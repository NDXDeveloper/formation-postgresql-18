# 14.7.3. Prometheus + postgres_exporter + Grafana

## Introduction

Surveiller une base de données PostgreSQL en production ne se limite pas à consulter occasionnellement les logs ou à exécuter quelques requêtes de diagnostic. Il faut un **système de monitoring continu, automatisé et visuel** qui vous alerte immédiatement en cas de problème.

**La stack Prometheus + postgres_exporter + Grafana** est devenue la solution de référence pour le monitoring moderne des bases de données PostgreSQL. Elle combine :

- 🔍 **Prometheus** : Collecte et stockage des métriques  
- 📊 **postgres_exporter** : Extraction des métriques PostgreSQL  
- 📈 **Grafana** : Visualisation et tableaux de bord interactifs

Cette stack est :
- ✅ **Open-source** et gratuite  
- ✅ **Standard de l'industrie** (utilisée par Google, Netflix, Uber...)  
- ✅ **Cloud-native** (parfaite pour Kubernetes)  
- ✅ **Hautement configurable** et extensible  
- ✅ **Puissante** pour l'alerting

### Pourquoi cette stack ?

**Avant** (approche traditionnelle) :
```
┌──────────────────────────────────────────┐
│ DBA connecté manuellement à PostgreSQL   │
│ ↓                                        │
│ SELECT * FROM pg_stat_activity;          │
│ ↓                                        │

│ "Pourquoi personne n'a rien vu ?!"       │

```

**Avec la stack Prometheus/Grafana** :
```

│ ↓                                        │

└──────────────────────────────────────────┘
```

## Comprendre les composants

### 1. Prometheus : Le collecteur de métriques

#### Qu'est-ce que Prometheus ?

**Prometheus** est un système de monitoring et d'alerting open-source créé par SoundCloud en 2012, devenu projet incubé par la Cloud Native Computing Foundation (CNCF).

**Philosophie** :
- Modèle **pull** : Prometheus interroge activement les sources de métriques
- Stockage **time-series** : Chaque métrique est associée à un timestamp
- Langage de requête puissant : **PromQL** (Prometheus Query Language)
- Architecture **distribuée** : Pas de point unique de défaillance

#### Comment fonctionne Prometheus ?

```
┌─────────────────────────────────────────────────────┐
│                   Prometheus Server                 │
│                                                     │
│  ┌──────────────┐       ┌──────────────┐            │
│  │   Scraper    │──────▶│   TSDB       │            │
│  │ (Collecteur) │       │(Time Series) │            │
│  └──────┬───────┘       └──────────────┘            │
│         │                                           │
│         │ Pull HTTP GET                             │
│         │ toutes les 15s                            │
└─────────┼───────────────────────────────────────────┘
          │
          ▼
┌─────────────────────┐    ┌─────────────────────┐
│ postgres_exporter   │    │ node_exporter       │
│ (métriques PG)      │    │ (métriques système) │
│ :9187/metrics       │    │ :9100/metrics       │
└─────────────────────┘    └─────────────────────┘
```

**Cycle de collecte** :
1. Prometheus envoie une requête HTTP GET vers `/metrics` toutes les N secondes (ex: 15s)  
2. L'exporter répond avec les métriques au format texte  
3. Prometheus stocke ces métriques avec timestamp dans sa base de données  
4. Les données sont conservées selon la rétention configurée (ex: 30 jours)

#### Format des métriques Prometheus

Les métriques sont exposées au format texte simple :

```
# HELP pg_up Whether the PostgreSQL server is up
# TYPE pg_up gauge
pg_up{server="production-db"} 1

# HELP pg_stat_database_numbackends Number of backends currently connected
# TYPE pg_stat_database_numbackends gauge
pg_stat_database_numbackends{datname="myapp"} 45

# HELP pg_stat_database_xact_commit Transactions committed
# TYPE pg_stat_database_xact_commit counter
pg_stat_database_xact_commit{datname="myapp"} 1234567
```

**Structure** :
- `# HELP` : Description de la métrique  
- `# TYPE` : Type de métrique (gauge, counter, histogram, summary)
- Nom de la métrique + labels + valeur

### 2. postgres_exporter : Le traducteur PostgreSQL

#### Qu'est-ce que postgres_exporter ?

**postgres_exporter** est un agent qui :
1. Se connecte à PostgreSQL  
2. Exécute des requêtes SQL prédéfinies  
3. Transforme les résultats en métriques Prometheus  
4. Expose ces métriques sur un endpoint HTTP

**Rôle** : C'est le pont entre PostgreSQL (qui ne parle pas nativement Prometheus) et Prometheus.

#### Architecture de postgres_exporter

```
┌──────────────────────────────────────────┐
│         postgres_exporter                │
│                                          │
│  ┌────────────────┐                      │
│  │ SQL Queries    │                      │
│  │ Executor       │                      │
│  └────────┬───────┘                      │
│           │                              │
│           ▼                              │
│  ┌────────────────┐    ┌──────────────┐  │
│  │ Metric         │───▶│ HTTP Server  │  │
│  │ Converter      │    │ :9187        │  │
│  └────────────────┘    └──────────────┘  │
└───────────┬──────────────────────────────┘
            │
            │ Connexion PostgreSQL
            ▼
┌──────────────────────────────────────────┐
│         PostgreSQL Server                │
│                                          │
│  • pg_stat_database                      │
│  • pg_stat_activity                      │
│  • pg_stat_bgwriter                      │
│  • pg_stat_replication                   │
│  • ... et bien d'autres vues système     │
└──────────────────────────────────────────┘
```

#### Métriques collectées par défaut

postgres_exporter collecte automatiquement des dizaines de métriques :

**Santé et disponibilité** :
- `pg_up` : Le serveur PostgreSQL répond-il ?  
- `pg_stat_database_numbackends` : Nombre de connexions actives  
- `pg_database_size_bytes` : Taille des bases de données

**Performance des requêtes** :
- `pg_stat_database_xact_commit` : Transactions committées  
- `pg_stat_database_xact_rollback` : Transactions annulées  
- `pg_stat_database_blks_read` : Blocs lus depuis le disque  
- `pg_stat_database_blks_hit` : Blocs lus depuis le cache

**Réplication** :
- `pg_stat_replication_lag` : Retard de réplication  
- `pg_stat_replication_state` : État de la réplication

**Tables et index** :
- `pg_stat_user_tables_n_tup_ins` : Lignes insérées  
- `pg_stat_user_tables_n_tup_upd` : Lignes modifiées  
- `pg_stat_user_indexes_idx_scan` : Scans d'index

### 3. Grafana : La vitrine visuelle

#### Qu'est-ce que Grafana ?

**Grafana** est une plateforme de visualisation et d'analyse de données open-source.

**Fonctionnalités principales** :
- 📊 **Dashboards** : Créer des tableaux de bord avec graphiques interactifs  
- 🔔 **Alerting** : Définir des alertes et notifications  
- 👥 **Multi-utilisateurs** : Gestion des permissions et équipes  
- 🔌 **Data sources** : Connexion à Prometheus, InfluxDB, PostgreSQL, etc.  
- 📤 **Export/Import** : Partager des dashboards avec la communauté

#### Pourquoi Grafana ?

Sans Grafana, vous interrogez Prometheus manuellement :
```
# Requête PromQL dans le terminal
curl 'http://prometheus:9090/api/v1/query?query=pg_up'

# Résultat brut JSON
{"status":"success","data":{"resultType":"vector","result":[...]}}
```

Avec Grafana, vous obtenez :
```
┌────────────────────────────────────────┐
│        Dashboard PostgreSQL            │
├────────────────────────────────────────┤
│  ●●●●  Database Health: ✓ UP           │
│                                        │
│  [===== CPU: 45% =====]                │
│  [===== Memory: 78% ===]               │
│                                        │
│  📊 Connections: 150/200               │
│  📈 QPS: 1,245 queries/sec             │
│  💾 Cache Hit Ratio: 99.2%             │
│                                        │
│  [Graphique temps réel]                │
└────────────────────────────────────────┘
```


### Flux de données

1. **PostgreSQL** expose ses statistiques via les vues système (`pg_stat_*`)  
2. **postgres_exporter** interroge ces vues et les transforme en métriques Prometheus  
3. **Prometheus** collecte ces métriques périodiquement et les stocke  
4. **Grafana** interroge Prometheus pour afficher les graphiques  
5. **Utilisateurs** consultent les dashboards Grafana dans leur navigateur

## Installation et configuration

### Prérequis

- **Système** : Linux (Ubuntu, Debian, CentOS, RHEL)  
- **PostgreSQL** : Version 10+ (idéalement 18)  
- **Mémoire** : 2GB minimum (4GB recommandé)  
- **Réseau** : Ports 9090 (Prometheus), 9187 (postgres_exporter), 3000 (Grafana)

### Option 1 : Installation via Docker (Recommandé pour débuter)

#### Étape 1 : Créer le fichier docker-compose.yml

```yaml
version: '3.8'

services:
  # PostgreSQL (si vous n'en avez pas déjà un)
  postgres:
    image: postgres:18
    container_name: postgres
    environment:
      POSTGRES_USER: myuser
      POSTGRES_PASSWORD: mypassword
      POSTGRES_DB: myapp
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    networks:
      - monitoring

  # postgres_exporter
  postgres_exporter:
    image: prometheuscommunity/postgres-exporter:latest
    container_name: postgres_exporter
    environment:
      DATA_SOURCE_NAME: "postgresql://myuser:mypassword@postgres:5432/myapp?sslmode=disable"
    ports:
      - "9187:9187"
    networks:
      - monitoring
    depends_on:
      - postgres

  # Prometheus
  prometheus:
    image: prom/prometheus:latest
    container_name: prometheus
    ports:
      - "9090:9090"
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
      - prometheus_data:/prometheus
    command:
      - '--config.file=/etc/prometheus/prometheus.yml'
      - '--storage.tsdb.path=/prometheus'
      - '--storage.tsdb.retention.time=30d'
    networks:
      - monitoring

  # Grafana
  grafana:
    image: grafana/grafana:latest
    container_name: grafana
    ports:
      - "3000:3000"
    environment:
      - GF_SECURITY_ADMIN_USER=admin
      - GF_SECURITY_ADMIN_PASSWORD=admin
      - GF_INSTALL_PLUGINS=grafana-clock-panel
    volumes:
      - grafana_data:/var/lib/grafana
    networks:
      - monitoring

volumes:
  postgres_data:
  prometheus_data:
  grafana_data:

networks:
  monitoring:
    driver: bridge
```

#### Étape 2 : Créer le fichier prometheus.yml

```yaml
# Configuration globale
global:
  scrape_interval: 15s      # Collecte toutes les 15 secondes
  evaluation_interval: 15s  # Évaluation des règles toutes les 15s

# Configuration des targets à scraper
scrape_configs:
  # Prometheus lui-même
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']

  # PostgreSQL via postgres_exporter
  - job_name: 'postgresql'
    static_configs:
      - targets: ['postgres_exporter:9187']
        labels:
          instance: 'production-db'
          environment: 'production'
```

#### Étape 3 : Démarrer la stack

```bash
# Démarrer tous les services
docker-compose up -d

# Vérifier que tout fonctionne
docker-compose ps

# Logs de postgres_exporter
docker-compose logs -f postgres_exporter
```

### Option 2 : Installation native (Production)

#### Étape 1 : Installer Prometheus

```bash
# Télécharger Prometheus
cd /tmp  
wget https://github.com/prometheus/prometheus/releases/download/v2.48.0/prometheus-2.48.0.linux-amd64.tar.gz  

# Extraire
tar xvfz prometheus-*.tar.gz  
cd prometheus-*  

# Déplacer les binaires
sudo mv prometheus promtool /usr/local/bin/

# Créer les répertoires
sudo mkdir -p /etc/prometheus /var/lib/prometheus

# Copier les fichiers de config
sudo mv prometheus.yml /etc/prometheus/  
sudo mv consoles/ console_libraries/ /etc/prometheus/  

# Créer un utilisateur système
sudo useradd --no-create-home --shell /bin/false prometheus

# Permissions
sudo chown -R prometheus:prometheus /etc/prometheus /var/lib/prometheus
```

**Créer le service systemd** : `/etc/systemd/system/prometheus.service`

```ini
[Unit]
Description=Prometheus Monitoring System  
Documentation=https://prometheus.io/docs/  
After=network.target  

[Service]
User=prometheus  
Group=prometheus  
Type=simple  
ExecStart=/usr/local/bin/prometheus \  
  --config.file=/etc/prometheus/prometheus.yml \
  --storage.tsdb.path=/var/lib/prometheus/ \
  --storage.tsdb.retention.time=30d \
  --web.console.templates=/etc/prometheus/consoles \
  --web.console.libraries=/etc/prometheus/console_libraries

Restart=on-failure  
RestartSec=5  

[Install]
WantedBy=multi-user.target
```

**Démarrer Prometheus** :
```bash
sudo systemctl daemon-reload  
sudo systemctl start prometheus  
sudo systemctl enable prometheus  
sudo systemctl status prometheus  
```

**Important** : Créer un utilisateur PostgreSQL dédié au monitoring :

```sql
-- Se connecter à PostgreSQL

-- Créer l'utilisateur de monitoring

-- Permissions minimales nécessaires
```

## Métriques essentielles à surveiller

### 1. Santé globale (Health)

#### pg_up

**Description** : PostgreSQL est-il accessible ?

**Valeur** :
- `1` : Base accessible ✅  
- `0` : Base inaccessible ❌

**PromQL** :
```promql
pg_up
```

**Alerte** :
```yaml
- alert: PostgreSQLDown
  expr: pg_up == 0
  for: 1m
  annotations:
    summary: "PostgreSQL is down"
```

#### pg_database_size_bytes

**Description** : Taille de chaque base de données en octets

**PromQL** :
```promql
# Taille en GB
pg_database_size_bytes / 1024 / 1024 / 1024

# Croissance sur 24h
delta(pg_database_size_bytes[24h]) / 1024 / 1024 / 1024
```

**Alerte** :
```yaml
- alert: DatabaseSizeGrowing
  expr: delta(pg_database_size_bytes[24h]) > 10737418240  # >10GB/jour
  for: 1h
  annotations:
    summary: "Database growing rapidly"
```

### 2. Connexions (Connections)

#### pg_stat_database_numbackends

**Description** : Nombre de connexions actives par base

**PromQL** :
```promql
# Connexions totales
sum(pg_stat_database_numbackends)

# Pourcentage d'utilisation
sum(pg_stat_database_numbackends) / pg_settings_max_connections * 100

# Par base de données
pg_stat_database_numbackends{datname="myapp"}
```

**Alerte** :
```yaml
- alert: TooManyConnections
  expr: sum(pg_stat_database_numbackends) / pg_settings_max_connections > 0.8
  for: 5m
  annotations:
    summary: "Connection pool usage > 80%"
```

#### pg_settings_max_connections

**Description** : Nombre maximum de connexions autorisées

**PromQL** :
```promql
pg_settings_max_connections
```

### 3. Performance des transactions

#### pg_stat_database_xact_commit / xact_rollback

**Description** : Nombre de transactions committées vs annulées

**PromQL** :
```promql
# Taux de commit par seconde
rate(pg_stat_database_xact_commit[5m])

# Taux de rollback par seconde
rate(pg_stat_database_xact_rollback[5m])

# Ratio rollback/commit (doit être faible)
rate(pg_stat_database_xact_rollback[5m]) / rate(pg_stat_database_xact_commit[5m])
```

**Alerte** :
```yaml
- alert: HighRollbackRate
  expr: rate(pg_stat_database_xact_rollback[5m]) / rate(pg_stat_database_xact_commit[5m]) > 0.1
  for: 10m
  annotations:
    summary: "Rollback rate > 10%"
```

### 4. Cache Hit Ratio (Performance I/O)

#### pg_stat_database_blks_hit / blks_read

**Description** : Ratio de lectures depuis le cache vs le disque

**PromQL** :
```promql
# Cache hit ratio (devrait être > 95%)
sum(rate(pg_stat_database_blks_hit[5m])) /
(sum(rate(pg_stat_database_blks_hit[5m])) + sum(rate(pg_stat_database_blks_read[5m]))) * 100

# Par base de données
rate(pg_stat_database_blks_hit{datname="myapp"}[5m]) /
(rate(pg_stat_database_blks_hit{datname="myapp"}[5m]) + rate(pg_stat_database_blks_read{datname="myapp"}[5m])) * 100
```

**Interprétation** :
- **> 99%** : Excellent ✅  
- **95-99%** : Bon ✅  
- **90-95%** : À surveiller ⚠️  
- **< 90%** : Problème, augmenter `shared_buffers` ❌

**Alerte** :
```yaml
- alert: LowCacheHitRatio
  expr: |
    sum(rate(pg_stat_database_blks_hit[5m])) /
    (sum(rate(pg_stat_database_blks_hit[5m])) + sum(rate(pg_stat_database_blks_read[5m]))) < 0.95
  for: 10m
  annotations:
    summary: "Cache hit ratio < 95%"
```

### 5. Réplication (pour les setups HA)

#### pg_stat_replication_lag

**Description** : Retard de réplication en secondes

**PromQL** :
```promql
# Retard en secondes
pg_stat_replication_lag_seconds

# Retard en bytes
pg_stat_replication_lag_bytes
```

**Alerte** :
```yaml

  expr: pg_stat_replication_lag_seconds > 60
  for: 5m
  annotations:
    summary: "Replication lag > 60s"
```

### 6. Locks (Verrous)

#### pg_stat_database_deadlocks

**Description** : Nombre de deadlocks détectés

**PromQL** :
```promql
# Deadlocks par seconde
rate(pg_stat_database_deadlocks[5m])

# Total sur 1h
increase(pg_stat_database_deadlocks[1h])
```

**Alerte** :
```yaml
- alert: FrequentDeadlocks
  expr: increase(pg_stat_database_deadlocks[1h]) > 5
  for: 5m
  annotations:
    summary: "More than 5 deadlocks in the last hour"
```

### 7. Tables et index

#### pg_stat_user_tables_n_tup_*

**Description** : Opérations sur les tables (INSERT, UPDATE, DELETE)

**PromQL** :
```promql
# Taux d'insertion par seconde
rate(pg_stat_user_tables_n_tup_ins[5m])

# Taux de mise à jour
rate(pg_stat_user_tables_n_tup_upd[5m])

# Taux de suppression
rate(pg_stat_user_tables_n_tup_del[5m])
```

#### pg_stat_user_tables_n_live_tup / n_dead_tup

**Description** : Lignes vivantes vs lignes mortes (bloat)

**PromQL** :
```promql
# Pourcentage de dead tuples
pg_stat_user_tables_n_dead_tup /
(pg_stat_user_tables_n_live_tup + pg_stat_user_tables_n_dead_tup) * 100
```

**Alerte** :
```yaml
- alert: HighTableBloat
  expr: |
    pg_stat_user_tables_n_dead_tup /
    (pg_stat_user_tables_n_live_tup + pg_stat_user_tables_n_dead_tup) > 0.2
  for: 30m
  annotations:
    summary: "Table bloat > 20%"
```

### 8. Checkpoints

#### pg_stat_bgwriter_checkpoints_*

**Description** : Statistiques sur les checkpoints

**PromQL** :
```promql
# Checkpoints planifiés (bons)
rate(pg_stat_bgwriter_checkpoints_timed[5m])

# Checkpoints forcés (mauvais si trop nombreux)
rate(pg_stat_bgwriter_checkpoints_req[5m])

# Ratio de checkpoints forcés
rate(pg_stat_bgwriter_checkpoints_req[5m]) /
(rate(pg_stat_bgwriter_checkpoints_timed[5m]) + rate(pg_stat_bgwriter_checkpoints_req[5m])) * 100
```

**Alerte** :
```yaml
- alert: TooManyForcedCheckpoints
  expr: |
    rate(pg_stat_bgwriter_checkpoints_req[5m]) /
    (rate(pg_stat_bgwriter_checkpoints_timed[5m]) + rate(pg_stat_bgwriter_checkpoints_req[5m])) > 0.5
  for: 10m
  annotations:
    summary: "Forced checkpoints > 50%"
    description: "Consider increasing max_wal_size"
```

### 2. Performance et dimensionnement

#### Optimiser la rétention Prometheus

**Problème** : Prometheus peut consommer beaucoup de disque.

**Configuration dans prometheus.yml** :
```yaml
global:
  scrape_interval: 15s         # Fréquence de collecte
  evaluation_interval: 15s

# Ligne de commande
--storage.tsdb.retention.time=30d     # Conserver 30 jours
--storage.tsdb.retention.size=50GB    # Limiter à 50GB
```

**Calcul de l'espace nécessaire** :
```
Espace = nb_métriques × échantillons_par_jour × taille_échantillon × rétention_jours

Exemple :
1000 métriques × 5760 échantillons/jour × 2 bytes × 30 jours = 345 MB
```

#### Optimiser les requêtes PromQL

**Mauvais** (calcule sur toutes les séries) :
```promql
rate(pg_stat_database_xact_commit[5m])
```

**Bon** (filtre dès le début) :
```promql
rate(pg_stat_database_xact_commit{datname="myapp"}[5m])
```

**Utiliser le recording rules** pour les calculs complexes :

```yaml
# prometheus.yml
rule_files:
  - "rules.yml"

# rules.yml
groups:
  - name: postgresql
    interval: 30s
    rules:
      - record: job:pg_cache_hit_ratio:rate5m
        expr: |
          sum(rate(pg_stat_database_blks_hit[5m])) /
          (sum(rate(pg_stat_database_blks_hit[5m])) + sum(rate(pg_stat_database_blks_read[5m])))
```

```yaml
scrape_configs:

      - targets: ['postgres-exporter-dev:9187']
        labels:
          instance: 'development'
          environment: 'dev'
          datacenter: 'eu-west-1'
```

**Dashboard avec variables** :

Variable `environment` :
```promql
label_values(pg_up, environment)
```

Panel filtré :
```promql
pg_stat_database_numbackends{environment="$environment"}
```

### 2. Alerting sur tendances

**Détecter une croissance anormale** :

```promql
# Croissance de la base > 10GB en 24h
delta(pg_database_size_bytes[24h]) > 10737418240

# Augmentation soudaine des connexions (>50% en 10min)
(pg_stat_database_numbackends - pg_stat_database_numbackends offset 10m) /
pg_stat_database_numbackends offset 10m > 0.5

# Dégradation progressive du cache hit ratio
(avg_over_time(job:pg_cache_hit_ratio:rate5m[7d] offset 7d) -
 avg_over_time(job:pg_cache_hit_ratio:rate5m[7d])) > 0.05
```

### 3. Corrélation avec les métriques système

**Ajouter node_exporter** pour les métriques OS :

```yaml
# docker-compose.yml
node_exporter:
  image: prom/node-exporter:latest
  ports:
    - "9100:9100"
  volumes:
    - /proc:/host/proc:ro
    - /sys:/host/sys:ro
  command:
    - '--path.procfs=/host/proc'
    - '--path.sysfs=/host/sys'
```

**Dashboard combiné** :

Panel 1 : CPU PostgreSQL vs CPU Système
```promql
# CPU PostgreSQL (via pg_stat_kcache si disponible)
rate(pg_stat_kcache_exec_user_time[5m])

# CPU Système total
100 - (avg(irate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)
```

Panel 2 : I/O PostgreSQL vs I/O Disque
```promql
# I/O PostgreSQL
rate(pg_stat_database_blks_read[5m])

# I/O Disque (device sda)
rate(node_disk_read_bytes_total{device="sda"}[5m])
```

### 4. SLO/SLI monitoring

**Définir des SLOs (Service Level Objectives)** :

**SLI 1** : Disponibilité > 99.9%
```promql
# Uptime ratio sur 30 jours
avg_over_time(pg_up[30d]) * 100
```

**SLI 2** : 95% des requêtes < 100ms
```promql
# Nécessite pg_stat_statements + custom exporter
histogram_quantile(0.95, rate(pg_query_duration_bucket[5m]))
```

**Dashboard SLO** avec "burn rate" :
```promql
# Error budget restant
1 - ((1 - avg_over_time(pg_up[30d])) / 0.001)
```

## Troubleshooting

### Problème 1 : Prometheus ne collecte pas les métriques

**Symptômes** :
- Target est en état "DOWN" dans Prometheus
- Pas de données dans Grafana

**Diagnostic** :
```bash
# Vérifier que postgres_exporter écoute
netstat -tulpn | grep 9187

# Tester l'endpoint manuellement
curl http://localhost:9187/metrics

# Vérifier les logs
docker logs postgres_exporter
# ou
journalctl -u postgres_exporter -f
```

**Causes courantes** :
1. postgres_exporter n'arrive pas à se connecter à PostgreSQL  
2. Firewall bloque le port 9187  
3. Configuration DATA_SOURCE_NAME incorrecte

**Solution** :
```bash
# Tester la connexion PostgreSQL
psql "postgresql://monitoring_user:password@localhost:5432/postgres"

# Vérifier les permissions
psql -U monitoring_user -d postgres -c "SELECT * FROM pg_stat_database LIMIT 1;"
```

### Problème 2 : Métriques manquantes

**Symptôme** : Certaines métriques ne s'affichent pas.

**Cause** : postgres_exporter ne collecte que les métriques de base par défaut.

**Solution** : Activer les métriques étendues.

**Créer un fichier queries.yaml** :
```yaml
pg_replication:
  query: "SELECT CASE WHEN pg_last_wal_receive_lsn() = pg_last_wal_replay_lsn() THEN 0 ELSE EXTRACT(EPOCH FROM NOW() - pg_last_xact_replay_timestamp()) END AS lag"
  master: true
  metrics:
    - lag:
        usage: "GAUGE"
        description: "Replication lag behind master in seconds"

pg_postmaster:
  query: "SELECT pg_postmaster_start_time() as start_time_seconds"
  metrics:
    - start_time_seconds:
        usage: "GAUGE"
        description: "Time at which postmaster started"
```

**Charger le fichier** :
```bash
# Ligne de commande
postgres_exporter --extend.query-path=/etc/postgres_exporter/queries.yaml

# Docker
docker run -v /path/to/queries.yaml:/queries.yaml \
  -e PG_EXPORTER_EXTEND_QUERY_PATH=/queries.yaml \
  postgres_exporter
```

### Problème 3 : Dashboards vides après import

**Cause** : Data source non configurée ou nom différent.

**Solution** :
1. Vérifier que la data source existe : Configuration → Data Sources  
2. Éditer le dashboard : Settings → Variables  
3. Changer la data source dans chaque variable/panel si nécessaire

### Problème 4 : Alertes qui ne se déclenchent pas

**Diagnostic** :
1. Vérifier l'état de l'alerte : Alerting → Alert Rules  
2. Tester la condition manuellement dans un panel  
3. Vérifier le canal de notification : Test

**Erreur courante** : Alerte configurée dans Grafana mais pas dans Prometheus.

**Solution** : Utiliser soit Grafana Alerting, soit Prometheus Alertmanager, pas les deux.

### Problème 5 : Performance dégradée de Prometheus

**Symptômes** :
- Prometheus lent à répondre
- Grafana timeout sur les requêtes
- CPU élevé sur le serveur Prometheus

**Causes** :
- Trop de métriques collectées
- Rétention trop longue
- Requêtes PromQL non optimisées

**Solutions** :
```yaml
# 1. Réduire la fréquence de scraping
global:
  scrape_interval: 30s  # Au lieu de 15s

# 2. Réduire la rétention
--storage.tsdb.retention.time=15d  # Au lieu de 30d

# 3. Filtrer les métriques inutiles
scrape_configs:
  - job_name: 'postgresql'
    metric_relabel_configs:
      - source_labels: [__name__]
        regex: 'pg_stat_user_tables_.*'
        action: drop  # Ne pas collecter les métriques par table
```

## Comparaison avec d'autres solutions

### Prometheus/Grafana vs Solutions propriétaires

| Critère | Prometheus/Grafana | Datadog | New Relic | CloudWatch |
|---------|-------------------|---------|-----------|------------|
| **Coût** | Gratuit | $15-$23/host/mois | $25+/host/mois | Pay-as-you-go |
| **Setup** | Manuel, complet | Simple, agent | Simple, agent | Automatique (AWS) |
| **Flexibilité** | Très élevée | Moyenne | Moyenne | Faible |
| **On-premise** | Oui | Non | Non | Non (AWS only) |
| **Communauté** | Très active | Supportée | Supportée | AWS |
| **Alerting** | Avancé | Avancé | Avancé | Basique |
| **Rétention** | Configurable | 15 mois | Variable | 15 mois |

**Quand choisir Prometheus/Grafana** :
- ✅ Budget limité (open-source)  
- ✅ On-premise ou multi-cloud  
- ✅ Besoin de contrôle total  
- ✅ Équipe technique capable de maintenir

**Quand choisir une solution propriétaire** :
- ✅ Budget disponible  
- ✅ Équipe petite, besoin de simplicité  
- ✅ Besoin d'APM intégré (traçage distribué)  
- ✅ Support 24/7 requis

### Prometheus vs InfluxDB

| Critère | Prometheus | InfluxDB |
|---------|-----------|----------|
| **Modèle** | Pull (scraping) | Push (agents) |
| **Stockage** | Time-series optimisé | Time-series avec SQL |
| **Requêtes** | PromQL | InfluxQL, Flux |
| **Clustering** | Fédération | Natif (InfluxDB Enterprise) |
| **Cas d'usage** | Monitoring infra/apps | IoT, analytics temps réel |

**Alerting** :
- [ ] Canaux de notification configurés (Slack, email)  
- [ ] Alertes critiques définies (PostgreSQL Down, High connections)  
- [ ] Alertes de performance (Low cache, High bloat)  
- [ ] Tests d'alertes effectués

**Sécurité** :
- [ ] Authentification activée sur tous les services  
- [ ] Permissions PostgreSQL minimales (pg_monitor)  
- [ ] Reverse proxy configuré (Nginx)  
- [ ] SSL/TLS activé

**Documentation** :
- [ ] Runbooks créés pour chaque alerte  
- [ ] Annotations ajoutées aux panels  
- [ ] Dashboards versionnés dans Git  
- [ ] Équipe formée à l'utilisation

### Prochaines étapes

Après avoir maîtrisé cette stack, vous pouvez explorer :

1. **Prometheus Alertmanager** : Routage avancé des alertes  
2. **Thanos** : Stockage long terme et haute disponibilité pour Prometheus  
3. **Loki** : Agrégation de logs (complément parfait)  
4. **Tempo** : Traçage distribué (APM)  
5. **Service Mesh** : Istio/Linkerd pour monitoring microservices

**La stack Prometheus/Grafana transforme votre monitoring réactif en monitoring proactif.** Vous détectez les problèmes avant qu'ils n'impactent vos utilisateurs.

## Glossaire

| Terme | Définition |
|-------|------------|
| **Scraping** | Action de collecter les métriques par pull HTTP |
| **Target** | Source de métriques (endpoint HTTP) |
| **TSDB** | Time-Series Database (base de données temporelle) |
| **PromQL** | Prometheus Query Language (langage de requête) |
| **Exporter** | Agent qui expose des métriques au format Prometheus |
| **Data source** | Source de données dans Grafana (Prometheus, InfluxDB...) |
| **Panel** | Widget individuel dans un dashboard Grafana |
| **Dashboard** | Tableau de bord composé de plusieurs panels |
| **SLO** | Service Level Objective (objectif de niveau de service) |
| **SLI** | Service Level Indicator (indicateur de niveau de service) |
| **Gauge** | Métrique qui peut monter et descendre |
| **Counter** | Métrique qui ne fait qu'augmenter |
| **Histogram** | Distribution de valeurs dans des buckets |
| **Recording rule** | Pré-calcul de requêtes PromQL complexes |
| **Alertmanager** | Composant Prometheus pour le routage des alertes |
