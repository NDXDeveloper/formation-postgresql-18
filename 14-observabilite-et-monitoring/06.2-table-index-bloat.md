# 14.6.2. Table et Index Bloat

## Introduction

Le **Bloat** (gonflement en français) est l'un des problèmes de performance les plus courants dans PostgreSQL, mais aussi l'un des moins bien compris par les débutants. Il s'agit d'un phénomène naturel qui se produit au fil du temps dans toutes les bases de données PostgreSQL actives.

Dans ce chapitre, nous allons comprendre ce qu'est le bloat, pourquoi il se produit, comment le détecter, et surtout comment le gérer efficacement.

---

## Comprendre le Concept de Bloat

### Une Analogie Simple : Le Parking

Imaginez un parking de 100 places :

**Situation normale :**
- 80 voitures garées
- 20 places vides réparties uniformément
- Vous trouvez facilement une place

**Situation avec "bloat" :**
- Seules 40 voitures sont réellement présentes
- Mais 40 autres emplacements sont marqués "réservés" avec des cônes alors qu'aucune voiture n'y est
- Le parking semble rempli mais il est en réalité à moitié vide
- Vous devez parcourir tout le parking pour trouver les vraies places disponibles

C'est exactement ce qui se passe avec le bloat dans PostgreSQL : de l'espace occupé inutilement qui ralentit l'accès aux vraies données.

### Le Bloat dans PostgreSQL

Le **bloat** est un espace disque occupé par des données obsolètes ou inutilisées qui n'ont pas encore été récupérées par PostgreSQL.

**En termes simples :**
- Votre table fait 10 GB sur le disque
- Mais seulement 6 GB contiennent des données réellement utilisables
- Les 4 GB restants sont du "bloat" : de l'espace gaspillé

---

## Pourquoi le Bloat Existe-t-il ?

### Le Principe MVCC (Multiversion Concurrency Control)

Pour comprendre le bloat, il faut comprendre comment PostgreSQL gère les modifications de données.

#### Scénario : Mise à Jour d'une Ligne

**Dans une base de données classique :**
```
Ligne originale : [id=1, nom="Alice", age=25]  
UPDATE users SET age = 26 WHERE id = 1;  
Résultat : La ligne est modifiée directement  
```

**Dans PostgreSQL (MVCC) :**
```
Ligne originale : [id=1, nom="Alice", age=25]  ← Marquée comme "ancienne version"  
UPDATE users SET age = 26 WHERE id = 1;  
Nouvelle ligne : [id=1, nom="Alice", age=26]   ← Créée comme nouvelle version  
```

**PostgreSQL ne supprime JAMAIS une ligne directement !**

Pourquoi ? Pour permettre aux transactions concurrentes de continuer à lire l'ancienne version si nécessaire (isolation transactionnelle).

### Les Opérations Qui Créent du Bloat

#### 1. UPDATE

Chaque `UPDATE` crée une nouvelle version de la ligne et marque l'ancienne comme obsolète.

**Exemple :**
```sql
-- Table initiale : 1000 lignes
SELECT pg_size_pretty(pg_relation_size('users'));
-- Résultat : 80 KB

-- 1000 updates
UPDATE users SET last_login = NOW();

-- Taille après updates
SELECT pg_size_pretty(pg_relation_size('users'));
-- Résultat : 160 KB (doublé !)
```

Les anciennes versions prennent de la place jusqu'à ce qu'elles soient nettoyées.

#### 2. DELETE

Les `DELETE` ne libèrent pas l'espace immédiatement. Les lignes sont marquées comme "mortes" mais restent physiquement sur le disque.

**Exemple :**
```sql
-- Table avec 10 000 lignes
DELETE FROM old_logs WHERE date < '2020-01-01';
-- 9 000 lignes supprimées

-- La taille de la table ne change pas immédiatement !
```

#### 3. INSERT Suivi de DELETE (Pattern Rotation)

Les tables avec beaucoup d'insertions et suppressions (logs, queues, sessions) accumulent rapidement du bloat.

**Exemple typique :**
```sql
-- Tous les jours :
INSERT INTO session_logs (...) VALUES (...);  -- 100 000 insertions  
DELETE FROM session_logs WHERE created_at < NOW() - INTERVAL '7 days';  -- 100 000 suppressions  
```

---

## Types de Bloat

### 1. Table Bloat (Bloat de Table)

De l'espace mort dans les fichiers de données des tables.

**Symptômes :**
- Tables anormalement volumineuses
- Scans séquentiels de plus en plus lents
- Sauvegardes de plus en plus grosses

### 2. Index Bloat (Bloat d'Index)

De l'espace mort dans les structures d'index.

**Symptômes :**
- Index très volumineux par rapport aux données
- Requêtes utilisant des index de plus en plus lentes
- Ratio taille_index / taille_table anormal

**Important :** Les index peuvent gonfler plus vite que les tables car ils référencent toutes les versions de lignes, même obsolètes.

---

## Impact du Bloat sur les Performances

### 1. Performances de Requêtes Dégradées

**Sans bloat :**
```
Table 100 GB, 100 millions de lignes utiles  
Scan complet : 10 secondes  
```

**Avec 50% de bloat :**
```
Table 200 GB, 100 millions de lignes utiles + 100 GB de bloat  
Scan complet : 20 secondes (2× plus lent !)  
```

PostgreSQL doit parcourir le bloat pour trouver les vraies données.

### 2. Utilisation Excessive du Disque

- Coûts de stockage augmentés
- Sauvegardes plus volumineuses et plus longues
- Réplication plus lente (plus de données à transférer)

### 3. Cache Moins Efficace

Le bloat occupe de la place dans le cache (shared_buffers), réduisant l'espace disponible pour les vraies données.

**Résultat :** Cache Hit Ratio plus faible.

### 4. VACUUM Plus Lents

Plus il y a de bloat, plus le VACUUM met de temps à s'exécuter, créant un cercle vicieux.

## Mesurer le Bloat

### Requête Simple : Estimation du Bloat

Voici une requête qui estime le pourcentage de bloat dans vos tables :

```sql
SELECT
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS total_size,
    pg_size_pretty(pg_relation_size(schemaname||'.'||tablename)) AS table_size,
    round(100 * pg_relation_size(schemaname||'.'||tablename) /
          NULLIF(pg_total_relation_size(schemaname||'.'||tablename), 0), 2) AS table_percent,
    n_dead_tup AS dead_tuples,
    n_live_tup AS live_tuples,
    round(100 * n_dead_tup::numeric / NULLIF(n_live_tup + n_dead_tup, 0), 2) AS dead_tuple_percent
FROM pg_stat_user_tables  
WHERE n_live_tup > 0  
ORDER BY n_dead_tup DESC;  
```

#### Interprétation des Colonnes

- **total_size** : Taille totale (table + index + TOAST)  
- **table_size** : Taille de la table seule  
- **dead_tuples** : Nombre de lignes mortes (obsolètes)  
- **live_tuples** : Nombre de lignes vivantes (actuelles)  
- **dead_tuple_percent** : Pourcentage de bloat estimé

#### Seuils d'Alerte

| Pourcentage | État | Action |
|-------------|------|--------|
| **< 10%** | 🟢 Normal | Aucune action nécessaire |
| **10-20%** | 🟡 Surveillé | Planifier un VACUUM |
| **20-40%** | 🟠 Préoccupant | VACUUM urgent recommandé |
| **> 40%** | 🔴 Critique | Action immédiate nécessaire |

### Requête Avancée : Estimation Plus Précise

Pour une estimation plus précise (mais plus complexe) :

```sql
SELECT
    schemaname,
    tablename,
    pg_size_pretty(bloat_size) AS bloat_size,
    round(bloat_ratio, 2) AS bloat_ratio_percent
FROM (
    SELECT
        schemaname,
        tablename,
        (bs*tblpages)::bigint AS real_size,
        ((tblpages-est_tblpages)*bs)::bigint AS bloat_size,
        CASE WHEN tblpages > 0
            THEN 100 * (tblpages-est_tblpages)::float / tblpages
            ELSE 0
        END AS bloat_ratio
    FROM (
        SELECT
            schemaname,
            tablename,
            cc.reltuples,
            cc.relpages AS tblpages,
            bs,
            CEIL((cc.reltuples*((datahdr+ma-
                (CASE WHEN datahdr%ma=0 THEN ma ELSE datahdr%ma END))+nullhdr2+4))/(bs-20::float)) AS est_tblpages
        FROM (
            SELECT
                ma, bs, schemaname, tablename,
                (datawidth+(hdr+ma-(case when hdr%ma=0 THEN ma ELSE hdr%ma END)))::numeric AS datahdr,
                (maxfracsum*(nullhdr+ma-(case when nullhdr%ma=0 THEN ma ELSE nullhdr%ma END))) AS nullhdr2
            FROM (
                SELECT
                    schemaname, tablename, hdr, ma, bs,
                    SUM((1-null_frac)*avg_width) AS datawidth,
                    MAX(null_frac) AS maxfracsum,
                    hdr+(
                        SELECT 1+count(*)/8
                        FROM pg_stats s2
                        WHERE null_frac<>0 AND s2.schemaname = s.schemaname AND s2.tablename = s.tablename
                    ) AS nullhdr
                FROM pg_stats s, (
                    SELECT
                        (SELECT current_setting('block_size')::numeric) AS bs,
                        CASE WHEN substring(v,12,3) IN ('8.0','8.1','8.2') THEN 27 ELSE 23 END AS hdr,
                        CASE WHEN v ~ 'mingw32' THEN 8 ELSE 4 END AS ma
                    FROM (SELECT version() AS v) AS foo
                ) AS constants
                GROUP BY 1,2,3,4,5
            ) AS foo
        ) AS rs
        JOIN pg_class cc ON cc.relname = rs.tablename
        JOIN pg_namespace nn ON cc.relnamespace = nn.oid AND nn.nspname = rs.schemaname
    ) AS sml
) AS bloat_info
WHERE bloat_ratio > 10  
ORDER BY bloat_size DESC;  
```

**Note :** Cette requête est complexe mais fournit une estimation plus réaliste basée sur l'analyse de la structure interne des données.

### Mesurer le Bloat des Index

```sql
SELECT
    schemaname,
    tablename,
    indexname,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
    idx_scan AS index_scans,
    idx_tup_read AS tuples_read,
    idx_tup_fetch AS tuples_fetched
FROM pg_stat_user_indexes  
WHERE idx_scan = 0  -- Index jamais utilisés (candidats au bloat ou à la suppression)  
ORDER BY pg_relation_size(indexrelid) DESC;  
```

Cette requête identifie :
- Les index volumineux
- Les index non utilisés (potentiellement inutiles ou gonflés)

## Le Processus VACUUM : Le Nettoyeur de PostgreSQL

### Qu'est-ce que VACUUM ?

**VACUUM** est le processus de nettoyage qui récupère l'espace occupé par les lignes mortes (bloat).

**Analogie :** C'est comme un éboueur qui passe régulièrement vider les poubelles du parking (enlever les cônes "réservés" inutiles).

### Les Deux Types de VACUUM

#### 1. VACUUM (Standard)

```sql
VACUUM table_name;
```

**Ce qu'il fait :**
- Marque l'espace des lignes mortes comme réutilisable
- Ne réduit PAS la taille du fichier sur disque
- Rapide et peu intrusif
- Peut s'exécuter en parallèle avec les requêtes normales

**Résultat :**
```
Avant : Table 200 MB (100 MB données + 100 MB bloat)  
Après : Table 200 MB (mais 100 MB réutilisables pour nouvelles données)  
```

L'espace est récupéré mais le fichier ne rétrécit pas.

#### 2. VACUUM FULL

```sql
VACUUM FULL table_name;
```

**Ce qu'il fait :**
- Réécrit complètement la table
- Élimine physiquement le bloat
- Réduit réellement la taille du fichier
- **Prend un verrou exclusif** (bloque toutes les opérations)
- Très lent et ressource-intensif

**Résultat :**
```
Avant : Table 200 MB (100 MB données + 100 MB bloat)  
Après : Table 100 MB (bloat complètement éliminé)  
```

**⚠️ Attention :** VACUUM FULL nécessite de l'espace disque temporaire égal à la taille de la table !

### Quand Utiliser Chaque Type ?

| Situation | Commande | Fréquence |
|-----------|----------|-----------|
| Maintenance régulière | `VACUUM` | Automatique (autovacuum) |
| Bloat modéré (< 30%) | `VACUUM` | Hebdomadaire/Mensuel |
| Bloat sévère (> 40%) | `VACUUM FULL` | Exceptionnel |
| Migration/Maintenance planifiée | `VACUUM FULL` | Fenêtre de maintenance |

## Autovacuum : Le Nettoyage Automatique

### Qu'est-ce que l'Autovacuum ?

**Autovacuum** est un processus d'arrière-plan qui exécute automatiquement VACUUM sur les tables qui en ont besoin.

**Avantage :** Vous n'avez (presque) rien à faire !

### Vérifier que l'Autovacuum est Actif

```sql
SHOW autovacuum;
-- Résultat attendu : on
```

Si c'est "off", activez-le immédiatement :

```sql
-- Dans postgresql.conf
autovacuum = on
```

### Paramètres Clés de l'Autovacuum

#### 1. autovacuum_vacuum_scale_factor

**Définition :** Pourcentage de lignes modifiées avant déclenchement.

```ini
autovacuum_vacuum_scale_factor = 0.2  # 20% par défaut
```

**Signification :** L'autovacuum se déclenche quand 20% de la table a été modifiée.

**Pour une table de 1 million de lignes :**
- Seuil = 200 000 lignes mortes

**⚠️ Problème sur grosses tables :**
- Table de 100 millions de lignes → Seuil = 20 millions !
- L'autovacuum attend trop longtemps

**Solution :** Réduire le facteur pour les grosses tables.

#### 2. autovacuum_vacuum_threshold

**Définition :** Nombre minimum de lignes mortes pour déclencher VACUUM.

```ini
autovacuum_vacuum_threshold = 50  # Par défaut
```

**Formule de déclenchement :**
```
Seuil = autovacuum_vacuum_threshold + (autovacuum_vacuum_scale_factor * nombre_lignes)
```

### Nouveautés PostgreSQL 18

#### autovacuum_worker_slots

PostgreSQL 18 introduit des **ajustements dynamiques** pour optimiser l'autovacuum.

#### autovacuum_vacuum_max_threshold

Nouveau paramètre permettant de plafonner le seuil de déclenchement pour les très grosses tables.

```ini
autovacuum_vacuum_max_threshold = 1000000  # Exemple
```

**Utilité :** Empêche l'autovacuum d'attendre trop longtemps sur des tables massives.

### Surveiller l'Autovacuum

#### Voir le Statut de l'Autovacuum

```sql
SELECT
    schemaname,
    relname,
    last_vacuum,
    last_autovacuum,
    n_tup_ins AS inserts,
    n_tup_upd AS updates,
    n_tup_del AS deletes,
    n_dead_tup AS dead_tuples
FROM pg_stat_user_tables  
ORDER BY n_dead_tup DESC;  
```

#### Colonnes Importantes

- **last_autovacuum** : Dernière exécution automatique  
- **dead_tuples** : Nombre de lignes mortes en attente

**⚠️ Alerte :** Si `last_autovacuum` est NULL ou ancien ET `dead_tuples` est élevé, l'autovacuum ne fonctionne pas correctement.

### Statistiques Nouvelles dans PostgreSQL 18

```sql
-- Nouvelles statistiques VACUUM et ANALYZE
SELECT
    schemaname,
    relname,
    vacuum_count,
    autovacuum_count,
    analyze_count,
    autoanalyze_count
FROM pg_stat_all_tables  
WHERE schemaname NOT IN ('pg_catalog', 'information_schema')  
ORDER BY autovacuum_count DESC;  
```

## Optimiser la Configuration de l'Autovacuum

### Configuration Globale vs Par Table

#### Configuration Globale (postgresql.conf)

```ini
# Nombre de workers autovacuum
autovacuum_max_workers = 3

# Délai entre les runs
autovacuum_naptime = 1min

# Coût et délais (throttling)
autovacuum_vacuum_cost_delay = 2ms  
autovacuum_vacuum_cost_limit = 200  
```

#### Configuration Par Table (Recommandé pour Tables Critiques)

```sql
-- Table avec beaucoup d'activité : autovacuum plus agressif
ALTER TABLE orders SET (
    autovacuum_vacuum_scale_factor = 0.05,  -- 5% au lieu de 20%
    autovacuum_vacuum_threshold = 1000
);

-- Table d'archivage : autovacuum moins fréquent
ALTER TABLE old_logs SET (
    autovacuum_vacuum_scale_factor = 0.5,   -- 50%
    autovacuum_vacuum_threshold = 5000
);
```

### Cas Pratiques de Tuning

#### Table avec Beaucoup d'UPDATEs

**Problème :** Table "users" avec 10 millions de lignes, mise à jour fréquente.

**Solution :**
```sql
ALTER TABLE users SET (
    autovacuum_vacuum_scale_factor = 0.02,     -- Déclenchement à 2%
    autovacuum_vacuum_cost_delay = 0,          -- Pas de throttling
    autovacuum_analyze_scale_factor = 0.05     -- Analyse plus fréquente
);
```

#### Table de Queue/Session

**Problème :** Table avec constant INSERT/DELETE.

**Solution :**
```sql
ALTER TABLE job_queue SET (
    autovacuum_vacuum_scale_factor = 0.01,     -- Très agressif
    autovacuum_vacuum_threshold = 100,         -- Seuil bas
    autovacuum_vacuum_cost_delay = 0           -- Vitesse maximale
);
```

## Alternatives à VACUUM FULL

### 1. pg_repack (Extension)

**pg_repack** permet de réorganiser les tables sans verrou exclusif.

**Avantages :**
- Pas de downtime
- Plus sûr que VACUUM FULL
- Peut se faire en production

**Installation :**
```sql
CREATE EXTENSION pg_repack;
```

**Usage :**
```bash
# En ligne de commande
pg_repack -d mydb -t mytable
```

**Fonctionnement :**
1. Crée une copie de la table sans bloat  
2. Applique les modifications en continu  
3. Bascule vers la nouvelle table  
4. Supprime l'ancienne

### 2. CLUSTER

**CLUSTER** réorganise physiquement une table selon un index.

```sql
-- Première fois : définir l'index de clustering
CLUSTER mytable USING mytable_idx;

-- Ensuite : re-cluster si nécessaire
CLUSTER mytable;
```

**⚠️ Attention :** Prend aussi un verrou exclusif comme VACUUM FULL.

**Bonus :** Améliore aussi la performance des requêtes ordonnées.

## Prévenir le Bloat

### 1. Dimensionner Correctement l'Autovacuum

**Règle d'or :** L'autovacuum doit s'exécuter AVANT que le bloat devienne problématique.

**Configuration recommandée pour production :**
```ini
autovacuum_max_workers = 5  
autovacuum_naptime = 30s  
autovacuum_vacuum_cost_delay = 2ms  
autovacuum_vacuum_cost_limit = 400  
```

### 2. Surveillance Proactive

**Script de Monitoring :**
```sql
-- Créer une vue pour surveillance quotidienne
CREATE VIEW bloat_monitor AS  
SELECT  
    schemaname,
    tablename,
    n_dead_tup AS dead_tuples,
    n_live_tup AS live_tuples,
    round(100 * n_dead_tup::numeric / NULLIF(n_live_tup + n_dead_tup, 0), 2) AS bloat_percent,
    last_autovacuum,
    CASE
        WHEN n_dead_tup::numeric / NULLIF(n_live_tup + n_dead_tup, 0) > 0.3 THEN '🔴 CRITICAL'
        WHEN n_dead_tup::numeric / NULLIF(n_live_tup + n_dead_tup, 0) > 0.15 THEN '🟠 WARNING'
        ELSE '🟢 OK'
    END AS status
FROM pg_stat_user_tables  
WHERE n_live_tup > 1000  -- Ignorer petites tables  
ORDER BY bloat_percent DESC NULLS LAST;  

-- Consultation
SELECT * FROM bloat_monitor WHERE status != '🟢 OK';
```

### 3. Optimiser les Schémas

#### Éviter les UPDATEs Fréquents

**Anti-pattern :**
```sql
-- Mauvais : UPDATE fréquent d'un compteur
UPDATE users SET login_count = login_count + 1 WHERE id = 123;
```

**Better pattern :**
```sql
-- Bon : Table séparée pour données volatiles
INSERT INTO user_logins (user_id, login_time) VALUES (123, NOW());
-- Le compteur est calculé par requête
```

#### Utiliser des Partitions

Les tables partitionnées permettent de :
- Supprimer des partitions entières (rapide, pas de bloat)
- Vacuum seulement les partitions actives

**Exemple :**
```sql
-- Table partitionnée par mois
CREATE TABLE logs (
    id BIGSERIAL,
    created_at TIMESTAMP,
    message TEXT
) PARTITION BY RANGE (created_at);

-- Partitions
CREATE TABLE logs_2024_11 PARTITION OF logs
    FOR VALUES FROM ('2024-11-01') TO ('2024-12-01');

CREATE TABLE logs_2024_12 PARTITION OF logs
    FOR VALUES FROM ('2024-12-01') TO ('2025-01-01');

-- Suppression d'anciennes données : instantané, pas de bloat
DROP TABLE logs_2024_10;
```

### 4. Planifier des VACUUM Manuels

Pour les tables critiques, compléter l'autovacuum par des VACUUM manuels pendant les heures creuses :

```bash
# Cron job quotidien
0 2 * * * psql -d mydb -c "VACUUM ANALYZE critical_table;"
```

## Stratégies de Remédiation du Bloat

### Scénario 1 : Bloat Modéré (10-20%)

**Action :** VACUUM standard

```sql
VACUUM ANALYZE table_name;
```

**Timing :** Peut être fait à tout moment.

### Scénario 2 : Bloat Important (20-40%)

**Action :** VACUUM répété + ajustement autovacuum

```sql
-- VACUUM immédiat
VACUUM ANALYZE table_name;

-- Ajuster les paramètres
ALTER TABLE table_name SET (
    autovacuum_vacuum_scale_factor = 0.05
);
```

**Timing :** Pendant heures creuses de préférence.

### Scénario 3 : Bloat Critique (> 40%)

**Option A : VACUUM FULL (Downtime acceptable)**

```sql
-- ⚠️ Bloque la table !
VACUUM FULL ANALYZE table_name;
```

**Pré-requis :**
- Fenêtre de maintenance planifiée
- Espace disque disponible = taille table
- Backup récent

**Option B : pg_repack (Pas de downtime)**

```bash
# Pas de blocage, mais consomme des ressources
pg_repack -d mydb -t table_name
```

**Option C : Recréation de Table**

Pour les très grosses tables :

```sql
BEGIN;

-- Créer nouvelle table
CREATE TABLE table_name_new (LIKE table_name INCLUDING ALL);

-- Copier données
INSERT INTO table_name_new SELECT * FROM table_name;

-- Recréer index
-- (déjà fait par INCLUDING ALL)

-- Renommer
DROP TABLE table_name;  
ALTER TABLE table_name_new RENAME TO table_name;  

COMMIT;
```

**⚠️ Attention :** Nécessite double espace disque et gestion des FK.

## Cas Particuliers et Pièges

### 1. TOAST et Bloat

Les données **TOAST** (The Oversized-Attribute Storage Technique) sont stockées séparément pour les colonnes volumineuses (TEXT, JSONB, etc.).

**Problème :** Le TOAST peut avoir son propre bloat !

**Vérification :**
```sql
SELECT
    relname,
    pg_size_pretty(pg_relation_size(oid)) as size
FROM pg_class  
WHERE relname LIKE '%toast%'  
ORDER BY pg_relation_size(oid) DESC;  
```

**Solution :**
```sql
-- Le VACUUM normal inclut le TOAST
VACUUM table_name;
```

### 2. Index Non Utilisés

Des index jamais utilisés mais gonflés occupent inutilement de l'espace.

**Détection :**
```sql
SELECT
    schemaname,
    tablename,
    indexname,
    pg_size_pretty(pg_relation_size(indexrelid)) AS size,
    idx_scan
FROM pg_stat_user_indexes  
WHERE idx_scan = 0  
ORDER BY pg_relation_size(indexrelid) DESC;  
```

**Action :** Supprimer les index inutiles
```sql
DROP INDEX IF EXISTS unused_index_name;
```

### 3. Transactions Longues

Les transactions longues empêchent VACUUM de nettoyer les lignes mortes.

**Pourquoi ?** VACUUM ne peut pas supprimer une ligne si une transaction active pourrait encore la voir (isolation MVCC).

**Détection :**
```sql
SELECT
    pid,
    now() - xact_start AS duration,
    state,
    query
FROM pg_stat_activity  
WHERE xact_start IS NOT NULL  
ORDER BY xact_start;  
```

**Solution :** Identifier et terminer (ou optimiser) les transactions trop longues.

### 4. Prepared Transactions

Les transactions préparées (2PC - Two-Phase Commit) non terminées bloquent VACUUM.

**Détection :**
```sql
SELECT * FROM pg_prepared_xacts;
```

**Solution :** Commit ou rollback les transactions préparées oubliées.

## Monitoring en Production

### Alertes Recommandées

#### Alerte 1 : Bloat Table > 20%

```sql
-- Requête pour alerting (Prometheus, Grafana, etc.)
SELECT
    COUNT(*) as tables_with_bloat
FROM pg_stat_user_tables  
WHERE n_dead_tup::numeric / NULLIF(n_live_tup + n_dead_tup, 0) > 0.20  
AND n_live_tup > 10000;  
```

**Seuil :** Si > 0, envoyer une alerte.

#### Alerte 2 : Autovacuum Non Exécuté

```sql
-- Tables non vacuum depuis > 24h
SELECT
    schemaname,
    relname,
    last_autovacuum
FROM pg_stat_user_tables  
WHERE last_autovacuum < NOW() - INTERVAL '24 hours'  
AND n_dead_tup > 10000;  
```

#### Alerte 3 : Espace Disque Critique

```sql
-- Taille des tables critiques
SELECT
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS size
FROM pg_tables  
WHERE schemaname NOT IN ('pg_catalog', 'information_schema')  
ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC  
LIMIT 20;  
```

## Questions Fréquentes (FAQ)

### Q1 : VACUUM FULL est-il dangereux ?

**Réponse :** Pas dangereux mais perturbant :
- Bloque complètement la table
- Nécessite de l'espace disque temporaire
- Peut prendre des heures sur grosses tables

**Recommandation :** Préférer pg_repack en production.

### Q2 : Pourquoi mon autovacuum ne se déclenche jamais ?

**Causes possibles :**
1. `autovacuum = off` (vérifier la config)  
2. Transactions longues qui bloquent le nettoyage  
3. Seuils trop élevés (scale_factor trop grand)  
4. Tous les workers autovacuum occupés

**Diagnostic :**
```sql
-- Vérifier config
SHOW autovacuum;

-- Vérifier workers
SELECT * FROM pg_stat_activity WHERE backend_type = 'autovacuum worker';
```

### Q3 : Peut-on avoir un bloat négatif ?

**Réponse :** Non, mais l'estimation peut être imprécise pour :
- Petites tables
- Tables récemment créées
- Tables avec beaucoup de données TOAST

### Q4 : VACUUM réduit-il réellement la taille du fichier ?

**Réponse :**
- **VACUUM** : Non, il marque seulement l'espace comme réutilisable  
- **VACUUM FULL** : Oui, il réduit physiquement le fichier

### Q5 : Quelle fréquence pour VACUUM manuel ?

**Réponse :** En théorie, l'autovacuum suffit. Mais pour tables critiques :
- OLTP intensif : VACUUM quotidien
- Tables modérées : VACUUM hebdomadaire
- Tables stables : Autovacuum uniquement

### Q6 : Le bloat peut-il corrompre les données ?

**Réponse :** Non, le bloat n'affecte pas l'intégrité des données. C'est uniquement un problème de performance et d'espace disque.

## Points Clés à Retenir

✅ **Le bloat est un phénomène naturel**
   - Causé par le fonctionnement MVCC de PostgreSQL
   - Se produit lors d'UPDATE et DELETE

✅ **Types de bloat :**
   - Table bloat : espace mort dans les tables
   - Index bloat : espace mort dans les index

✅ **Mesure du bloat :**
   - Surveiller `n_dead_tup` dans pg_stat_user_tables
   - Seuils : < 10% OK, 10-20% à surveiller, > 20% action requise

✅ **VACUUM : le nettoyeur de PostgreSQL**
   - VACUUM standard : récupère l'espace sans réduire le fichier
   - VACUUM FULL : réduit le fichier mais bloque la table

✅ **Autovacuum : votre meilleur allié**
   - Activé par défaut, s'exécute automatiquement
   - Ajuster les paramètres pour tables à forte activité
   - PostgreSQL 18 : ajustements dynamiques améliorés

✅ **Prévention du bloat :**
   - Optimiser les schémas (éviter UPDATEs inutiles)
   - Utiliser le partitionnement
   - Surveiller régulièrement
   - Configurer l'autovacuum de manière agressive

✅ **Remédiation :**
   - Bloat modéré : VACUUM standard
   - Bloat important : VACUUM répété + tuning autovacuum
   - Bloat critique : VACUUM FULL (fenêtre maintenance) ou pg_repack

✅ **Monitoring essentiel :**
   - Bloat percentage par table
   - Fréquence d'exécution de l'autovacuum
   - Dead tuples count
   - Espace disque disponible

## Conclusion

Le **bloat** est un aspect inévitable du fonctionnement de PostgreSQL, mais il est parfaitement gérable avec les bons outils et les bonnes pratiques.

La clé du succès réside dans :
1. **La compréhension** : Savoir pourquoi le bloat se produit  
2. **La surveillance** : Mesurer régulièrement le bloat  
3. **La prévention** : Configurer correctement l'autovacuum  
4. **L'action** : Intervenir rapidement quand nécessaire

Un autovacuum bien configuré et surveillé permet de maintenir votre base de données PostgreSQL en bonne santé sans intervention manuelle dans 95% des cas.

Les 5% restants concernent des situations exceptionnelles (migrations, charges inhabituelles) où des actions manuelles ciblées permettent de résoudre rapidement les problèmes.
