# 14.6.3. I/O Wait et Disk Latency

## Introduction

**I/O Wait** (attente d'entrée/sortie) et **Disk Latency** (latence disque) sont deux métriques fondamentales pour comprendre les performances de votre base de données PostgreSQL. Ces indicateurs révèlent si votre système passe trop de temps à attendre les données stockées sur le disque.

Dans ce chapitre, nous allons démystifier ces concepts souvent mal compris et vous donner les clés pour identifier, mesurer et résoudre les problèmes de performance liés au disque.

---

## Comprendre les Concepts de Base

### Analogie : La Bibliothèque

Imaginez que vous travaillez dans une bibliothèque :

**Scénario 1 : Bibliothèque bien organisée**
- Vous demandez un livre
- Le bibliothécaire le trouve en 5 secondes
- Vous pouvez continuer votre travail rapidement

**Scénario 2 : Bibliothèque désorganisée**
- Vous demandez un livre
- Le bibliothécaire cherche pendant 10 minutes
- **Vous attendez** sans rien pouvoir faire (I/O Wait)
- Le temps de recherche est long (Latency)

C'est exactement ce qui se passe avec PostgreSQL et le disque !

### Les Acteurs en Présence

#### 1. Le CPU (Processeur)

Le cerveau de votre serveur qui exécute les calculs et traite les requêtes SQL.

**Caractéristique :** Ultra-rapide (milliards d'opérations par seconde)

#### 2. La RAM (Mémoire)

Le cache où sont stockées les données fréquemment utilisées.

**Caractéristique :** Très rapide (~100 nanosecondes par accès)

#### 3. Le Disque (SSD ou HDD)

Le stockage permanent où résident toutes vos données.

**Caractéristique :** Beaucoup plus lent que la RAM
- **SSD** : ~100 microsecondes (1000× plus lent que la RAM)  
- **HDD** : ~10 millisecondes (100 000× plus lent que la RAM)

### Le Problème de Vitesse

PostgreSQL a besoin de données stockées sur le disque, mais :

```
Vitesse CPU    : 1 000 000 000 opérations/seconde  
Vitesse RAM    :    10 000 000 accès/seconde  
Vitesse SSD    :        10 000 accès/seconde  
Vitesse HDD    :           100 accès/seconde  
```

Le disque est le **goulot d'étranglement** majeur des bases de données.

---

## Qu'est-ce que l'I/O Wait ?

### Définition Simple

**I/O Wait** (Input/Output Wait) est le **pourcentage de temps pendant lequel le CPU attend que le disque réponde** au lieu de traiter des données.

**En termes simples :** C'est du temps perdu à attendre.

### Visualisation

```
Sans I/O Wait (Idéal) :  
CPU : [████████████████████████████████] 100% productif  
Disque : Rapide, pas d'attente  

Avec I/O Wait (Problématique) :  
CPU : [████░░░░████░░░░████░░░░████░░░░] 50% productif, 50% en attente  
       ↑     ↑     ↑     ↑
       Travail Attend Travail Attend
```

Les zones `░░░░` représentent le temps où le CPU ne fait **rien** et attend simplement que le disque fournisse les données.

### Pourquoi c'est un Problème

#### Gaspillage de Ressources

Vous payez pour un CPU puissant qui passe la moitié de son temps à attendre !

**Exemple concret :**
- Serveur avec CPU à 3 GHz (puissant)
- I/O Wait à 60%
- **Résultat :** Vous n'utilisez que 40% de votre CPU

#### Performances Dégradées

Les requêtes prennent beaucoup plus de temps à s'exécuter.

**Exemple :**
- Requête qui devrait prendre 100 ms de calcul CPU
- Avec 80% d'I/O Wait : **500 ms réels** (5× plus lent)

---

## Qu'est-ce que la Disk Latency ?

### Définition Simple

La **Disk Latency** (latence disque) est le **temps nécessaire pour que le disque réponde à une demande de lecture ou d'écriture**.

**En termes simples :** C'est le délai de réponse du disque.

### Composantes de la Latency

#### 1. Seek Time (Temps de Recherche) - HDD uniquement

Le temps pour que la tête de lecture se positionne au bon endroit sur le plateau.

**HDD typique :** 5-10 millisecondes

**SSD :** N/A (pas de pièces mobiles)

#### 2. Rotational Latency (Latence de Rotation) - HDD uniquement

Le temps d'attente pour que le secteur passe sous la tête de lecture.

**HDD 7200 RPM :** ~4 millisecondes en moyenne

**SSD :** N/A

#### 3. Transfer Time (Temps de Transfert)

Le temps pour transférer les données du disque vers la RAM.

**Dépend de :**
- Taille des données
- Interface (SATA, NVMe, etc.)
- Bande passante disponible

### Ordres de Grandeur

| Type de Disque | Latency Typique | Utilisation |
|----------------|-----------------|-------------|
| **HDD 7200 RPM** | 10-15 ms | Archivage, backup |
| **HDD 15000 RPM** | 3-5 ms | Serveurs (obsolète) |
| **SSD SATA** | 0.1-1 ms | Standard moderne |
| **SSD NVMe** | 0.02-0.1 ms | Haute performance |
| **RAM** | 0.0001 ms (100 ns) | Cache |

**Observation :** Même un SSD est 1000× plus lent que la RAM !

---

## Impact sur PostgreSQL

### Opérations Sensibles à l'I/O

#### 1. Scans Séquentiels (Sequential Scans)

Lorsque PostgreSQL doit lire une table entière sans index.

**Exemple :**
```sql
SELECT * FROM big_table WHERE status = 'active';
-- Sans index sur 'status' : scan complet
```

**Impact I/O :** Très élevé si la table est volumineuse et ne tient pas en cache.

#### 2. Index Scans sur Données Non Cachées

Même avec des index, si les données ne sont pas en cache, il faut les lire depuis le disque.

**Exemple :**
```sql
SELECT * FROM orders WHERE customer_id = 12345;
-- Avec index : rapide si en cache, lent si sur disque
```

#### 3. Écritures (Writes)

Les `INSERT`, `UPDATE`, `DELETE` doivent écrire sur le disque.

**Particulièrement coûteux :**
- Écriture dans le WAL (Write-Ahead Log)
- Mise à jour des index
- Checkpoints (synchronisation disque)

#### 4. Checkpoints

Les checkpoints synchronisent les données de la RAM vers le disque.

**Impact :** Pic d'écriture disque, potentiellement forte latency.

#### 5. VACUUM

Le nettoyage des lignes mortes nécessite de lire et réécrire des pages.

**Impact :** I/O élevé, surtout sur tables volumineuses.

### Symptômes de Problèmes I/O

| Symptôme | Signification |
|----------|---------------|
| Requêtes lentes malgré bons index | Données pas en cache, latency élevée |
| Pics de lenteur réguliers | Probablement pendant les checkpoints |
| Dégradation progressive | Bloat ou cache insuffisant |
| Serveur "idle" mais lent | CPU en I/O Wait |
| Sauvegardes très longues | Disque saturé ou lent |

## Mesurer l'I/O Wait

### Au Niveau Système (Linux)

#### 1. La Commande `top`

```bash
top
```

**Regardez la ligne `%Cpu(s)` :**
```
%Cpu(s):  15.2 us,  2.1 sy,  0.0 ni, 45.3 id, 37.4 wa,  0.0 hi,  0.0 si,  0.0 st
                                              ↑↑↑↑
                                              I/O Wait = 37.4%
```

**Interprétation :**
- **us** (user) : 15.2% en calculs utilisateur  
- **sy** (system) : 2.1% en appels système  
- **id** (idle) : 45.3% inactif  
- **wa** (wait) : **37.4% en attente I/O** ← Problème !

#### Seuils d'Interprétation

| I/O Wait | État | Action |
|----------|------|--------|
| **< 5%** | 🟢 Normal | RAS |
| **5-15%** | 🟡 Acceptable | Surveiller |
| **15-30%** | 🟠 Préoccupant | Investiguer |
| **> 30%** | 🔴 Critique | Action urgente |

#### 2. La Commande `iostat`

```bash
iostat -x 2
```

**Sortie exemple :**
```
Device    r/s   w/s   rkB/s   wkB/s  await  svctm  %util  
sda      150.0  80.0  6000.0  3200.0  25.50  12.30  95.00  
```

**Colonnes importantes :**
- **r/s** : Lectures par seconde  
- **w/s** : Écritures par seconde  
- **await** : Temps d'attente moyen (ms) ← **Latency**  
- **%util** : Utilisation du disque (%)

**Interprétation :**
```
await = 25.50 ms  → Latency très élevée (problème !)
%util = 95%       → Disque saturé
```

#### Seuils pour `await` (Latency)

| Type Disque | Normal | Attention | Problème |
|-------------|--------|-----------|----------|
| **SSD NVMe** | < 1 ms | 1-5 ms | > 5 ms |
| **SSD SATA** | < 5 ms | 5-10 ms | > 10 ms |
| **HDD** | < 10 ms | 10-20 ms | > 20 ms |

#### 3. La Commande `iotop`

Comme `top`, mais pour l'I/O :

```bash
sudo iotop -o
```

**Affiche :**
- Quels processus génèrent de l'I/O
- Vitesse de lecture/écriture par processus
- Pourcentage d'I/O par processus

**Utile pour :** Identifier quel processus PostgreSQL consomme le plus d'I/O.

### Au Niveau PostgreSQL

#### 1. Extension pg_stat_statements

Cette extension fournit des statistiques sur les requêtes, y compris l'I/O.

**Installation :**
```sql
CREATE EXTENSION pg_stat_statements;
```

**Prérequis :** `track_io_timing = on` dans `postgresql.conf` pour que les temps I/O soient mesurés (sinon les colonnes restent à 0).

**Requête pour voir les requêtes consommant le plus d'I/O :**

> ⚠️ **PostgreSQL 17+** : les colonnes `blk_read_time` et `blk_write_time` ont été renommées et scindées par type de bloc : `shared_blk_read_time`, `shared_blk_write_time`, `local_blk_read_time`, `local_blk_write_time`, `temp_blk_read_time`, `temp_blk_write_time`.

```sql
SELECT
    query,
    calls,
    total_exec_time / 1000 AS total_time_seconds,
    shared_blk_read_time / 1000 AS shared_read_time_s,
    shared_blk_write_time / 1000 AS shared_write_time_s,
    local_blk_read_time / 1000 AS local_read_time_s,
    temp_blk_read_time / 1000 AS temp_read_time_s,
    round(
        ((shared_blk_read_time + shared_blk_write_time
         + local_blk_read_time + local_blk_write_time
         + temp_blk_read_time + temp_blk_write_time)
        / NULLIF(total_exec_time, 0) * 100)::numeric,   -- cast obligatoire : ces colonnes sont double precision, et round(double precision, N) n'existe pas
        2
    ) AS io_percent
FROM pg_stat_statements  
WHERE total_exec_time > 0  
ORDER BY (shared_blk_read_time + shared_blk_write_time  
          + local_blk_read_time + local_blk_write_time
          + temp_blk_read_time + temp_blk_write_time) DESC
LIMIT 20;
```

**Colonnes :**
- **shared_read_time_s** / **shared_write_time_s** : temps I/O sur les blocs partagés (cache `shared_buffers`)
- **local_read_time_s** : temps I/O sur les blocs locaux (tables temporaires de session)
- **temp_read_time_s** : temps I/O sur les blocs temporaires (tris, hash joins externes)
- **io_percent** : pourcentage du temps total passé en I/O

**Interprétation :**
```
io_percent > 50% → La requête passe plus de temps en I/O qu'en calcul
→ Optimisation nécessaire (index, cache, requête)
```

#### 2. Nouveautés PostgreSQL 18 : Statistiques I/O par Backend

PostgreSQL 18 introduit la fonction **`pg_stat_get_backend_io(pid)`** qui renvoie les statistiques I/O détaillées (au même format que `pg_stat_io`) pour un **backend individuel**, identifié par son PID. Avant PG 18, `pg_stat_io` n'agrégeait les compteurs que par `backend_type`, sans possibilité de cibler une session particulière.

> ⚠️ **Attention :** `pg_stat_io` ne contient PAS de colonnes `pid`, `usename`, `application_name` ou `state`. Pour obtenir l'I/O par session il faut joindre `pg_stat_activity` à la fonction `pg_stat_get_backend_io(pid)` via `LATERAL`.

**Requête (PG 18) : I/O des sessions actives, classées par temps I/O décroissant**

```sql
SELECT
    a.pid,
    a.usename,
    a.application_name,
    a.state,
    io.object,
    io.context,
    io.reads,
    io.read_time,
    io.writes,
    io.write_time
FROM pg_stat_activity a  
CROSS JOIN LATERAL pg_stat_get_backend_io(a.pid) AS io  
WHERE a.backend_type = 'client backend'  
  AND (io.reads > 0 OR io.writes > 0)
ORDER BY (COALESCE(io.read_time, 0) + COALESCE(io.write_time, 0)) DESC  
LIMIT 50;  
```

**Utilité :** identifier en temps réel quelle session précise génère le plus d'I/O, ce qui n'était pas possible avec la seule vue `pg_stat_io` (qui n'expose que des totaux par `backend_type`).

> 💡 Une fonction complémentaire `pg_stat_reset_backend_stats(pid)` (PG 18) permet de remettre à zéro les compteurs d'un backend pour mesurer un intervalle précis.

#### 3. Vues Système pour l'I/O

**pg_statio_user_tables** : Statistiques I/O par table

```sql
SELECT
    schemaname,
    relname,
    heap_blks_read AS disk_reads,
    heap_blks_hit AS cache_hits,
    idx_blks_read AS index_disk_reads,
    idx_blks_hit AS index_cache_hits,
    round(
        (heap_blks_hit::numeric / NULLIF(heap_blks_hit + heap_blks_read, 0)) * 100,
        2
    ) AS cache_hit_ratio
FROM pg_statio_user_tables  
WHERE heap_blks_read > 0  
ORDER BY heap_blks_read DESC;  
```

**Interprétation :**
- **heap_blks_read** élevé → Beaucoup de lectures disque sur cette table  
- **cache_hit_ratio** faible → Données rarement en cache

## Mesurer la Disk Latency

### Au Niveau Système

#### 1. Avec `iostat` (Déjà vu)

```bash
iostat -x 1
```

La colonne **await** donne la latency moyenne.

#### 2. Benchmark avec `fio` (Flexible I/O Tester)

**fio** permet de tester les performances réelles de votre disque.

**Installation :**
```bash
sudo apt install fio  # Ubuntu/Debian
```

**Test de Lecture Aléatoire (Simule lecture BDD) :**
```bash
fio --name=random-read \
    --ioengine=libaio \
    --iodepth=16 \
    --rw=randread \
    --bs=8k \
    --direct=1 \
    --size=1G \
    --numjobs=4 \
    --runtime=60 \
    --group_reporting
```

**Résultat exemple :**
```
read: IOPS=25000, BW=195MiB/s (205MB/s)
  lat (usec): min=50, max=5000, avg=250.5
```

**Interprétation :**
- **IOPS** : 25 000 opérations de lecture par seconde  
- **avg latency** : 250 µs (0.25 ms) ← Excellent pour un SSD

#### Seuils de Performance

| Métrique | SSD NVMe | SSD SATA | HDD |
|----------|----------|----------|-----|
| **IOPS Random Read** | 100K-500K | 10K-100K | 100-200 |
| **Latency** | < 0.1 ms | 0.1-1 ms | 5-15 ms |
| **Throughput** | 2-7 GB/s | 500 MB/s | 150 MB/s |

#### 3. Avec `dd` (Rapide mais moins précis)

```bash
# Test écriture séquentielle
dd if=/dev/zero of=/tmp/testfile bs=1M count=1000 oflag=direct
```

**Résultat :**
```
1048576000 bytes (1.0 GB) copied, 2.5 s, 419 MB/s
```

**Utile pour :** Vérifier rapidement si le disque fonctionne normalement.

### Au Niveau PostgreSQL

#### Vue pg_stat_io (PostgreSQL 16+)

> ℹ️ `pg_stat_io` n'est pas une extension à installer mais une **vue système** intégrée, disponible depuis PostgreSQL 16. PG 18 l'enrichit avec des colonnes en octets (`read_bytes`, `write_bytes`, `extend_bytes`) et de nouvelles lignes pour l'I/O **WAL**.

```sql
SELECT * FROM pg_stat_io;
```

**Colonnes pertinentes :**
- **reads** : Nombre de lectures  
- **writes** : Nombre d'écritures  
- **read_time** : Temps total de lecture (ms)  
- **write_time** : Temps total d'écriture (ms)

**Calculer la latency moyenne :**
```sql
SELECT
    backend_type,
    reads,
    read_time,
    CASE WHEN reads > 0
        THEN round(read_time::numeric / reads, 2)
        ELSE 0
    END AS avg_read_latency_ms
FROM pg_stat_io  
WHERE reads > 0  
ORDER BY avg_read_latency_ms DESC;  
```

## Causes Fréquentes de Problèmes I/O

### 1. Cache Insuffisant (shared_buffers trop petit)

**Problème :** Les données les plus utilisées ne tiennent pas en mémoire.

**Symptôme :**
- Cache Hit Ratio < 95%
- I/O Wait élevé

**Solution :**
```ini
# Dans postgresql.conf
shared_buffers = 8GB  # Augmenter (25% de RAM recommandé)
```

### 2. Requêtes Non Optimisées

**Problème :** Scans séquentiels sur de grosses tables sans index.

**Symptôme :**
```sql
EXPLAIN ANALYZE SELECT * FROM big_table WHERE status = 'active';
-- Résultat : Seq Scan on big_table (rows=10000000)
```

**Solution :**
```sql
CREATE INDEX idx_status ON big_table(status);
```

### 3. Disque Lent ou Saturé

**Problème :** Disque HDD ou SSD vieillissant.

**Symptôme :**
- `iostat` montre `%util` proche de 100%  
- `await` très élevé

**Solution :**
- Migrer vers SSD NVMe
- Utiliser RAID pour paralléliser les I/O
- Distribuer les données sur plusieurs disques

### 4. Checkpoints Trop Fréquents

**Problème :** Les checkpoints écrivent massivement sur le disque.

**Symptôme :**
- Pics de lenteur réguliers (ex: toutes les 5 minutes)
- Logs montrant des checkpoints fréquents

**Vérification (PostgreSQL 17+) :**

> ⚠️ Depuis PG 17, les compteurs de checkpoints ont quitté `pg_stat_bgwriter` pour la nouvelle vue **`pg_stat_checkpointer`** (colonnes renommées : `num_timed`, `num_requested`, `write_time`, `sync_time`).

```sql
SELECT
    num_timed,             -- checkpoints planifiés (anciennement checkpoints_timed)
    num_requested,         -- checkpoints forcés (anciennement checkpoints_req)
    write_time,            -- temps cumulé d'écriture (ms)
    sync_time,             -- temps cumulé de fsync (ms)
    buffers_written        -- buffers écrits pendant les checkpoints
FROM pg_stat_checkpointer;
```

**Solution :**
```ini
# Dans postgresql.conf
max_wal_size = 4GB               # Augmenter  
checkpoint_timeout = 15min       # Augmenter  
checkpoint_completion_target = 0.9  # Lisser les écritures  
```

### 5. WAL sur le Même Disque que les Données

**Problème :** Les écritures WAL et les données se font concurrence.

**Symptôme :**
- Forte contention I/O lors d'écritures massives

**Solution :**
- Placer le WAL sur un disque dédié (ou partition)
```bash
# Déplacer pg_wal sur un autre disque
mv /var/lib/postgresql/data/pg_wal /mnt/fast-disk/pg_wal  
ln -s /mnt/fast-disk/pg_wal /var/lib/postgresql/data/pg_wal  
```

### 6. Autovacuum Trop Agressif ou Insuffisant

**Problème 1 (Trop agressif) :**
- Autovacuum consomme trop d'I/O en parallèle des requêtes

**Solution :**
```ini
autovacuum_vacuum_cost_delay = 10ms  # Ralentir l'autovacuum
```

**Problème 2 (Insuffisant) :**
- Bloat élevé → Plus de données à lire → Plus d'I/O

**Solution :**
```sql
-- Voir section 14.6.2 sur le Bloat
```

### 7. RAID Mal Configuré

**Problème :** RAID software mal paramétré ou RAID hardware avec cache désactivé.

**Solution :**
- Activer le write cache sur RAID hardware (avec BBU)
- Utiliser RAID 10 plutôt que RAID 5 pour PostgreSQL
- Vérifier les paramètres kernel (scheduler I/O)

## Optimisations I/O pour PostgreSQL

### 1. Augmenter le Cache PostgreSQL

```ini
# postgresql.conf
shared_buffers = 8GB                # 25% de la RAM  
effective_cache_size = 24GB         # 75% de la RAM (estimation pour planificateur)  
```

**Effet :** Plus de données en RAM = Moins d'I/O disque.

### 2. Optimiser les Paramètres WAL

```ini
# Écriture WAL
wal_buffers = 16MB                  # Buffer pour WAL  
wal_writer_delay = 200ms            # Fréquence d'écriture WAL  

# Checkpoints
max_wal_size = 4GB  
checkpoint_timeout = 15min  
checkpoint_completion_target = 0.9  # Lisser les écritures sur 90% de l'intervalle  
```

**Effet :** Réduit les pics d'écriture, améliore la latency.

### 3. Utiliser Partitionnement et Tablespaces

**Partitionnement :** Diviser les grosses tables.

```sql
-- Exemple : Table de logs partitionnée par mois
CREATE TABLE logs (
    id BIGSERIAL,
    created_at TIMESTAMP,
    message TEXT
) PARTITION BY RANGE (created_at);

-- Partitions
CREATE TABLE logs_2024_11 PARTITION OF logs
    FOR VALUES FROM ('2024-11-01') TO ('2024-12-01');
```

**Tablespaces :** Placer différentes tables sur différents disques.

```sql
-- Créer tablespace sur disque rapide
CREATE TABLESPACE fast_storage LOCATION '/mnt/nvme/pg_data';

-- Déplacer table critique
ALTER TABLE critical_table SET TABLESPACE fast_storage;
```

**Effet :** Distribue l'I/O, réduit la contention.

### 4. Nouveauté PostgreSQL 18 : Sous-système I/O Asynchrone (AIO)

PostgreSQL 18 introduit un véritable sous-système d'I/O **asynchrone** au moteur, qui peut améliorer significativement les performances sur les charges de lecture intensives (sequential scans, bitmap heap scans, vacuum…). Le gain typique observé est de l'ordre de 1,5 à 3× selon la charge et le stockage sous-jacent.

**Paramètres clés :**

```ini
# postgresql.conf

# Implémentation à utiliser
io_method = 'worker'        # Défaut PG 18
# io_method = 'io_uring'    # Linux récent ; PG doit être compilé avec --with-liburing
# io_method = 'sync'        # Comportement legacy (équivalent pré-PG 18)

# Nombre de workers I/O (uniquement utilisé si io_method = 'worker')
io_workers = 3              # Défaut PG 18
```

**Restrictions** : `io_method` ne peut être changé qu'au **démarrage** du serveur. `io_uring` nécessite à la fois un noyau Linux récent (≥ 5.1, idéalement ≥ 6.0) **et** un binaire PostgreSQL compilé avec le support `liburing` (option `--with-liburing` au `configure`).

**Principe :** au lieu d'attendre chaque opération I/O individuellement (modèle synchrone), PostgreSQL peut désormais émettre plusieurs requêtes I/O en parallèle, les workers (ou io_uring) les exécutent, et le backend récupère les résultats à mesure qu'ils arrivent.

**Bénéfices :**
- Réduction de l'I/O Wait visible au niveau OS
- Meilleur parallélisme des lectures, surtout sur SSD NVMe à forte profondeur de file
- Vacuum et bulk reads accélérés

**Observer les I/O en cours (PG 18) :** la nouvelle vue système **`pg_aios`** liste les handles d'I/O asynchrones actifs (un par opération) :

```sql
SELECT pid, io_id, state, operation, length, target  
FROM pg_aios  
ORDER BY state;  
```

> ℹ️ `pg_aios` est avant tout un outil de diagnostic bas-niveau pour comprendre l'activité AIO instantanée — utile pour repérer un goulot dans le sous-système I/O (ex : beaucoup d'I/O en état `SUBMITTED` qui n'avancent pas). L'accès nécessite un membre du rôle `pg_read_all_stats`.

**Nouveaux wait events AIO (PG 18)** : si vous voyez ces wait events dans `pg_stat_activity`, c'est que des backends attendent une opération AIO :

| Wait event | Signification |
|------------|---------------|
| `IO:AioIoCompletion` | Le backend attend qu'un autre processus termine une I/O qu'il a partagée |
| `IO:AioIoUringSubmit` | Soumission d'une I/O via `io_uring` (rare, doit être très court) |
| `IO:AioIoUringExecution` | Exécution effective côté kernel via `io_uring` |

```sql
-- Voir les backends actuellement bloqués sur de l'AIO
SELECT pid, wait_event_type, wait_event, state, LEFT(query, 60) AS query  
FROM pg_stat_activity  
WHERE wait_event_type = 'IO'  
  AND wait_event LIKE 'Aio%';
```

> 💡 Sur un système sain avec AIO actif, ces wait events doivent être **fugaces** (jamais bloquants longtemps). Un `AioIoCompletion` qui dure plusieurs secondes indique généralement une saturation disque sous-jacente, à corréler avec `iostat -x` côté système.

**Paramètres associés : `effective_io_concurrency` et `maintenance_io_concurrency`**

Ces deux paramètres indiquent à PostgreSQL **combien d'I/O concurrentes** le disque peut soutenir — ils contrôlent la profondeur du **prefetch** sur les opérations comme les bitmap heap scans (et, en PG 18, sont aussi exploités par le moteur AIO).

```ini
# postgresql.conf

# Pour les requêtes utilisateur (Bitmap Heap Scan, etc.)
effective_io_concurrency = 200     # SSD NVMe ; défaut 16 en PG 18 (valait 1 de PG 8.4 à PG 17)

# Pour les opérations de maintenance (VACUUM, ANALYZE, CREATE INDEX)
maintenance_io_concurrency = 200   # défaut 16 en PG 18 (valait 10 de PG 13 à PG 17)
```

| Stockage | `effective_io_concurrency` recommandé |
|----------|---------------------------------------|
| HDD seul | 1 — pas de gain à paralléliser |
| HDD en RAID 10 | 4-8 (nombre de disques) |
| SSD SATA | 100-200 |
| SSD NVMe | 200-1000 |
| Cloud (EBS gp3, Premium SSD…) | 100-300 selon les IOPS provisionnées |

> 💡 **Nouveauté PG 18** : le défaut d'`effective_io_concurrency` est passé de **1** (PG 8.4 → 17) à **16** en PG 18, et celui de `maintenance_io_concurrency` de **10** (PG 13 → 17) à **16** — en lien avec le nouveau sous-système AIO. Si vous tournez sur SSD/NVMe, monter à 200+ donne un gain mesurable sur les bitmap scans et les vacuum.

### 5. Optimiser les Requêtes

#### Utiliser EXPLAIN ANALYZE

```sql
EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM orders WHERE customer_id = 123;  
```

> 💡 **Nouveauté PG 18** : `BUFFERS` est désormais **inclus automatiquement** quand `ANALYZE` est utilisé. Un simple `EXPLAIN ANALYZE` suffit donc en PG 18 pour obtenir les statistiques de buffers — `EXPLAIN (ANALYZE, BUFFERS)` reste compatible et explicite.

**Autres nouveautés EXPLAIN ANALYZE en PG 18** :

- **Nombre de lookups par Index Scan** : chaque nœud `Index Scan` indique combien de fois l'index a été consulté pendant l'exécution. Permet de repérer les **boucles cachées** (ex : un nested loop qui consulte 1M fois le même index).
- **Compteurs `actual rows` fractionnaires** : `actual time=… rows=4876.50 loops=2` au lieu de `rows=4876` arrondi — plus précis sur les nœuds exécutés plusieurs fois.
- **Nœuds `disabled` clairement marqués** : quand un nœud est forcé par `enable_seqscan = off` (ou équivalent), le plan le mentionne explicitement — utile pour comprendre pourquoi un plan a été choisi.
- **Statistiques du cache des workers Parallel Bitmap Heap Scan** : profondeur de cache, hits, etc. par worker — utile pour diagnostiquer les bitmap scans parallèles déséquilibrés.

**Regardez :**
- **Buffers: shared read=XXX** → nombre de blocs lus depuis le disque
- **Buffers: shared hit=XXX** → blocs trouvés dans `shared_buffers`
- **I/O Timings: read=XX.XX** → temps passé en I/O (si `track_io_timing = on`)

**Activation du tracking I/O :**
```sql
ALTER SYSTEM SET track_io_timing = on;  
SELECT pg_reload_conf();  
```

##### Options EXPLAIN récentes (PG 17+/18+)

PostgreSQL 17 et 18 introduisent deux options EXPLAIN très utiles, peu connues :

```sql
-- (PG 17+) SERIALIZE : mesurer le coût de sérialisation des résultats
-- (conversion en texte/binary avant envoi au client — souvent oublié)
EXPLAIN (ANALYZE, SERIALIZE TEXT)  
SELECT * FROM big_table LIMIT 1000;  

-- Renvoie en plus :
-- Serialization: time=12.345 ms  output=4096 kB  format=text

-- (PG 17+) MEMORY : voir la consommation mémoire de la phase planning
EXPLAIN (MEMORY)  
SELECT … FROM table_avec_beaucoup_de_partitions WHERE … ;  

-- Renvoie en plus :
-- Planning:
--   Memory: used=125 kB allocated=128 kB
```

**Utilité pratique** :
- **`SERIALIZE`** : sur les requêtes qui renvoient beaucoup de lignes/colonnes (ex : exports), `SERIALIZE` révèle un coût souvent invisible — un `SELECT * FROM logs LIMIT 1000000` peut passer plus de temps à sérialiser qu'à lire. Aide à choisir un format binaire plus efficace côté driver.
- **`MEMORY`** (PG 17) : utile sur les requêtes sur tables fortement **partitionnées** où la planification peut consommer beaucoup de mémoire (un partition pruning lent peut être visible ici).

> 💡 Combiner avec `ANALYZE` quand c'est utile : `EXPLAIN (ANALYZE, MEMORY, SERIALIZE TEXT)`.

#### Ajouter des Index

```sql
-- Avant : Seq Scan (lent)
SELECT * FROM users WHERE email = 'test@example.com';

-- Créer index
CREATE INDEX idx_users_email ON users(email);

-- Après : Index Scan (rapide, moins d'I/O)
```

### 6. Connection Pooling

**Problème :** Trop de connexions simultanées = contention I/O.

**Solution :** Utiliser PgBouncer pour limiter les connexions actives.

```ini
# pgbouncer.ini
default_pool_size = 25        # Nombre de connexions PostgreSQL  
max_client_conn = 500         # Nombre de clients acceptés  
pool_mode = transaction       # Mode transaction  
```

**Effet :** Réduit la charge globale, améliore les performances I/O.

**Alerte :**
```

```

#### 3. IOPS et Throughput

**Métriques :**
- Lectures/sec
- Écritures/sec
- MB/sec lus
- MB/sec écrits

**Alerte :** Détecter les pics anormaux.

#### 4. Checkpoint Frequency (PostgreSQL 17+)

**Requête (vue `pg_stat_checkpointer` introduite en PG 17) :**
```sql
SELECT
    num_timed,
    num_requested,
    round(
        num_requested::numeric / NULLIF(num_timed + num_requested, 0) * 100,
        2
    ) AS forced_checkpoint_pct
FROM pg_stat_checkpointer;
```

**Seuil :** `forced_checkpoint_pct` > 10% → augmenter `max_wal_size`

#### 5. Top Requêtes I/O (PostgreSQL)

**Requête automatisée (PG 17+) :**

```sql
-- À exécuter régulièrement (track_io_timing = on requis)
WITH io_total AS (
    SELECT
        queryid,
        query,
        calls,
        shared_blk_read_time + shared_blk_write_time
            + local_blk_read_time + local_blk_write_time
            + temp_blk_read_time  + temp_blk_write_time AS total_io_ms
    FROM pg_stat_statements
)
SELECT
    substring(query, 1, 80) AS query_short,
    calls,
    round((total_io_ms / 1000)::numeric, 2) AS total_io_seconds,
    round((total_io_ms / NULLIF(calls, 0))::numeric, 2) AS avg_io_ms_per_call
FROM io_total  
WHERE total_io_ms > 0  
ORDER BY total_io_ms DESC  
LIMIT 10;  
```

## Troubleshooting : Scénarios Courants

### Scénario 1 : I/O Wait soudainement élevé

**Symptômes :**
- I/O Wait passe de 5% à 40%
- Requêtes lentes
- Utilisateurs se plaignent

**Diagnostic :**

```bash
# 1. Vérifier les processus
sudo iotop -o

# 2. Vérifier PostgreSQL
SELECT * FROM pg_stat_activity WHERE state = 'active';

# 3. Vérifier les requêtes lentes
SELECT * FROM pg_stat_statements ORDER BY total_exec_time DESC LIMIT 10;
```

**Causes fréquentes :**
- Grosse requête analytique lancée
- VACUUM FULL en cours
- Backup/dump en cours
- Application qui fait du batch processing

**Action :**
- Tuer la requête problématique si nécessaire
- Planifier ces opérations en heures creuses

### Scénario 2 : Latency élevée constante

**Symptômes :**
- `await` dans iostat > 50ms en permanence
- Base de données globalement lente

**Diagnostic :**

```bash
# Vérifier état disque
iostat -x 2 10

# Vérifier RAID
cat /proc/mdstat  # Pour software RAID

# Tester performance disque
fio --name=test --rw=randread --bs=8k --size=1G
```

**Causes fréquentes :**
- Disque défaillant (à remplacer)
- RAID dégradé
- Firmware disque obsolète
- Saturation I/O constante

**Action :**
- Vérifier santé disque (SMART)
- Remplacer disque si nécessaire
- Upgrader vers SSD si HDD

### Scénario 3 : Performances OK puis dégradation progressive

**Symptômes :**
- Performances dégradent sur plusieurs jours/semaines
- I/O Wait augmente progressivement

**Diagnostic :**

```sql
-- Vérifier bloat
SELECT * FROM bloat_monitor;  -- Vue créée section 14.6.2

-- Vérifier cache hit ratio
SELECT * FROM cache_hit_ratio_view;  -- Vue créée section 14.6.1

-- Vérifier taille tables
SELECT
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS size
FROM pg_tables  
ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC  
LIMIT 20;  
```

**Causes fréquentes :**
- Bloat qui s'accumule
- Données qui grossissent et ne tiennent plus en cache
- Index fragmentés

**Action :**
- Exécuter VACUUM ANALYZE
- Éventuellement VACUUM FULL en maintenance
- Augmenter shared_buffers si possible
- REINDEX si index très fragmentés

## Questions Fréquentes (FAQ)

### Q1 : Est-ce normal d'avoir un peu d'I/O Wait ?

**Réponse :** Oui, un I/O Wait de 1-5% est normal et acceptable. C'est au-delà de 15% que cela devient préoccupant.

### Q2 : Mon I/O Wait est à 0% mais ma base est lente, pourquoi ?

**Réponse :** Le problème est ailleurs :
- CPU saturé (calculs lourds)
- Verrous (locks)
- Requêtes mal optimisées avec beaucoup de calculs
- Réseau lent (pour connexions distantes)

### Q3 : SSD vs HDD : quelle différence en pratique ?

**Réponse :**
- **SSD** : Latency < 1ms, parfait pour BDD  
- **HDD** : Latency 10-15ms, 10-100× plus lent

Pour PostgreSQL, un SSD fait une **différence énorme**.

### Q4 : Puis-je avoir 0% d'I/O Wait ?

**Réponse :** Théoriquement oui, si :
- Toutes les données tiennent en RAM (cache 100%)
- Pas d'écritures en cours

En pratique, 0-2% est déjà excellent.

### Q5 : RAID améliore-t-il les performances I/O ?

**Réponse :** Dépend du type :
- **RAID 0** : Oui (mais aucune redondance)  
- **RAID 1** : Lectures améliorées, écritures identiques  
- **RAID 10** : Oui, meilleur compromis perf/sécurité  
- **RAID 5/6** : Pénalité sur écritures (éviter pour PostgreSQL)

### Q6 : track_io_timing a-t-il un impact sur les performances ?

**Réponse :** Très faible (< 1% overhead). Les bénéfices du monitoring dépassent largement le coût.

```sql
ALTER SYSTEM SET track_io_timing = on;  -- Recommandé
```

## Points Clés à Retenir

✅ **I/O Wait = Temps où le CPU attend le disque**
   - Métrique système cruciale
   - Seuils : < 5% OK, > 15% problématique

✅ **Disk Latency = Temps de réponse du disque**
   - SSD : < 1ms, HDD : 10-15ms
   - Mesurable avec iostat (colonne await)

✅ **Impact sur PostgreSQL :**
   - Scans séquentiels très sensibles à l'I/O
   - Checkpoints = pics d'écriture disque
   - Cache insuffisant → Plus d'I/O

✅ **Mesure :**
   - Système : top, iostat, iotop
   - PostgreSQL : pg_stat_statements, pg_stat_io
   - PostgreSQL 18 : Statistiques I/O par backend

✅ **Optimisations principales :**
   - Augmenter shared_buffers (cache)
   - Utiliser des index appropriés
   - Migrer vers SSD (NVMe si possible)
   - Optimiser checkpoints
   - PostgreSQL 18 : Activer I/O asynchrone

✅ **Monitoring continu essentiel :**
   - Alerte si I/O Wait > 15%
   - Alerte si Latency anormale pour type disque
   - Surveiller top requêtes I/O

✅ **Causes fréquentes :**
   - Cache trop petit
   - Requêtes non optimisées
   - Disque lent ou saturé
   - Bloat excessif
   - Checkpoints trop fréquents

## Conclusion

L'**I/O Wait** et la **Disk Latency** sont des indicateurs vitaux de la santé de votre base de données PostgreSQL. Un système avec un I/O Wait élevé ou une latency importante verra ses performances drastiquement réduites, peu importe la puissance du CPU ou la qualité de vos requêtes SQL.

Les points essentiels à retenir :

1. **Comprendre** : Le disque est le goulot d'étranglement principal  
2. **Mesurer** : Utiliser iostat, top et pg_stat_statements régulièrement  
3. **Optimiser** : Cache, SSD, index, configuration WAL  
4. **Monitorer** : Surveillance continue avec alertes appropriées

Les investissements les plus rentables pour réduire l'I/O :
- 🥇 **Migration vers SSD NVMe** : Impact massif  
- 🥈 **Augmentation de RAM** : Plus de cache = moins d'I/O  
- 🥉 **Optimisation requêtes et index** : Réduit les lectures inutiles

Avec PostgreSQL 18 et son nouveau sous-système I/O asynchrone, les performances I/O peuvent être améliorées jusqu'à 3× sur certaines charges, rendant le SGBD encore plus performant sur du matériel moderne.

Un monitoring proactif et une bonne compréhension de ces métriques vous permettront de maintenir votre PostgreSQL rapide et réactif, même sous forte charge.
