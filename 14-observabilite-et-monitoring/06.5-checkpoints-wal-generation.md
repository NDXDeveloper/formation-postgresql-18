# 14.6.5. Checkpoints et WAL generation

## Introduction

Les **Checkpoints** et le **WAL (Write-Ahead Log)** sont deux mécanismes fondamentaux qui garantissent la **durabilité** et la **performance** de PostgreSQL. Bien qu'invisibles pour la plupart des utilisateurs, ils jouent un rôle crucial dans la fiabilité de votre base de données.

Dans ce chapitre, nous allons démystifier ces concepts souvent mal compris et vous donner les clés pour optimiser leur fonctionnement.

---

## Le Concept de WAL (Write-Ahead Log)

### Qu'est-ce que le WAL ?

Le **WAL** (Write-Ahead Log), aussi appelé **journal de transactions**, est un fichier séquentiel où PostgreSQL enregistre **toutes les modifications** avant de les appliquer réellement aux fichiers de données.

### Analogie : Le Cahier de Brouillon

Imaginez que vous écrivez un livre important :

**Méthode risquée (sans WAL) :**
- Vous écrivez directement dans le livre définitif
- Si vous renversez du café → Catastrophe irréversible
- Si vous faites une erreur → Difficile à corriger

**Méthode sécurisée (avec WAL) :**
- Vous écrivez d'abord sur un cahier de brouillon (WAL)
- Une fois que tout est validé, vous recopiez proprement dans le livre
- Si accident → Vous avez toujours le brouillon pour tout refaire
- Si erreur → Facile d'annuler dans le brouillon

Le WAL est le **cahier de brouillon** de PostgreSQL.

### Le Principe Write-Ahead

**"Write-Ahead"** signifie **"Écrire en avance"**.

**Ordre des opérations :**

```
1. Application demande : UPDATE users SET name = 'Alice' WHERE id = 1
2. PostgreSQL écrit d'abord dans le WAL : "Modifier ligne 1, name = Alice"
3. COMMIT
4. PostgreSQL confirme à l'application : ✅ "Transaction validée"
5. Plus tard, PostgreSQL applique réellement aux fichiers de données
```

**Point crucial :** À l'étape 4, les données ne sont PAS encore dans les fichiers définitifs, mais **c'est garanti** qu'elles le seront grâce au WAL.

### Pourquoi le WAL est Essentiel

#### 1. Garantie de Durabilité (ACID - Durability)

**Sans WAL :**
```
Application : UPDATE ...  
PostgreSQL : OK, en train d'écrire...  
💥 CRASH serveur
Résultat : Données perdues ❌
```

**Avec WAL :**
```
Application : UPDATE ...  
PostgreSQL : Écrit dans WAL → Sync disque → COMMIT  
Application : Reçoit confirmation ✅  
💥 CRASH serveur
Au redémarrage : PostgreSQL relit le WAL et applique les changements ✅  
Résultat : Aucune donnée perdue ✅  
```

#### 2. Performance

Écrire dans le WAL est **beaucoup plus rapide** qu'écrire dans les fichiers de données :

**WAL :**
- Écriture séquentielle (un fichier qui grandit)
- Très rapide, même sur HDD
- Pas de recherche de position

**Fichiers de données :**
- Écriture aléatoire (données dispersées)
- Beaucoup plus lent
- Nécessite recherche de position

**Analogie :** C'est comme écrire dans un cahier de bas en haut (rapide) vs écrire sur des post-it éparpillés dans tout le bureau (lent).

#### 3. Cohérence

Le WAL garantit que les transactions sont **atomiques** (tout ou rien).

**Exemple :**
```sql
BEGIN;  
UPDATE accounts SET balance = balance - 100 WHERE id = 1;  
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  
COMMIT;  
```

**Sans WAL :** Risque que la première UPDATE réussisse et la seconde échoue (incohérence).

**Avec WAL :** Soit les deux sont appliquées, soit aucune (atomicité garantie).

## Structure et Fonctionnement du WAL

### Les Fichiers WAL

#### Emplacement

Les fichiers WAL se trouvent dans :
```
/var/lib/postgresql/{version}/main/pg_wal/
```

**Exemple :**
```bash
ls -lh /var/lib/postgresql/18/main/pg_wal/
```

**Sortie typique :**
```
-rw------- 1 postgres postgres 16M Nov 21 10:00 000000010000000000000001
-rw------- 1 postgres postgres 16M Nov 21 10:15 000000010000000000000002
-rw------- 1 postgres postgres 16M Nov 21 10:30 000000010000000000000003
-rw------- 1 postgres postgres 16M Nov 21 10:45 000000010000000000000004
```

#### Caractéristiques

**Taille fixe :** Chaque fichier WAL fait **16 MB** (par défaut, configurable).

**Rotation :** Quand un fichier est plein, PostgreSQL crée le suivant.

**Nommage séquentiel :** Les fichiers sont numérotés de manière continue.

### Cycle de Vie d'une Modification

**Étapes détaillées :**

```
1. Transaction BEGIN
   ↓
2. Commandes SQL (INSERT, UPDATE, DELETE)
   ↓
3. Modifications écrites dans le WAL buffer (RAM)
   ↓
4. COMMIT déclenche :
   a. Flush du WAL buffer vers le disque (fsync)
   b. Confirmation à l'application
   ↓
5. Plus tard (de manière asynchrone) :
   Modifications appliquées aux fichiers de données
   ↓
6. Checkpoint :
   Synchronise tout et marque les WAL comme recyclables
```

### Types de Records WAL

Le WAL contient différents types d'entrées :

| Type | Description | Exemple |
|------|-------------|---------|
| **INSERT** | Nouvelle ligne | "Insérer ligne X dans table Y" |
| **UPDATE** | Modification | "Modifier ligne X, colonne A = nouvelle valeur" |
| **DELETE** | Suppression | "Supprimer ligne X de table Y" |
| **COMMIT** | Validation transaction | "Transaction 12345 validée" |
| **CHECKPOINT** | Point de sauvegarde | "Checkpoint démarré" |
| **VACUUM** | Nettoyage | "VACUUM sur table Z" |

### Monitoring du WAL

#### Vérifier la Génération de WAL

```sql
-- Emplacement actuel dans le WAL
SELECT pg_current_wal_lsn();
```

**Résultat exemple :**
```
 pg_current_wal_lsn
--------------------
 0/1A234567
```

**LSN** = Log Sequence Number (numéro de séquence dans le journal)

> 💡 **Trois fonctions à connaître pour les LSN courants** :  
>  
> | Fonction | Renvoie |  
> |----------|---------|  
> | `pg_current_wal_insert_lsn()` | LSN d'**insertion** : où en est PostgreSQL dans la construction des records WAL (en mémoire) |  
> | `pg_current_wal_lsn()` | LSN **écrit** : ce qui a été passé à l'OS via `write()` |  
> | `pg_current_wal_flush_lsn()` | LSN **flushed** : ce qui a effectivement atteint le disque via `fsync()` |  
>  
> Ordre toujours respecté : `insert_lsn ≥ current_lsn ≥ flush_lsn`. L'écart entre `insert` et `flush` mesure le « retard » du `walwriter` ; un écart important peut indiquer un disque WAL saturé ou un `wal_writer_delay` trop élevé.  
>  
> Sur un standby, les fonctions équivalentes sont `pg_last_wal_receive_lsn()` et `pg_last_wal_replay_lsn()` (déjà vues dans la section diagnostic standby).

#### Calculer le Volume de WAL Généré

```sql
-- WAL généré depuis un certain moment
SELECT
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), '0/1A000000')
    ) AS wal_generated;
```

#### Statistiques WAL globales : `pg_stat_wal`

Cette vue (PG 14+) donne les compteurs **cumulés** de génération WAL pour toute l'instance :

```sql
SELECT * FROM pg_stat_wal;
```

| Colonne | Description |
|---------|-------------|
| `wal_records` | Nombre total d'enregistrements WAL générés |
| `wal_fpi` | Nombre total de Full Page Images écrites |
| `wal_bytes` | Volume total de WAL généré (en octets) |
| `wal_buffers_full` | Nombre de fois où le buffer WAL a dû être flushé parce qu'il était plein |
| `stats_reset` | Dernier reset des statistiques |

> ⚠️ **Changement PG 18** : les colonnes `wal_write`, `wal_sync`, `wal_write_time` et `wal_sync_time` ont été **retirées** de `pg_stat_wal`. Les statistiques de timing WAL ont migré dans **`pg_stat_io`** (lignes où `object = 'wal'`), avec les colonnes `writes`, `write_time`, `fsyncs`, `fsync_time`. C'est cohérent avec l'enrichissement WAL de `pg_stat_io` en PG 18.

**Exemple — calculer le débit WAL et détecter une saturation de buffer :**

```sql
-- Volume WAL généré depuis le dernier reset, par seconde
SELECT
    pg_size_pretty(wal_bytes) AS total_wal,
    wal_records,
    wal_fpi,
    wal_buffers_full,
    NOW() - stats_reset AS depuis,
    pg_size_pretty(
        (wal_bytes / GREATEST(EXTRACT(EPOCH FROM NOW() - stats_reset), 1))::bigint
    ) AS wal_par_sec
FROM pg_stat_wal;
```

> 💡 Un `wal_buffers_full` qui croît rapidement indique que le buffer WAL (`wal_buffers`, défaut ~16 MB) est sous-dimensionné pour votre charge. Augmenter `wal_buffers` à 64 MB ou plus peut réduire les contentions sur le `WALWriteLock`.

#### Nouveauté PostgreSQL 18 : Statistiques WAL par Backend

PostgreSQL 18 introduit la fonction **`pg_stat_get_backend_wal(pid)`** qui retourne les compteurs de génération de WAL pour un backend individuel : `wal_records`, `wal_fpi` (full-page images), `wal_bytes` et `stats_reset`.

> ⚠️ **Attention :** la vue `pg_stat_activity` ne contient PAS de colonnes `wal_records` / `wal_bytes` / `wal_fpi`. Il faut passer par la fonction `pg_stat_get_backend_wal(pid)` jointe à `pg_stat_activity` via `LATERAL`.

**Requête (PG 18) : top des sessions par volume de WAL généré**

```sql
SELECT
    a.pid,
    a.usename,
    a.application_name,
    a.state,
    w.wal_records,
    w.wal_fpi,
    pg_size_pretty(w.wal_bytes) AS wal_bytes_pretty
FROM pg_stat_activity a  
CROSS JOIN LATERAL pg_stat_get_backend_wal(a.pid) AS w  
WHERE a.backend_type = 'client backend'  
  AND w.wal_bytes > 0
ORDER BY w.wal_bytes DESC  
LIMIT 20;  
```

**Utilité :** identifier en temps réel quelles sessions génèrent le plus de WAL — utile pour repérer une application qui sature la réplication ou les checkpoints à cause d'écritures massives.

> 💡 La fonction complémentaire `pg_stat_reset_backend_stats(pid)` (PG 18) permet de remettre à zéro les compteurs d'un backend pour mesurer un intervalle précis.

## Qu'est-ce qu'un Checkpoint ?

### Définition

Un **checkpoint** est un processus qui synchronise toutes les modifications en mémoire (dirty buffers) vers le disque.

**En termes simples :** C'est le moment où PostgreSQL recopie proprement du brouillon (WAL) vers le livre définitif (fichiers de données).

### Analogie : La Sauvegarde de Jeu Vidéo

Quand vous jouez à un jeu vidéo :

**Sans sauvegarde (sans checkpoint) :**
- Vous jouez pendant 2 heures
- 💥 Crash
- Vous devez tout recommencer depuis le début

**Avec sauvegardes régulières (checkpoints) :**
- Le jeu se sauvegarde toutes les 15 minutes
- 💥 Crash
- Vous reprenez depuis la dernière sauvegarde (max 15 min perdues)

Les checkpoints sont les **points de sauvegarde** de PostgreSQL.

### Pourquoi les Checkpoints Existent

#### 1. Récupération Rapide Après Crash

**Sans checkpoint :**
```
Crash serveur → Redémarrage  
PostgreSQL doit relire TOUT le WAL depuis le début (peut-être des Go)  
Temps de récupération : Des heures ❌  
```

**Avec checkpoints réguliers :**
```
Crash serveur → Redémarrage  
PostgreSQL relit seulement depuis le dernier checkpoint (quelques MB)  
Temps de récupération : Quelques secondes ✅  
```

#### 2. Recyclage du WAL

Les fichiers WAL peuvent être réutilisés après un checkpoint :

```
Checkpoint N → Tout avant le checkpoint N est sûrement sur disque
            → Anciens fichiers WAL peuvent être supprimés/recyclés
            → Espace disque récupéré
```

#### 3. Cohérence des Données

Un checkpoint garantit que les données sur disque sont dans un état **cohérent** à un instant T.

## Types de Checkpoints

### 1. Timed Checkpoints (Checkpoint Temporisés)

Déclenchés automatiquement selon un **intervalle de temps**.

**Paramètre :**
```ini
checkpoint_timeout = 5min  # Par défaut
```

**Exemple :**
```
10:00 → Checkpoint
10:05 → Checkpoint
10:10 → Checkpoint
...
```

**Avantage :** Prévisible et régulier.

### 2. Requested Checkpoints (Checkpoint Requis)

Déclenchés automatiquement quand le WAL atteint une **taille limite**.

**Paramètre :**
```ini
max_wal_size = 1GB  # Par défaut
```

**Exemple :**
```
WAL atteint 1 GB → Checkpoint déclenché immédiatement
```

**Problème :** Si votre charge est intense, le WAL peut atteindre la limite avant le timeout → Checkpoints plus fréquents que prévu.

### 3. Manual Checkpoints (Checkpoint Manuels)

Déclenchés par commande SQL.

```sql
CHECKPOINT;
```

**Usage :** Rarement nécessaire, sauf pour :
- Avant une sauvegarde physique
- Avant maintenance planifiée
- Tests

## Impact Performance des Checkpoints

### Le Problème des "Checkpoint Spikes"

**Symptôme :** Pics de lenteur périodiques et prévisibles.

**Cause :** Pendant un checkpoint, PostgreSQL écrit massivement sur le disque.

**Visualisation :**
```
Performance normale : ████████████████████████████████████████  
Checkpoint démarre  : ██░░░░░░░░░░░░░░░░░░░░██████████████████  
                         ↑
                    Ralentissement
```

**Utilisateurs :** "Pourquoi l'application est lente toutes les 5 minutes ?!"

### Pourquoi les Checkpoints Ralentissent

#### 1. I/O Burst (Rafale d'Écritures)

Le checkpoint doit écrire tous les "dirty buffers" (pages modifiées en mémoire) sur le disque.

**Exemple :**
```
Shared buffers = 8 GB
50% de pages modifiées = 4 GB à écrire
Checkpoint a 5 minutes pour tout écrire
→ 800 MB/sec d'écritures pendant 5 minutes
```

Sur un disque limité à 500 MB/sec → Saturation → Ralentissement global.

#### 2. Contention I/O

Les écritures du checkpoint entrent en compétition avec :
- Les requêtes normales
- Les VACUUM
- Les index scans
- Les autres processus

**Résultat :** Tout ralentit.

#### 3. Flush Synchrone

À la fin du checkpoint, PostgreSQL fait un **fsync** (force la synchronisation disque).

**Sur certains systèmes :** Peut bloquer pendant plusieurs secondes.

## Configuration et Optimisation

### Paramètres Clés

#### 1. checkpoint_timeout

**Définition :** Temps maximum entre deux checkpoints.

```ini
# postgresql.conf
checkpoint_timeout = 5min   # Défaut : 5 minutes
```

**Recommandations :**

| Charge | Valeur | Raison |
|--------|--------|--------|
| **Faible** | 5-10 min | Défaut suffisant |
| **Moyenne** | 10-15 min | Réduit fréquence |
| **Élevée (OLTP)** | 15-30 min | Moins d'interruptions |
| **Très élevée** | 30-60 min | Max avant risques |

**⚠️ Attention :** Augmenter trop → Temps de récupération après crash plus long.

#### 2. max_wal_size

**Définition :** Taille max de WAL avant déclencher un checkpoint.

```ini
# postgresql.conf
max_wal_size = 1GB   # Défaut : 1 GB
```

**Recommandations :**

| Charge | Valeur | Raison |
|--------|--------|--------|
| **Faible** | 1-2 GB | Défaut suffisant |
| **Moyenne** | 2-4 GB | Plus de marge |
| **Élevée** | 4-8 GB | Évite checkpoints requis |
| **Très élevée (batch)** | 8-16 GB | Grosses transactions |

**Calcul :**
```
Si vous générez 200 MB WAL/minute  
max_wal_size = 4 GB  
→ Checkpoint requis toutes les 20 minutes environ
```

**💡 Astuce :** Augmenter `max_wal_size` réduit la fréquence des checkpoints requis.

#### 3. checkpoint_completion_target

**Définition :** Pourcentage du checkpoint_timeout pendant lequel étaler les écritures.

```ini
# postgresql.conf
checkpoint_completion_target = 0.9   # Défaut : 0.9 (90%)
```

**Fonctionnement :**

```
checkpoint_timeout = 10 min  
checkpoint_completion_target = 0.9  

Checkpoint démarre à 10:00  
Objectif : Finir en 9 minutes (90% de 10 min)  
→ Écritures étalées entre 10:00 et 10:09
→ Dernière minute (10:09-10:10) : Finalisation et fsync
```

**Effet :** Lisse les écritures au lieu d'avoir un pic brutal.

**Visualisation :**

**Sans lissage (target = 0.1) :**
```
I/O : ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░████████████████
      ↑                              ↑
      Checkpoint                   Tout écrit d'un coup
```

**Avec lissage (target = 0.9) :**
```
I/O : ░░░░░░░░░░░░░░░░░░░░░░░░░░▁▂▃▄▅▆▇█▇▆▅▄▃▂▁░░░░
      ↑                          ↑              ↑
      Checkpoint             Étalé            Fin
```

**Recommandation :** Garder à 0.9 (valeur optimale pour la plupart des cas).

#### 4. min_wal_size

**Définition :** Taille minimale de WAL à conserver.

```ini
# postgresql.conf
min_wal_size = 80MB   # Défaut : 80 MB
```

**Usage :** PostgreSQL garde toujours au moins cette quantité de WAL, même après un checkpoint.

**Pourquoi ?** Évite de recréer constamment des fichiers WAL (coûteux).

#### 5. wal_level

**Définition :** Quantité d'informations écrites dans le WAL.

```ini
# postgresql.conf
wal_level = replica   # Défaut : replica
```

**Valeurs possibles :**

| Valeur | Description | Usage |
|--------|-------------|-------|
| **minimal** | WAL minimal | Standalone, pas de réplication |
| **replica** | Streaming replication | Défaut, recommandé |
| **logical** | Réplication logique | Publications/Subscriptions |

**Recommandation :** Garder `replica` ou `logical` selon besoins de réplication.

#### 6. wal_keep_size (PG 13+)

**Définition :** Quantité **minimale** de WAL conservée pour les replicas sans slot de réplication.

```ini
# postgresql.conf
wal_keep_size = 0   # Défaut : 0 (désactivé)
# wal_keep_size = 2GB
```

> ⚠️ **Renommage PG 13** : ce paramètre s'appelait `wal_keep_segments` jusqu'en PG 12 (en nombre de fichiers de 16 MB) — il est maintenant exprimé directement en taille (`wal_keep_size`, en MB par défaut).

**Quand l'utiliser ?**
- Vous avez un standby **sans slot de réplication** (architecture simple, sans risque de slot oublié) — `wal_keep_size` garantit qu'il reste un peu de marge si le standby décroche brièvement.
- **Avec un slot de réplication**, ce paramètre est généralement inutile : le slot lui-même garantit la rétention. Préférer alors `max_slot_wal_keep_size` pour plafonner la rétention.

> 💡 **Compromis** : `wal_keep_size = 0` (défaut) est correct pour la plupart des architectures modernes utilisant des slots. Ne le passer à 1-2 GB qu'en cas de standby sans slot, et en acceptant la sur-occupation disque correspondante.

### Configuration Optimisée par Type de Charge

#### Configuration OLTP (Haute Performance)

```ini
# postgresql.conf

# WAL
wal_level = replica  
wal_buffers = 16MB  
max_wal_size = 4GB  
min_wal_size = 1GB  

# Checkpoints
checkpoint_timeout = 15min  
checkpoint_completion_target = 0.9  

# Permet plus de workers pour checkpoints longs
max_wal_senders = 10
```

#### Configuration OLAP (Analytics)

```ini
# postgresql.conf

# WAL
max_wal_size = 8GB  
min_wal_size = 2GB  

# Checkpoints
checkpoint_timeout = 30min  
checkpoint_completion_target = 0.9  

# Grosses transactions OK
```

#### Configuration Batch Processing

```ini
# postgresql.conf

# WAL
max_wal_size = 16GB      # Très élevé  
checkpoint_timeout = 60min  

# Pour imports massifs
maintenance_work_mem = 2GB
```

## Monitoring des Checkpoints

### Statistiques de Base

#### Vues pg_stat_checkpointer et pg_stat_bgwriter

> ⚠️ **Changement important PostgreSQL 17+** : les statistiques de checkpoints **ne sont plus** dans `pg_stat_bgwriter`. Elles ont été déplacées dans une nouvelle vue dédiée **`pg_stat_checkpointer`**. La vue `pg_stat_bgwriter` ne contient désormais que les compteurs propres au background writer (`buffers_clean`, `maxwritten_clean`, `buffers_alloc`). Le compteur `buffers_backend` a été retiré (l'information équivalente se trouve maintenant dans `pg_stat_io`, contexte `normal`, opération `writes` pour les backends client).

**Requête PG 17+ pour les statistiques de checkpoints :**

```sql
SELECT
    num_timed,           -- checkpoints planifiés (timeout)
    num_requested,       -- checkpoints forcés (max_wal_size atteint)
    num_done,            -- (PG 18+) checkpoints terminés
    write_time,          -- ms passés à écrire les buffers
    sync_time,           -- ms passés en fsync
    buffers_written,     -- shared buffers écrits pendant checkpoints
    slru_written         -- (PG 18+) SLRU buffers écrits pendant checkpoints
FROM pg_stat_checkpointer;
```

> 💡 **Distinction PG 18** : `num_timed` et `num_requested` comptent les checkpoints **demandés** (qu'ils aient été menés à terme ou ignorés/coalescés), tandis que `num_done` (nouveau PG 18) compte uniquement ceux **réellement effectués**. Un écart `num_timed + num_requested > num_done` indique que des checkpoints ont été annulés (par exemple parce qu'aucune écriture n'avait eu lieu depuis le précédent).  
>  
> 💡 **`slru_written` (PG 18+)** : compte les buffers SLRU (CLOG, subtransaction log, multixact, etc.) écrits pendant les checkpoints. Auparavant ces écritures étaient comptées avec les buffers partagés — la séparation permet d'isoler la pression sur les SLRU des écritures de pages de données. Le message de log « checkpoint complete » a aussi été modifié en PG 18 pour les rapporter séparément.

> 🔧 **Paramètres SLRU configurables (PG 17+)** : avant PG 17, la taille des caches SLRU était figée (128 kB pour la plupart). Beaucoup de contentions visibles dans `pg_stat_activity` (wait events `LWLock:MultiXactOffsetSLRU`, `LWLock:SubtransSLRU`…) en sont la conséquence. PG 17 expose ces tailles comme paramètres serveur (en blocs de 8 kB, modifiables au démarrage uniquement) :  
>  
> | Paramètre | Cache | Défaut |  
> |-----------|-------|--------|  
> | `transaction_buffers` | `pg_xact` (CLOG) | auto (jusqu'à 1024 blocs) |  
> | `subtransaction_buffers` | `pg_subtrans` | auto (jusqu'à 1024 blocs) |  
> | `commit_timestamp_buffers` | `pg_commit_ts` | auto (jusqu'à 1024 blocs) |  
> | `multixact_offset_buffers` | `pg_multixact/offsets` | 16 blocs |  
> | `multixact_member_buffers` | `pg_multixact/members` | 32 blocs |  
> | `notify_buffers` | `pg_notify` | 16 blocs |  
> | `serializable_buffers` | `pg_serial` | 32 blocs |  
>  
> Si vous voyez beaucoup de wait events `LWLock:<xxx>SLRU` dans `pg_stat_activity`, augmenter le buffer correspondant peut éliminer la contention. Multiplier par 4-8 est en général sûr et efficace.

#### Diagnostiquer les SLRU : `pg_stat_slru` (PG 13+)

Cette vue donne une ligne par cache SLRU avec les compteurs hit / miss / read / write — exactement ce qu'il faut pour valider si un cache SLRU est trop petit :

```sql
SELECT
    name,
    blks_zeroed,
    blks_hit,
    blks_read,
    blks_written,
    blks_exists,
    flushes,
    truncates,
    ROUND(100.0 * blks_hit / NULLIF(blks_hit + blks_read, 0), 2) AS hit_ratio
FROM pg_stat_slru  
ORDER BY blks_read DESC;  
```

| `name` | Cache concerné |
|--------|---------------|
| `commit_timestamp` | `pg_commit_ts` |
| `multixact_member` | `pg_multixact/members` |
| `multixact_offset` | `pg_multixact/offsets` |
| `notify` | `pg_notify` (LISTEN/NOTIFY) |
| `serializable` | `pg_serial` |
| `subtransaction` | `pg_subtrans` |
| `transaction` | `pg_xact` (CLOG) |
| `other` | autres caches SLRU internes |

> ⚠️ **Renommage PG 17** : ces valeurs de `name` ont été **renommées** en PG 17 pour s'aligner sur les nouveaux paramètres `*_buffers`. Avant PG 17, elles s'appelaient `CommitTs`, `MultiXactMember`, `MultiXactOffset`, `Notify`, `Serial`, `Subtrans`, `Xact`. Adaptez vos scripts de monitoring si vous migrez depuis PG ≤ 16.

> 💡 **Diagnostic** : un `hit_ratio < 95 %` sur un cache (typiquement `Subtrans` ou `MultiXactMember`) confirme un sous-dimensionnement. Augmenter le `xxx_buffers` correspondant (PG 17+) résout immédiatement la contention.

**Et pour les statistiques du background writer lui-même :**

```sql
SELECT
    buffers_clean,        -- buffers écrits par le bgwriter
    maxwritten_clean,     -- nombre de fois où le bgwriter s'est arrêté (quota dépassé)
    buffers_alloc         -- buffers alloués
FROM pg_stat_bgwriter;
```

#### Calculer le Ratio de Checkpoints Forcés (PG 17+)

```sql
SELECT
    num_timed,
    num_requested,
    round(
        100.0 * num_requested /
        NULLIF(num_timed + num_requested, 0),
        2
    ) AS forced_checkpoint_pct
FROM pg_stat_checkpointer;
```

**Interprétation :**

| Pourcentage | État | Action |
|-------------|------|--------|
| **< 10%** | 🟢 Excellent | RAS |
| **10-30%** | 🟡 Acceptable | Surveiller |
| **30-50%** | 🟠 Préoccupant | Augmenter max_wal_size |
| **> 50%** | 🔴 Problématique | Action urgente |

**Objectif :** Avoir < 10% de checkpoints forcés.

**Pourquoi ?** Les checkpoints forcés sont non planifiés et peuvent causer des pics de latence.

**Diagnostic :**


**Causes possibles** (par ordre de fréquence) :

2. Réplication en retard (standby lent ou tombé)

#### Diagnostic 1 : les replication slots

C'est **la cause n°1** de `pg_wal/` qui explose. Un slot de réplication marqué « inactif » empêche PostgreSQL de recycler le WAL tant qu'il n'a pas été acquitté.

```sql
SELECT
    slot_name,
    slot_type,             -- 'physical' ou 'logical'
    active,                -- true si un consommateur est connecté
    active_pid,            -- PID du wal sender associé (si active = true)
    restart_lsn,           -- LSN à partir duquel PG doit conserver le WAL
    confirmed_flush_lsn,   -- LSN confirmé par le consommateur (slots logiques)
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)
    ) AS wal_retenu,
    wal_status,            -- 'reserved', 'extended', 'unreserved', 'lost'
    pg_size_pretty(safe_wal_size) AS safe_wal_size,
    invalidation_reason    -- (PG 16+/18+) raison de l'invalidation si lost
FROM pg_replication_slots  
ORDER BY pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) DESC NULLS LAST;  
```

**Interprétation de `wal_status`** :

| Statut | Signification |
|--------|---------------|
| `reserved` | Le WAL retenu tient dans `max_wal_size`, situation normale |
| `extended` | Le WAL retenu dépasse `max_wal_size` — alerte |
| `unreserved` | Le WAL retenu approche `max_slot_wal_keep_size` (PG 13+) — danger imminent |
| `lost` | Le slot est **invalidé** : le WAL nécessaire a été supprimé, le consommateur ne pourra plus rattraper |

> 💡 **Nouveauté PG 18 — `idle_replication_slot_timeout`** : nouveau paramètre serveur qui invalide automatiquement les slots de réplication restés inactifs trop longtemps. Empêche les slots oubliés de saturer le disque. Défaut : `0` (désactivé). Recommandation production : `24h` ou `7d` selon le contexte.  
>  
> ```ini  
> # postgresql.conf (PG 18+)  
> idle_replication_slot_timeout = '1d'  
> ```  
>  
> Quand un slot est invalidé pour cette raison, `invalidation_reason = 'idle_timeout'`.

**Statistiques détaillées par slot logique : `pg_stat_replication_slots` (PG 14+)** :

Pour les slots **logiques** (utilisés par `pg_logical` ou les `SUBSCRIPTION`), une vue complémentaire donne des compteurs précieux :

```sql
SELECT
    slot_name,
    spill_txns, spill_count, pg_size_pretty(spill_bytes) AS spill_size,
    stream_txns, stream_count, pg_size_pretty(stream_bytes) AS stream_size,
    total_txns, pg_size_pretty(total_bytes) AS total_size
FROM pg_stat_replication_slots;
```

| Colonne | Signification |
|---------|---------------|
| `spill_*` | Transactions ayant **débordé sur disque** car `logical_decoding_work_mem` insuffisant |
| `stream_*` | Transactions **streamées en cours** (PG 14+) — bonne nouvelle, c'est efficace |
| `total_*` | Volume total traité par ce slot |

> 💡 Un `spill_count` élevé indique que `logical_decoding_work_mem` (défaut 64 MB) est trop petit. Augmenter à 256 MB ou plus permet de garder plus de transactions en mémoire et accélère la réplication logique.

#### Diagnostic 2 : les replicas en retard

```sql
SELECT
    application_name,
    client_addr,
    state,                          -- 'streaming', 'catchup', 'startup'…
    sync_state,                     -- 'sync', 'async', 'quorum'…
    -- Retard en VOLUME (différence de LSN, exprimée en taille)
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), sent_lsn)
    ) AS send_byte_lag,
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), write_lsn)
    ) AS write_byte_lag,
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), flush_lsn)
    ) AS flush_byte_lag,
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn)
    ) AS replay_byte_lag,
    -- Retard en TEMPS (colonnes natives de pg_stat_replication, type interval)
    write_lag    AS write_time_lag,
    flush_lag    AS flush_time_lag,
    replay_lag   AS replay_time_lag
FROM pg_stat_replication;
```

> 💡 **Différence importante** : `pg_wal_lsn_diff()` mesure le retard **en octets** (combien de WAL le standby doit encore lire), tandis que les colonnes `write_lag` / `flush_lag` / `replay_lag` (PG 10+) mesurent le retard **en temps** (combien de temps une transaction commitée sur le primaire attendra côté replica). Les deux sont utiles : un retard en octets de quelques MB n'est pas grave, mais un `replay_lag > 1 minute` rend les requêtes sur le replica obsolètes.

Un `replay_byte_lag` important indique un standby qui n'arrive pas à suivre — la cause habituelle d'un `pg_wal/` qui croît sur le primaire.

**Diagnostic côté standby** (requête à exécuter sur le replica) :

```sql
-- Sur le standby : suis-je vraiment un standby et quel est mon état ?
SELECT
    pg_is_in_recovery()                AS in_recovery,
    pg_is_wal_replay_paused()          AS replay_paused,
    pg_last_wal_receive_lsn()          AS last_received_lsn,
    pg_last_wal_replay_lsn()           AS last_replayed_lsn,
    pg_last_xact_replay_timestamp()    AS last_replayed_tx_time,
    -- Combien de temps depuis la dernière transaction rejouée ?
    NOW() - pg_last_xact_replay_timestamp() AS staleness,
    -- Volume WAL reçu mais pas encore rejoué
    pg_size_pretty(
        pg_wal_lsn_diff(pg_last_wal_receive_lsn(), pg_last_wal_replay_lsn())
    ) AS receive_replay_gap;
```

**Interprétation** :
- `pg_is_in_recovery() = false` sur ce qui devrait être un standby → c'est en réalité un primaire (failover oublié ?).
- `staleness` croît continûment → le standby applique le WAL plus lentement qu'il n'arrive.
- `receive_replay_gap` important (> 100 MB) → le replay est le goulot (rare en streaming, plus fréquent en logical replication).
- `pg_is_wal_replay_paused() = true` → quelqu'un a explicitement mis en pause le replay (`pg_wal_replay_pause()`).

#### Diagnostic 3 : `archive_command` en échec

```sql
SELECT
    archived_count,
    last_archived_wal,
    last_archived_time,
    failed_count,
    last_failed_wal,
    last_failed_time,
    stats_reset
FROM pg_stat_archiver;
```

Si `failed_count` croît et `last_archived_time` n'avance plus, l'`archive_command` échoue en boucle → PG ne recycle pas le WAL.

**Actions correctives** :

```sql
-- Plafonner ce qu'un slot peut retenir (PG 13+) :
ALTER SYSTEM SET max_slot_wal_keep_size = '64GB';  
SELECT pg_reload_conf();  
-- Les slots qui dépassent passent en wal_status = 'lost' et leur WAL est recyclé.

-- Supprimer un slot dont on est SÛR qu'il n'est plus utilisé :
-- (ne pas drop un slot encore consommé : le consommateur ne pourra plus repartir)
SELECT pg_drop_replication_slot('old_slot_name');

-- Forcer un checkpoint après libération
CHECKPOINT;
```

> ⚠️ **Avant de supprimer un slot**, vérifiez `active = false` ET qu'aucun consommateur légitime n'en dépend (replica déconnecté temporairement, subscription logique en pause, etc.). Un slot supprimé à tort impose une **re-synchronisation complète** du replica.

### Problème 4 : WAL Generation Excessive

**Symptômes :**
- Génération de plusieurs GB de WAL en quelques minutes
- Checkpoints très fréquents malgré configuration correcte

**Diagnostic (PostgreSQL 18) :**

```sql
-- PG 18 : pg_stat_get_backend_wal(pid) joint à pg_stat_activity
SELECT
    a.pid,
    a.usename,
    a.application_name,
    pg_size_pretty(w.wal_bytes) AS wal_generated,
    w.wal_records,
    w.wal_fpi,
    a.query
FROM pg_stat_activity a  
CROSS JOIN LATERAL pg_stat_get_backend_wal(a.pid) AS w  
WHERE w.wal_bytes > 0  
ORDER BY w.wal_bytes DESC  
LIMIT 10;  
```

**Causes fréquentes :**
1. Grosses transactions (COPY, bulk INSERT)  
2. Nombreux UPDATE sur colonnes indexées  
3. VACUUM FULL en cours

**Solutions :**

**Pour imports massifs :**
```sql
-- Temporairement désactiver autovacuum
ALTER TABLE big_table SET (autovacuum_enabled = false);

-- Import
COPY big_table FROM '/data/file.csv';

-- Réactiver
ALTER TABLE big_table SET (autovacuum_enabled = true);  
VACUUM ANALYZE big_table;  
```

**Pour updates massifs :**
```sql
-- Utiliser batching
DO $$  
DECLARE  
    batch_size INT := 10000;
BEGIN
    LOOP
        UPDATE my_table
        SET colonne = nouvelle_valeur          -- « column » est un mot réservé : utiliser un identifiant non réservé
        WHERE ctid IN (
            SELECT ctid
            FROM my_table
            WHERE condition
            LIMIT batch_size
        );

        EXIT WHEN NOT FOUND;
        COMMIT;
    END LOOP;
END $$;
```

## WAL Archiving et Point-in-Time Recovery

### Concept de WAL Archiving

**WAL Archiving** permet de sauvegarder les fichiers WAL avant qu'ils soient recyclés.

**Utilité :**
- Restauration à un instant précis (PITR)
- Réplication vers standby distant
- Audit et compliance

### Configuration Basique

```ini
# postgresql.conf
archive_mode = on  
archive_command = 'cp %p /mnt/backup/wal/%f'  
```

**Variables :**
- `%p` : Chemin complet du fichier WAL  
- `%f` : Nom du fichier WAL

### Script d'Archivage Robuste

```bash
#!/bin/bash
# /usr/local/bin/archive_wal.sh

WAL_FILE=$1  
DEST_DIR="/mnt/backup/wal"  

# Copier avec vérification
cp "$WAL_FILE" "$DEST_DIR/" || exit 1

# Vérifier checksum
md5sum "$WAL_FILE" > "$DEST_DIR/$(basename $WAL_FILE).md5"

exit 0
```

**Configuration :**
```ini
archive_command = '/usr/local/bin/archive_wal.sh %p'
```

### Point-in-Time Recovery (PITR)

**Scénario :** Suppression accidentelle de données à 14:30.

**Restauration à 14:25 (avant l'erreur) :**

```bash
# 1. Arrêter PostgreSQL
sudo systemctl stop postgresql

# 2. Restaurer backup de base (ex: backup de ce matin)
rm -rf /var/lib/postgresql/18/main/*  
tar xzf /backup/base_backup_20241121_0600.tar.gz -C /var/lib/postgresql/18/main/  

# 3. Créer recovery.signal
touch /var/lib/postgresql/18/main/recovery.signal

# 4. Configurer recovery
cat > /var/lib/postgresql/18/main/postgresql.auto.conf << EOF  
restore_command = 'cp /mnt/backup/wal/%f %p'  
recovery_target_time = '2024-11-21 14:25:00'  
EOF  

# 5. Redémarrer
sudo systemctl start postgresql

# PostgreSQL va :
# - Appliquer le backup de base
# - Rejouer les WAL jusqu'à 14:25
# - S'arrêter à 14:25
```

## Optimisations Avancées

### 1. Séparer WAL et Données

**Principe :** Placer WAL et fichiers de données sur des disques différents.

**Avantages :**
- Réduit contention I/O
- WAL sur disque rapide (SSD) même si données sur HDD
- Parallélisation des écritures

**Configuration :**

```bash
# Déplacer pg_wal
sudo systemctl stop postgresql  
mv /var/lib/postgresql/18/main/pg_wal /mnt/fast-ssd/pg_wal  
ln -s /mnt/fast-ssd/pg_wal /var/lib/postgresql/18/main/pg_wal  
sudo systemctl start postgresql  
```

### 2. Optimiser fsync

**Le problème du fsync :** Force synchronisation disque, peut bloquer.

**Paramètres :**

```ini
# postgresql.conf
fsync = on   # Ne JAMAIS désactiver en production !

# Méthode de sync
wal_sync_method = fdatasync   # Linux: fdatasync (défaut et optimal)
```

**Options wal_sync_method :**

| Méthode | Performance | Sécurité | OS |
|---------|-------------|----------|-----|
| **fdatasync** | Rapide | Élevée | Linux (recommandé) |
| **fsync** | Moyen | Élevée | Tous |
| **open_sync** | Lent | Élevée | Tous |
| **open_datasync** | Lent | Élevée | Certains Unix |

**Recommandation :** Garder `fdatasync` sur Linux.

### 3. Ajuster wal_buffers

**Définition :** Taille du buffer WAL en mémoire avant écriture disque.

```ini
# postgresql.conf
wal_buffers = 16MB   # Défaut : -1 (auto, 1/32 de shared_buffers)
```

**Recommandations :**

| shared_buffers | wal_buffers | Charge |
|----------------|-------------|--------|
| < 1 GB | 16 MB | Légère |
| 4 GB | 16 MB | Moyenne |
| 8 GB | 16 MB | Élevée |
| 16+ GB | 32 MB | Très élevée |

**💡 Astuce :** Rarement besoin de dépasser 16 MB, sauf charge extrême.

### 4. Commit Asynchrone (⚠️ Risqué)

Pour applications acceptant perte de quelques transactions en cas de crash.

```ini
# postgresql.conf
synchronous_commit = off   # ⚠️ Perte potentielle de données
```

**Gain :** Performance×2-5 sur écritures.

**Risque :** Perte des dernières transactions (< 1 sec) en cas de crash.

**Usage :** Acceptable pour :
- Logs non critiques
- Sessions temporaires
- Données recalculables

**Ne JAMAIS utiliser pour :** Données financières, commandes, etc.

### 5. Compression WAL

> ℹ️ **Histoire** : `wal_compression` existe depuis **PostgreSQL 9.5**, mais à l'époque c'était un booléen `on/off` qui n'utilisait que l'algorithme `pglz`. **PostgreSQL 15+** a élargi le paramètre pour choisir l'algorithme : `off`, `pglz`, `lz4`, `zstd` (et `on` reste un alias pour `pglz`).

Le paramètre compresse les **Full Page Images** (FPI) — pas la totalité du WAL. C'est sur ces FPI que la compression apporte le plus de gain car ils représentent l'essentiel du volume après un checkpoint.

```ini
# postgresql.conf

# Option historique (PG 9.5+)
wal_compression = pglz

# Recommandée en PG 15+ (meilleur compromis débit/CPU)
wal_compression = lz4

# Ratio maximum, plus de CPU (PG 15+)
wal_compression = zstd

# Désactivée
wal_compression = off
```

| Algorithme | Disponible | Compression | Coût CPU |
|------------|------------|-------------|----------|
| `pglz` | PG 9.5+ | Modérée (~30 %) | Faible |
| `lz4`  | PG 15+ | Bonne (~50 %)   | Très faible (recommandé) |
| `zstd` | PG 15+ | Excellente (~70 %) | Modéré |

**Recommandation :** activer `lz4` (PG 15+) — coût CPU quasi nul et gain WAL substantiel, particulièrement bénéfique pour le streaming replication et l'archiving.

```promql
rate(pg_stat_checkpointer_num_requested_total[5m])  
```

**Forced checkpoint ratio :**
```promql
pg_stat_checkpointer_num_requested_total /
(pg_stat_checkpointer_num_timed_total + pg_stat_checkpointer_num_requested_total)
```

## Questions Fréquentes (FAQ)

### Q1 : Puis-je désactiver fsync pour plus de performances ?

**Réponse :** **NON, JAMAIS en production !**

Sans fsync, un crash peut corrompre toute votre base de données.

**Exception :** Environnements de test/dev uniquement.

### Q2 : Pourquoi mes checkpoints prennent 15 minutes ?

**Réponses possibles :**
1. `shared_buffers` trop grand (> 25% RAM)  
2. Disque lent (HDD)  
3. Beaucoup de données modifiées  
4. `checkpoint_timeout` trop long

**Solution :** Réduire checkpoint_timeout ou améliorer I/O.

### Q3 : Mon pg_wal/ fait 50 GB, est-ce normal ?

**Réponse :** Non, c'est anormal.

**Causes :**
- Réplication en retard
- Slots de réplication abandonnés
- Archiving défaillant

**Action :** Vérifier `pg_replication_slots` et corriger.

### Q4 : Quelle est la taille idéale de max_wal_size ?

**Réponse :** Dépend de votre charge.

**Formule approximative :**
```
max_wal_size = WAL généré par minute × checkpoint_timeout × 1.5
```

**Exemple :**
```
Génération : 100 MB/min  
checkpoint_timeout : 15 min  
max_wal_size = 100 × 15 × 1.5 = 2.25 GB → Configurer 3 GB  
```

### Q5 : Les checkpoints peuvent-ils causer un downtime ?

**Réponse :** Non, jamais.

Ils ralentissent les performances mais ne bloquent pas PostgreSQL.

### Q6 : Dois-je activer wal_compression ?

**Réponse :** Oui, dans la plupart des cas.

**Avantages :** Moins d'I/O, moins d'espace disque  
**Coût :** Négligeable en CPU  

**Exception :** Si CPU déjà saturé.

## Points Clés à Retenir

✅ **WAL (Write-Ahead Log) = Journal de transactions**
   - Toutes modifications écrites d'abord dans le WAL
   - Garantit durabilité (ACID)
   - Écriture séquentielle = rapide

✅ **Checkpoints = Points de sauvegarde**
   - Synchronisent modifications mémoire → disque
   - Permettent récupération rapide après crash
   - Permettent recyclage du WAL

✅ **Types de checkpoints :**
   - Timed (par timeout) : Prévisible
   - Requested (par taille WAL) : À minimiser
   - Manual : Rarement nécessaire

✅ **Impact performance :**
   - Checkpoints génèrent I/O burst
   - Peuvent causer pics de latence
   - Lissés avec checkpoint_completion_target

✅ **Configuration clés :**  
   - `max_wal_size` : 2-8 GB selon charge  
   - `checkpoint_timeout` : 10-30 min  
   - `checkpoint_completion_target` : 0.9 (optimal)  
   - `log_checkpoints` : on (monitoring)

✅ **PostgreSQL 18 :**
   - Statistiques WAL par backend
   - Meilleure observabilité
   - I/O asynchrone améliore checkpoints

✅ **Monitoring essentiel :**
   - Forced checkpoint ratio < 10%
   - Durée checkpoint < timeout
   - Taille pg_wal/ stable
   - Logs consultés régulièrement

✅ **Optimisations :**
   - Séparer WAL et données (disques différents)
   - Compression WAL (si I/O limité)
   - SSD pour meilleurs performances
   - Ne JAMAIS désactiver fsync en production

## Conclusion

Le **WAL** et les **Checkpoints** sont les gardiens invisibles de votre base de données PostgreSQL. Le WAL garantit que chaque transaction validée est **définitivement** sauvegardée, même en cas de crash, tandis que les checkpoints permettent une **récupération rapide** et le **recyclage efficace** de l'espace disque.

Comprendre ces mécanismes vous permet de :

1. **Optimiser les performances** en réduisant l'impact des checkpoints  
2. **Garantir la fiabilité** avec une configuration WAL appropriée  
3. **Monitorer efficacement** pour détecter les problèmes avant qu'ils deviennent critiques  
4. **Récupérer rapidement** en cas de désastre grâce au WAL archiving

Les configurations par défaut de PostgreSQL sont bonnes, mais les ajuster selon votre charge spécifique peut apporter des gains significatifs en performance et stabilité.

**Règle d'or :** Ne sacrifiez JAMAIS la durabilité pour la performance. Un système rapide mais qui perd des données n'a aucune valeur.

Avec PostgreSQL 18 et ses améliorations sur les statistiques WAL et l'I/O asynchrone, vous avez maintenant tous les outils nécessaires pour optimiser et surveiller ces aspects critiques de votre base de données.
