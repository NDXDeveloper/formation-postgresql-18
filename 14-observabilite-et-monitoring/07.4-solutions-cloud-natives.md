# 14.7.4. Solutions cloud natives (CloudWatch, Azure Monitor, GCP)

## Introduction

Lorsque vous hébergez PostgreSQL dans le cloud (AWS, Azure ou Google Cloud), vous avez accès à des **solutions de monitoring intégrées** fournies directement par les fournisseurs cloud. Ces solutions sont **cloud-natives** : elles sont conçues pour s'intégrer naturellement avec l'infrastructure cloud et les services managés.

### Cloud-native vs Self-hosted

**Monitoring self-hosted** (comme Prometheus/Grafana) :
```
Vous installez et gérez :
┌─────────────────────────────────────┐
│  Votre infrastructure               │
│  • Serveurs de monitoring           │
│  • Configuration manuelle           │
│  • Maintenance et mises à jour      │
│  • Sécurité à gérer                 │
│  • Scalabilité à planifier          │
└─────────────────────────────────────┘
```

**Monitoring cloud-native** :
```
Le cloud provider gère pour vous :
┌─────────────────────────────────────┐
│  Service managé                     │
│  • Aucun serveur à provisionner     │
│  • Intégration automatique          │
│  • Mises à jour transparentes       │
│  • Sécurité incluse                 │
│  • Scaling automatique              │
└─────────────────────────────────────┘
```

### Pourquoi utiliser les solutions cloud natives ?

**Avantages** :
- ✅ **Intégration native** : Métriques PostgreSQL automatiquement collectées  
- ✅ **Zéro infrastructure** : Pas de serveurs de monitoring à gérer  
- ✅ **Démarrage rapide** : Opérationnel en quelques clics  
- ✅ **Scalabilité automatique** : Gère des millions de métriques  
- ✅ **Sécurité intégrée** : IAM, encryption, audit logs  
- ✅ **Écosystème complet** : Logs, métriques, traces au même endroit

**Inconvénients** :
- ⚠️ **Coût** : Généralement plus cher qu'une solution self-hosted  
- ⚠️ **Vendor lock-in** : Dépendance au fournisseur cloud  
- ⚠️ **Moins de flexibilité** : Fonctionnalités pré-définies  
- ⚠️ **Données dans le cloud** : Pas de contrôle total sur l'hébergement

### Les trois grandes solutions

| Fournisseur | Service principal | Lancement | Force principale |
|-------------|-------------------|-----------|------------------|
| **AWS** | CloudWatch | 2009 | Écosystème AWS riche |
| **Azure** | Azure Monitor | 2014 | Intégration Microsoft |
| **Google Cloud** | Cloud Monitoring | 2014 | Analytics avancés |

### Métriques CloudWatch pour RDS PostgreSQL

#### Métriques standard (incluses gratuitement)

CloudWatch collecte automatiquement des métriques toutes les **1 minute** pour RDS :

**Métriques CPU et mémoire** :

| Métrique | Description | Unité | Seuil recommandé |
|----------|-------------|-------|------------------|
| `CPUUtilization` | Utilisation CPU | Pourcentage | < 80% |
| `FreeableMemory` | Mémoire disponible | Bytes | > 512MB |
| `SwapUsage` | Utilisation du swap | Bytes | 0 (idéalement) |

**Métriques stockage** :

| Métrique | Description | Unité | Seuil recommandé |
|----------|-------------|-------|------------------|
| `FreeStorageSpace` | Espace disque disponible | Bytes | > 20% |
| `ReadIOPS` | Opérations I/O lecture | Count/sec | Surveiller tendances |
| `WriteIOPS` | Opérations I/O écriture | Count/sec | Surveiller tendances |
| `ReadLatency` | Latence de lecture | Secondes | < 10ms |
| `WriteLatency` | Latence d'écriture | Secondes | < 10ms |
| `ReadThroughput` | Débit de lecture | Bytes/sec | - |
| `WriteThroughput` | Débit d'écriture | Bytes/sec | - |

**Métriques réseau** :

| Métrique | Description | Unité |
|----------|-------------|-------|
| `NetworkReceiveThroughput` | Trafic entrant | Bytes/sec |
| `NetworkTransmitThroughput` | Trafic sortant | Bytes/sec |

**Métriques PostgreSQL spécifiques** :

| Métrique | Description | Unité | Seuil recommandé |
|----------|-------------|-------|------------------|
| `DatabaseConnections` | Nombre de connexions | Count | < 80% de max_connections |
| `TransactionLogsDiskUsage` | Espace WAL utilisé | MB | < 50% de l'espace |
| `MaximumUsedTransactionIDs` | Transaction IDs utilisés | Count | Surveiller wraparound |
| `OldestReplicationSlotLag` | Retard du slot de réplication le plus ancien | MB | < 10GB |

#### Enhanced Monitoring (métriques avancées)

**Enhanced Monitoring** fournit des métriques au niveau système d'exploitation avec une granularité jusqu'à **1 seconde**.

**Métriques supplémentaires** :
- Utilisation CPU par processus
- Mémoire détaillée (active, inactive, buffers, cached)
- I/O par device
- Statistiques système (load average, tasks, swap)
- Liste des processus actifs

**Activation** :
```bash
# Via AWS CLI
aws rds modify-db-instance \
  --db-instance-identifier my-postgres-instance \
  --monitoring-interval 60 \
  --monitoring-role-arn arn:aws:iam::123456789012:role/rds-monitoring-role
```

**Coût** : Enhanced Monitoring est facturé séparément (~$2.50/mois par instance pour un intervalle de 60s).

### Performance Insights

**Performance Insights** est un outil AWS spécialement conçu pour l'analyse des performances des bases de données.

**Fonctionnalités** :
- 📊 **Dashboard temps réel** : Visualisation de la charge base de données  
- 🔍 **Top SQL queries** : Requêtes les plus consommatrices  
- ⏱️ **Wait events** : Analyse des événements d'attente  
- 📈 **Métriques historiques** : Rétention jusqu'à 2 ans (avec coût)  
- 🎯 **Analyse des dimensions** : Par utilisateur, host, base de données

**Métriques clés** :
- **DB Load** : Charge active moyenne de la base (Average Active Sessions)  
- **Top SQL** : Les requêtes triées par temps total, appels, latence  
- **Wait events** : CPU, I/O, locks, network

**Activation** :
```bash
aws rds modify-db-instance \
  --db-instance-identifier my-postgres-instance \
  --enable-performance-insights \
  --performance-insights-retention-period 7  # Jours (7 gratuit, jusqu'à 731 payant)
```

**Coût** :
- 7 jours de rétention : **Gratuit**
- Rétention longue (jusqu'à 2 ans) : ~$0.02/vCPU/heure

### CloudWatch Logs Insights

**CloudWatch Logs Insights** permet d'analyser les logs PostgreSQL avec un langage de requête puissant.

**Exemples de requêtes** :

**1. Requêtes lentes** :
```
fields @timestamp, @message
| filter @message like /duration: [0-9]{4,}/
| parse @message /duration: (?<duration>[0-9]+)/
| sort duration desc
| limit 20
```

**2. Erreurs les plus fréquentes** :
```
fields @timestamp, @message
| filter @message like /ERROR/
| stats count() by @message
| sort count desc
| limit 10
```

**3. Connexions par base** :
```
fields @timestamp, @message
| filter @message like /connection received/
| parse @message /database=(?<database>[^ ]+)/
| stats count() by database
```

### CloudWatch Alarms

**Créer une alarme** :

**Exemple 1 : CPU élevé**
```bash
aws cloudwatch put-metric-alarm \
  --alarm-name high-cpu-alarm \
  --alarm-description "CPU > 80% for 5 minutes" \
  --metric-name CPUUtilization \
  --namespace AWS/RDS \
  --statistic Average \
  --period 300 \
  --threshold 80 \
  --comparison-operator GreaterThanThreshold \
  --evaluation-periods 1 \
  --dimensions Name=DBInstanceIdentifier,Value=my-postgres-instance \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:my-sns-topic
```

**Exemple 2 : Espace disque faible**
```bash
aws cloudwatch put-metric-alarm \
  --alarm-name low-storage-alarm \
  --metric-name FreeStorageSpace \
  --namespace AWS/RDS \
  --statistic Average \
  --period 300 \
  --threshold 10737418240 \  # 10GB
  --comparison-operator LessThanThreshold \
  --evaluation-periods 1 \
  --dimensions Name=DBInstanceIdentifier,Value=my-postgres-instance
```

**Actions possibles** :
- 📧 **SNS Topic** : Envoyer email/SMS  
- 🔔 **Lambda** : Exécuter une fonction (ex: scale automatique)  
- 📱 **Systems Manager** : Exécuter un runbook  
- 🚨 **Auto Scaling** : Augmenter les ressources

### Diagnostic Settings (Logs)

**Activer les logs** :

Via le portail Azure :
1. Azure Database for PostgreSQL → Diagnostic settings  
2. Add diagnostic setting  
3. Sélectionner les logs à capturer :  
   - `PostgreSQLLogs` : Logs PostgreSQL standard  
   - `QueryStoreRuntimeStatistics` : Statistiques Query Store  
   - `QueryStoreWaitStatistics` : Wait events  
4. Destination : Log Analytics workspace, Storage account, ou Event Hub

**Via Azure CLI** :
```bash
az monitor diagnostic-settings create \
  --resource /subscriptions/{subscription-id}/resourceGroups/{rg}/providers/Microsoft.DBforPostgreSQL/servers/{server-name} \
  --name DiagnosticSetting1 \
  --logs '[
    {
      "category": "PostgreSQLLogs",
      "enabled": true
    },
    {
      "category": "QueryStoreRuntimeStatistics",
      "enabled": true
    }
  ]' \
  --workspace {log-analytics-workspace-id}
```

### Query Performance Insights

**Query Performance Insights** (similaire à AWS Performance Insights) analyse les performances des requêtes.

**Fonctionnalités** :
- 📊 **Top queries** : Requêtes les plus consommatrices  
- ⏱️ **Wait statistics** : Analyse des temps d'attente  
- 📈 **Métriques historiques** : Jusqu'à 7 jours (gratuit)  
- 🎯 **Drill-down** : Détails par requête

**Activation** :
1. Portail Azure → Azure Database for PostgreSQL  
2. Intelligent Performance → Query Performance Insight  
3. Activer Query Store si ce n'est pas déjà fait

**Configuration Query Store** :
```sql
-- Configuration dans PostgreSQL
ALTER DATABASE mydb SET pg_qs.query_capture_mode = 'TOP';  
ALTER DATABASE mydb SET pgms_wait_sampling.query_capture_mode = 'ALL';  
```

**Workbooks** sont des dashboards interactifs et personnalisables.

**Templates disponibles** :

**Créer un workbook personnalisé** :

**Exemple de requête KQL pour les connexions échouées** :
```kql
AzureDiagnostics
| where ResourceProvider == "MICROSOFT.DBFORPOSTGRESQL"
| where Category == "PostgreSQLLogs"
| where Message contains "connection"
| where Message contains "failed"
| summarize FailedConnections = count() by bin(TimeGenerated, 5m)
| render timechart
```

### Azure Alerts

**Créer une alerte sur métrique** :

Via le portail :
1. Azure Database for PostgreSQL → Alerts → New alert rule  
2. Sélectionner une condition (ex: CPU > 80%)  
3. Configurer la logique :
   - Seuil : 80
   - Agrégation : Average
   - Période : 5 minutes
4. Associer un Action Group (email, SMS, webhook, Logic App)

**Via Azure CLI** :
```bash
# Créer une règle d'alerte
az monitor metrics alert create \
  --name high-cpu-alert \
  --resource-group myResourceGroup \
  --scopes /subscriptions/{sub-id}/resourceGroups/{rg}/providers/Microsoft.DBforPostgreSQL/servers/{server} \
  --condition "avg cpu_percent > 80" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --action {action-group-id}
```

**Action Groups** :

Actions possibles :
- 📧 **Email/SMS** : Notification directe  
- 📱 **Azure mobile app** : Push notification  
- 🔔 **Webhook** : Appeler une API externe  
- ⚡ **Azure Function** : Exécuter du code custom  
- 🤖 **Logic App** : Workflow d'automatisation  
- 🎫 **ITSM** : Créer un ticket (ServiceNow, etc.)

### Azure Monitor Logs (KQL)

**Kusto Query Language (KQL)** est le langage de requête d'Azure Monitor.

**Exemples de requêtes** :

**1. Requêtes lentes** :
```kql
AzureDiagnostics
| where ResourceProvider == "MICROSOFT.DBFORPOSTGRESQL"
| where Category == "PostgreSQLLogs"
| where Message contains "duration:"
| parse Message with * "duration: " Duration:real " ms" *
| where Duration > 1000  // > 1 seconde
| project TimeGenerated, Duration, Message
| order by Duration desc
| take 20
```

**2. Taux d'erreurs par heure** :
```kql
AzureDiagnostics
| where ResourceProvider == "MICROSOFT.DBFORPOSTGRESQL"
| where Category == "PostgreSQLLogs"
| where Level == "ERROR"
| summarize ErrorCount = count() by bin(TimeGenerated, 1h)
| render columnchart
```

**3. Top bases de données par connexions** :
```kql
AzureMetrics
| where ResourceProvider == "MICROSOFT.DBFORPOSTGRESQL"
| where MetricName == "active_connections"
| summarize AvgConnections = avg(Average) by Resource
| order by AvgConnections desc
```

### MQL (Monitoring Query Language)

**MQL** est le langage de requête avancé de Google Cloud Monitoring, similaire à PromQL.

**Exemples de requêtes MQL** :

**1. Taux de connexions** :
```
fetch cloudsql_database
| metric 'cloudsql.googleapis.com/database/postgresql/num_backends'
| group_by 1m, [value_num_backends_mean: mean(value.num_backends)]
| every 1m
```

**2. Utilisation CPU avec seuil** :
```
fetch cloudsql_database
| metric 'cloudsql.googleapis.com/database/cpu/utilization'
| filter resource.database_id == 'my-project:my-instance'
| group_by 5m, [value_utilization_mean: mean(value.utilization)]
| condition value_utilization_mean > 0.80
```

**3. Comparaison avec semaine précédente** :
```
fetch cloudsql_database
| metric 'cloudsql.googleapis.com/database/postgresql/transaction_count'
| align rate(1m)
| group_by [resource.database_id], [value: sum(value)]
| outer_join 0,
  (fetch cloudsql_database
   | metric 'cloudsql.googleapis.com/database/postgresql/transaction_count'
   | align rate(1m)
   | group_by [resource.database_id], [value: sum(value)]
   | time_shift(7d))
```

## Bonnes pratiques

### 1. Configuration des logs

**Quoi logger** :

| Niveau | Événements | Environnement |
|--------|-----------|---------------|
| **Minimal** | Erreurs uniquement | Production stable |
| **Standard** | Erreurs + Connexions + Requêtes lentes (>1s) | Production typique |
| **Détaillé** | + Toutes les requêtes | Développement, Debug |
| **Verbeux** | Tout (attention au volume !) | Troubleshooting ponctuel |

**Configuration PostgreSQL recommandée** :
```sql
-- Pour RDS/Cloud SQL, via paramètres
log_connections = on  
log_disconnections = on  
log_min_duration_statement = 1000  -- 1 seconde  
log_line_prefix = '%t [%p]: [%l-1] user=%u,db=%d,app=%a,client=%h '  
log_statement = 'ddl'  -- Log des DDL uniquement  
```

### 2. Dimensionnement des alertes

**Principe** : Alerter sur ce qui nécessite une action, pas sur tout.

**Alertes critiques** (notification immédiate, 24/7) :
- Base de données inaccessible
- Espace disque < 10%
- CPU > 90% pendant 10+ minutes
- Réplication en échec

**Alertes warning** (notification pendant heures de travail) :
- CPU > 80% pendant 30 minutes
- Espace disque < 20%
- Cache hit ratio < 90%
- Connexions > 80% du max

**Alertes info** (pas de notification, dashboard seulement) :
- Tendances à surveiller
- Métriques de capacité

### 3. Organisation des dashboards

**Hiérarchie recommandée** :

```
📊 PostgreSQL (Dossier principal)
├── 🌐 Overview (Vue d'ensemble multi-instances)
├── 📈 Production
│   ├── Production-DB-1 (détaillé)
│   ├── Production-DB-2 (détaillé)
│   └── Production-Replicas
├── 🧪 Staging
│   └── Staging-DB
├── 🔧 Development
│   └── Dev-DBs
└── 🚨 Alerting Status (état des alertes)
```

**Un dashboard = Un objectif** :
- **Overview** : Santé globale, toutes instances  
- **Performance** : CPU, I/O, cache, requêtes lentes  
- **Capacity** : Stockage, connexions, croissance  
- **Replication** : Lag, état des replicas  
- **Troubleshooting** : Métriques détaillées pour debug

### 4. Rétention des données

**Stratégie 3-2-1 adaptée au monitoring** :

```
3 copies de vos métriques critiques
2 types de stockage différents
1 copie hors du cloud principal
```

**Exemple** :
```
• Métriques en temps réel : 7 jours (haute résolution)
• Métriques agrégées : 90 jours (résolution 5min)
• Métriques archivées : 1 an (résolution 1h)
• Export mensuel : Backup hors ligne
```

**Configuration** :

AWS CloudWatch :
```bash
# Métriques standard: 15 mois automatique
# Logs: Configurer la rétention
aws logs put-retention-policy \
  --log-group-name /aws/rds/instance/my-postgres/postgresql \
  --retention-in-days 90
```

Azure Monitor :
```bash
# Log Analytics workspace retention
az monitor log-analytics workspace update \
  --resource-group myResourceGroup \
  --workspace-name myWorkspace \
  --retention-time 90
```

GCP :
```bash
# Cloud Logging retention
gcloud logging sinks create my-export-sink \
  storage.googleapis.com/my-log-bucket \
  --log-filter='resource.type="cloudsql_database"'
```

### 5. Coûts et optimisation

**Stratégies pour réduire les coûts** :

1. **Filtrer les logs inutiles** :
   ```
   # Ne pas logger les requêtes SELECT simples
   log_min_duration_statement = 1000
   log_statement = 'ddl'  # Pas 'all'
   ```

2. **Agréger les métriques** :
   - Haute résolution (1min) : 7 jours seulement
   - Résolution normale (5min) : 30 jours
   - Agrégations (1h) : 1 an

3. **Archiver plutôt que stocker** :
   - Exporter vers S3/Blob Storage (beaucoup moins cher)
   - Compression des logs

4. **Limiter les dashboards payants** (AWS) :
   - Consolider plusieurs métriques dans un dashboard
   - Partager les dashboards entre équipes

   ```
   Permettre l'extension à 7 jours si nécessaire
   ```

3. **Utiliser des variables** :
   ```
   Filtrer par instance_id, database_name
   → Réduire le volume de données
   ```

## Conclusion

### Récapitulatif

Les **solutions cloud natives** (CloudWatch, Azure Monitor, GCP Cloud Monitoring) offrent un monitoring **puissant et intégré** pour PostgreSQL :

- ✅ **Zéro infrastructure** : Pas de serveurs à gérer  
- ✅ **Intégration native** : Métriques automatiques  
- ✅ **Scalabilité** : Gère des millions de métriques  
- ✅ **Sécurité** : IAM, chiffrement, audit  
- ✅ **Écosystème complet** : Métriques + Logs + Traces

**Points de vigilance** :
- ⚠️ **Coût** : Peut devenir élevé avec beaucoup de logs  
- ⚠️ **Vendor lock-in** : Dépendance au cloud provider  
- ⚠️ **Flexibilité limitée** : Moins personnalisable que Prometheus/Grafana

### Choix de solution

**Matrice de décision** :

| Situation | Solution recommandée |
|-----------|---------------------|
| **PostgreSQL sur AWS RDS/Aurora** | AWS CloudWatch + Performance Insights |
| **PostgreSQL sur Azure** | Azure Monitor + Query Performance Insights |
| **PostgreSQL sur GCP** | Cloud Monitoring (meilleur rapport qualité/prix) |
| **Multi-cloud** | Grafana centralisé + exporters |
| **Hybride (on-prem + cloud)** | Prometheus/Grafana ou Datadog |
| **Budget très limité** | GCP (tier gratuit généreux) ou Prometheus self-hosted |
| **Besoin APM complet** | Datadog, New Relic (payant) |

| **Cloud-native** | Conçu spécifiquement pour fonctionner dans le cloud |

| **Query Insights** | Outil d'analyse de performances requêtes (GCP) |
| **Workbook** | Dashboard interactif (Azure Monitor) |
| **CloudWatch Logs Insights** | Outil d'analyse de logs (AWS) |
| **Log Analytics** | Service d'analyse de logs (Azure) |
