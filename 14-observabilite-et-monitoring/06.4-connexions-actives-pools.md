# 14.6.4. Connexions actives et pools saturés

## Introduction

La gestion des **connexions** est l'un des aspects les plus critiques mais souvent négligés d'une base de données PostgreSQL en production. Une mauvaise gestion des connexions peut transformer une base de données performante en système saturé et inutilisable, même avec un excellent matériel.

Dans ce chapitre, nous allons comprendre comment PostgreSQL gère les connexions, pourquoi il existe des limites, et surtout comment utiliser le **connection pooling** pour éviter les problèmes de saturation.

---

## Comprendre les Connexions PostgreSQL

### Analogie : Le Restaurant

Imaginez un restaurant avec 100 places assises :

**Scénario 1 : Restaurant bien géré**
- 80 clients présents
- Service fluide
- Tout le monde est servi rapidement

**Scénario 2 : Restaurant surchargé**
- 150 clients se présentent en même temps
- Pas assez de places
- Les serveurs sont débordés
- 50 clients attendent dehors (frustration)
- Le service ralentit pour tout le monde

C'est exactement ce qui se passe quand votre PostgreSQL reçoit trop de connexions simultanées !

### Architecture Client-Serveur de PostgreSQL

#### Le Modèle Process-Per-Connection

PostgreSQL utilise un modèle **un processus par connexion** :

```
Application 1 ──→ Connexion 1 ──→ Backend Process 1  
Application 2 ──→ Connexion 2 ──→ Backend Process 2  
Application 3 ──→ Connexion 3 ──→ Backend Process 3  
     ...              ...              ...
Application N ──→ Connexion N ──→ Backend Process N
```

**Particularité importante :** Chaque connexion crée un **processus système complet** sur le serveur.

### Coût d'une Connexion

#### Ressources Consommées par Connexion

Chaque connexion PostgreSQL consomme :

1. **Mémoire** : 5-10 MB minimum par connexion
   - work_mem (mémoire de travail)
   - temp_buffers (buffers temporaires)
   - Structures internes

2. **Processus Système** : 1 processus Linux/Unix complet
   - PID (Process ID)
   - Context switching overhead
   - Descripteurs de fichiers

3. **Ressources Kernel** :
   - Sockets réseau
   - Tables de routage

**Calcul simple :**
```
100 connexions × 10 MB/connexion = 1 GB de RAM minimum
500 connexions × 10 MB/connexion = 5 GB de RAM minimum
```

**Et cela n'inclut même pas les données réelles traitées !**

#### Problèmes avec Trop de Connexions

##### 1. Saturation Mémoire

```
Serveur avec 32 GB RAM  
shared_buffers = 8 GB  
1000 connexions × 10 MB = 10 GB
────────────────────────────────
18 GB utilisés juste pour les connexions !
```

**Résultat :** Mémoire restante insuffisante → Swap → Performances catastrophiques

##### 2. Context Switching Excessif

Le CPU passe son temps à basculer entre les processus au lieu d'exécuter réellement du travail.

**Analogie :** Un serveur de restaurant qui court entre 100 tables sans jamais prendre de commande complète.

##### 3. Contention sur les Ressources Partagées

- Lock tables saturées
- Buffer cache thrashing
- I/O contention

## Limites de Connexions dans PostgreSQL

### Le Paramètre max_connections

**max_connections** définit le nombre maximum de connexions simultanées acceptées par PostgreSQL.

**Vérifier la valeur actuelle :**
```sql
SHOW max_connections;
-- Résultat typique : 100
```

**Valeur par défaut :** 100 connexions

### Pourquoi Cette Limite Existe ?

C'est une **protection** contre la surcharge du serveur.

**Exemple concret :**

```
max_connections = 100

Connexion 1-100 : ✅ Acceptée  
Connexion 101   : ❌ FATAL: sorry, too many clients already  
```

Le message d'erreur complet :
```
FATAL: sorry, too many clients already  
HINT: Increase "max_connections" and "shared_memory"  
```

### Connexions Réservées

PostgreSQL réserve des connexions pour les superutilisateurs :

**superuser_reserved_connections** : 3 par défaut

**Calcul :**
```
Connexions disponibles pour utilisateurs normaux = max_connections - superuser_reserved_connections
                                                  = 100 - 3
                                                  = 97
```

**Pourquoi ?** Permet à un admin de se connecter même quand le système est saturé pour investiguer.

### Modifier max_connections

**⚠️ Attention :** Augmenter max_connections n'est PAS toujours la solution !

```ini
# Dans postgresql.conf
max_connections = 200

# Redémarrage nécessaire
sudo systemctl restart postgresql
```

**Conséquences :**
- ✅ Plus de connexions acceptées  
- ❌ Plus de mémoire consommée  
- ❌ Plus de context switching  
- ❌ Performances possiblement dégradées

**Règle d'or :** Ne pas dépasser 200-300 connexions sans connection pooling.

## Le Concept de Connection Pooling

### Qu'est-ce que le Connection Pooling ?

Le **connection pooling** est une technique qui réutilise un nombre limité de connexions PostgreSQL pour servir un grand nombre de clients.

### Analogie : Le Central Téléphonique

**Sans pooling (ancien système) :**
- Chaque personne a sa propre ligne téléphonique dédiée
- 1000 employés = 1000 lignes physiques nécessaires
- Coûteux et inefficace

**Avec pooling (central moderne) :**
- 50 lignes partagées par 1000 employés
- Le central route intelligemment les appels
- Beaucoup moins cher et tout aussi efficace
- Car tout le monde n'appelle pas en même temps

### Principe du Pooling pour PostgreSQL

```
5000 clients application ──┐
                           │
                           ├──→ Connection Pool (100 connexions)
                           │
                           └──→ PostgreSQL (100 connexions actives)
```

**Magie :** 5000 clients peuvent travailler avec seulement 100 connexions PostgreSQL réelles !

**Comment ?** La plupart des clients passent 90% de leur temps à ne PAS utiliser activement la connexion.

### Avantages du Connection Pooling

#### 1. Réduction de la Charge Serveur

```
Sans pooling :
1000 clients → 1000 connexions PostgreSQL → Serveur saturé

Avec pooling :
1000 clients → Pool (50 connexions) → PostgreSQL léger
```

#### 2. Gestion Intelligente

- Réutilisation des connexions (pas de création/destruction constante)
- File d'attente quand le pool est plein
- Timeout automatique des connexions inactives

#### 3. Meilleure Performance Globale

- Moins de context switching
- Moins de mémoire consommée
- Plus de stabilité

#### 4. Protection Contre les Connection Storms

Lors d'un redémarrage d'application, toutes les instances essaient de se connecter simultanément.

**Sans pooling :**
```
Redémarrage app → 500 connexions en 2 secondes → PostgreSQL saturé → Crash
```

**Avec pooling :**
```
Redémarrage app → Pool gère progressivement → PostgreSQL stable
```

## PgBouncer : Le Pool de Connexions de Référence

### Présentation

**PgBouncer** est le connection pooler le plus populaire pour PostgreSQL.

**Caractéristiques :**
- Léger (écrit en C)
- Ultra-rapide
- Battle-tested (utilisé par les plus gros sites)
- Open source

### Architecture

```
Applications (1000 clients)
         ↓
   PgBouncer (middleware)
    Pool : 50 connexions
         ↓
   PostgreSQL (50 connexions)
```

### Les Modes de Pooling

PgBouncer propose 3 modes de pooling :

#### 1. Session Pooling (Mode Session)

**Principe :** Une connexion PostgreSQL est attribuée à un client pour toute la durée de sa session.

```ini
pool_mode = session
```

**Comportement :**
- Client se connecte → Récupère une connexion du pool
- Client travaille (peut faire plusieurs transactions)
- Client se déconnecte → Connexion retourne au pool

**Avantages :**
- ✅ Compatible avec tout PostgreSQL (prepared statements, LISTEN/NOTIFY, etc.)  
- ✅ Comportement transparent

**Inconvénients :**
- ❌ Moins efficace (un client monopolise une connexion même inactif)

**Usage :** Applications avec longues sessions interactives.

#### 2. Transaction Pooling (Mode Transaction)

**Principe :** Une connexion PostgreSQL est attribuée seulement pendant une transaction.

```ini
pool_mode = transaction  # Recommandé
```

**Comportement :**
- Client commence transaction → Récupère connexion
- Transaction commit/rollback → Connexion retourne immédiatement au pool
- Client peut faire plusieurs transactions successives

**Avantages :**
- ✅ Très efficace (réutilisation maximale)  
- ✅ Recommandé pour la plupart des applications

**Inconvénients :**
- ❌ Incompatible avec prepared statements nommés  
- ❌ Incompatible avec LISTEN/NOTIFY  
- ❌ Pas de variables de session temporaires

**Usage :** Applications web standards (REST APIs, sites web).

#### 3. Statement Pooling (Mode Statement)

**Principe :** Une connexion est retournée au pool après chaque requête SQL.

```ini
pool_mode = statement  # Rarement utilisé
```

**Comportement :**
- Plus agressif que transaction
- Même les transactions multi-statements sont interrompues

**Avantages :**
- ✅ Réutilisation maximale absolue

**Inconvénients :**
- ❌ Casse les transactions  
- ❌ Très peu d'applications compatibles

**Usage :** Très rare, seulement pour requêtes READ-ONLY ultra-simples.

### Configuration de PgBouncer

#### Installation

**Ubuntu/Debian :**
```bash
sudo apt install pgbouncer
```

**RHEL/CentOS :**
```bash
sudo yum install pgbouncer
```

#### Configuration de Base

**Fichier `/etc/pgbouncer/pgbouncer.ini` :**

```ini
[databases]
myapp = host=localhost port=5432 dbname=myapp_production

[pgbouncer]
# Écoute sur toutes les interfaces, port 6432
listen_addr = *  
listen_port = 6432  

# Mode pooling (transaction recommandé)
pool_mode = transaction

# Taille du pool
default_pool_size = 25          # Connexions PostgreSQL par base  
max_client_conn = 500           # Clients maximum acceptés  

# Connexions de réserve
reserve_pool_size = 5           # Connexions en réserve  
reserve_pool_timeout = 3        # Attente avant utiliser réserve (sec)  

# Timeouts
server_idle_timeout = 600       # Fermer connexion inactive 10 min  
query_timeout = 60              # Timeout requête (60 sec)  

# Logs
log_connections = 1  
log_disconnections = 1  
log_pooler_errors = 1  

# Admin
admin_users = postgres  
stats_users = monitoring_user  
```

Authentification des utilisateurs :

```

```

**Générer le hash MD5 :**
```bash

```

#### Démarrage

```bash
sudo systemctl start pgbouncer  
sudo systemctl enable pgbouncer  
sudo systemctl status pgbouncer  
```

### Connexion via PgBouncer

**Sans PgBouncer (direct à PostgreSQL) :**
```bash
psql -h localhost -p 5432 -U appuser -d myapp_production
```

**Avec PgBouncer :**
```bash
psql -h localhost -p 6432 -U appuser -d myapp_production
# Port 6432 au lieu de 5432 !
```

**Dans votre application (exemple Python) :**
```python
import psycopg2

# Avant (direct)
conn = psycopg2.connect(
    host="localhost",
    port=5432,
    database="myapp_production",
    user="appuser",
    password="secret"
)

# Après (via PgBouncer)
conn = psycopg2.connect(
    host="localhost",
    port=6432,  # Port PgBouncer
    database="myapp_production",
    user="appuser",
    password="secret"
)
```

**Transparent pour l'application !**

## Dimensionner le Pool

### Formules de Calcul

#### Pool Size (Taille du Pool)

**Règle générale :**

```
pool_size = (2 × Nombre de CPU cores) + Nombre de disques effectifs
```

**Exemple :**
- Serveur avec 8 cores
- 2 disques (RAID 1)

```
pool_size = (2 × 8) + 2 = 18 connexions
```

**Recommandations par type d'application :**

| Type | Pool Size Recommandé |
|------|---------------------|
| **Web app OLTP** | 20-30 |
| **API REST** | 15-25 |
| **Mixed (OLTP + analytique)** | 30-50 |
| **Serveur dédié analytique** | 50-100 |

#### Max Client Connections

**Règle :**
```
max_client_conn = pool_size × 10 (minimum)
                = pool_size × 20 (confortable)
```

**Exemple avec pool_size = 25 :**
```
max_client_conn = 25 × 20 = 500 clients
```

## Détecter les Problèmes de Connexions

### Au Niveau PostgreSQL

#### 1. Voir les Connexions Actives

```sql
SELECT
    pid,
    usename,
    application_name,
    client_addr,
    state,
    backend_start,
    state_change,
    query
FROM pg_stat_activity  
WHERE state != 'idle'  
ORDER BY backend_start;  
```

**Colonnes importantes :**
- **pid** : Process ID  
- **state** : État (active, idle, idle in transaction)  
- **backend_start** : Moment de connexion  
- **query** : Requête en cours

#### 2. Compter les Connexions par État

```sql
SELECT
    state,
    count(*) AS connection_count
FROM pg_stat_activity  
GROUP BY state  
ORDER BY connection_count DESC;  
```

**Exemple de sortie :**
```
       state        | connection_count
--------------------+-----------------
 idle               |               45
 active             |               12
 idle in transaction|                3
```

**⚠️ Alerte si :**
- `idle in transaction` > 10 : Transactions oubliées (connection leaks)

#### 3. Connexions les Plus Anciennes

```sql
SELECT
    pid,
    usename,
    application_name,
    state,
    now() - backend_start AS connection_age,
    now() - state_change AS state_age
FROM pg_stat_activity  
WHERE state != 'idle'  
ORDER BY backend_start  
LIMIT 20;  
```

**⚠️ Alerte si :**
- Connexion ouverte depuis plusieurs heures en `idle in transaction`

#### 4. Utilisation vs Limite

```sql
SELECT
    (SELECT count(*) FROM pg_stat_activity) AS current_connections,
    (SELECT setting::int FROM pg_settings WHERE name = 'max_connections') AS max_connections,
    round(
        100.0 * (SELECT count(*) FROM pg_stat_activity) /
        (SELECT setting::int FROM pg_settings WHERE name = 'max_connections'),
        2
    ) AS usage_percent;
```

**Seuils :**
- < 70% : 🟢 OK
- 70-85% : 🟡 Surveillé
- 85-95% : 🟠 Attention
- > 95% : 🔴 Critique (proche de la saturation)

### Symptômes de Saturation

#### Côté Application

**Erreurs visibles :**
```
FATAL: sorry, too many clients already
```

**Ou :**
```
Connection timeout  
Could not connect to database  
```

**Comportement :**
- Timeouts fréquents
- Erreurs intermittentes
- Performances en dents de scie

#### Côté Serveur

**Commande système :**
```bash
ps aux | grep postgres | wc -l
# Si proche de max_connections → Problème
```

**Logs PostgreSQL :**
```

```

**Chercher :**
```bash

```

## Connection Leaks (Fuites de Connexions)

### Qu'est-ce qu'un Connection Leak ?

Un **connection leak** se produit quand une application ouvre une connexion mais ne la ferme jamais.

**Analogie :** Ouvrir un robinet et partir sans le fermer.

### Causes Courantes

#### 1. Exceptions Non Gérées

**Code problématique (Python) :**
```python
# ❌ MAUVAIS
def mauvaise_fonction():
    conn = psycopg2.connect(...)
    cursor = conn.cursor()
    cursor.execute("SELECT * FROM users")
    # Si une exception se produit ici
    # la connexion n'est jamais fermée !
    data = cursor.fetchall()
    conn.close()  # Cette ligne peut ne jamais s'exécuter
```

**Code correct :**
```python
# ✅ BON
def bonne_fonction():
    conn = None
    try:
        conn = psycopg2.connect(...)
        cursor = conn.cursor()
        cursor.execute("SELECT * FROM users")
        data = cursor.fetchall()
        return data
    finally:
        if conn:
            conn.close()  # TOUJOURS exécuté
```

**Encore mieux (Python) :**
```python
# ✅ MEILLEUR - Context manager
def meilleure_fonction():
    with psycopg2.connect(...) as conn:
        with conn.cursor() as cursor:
            cursor.execute("SELECT * FROM users")
            return cursor.fetchall()
    # Fermeture automatique garantie
```

#### 2. Transactions Oubliées

**Code problématique :**
```python
# ❌ MAUVAIS
conn = psycopg2.connect(...)  
conn.execute("BEGIN")  
conn.execute("UPDATE users SET ...")  
# Oubli de COMMIT ou ROLLBACK
# Connexion reste en "idle in transaction"
```

**Code correct :**
```python
# ✅ BON
conn = psycopg2.connect(...)  
try:  
    conn.execute("BEGIN")
    conn.execute("UPDATE users SET ...")
    conn.execute("COMMIT")
except:
    conn.execute("ROLLBACK")
    raise
finally:
    conn.close()
```

#### 3. Pooling Applicatif Mal Géré

Certains frameworks ont leur propre pooling. Si mal configuré :

```python
# Configuration ORM (exemple)
# ❌ MAUVAIS
pool_size = 100      # Trop grand  
max_overflow = 100   # Trop grand  
```

### Détecter les Leaks

#### 1. Connexions "idle in transaction" Prolongées

```sql
SELECT
    pid,
    usename,
    application_name,
    client_addr,
    now() - state_change AS idle_time,
    query
FROM pg_stat_activity  
WHERE state = 'idle in transaction'  
    AND now() - state_change > interval '5 minutes'
ORDER BY idle_time DESC;
```

**Action :** Identifier l'application et corriger le code.

#### 2. Croissance Continue des Connexions

**Monitoring sur 1 heure :**
```bash
# Script de monitoring
while true; do
    echo "$(date): $(psql -t -c "SELECT count(*) FROM pg_stat_activity")"
    sleep 60
done
```

**Si le nombre augmente constamment :** Connection leak.

### Tuer des Connexions Bloquées

**⚠️ À utiliser avec prudence !**

#### Tuer une Connexion Spécifique

```sql
-- 1. Identifier le PID
SELECT pid, usename, state, query  
FROM pg_stat_activity  
WHERE state = 'idle in transaction'  
    AND now() - state_change > interval '30 minutes';

-- 2. Terminer gentiment
SELECT pg_terminate_backend(12345);  -- Remplacer 12345 par le PID

-- 3. Si ça ne fonctionne pas, forcer
SELECT pg_cancel_backend(12345);
```

#### Tuer Toutes les Connexions Idle

```sql
-- ⚠️ DANGEREUX - À utiliser en dernier recours
SELECT pg_terminate_backend(pid)  
FROM pg_stat_activity  
WHERE state = 'idle in transaction'  
    AND now() - state_change > interval '1 hour'
    AND pid != pg_backend_pid();  -- Ne pas se tuer soi-même !
```

## Stratégies de Dimensionnement

### Approche par Type d'Application

#### Application Web (REST API)

**Caractéristiques :**
- Requêtes courtes (< 100ms)
- Trafic en rafales
- Nombreux clients simultanés

**Configuration recommandée :**

**PostgreSQL :**
```ini
max_connections = 100
```

**PgBouncer :**
```ini
pool_mode = transaction  
default_pool_size = 25  
max_client_conn = 1000  
```

**Ratio :** 1 connexion PostgreSQL peut servir 40 clients.

#### Application Analytics/BI

**Caractéristiques :**
- Requêtes longues (plusieurs secondes)
- Peu de clients simultanés
- Grosses lectures

**Configuration recommandée :**

**PostgreSQL :**
```ini
max_connections = 50
```

**PgBouncer :**
```ini
pool_mode = session  # Requêtes longues  
default_pool_size = 40  
max_client_conn = 100  
```

#### Application Mixte (OLTP + Analytics)

**Solution :** Séparer les pools !

**PgBouncer :**
```ini
[databases]
myapp_oltp = host=localhost port=5432 dbname=myapp pool_size=25  
myapp_analytics = host=localhost port=5432 dbname=myapp pool_size=15  

[pgbouncer]
pool_mode = transaction  # Par défaut pour OLTP
```

**Usage :**
- Applications web → Connecter à `myapp_oltp`
- Rapports/Analytics → Connecter à `myapp_analytics`

### Formule de Calcul Complète

```
Total connexions nécessaires =
    (Nombre d'app servers × Connexions par server) +
    (Nombre de workers background) +
    (Connexions outils de monitoring) +
    (Marge de sécurité 20%)

Exemple :
    (10 app servers × 20 connexions) +
    (5 workers) +
    (3 monitoring) +
    (20% de 208) ≈ 250 connexions max app

Avec pooling :
    Pool PgBouncer = 30-50 connexions PostgreSQL
    max_client_conn = 500-1000
```

## Alternatives et Outils Complémentaires

### 1. PgPool-II

Alternative à PgBouncer avec fonctionnalités additionnelles :
- Connection pooling
- Load balancing
- Réplication management

**Quand utiliser :**
- Besoin de load balancing automatique
- Architectures HA complexes

**Inconvénient :** Plus complexe à configurer que PgBouncer.

### 2. Odyssey (Yandex)

Pooler moderne développé par Yandex.

**Avantages :**
- Multithreadé (meilleures perfs que PgBouncer sur multi-core)
- Configuration dynamique

**Inconvénient :** Moins mature que PgBouncer.

### 3. Connection Pooling Intégré (AWS RDS Proxy, etc.)

Les clouds providers proposent des solutions intégrées :

**AWS RDS Proxy :**
- Pooling géré
- Failover automatique
- Scaling transparent

**Avantages :**
- Zéro configuration
- Maintenance gérée

**Inconvénients :**
- Coût supplémentaire
- Moins de contrôle

### 4. Application-Level Pooling

De nombreux frameworks ont leur propre pooling :

**Python (SQLAlchemy) :**
```python
from sqlalchemy import create_engine

engine = create_engine(
    'postgresql://user:pass@localhost/db',
    pool_size=10,          # Taille du pool
    max_overflow=20,       # Connexions supplémentaires possibles
    pool_timeout=30,       # Timeout si pool plein
    pool_recycle=3600      # Recycler connexions après 1h
)
```

**Java (HikariCP) :**
```java
HikariConfig config = new HikariConfig();  
config.setJdbcUrl("jdbc:postgresql://localhost:5432/mydb");  
config.setUsername("user");  
config.setPassword("password");  
config.setMaximumPoolSize(20);  
config.setMinimumIdle(5);  

HikariDataSource ds = new HikariDataSource(config);
```

**Node.js (node-postgres) :**
```javascript
const { Pool } = require('pg');

const pool = new Pool({
    host: 'localhost',
    port: 5432,
    database: 'mydb',
    user: 'user',
    password: 'password',
    max: 20,              // Taille max du pool
    idleTimeoutMillis: 30000,
    connectionTimeoutMillis: 2000,
});
```

**⚠️ Important :**
- Ne pas combiner pooling applicatif ET PgBouncer sans réflexion
- Risque de sur-pooling : 10 app servers × 20 connexions = 200 connexions réelles

**Recommandation :**
- **Petite app (1 serveur)** : Pooling applicatif suffit  
- **Moyenne app (2-5 serveurs)** : Pooling applicatif + PgBouncer  
- **Grande app (10+ serveurs)** : PgBouncer obligatoire

## Monitoring en Production

### Métriques Essentielles

#### 1. Nombre de Connexions Actives

**Requête PostgreSQL :**
```sql
SELECT count(*) as active_connections  
FROM pg_stat_activity  
WHERE state = 'active';  
```

**Alerte :** Si > 80% de max_connections.

#### 2. Connexions Idle in Transaction

```sql
SELECT count(*) as idle_in_tx  
FROM pg_stat_activity  
WHERE state = 'idle in transaction';  
```

**Alerte :** Si > 10.

#### 3. État du Pool PgBouncer

```sql
-- Via console admin PgBouncer
SHOW POOLS;
```

**Alerte :** Si `cl_waiting > 0`.

#### 4. Max Wait Time

**PgBouncer :**
```sql
SELECT database, user, maxwait  
FROM SHOW POOLS  
WHERE maxwait > 5;  
```

**Alerte :** Si `maxwait > 5 secondes`.

### Dashboard Grafana Recommandé

**Panels à créer :**

1. **Active Connections** (Gauge)
   - Valeur actuelle vs max_connections
   - Seuil warning à 70%, critical à 85%

2. **Connections by State** (Pie chart)
   - active, idle, idle in transaction

3. **Connection Age Distribution** (Heatmap)
   - Durée de vie des connexions

4. **PgBouncer Pool Status** (Time series)
   - cl_active, cl_waiting, sv_active, sv_idle

5. **Connection Rate** (Time series)
   - Nouvelles connexions / minute

6. **Top Applications by Connections** (Table)
   - Group by application_name

### Alertes Recommandées

#### Alerte 1 : Saturation Proche

```sql
-- Prometheus QL
(pg_stat_activity_count / pg_settings_max_connections) > 0.85
```

**Action :** Vérifier origine des connexions, envisager scaling.

#### Alerte 2 : Connection Leaks

```sql
pg_stat_activity_idle_in_transaction_count > 10
```

**Action :** Investiguer applications, corriger code.

#### Alerte 3 : PgBouncer Pool Saturé

```
pgbouncer_pools_cl_waiting > 0
```

**Action :** Augmenter pool_size ou investiguer requêtes lentes.

#### Alerte 4 : Connexions Anciennes

```sql
SELECT count(*)  
FROM pg_stat_activity  
WHERE state = 'active'  
    AND now() - query_start > interval '5 minutes';
```

**Action :** Identifier et optimiser requêtes lentes.

## Troubleshooting : Scénarios Courants

### Scénario 1 : "Too many clients already"

**Symptômes :**
- Erreur `FATAL: sorry, too many clients already`
- Application ne peut plus se connecter

**Diagnostic :**

```sql
-- Vérifier utilisation
SELECT count(*) as current,
       (SELECT setting::int FROM pg_settings WHERE name = 'max_connections') as max
FROM pg_stat_activity;
```

**Causes possibles :**
1. Pas de connection pooling  
2. Connection leaks dans le code  
3. max_connections trop bas

**Solutions :**

**Court terme :**
```sql
-- Tuer connexions idle longues
SELECT pg_terminate_backend(pid)  
FROM pg_stat_activity  
WHERE state = 'idle'  
    AND now() - state_change > interval '30 minutes';
```

**Moyen terme :**
- Implémenter PgBouncer

**Long terme :**
- Corriger connection leaks dans le code
- Dimensionner correctement

### Scénario 2 : Performances Dégradées avec Peu de Connexions

**Symptômes :**
- Seulement 30 connexions actives
- Mais système très lent

**Diagnostic :**

```sql
-- Identifier requêtes longues
SELECT pid, now() - query_start as duration, query  
FROM pg_stat_activity  
WHERE state = 'active'  
    AND now() - query_start > interval '10 seconds'
ORDER BY duration DESC;
```

**Cause :** Requêtes lentes bloquant les connexions.

**Solution :**
- Optimiser les requêtes (voir section 13 Indexation)
- Augmenter le timeout
- Scaler les connexions disponibles

### Scénario 3 : PgBouncer avec Clients en Attente

**Symptômes :**
- `cl_waiting > 0` dans SHOW POOLS
- Timeouts côté application

**Diagnostic :**

```sql
-- Console PgBouncer
SHOW POOLS;
```

**Causes possibles :**
1. Pool trop petit  
2. Requêtes qui prennent trop de temps  
3. Mode pooling inadapté

**Solutions :**

```ini
# Augmenter pool
default_pool_size = 50  # Au lieu de 25

# Augmenter réserve
reserve_pool_size = 10

# Augmenter timeout si requêtes légitimement longues
query_timeout = 120
```

### Scénario 4 : Connexions Zombie

**Symptômes :**
- Connexions `idle in transaction` depuis des heures
- Bloat qui augmente (VACUUM bloqué)

**Diagnostic :**

```sql
SELECT pid, usename, application_name, client_addr,
       now() - state_change as idle_duration, query
FROM pg_stat_activity  
WHERE state = 'idle in transaction'  
ORDER BY state_change;  
```

**Cause :** Application ne commit/rollback pas les transactions.

**Solution :**

**Immédiat :**
```sql
-- Tuer ces connexions
SELECT pg_terminate_backend(pid)  
FROM pg_stat_activity  
WHERE state = 'idle in transaction'  
    AND now() - state_change > interval '1 hour';
```

**Prévention :**
```ini
# postgresql.conf
idle_in_transaction_session_timeout = 300000  # 5 minutes en ms
```

## Bonnes Pratiques

### 1. Toujours Utiliser Connection Pooling en Production

**Sans exception**, dès que vous avez plus d'un serveur d'application.

### 2. Dimensionner Correctement

**Règles :**
- Ne jamais dépasser 200 connexions PostgreSQL sans très bonne raison
- Pool PgBouncer : 2-4× le nombre de cores CPU
- Prévoir 20% de marge

### 3. Gérer Proprement les Connexions dans le Code

```python
# ✅ BON - Toujours utiliser context managers
with get_db_connection() as conn:
    with conn.cursor() as cursor:
        cursor.execute(...)
# Fermeture automatique garantie
```

### 4. Monitorer en Continu

**Métriques critiques :**
- Nombre de connexions actives
- Connexions idle in transaction
- État du pool PgBouncer
- Temps d'attente (maxwait)

### 5. Configurer des Timeouts

**PostgreSQL :**
```ini
statement_timeout = 30s               # Timeout requête  
idle_in_transaction_session_timeout = 5min  # Timeout transaction idle  
```

**PgBouncer :**
```ini
query_timeout = 60        # 60 secondes max par requête  
server_idle_timeout = 600 # 10 minutes  
```

### 6. Séparer les Workloads

**Utilisez des pools différents pour :**
- Requêtes OLTP (rapides)
- Requêtes Analytics (lentes)
- Batch jobs
- Monitoring

### 7. Tester sous Charge

**Avant la production :**
- Load testing avec JMeter, Gatling, etc.
- Simuler connection storms
- Vérifier comportement en saturation

## Questions Fréquentes (FAQ)

### Q1 : Combien de connexions max_connections dois-je configurer ?

**Réponse :**
- **Avec PgBouncer** : 100-200 (rarement plus)  
- **Sans pooling** : 2× le nombre d'app servers × connexions par serveur

### Q2 : PgBouncer ou pooling applicatif ?

**Réponse :**
- **< 50 connexions totales** : Pooling applicatif suffit  
- **> 50 connexions** : PgBouncer recommandé  
- **> 100 connexions** : PgBouncer obligatoire

### Q3 : Quel mode PgBouncer choisir ?

**Réponse :**
- **Transaction** : 90% des cas (recommandé)  
- **Session** : Si vous utilisez prepared statements ou LISTEN/NOTIFY  
- **Statement** : Très rare

### Q4 : Connection leaks : comment les éviter ?

**Réponse :**
- Toujours utiliser try/finally ou context managers
- Configurer `idle_in_transaction_session_timeout`
- Monitorer les connexions idle in transaction

### Q5 : Puis-je augmenter max_connections à 1000 ?

**Réponse :**  
**Techniquement oui, mais ne le faites pas !**  
- Chaque connexion = 10 MB RAM
- 1000 connexions = 10 GB rien que pour les connexions
- Context switching catastrophique
- Utilisez PgBouncer à la place

### Q6 : Mon pool PgBouncer est toujours plein, que faire ?

**Réponses possibles :**
1. Augmenter pool_size (jusqu'à 2-4× CPU cores)  
2. Optimiser les requêtes lentes  
3. Vérifier pas de connexions bloquées  
4. Scaler horizontalement (plus de serveurs PostgreSQL)

## Points Clés à Retenir

✅ **PostgreSQL = 1 processus par connexion**
   - Coûteux en ressources (mémoire, CPU)
   - Limitation par max_connections (défaut : 100)

✅ **Connection Pooling = Solution incontournable**
   - Réutilise un petit pool pour servir beaucoup de clients
   - PgBouncer = Standard de facto

✅ **Modes de pooling :**
   - Transaction : Recommandé (90% des cas)
   - Session : Compatible tout PostgreSQL
   - Statement : Très rare

✅ **Dimensionnement :**
   - Pool size : 2-4× CPU cores
   - max_client_conn : 10-20× pool_size
   - Ne jamais dépasser 200 connexions PostgreSQL sans raison

✅ **Connection leaks = Problème fréquent**
   - Toujours fermer les connexions (try/finally)
   - Monitorer "idle in transaction"
   - Configurer timeouts

✅ **Monitoring essentiel :**
   - Nombre connexions actives vs max
   - État du pool PgBouncer
   - Connexions idle in transaction
   - Temps d'attente (maxwait)

✅ **Signaux d'alerte :**  
   - "Too many clients already"
   - cl_waiting > 0 dans PgBouncer
   - Connexions > 80% de max
   - Idle in transaction > 10

## Conclusion

La gestion des **connexions** est un aspect critique souvent sous-estimé de PostgreSQL. Une mauvaise configuration peut transformer une base de données puissante en goulot d'étranglement majeur.

Les points essentiels à retenir :

1. **Comprendre le coût** : Chaque connexion est un processus complet avec son overhead  
2. **Utiliser le pooling** : PgBouncer est votre meilleur ami en production  
3. **Dimensionner intelligemment** : Moins c'est souvent mieux (qualité > quantité)  
4. **Éviter les leaks** : Code propre avec gestion d'erreurs appropriée  
5. **Monitorer activement** : Détecter les problèmes avant qu'ils deviennent critiques

Avec **PgBouncer** correctement configuré, vous pouvez facilement supporter des milliers de clients applicatifs avec seulement 50-100 connexions PostgreSQL réelles. C'est la différence entre un système qui scale et un système qui s'effondre sous la charge.

L'investissement dans une bonne architecture de connexions se paye au centuple en stabilité, performances et sérénité opérationnelle.
