# 14.6.1. Cache Hit Ratio (Buffer Cache)

## Introduction

Le **Cache Hit Ratio** (taux de succès du cache) est l'une des métriques les plus importantes pour évaluer la performance de votre base de données PostgreSQL. C'est un indicateur qui vous permet de savoir si votre base de données utilise efficacement la mémoire disponible.

Dans ce chapitre, nous allons comprendre ce qu'est le cache, pourquoi il est crucial, et comment interpréter cette métrique pour optimiser les performances de votre système.

---

## Comprendre le Concept de Cache

### Qu'est-ce qu'un Cache ?

Imaginez que vous travaillez sur un projet et que vous devez consulter régulièrement certains documents. Vous avez deux options :

1. **Ranger les documents sur votre bureau** (accès rapide)  
2. **Les ranger dans une armoire à l'autre bout du bâtiment** (accès lent)

Naturellement, vous garderiez les documents les plus utilisés sur votre bureau pour y accéder rapidement. C'est exactement le principe d'un **cache** en informatique.

### Le Cache dans PostgreSQL

PostgreSQL fonctionne de la même manière avec vos données :

- **Le disque dur** (ou SSD) = L'armoire lointaine (lent mais grande capacité)  
- **La mémoire RAM** (cache) = Votre bureau (rapide mais espace limité)

PostgreSQL utilise une zone de mémoire appelée **Shared Buffers** (ou Buffer Cache) pour stocker temporairement les données les plus fréquemment utilisées.

### Pourquoi c'est Important ?

La différence de vitesse est considérable :

- **Lecture depuis la RAM** : ~100 nanosecondes  
- **Lecture depuis un SSD** : ~100 microsecondes (1000× plus lent)  
- **Lecture depuis un disque dur** : ~10 millisecondes (100 000× plus lent !)

Un bon taux de cache peut donc améliorer vos performances de manière spectaculaire.

---

## Le Cache Hit Ratio : Définition

### Qu'est-ce que le Cache Hit Ratio ?

Le **Cache Hit Ratio** mesure le pourcentage de fois où PostgreSQL trouve les données dont il a besoin directement dans la mémoire (cache), sans avoir à aller les chercher sur le disque.

**Formule simplifiée :**

```
Cache Hit Ratio = (Lectures depuis le cache / Lectures totales) × 100
```

### Les Deux Scénarios

#### Cache Hit (Succès) ✅

PostgreSQL cherche une donnée → Elle est dans la RAM → Accès ultra-rapide

#### Cache Miss (Échec) ❌

PostgreSQL cherche une donnée → Elle n'est PAS dans la RAM → Il doit aller la chercher sur le disque → Beaucoup plus lent

---

## Comment Mesurer le Cache Hit Ratio

### La Requête de Base

PostgreSQL fournit des statistiques système qui nous permettent de calculer ce ratio. Voici la requête à utiliser :

```sql
SELECT
    sum(heap_blks_read) as heap_read,
    sum(heap_blks_hit) as heap_hit,
    round(
        sum(heap_blks_hit) / nullif(sum(heap_blks_hit) + sum(heap_blks_read), 0) * 100,
        2
    ) as cache_hit_ratio
FROM pg_statio_user_tables;
```

#### Explication des Termes

- **heap_blks_read** : Nombre de blocs (morceaux de données) lus depuis le disque  
- **heap_blks_hit** : Nombre de blocs trouvés directement dans le cache  
- **cache_hit_ratio** : Le pourcentage de succès du cache

### Requête Plus Détaillée (Par Table)

Si vous voulez voir le ratio pour chaque table individuellement :

```sql
SELECT
    schemaname,
    relname as table_name,
    heap_blks_read as disk_reads,
    heap_blks_hit as cache_reads,
    round(
        heap_blks_hit::numeric / nullif(heap_blks_hit + heap_blks_read, 0) * 100,
        2
    ) as cache_hit_ratio_percent
FROM pg_statio_user_tables  
WHERE heap_blks_read + heap_blks_hit > 0  
ORDER BY cache_hit_ratio_percent ASC, heap_blks_read DESC;  
```

Cette requête affiche :
- Le nom du schéma et de la table
- Le nombre de lectures disque
- Le nombre de lectures cache
- Le ratio en pourcentage

Les tables avec les ratios les plus faibles apparaissent en premier, ce qui vous permet d'identifier rapidement les tables problématiques.

## Interpréter les Résultats

### Les Seuils de Référence

Voici comment interpréter votre Cache Hit Ratio :

| Ratio | État | Signification |
|-------|------|---------------|
| **> 99%** | 🟢 Excellent | Votre cache est très efficace |
| **95-99%** | 🟡 Bon | Acceptable, mais peut être amélioré |
| **90-95%** | 🟠 Moyen | Des optimisations sont recommandées |
| **< 90%** | 🔴 Problématique | Action nécessaire |

### Exemple d'Interprétation

#### Scénario 1 : Bon Ratio

```
cache_hit_ratio: 99.5%
```

**Signification :** Sur 1000 lectures, 995 sont servies depuis la RAM et seulement 5 nécessitent un accès disque. C'est excellent !

#### Scénario 2 : Ratio Faible

```
cache_hit_ratio: 85%
```

**Signification :** Sur 1000 lectures, 150 nécessitent un accès disque. Cela signifie que votre système attend souvent après le disque, ce qui ralentit les performances.

---

## Cas Particuliers et Exceptions

### Quand un Ratio Faible est Normal

Il existe des situations où un ratio plus faible est acceptable :

#### 1. Démarrage Récent de PostgreSQL

Juste après le démarrage, le cache est vide. Il faut du temps pour le "réchauffer" (warm-up).

**Que faire :** Attendre quelques heures d'utilisation normale avant de s'inquiéter.

#### 2. Première Requête sur de Nouvelles Données

Si vous interrogez des données qui n'ont jamais été accédées, elles ne peuvent pas être dans le cache.

**C'est normal :** Le premier accès sera toujours depuis le disque.

#### 3. Tables Très Volumineuses et Rarement Accédées

Si vous avez des tables d'archivage de 500 Go consultées une fois par mois, elles ne resteront pas en cache.

**C'est normal :** Ces données ne justifient pas d'occuper le cache.

#### 4. Scans Séquentiels Massifs

Lorsque vous faites une sauvegarde complète (pg_dump) ou un scan complet d'une grosse table, PostgreSQL peut volontairement ne pas mettre toutes ces données en cache pour ne pas "polluer" le cache avec des données utilisées une seule fois.

**C'est une protection :** PostgreSQL est intelligent et évite de gaspiller le cache.

## Facteurs Influençant le Cache Hit Ratio

### 1. Taille du Shared Buffers

Le paramètre **shared_buffers** détermine la taille du cache PostgreSQL.

**Configuration typique :**

```ini
# Dans postgresql.conf
shared_buffers = 256MB    # Serveur petit  
shared_buffers = 4GB      # Serveur moyen  
shared_buffers = 16GB     # Serveur puissant  
```

**Règle générale :**
- Serveur dédié : 25% de la RAM totale
- Serveur partagé : 10-15% de la RAM totale

### 2. Taille de la Base de Données

Si votre base de données fait 100 GB et que vous n'avez que 4 GB de shared_buffers, toutes les données ne pourront jamais être en cache simultanément.

**Logique :**
- Base petite (< taille cache) → Ratio peut atteindre 99%+
- Base grande (>>> taille cache) → Ratio dépendra des patterns d'accès

### 3. Pattern d'Accès aux Données

#### Hot Data (Données Chaudes)

Les données fréquemment accédées (ex: table "utilisateurs actifs").

**Impact :** Ratio élevé car ces données restent en cache.

#### Cold Data (Données Froides)

Les données rarement accédées (ex: archives de 5 ans).

**Impact :** Ratio plus faible car elles sont rechargées à chaque accès.

### 4. Type de Charge de Travail (Workload)

#### OLTP (Online Transaction Processing)

- Nombreuses petites transactions
- Accès répétés aux mêmes données
- **Ratio attendu :** 95-99%+

#### OLAP (Online Analytical Processing)

- Grosses requêtes analytiques
- Scans complets de tables
- **Ratio attendu :** Peut être plus faible (85-95%)

## Améliorer le Cache Hit Ratio

Si votre ratio est insuffisant, voici les actions possibles :

### 1. Augmenter shared_buffers

**Comment :**

```ini
# Dans postgresql.conf
shared_buffers = 8GB  # Augmentation de la taille
```

**Attention :** Après modification, vous devez redémarrer PostgreSQL.

**Limites :**
- Ne pas dépasser 40% de la RAM totale
- Au-delà de 16-32 GB, les gains diminuent

### 2. Optimiser les Requêtes

Des requêtes inefficaces peuvent parcourir inutilement beaucoup de données.

**Actions :**
- Ajouter des index appropriés
- Éviter les `SELECT *` quand vous n'avez besoin que de quelques colonnes
- Utiliser des `WHERE` pour filtrer les données

### 3. Ajouter de la RAM au Serveur

Si votre serveur manque de mémoire physique, augmenter la RAM est la solution la plus efficace.

**Exemple :** Passer de 8 GB à 32 GB de RAM permettra un shared_buffers plus grand.

### 4. Partitionner les Grandes Tables

Si vous avez des tables très volumineuses, le partitionnement permet de ne charger en cache que les partitions pertinentes.

**Exemple :** Table de logs partitionnée par mois → Seul le mois en cours reste en cache.

### 5. Utiliser pg_prewarm

L'extension **pg_prewarm** permet de "préchauffer" le cache en chargeant des tables spécifiques en mémoire au démarrage.

**Usage :**

```sql
-- Charger une table en cache
SELECT pg_prewarm('ma_table_importante');
```

## Cache Hit Ratio vs Autres Métriques

Le Cache Hit Ratio est important, mais il ne raconte pas toute l'histoire. Voici comment il s'intègre dans une vision globale :

### Métriques Complémentaires

#### 1. I/O Wait

Même avec un bon cache hit ratio, si les 1% de cache misses génèrent beaucoup d'attente I/O, vous avez un problème de disque.

**À surveiller :** Le temps d'attente disque (iostat, iotop)

#### 2. Query Performance

Un ratio élevé ne garantit pas des requêtes rapides si celles-ci sont mal optimisées.

**À surveiller :** Les requêtes lentes (pg_stat_statements)

#### 3. Index Usage

Des index non utilisés ou manquants peuvent forcer des scans de table entière, réduisant le ratio.

**À surveiller :** pg_stat_user_indexes

## Le Cache de l'OS (Operating System)

### Il N'y a Pas Qu'un Seul Cache !

Outre le shared_buffers de PostgreSQL, le système d'exploitation (Linux, Windows) maintient son propre cache disque appelé **Page Cache** (ou File System Cache).

**Architecture en couches :**

```
Application → PostgreSQL Shared Buffers → OS Page Cache → Disque
```

### Implication Pratique

Même si une donnée n'est pas dans les shared_buffers de PostgreSQL, elle peut être dans le cache de l'OS, ce qui reste beaucoup plus rapide que d'aller sur le disque.

**Ratio réel :**
- Ratio PostgreSQL : 95%
- Ratio OS Cache : 4%
- Vraies lectures disque : 1%

### Mesurer le Cache de l'OS

Vous pouvez voir l'utilisation du cache système avec :

**Linux :**
```bash
free -h
# Regardez la ligne "buff/cache"
```

**PostgreSQL offre aussi des métriques :**
```sql
SELECT * FROM pg_statio_user_tables;
-- Colonne "heap_blks_read" = lectures ayant traversé les caches
```

## Monitoring Continu

### Mettre en Place une Surveillance

Il est recommandé de surveiller ce ratio de manière continue, pas seulement ponctuellement.

#### Solution Simple : Script Périodique

```sql
-- Vue personnalisée pour faciliter le monitoring
CREATE OR REPLACE VIEW cache_hit_ratio_view AS  
SELECT  
    'Global' as scope,
    round(
        sum(heap_blks_hit)::numeric /
        nullif(sum(heap_blks_hit) + sum(heap_blks_read), 0) * 100,
        2
    ) as cache_hit_ratio_percent,
    sum(heap_blks_read) as total_disk_reads,
    sum(heap_blks_hit) as total_cache_reads
FROM pg_statio_user_tables;

-- Consulter facilement
SELECT * FROM cache_hit_ratio_view;
```

#### Solution Professionnelle : Monitoring Tools

Des outils comme **Grafana + Prometheus + postgres_exporter** permettent de visualiser l'évolution du ratio au fil du temps avec des graphiques et des alertes.

**Avantages :**
- Détection automatique de dégradations
- Historique des performances
- Alertes en cas de problème

## Questions Fréquentes (FAQ)

### Q1 : Mon ratio est de 99.9%, dois-je encore l'optimiser ?

**Réponse :** Non, c'est excellent ! Concentrez-vous sur d'autres aspects (requêtes lentes, index, etc.).

### Q2 : Mon ratio a chuté brutalement, que faire ?

**Réponse :** Vérifiez si :
1. PostgreSQL a redémarré (cache vidé)  
2. Une grosse requête analytique a été lancée  
3. Une nouvelle table volumineuse a été créée et massivement utilisée

### Q3 : Quelle est la taille idéale pour shared_buffers ?

**Réponse :** Il n'y a pas de valeur magique, mais :
- Démarrez avec 25% de la RAM
- Surveillez le ratio
- Ajustez progressivement si nécessaire

### Q4 : Un ratio de 100% est-il possible ?

**Réponse :** Théoriquement oui, si votre base complète tient en mémoire et que toutes les données sont constamment accédées. En pratique, 99.5%+ est déjà excellent.

### Q5 : Le ratio varie-t-il dans la journée ?

**Réponse :** Oui, c'est normal ! Il peut baisser pendant :
- Les sauvegardes
- Les rapports de fin de mois
- Les imports de données
- Le démarrage (warm-up)

## Points Clés à Retenir

✅ **Le Cache Hit Ratio mesure l'efficacité de votre mémoire cache**
   - Plus il est élevé, moins PostgreSQL attend après le disque

✅ **Cible : > 95%, idéalement > 99%**
   - Mais le contexte compte (OLTP vs OLAP)

✅ **Facteurs influençant le ratio :**
   - Taille du shared_buffers
   - Taille de la base de données
   - Patterns d'accès aux données
   - Type de workload

✅ **Actions d'amélioration :**
   - Augmenter shared_buffers (dans les limites raisonnables)
   - Optimiser les requêtes et les index
   - Ajouter de la RAM au serveur
   - Partitionner les grandes tables

✅ **Ne pas oublier :**
   - Le cache de l'OS joue aussi un rôle
   - Un bon ratio n'est qu'une métrique parmi d'autres
   - Le monitoring continu est essentiel

---

## Conclusion

Le **Cache Hit Ratio** est une métrique fondamentale pour comprendre les performances de votre base de données PostgreSQL. C'est un indicateur simple mais puissant qui vous dit si votre système exploite efficacement la mémoire disponible.

Un bon ratio (> 99%) signifie que votre application passe la majorité de son temps à traiter des données, pas à attendre le disque. C'est la différence entre une application réactive et une application frustrante pour les utilisateurs.

Maintenant que vous comprenez cette métrique, vous pouvez :
1. La mesurer régulièrement  
2. L'interpréter correctement  
3. Agir en conséquence pour optimiser vos performances

Dans la suite de cette formation, nous verrons d'autres métriques vitales qui complètent cette vision globale des performances de PostgreSQL.
