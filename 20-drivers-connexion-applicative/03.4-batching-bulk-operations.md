# 20.3.4. Batching et Bulk Operations

## Introduction

Lorsque vous devez manipuler des milliers ou des millions de lignes dans PostgreSQL, la façon dont vous procédez fait **toute la différence**. Insérer 100,000 lignes une par une peut prendre plusieurs heures, alors que les opérations en masse (bulk operations) peuvent accomplir la même tâche en quelques secondes.

**Objectif de ce chapitre :** Maîtriser les techniques de traitement en masse pour manipuler efficacement de grands volumes de données avec PostgreSQL.

---

## 1. Le Problème : Opérations Une par Une

### 1.1. Définition du Problème

**Opération unitaire :** Exécuter une requête SQL pour chaque ligne individuellement.

**Exemple :** Importer 10,000 utilisateurs depuis un fichier CSV.

### 1.2. L'Approche Naïve (Problématique)

```python
# Python avec Django ORM
import csv

with open('users.csv') as f:
    reader = csv.DictReader(f)
    for row in reader:
        User.objects.create(
            name=row['name'],
            email=row['email'],
            age=int(row['age'])
        )
# Pour 10,000 lignes : 10,000 INSERT !
```

**SQL généré (10,000 fois) :**
```sql
INSERT INTO users (name, email, age) VALUES ('Alice', 'alice@example.com', 25);  
INSERT INTO users (name, email, age) VALUES ('Bob', 'bob@example.com', 30);  
INSERT INTO users (name, email, age) VALUES ('Charlie', 'charlie@example.com', 35);  
-- ... 9,997 fois de plus !
```

### 1.3. Pourquoi C'est Problématique ?

**Overhead par requête :**
1. **Parsing SQL** : PostgreSQL doit analyser chaque requête  
2. **Planification** : Créer un plan d'exécution  
3. **Commit** : Écrire dans le WAL (Write-Ahead Log)  
4. **Lock/Unlock** : Acquisition et libération de verrous  
5. **Latence réseau** : Aller-retour application ↔ base

**Impact sur les performances :**
```
1 INSERT    = ~1ms (parsing + exec + commit)
10,000 fois = 10,000ms = 10 secondes minimum
```

**En réalité, souvent bien pire :**
- Locks cumulés
- Fragmentation du WAL
- Contention sur les index

**Temps réel observé : 30-60 secondes** pour 10,000 lignes ! 😱

### 1.4. Le Coût Caché : Les Index

Chaque INSERT déclenche la mise à jour de **tous les index** de la table.

```sql
-- Table avec 3 index
CREATE TABLE users (
    id SERIAL PRIMARY KEY,                    -- Index 1
    email VARCHAR(100) UNIQUE,                -- Index 2
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_users_created_at ON users(created_at);  -- Index 3
```

**1 INSERT = 3 mises à jour d'index !**  
**10,000 INSERT = 30,000 mises à jour d'index !**

## 2. La Solution : Bulk Operations

### 2.1. Définition

Les **bulk operations** (opérations en masse) regroupent plusieurs opérations en une seule requête ou en un lot optimisé.

**Principe :** Au lieu de 10,000 requêtes → 1 requête (ou quelques requêtes).

### 2.2. Types d'Opérations en Masse

1. **Bulk INSERT** : Insérer plusieurs lignes d'un coup  
2. **Bulk UPDATE** : Mettre à jour plusieurs lignes avec des valeurs différentes  
3. **Bulk DELETE** : Supprimer plusieurs lignes selon des critères  
4. **Batching** : Découper en lots pour gérer de très gros volumes

### 2.3. Avantages

- ✅ **Performance multipliée par 10 à 100×**  
- ✅ **Moins de pression sur le WAL**  
- ✅ **Moins de locks/unlocks**  
- ✅ **Moins de fragmentation**  
- ✅ **Meilleure utilisation du cache**

## 3. Bulk INSERT : Insérer en Masse

### 3.1. Approche 1 : INSERT avec VALUES Multiples

**SQL standard :**
```sql
-- Au lieu de 3 INSERT séparés
INSERT INTO users (name, email, age) VALUES ('Alice', 'alice@example.com', 25);  
INSERT INTO users (name, email, age) VALUES ('Bob', 'bob@example.com', 30);  
INSERT INTO users (name, email, age) VALUES ('Charlie', 'charlie@example.com', 35);  

-- Faire 1 seul INSERT avec plusieurs VALUES
INSERT INTO users (name, email, age) VALUES
    ('Alice', 'alice@example.com', 25),
    ('Bob', 'bob@example.com', 30),
    ('Charlie', 'charlie@example.com', 35);
```

**Avantages :**
- 1 parsing, 1 plan, 1 commit
- Beaucoup plus rapide

**Performance :**
- 3 INSERT séparés : ~3ms
- 1 INSERT groupé : ~0.5ms
- **Gain : 6×**

### 3.2. Approche 2 : ORM Bulk Create

#### Python - Django
```python
# ❌ Lent : 1 INSERT par ligne
for row in data:
    User.objects.create(name=row['name'], email=row['email'])

# ✅ Rapide : bulk_create
users = [
    User(name=row['name'], email=row['email'])
    for row in data
]
User.objects.bulk_create(users)
```

**SQL généré par bulk_create :**
```sql
INSERT INTO users (name, email) VALUES
    ('Alice', 'alice@example.com'),
    ('Bob', 'bob@example.com'),
    ('Charlie', 'charlie@example.com'),
    -- ... toutes les lignes d'un coup
```

**Options importantes :**
```python
User.objects.bulk_create(
    users,
    batch_size=1000,        # Découper en lots de 1000
    ignore_conflicts=True   # Ignorer les doublons (PG 9.5+)
)
```

#### Python - SQLAlchemy
```python
from sqlalchemy.orm import Session

# ❌ Lent
for row in data:
    session.add(User(name=row['name'], email=row['email']))
session.commit()

# ✅ Rapide
session.bulk_insert_mappings(
    User,
    [{'name': row['name'], 'email': row['email']} for row in data]
)
session.commit()
```

#### JavaScript - Prisma
```javascript
// ❌ Lent
for (const row of data) {
  await prisma.user.create({
    data: { name: row.name, email: row.email }
  });
}

// ✅ Rapide
await prisma.user.createMany({
  data: data.map(row => ({
    name: row.name,
    email: row.email
  }))
});
```

#### Java - Hibernate
```java
// ❌ Lent
for (User user : users) {
    session.save(user);
}

// ✅ Rapide avec batching
Session session = sessionFactory.openSession();  
Transaction tx = session.beginTransaction();  

for (int i = 0; i < users.size(); i++) {
    session.save(users.get(i));

    if (i % 50 == 0) {  // Batch de 50
        session.flush();
        session.clear();
    }
}

tx.commit();
```

**Configuration Hibernate :**
```properties
hibernate.jdbc.batch_size=50  
hibernate.order_inserts=true  
hibernate.order_updates=true  
```

### 3.3. Approche 3 : COPY (La Plus Rapide)

**COPY** est la commande native PostgreSQL pour l'import massif. C'est **la méthode la plus rapide**.

#### Syntaxe de base
```sql
COPY users(name, email, age)  
FROM '/tmp/users.csv'  
WITH (FORMAT csv, HEADER true);  
```

**Variante : COPY depuis STDIN**
```python
# Python avec psycopg3
import csv  
import psycopg  

conn = psycopg.connect("dbname=mydb")  
cursor = conn.cursor()  

with open('users.csv', 'r') as f:
    cursor.copy(
        "COPY users(name, email, age) FROM STDIN WITH CSV HEADER",
        f
    )

conn.commit()
```

#### Performance de COPY

**Benchmark pour 1,000,000 lignes :**

| Méthode | Temps | Vitesse |
|---------|-------|---------|
| INSERT une par une | ~60 min | 278 lignes/s |
| INSERT groupé (1000) | ~3 min | 5,555 lignes/s |
| bulk_create (Django) | ~2 min | 8,333 lignes/s |
| **COPY** | **~20 sec** | **50,000 lignes/s** |

**COPY est 180× plus rapide que INSERT unitaire !**

#### Pourquoi COPY est si rapide ?

1. **Format binaire optimisé** (moins de parsing)  
2. **Bypass des triggers** (optionnel)  
3. **Buffering massif** en mémoire  
4. **Moins de locks** (verrou unique)  
5. **Optimisation des index** (reconstruction en masse)

### 3.4. Comparaison des Approches

| Approche | Vitesse | Simplicité | Contrôle | Cas d'usage |
|----------|---------|------------|----------|-------------|
| INSERT unitaire | ⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | < 100 lignes |
| INSERT groupé | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 100-10k lignes |
| ORM bulk_create | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | 1k-100k lignes |
| **COPY** | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐ | > 10k lignes |

## 4. Bulk UPDATE : Mettre à Jour en Masse

### 4.1. Le Problème

Vous devez mettre à jour 10,000 lignes avec des **valeurs différentes**.

**Approche naïve :**
```python
# ❌ 10,000 UPDATE
for user_id, new_email in updates:
    User.objects.filter(id=user_id).update(email=new_email)
```

### 4.2. Solution 1 : UPDATE avec CASE

**SQL :**
```sql
UPDATE users  
SET email = CASE  
    WHEN id = 1 THEN 'alice_new@example.com'
    WHEN id = 2 THEN 'bob_new@example.com'
    WHEN id = 3 THEN 'charlie_new@example.com'
    ELSE email
END  
WHERE id IN (1, 2, 3);  
```

**Générer depuis Python :**
```python
from django.db import connection

def bulk_update_emails(updates):
    """
    updates = [(user_id, new_email), ...]
    """
    if not updates:
        return

    case_sql = "CASE\n"
    ids = []

    for user_id, new_email in updates:
        case_sql += f"  WHEN id = %s THEN %s\n"
        ids.append(user_id)
        ids.append(new_email)

    case_sql += "  ELSE email\nEND"

    user_ids = [u[0] for u in updates]

    sql = f"""
        UPDATE users
        SET email = {case_sql}
        WHERE id IN ({','.join(['%s'] * len(user_ids))})
    """

    with connection.cursor() as cursor:
        cursor.execute(sql, ids + user_ids)
```

### 4.3. Solution 2 : UPDATE FROM (PostgreSQL)

**Plus propre et plus performant :**

```sql
-- Créer une table temporaire
CREATE TEMP TABLE temp_updates (
    id INTEGER,
    new_email VARCHAR(100)
);

-- Insérer les nouvelles valeurs
INSERT INTO temp_updates VALUES
    (1, 'alice_new@example.com'),
    (2, 'bob_new@example.com'),
    (3, 'charlie_new@example.com');

-- UPDATE avec JOIN
UPDATE users  
SET email = temp_updates.new_email  
FROM temp_updates  
WHERE users.id = temp_updates.id;  

-- Nettoyer
DROP TABLE temp_updates;
```

**Avec Python (psycopg3) :**
```python
def bulk_update_emails(cursor, updates):
    # Créer table temporaire
    cursor.execute("""
        CREATE TEMP TABLE temp_updates (
            id INTEGER,
            new_email VARCHAR(100)
        ) ON COMMIT DROP
    """)

    # Insérer avec COPY (rapide !)
    with cursor.copy("COPY temp_updates(id, new_email) FROM STDIN") as copy:
        for user_id, new_email in updates:
            copy.write_row([user_id, new_email])

    # UPDATE avec JOIN
    cursor.execute("""
        UPDATE users
        SET email = temp_updates.new_email
        FROM temp_updates
        WHERE users.id = temp_updates.id
    """)
```

**Performance :**
- 10,000 UPDATE unitaires : ~30 secondes
- 1 UPDATE avec CASE : ~2 secondes
- 1 UPDATE FROM temp table : ~0.5 secondes

### 4.4. Solution 3 : ORM bulk_update (Django 2.2+)

```python

users = User.objects.filter(id__in=[1, 2, 3])

# Modifier en mémoire
```

## 5. Bulk DELETE : Supprimer en Masse

### 5.1. DELETE Simple

**Si tous les critères sont identiques :**
```sql
-- ✅ Efficient : 1 seule requête
DELETE FROM users WHERE created_at < '2020-01-01';
```

**ORM équivalent :**
```python
# Django
User.objects.filter(created_at__lt='2020-01-01').delete()

# SQLAlchemy
session.query(User).filter(User.created_at < '2020-01-01').delete()
```

### 5.2. DELETE avec Liste d'IDs

**SQL :**
```sql
DELETE FROM users WHERE id IN (1, 2, 3, 4, 5, ..., 10000);
```

**Attention :** PostgreSQL a une limite sur la longueur des requêtes.

**Solution : Découper en lots**
```python
def bulk_delete_by_ids(user_ids, batch_size=1000):
    for i in range(0, len(user_ids), batch_size):
        batch = user_ids[i:i + batch_size]
        User.objects.filter(id__in=batch).delete()
```

### 5.3. DELETE avec Table Temporaire

Pour des critères complexes :

```sql
-- Créer une table temporaire avec les IDs à supprimer
CREATE TEMP TABLE to_delete (id INTEGER);

INSERT INTO to_delete  
SELECT id FROM users  
WHERE some_complex_condition;  

-- Supprimer
DELETE FROM users  
WHERE id IN (SELECT id FROM to_delete);  
```

### 5.4. TRUNCATE vs DELETE

**TRUNCATE** : Vider complètement une table (ultra rapide).

```sql
-- ❌ Lent pour 10M lignes
DELETE FROM users;

-- ✅ Instantané
TRUNCATE TABLE users;
```

**Différences :**

| Critère | DELETE | TRUNCATE |
|---------|--------|----------|
| Vitesse | Lente (scan complet) | Instantanée |
| Conditions WHERE | ✅ Oui | ❌ Non |
| Triggers | ✅ Exécutés | ❌ Ignorés |
| Rollback | ✅ Possible | ⚠️ Possible mais moins sûr |
| Réinitialise AUTO_INCREMENT | ❌ Non | ✅ Oui |

**Quand utiliser TRUNCATE :**
- Vider complètement une table
- Pas besoin de conditions
- Performance critique

## 6. Batching : Gérer de Gros Volumes

### 6.1. Qu'est-ce que le Batching ?

**Batching** : Découper une opération massive en **lots (batches)** plus petits.

**Pourquoi :**
1. **Éviter la saturation mémoire** (application + PostgreSQL)  
2. **Permettre le commit progressif** (rollback partiel possible)  
3. **Réduire la durée des locks** (pas de verrouillage trop long)  
4. **Gérer les timeouts** (requêtes très longues)

### 6.2. Exemple : Import de 10 Millions de Lignes

**Sans batching (problématique) :**
```python
# ❌ Tout en mémoire : OOM (Out Of Memory)
users = [User(name=row['name'], email=row['email']) for row in all_10M_rows]  
User.objects.bulk_create(users)  # 💥 Crash !  
```

**Avec batching :**
```python
BATCH_SIZE = 10000

def import_users_batched(csv_file):
    batch = []

    with open(csv_file) as f:
        reader = csv.DictReader(f)

        for row in reader:
            batch.append(User(name=row['name'], email=row['email']))

            if len(batch) >= BATCH_SIZE:
                User.objects.bulk_create(batch)
                batch = []  # Vider le batch

        # Dernier batch (reste)
        if batch:
            User.objects.bulk_create(batch)
```

**Avantages :**
- Mémoire constante (~10k lignes max)
- Commit tous les 10k (récupérable en cas d'erreur)
- Pas de timeout

### 6.3. Batching avec Progrès

**Afficher la progression :**
```python
import csv  
from tqdm import tqdm  # Barre de progression  

def import_users_with_progress(csv_file):
    BATCH_SIZE = 10000
    batch = []

    # Compter les lignes d'abord
    with open(csv_file) as f:
        total_lines = sum(1 for _ in f) - 1  # -1 pour le header

    with open(csv_file) as f:
        reader = csv.DictReader(f)

        for row in tqdm(reader, total=total_lines, desc="Importing users"):
            batch.append(User(name=row['name'], email=row['email']))

            if len(batch) >= BATCH_SIZE:
                User.objects.bulk_create(batch)
                batch = []

        if batch:
            User.objects.bulk_create(batch)

# Affiche : Importing users: 45%|████████████████         | 4523891/10000000 [02:15<02:44, 33432.89it/s]
```

### 6.4. Batching avec Gestion d'Erreurs

**Principe :** Si un batch échoue, continuer avec les suivants.

```python
def import_users_resilient(csv_file):
    BATCH_SIZE = 10000
    batch = []
    errors = []

    with open(csv_file) as f:
        reader = csv.DictReader(f)

        for i, row in enumerate(reader):
            try:
                batch.append(User(name=row['name'], email=row['email']))

                if len(batch) >= BATCH_SIZE:
                    try:
                        User.objects.bulk_create(batch)
                    except Exception as e:
                        errors.append({
                            'batch_start': i - BATCH_SIZE,
                            'batch_end': i,
                            'error': str(e)
                        })
                    finally:
                        batch = []

            except Exception as e:
                errors.append({
                    'line': i,
                    'row': row,
                    'error': str(e)
                })

        # Dernier batch
        if batch:
            try:
                User.objects.bulk_create(batch)
            except Exception as e:
                errors.append({
                    'batch_start': i - len(batch),
                    'batch_end': i,
                    'error': str(e)
                })

    return errors
```

## 7. Transactions et Atomicité

### 7.1. Transaction Globale

**Tout ou rien :**
```python
from django.db import transaction

@transaction.atomic
def import_users(csv_file):
    # Si une erreur survient, TOUT est rollback
    users = []
    with open(csv_file) as f:
        for row in csv.DictReader(f):
            users.append(User(name=row['name'], email=row['email']))

    User.objects.bulk_create(users)
```

**Problème :** Si l'import échoue à 99%, tout est perdu.

### 7.2. Transaction par Batch

**Meilleur compromis :**
```python
def import_users_with_batch_transaction(csv_file):
    BATCH_SIZE = 10000
    batch = []

    with open(csv_file) as f:
        reader = csv.DictReader(f)

        for row in reader:
            batch.append(User(name=row['name'], email=row['email']))

            if len(batch) >= BATCH_SIZE:
                with transaction.atomic():
                    User.objects.bulk_create(batch)
                batch = []

        if batch:
            with transaction.atomic():
                User.objects.bulk_create(batch)
```

**Avantage :** Si batch #42 échoue, les 41 premiers sont conservés.

### 7.3. Savepoints pour Rollback Partiel

```python
def import_users_with_savepoint(csv_file):
    with transaction.atomic():
        batch = []

        with open(csv_file) as f:
            for row in csv.DictReader(f):
                batch.append(User(name=row['name'], email=row['email']))

                if len(batch) >= 10000:
                    sid = transaction.savepoint()  # Point de sauvegarde

                    try:
                        User.objects.bulk_create(batch)
                        transaction.savepoint_commit(sid)
                    except Exception:
                        transaction.savepoint_rollback(sid)  # Rollback partiel

                    batch = []
```

## 8. Performance et Optimisations

### 8.1. Désactiver les Contraintes Temporairement

**Pour import initial massif SEULEMENT :**

```sql
-- Désactiver les triggers
ALTER TABLE users DISABLE TRIGGER ALL;

-- Import massif
COPY users FROM '/tmp/users.csv' WITH CSV;

-- Réactiver
ALTER TABLE users ENABLE TRIGGER ALL;
```

**⚠️ ATTENTION :** Les contraintes ne sont plus vérifiées !

### 8.2. Désactiver les Index Temporairement

**Pour tables sans données existantes :**

```sql
-- Supprimer les index
DROP INDEX idx_users_email;  
DROP INDEX idx_users_created_at;  

-- Import massif
COPY users FROM '/tmp/users.csv' WITH CSV;

-- Recréer les index (plus rapide en une fois)
CREATE INDEX idx_users_email ON users(email);  
CREATE INDEX idx_users_created_at ON users(created_at);  
```

**Performance :** 3-5× plus rapide pour import initial.

### 8.3. Augmenter maintenance_work_mem

**Temporairement pour l'import :**

```sql
-- Augmenter la mémoire pour les opérations de maintenance
SET maintenance_work_mem = '2GB';

-- Import
COPY users FROM '/tmp/users.csv' WITH CSV;

-- Réinitialiser
RESET maintenance_work_mem;
```

### 8.4. UNLOGGED Tables (Temporaire)

**Pour imports temporaires :**

```sql
-- Créer une table UNLOGGED (pas de WAL)
CREATE UNLOGGED TABLE users_temp (LIKE users INCLUDING ALL);

-- Import ultra-rapide
COPY users_temp FROM '/tmp/users.csv' WITH CSV;

-- Convertir en table normale
ALTER TABLE users_temp SET LOGGED;
```

**Performance :** 2-3× plus rapide.

**⚠️ RISQUE :** Données perdues en cas de crash pendant l'import.

## 9. Gestion des Erreurs et Validation

### 9.1. Validation Avant Import

```python
def validate_csv(csv_file):
    errors = []

    with open(csv_file) as f:
        reader = csv.DictReader(f)

        for i, row in enumerate(reader, start=2):  # start=2 car ligne 1 = header
            # Vérifier les champs obligatoires
            if not row.get('name'):
                errors.append(f"Ligne {i}: 'name' manquant")

            # Vérifier le format email
            if not re.match(r'^[\w\.-]+@[\w\.-]+\.\w+$', row.get('email', '')):
                errors.append(f"Ligne {i}: email invalide '{row.get('email')}'")

            # Vérifier la plage de valeurs
            try:
                age = int(row.get('age', 0))
                if age < 0 or age > 150:
                    errors.append(f"Ligne {i}: âge invalide {age}")
            except ValueError:
                errors.append(f"Ligne {i}: âge doit être un nombre")

    return errors

# Utilisation
errors = validate_csv('users.csv')  
if errors:

```

### 9.2. ignore_conflicts : Ignorer les Doublons

```python
# Django : ignorer les doublons (ON CONFLICT DO NOTHING)
User.objects.bulk_create(users, ignore_conflicts=True)
```

**SQL généré :**
```sql
INSERT INTO users (name, email) VALUES
    ('Alice', 'alice@example.com'),
    ('Bob', 'bob@example.com')
ON CONFLICT DO NOTHING;
```

### 9.3. update_conflicts : Upsert en Masse

```python
# Django 4.1+ : upsert (INSERT ou UPDATE)
User.objects.bulk_create(
    users,
    update_conflicts=True,
    update_fields=['name'],
    unique_fields=['email']
)
```

**SQL généré :**
```sql
INSERT INTO users (name, email) VALUES
    ('Alice', 'alice@example.com'),
    ('Bob', 'bob@example.com')
ON CONFLICT (email) DO UPDATE SET name = EXCLUDED.name;
```

## 10. Cas Pratiques par Scénario

### 10.1. Import CSV de 100,000 Lignes

**Solution optimale : COPY**

```python
import psycopg

def import_csv_fast(csv_file, table_name):
    conn = psycopg.connect("dbname=mydb")
    cursor = conn.cursor()

    with open(csv_file, 'r') as f:
        cursor.copy(
            f"COPY {table_name} FROM STDIN WITH CSV HEADER",
            f
        )

    conn.commit()
    conn.close()

# Temps : ~2 secondes pour 100k lignes
```

### 10.2. Synchronisation depuis API Externe

**Contexte :** Récupérer 50,000 produits depuis une API et les mettre à jour.

```python
import requests

def sync_products_from_api():
    BATCH_SIZE = 1000

    # Récupérer depuis l'API
    response = requests.get('https://api.example.com/products')
    products_data = response.json()

    # Préparer les objets
    products = [
        Product(
            external_id=p['id'],
            name=p['name'],
            price=p['price']
        )
        for p in products_data
    ]

    # Bulk upsert par batch
    for i in range(0, len(products), BATCH_SIZE):
        batch = products[i:i + BATCH_SIZE]

        Product.objects.bulk_create(
            batch,
            update_conflicts=True,
            update_fields=['name', 'price'],
            unique_fields=['external_id']
        )

    print(f"✅ {len(products)} produits synchronisés")
```

### 10.3. Calcul de Statistiques sur Millions de Lignes

**Contexte :** Calculer les statistiques mensuelles sur 10M de transactions.

```python
def calculate_monthly_stats():
    # ❌ Charger tout en mémoire : OOM
    # transactions = Transaction.objects.all()

    # ✅ Itérer par batch
    BATCH_SIZE = 10000
    offset = 0
    stats = {}

    while True:
        batch = Transaction.objects.all()[offset:offset + BATCH_SIZE]

        if not batch:
            break

        for transaction in batch:
            month = transaction.date.strftime('%Y-%m')
            stats.setdefault(month, {'total': 0, 'count': 0})
            stats[month]['total'] += transaction.amount
            stats[month]['count'] += 1

        offset += BATCH_SIZE

    return stats
```

**Mieux : Faire le calcul en SQL**
```python
from django.db.models import Sum, Count  
from django.db.models.functions import TruncMonth  

stats = Transaction.objects.annotate(
    month=TruncMonth('date')
).values('month').annotate(
    total=Sum('amount'),
    count=Count('id')
).order_by('month')
```

## 11. Checklist et Bonnes Pratiques

### 11.1. Checklist de Performance

Avant de lancer une opération massive :

- [ ] **Volume estimé** : Combien de lignes ?  
- [ ] **Méthode choisie** : INSERT groupé, bulk_create, ou COPY ?  
- [ ] **Batching configuré** : Taille de batch adaptée (1000-10000)  
- [ ] **Transactions** : Par batch ou globale ?  
- [ ] **Validation** : Données validées avant import ?  
- [ ] **Gestion erreurs** : Stratégie de récupération ?  
- [ ] **Index** : Désactiver temporairement si table vide ?  
- [ ] **Monitoring** : Surveiller l'utilisation mémoire et CPU ?  
- [ ] **Backup** : Sauvegarde avant modification massive ?  
- [ ] **Tests** : Testé sur sous-ensemble d'abord ?

### 11.2. Tailles de Batch Recommandées

| Volume Total | Batch Size | Raison |
|--------------|------------|--------|
| < 1,000 | Pas de batching | Overhead inutile |
| 1,000 - 10,000 | 1,000 | Équilibre performance/mémoire |
| 10,000 - 100,000 | 5,000 | Bonne performance |
| 100,000 - 1M | 10,000 | Optimal pour gros volumes |
| > 1M | 10,000 - 50,000 | Selon mémoire disponible |

### 11.3. Guide de Choix de Méthode

```
Vous devez insérer combien de lignes ?
│
├─ < 100
│  └─→ INSERT unitaire OK (simplicité)
│
├─ 100 - 1,000
│  └─→ ORM bulk_create (bon compromis)
│
├─ 1,000 - 10,000
│  └─→ ORM bulk_create avec batching
│
├─ 10,000 - 100,000
│  ├─ Données déjà en CSV/fichier ?
│  │  └─→ COPY (optimal)
│  └─ Données depuis API/traitement ?
│     └─→ bulk_create avec batching
│
└─ > 100,000
   └─→ COPY + batching + désactiver index temporairement
```

### 11.4. Anti-Patterns à Éviter

#### ❌ Anti-Pattern 1 : Pas de Batching

```python
# ❌ Charger 1M lignes en mémoire
users = [User(...) for i in range(1_000_000)]  
User.objects.bulk_create(users)  # OOM !  
```

#### ❌ Anti-Pattern 2 : Batch Trop Petit

```python
# ❌ Batch de 10 : trop d'overhead
for i in range(0, len(users), 10):
    User.objects.bulk_create(users[i:i+10])
```

#### ❌ Anti-Pattern 3 : Ignorer les Erreurs

```python
# ❌ Aucune gestion d'erreur
try:
    User.objects.bulk_create(users)
except Exception:
    pass  # Données perdues !
```

#### ❌ Anti-Pattern 4 : Transaction Globale Massive

```python
# ❌ Si échec à 99%, tout est perdu
@transaction.atomic
def import_10M_users():
    # ... 10M lignes
    pass
```

```python
import time

def benchmark_import(method, data):
    start = time.time()
    method(data)
    elapsed = time.time() - start

    rows_per_sec = len(data) / elapsed
    print(f"⏱️  Temps : {elapsed:.2f}s")
    print(f"📊 Vitesse : {rows_per_sec:.0f} lignes/s")

# Comparer les méthodes
```

## 13. Résumé et Synthèse

### 13.1. Points Clés à Retenir

1. **Les opérations unitaires sont 100× plus lentes** que les bulk operations  
2. **COPY est la méthode la plus rapide** pour l'import massif (50,000 lignes/s)  
3. **Toujours utiliser le batching** pour > 1,000 lignes  
4. **Taille de batch optimale : 1,000 à 10,000** selon le volume  
5. **Transaction par batch** pour permettre la récupération partielle  
6. **Valider avant d'importer** pour éviter les erreurs en cours de route  
7. **Désactiver les index temporairement** pour import initial massif

### 13.2. Tableau de Performance

**Benchmark : 100,000 lignes**

| Méthode | Temps | Lignes/s | Mémoire | Complexité |
|---------|-------|----------|---------|------------|
| INSERT unitaire | 600s | 167 | Faible | Simple |
| INSERT groupé (x100) | 60s | 1,667 | Moyenne | Simple |
| ORM bulk_create | 12s | 8,333 | Moyenne | Simple |
| ORM bulk_create + batch | 15s | 6,667 | Faible | Moyenne |
| **COPY** | **2s** | **50,000** | Faible | Moyenne |

### 13.3. Règles d'Or

#### 1. Volume < 1,000 : Simplicité
Pas besoin d'optimisation, INSERT classique suffit.

#### 2. Volume 1,000-10,000 : ORM Bulk
Utilisez `bulk_create` ou équivalent.

#### 3. Volume > 10,000 : COPY
Privilégiez COPY pour la performance maximale.

#### 4. Toujours Batcher
Au-delà de 1,000 lignes, découpez en lots.

#### 5. Valider d'Abord
Ne jamais importer sans validation préalable.

**Situation :** Vous devez synchroniser quotidiennement 500,000 produits depuis un API externe vers PostgreSQL. Certains produits existent déjà, d'autres sont nouveaux.

```python
def sync_products_daily():

    products_data = fetch_from_api()  # 500k produits
```
