# 20.3.2. ORM vs SQL brut : Quand utiliser quoi

## Introduction

Le débat entre **ORM** (Object-Relational Mapping) et **SQL brut** est l'un des plus anciens et des plus passionnés dans le monde du développement. Certains développeurs jurent que par les ORM, d'autres les fuient comme la peste. La vérité ? **Les deux ont leur place**, et savoir quand utiliser l'un ou l'autre est une compétence essentielle.

**Objectif de ce chapitre :** Comprendre ce qu'est un ORM, ses avantages et limites, et développer un sens critique pour choisir la bonne approche selon le contexte.

---

## 1. Qu'est-ce qu'un ORM ?

### 1.1. Définition Simple

Un **ORM** (Object-Relational Mapping) est une bibliothèque qui fait le pont entre votre code orienté objet et votre base de données relationnelle.

**Analogie :** C'est comme un traducteur automatique entre votre langage de programmation et SQL.

**Concrètement :**
- Vous manipulez des **objets** (classes, instances)
- L'ORM génère et exécute du **SQL** pour vous
- Vous n'écrivez (presque) jamais de SQL directement

### 1.2. Exemple Visuel

**Sans ORM (SQL brut) :**
```python
# Python avec psycopg3
cursor.execute("SELECT * FROM users WHERE email = %s", (email,))  
row = cursor.fetchone()  
user = {  
    'id': row[0],
    'name': row[1],
    'email': row[2]
}
```

**Avec ORM (exemple Django) :**
```python
# Python avec Django ORM
user = User.objects.get(email=email)
# user.id, user.name, user.email sont directement accessibles
```

**Résultat :** Le code avec ORM est plus court, plus lisible, plus "pythonique".

### 1.3. ORM Populaires par Langage

| Langage | ORM Principal | Alternatives |
|---------|---------------|--------------|
| **Python** | Django ORM, SQLAlchemy | Peewee, Tortoise ORM, Pony ORM |
| **Java** | Hibernate | JPA, MyBatis, jOOQ |
| **JavaScript/TypeScript** | Prisma, TypeORM | Sequelize, Knex.js, Drizzle |
| **PHP** | Eloquent (Laravel), Doctrine | Propel, RedBeanPHP |
| **Ruby** | Active Record (Rails) | Sequel, DataMapper |
| **C#** | Entity Framework | Dapper, NHibernate |
| **Go** | GORM | SQLBoiler, Ent |

### 1.4. Que Fait un ORM Concrètement ?

**Fonctionnalités typiques :**

1. **Mapping Objet-Table**
   ```python
   class User:  # Classe Python
       id: int
       name: str
       email: str
   # → Table 'users' en base de données
   ```

2. **Génération de SQL**
   ```python
   User.objects.filter(age__gte=18)
   # → SELECT * FROM users WHERE age >= 18
   ```

3. **Gestion des Relations**
   ```python
   user.articles.all()  # Articles de cet utilisateur
   # → SELECT * FROM articles WHERE user_id = ?
   ```

4. **Migrations de Schéma**
   ```bash
   python manage.py makemigrations
   # → Génère un fichier de migration SQL
   ```

5. **Validation et Contraintes**
   ```python
   user = User(email="invalid")  # Validation automatique
   user.save()  # → Erreur si email invalide
   ```

## 2. SQL Brut : La Fondation

### 2.1. Qu'est-ce que le SQL Brut ?

**SQL brut** (ou raw SQL) signifie écrire directement vos requêtes SQL et les exécuter via un driver de base de données.

**Exemple avec Python (psycopg3) :**
```python
import psycopg

# Connexion
conn = psycopg.connect("dbname=mydb user=postgres")  
cursor = conn.cursor()  

# Requête SQL brute
cursor.execute("""
    SELECT u.name, COUNT(a.id) as article_count
    FROM users u
    LEFT JOIN articles a ON u.id = a.user_id
    GROUP BY u.id, u.name
    HAVING COUNT(a.id) > 5
    ORDER BY article_count DESC
""")

results = cursor.fetchall()  
for row in results:  
    print(f"{row[0]}: {row[1]} articles")

cursor.close()  
conn.close()  
```

### 2.2. Avantages du SQL Brut

#### ✅ 1. Contrôle Total

Vous écrivez **exactement** le SQL que vous voulez, sans surprise.

**Exemple :** Requête optimisée avec window functions
```sql
SELECT
    user_id,
    name,
    article_title,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY published_at DESC) as rank
FROM users u  
JOIN articles a ON u.id = a.user_id  
WHERE rank <= 3;  -- Top 3 articles par utilisateur  
```

Difficile (voire impossible) à faire proprement avec certains ORM.

#### ✅ 2. Performance Optimale

Vous pouvez utiliser **toutes les features** de PostgreSQL :
- CTE récursives
- Window functions avancées
- LATERAL joins
- Optimisations spécifiques (EXPLAIN, hints)

#### ✅ 3. Pas de "Magie Noire"

Ce qui s'exécute = ce que vous avez écrit. Pas de surprise sur :
- Le SQL généré
- Le nombre de requêtes
- Les jointures effectuées

#### ✅ 4. Courbe d'Apprentissage SQL

Écrire du SQL brut **améliore votre maîtrise de SQL**, une compétence transférable.

### 2.3. Inconvénients du SQL Brut

#### ❌ 1. Verbosité

Beaucoup de code "boilerplate" :
- Gestion des connexions
- Gestion des curseurs
- Mapping manuel des résultats
- Gestion des erreurs

#### ❌ 2. Risque d'Injection SQL

Sans précautions, le code devient vulnérable :

```python
# ⚠️ DANGEREUX : Injection SQL possible
query = f"SELECT * FROM users WHERE email = '{email}'"  
cursor.execute(query)  

# ✅ SÛR : Paramètres liés
cursor.execute("SELECT * FROM users WHERE email = %s", (email,))
```

#### ❌ 3. Dépendance à la Base de Données

Votre SQL est souvent spécifique à PostgreSQL. Migrer vers MySQL = réécriture.

#### ❌ 4. Pas de Migrations Automatiques

Vous devez gérer manuellement :
- Les fichiers de migration SQL
- Les versions du schéma
- Le rollback en cas d'erreur

## 3. Les ORM : Productivité vs Contrôle

### 3.1. Avantages des ORM

#### ✅ 1. Productivité Accélérée

**Moins de code pour les opérations courantes :**

```python
# Créer un utilisateur
user = User.objects.create(name="Alice", email="alice@example.com")

# Récupérer tous les utilisateurs actifs
active_users = User.objects.filter(is_active=True)

# Compter les articles
article_count = Article.objects.count()
```

**Équivalent SQL brut :** 3× plus de lignes de code.

#### ✅ 2. Abstraction de la Base de Données

Le même code ORM fonctionne sur PostgreSQL, MySQL, SQLite... (avec quelques limites).

```python
# Ce code fonctionne sur plusieurs SGBD
users = User.objects.filter(created_at__gte=datetime(2024, 1, 1))
```

#### ✅ 3. Gestion Automatique des Relations

```python
# Django ORM : Récupérer les articles d'un utilisateur
user = User.objects.get(id=1)  
articles = user.articles.all()  # Relation 1-N automatique  

# L'ORM génère le JOIN pour vous
```

#### ✅ 4. Migrations de Schéma Automatisées

```python
# Modifier un modèle
class User(models.Model):
    name = models.CharField(max_length=100)
    email = models.EmailField()
    bio = models.TextField()  # ← Nouveau champ

# Générer et appliquer la migration
# python manage.py makemigrations
# python manage.py migrate
```

L'ORM génère le SQL `ALTER TABLE` pour vous.

#### ✅ 5. Validation et Contraintes

```python
class User(models.Model):
    email = models.EmailField(unique=True)  # Validation automatique
    age = models.IntegerField(validators=[MinValueValidator(0)])

# Si email invalide ou age < 0 → Exception avant même d'aller en base
```

#### ✅ 6. Protection Contre les Injections SQL

Les ORM paramètrent automatiquement les requêtes :

```python
# Automatiquement sécurisé
User.objects.filter(email=user_input)
# L'ORM génère : WHERE email = %s avec paramètre lié
```

### 3.2. Inconvénients des ORM

#### ❌ 1. Courbe d'Apprentissage Double

Vous devez apprendre :
1. **SQL** (pour comprendre ce qui se passe)  
2. **L'API de l'ORM** (syntaxe spécifique)

**Exemple :** Faire un `OUTER JOIN` en Django ORM vs SQL :

```python
# Django ORM (pas immédiatement évident)
User.objects.annotate(
    article_count=Count('articles')
).filter(article_count__gte=5)

# SQL (plus clair si on connaît SQL)
SELECT u.*, COUNT(a.id) as article_count  
FROM users u  
LEFT JOIN articles a ON u.id = a.user_id  
GROUP BY u.id  
HAVING COUNT(a.id) >= 5  
```

#### ❌ 2. Performance : Requêtes Suboptimales

Les ORM génèrent parfois du SQL inefficace.

**Exemple 1 : N+1 queries** (vu au chapitre précédent)

```python
# ❌ Génère N+1 requêtes
for article in Article.objects.all():
    print(article.author.name)

# ✅ Solution : select_related
for article in Article.objects.select_related('author'):
    print(article.author.name)
```

**Exemple 2 : SELECT * inutile**

```python
# ORM génère : SELECT * FROM users (tous les champs)
users = User.objects.all()

# Alors qu'on veut juste :
# SELECT name, email FROM users
users = User.objects.values('name', 'email')
```

#### ❌ 3. Limitations sur Requêtes Complexes

Certaines requêtes avancées sont difficiles voire impossibles :
- CTE récursives
- Window functions complexes
- LATERAL joins
- Optimisations PostgreSQL spécifiques

**Solution :** Revenir au SQL brut pour ces cas.

#### ❌ 4. Abstraction qui Fuit (Leaky Abstraction)

L'ORM cache la complexité... jusqu'à ce que ça ne marche plus.

**Symptôme :** "Ça marchait avec 100 lignes, mais avec 100,000 c'est très lent !"

**Cause :** L'ORM génère du SQL inefficace qu'il faut investiguer.

#### ❌ 5. Dépendance à l'ORM

Votre code devient dépendant de l'ORM choisi. Changer d'ORM = réécriture majeure.

## 4. Arbre de Décision : ORM ou SQL Brut ?

### 4.1. Quand Utiliser un ORM ?

#### ✅ Scénario 1 : CRUD Basique (Create, Read, Update, Delete)

**Opérations simples sur une seule table ou relations 1-1 :**

```python
# Créer
user = User.objects.create(name="Bob", email="bob@example.com")

# Lire
user = User.objects.get(id=1)

# Mettre à jour
user.name = "Robert"  
user.save()  

# Supprimer
user.delete()
```

**Verdict :** ORM parfait. Rapide, lisible, sécurisé.

#### ✅ Scénario 2 : Application CRUD Standard

**Blog, CMS, backoffice, dashboard simples** où :
- Peu de jointures complexes
- Requêtes prévisibles
- Volume de données modéré (< 1M lignes par table)

**Verdict :** ORM recommandé. Productivité maximale.

#### ✅ Scénario 3 : Prototypage Rapide

**MVP, POC, démos** où la vitesse de développement prime.

**Verdict :** ORM optimal. Itérez vite, optimisez plus tard.

#### ✅ Scénario 4 : Équipe Junior ou Mixte

**Si l'équipe connaît peu SQL** ou préfère le code orienté objet.

**Verdict :** ORM avec formation sur les pièges (N+1, etc.).

### 4.2. Quand Utiliser SQL Brut ?

#### ✅ Scénario 1 : Requêtes Analytiques Complexes

**Reporting, BI, dashboards** avec :
- Multiples jointures
- Agrégations complexes
- Window functions
- CTE récursives

**Exemple :**
```sql
WITH monthly_stats AS (
    SELECT
        DATE_TRUNC('month', created_at) as month,
        user_id,
        COUNT(*) as article_count,
        AVG(view_count) as avg_views
    FROM articles
    GROUP BY 1, 2
),
ranked_users AS (
    SELECT
        month,
        user_id,
        article_count,
        RANK() OVER (PARTITION BY month ORDER BY article_count DESC) as rank
    FROM monthly_stats
)
SELECT * FROM ranked_users WHERE rank <= 10;
```

**Verdict :** SQL brut indispensable.

#### ✅ Scénario 2 : Performance Critique

**Endpoints à haute fréquence** (> 1000 req/s) où chaque milliseconde compte.

**Exemple :** API de prix en temps réel, flux de trading, jeux en ligne.

**Verdict :** SQL brut, optimisé au maximum avec EXPLAIN ANALYZE.

#### ✅ Scénario 3 : Bulk Operations

**Import/export massif** de données (millions de lignes).

**Exemple :**
```sql
COPY users(name, email, created_at)  
FROM '/tmp/users.csv'  
WITH (FORMAT csv, HEADER true);  
```

**Verdict :** SQL brut + commandes PostgreSQL natives.

#### ✅ Scénario 4 : Migrations Complexes

**Transformations de schéma** avec logique métier embarquée.

**Exemple :**
```sql
-- Migration : scinder le champ "full_name" en "first_name" et "last_name"
UPDATE users  
SET  
    first_name = SPLIT_PART(full_name, ' ', 1),
    last_name = SPLIT_PART(full_name, ' ', 2)
WHERE full_name IS NOT NULL;
```

**Verdict :** SQL brut pour contrôle total.

#### ✅ Scénario 5 : Features PostgreSQL Avancées

**Si vous devez utiliser :**
- PostGIS (requêtes spatiales)
- Full-Text Search avancé
- JSONB avec opérateurs spécifiques
- Extensions spécifiques (pg_trgm, hstore)

**Verdict :** SQL brut ou mix ORM + SQL brut.

### 4.3. L'Approche Hybride (Recommandée)

**La vraie réponse :** Utilisez **les deux** !

```python
# Python avec Django

# ✅ ORM pour le CRUD simple
user = User.objects.get(email=email)  
user.last_login = timezone.now()  
user.save()  

# ✅ SQL brut pour les requêtes complexes
from django.db import connection

with connection.cursor() as cursor:
    cursor.execute("""
        SELECT
            u.name,
            COUNT(DISTINCT a.id) as article_count,
            AVG(a.view_count) as avg_views
        FROM users u
        LEFT JOIN articles a ON u.id = a.user_id
        WHERE u.created_at >= %s
        GROUP BY u.id, u.name
        HAVING COUNT(DISTINCT a.id) > 10
        ORDER BY avg_views DESC
        LIMIT 20
    """, [start_date])

    top_users = cursor.fetchall()
```

**Principe :** ORM par défaut, SQL brut quand nécessaire.

## 6. Cas Pratiques : ORM vs SQL Brut

### 6.1. Cas 1 : Afficher la Liste des Utilisateurs

**Besoin :** Afficher tous les utilisateurs actifs avec pagination.

**ORM (Django) :**
```python
from django.core.paginator import Paginator

users = User.objects.filter(is_active=True).order_by('-created_at')  
paginator = Paginator(users, 25)  # 25 par page  
page_obj = paginator.get_page(page_number)  
```

**SQL Brut :**
```python
cursor.execute("""
    SELECT id, name, email, created_at
    FROM users
    WHERE is_active = true
    ORDER BY created_at DESC
    LIMIT %s OFFSET %s
""", [25, (page_number - 1) * 25])

users = cursor.fetchall()
# + Code pour compter le total et calculer les pages
```

**Verdict :** ORM gagne haut la main. Plus simple, plus lisible.

### 6.2. Cas 2 : Rapport Mensuel des Ventes

**Besoin :** CA mensuel par région avec comparaison année précédente.

**ORM (Django) :**
```python
# Possible mais complexe et peu lisible
from django.db.models import Sum, F  
from django.db.models.functions import TruncMonth  

sales = Sale.objects.annotate(
    month=TruncMonth('date')
).values('month', 'region').annotate(
    total=Sum('amount')
).order_by('month', 'region')

# Comparaison année précédente ? Très compliqué en ORM...
```

**SQL Brut :**
```sql
WITH current_year AS (
    SELECT
        DATE_TRUNC('month', date) as month,
        region,
        SUM(amount) as total
    FROM sales
    WHERE EXTRACT(YEAR FROM date) = 2025
    GROUP BY 1, 2
),
previous_year AS (
    SELECT
        DATE_TRUNC('month', date) + INTERVAL '1 year' as month,
        region,
        SUM(amount) as total
    FROM sales
    WHERE EXTRACT(YEAR FROM date) = 2024
    GROUP BY 1, 2
)
SELECT
    c.month,
    c.region,
    c.total as current_total,
    p.total as previous_total,
    ROUND(((c.total - p.total) / p.total * 100)::numeric, 2) as growth_pct
FROM current_year c  
LEFT JOIN previous_year p ON c.month = p.month AND c.region = p.region  
ORDER BY c.month, c.region;  
```

**Verdict :** SQL brut indispensable. L'ORM ne peut pas gérer ça proprement.

### 6.3. Cas 3 : Top 10 Articles par Catégorie

**Besoin :** Les 10 articles les plus vus dans chaque catégorie.

**ORM (très difficile) :**
```python
# Nécessite plusieurs requêtes ou du raw SQL
categories = Category.objects.all()  
result = {}  
for category in categories:  
    top_articles = category.articles.order_by('-view_count')[:10]
    result[category] = list(top_articles)
# → N+1 queries !
```

**SQL Brut (avec Window Function) :**
```sql
WITH ranked_articles AS (
    SELECT
        a.*,
        c.name as category_name,
        ROW_NUMBER() OVER (
            PARTITION BY a.category_id
            ORDER BY a.view_count DESC
        ) as rank
    FROM articles a
    JOIN categories c ON a.category_id = c.id
)
SELECT *  
FROM ranked_articles  
WHERE rank <= 10  
ORDER BY category_name, rank;  
```

**Verdict :** SQL brut. Les window functions sont hors de portée de la plupart des ORM.

### 6.4. Cas 4 : Import Massif de Données

**Besoin :** Importer 10 millions de lignes depuis un CSV.

**ORM :**
```python
# ❌ TRÈS LENT : 1 INSERT par ligne
with open('data.csv') as f:
    for line in f:
        User.objects.create(name=line['name'], email=line['email'])
# Temps : plusieurs heures !

# ⚠️ Mieux : bulk_create (mais limité)
users = []  
with open('data.csv') as f:  
    for line in f:
        users.append(User(name=line['name'], email=line['email']))
        if len(users) >= 1000:
            User.objects.bulk_create(users)
            users = []
# Temps : ~30 minutes
```

**SQL Brut (COPY) :**
```python
cursor.execute("""
    COPY users(name, email)
    FROM '/tmp/data.csv'
    WITH (FORMAT csv, HEADER true);
""")
# Temps : ~2 minutes !
```

**Verdict :** SQL brut avec COPY. 15× plus rapide.

## 7. Bonnes Pratiques et Recommandations

### 7.1. Règle des 80/20

**Principe :**
- **80% de votre code** : ORM (CRUD, relations simples)  
- **20% critique** : SQL brut (performance, complexité)

### 7.2. Stratégie Recommandée

#### Étape 1 : Commencer avec l'ORM

Utilisez l'ORM par défaut pour :
- Définir vos modèles
- CRUD basique
- Relations simples
- Migrations de schéma

#### Étape 2 : Identifier les Hotspots

Avec monitoring (pg_stat_statements, APM) :
- Requêtes lentes (> 100ms)
- Requêtes fréquentes (> 1000/min)
- N+1 queries

#### Étape 3 : Optimiser Sélectivement

Pour les hotspots identifiés :
1. Essayer d'optimiser avec l'ORM (`select_related`, `prefetch_related`, `only()`)  
2. Si impossible ou insuffisant → SQL brut  
3. Mesurer l'amélioration avec EXPLAIN ANALYZE

#### Étape 4 : Documenter les Exceptions

```python
# users/queries.py

def get_user_stats_report(start_date, end_date):
    """
    Requête SQL brute pour performance.

    L'ORM génère un plan suboptimal avec trop de subqueries.
    SQL brut : 80ms vs ORM : 450ms

    Dernière révision : 2025-11-23
    """
    with connection.cursor() as cursor:
        cursor.execute(RAW_USER_STATS_QUERY, [start_date, end_date])
        return cursor.fetchall()
```

### 7.3. Checklist de Décision

Avant d'utiliser SQL brut, demandez-vous :

- [ ] L'ORM génère-t-il vraiment du SQL inefficace ? (vérifier avec logging)  
- [ ] Ai-je essayé toutes les optimisations ORM ? (`select_related`, `annotate`, etc.)  
- [ ] Le gain de performance justifie-t-il la perte de lisibilité ?  
- [ ] Puis-je isoler ce SQL dans une fonction dédiée et documentée ?  
- [ ] Ai-je testé et mesuré l'amélioration ?

### 7.4. Anti-Patterns à Éviter

#### ❌ Anti-Pattern 1 : "ORM Partout"

```python
# ❌ Forcer l'ORM pour des requêtes complexes
result = User.objects.annotate(
    article_count=Count('articles')
).annotate(
    avg_views=Avg('articles__view_count')
).filter(
    article_count__gte=10
).annotate(
    rank=Window(
        expression=RowNumber(),
        order_by=F('avg_views').desc()
    )
)
# Code illisible et SQL généré peut-être inefficace
```

**✅ Solution :** SQL brut pour cette requête complexe.

#### ❌ Anti-Pattern 2 : "SQL Brut Partout"

```python
# ❌ Utiliser SQL brut pour du CRUD simple
cursor.execute("SELECT * FROM users WHERE id = %s", [user_id])  
row = cursor.fetchone()  
user = {  
    'id': row[0],
    'name': row[1],
    'email': row[2]
}
```

**✅ Solution :** ORM est parfait pour ça.
```python
user = User.objects.get(id=user_id)
```

#### ❌ Anti-Pattern 3 : "Mix Chaotique"

```python
# ❌ Mélanger ORM et SQL sans logique
def get_user_articles(user_id):
    cursor.execute("SELECT * FROM users WHERE id = %s", [user_id])
    user = cursor.fetchone()

    articles = Article.objects.filter(user_id=user_id)

    cursor.execute("SELECT COUNT(*) FROM comments WHERE user_id = %s", [user_id])
    comment_count = cursor.fetchone()[0]

    # Pourquoi ce mélange ?
```

**✅ Solution :** Cohérence. Tout en ORM ou tout en SQL pour une fonction donnée.

### 7.5. Organisation du Code

**Structure recommandée :**

```
myapp/
├── models.py          # Définitions ORM
├── queries.py         # SQL brut complexe
├── views.py           # Logique métier (utilise models et queries)
└── tests/
    ├── test_models.py
    └── test_queries.py
```

**Exemple queries.py :**
```python
# myapp/queries.py

from django.db import connection

def get_monthly_sales_report(year):
    """
    Rapport mensuel des ventes avec comparaison année précédente.

    Utilise SQL brut car nécessite des CTE et window functions
    que Django ORM ne gère pas élégamment.
    """
    query = """
        WITH current_year AS (
            SELECT DATE_TRUNC('month', date) as month, ...
        )
        SELECT * FROM current_year ...
    """

    with connection.cursor() as cursor:
        cursor.execute(query, [year])
        return cursor.fetchall()
```

## 8. ORM Avancés : Query Builders

### 8.1. Qu'est-ce qu'un Query Builder ?

**Définition :** Un compromis entre ORM et SQL brut.

**Caractéristiques :**
- API fluide pour construire des requêtes
- Génère du SQL mais avec contrôle fin
- Moins de "magie" qu'un ORM complet

**Exemples :**
- **Knex.js** (JavaScript)  
- **jOOQ** (Java)  
- **Ecto** (Elixir)

### 8.2. Exemple avec Knex.js

```javascript
// Query Builder (Knex.js)
const users = await knex('users')
  .select('users.name', 'users.email')
  .count('articles.id as article_count')
  .leftJoin('articles', 'users.id', 'articles.user_id')
  .where('users.is_active', true)
  .groupBy('users.id')
  .having('article_count', '>', 5)
  .orderBy('article_count', 'desc');

// Équivalent SQL généré
// SELECT users.name, users.email, COUNT(articles.id) as article_count
// FROM users
// LEFT JOIN articles ON users.id = articles.user_id
// WHERE users.is_active = true
// GROUP BY users.id
// HAVING COUNT(articles.id) > 5
// ORDER BY article_count DESC
```

**Avantages :**
- Plus flexible qu'un ORM classique
- Moins verbeux que du SQL brut
- Toujours type-safe (en TypeScript)

### 8.3. Quand Utiliser un Query Builder ?

**✅ Bon choix si :**
- Vous voulez le contrôle du SQL brut
- Mais avec une API plus ergonomique
- Et des garanties de type (TypeScript, Scala)

**❌ Moins bon si :**
- Vous voulez la simplicité totale d'un ORM
- Ou le contrôle absolu du SQL brut

## 10. Migration : ORM → SQL Brut ou Inverse

### 10.1. Migrer d'ORM vers SQL Brut (Optimisation)

**Étapes :**

1. **Identifier la requête ORM lente**
   ```python
   # Requête ORM actuelle
   users = User.objects.annotate(
       article_count=Count('articles')
   ).filter(article_count__gte=10)
   ```

2. **Activer le logging et récupérer le SQL généré**
   ```sql
   SELECT users.*, COUNT(articles.id) as article_count
   FROM users
   LEFT JOIN articles ON ...
   GROUP BY users.id
   HAVING COUNT(articles.id) >= 10
   ```

3. **Optimiser le SQL**
   ```sql
   -- Version optimisée avec index
   SELECT u.id, u.name, u.email, COUNT(a.id) as article_count
   FROM users u
   LEFT JOIN articles a ON u.id = a.user_id
   WHERE u.is_active = true
   GROUP BY u.id
   HAVING COUNT(a.id) >= 10
   ORDER BY article_count DESC;
   ```

4. **Remplacer par du SQL brut**
   ```python
   from django.db import connection

   def get_prolific_users(min_articles=10):
       with connection.cursor() as cursor:
           cursor.execute(OPTIMIZED_QUERY, [min_articles])
           return cursor.fetchall()
   ```

5. **Mesurer l'amélioration**
   - Avant : 450ms
   - Après : 80ms
   - Gain : 82%

### 10.2. Migrer de SQL Brut vers ORM (Simplification)

**Quand :** Si le SQL brut est devenu du boilerplate pour des requêtes simples.

**Avant (SQL brut) :**
```python
cursor.execute("SELECT * FROM users WHERE is_active = %s ORDER BY created_at DESC", [True])  
users = cursor.fetchall()  
# + Code de mapping vers des objets
```

**Après (ORM) :**
```python
users = User.objects.filter(is_active=True).order_by('-created_at')
```

**Gain :** Lisibilité, maintenabilité.

## 11. Cas Particuliers et Niches

### 11.1. Stored Procedures et Functions

**Scénario :** Logique métier complexe en base de données.

```sql
CREATE FUNCTION calculate_user_score(user_id INT)  
RETURNS DECIMAL AS $$  
DECLARE  
    score DECIMAL;
BEGIN
    SELECT
        (article_count * 10) + (comment_count * 2) + (like_count * 0.5)
    INTO score
    FROM user_stats
    WHERE id = user_id;

    RETURN COALESCE(score, 0);
END;
$$ LANGUAGE plpgsql;
```

**Appel depuis ORM vs SQL Brut :**

```python
# SQL Brut (direct)
cursor.callproc('calculate_user_score', [user_id])  
score = cursor.fetchone()[0]  

# ORM (Django) : nécessite du raw SQL
from django.db import connection  
score = connection.cursor().callproc('calculate_user_score', [user_id])  
```

**Verdict :** Les stored procedures nécessitent souvent du SQL brut.

### 11.2. Transactions Complexes

**Scénario :** Transaction avec logique conditionnelle et rollback partiel.

**Avec ORM (Django) :**
```python
from django.db import transaction

try:
    with transaction.atomic():
        user = User.objects.create(name="Alice")
        account = Account.objects.create(user=user, balance=100)

        if account.balance < 0:
            raise ValueError("Balance cannot be negative")

        # Si exception : rollback automatique
except ValueError:
    # Gestion d'erreur
    pass
```

**Avec SQL Brut :**
```python
try:
    cursor.execute("BEGIN;")
    cursor.execute("INSERT INTO users (name) VALUES (%s) RETURNING id", ["Alice"])
    user_id = cursor.fetchone()[0]

    cursor.execute("INSERT INTO accounts (user_id, balance) VALUES (%s, %s)", [user_id, 100])

    cursor.execute("SELECT balance FROM accounts WHERE user_id = %s", [user_id])
    balance = cursor.fetchone()[0]

    if balance < 0:
        cursor.execute("ROLLBACK;")
        raise ValueError("Balance cannot be negative")

    cursor.execute("COMMIT;")
except Exception as e:
    cursor.execute("ROLLBACK;")
    raise
```

**Verdict :** ORM plus simple pour les transactions standards.

### 11.3. Avec des Extensions PostgreSQL (PostGIS, pg_trgm)

**Exemple avec PostGIS (requêtes spatiales) :**

```sql
-- Trouver les restaurants dans un rayon de 5km
SELECT name, ST_Distance(location, ST_Point(-122.4194, 37.7749)) as distance  
FROM restaurants  
WHERE ST_DWithin(location, ST_Point(-122.4194, 37.7749), 5000)  
ORDER BY distance;  
```

**Avec ORM (Django + GeoDjango) :**
```python
from django.contrib.gis.geos import Point  
from django.contrib.gis.measure import D  

user_location = Point(-122.4194, 37.7749, srid=4326)  
restaurants = Restaurant.objects.filter(  
    location__distance_lte=(user_location, D(km=5))
).annotate(
    distance=Distance('location', user_location)
).order_by('distance')
```

**Verdict :** Les ORM spécialisés (GeoDjango) gèrent bien les extensions. Sinon, SQL brut.

## 12. Résumé et Synthèse

### 12.1. Les Grandes Vérités

1. **Il n'y a pas de "meilleur" choix absolu** : tout dépend du contexte  
2. **Les ORM sont parfaits pour 80% des cas** : CRUD, relations simples  
3. **Le SQL brut est indispensable pour 20% critique** : performance, complexité  
4. **Connaître SQL est obligatoire** même avec un ORM (pour débugger, optimiser)  
5. **La vraie compétence : savoir quand utiliser quoi**

### 12.2. Principes de Décision

**Utilisez l'ORM par défaut, sauf si :**
- ❌ Requête trop complexe (CTE, window functions)  
- ❌ Performance critique identifiée  
- ❌ Opérations en masse (bulk)  
- ❌ Extensions PostgreSQL spécifiques

**Revenez au SQL brut quand :**
- ✅ EXPLAIN montre un plan inefficace  
- ✅ Temps de réponse > 100ms sur hotspot  
- ✅ ORM génère N+1 non corrigeable  
- ✅ Requête analytique complexe

### 12.3. Matrice de Décision Finale

| Type de Requête | Recommandation | Raison |
|-----------------|----------------|--------|
| CRUD simple (1 table) | **ORM** | Productivité maximale |
| Relations 1-1 ou N-1 | **ORM** | `select_related` suffit |
| Listes paginées | **ORM** | Gestion native pagination |
| Recherche full-text simple | **ORM** | Bonne abstraction |
| Dashboard avec agrégations | **SQL Brut** | Requêtes analytiques |
| Top N par groupe | **SQL Brut** | Window functions |
| Import/export masse | **SQL Brut** | COPY natif |
| Migrations complexes | **SQL Brut** | Contrôle total nécessaire |
| API haute fréquence | **SQL Brut** | Performance critique |
| Prototypage rapide | **ORM** | Itération rapide |

### 12.4. Checklist Finale

Avant de choisir, répondez à ces questions :

**Questions ORM :**
- [ ] Mon équipe connaît-elle bien cet ORM ?  
- [ ] La requête est-elle un CRUD simple ?  
- [ ] Les performances actuelles sont-elles acceptables ?  
- [ ] La lisibilité du code ORM est-elle bonne ?

**Questions SQL Brut :**
- [ ] Ai-je vraiment besoin de features PostgreSQL avancées ?  
- [ ] Les performances ORM sont-elles insuffisantes ?  
- [ ] Puis-je documenter et maintenir ce SQL ?  
- [ ] Le gain justifie-t-il la complexité ajoutée ?
