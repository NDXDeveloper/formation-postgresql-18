# 20.2.2. PgBouncer : Transaction vs Session Pooling

## Introduction

Dans la section précédente (20.2.1), nous avons vu le **connection pooling côté application**. Maintenant, nous allons découvrir **PgBouncer**, un outil de pooling **externe** qui se place entre votre application et PostgreSQL.

### Pourquoi un pooling externe ?

Le pooling côté application a des limites :
- Chaque instance d'application a son propre pool
- En microservices : multiplication des pools
- Difficile de gérer globalement les connexions

**PgBouncer** résout ces problèmes en centralisant le pooling.

---

## Qu'est-ce que PgBouncer ?

### Définition

**PgBouncer** est un **connection pooler léger** qui agit comme un **proxy** entre vos applications et PostgreSQL.

### Architecture

```
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│             │      │             │      │             │
│ Application │─────▶│  PgBouncer  │─────▶│ PostgreSQL  │
│             │      │   (Proxy)   │      │             │
└─────────────┘      └─────────────┘      └─────────────┘
   1000 clients         100 connexions       10 connexions
                        virtuelles           réelles
```

**Analogie** : PgBouncer est comme un standard téléphonique :
- 1000 téléphones (clients) peuvent appeler
- Le standard n'a que 10 lignes sortantes (connexions PostgreSQL)
- Il gère intelligemment qui obtient une ligne et quand

### Avantages de PgBouncer

- ✅ **Centralisation** : Un seul point de pooling pour toutes les applications  
- ✅ **Efficacité** : Réutilisation maximale des connexions PostgreSQL  
- ✅ **Scalabilité** : Supporte des milliers de connexions clientes avec peu de connexions serveur  
- ✅ **Léger** : Consomme très peu de ressources (RAM, CPU)  
- ✅ **Transparent** : L'application ne voit pas la différence

## Les trois modes de pooling

PgBouncer propose **trois modes** de pooling, chacun avec des caractéristiques différentes.

### Vue d'ensemble

| Mode | Granularité | Performance | Compatibilité | Cas d'usage |
|------|-------------|-------------|---------------|-------------|
| **Session** | Par session | ⭐⭐ | ⭐⭐⭐⭐⭐ | Applications complexes |
| **Transaction** | Par transaction | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | Applications web/API |
| **Statement** | Par requête | ⭐⭐⭐⭐⭐ | ⭐ | Cas très spécifiques |

Nous allons nous concentrer sur les deux modes principaux : **Session** et **Transaction**.

---

## Mode Session Pooling

### Principe de fonctionnement

En mode **session**, une connexion PostgreSQL est attribuée à un client **pour toute la durée de sa session**.

```
Client connecte → Obtient une connexion PostgreSQL
                → Garde cette connexion jusqu'à déconnexion
                → La connexion retourne au pool
```

**Analogie** : Comme louer une voiture pour toute la journée. Vous l'utilisez ou non, elle reste à vous jusqu'à ce que vous la rendiez le soir.

### Diagramme

```
┌──────────────────────────────────────────────────────────┐
│ Session du Client (jusqu'à déconnexion)                  │
│                                                          │
│  BEGIN; SELECT...; UPDATE...; COMMIT;                    │
│  BEGIN; DELETE...; COMMIT;                               │
│  SELECT...;                                              │
│  (pause de 10 secondes)                                  │
│  INSERT...;                                              │
│                                                          │
└──────────────────────────────────────────────────────────┘
         ↕ (même connexion PostgreSQL tout le long)
┌──────────────────────────────────────────────────────────┐
│ Connexion PostgreSQL dédiée                              │
└──────────────────────────────────────────────────────────┘
```

### Comportement

**Quand le client se connecte** :
1. PgBouncer attribue une connexion PostgreSQL du pool  
2. Le client garde cette connexion

**Quand le client se déconnecte** :
1. PgBouncer libère la connexion PostgreSQL  
2. La connexion retourne au pool pour un autre client

### Avantages

- ✅ **Compatibilité maximale** : Supporte toutes les fonctionnalités PostgreSQL  
- ✅ **Variables de session** : Conservées (SET search_path, SET timezone, etc.)  
- ✅ **Transactions longues** : Pas de problème  
- ✅ **Prepared statements** : Fonctionnent parfaitement  
- ✅ **LISTEN/NOTIFY** : Supporté  
- ✅ **Advisory locks** : Supporté

### Inconvénients

- ❌ **Efficacité limitée** : Une connexion par client connecté  
- ❌ **Scalabilité réduite** : Si 100 clients connectés = 100 connexions PostgreSQL  
- ❌ **Gaspillage** : Clients inactifs gardent leur connexion

### Configuration

**Fichier pgbouncer.ini** :
```ini
[databases]
mydb = host=localhost port=5432 dbname=mydb

[pgbouncer]
listen_addr = 0.0.0.0  
listen_port = 6432  
auth_type = md5  
auth_file = /etc/pgbouncer/userlist.txt  

# MODE SESSION
pool_mode = session

# Taille du pool
default_pool_size = 25        # 25 connexions PostgreSQL par base  
max_client_conn = 1000        # Maximum 1000 clients connectés  

# Timeouts
server_idle_timeout = 600     # Fermer connexions inactives après 10 min  
server_lifetime = 3600        # Renouveler après 1h  
```

### Cas d'usage

**✅ Utiliser le mode session si** :
- Vous utilisez des **prepared statements** nommés
- Vous utilisez **LISTEN/NOTIFY**
- Vous utilisez des **advisory locks**
- Vous modifiez des **variables de session** (SET commands)
- Vous avez des **transactions longues**
- Vous utilisez des **temporary tables**

**Exemple typique** : Application batch avec transactions longues

## Mode Transaction Pooling

### Principe de fonctionnement

En mode **transaction**, une connexion PostgreSQL est attribuée à un client **seulement pendant la durée d'une transaction**.

```
Client: BEGIN;           → Obtient connexion PostgreSQL  
Client: SELECT...;       → Utilise la connexion  
Client: UPDATE...;       → Utilise la connexion  
Client: COMMIT;          → Libère la connexion immédiatement  
Client: (fait autre chose) → Pas de connexion PostgreSQL  
Client: BEGIN;           → Obtient connexion (peut-être différente)  
Client: INSERT...;       → Utilise la connexion  
Client: COMMIT;          → Libère la connexion  
```

**Analogie** : Comme prendre un taxi. Vous prenez un taxi pour un trajet, descendez, et quelqu'un d'autre peut prendre le même taxi immédiatement après.

### Diagramme

```
Client                      PgBouncer               PostgreSQL

BEGIN;                     ─────────▶ Attribue Connexion #1  
SELECT...;                                         Connexion #1  
UPDATE...;                                         Connexion #1  
COMMIT;                    ─────────▶ Libère Connexion #1  

(pas de transaction)       (pas de connexion)
SELECT...; (autocommit)    ─────────▶ Attribue Connexion #2
                           ─────────▶ Libère Connexion #2

BEGIN;                     ─────────▶ Attribue Connexion #3  
INSERT...;                                         Connexion #3  
COMMIT;                    ─────────▶ Libère Connexion #3  
```

### Comportement

**Pendant une transaction** :
1. Le client exécute `BEGIN;`  
2. PgBouncer attribue une connexion PostgreSQL  
3. Toutes les commandes de la transaction utilisent cette connexion  
4. Le client exécute `COMMIT;` ou `ROLLBACK;`  
5. **Immédiatement**, PgBouncer libère la connexion

**Entre les transactions** :
- Le client n'a **pas** de connexion PostgreSQL
- La connexion est disponible pour d'autres clients
- Maximum de réutilisation

### Avantages

- ✅ **Efficacité maximale** : 10 connexions PostgreSQL peuvent servir 1000 clients  
- ✅ **Scalabilité** : Supporte énormément de clients simultanés  
- ✅ **Performance** : Les connexions sont constamment utilisées  
- ✅ **Économie de ressources** : Minimal de connexions côté PostgreSQL

### Inconvénients

- ❌ **Variables de session perdues** : `SET` commands réinitialisés entre transactions  
- ❌ **Prepared statements** : Seulement les unnamed (automatiques)  
- ❌ **LISTEN/NOTIFY** : Non supporté  
- ❌ **Advisory locks** : Non supporté  
- ❌ **Temporary tables** : Ne survivent pas entre transactions

### Configuration

**Fichier pgbouncer.ini** :
```ini
[databases]
mydb = host=localhost port=5432 dbname=mydb

[pgbouncer]
listen_addr = 0.0.0.0  
listen_port = 6432  
auth_type = md5  
auth_file = /etc/pgbouncer/userlist.txt  

# MODE TRANSACTION
pool_mode = transaction

# Taille du pool (peut être très petit)
default_pool_size = 10        # Seulement 10 connexions PostgreSQL !  
max_client_conn = 5000        # Peut servir 5000 clients  

# Timeouts
server_idle_timeout = 600  
server_lifetime = 3600  

# Recommandé en mode transaction
server_reset_query = DISCARD ALL
```

### Cas d'usage

**✅ Utiliser le mode transaction si** :
- Vous avez une **application web/API REST** classique
- Chaque requête HTTP = 1 ou plusieurs transactions courtes
- Vous n'utilisez **pas** de variables de session
- Vous n'utilisez **pas** de prepared statements nommés
- Vous avez besoin de **haute scalabilité**
- Vous voulez **minimiser** les connexions PostgreSQL

**Exemple typique** : API REST avec Node.js/Express, Python/FastAPI, Java/Spring Boot

## Comparaison détaillée : Session vs Transaction

### Tableau comparatif

| Fonctionnalité | Session | Transaction |
|----------------|---------|-------------|
| **Transactions longues** | ✅ Supporté | ❌ Déconseillé |
| **SET variables** | ✅ Persistées | ❌ Perdues |
| **Prepared statements (named)** | ✅ Supporté | ❌ Non supporté |
| **Prepared statements (unnamed)** | ✅ Supporté | ✅ Supporté |
| **LISTEN/NOTIFY** | ✅ Supporté | ❌ Non supporté |
| **Advisory locks** | ✅ Supporté | ❌ Non supporté |
| **Temporary tables** | ✅ Persistées | ❌ Perdues |
| **DISCARD ALL** | Pas nécessaire | ✅ Recommandé |
| **Ratio client/connexion** | 1:1 ou 2:1 | 100:1 ou plus |
| **Scalabilité** | ⭐⭐ Moyenne | ⭐⭐⭐⭐⭐ Excellente |
| **Efficacité** | ⭐⭐ Moyenne | ⭐⭐⭐⭐⭐ Excellente |
| **Complexité** | ⭐ Simple | ⭐⭐⭐ Modérée |

### Exemple concret : API REST

#### Scénario

Une API REST qui :
- Reçoit 500 requêtes/seconde
- Chaque requête fait 1-5 transactions SQL
- Temps moyen de traitement : 50ms

#### Avec Session Pooling

```
500 requêtes/sec × 0.05 sec = 25 requêtes simultanées en moyenne

→ Besoin d'au moins 25-30 connexions PostgreSQL
→ En pics : peut-être 50-100 connexions
```

**Configuration** :
```ini
pool_mode = session  
default_pool_size = 50  
max_client_conn = 200  
```

#### Avec Transaction Pooling

```
Chaque transaction dure ~10ms (pas 50ms, car temps sans connexion)
500 req/sec × 2 transactions × 0.01 sec = 10 transactions simultanées

→ Besoin d'environ 10 connexions PostgreSQL !
```

**Configuration** :
```ini
pool_mode = transaction  
default_pool_size = 10  
max_client_conn = 2000  
```

**Résultat** :
- Transaction pooling = **5× moins de connexions PostgreSQL**
- Peut gérer **10× plus de clients**
- Performance équivalente ou meilleure

## Configuration complète de PgBouncer

### Fichier pgbouncer.ini complet

```ini
;; Fichier de configuration PgBouncer
;; Emplacement typique : /etc/pgbouncer/pgbouncer.ini

;;
;; Section [databases] : Définir les bases de données
;;
[databases]
; Format: dbname = paramètres de connexion
myapp_db = host=127.0.0.1 port=5432 dbname=myapp user=myapp_user  
analytics_db = host=10.0.1.5 port=5432 dbname=analytics  

; Base spéciale : * = configuration par défaut pour toutes les bases
; * = host=127.0.0.1 port=5432

;;
;; Section [pgbouncer] : Configuration principale
;;
[pgbouncer]

;;;
;;; Configuration réseau
;;;

; Adresse d'écoute (0.0.0.0 = toutes les interfaces)
listen_addr = 0.0.0.0

; Port d'écoute (par défaut : 6432)
listen_port = 6432

; Socket Unix (optionnel, pour connexions locales)
unix_socket_dir = /var/run/postgresql

;;;
;;; Authentification
;;;

; Type d'authentification
; Valeurs possibles: md5, scram-sha-256, plain, trust, cert
auth_type = scram-sha-256

; Fichier contenant les utilisateurs et mots de passe
auth_file = /etc/pgbouncer/userlist.txt

; Requête pour obtenir les mots de passe depuis PostgreSQL (optionnel)
; auth_query = SELECT usename, passwd FROM pg_shadow WHERE usename=$1

;;;
;;; Mode de pooling (CRITIQUE)
;;;

; Mode: session, transaction, ou statement
pool_mode = transaction

;;;
;;; Configuration du pool
;;;

; Taille par défaut du pool par base de données
default_pool_size = 10

; Taille minimale du pool (connexions toujours maintenues)
min_pool_size = 2

; Pool de réserve (connexions supplémentaires si nécessaire)
reserve_pool_size = 5

; Délai avant d'utiliser le reserve pool (secondes)
reserve_pool_timeout = 5

;;;
;;; Limites de connexions
;;;

; Nombre maximum de connexions clientes
max_client_conn = 1000

; Nombre maximum de connexions par utilisateur
; max_user_connections = 100

; Nombre maximum de connexions par base de données
; max_db_connections = 50

;;;
;;; Timeouts (en secondes)
;;;

; Timeout pour obtenir une connexion serveur du pool
server_connect_timeout = 15

; Temps d'inactivité avant de fermer une connexion serveur
server_idle_timeout = 600

; Durée de vie maximale d'une connexion serveur (secondes)
server_lifetime = 3600

; Temps d'inactivité avant de fermer une connexion client
client_idle_timeout = 0

; Timeout pour une requête (0 = désactivé)
query_timeout = 0

; Timeout d'attente client pour obtenir connexion
query_wait_timeout = 120

;;;
;;; Comportement en mode transaction
;;;

; Commande exécutée lors de la libération d'une connexion
; DISCARD ALL nettoie : prepared statements, LISTEN, temp tables, etc.
server_reset_query = DISCARD ALL

; Alternative moins stricte (plus rapide)
; server_reset_query = RESET ALL

;;;
;;; Logging
;;;

; Emplacement des logs
logfile = /var/log/pgbouncer/pgbouncer.log

; Niveau de log: debug, info, warning, error
log_connections = 1  
log_disconnections = 1  
log_pooler_errors = 1  

; Statistiques
stats_period = 60

;;;
;;; Sécurité
;;;

; Utilisateur système sous lequel PgBouncer s'exécute
; user = pgbouncer

; Ignorer les erreurs de démarrage pour certaines bases
ignore_startup_parameters = extra_float_digits

;;;
;;; Administration
;;;

; Base de données d'administration (virtuelle)
; Connexion: psql -h localhost -p 6432 -U admin pgbouncer
admin_users = admin, pgbouncer

; Utilisateurs avec privilèges de stats
stats_users = admin, monitoring
```

### Fichier userlist.txt

Le fichier `userlist.txt` contient les utilisateurs et leurs mots de passe :

```txt
; Format: "username" "mot_de_passe_hashé"

; Pour md5: md5 + md5(password + username)
"myapp_user" "md5d8578edf8458ce06fbc5bb76a58c5ca4"

; Pour scram-sha-256: utiliser SCRAM-SHA-256$<iterations>:<salt>$<StoredKey>:<ServerKey>
"api_user" "SCRAM-SHA-256$4096:rGbSEfZ+..."

; Pour plain text (NON RECOMMANDÉ en production)
; "dev_user" "plain_password_123"
```

**Génération d'un hash md5** :
```bash
# Formule: md5(password + username)
echo -n "mypasswordmyapp_user" | md5sum
# Ajouter "md5" devant le résultat
```

**Meilleure pratique** : Utiliser `auth_query` pour récupérer les mots de passe depuis PostgreSQL.

```bash
# Test de connexion via PgBouncer
```

```sql
-- Connexions en attente (signe de sous-dimensionnement)
SELECT * FROM pgbouncer.show_pools WHERE cl_waiting > 0;

-- Utilisation du pool
SELECT
    database,
    cl_active,
    sv_active,
    sv_idle,
    (sv_active::float / (sv_active + sv_idle)) * 100 as utilization_pct
FROM pgbouncer.show_pools;
```

## Cas pratiques et recommandations

### Cas 1 : API REST avec microservices

**Contexte** :
- 5 microservices Node.js
- Chaque service a 3 instances (total : 15 instances)
- Chaque instance sans PgBouncer aurait un pool de 10 connexions

**Sans PgBouncer** :
```
15 instances × 10 connexions = 150 connexions PostgreSQL
```

**Avec PgBouncer (transaction mode)** :
```ini
pool_mode = transaction  
default_pool_size = 20  
max_client_conn = 500  

→ Seulement 20 connexions PostgreSQL pour 500 clients possibles !
→ Économie : 150 → 20 = 86% de connexions en moins
```

**Configuration recommandée** :
```ini
[databases]
myapp = host=postgres-server port=5432 dbname=myapp

[pgbouncer]
pool_mode = transaction  
default_pool_size = 20  
max_client_conn = 500  
server_reset_query = DISCARD ALL  
server_idle_timeout = 300  
server_lifetime = 3600  
```

### Cas 2 : Application legacy avec prepared statements

**Contexte** :
- Application Java avec JDBC
- Utilise des prepared statements nommés
- Transactions longues (plusieurs secondes)

**Configuration recommandée** :
```ini
[databases]
legacy_db = host=postgres-server port=5432 dbname=legacy

[pgbouncer]
pool_mode = session        # MODE SESSION obligatoire  
default_pool_size = 50     # Plus de connexions nécessaires  
max_client_conn = 200  
server_idle_timeout = 1800 # 30 min (transactions longues)  
server_lifetime = 7200     # 2h  
```

### Cas 3 : Application mixte (web + batch)

**Contexte** :
- Partie web : transactions courtes
- Partie batch : transactions longues

**Solution** : **Deux instances de PgBouncer**

**PgBouncer Web (port 6432)** :
```ini
[databases]
webapp = host=postgres-server port=5432 dbname=myapp

[pgbouncer]
listen_port = 6432  
pool_mode = transaction  
default_pool_size = 10  
```

**PgBouncer Batch (port 6433)** :
```ini
[databases]
batch = host=postgres-server port=5432 dbname=myapp

[pgbouncer]
listen_port = 6433  
pool_mode = session  
default_pool_size = 5  
```

**Connexions applicatives** :
```python
# Web app
web_pool = ConnectionPool("postgresql://user:pass@localhost:6432/webapp")

# Batch jobs
batch_conn = psycopg.connect("postgresql://user:pass@localhost:6433/batch")
```

### Cas 4 : Serverless (AWS Lambda, Cloud Functions)

**Problème** :
- Chaque invocation Lambda = nouvelle connexion
- Saturation rapide de PostgreSQL

**Solution avec PgBouncer** :
```
Lambda → PgBouncer (EC2/ECS) → RDS PostgreSQL
```

**Configuration PgBouncer** :
```ini
[databases]
lambda_db = host=my-rds.amazonaws.com port=5432 dbname=mydb

[pgbouncer]
pool_mode = transaction  
default_pool_size = 5      # Très petit pool  
max_client_conn = 1000     # Beaucoup de Lambdas  
server_reset_query = DISCARD ALL  
server_idle_timeout = 60   # Courte durée  
```

**Alternative** : AWS RDS Proxy (équivalent managé de PgBouncer)

## Monitoring et troubleshooting

### Métriques à surveiller

#### 1. Clients en attente (cl_waiting)

```sql
-- Voir les clients en attente
SHOW POOLS;
```

**Problème si** : `cl_waiting > 0` de manière récurrente

**Solutions** :
- Augmenter `default_pool_size`
- Augmenter `reserve_pool_size`
- Vérifier les slow queries côté PostgreSQL

#### 2. Utilisation du pool

```sql
-- Taux d'utilisation
SELECT
    database,
    user,
    sv_active,
    sv_idle,
    sv_used,
    ROUND((sv_active::float / (sv_active + sv_idle + sv_used)) * 100, 2) as utilization
FROM pgbouncer.show_pools  
WHERE database != 'pgbouncer';  
```

**Sain** : 60-80% d'utilisation en charge normale  
**Problème** : >90% constamment → Pool trop petit  

#### 3. Transactions longues en mode transaction

```sql
-- Connexions avec transactions longues (> 1 sec)
SHOW SERVERS;
```

En mode `transaction`, si vous voyez des connexions actives pendant plusieurs secondes, c'est suspect.

**Solution** :
- Optimiser les requêtes lentes
- Considérer le mode `session` si nécessaire

#### 4. Statistiques globales

```sql
-- Performance globale
SHOW STATS;
```

**Colonnes importantes** :
- `avg_xact_time` : Temps moyen de transaction (doit être <100ms)  
- `avg_query_time` : Temps moyen de requête  
- `avg_wait_time` : Temps d'attente pour obtenir une connexion (doit être ~0)

### Problèmes courants

#### Problème 1 : "no more connections allowed"

**Symptôme** :
```
FATAL: no more connections allowed (max_client_conn)
```

**Cause** : Trop de clients essaient de se connecter

**Solution** :
```ini
# Augmenter la limite
max_client_conn = 2000  # Au lieu de 1000
```

#### Problème 2 : "sorry, too many clients already"

**Symptôme** : Erreur venant de PostgreSQL

**Cause** : PgBouncer envoie trop de connexions à PostgreSQL

**Solution** :
```ini
# Réduire le pool côté PgBouncer
default_pool_size = 10  # Au lieu de 50

# Ou augmenter max_connections côté PostgreSQL (dans postgresql.conf)
max_connections = 200
```

#### Problème 3 : Transactions qui échouent en mode transaction

**Symptôme** :
```
ERROR: prepared statement "..." does not exist  
ERROR: cursor "..." does not exist  
```

**Cause** : Utilisation de features incompatibles avec le mode transaction

**Solution** :
- Passer en mode `session`
- Ou modifier le code pour éviter prepared statements nommés

#### Problème 4 : Performance dégradée après passage à PgBouncer

**Causes possibles** :
1. Pool trop petit → Augmenter `default_pool_size`  
2. `server_reset_query` trop lourd → Utiliser `RESET ALL` au lieu de `DISCARD ALL`  
3. Latence réseau ajoutée → Déployer PgBouncer sur le même serveur que l'app

## PgBouncer vs Pooling applicatif

### Quand utiliser les deux ?

**Configuration optimale** :
```
Application (pool=5) → PgBouncer (pool=20) → PostgreSQL
```

**Avantages de la combinaison** :
- **Application** : Réduit la latence (connexions locales)  
- **PgBouncer** : Mutualise les connexions entre toutes les applications

### Dimensionnement combiné

**Règle** :
```
pool_size_app = 5-10 connexions par instance  
pool_size_pgbouncer = nb_instances_app × 2  

Exemple:
- 10 instances d'application
- Pool app = 5
- Pool PgBouncer = 10 × 2 = 20
```

## Haute disponibilité avec PgBouncer

### Architecture HA

```
                      ┌─────────────────┐
                      │   Load Balancer │
                      │   (HAProxy)     │
                      └────────┬────────┘
                               │
                ┌──────────────┼──────────────┐
                │              │              │
         ┌──────▼─────┐  ┌─────▼──────┐ ┌─────▼──────┐
         │ PgBouncer 1│  │ PgBouncer 2│ │ PgBouncer 3│
         └──────┬─────┘  └─────┬──────┘ └─────┬──────┘
                │              │              │
                └──────────────┼──────────────┘
                               │
                      ┌────────▼────────┐
                      │   PostgreSQL    │
                      │   Primary       │
                      └─────────────────┘
```

### Configuration HAProxy pour PgBouncer

```conf
global
    maxconn 10000

defaults
    mode tcp
    timeout connect 5s
    timeout client 30s
    timeout server 30s

# Frontend pour les connexions PostgreSQL
frontend postgres_frontend
    bind *:5432
    default_backend pgbouncer_backend

# Backend avec plusieurs PgBouncer
backend pgbouncer_backend
    balance roundrobin
    option tcp-check

    server pgbouncer1 10.0.1.10:6432 check
    server pgbouncer2 10.0.1.11:6432 check
    server pgbouncer3 10.0.1.12:6432 check
```

## Résumé : Points clés à retenir

### ✨ Quand utiliser quel mode

| Votre cas | Mode recommandé | Pourquoi |
|-----------|----------------|----------|
| API REST/Web moderne | **Transaction** | Scalabilité maximale |
| Microservices | **Transaction** | Mutualisation des connexions |
| Application avec prepared statements | **Session** | Compatibilité complète |
| Transactions longues (>1s) | **Session** | Évite les timeouts |
| Batch jobs | **Session** | Flexibilité nécessaire |
| Serverless (Lambda) | **Transaction** | Gestion des pics |

### 🎯 Configuration type : API REST

```ini
[pgbouncer]
pool_mode = transaction  
default_pool_size = 10  
max_client_conn = 2000  
server_reset_query = DISCARD ALL  
server_idle_timeout = 600  
```

### 🎯 Configuration type : Application legacy

```ini
[pgbouncer]
pool_mode = session  
default_pool_size = 50  
max_client_conn = 200  
server_idle_timeout = 1800  
```

### 💡 Bonnes pratiques

1. **Commencer petit** : `default_pool_size = 10`, puis ajuster  
2. **Monitorer** : Surveiller `cl_waiting` et `avg_wait_time`  
3. **Tester** : Valider en dev avant la prod  
4. **Documenter** : Noter la configuration et les raisons  
5. **Sauvegarder** : Garder l'ancienne connection string en backup

### ⚠️ Pièges à éviter

- ❌ Pool trop grand en mode transaction (gaspillage)  
- ❌ Pool trop petit → clients en attente constante  
- ❌ Mode transaction avec prepared statements nommés  
- ❌ Oublier `server_reset_query` en mode transaction  
- ❌ Ne pas monitorer les métriques PgBouncer
