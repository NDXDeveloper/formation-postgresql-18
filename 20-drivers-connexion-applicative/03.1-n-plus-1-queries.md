# 20.3.1. N+1 Queries : Détection et Correction (JOIN, LATERAL)

## Introduction

Le problème des **N+1 queries** est l'un des pièges de performance les plus courants dans le développement d'applications avec bases de données. Ce problème survient souvent de manière invisible et peut transformer une application rapide en cauchemar de performance une fois en production.

**Objectif de ce chapitre :** Comprendre le problème N+1, savoir le détecter et maîtriser les techniques pour le corriger avec PostgreSQL.

---

## 1. Qu'est-ce que le Problème N+1 ?

### 1.1. Définition

Le problème N+1 se produit lorsqu'une application exécute :
- **1 requête** pour récupérer une liste d'entités principales
- Puis **N requêtes supplémentaires** (une pour chaque entité) pour récupérer des données liées

Si vous avez 100 entités, vous exécutez donc **101 requêtes** au lieu d'une seule requête optimisée.

### 1.2. Exemple Concret

Imaginons un blog avec des articles et leurs auteurs.

**Schéma de base :**
```sql
-- Table des auteurs
CREATE TABLE authors (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(100)
);

-- Table des articles
CREATE TABLE articles (
    id SERIAL PRIMARY KEY,
    title VARCHAR(200) NOT NULL,
    content TEXT,
    author_id INTEGER REFERENCES authors(id),
    published_at TIMESTAMP DEFAULT NOW()
);
```

**Scénario problématique :**

Vous voulez afficher une liste d'articles avec le nom de leur auteur.

```sql
-- Requête 1 : Récupérer tous les articles
SELECT id, title, author_id FROM articles LIMIT 100;
```

Cette requête retourne 100 articles. Ensuite, pour chaque article, l'application exécute :

```sql
-- Requête 2 (pour article 1)
SELECT name FROM authors WHERE id = 1;

-- Requête 3 (pour article 2)
SELECT name FROM authors WHERE id = 2;

-- Requête 4 (pour article 3)
SELECT name FROM authors WHERE id = 3;

-- ... et ainsi de suite jusqu'à 100 !
```

**Résultat :** 101 requêtes au lieu d'une seule !

### 1.3. Pourquoi est-ce un Problème ?

Chaque requête vers la base de données a un coût :
- **Latence réseau** (même minime)  
- **Overhead de connexion** et de parsing SQL  
- **Changements de contexte** sur le serveur PostgreSQL  
- **Locks et concurrence**

Même si chaque requête individuelle prend 1ms, 100 requêtes = 100ms au minimum, contre potentiellement 5-10ms pour une requête optimisée avec JOIN.

**En production :**
- 1 utilisateur = 101 requêtes
- 100 utilisateurs simultanés = 10,100 requêtes
- Saturation rapide de la base de données

## 2. Comment Détecter le Problème N+1

### 2.1. Observation des Logs d'Application

**Symptômes caractéristiques :**
- De nombreuses requêtes SQL identiques avec seulement les paramètres qui changent
- Temps de réponse qui augmente proportionnellement au nombre d'entités
- Patterns répétitifs dans les logs

**Exemple de logs problématiques :**
```
[SQL] SELECT * FROM articles LIMIT 100;
[SQL] SELECT * FROM authors WHERE id = 1;
[SQL] SELECT * FROM authors WHERE id = 2;
[SQL] SELECT * FROM authors WHERE id = 3;
[SQL] SELECT * FROM authors WHERE id = 4;
...
```

### 2.2. Utilisation de pg_stat_statements

PostgreSQL fournit l'extension `pg_stat_statements` qui enregistre les statistiques d'exécution de toutes les requêtes.

```sql
-- Activer l'extension (à faire une seule fois)
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- Identifier les requêtes exécutées très fréquemment
SELECT
    calls,
    mean_exec_time,
    query
FROM pg_stat_statements  
WHERE query LIKE '%authors%'  
ORDER BY calls DESC  
LIMIT 10;  
```

**Ce qu'on recherche :**
- Requêtes avec un très grand nombre d'appels (`calls`)
- Temps d'exécution individuel faible mais multiplié par le nombre d'appels
- Patterns de requêtes `SELECT ... WHERE id = ?` répétitives

### 2.3. Outils de Monitoring ORM

La plupart des ORM (Object-Relational Mapping) modernes offrent des outils de détection :

**Django (Python) :**
```python
# Activer le logging SQL en développement
LOGGING = {
    'loggers': {
        'django.db.backends': {
            'level': 'DEBUG',
        }
    }
}
```

**SQLAlchemy (Python) :**
```python
import logging  
logging.basicConfig()  
logging.getLogger('sqlalchemy.engine').setLevel(logging.INFO)  
```

**Hibernate (Java) :**
```properties
# application.properties
spring.jpa.show-sql=true  
spring.jpa.properties.hibernate.format_sql=true  
```

### 2.4. Mesure des Performances en Développement

**Technique simple :**
1. Activer le logging SQL  
2. Exécuter une fonctionnalité qui affiche une liste  
3. Compter manuellement les requêtes SQL générées  
4. Si le nombre de requêtes = 1 + nombre d'éléments → Vous avez un N+1

## 3. Solution 1 : Les Jointures (JOIN)

### 3.1. Principe

Au lieu d'exécuter plusieurs requêtes, on **combine les données** de plusieurs tables en une seule requête grâce aux jointures.

### 3.2. Correction avec INNER JOIN

**Version problématique (N+1) :**
```sql
-- Requête 1
SELECT id, title, author_id FROM articles;

-- Puis N requêtes
SELECT name FROM authors WHERE id = ?;
```

**Version optimisée (1 requête) :**
```sql
SELECT
    a.id,
    a.title,
    a.published_at,
    au.name AS author_name,
    au.email AS author_email
FROM articles a  
INNER JOIN authors au ON a.author_id = au.id  
ORDER BY a.published_at DESC;  
```

**Résultat :** Toutes les données sont récupérées en une seule requête !

### 3.3. LEFT JOIN pour Gérer les Valeurs Manquantes

Si certains articles n'ont **pas d'auteur** (author_id NULL), `INNER JOIN` les exclura.

**Solution :** Utiliser `LEFT JOIN`

```sql
SELECT
    a.id,
    a.title,
    COALESCE(au.name, 'Auteur inconnu') AS author_name
FROM articles a  
LEFT JOIN authors au ON a.author_id = au.id;  
```

**Explication :**
- `LEFT JOIN` conserve tous les articles, même sans auteur  
- `COALESCE` fournit une valeur par défaut si l'auteur est NULL

### 3.4. Cas avec Plusieurs Relations

Imaginons un article avec un auteur **et** une catégorie :

```sql
CREATE TABLE categories (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50) NOT NULL
);

ALTER TABLE articles ADD COLUMN category_id INTEGER REFERENCES categories(id);
```

**Requête optimisée avec plusieurs JOIN :**
```sql
SELECT
    a.id,
    a.title,
    au.name AS author_name,
    c.name AS category_name
FROM articles a  
LEFT JOIN authors au ON a.author_id = au.id  
LEFT JOIN categories c ON a.category_id = c.id  
ORDER BY a.published_at DESC;  
```

**Résultat :** Une seule requête récupère articles, auteurs ET catégories.

### 3.5. Performance des JOIN

**Conditions pour de bonnes performances :**
1. **Index sur les clés étrangères**
   ```sql
   CREATE INDEX idx_articles_author_id ON articles(author_id);
   CREATE INDEX idx_articles_category_id ON articles(category_id);
   ```

2. **Statistiques à jour**
   ```sql
   ANALYZE articles;
   ANALYZE authors;
   ANALYZE categories;
   ```

3. **Vérifier le plan d'exécution**
   ```sql
   EXPLAIN ANALYZE
   SELECT a.*, au.name
   FROM articles a
   LEFT JOIN authors au ON a.author_id = au.id;
   ```

## 4. Solution 2 : LATERAL JOIN (Corrélation Avancée)

### 4.1. Qu'est-ce qu'un LATERAL JOIN ?

`LATERAL` est une fonctionnalité avancée de PostgreSQL qui permet à une sous-requête de **référencer des colonnes** de la requête principale.

**Analogie :** C'est comme une boucle FOR EACH en SQL, mais optimisée par PostgreSQL.

### 4.2. Cas d'Usage : Relations 1-à-N Complexes

Imaginons que vous voulez récupérer chaque auteur avec ses **3 derniers articles**.

**Problème avec JOIN classique :**
```sql
-- Cette requête ne fonctionne pas correctement
SELECT
    au.name,
    a.title,
    a.published_at
FROM authors au  
LEFT JOIN articles a ON au.id = a.author_id  
ORDER BY a.published_at DESC  
LIMIT 3; -- ❌ Limite globale, pas par auteur !  
```

### 4.3. Solution avec LATERAL

```sql
SELECT
    au.id,
    au.name,
    recent_articles.title,
    recent_articles.published_at
FROM authors au  
LEFT JOIN LATERAL (  
    SELECT title, published_at
    FROM articles
    WHERE author_id = au.id  -- ✅ Référence à la table externe !
    ORDER BY published_at DESC
    LIMIT 3  -- Les 3 derniers articles PAR AUTEUR
) AS recent_articles ON true;
```

**Explication :**
- La sous-requête `LATERAL` peut accéder à `au.id` de la requête principale
- Pour chaque auteur, PostgreSQL exécute la sous-requête
- Mais PostgreSQL optimise cela intelligemment (pas un vrai N+1)

### 4.4. LATERAL vs N+1 : Quelle Différence ?

**Pourquoi LATERAL n'est pas un N+1 :**
1. **Planification unique :** PostgreSQL compile une seule fois la requête  
2. **Optimisations possibles :** Index scan, batch fetching  
3. **Pas d'overhead réseau :** Tout se passe côté serveur  
4. **Transactions cohérentes :** Tout dans la même transaction

**Un vrai N+1 :**
- N+1 connexions réseau
- N+1 compilations SQL
- N+1 changements de contexte

### 4.5. Cas Pratique : Top N par Groupe

**Objectif :** Les 5 articles les plus récents dans chaque catégorie.

```sql
SELECT
    c.name AS category_name,
    top_articles.title,
    top_articles.published_at
FROM categories c  
LEFT JOIN LATERAL (  
    SELECT title, published_at
    FROM articles
    WHERE category_id = c.id
    ORDER BY published_at DESC
    LIMIT 5
) AS top_articles ON true
ORDER BY c.name, top_articles.published_at DESC;
```

### 4.6. LATERAL avec Fonctions d'Agrégation

**Exemple :** Pour chaque auteur, compter ses articles ET récupérer le titre du plus récent.

```sql
SELECT
    au.name,
    article_stats.total_articles,
    article_stats.latest_title
FROM authors au  
LEFT JOIN LATERAL (  
    SELECT
        COUNT(*) AS total_articles,
        (SELECT title
         FROM articles
         WHERE author_id = au.id
         ORDER BY published_at DESC
         LIMIT 1) AS latest_title
    FROM articles
    WHERE author_id = au.id
) AS article_stats ON true;
```

### 4.7. Performance de LATERAL

**Quand utiliser LATERAL :**
- ✅ Relations 1-à-N avec limite par groupe (Top N)  
- ✅ Calculs complexes dépendant de la ligne parente  
- ✅ Besoin de corrélation avancée

**Quand préférer JOIN classique :**
- ✅ Relations simples 1-à-1 ou N-à-1  
- ✅ Pas besoin de LIMIT par groupe  
- ✅ Meilleure lisibilité

**Optimisation LATERAL :**
```sql
-- Index sur la colonne de jointure et de tri
CREATE INDEX idx_articles_author_published  
ON articles(author_id, published_at DESC);  
```

## 5. Comparaison des Solutions

### 5.1. Tableau Récapitulatif

| Critère | N+1 (Problème) | JOIN Classique | LATERAL JOIN |
|---------|----------------|----------------|--------------|
| **Nombre de requêtes** | 1 + N | 1 | 1 |
| **Performance** | ❌ Très mauvaise | ✅ Excellente | ✅ Bonne à excellente |
| **Complexité** | Simple (mais mauvais) | Simple | Moyenne |
| **Top N par groupe** | ❌ Impossible | ❌ Difficile | ✅ Facile |
| **Relations 1-1 ou N-1** | ❌ | ✅ Optimal | ⚠️ Overkill |
| **Relations 1-N** | ❌ | ✅ Si on veut tout | ✅ Si limite par parent |

### 5.2. Arbre de Décision

```
Vous avez une relation entre tables ?
│
├─ Relation 1-1 ou N-1 (ex: article → auteur)
│  └─→ Utiliser INNER JOIN ou LEFT JOIN
│
├─ Relation 1-N simple (ex: auteur → tous ses articles)
│  ├─ Vous voulez TOUS les articles ?
│  │  └─→ JOIN classique
│  │
│  └─ Vous voulez un TOP N par parent (ex: 5 derniers) ?
│     └─→ LATERAL JOIN
│
└─ Calculs complexes dépendant du parent ?
   └─→ LATERAL JOIN
```

## 6. Intégration dans le Code Applicatif

### 6.1. Avec les ORM (Eager Loading)

La plupart des ORM offrent des mécanismes pour éviter le N+1.

#### Python - Django
```python
# ❌ Problème N+1
articles = Article.objects.all()  
for article in articles:  
    print(article.author.name)  # 1 requête par itération !

# ✅ Solution : select_related (pour FK et OneToOne)
articles = Article.objects.select_related('author').all()  
for article in articles:  
    print(article.author.name)  # Tout en 1 requête (JOIN)
```

#### Python - SQLAlchemy
```python
from sqlalchemy.orm import joinedload

# ❌ N+1
articles = session.query(Article).all()

# ✅ Solution
articles = session.query(Article).options(
    joinedload(Article.author)
).all()
```

#### JavaScript - Prisma
```javascript
// ❌ N+1
const articles = await prisma.article.findMany();  
for (const article of articles) {  
  const author = await prisma.author.findUnique({
    where: { id: article.authorId }
  });
}

// ✅ Solution : include
const articles = await prisma.article.findMany({
  include: {
    author: true
  }
});
```

#### Java - Hibernate
```java
// ❌ N+1 (lazy loading par défaut)
List<Article> articles = session.createQuery("FROM Article", Article.class).list();  
for (Article article : articles) {  
    System.out.println(article.getAuthor().getName()); // N requêtes !
}

// ✅ Solution : JOIN FETCH
List<Article> articles = session.createQuery(
    "SELECT a FROM Article a JOIN FETCH a.author",
    Article.class
).list();
```

### 6.2. Quand Ne PAS Éviter le N+1

**Cas rares où N+1 peut être acceptable :**

1. **Très petit N (< 5)** et requêtes en cache  
2. **Connexion ultra-rapide** (localhost) et données complexes  
3. **Données déjà chargées** dans un cache applicatif

**Mais en général :** Toujours optimiser, même pour un petit N.

## 8. Exercice Mental (Sans Code)

### Scénario

Vous développez une API qui doit retourner :
- Une liste de commandes (orders)
- Pour chaque commande : le client (customer)
- Pour chaque commande : les produits commandés (order_items → products)

**Schéma :**
```
orders → customer (N-1)  
orders → order_items (1-N) → products (N-1)  
```

### Questions de Réflexion

1. **Combien de requêtes si on fait du N+1 pur pour 10 commandes avec 5 produits chacune ?**
   - Réponse : 1 (orders) + 10 (customers) + 10 (order_items) + 50 (products) = 71 requêtes

2. **Comment optimiser avec JOIN ?**
   - Réponse :
   ```sql
   SELECT o.*, c.*, oi.*, p.*
   FROM orders o
   JOIN customers c ON o.customer_id = c.id
   JOIN order_items oi ON o.id = oi.order_id
   JOIN products p ON oi.product_id = p.id
   ```

3. **Problème potentiel de cette requête ?**
   - Réponse : Explosion cartésienne. Si 1 commande = 5 produits, vous aurez 5 lignes pour cette commande.
   - Solution : Grouper côté application ou utiliser des agrégations JSON.

---

## 9. Techniques Avancées de Correction

### 9.1. Agrégation JSON pour Relations 1-N

Au lieu d'avoir plusieurs lignes par commande, on peut **agréger** les produits en JSON.

```sql
SELECT
    o.id AS order_id,
    o.total_amount,
    c.name AS customer_name,
    json_agg(
        json_build_object(
            'product_name', p.name,
            'quantity', oi.quantity,
            'price', oi.price
        )
    ) AS products
FROM orders o  
JOIN customers c ON o.customer_id = c.id  
JOIN order_items oi ON o.id = oi.order_id  
JOIN products p ON oi.product_id = p.id  
GROUP BY o.id, o.total_amount, c.name;  
```

**Résultat :**
```json
{
  "order_id": 1,
  "total_amount": 150.00,
  "customer_name": "John Doe",
  "products": [
    {"product_name": "Laptop", "quantity": 1, "price": 100.00},
    {"product_name": "Mouse", "quantity": 2, "price": 25.00}
  ]
}
```

**Avantages :**
- ✅ Une seule ligne par commande  
- ✅ Pas de duplication de données  
- ✅ Facile à parser côté application

### 9.2. Subqueries Corrélées (Alternative à LATERAL)

```sql
SELECT
    au.name,
    (SELECT COUNT(*)
     FROM articles
     WHERE author_id = au.id) AS article_count,
    (SELECT title
     FROM articles
     WHERE author_id = au.id
     ORDER BY published_at DESC
     LIMIT 1) AS latest_article
FROM authors au;
```

**Attention :** Peut être moins performant que LATERAL si plusieurs sous-requêtes.

### 9.3. Window Functions pour Top N

Pour récupérer le top N par groupe sans LATERAL :

```sql
WITH ranked_articles AS (
    SELECT
        a.*,
        au.name AS author_name,
        ROW_NUMBER() OVER (
            PARTITION BY a.author_id
            ORDER BY a.published_at DESC
        ) AS rn
    FROM articles a
    JOIN authors au ON a.author_id = au.id
)
SELECT *  
FROM ranked_articles  
WHERE rn <= 3;  -- Top 3 par auteur  
```

## 10. Résumé et Bonnes Pratiques

### 10.1. Points Clés à Retenir

1. **Le N+1 est un problème de performance majeur** qui multiplie le nombre de requêtes inutilement  
2. **Toujours utiliser JOIN** pour les relations simples (1-1, N-1)  
3. **LATERAL est parfait** pour les Top N par groupe ou corrélations complexes  
4. **Les ORM ne sont pas magiques** : connaître SQL est essentiel  
5. **Mesurer et monitorer** avec pg_stat_statements et les logs

### 10.2. Checklist de Prévention

- [ ] **En développement :** Activer le logging SQL détaillé  
- [ ] **En code review :** Vérifier les boucles avec requêtes  
- [ ] **En test :** Compter les requêtes générées pour chaque endpoint  
- [ ] **En production :** Monitorer pg_stat_statements  
- [ ] **Sur les ORM :** Utiliser eager loading (select_related, joinedload, include)  
- [ ] **Sur les index :** Créer des index sur toutes les clés étrangères

### 10.3. Règles d'Or

1. **"1 opération = 1 requête"** : Si vous affichez une liste, une seule requête devrait suffire  
2. **"Joindre, pas boucler"** : Préférer JOIN à des boucles applicatives  
3. **"Mesurer avant d'optimiser"** : Mais le N+1 est presque toujours à corriger  
4. **"L'ORM ne remplace pas SQL"** : Comprendre ce qui est généré

## Conclusion

Le problème N+1 est l'un des anti-patterns les plus courants et les plus dévastateurs en termes de performance. Avec PostgreSQL, vous disposez d'outils puissants comme **JOIN** et **LATERAL** pour le corriger élégamment.

**Retenez :** Une application bien conçue doit **minimiser le nombre de requêtes** tout en **maximisant la quantité de données utiles** récupérées en une seule fois. C'est l'essence même de l'optimisation SQL.
