# 20.4.3. Schema Versioning

## Introduction

Le **Schema Versioning** (versionnement de schéma) est l'art de gérer et tracer l'évolution de la structure de votre base de données PostgreSQL dans le temps. Alors que les migrations (chapitre précédent 20.4.2) sont les **mécanismes** qui appliquent les changements, le schema versioning est la **stratégie** qui organise ces changements.

### Métaphore : Le livre d'histoire de votre base de données

Imaginez votre base de données PostgreSQL comme un bâtiment :
- **Les migrations** sont les travaux de rénovation (ajouter une pièce, changer une porte)  
- **Le schema versioning** est le système de numérotation qui identifie chaque version du bâtiment (v1.0, v1.1, v2.0)

Le schema versioning répond à des questions cruciales :
- Quelle version du schéma est déployée en production ?
- Mon code en version 2.3 est-il compatible avec le schéma en version 2.1 ?
- Comment documenter les breaking changes entre versions ?
- Comment gérer plusieurs versions simultanément (migration progressive) ?

## Pourquoi le schema versioning est-il important ?

### Problème 1 : Coordination Code ↔ Base de données

Votre application évolue en parallèle de votre base de données :

```plaintext
Situation sans versioning :
─────────────────────────────────────
Timeline:
  Lundi    : Déploiement application v1.5
  Mardi    : Quelqu'un applique une migration
  Mercredi : Application plante en production
  Jeudi    : Investigation : Quelle version du schéma ?
  Vendredi : Impossible à déterminer → Chaos

Situation avec versioning :
─────────────────────────────────────
Timeline:
  Lundi    : Déploiement application v1.5 (requiert schéma v1.4+)
  Mardi    : Migration appliquée → Schéma passe à v1.5
  Mercredi : Application fonctionne parfaitement

  → Traçabilité claire : App v1.5 ↔ Schéma v1.5
```

### Problème 2 : Déploiements progressifs

Dans les architectures modernes (microservices, blue/green deployment), plusieurs versions de votre application peuvent coexister :

```plaintext
Sans schema versioning :
  Production A : Application v2.0
  Production B : Application v2.1 (déploiement en cours)
  → Les deux versions doivent parler à la même base
  → Quelle version du schéma utiliser ? Mystère...

Avec schema versioning :
  Production A : App v2.0 (compatible schéma v2.0 à v2.2)
  Production B : App v2.1 (compatible schéma v2.1 à v2.2)
  Schéma actuel : v2.1
  → Compatibilité garantie pour les deux versions
```

### Problème 3 : Communication d'équipe

```plaintext
Développeur Backend : "J'ai besoin de la colonne user_status dans users"  
Développeur Frontend : "Quelle version de l'API dois-je utiliser ?"  
DevOps : "Quelle version du schéma est en staging ?"  
DBA : "Quand peut-on supprimer l'ancienne colonne legacy_field ?"  

→ Sans versioning : Confusion totale
→ Avec versioning : Réponses claires grâce aux numéros de version
```

### Problème 4 : Rollback et dégradation

```plaintext
Déploiement v3.0 échoue en production :

Sans versioning :
  → On ne sait pas à quelle version revenir
  → Risque de casser d'autres services

Avec versioning :
  → Schéma v3.0 est incompatible
  → Rollback au schéma v2.5 (version stable connue)
  → Application retourne en v2.5 aussi
  → Service restauré en toute confiance
```

## Les stratégies de versionnement

### 1. Sequential Versioning (Versionnement séquentiel)

C'est l'approche la plus simple : chaque migration incrémente un compteur.

```plaintext
Format : Version = Numéro séquentiel

Exemples :
  Version 1  : Création initiale des tables
  Version 2  : Ajout colonne email
  Version 3  : Ajout table orders
  Version 4  : Ajout index sur email
  Version 5  : Renommage colonne name → full_name
  ...
  Version 247 : État actuel
```

#### Avantages
- ✅ **Simplicité** : Facile à comprendre (1, 2, 3, 4...)  
- ✅ **Ordre clair** : On sait toujours quelle version vient avant/après  
- ✅ **Peu de conflits** : Les numéros ne se chevauchent pas

#### Inconvénients
- ❌ **Pas de signification** : Le numéro 247 ne dit rien sur l'importance des changements  
- ❌ **Difficile pour les branches** : Si deux développeurs créent la version 50 en parallèle → conflit  
- ❌ **Pas d'indication de compatibilité** : Impossible de savoir si v100 casse v99

#### Cas d'usage
- Projets simples avec une seule équipe
- Développement linéaire (pas de branches parallèles)
- Petites applications

**Exemple avec Flyway :**
```plaintext
V1__create_users_table.sql  
V2__add_email_to_users.sql  
V3__create_orders_table.sql  
V4__add_foreign_key_orders_users.sql  
```

### 2. Timestamp Versioning (Versionnement par date)

Utilise la date et l'heure comme numéro de version.

```plaintext
Format : Version = YYYYMMDDHHmmSS

Exemples :
  Version 20250115103000  : 15 janvier 2025, 10h30
  Version 20250115143000  : 15 janvier 2025, 14h30
  Version 20250120091500  : 20 janvier 2025, 09h15
  Version 20250203140000  : 3 février 2025, 14h00
```

#### Avantages
- ✅ **Unicité garantie** : Deux développeurs ne peuvent pas créer le même timestamp  
- ✅ **Contexte temporel** : On sait quand la migration a été créée  
- ✅ **Ordre chronologique** : Tri naturel par date

#### Inconvénients
- ❌ **Numéros longs** : 20250115103000 est moins lisible que 5  
- ❌ **Pas de signification sémantique** : Impossible de savoir si c'est un changement majeur ou mineur  
- ❌ **Conflits possibles** : Si deux développeurs créent une migration à la même seconde

#### Cas d'usage
- Équipes distribuées travaillant en parallèle
- Projets avec beaucoup de branches de développement
- Besoin de traçabilité temporelle

**Exemple avec Alembic :**
```plaintext
20250115103000_create_users_table.py
20250115143000_add_email_to_users.py
20250120091500_create_orders_table.py
```

### 3. Semantic Versioning (Versionnement sémantique)

Utilise le standard **SemVer** : `MAJOR.MINOR.PATCH`

```plaintext
Format : MAJOR.MINOR.PATCH

Exemples :
  Version 1.0.0  : Première version stable
  Version 1.1.0  : Ajout de fonctionnalités compatibles
  Version 1.1.1  : Correction de bug (pas de changement de schéma)
  Version 2.0.0  : Breaking change (incompatible avec v1.x)
  Version 2.1.0  : Nouvelles fonctionnalités (compatible v2.0.0)
```

#### Règles SemVer pour les bases de données

**MAJOR (X.0.0)** - Breaking change
```plaintext
Changements qui cassent la compatibilité :
  - Renommer une table ou colonne
  - Supprimer une table ou colonne
  - Changer le type d'une colonne (VARCHAR → INTEGER)
  - Modifier une contrainte (nullable → NOT NULL)
  - Supprimer un index utilisé

Exemple :
  v1.5.2 → v2.0.0
  ALTER TABLE users RENAME COLUMN name TO full_name;
  → Les applications lisant "name" vont planter
```

**MINOR (0.X.0)** - Ajout rétrocompatible
```plaintext
Changements qui ajoutent sans casser :
  - Ajouter une nouvelle table
  - Ajouter une colonne nullable
  - Ajouter un index
  - Créer une vue
  - Ajouter une fonction

Exemple :
  v1.4.0 → v1.5.0
  ALTER TABLE users ADD COLUMN phone VARCHAR(20);
  → Les applications existantes continuent de fonctionner
```

**PATCH (0.0.X)** - Corrections
```plaintext
Corrections sans impact sur le schéma :
  - Correction de données (UPDATE)
  - Optimisation d'index existant
  - Correction de commentaires
  - Mise à jour de statistiques (ANALYZE)

Exemple :
  v1.4.2 → v1.4.3
  UPDATE users SET status = 'active' WHERE status IS NULL;
  → Pas de changement structurel
```

#### Avantages
- ✅ **Signification claire** : On comprend immédiatement l'impact du changement  
- ✅ **Compatibilité prévisible** : v2.3.0 est compatible avec toute app supportant v2.0.0+  
- ✅ **Standard reconnu** : SemVer est utilisé partout (npm, Maven, etc.)  
- ✅ **Communication facilitée** : "On passe à la v2.0" = breaking change évident

#### Inconvénients
- ❌ **Complexité** : Nécessite de comprendre SemVer  
- ❌ **Subjectivité** : Débats sur ce qui est "breaking" ou pas  
- ❌ **Gestion manuelle** : Il faut décider consciemment du numéro de version

#### Cas d'usage
- Projets avec plusieurs équipes consommatrices (APIs, microservices)
- Applications nécessitant une compatibilité stricte
- Documentation et communication importantes
- Projets matures avec cycle de release structuré

**Exemple d'organisation :**
```plaintext
migrations/
├── v1/
│   ├── 1.0.0__initial_schema.sql
│   ├── 1.1.0__add_orders_table.sql
│   ├── 1.2.0__add_payments_table.sql
│   └── 1.2.1__fix_payment_constraints.sql
├── v2/
│   ├── 2.0.0__rename_users_columns.sql  ← Breaking
│   ├── 2.1.0__add_notifications.sql
│   └── 2.2.0__add_audit_logging.sql
└── current_version.txt → "2.2.0"
```

### 4. Hybrid Versioning (Versionnement hybride)

Combine plusieurs approches : SemVer + Timestamp ou Sequential + SemVer.

```plaintext
Format : MAJOR.MINOR.PATCH-TIMESTAMP

Exemples :
  Version 1.0.0-20250115
  Version 1.1.0-20250120
  Version 2.0.0-20250203

Ou : MAJOR.MINOR-SEQUENTIAL

Exemples :
  Version 1.0-001
  Version 1.0-002
  Version 1.1-001
  Version 2.0-001
```

#### Avantages
- ✅ **Meilleur des deux mondes** : Sémantique + unicité temporelle  
- ✅ **Flexibilité** : Adapté aux besoins spécifiques

#### Inconvénients
- ❌ **Complexité accrue** : Plus difficile à maintenir  
- ❌ **Conventions à définir** : L'équipe doit créer ses propres règles

#### Cas d'usage
- Grands projets avec multiples équipes
- Besoins de traçabilité extrême
- Déploiements complexes multi-environnements

## Compatibilité entre versions

### Backward Compatibility (Rétrocompatibilité)

Une nouvelle version de schéma est **rétrocompatible** si les anciennes versions de l'application continuent de fonctionner.

```plaintext
Schéma v1.0 :
  Table users (id, name, email)

Application v1.0 :
  SELECT id, name, email FROM users;  ✅

Migration vers Schéma v1.1 :
  ALTER TABLE users ADD COLUMN phone VARCHAR(20);

Application v1.0 (inchangée) :
  SELECT id, name, email FROM users;  ✅ Fonctionne toujours !
  → Phone est ignoré, pas de problème

→ Schéma v1.1 est BACKWARD COMPATIBLE avec App v1.0
```

**Principe :** Ajouter sans supprimer ni modifier.

#### Changements rétrocompatibles

✅ **Ajouter une colonne nullable**
```sql
ALTER TABLE users ADD COLUMN phone VARCHAR(20);
-- Anciennes requêtes fonctionnent (ignorent phone)
```

✅ **Ajouter une nouvelle table**
```sql
CREATE TABLE notifications (...);
-- Anciennes requêtes ne touchent pas cette table
```

✅ **Ajouter un index**
```sql
CREATE INDEX idx_users_email ON users(email);
-- Transparent pour l'application
```

✅ **Créer une vue**
```sql
CREATE VIEW active_users AS SELECT * FROM users WHERE deleted_at IS NULL;
-- Les requêtes sur users continuent de fonctionner
```

#### Changements NON rétrocompatibles (Breaking changes)

❌ **Renommer une colonne**
```sql
ALTER TABLE users RENAME COLUMN name TO full_name;
-- SELECT name FROM users; → ERREUR : colonne "name" n'existe plus
```

❌ **Supprimer une colonne**
```sql
ALTER TABLE users DROP COLUMN phone;
-- SELECT phone FROM users; → ERREUR
```

❌ **Changer le type d'une colonne**
```sql
ALTER TABLE users ALTER COLUMN age TYPE VARCHAR(10);
-- Requêtes attendant un INTEGER vont échouer
```

❌ **Ajouter une contrainte NOT NULL**
```sql
ALTER TABLE users ALTER COLUMN email SET NOT NULL;
-- INSERT INTO users (name) VALUES ('John'); → ERREUR : email manquant
```

### Forward Compatibility (Compatibilité future)

Une ancienne version de schéma est **compatible avec les futures versions** de l'application.

```plaintext
Schéma v1.0 :
  Table users (id, name, email)

Application v1.1 (nouvelle) :
  SELECT id, name, email, phone FROM users;
  → ERREUR si phone n'existe pas dans schéma v1.0

→ Schéma v1.0 n'est PAS FORWARD COMPATIBLE avec App v1.1

Solution :
  Application v1.1 gère gracieusement l'absence de phone :
  SELECT id, name, email, COALESCE(phone, '') as phone FROM users;
```

**Principe :** Le code doit être tolérant aux changements futurs du schéma.

### Stratégie de compatibilité idéale

```plaintext
Golden Rule : Expand-Migrate-Contract

Phase 1 - EXPAND (Étendre) :
  Schéma v1.1 : Ajouter nouvelle colonne full_name (nullable)
  Application v1.0 : Continue d'utiliser name
  Application v1.1 : Utilise full_name si disponible, sinon name
  → Les deux versions coexistent

Phase 2 - MIGRATE (Migrer) :
  Schéma v1.1 : Copier les données de name vers full_name
  Application v1.2 : Utilise uniquement full_name
  → Migration progressive

Phase 3 - CONTRACT (Retirer) :
  Schéma v2.0 : Supprimer l'ancienne colonne name
  → Breaking change contrôlé, tout le monde est prêt
```

Cette stratégie permet des **déploiements sans interruption**.

## Traçabilité et documentation

### 1. Fichier de version central

Créez un fichier qui documente la version actuelle du schéma :

**Fichier : schema_version.txt**
```plaintext
CURRENT_VERSION=2.5.0  
LAST_MIGRATION=20250203_add_notifications_table  
COMPATIBLE_APP_VERSIONS=2.3.0,2.4.0,2.5.0  
BREAKING_CHANGES_FROM=1.x  
```

Ce fichier doit être :
- ✅ Versionné dans Git  
- ✅ Mis à jour automatiquement par les migrations  
- ✅ Consulté par les scripts de déploiement

### 2. Table de métadonnées PostgreSQL

Créez une table dédiée au versionnement :

```sql
CREATE TABLE schema_version (
    version VARCHAR(20) PRIMARY KEY,
    applied_at TIMESTAMPTZ DEFAULT NOW(),
    description TEXT,
    breaking_change BOOLEAN DEFAULT false,
    compatible_with VARCHAR(100),  -- Versions compatibles
    migration_script VARCHAR(255),
    applied_by VARCHAR(100)
);
```

**Exemple de contenu :**
```sql
INSERT INTO schema_version (version, description, breaking_change, compatible_with) VALUES
('1.0.0', 'Initial schema', false, 'app>=1.0.0'),
('1.1.0', 'Added orders table', false, 'app>=1.0.0'),
('1.2.0', 'Added payments', false, 'app>=1.0.0'),
('2.0.0', 'Renamed users.name to full_name', true, 'app>=2.0.0'),
('2.1.0', 'Added notifications', false, 'app>=2.0.0');
```

**Requête pour vérifier la version :**
```sql
SELECT version, applied_at, breaking_change  
FROM schema_version  
ORDER BY applied_at DESC  
LIMIT 1;  
```

### 4. Tags Git pour les versions majeures

Utilisez les tags Git pour marquer les versions importantes du schéma :

```bash
# Créer un tag pour une version majeure
git tag -a schema-v2.0.0 -m "Schema version 2.0.0 - Breaking changes"  
git push origin schema-v2.0.0  

# Lister tous les tags de schéma
git tag -l "schema-v*"
```

Cela permet de retrouver facilement l'état du schéma à n'importe quelle version.

### 5. Commentaires PostgreSQL

Utilisez les commentaires natifs PostgreSQL pour documenter :

```sql
-- Documenter une table
COMMENT ON TABLE users IS
'User accounts. Schema version: 2.0.0.
Breaking change from v1.x: column "name" renamed to "full_name".';

-- Documenter une colonne
COMMENT ON COLUMN users.full_name IS
'User full name. Added in v2.0.0 (renamed from "name").
Required for all new users.';

-- Documenter une contrainte
COMMENT ON CONSTRAINT users_email_unique ON users IS
'Email must be unique. Enforced since v1.0.0.';
```

**Avantage :** Ces commentaires sont stockés directement dans PostgreSQL et visibles via `\d+` dans psql.

## Gestion multi-environnements

### Stratégie de déploiement progressif

```plaintext
Environnements avec versions différentes :

Développement local :
  Schéma v2.2.0 (bleeding edge)
  Application v2.2.0-dev
  → Développeurs testent les dernières features

CI/CD :
  Schéma v2.2.0
  Application v2.2.0
  → Tests automatisés

Staging :
  Schéma v2.1.0
  Application v2.1.0
  → Réplique de production pour tests finaux

Production :
  Schéma v2.0.0
  Application v2.0.0
  → Version stable éprouvée

→ Déploiement progressif : Dev → CI → Staging → Production
```

### Matrice de compatibilité

Documentez quelle version d'application fonctionne avec quelle version de schéma :

```plaintext
Matrice de Compatibilité :

| Schema Version | App v2.0 | App v2.1 | App v2.2 | App v3.0 |
|----------------|----------|----------|----------|----------|
| Schema v2.0    | ✅       | ✅       | ✅       | ❌       |
| Schema v2.1    | ✅       | ✅       | ✅       | ❌       |
| Schema v2.2    | ⚠️       | ✅       | ✅       | ❌       |
| Schema v3.0    | ❌       | ❌       | ❌       | ✅       |

✅ Compatible
⚠️ Compatible avec limitations
❌ Incompatible
```

Cette matrice aide à planifier les déploiements et éviter les incompatibilités.

### Scripts de vérification de version

Créez des scripts qui vérifient la compatibilité avant déploiement :

**Exemple conceptuel : verify_compatibility.sh**
```bash
#!/bin/bash
# Script de vérification de compatibilité

APP_VERSION="2.1.0"  
SCHEMA_VERSION=$(psql -t -c "SELECT version FROM schema_version ORDER BY applied_at DESC LIMIT 1")  

if [ "$SCHEMA_VERSION" = "2.0.0" ] || [ "$SCHEMA_VERSION" = "2.1.0" ]; then
    echo "✅ Compatible : App $APP_VERSION avec Schema $SCHEMA_VERSION"
    exit 0
else
    echo "❌ INCOMPATIBLE : App $APP_VERSION requiert Schema v2.0.0 ou v2.1.0"
    echo "   Schema actuel : $SCHEMA_VERSION"
    exit 1
fi
```

Ce script peut être intégré dans votre pipeline CI/CD.

## Patterns et bonnes pratiques

### Pattern 1 : Version Pinning (Épinglage de version)

L'application déclare explicitement quelle version de schéma elle requiert.

**Dans le code de l'application :**
```plaintext
Configuration de l'application :

REQUIRED_SCHEMA_VERSION = "2.1.0"  
MINIMUM_SCHEMA_VERSION = "2.0.0"  
MAXIMUM_SCHEMA_VERSION = "2.3.0"  

Au démarrage de l'application :
  1. Se connecter à PostgreSQL
  2. Vérifier la version du schéma
  3. Si version < MINIMUM ou > MAXIMUM → ERREUR et arrêt
  4. Sinon → Démarrage normal
```

**Avantages :**
- ✅ Prévient les incompatibilités silencieuses  
- ✅ Échec rapide (fail-fast) en cas de problème  
- ✅ Documentation explicite des dépendances

### Pattern 2 : Feature Flags sur le schéma

Activer/désactiver des fonctionnalités selon la version du schéma.

**Exemple conceptuel :**
```plaintext
Dans l'application :

if (schemaVersion >= "2.1.0") {
    // Utiliser la nouvelle colonne full_name
    query = "SELECT id, full_name FROM users";
} else {
    // Fallback sur l'ancienne colonne name
    query = "SELECT id, name as full_name FROM users";
}

→ L'application s'adapte à la version du schéma
```

**Avantages :**
- ✅ Permet la coexistence de versions  
- ✅ Déploiement progressif facilité  
- ✅ Rollback sans casse

**Inconvénients :**
- ❌ Code plus complexe  
- ❌ Maintenance de legacy code  
- ❌ Dette technique si non nettoyé

### Pattern 3 : Schema Adapters

Créer des adaptateurs qui traduisent entre versions de schéma.

```plaintext
Architecture :

Application (attend schéma v2.0)
    ↓
SchemaAdapter
    ↓ (détecte schéma v1.5)
    ↓ (adapte automatiquement)
PostgreSQL (schéma v1.5)

Exemple :
  Application demande : SELECT full_name FROM users
  Adapter détecte : schéma v1.5 n'a que "name"
  Adapter traduit : SELECT name as full_name FROM users
  Application reçoit : full_name (transparence totale)
```

**Avantages :**
- ✅ Découplage total entre code et schéma  
- ✅ Très flexible

**Inconvénients :**
- ❌ Complexité élevée  
- ❌ Performance (overhead de traduction)  
- ❌ Maintenance difficile

### Pattern 4 : Blue-Green Schema Deployment

Déployer une nouvelle version de schéma en parallèle de l'ancienne.

```plaintext
État initial :
  Schema v1.0 dans namespace "public"
  Application pointe vers "public"

Étape 1 : Créer nouveau schéma
  CREATE SCHEMA public_v2;
  -- Appliquer migrations dans public_v2

Étape 2 : Tester
  Application de test pointe vers "public_v2"
  Validation complète

Étape 3 : Basculer
  Renommer public → public_v1_old
  Renommer public_v2 → public
  → Bascule instantanée !

Étape 4 : Rollback facile si problème
  Renommer public → public_v2_failed
  Renommer public_v1_old → public
  → Retour à l'ancien schéma
```

**Avantages :**
- ✅ Zero-downtime  
- ✅ Rollback instantané  
- ✅ Tests en conditions réelles

**Inconvénients :**
- ❌ Double espace disque  
- ❌ Synchronisation des données complexe  
- ❌ Ne fonctionne pas pour tous les changements

### Pattern 5 : Deprecation Policy (Politique de dépréciation)

Définir un processus clair pour supprimer des éléments de schéma.

```plaintext
Cycle de vie d'une colonne deprecated :

Version 1.5.0 :
  - Ajouter nouvelle colonne "full_name"
  - Marquer "name" comme deprecated
  - Documentation : "name sera supprimé en v2.0.0"

Version 1.6.0 - 1.9.0 :
  - Les deux colonnes coexistent
  - Logs de warning si "name" est utilisé
  - Migration progressive du code

Version 2.0.0 :
  - Suppression de "name"
  - Breaking change documenté et annoncé depuis v1.5.0

→ Minimum 3-6 mois entre deprecation et suppression
```

**Exemple de table de deprecation :**
```sql
CREATE TABLE schema_deprecations (
    element_type VARCHAR(50),  -- 'column', 'table', 'function'
    element_name VARCHAR(255),
    deprecated_in_version VARCHAR(20),
    removal_planned_version VARCHAR(20),
    reason TEXT,
    replacement TEXT
);

INSERT INTO schema_deprecations VALUES
('column', 'users.name', '1.5.0', '2.0.0',
 'Consolidation naming', 'Use users.full_name instead');
```

### Pattern 6 : Semantic Commit Messages

Utiliser des messages de commit Git pour automatiquement déduire la version.

```plaintext
Commit messages avec convention :

feat(schema): add notifications table
  → MINOR version bump (1.0.0 → 1.1.0)

fix(schema): correct payment_status constraint
  → PATCH version bump (1.1.0 → 1.1.1)

BREAKING CHANGE: rename users.name to full_name
  → MAJOR version bump (1.1.1 → 2.0.0)

→ Scripts automatiques calculent la prochaine version
```

## Cas d'usage avancés

### Cas 1 : API versionnée avec schéma versionnée

```plaintext
Architecture :

API v1 (/v1/users)
  └─> Schéma v1.x (utilise users.name)

API v2 (/v2/users)
  └─> Schéma v2.x (utilise users.full_name)

Stratégie :
  1. Schéma v1.5 : Ajouter full_name (nullable)
  2. Schéma v1.6 : Copier name → full_name
  3. Déployer API v2 (utilise full_name)
  4. API v1 continue de fonctionner (utilise name)
  5. Schéma v2.0 : Supprimer name (API v1 deprecated)
  6. Retirer API v1

→ Coexistence pacifique des versions
```

### Cas 2 : Microservices avec schéma partagé

```plaintext
Problème :
  Service A v1.0 → PostgreSQL (schéma v2.0)
  Service B v1.5 → PostgreSQL (schéma v2.0)
  Service C v2.0 → PostgreSQL (schéma v2.0)

Tous partagent la même base, mais ont des versions différentes.

Solution :
  Schéma v2.0 doit être compatible avec :
    - Service A v1.0+ (minimum)
    - Service B v1.5+ (minimum)
    - Service C v2.0+ (minimum)

  Matrice de compatibilité obligatoire :
    schema_version table contient :
      compatible_services = "A>=1.0,B>=1.5,C>=2.0"

  Before deploying schema v2.1:
    1. Vérifier que tous les services supportent v2.1
    2. Sinon, mettre à jour les services d'abord
    3. Puis migrer le schéma
```

### Cas 3 : Multi-tenant avec versions différentes

```plaintext
Base de données multi-tenant :

Tenant A (schema_a) → Version 2.0.0  
Tenant B (schema_b) → Version 2.1.0  
Tenant C (schema_c) → Version 1.9.0  

Chaque tenant peut être à une version différente.

Gestion :
  Table globale : tenant_schema_versions
    tenant_id | schema_version | last_updated
    ----------|----------------|-------------
    A         | 2.0.0          | 2025-01-15
    B         | 2.1.0          | 2025-02-01
    C         | 1.9.0          | 2024-12-20

  Application détecte le tenant :
    1. Identifier le tenant (ex: tenant_id dans JWT)
    2. Consulter tenant_schema_versions
    3. Adapter le code selon la version du schéma
    4. Exécuter les requêtes appropriées

  Migration progressive :
    - Migrer tenant par tenant
    - Rollback facile par tenant
    - Flexibilité maximale
```

## Outils et automatisation

### 1. Scripts de versionnement

**Script pour incrémenter la version (conceptuel) :**

```bash
#!/bin/bash
# bump_version.sh

CURRENT_VERSION=$(cat schema_version.txt)  
CHANGE_TYPE=$1  # major | minor | patch  

# Parser la version actuelle
IFS='.' read -r -a version <<< "$CURRENT_VERSION"  
MAJOR=${version[0]}  
MINOR=${version[1]}  
PATCH=${version[2]}  

# Incrémenter selon le type
case $CHANGE_TYPE in
  major)
    MAJOR=$((MAJOR + 1))
    MINOR=0
    PATCH=0
    ;;
  minor)
    MINOR=$((MINOR + 1))
    PATCH=0
    ;;
  patch)
    PATCH=$((PATCH + 1))
    ;;
esac

NEW_VERSION="$MAJOR.$MINOR.$PATCH"  
echo $NEW_VERSION > schema_version.txt  

echo "Version incrémentée : $CURRENT_VERSION → $NEW_VERSION"
```

### 2. Hooks Git

**Hook pre-commit pour vérifier la cohérence :**

```bash
#!/bin/bash
# .git/hooks/pre-commit

# Vérifier que CHANGELOG.md est à jour
if git diff --cached --name-only | grep -q "migrations/"; then
    if ! git diff --cached --name-only | grep -q "CHANGELOG.md"; then
        echo "❌ Erreur : Migration détectée mais CHANGELOG.md non modifié"
        exit 1
    fi
fi

# Vérifier que schema_version.txt est à jour
if git diff --cached --name-only | grep -q "migrations/"; then
    if ! git diff --cached --name-only | grep -q "schema_version.txt"; then
        echo "❌ Erreur : Migration détectée mais schema_version.txt non modifié"
        exit 1
    fi
fi

echo "✅ Checks passed"  
exit 0  
```

### 3. Génération automatique de documentation

**Script pour générer la matrice de compatibilité :**

```python
# generate_compatibility_matrix.py (conceptuel)

def generate_matrix():
    # Lire toutes les versions de schéma
    schema_versions = read_schema_versions()

    # Lire toutes les versions d'application
    app_versions = read_app_versions()

    # Générer la matrice
    matrix = []
    for schema_v in schema_versions:
        row = [schema_v]
        for app_v in app_versions:
            compatible = check_compatibility(schema_v, app_v)
            row.append('✅' if compatible else '❌')
        matrix.append(row)

    # Générer Markdown
    return format_as_markdown(matrix)
```

## Erreurs courantes à éviter

### ❌ Erreur 1 : Modifier une migration appliquée

```plaintext
Mauvais :
  V001__create_users.sql déjà appliqué en production
  → Quelqu'un modifie V001__create_users.sql
  → Checksum mismatch → Erreur

Bon :
  Créer V005__fix_users_table.sql pour corriger
```

### ❌ Erreur 2 : Sauter des versions

```plaintext
Mauvais :
  v1.0.0 → v1.2.0 (où est v1.1.0 ?)

Bon :
  v1.0.0 → v1.1.0 → v1.2.0 (séquence complète)
```

### ❌ Erreur 3 : Breaking change en MINOR/PATCH

```plaintext
Mauvais :
  v1.2.0 → v1.3.0 : DROP COLUMN users.name (breaking !)

Bon :
  v1.2.0 → v2.0.0 : DROP COLUMN users.name (MAJOR bump)
```

### ❌ Erreur 4 : Pas de rollback testé

```plaintext
Mauvais :
  Déployer v2.0.0 sans tester le rollback
  → Problème en production → Impossible de revenir

Bon :
  Avant déploiement :
    1. Appliquer v2.0.0 en staging
    2. Tester le rollback vers v1.9.0
    3. Ré-appliquer v2.0.0
    4. Valider → Déployer en production
```

### ❌ Erreur 5 : Documentation obsolète

```plaintext
Mauvais :
  CHANGELOG.md : dernière entrée v1.5.0
  Schéma réel : v2.3.0
  → Personne ne sait ce qui a changé

Bon :
  CHANGELOG.md synchronisé avec chaque migration
  Automatiser la mise à jour si possible
```

### ❌ Erreur 6 : Versions incohérentes entre environnements

```plaintext
Mauvais :
  Dev : v2.5.0
  Staging : v2.2.0
  Production : v2.4.0
  → Chaos total

Bon :
  Dev >= Staging >= Production
  Flow unidirectionnel : Dev → Staging → Production
```

## Conclusion

Le **schema versioning** est un pilier fondamental de la gestion professionnelle de bases de données PostgreSQL. Il transforme une base de données "boîte noire" en un système **traçable, prévisible et collaboratif**.

### Principes clés à retenir

1. **Choisissez une stratégie** : Sequential, Timestamp, ou Semantic (SemVer recommandé)  
2. **Documentez tout** : CHANGELOG.md, commentaires, table de métadonnées  
3. **Compatibilité d'abord** : Expand-Migrate-Contract pour éviter les breaking changes brutaux  
4. **Automatisez** : CI/CD, checks, génération de documentation  
5. **Testez les rollbacks** : Même si vous ne les utilisez jamais, ils doivent fonctionner  
6. **Communiquez** : Matrice de compatibilité, deprecation policy, notifications

### Recommandations finales

**Pour les petits projets :**
- Sequential versioning (simple et efficace)
- Migrations Flyway ou Alembic
- Documentation minimale (CHANGELOG.md)

**Pour les projets moyens :**
- Semantic Versioning (MAJOR.MINOR.PATCH)
- Table schema_version dans PostgreSQL
- CI/CD avec checks automatiques
- Matrice de compatibilité

**Pour les grands projets / microservices :**
- Semantic Versioning strict
- Deprecation policy formalisée
- Schema adapters ou Feature flags
- Dashboard de monitoring
- Documentation exhaustive et automatisée
- Blue-Green deployments

Le schema versioning n'est pas une contrainte, c'est une **libération** : il vous permet de faire évoluer votre base de données en toute confiance, de collaborer efficacement, et de dormir tranquille sachant que vous pouvez toujours revenir en arrière.
