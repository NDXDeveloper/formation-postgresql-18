# 20.2.1. Connection Pooling Côté Application

## Introduction

Le **connection pooling côté application** est une technique fondamentale pour optimiser les performances de vos applications utilisant PostgreSQL. Cette section vous expliquera pourquoi cette approche est essentielle et comment l'implémenter efficacement.

---

## Qu'est-ce qu'une connexion à une base de données ?

Avant de parler de pooling, commençons par les bases.

### Le concept de connexion

Lorsqu'une application veut communiquer avec PostgreSQL, elle doit établir une **connexion** :

1. **Initiation** : L'application envoie une demande de connexion au serveur PostgreSQL  
2. **Authentification** : PostgreSQL vérifie les identifiants (utilisateur, mot de passe)  
3. **Établissement** : Une fois validée, la connexion est active et utilisable  
4. **Communication** : L'application peut maintenant envoyer des requêtes SQL  
5. **Fermeture** : Quand l'application a fini, elle ferme la connexion

### Le coût d'une connexion

Créer une connexion n'est **pas gratuit** en termes de ressources :

- **Temps** : L'établissement prend entre 10 et 100 ms (millisecondes)  
- **Mémoire** : Chaque connexion consomme ~10 MB de RAM côté serveur  
- **CPU** : L'authentification et l'initialisation consomment du processeur  
- **Processus** : PostgreSQL crée un processus système pour chaque connexion

**Analogie** : Imaginez que vous allez au restaurant. À chaque fois, vous devez :
1. Faire la queue à l'entrée  
2. Montrer votre réservation  
3. Être installé à une table  
4. Commander et manger  
5. Payer et partir

Si vous faites ça 100 fois par heure pour chaque petit plat, c'est extrêmement inefficace !

---

## Le problème : L'approche naïve

### Scénario classique sans pooling

Voici ce qui se passe dans une application **sans** connection pooling :

```
Requête 1 arrive → Ouvrir connexion → Exécuter SQL → Fermer connexion  
Requête 2 arrive → Ouvrir connexion → Exécuter SQL → Fermer connexion  
Requête 3 arrive → Ouvrir connexion → Exécuter SQL → Fermer connexion  
...
```

### Les conséquences

#### 1. **Performance dégradée**
- Chaque requête attend l'établissement de la connexion
- Latence ajoutée de 10-100 ms par requête
- L'utilisateur attend plus longtemps

#### 2. **Surcharge du serveur**
- Création/destruction constante de processus
- Consommation excessive de CPU et RAM
- Le serveur peut saturer avec seulement quelques dizaines d'utilisateurs

#### 3. **Limite de connexions**
PostgreSQL a une limite de connexions simultanées (paramètre `max_connections`, souvent 100-200 par défaut).

**Exemple concret** :
```
Votre application web a 50 utilisateurs simultanés.  
Chaque page fait 20 requêtes SQL.  
Sans pooling : 50 × 20 = 1000 connexions nécessaires !  
→ PostgreSQL va refuser les connexions → Erreurs applicatives
```

---

## La solution : Connection Pooling

### Principe fondamental

Le **connection pooling** consiste à maintenir un ensemble (un "pool") de connexions **réutilisables** :

```
Application démarrage → Créer 10 connexions → Les garder ouvertes dans un pool

Requête 1 arrive → Emprunter une connexion du pool → Exécuter SQL → Rendre la connexion au pool  
Requête 2 arrive → Emprunter une connexion du pool → Exécuter SQL → Rendre la connexion au pool  
Requête 3 arrive → Emprunter une connexion du pool → Exécuter SQL → Rendre la connexion au pool  
```

**Analogie** : C'est comme un parking de taxis. Les taxis (connexions) attendent au parking (pool). Quand un client arrive, il prend un taxi disponible, fait son trajet, et le taxi retourne au parking pour le prochain client.

### Avantages immédiats

- ✅ **Performance** : Pas de délai de création de connexion pour chaque requête  
- ✅ **Efficacité** : Réutilisation des ressources existantes  
- ✅ **Scalabilité** : 10 connexions peuvent servir 1000 requêtes  
- ✅ **Stabilité** : Pas de saturation du serveur PostgreSQL

---

## Configuration du Pool : Les paramètres clés

### 1. Taille du pool (Pool Size)

C'est le nombre de connexions **maintenues ouvertes** dans le pool.

**Exemple de configuration** :
```
pool_size = 10  → Le pool maintient 10 connexions actives
```

**Comment choisir la taille ?**

**Règle générale** :
```
pool_size = nombre_de_cores_CPU × 2 + nombre_de_disques
```

**Exemples pratiques** :
- Serveur avec 4 cores CPU et 1 disque : `4 × 2 + 1 = 9` → **pool_size = 10**
- Serveur avec 8 cores CPU et 2 disques : `8 × 2 + 2 = 18` → **pool_size = 20**

⚠️ **Attention** : Plus n'est pas toujours mieux ! Trop de connexions saturent PostgreSQL.

### 2. Max overflow (Débordement maximum)

Nombre de connexions **supplémentaires** créées temporairement si le pool est saturé.

**Exemple** :
```
pool_size = 10  
max_overflow = 5  

→ Maximum absolu : 10 + 5 = 15 connexions
```

**Comportement** :
- Les 10 connexions du pool sont prises → Créer jusqu'à 5 connexions temporaires
- Ces connexions temporaires sont **fermées** après utilisation (pas réutilisées)

### 3. Timeout (Délai d'attente)

Temps maximum qu'une requête attend pour obtenir une connexion disponible.

**Exemple** :
```
pool_timeout = 30  → Attendre 30 secondes maximum
```

**Comportement** :
- Le pool est plein (toutes les connexions utilisées)
- Une nouvelle requête arrive
- Elle attend jusqu'à 30 secondes qu'une connexion se libère
- Si timeout expiré → **Erreur** : "Timeout waiting for connection from pool"

### 4. Connection lifetime (Durée de vie)

Durée maximale qu'une connexion peut rester dans le pool avant d'être renouvelée.

**Exemple** :
```
max_lifetime = 3600  → Renouveler les connexions toutes les heures (3600 secondes)
```

**Pourquoi ?**
- Éviter les connexions "stale" (périmées)
- Forcer la reconnexion pour récupérer les changements de configuration
- Libérer les ressources accumulées côté serveur

## Implémentations par langage

### Python : psycopg3 avec pool intégré

**Installation** :
```bash
pip install psycopg[pool]
```

**Code exemple** :
```python
from psycopg_pool import ConnectionPool

# Créer le pool au démarrage de l'application
pool = ConnectionPool(
    conninfo="postgresql://user:password@localhost/mydb",
    min_size=5,        # Minimum 5 connexions maintenues
    max_size=20,       # Maximum 20 connexions au total
    timeout=30,        # Attendre 30s pour une connexion
    max_idle=600,      # Fermer connexions inactives après 10 min
    max_lifetime=3600  # Renouveler après 1 heure
)

# Utiliser une connexion du pool
def get_users():
    with pool.connection() as conn:
        with conn.cursor() as cur:
            cur.execute("SELECT * FROM users")
            return cur.fetchall()
    # La connexion retourne automatiquement au pool ici

# Fermer le pool à l'arrêt de l'application
pool.close()
```

**Points clés** :
- `with pool.connection()` : Emprunte une connexion
- La connexion est **automatiquement rendue** au pool à la fin du bloc `with`
- Pas besoin de fermer manuellement (`conn.close()` ne ferme pas vraiment, mais rend au pool)

### Node.js : node-postgres (pg)

**Installation** :
```bash
npm install pg
```

**Code exemple** :
```javascript
const { Pool } = require('pg');

// Créer le pool au démarrage
const pool = new Pool({
  host: 'localhost',
  database: 'mydb',
  user: 'user',
  password: 'password',
  port: 5432,

  // Configuration du pool
  max: 20,                    // Maximum 20 connexions
  min: 5,                     // Minimum 5 connexions maintenues
  idleTimeoutMillis: 30000,   // Fermer connexions inactives après 30s
  connectionTimeoutMillis: 5000, // Timeout d'attente 5s
  maxLifetimeSeconds: 3600    // Renouveler après 1h
});

// Utiliser une connexion du pool
async function getUsers() {
  const client = await pool.connect(); // Emprunter une connexion

  try {
    const result = await client.query('SELECT * FROM users');
    return result.rows;
  } finally {
    client.release(); // IMPORTANT : Rendre la connexion au pool
  }
}

// Alternative plus simple avec pool.query()
async function getUsersSimple() {
  // pool.query() gère automatiquement connect() et release()
  const result = await pool.query('SELECT * FROM users');
  return result.rows;
}

// Fermer le pool à l'arrêt
await pool.end();
```

**Points clés** :
- `pool.connect()` : Emprunte une connexion (retourne une Promise)  
- `client.release()` : **CRITIQUE** - Rend la connexion au pool  
- `pool.query()` : Shortcut qui gère automatiquement connect/release

⚠️ **Erreur fréquente** : Oublier `client.release()` → Fuite de connexions → Pool saturé

### Java : HikariCP (Le pool le plus performant)

**Maven dependency** :
```xml
<dependency>
    <groupId>com.zaxxer</groupId>
    <artifactId>HikariCP</artifactId>
    <version>5.1.0</version>
</dependency>
<dependency>
    <groupId>org.postgresql</groupId>
    <artifactId>postgresql</artifactId>
    <version>42.7.0</version>
</dependency>
```

**Code exemple** :
```java
import com.zaxxer.hikari.HikariConfig;  
import com.zaxxer.hikari.HikariDataSource;  
import java.sql.Connection;  
import java.sql.ResultSet;  
import java.sql.Statement;  

public class DatabasePool {
    private static HikariDataSource dataSource;

    static {
        // Configuration du pool au démarrage
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:postgresql://localhost:5432/mydb");
        config.setUsername("user");
        config.setPassword("password");

        // Configuration du pool
        config.setMaximumPoolSize(20);           // Max 20 connexions
        config.setMinimumIdle(5);                // Min 5 connexions en attente
        config.setConnectionTimeout(30000);      // Timeout 30s
        config.setIdleTimeout(600000);           // Inactivité 10 min
        config.setMaxLifetime(1800000);          // Durée de vie 30 min
        config.setAutoCommit(true);              // Auto-commit activé
        config.setPoolName("PostgreSQL-Pool");

        dataSource = new HikariDataSource(config);
    }

    public static List<User> getUsers() throws SQLException {
        // Emprunter une connexion avec try-with-resources
        try (Connection conn = dataSource.getConnection();
             Statement stmt = conn.createStatement();
             ResultSet rs = stmt.executeQuery("SELECT * FROM users")) {

            List<User> users = new ArrayList<>();
            while (rs.next()) {
                users.add(new User(rs.getInt("id"), rs.getString("name")));
            }
            return users;
        }
        // La connexion retourne automatiquement au pool ici
    }

    // Fermer le pool à l'arrêt de l'application
    public static void shutdown() {
        if (dataSource != null) {
            dataSource.close();
        }
    }
}
```

**Points clés** :
- `try-with-resources` : Rend automatiquement la connexion au pool
- HikariCP est **extrêmement performant** (le plus rapide du marché)
- Configuration via `HikariConfig` ou fichier properties

### Go : pgx (Pool natif)

**Installation** :
```bash
go get github.com/jackc/pgx/v5/pgxpool
```

**Code exemple** :
```go
package main

import (
    "context"
    "log"
    "github.com/jackc/pgx/v5/pgxpool"
)

var pool *pgxpool.Pool

func initPool() {
    // Configuration du pool
    config, err := pgxpool.ParseConfig(
        "postgres://user:password@localhost:5432/mydb" +
        "?pool_max_conns=20" +           // Max 20 connexions
        "&pool_min_conns=5" +             // Min 5 connexions
        "&pool_max_conn_lifetime=1h" +   // Durée de vie 1h
        "&pool_max_conn_idle_time=30m",  // Inactivité 30 min
    )
    if err != nil {
        log.Fatal(err)
    }

    pool, err = pgxpool.NewWithConfig(context.Background(), config)
    if err != nil {
        log.Fatal(err)
    }
}

func getUsers(ctx context.Context) ([]User, error) {
    // Exécuter une requête (gère automatiquement le pool)
    rows, err := pool.Query(ctx, "SELECT id, name FROM users")
    if err != nil {
        return nil, err
    }
    defer rows.Close()

    var users []User
    for rows.Next() {
        var u User
        err := rows.Scan(&u.ID, &u.Name)
        if err != nil {
            return nil, err
        }
        users = append(users, u)
    }

    return users, rows.Err()
}

func main() {
    initPool()
    defer pool.Close() // Fermer à l'arrêt

    users, err := getUsers(context.Background())
    if err != nil {
        log.Fatal(err)
    }
    // ...
}
```

**Points clés** :
- `pool.Query()` et `pool.Exec()` : Gèrent automatiquement le pool
- Pas besoin d'acquérir/relâcher manuellement les connexions
- Support natif du context Go pour les timeouts

### .NET : Npgsql avec pool intégré

**Installation (NuGet)** :
```bash
dotnet add package Npgsql
```

**Code exemple** :
```csharp
using Npgsql;  
using System.Collections.Generic;  

public class DatabaseService
{
    // Connection string avec paramètres de pool
    private const string ConnectionString =
        "Host=localhost;" +
        "Database=mydb;" +
        "Username=user;" +
        "Password=password;" +
        "Pooling=true;" +              // Activer le pooling (true par défaut)
        "Minimum Pool Size=5;" +       // Min 5 connexions
        "Maximum Pool Size=20;" +      // Max 20 connexions
        "Connection Idle Lifetime=600;" + // Inactivité 10 min (secondes)
        "Connection Pruning Interval=10;" // Vérification toutes les 10s
        "Connection Lifetime=3600;";   // Durée de vie 1h

    public List<User> GetUsers()
    {
        var users = new List<User>();

        // using garantit que la connexion retourne au pool
        using (var conn = new NpgsqlConnection(ConnectionString))
        {
            conn.Open();

            using (var cmd = new NpgsqlCommand("SELECT id, name FROM users", conn))
            using (var reader = cmd.ExecuteReader())
            {
                while (reader.Read())
                {
                    users.Add(new User
                    {
                        Id = reader.GetInt32(0),
                        Name = reader.GetString(1)
                    });
                }
            }
        } // La connexion retourne au pool ici

        return users;
    }
}
```

**Points clés** :
- Le pooling est **activé par défaut** dans Npgsql
- Configuration via connection string
- `using` statement : Libère automatiquement la connexion vers le pool

## Bonnes pratiques

### 1. ✅ Toujours libérer les connexions

**Mauvais exemple (Node.js)** :
```javascript
async function badExample() {
  const client = await pool.connect();
  const result = await client.query('SELECT * FROM users');
  return result.rows;
  // OUBLI : client.release() non appelé → Fuite de connexion !
}
```

**Bon exemple** :
```javascript
async function goodExample() {
  const client = await pool.connect();
  try {
    const result = await client.query('SELECT * FROM users');
    return result.rows;
  } finally {
    client.release(); // TOUJOURS dans un finally
  }
}
```

### 2. ✅ Utiliser les méthodes de haut niveau quand possible

**Préférez** :
```javascript
// Pool gère tout automatiquement
const result = await pool.query('SELECT * FROM users');
```

**Au lieu de** :
```javascript
// Gestion manuelle (plus de risques d'erreur)
const client = await pool.connect();  
try {  
  const result = await client.query('SELECT * FROM users');
  return result;
} finally {
  client.release();
}
```

### 3. ✅ Dimensionner correctement le pool

**Formule de base** :
```
pool_size ≈ nombre_de_CPU_cores × 2
```

**Exemple** : Serveur avec 4 cores → **pool_size = 8 à 10**

⚠️ **Éviter** :
- Pool trop petit → Requêtes en attente, timeouts
- Pool trop grand → Surcharge PostgreSQL, performances dégradées

### 4. ✅ Surveiller les métriques du pool

**Métriques importantes à monitorer** :
- Nombre de connexions actives
- Nombre de connexions en attente
- Temps d'attente moyen pour obtenir une connexion
- Nombre de timeouts

**Exemple avec HikariCP (Java)** :
```java
HikariPoolMXBean poolMBean = dataSource.getHikariPoolMXBean();  
System.out.println("Active: " + poolMBean.getActiveConnections());  
System.out.println("Idle: " + poolMBean.getIdleConnections());  
System.out.println("Waiting: " + poolMBean.getThreadsAwaitingConnection());  
System.out.println("Total: " + poolMBean.getTotalConnections());  
```

### 5. ✅ Configurer des timeouts appropriés

**Recommandations** :
- `connection_timeout` : 5-30 secondes (temps d'attente pour obtenir une connexion)  
- `statement_timeout` : 30-60 secondes (temps max d'exécution d'une requête)  
- `idle_timeout` : 10-30 minutes (fermer les connexions inactives)  
- `max_lifetime` : 30-60 minutes (renouveler les connexions périodiquement)

### 6. ✅ Gérer les erreurs de pool saturé

**Exemple de gestion d'erreur (Python)** :
```python
from psycopg_pool import PoolTimeout

try:
    with pool.connection() as conn:
        # Votre code ici
        pass
except PoolTimeout:
    # Pool saturé - loguer et alerter
    logger.error("Pool exhausted - too many concurrent requests")
    # Possiblement : augmenter pool_size ou max_overflow
    raise
```

### 7. ✅ Utiliser un seul pool par application

**Architecture recommandée** :
```
Application (1 instance)
    └── Connection Pool (1 seul pool, singleton)
        └── PostgreSQL (1 serveur)
```

**À éviter** : Créer plusieurs pools dans la même application
```
Application
    ├── Pool 1 (10 connexions)
    ├── Pool 2 (10 connexions)
    └── Pool 3 (10 connexions)
        → Total : 30 connexions gaspillées !
```

### 8. ✅ Fermer le pool à l'arrêt de l'application

**Exemple (Node.js avec graceful shutdown)** :
```javascript
const pool = new Pool({...});

// Gérer les signaux d'arrêt
process.on('SIGTERM', async () => {
  console.log('SIGTERM received, closing pool...');
  await pool.end();
  process.exit(0);
});

process.on('SIGINT', async () => {
  console.log('SIGINT received, closing pool...');
  await pool.end();
  process.exit(0);
});
```

## Diagnostic : Comment savoir si votre pool fonctionne bien ?

### Indicateurs de problèmes

#### 🔴 **Pool trop petit**
**Symptômes** :
- Erreurs "Timeout waiting for connection"
- Requêtes qui prennent soudainement beaucoup plus de temps
- Pics de latence inexpliqués

**Solution** : Augmenter `pool_size` et/ou `max_overflow`

#### 🔴 **Pool trop grand**
**Symptômes** :
- PostgreSQL montre beaucoup de connexions idle
- CPU élevé côté PostgreSQL sans raison
- Performances qui se dégradent avec plus de connexions

**Solution** : Réduire `pool_size`

#### 🔴 **Fuites de connexions**
**Symptômes** :
- Connexions actives qui augmentent constamment
- Pool qui finit par être saturé même avec peu de requêtes
- Application qui devient lente avec le temps

**Solution** : Chercher les `client.release()` manquants ou les `conn.close()` oubliés

### Commandes PostgreSQL pour diagnostic

**Voir toutes les connexions actives** :
```sql
SELECT
    pid,
    usename,
    application_name,
    client_addr,
    state,
    state_change,
    query,
    now() - state_change AS duration
FROM pg_stat_activity  
WHERE datname = 'mydb'  
ORDER BY state_change;  
```

**Compter les connexions par état** :
```sql
SELECT
    state,
    COUNT(*) as count
FROM pg_stat_activity  
WHERE datname = 'mydb'  
GROUP BY state;  
```

**États possibles** :
- `active` : Connexion en train d'exécuter une requête  
- `idle` : Connexion inactive (dans le pool, en attente)  
- `idle in transaction` : ⚠️ Transaction ouverte mais inactive (problème !)

## Cas d'usage : Application web typique

### Scénario

Vous développez une **API REST** avec :
- 100 requêtes HTTP par seconde en moyenne
- Chaque requête fait 1-3 requêtes SQL
- Serveur avec 4 cores CPU

### Configuration recommandée

**Node.js (express + pg)** :
```javascript
const { Pool } = require('pg');

const pool = new Pool({
  host: 'localhost',
  database: 'api_db',
  user: 'api_user',
  password: 'secure_password',

  // Configuration optimale pour ce cas
  max: 10,                          // 4 cores × 2 = 8, arrondi à 10
  min: 2,                           // Toujours 2 connexions prêtes
  idleTimeoutMillis: 30000,         // Fermer inactives après 30s
  connectionTimeoutMillis: 5000,    // Timeout 5s
  maxLifetimeSeconds: 3600          // Renouveler après 1h
});

// Middleware express pour vérifier le pool
app.use((req, res, next) => {
  console.log('Pool stats:', {
    total: pool.totalCount,
    idle: pool.idleCount,
    waiting: pool.waitingCount
  });
  next();
});

// Route exemple
app.get('/api/users', async (req, res) => {
  try {
    const result = await pool.query('SELECT * FROM users LIMIT 100');
    res.json(result.rows);
  } catch (error) {
    console.error('Database error:', error);
    res.status(500).json({ error: 'Database error' });
  }
});
```

### Résultats attendus

Avec cette configuration :
- ✅ Les 10 connexions peuvent traiter **1000+ requêtes/seconde**  
- ✅ Latence moyenne : 2-5 ms (vs 10-100 ms sans pool)  
- ✅ Utilisation CPU PostgreSQL : ~20-30% (vs 80-90% sans pool)  
- ✅ Pas de saturation de connexions

## Résumé : Points clés à retenir

### ✨ Pourquoi le connection pooling est essentiel
1. **Performance** : Évite le coût de création/fermeture de connexions  
2. **Scalabilité** : Permet de servir beaucoup de requêtes avec peu de connexions  
3. **Stabilité** : Évite la saturation du serveur PostgreSQL

### 🔧 Configuration de base
- `pool_size` : Nombre de CPU cores × 2  
- `max_overflow` : 25-50% du pool_size  
- `timeout` : 5-30 secondes  
- `max_lifetime` : 30-60 minutes

### 💡 Bonnes pratiques
1. **Toujours** libérer les connexions (try/finally, using, with)  
2. **Utiliser** les méthodes de haut niveau quand possible  
3. **Monitorer** les métriques du pool  
4. **Configurer** des timeouts appropriés  
5. **Fermer** le pool à l'arrêt de l'application

### ⚠️ Erreurs à éviter
- Oublier de libérer les connexions → Fuites
- Pool trop grand → Surcharge PostgreSQL
- Pool trop petit → Timeouts et latence
- Créer plusieurs pools dans la même application
- Ne pas gérer les erreurs de pool saturé
