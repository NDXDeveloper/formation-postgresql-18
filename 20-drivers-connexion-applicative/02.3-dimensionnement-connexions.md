# 20.2.3. Dimensionnement (max_connections vs pool_size)

## Introduction

Le dimensionnement correct des connexions est **crucial** pour la performance et la stabilité de vos applications PostgreSQL. Trop peu de connexions et votre application sera lente, trop et PostgreSQL sera surchargé.

Dans cette section, nous allons comprendre :
- Qu'est-ce que `max_connections` (côté PostgreSQL)
- Qu'est-ce que `pool_size` (côté application/PgBouncer)
- Comment les dimensionner correctement
- Les formules et calculs pratiques

---

## Les deux types de connexions

### Vue d'ensemble

Il existe **deux niveaux** de connexions à considérer :

```
┌─────────────────┐          ┌─────────────────┐          ┌─────────────────┐
│   Application   │          │   PgBouncer     │          │   PostgreSQL    │
│                 │          │                 │          │                 │
│  Connexions     │◄────────►│  Pool           │◄────────►│  max_connections│
│  clientes       │          │  (pool_size)    │          │                 │
└─────────────────┘          └─────────────────┘          └─────────────────┘
    100-10000                      10-100                       50-200
```

**Analogie** : Imaginez un restaurant :
- **Clients** = Connexions de l'application (peuvent être des milliers)  
- **Serveurs** = Pool PgBouncer (une dizaine)  
- **Cuisiniers** = Connexions PostgreSQL (limité par la cuisine)

## max_connections : La limite de PostgreSQL

### Qu'est-ce que max_connections ?

`max_connections` est un paramètre de **configuration PostgreSQL** qui définit le **nombre maximum de connexions simultanées** que le serveur acceptera.

**Emplacement** : Fichier `postgresql.conf`
```conf
max_connections = 100    # Valeur par défaut
```

### Comment ça fonctionne ?

Quand PostgreSQL démarre :
1. Il alloue les ressources pour `max_connections` connexions  
2. Chaque connexion consomme de la mémoire (même si inutilisée)  
3. Si une nouvelle connexion arrive et dépasse la limite → **Rejet**

**Erreur typique** :
```
FATAL: sorry, too many clients already
```

### Le coût d'une connexion PostgreSQL

Chaque connexion PostgreSQL consomme des ressources :

#### Mémoire par connexion

```
Mémoire par connexion = work_mem + (shared_buffers / max_connections) + overhead

Exemple typique :
- work_mem = 4 MB
- shared_buffers = 2 GB = 2048 MB
- max_connections = 100
- overhead ≈ 2 MB

Mémoire ≈ 4 + (2048 / 100) + 2 = 26 MB par connexion
```

**Calcul total** :
```
max_connections = 100
→ Mémoire totale ≈ 100 × 26 MB = 2.6 GB rien que pour les connexions !
```

#### Processus système

Chaque connexion PostgreSQL = **1 processus système Unix/Linux**

```bash
# Voir les processus PostgreSQL
ps aux | grep postgres

# Exemple de sortie avec 50 connexions actives :
postgres  1234  ... postgres: postgres mydb 192.168.1.10(45678) idle  
postgres  1235  ... postgres: postgres mydb 192.168.1.10(45679) idle  
postgres  1236  ... postgres: postgres mydb 192.168.1.10(45680) SELECT  
...
# 50 lignes au total
```

**Coût** :
- Création/destruction de processus = coût CPU
- Context switching entre processus = overhead

### Valeurs par défaut selon les installations

| Installation | max_connections par défaut |
|-------------|----------------------------|
| PostgreSQL compilation standard | 100 |
| Debian/Ubuntu (apt) | 100 |
| Red Hat/CentOS (yum) | 100 |
| AWS RDS (db.t3.micro) | 87 |
| AWS RDS (db.t3.medium) | 112 |
| AWS RDS (db.m5.large) | 145 |
| Azure Database | Variable selon vCores |
| Docker image officiel | 100 |

### Modifier max_connections

#### Méthode 1 : Éditer postgresql.conf

```bash
# Trouver le fichier de configuration
sudo -u postgres psql -c "SHOW config_file;"

# Éditer le fichier
sudo nano /etc/postgresql/18/main/postgresql.conf

# Chercher et modifier la ligne
max_connections = 200    # Au lieu de 100

# Redémarrer PostgreSQL (OBLIGATOIRE)
sudo systemctl restart postgresql
```

⚠️ **Important** : Modifier `max_connections` nécessite un **redémarrage** de PostgreSQL.

#### Méthode 2 : Avec ALTER SYSTEM (PostgreSQL 9.4+)

```sql
-- Modifier via SQL
ALTER SYSTEM SET max_connections = 200;

-- Recharger la configuration (pas suffisant, redémarrage nécessaire)
SELECT pg_reload_conf();

-- Redémarrage obligatoire
-- sudo systemctl restart postgresql
```

### Limites pratiques de max_connections

#### Limite théorique

PostgreSQL peut techniquement supporter **jusqu'à 8388607 connexions**, mais c'est **totalement irréaliste**.

#### Limites pratiques

**Serveurs typiques** :

| Taille serveur | RAM | CPU | max_connections recommandé |
|----------------|-----|-----|---------------------------|
| Petit (dev/test) | 4 GB | 2 cores | 50-100 |
| Moyen | 16 GB | 4 cores | 100-200 |
| Grand | 64 GB | 8 cores | 200-300 |
| Très grand | 128+ GB | 16+ cores | 300-500 |

⚠️ **Au-delà de 500 connexions**, vous avez probablement un problème d'architecture.

#### Pourquoi pas 1000+ connexions ?

**Problèmes avec beaucoup de connexions** :

1. **Mémoire excessive**
   ```
   max_connections = 1000
   26 MB par connexion
   → 26 GB rien que pour les connexions !
   ```

2. **CPU surchargé**
   - 1000 processus à gérer
   - Context switching excessif
   - Performance qui s'effondre

3. **Latence accrue**
   - Le scheduler du système peine
   - Chaque requête attend plus longtemps

**Résultat** : Les performances se **dégradent** quand on augmente trop `max_connections`.

## pool_size : La taille des pools

### Qu'est-ce que pool_size ?

`pool_size` est le nombre de connexions **maintenues dans un pool** par :
- Votre application (connection pooling applicatif)
- PgBouncer (connection pooler externe)

### Différents pool_size selon le contexte

#### 1. Pool_size applicatif (par instance)

**Python (psycopg3)** :
```python
pool = ConnectionPool(
    conninfo="postgresql://...",
    min_size=5,      # Minimum 5 connexions maintenues
    max_size=20      # Maximum 20 connexions
)
```

**Node.js (pg)** :
```javascript
const pool = new Pool({
  max: 20,  // pool_size = 20
  min: 5
});
```

**Java (HikariCP)** :
```java
config.setMaximumPoolSize(20);  // pool_size = 20  
config.setMinimumIdle(5);  
```

#### 2. Pool_size PgBouncer (global)

**pgbouncer.ini** :
```ini
[pgbouncer]
default_pool_size = 10      # Pool de 10 connexions PostgreSQL  
reserve_pool_size = 5       # 5 connexions de réserve  
```

### Calcul du nombre total de connexions

#### Scénario 1 : Application directe (sans PgBouncer)

```
┌─────────────────────────────────────────────┐
│ Application (3 instances)                   │
│                                             │
│  Instance 1: pool_size=10 ──┐               │
│  Instance 2: pool_size=10 ──┼──►            │
│  Instance 3: pool_size=10 ──┘               │
│                                             │
│  Total connexions PostgreSQL = 3 × 10 = 30  │
└─────────────────────────────────────────────┘
                    │
                    ▼
        ┌───────────────────────┐
        │ PostgreSQL            │
        │ max_connections = 50  │
        │ (OK, 30 < 50)         │
        └───────────────────────┘
```

**Formule** :
```
Connexions totales = nombre_instances × pool_size_par_instance
```

#### Scénario 2 : Avec PgBouncer

```
┌─────────────────────────────────────────────┐
│ Applications (plusieurs instances)          │
│                                             │
│  App1 (3 inst): 3×10 = 30 connexions  ──┐   │
│  App2 (2 inst): 2×10 = 20 connexions  ──┼──►│
│  App3 (5 inst): 5×10 = 50 connexions  ──┘   │
│                                             │
│  Total : 100 connexions vers PgBouncer      │
└─────────────────────────────────────────────┘
                    │
                    ▼
        ┌───────────────────────┐
        │ PgBouncer             │
        │ pool_size = 20        │
        │                       │
        │ 100 clients → 20 cnx  │
        └───────────────────────┘
                    │
                    ▼
        ┌───────────────────────┐
        │ PostgreSQL            │
        │ max_connections = 50  │
        │ (OK, 20 < 50)         │
        └───────────────────────┘
```

**Avantage** : Mutualisation massive des connexions !

## La relation max_connections vs pool_size

### Règle d'or

```
┌────────────────────────────────────────────────────────┐
│  Somme de tous les pool_size < max_connections         │
│                                                        │
│  Avec marge de sécurité :                              │
│  Somme des pools ≤ (max_connections × 0.8)             │
└────────────────────────────────────────────────────────┘
```

**Pourquoi 80% ?**
- Réserver des connexions pour les admins
- Marge pour les pics temporaires
- Connexions système (autovacuum, background workers)

### Exemples de calcul

#### Exemple 1 : Application simple

**Configuration** :
- 1 application web
- 3 instances (load balancing)
- Pool par instance = 10 connexions

**Calcul** :
```
Total connexions = 3 × 10 = 30

max_connections nécessaire = 30 / 0.8 = 37.5
→ max_connections = 50 (arrondi supérieur)
```

#### Exemple 2 : Plusieurs applications

**Configuration** :
- Application Web : 5 instances × 10 = 50 connexions
- Worker background : 2 instances × 5 = 10 connexions
- Reporting : 1 instance × 5 = 5 connexions

**Calcul** :
```
Total connexions = 50 + 10 + 5 = 65

max_connections nécessaire = 65 / 0.8 = 81.25
→ max_connections = 100
```

#### Exemple 3 : Avec PgBouncer

**Configuration** :
- 10 microservices
- Chaque service : 3 instances × 10 = 30 connexions
- Total vers PgBouncer : 10 × 30 = 300 connexions
- PgBouncer pool_size = 20

**Calcul** :
```
Connexions vers PostgreSQL = 20 (pool PgBouncer)

max_connections nécessaire = 20 / 0.8 = 25
→ max_connections = 50 (avec beaucoup de marge)
```

**Économie massive** : 300 → 20 connexions !

## Formules de dimensionnement

### Formule générale pour pool_size (sans PgBouncer)

```
pool_size = (nombre_de_CPU_cores × 2) + nombre_de_disques
```

**Explication** : Cette formule vient des contraintes hardware :
- **CPU** : Nombre de tâches parallèles possibles  
- **Disques** : I/O en attente pendant les lectures/écritures

**Exemples** :

| Serveur | CPU Cores | Disques | pool_size recommandé |
|---------|-----------|---------|---------------------|
| Petit | 2 | 1 | (2×2) + 1 = 5 |
| Moyen | 4 | 1 | (4×2) + 1 = 9 ≈ **10** |
| Grand | 8 | 2 | (8×2) + 2 = 18 ≈ **20** |
| Très grand | 16 | 4 | (16×2) + 4 = 36 ≈ **40** |

### Formule pour max_connections

```
max_connections = (total_de_tous_les_pools / 0.8) + connexions_réservées

où :
- total_de_tous_les_pools = somme de tous les pool_size
- 0.8 = coefficient de sécurité (80%)
- connexions_réservées = ~10 pour admin/monitoring/maintenance
```

**Exemple détaillé** :
```
Applications :
- API REST : 5 instances × 10 = 50
- Worker : 2 instances × 5 = 10
- Total pools : 60

max_connections = (60 / 0.8) + 10
                = 75 + 10
                = 85
→ Configurer max_connections = 100
```

### Formule avec PgBouncer

```
pool_size_app = 5 à 10 par instance  
pool_size_pgbouncer = (nb_instances_totales × 2) / 10  

max_connections = (pool_size_pgbouncer / 0.8) + 10
```

**Exemple** :
```
Applications :
- 20 instances au total
- pool_size_app = 5 par instance

pool_size_pgbouncer = (20 × 2) / 10 = 4
→ Utiliser pool_size = 10 (minimum recommandé)

max_connections = (10 / 0.8) + 10 = 22.5
→ Configurer max_connections = 50 (avec marge)
```

## Cas d'usage réels

### Cas 1 : Startup - Application web simple

**Contexte** :
- 1 serveur web (Node.js/Express)
- 1 serveur PostgreSQL (4 cores, 16 GB RAM)
- Trafic : 100 requêtes/seconde
- Utilisateurs : ~1000

**Dimensionnement** :

```
Serveur web :
- 1 instance Node.js
- pool_size = 10

PostgreSQL :
- 4 cores → pool optimal = (4×2) + 1 = 9 ≈ 10
- max_connections = (10 / 0.8) + 10 = 22.5

Configuration :
- Pool app : 10
- max_connections : 50 (avec marge pour croissance)
```

**Fichiers de configuration** :

**Node.js** :
```javascript
const pool = new Pool({
  max: 10,
  min: 2
});
```

**postgresql.conf** :
```conf
max_connections = 50  
shared_buffers = 4GB      # 25% de la RAM  
work_mem = 32MB  
```

### Cas 2 : Scale-up - Application en croissance

**Contexte** :
- 3 serveurs web (load balanced)
- 1 serveur PostgreSQL (8 cores, 32 GB RAM)
- Trafic : 500 requêtes/seconde
- Utilisateurs : ~10000

**Dimensionnement** :

```
Serveurs web :
- 3 instances × pool_size 10 = 30 connexions

PostgreSQL :
- 8 cores → pool optimal = (8×2) + 1 = 17 ≈ 20
- max_connections = (30 / 0.8) + 10 = 47.5

Configuration :
- Pool par instance : 10
- max_connections : 100 (anticipe la croissance)
```

**Problème potentiel** : Si on passe à 5 instances → 50 connexions
- Solution 1 : Augmenter max_connections à 100
- Solution 2 : Introduire PgBouncer

### Cas 3 : Entreprise - Microservices

**Contexte** :
- 15 microservices (Node.js, Python, Java)
- Chaque service : 3-5 instances
- Total : ~60 instances d'application
- 1 serveur PostgreSQL (16 cores, 64 GB RAM)

**Sans PgBouncer (mauvaise approche)** :
```
60 instances × 10 connexions = 600 connexions
→ IMPOSSIBLE ! PostgreSQL ne peut pas gérer 600 connexions efficacement
```

**Avec PgBouncer (bonne approche)** :
```
Applications :
- 60 instances × 5 connexions = 300 connexions vers PgBouncer

PgBouncer :
- pool_mode = transaction
- pool_size = 20

PostgreSQL :
- max_connections = (20 / 0.8) + 10 = 35

Configuration :
- Pool par app : 5
- PgBouncer pool : 20
- max_connections : 50
```

**Architecture** :
```
┌────────────────────────────────────┐
│ 60 instances app (300 connexions)  │
└──────────────┬─────────────────────┘
               │
               ▼
        ┌──────────────┐
        │  PgBouncer   │
        │  pool = 20   │
        └──────┬───────┘
               │
               ▼
        ┌──────────────┐
        │ PostgreSQL   │
        │ max_conn=50  │
        └──────────────┘
```

### Cas 4 : Serverless (AWS Lambda, Cloud Functions)

**Contexte** :
- Fonctions Lambda déployées
- Chaque invocation = nouvelle connexion potentielle
- Trafic variable (0 à 1000 req/sec)

**Problème** :
```
Pic de 1000 requêtes simultanées
→ 1000 connexions PostgreSQL demandées
→ CRASH !
```

**Solution** :
```
Lambda → RDS Proxy (ou PgBouncer sur EC2/ECS) → RDS PostgreSQL

Configuration :
- RDS Proxy pool : 20-30 connexions
- max_connections : 50
```

**Exemple AWS RDS Proxy** :
```json
{
  "MaxConnectionsPercent": 50,
  "MaxIdleConnectionsPercent": 20,
  "ConnectionBorrowTimeout": 120
}
```

### Cas 5 : Application legacy + nouvelles applications

**Contexte** :
- Application legacy (Java) : 5 instances × 20 = 100 connexions
- Nouvelles APIs (Node.js) : 10 instances × 10 = 100 connexions
- Total : 200 connexions

**Problème** :
```
200 connexions directes = PostgreSQL surchargé
```

**Solution progressive** :

**Phase 1 : PgBouncer pour nouvelles apps seulement**
```
Legacy (Java) : 100 connexions directes  
Nouvelles APIs → PgBouncer (pool=20) → PostgreSQL  

Total vers PostgreSQL : 100 + 20 = 120  
max_connections = 150  
```

**Phase 2 : Migration complète vers PgBouncer**
```
Legacy → PgBouncer #1 (session mode, pool=30)  
APIs → PgBouncer #2 (transaction mode, pool=20)  

Total vers PostgreSQL : 30 + 20 = 50  
max_connections = 100  
```

## Problèmes courants et solutions

### Problème 1 : Pool trop petit

**Symptômes** :
```
- Erreurs "timeout waiting for connection"
- Latence élevée et variable
- Logs montrant beaucoup de connexions en attente
```

**Diagnostic** :
```sql
-- PgBouncer : Voir les clients en attente
SHOW POOLS;

-- Application : Logs
"PoolTimeout: Timed out waiting for connection from pool"
```

**Solution** :
```
Augmenter progressivement le pool_size :
- Était 10 → Essayer 15
- Monitorer
- Puis 20 si nécessaire
```

### Problème 2 : Pool trop grand

**Symptômes** :
```
- CPU élevé sur PostgreSQL sans charge apparente
- Beaucoup de connexions "idle"
- Performance qui se dégrade avec plus de connexions
```

**Diagnostic** :
```sql
-- Voir les connexions idle
SELECT
    count(*) as idle_count
FROM pg_stat_activity  
WHERE state = 'idle';  

-- Si idle_count > pool_size × 0.5 : pool trop grand
```

**Solution** :
```
Réduire le pool_size :
- Était 50 → Essayer 30
- Était 30 → Essayer 20
```

### Problème 3 : max_connections atteint

**Symptôme** :
```
FATAL: sorry, too many clients already  
FATAL: remaining connection slots are reserved for non-replication superuser connections  
```

**Diagnostic** :
```sql
-- Voir le nombre de connexions actuelles
SELECT
    count(*) as current,
    current_setting('max_connections')::int as max,
    (count(*)::float / current_setting('max_connections')::int * 100)::numeric(5,2) as percent
FROM pg_stat_activity;

-- Si percent > 90% : problème imminent
```

**Solutions** :

**Solution immédiate** :
```sql
-- Tuer les connexions idle depuis longtemps
SELECT pg_terminate_backend(pid)  
FROM pg_stat_activity  
WHERE state = 'idle'  
  AND state_change < now() - interval '1 hour';
```

**Solution à long terme** :
```
Option 1 : Augmenter max_connections
- Éditer postgresql.conf
- Redémarrer PostgreSQL

Option 2 : Introduire PgBouncer (recommandé)
- Réduire les connexions directes
- Mutualiser via PgBouncer
```

### Problème 4 : Croissance non contrôlée

**Scénario** :
```
Déploiement d'une nouvelle version de l'app
→ Doublement des instances (scaling horizontal)
→ Doublement des connexions
→ Saturation de PostgreSQL
```

**Prévention** :

**Sans PgBouncer** :
```
Avant déploiement, calculer :  
nouvelles_connexions = nouvelle_instance_count × pool_size  

Si nouvelles_connexions > (max_connections × 0.8) :
→ Augmenter max_connections AVANT le déploiement
```

**Avec PgBouncer** :
```
PgBouncer absorbe la croissance automatiquement :
- 10 instances → 50 instances
- PgBouncer maintient toujours le même pool_size
→ Pas d'impact sur PostgreSQL
```

### Problème 5 : Connexions bloquées (idle in transaction)

**Symptôme** :
```sql
SELECT count(*)  
FROM pg_stat_activity  
WHERE state = 'idle in transaction';  

-- Si count > 10 : problème
```

**Cause** :
- Application qui ouvre une transaction mais ne la ferme pas
- Connection leak dans le code

**Solution** :
```sql
-- Configurer un timeout automatique
ALTER DATABASE mydb SET idle_in_transaction_session_timeout = '5min';

-- Ou dans postgresql.conf
idle_in_transaction_session_timeout = 300000  # 5 minutes en ms
```

## Monitoring et métriques

### Métriques essentielles à surveiller

#### 1. Utilisation de max_connections

**Requête** :
```sql
SELECT
    count(*) AS used,
    current_setting('max_connections')::int AS max,
    ROUND(
        count(*)::numeric / current_setting('max_connections')::int * 100,
        2
    ) AS percent_used
FROM pg_stat_activity;
```

**Seuils d'alerte** :
- < 50% : ✅ OK
- 50-70% : ⚠️ Surveiller
- 70-85% : ⚠️⚠️ Attention
- > 85% : 🔴 Critique - Agir immédiatement

#### 2. Connexions par état

**Requête** :
```sql
SELECT
    state,
    count(*) as count
FROM pg_stat_activity  
WHERE datname = 'mydb'  
GROUP BY state  
ORDER BY count DESC;  
```

**Résultat sain** :
```
     state     | count
---------------+-------
 active        |    12
 idle          |    45
 idle in trans |     2
```

**Problèmes** :
- `idle in transaction` > 10 : Connexions qui ne finissent pas leurs transactions  
- `idle` > pool_size × 1.5 : Pool trop grand

#### 3. Durée des connexions

**Requête** :
```sql
SELECT
    pid,
    usename,
    application_name,
    state,
    now() - backend_start as connection_age,
    now() - state_change as time_in_state
FROM pg_stat_activity  
WHERE datname = 'mydb'  
ORDER BY backend_start  
LIMIT 10;  
```

**Attention si** :
- `connection_age` > 1 heure pour connexions idle (connection leak ?)  
- `time_in_state` > 5 min pour "idle in transaction"

### Dashboard de monitoring

**Requête complète pour tableau de bord** :
```sql
WITH connection_stats AS (
    SELECT
        count(*) FILTER (WHERE state = 'active') AS active,
        count(*) FILTER (WHERE state = 'idle') AS idle,
        count(*) FILTER (WHERE state = 'idle in transaction') AS idle_in_trans,
        count(*) AS total
    FROM pg_stat_activity
    WHERE datname = current_database()
)
SELECT
    cs.*,
    current_setting('max_connections')::int AS max_connections,
    ROUND((cs.total::numeric / current_setting('max_connections')::int * 100), 2) AS percent_used,
    current_setting('max_connections')::int - cs.total AS remaining
FROM connection_stats cs;
```

**Résultat** :
```
 active | idle | idle_in_trans | total | max_connections | percent_used | remaining
--------+------+---------------+-------+-----------------+--------------+-----------
     15 |   32 |             2 |    49 |             100 |        49.00 |        51
```

### Alerting avec Prometheus

**Métriques à exporter** (via postgres_exporter) :

```yaml
# Taux d'utilisation max_connections
- alert: PostgreSQLConnectionsNearLimit
  expr: |
    (pg_stat_database_numbackends / pg_settings_max_connections) > 0.85
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "PostgreSQL connections near limit"

# Connexions idle in transaction
- alert: PostgreSQLIdleInTransaction
  expr: |
    pg_stat_activity_idle_in_transaction_count > 10
  for: 5m
  labels:
    severity: warning
```

## Stratégies d'optimisation

### Stratégie 1 : Start Small, Scale Smart

**Principe** : Commencer avec des valeurs conservatrices, puis ajuster.

**Phase 1 : Développement**
```
pool_size = 5  
max_connections = 20  
```

**Phase 2 : Staging/Test**
```
pool_size = 10  
max_connections = 50  
→ Tester en charge, monitorer
```

**Phase 3 : Production**
```
Selon résultats tests :  
pool_size = 10-20  
max_connections = 100  

Avec PgBouncer si nécessaire
```

### Stratégie 2 : Règle du 80/20

**Principe** : 80% du trafic nécessite 20% des ressources

**Application** :
```
max_connections = 100

Réservation :
- 80 connexions pour applications (80%)
- 10 connexions pour admin/monitoring (10%)
- 10 connexions de marge (10%)
```

### Stratégie 3 : Pool par type de workload

**Principe** : Différents pools pour différents usages

**Exemple** :
```
┌─────────────────────────────────────────┐
│ PgBouncer - Port 6432                   │
│ Mode: transaction                       │
│ pool_size: 20                           │
│ Usage: API REST (lectures/écritures)    │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│ PgBouncer - Port 6433                   │
│ Mode: session                           │
│ pool_size: 5                            │
│ Usage: Jobs batch (longues transactions)│

│ Connexion directe                       │
│ Usage: Admin, monitoring                │

```

### Stratégie 4 : Auto-scaling avec limites

**Principe** : Permettre la croissance tout en protégeant PostgreSQL

**Configuration HikariCP (Java)** :
```java
// Pool qui grandit avec la charge
config.setMinimumIdle(5);           // Toujours 5 connexions  
config.setMaximumPoolSize(20);      // Jamais plus de 20  
config.setConnectionTimeout(3000);   // Timeout rapide si saturé  
```

**Configuration avec PgBouncer** :
```ini
default_pool_size = 10  
reserve_pool_size = 5        # 5 connexions de réserve  
reserve_pool_timeout = 3     # Utiliser après 3 secondes d'attente  
```

## Checklist de configuration

### Pour l'application

```
☐ Définir pool_size par instance
   - Formule : (CPU_cores × 2) + disques
   - Valeur typique : 10-20

☐ Configurer les timeouts
   - Connection timeout : 5-10 secondes
   - Statement timeout : 30-60 secondes

☐ Implémenter la libération des connexions
   - Utiliser try/finally ou with statements
   - Jamais de connection leaks

☐ Monitorer le pool applicatif
   - Connexions actives
   - Connexions en attente
   - Temps d'attente moyen
```

### Pour PgBouncer (si utilisé)

```
☐ Choisir le mode approprié
   - Transaction : API REST, microservices
   - Session : Legacy, prepared statements

☐ Dimensionner le pool PgBouncer
   - Formule : (nb_instances × 2) / 10
   - Minimum : 10

☐ Configurer server_reset_query
   - Transaction mode : DISCARD ALL
   - Performance : RESET ALL

☐ Définir les timeouts
   - server_idle_timeout : 600s (10 min)
   - server_lifetime : 3600s (1h)
```

### Pour PostgreSQL

```
☐ Calculer max_connections nécessaire
   - Formule : (total_pools / 0.8) + 10
   - Minimum : 50
   - Maximum pratique : 300-500

☐ Ajuster shared_buffers
   - Formule : 25% de la RAM
   - Exemple : 16 GB RAM → 4 GB shared_buffers

☐ Configurer work_mem
   - Formule : (RAM - shared_buffers) / (max_connections × 2)
   - Exemple : (16GB - 4GB) / (100 × 2) = 60 MB

☐ Activer connection timeout
   idle_in_transaction_session_timeout = 5min

☐ Configurer les logs
   log_connections = on
   log_disconnections = on
```

### Monitoring

```
☐ Surveiller l'utilisation de max_connections
   - Alerte si > 85%

☐ Tracer les connexions idle in transaction
   - Alerte si > 10

☐ Monitorer les timeouts d'application
   - Tracer les erreurs de pool exhausté

☐ Dashboard avec métriques essentielles
   - Connexions actives/idle
   - Taux d'utilisation
   - Temps d'attente
```

## Résumé : Points clés à retenir

### ✨ Les 3 nombres magiques

1. **pool_size** (application) : 5-20 par instance  
2. **pool_size** (PgBouncer) : 10-50 selon charge  
3. **max_connections** (PostgreSQL) : 50-300 max

### 🎯 Règles d'or

1. **Plus n'est pas mieux** : Trop de connexions = performance dégradée  
2. **Utiliser PgBouncer** : Au-delà de 3-5 instances d'application  
3. **Garder de la marge** : Rester sous 80% de max_connections  
4. **Monitorer activement** : Surveiller les métriques en continu

### ⚖️ Trade-offs

| Approche | Avantages | Inconvénients |
|----------|-----------|---------------|
| **Pool petit** | Économie ressources | Risque de timeouts |
| **Pool grand** | Pas de timeouts | Surcharge PostgreSQL |
| **Sans PgBouncer** | Architecture simple | Scalabilité limitée |
| **Avec PgBouncer** | Scalabilité maximale | Complexité ajoutée |

### 💡 Quand introduire PgBouncer ?

✅ **Oui, utiliser PgBouncer si** :
- Plus de 5 instances d'application
- Microservices (plusieurs applications)
- Serverless (Lambda, Cloud Functions)
- Besoin de scaler rapidement

❌ **Non, pas nécessaire si** :
- 1-2 instances d'application
- Trafic stable et prévisible
- < 50 connexions totales
