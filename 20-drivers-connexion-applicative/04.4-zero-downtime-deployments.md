# 20.4.4. Zero-Downtime Deployments

## Introduction

Un **Zero-Downtime Deployment** (déploiement sans interruption) est une technique qui permet de mettre à jour votre application et votre base de données PostgreSQL **sans arrêter le service**, même pendant quelques secondes. Les utilisateurs continuent d'utiliser l'application normalement pendant toute la durée du déploiement.

### Métaphore : Changer les roues d'une voiture en mouvement

Imaginez que vous devez changer les pneus d'une voiture :

**Déploiement classique (avec downtime) :**
```plaintext
1. Arrêter la voiture (maintenance mode)
2. Changer les 4 pneus
3. Redémarrer la voiture
⏱️ Durée d'arrêt : 30 minutes

→ Les passagers doivent attendre, service interrompu
```

**Zero-downtime deployment :**
```plaintext
1. La voiture continue de rouler
2. Changer un pneu à la fois, sans arrêter
3. Vérifier chaque pneu individuellement
4. Continuer jusqu'à avoir changé les 4
⏱️ Durée d'arrêt : 0 seconde

→ Les passagers ne remarquent rien, service continu
```

Le zero-downtime deployment applique ce principe à votre stack technologique : application + base de données PostgreSQL.

## Pourquoi le zero-downtime est-il crucial ?

### 1. Exigences métier modernes

```plaintext
Applications 24/7/365 :
  - E-commerce : Chaque minute d'arrêt = perte de revenus
  - Services financiers : Transactions critiques ne peuvent pas attendre
  - SaaS B2B : Clients dans différents fuseaux horaires
  - Applications mobiles : Utilisateurs globaux à toute heure

Exemple concret :
  Site e-commerce avec 1000 commandes/heure
  Downtime de 30 minutes = 500 commandes perdues
  → Coût : 50 000€+ en revenus + insatisfaction client
```

### 2. Attentes utilisateurs

```plaintext
Années 2000 : "Maintenance de 2h à 3h du matin acceptable"  
Années 2025 : "Service doit être disponible en permanence"  

Les utilisateurs ne tolèrent plus les interruptions :
  ❌ "Le site sera en maintenance de 2h à 4h"
  ✅ "Mise à jour transparente, pas d'interruption"
```

### 3. Fréquence de déploiement

```plaintext
Ancienne approche :
  1 déploiement par mois → 1 maintenance mensuelle acceptable

Approche moderne (CI/CD) :
  10+ déploiements par jour → Maintenance impossible !

Exemple :
  Équipe avec 5 déploiements/jour
  Avec 5 min de downtime par déploiement
  → 25 minutes d'interruption quotidienne
  → 750 minutes/mois = 12,5 heures ! ❌ Inacceptable
```

### 4. SLA et réputation

```plaintext
SLA (Service Level Agreement) typique :
  99.9%  uptime = 43 minutes de downtime/mois ✅ Acceptable
  99.99% uptime = 4 minutes de downtime/mois ⚠️ Serré
  99.999% uptime = 26 secondes de downtime/mois 🎯 Excellence

Zero-downtime = 99.999%+ uptime garanti
```

## Les défis spécifiques avec PostgreSQL

Contrairement à une application stateless (sans état), PostgreSQL est **stateful** (avec état) : il stocke des données persistantes qui doivent rester cohérentes pendant les déploiements.

### Défi 1 : Changements de schéma

```plaintext
Problème :
  Application v1 attend : users(id, name, email)
  Migration renomme : name → full_name
  Application v2 attend : users(id, full_name, email)

  Pendant la migration :
    - App v1 déployée : SELECT name → ERREUR (colonne n'existe plus)
    - App v2 déployée : SELECT full_name → ERREUR (pas encore créée)

  → Période d'incompatibilité = downtime
```

### Défi 2 : Verrous de table

```plaintext
Certaines opérations PostgreSQL verrouillent les tables :

ALTER TABLE users ADD COLUMN phone VARCHAR(20) NOT NULL;
  → Verrouille la table users en écriture
  → Durée : proportionnelle à la taille de la table
  → Table de 100M lignes = plusieurs minutes de verrou
  → Pendant ce temps : INSERT/UPDATE/DELETE bloqués = downtime !
```

### Défi 3 : Versions d'application multiples

```plaintext
Dans une architecture distribuée :
  Instance A : App v2.0 (ancienne)
  Instance B : App v2.1 (nouvelle, en cours de déploiement)
  Instance C : App v2.0 (ancienne)

  Toutes parlent à la même base PostgreSQL
  → Le schéma doit être compatible avec v2.0 ET v2.1 simultanément
```

### Défi 4 : Rollback

```plaintext
Déploiement échoue en production :
  Schéma modifié : users.name → users.full_name
  Application v2 déployée : plante

  Rollback de l'application : facile (déployer v1)
  Rollback du schéma : complexe !
    - Renommer full_name → name ?
    - Et si des données ont été insérées entre temps ?
    - Intégrité des données compromise ?
```

## Principes fondamentaux du zero-downtime

### Principe 1 : Backward Compatibility (Rétrocompatibilité)

**Définition :** Les nouvelles versions du schéma doivent fonctionner avec les anciennes versions de l'application.

```plaintext
Schéma v2 déployé en premier
  ↓
Application v1 continue de fonctionner (backward compatible)
  ↓
Application v2 déployée progressivement
  ↓
Tout fonctionne en permanence ✅
```

**Exemple concret :**
```sql
-- ✅ Backward compatible : Ajouter colonne nullable
ALTER TABLE users ADD COLUMN phone VARCHAR(20);

-- Application v1 : SELECT id, name, email FROM users;
--   → Fonctionne (ignore phone)

-- Application v2 : SELECT id, name, email, phone FROM users;
--   → Fonctionne (lit phone)
```

```sql
-- ❌ PAS backward compatible : Renommer colonne
ALTER TABLE users RENAME COLUMN name TO full_name;

-- Application v1 : SELECT name FROM users;
--   → ERREUR : column "name" does not exist
--   → Downtime ! ❌
```

### Principe 2 : Forward Compatibility (Compatibilité future)

**Définition :** Les anciennes versions du schéma doivent tolérer les nouvelles versions de l'application.

```plaintext
Application v2 déployée en premier
  ↓
Schéma v1 doit tolérer les requêtes de v2 (forward compatible)
  ↓
Schéma v2 déployé ensuite
  ↓
Tout fonctionne en permanence ✅
```

**Exemple :**
```plaintext
Schéma v1 : users(id, name, email)

Application v2 code :
  // Tolérant : utilise COALESCE pour gérer l'absence de phone
  SELECT id, name, email, COALESCE(phone, '') as phone FROM users;

  → Fonctionne même si phone n'existe pas encore
  → Forward compatible ✅
```

### Principe 3 : Expand-Migrate-Contract (Étendre-Migrer-Réduire)

C'est la **stratégie clé** du zero-downtime. Toute modification breaking est décomposée en 3 phases.

```plaintext
Phase 1 - EXPAND (Étendre) :
  Ajouter les nouveaux éléments sans supprimer les anciens
  → Compatibilité totale maintenue

Phase 2 - MIGRATE (Migrer) :
  Déployer progressivement le nouveau code
  Copier/transformer les données si nécessaire
  → Coexistence des deux versions

Phase 3 - CONTRACT (Réduire) :
  Supprimer les anciens éléments devenus inutiles
  → Nettoyage en toute sécurité
```

**Exemple complet :**

**Phase 1 - EXPAND :**
```sql
-- Déploiement 1 : Ajouter nouvelle colonne
ALTER TABLE users ADD COLUMN full_name VARCHAR(255);

-- Application v1 continue : SELECT name FROM users;
--   → Fonctionne ✅
-- Application v2 peut être déployée : SELECT full_name FROM users;
--   → Fonctionne ✅ (peut gérer NULL)
```

**Phase 2 - MIGRATE :**
```sql
-- Déploiement 2 : Copier les données
UPDATE users SET full_name = name WHERE full_name IS NULL;

-- Application v2 complètement déployée sur toutes les instances
-- Les deux colonnes coexistent et sont synchronisées
```

**Phase 3 - CONTRACT :**
```sql
-- Déploiement 3 (quelques jours/semaines plus tard)
-- Application v1 complètement retirée
-- Supprimer l'ancienne colonne
ALTER TABLE users DROP COLUMN name;

-- Maintenant seule full_name existe
```

**Avantages :**
- ✅ Zéro downtime à chaque phase  
- ✅ Rollback possible à chaque étape  
- ✅ Transition progressive et contrôlée

### Principe 4 : Blue-Green Deployment

Deux environnements identiques : **Blue** (actif) et **Green** (nouveau).

```plaintext
État initial :
  Blue (actif) : App v1 + Schéma v1
  Green (inactif) : vide

Étape 1 : Préparer Green
  Blue (actif) : App v1 + Schéma v1
  Green : App v2 + Schéma v2 (tests, validations)

Étape 2 : Basculer le trafic
  Router : 100% Blue → 100% Green
  Bascule instantanée !
  Blue (inactif) : App v1 + Schéma v1
  Green (actif) : App v2 + Schéma v2

Étape 3 : Rollback si problème
  Router : 100% Green → 100% Blue
  Retour instantané !
```

**Avec PostgreSQL :**
```plaintext
Option 1 : Deux bases distinctes
  Blue : PostgreSQL instance 1
  Green : PostgreSQL instance 2
  → Synchronisation complexe

Option 2 : Même base, deux schémas
  Blue : namespace "public"
  Green : namespace "public_v2"
  → Gestion des données partagées complexe

Option 3 : Même base, même schéma compatible
  Blue : App v1 (compatible schéma v1.5)
  Green : App v2 (compatible schéma v1.5)
  → Approche recommandée avec expand-migrate-contract
```

### Principe 5 : Rolling Deployment (Déploiement progressif)

Mettre à jour les instances une par une.

```plaintext
État initial :
  Instance A, B, C, D : App v1 + Schéma v1

Étape 1 : Migrer schéma (compatible avec v1 et v2)
  Schéma v1.5 déployé (backward et forward compatible)

Étape 2 : Rolling deployment de l'application
  Instance A : App v2 ✅
  Instance B : App v1
  Instance C : App v1
  Instance D : App v1

  (attendre validation, monitoring)

  Instance A : App v2 ✅
  Instance B : App v2 ✅
  Instance C : App v1
  Instance D : App v1

  (continuer progressivement)

  Instance A, B, C, D : App v2 ✅

Étape 3 : Nettoyer schéma (optionnel, plus tard)
  Schéma v2.0 (suppression des éléments v1)

Avantages :
  - Risque distribué (1 instance à la fois)
  - Rollback facile (revenir à v1 sur instances non migrées)
  - Détection précoce de problèmes
```

## Techniques spécifiques PostgreSQL

### Technique 1 : Ajouter des colonnes sans verrou

**Problème :**
```sql
-- ❌ Verrou exclusif sur toute la table
ALTER TABLE users ADD COLUMN phone VARCHAR(20) NOT NULL DEFAULT '';
-- Sur 100M lignes : plusieurs minutes de verrou
-- Toutes les requêtes bloquées = downtime !
```

**Solution : Ajout en plusieurs étapes**

**Étape 1 : Ajouter colonne nullable (rapide)**
```sql
-- Très rapide : ajoute juste metadata, pas de réécriture de table
ALTER TABLE users ADD COLUMN phone VARCHAR(20);
-- ✅ Pas de verrou significatif
```

**Étape 2 : Définir la valeur par défaut pour les nouvelles lignes**
```sql
-- Les futures insertions auront une valeur par défaut
ALTER TABLE users ALTER COLUMN phone SET DEFAULT '';
-- ✅ Instantané, pas de verrou
```

**Étape 3 : Remplir progressivement les lignes existantes (par batch)**
```sql
-- Traiter par petits lots pour éviter les verrous longs
UPDATE users  
SET phone = ''  
WHERE id BETWEEN 1 AND 100000  
  AND phone IS NULL;
-- Répéter pour tous les ranges

-- Ou utiliser un job background qui traite 1000 lignes par seconde
-- Sur 100M lignes : ~27 heures, mais sans bloquer le service
```

**Étape 4 : Rendre NOT NULL une fois tout rempli**
```sql
-- Une fois toutes les lignes remplies
ALTER TABLE users ALTER COLUMN phone SET NOT NULL;
-- ✅ Rapide car PostgreSQL vérifie juste qu'il n'y a pas de NULL
```

**Résultat :** Zéro downtime, migration progressive.

### Technique 2 : Créer des index sans bloquer

**Problème :**
```sql
-- ❌ Verrou exclusif pendant la création d'index
CREATE INDEX idx_users_email ON users(email);
-- Sur 100M lignes : 30+ minutes de verrou
-- Toutes les écritures bloquées = downtime !
```

**Solution : CREATE INDEX CONCURRENTLY**
```sql
-- ✅ Création d'index sans bloquer les écritures
CREATE INDEX CONCURRENTLY idx_users_email ON users(email);
-- Prend plus de temps, mais le service reste disponible
-- Aucune interruption pour les utilisateurs
```

**Fonctionnement :**
```plaintext
CREATE INDEX classique :
  1. Verrouiller la table (EXCLUSIVE LOCK)
  2. Scanner toute la table
  3. Construire l'index
  4. Déverrouiller
  → Écritures bloquées pendant toute la durée

CREATE INDEX CONCURRENTLY :
  1. Prendre un verrou léger (SHARE UPDATE EXCLUSIVE)
  2. Scanner la table (les écritures continuent)
  3. Construire l'index
  4. Scanner à nouveau les nouvelles données
  5. Finaliser l'index
  → Écritures jamais bloquées ✅
```

**Précautions :**
```sql
-- ⚠️ Vérifier que l'index a été créé correctement
SELECT * FROM pg_indexes WHERE indexname = 'idx_users_email';

-- Si échec : l'index reste en état "invalid"
-- Le supprimer et recommencer :
DROP INDEX CONCURRENTLY IF EXISTS idx_users_email;
```

### Technique 3 : Renommer sans downtime (Expand-Migrate-Contract)

**Objectif :** Renommer `users.name` en `users.full_name` sans interruption.

**❌ Approche naïve (causera du downtime) :**
```sql
ALTER TABLE users RENAME COLUMN name TO full_name;
-- Instant où ni "name" ni "full_name" n'existent pour l'ancienne app
-- → Downtime !
```

**✅ Approche zero-downtime :**

**Déploiement 1 : EXPAND**
```sql
-- Ajouter nouvelle colonne
ALTER TABLE users ADD COLUMN full_name VARCHAR(255);

-- Créer un trigger pour synchroniser
CREATE OR REPLACE FUNCTION sync_full_name()  
RETURNS TRIGGER AS $$  
BEGIN  
    -- Lors d'INSERT/UPDATE, synchroniser name ↔ full_name
    IF NEW.full_name IS NULL AND NEW.name IS NOT NULL THEN
        NEW.full_name := NEW.name;
    END IF;
    IF NEW.name IS NULL AND NEW.full_name IS NOT NULL THEN
        NEW.name := NEW.full_name;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER sync_users_name
    BEFORE INSERT OR UPDATE ON users
    FOR EACH ROW EXECUTE FUNCTION sync_full_name();
```

**Déploiement 2 : MIGRATE**
```sql
-- Copier les données existantes
UPDATE users SET full_name = name WHERE full_name IS NULL;

-- Déployer progressivement App v2 qui utilise full_name
-- App v1 continue d'utiliser name
-- Le trigger maintient la synchronisation ✅
```

**Déploiement 3 : CONTRACT (quelques semaines plus tard)**
```sql
-- App v1 complètement retirée
-- Supprimer le trigger
DROP TRIGGER IF EXISTS sync_users_name ON users;  
DROP FUNCTION IF EXISTS sync_full_name();  

-- Supprimer l'ancienne colonne
ALTER TABLE users DROP COLUMN name;

-- Maintenant seule full_name existe
```

**Avantages :**
- ✅ Zéro downtime à chaque étape  
- ✅ Rollback possible à tout moment  
- ✅ Coexistence des deux versions d'application

### Technique 4 : Modifier le type d'une colonne

**Objectif :** Changer `orders.amount` de `INTEGER` à `NUMERIC(10,2)`.

**❌ Approche naïve (causera du downtime) :**
```sql
-- ❌ Réécriture complète de la table = verrou exclusif long
ALTER TABLE orders ALTER COLUMN amount TYPE NUMERIC(10,2);
-- Sur 50M lignes : 15+ minutes de verrou = downtime !
```

**✅ Approche zero-downtime :**

**Déploiement 1 : EXPAND**
```sql
-- Ajouter nouvelle colonne
ALTER TABLE orders ADD COLUMN amount_decimal NUMERIC(10,2);

-- Créer trigger de synchronisation
CREATE OR REPLACE FUNCTION sync_amount()  
RETURNS TRIGGER AS $$  
BEGIN  
    NEW.amount_decimal := NEW.amount::NUMERIC(10,2);
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER sync_orders_amount
    BEFORE INSERT OR UPDATE ON orders
    FOR EACH ROW EXECUTE FUNCTION sync_amount();

-- Remplir progressivement par batch
UPDATE orders  
SET amount_decimal = amount::NUMERIC(10,2)  
WHERE id BETWEEN 1 AND 100000;  
-- Répéter pour tous les ranges
```

**Déploiement 2 : MIGRATE**
```sql
-- App v2 déployée progressivement (utilise amount_decimal)
-- App v1 continue (utilise amount)
-- Trigger maintient la cohérence
```

**Déploiement 3 : CONTRACT**
```sql
-- App v1 retirée
DROP TRIGGER sync_orders_amount ON orders;  
DROP FUNCTION sync_amount();  

-- Renommer amount_decimal → amount
ALTER TABLE orders RENAME COLUMN amount TO amount_old;  
ALTER TABLE orders RENAME COLUMN amount_decimal TO amount;  
ALTER TABLE orders DROP COLUMN amount_old;  
```

### Technique 5 : Supprimer une colonne

**Objectif :** Supprimer `users.legacy_field` qui n'est plus utilisée.

**✅ Approche zero-downtime :**

**Déploiement 1 : Arrêter d'écrire**
```sql
-- App v2 déployée : ne lit/écrit plus legacy_field
-- Mais la colonne existe toujours (tolérée)
-- Monitoring : vérifier qu'aucune requête n'utilise cette colonne
```

**Déploiement 2 : Marquer comme deprecated**
```sql
-- Documenter la deprecation
COMMENT ON COLUMN users.legacy_field IS
'DEPRECATED: Will be removed in v3.0. Do not use.';

-- Créer une alerte si quelqu'un l'utilise (optionnel)
-- Via pg_stat_statements ou trigger
```

**Déploiement 3 : CONTRACT (plusieurs semaines/mois plus tard)**
```sql
-- Aucune requête n'utilise legacy_field depuis longtemps
-- Suppression sûre
ALTER TABLE users DROP COLUMN legacy_field;
-- ✅ Instantané, pas de downtime
```

**Pourquoi attendre ?**
- Sécurité : S'assurer qu'aucun code legacy oublié n'utilise la colonne
- Rollback : Si problème avec v2, on peut revenir à v1 sans recréer la colonne

### Technique 6 : Ajouter une contrainte

**Objectif :** Rendre `users.email` `NOT NULL`.

**❌ Approche naïve (peut causer downtime) :**
```sql
-- ❌ Scanne toute la table pour vérifier + verrou
ALTER TABLE users ALTER COLUMN email SET NOT NULL;
-- Sur 100M lignes : plusieurs minutes = risque de downtime
```

**✅ Approche zero-downtime :**

**Déploiement 1 : Ajouter CHECK constraint NOT VALID**
```sql
-- Créer la contrainte sans valider les données existantes
ALTER TABLE users  
ADD CONSTRAINT users_email_not_null  
CHECK (email IS NOT NULL) NOT VALID;  
-- ✅ Instantané, pas de scan de table
-- Les NOUVELLES lignes doivent respecter la contrainte
-- Les anciennes lignes ne sont pas vérifiées
```

**Déploiement 2 : Nettoyer les données existantes**
```sql
-- Remplir progressivement par batch
UPDATE users  
SET email = 'placeholder@example.com'  
WHERE id BETWEEN 1 AND 100000  
  AND email IS NULL;
-- Répéter pour tous les ranges
-- Ou corriger manuellement les quelques cas problématiques
```

**Déploiement 3 : Valider la contrainte**
```sql
-- Valider que TOUTES les lignes respectent maintenant la contrainte
ALTER TABLE users VALIDATE CONSTRAINT users_email_not_null;
-- Scanne la table, mais pas de verrou exclusif
-- Les écritures continuent ✅

-- Ensuite, rendre NOT NULL natif
ALTER TABLE users ALTER COLUMN email SET NOT NULL;
-- Instantané car PostgreSQL sait déjà que pas de NULL

-- Supprimer la contrainte CHECK devenue redondante
ALTER TABLE users DROP CONSTRAINT users_email_not_null;
```

**Avantages :**
- ✅ Pas de verrou exclusif long  
- ✅ Validation progressive  
- ✅ Service disponible en permanence

## Patterns de déploiement avancés

### Pattern 1 : Dual-Write (Double écriture)

Pendant une migration, écrire simultanément dans l'ancienne et la nouvelle structure.

**Cas d'usage :** Migrer de `users.address` (texte) vers table `addresses` (normalisée).

**Phase 1 - EXPAND :**
```sql
-- Créer nouvelle table
CREATE TABLE addresses (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id),
    street VARCHAR(255),
    city VARCHAR(100),
    country VARCHAR(50)
);
```

**Phase 2 - MIGRATE (Dual-Write) :**
```plaintext
Application v2 :
  Lors d'un UPDATE d'adresse :
    1. Écrire dans users.address (ancien format)
    2. Parser l'adresse
    3. Écrire dans table addresses (nouveau format)

  Lors d'une lecture :
    1. Lire depuis addresses si existe
    2. Sinon fallback sur users.address

  → Les deux structures coexistent
  → Données synchronisées
```

**Phase 3 - CONTRACT :**
```sql
-- Migrer toutes les anciennes données
-- (script batch pour parser users.address et populer addresses)

-- App v3 : Ne lit/écrit que dans addresses
-- Supprimer users.address
ALTER TABLE users DROP COLUMN address;
```

**Avantages :**
- ✅ Migration progressive des données  
- ✅ Rollback facile (les deux versions existent)  
- ✅ Validation en production avant suppression

**Inconvénients :**
- ❌ Complexité du code (dual-write logic)  
- ❌ Risque de désynchronisation  
- ❌ Performance (double écriture)

### Pattern 2 : Shadow Tables (Tables miroir)

Créer une copie de la table avec la nouvelle structure, synchroniser, puis basculer.

**Cas d'usage :** Restructuration complète de la table `orders`.

**Phase 1 :**
```sql
-- Créer table shadow avec nouvelle structure
CREATE TABLE orders_new (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT REFERENCES users(id),
    total_amount NUMERIC(10,2),
    status VARCHAR(20),
    created_at TIMESTAMPTZ DEFAULT NOW(),
    -- Nouveaux champs
    currency VARCHAR(3) DEFAULT 'EUR',
    tax_amount NUMERIC(10,2) DEFAULT 0
);
```

**Phase 2 :**
```sql
-- Trigger pour synchroniser orders → orders_new
CREATE OR REPLACE FUNCTION sync_to_orders_new()  
RETURNS TRIGGER AS $$  
BEGIN  
    IF TG_OP = 'INSERT' THEN
        INSERT INTO orders_new (id, user_id, total_amount, status, created_at)
        VALUES (NEW.id, NEW.user_id, NEW.total_amount, NEW.status, NEW.created_at);
    ELSIF TG_OP = 'UPDATE' THEN
        UPDATE orders_new
        SET user_id = NEW.user_id,
            total_amount = NEW.total_amount,
            status = NEW.status
        WHERE id = NEW.id;
    ELSIF TG_OP = 'DELETE' THEN
        DELETE FROM orders_new WHERE id = OLD.id;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER sync_orders
    AFTER INSERT OR UPDATE OR DELETE ON orders
    FOR EACH ROW EXECUTE FUNCTION sync_to_orders_new();

-- Copier données historiques
INSERT INTO orders_new (id, user_id, total_amount, status, created_at)  
SELECT id, user_id, amount, status, created_at FROM orders;  
```

**Phase 3 :**
```sql
-- Basculer atomiquement
BEGIN;
    ALTER TABLE orders RENAME TO orders_old;
    ALTER TABLE orders_new RENAME TO orders;
COMMIT;
-- ✅ Bascule en quelques millisecondes

-- App continue de fonctionner avec la nouvelle structure
```

**Phase 4 :**
```sql
-- Nettoyer après validation
DROP TRIGGER IF EXISTS sync_orders ON orders_old;  
DROP TABLE orders_old;  
```

**Avantages :**
- ✅ Restructuration complète possible  
- ✅ Bascule quasi-instantanée  
- ✅ Rollback facile (renommer orders → orders_new, orders_old → orders)

**Inconvénients :**
- ❌ Double espace disque  
- ❌ Overhead de synchronisation (trigger)  
- ❌ Complexité

### Pattern 3 : Feature Toggles (Bascules de fonctionnalités)

Déployer le nouveau code désactivé, activer progressivement via configuration.

**Cas d'usage :** Migrer vers une nouvelle table `notifications`.

**Application v2 avec feature toggle :**
```plaintext
Configuration :
  USE_NEW_NOTIFICATIONS_TABLE = false (par défaut)

Code :
  if (USE_NEW_NOTIFICATIONS_TABLE) {
      // Utiliser nouvelle table notifications
      query = "SELECT * FROM notifications WHERE user_id = ?";
  } else {
      // Utiliser ancienne table user_alerts
      query = "SELECT * FROM user_alerts WHERE user_id = ?";
  }
```

**Déploiement progressif :**
```plaintext
Jour 1 : Déployer App v2 avec toggle=false
  → Toutes les instances utilisent user_alerts

Jour 2 : Activer pour 5% des utilisateurs
  USE_NEW_NOTIFICATIONS_TABLE = true (5% traffic)
  → Observer metrics, erreurs

Jour 3 : Activer pour 25% des utilisateurs
  → Continuer validation

Jour 5 : Activer pour 100%
  USE_NEW_NOTIFICATIONS_TABLE = true (100%)
  → Migration complète

Jour 10 : Supprimer le toggle et l'ancien code
  → Nettoyage du legacy
```

**Avantages :**
- ✅ Déploiement découplé de l'activation  
- ✅ Rollback instantané (toggle=false)  
- ✅ Test en production avec risque contrôlé (canary testing)  
- ✅ Validation progressive

**Inconvénients :**
- ❌ Complexité du code (if/else partout)  
- ❌ Dette technique (nettoyage nécessaire)  
- ❌ Gestion de configuration distribuée

### Pattern 4 : Read-Write Split (Séparation lecture/écriture)

Séparer les flux de lecture et d'écriture pour minimiser l'impact des migrations.

**Architecture :**
```plaintext
Application
    ↓
    ├─> Master PostgreSQL (écritures)
    │   ├─> Migration appliquée ici en premier
    │   └─> Peut avoir verrou court acceptable
    │
    └─> Replica PostgreSQL (lectures)
        ├─> Réplication avec léger délai
        └─> 95% du trafic = lectures sans verrou
```

**Workflow de migration :**
```plaintext
1. Appliquer migration sur Master
   → Verrou de 30 secondes sur Master
   → Lectures continuent sur Replica ✅
   → 95% du trafic non affecté

2. Réplication vers Replica
   → Quelques secondes de délai
   → Lectures basculent automatiquement

3. Migration complète
   → Zero-downtime perçu par 95% des utilisateurs
```

**Précaution :**
```plaintext
Les 5% d'écritures :
  - Nécessitent toujours des techniques zero-downtime
  - CREATE INDEX CONCURRENTLY
  - Expand-Migrate-Contract
  - Pas de verrous exclusifs longs
```

## Monitoring et validation

### Métriques clés pendant un déploiement

**1. Latence des requêtes**
```plaintext
Avant déploiement : P95 = 50ms  
Pendant déploiement : P95 = 52ms ✅ Acceptable  
Si P95 > 100ms → Problème ! Rollback  
```

**2. Taux d'erreur**
```plaintext
Avant : 0.01% d'erreurs  
Pendant : 0.01% d'erreurs ✅ Stable  
Si > 0.1% → Problème ! Rollback  
```

**3. Verrous PostgreSQL**
```sql
-- Surveiller les verrous en temps réel
SELECT
    relation::regclass as table_name,
    mode,
    granted,
    pid
FROM pg_locks  
WHERE granted = false;  

-- Si verrous non accordés > 5 secondes → Alerte
```

**4. Connexions actives**
```sql
-- Surveiller les connexions bloquées
SELECT
    pid,
    usename,
    application_name,
    state,
    wait_event,
    query,
    state_change
FROM pg_stat_activity  
WHERE state = 'active'  
  AND wait_event IS NOT NULL;
```

**5. Réplication lag (si replica)**
```sql
-- Surveiller le délai de réplication
SELECT
    client_addr,
    state,
    sync_state,
    replay_lag
FROM pg_stat_replication;

-- Si replay_lag > 10 secondes → Ralentir le déploiement
```

### Health checks applicatifs

**Health check endpoint :**
```plaintext
GET /health

Vérifications :
  1. Connexion PostgreSQL OK ?
  2. Schéma version compatible ?
  3. Tables essentielles accessibles ?
  4. Requête test réussie ?

Réponse :
  {
    "status": "healthy",
    "database": {
      "connected": true,
      "schema_version": "2.1.0",
      "compatible": true,
      "response_time_ms": 15
    }
  }

Load balancer :
  Si status != "healthy" → Retirer instance du pool
  → Trafic redirigé vers instances saines
```

### Smoke tests post-déploiement

**Tests automatiques après chaque étape :**
```plaintext
1. Test CRUD basique
   - INSERT un utilisateur de test
   - SELECT pour vérifier
   - UPDATE l'utilisateur
   - DELETE l'utilisateur
   → Si échec : ROLLBACK immédiat

2. Test requêtes critiques
   - Top 10 requêtes business
   - Vérifier résultats corrects
   → Si échec : ROLLBACK immédiat

3. Test intégrité référentielle
   - Contraintes FK respectées ?
   - Contraintes UNIQUE respectées ?
   → Si échec : ROLLBACK immédiat

4. Test performance
   - Requêtes dans les SLA ?
   - Pas de régression > 20% ?
   → Si échec : Alerter équipe
```

## Stratégie de rollback

### Plan de rollback préparé à l'avance

**Avant chaque déploiement, documenter :**

```plaintext
ROLLBACK PLAN - Déploiement v2.1.0
═══════════════════════════════════════

Changements :
  - Migration 050 : Ajout colonne users.phone
  - Migration 051 : Ajout table notifications
  - Application v2.1.0 déployée

Rollback Application :
  1. Réduire trafic vers v2.1.0 à 0%
  2. Augmenter trafic vers v2.0.0 à 100%
  3. Arrêter instances v2.1.0
  ⏱️ Durée : 2 minutes

Rollback Schéma (si nécessaire) :
  1. Exécuter migration DOWN :
     - DROP TABLE notifications;
     - ALTER TABLE users DROP COLUMN phone;
  2. Vérifier schéma version revenu à 2.0.0
  ⏱️ Durée : 30 secondes

Critères de rollback :
  - Taux d'erreur > 0.5%
  - Latence P95 > 200ms
  - Indisponibilité > 1 minute
  - Corruption de données détectée

Contact d'urgence :
  - Lead Tech : +33 6 XX XX XX XX
  - DBA on-call : +33 6 YY YY YY YY
  - PagerDuty : #incidents
```

### Types de rollback

**1. Rollback applicatif (simple)**
```plaintext
Redéployer l'ancienne version de l'application
  → Le schéma est backward compatible
  → Pas de modification de schéma nécessaire
  → Rapide (2-5 minutes)

Exemple :
  App v2.1 → App v2.0
  Schéma v2.1 reste (compatible avec v2.0)
```

**2. Rollback schéma (complexe)**
```plaintext
Annuler les migrations de schéma
  → Nécessite migrations DOWN préparées
  → Risque de perte de données
  → Plus lent (5-30 minutes)

Exemple :
  Schéma v2.1 → Schéma v2.0
  Via : alembic downgrade -1
```

**3. Rollback complet (extrême)**
```plaintext
Restaurer depuis backup
  → Dernier recours
  → Perte de données récentes
  → Très lent (heures)

Exemple :
  Restaurer backup de 1h avant le déploiement
  Perdre 1h de données
```

### Principes de rollback safe

**✅ À faire :**
- Tester le rollback en staging avant production
- Documenter la procédure de rollback
- Avoir des migrations DOWN pour chaque migration UP
- Monitorer en continu pendant et après déploiement
- Définir des seuils d'alerte clairs
- Automatiser le rollback quand possible

**❌ À éviter :**
- Rollback précipité sans analyse
- Rollback schéma si données déjà modifiées (risque corruption)
- Rollback multiple (v2.1 → v2.0 → v1.9) = chaos
- Rollback sans communication à l'équipe

## Checklist complète zero-downtime

### Phase 1 : Préparation (Avant déploiement)

- [ ] **Architecture compatible**

  - [ ] Utilisation de `NOT VALID` pour contraintes  
  - [ ] Migrations DOWN testées (rollback)

- [ ] **Tests complets**  
  - [ ] Tests automatisés passent sur nouvelle version

  - [ ] Plan de rollback écrit et validé

  - [ ] Contacts d'urgence à jour

### Phase 2 : Déploiement (Pendant)

- [ ] **Déploiement schéma**

- [ ] **Déploiement application**

### Scripts utiles

**Script de validation pre-deployment :**
```bash
#!/bin/bash
# pre_deploy_check.sh

echo "Vérification pre-déploiement..."

# Check 1 : Connexion PostgreSQL
psql -c "SELECT 1" > /dev/null 2>&1  
if [ $? -ne 0 ]; then  
    echo "❌ PostgreSQL inaccessible"
    exit 1
fi

# Check 2 : Version schéma compatible
CURRENT_SCHEMA=$(psql -t -c "SELECT version FROM schema_version ORDER BY applied_at DESC LIMIT 1")  
REQUIRED_SCHEMA="2.0.0"  
if [ "$CURRENT_SCHEMA" != "$REQUIRED_SCHEMA" ]; then  
    echo "⚠️  Schéma version mismatch : $CURRENT_SCHEMA vs $REQUIRED_SCHEMA"
    exit 1
fi

# Check 3 : Backup récent existe
LAST_BACKUP=$(ls -t /backups/*.sql 2>/dev/null | head -1)  
if [ -z "$LAST_BACKUP" ]; then  
    echo "❌ Pas de backup récent"
    exit 1
fi

# Check 4 : Migrations dry-run
flyway validate  
if [ $? -ne 0 ]; then  
    echo "❌ Validation migrations échouée"
    exit 1
fi

echo "✅ Tous les checks passés"  
exit 0  
```

**Script de monitoring during deployment :**
```bash
#!/bin/bash
# monitor_deployment.sh

THRESHOLD_ERROR_RATE=0.5  # 0.5%  
THRESHOLD_LATENCY=200     # 200ms  

while true; do
    # Récupérer métriques
    ERROR_RATE=$(curl -s http://metrics/error_rate)
    LATENCY_P95=$(curl -s http://metrics/latency_p95)

    # Vérifier seuils
    if (( $(echo "$ERROR_RATE > $THRESHOLD_ERROR_RATE" | bc -l) )); then
        echo "🚨 ALERTE : Taux d'erreur trop élevé : $ERROR_RATE%"
        # Trigger rollback automatique
        ./rollback.sh
        exit 1
    fi

    if (( $(echo "$LATENCY_P95 > $THRESHOLD_LATENCY" | bc -l) )); then
        echo "⚠️  ATTENTION : Latence élevée : ${LATENCY_P95}ms"
    fi

    echo "✅ Métriques OK : Errors=$ERROR_RATE%, Latency=$LATENCY_P95ms"
    sleep 10
done
```

## Cas d'étude : Déploiement complet zero-downtime

### Scénario réel

**Objectif :** Renommer `users.name` en `users.full_name` et restructurer `orders`.

**Contexte :**
- Application web avec 100 000 utilisateurs actifs
- 5 millions de lignes dans `users`
- 50 millions de lignes dans `orders`
- Trafic : 1000 req/sec en moyenne, pics à 5000 req/sec
- SLA : 99.99% uptime (4 minutes de downtime/mois max)

### Timeline du déploiement

**Semaine 1 : Préparation**
```plaintext
Lundi :
  - Création des migrations (Expand-Migrate-Contract)
  - Tests automatisés sur base de dev
  - Code review des migrations

Mercredi :
  - Déploiement sur environnement de staging
  - Tests de charge (simule 2× le trafic production)
  - Test du rollback complet

Vendredi :
  - Documentation finalisée (runbook, rollback plan)
  - Validation par l'équipe et le management
  - Communication aux utilisateurs (aucun downtime prévu)
```

**Semaine 2 : Déploiement**

**Lundi 10h00 : Phase EXPAND (Schéma)**
```sql
-- Migration 050 : Ajouter full_name
ALTER TABLE users ADD COLUMN full_name VARCHAR(255);

-- Trigger de synchronisation
CREATE TRIGGER sync_users_name
    BEFORE INSERT OR UPDATE ON users
    FOR EACH ROW EXECUTE FUNCTION sync_full_name();
```
- Durée : 2 secondes (ajout de métadonnées)
- Impact : Aucun (backward compatible)
- Monitoring : ✅ Aucune dégradation

**Lundi 11h00 : Phase EXPAND (Application)**
```plaintext
Deploy App v2.1 avec feature toggle :
  USE_FULL_NAME = false (par défaut)

Rolling deployment :
  - Instance 1/10 : v2.1 ✅ (10h01)
  - Monitoring 15 minutes : OK
  - Instances 2-10 : v2.1 ✅ (10h15-11h00)

État :
  - App v2.1 déployée partout
  - Utilise toujours "name" (toggle=false)
  - Colonne full_name existe mais vide
```

**Lundi 14h00 : Phase MIGRATE (Données)**
```sql
-- Job background : Copier name → full_name
-- Traitement par batch de 100k lignes
UPDATE users  
SET full_name = name  
WHERE id BETWEEN 1 AND 100000  
  AND full_name IS NULL;
-- Répété automatiquement pour tous les ranges

-- Durée totale : 2 heures (batch processing)
-- Impact : Aucun (pas de verrou long, requêtes continuent)
```

**Lundi 16h30 : Phase MIGRATE (Activation progressive)**
```plaintext
Activer feature toggle progressivement :

16h30 : USE_FULL_NAME = true pour 1% utilisateurs
  - Monitoring 30 minutes
  - Métriques : ✅ Stables

17h00 : 5% utilisateurs

  - Métriques : ✅ Stables

17h30 : 25% utilisateurs
  - Monitoring 1 heure
  - Métriques : ✅ Stables

18h30 : 100% utilisateurs
  - USE_FULL_NAME = true pour tous
  - Colonne "name" toujours synchronisée (trigger)
  - Monitoring intensif toute la soirée
  - Métriques : ✅ Stables
```

**Mardi-Jeudi : Observation**

```plaintext
  - Consommation ressources : normale

  - Aucun incident
  - Uptime : 100% maintenu
```

**Vendredi : Phase CONTRACT (Nettoyage)**
```plaintext

  - Logs confirmés : 100% des requêtes sur full_name

10h00 : Deploy App v2.2 (toggle retiré, code nettoyé)
  - Rolling deployment standard
  - Durée : 30 minutes
```

## Conclusion

Le **zero-downtime deployment** n'est plus un luxe, c'est une nécessité pour les applications modernes. Avec PostgreSQL, c'est parfaitement réalisable en combinant :

### Les 5 piliers du zero-downtime

1. **Backward compatibility** : Toujours déployer le schéma avant l'application  
2. **Expand-Migrate-Contract** : Décomposer les breaking changes en 3 phases  
3. **Verrous minimisés** : CREATE INDEX CONCURRENTLY, NOT VALID, batch processing  
4. **Monitoring continu** : Détecter les problèmes en temps réel  
5. **Rollback plan** : Toujours avoir un plan B testé

### Principes à retenir

✅ **Toujours faire :**
- Planifier méticuleusement
- Tester en staging (y compris le rollback)
- Déployer progressivement (canary → rolling)
- Monitorer intensivement
- Documenter tout

❌ **Ne jamais faire :**
- ALTER TABLE avec verrou exclusif long
- Breaking change brutal sans transition
- Déployer sans backup
- Déployer sans plan de rollback
- Précipiter un rollback sans analyse

### Investissement vs bénéfice

```plaintext
Investissement initial :
  - Setup infrastructure (load balancer, monitoring) : 1-2 semaines
  - Formation de l'équipe : 2-3 jours
  - Procédures et documentation : 1 semaine

Bénéfices à long terme :
  - Déploiements plus fréquents (10×)
  - Confiance de l'équipe (stress réduit)
  - Satisfaction utilisateurs (pas d'interruption)
  - Respect des SLA (99.99%+)
  - ROI : > 200% en général
```

### Recommandations finales

**Pour les petits projets :**
- Start simple : Expand-Migrate-Contract basique
- CREATE INDEX CONCURRENTLY systématiquement
- Health checks et monitoring minimal

**Pour les projets moyens :**
- Rolling deployments automatisés
- Feature toggles pour activations progressives
- Monitoring avancé (métriques, logs, alertes)
- Replica PostgreSQL (read-write split)

**Pour les grands projets :**
- Blue-green ou canary deployments
- Shadow tables pour restructurations complexes
- Automation complète (CI/CD, rollback automatique)
- Chaos engineering (tester les pannes)

Le zero-downtime n'est pas parfait à 100% (incidents imprévus possibles), mais il réduit drastiquement les risques et améliore la résilience de votre système. C'est un investissement qui se rembourse rapidement en fiabilité et sérénité d'esprit.
