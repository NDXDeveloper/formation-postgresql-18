# 20.4.1. Repository Pattern

## Introduction

Le **Repository Pattern** (ou "Motif de Dépôt") est un patron de conception (design pattern) qui s'interpose entre votre logique métier (business logic) et votre couche d'accès aux données. Son objectif principal est de créer une **abstraction** entre votre application et la base de données PostgreSQL.

Imaginez que votre base de données PostgreSQL soit une bibliothèque. Le Repository Pattern, c'est comme avoir un bibliothécaire : au lieu d'aller vous-même chercher les livres dans les rayons (écrire du SQL directement), vous demandez au bibliothécaire qui connaît parfaitement l'organisation de la bibliothèque.

## Pourquoi utiliser le Repository Pattern ?

### 1. Séparation des préoccupations (Separation of Concerns)

Sans Repository Pattern, votre code métier contient directement des requêtes SQL :

```plaintext
❌ Code métier = SQL + Logique + Règles métier (tout mélangé)
```

Avec Repository Pattern :

```plaintext
✅ Code métier = Logique + Règles métier
✅ Repository = SQL et accès aux données
```

### 2. Facilite les tests

Vous pouvez facilement créer un "faux" repository (mock) pour tester votre logique métier sans avoir besoin d'une vraie base de données PostgreSQL. Cela rend vos tests plus rapides et plus fiables.

### 3. Réutilisabilité du code

Les opérations courantes (rechercher un utilisateur, lister des produits, etc.) sont écrites une seule fois dans le repository et réutilisées partout dans votre application.

### 4. Facilite les changements

Si vous devez modifier la structure de votre base de données ou même changer de système de base de données, vous ne modifiez que le repository, pas tout votre code métier.

## Architecture du Repository Pattern

### Vue d'ensemble

```
┌─────────────────────────────────────────┐
│   Application / Controller              │
│   (Logique métier)                      │
└──────────────────┬──────────────────────┘
                   │
                   │ Utilise
                   ▼
┌─────────────────────────────────────────┐
│   Repository (Interface/Abstraction)    │
│   - findById(id)                        │
│   - findAll()                           │
│   - save(entity)                        │
│   - delete(id)                          │
└──────────────────┬──────────────────────┘
                   │
                   │ Implémente
                   ▼
┌─────────────────────────────────────────┐
│   Repository PostgreSQL (Implémentation)│
│   - SQL queries                         │
│   - Connexion à PostgreSQL              │
└──────────────────┬──────────────────────┘
                   │
                   │ Communique avec
                   ▼
┌─────────────────────────────────────────┐
│   Base de données PostgreSQL            │
└─────────────────────────────────────────┘
```

### Les composants clés

#### 1. L'entité (Entity)

C'est la représentation d'un objet métier dans votre code. Par exemple, un `User` (utilisateur) :

```plaintext
User
  - id: integer
  - email: string
  - name: string
  - created_at: timestamp
```

Cette entité correspond généralement à une table PostgreSQL :

```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    name VARCHAR(255) NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);
```

#### 2. L'interface du Repository

L'interface définit le "contrat" : quelles opérations sont disponibles pour manipuler les entités.

```plaintext
UserRepository (Interface)
  - findById(userId: int): User
  - findByEmail(email: string): User
  - findAll(): List<User>
  - save(user: User): User
  - update(user: User): User
  - delete(userId: int): boolean
```

#### 3. L'implémentation du Repository

C'est ici que se trouve le code qui communique réellement avec PostgreSQL. Chaque méthode de l'interface est implémentée avec du SQL.

Exemple conceptuel de `findById` :

```plaintext
Méthode: findById(userId: 123)
  1. Préparer la requête SQL:
     SELECT id, email, name, created_at
     FROM users
     WHERE id = $1

  2. Exécuter la requête avec le paramètre userId

  3. Récupérer le résultat

  4. Transformer les données PostgreSQL en objet User

  5. Retourner l'objet User
```

## Opérations CRUD classiques

Le Repository Pattern implémente généralement les opérations **CRUD** (Create, Read, Update, Delete) :

### 1. Create (Créer)

**Méthode :** `save(entity)` ou `create(entity)`

**Objectif :** Insérer une nouvelle ligne dans la table PostgreSQL

**SQL sous-jacent :**
```sql
INSERT INTO users (email, name)  
VALUES ($1, $2)  
RETURNING id, email, name, created_at;  
```

**Flux :**
- Application crée un objet User
- Appelle `repository.save(user)`
- Repository exécute l'INSERT
- PostgreSQL retourne l'enregistrement créé (avec son ID généré)
- Repository transforme le résultat en objet User
- Application reçoit l'objet User complet

### 2. Read (Lire)

**Méthodes :** `findById()`, `findAll()`, `findByEmail()`, etc.

**Objectif :** Récupérer des données depuis PostgreSQL

**SQL sous-jacent (exemple findById) :**
```sql
SELECT id, email, name, created_at  
FROM users  
WHERE id = $1;  
```

**Flux :**
- Application demande un utilisateur par son ID
- Appelle `repository.findById(123)`
- Repository exécute le SELECT
- PostgreSQL retourne les colonnes
- Repository transforme les données en objet User
- Application reçoit l'objet User

### 3. Update (Mettre à jour)

**Méthode :** `update(entity)` ou `save(entity)` (si l'ID existe)

**Objectif :** Modifier une ligne existante dans PostgreSQL

**SQL sous-jacent :**
```sql
UPDATE users  
SET email = $1, name = $2  
WHERE id = $3  
RETURNING id, email, name, created_at;  
```

**Flux :**
- Application modifie un objet User existant
- Appelle `repository.update(user)`
- Repository exécute l'UPDATE
- PostgreSQL retourne l'enregistrement mis à jour
- Repository transforme le résultat en objet User
- Application reçoit l'objet User à jour

### 4. Delete (Supprimer)

**Méthode :** `delete(id)` ou `deleteById(id)`

**Objectif :** Supprimer une ligne de PostgreSQL

**SQL sous-jacent :**
```sql
DELETE FROM users WHERE id = $1;
```

**Flux :**
- Application décide de supprimer un utilisateur
- Appelle `repository.delete(123)`
- Repository exécute le DELETE
- PostgreSQL supprime la ligne
- Repository retourne un booléen (true si supprimé, false sinon)
- Application reçoit la confirmation

## Méthodes de recherche avancées

Au-delà du CRUD basique, un Repository peut proposer des méthodes de recherche personnalisées :

### Exemples de méthodes personnalisées

#### 1. Recherche par critère unique
```plaintext
findByEmail(email: string): User
→ SELECT * FROM users WHERE email = $1
```

#### 2. Recherche par critères multiples
```plaintext
findByNameAndEmail(name: string, email: string): User
→ SELECT * FROM users WHERE name = $1 AND email = $2
```

#### 3. Recherche avec pattern matching
```plaintext
findByNameContaining(pattern: string): List<User>
→ SELECT * FROM users WHERE name ILIKE '%' || $1 || '%'
```

#### 4. Recherche avec limite et tri
```plaintext
findTopNByOrderByCreatedAtDesc(limit: int): List<User>
→ SELECT * FROM users ORDER BY created_at DESC LIMIT $1
```

#### 5. Comptage
```plaintext
countByEmailDomain(domain: string): int
→ SELECT COUNT(*) FROM users WHERE email LIKE '%@' || $1
```

## Avantages du Repository Pattern

### ✅ Testabilité améliorée

Sans Repository Pattern, tester votre logique métier nécessite une base de données PostgreSQL :

```plaintext
Test sans Repository:
  1. Démarrer PostgreSQL de test
  2. Créer les tables
  3. Insérer des données de test
  4. Exécuter le test
  5. Nettoyer la base
  ⏱️ Lent et complexe
```

Avec Repository Pattern, vous utilisez un mock :

```plaintext
Test avec Repository:
  1. Créer un faux repository (mock)
  2. Définir les comportements attendus
  3. Exécuter le test
  ⚡ Rapide et simple
```

### ✅ Maintenance simplifiée

Si votre schéma PostgreSQL change (renommer une colonne, ajouter un index, etc.), vous ne modifiez que le repository, pas 50 fichiers de code métier.

**Exemple :** Vous renommez `email` en `email_address` :
- Sans Repository : Vous devez modifier toutes les requêtes SQL dans toute l'application
- Avec Repository : Vous modifiez uniquement l'implémentation du repository

### ✅ Abstraction de la complexité SQL

Votre code métier ne voit pas le SQL complexe. Compare :

**Sans Repository Pattern :**
```plaintext
Code métier contient:
  SELECT u.id, u.name, COUNT(o.id) as order_count
  FROM users u
  LEFT JOIN orders o ON u.id = o.user_id
  WHERE u.created_at > NOW() - INTERVAL '30 days'
  GROUP BY u.id, u.name
  HAVING COUNT(o.id) > 5
  ORDER BY order_count DESC;
```

**Avec Repository Pattern :**
```plaintext
Code métier contient:
  userRepository.findActiveUsersWithManyOrders()

→ Le SQL complexe est caché dans le repository
```

### ✅ Réutilisabilité

Une fois que vous avez écrit `userRepository.findById()`, vous l'utilisez partout dans votre application :
- Dans le controller web
- Dans les jobs background
- Dans les scripts d'administration
- Dans les tests

Pas de duplication de code SQL.

### ✅ Changement de base de données facilité

Si un jour vous devez migrer de PostgreSQL vers un autre système (rare, mais possible), vous ne changez que les repositories, pas votre logique métier.

```plaintext
Application → UserRepository (Interface) → PostgreSQLUserRepository
                                        → MongoDBUserRepository (nouvelle implémentation)
                                        → MySQLUserRepository (nouvelle implémentation)
```

## Inconvénients et limites

### ❌ Abstraction excessive

Pour des requêtes très simples, le Repository Pattern peut sembler une sur-ingénierie :

```plaintext
Au lieu de:
  SELECT * FROM users WHERE id = 123

Vous écrivez:
  - Une interface UserRepository
  - Une implémentation PostgreSQLUserRepository
  - Une méthode findById()
  - Des tests pour cette méthode
```

Pour des projets très simples ou des scripts, c'est parfois trop.

### ❌ Performance potentielle

Un Repository générique peut charger plus de données que nécessaire. Par exemple :

```plaintext
userRepository.findById(123)
→ Charge TOUTES les colonnes de l'utilisateur
```

Alors que parfois vous n'avez besoin que de l'email :

```plaintext
SELECT email FROM users WHERE id = 123  (plus rapide)
```

Solution : Créer des méthodes spécifiques comme `findEmailById(id)`.

### ❌ Requêtes complexes difficiles à abstraire

Certaines requêtes analytiques très complexes avec des jointures multiples, des CTE, des window functions, etc., peuvent être difficiles à encapsuler dans un Repository de manière élégante.

Dans ces cas, il peut être préférable de garder le SQL dans un fichier dédié ou d'utiliser des objets Query Builder.

### ❌ Courbe d'apprentissage

Pour les développeurs débutants, comprendre l'abstraction du Repository Pattern demande un effort initial. Il faut comprendre :
- Les interfaces
- L'injection de dépendances
- Les mocks pour les tests
- La séparation des couches

## Bonnes pratiques avec PostgreSQL

### 1. Utiliser les requêtes préparées (Prepared Statements)

Toutes les requêtes dans votre repository doivent utiliser des paramètres ($1, $2, etc.) pour éviter les injections SQL :

```plaintext
✅ Bon:
  SELECT * FROM users WHERE email = $1
  (paramètre: email)

❌ Mauvais:
  SELECT * FROM users WHERE email = '" + email + "'"
  (vulnérable aux injections SQL)
```

### 2. Gérer correctement les transactions

Le repository doit permettre d'exécuter plusieurs opérations dans une transaction PostgreSQL :

```plaintext
Exemple: Transférer de l'argent entre deux comptes

BEGIN;
  UPDATE accounts SET balance = balance - 100 WHERE id = 1;
  UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

→ Le repository doit supporter ce pattern
```

### 3. Utiliser RETURNING pour récupérer les données

Lors d'un INSERT ou UPDATE, PostgreSQL peut retourner les données créées/modifiées :

```sql
INSERT INTO users (email, name)  
VALUES ($1, $2)  
RETURNING id, created_at;  
```

Cela évite un SELECT supplémentaire pour récupérer l'ID généré.

### 4. Gérer les erreurs PostgreSQL

Le repository doit traduire les erreurs PostgreSQL en exceptions applicatives compréhensibles :

```plaintext
PostgreSQL retourne:
  ERROR: duplicate key value violates unique constraint "users_email_key"

Repository traduit en:
  UserAlreadyExistsException("Un utilisateur avec cet email existe déjà")
```

### 5. Pagination efficace

Pour les listes longues, utiliser LIMIT et OFFSET (ou mieux, la pagination par curseur) :

```plaintext
findAll(page: int, pageSize: int): List<User>
→ SELECT * FROM users
  ORDER BY id
  LIMIT $1 OFFSET $2
```

**Note :** Pour de grandes tables, préférer la pagination par curseur (WHERE id > $1 LIMIT $2) pour de meilleures performances.

### 6. Projection sélective

Offrir des méthodes qui ne chargent que les colonnes nécessaires :

```plaintext
findById(id): User
  → Charge toutes les colonnes

findUserSummaryById(id): UserSummary
  → Charge uniquement id, name, email (plus rapide)
```

### 7. Gestion du cache

Le repository peut intégrer une couche de cache (Redis, Memcached) pour éviter des requêtes répétitives à PostgreSQL :

```plaintext
findById(id):
  1. Vérifier le cache
  2. Si trouvé → retourner depuis le cache
  3. Sinon → requête PostgreSQL + stocker en cache
```

### 8. Connexion à la base

Le repository ne doit **PAS** gérer directement les connexions. Utilisez un pool de connexions partagé :

```plaintext
❌ Mauvais:
  Chaque méthode du repository ouvre/ferme une connexion

✅ Bon:
  Le repository reçoit une connexion depuis un pool
  Connection pooling (PgBouncer, HikariCP, etc.)
```

## Variantes du Repository Pattern

### Repository générique

Un repository de base avec des opérations CRUD communes pour toutes les entités :

```plaintext
GenericRepository<T>
  - findById(id): T
  - findAll(): List<T>
  - save(entity: T): T
  - delete(id): void

UserRepository extends GenericRepository<User>  
ProductRepository extends GenericRepository<Product>  
```

### Repository spécifique

Chaque entité a son repository avec ses méthodes spécifiques :

```plaintext
UserRepository
  - findById(id): User
  - findByEmail(email): User
  - findActiveUsers(): List<User>

OrderRepository
  - findById(id): Order
  - findByUserId(userId): List<Order>
  - findPendingOrders(): List<Order>
```

### Unit of Work Pattern (combiné)

Coordonne plusieurs repositories dans une transaction unique :

```plaintext
UnitOfWork
  - userRepository
  - orderRepository
  - productRepository

  commit(): Valide TOUTES les modifications en une transaction
  rollback(): Annule TOUTES les modifications
```

## Cas d'usage concrets

### Cas 1 : Application e-commerce

```plaintext
Repositories nécessaires:
  - UserRepository
  - ProductRepository
  - OrderRepository
  - PaymentRepository
  - InventoryRepository

Exemple de flux:
  1. UserRepository.findById(123) → Récupérer l'utilisateur
  2. ProductRepository.findByIds([1,2,3]) → Récupérer les produits du panier
  3. InventoryRepository.checkAvailability(products) → Vérifier le stock
  4. OrderRepository.create(order) → Créer la commande
  5. PaymentRepository.processPayment(order) → Traiter le paiement
  6. InventoryRepository.decrementStock(products) → Déduire le stock

→ Tout cela peut être coordonné dans une transaction PostgreSQL
```

### Cas 2 : API REST

```plaintext
GET /api/users/{id}
  → userRepository.findById(id)
  → Retourner JSON

POST /api/users
  → userRepository.save(newUser)
  → Retourner JSON (201 Created)

PUT /api/users/{id}
  → userRepository.update(user)
  → Retourner JSON

DELETE /api/users/{id}
  → userRepository.delete(id)
  → Retourner 204 No Content
```

Le controller REST appelle uniquement le repository, sans écrire de SQL.

### Cas 3 : Job de traitement batch

```plaintext
Job: Envoyer des emails aux utilisateurs inactifs

1. userRepository.findInactiveUsers(30days)
   → Récupérer les utilisateurs inactifs depuis 30 jours

2. Pour chaque utilisateur:
   - emailService.sendReminder(user.email)
   - userRepository.updateLastReminderDate(user.id)

→ Le job n'a pas de SQL, seulement des appels au repository
```

## Relation avec les ORM (Object-Relational Mapping)

Les **ORM** (comme Hibernate en Java, Entity Framework en .NET, SQLAlchemy en Python, TypeORM en Node.js) implémentent souvent le Repository Pattern de manière automatique ou facilitée.

### ORM vs Repository "manuel"

**ORM (Repository automatique) :**
- L'ORM génère automatiquement les méthodes CRUD
- Mapping automatique entre objets et tables PostgreSQL
- Query builder intégré
- Cache de premier niveau inclus
- **Avantage :** Rapide à mettre en place  
- **Inconvénient :** Moins de contrôle sur le SQL généré

**Repository manuel :**
- Vous écrivez vous-même le SQL
- Contrôle total sur les requêtes PostgreSQL
- Optimisations fines possibles
- **Avantage :** Performance optimale, pas de "magie"  
- **Inconvénient :** Plus de code à écrire

### Compromis : Repository avec Query Builder

Utiliser un Query Builder (comme Knex.js, jOOQ) qui offre une API fluide pour construire du SQL sans l'écrire en chaînes de caractères :

```plaintext
Au lieu de:
  "SELECT * FROM users WHERE email = $1 AND active = true"

Query Builder:
  query.select('*')
       .from('users')
       .where('email', email)
       .andWhere('active', true)

→ Type-safe et plus lisible
```

## Résumé des points clés

### ✨ Le Repository Pattern c'est :
- Une **abstraction** entre votre code métier et PostgreSQL
- Une **interface** définissant les opérations disponibles
- Une **implémentation** contenant le SQL réel
- Un moyen de **séparer les responsabilités**

### 🎯 Quand l'utiliser :
- Applications web avec logique métier complexe
- Projets nécessitant des tests unitaires
- Architecture modulaire / microservices
- Équipes avec développeurs de niveaux variés

### 🚫 Quand l'éviter :
- Scripts simples one-shot
- Prototypes rapides
- Requêtes analytiques très complexes (mieux vaut du SQL pur)
- Projets avec une seule table

### 📚 Concepts liés à approfondir :
- **Injection de dépendances** : Comment passer le repository aux classes qui en ont besoin  
- **Unit of Work Pattern** : Coordonner plusieurs repositories dans une transaction  
- **Specification Pattern** : Construire des critères de recherche dynamiques  
- **CQRS** : Séparer les repositories de lecture et d'écriture  
- **Data Transfer Objects (DTO)** : Objets pour transporter les données entre couches

## Conclusion

Le Repository Pattern est un outil puissant pour structurer vos applications utilisant PostgreSQL. Il ne s'agit pas d'une règle absolue, mais d'un **choix architectural** qui apporte clarté, testabilité et maintenabilité au prix d'une légère complexité initiale.

Pour les débutants, commencez simple : créez un repository pour une seule entité (par exemple `UserRepository`), implémentez les méthodes CRUD de base, et observez comment cela simplifie votre code métier. Vous pourrez ensuite étendre progressivement le concept à d'autres entités et ajouter des méthodes personnalisées selon vos besoins.
