# 21.2.1 — Event Store Pattern

## Introduction

Dans les architectures traditionnelles, nous stockons l'**état actuel** des données. Une table `accounts` contient le solde actuel de chaque compte bancaire. Quand un client effectue un retrait, nous mettons à jour ce solde. L'historique ? Perdu, ou stocké séparément dans des logs difficiles à exploiter.

L'**Event Sourcing** propose une approche radicalement différente : au lieu de stocker l'état actuel, nous stockons **tous les événements** qui ont conduit à cet état. Le solde d'un compte n'est plus une donnée stockée, mais une donnée **calculée** à partir de la séquence de dépôts et retraits.

L'**Event Store** est la base de données spécialisée qui stocke ces événements. Dans ce chapitre, nous allons découvrir comment implémenter un Event Store robuste avec PostgreSQL.

---

## Du CRUD à l'Event Sourcing

### L'Approche Traditionnelle : CRUD

Dans une application classique, nous manipulons les données avec les opérations **CRUD** (Create, Read, Update, Delete) :

```sql
-- Créer un compte
INSERT INTO accounts (id, owner, balance)  
VALUES ('acc-001', 'Alice', 1000.00);  

-- Effectuer un retrait : MISE À JOUR destructive
UPDATE accounts SET balance = balance - 200 WHERE id = 'acc-001';

-- État actuel : balance = 800
-- Question : Pourquoi 800 ? Quand le retrait a-t-il eu lieu ?
-- Réponse : On ne sait pas, l'information est perdue.
```

```
État dans le temps (CRUD) :

    t0              t1              t2              t3
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ balance │    │ balance │    │ balance │    │ balance │
│  1000   │───►│   800   │───►│  1300   │───►│  1100   │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
                   │              │              │
              Retrait ?      Dépôt ?        Retrait ?
              Quand ?        Combien ?      Pourquoi ?
              Par qui ?      D'où ?         Autorisé ?

         ▲
         │
    L'historique est PERDU à chaque UPDATE
```

### L'Approche Event Sourcing

Avec l'Event Sourcing, nous ne stockons jamais l'état directement. Nous stockons les **faits** (événements) qui se sont produits :

```sql
-- Au lieu de UPDATE, nous INSÉRONS des événements
INSERT INTO events (stream_id, event_type, data, occurred_at) VALUES
('acc-001', 'AccountOpened',   '{"owner": "Alice", "initial_balance": 1000}', '2025-01-01 10:00:00'),
('acc-001', 'MoneyWithdrawn',  '{"amount": 200, "reason": "ATM"}',            '2025-01-01 14:30:00'),
('acc-001', 'MoneyDeposited',  '{"amount": 500, "source": "salary"}',         '2025-01-15 09:00:00'),
('acc-001', 'MoneyWithdrawn',  '{"amount": 200, "reason": "transfer"}',       '2025-01-20 16:45:00');

-- L'état actuel (balance = 1100) se CALCULE à partir des événements
```

```
État dans le temps (Event Sourcing) :

    Event 1           Event 2           Event 3           Event 4
┌─────────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│AccountOpened│   │MoneyWithdrawn│   │MoneyDeposited│   │MoneyWithdrawn│
│ +1000       │──►│ -200         │──►│ +500         │──►│ -200         │
│ 01/01 10:00 │   │ 01/01 14:30  │   │ 15/01 09:00  │   │ 20/01 16:45  │
│ initial     │   │ ATM          │   │ salary       │   │ transfer     │
└─────────────┘   └──────────────┘   └──────────────┘   └──────────────┘
      │                 │                 │                 │
      ▼                 ▼                 ▼                 ▼
   balance           balance           balance           balance
    =1000             =800             =1300             =1100

         ▲
         │
    L'historique COMPLET est préservé, RIEN n'est perdu
```

## Qu'est-ce qu'un Event Store ?

### Définition

Un **Event Store** est un système de stockage optimisé pour l'Event Sourcing. Il s'agit d'une base de données où :

- Les données sont **immutables** (append-only) : on ajoute, on ne modifie jamais
- Les événements sont ordonnés chronologiquement
- Chaque événement appartient à un **stream** (flux) identifiant une entité
- L'état d'une entité se reconstruit en "rejouant" ses événements

### Analogie : Le Grand Livre Comptable

L'Event Store fonctionne comme un **grand livre comptable** (ledger) :

- Chaque transaction est inscrite définitivement
- On ne "gomme" jamais une écriture
- Pour corriger une erreur, on ajoute une écriture inverse
- Le solde se calcule en additionnant toutes les écritures

```
┌──────────────────────────────────────────────────────────────────┐
│                    GRAND LIVRE - Compte ACC-001                  │
├──────────┬──────────────────┬───────────┬───────────┬────────────┤
│   Date   │    Libellé       │   Débit   │   Crédit  │   Solde    │
├──────────┼──────────────────┼───────────┼───────────┼────────────┤
│ 01/01/25 │ Ouverture        │           │  1000.00  │   1000.00  │
│ 01/01/25 │ Retrait DAB      │   200.00  │           │    800.00  │
│ 15/01/25 │ Virement salaire │           │   500.00  │   1300.00  │
│ 20/01/25 │ Virement émis    │   200.00  │           │   1100.00  │
├──────────┴──────────────────┴───────────┴───────────┴────────────┤
│ ⚠️  Aucune ligne ne peut être modifiée ou supprimée              │
│ ✅  Pour corriger : ajouter une écriture de régularisation       │
└──────────────────────────────────────────────────────────────────┘
```

### Concepts Clés

| Concept | Description |
|---------|-------------|
| **Event** | Fait immutable qui s'est produit (passé) |
| **Stream** | Séquence ordonnée d'événements pour une entité |
| **Stream ID** | Identifiant unique du stream (ex: `account-123`) |
| **Event Type** | Nature de l'événement (ex: `MoneyDeposited`) |
| **Payload** | Données associées à l'événement (JSON) |
| **Version** | Numéro de séquence dans le stream |
| **Timestamp** | Moment où l'événement s'est produit |
| **Metadata** | Informations contextuelles (user, correlation ID) |

## Implémentation d'un Event Store avec PostgreSQL

PostgreSQL est parfaitement adapté pour construire un Event Store grâce à :

- Son support JSONB pour les payloads flexibles
- Ses séquences pour le versioning
- Ses index performants
- Ses transactions ACID
- Son mécanisme LISTEN/NOTIFY pour les notifications

### Structure de Base

#### Table des Événements

```sql
-- Table principale des événements
CREATE TABLE events (
    -- Identifiant global unique de l'événement.
    -- 💡 PG 18 : préférer `uuidv7()` à `gen_random_uuid()` (UUIDv4) pour un
    --    event store. UUIDv7 encode un timestamp dans les 48 premiers bits,
    --    donc les nouveaux UUID s'insèrent en fin d'index B-tree (séquentiel)
    --    au lieu de tomber à des positions aléatoires (UUIDv4). Résultat :
    --      - pas de page splits → moins de fragmentation de l'index
    --      - meilleure cache locality → INSERT plus rapides
    --      - tri naturel par création → simplifie les requêtes temporelles
    --    Particulièrement pertinent pour un event store qui INSERT en masse.
    event_id        UUID PRIMARY KEY DEFAULT uuidv7(),

    -- Identifiant du stream (entité concernée)
    stream_id       VARCHAR(255) NOT NULL,

    -- Type de stream (pour filtrage par catégorie)
    stream_type     VARCHAR(100) NOT NULL,

    -- Version de l'événement dans le stream (1, 2, 3...)
    stream_version  INTEGER NOT NULL,

    -- Type d'événement
    event_type      VARCHAR(100) NOT NULL,

    -- Données de l'événement (payload)
    data            JSONB NOT NULL,

    -- Métadonnées (user, correlation, causation)
    metadata        JSONB DEFAULT '{}',

    -- Horodatage
    occurred_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    recorded_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    -- Contrainte d'unicité : un seul événement par version dans un stream
    CONSTRAINT uq_stream_version UNIQUE (stream_id, stream_version)
);

-- Index pour les lectures fréquentes.
-- Note : la contrainte UNIQUE (stream_id, stream_version) ci-dessus crée
-- DÉJÀ un index B-tree sur ces deux colonnes — pas besoin d'un index
-- supplémentaire `(stream_id, stream_version)`, ce serait un doublon
-- qui gaspille espace et ralentit les écritures.
CREATE INDEX idx_events_stream_type ON events(stream_type, recorded_at);  
CREATE INDEX idx_events_event_type  ON events(event_type, recorded_at);  
CREATE INDEX idx_events_occurred_at ON events(occurred_at);  

-- Index GIN pour recherche dans le payload JSON
CREATE INDEX idx_events_data ON events USING GIN(data);
```

#### Séquence Globale pour l'Ordre Total

```sql
-- Position globale pour donner un ordre aux événements
CREATE SEQUENCE events_global_position_seq;

-- Ajouter la colonne de position globale
ALTER TABLE events
    ADD COLUMN global_position BIGINT
    DEFAULT nextval('events_global_position_seq');

CREATE INDEX idx_events_global_position ON events(global_position);
```

> ⚠️ **Subtilité importante — l'ordre de séquence ≠ l'ordre de commit**  
>  
> `nextval()` alloue un numéro AVANT le `COMMIT`. Si deux transactions T1 et T2  
> appellent `nextval()` dans cet ordre, mais que T2 commite avant T1 :  
>  
> - T1 obtient `global_position=42`, T2 obtient `global_position=43`  
> - T2 commite à 10h00:00.5, T1 commite à 10h00:01.0  
> - Un consumer qui lit dans l'ordre 42 → 43 verra **T2 avant T1** ou laissera  
>   un "trou" temporaire (T2 visible mais T1 invisible quelques ms).  
>  
> Pour un **vrai ordre total cohérent avec les commits**, deux approches :  
> 1. **Logical Decoding** (cf. 21.2.3) : PostgreSQL garantit l'ordre des  
>    commits dans le WAL ; les consumers CDC voient l'ordre de commit réel.  
> 2. **Sérialiser les writes** sur l'event store (un seul writer) — sacrifie  
>    le throughput pour l'ordre strict.  
>  
> Pour la plupart des cas d'usage (projections eventually-consistent),  
> l'ordre par séquence reste acceptable : les "trous" se résorbent en  
> quelques ms et le consumer peut tolérer ce délai.

### Explication des Colonnes

| Colonne | Rôle |
|---------|------|
| `event_id` | Identifiant unique universel de l'événement |
| `stream_id` | Identifie l'entité (ex: `order-12345`) |
| `stream_type` | Catégorie du stream (ex: `Order`, `Account`) |
| `stream_version` | Position dans le stream (contrôle de concurrence) |
| `event_type` | Type métier de l'événement |
| `data` | Payload JSON avec les détails |
| `metadata` | Contexte : qui, quand, pourquoi, correlation |
| `occurred_at` | Quand l'événement s'est réellement produit |
| `recorded_at` | Quand il a été enregistré dans le store |
| `global_position` | Ordre total pour la projection |

## Opérations Fondamentales

### 1. Append : Ajouter des Événements

L'opération principale est l'**append** (ajout) d'événements à un stream.

```sql
-- Fonction pour ajouter un événement avec contrôle de version
CREATE OR REPLACE FUNCTION append_event(
    p_stream_id VARCHAR(255),
    p_stream_type VARCHAR(100),
    p_expected_version INTEGER,  -- Version attendue (pour optimistic locking)
    p_event_type VARCHAR(100),
    p_data JSONB,
    p_metadata JSONB DEFAULT '{}'
) RETURNS UUID AS $$
DECLARE
    v_current_version INTEGER;
    v_new_event_id UUID;
BEGIN
    -- Verrouiller le stream pour cette transaction via advisory lock
    -- (libéré automatiquement au COMMIT/ROLLBACK).
    -- Note : `FOR UPDATE` est interdit avec une fonction d'agrégat
    --        (`MAX(stream_version)`), d'où l'usage d'un advisory lock.
    PERFORM pg_advisory_xact_lock(hashtext(p_stream_id));

    -- Obtenir la version actuelle du stream
    SELECT COALESCE(MAX(stream_version), 0) INTO v_current_version
    FROM events
    WHERE stream_id = p_stream_id;

    -- Vérifier la version attendue (optimistic concurrency).
    -- p_expected_version = -1 → mode "ignorer la vérification" (append libre).
    IF p_expected_version != -1 AND v_current_version != p_expected_version THEN
        RAISE EXCEPTION 'Concurrency conflict on stream % : expected version %, got %',
            p_stream_id, p_expected_version, v_current_version
            USING ERRCODE = 'serialization_failure';
    END IF;

    -- Insérer le nouvel événement
    -- La contrainte UNIQUE (stream_id, stream_version) reste un deuxième filet
    -- de sécurité : si deux transactions parallèles franchissent miraculeusement
    -- l'advisory lock, l'une d'elles échouera sur SQLSTATE 23505 (unique_violation).
    INSERT INTO events (
        stream_id,
        stream_type,
        stream_version,
        event_type,
        data,
        metadata
    ) VALUES (
        p_stream_id,
        p_stream_type,
        v_current_version + 1,
        p_event_type,
        p_data,
        p_metadata
    )
    RETURNING event_id INTO v_new_event_id;

    RETURN v_new_event_id;
END;
$$ LANGUAGE plpgsql;
```

> **Pourquoi `pg_advisory_xact_lock(hashtext(...))` ?**  
> - `FOR UPDATE` est interdit avec `MAX()` (`ERROR: FOR UPDATE is not allowed with aggregate functions`).  
> - Verrouiller toute la table avec `LOCK TABLE` sérialiserait *tous* les streams (catastrophique).  
> - L'advisory lock par `hashtext(stream_id)` sérialise uniquement les appends sur **le même stream** : les autres streams peuvent continuer à appender en parallèle.  
> - Le suffixe `_xact_` libère le lock automatiquement au COMMIT/ROLLBACK — pas de fuite possible.  
>  
> ⚠️ **Collisions de hash** : `hashtext()` produit un INTEGER 32 bits. Avec  
> beaucoup de streams actifs (~65 000 selon le paradoxe des anniversaires),  
> deux `stream_id` distincts peuvent partager le même hash et subir une  
> **fausse contention** (sérialisation inutile). Ce n'est pas un bug de  
> correction (la contrainte UNIQUE garantit toujours l'intégrité), juste  
> une perte de parallélisme marginale. Pour des charges extrêmes, utiliser  
> la variante deux-arguments : `pg_advisory_xact_lock(hashtextextended(p_stream_id, 0))`  
> (64 bits, collisions négligeables) ou indexer un identifiant entier dédié.

#### Utilisation

```sql
-- Créer un nouveau compte (premier événement du stream)
SELECT append_event(
    'account-001',                              -- stream_id
    'Account',                                  -- stream_type
    0,                                          -- expected_version (0 = nouveau stream)
    'AccountOpened',                            -- event_type
    '{"owner": "Alice", "initial_balance": 1000, "currency": "EUR"}'::jsonb,
    '{"user_id": "user-42", "ip": "192.168.1.1"}'::jsonb
);

-- Ajouter un retrait (après le premier événement)
SELECT append_event(
    'account-001',
    'Account',
    1,                                          -- expected_version = 1
    'MoneyWithdrawn',
    '{"amount": 200, "reason": "ATM withdrawal", "atm_id": "ATM-789"}'::jsonb,
    '{"user_id": "user-42", "correlation_id": "tx-abc123"}'::jsonb
);

-- Ajouter un dépôt
SELECT append_event(
    'account-001',
    'Account',
    2,                                          -- expected_version = 2
    'MoneyDeposited',
    '{"amount": 500, "source": "salary", "reference": "SAL-2025-01"}'::jsonb,
    '{"user_id": "system", "correlation_id": "payroll-001"}'::jsonb
);
```

### 2. Load : Charger un Stream

Pour reconstruire l'état d'une entité, nous chargeons tous ses événements.

```sql
-- Fonction pour charger tous les événements d'un stream
CREATE OR REPLACE FUNCTION load_stream(
    p_stream_id VARCHAR(255),
    p_from_version INTEGER DEFAULT 0,
    p_to_version INTEGER DEFAULT NULL
) RETURNS TABLE (
    event_id UUID,
    stream_version INTEGER,
    event_type VARCHAR(100),
    data JSONB,
    metadata JSONB,
    occurred_at TIMESTAMPTZ
) AS $$
BEGIN
    RETURN QUERY
    SELECT
        e.event_id,
        e.stream_version,
        e.event_type,
        e.data,
        e.metadata,
        e.occurred_at
    FROM events e
    WHERE e.stream_id = p_stream_id
    AND e.stream_version > p_from_version
    AND (p_to_version IS NULL OR e.stream_version <= p_to_version)
    ORDER BY e.stream_version ASC;
END;
$$ LANGUAGE plpgsql;
```

#### Utilisation

```sql
-- Charger tous les événements du compte
SELECT * FROM load_stream('account-001');

-- Résultat :
-- event_id | stream_version | event_type      | data                          | ...
-- ---------|----------------|-----------------|-------------------------------|----
-- uuid-1   | 1              | AccountOpened   | {"owner": "Alice", ...}       | ...
-- uuid-2   | 2              | MoneyWithdrawn  | {"amount": 200, ...}          | ...
-- uuid-3   | 3              | MoneyDeposited  | {"amount": 500, ...}          | ...
```

### 3. Reconstruire l'État (Rehydration)

L'état actuel se calcule en "rejouant" les événements.

```sql
-- Vue pour calculer le solde actuel de chaque compte
CREATE OR REPLACE VIEW account_balances AS  
SELECT  
    stream_id AS account_id,
    -- Extraire le propriétaire du premier événement
    (SELECT data->>'owner'
     FROM events e2
     WHERE e2.stream_id = e.stream_id
     AND e2.event_type = 'AccountOpened'
     LIMIT 1) AS owner,
    -- Calculer le solde en sommant les mouvements
    SUM(
        CASE
            WHEN event_type = 'AccountOpened' THEN (data->>'initial_balance')::numeric
            WHEN event_type = 'MoneyDeposited' THEN (data->>'amount')::numeric
            WHEN event_type = 'MoneyWithdrawn' THEN -(data->>'amount')::numeric
            ELSE 0
        END
    ) AS balance,
    -- Dernière activité
    MAX(occurred_at) AS last_activity,
    -- Nombre de transactions
    COUNT(*) FILTER (WHERE event_type IN ('MoneyDeposited', 'MoneyWithdrawn')) AS transaction_count
FROM events e  
WHERE stream_type = 'Account'  
GROUP BY stream_id;  
```

```sql
-- Consulter l'état actuel
SELECT * FROM account_balances WHERE account_id = 'account-001';

-- Résultat :
-- account_id   | owner | balance | last_activity       | transaction_count
-- -------------|-------|---------|---------------------|------------------
-- account-001  | Alice | 1300.00 | 2025-01-15 09:00:00 | 2
```

### 4. Fonction de Reconstruction Complète

Pour une logique métier plus complexe, une fonction PL/pgSQL :

```sql
-- Type pour représenter l'état d'un compte
CREATE TYPE account_state AS (
    account_id VARCHAR(255),
    owner VARCHAR(255),
    balance NUMERIC(15,2),
    currency VARCHAR(3),
    status VARCHAR(20),
    opened_at TIMESTAMPTZ,
    version INTEGER
);

-- Fonction pour reconstruire l'état d'un compte
CREATE OR REPLACE FUNCTION get_account_state(p_account_id VARCHAR(255))  
RETURNS account_state AS $$  
DECLARE  
    v_state account_state;
    v_event RECORD;
BEGIN
    -- Initialiser l'état
    v_state.account_id := p_account_id;
    v_state.balance := 0;
    v_state.status := 'unknown';
    v_state.version := 0;

    -- Rejouer chaque événement
    FOR v_event IN (SELECT * FROM load_stream(p_account_id)) LOOP
        v_state.version := v_event.stream_version;

        CASE v_event.event_type
            WHEN 'AccountOpened' THEN
                v_state.owner := v_event.data->>'owner';
                v_state.balance := (v_event.data->>'initial_balance')::numeric;
                v_state.currency := COALESCE(v_event.data->>'currency', 'EUR');
                v_state.status := 'active';
                v_state.opened_at := v_event.occurred_at;

            WHEN 'MoneyDeposited' THEN
                v_state.balance := v_state.balance + (v_event.data->>'amount')::numeric;

            WHEN 'MoneyWithdrawn' THEN
                v_state.balance := v_state.balance - (v_event.data->>'amount')::numeric;

            WHEN 'AccountClosed' THEN
                v_state.status := 'closed';

            WHEN 'AccountFrozen' THEN
                v_state.status := 'frozen';

            ELSE
                -- Événement inconnu, ignorer silencieusement
                NULL;
        END CASE;
    END LOOP;

    RETURN v_state;
END;
$$ LANGUAGE plpgsql;
```

```sql
-- Utilisation
SELECT * FROM get_account_state('account-001');

-- Résultat :
-- account_id  | owner | balance | currency | status | opened_at           | version
-- ------------|-------|---------|----------|--------|---------------------|--------
-- account-001 | Alice | 1300.00 | EUR      | active | 2025-01-01 10:00:00 | 3
```

## Contrôle de Concurrence Optimiste

### Le Problème

Que se passe-t-il si deux processus tentent de modifier le même compte simultanément ?

```
Processus A                    Processus B
    │                              │
    ▼                              ▼
Lire account-001               Lire account-001
(version 3, solde 1300)        (version 3, solde 1300)
    │                              │
    ▼                              ▼
Retrait de 500                 Dépôt de 200  
Nouveau solde: 800             Nouveau solde: 1500  
    │                              │
    ▼                              ▼
Append (expected=3)            Append (expected=3)
    │                              │
    ▼                              ▼
Succès (version 4)             ???

Problème : B a travaillé sur un état obsolète !
```

### La Solution : Expected Version

Le paramètre `expected_version` résout ce problème :

```sql
-- Processus A réussit
SELECT append_event('account-001', 'Account', 3, 'MoneyWithdrawn',
    '{"amount": 500}'::jsonb);
-- OK, version devient 4

-- Processus B échoue
SELECT append_event('account-001', 'Account', 3, 'MoneyDeposited',
    '{"amount": 200}'::jsonb);
-- ERREUR: Concurrency conflict: expected version 3, got 4

-- Processus B doit recharger l'état et réessayer
SELECT * FROM load_stream('account-001');  -- Recharger
-- Recalculer avec le nouveau solde
SELECT append_event('account-001', 'Account', 4, 'MoneyDeposited',
    '{"amount": 200}'::jsonb);
-- OK, version devient 5
```

### Modes de Vérification

| Mode | Valeur | Comportement |
|------|--------|--------------|
| **Strict** | `expected_version = N` | Échoue si version actuelle ≠ N |
| **Any** | `expected_version = -1` | Accepte quelle que soit la version |
| **No Stream** | `expected_version = 0` | Échoue si le stream existe déjà |

## Snapshots : Optimiser la Reconstruction

### Le Problème des Longs Streams

Si un compte a 10 000 événements, reconstruire son état à chaque lecture devient coûteux.

```
Stream avec 10 000 événements :

┌───┐ ┌───┐ ┌───┐ ┌───┐         ┌────┐ ┌───┐
│ 1 │─│ 2 │─│ 3 │─│ 4 │─ ... ───│9999│─│10K│
└───┘ └───┘ └───┘ └───┘         └────┘ └───┘
  │                                       │
  └───────────────────────────────────────┘
        Rejouer 10 000 événements ?
                  😰
```

### La Solution : Snapshots

Un **snapshot** capture l'état à un instant donné. On ne rejoue que les événements après le snapshot.

```
Avec snapshot à la version 9000 :

┌───┐     ┌────┐ ┌────┐         ┌────┐ ┌───┐
│ 1 │─...─│9000│─│9001│─ ... ── │9999│─│10K│
└───┘     └────┘ └────┘         └────┘ └───┘
            │
    ┌───────┴───────┐
    │   SNAPSHOT    │
    │ version: 9000 │
    │ balance: 5000 │
    │ owner: Alice  │
    └───────────────┘
            │
            └──► Rejouer seulement 1000 événements
                         😊
```

### Implémentation des Snapshots

```sql
-- Table des snapshots
CREATE TABLE snapshots (
    stream_id       VARCHAR(255) NOT NULL,
    stream_type     VARCHAR(100) NOT NULL,
    stream_version  INTEGER NOT NULL,
    state           JSONB NOT NULL,
    created_at      TIMESTAMPTZ DEFAULT NOW(),

    PRIMARY KEY (stream_id, stream_version)
);

-- Note : PostgreSQL peut scanner l'index de la PRIMARY KEY en sens inverse
-- pour répondre à `ORDER BY stream_version DESC LIMIT 1`. Pas besoin d'un
-- index dédié `(stream_id, stream_version DESC)` — ce serait un doublon.
```

```sql
-- Fonction pour sauvegarder un snapshot
CREATE OR REPLACE FUNCTION save_snapshot(
    p_stream_id VARCHAR(255),
    p_stream_type VARCHAR(100),
    p_version INTEGER,
    p_state JSONB
) RETURNS VOID AS $$
BEGIN
    INSERT INTO snapshots (stream_id, stream_type, stream_version, state)
    VALUES (p_stream_id, p_stream_type, p_version, p_state)
    ON CONFLICT (stream_id, stream_version)
    DO UPDATE SET state = p_state, created_at = NOW();
END;
$$ LANGUAGE plpgsql;

-- Fonction pour charger l'état avec snapshot
CREATE OR REPLACE FUNCTION get_account_state_with_snapshot(p_account_id VARCHAR(255))  
RETURNS account_state AS $$  
DECLARE  
    v_snapshot RECORD;
    v_state account_state;
    v_event RECORD;
    v_from_version INTEGER := 0;
BEGIN
    -- Chercher le dernier snapshot
    SELECT * INTO v_snapshot
    FROM snapshots
    WHERE stream_id = p_account_id
    ORDER BY stream_version DESC
    LIMIT 1;

    IF FOUND THEN
        -- Initialiser depuis le snapshot
        v_state.account_id := p_account_id;
        v_state.owner := v_snapshot.state->>'owner';
        v_state.balance := (v_snapshot.state->>'balance')::numeric;
        v_state.currency := v_snapshot.state->>'currency';
        v_state.status := v_snapshot.state->>'status';
        v_state.opened_at := (v_snapshot.state->>'opened_at')::timestamptz;
        v_state.version := v_snapshot.stream_version;
        v_from_version := v_snapshot.stream_version;
    ELSE
        -- Pas de snapshot, initialiser à vide
        v_state.account_id := p_account_id;
        v_state.balance := 0;
        v_state.status := 'unknown';
        v_state.version := 0;
    END IF;

    -- Rejouer les événements APRÈS le snapshot
    FOR v_event IN (SELECT * FROM load_stream(p_account_id, v_from_version)) LOOP
        v_state.version := v_event.stream_version;

        CASE v_event.event_type
            WHEN 'AccountOpened' THEN
                v_state.owner := v_event.data->>'owner';
                v_state.balance := (v_event.data->>'initial_balance')::numeric;
                v_state.currency := COALESCE(v_event.data->>'currency', 'EUR');
                v_state.status := 'active';
                v_state.opened_at := v_event.occurred_at;
            WHEN 'MoneyDeposited' THEN
                v_state.balance := v_state.balance + (v_event.data->>'amount')::numeric;
            WHEN 'MoneyWithdrawn' THEN
                v_state.balance := v_state.balance - (v_event.data->>'amount')::numeric;
            WHEN 'AccountClosed' THEN
                v_state.status := 'closed';
            WHEN 'AccountFrozen' THEN
                v_state.status := 'frozen';
            ELSE
                NULL;
        END CASE;
    END LOOP;

    RETURN v_state;
END;
$$ LANGUAGE plpgsql;
```

### Stratégies de Création de Snapshots

| Stratégie | Description |
|-----------|-------------|
| **Périodique** | Toutes les N versions (ex: tous les 100 événements) |
| **Temporelle** | Toutes les N heures/jours |
| **À la demande** | Lors d'opérations de maintenance |
| **Conditionnelle** | Après certains types d'événements importants |

```sql
-- Exemple : Créer un snapshot si > 100 événements depuis le dernier
CREATE OR REPLACE FUNCTION maybe_create_snapshot(p_account_id VARCHAR(255))  
RETURNS VOID AS $$  
DECLARE  
    v_state account_state;
    v_last_snapshot_version INTEGER;
    v_current_version INTEGER;
BEGIN
    -- Version du dernier snapshot
    SELECT COALESCE(MAX(stream_version), 0) INTO v_last_snapshot_version
    FROM snapshots WHERE stream_id = p_account_id;

    -- Version actuelle du stream
    SELECT COALESCE(MAX(stream_version), 0) INTO v_current_version
    FROM events WHERE stream_id = p_account_id;

    -- Créer snapshot si > 100 événements d'écart
    IF v_current_version - v_last_snapshot_version > 100 THEN
        v_state := get_account_state_with_snapshot(p_account_id);

        PERFORM save_snapshot(
            p_account_id,
            'Account',
            v_state.version,
            jsonb_build_object(
                'owner', v_state.owner,
                'balance', v_state.balance,
                'currency', v_state.currency,
                'status', v_state.status,
                'opened_at', v_state.opened_at
            )
        );
    END IF;
END;
$$ LANGUAGE plpgsql;
```

## Notifications en Temps Réel

PostgreSQL permet de notifier les applications quand de nouveaux événements sont ajoutés.

### Avec LISTEN/NOTIFY

```sql
-- Trigger pour notifier les nouveaux événements
CREATE OR REPLACE FUNCTION notify_new_event()  
RETURNS TRIGGER AS $$  
BEGIN  
    -- Notifier sur un canal par type de stream
    PERFORM pg_notify(
        'events_' || NEW.stream_type,
        json_build_object(
            'event_id', NEW.event_id,
            'stream_id', NEW.stream_id,
            'stream_version', NEW.stream_version,
            'event_type', NEW.event_type,
            'global_position', NEW.global_position
        )::text
    );

    -- Notifier aussi sur un canal global
    PERFORM pg_notify(
        'all_events',
        json_build_object(
            'event_id', NEW.event_id,
            'stream_type', NEW.stream_type,
            'event_type', NEW.event_type,
            'global_position', NEW.global_position
        )::text
    );

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_notify_new_event  
AFTER INSERT ON events  
FOR EACH ROW  
EXECUTE FUNCTION notify_new_event();  
```

## Requêtes Utiles pour l'Event Store

### Statistiques Globales

```sql
-- Nombre d'événements par type
SELECT
    event_type,
    COUNT(*) as count,
    MIN(occurred_at) as first_occurrence,
    MAX(occurred_at) as last_occurrence
FROM events  
GROUP BY event_type  
ORDER BY count DESC;  

-- Activité par jour
SELECT
    DATE(occurred_at) as day,
    COUNT(*) as events_count,
    COUNT(DISTINCT stream_id) as active_streams
FROM events  
GROUP BY DATE(occurred_at)  
ORDER BY day DESC  
LIMIT 30;  
```

### Recherche dans les Événements

```sql
-- Trouver tous les gros dépôts
SELECT
    stream_id,
    event_type,
    data->>'amount' as amount,
    occurred_at
FROM events  
WHERE event_type = 'MoneyDeposited'  
AND (data->>'amount')::numeric > 10000  
ORDER BY occurred_at DESC;  

-- Historique complet d'un compte
SELECT
    occurred_at,
    event_type,
    CASE event_type
        WHEN 'AccountOpened' THEN '+'||(data->>'initial_balance')
        WHEN 'MoneyDeposited' THEN '+'||(data->>'amount')
        WHEN 'MoneyWithdrawn' THEN '-'||(data->>'amount')
        ELSE NULL
    END as movement,
    data->>'reason' as reason,
    metadata->>'user_id' as performed_by
FROM events  
WHERE stream_id = 'account-001'  
ORDER BY stream_version;  
```

### Audit et Traçabilité

```sql
-- Toutes les actions d'un utilisateur
SELECT
    e.occurred_at,
    e.stream_type,
    e.stream_id,
    e.event_type,
    e.data
FROM events e  
WHERE e.metadata->>'user_id' = 'user-42'  
ORDER BY e.occurred_at DESC  
LIMIT 100;  

-- Corrélation : suivre une transaction à travers les services
SELECT
    e.stream_type,
    e.stream_id,
    e.event_type,
    e.occurred_at
FROM events e  
WHERE e.metadata->>'correlation_id' = 'order-tx-abc123'  
ORDER BY e.global_position;
```

## Bonnes Pratiques

### 1. Nommer les Événements au Passé

```sql
-- ✅ Bon : décrit ce qui S'EST passé
'OrderPlaced', 'PaymentReceived', 'ItemShipped'

-- ❌ Mauvais : décrit une intention
'PlaceOrder', 'ReceivePayment', 'ShipItem'
```

### 2. Événements Autonomes et Complets

```sql
-- ✅ Bon : contient toutes les infos nécessaires
{"event_type": "OrderPlaced", "data": {
    "order_id": "ord-123",
    "customer_id": "cust-456",
    "customer_name": "Alice",        -- Dénormalisé
    "items": [...],
    "total": 150.00,
    "currency": "EUR"
}}

-- ❌ Mauvais : nécessite des lookups
{"event_type": "OrderPlaced", "data": {
    "order_id": "ord-123",
    "customer_id": "cust-456"        -- Juste l'ID, pas le nom
}}
```

### 3. Ne Jamais Modifier un Événement

```sql
-- ❌ INTERDIT : UPDATE sur la table events
UPDATE events SET data = '...' WHERE event_id = 'xxx';

-- ✅ Correct : ajouter un événement de correction
INSERT INTO events (...) VALUES (
    ..., 'OrderCorrected', '{"reason": "prix incorrect", "old_total": 150, "new_total": 140}'
);
```

## Conclusion

L'**Event Store** est le fondement de l'Event Sourcing. En stockant les événements plutôt que l'état, nous obtenons un système avec :

- Un **historique complet** et immutable
- Une **traçabilité parfaite** pour l'audit
- La capacité de **reconstruire l'état** à n'importe quel moment
- Une **intégration naturelle** avec d'autres systèmes via les événements

PostgreSQL, avec son support JSONB, ses transactions ACID et son mécanisme LISTEN/NOTIFY, constitue une excellente plateforme pour construire un Event Store robuste et performant.

Dans les prochaines sections, nous verrons comment exploiter ces événements avec **NOTIFY/LISTEN** pour le temps réel, et comment construire des **projections optimisées** avec le pattern **CQRS**.

## Points Clés à Retenir

- **Event Sourcing** : Stocker les événements, pas l'état  
- **Event Store** : Base de données append-only pour les événements  
- **Stream** : Séquence d'événements pour une entité  
- **Reconstruction** : L'état se calcule en rejouant les événements  
- **Optimistic Locking** : `expected_version` pour la concurrence  
- **Snapshots** : Caches d'état pour optimiser la reconstruction  
- **Immutabilité** : Ne jamais modifier ou supprimer un événement  
- **LISTEN/NOTIFY** : Notifications temps réel avec PostgreSQL
