# 21.1.1 Résumé pour Débutants : SQL, DDL, DML et Contraintes

## Introduction

Bienvenue dans ce guide de synthèse destiné aux débutants en PostgreSQL. Ce chapitre récapitule les concepts fondamentaux que tout développeur ou administrateur de bases de données doit maîtriser avant de progresser vers des sujets plus avancés.

Nous allons couvrir quatre piliers essentiels :

1. **SQL** : Le langage universel des bases de données relationnelles  
2. **DDL** : La définition des structures de données  
3. **DML** : La manipulation des données  
4. **Contraintes** : Les règles garantissant l'intégrité des données

---

## 1. SQL : Le Langage Universel des Bases de Données

### 1.1 Qu'est-ce que SQL ?

**SQL** (Structured Query Language, ou Langage de Requête Structuré) est le langage standard utilisé pour communiquer avec les bases de données relationnelles. Créé dans les années 1970, il est aujourd'hui utilisé par pratiquement tous les systèmes de gestion de bases de données relationnelles (SGBDR), dont PostgreSQL.

SQL vous permet de :

- Créer et modifier la structure de vos bases de données
- Insérer, modifier et supprimer des données
- Interroger vos données pour obtenir des informations précises
- Contrôler les accès et les permissions

### 1.2 Les Catégories de Commandes SQL

SQL se divise en plusieurs sous-langages, chacun ayant un rôle spécifique :

| Catégorie | Nom Complet | Rôle | Exemples de Commandes |
|-----------|-------------|------|----------------------|
| **DDL** | Data Definition Language | Définir la structure | `CREATE`, `ALTER`, `DROP` |
| **DML** | Data Manipulation Language | Manipuler les données | `INSERT`, `UPDATE`, `DELETE` |
| **DQL** | Data Query Language | Interroger les données | `SELECT` |
| **DCL** | Data Control Language | Gérer les permissions | `GRANT`, `REVOKE` |
| **TCL** | Transaction Control Language | Contrôler les transactions | `BEGIN`, `COMMIT`, `ROLLBACK` |

> **Note** : Certains considèrent DQL comme faisant partie de DML. Dans ce guide, nous nous concentrons sur DDL et DML, en incluant `SELECT` dans notre discussion sur DML pour des raisons pratiques.

### 1.3 Syntaxe de Base SQL

SQL utilise une syntaxe déclarative : vous décrivez **ce que vous voulez obtenir**, et non **comment l'obtenir**. Voici quelques règles fondamentales :

- Les commandes SQL ne sont **pas sensibles à la casse** (majuscules/minuscules)
- Par convention, les mots-clés SQL s'écrivent en MAJUSCULES
- Chaque instruction se termine par un **point-virgule** (`;`)
- Les chaînes de caractères sont entourées de **guillemets simples** (`'texte'`)
- Les commentaires utilisent `--` (une ligne) ou `/* ... */` (plusieurs lignes)

```sql
-- Ceci est un commentaire sur une ligne

/* Ceci est un commentaire
   sur plusieurs lignes */

SELECT nom, prenom   -- Les mots-clés en majuscules  
FROM utilisateurs    -- Les noms de tables/colonnes en minuscules  
WHERE age >= 18;     -- Point-virgule pour terminer  
```

## 2. DDL : Data Definition Language (Langage de Définition des Données)

Le DDL permet de créer, modifier et supprimer les **structures** qui contiennent vos données : bases de données, schémas, tables, index, etc.

### 2.1 Hiérarchie des Objets PostgreSQL

Avant de créer des objets, comprenons comment PostgreSQL organise les données :

```
Instance PostgreSQL (Serveur)
    └── Database (Base de données)
            └── Schema (Schéma / Espace de noms)
                    └── Table (Table)
                            └── Column (Colonne)
```

- **Instance** : Le serveur PostgreSQL lui-même  
- **Database** : Une base de données isolée (vous pouvez en avoir plusieurs)  
- **Schema** : Un espace de noms pour organiser les objets (par défaut : `public`)  
- **Table** : La structure qui contient les données en lignes et colonnes

### 2.2 CREATE : Créer des Objets

#### Créer une Base de Données

```sql
CREATE DATABASE ma_boutique;
```

#### Créer un Schéma

```sql
CREATE SCHEMA ventes;
```

#### Créer une Table

La création de table est l'opération DDL la plus courante. Voici la syntaxe de base :

```sql
CREATE TABLE nom_de_table (
    nom_colonne1 type_donnees contraintes,
    nom_colonne2 type_donnees contraintes,
    ...
);
```

**Exemple concret** : Créons une table pour stocker des clients.

```sql
CREATE TABLE clients (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100) NOT NULL,
    prenom VARCHAR(100) NOT NULL,
    email VARCHAR(255) UNIQUE,
    date_naissance DATE,
    date_inscription TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

Décortiquons cet exemple :

| Colonne | Type | Explication |
|---------|------|-------------|
| `id` | `SERIAL` | Entier auto-incrémenté (1, 2, 3...) |
| `nom` | `VARCHAR(100)` | Texte variable jusqu'à 100 caractères |
| `prenom` | `VARCHAR(100)` | Texte variable jusqu'à 100 caractères |
| `email` | `VARCHAR(255)` | Texte variable jusqu'à 255 caractères |
| `date_naissance` | `DATE` | Date (année-mois-jour) |
| `date_inscription` | `TIMESTAMP` | Date et heure |

### 2.3 Les Types de Données Fondamentaux

PostgreSQL offre de nombreux types de données. Voici les plus courants pour débuter :

#### Types Numériques

| Type | Description | Plage de Valeurs |
|------|-------------|------------------|
| `SMALLINT` | Petit entier | -32 768 à 32 767 |
| `INTEGER` ou `INT` | Entier standard | -2 147 483 648 à 2 147 483 647 |
| `BIGINT` | Grand entier | Très grande plage |
| `SERIAL` | Entier auto-incrémenté | Comme INTEGER mais automatique |
| `NUMERIC(p,s)` | Nombre décimal précis | `p` chiffres, `s` décimales |
| `REAL` | Nombre à virgule flottante | Précision simple |
| `DOUBLE PRECISION` | Nombre à virgule flottante | Précision double |

```sql
-- Exemples d'utilisation
CREATE TABLE produits (
    id SERIAL,
    quantite INTEGER,
    prix NUMERIC(10, 2),    -- 10 chiffres dont 2 décimales (ex: 12345678.99)
    poids REAL
);
```

#### Types Texte

| Type | Description | Utilisation |
|------|-------------|-------------|
| `CHAR(n)` | Chaîne fixe de n caractères | Codes fixes (ex: code postal) |
| `VARCHAR(n)` | Chaîne variable jusqu'à n caractères | Texte de longueur variable |
| `TEXT` | Chaîne de longueur illimitée | Descriptions, contenus longs |

```sql
-- Exemples d'utilisation
CREATE TABLE articles (
    code_postal CHAR(5),        -- Toujours 5 caractères
    titre VARCHAR(200),          -- Jusqu'à 200 caractères
    contenu TEXT                 -- Sans limite
);
```

> **Conseil PostgreSQL** : En pratique, `VARCHAR` sans limite de taille et `TEXT` sont équivalents en termes de performance dans PostgreSQL. Utilisez `VARCHAR(n)` si vous voulez imposer une limite métier.

#### Types Temporels

| Type | Description | Exemple |
|------|-------------|---------|
| `DATE` | Date seule | `'2025-11-24'` |
| `TIME` | Heure seule | `'14:30:00'` |
| `TIMESTAMP` | Date et heure | `'2025-11-24 14:30:00'` |
| `TIMESTAMPTZ` | Date, heure et fuseau horaire | `'2025-11-24 14:30:00+01'` |
| `INTERVAL` | Durée | `'2 hours 30 minutes'` |

```sql
-- Exemples d'utilisation
CREATE TABLE evenements (
    date_evenement DATE,
    heure_debut TIME,
    cree_le TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    cree_le_tz TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);
```

> **Bonne pratique** : Privilégiez `TIMESTAMPTZ` pour stocker des dates avec heures. PostgreSQL convertira automatiquement selon le fuseau horaire de la session.

#### Types Booléen et Spéciaux

| Type | Description | Valeurs Possibles |
|------|-------------|-------------------|
| `BOOLEAN` | Vrai ou faux | `TRUE`, `FALSE`, `NULL` |
| `UUID` | Identifiant unique universel | `'a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11'` |
| `JSON` / `JSONB` | Données JSON | `'{"nom": "Dupont"}'` |

```sql
-- Exemples d'utilisation
CREATE TABLE utilisateurs (
    id UUID DEFAULT gen_random_uuid(),
    actif BOOLEAN DEFAULT TRUE,
    preferences JSONB
);
```

### 2.4 ALTER : Modifier des Objets Existants

La commande `ALTER` permet de modifier la structure d'objets existants sans les supprimer.

#### Ajouter une Colonne

```sql
ALTER TABLE clients  
ADD COLUMN telephone VARCHAR(20);  
```

#### Supprimer une Colonne

```sql
ALTER TABLE clients  
DROP COLUMN telephone;  
```

#### Modifier le Type d'une Colonne

```sql
ALTER TABLE clients  
ALTER COLUMN nom TYPE VARCHAR(150);  
```

#### Renommer une Colonne

```sql
ALTER TABLE clients  
RENAME COLUMN nom TO nom_famille;  
```

#### Renommer une Table

```sql
ALTER TABLE clients  
RENAME TO customers;  
```

#### Ajouter une Contrainte

```sql
ALTER TABLE clients  
ADD CONSTRAINT email_unique UNIQUE (email);  
```

#### Supprimer une Contrainte

```sql
ALTER TABLE clients  
DROP CONSTRAINT email_unique;  
```

### 2.5 DROP : Supprimer des Objets

La commande `DROP` supprime définitivement un objet et toutes ses données.

```sql
-- Supprimer une table
DROP TABLE clients;

-- Supprimer une table seulement si elle existe (évite les erreurs)
DROP TABLE IF EXISTS clients;

-- Supprimer une table et tous les objets qui en dépendent
DROP TABLE clients CASCADE;
```

> **Attention** : `DROP` est irréversible ! Utilisez toujours `IF EXISTS` pour éviter les erreurs dans vos scripts, et `CASCADE` avec précaution.

### 2.6 TRUNCATE : Vider une Table

`TRUNCATE` supprime toutes les lignes d'une table mais conserve sa structure.

```sql
TRUNCATE TABLE clients;

-- Réinitialiser également les séquences (compteurs auto-incrémentés)
TRUNCATE TABLE clients RESTART IDENTITY;
```

> **Différence avec DELETE** : `TRUNCATE` est beaucoup plus rapide car il ne parcourt pas chaque ligne. Cependant, il ne déclenche pas les triggers `ON DELETE` et ne peut pas être filtré avec `WHERE`.

## 3. DML : Data Manipulation Language (Langage de Manipulation des Données)

Le DML permet d'interagir avec les **données** contenues dans vos tables : insertion, modification, suppression et consultation.

### 3.1 INSERT : Insérer des Données

#### Insertion Simple

```sql
INSERT INTO clients (nom, prenom, email)  
VALUES ('Dupont', 'Marie', 'marie.dupont@email.com');  
```

#### Insertion avec Toutes les Colonnes

Si vous insérez une valeur pour chaque colonne (dans l'ordre de création), vous pouvez omettre la liste des colonnes :

```sql
-- Attention : l'ordre doit correspondre exactement à la structure de la table
INSERT INTO clients  
VALUES (1, 'Dupont', 'Marie', 'marie.dupont@email.com', '1990-05-15', NOW());  
```

> **Bonne pratique** : Spécifiez toujours explicitement les colonnes. Cela rend le code plus lisible et résistant aux modifications de schéma.

#### Insertion Multiple

Insérez plusieurs lignes en une seule commande :

```sql
INSERT INTO clients (nom, prenom, email)  
VALUES  
    ('Martin', 'Pierre', 'pierre.martin@email.com'),
    ('Bernard', 'Sophie', 'sophie.bernard@email.com'),
    ('Petit', 'Lucas', 'lucas.petit@email.com');
```

#### Insertion avec Valeurs par Défaut

```sql
-- Utilise les valeurs DEFAULT définies dans la table
INSERT INTO clients (nom, prenom)  
VALUES ('Durand', 'Alice');  
-- date_inscription prendra la valeur CURRENT_TIMESTAMP automatiquement
```

#### La Clause RETURNING

PostgreSQL offre une fonctionnalité puissante : récupérer les valeurs générées lors de l'insertion.

```sql
INSERT INTO clients (nom, prenom, email)  
VALUES ('Moreau', 'Julie', 'julie.moreau@email.com')  
RETURNING id, date_inscription;  
```

Cette requête insère la ligne ET retourne l'`id` auto-généré ainsi que la date d'inscription.

### 3.2 SELECT : Interroger les Données

`SELECT` est la commande la plus utilisée en SQL. Elle permet de lire et filtrer les données.

#### Sélection Basique

```sql
-- Sélectionner toutes les colonnes
SELECT * FROM clients;

-- Sélectionner des colonnes spécifiques
SELECT nom, prenom, email FROM clients;
```

#### Alias de Colonnes

```sql
SELECT
    nom AS nom_famille,
    prenom AS prenom_usuel,
    email AS adresse_email
FROM clients;
```

#### Filtrage avec WHERE

```sql
-- Égalité
SELECT * FROM clients WHERE nom = 'Dupont';

-- Comparaisons
SELECT * FROM clients WHERE date_naissance > '1990-01-01';

-- Opérateurs logiques
SELECT * FROM clients  
WHERE nom = 'Dupont' AND prenom = 'Marie';  

SELECT * FROM clients  
WHERE nom = 'Dupont' OR nom = 'Martin';  

-- Négation
SELECT * FROM clients WHERE nom != 'Dupont';  
SELECT * FROM clients WHERE NOT nom = 'Dupont';  
```

#### Opérateurs de Comparaison Courants

| Opérateur | Signification |
|-----------|---------------|
| `=` | Égal à |
| `!=` ou `<>` | Différent de |
| `<` | Inférieur à |
| `>` | Supérieur à |
| `<=` | Inférieur ou égal à |
| `>=` | Supérieur ou égal à |
| `BETWEEN ... AND ...` | Dans une plage |
| `IN (...)` | Dans une liste |
| `LIKE` | Correspondance de motif |
| `IS NULL` | Est nul |
| `IS NOT NULL` | N'est pas nul |

#### Exemples d'Opérateurs

```sql
-- BETWEEN : dans une plage (inclusif)
SELECT * FROM clients  
WHERE date_naissance BETWEEN '1980-01-01' AND '1990-12-31';  

-- IN : dans une liste de valeurs
SELECT * FROM clients  
WHERE nom IN ('Dupont', 'Martin', 'Bernard');  

-- LIKE : recherche par motif
SELECT * FROM clients WHERE nom LIKE 'D%';      -- Commence par D  
SELECT * FROM clients WHERE nom LIKE '%ont';    -- Finit par "ont"  
SELECT * FROM clients WHERE nom LIKE '%up%';    -- Contient "up"  
SELECT * FROM clients WHERE nom LIKE 'D_pont';  -- D + 1 caractère + pont  

-- ILIKE : comme LIKE mais insensible à la casse (spécifique PostgreSQL)
SELECT * FROM clients WHERE nom ILIKE 'd%';     -- d ou D

-- IS NULL / IS NOT NULL
SELECT * FROM clients WHERE email IS NULL;  
SELECT * FROM clients WHERE email IS NOT NULL;  
```

> **Important** : Pour tester si une valeur est `NULL`, utilisez toujours `IS NULL` ou `IS NOT NULL`. L'opérateur `=` ne fonctionne pas avec `NULL`.

#### Tri avec ORDER BY

```sql
-- Tri ascendant (par défaut)
SELECT * FROM clients ORDER BY nom;  
SELECT * FROM clients ORDER BY nom ASC;  

-- Tri descendant
SELECT * FROM clients ORDER BY date_inscription DESC;

-- Tri multiple
SELECT * FROM clients ORDER BY nom ASC, prenom ASC;

-- Gestion des NULL dans le tri
SELECT * FROM clients ORDER BY email NULLS LAST;   -- NULL à la fin  
SELECT * FROM clients ORDER BY email NULLS FIRST;  -- NULL au début  
```

#### Limitation des Résultats

```sql
-- Limiter le nombre de résultats
SELECT * FROM clients LIMIT 10;

-- Pagination : sauter les N premières lignes
SELECT * FROM clients LIMIT 10 OFFSET 20;  -- Lignes 21 à 30
```

> **Attention** : La pagination avec `OFFSET` devient lente sur de grandes tables. Pour de meilleures performances, utilisez une pagination par curseur (basée sur une valeur de colonne).

#### Éliminer les Doublons

```sql
-- Valeurs uniques d'une colonne
SELECT DISTINCT nom FROM clients;

-- Combinaisons uniques de colonnes
SELECT DISTINCT nom, prenom FROM clients;
```

### 3.3 UPDATE : Modifier des Données

`UPDATE` modifie les valeurs de colonnes pour les lignes correspondant à une condition.

#### Mise à Jour Simple

```sql
UPDATE clients  
SET email = 'nouveau.email@example.com'  
WHERE id = 1;  
```

#### Mise à Jour de Plusieurs Colonnes

```sql
UPDATE clients  
SET  
    email = 'marie.dupont.nouveau@email.com',
    nom = 'Dupont-Martin'
WHERE id = 1;
```

#### Mise à Jour avec Calcul

```sql
-- Supposons une table produits avec un prix
UPDATE produits  
SET prix = prix * 1.10  -- Augmentation de 10%  
WHERE categorie = 'électronique';  
```

#### UPDATE avec RETURNING

```sql
UPDATE clients  
SET email = 'nouveau@email.com'  
WHERE id = 1  
RETURNING id, nom, email;  -- Retourne les valeurs après modification  
```

> **Attention** : Un `UPDATE` sans clause `WHERE` modifie **toutes** les lignes de la table ! Vérifiez toujours votre condition avant d'exécuter.

```sql
-- DANGER : modifie TOUS les clients !
UPDATE clients SET actif = FALSE;
```

### 3.4 DELETE : Supprimer des Données

`DELETE` supprime les lignes correspondant à une condition.

#### Suppression avec Condition

```sql
DELETE FROM clients  
WHERE id = 1;  
```

#### Suppression avec Plusieurs Conditions

```sql
DELETE FROM clients  
WHERE date_inscription < '2020-01-01'  
AND actif = FALSE;  
```

#### DELETE avec RETURNING

```sql
DELETE FROM clients  
WHERE id = 1  
RETURNING *;  -- Retourne la ligne supprimée  
```

> **Attention** : Comme pour `UPDATE`, un `DELETE` sans `WHERE` supprime **toutes** les lignes !

```sql
-- DANGER : supprime TOUS les clients !
DELETE FROM clients;
```

### 3.5 UPSERT : INSERT ... ON CONFLICT

L'opération "UPSERT" (UPDATE + INSERT) permet d'insérer une ligne, ou de la mettre à jour si elle existe déjà.

#### Ignorer les Conflits

```sql
INSERT INTO clients (id, nom, prenom, email)  
VALUES (1, 'Dupont', 'Marie', 'marie@email.com')  
ON CONFLICT (id) DO NOTHING;  -- Si l'id existe, ne rien faire  
```

#### Mettre à Jour en Cas de Conflit

```sql
INSERT INTO clients (id, nom, prenom, email)  
VALUES (1, 'Dupont', 'Marie', 'marie.nouveau@email.com')  
ON CONFLICT (id)  
DO UPDATE SET  
    email = EXCLUDED.email,  -- EXCLUDED référence les nouvelles valeurs
    nom = EXCLUDED.nom;
```

`EXCLUDED` est un mot-clé spécial qui fait référence aux valeurs que vous tentiez d'insérer.

## 4. Contraintes d'Intégrité

Les contraintes sont des règles appliquées aux colonnes ou aux tables pour garantir la **validité** et la **cohérence** des données. Elles constituent un filet de sécurité essentiel.

### 4.1 Pourquoi les Contraintes ?

Les contraintes permettent de :

- **Garantir la qualité** des données à la source  
- **Éviter les erreurs** applicatives en amont  
- **Documenter** les règles métier dans le schéma  
- **Optimiser** les performances (les clés primaires créent automatiquement des index)

### 4.2 PRIMARY KEY : Clé Primaire

La clé primaire identifie de manière **unique** chaque ligne d'une table. Elle ne peut pas être `NULL` et chaque valeur doit être unique.

```sql
-- Méthode 1 : Contrainte au niveau de la colonne
CREATE TABLE clients (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100)
);

-- Méthode 2 : Contrainte au niveau de la table
CREATE TABLE clients (
    id SERIAL,
    nom VARCHAR(100),
    PRIMARY KEY (id)
);

-- Clé primaire composite (plusieurs colonnes)
CREATE TABLE commande_produits (
    commande_id INTEGER,
    produit_id INTEGER,
    quantite INTEGER,
    PRIMARY KEY (commande_id, produit_id)
);
```

> **Bonne pratique** : Chaque table devrait avoir une clé primaire. Préférez un identifiant technique (`id SERIAL` ou `UUID`) plutôt qu'une donnée métier qui pourrait changer.

### 4.3 FOREIGN KEY : Clé Étrangère

La clé étrangère crée une **relation** entre deux tables. Elle garantit que la valeur référence une ligne existante dans une autre table.

```sql
-- Table parente
CREATE TABLE categories (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100) NOT NULL
);

-- Table enfant avec clé étrangère
CREATE TABLE produits (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100) NOT NULL,
    categorie_id INTEGER REFERENCES categories(id)
);
```

#### Syntaxe Complète

```sql
CREATE TABLE produits (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100) NOT NULL,
    categorie_id INTEGER,
    FOREIGN KEY (categorie_id) REFERENCES categories(id)
        ON DELETE CASCADE      -- Que faire si la catégorie est supprimée ?
        ON UPDATE CASCADE      -- Que faire si l'id de la catégorie change ?
);
```

#### Actions Référentielles

| Action | Comportement |
|--------|--------------|
| `CASCADE` | Propager l'action aux lignes enfants |
| `RESTRICT` | Empêcher l'action si des enfants existent (par défaut) |
| `SET NULL` | Mettre la clé étrangère à NULL |
| `SET DEFAULT` | Mettre la clé étrangère à sa valeur par défaut |
| `NO ACTION` | Similaire à RESTRICT (vérification différée possible) |

**Exemple concret** :

```sql
-- Si on supprime une catégorie, tous ses produits sont aussi supprimés
ON DELETE CASCADE

-- Si on supprime une catégorie, l'opération est bloquée s'il y a des produits
ON DELETE RESTRICT

-- Si on supprime une catégorie, les produits ont categorie_id = NULL
ON DELETE SET NULL
```

### 4.4 UNIQUE : Valeur Unique

La contrainte `UNIQUE` garantit que toutes les valeurs d'une colonne (ou combinaison de colonnes) sont distinctes.

```sql
-- Contrainte sur une colonne
CREATE TABLE clients (
    id SERIAL PRIMARY KEY,
    email VARCHAR(255) UNIQUE
);

-- Contrainte sur plusieurs colonnes
CREATE TABLE employes (
    id SERIAL PRIMARY KEY,
    prenom VARCHAR(100),
    nom VARCHAR(100),
    UNIQUE (prenom, nom)  -- La combinaison prénom + nom doit être unique
);
```

> **Note** : Contrairement à `PRIMARY KEY`, une contrainte `UNIQUE` autorise les valeurs `NULL`. Plusieurs lignes peuvent avoir `NULL` dans une colonne `UNIQUE`.

### 4.5 NOT NULL : Valeur Obligatoire

`NOT NULL` interdit les valeurs nulles dans une colonne.

```sql
CREATE TABLE clients (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100) NOT NULL,      -- Obligatoire
    prenom VARCHAR(100) NOT NULL,   -- Obligatoire
    email VARCHAR(255)              -- Facultatif (NULL autorisé)
);
```

> **Conseil** : Réfléchissez bien à quelles colonnes doivent être obligatoires. Une fois la table remplie, ajouter `NOT NULL` peut être complexe si des valeurs nulles existent.

### 4.6 CHECK : Validation Personnalisée

`CHECK` permet de définir une condition que chaque valeur doit respecter.

```sql
CREATE TABLE produits (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100) NOT NULL,
    prix NUMERIC(10, 2) CHECK (prix >= 0),           -- Prix positif ou nul
    stock INTEGER CHECK (stock >= 0),                 -- Stock positif ou nul
    date_creation DATE CHECK (date_creation <= CURRENT_DATE)  -- Pas dans le futur
);

-- Contrainte CHECK nommée (facilite la maintenance)
CREATE TABLE employes (
    id SERIAL PRIMARY KEY,
    salaire NUMERIC(10, 2),
    CONSTRAINT salaire_positif CHECK (salaire > 0)
);

-- CHECK sur plusieurs colonnes
CREATE TABLE promotions (
    id SERIAL PRIMARY KEY,
    date_debut DATE,
    date_fin DATE,
    CONSTRAINT dates_valides CHECK (date_fin > date_debut)
);
```

### 4.7 DEFAULT : Valeur par Défaut

`DEFAULT` définit la valeur utilisée si aucune valeur n'est spécifiée lors de l'insertion.

```sql
CREATE TABLE commandes (
    id SERIAL PRIMARY KEY,
    date_commande TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    statut VARCHAR(20) DEFAULT 'en_attente',
    montant NUMERIC(10, 2) DEFAULT 0.00,
    actif BOOLEAN DEFAULT TRUE
);
```

### 4.8 Nommer les Contraintes

Il est recommandé de nommer explicitement vos contraintes pour faciliter la maintenance et les messages d'erreur.

```sql
CREATE TABLE produits (
    id SERIAL,
    nom VARCHAR(100),
    prix NUMERIC(10, 2),
    categorie_id INTEGER,

    -- Contraintes nommées
    CONSTRAINT pk_produits PRIMARY KEY (id),
    CONSTRAINT uq_produits_nom UNIQUE (nom),
    CONSTRAINT chk_produits_prix CHECK (prix >= 0),
    CONSTRAINT fk_produits_categorie FOREIGN KEY (categorie_id)
        REFERENCES categories(id) ON DELETE SET NULL
);
```

**Convention de nommage suggérée** :

- `pk_` : Primary Key  
- `fk_` : Foreign Key  
- `uq_` : Unique  
- `chk_` : Check
- Suivi du nom de la table et des colonnes concernées

### 4.9 Ajouter et Supprimer des Contraintes

#### Ajouter une Contrainte à une Table Existante

```sql
-- Ajouter NOT NULL
ALTER TABLE clients  
ALTER COLUMN nom SET NOT NULL;  

-- Ajouter une contrainte UNIQUE
ALTER TABLE clients  
ADD CONSTRAINT uq_clients_email UNIQUE (email);  

-- Ajouter une contrainte CHECK
ALTER TABLE produits  
ADD CONSTRAINT chk_produits_prix CHECK (prix >= 0);  

-- Ajouter une clé étrangère
ALTER TABLE produits  
ADD CONSTRAINT fk_produits_categorie  
    FOREIGN KEY (categorie_id) REFERENCES categories(id);
```

#### Supprimer une Contrainte

```sql
-- Supprimer une contrainte nommée
ALTER TABLE clients  
DROP CONSTRAINT uq_clients_email;  

-- Supprimer NOT NULL
ALTER TABLE clients  
ALTER COLUMN nom DROP NOT NULL;  
```

### 4.10 Tableau Récapitulatif des Contraintes

| Contrainte | Rôle | Permet NULL ? | Crée un Index ? |
|------------|------|---------------|-----------------|
| `PRIMARY KEY` | Identifiant unique de la ligne | Non | Oui (automatique) |
| `FOREIGN KEY` | Référence vers une autre table | Oui | Non (à créer manuellement) |
| `UNIQUE` | Valeurs distinctes | Oui | Oui (automatique) |
| `NOT NULL` | Valeur obligatoire | Non | Non |
| `CHECK` | Validation personnalisée | Oui | Non |
| `DEFAULT` | Valeur par défaut | Oui | Non |

> **Conseil Performance** : Créez toujours un index sur les colonnes de clé étrangère qui seront utilisées dans des jointures. PostgreSQL ne le fait pas automatiquement.

```sql
-- Après avoir créé une clé étrangère
CREATE INDEX idx_produits_categorie_id ON produits(categorie_id);
```

## 5. Bonnes Pratiques

### 5.1 Conventions de Nommage

- Utilisez des noms en **minuscules** avec des **underscores** (`_`) comme séparateurs
- Nommez les tables au **pluriel** (`clients`, `produits`, `commandes`)
- Nommez les colonnes au **singulier** (`nom`, `prix`, `date_creation`)
- Soyez **cohérent** dans tout votre schéma

```sql
-- Bon
CREATE TABLE commandes (
    id SERIAL PRIMARY KEY,
    client_id INTEGER REFERENCES clients(id),
    date_commande TIMESTAMP,
    montant_total NUMERIC(10, 2)
);

-- À éviter
CREATE TABLE Commande (
    ID SERIAL PRIMARY KEY,
    ClientID INTEGER,
    dateCommande TIMESTAMP,
    MontantTotal NUMERIC(10, 2)
);
```

### 5.2 Toujours Utiliser des Transactions

Pour les opérations critiques, encadrez vos commandes dans une transaction :

```sql
BEGIN;

UPDATE comptes SET solde = solde - 100 WHERE id = 1;  
UPDATE comptes SET solde = solde + 100 WHERE id = 2;  

-- Si tout est OK
COMMIT;

-- Ou en cas de problème
-- ROLLBACK;
```

### 5.3 Tester Avant d'Exécuter

Avant un `UPDATE` ou `DELETE`, vérifiez les lignes concernées avec un `SELECT` :

```sql
-- D'abord, vérifier ce qui sera affecté
SELECT * FROM clients  
WHERE date_inscription < '2020-01-01' AND actif = FALSE;  

-- Puis, si c'est correct, exécuter
DELETE FROM clients  
WHERE date_inscription < '2020-01-01' AND actif = FALSE;  
```

### 5.4 Commenter Votre Schéma

PostgreSQL permet d'ajouter des commentaires aux objets :

```sql
COMMENT ON TABLE clients IS 'Table des clients de la boutique en ligne';  
COMMENT ON COLUMN clients.email IS 'Adresse email unique, utilisée pour la connexion';  
```
