# 21.3.1. I/O Asynchrone : Jusqu'à 3× Plus Rapide

## Introduction

L'une des améliorations les plus significatives de PostgreSQL 18 est l'introduction d'un **sous-système d'entrées/sorties asynchrones** (Asynchronous I/O ou AIO). Cette évolution architecturale majeure permet d'améliorer considérablement les performances de lecture, avec des gains pouvant atteindre **2 à 3 fois** les performances des versions précédentes dans certains scénarios.

Cette section vous explique, de manière accessible, ce qu'est l'I/O asynchrone, pourquoi c'est important, et comment PostgreSQL 18 l'implémente.

---

## Comprendre le Problème : L'I/O Synchrone Traditionnel

### Une Analogie Simple

Imaginez que vous êtes dans une bibliothèque et que vous devez consulter 100 livres rangés dans différentes étagères. Avec l'approche **synchrone** (celle utilisée par PostgreSQL avant la version 18), vous procéderiez ainsi :

1. Vous demandez le livre n°1 au bibliothécaire  
2. Vous **attendez** qu'il aille le chercher  
3. Vous recevez le livre et le lisez  
4. Vous demandez le livre n°2  
5. Vous **attendez** à nouveau...  
6. Et ainsi de suite, un par un

Le problème ? Vous passez beaucoup de temps à **attendre** plutôt qu'à lire.

### Le Modèle Synchrone dans PostgreSQL

Dans les versions précédentes de PostgreSQL, le fonctionnement était similaire :

1. PostgreSQL demande à lire un bloc de données (8 Ko) sur le disque  
2. Le processus **se bloque** et attend que le disque réponde  
3. Une fois les données reçues, PostgreSQL les traite  
4. PostgreSQL demande le bloc suivant  
5. Nouveau blocage en attendant la réponse...

Ce modèle est appelé **I/O synchrone bloquant**. Le processeur reste inactif pendant que le disque travaille, ce qui représente un gaspillage de ressources significatif.

---

## La Solution : L'I/O Asynchrone

### Le Nouveau Paradigme

Avec l'I/O asynchrone, PostgreSQL peut maintenant :

1. **Soumettre plusieurs requêtes de lecture simultanément**  
2. **Continuer à travailler** pendant que les données arrivent  
3. **Traiter les données au fur et à mesure** qu'elles deviennent disponibles

### Reprenons Notre Analogie

Avec l'approche **asynchrone**, vous procédez différemment dans la bibliothèque :

1. Vous donnez au bibliothécaire une liste de 10 livres à récupérer  
2. Pendant qu'il va les chercher, vous commencez à lire le livre que vous avez déjà  
3. Le bibliothécaire vous apporte les livres au fur et à mesure  
4. Vous continuez à lire sans interruption

Résultat : vous passez votre temps à **lire** plutôt qu'à **attendre**.

### Les Bénéfices Concrets

L'I/O asynchrone permet à PostgreSQL de :

- **Anticiper** les besoins en données grâce aux plans d'exécution des requêtes  
- **Paralléliser** les opérations de lecture avec le traitement des données  
- **Optimiser** l'utilisation du matériel (CPU, disques, mémoire)

---

## Les Trois Méthodes d'I/O dans PostgreSQL 18

PostgreSQL 18 introduit un nouveau paramètre de configuration appelé `io_method` qui permet de choisir parmi trois méthodes d'I/O.

### 1. La Méthode `sync` (Synchrone)

```
io_method = 'sync'
```

Cette méthode reproduit le comportement des versions précédentes de PostgreSQL :

- Les lectures sont **synchrones et bloquantes**
- Utilise `posix_fadvise()` pour le read-ahead du noyau Linux
- **Cas d'usage** : Débogage, comparaison de performances, ou si vous rencontrez des problèmes avec les méthodes asynchrones

### 2. La Méthode `worker` (Par Défaut)

```
io_method = 'worker'
```

C'est la **méthode par défaut** dans PostgreSQL 18 :

- Des **processus worker dédiés** gèrent les opérations d'I/O en arrière-plan
- Le processus principal soumet les requêtes de lecture et continue son travail
- Les workers récupèrent les données indépendamment de l'exécution des requêtes
- **Compatible** avec tous les systèmes d'exploitation supportés par PostgreSQL

Le nombre de workers est contrôlé par le paramètre `io_workers` (par défaut : 3).

### 3. La Méthode `io_uring` (Linux Uniquement)

```
io_method = 'io_uring'
```

C'est la méthode la **plus performante**, mais avec des prérequis :

- Utilise l'interface moderne `io_uring` du noyau Linux
- Les requêtes sont soumises **directement au noyau** via un buffer circulaire partagé
- **Prérequis** :
  - Linux avec noyau 5.1 ou supérieur
  - PostgreSQL compilé avec l'option `--with-liburing`
  - La fonctionnalité io_uring doit être activée dans le système

Cette méthode élimine le besoin de processus workers intermédiaires et offre les meilleures performances, particulièrement sur les systèmes de stockage à haute latence (comme dans le cloud).

---

## Paramètres de Configuration

### Les Paramètres Principaux

| Paramètre | Description | Valeur par défaut |
|-----------|-------------|-------------------|
| `io_method` | Méthode d'I/O à utiliser (`sync`, `worker`, `io_uring`) | `worker` |
| `io_workers` | Nombre de processus workers pour l'I/O asynchrone | 3 |
| `effective_io_concurrency` | Nombre de requêtes d'I/O concurrentes pour les opérations normales | 16 |
| `maintenance_io_concurrency` | Nombre de requêtes d'I/O concurrentes pour les opérations de maintenance (VACUUM, etc.) | 16 |
| `io_combine_limit` | Taille maximale des I/O combinées | Variable selon le système |

### Points Importants

- Le paramètre `io_method` nécessite un **redémarrage du serveur** pour être pris en compte
- Les autres paramètres peuvent généralement être modifiés dynamiquement
- Sur les systèmes cloud ou avec des disques SSD/NVMe, des valeurs plus élevées pour `effective_io_concurrency` peuvent améliorer les performances

---

## Quelles Opérations Bénéficient de l'I/O Asynchrone ?

Dans PostgreSQL 18, l'I/O asynchrone est implémenté pour les **opérations de lecture** suivantes :

### Opérations Supportées

1. **Parcours séquentiels** (Sequential Scans) : Lecture complète d'une table  
2. **Parcours par bitmap** (Bitmap Heap Scans) : Lecture de blocs identifiés par un index bitmap  
3. **Opérations de maintenance** : VACUUM, ANALYZE, et autres opérations de maintenance  
4. **Opérations de sauvegarde** : pg_basebackup et similaires

### Limitations Actuelles

Il est important de noter que dans PostgreSQL 18 :

- **Seules les lectures** sont asynchrones  
- **Les écritures restent synchrones**, y compris les écritures WAL (Write-Ahead Log)
- D'autres types d'accès (comme certains types d'index) pourraient bénéficier de l'AIO dans les versions futures

## Gains de Performance : Les Chiffres

### Benchmarks Officiels

Les tests de performance réalisés par la communauté PostgreSQL et différents éditeurs montrent des améliorations significatives :

| Scénario | Amélioration Typique |
|----------|---------------------|
| Parcours séquentiels sur tables volumineuses | 2× à 3× plus rapide |
| Workloads intensifs en lecture (lecture depuis le disque) | Jusqu'à 3× plus rapide |
| Environnements cloud (stockage à haute latence) | Gains les plus importants |

### Conditions pour Maximiser les Gains

Les gains de performance sont maximisés lorsque :

1. **Les données ne tiennent pas en mémoire** : Si tout est en cache (shared_buffers), l'I/O asynchrone n'apporte pas d'avantage  
2. **Le stockage a une latence significative** : Les environnements cloud bénéficient particulièrement de cette amélioration  
3. **Les requêtes effectuent beaucoup de lectures** : Requêtes analytiques, rapports, exports de données  
4. **Le CPU n'est pas saturé** : Le CPU doit pouvoir traiter les données pendant que d'autres arrivent

### Quand les Gains Sont Moindres

L'I/O asynchrone apporte peu ou pas d'amélioration quand :

- Les données sont déjà en cache mémoire
- Les requêtes sont limitées par le CPU (calculs intensifs)
- Le stockage est déjà saturé (limite physique atteinte)
- Les requêtes utilisent principalement des index (peu de lectures séquentielles)

## Observabilité et Monitoring

PostgreSQL 18 fournit de nouveaux outils pour surveiller l'I/O asynchrone.

### La Vue Système `pg_aios`

Cette nouvelle vue permet de voir les opérations d'I/O asynchrone en cours :

```sql
SELECT * FROM pg_aios;
```

Elle affiche les handles de fichiers utilisés pour l'I/O asynchrone et l'état des requêtes en cours.

### La Fonction `pg_get_aios()`

Cette fonction retourne des informations détaillées sur les I/O planifiées :

```sql
SELECT * FROM pg_get_aios();
```

Pour chaque requête d'I/O planifiée, vous obtenez une entrée permettant de comprendre ce qui se passe en coulisses.

### Nouveaux Événements d'Attente

Avec la méthode `worker`, de nouveaux événements d'attente apparaissent dans `pg_stat_activity` :

- `IO/AioIoCompletion` : Le backend attend la fin d'une opération d'I/O asynchrone  
- `IO/DataFileRead` : Un worker I/O effectue une lecture de fichier de données

Ces événements permettent de diagnostiquer où le temps est passé lors de l'exécution des requêtes.

## Comparaison des Méthodes d'I/O

| Caractéristique | `sync` | `worker` | `io_uring` |
|-----------------|--------|----------|------------|
| **Modèle** | Synchrone bloquant | Asynchrone via workers | Asynchrone natif |
| **Compatibilité** | Tous OS | Tous OS | Linux 5.1+ uniquement |
| **Performance** | Baseline | Bonne | Excellente |
| **Overhead système** | Minimal | Modéré (processus workers) | Minimal |
| **Complexité** | Simple | Moyenne | Plus complexe à configurer |
| **Cas d'usage idéal** | Débogage, anciens systèmes | Production générale | Production Linux haute performance |

## Impact sur les Environnements Cloud

L'I/O asynchrone est particulièrement bénéfique dans les environnements cloud pour plusieurs raisons :

### Latence du Stockage

Dans le cloud, le stockage (EBS sur AWS, Persistent Disk sur GCP, etc.) présente souvent une latence plus élevée que le stockage local. L'I/O asynchrone permet de :

- **Masquer cette latence** en soumettant plusieurs requêtes en parallèle  
- **Maintenir le débit** même avec une latence individuelle élevée  
- **Optimiser les coûts** en utilisant mieux les IOPS provisionnées

### Stockage Réseau

Les bases de données dans le cloud utilisent généralement du stockage réseau. L'I/O asynchrone est conçu pour tirer parti de la capacité de ces systèmes à traiter de nombreuses requêtes en parallèle.

---

## Résumé

L'introduction de l'I/O asynchrone dans PostgreSQL 18 représente une **évolution architecturale majeure** qui :

1. **Améliore significativement les performances** : Jusqu'à 3× plus rapide pour les opérations de lecture intensives

2. **S'adapte aux infrastructures modernes** : Particulièrement efficace dans les environnements cloud et sur les systèmes de stockage modernes

3. **Reste compatible** : La méthode `sync` permet de revenir au comportement traditionnel si nécessaire

4. **Offre de la flexibilité** : Trois méthodes au choix selon votre environnement et vos besoins

5. **Améliore l'observabilité** : Nouveaux outils de monitoring pour comprendre le comportement I/O

### Points Clés à Retenir

- **Par défaut**, PostgreSQL 18 utilise la méthode `worker` (asynchrone)  
- **Les lectures** sont asynchrones, **les écritures** restent synchrones
- Les gains sont maximaux pour les **workloads intensifs en lecture** où les données ne tiennent pas en cache
- Sur **Linux moderne**, la méthode `io_uring` offre les meilleures performances
- Cette amélioration est **transparente** : vos applications existantes en bénéficient automatiquement
