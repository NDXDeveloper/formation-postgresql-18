# 21.5. Roadmap de Montée en Compétence

## 21.5.3. Parcours DBA (Database Administrator) PostgreSQL

Ce parcours est conçu pour guider les futurs administrateurs de bases de données (DBA) dans leur apprentissage de PostgreSQL. Le rôle de DBA combine des compétences techniques profondes en SQL, en optimisation de performances, en modélisation de données et en administration système, avec une dimension de conseil et de gouvernance auprès des équipes de développement.

> **Note importante** : Les durées indiquées sont des estimations basées sur un apprentissage régulier et intensif. Le rôle de DBA requiert une expertise approfondie qui se construit avec le temps et l'expérience sur des cas réels.

> **Prérequis** : Ce parcours suppose une bonne maîtrise du SQL (équivalent à la Phase 2 du parcours Développeur) et des bases en administration système Linux.

---

## Qu'est-ce qu'un DBA PostgreSQL ?

Avant de commencer, il est important de comprendre ce qui distingue le rôle de DBA des autres métiers liés aux bases de données.

### Les responsabilités d'un DBA

Le DBA est le gardien de la base de données. Ses responsabilités incluent :

- **Performance** : S'assurer que la base de données répond aux exigences de performance, optimiser les requêtes et l'infrastructure  
- **Disponibilité** : Garantir que la base est accessible quand les applications en ont besoin  
- **Intégrité** : Protéger les données contre la corruption, les pertes et les accès non autorisés  
- **Évolutivité** : Planifier la croissance et adapter l'infrastructure aux besoins futurs  
- **Conseil** : Accompagner les équipes de développement dans la conception des schémas et l'écriture de requêtes efficaces

### DBA vs DevOps/SRE

Bien que les rôles se chevauchent parfois, le DBA se distingue par :

- Une expertise plus profonde en SQL et en optimisation de requêtes
- Une connaissance approfondie des mécanismes internes de PostgreSQL
- Un rôle de conseil auprès des développeurs sur la modélisation et les performances
- Une responsabilité sur la gouvernance et la qualité des données

Le DevOps/SRE se concentre davantage sur l'automatisation, le déploiement et l'infrastructure.

## Phase 1 : Fondations DBA (0-6 mois)

### Objectif de cette phase

Acquérir les bases essentielles de l'administration PostgreSQL : comprendre l'architecture interne, maîtriser la configuration, effectuer les opérations de maintenance courantes et développer une expertise SQL solide. À la fin de cette phase, vous serez capable de gérer une instance PostgreSQL en environnement de développement ou de test.

### Compétences à développer

#### Mois 1-2 : Architecture Interne et Mécanismes Fondamentaux

**Comprendre l'architecture PostgreSQL en profondeur :**

En tant que DBA, vous devez comprendre non seulement comment utiliser PostgreSQL, mais comment il fonctionne en interne.

- **Le processus Postmaster** :
  - Point d'entrée de toutes les connexions
  - Spawne un processus backend par connexion client
  - Gère le cycle de vie des processus d'arrière-plan

- **Processus d'arrière-plan et leur rôle** :  
  - `background writer` : écrit les pages "dirty" du shared buffer vers le disque de manière asynchrone  
  - `checkpointer` : crée des points de contrôle (checkpoints) qui garantissent la cohérence des données sur disque  
  - `walwriter` : écrit les journaux WAL de manière asynchrone  
  - `autovacuum launcher` : coordonne les workers autovacuum pour la maintenance automatique  
  - `stats collector` : collecte les statistiques d'utilisation  
  - `logical replication launcher` : gère les workers de réplication logique

- **Architecture mémoire** :

  Mémoire partagée (Shared Memory) :
  - `shared_buffers` : cache des pages de données, partagé entre tous les processus  
  - `wal_buffers` : buffer pour les écritures WAL
  - Tables de verrous (lock tables)
  - Structures de contrôle des processus

  Mémoire locale (par processus backend) :
  - `work_mem` : mémoire pour les opérations de tri et de hash  
  - `maintenance_work_mem` : mémoire pour les opérations de maintenance (VACUUM, CREATE INDEX)  
  - `temp_buffers` : cache pour les tables temporaires

- **Structure physique des données** :

  Le répertoire de données (PGDATA) contient :
  - `base/` : données des bases (un sous-répertoire par base, identifié par OID)  
  - `global/` : objets partagés entre toutes les bases (rôles, tablespaces)  
  - `pg_wal/` : journaux de transactions (WAL)  
  - `pg_xact/` : statut des transactions (commit/abort)  
  - `pg_stat/` : statistiques persistantes  
  - `pg_stat_tmp/` : statistiques temporaires
  - Fichiers de configuration : `postgresql.conf`, `pg_hba.conf`, `pg_ident.conf`

**Comprendre le Write-Ahead Logging (WAL) :**

Le WAL est le mécanisme qui garantit la durabilité des données (le "D" d'ACID).

- **Principe** : toute modification est d'abord écrite dans le journal WAL avant d'être appliquée aux fichiers de données  
- **Avantages** :
  - Récupération après crash : rejouer les WAL pour retrouver un état cohérent
  - Base de la réplication : les standbys reçoivent et appliquent les WAL
  - Point-In-Time Recovery : restaurer à n'importe quel instant

- **Structure des WAL** :
  - Fichiers de 16 MB par défaut (configurable avec `--wal-segsize` lors de l'initdb)
  - Nommage : timeline + segment (ex: 000000010000000000000001)
  - LSN (Log Sequence Number) : position dans le flux WAL

- **Paramètres clés** :  
  - `wal_level` : minimal, replica, logical  
  - `max_wal_size` : taille maximale avant checkpoint forcé  
  - `min_wal_size` : taille minimale conservée  
  - `checkpoint_timeout` : intervalle maximum entre checkpoints

**Comprendre MVCC (Multiversion Concurrency Control) :**

MVCC est le mécanisme central de gestion de la concurrence dans PostgreSQL.

- **Principe** : chaque transaction voit un "snapshot" cohérent des données  
- **Implémentation PostgreSQL** :
  - Chaque ligne (tuple) possède des champs système : `xmin`, `xmax`, `ctid`
  - `xmin` : ID de la transaction qui a créé le tuple  
  - `xmax` : ID de la transaction qui a supprimé/modifié le tuple (0 si actif)
  - Un `UPDATE` crée une nouvelle version du tuple (l'ancienne reste avec un `xmax`)
  - Un `DELETE` marque le tuple avec un `xmax`

- **Visibilité des tuples** :
  - Un tuple est visible si sa transaction créatrice (`xmin`) est commitée et antérieure au snapshot
  - Et si sa transaction de suppression (`xmax`) n'existe pas ou n'est pas commitée

- **Conséquences** :
  - Les anciennes versions s'accumulent ("dead tuples")
  - VACUUM est nécessaire pour récupérer l'espace
  - Le "bloat" est l'espace perdu à cause des dead tuples non nettoyés

#### Mois 3-4 : Configuration et Tuning de Base

**Maîtrise du fichier postgresql.conf :**

Le DBA doit comprendre chaque paramètre important et savoir l'ajuster selon le contexte.

- **Paramètres de connexion** :
  ```
  listen_addresses = '*'          # Interfaces réseau
  port = 5432                     # Port d'écoute
  max_connections = 200           # Connexions simultanées max
  superuser_reserved_connections = 3  # Connexions réservées au superuser
  ```

- **Paramètres mémoire** :

  Règles de base pour commencer (à affiner selon le workload) :
  ```
  # Pour un serveur dédié PostgreSQL avec 32 GB de RAM
  shared_buffers = 8GB            # ~25% de la RAM
  effective_cache_size = 24GB     # ~75% de la RAM (estimation du cache OS)
  work_mem = 64MB                 # Attention : multiplié par le nombre d'opérations
  maintenance_work_mem = 1GB      # Pour VACUUM, CREATE INDEX, etc.
  ```

  Comprendre `work_mem` :
  - Utilisé par les tris, hash joins, hash aggregates
  - Peut être alloué plusieurs fois par requête (une par opération)
  - Une requête complexe peut utiliser N × work_mem
  - Trop élevé : risque d'OOM avec beaucoup de connexions
  - Trop bas : utilisation du disque (fichiers temporaires)

- **Paramètres WAL et checkpoints** :
  ```
  wal_level = replica             # Nécessaire pour la réplication
  max_wal_size = 4GB              # Taille avant checkpoint forcé
  min_wal_size = 1GB              # Espace WAL minimum conservé
  checkpoint_timeout = 10min      # Intervalle max entre checkpoints
  checkpoint_completion_target = 0.9  # Étaler les écritures sur 90% de l'intervalle
  ```

- **Paramètres autovacuum** :
  ```
  autovacuum = on                 # Toujours activer en production
  autovacuum_max_workers = 3      # Workers simultanés
  autovacuum_naptime = 1min       # Intervalle de vérification
  autovacuum_vacuum_threshold = 50
  autovacuum_vacuum_scale_factor = 0.1  # 10% de modifications déclenche VACUUM
  autovacuum_analyze_threshold = 50
  autovacuum_analyze_scale_factor = 0.05  # 5% de modifications déclenche ANALYZE
  ```

- **Paramètres de logging** :
  ```
  log_destination = 'stderr'
  logging_collector = on
  log_directory = 'pg_log'
  log_filename = 'postgresql-%Y-%m-%d_%H%M%S.log'
  log_rotation_age = 1d
  log_rotation_size = 100MB
  log_min_duration_statement = 1000  # Logger les requêtes > 1 seconde
  log_line_prefix = '%t [%p]: user=%u,db=%d,app=%a,client=%h '
  log_checkpoints = on
  log_connections = on
  log_disconnections = on
  log_lock_waits = on
  log_temp_files = 0              # Logger tous les fichiers temporaires
  ```

- **Paramètres nécessitant un redémarrage** :  
  - `shared_buffers`, `max_connections`, `max_wal_senders`  
  - `shared_preload_libraries`  
  - `wal_level`, `max_worker_processes`

  Vérifier avec :
  ```sql
  SELECT name, setting, context FROM pg_settings WHERE context = 'postmaster';
  ```

**Configuration de la sécurité (pg_hba.conf) :**

- **Comprendre l'ordre d'évaluation** : première règle correspondante s'applique  
- **Bonnes pratiques** :
  - Commencer par les règles les plus spécifiques
  - Utiliser `scram-sha-256` (pas `md5`, déprécié)
  - Toujours utiliser SSL pour les connexions réseau
  - Restreindre les accès par IP autant que possible

**Outils d'aide à la configuration :**

- **PGTune** (pgtune.leopard.in.ua) : génère une configuration de base selon les ressources  
- **Limites de PGTune** : ne connaît pas votre workload, ajustement nécessaire  
- **pgBadger** : analyse des logs pour identifier les besoins

#### Mois 5-6 : Maintenance et Gestion de l'Espace

**VACUUM en profondeur :**

En tant que DBA, vous devez maîtriser parfaitement VACUUM.

- **Types de VACUUM** :

  `VACUUM` (standard) :
  - Marque l'espace des dead tuples comme réutilisable
  - Met à jour la visibility map et la free space map
  - Ne rend PAS l'espace au système de fichiers
  - Ne bloque pas les lectures/écritures (sauf conflit sur les mêmes pages)

  `VACUUM FULL` :
  - Réécrit entièrement la table
  - Récupère l'espace et le rend au système de fichiers
  - Requiert un verrou exclusif (ACCESS EXCLUSIVE)
  - Nécessite de l'espace disque supplémentaire (réécriture complète)
  - À utiliser rarement, en maintenance planifiée

  `VACUUM FREEZE` :
  - Force le "freezing" des anciennes transactions
  - Prévention du transaction ID wraparound
  - Automatiquement déclenché par autovacuum quand nécessaire

- **Visibility Map** :
  - Un bit par page indiquant si tous les tuples sont visibles par toutes les transactions
  - Permet les Index-Only Scans (évite de lire la table si l'index suffit)
  - Mise à jour par VACUUM

- **Free Space Map** :
  - Indique l'espace disponible dans chaque page
  - Utilisée pour savoir où insérer de nouveaux tuples
  - Mise à jour par VACUUM

- **Monitoring de VACUUM** :
  ```sql
  -- État des tables vis-à-vis de VACUUM
  SELECT schemaname, relname,
         n_live_tup, n_dead_tup,
         n_dead_tup::float / NULLIF(n_live_tup, 0) as dead_ratio,
         last_vacuum, last_autovacuum,
         last_analyze, last_autoanalyze
  FROM pg_stat_user_tables
  ORDER BY n_dead_tup DESC;

  -- Progression d'un VACUUM en cours
  SELECT * FROM pg_stat_progress_vacuum;
  ```

- **Nouveautés PostgreSQL 18** :
  - Statistiques de VACUUM et ANALYZE dans `pg_stat_all_tables`
  - Paramètre `autovacuum_vacuum_max_threshold` pour les grandes tables

**ANALYZE et statistiques du planificateur :**

- **Rôle d'ANALYZE** :
  - Collecte des statistiques sur la distribution des données
  - Le planificateur utilise ces statistiques pour choisir le meilleur plan
  - Sans statistiques à jour, plans d'exécution potentiellement catastrophiques

- **Histogrammes et MCV (Most Common Values)** :
  - PostgreSQL stocke les N valeurs les plus fréquentes
  - Et un histogramme pour estimer la distribution des autres valeurs
  - Configurable par colonne avec `ALTER TABLE ... SET STATISTICS`

- **Quand lancer ANALYZE manuellement** :
  - Après un import massif de données
  - Après une modification significative de la distribution
  - Si le planificateur fait de mauvais choix (estimations très différentes du réel)

**Gestion du bloat :**

Le bloat est l'espace perdu à cause des dead tuples non récupérés.

- **Causes du bloat** :
  - VACUUM pas assez fréquent ou efficace
  - Transactions longues empêchant VACUUM de nettoyer
  - `autovacuum_vacuum_scale_factor` trop élevé pour les grandes tables

- **Mesurer le bloat** :
  ```sql
  -- Estimation avec pgstattuple (extension)
  CREATE EXTENSION pgstattuple;
  SELECT * FROM pgstattuple('ma_table');

  -- dead_tuple_percent indique le pourcentage de bloat
  ```

- **Réduire le bloat** :
  - Ajuster les paramètres autovacuum (surtout pour les grandes tables)
  - `VACUUM FULL` (avec verrou exclusif, à planifier)  
  - `pg_repack` (extension) : réorganisation sans verrou exclusif

**Transaction ID Wraparound :**

C'est l'un des problèmes les plus critiques qu'un DBA doit prévenir.

- **Le problème** :
  - Les XIDs (transaction IDs) sont sur 32 bits (~4 milliards)
  - PostgreSQL compare les XIDs de manière circulaire
  - À ~2 milliards de transactions, les "anciennes" deviennent "futures" (problème de visibilité)
  - À l'approche de la limite, PostgreSQL refuse de nouvelles transactions

- **Le mécanisme de protection : freezing** :
  - VACUUM remplace les anciens XIDs par `FrozenTransactionId`
  - Un tuple "frozen" est visible par toutes les transactions futures
  - `vacuum_freeze_min_age` : âge minimum avant freezing  
  - `autovacuum_freeze_max_age` : force un VACUUM anti-wraparound

- **Monitoring** :
  ```sql
  -- Âge des XIDs par base
  SELECT datname, age(datfrozenxid) as xid_age,
         2147483647 - age(datfrozenxid) as remaining
  FROM pg_database
  ORDER BY xid_age DESC;

  -- Âge par table
  SELECT schemaname, relname, age(relfrozenxid) as xid_age
  FROM pg_stat_user_tables
  ORDER BY xid_age DESC
  LIMIT 20;
  ```

- **Alertes recommandées** :
  - Warning à 500 millions (25%)
  - Critical à 1 milliard (50%)
  - Emergency à 1.5 milliard (75%)

## Phase 2 : Optimisation et Modélisation (6-12 mois)

### Objectif de cette phase

Développer une expertise en optimisation de requêtes et en modélisation de données. Le DBA doit pouvoir analyser des requêtes problématiques, proposer des index appropriés et conseiller les développeurs sur la conception des schémas. Cette phase vous permettra de devenir un référent technique sur les performances PostgreSQL.

### Compétences à développer

#### Mois 7-8 : Planificateur et Plans d'Exécution

**Comprendre le planificateur (Query Planner) :**

Le planificateur est le composant qui décide comment exécuter une requête.

- **Étapes du traitement d'une requête** :  
  1. Parsing : analyse syntaxique → arbre de parsing  
  2. Analyse : résolution des noms, vérification des types → arbre de requête  
  3. Réécriture : application des règles (vues, etc.) → arbre réécrit  
  4. Planification : choix du plan optimal → arbre de plan  
  5. Exécution : exécution du plan → résultats

- **Principe du planificateur basé sur le coût** :
  - Génère plusieurs plans possibles
  - Estime le coût de chaque plan (I/O, CPU)
  - Choisit le plan avec le coût estimé le plus bas
  - Utilise les statistiques (pg_stats) pour les estimations

- **Paramètres influençant le planificateur** :
  ```
  random_page_cost = 4.0      # Coût d'une lecture aléatoire (SSD : 1.1-1.5)
  seq_page_cost = 1.0         # Coût d'une lecture séquentielle
  cpu_tuple_cost = 0.01       # Coût CPU par tuple
  cpu_index_tuple_cost = 0.005
  cpu_operator_cost = 0.0025
  effective_cache_size = 24GB # Estimation du cache disponible
  ```

  Note : sur SSD, réduire `random_page_cost` (souvent 1.1-1.5)

**Maîtriser EXPLAIN :**

EXPLAIN est l'outil principal du DBA pour comprendre et optimiser les requêtes.

- **Syntaxe et options** :
  ```sql
  EXPLAIN [options] requête;

  -- Options principales
  EXPLAIN (ANALYZE)        -- Exécute réellement la requête
  EXPLAIN (BUFFERS)        -- Affiche les accès aux buffers
  EXPLAIN (TIMING)         -- Affiche le temps par nœud (activé par défaut avec ANALYZE)
  EXPLAIN (VERBOSE)        -- Plus de détails
  EXPLAIN (FORMAT JSON)    -- Sortie JSON (utile pour les outils)
  EXPLAIN (SETTINGS)       -- Affiche les paramètres non-défaut

  -- Combinaison typique pour le debug
  EXPLAIN (ANALYZE, BUFFERS, VERBOSE) SELECT ...;
  ```

- **Nouveauté PostgreSQL 18** : EXPLAIN ANALYZE affiche automatiquement les buffers

- **Lire un plan d'exécution** :
  ```
  Seq Scan on users  (cost=0.00..15.00 rows=500 width=100) (actual time=0.010..0.500 rows=523 loops=1)
    Filter: (status = 'active')
    Rows Removed by Filter: 77
  ```

  Éléments à comprendre :
  - Type de nœud : `Seq Scan`, `Index Scan`, `Nested Loop`, etc.
  - `cost=startup..total` : coût estimé (unités arbitraires)  
  - `rows` estimé vs `actual rows` : écart = potentiel problème de statistiques  
  - `loops` : nombre d'exécutions du nœud (important pour les jointures)  
  - `Buffers: shared hit=X read=Y` : pages en cache vs lues sur disque

- **Types de nœuds de scan** :  
  - `Seq Scan` : lecture séquentielle de toute la table  
  - `Index Scan` : utilisation d'un index, puis lecture de la table  
  - `Index Only Scan` : données obtenues uniquement depuis l'index  
  - `Bitmap Index Scan` + `Bitmap Heap Scan` : construction d'une bitmap puis lecture  
  - `TID Scan` : accès direct par tuple ID (rare)

- **Types de jointures** :  
  - `Nested Loop` : pour chaque ligne externe, parcourir les lignes internes
    - Efficace quand la table interne est petite ou bien indexée
  - `Hash Join` : construction d'une table de hash, puis probe
    - Efficace pour les grandes jointures d'égalité
  - `Merge Join` : fusion de deux ensembles triés
    - Efficace quand les données sont déjà triées

- **Identifier les problèmes** :
  - Écart important entre `rows` estimé et `actual rows`
  - `Seq Scan` sur une grande table avec peu de lignes retournées  
  - `Nested Loop` avec `loops` très élevé  
  - `Sort` ou `Hash` avec `Buffers: temp read/written` (débordement mémoire)

#### Mois 9-10 : Indexation Avancée

**Stratégie d'indexation :**

Un DBA doit savoir quand et comment créer des index, mais aussi quand NE PAS en créer.

- **Coût des index** :
  - Espace disque supplémentaire
  - Overhead sur les écritures (INSERT, UPDATE, DELETE)
  - Temps de création (peut verrouiller la table)
  - Maintenance (VACUUM doit aussi traiter les index)

- **Règle d'or** : créer un index uniquement s'il sera utilisé et apportera un bénéfice mesurable

**Types d'index et cas d'usage :**

- **B-Tree** (défaut) :
  - Opérateurs : `=`, `<`, `>`, `<=`, `>=`, `BETWEEN`, `IN`, `IS NULL`
  - Pattern `LIKE 'prefix%'` (mais pas `LIKE '%suffix'`)
  - Tri (`ORDER BY`)
  - Cas d'usage : la plupart des situations

- **Hash** :
  - Opérateur : `=` uniquement
  - Plus compact que B-Tree pour l'égalité stricte
  - Rarement utilisé (B-Tree souvent suffisant)

- **GIN (Generalized Inverted Index)** :
  - JSONB : recherche de clés/valeurs
  - Arrays : opérateurs `@>`, `<@`, `&&`
  - Full-text search : `tsvector`
  - Extensions : `pg_trgm` pour la similarité de texte

  ```sql
  -- Index GIN sur JSONB
  CREATE INDEX idx_data_gin ON documents USING gin (data);

  -- Index GIN pour full-text
  CREATE INDEX idx_fts ON articles USING gin (to_tsvector('french', content));

  -- Index trigram pour LIKE '%pattern%'
  CREATE INDEX idx_trgm ON products USING gin (name gin_trgm_ops);
  ```

- **GiST (Generalized Search Tree)** :
  - Données géométriques (PostGIS)
  - Ranges (tsrange, numrange, etc.)
  - Full-text search (alternative à GIN, meilleur pour les mises à jour fréquentes)
  - `ltree` (données hiérarchiques)

- **BRIN (Block Range Index)** :
  - Données naturellement ordonnées (timestamps séquentiels, IDs croissants)
  - Très compact (stocke min/max par groupe de pages)
  - Idéal pour les très grandes tables avec données corrélées physiquement

  ```sql
  -- BRIN sur une colonne temporelle
  CREATE INDEX idx_brin_created ON logs USING brin (created_at);
  ```

- **SP-GiST (Space-Partitioned GiST)** :
  - Structures de données partitionnées dans l'espace
  - Points, rectangles (géométrie)
  - Arbres de recherche (quad-trees, k-d trees)

**Stratégies d'indexation avancées :**

- **Index partiels** :
  ```sql
  -- Indexer uniquement les commandes actives
  CREATE INDEX idx_orders_active ON orders (customer_id)
  WHERE status = 'active';
  ```
  - Réduit la taille de l'index
  - Améliore les performances d'écriture
  - La requête doit inclure la condition du WHERE pour utiliser l'index

- **Index sur expression** :
  ```sql
  -- Recherche insensible à la casse
  CREATE INDEX idx_email_lower ON users ((lower(email)));

  -- Extraction de données JSON
  CREATE INDEX idx_json_field ON documents ((data->>'type'));
  ```
  - La requête doit utiliser exactement la même expression

- **Index multi-colonnes** :
  ```sql
  CREATE INDEX idx_multi ON orders (customer_id, order_date);
  ```
  - Ordre des colonnes crucial : le plus sélectif en premier (souvent)
  - Peut servir pour les requêtes sur le préfixe (customer_id seul)
  - Ne sert pas pour les requêtes sur les colonnes de droite seules

- **Index couvrants (INCLUDE)** :
  ```sql
  CREATE INDEX idx_covering ON orders (customer_id) INCLUDE (total, status);
  ```
  - Les colonnes INCLUDE sont dans l'index mais pas dans l'arbre de recherche
  - Permet des Index Only Scans pour les requêtes incluant ces colonnes
  - Évite de lire la table (heap)

- **Nouveauté PostgreSQL 18 : Skip Scan** :
  - Optimisation pour les index multi-colonnes
  - Permet d'utiliser un index `(a, b)` pour une requête sur `b` seul
  - Efficace quand la première colonne a peu de valeurs distinctes

**Identifier les index manquants et inutilisés :**

```sql
-- Index inutilisés (candidats à la suppression)
SELECT schemaname, relname, indexrelname, idx_scan  
FROM pg_stat_user_indexes  
WHERE idx_scan = 0  
AND indexrelname NOT LIKE '%_pkey';  

-- Tables avec beaucoup de Seq Scans (candidats à l'indexation)
SELECT schemaname, relname, seq_scan, idx_scan,
       seq_scan::float / NULLIF(seq_scan + idx_scan, 0) as seq_ratio
FROM pg_stat_user_tables  
WHERE seq_scan > 1000  
ORDER BY seq_scan DESC;  
```

**Extension HypoPG pour les index hypothétiques :**

```sql
-- Créer un index hypothétique (pas réellement créé)
SELECT * FROM hypopg_create_index('CREATE INDEX ON orders (customer_id)');

-- Tester si le planificateur l'utiliserait
EXPLAIN SELECT * FROM orders WHERE customer_id = 123;

-- Supprimer les index hypothétiques
SELECT hypopg_reset();
```

#### Mois 11-12 : Modélisation et Conseil aux Développeurs

**Normalisation vs Dénormalisation :**

- **Normalisation** (3NF/BCNF) :
  - Élimine la redondance
  - Garantit la cohérence
  - Facilite les mises à jour
  - Mais : peut nécessiter plus de jointures

- **Dénormalisation stratégique** :
  - Dupliquer certaines données pour éviter les jointures
  - Utiliser des colonnes calculées ou matérialisées
  - Stocker des agrégats pré-calculés
  - Trade-off : performance en lecture vs complexité des écritures

- **Conseil DBA** : commencer normalisé, dénormaliser seulement avec des preuves de besoin (benchmarks)

**Modélisation JSONB :**

Quand utiliser JSONB vs colonnes relationnelles :

- **JSONB approprié** :
  - Données à structure variable
  - Attributs optionnels nombreux
  - Données importées d'APIs externes
  - Prototypage rapide

- **Colonnes relationnelles préférables** :
  - Données interrogées fréquemment dans les WHERE/JOIN
  - Contraintes d'intégrité nécessaires
  - Performances critiques
  - Structure stable et connue

**Partitionnement :**

Le DBA doit conseiller sur les stratégies de partitionnement.

- **Quand partitionner** :
  - Tables > 100 GB (ordre de grandeur)
  - Requêtes filtrant presque toujours sur la clé de partitionnement
  - Besoin de supprimer rapidement des données anciennes (DROP PARTITION)
  - Maintenance plus efficace (VACUUM par partition)

- **Stratégies** :  
  - `RANGE` : dates, timestamps, IDs séquentiels  
  - `LIST` : régions, statuts, types discrets  
  - `HASH` : distribution uniforme quand pas de clé naturelle

- **Partition Pruning** : PostgreSQL ignore automatiquement les partitions non pertinentes

- **Considérations** :
  - La clé de partitionnement doit être dans la clause WHERE pour bénéficier du pruning
  - Les jointures entre partitions peuvent être coûteuses
  - Gérer le nombre de partitions (pas des milliers)

**Vues matérialisées :**

- **Cas d'usage** :
  - Agrégations coûteuses utilisées fréquemment
  - Données rafraîchies périodiquement (pas temps réel)
  - Rapports et dashboards

- **Rafraîchissement** :
  ```sql
  -- Bloquant (verrou exclusif pendant le refresh)
  REFRESH MATERIALIZED VIEW mv_stats;

  -- Non bloquant (nécessite un index UNIQUE)
  REFRESH MATERIALIZED VIEW CONCURRENTLY mv_stats;
  ```

- **Indexation** : les vues matérialisées peuvent et doivent être indexées

**Colonnes générées :**

- **Colonnes stockées (STORED)** :
  ```sql
  CREATE TABLE products (
      price numeric,
      quantity integer,
      total numeric GENERATED ALWAYS AS (price * quantity) STORED
  );
  ```
  - Calculée à l'insertion/mise à jour
  - Stockée physiquement
  - Peut être indexée

- **Nouveauté PostgreSQL 18 : Colonnes virtuelles (VIRTUAL)** :
  ```sql
  CREATE TABLE products (
      price numeric,
      quantity integer,
      total numeric GENERATED ALWAYS AS (price * quantity) VIRTUAL
  );
  ```
  - Calculée à la lecture
  - Pas de stockage supplémentaire
  - Idéal pour les calculs simples

**Rôle de conseil du DBA :**

Le DBA n'est pas seulement un opérateur technique, il est aussi un conseiller :

- **Revue de schéma** : valider les choix de modélisation avant l'implémentation  
- **Revue de requêtes** : identifier les problèmes de performance potentiels  
- **Documentation** : maintenir un dictionnaire de données  
- **Formation** : partager les bonnes pratiques avec les équipes de développement  
- **Standards** : définir des conventions de nommage, types recommandés, etc.

### Indicateurs de progression (Phase 2)

À la fin de cette phase, vous devriez être capable de :

## Phase 3 : Expertise et Leadership (12-24 mois)

### Objectif de cette phase

Devenir un expert reconnu capable de gérer des environnements PostgreSQL critiques, de résoudre des problèmes complexes et de définir l'architecture et la gouvernance des données. Cette phase vous prépare à un rôle de DBA senior ou de référent technique.

### Compétences à développer

#### Mois 13-16 : Réplication et Haute Disponibilité

**Réplication physique en profondeur :**

- **Configuration avancée** :
  ```
  # Primary
  wal_level = replica
  max_wal_senders = 10
  max_replication_slots = 10
  synchronous_standby_names = 'standby1'  # Pour réplication synchrone

  # Standby
  primary_conninfo = 'host=primary port=5432 user=replication'
  hot_standby = on
  hot_standby_feedback = on  # Évite les conflits de requêtes
  ```

- **Réplication synchrone** :
  - Garantit la durabilité sur au moins un standby
  - Impact sur la latence des écritures
  - Modes : `remote_write`, `remote_apply`, `on`
  - Quorum-based : `ANY 2 (standby1, standby2, standby3)`

- **Gestion des conflits** :
  - Les requêtes longues sur un standby peuvent entrer en conflit avec le replay WAL
  - `max_standby_streaming_delay` : délai max avant annulation  
  - `hot_standby_feedback` : informe le primary des requêtes en cours

- **Slots de réplication** :
  - Garantissent la rétention des WAL nécessaires
  - Attention : peuvent causer une accumulation infinie si le standby est déconnecté
  - `max_slot_wal_keep_size` (PG 13+) : limite l'accumulation

**Réplication logique avancée :**

- **Cas d'usage DBA** :
  - Migration de version majeure avec temps d'arrêt minimal
  - Consolidation de données de plusieurs sources
  - Réplication sélective (certaines tables/colonnes)
  - Transformation des données pendant la réplication

- **Limitations à connaître** :
  - Ne réplique pas les DDL (CREATE TABLE, ALTER, etc.)
  - Ne réplique pas les séquences (besoin de synchronisation manuelle)
  - Les TRUNCATE nécessitent une configuration spécifique
  - Les Large Objects ne sont pas répliqués

- **Initial data sync** :
  - Par défaut, copie initiale des données existantes
  - Peut être désactivé : `WITH (copy_data = false)`

**Patroni et haute disponibilité :**

- **Architecture Patroni** :
  - Utilise un système de consensus distribué (etcd, Consul, ZooKeeper)
  - Leader election automatique
  - Failover automatique avec politiques configurables
  - API REST pour le management

- **Configuration clé** :
  ```yaml
  scope: postgres-cluster
  name: node1

  bootstrap:
    dcs:
      ttl: 30
      loop_wait: 10
      retry_timeout: 10
      maximum_lag_on_failover: 1048576
      synchronous_mode: true

    postgresql:
      parameters:
        max_connections: 200
        shared_buffers: 8GB
  ```

- **Opérations Patroni** :
  ```bash
  # État du cluster
  patronictl list

  # Switchover planifié
  patronictl switchover --master node1 --candidate node2 --scheduled "2025-11-20T22:00:00"

  # Failover d'urgence
  patronictl failover

  # Réinitialiser un nœud
  patronictl reinit postgres-cluster node3
  ```

#### Mois 17-20 : Diagnostic Avancé et Troubleshooting

**Diagnostic des problèmes de verrous :**

```sql
-- Identifier les sessions bloquantes
SELECT
    blocked.pid AS blocked_pid,
    blocked.usename AS blocked_user,
    blocked.query AS blocked_query,
    blocking.pid AS blocking_pid,
    blocking.usename AS blocking_user,
    blocking.query AS blocking_query,
    blocking.state AS blocking_state
FROM pg_stat_activity blocked  
JOIN pg_locks blocked_locks ON blocked.pid = blocked_locks.pid  
JOIN pg_locks blocking_locks ON blocked_locks.locktype = blocking_locks.locktype  
    AND blocked_locks.database IS NOT DISTINCT FROM blocking_locks.database
    AND blocked_locks.relation IS NOT DISTINCT FROM blocking_locks.relation
    AND blocked_locks.page IS NOT DISTINCT FROM blocking_locks.page
    AND blocked_locks.tuple IS NOT DISTINCT FROM blocking_locks.tuple
    AND blocked_locks.virtualxid IS NOT DISTINCT FROM blocking_locks.virtualxid
    AND blocked_locks.transactionid IS NOT DISTINCT FROM blocking_locks.transactionid
    AND blocked_locks.classid IS NOT DISTINCT FROM blocking_locks.classid
    AND blocked_locks.objid IS NOT DISTINCT FROM blocking_locks.objid
    AND blocked_locks.objsubid IS NOT DISTINCT FROM blocking_locks.objsubid
JOIN pg_stat_activity blocking ON blocking_locks.pid = blocking.pid  
WHERE NOT blocked_locks.granted  
AND blocked.pid != blocking.pid;  

-- Version simplifiée avec pg_blocking_pids (PG 9.6+)
SELECT pid, usename, query, pg_blocking_pids(pid) AS blocking_pids  
FROM pg_stat_activity  
WHERE cardinality(pg_blocking_pids(pid)) > 0;  
```

**Diagnostic des problèmes de mémoire :**

- **Shared buffers** :
  ```sql
  -- Utilisation des shared buffers par table
  SELECT c.relname,
         pg_size_pretty(count(*) * 8192) as buffered_size,
         round(100.0 * count(*) / (SELECT setting::int FROM pg_settings WHERE name='shared_buffers'), 2) as percent
  FROM pg_buffercache b
  JOIN pg_class c ON b.relfilenode = pg_relation_filenode(c.oid)
  GROUP BY c.relname
  ORDER BY count(*) DESC
  LIMIT 20;
  ```

- **Fichiers temporaires** :
  ```sql
  -- Sessions utilisant des fichiers temporaires
  SELECT pid, usename, query, temp_files, pg_size_pretty(temp_bytes) as temp_size
  FROM pg_stat_activity a
  JOIN pg_stat_database d ON a.datid = d.datid
  WHERE temp_files > 0;
  ```

- **Signes de mémoire insuffisante** :  
  - `Sort Method: external merge` dans EXPLAIN  
  - `Buffers: temp read/written` élevé
  - Fichiers dans `pgsql_tmp/`

**Diagnostic des problèmes I/O :**

- **Identifier les requêtes I/O-intensives** :
  ```sql
  -- Avec pg_stat_statements
  SELECT query, calls,
         shared_blks_read + local_blks_read + temp_blks_read as total_reads,
         shared_blks_written + local_blks_written + temp_blks_written as total_writes
  FROM pg_stat_statements
  ORDER BY shared_blks_read DESC
  LIMIT 10;
  ```

- **Nouveauté PostgreSQL 18** : Statistiques I/O par backend

- **Cache hit ratio** :
  ```sql
  -- Par base
  SELECT datname,
         round(100.0 * blks_hit / NULLIF(blks_hit + blks_read, 0), 2) as cache_hit_ratio
  FROM pg_stat_database
  WHERE datname NOT LIKE 'template%';

  -- Par table
  SELECT schemaname, relname,
         round(100.0 * heap_blks_hit / NULLIF(heap_blks_hit + heap_blks_read, 0), 2) as cache_hit_ratio
  FROM pg_statio_user_tables
  WHERE heap_blks_read > 0
  ORDER BY heap_blks_read DESC;
  ```

**Corruption de données et recovery :**

- **Détection avec checksums** :
  ```bash
  # Vérifier si les checksums sont activés
  pg_controldata /path/to/data | grep checksum

  # Vérifier l'intégrité (PostgreSQL arrêté)
  pg_checksums --check -D /path/to/data
  ```

- **Nouveauté PostgreSQL 18** : checksums activés par défaut

- **pg_amcheck pour les index** :
  ```bash
  pg_amcheck --all --heapallindexed dbname
  ```

- **Recovery en cas de corruption** :
  - Identifier l'étendue de la corruption
  - Restaurer depuis une sauvegarde si possible
  - En dernier recours : `pg_resetwal` (perte de données possible)
  - Documenter et analyser la cause (hardware défaillant ?)

**Analyse des logs avec pgBadger :**

```bash
# Générer un rapport
pgbadger /var/log/postgresql/postgresql-*.log -o report.html

# Options utiles
pgbadger --begin "2025-11-15 08:00:00" --end "2025-11-15 18:00:00" ...  
pgbadger --top 20  # Top 20 requêtes au lieu de 10  
pgbadger --exclude-query "SELECT 1"  # Exclure les health checks  
```

#### Mois 21-24 : Gouvernance, Architecture et Leadership

**Sécurité avancée :**

- **Row-Level Security (RLS)** :
  ```sql
  -- Activer RLS
  ALTER TABLE documents ENABLE ROW LEVEL SECURITY;

  -- Politique d'isolation par tenant
  CREATE POLICY tenant_isolation ON documents
      USING (tenant_id = current_setting('app.current_tenant')::int);

  -- Politique pour les admins
  CREATE POLICY admin_full_access ON documents
      TO admin_role
      USING (true);
  ```

- **Audit avec pgAudit** :
  ```
  # postgresql.conf
  shared_preload_libraries = 'pgaudit'
  pgaudit.log = 'write, ddl'
  pgaudit.log_relation = on
  ```

- **Chiffrement** :
  - En transit : SSL/TLS obligatoire
  - Au repos : chiffrement du système de fichiers ou TDE
  - Données sensibles : `pgcrypto` pour le chiffrement applicatif

**Planification de capacité :**

- **Métriques à suivre** :
  - Croissance des données (taille des tables, index)
  - Évolution du nombre de transactions
  - Utilisation CPU, mémoire, I/O
  - Nombre de connexions

- **Modélisation** :
  - Projeter la croissance sur 1-3 ans
  - Identifier les goulots d'étranglement potentiels
  - Planifier les upgrades d'infrastructure

- **Tests de charge** :  
  - `pgbench` pour les tests synthétiques
  - Replay de workload réel (avec obfuscation des données)

**Migrations et upgrades :**

- **Stratégies de migration de version** :

  1. **pg_upgrade** (le plus rapide) :
     ```bash
     pg_upgrade --check -b /old/bin -B /new/bin -d /old/data -D /new/data
     pg_upgrade -b /old/bin -B /new/bin -d /old/data -D /new/data
     ```
     - Nouveauté PG 18 : `--swap` pour encore plus rapide
     - Nouveauté PG 18 : préservation des statistiques

  2. **Réplication logique** (temps d'arrêt minimal) :
     - Configurer la réplication vers la nouvelle version
     - Synchroniser
     - Basculer les applications

  3. **Dump/Restore** (le plus sûr, mais le plus lent) :
     - Adapté aux petites bases
     - Permet une réorganisation complète

- **Tests de migration** :
  - Toujours tester sur un environnement de pré-production
  - Vérifier les performances après migration
  - Avoir un plan de rollback

**Documentation et processus :**

- **Ce qu'un DBA senior doit documenter** :
  - Architecture des bases (schémas, relations)
  - Dictionnaire de données
  - Procédures opérationnelles (runbooks)
  - Politiques de sauvegarde et rétention
  - Procédures de disaster recovery
  - Standards de développement SQL

- **Revue et amélioration continue** :
  - Revues régulières des requêtes les plus coûteuses
  - Audits de sécurité
  - Tests de restauration
  - Post-mortems des incidents

**Leadership technique :**

- **Mentorat** : former les DBA juniors et les développeurs  
- **Communication** : traduire les concepts techniques pour les non-techniques  
- **Veille** : suivre les évolutions de PostgreSQL et de l'écosystème  
- **Contribution** : participer à la communauté (bugs, patches, documentation)

## Le DBA dans l'Ère Moderne

### L'évolution du rôle de DBA

Le rôle de DBA évolue avec les technologies cloud et DevOps :

- **DBA traditionnel** : focus sur l'administration système, les sauvegardes, la haute disponibilité  
- **DBA moderne** : plus orienté performance, modélisation, conseil, automatisation

### DBA et Cloud

Avec les services managés (RDS, Cloud SQL), certaines tâches opérationnelles sont déléguées au provider. Le DBA se concentre alors sur :

- Optimisation des requêtes et des schémas
- Configuration avancée (paramètres accessibles)
- Sécurité et conformité
- Planification de capacité et coûts
- Migration et architecture multi-cloud

### DBA et DevOps

La collaboration DBA-DevOps est essentielle :

- Le DBA apporte l'expertise base de données
- Le DevOps apporte l'automatisation et l'infrastructure
- Ensemble, ils construisent des pipelines de déploiement incluant les migrations de schéma
- Database as Code : versionner les schémas, automatiser les migrations

### Compétences complémentaires recommandées

- **Scripting** : Python, Bash pour l'automatisation  
- **Infrastructure as Code** : Terraform, Ansible  
- **Conteneurs** : Docker, Kubernetes (comprendre, pas nécessairement maîtriser)  
- **Monitoring** : Prometheus, Grafana  
- **CI/CD** : intégrer les migrations de schéma dans les pipelines

## Conseils pour Devenir un Excellent DBA

### Cultiver la curiosité

Cherchez à comprendre le "pourquoi" derrière chaque comportement de PostgreSQL. Lisez le code source si nécessaire.

### Documenter systématiquement

Chaque incident résolu, chaque optimisation réussie mérite une documentation. Vous construisez ainsi une base de connaissances précieuse.

### Automatiser avec discernement

Automatisez les tâches répétitives, mais comprenez ce que font vos scripts. En cas de problème, vous devrez intervenir manuellement.

### Communiquer efficacement

Un DBA doit pouvoir expliquer des concepts techniques à des non-techniques (managers, développeurs débutants). Cette compétence est aussi importante que l'expertise technique.

### Rester humble

Les bases de données sont complexes. Même les experts font des erreurs. L'important est d'apprendre de chaque incident.

### Participer à la communauté

La communauté PostgreSQL est accueillante et riche en connaissances. Participez aux forums, lisez les blogs, assistez aux conférences. Vous apprendrez et vous contribuerez.
