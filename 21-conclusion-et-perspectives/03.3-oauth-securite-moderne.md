# 21.3.3. OAuth et Sécurité Moderne

## Introduction

PostgreSQL 18 marque un tournant majeur dans la sécurité des bases de données en introduisant le support natif de **OAuth 2.0** et en renforçant l'ensemble de son architecture de sécurité. Ces améliorations permettent à PostgreSQL de s'intégrer aux systèmes d'authentification modernes utilisés par les entreprises.

Cette section présente les nouveautés en matière de sécurité de PostgreSQL 18 de manière accessible, même si vous n'êtes pas expert en sécurité informatique.

---

## Les Concepts Fondamentaux de la Sécurité

Avant d'explorer les nouveautés, clarifions quelques concepts essentiels.

### Authentification vs Autorisation

Ces deux termes sont souvent confondus, mais désignent des processus distincts :

| Concept | Question Posée | Exemple |
|---------|----------------|---------|
| **Authentification** | "Qui êtes-vous ?" | Vérifier l'identité avec un mot de passe |
| **Autorisation** | "Qu'avez-vous le droit de faire ?" | Vérifier les permissions sur une table |

PostgreSQL gère ces deux aspects :
- **Authentification** : Configurée dans `pg_hba.conf`  
- **Autorisation** : Gérée par les rôles et les privilèges (`GRANT`/`REVOKE`)

### Les Méthodes d'Authentification Traditionnelles

Historiquement, PostgreSQL proposait plusieurs méthodes :

| Méthode | Description | Sécurité |
|---------|-------------|----------|
| `trust` | Aucune vérification | ❌ Aucune |
| `password` | Mot de passe en clair | ❌ Très faible |
| `md5` | Mot de passe haché en MD5 | ⚠️ Obsolète |
| `scram-sha-256` | Protocole challenge-response moderne | ✅ Forte |
| `cert` | Certificat SSL client | ✅ Très forte |
| `ldap` | Annuaire d'entreprise | ✅ Bonne |
| `oauth` | **Nouveau PG 18** — Tokens OAuth 2.0 | ✅ Moderne |

# Partie 1 : OAuth 2.0 — L'Authentification Moderne

## Qu'est-ce que OAuth 2.0 ?

### Une Analogie Simple

Imaginez que vous souhaitez entrer dans un immeuble de bureaux sécurisé :

**Méthode traditionnelle (mot de passe) :**
1. Vous donnez votre nom et un code secret au gardien  
2. Le gardien vérifie dans sa liste  
3. Si le code correspond, vous entrez

**Méthode OAuth 2.0 :**
1. Vous vous êtes déjà identifié auprès du service de sécurité central de l'entreprise  
2. Ce service vous a donné un **badge temporaire** (le token)  
3. Vous présentez ce badge au gardien de l'immeuble  
4. Le gardien vérifie que le badge est valide auprès du service central  
5. Si c'est bon, vous entrez

### Les Avantages de OAuth

| Avantage | Explication |
|----------|-------------|
| **Centralisation** | Un seul système gère toutes les identités |
| **Single Sign-On (SSO)** | Une seule connexion pour tous les services |
| **Tokens temporaires** | Plus sûrs que des mots de passe permanents |
| **Révocation facile** | Un token compromis peut être annulé instantanément |
| **Pas de mots de passe partagés** | La base de données ne stocke jamais votre mot de passe |

### Le Vocabulaire OAuth

Comprendre OAuth nécessite de connaître quelques termes :

| Terme | Description | Exemple |
|-------|-------------|---------|
| **Resource Owner** | L'utilisateur qui possède les droits | Vous, le développeur |
| **Client** | L'application qui veut accéder | psql, votre application |
| **Authorization Server** | Le serveur qui délivre les tokens | Google, Azure AD, Keycloak |
| **Resource Server** | Le service protégé | PostgreSQL (votre base de données) |
| **Identity Provider (IdP)** | Le fournisseur d'identité | Google, Microsoft, Okta |
| **Bearer Token** | Le "badge" d'accès temporaire | Une chaîne de caractères opaque |
| **Issuer** | L'émetteur du token | URL du fournisseur OAuth |

---

## OAuth 2.0 dans PostgreSQL 18

### Comment Ça Fonctionne

PostgreSQL 18 agit comme un **Resource Server** : il ne délivre pas de tokens, mais il peut les **valider**.

Le flux d'authentification se déroule ainsi :

```
┌─────────────┐     1. Demande de token        ┌──────────────────┐
│   Client    │ ─────────────────────────────▶ │  Authorization   │
│   (psql)    │                                │     Server       │
│             │ ◀───────────────────────────── │  (Google, etc.)  │
└─────────────┘     2. Reçoit le token         └──────────────────┘
       │
       │ 3. Connexion avec le token
       ▼
┌─────────────┐     4. Valide le token         ┌──────────────────┐
│  PostgreSQL │ ─────────────────────────────▶ │    Validator     │
│   (serveur) │                                │     Module       │
│             │ ◀───────────────────────────── │                  │
└─────────────┘     5. Token OK/KO             └──────────────────┘
       │
       │ 6. Connexion acceptée ou refusée
       ▼
   [Accès à la base de données]
```

### Les Deux Modes de Connexion

PostgreSQL 18 supporte deux façons de s'authentifier avec OAuth :

**Mode 1 : Token pré-obtenu**
- L'application obtient le token en dehors de PostgreSQL
- Elle envoie directement le token lors de la connexion
- Idéal pour les applications serveur et les scripts automatisés

**Mode 2 : Device Authorization Flow**
- Le client (psql) initie le flux OAuth
- Un navigateur s'ouvre pour l'authentification
- L'utilisateur se connecte sur le site du fournisseur
- Le client récupère le token automatiquement
- Idéal pour les outils interactifs comme psql

### Configuration Côté Serveur

#### Prérequis

Pour utiliser OAuth, PostgreSQL doit être compilé avec :
- Support OpenSSL (`--with-openssl`)
- Support libcurl (`--with-libcurl`)

#### Étape 1 : Configurer postgresql.conf

```ini
# Activer les modules de validation OAuth
oauth_validator_libraries = 'mon_validateur'
```

Le paramètre `oauth_validator_libraries` liste les bibliothèques qui savent valider les tokens de votre fournisseur OAuth.

#### Étape 2 : Configurer pg_hba.conf

Le fichier `pg_hba.conf` contrôle qui peut se connecter et comment :

```
# TYPE   DATABASE   USER        ADDRESS         METHOD    OPTIONS
host     myapp      all         0.0.0.0/0       oauth     issuer="https://accounts.google.com" scope="openid profile"
```

Les paramètres OAuth dans pg_hba.conf :

| Paramètre | Description | Obligatoire |
|-----------|-------------|-------------|
| `issuer` | URL du fournisseur OAuth | Oui |
| `scope` | Scopes OAuth requis | Oui |
| `validator` | Module de validation à utiliser | Si plusieurs modules |
| `map` | Mapping des identités (pg_ident.conf) | Non |

#### Étape 3 : Mapping des Identités (Optionnel)

Le fichier `pg_ident.conf` permet de faire correspondre les identités OAuth aux rôles PostgreSQL :

```
# MAPNAME       SYSTEM-USERNAME              PG-USERNAME
oauth_map       alice@example.com            alice  
oauth_map       bob@example.com              bob  
oauth_map       /^(.*)@example\.com$         \1  
```

La dernière ligne utilise une expression régulière pour mapper automatiquement tous les utilisateurs du domaine `example.com`.

### Exemple Complet de Configuration

**postgresql.conf :**
```ini
# Module de validation OAuth
oauth_validator_libraries = 'oauth_validator_google'

# TLS obligatoire pour OAuth
ssl = on  
ssl_cert_file = '/etc/postgresql/server.crt'  
ssl_key_file = '/etc/postgresql/server.key'  
```

**pg_hba.conf :**
```
# Connexions OAuth pour l'application "production"
hostssl    production    all    0.0.0.0/0    oauth    issuer="https://accounts.google.com" scope="openid email" map=google_map

# Fallback SCRAM pour les anciens clients
hostssl    production    all    0.0.0.0/0    scram-sha-256
```

**pg_ident.conf :**
```
# Mapping Google vers rôles PostgreSQL
google_map    alice@mycompany.com     app_admin  
google_map    /^(.*)@mycompany\.com$  app_user  
```

### Cas d'Usage Typiques

#### SSO d'Entreprise

Intégration avec Azure AD, Okta, ou un autre IdP d'entreprise :

```
# pg_hba.conf
hostssl  all  all  10.0.0.0/8  oauth  issuer="https://login.microsoftonline.com/tenant-id/v2.0" scope="openid profile"
```

Avantages :
- Les employés utilisent leurs identifiants d'entreprise
- Révocation automatique quand un employé quitte l'entreprise
- Audit centralisé des accès

#### Applications Cloud-Native

Pour des microservices qui accèdent à PostgreSQL :

```
# pg_hba.conf
hostssl  orders_db  order_service  0.0.0.0/0  oauth  issuer="https://auth.mycompany.com" scope="database:orders:read"
```

Avantages :
- Chaque service a son propre token avec des scopes limités
- Rotation automatique des credentials
- Pas de secrets stockés dans les configurations

# Partie 2 : Dépréciation de MD5 et Migration vers SCRAM

## Pourquoi MD5 Est Obsolète

### Les Faiblesses de MD5

L'algorithme MD5 (Message Digest 5) a été conçu en 1991. Depuis, de nombreuses failles ont été découvertes :

| Problème | Explication |
|----------|-------------|
| **Collisions** | Deux entrées différentes peuvent produire le même hash |
| **Vitesse** | Trop rapide, facilitant les attaques par force brute |
| **Rainbow Tables** | Des tables pré-calculées existent pour casser les hash MD5 |
| **Pas de sel (salt)** | Le hash est toujours le même pour un mot de passe donné |

### Ce Qui Change dans PostgreSQL 18

PostgreSQL 18 **déprécie officiellement** l'authentification MD5 :

- ⚠️ MD5 fonctionne encore, mais génère des avertissements  
- ⚠️ MD5 sera **supprimé** dans une future version majeure  
- ✅ SCRAM-SHA-256 est maintenant la méthode recommandée

## SCRAM-SHA-256 : Le Remplacement

### Comment Fonctionne SCRAM

**SCRAM** (Salted Challenge Response Authentication Mechanism) est défini dans la RFC 7677. Son fonctionnement est plus sophistiqué :

```
Client                                    Serveur
   │                                         │
   │  1. "Je veux me connecter"              │
   ├────────────────────────────────────────▶│
   │                                         │
   │  2. "Voici un défi (salt + iterations)" │
   │◀────────────────────────────────────────┤
   │                                         │
   │  3. Calcul local avec le mot de passe   │
   │     + salt + itérations                 │
   │                                         │
   │  4. "Voici ma preuve"                   │
   ├────────────────────────────────────────▶│
   │                                         │
   │  5. Vérification côté serveur           │
   │                                         │
   │  6. "OK, vous êtes authentifié"         │
   │◀────────────────────────────────────────┤
```

### Avantages de SCRAM sur MD5

| Aspect | MD5 | SCRAM-SHA-256 |
|--------|-----|---------------|
| **Algorithme de hash** | MD5 (obsolète) | SHA-256 (moderne) |
| **Sel (salt)** | Fixe (nom d'utilisateur) | Aléatoire par utilisateur |
| **Itérations** | 1 | Configurable (4096 par défaut) |
| **Protection contre interception** | Partielle | Complète |
| **Attaque par rejeu** | Possible | Impossible |
| **Channel Binding** | Non | Oui (avec TLS) |

### Channel Binding : Protection Supplémentaire

SCRAM-SHA-256 supporte le **channel binding** (SCRAM-SHA-256-PLUS), qui lie l'authentification au certificat TLS du serveur :

- Empêche les attaques "man-in-the-middle"
- Le token d'authentification est lié à la connexion TLS spécifique
- Un attaquant ne peut pas réutiliser les données d'authentification capturées

## Migration de MD5 vers SCRAM

### Étape 1 : Vérifier les Clients

Avant de migrer, assurez-vous que tous vos clients supportent SCRAM :

- **libpq** : Version 10+ requise  
- **psycopg2/psycopg3** : Versions récentes  
- **JDBC** : Driver PostgreSQL 42.2.0+  
- **node-postgres** : Version 7.0+  
- **npgsql** : Version 4.0+

### Étape 2 : Vérifier l'État Actuel des Mots de Passe

```sql
-- Voir quels utilisateurs ont encore des mots de passe MD5
SELECT
    rolname,
    CASE
        WHEN rolpassword LIKE 'md5%' THEN 'MD5 (à migrer)'
        WHEN rolpassword LIKE 'SCRAM-SHA-256$%' THEN 'SCRAM (OK)'
        ELSE 'Autre/Aucun'
    END AS password_type
FROM pg_authid  
WHERE rolcanlogin;  
```

### Étape 3 : Configurer le Chiffrement des Nouveaux Mots de Passe

Dans `postgresql.conf` :

```ini
# Utiliser SCRAM pour tous les nouveaux mots de passe
password_encryption = scram-sha-256
```

Puis recharger la configuration :

```sql
SELECT pg_reload_conf();
```

### Étape 4 : Réinitialiser les Mots de Passe des Utilisateurs

Chaque utilisateur doit définir un nouveau mot de passe pour qu'il soit haché en SCRAM :

```sql
-- En tant qu'administrateur
ALTER USER alice PASSWORD 'nouveau_mot_de_passe';

-- Ou l'utilisateur peut le faire lui-même
\password
```

### Étape 5 : Mettre à Jour pg_hba.conf

Remplacer `md5` par `scram-sha-256` :

```
# Avant (obsolète)
host    all    all    0.0.0.0/0    md5

# Après (recommandé)
host    all    all    0.0.0.0/0    scram-sha-256
```

### Compatibilité Automatique

PostgreSQL offre une fonctionnalité de transition : si vous gardez `md5` dans pg_hba.conf mais qu'un utilisateur a un mot de passe SCRAM, PostgreSQL utilisera automatiquement SCRAM.

Cela permet une migration progressive, utilisateur par utilisateur.

# Partie 3 : TLS 1.3 et Chiffrement des Connexions

## Nouveautés TLS dans PostgreSQL 18

### Le Paramètre ssl_tls13_ciphers

PostgreSQL 18 introduit un nouveau paramètre pour contrôler précisément les suites de chiffrement TLS 1.3 :

```ini
# postgresql.conf

# Activer TLS
ssl = on

# Versions TLS autorisées
ssl_min_protocol_version = 'TLSv1.2'  
ssl_max_protocol_version = 'TLSv1.3'  

# Suites de chiffrement TLS 1.2 (paramètre existant)
ssl_ciphers = 'HIGH:!aNULL:!MD5'

# NOUVEAU PG 18 : Suites de chiffrement TLS 1.3
ssl_tls13_ciphers = 'TLS_AES_256_GCM_SHA384:TLS_AES_128_GCM_SHA256'
```

### Suites de Chiffrement TLS 1.3 Recommandées

| Suite | Algorithme | Sécurité |
|-------|------------|----------|
| `TLS_AES_256_GCM_SHA384` | AES 256 bits + SHA-384 | ✅ Très forte |
| `TLS_AES_128_GCM_SHA256` | AES 128 bits + SHA-256 | ✅ Forte |
| `TLS_CHACHA20_POLY1305_SHA256` | ChaCha20 + Poly1305 | ✅ Forte (optimisé mobile) |

### Vérifier la Configuration TLS

Après connexion, vous pouvez vérifier les paramètres TLS :

```sql
-- Avec psql
\conninfo

-- Résultat (extrait) :
-- SSL Connection     | true
-- SSL Protocol       | TLSv1.3
-- SSL Cipher         | TLS_AES_256_GCM_SHA384
```

Ou via une requête :

```sql
SELECT * FROM pg_stat_ssl WHERE pid = pg_backend_pid();
```

## Mode FIPS dans PostgreSQL 18

### Qu'est-ce que FIPS ?

**FIPS 140-2** (Federal Information Processing Standards) est une norme américaine qui définit les exigences de sécurité pour les modules cryptographiques. Certaines organisations (gouvernement, finance, santé) doivent utiliser des systèmes conformes FIPS.

### Support FIPS dans PostgreSQL 18

PostgreSQL 18 améliore la validation du mode FIPS :

- **Détection automatique** : PostgreSQL détecte si OpenSSL est en mode FIPS  
- **Tests validés** : Toutes les suites de tests PostgreSQL passent en mode FIPS  
- **MD5 bloqué** : En mode FIPS, MD5 est automatiquement désactivé (car non approuvé FIPS)

### Implications Pratiques

Si vous activez le mode FIPS dans OpenSSL :

```bash
# Vérifier si FIPS est activé (côté système)
openssl version -a | grep FIPS
```

Alors dans PostgreSQL :
- ❌ L'authentification MD5 échouera  
- ❌ Les fonctions pgcrypto utilisant MD5 échoueront  
- ✅ SCRAM-SHA-256 fonctionnera normalement  
- ✅ TLS avec suites approuvées FIPS fonctionnera

### Paramètre builtin_crypto_enabled

PostgreSQL 18 ajoute ce paramètre dans pgcrypto :

```sql
-- Désactiver les fonctions crypto non-FIPS
SET pgcrypto.builtin_crypto_enabled = off;
```

Cela garantit que seules les fonctions cryptographiques approuvées FIPS sont utilisables.

# Partie 4 : Autres Améliorations de Sécurité

## SCRAM Passthrough avec postgres_fdw et dblink

### Le Problème Précédent

Avant PostgreSQL 18, lors de connexions à des serveurs distants via `postgres_fdw` ou `dblink`, l'authentification SCRAM ne fonctionnait pas correctement car le serveur intermédiaire ne pouvait pas transmettre les credentials SCRAM.

### La Solution PostgreSQL 18

PostgreSQL 18 supporte le **SCRAM passthrough** :

```sql
-- Création d'un serveur distant avec authentification SCRAM
CREATE SERVER serveur_distant
    FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (host 'remote.example.com', dbname 'production');

-- Mapping d'utilisateur avec transmission SCRAM
CREATE USER MAPPING FOR utilisateur_local
    SERVER serveur_distant
    OPTIONS (user 'utilisateur_distant', password 'mot_de_passe');
```

Le flux SCRAM est maintenant correctement transmis au serveur distant, permettant une authentification sécurisée de bout en bout.

## Data Checksums Activés par Défaut

### Changement Important

Dans PostgreSQL 18, les **checksums de données** sont **activés par défaut** lors de l'initialisation d'un nouveau cluster :

```bash
# PostgreSQL 18 : checksums activés par défaut
initdb --pgdata=/var/lib/postgresql/18/main

# Pour désactiver explicitement (non recommandé)
initdb --pgdata=/var/lib/postgresql/18/main --no-data-checksums
```

### Pourquoi C'est Important

Les checksums détectent la **corruption silencieuse** des données :

- Erreurs matérielles (disques, mémoire)
- Bugs de firmware
- Corruption lors de transferts réseau

Sans checksums, PostgreSQL pourrait lire et retourner des données corrompues sans s'en apercevoir.

### Vérifier l'État des Checksums

```sql
SHOW data_checksums;
-- Résultat : on (si activés)
```

## SHA-2 dans pgcrypto

PostgreSQL 18 améliore l'extension pgcrypto avec le support de **SHA-2 pour le hachage des mots de passe** :

```sql
-- Hacher un mot de passe avec SHA-512 (plus sécurisé que MD5)
SELECT crypt('mon_mot_de_passe', gen_salt('sha512'));
```

Les algorithmes disponibles :
- `bf` : Blowfish (recommandé pour les mots de passe)  
- `sha256` : SHA-256  
- `sha512` : SHA-512  
- `md5` : MD5 (déprécié, éviter)

# Partie 5 : Bonnes Pratiques de Sécurité

## Checklist de Sécurité PostgreSQL 18

### Configuration de Base

```ini
# postgresql.conf

# 1. Forcer les connexions chiffrées
ssl = on  
ssl_min_protocol_version = 'TLSv1.2'  
ssl_prefer_server_ciphers = on  

# 2. Utiliser SCRAM pour les mots de passe
password_encryption = scram-sha-256

# 3. Journalisation des connexions
log_connections = on  
log_disconnections = on  

# 4. Limiter les connexions superutilisateur
superuser_reserved_connections = 3
```

### Configuration pg_hba.conf Sécurisée

```
# TYPE   DATABASE        USER            ADDRESS                 METHOD

# Connexions locales (socket Unix)
local   all             postgres                                peer

# Rejeter les connexions en clair
hostnossl  all          all             0.0.0.0/0               reject

# OAuth pour les applications modernes
hostssl    app_db       app_users       10.0.0.0/8              oauth     issuer="..."

# SCRAM pour les connexions traditionnelles
hostssl    all          all             10.0.0.0/8              scram-sha-256

# Tout le reste est rejeté implicitement
```

### Principes de Sécurité

| Principe | Application |
|----------|-------------|
| **Moindre privilège** | N'accordez que les droits nécessaires |
| **Défense en profondeur** | Combinez plusieurs mécanismes (TLS + Auth + RLS) |
| **Rotation des secrets** | Changez régulièrement les mots de passe/tokens |
| **Audit** | Journalisez les accès et les modifications |
| **Séparation des environnements** | Production ≠ Développement |

## Comparatif des Méthodes d'Authentification

| Méthode | Sécurité | Facilité | Cas d'Usage |
|---------|----------|----------|-------------|
| `trust` | ❌ Aucune | ✅ Très facile | Développement local uniquement |
| `md5` | ⚠️ Faible | ✅ Facile | **Déprécié** — À éviter |
| `scram-sha-256` | ✅ Forte | ✅ Facile | Applications traditionnelles |
| `cert` | ✅ Très forte | ⚠️ Complexe | Services automatisés, IoT |
| `oauth` | ✅ Moderne | ⚠️ Complexe | SSO entreprise, Cloud |
| `ldap` | ✅ Bonne | ⚠️ Moyenne | Intégration Active Directory |

## Conclusion

PostgreSQL 18 représente une avancée significative en matière de sécurité :

1. **OAuth 2.0** permet l'intégration avec les systèmes d'authentification modernes et le SSO d'entreprise

2. **La dépréciation de MD5** encourage la migration vers des méthodes plus sûres comme SCRAM-SHA-256

3. **Le contrôle TLS 1.3** offre une flexibilité accrue pour les environnements à haute sécurité

4. **Les checksums par défaut** protègent contre la corruption silencieuse des données

Ces améliorations positionnent PostgreSQL comme une base de données adaptée aux exigences de sécurité modernes, que ce soit pour les startups cloud-native ou les grandes entreprises soumises à des réglementations strictes.
