## Partie 1 : MVCC (Multiversion Concurrency Control)

Le MVCC est le cœur de PostgreSQL. C'est le mécanisme qui permet à plusieurs transactions de travailler simultanément sur les mêmes données sans se bloquer mutuellement.

### 1.1 Qu'est-ce que le MVCC ?

**MVCC** (Multiversion Concurrency Control) est une technique de gestion de la concurrence qui maintient **plusieurs versions** de chaque ligne de données. Au lieu de verrouiller les lignes pendant la lecture, PostgreSQL permet à chaque transaction de voir une **photo cohérente** (snapshot) de la base de données à un instant donné.

#### Le Principe Fondamental

```
┌────────────────────────────────────────────────────────────────────────┐
│                         MVCC : Le Concept                              │
│                                                                        │
│   Transaction A (lecture)         Transaction B (écriture)             │
│         │                               │                              │
│         │  ┌──────────────┐             │                              │
│         │  │ Version 1    │◄────────────┤ Crée Version 2               │
│         │  │ (ancienne)   │             │                              │
│         │  └──────────────┘             │                              │
│         │                               ▼                              │
│         │                        ┌──────────────┐                      │
│         │                        │ Version 2    │                      │
│         └───────────────────────►│ (nouvelle)   │                      │
│           Voit Version 1         └──────────────┘                      │
│           (snapshot cohérent)                                          │
│                                                                        │
│   ➜ Les lecteurs ne bloquent pas les écrivains                         │
│   ➜ Les écrivains ne bloquent pas les lecteurs                         │
└────────────────────────────────────────────────────────────────────────┘
```

**Avantages du MVCC** :
- Les lectures ne bloquent jamais les écritures
- Les écritures ne bloquent jamais les lectures
- Chaque transaction voit un état cohérent de la base
- Excellente performance en environnement concurrent

### 1.2 Les Identifiants de Transaction (XID)

Chaque transaction dans PostgreSQL reçoit un identifiant unique appelé **XID** (Transaction ID).

```sql
-- Voir l'identifiant de la transaction courante
SELECT txid_current();
```

PostgreSQL utilise ces XIDs pour déterminer quelles versions de lignes sont visibles pour chaque transaction.

#### Structure Interne d'une Ligne

Chaque ligne (tuple) dans PostgreSQL contient des colonnes système invisibles :

| Colonne | Description |
|---------|-------------|
| `xmin` | XID de la transaction qui a créé cette version |
| `xmax` | XID de la transaction qui a supprimé/modifié cette version (0 si active) |
| `ctid` | Identifiant physique de la ligne (page, offset) |

```sql
-- Voir les colonnes système d'une ligne
SELECT xmin, xmax, ctid, * FROM clients LIMIT 5;
```

### 1.3 Cycle de Vie d'une Ligne

#### Insertion

Quand vous insérez une ligne :
- `xmin` = XID de votre transaction  
- `xmax` = 0 (non supprimée)

```sql
BEGIN;  
INSERT INTO clients (nom) VALUES ('Alice');  
-- xmin = 12345 (XID courant), xmax = 0
COMMIT;
```

#### Mise à Jour

Quand vous modifiez une ligne, PostgreSQL ne modifie pas la ligne existante. Il :
1. Marque l'ancienne version comme "supprimée" (`xmax` = XID courant)  
2. Crée une **nouvelle version** avec les nouvelles valeurs

```sql
BEGIN;  
UPDATE clients SET nom = 'Alice Martin' WHERE id = 1;  
-- Ancienne ligne : xmax = 12346 (marquée comme supprimée)
-- Nouvelle ligne : xmin = 12346, xmax = 0
COMMIT;
```

C'est pourquoi les UPDATEs fréquents créent des "lignes mortes" (dead tuples).

#### Suppression

Quand vous supprimez une ligne :
- PostgreSQL marque simplement `xmax` avec le XID courant
- La ligne reste physiquement présente jusqu'au VACUUM

```sql
BEGIN;  
DELETE FROM clients WHERE id = 1;  
-- La ligne est marquée xmax = 12347, mais existe encore physiquement
COMMIT;
```

### 1.4 Visibilité des Lignes

Une ligne est **visible** pour une transaction si :
1. `xmin` est une transaction **validée** (committed) avant le début de notre snapshot  
2. `xmax` est soit 0, soit une transaction **non validée** ou validée **après** notre snapshot

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    Règles de Visibilité                                 │
│                                                                         │
│   Transaction courante : XID = 100                                      │
│                                                                         │
│   Ligne A : xmin=50 (committed), xmax=0        → VISIBLE ✓              │
│   Ligne B : xmin=50 (committed), xmax=80       → NON VISIBLE ✗          │
│   Ligne C : xmin=110 (en cours), xmax=0        → NON VISIBLE ✗          │
│   Ligne D : xmin=50 (committed), xmax=110      → VISIBLE ✓              │
│             (xmax en cours = suppression non validée)                   │
└─────────────────────────────────────────────────────────────────────────┘
```

### 1.5 Transactions et Niveaux d'Isolation

Les niveaux d'isolation définissent quelles données une transaction peut voir.

#### Les Quatre Niveaux ANSI SQL

| Niveau | Dirty Read | Non-Repeatable Read | Phantom Read |
|--------|------------|---------------------|--------------|
| Read Uncommitted | Possible | Possible | Possible |
| Read Committed | Impossible | Possible | Possible |
| Repeatable Read | Impossible | Impossible | Possible* |
| Serializable | Impossible | Impossible | Impossible |

> **Note PostgreSQL** : PostgreSQL n'implémente pas réellement "Read Uncommitted" - il se comporte comme "Read Committed". De plus, "Repeatable Read" dans PostgreSQL prévient également les Phantom Reads.

#### Read Committed (Défaut)

Chaque instruction voit un nouveau snapshot de la base. C'est le niveau par défaut.

```sql
-- Session 1
BEGIN;  
SELECT * FROM comptes WHERE id = 1;  -- solde = 100  

-- Session 2 (pendant ce temps)
UPDATE comptes SET solde = 200 WHERE id = 1;  
COMMIT;  

-- Session 1 (continue)
SELECT * FROM comptes WHERE id = 1;  -- solde = 200 (voit le changement!)  
COMMIT;  
```

#### Repeatable Read

Le snapshot est pris au début de la **première instruction** et reste fixe.

```sql
-- Session 1
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;  
SELECT * FROM comptes WHERE id = 1;  -- solde = 100  

-- Session 2 (pendant ce temps)
UPDATE comptes SET solde = 200 WHERE id = 1;  
COMMIT;  

-- Session 1 (continue)
SELECT * FROM comptes WHERE id = 1;  -- solde = 100 (même résultat!)  
COMMIT;  
```

#### Serializable

Le niveau le plus strict. Garantit que les transactions s'exécutent comme si elles étaient séquentielles.

```sql
BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;
-- PostgreSQL vérifie les conflits et peut annuler la transaction
-- avec l'erreur "could not serialize access"
```

### 1.6 Anomalies Transactionnelles

#### Dirty Read (Lecture Sale)

Lire des données non validées d'une autre transaction. **Impossible dans PostgreSQL**.

#### Non-Repeatable Read (Lecture Non Répétable)

La même requête retourne des résultats différents au sein d'une même transaction.

```sql
-- En Read Committed, ce problème peut survenir
BEGIN;  
SELECT solde FROM comptes WHERE id = 1;  -- 100  
-- Autre transaction modifie et commit
SELECT solde FROM comptes WHERE id = 1;  -- 200 (différent!)  
COMMIT;  
```

#### Phantom Read (Lecture Fantôme)

De nouvelles lignes apparaissent entre deux exécutions de la même requête.

```sql
-- En Read Committed
BEGIN;  
SELECT COUNT(*) FROM clients WHERE ville = 'Paris';  -- 10  
-- Autre transaction insère un nouveau client parisien
SELECT COUNT(*) FROM clients WHERE ville = 'Paris';  -- 11 (fantôme!)  
COMMIT;  
```

### 1.7 Verrouillage (Locks)

Bien que MVCC réduise le besoin de verrous, PostgreSQL utilise encore des verrous pour certaines opérations.

#### Types de Verrous sur les Lignes

| Mode | SELECT | SELECT FOR UPDATE | UPDATE | DELETE |
|------|--------|-------------------|--------|--------|
| FOR KEY SHARE | ✓ | ✓ | ✓* | ✗ |
| FOR SHARE | ✓ | ✓ | ✗ | ✗ |
| FOR NO KEY UPDATE | ✓ | ✗ | ✓* | ✗ |
| FOR UPDATE | ✓ | ✗ | ✗ | ✗ |

```sql
-- Verrouiller des lignes pour mise à jour
SELECT * FROM comptes WHERE id = 1 FOR UPDATE;

-- Verrouiller sans bloquer (retourne immédiatement si verrouillé)
SELECT * FROM comptes WHERE id = 1 FOR UPDATE NOWAIT;

-- Sauter les lignes verrouillées
SELECT * FROM comptes FOR UPDATE SKIP LOCKED;
```

#### Verrous de Table

| Mode | Conflit avec |
|------|--------------|
| ACCESS SHARE | ACCESS EXCLUSIVE |
| ROW SHARE | EXCLUSIVE, ACCESS EXCLUSIVE |
| ROW EXCLUSIVE | SHARE, SHARE ROW EXCLUSIVE, EXCLUSIVE, ACCESS EXCLUSIVE |
| SHARE | ROW EXCLUSIVE, SHARE ROW EXCLUSIVE, EXCLUSIVE, ACCESS EXCLUSIVE |
| ACCESS EXCLUSIVE | Tous |

```sql
-- Verrouiller explicitement une table
LOCK TABLE clients IN EXCLUSIVE MODE;

-- Voir les verrous actifs
SELECT * FROM pg_locks WHERE relation = 'clients'::regclass;
```

#### Deadlocks (Interblocages)

Un deadlock survient quand deux transactions s'attendent mutuellement.

```sql
-- Session 1
BEGIN;  
UPDATE comptes SET solde = solde - 100 WHERE id = 1;  
-- Attend Session 2...

-- Session 2
BEGIN;  
UPDATE comptes SET solde = solde - 50 WHERE id = 2;  
UPDATE comptes SET solde = solde + 50 WHERE id = 1;  -- Bloqué par Session 1  

-- Session 1 (continue)
UPDATE comptes SET solde = solde + 100 WHERE id = 2;  -- DEADLOCK!
```

PostgreSQL détecte automatiquement les deadlocks (via `deadlock_timeout`, défaut 1s) et annule une des transactions.

**Prévention** : Toujours accéder aux ressources dans le **même ordre**.

### 1.8 Advisory Locks (Verrous Applicatifs)

Les advisory locks sont des verrous personnalisés gérés par votre application.

```sql
-- Obtenir un verrou (bloquant)
SELECT pg_advisory_lock(12345);

-- Tenter d'obtenir un verrou (non bloquant)
SELECT pg_try_advisory_lock(12345);  -- Retourne true/false

-- Libérer le verrou
SELECT pg_advisory_unlock(12345);

-- Verrous au niveau session vs transaction
SELECT pg_advisory_xact_lock(12345);  -- Libéré automatiquement au COMMIT
```

**Cas d'usage** : Éviter les traitements en double (jobs cron), mutex applicatifs.

### 1.9 VACUUM : Nettoyer les Versions Obsolètes

Le MVCC crée des "dead tuples" (lignes mortes). **VACUUM** les nettoie.

#### Pourquoi VACUUM est Essentiel

1. **Récupérer l'espace** des lignes mortes  
2. **Mettre à jour les statistiques** pour le planificateur  
3. **Prévenir le Transaction ID Wraparound** (voir ci-dessous)

```sql
-- VACUUM simple (récupère l'espace pour réutilisation)
VACUUM clients;

-- VACUUM avec mise à jour des statistiques
VACUUM ANALYZE clients;

-- VACUUM FULL (réécrit toute la table, récupère l'espace disque)
-- ATTENTION : Verrouille la table en exclusif !
VACUUM FULL clients;

-- Voir les statistiques de VACUUM
SELECT
    relname,
    n_dead_tup,
    n_live_tup,
    last_vacuum,
    last_autovacuum
FROM pg_stat_user_tables;
```

#### Autovacuum

PostgreSQL exécute automatiquement VACUUM via le processus **autovacuum**.

```sql
-- Paramètres principaux (postgresql.conf)
autovacuum = on                          -- Activer l'autovacuum  
autovacuum_vacuum_threshold = 50         -- Lignes mortes minimum  
autovacuum_vacuum_scale_factor = 0.2     -- 20% de la table  
autovacuum_analyze_threshold = 50  
autovacuum_analyze_scale_factor = 0.1  
```

Une table est vacuum quand : `dead_tuples > threshold + scale_factor * n_live_tuples`

#### PostgreSQL 18 : Améliorations Autovacuum

PostgreSQL 18 introduit :
- `autovacuum_vacuum_max_threshold` : Limite haute pour déclencher le vacuum  
- `autovacuum_worker_slots` : Allocation dynamique des workers

### 1.10 Transaction ID Wraparound

Les XIDs sont stockés sur 32 bits (≈4 milliards de valeurs). Quand on approche de la limite, PostgreSQL doit "geler" les anciennes transactions.

```
┌─────────────────────────────────────────────────────────────────────────┐
│                   Transaction ID Wraparound                             │
│                                                                         │
│   XIDs : 0 ─────────────────────────────────────────────► 4 milliards   │
│              │                                        │                 │
│              │  "Passé"        │  "Présent"           │  "Futur"        │
│              │  (visible)      │  (courant)           │  (invisible)    │
│              └────────────────────────────────────────┘                 │
│                                                                         │
│   Sans VACUUM : les vieilles transactions deviennent "futures"          │
│   = Perte de données visibles !                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**VACUUM FREEZE** marque les lignes comme "toujours visibles" (frozen).

```sql
-- Voir l'âge des transactions
SELECT
    datname,
    age(datfrozenxid) AS age_xid,
    2147483647 - age(datfrozenxid) AS transactions_restantes
FROM pg_database  
ORDER BY age_xid DESC;  

-- Forcer le freeze
VACUUM FREEZE clients;
```

> **Alerte Critique** : Si `age(datfrozenxid)` approche de 2 milliards, PostgreSQL se mettra en mode "single-user" pour forcer le VACUUM. C'est une urgence !

## Partie 2 : Réplication

La réplication permet de maintenir des copies synchronisées de vos données sur plusieurs serveurs pour la haute disponibilité et la répartition de charge.

### 2.1 Concepts Fondamentaux

#### Terminologie

| Terme | Description |
|-------|-------------|
| **Primary** (Master) | Serveur principal qui accepte les écritures |
| **Standby** (Replica) | Serveur secondaire en lecture seule |
| **WAL** | Write-Ahead Log, journal des modifications |
| **LSN** | Log Sequence Number, position dans le WAL |
| **Failover** | Promotion d'un standby en primary |
| **Switchover** | Basculement planifié vers un standby |

#### Le WAL (Write-Ahead Log)

Le WAL est le journal de toutes les modifications. Avant d'écrire les données, PostgreSQL écrit d'abord dans le WAL.

```
┌────────────────────────────────────────────────────────────────────────┐
│                        Architecture WAL                                │
│                                                                        │
│   Application                                                          │
│       │                                                                │
│       ▼                                                                │
│   ┌─────────┐    1. Écrire WAL    ┌─────────┐                          │
│   │ Primary │ ──────────────────► │   WAL   │                          │
│   │ Server  │                     │  Files  │                          │
│   └─────────┘    2. Écrire Data   └────┬────┘                          │
│       │         ◄──────────────────    │                               │
│       ▼                                │  3. Répliquer                 │
│   ┌─────────┐                          ▼                               │
│   │  Data   │                     ┌─────────┐                          │
│   │  Files  │                     │ Standby │                          │
│   └─────────┘                     │ Server  │                          │
│                                   └─────────┘                          │
└────────────────────────────────────────────────────────────────────────┘
```

### 2.2 Réplication Physique (Streaming Replication)

La réplication physique copie le WAL byte par byte. Le standby est une copie exacte du primary.

#### Configuration du Primary

```ini
# postgresql.conf sur le Primary
wal_level = replica              # Niveau de journalisation (minimal, replica, logical)  
max_wal_senders = 10            # Nombre maximum de connexions de réplication  
max_replication_slots = 10      # Slots de réplication  
wal_keep_size = 1GB             # WAL à conserver pour les standbys  
```

```ini
# pg_hba.conf - Autoriser les connexions de réplication
host    replication    replicator    192.168.1.0/24    scram-sha-256
```

```sql
-- Créer un utilisateur de réplication
CREATE USER replicator WITH REPLICATION ENCRYPTED PASSWORD 'motdepasse';

-- Créer un slot de réplication (optionnel mais recommandé)
SELECT pg_create_physical_replication_slot('standby1_slot');
```

#### Configuration du Standby

```bash
# Créer une copie de base du primary
pg_basebackup -h primary_host -D /var/lib/postgresql/data -U replicator -P -R
```

L'option `-R` crée automatiquement le fichier de configuration pour le standby.

```ini
# postgresql.auto.conf généré (ou postgresql.conf)
primary_conninfo = 'host=primary_host port=5432 user=replicator password=motdepasse'  
primary_slot_name = 'standby1_slot'  
```

```bash
# Créer le fichier signal pour indiquer le mode standby
touch /var/lib/postgresql/data/standby.signal
```

#### Vérifier la Réplication

```sql
-- Sur le Primary : voir les standbys connectés
SELECT
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    pg_wal_lsn_diff(sent_lsn, replay_lsn) AS lag_bytes
FROM pg_stat_replication;

-- Sur le Standby : voir l'état de réception
SELECT
    status,
    received_lsn,
    latest_end_lsn,
    pg_wal_lsn_diff(latest_end_lsn, received_lsn) AS lag_bytes
FROM pg_stat_wal_receiver;
```

### 2.3 Réplication Synchrone vs Asynchrone

#### Réplication Asynchrone (Défaut)

Le primary n'attend pas la confirmation du standby. Risque de perte de données en cas de crash.

```
Primary                     Standby
   │                           │
   │  COMMIT ──────────────►   │
   │  (n'attend pas)           │
   ▼                           ▼
 Client                    Peut avoir
 notifié                   du retard
```

#### Réplication Synchrone

Le primary attend que le standby confirme avant de valider le COMMIT.

```ini
# postgresql.conf sur le Primary
synchronous_commit = on                           # Attendre la confirmation  
synchronous_standby_names = 'standby1'            # Nom du standby synchrone  
```

```sql
-- Sur le Standby, définir un nom d'application
-- primary_conninfo = '... application_name=standby1'
```

**Niveaux de synchronisation** :

| Paramètre | Garantie | Performance |
|-----------|----------|-------------|
| `off` | Aucune | Très rapide |
| `local` | Écriture locale | Rapide |
| `remote_write` | Standby a reçu | Moyenne |
| `on` (défaut) | Standby a écrit sur disque | Lente |
| `remote_apply` | Standby a appliqué | Très lente |

#### Quorum-Based Synchronous Replication

Avec plusieurs standbys, vous pouvez définir un quorum :

```ini
# Au moins 2 parmi 3 standbys doivent confirmer
synchronous_standby_names = 'ANY 2 (standby1, standby2, standby3)'

# Les standbys dans l'ordre de priorité
synchronous_standby_names = 'FIRST 1 (standby1, standby2)'
```

### 2.4 Réplication Logique

La réplication logique transmet les **changements logiques** (INSERT, UPDATE, DELETE) plutôt que les blocs WAL. Plus flexible mais plus complexe.

#### Avantages de la Réplication Logique

- Réplication sélective (certaines tables uniquement)
- Réplication entre versions PostgreSQL différentes
- Possibilité d'écrire sur le subscriber (réplica)
- Transformation des données pendant la réplication

#### Configuration

**Sur le Publisher (source)** :

```ini
# postgresql.conf
wal_level = logical  
max_replication_slots = 10  
max_wal_senders = 10  
```

```sql
-- Créer une publication
CREATE PUBLICATION ma_publication FOR TABLE clients, commandes;

-- Ou publier toutes les tables
CREATE PUBLICATION toutes_tables FOR ALL TABLES;
```

**Sur le Subscriber (destination)** :

```sql
-- Créer une souscription
CREATE SUBSCRIPTION ma_souscription  
CONNECTION 'host=publisher_host dbname=mabase user=replicator password=xxx'  
PUBLICATION ma_publication;  
```

#### Vérifier la Réplication Logique

```sql
-- Sur le Publisher
SELECT * FROM pg_publication;  
SELECT * FROM pg_publication_tables;  
SELECT * FROM pg_stat_replication;  

-- Sur le Subscriber
SELECT * FROM pg_subscription;  
SELECT * FROM pg_stat_subscription;  
```

### 2.5 Slots de Réplication

Les slots garantissent que le primary conserve les WAL nécessaires aux standbys.

```sql
-- Créer un slot physique
SELECT pg_create_physical_replication_slot('standby1_slot');

-- Créer un slot logique
SELECT pg_create_logical_replication_slot('logical_slot', 'pgoutput');

-- Voir les slots
SELECT
    slot_name,
    slot_type,
    active,
    pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) AS lag_bytes
FROM pg_replication_slots;

-- Supprimer un slot (ATTENTION : libère le WAL retenu !)
SELECT pg_drop_replication_slot('standby1_slot');
```

> **Attention** : Un slot inactif peut accumuler des WAL indéfiniment et remplir le disque !

### 2.6 Failover et Promotion

#### Promotion Manuelle

```bash
# Promouvoir un standby en primary
pg_ctl promote -D /var/lib/postgresql/data

# Ou via SQL (PostgreSQL 12+)
SELECT pg_promote();
```

#### Outils de Haute Disponibilité

**Patroni** : Solution complète avec consensus distribué (etcd, Consul, ZooKeeper)

```yaml
# Exemple de configuration Patroni
scope: mon_cluster  
name: node1  

restapi:
  listen: 0.0.0.0:8008

etcd:
  hosts: etcd1:2379,etcd2:2379,etcd3:2379

postgresql:
  listen: 0.0.0.0:5432
  data_dir: /var/lib/postgresql/data

  parameters:
    max_connections: 100
    shared_buffers: 1GB
```

**Repmgr** : Outil plus simple pour la gestion de réplication

```bash
# Enregistrer le primary
repmgr primary register

# Cloner et enregistrer un standby
repmgr standby clone -h primary_host -U repmgr  
repmgr standby register  

# Failover automatique
repmgr standby switchover --siblings-follow
```

### 2.7 Architectures Haute Disponibilité

#### Architecture Simple (1 Primary + 1 Standby)

```
┌──────────────┐         ┌──────────────┐
│   Primary    │────────►│   Standby    │
│  (R/W)       │   WAL   │  (R/O)       │
└──────────────┘         └──────────────┘
       ▲
       │
   Applications
```

#### Architecture avec Load Balancer

```
                    ┌──────────────┐
                    │  HAProxy /   │
                    │  PgBouncer   │
                    └──────┬───────┘
                           │
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
    ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
    │   Primary    │ │  Standby 1   │ │  Standby 2   │
    │  (R/W)       │ │  (R/O)       │ │  (R/O)       │
    └──────────────┘ └──────────────┘ └──────────────┘
```

Configuration HAProxy pour séparer lectures et écritures :

```haproxy
frontend pg_write
    bind *:5432
    default_backend pg_primary

frontend pg_read
    bind *:5433
    default_backend pg_standbys

backend pg_primary
    option httpchk GET /master
    server primary 192.168.1.10:5432 check port 8008

backend pg_standbys
    balance roundrobin
    option httpchk GET /replica
    server standby1 192.168.1.11:5432 check port 8008
    server standby2 192.168.1.12:5432 check port 8008
```

## Partie 3 : Optimisation Avancée

L'optimisation va au-delà de la simple indexation. Elle englobe la configuration système, l'analyse des requêtes et le tuning fin.

### 3.1 Configuration Mémoire

#### Shared Buffers

Cache partagé pour les pages de données. Règle générale : 25% de la RAM.

```ini
# postgresql.conf
shared_buffers = 4GB    # 25% de 16GB de RAM
```

```sql
-- Vérifier le cache hit ratio
SELECT
    sum(heap_blks_read) AS blocs_lus_disque,
    sum(heap_blks_hit) AS blocs_lus_cache,
    round(sum(heap_blks_hit) / (sum(heap_blks_hit) + sum(heap_blks_read)) * 100, 2) AS cache_hit_ratio
FROM pg_statio_user_tables;
```

> **Objectif** : Cache hit ratio > 99%

#### Work Mem

Mémoire pour les opérations de tri et hash par opération.

```ini
work_mem = 64MB    # Par opération, pas par connexion !
```

> **Attention** : Une requête complexe peut utiliser plusieurs fois `work_mem`. Avec 100 connexions × 5 opérations × 64MB = 32GB !

#### Maintenance Work Mem

Mémoire pour les opérations de maintenance (VACUUM, CREATE INDEX, etc.)

```ini
maintenance_work_mem = 1GB
```

#### Effective Cache Size

Estimation de la mémoire totale disponible pour le cache (PostgreSQL + OS).

```ini
effective_cache_size = 12GB    # ~75% de la RAM
```

### 3.6 Identifier et Résoudre les Problèmes Courants

#### Problème : Estimation de Lignes Incorrecte

```sql
-- Le planificateur estime 1 ligne mais 10000 sont retournées
Seq Scan on clients  (cost=0.00..1.05 rows=1 width=236)
                     (actual time=0.015..50.016 rows=10000 loops=1)
```

**Solution** : Mettre à jour les statistiques

```sql
ANALYZE clients;

-- Augmenter la précision des statistiques pour une colonne
ALTER TABLE clients ALTER COLUMN ville SET STATISTICS 1000;  
ANALYZE clients;  
```

#### Problème : Seq Scan sur Grande Table

**Solution** : Créer un index approprié

```sql
-- Vérifier que l'index est utilisé
SET enable_seqscan = off;  -- Temporairement pour tester  
EXPLAIN SELECT * FROM clients WHERE ville = 'Paris';  
RESET enable_seqscan;  
```

#### Problème : Index Non Utilisé

Causes possibles :
- Le planificateur estime qu'un Seq Scan est plus rapide
- L'index ne correspond pas à la requête
- Statistiques obsolètes

```sql

SET enable_bitmapscan = off;  
```

### 3.7 Optimisation des Requêtes N+1

Le pattern N+1 est un tueur de performances classique.

```sql
-- PROBLÈME : N+1 queries
-- Application fait :
-- 1. SELECT * FROM clients
-- 2. Pour chaque client : SELECT * FROM commandes WHERE client_id = ?

-- SOLUTION 1 : JOIN
SELECT c.*, cmd.*  
FROM clients c  
LEFT JOIN commandes cmd ON c.id = cmd.client_id;  

-- SOLUTION 2 : Sous-requête avec agrégation
SELECT
    c.*,
    (SELECT COUNT(*) FROM commandes cmd WHERE cmd.client_id = c.id) AS nb_commandes
FROM clients c;

-- SOLUTION 3 : LATERAL (plus flexible)
SELECT c.nom, recent.*  
FROM clients c  
LEFT JOIN LATERAL (  
    SELECT date_commande, montant
    FROM commandes cmd
    WHERE cmd.client_id = c.id
    ORDER BY date_commande DESC
    LIMIT 3
) recent ON TRUE;
```

### 3.8 Partitionnement pour les Performances

Le partitionnement divise une grande table en morceaux plus petits.

```sql
-- Créer une table partitionnée par plage de dates
CREATE TABLE logs (
    id BIGSERIAL,
    created_at TIMESTAMP NOT NULL,
    message TEXT,
    level VARCHAR(10)
) PARTITION BY RANGE (created_at);

-- Créer les partitions
CREATE TABLE logs_2024 PARTITION OF logs
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

CREATE TABLE logs_2025 PARTITION OF logs
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');

-- Partition par défaut pour les données non couvertes
CREATE TABLE logs_default PARTITION OF logs DEFAULT;
```

**Avantages** :
- **Partition Pruning** : Le planificateur ne scanne que les partitions pertinentes  
- **Maintenance** : VACUUM/ANALYZE plus rapides par partition  
- **Archivage** : Détacher et supprimer les anciennes partitions

```sql
-- Détacher une partition pour archivage
ALTER TABLE logs DETACH PARTITION logs_2024;

-- Supprimer (rapide car pas de DELETE ligne par ligne)
DROP TABLE logs_2024;
```

### 3.9 Connection Pooling

PostgreSQL crée un processus par connexion, ce qui est coûteux. Un pooler de connexions est essentiel en production.

#### PgBouncer

```ini
# pgbouncer.ini
[databases]
mydb = host=127.0.0.1 port=5432 dbname=mydb

[pgbouncer]
listen_addr = *  
listen_port = 6432  
auth_type = scram-sha-256  
auth_file = /etc/pgbouncer/userlist.txt  

# Mode de pooling
pool_mode = transaction    # session, transaction, statement

# Taille du pool
default_pool_size = 20  
max_client_conn = 1000  
```

**Modes de pooling** :

| Mode | Description | Quand utiliser |
|------|-------------|----------------|
| `session` | Une connexion par session client | Applications avec sessions longues |
| `transaction` | Connexion partagée entre transactions | La plupart des cas |
| `statement` | Connexion partagée entre requêtes | Requêtes simples uniquement |

> **Attention** : En mode `transaction`, les variables de session, les prepared statements et les curseurs ne persistent pas entre les transactions.

### 4.2 Sauvegardes et Restauration


#### Sauvegarde Physique (pg_basebackup)

```bash

pg_basebackup \
    -h primary_host \
    -D /backup/base_backup \
    -U replicator \
    -P \                  # Progress
    -Ft \                 # Format tar
    -z \                  # Compression gzip
    -Xs                   # Stream WAL

pg_basebackup -D /backup/base_backup --checkpoint=fast
```

#### Point-In-Time Recovery (PITR)

Permet de restaurer la base à n'importe quel moment dans le temps.

```ini

```

```bash

restore_command = 'cp /archive/%f %p'
```

#### Diagnostic de l'Espace Disque

```sql
-- Espace utilisé par base
SELECT
    datname,
    pg_size_pretty(pg_database_size(datname)) AS size
FROM pg_database  
ORDER BY pg_database_size(datname) DESC;  

-- Espace utilisé par table
SELECT
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname || '.' || tablename)) AS total,
    pg_size_pretty(pg_relation_size(schemaname || '.' || tablename)) AS table,
    pg_size_pretty(pg_indexes_size(schemaname || '.' || tablename)) AS indexes
FROM pg_tables  
WHERE schemaname NOT IN ('pg_catalog', 'information_schema')  
ORDER BY pg_total_relation_size(schemaname || '.' || tablename) DESC  
LIMIT 20;  

-- Tables avec beaucoup de bloat
SELECT
    schemaname || '.' || relname AS table_name,
    pg_size_pretty(pg_relation_size(relid)) AS table_size,
    n_dead_tup AS dead_tuples,
    n_live_tup AS live_tuples,
    CASE
        WHEN n_live_tup > 0
        THEN round(n_dead_tup * 100.0 / n_live_tup, 2)
        ELSE 0
    END AS dead_pct
FROM pg_stat_user_tables  
WHERE n_dead_tup > 10000  
ORDER BY n_dead_tup DESC;  
```

- [ ] **Sauvegardes** : Automatisées, testées, off-site  
- [ ] **Monitoring** : Alertes sur espace disque, connexions, locks, replication lag  
- [ ] **Sécurité** : SSL activé, SCRAM-SHA-256, RLS si multi-tenant  
- [ ] **Haute Disponibilité** : Réplication configurée, failover testé  
- [ ] **Performance** : pg_stat_statements activé, connexion pooling  
- [ ] **Maintenance** : Autovacuum configuré, ANALYZE régulier  
- [ ] **Logs** : Rotation configurée, requêtes lentes loggées  
- [ ] **Documentation** : Runbooks pour les incidents courants

## Résumé

Ce chapitre avancé vous a présenté les concepts essentiels pour maîtriser PostgreSQL en production :

| Concept | Ce qu'il faut retenir |
|---------|----------------------|
| **MVCC** | Système de versions multiples, XID, VACUUM essentiel |
| **Réplication** | Streaming (physique) vs Logique, synchrone vs asynchrone |
| **Optimisation** | Configuration mémoire, pg_stat_statements, EXPLAIN |
| **Production** | Monitoring, sauvegardes, sécurité, troubleshooting |

### Points Clés à Retenir

- **MVCC** permet une concurrence élevée mais nécessite VACUUM régulier  
- **Réplication** : Choisissez entre haute disponibilité (synchrone) et performance (asynchrone)  
- **Optimisation** : Mesurez avant d'optimiser, utilisez pg_stat_statements  
- **Production** : Automatisez tout, testez vos sauvegardes et failovers
