# 21.3.4. Upgrade Simplifié avec Préservation des Statistiques

## Introduction

Mettre à jour PostgreSQL vers une nouvelle version majeure a longtemps été une opération délicate, notamment en raison de la perte des **statistiques de l'optimiseur**. PostgreSQL 18 révolutionne ce processus en introduisant la **préservation automatique des statistiques** et plusieurs améliorations de l'outil `pg_upgrade`.

Cette section explique ces nouveautés de manière accessible, même si vous n'avez jamais effectué de mise à jour majeure de PostgreSQL.

---

## Comprendre les Enjeux d'une Mise à Jour Majeure

### Qu'est-ce qu'une Mise à Jour Majeure ?

PostgreSQL utilise un système de versioning à trois chiffres : **X.Y.Z**

| Composant | Signification | Exemple |
|-----------|---------------|---------|
| **X** (Majeur) | Version majeure, changements importants | 17 → **18** |
| **Y** (Mineur) | Corrections de bugs et sécurité | 18.0 → 18.**1** |

Une **mise à jour majeure** (17 → 18) nécessite une migration des données, contrairement à une mise à jour mineure (18.0 → 18.1) qui est transparente.

### Le Problème Historique : La Perte des Statistiques

Avant PostgreSQL 18, après chaque mise à jour majeure, vous perdiez toutes les **statistiques de l'optimiseur**. C'était comme si PostgreSQL oubliait tout ce qu'il savait sur vos données.

**Analogie : Le GPS Amnésique**

Imaginez un GPS qui connaît parfaitement votre ville : les embouteillages habituels, les raccourcis, les routes les plus rapides selon l'heure. Après une mise à jour logicielle, ce GPS oublie tout et doit réapprendre chaque trajet.

C'est exactement ce qui se passait avec PostgreSQL :
- Avant la mise à jour : PostgreSQL sait que la table `clients` contient 1 million de lignes, que 80% sont en France, que l'index sur `email` est très sélectif...
- Après la mise à jour : PostgreSQL ne sait plus rien et doit tout réapprendre via `ANALYZE`

### Les Conséquences

| Problème | Impact |
|----------|--------|
| **Plans de requête sous-optimaux** | Requêtes 10× à 100× plus lentes |
| **Temps d'ANALYZE** | Heures voire jours sur de grosses bases |
| **Indisponibilité prolongée** | Performances dégradées pendant l'analyse |
| **Risques opérationnels** | Incidents en production post-upgrade |

## Les Deux Types de Statistiques

Avant d'aller plus loin, il est crucial de distinguer deux types de statistiques dans PostgreSQL.

### 1. Statistiques de l'Optimiseur (Planner Statistics)

Ce sont les informations sur le **contenu et la distribution des données** :

| Information | Stockage | Utilisée par |
|-------------|----------|--------------|
| Nombre de lignes par table | `pg_class.reltuples` | Planificateur |
| Distribution des valeurs | `pg_statistic` | Planificateur |
| Statistiques étendues | `pg_statistic_ext_data` | Planificateur |
| Corrélation physique | `pg_statistic` | Planificateur |

Ces statistiques sont collectées par la commande `ANALYZE` et permettent au planificateur de choisir les meilleurs plans d'exécution.

**PostgreSQL 18 préserve ces statistiques lors des mises à jour !** ✅

### 2. Statistiques de Monitoring (Activity Statistics)

Ce sont les informations sur l'**activité de la base de données** :

| Information | Stockage | Utilisée par |
|-------------|----------|--------------|
| Nombre de scans séquentiels | `pg_stat_user_tables` | DBA / Monitoring |
| Nombre de tuples insérés/modifiés | `pg_stat_user_tables` | Autovacuum |
| Dernière date de vacuum | `pg_stat_user_tables` | Autovacuum |
| Requêtes les plus lentes | `pg_stat_statements` | DBA / Tuning |

Ces statistiques sont stockées en mémoire partagée et persistées dans `pg_stat/`.

**PostgreSQL 18 ne préserve PAS ces statistiques lors des mises à jour.** ❌

### Impact de la Non-Préservation des Statistiques de Monitoring

Après une mise à jour, l'autovacuum et l'autoanalyze "oublient" quelles tables ont besoin de maintenance :

```sql
-- Avant upgrade : PostgreSQL sait que cette table a beaucoup changé
SELECT relname, n_tup_ins, n_mod_since_analyze  
FROM pg_stat_user_tables  
WHERE relname = 'orders';  
-- n_tup_ins: 500000, n_mod_since_analyze: 50000

-- Après upgrade : Tout est remis à zéro
-- n_tup_ins: 0, n_mod_since_analyze: 0
```

**Recommandation** : Après une mise à jour, lancez manuellement un `VACUUM ANALYZE` sur vos tables les plus actives.

## La Révolution PostgreSQL 18 : Préservation des Statistiques

### Comment Ça Fonctionne

PostgreSQL 18 introduit un mécanisme pour **exporter et réimporter** les statistiques de l'optimiseur.

#### Lors du Dump (Export)

`pg_dump` génère maintenant des instructions spéciales pour sauvegarder les statistiques :

```sql
-- Exemple simplifié de ce que pg_dump génère
SELECT pg_catalog.pg_restore_relation_stats(
    'schemaname', 'public',
    'relname', 'clients',
    'reltuples', 1000000::real,
    'relpages', 50000::integer
    -- ... autres statistiques
);

SELECT pg_catalog.pg_restore_attribute_stats(
    'schemaname', 'public',
    'relname', 'clients',
    'attname', 'country',
    'null_frac', 0.01::real,
    'n_distinct', 195::real
    -- ... autres statistiques
);
```

#### Lors du Restore (Import)

Ces fonctions système réécrivent les statistiques dans les tables catalogue (`pg_class`, `pg_statistic`), permettant à PostgreSQL de retrouver immédiatement ses connaissances sur les données.

### Nouvelles Options de pg_dump et pg_restore

PostgreSQL 18 ajoute de nouvelles options pour gérer les statistiques :

| Option | Description |
|--------|-------------|
| `--statistics-only` | Exporte uniquement les statistiques (pas le schéma ni les données) |
| `--no-statistics` | Exporte le schéma et les données sans les statistiques |
| `--no-data` | Exporte uniquement le schéma (existait déjà) |
| `--no-schema` | Exporte uniquement les données (existait déjà) |

#### Exemples d'Utilisation

```bash
# Export complet avec statistiques (comportement par défaut)
pg_dump -Fc mydb > mydb_full.dump

# Export uniquement les statistiques
pg_dump --statistics-only -Fc mydb > mydb_stats.dump

# Export sans statistiques (utile pour les environnements de test)
pg_dump --no-statistics -Fc mydb > mydb_no_stats.dump

# Restaurer uniquement les statistiques sur une base existante
pg_restore --statistics-only -d mydb mydb_stats.dump
```

### Limitations Actuelles

Certains types de statistiques ne sont **pas encore préservés** :

| Type | Préservé ? | Remarque |
|------|------------|----------|
| Statistiques de colonnes | ✅ Oui | Via `pg_statistic` |
| Statistiques de tables | ✅ Oui | Via `pg_class` |
| Statistiques étendues | ❌ Non | CREATE STATISTICS |
| Statistiques d'extensions | ❌ Non | Dépend de l'extension |

Pour régénérer les statistiques manquantes après une mise à jour, utilisez :

```bash
# Analyser uniquement les objets sans statistiques
vacuumdb --all --analyze-in-stages --missing-stats-only

# Puis compléter avec une analyse complète
vacuumdb --all --analyze-only
```

## Les Améliorations de pg_upgrade

### Rappel : Qu'est-ce que pg_upgrade ?

`pg_upgrade` est l'outil officiel pour effectuer des mises à jour majeures de PostgreSQL. Il permet de migrer d'une version à une autre sans avoir à faire un dump/restore complet.

### Les Modes de Transfert de Données

pg_upgrade propose plusieurs modes pour transférer les fichiers de données :

| Mode | Option | Vitesse | Espace Disque | Réversibilité |
|------|--------|---------|---------------|---------------|
| **Copy** | (défaut) | Lent | 2× l'espace | ✅ Ancien cluster intact |
| **Link** | `--link` | Rapide | 1× l'espace | ❌ Ancien cluster inutilisable |
| **Clone** | `--clone` | Rapide | 1× l'espace | ❌ Ancien cluster inutilisable |
| **Swap** | `--swap` | **Très rapide** | 1× l'espace | ❌ Ancien cluster inutilisable |

### Nouveauté PostgreSQL 18 : L'Option --swap

Le mode `--swap` est la nouveauté la plus spectaculaire pour les performances de mise à jour.

#### Comment Ça Fonctionne

Au lieu de copier ou lier les fichiers, `--swap` **échange les répertoires de données** :

```
Avant --swap :
├── /data/pg17/  (ancien cluster avec vos données)
└── /data/pg18/  (nouveau cluster vide)

Après --swap :
├── /data/pg17/  (nouveau cluster avec vos données !)
└── /data/pg17.old/  (ancien cluster, renommé)
```

#### Avantages

- **Extrêmement rapide** : Quelques secondes au lieu de minutes/heures  
- **Aucune copie de fichiers** : Juste un renommage de répertoires  
- **Minimal d'espace disque** : Pas de duplication des données

#### Prérequis

- Les deux répertoires doivent être sur le **même système de fichiers**
- L'ancien cluster devient **inutilisable** après le swap

#### Exemple d'Utilisation

```bash
# Mise à jour avec swap (le plus rapide)
pg_upgrade \
    --old-datadir=/var/lib/postgresql/17/main \
    --new-datadir=/var/lib/postgresql/18/main \
    --old-bindir=/usr/lib/postgresql/17/bin \
    --new-bindir=/usr/lib/postgresql/18/bin \
    --swap
```

### Nouveauté PostgreSQL 18 : Vérifications Parallèles avec --jobs

Avant de migrer les données, `pg_upgrade` effectue de nombreuses vérifications de compatibilité. PostgreSQL 18 permet d'exécuter ces vérifications **en parallèle**.

#### Le Paramètre --jobs

```bash
# Vérifications sur 4 processus parallèles
pg_upgrade \
    --old-datadir=/var/lib/postgresql/17/main \
    --new-datadir=/var/lib/postgresql/18/main \
    --old-bindir=/usr/lib/postgresql/17/bin \
    --new-bindir=/usr/lib/postgresql/18/bin \
    --jobs=4
```

#### Recommandation

Utilisez le nombre de cœurs CPU de votre machine comme point de départ :

```bash
# Détecter le nombre de cœurs
nproc
# Résultat : 8

# Utiliser ce nombre pour --jobs
pg_upgrade ... --jobs=8
```

Les gains sont particulièrement importants pour les serveurs avec :
- **Plusieurs bases de données**  
- **Plusieurs tablespaces**  
- **De nombreux objets** (tables, séquences, fonctions)

### Option --no-statistics

Si vous préférez repartir avec des statistiques fraîches :

```bash
# Mise à jour sans préserver les statistiques
pg_upgrade \
    --old-datadir=/var/lib/postgresql/17/main \
    --new-datadir=/var/lib/postgresql/18/main \
    --old-bindir=/usr/lib/postgresql/17/bin \
    --new-bindir=/usr/lib/postgresql/18/bin \
    --no-statistics
```

Cela peut être utile si :
- Les statistiques de l'ancien cluster étaient obsolètes
- Vous avez significativement modifié les données pendant la migration
- Vous préférez un `ANALYZE` complet après la mise à jour

## Procédure de Mise à Jour Complète

### Étape 1 : Préparation

```bash
# Vérifier la version actuelle
psql -c "SELECT version();"

# Vérifier l'état des checksums (important pour PG 18)
pg_controldata /var/lib/postgresql/17/main | grep "Data page checksum"
```

**Note importante** : PostgreSQL 18 active les checksums par défaut. Si votre ancien cluster n'a pas de checksums, vous devrez initialiser le nouveau cluster avec `--no-data-checksums` :

```bash
# Si l'ancien cluster n'a pas de checksums
initdb --pgdata=/var/lib/postgresql/18/main --no-data-checksums
```

### Étape 2 : Installer PostgreSQL 18

```bash
# Exemple sur Debian/Ubuntu
sudo apt update  
sudo apt install postgresql-18  
```

### Étape 3 : Arrêter les Deux Clusters

```bash
# Arrêter l'ancien cluster
sudo systemctl stop postgresql@17-main

# S'assurer que le nouveau cluster est arrêté
sudo systemctl stop postgresql@18-main
```

### Étape 4 : Vérifications Préalables

```bash
# Exécuter pg_upgrade en mode vérification uniquement
sudo -u postgres /usr/lib/postgresql/18/bin/pg_upgrade \
    --old-datadir=/var/lib/postgresql/17/main \
    --new-datadir=/var/lib/postgresql/18/main \
    --old-bindir=/usr/lib/postgresql/17/bin \
    --new-bindir=/usr/lib/postgresql/18/bin \
    --check \
    --jobs=4
```

Cette commande vérifie la compatibilité sans effectuer la migration.

### Étape 5 : Exécuter la Migration

```bash
# Migration avec préservation des statistiques et mode swap
sudo -u postgres /usr/lib/postgresql/18/bin/pg_upgrade \
    --old-datadir=/var/lib/postgresql/17/main \
    --new-datadir=/var/lib/postgresql/18/main \
    --old-bindir=/usr/lib/postgresql/17/bin \
    --new-bindir=/usr/lib/postgresql/18/bin \
    --swap \
    --jobs=4
```

### Étape 6 : Post-Migration

Après la migration, pg_upgrade vous indiquera les commandes à exécuter :

```bash
# 1. Régénérer les statistiques manquantes (statistiques étendues)
/usr/lib/postgresql/18/bin/vacuumdb --all --analyze-in-stages --missing-stats-only

# 2. Mettre à jour les statistiques cumulatives pour autovacuum
/usr/lib/postgresql/18/bin/vacuumdb --all --analyze-only

# 3. Optionnel : Accélérer en désactivant le throttling
PGOPTIONS='-c vacuum_cost_delay=0' vacuumdb --all --analyze-only --jobs=4
```

### Étape 7 : Démarrer et Vérifier

```bash
# Démarrer le nouveau cluster
sudo systemctl start postgresql@18-main

# Vérifier la version
psql -c "SELECT version();"

# Vérifier que les statistiques sont présentes
psql -c "SELECT relname, reltuples FROM pg_class WHERE relkind = 'r' LIMIT 5;"
```

### Étape 8 : Nettoyage

Une fois satisfait de la migration :

```bash
# Supprimer l'ancien cluster (script généré par pg_upgrade)
./delete_old_cluster.sh
```

## Comparaison : Avant et Après PostgreSQL 18

### Temps d'Indisponibilité Typique

| Scénario | PostgreSQL 17 | PostgreSQL 18 |
|----------|---------------|---------------|
| **Migration (100 Go)** | ~30 min (link) | ~30 sec (swap) |
| **Vérifications** | Séquentielles | Parallèles (--jobs) |
| **ANALYZE post-upgrade** | Obligatoire (heures) | Optionnel (minutes) |
| **Performance optimale** | Après ANALYZE complet | Immédiate |

### Exemple Concret

Base de données de 500 Go avec 200 tables :

| Étape | Avant PG 18 | Avec PG 18 |
|-------|-------------|------------|
| Vérifications | 10 min | 2 min (--jobs=8) |
| Transfert données | 45 min (link) | 5 sec (swap) |
| ANALYZE obligatoire | 3 heures | 15 min (stats manquantes) |
| **Total indisponibilité** | **~4 heures** | **~20 minutes** |

## Gestion des Cas Particuliers

### Checksums Non Alignés

Si votre ancien cluster n'a pas de checksums et que vous voulez les activer :

```bash
# Option 1 : Désactiver les checksums sur le nouveau cluster
initdb --pgdata=/var/lib/postgresql/18/main --no-data-checksums

# Option 2 : Activer les checksums sur l'ancien cluster avant migration
# (nécessite un arrêt prolongé)
pg_checksums --enable --pgdata=/var/lib/postgresql/17/main
```

### Statistiques Étendues

Si vous utilisez `CREATE STATISTICS`, ces statistiques devront être régénérées :

```bash
# Après migration, régénérer les statistiques étendues
vacuumdb --all --analyze-in-stages --missing-stats-only
```

### Tables Très Volumineuses

Pour les tables de plusieurs To, considérez une analyse en plusieurs étapes :

```bash
# Étape 1 : Statistiques minimales (très rapide)
vacuumdb --all --analyze-in-stages --missing-stats-only

# Étape 2 : Analyse complète des tables critiques
psql -c "ANALYZE VERBOSE ma_grande_table;"
```

## Bonnes Pratiques

### Avant la Migration

1. **Testez d'abord** sur un environnement de pré-production  
2. **Sauvegardez** toujours avant une mise à jour majeure  
3. **Vérifiez la compatibilité** avec `--check`  
4. **Planifiez une fenêtre de maintenance** même si elle sera courte

### Pendant la Migration

1. **Utilisez `--swap`** si possible (même système de fichiers)  
2. **Utilisez `--jobs`** selon votre nombre de cœurs CPU  
3. **Surveillez l'espace disque** surtout en mode copy

### Après la Migration

1. **Exécutez les scripts post-migration** fournis par pg_upgrade  
2. **Vérifiez les performances** des requêtes critiques  
3. **Surveillez l'autovacuum** pendant les premiers jours  
4. **Gardez l'ancien cluster** quelques jours avant de le supprimer

## Résumé des Nouveautés

### Préservation des Statistiques

| Aspect | Description |
|--------|-------------|
| **Fonctionnalité** | Les statistiques de l'optimiseur sont conservées |
| **Bénéfice** | Performances optimales immédiatement après upgrade |
| **Limitation** | Statistiques étendues (CREATE STATISTICS) non préservées |
| **Option** | `--no-statistics` pour désactiver |

### Option --swap

| Aspect | Description |
|--------|-------------|
| **Fonctionnalité** | Échange les répertoires au lieu de copier |
| **Bénéfice** | Migration quasi-instantanée |
| **Prérequis** | Même système de fichiers |
| **Attention** | Ancien cluster inutilisable après |

### Vérifications Parallèles

| Aspect | Description |
|--------|-------------|
| **Fonctionnalité** | Exécution parallèle des vérifications |
| **Option** | `--jobs=N` |
| **Recommandation** | N = nombre de cœurs CPU |
| **Bénéfice** | Vérifications plusieurs fois plus rapides |

### Nouvelles Commandes vacuumdb

| Commande | Description |
|----------|-------------|
| `--missing-stats-only` | Analyse uniquement les objets sans statistiques |
| `--analyze-in-stages` | Analyse progressive (rapide puis détaillée) |

## Conclusion

PostgreSQL 18 transforme radicalement l'expérience de mise à jour majeure :

1. **Préservation des statistiques** : Plus besoin d'attendre des heures pour retrouver des performances optimales

2. **Mode swap** : Les migrations qui prenaient des heures se font maintenant en secondes

3. **Vérifications parallèles** : La préparation de la migration est accélérée

4. **Outils post-migration** : `vacuumdb --missing-stats-only` permet de ne régénérer que le strict nécessaire

Ces améliorations réduisent considérablement les risques et la durée d'indisponibilité lors des mises à jour, rendant plus facile de rester à jour avec les dernières versions de PostgreSQL.
