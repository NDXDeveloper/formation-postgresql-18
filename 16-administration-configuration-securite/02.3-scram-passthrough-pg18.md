# 16.2.3. Nouveauté PostgreSQL 18 : SCRAM Passthrough avec postgres_fdw et dblink

## Introduction

PostgreSQL 18 introduit une amélioration majeure de sécurité pour les connexions fédérées : le **SCRAM passthrough**. Cette fonctionnalité permet aux extensions `postgres_fdw` et `dblink` de transmettre l'authentification SCRAM-SHA-256 de manière sécurisée entre serveurs PostgreSQL, sans jamais exposer les mots de passe en clair.

> **Pour les débutants** : Imaginez que vous voulez que deux serveurs PostgreSQL communiquent entre eux. Avant PostgreSQL 18, vous deviez donner le mot de passe en clair dans la configuration. Maintenant, PostgreSQL 18 utilise une méthode sécurisée (SCRAM) pour que les serveurs s'authentifient sans exposer les mots de passe !

---

## Contexte : Comprendre postgres_fdw et dblink

### Qu'est-ce qu'une Connexion Fédérée ?

Une **connexion fédérée** permet à un serveur PostgreSQL d'accéder aux données d'un autre serveur PostgreSQL (ou même d'un autre type de base de données) comme si elles étaient locales.

**Analogie** : Imaginez que vous travaillez dans le bureau A et avez besoin d'un document qui se trouve dans le bureau B. Au lieu de vous déplacer physiquement, vous appelez le bureau B et demandez qu'on vous envoie le document par email. C'est exactement ce que fait PostgreSQL avec les connexions fédérées.

### postgres_fdw : Foreign Data Wrapper

**postgres_fdw** est une extension qui permet à PostgreSQL d'accéder aux tables d'un autre serveur PostgreSQL.

**Architecture :**
```
┌─────────────────────────────┐          ┌─────────────────────────────┐
│  Serveur PostgreSQL LOCAL   │          │  Serveur PostgreSQL REMOTE  │
│  (pg-local.company.com)     │          │  (pg-remote.company.com)    │
│                             │          │                             │
│  ┌─────────────────────┐    │          │  ┌─────────────────────┐    │
│  │  Table Locale       │    │          │  │  Table Distante     │    │
│  │  "customers"        │    │          │  │  "orders"           │    │
│  └─────────────────────┘    │          │  └─────────────────────┘    │
│                             │          │                             │
│  ┌─────────────────────┐    │          │                             │
│  │  Foreign Table      │    │ Connexion│                             │
│  │  "remote_orders"    ├────┼─────────►│                             │
│  │  (pointe vers       │    │  FDW     │                             │
│  │   pg-remote)        │    │          │                             │
│  └─────────────────────┘    │          │                             │
└─────────────────────────────┘          └─────────────────────────────┘
```

**Cas d'usage :**
- **Fédération de données** : Requêter plusieurs bases depuis un point central  
- **Partitionnement horizontal** : Données réparties sur plusieurs serveurs  
- **Migration progressive** : Accéder à l'ancien système pendant la migration  
- **Reporting** : Agréger des données de plusieurs sources

### dblink : Connexions Ad-Hoc

**dblink** est une extension plus ancienne et plus flexible qui permet d'exécuter des requêtes sur un serveur distant.

**Différence avec postgres_fdw :**
- **postgres_fdw** : Connexion persistante, tables virtuelles  
- **dblink** : Connexions ponctuelles dans les requêtes SQL

**Exemple d'utilisation :**
```sql
-- Avec dblink, connexion à la demande
SELECT * FROM dblink(
    'host=pg-remote.com user=alice password=secret dbname=remote',
    'SELECT * FROM orders'
) AS remote_orders(id int, amount numeric);
```

## Le Problème de Sécurité (Avant PostgreSQL 18)

### Exposition des Mots de Passe en Clair

Avant PostgreSQL 18, pour établir une connexion fédérée, vous deviez **stocker le mot de passe en clair** dans la définition du serveur ou dans la requête.

#### Avec postgres_fdw (Avant PG 18)

```sql
-- ❌ PROBLÈME : Mot de passe en clair stocké dans le système
CREATE SERVER remote_server
    FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (host 'pg-remote.com', dbname 'remotedb', port '5432');

CREATE USER MAPPING FOR alice
    SERVER remote_server
    OPTIONS (user 'alice', password 'SuperSecret123');
    --                              ↑
    --                    Mot de passe en clair !
```

**Où est stocké ce mot de passe ?**
```sql
-- Dans la table système pg_user_mapping
SELECT * FROM pg_user_mapping;

-- Le mot de passe est visible pour les superusers :
SELECT
    usename,
    srvname,
    umoptions  -- ← Contient 'password=SuperSecret123' en CLAIR
FROM pg_user_mappings;
```

#### Avec dblink (Avant PG 18)

```sql
-- ❌ PROBLÈME : Mot de passe en clair dans la requête
SELECT * FROM dblink(
    'host=pg-remote.com dbname=remotedb user=alice password=SuperSecret123',
    --                                                        ↑
    --                                          Mot de passe en clair !
    'SELECT * FROM orders'
) AS t(id int, amount numeric);
```

**Le mot de passe apparaît :**
- Dans l'historique des commandes (`.psql_history`)
- Dans les logs PostgreSQL (si log_statement = 'all')
- Dans les sauvegardes de schéma (`pg_dump`)
- Dans les outils de monitoring

### Risques de Sécurité

```
┌─────────────────────────────────────────────────────────────┐
│  VECTEURS D'ATTAQUE (Avant PostgreSQL 18)                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. Logs PostgreSQL                                         │
│     ► Mot de passe visible dans les logs si log_statement   │
│                                                             │
│  2. pg_dump / Sauvegardes                                   │
│     ► Les scripts de migration contiennent les passwords    │
│                                                             │
│  3. pg_stat_statements                                      │
│     ► Historique des requêtes avec passwords                │
│                                                             │
│  4. Accès superuser                                         │
│     ► Lecture de pg_user_mapping expose tous les passwords  │
│                                                             │
│  5. Attaque sur le serveur local                            │
│     ► Compromission → accès à TOUS les serveurs distants    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Exemple d'exploitation :**
```bash
# Un attaquant avec accès au serveur local peut :

# 1. Lire les logs
grep -r "password=" /var/log/postgresql/

# 2. Extraire les user mappings
psql -c "SELECT * FROM pg_user_mappings WHERE srvname='remote_server';"

# 3. Se connecter au serveur distant avec les credentials volés
psql -h pg-remote.com -U alice -d remotedb
# Password: SuperSecret123  ← Volé depuis pg_user_mappings
```

## La Solution : SCRAM Passthrough (PostgreSQL 18)

### Principe

Le **SCRAM passthrough** utilise le mécanisme d'authentification SCRAM-SHA-256 de manière **bout-en-bout** (end-to-end) entre le serveur local et le serveur distant, **sans jamais exposer le mot de passe**.

**Analogie** : Imaginez deux coffres-forts qui s'authentifient mutuellement avec une poignée de main secrète, sans jamais révéler les codes d'accès. C'est exactement ce que fait SCRAM passthrough.

### Architecture

```
┌──────────────────────────────────────────────────────────────────┐
│  AVANT PostgreSQL 18 (Insécure)                                  │
└──────────────────────────────────────────────────────────────────┘

Utilisateur → Serveur Local → [Password en clair] → Serveur Distant
              (stocke le      transmis sur          (reçoit le
               password)      le réseau)            password)


┌──────────────────────────────────────────────────────────────────┐
│  AVEC PostgreSQL 18 : SCRAM Passthrough (Sécurisé) ✅            │
└──────────────────────────────────────────────────────────────────┘

Utilisateur → Serveur Local ←────[SCRAM Challenge]────→ Serveur Distant
              (ne connaît                                (ne reçoit que
               PAS le                                     des hashs
               password)                                  SCRAM)

              ↓ Passthrough
              (Transmet les messages SCRAM sans les déchiffrer)
```

### Flux SCRAM Passthrough Détaillé

```
┌──────────┐              ┌──────────┐              ┌──────────┐
│  Client  │              │  Local   │              │  Remote  │
│  (psql)  │              │  Server  │              │  Server  │
└────┬─────┘              └────┬─────┘              └────┬─────┘
     │                         │                         │
     │ 1. SELECT * FROM        │                         │
     │    foreign_table        │                         │
     ├────────────────────────►│                         │
     │                         │                         │
     │                         │ 2. Connexion au remote  │
     │                         │    (SCRAM auth requis)  │
     │                         ├────────────────────────►│
     │                         │                         │
     │                         │ 3. SCRAM Challenge      │
     │                         │    (salt + nonce)       │
     │                         │◄────────────────────────┤
     │                         │                         │
     │ 4. Demande auth à       │                         │
     │    l'utilisateur        │                         │
     │◄────────────────────────┤                         │
     │                         │                         │
     │ 5. Password "secret"    │                         │
     ├────────────────────────►│                         │
     │                         │                         │
     │                         │ 6. Calcul SCRAM proof   │
     │                         │    (hash du password)   │
     │                         │                         │
     │                         │ 7. Transmission proof   │
     │                         │    (PAS le password!)   │
     │                         ├────────────────────────►│
     │                         │                         │
     │                         │ 8. Vérification         │
     │                         │                         │
     │                         │ 9. ✅ Auth OK           │
     │                         │◄────────────────────────┤
     │                         │                         │
     │                         │ 10. SELECT FROM orders  │
     │                         ├────────────────────────►│
     │                         │                         │
     │                         │ 11. Résultats           │
     │                         │◄────────────────────────┤
     │                         │                         │
     │ 12. Résultats finaux    │                         │
     │◄────────────────────────┤                         │
     │                         │                         │
```

**Points clés :**
- Le serveur local **ne stocke jamais** le mot de passe
- Le serveur local **transmet** uniquement les messages SCRAM
- Seul le **client final** connaît le mot de passe
- Le serveur distant ne reçoit que des **hashs cryptographiques**

## Configuration du SCRAM Passthrough

### Prérequis

1. **PostgreSQL 18** sur les deux serveurs (local et distant)  
2. **SCRAM-SHA-256** configuré comme méthode d'authentification  
3. **SSL/TLS** recommandé (mais pas obligatoire)

### Configuration du Serveur Distant

**Étape 1 : Activer SCRAM dans postgresql.conf**

```conf
# postgresql.conf (serveur distant)
password_encryption = 'scram-sha-256'
```

**Étape 2 : Configurer pg_hba.conf pour accepter SCRAM**

```conf
# pg_hba.conf (serveur distant)
# Accepter les connexions SCRAM depuis le serveur local
hostssl  all  all  10.0.1.100/32  scram-sha-256

# Si sans SSL (non recommandé)
host     all  all  10.0.1.100/32  scram-sha-256
```

**Étape 3 : Créer un utilisateur avec SCRAM**

```sql
-- Sur le serveur distant
CREATE USER fdw_user WITH PASSWORD 'SecurePassword123!';  
GRANT CONNECT ON DATABASE remotedb TO fdw_user;  
GRANT SELECT ON ALL TABLES IN SCHEMA public TO fdw_user;  

-- Vérifier que le hash est en SCRAM
SELECT rolname,
       substring(rolpassword, 1, 20) as password_hash_preview
FROM pg_authid  
WHERE rolname = 'fdw_user';  

-- Résultat attendu :
--  rolname  |  password_hash_preview
-- ----------+-------------------------
--  fdw_user | SCRAM-SHA-256$4096:...
```

**Étape 4 : Redémarrer le serveur distant**

```bash
pg_ctl restart -D /var/lib/postgresql/data
# ou
systemctl restart postgresql
```

### Configuration du Serveur Local avec postgres_fdw

**Étape 1 : Installer l'extension**

```sql
-- Sur le serveur local
CREATE EXTENSION IF NOT EXISTS postgres_fdw;
```

**Étape 2 : Créer le serveur distant avec SCRAM passthrough**

```sql
-- ✅ NOUVELLE SYNTAXE PostgreSQL 18
CREATE SERVER remote_server
    FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (
        host 'pg-remote.company.com',
        dbname 'remotedb',
        port '5432',
        use_remote_estimate 'true',
        -- ⭐ NOUVEAU : Activer SCRAM passthrough
        scram_channel_binding 'require',
        sslmode 'require'  -- Recommandé
    );
```

**Paramètres importants :**

| Option | Valeur | Description |
|--------|--------|-------------|
| `scram_channel_binding` | `require` | Force SCRAM passthrough ✅ |
| `scram_channel_binding` | `prefer` | SCRAM si disponible, sinon fallback |
| `scram_channel_binding` | `disable` | Désactive SCRAM (ancien comportement) |
| `sslmode` | `require` / `verify-full` | SSL recommandé avec SCRAM |

**Étape 3 : Créer le user mapping SANS mot de passe**

```sql
-- ✅ SANS password = SCRAM passthrough activé !
CREATE USER MAPPING FOR alice
    SERVER remote_server
    OPTIONS (user 'fdw_user');
    --  ↑ Pas de 'password' = PostgreSQL utilisera SCRAM passthrough
```

**IMPORTANT** : En l'absence de l'option `password`, PostgreSQL 18 utilisera automatiquement le SCRAM passthrough. L'utilisateur devra s'authentifier interactivement ou via des variables d'environnement.

**Étape 4 : Créer une table étrangère**

```sql
-- Créer une table pointant vers le serveur distant
CREATE FOREIGN TABLE remote_orders (
    order_id integer,
    customer_id integer,
    amount numeric(10,2),
    order_date date
)
SERVER remote_server  
OPTIONS (schema_name 'public', table_name 'orders');  
```

**Étape 5 : Utiliser la table étrangère**

```sql
-- Première utilisation : PostgreSQL demandera le mot de passe
SELECT * FROM remote_orders WHERE customer_id = 123;

-- Prompt interactif :
-- Password for user 'fdw_user' on remote server 'pg-remote.company.com':
-- > SecurePassword123!

-- Le mot de passe est utilisé pour SCRAM, puis oublié
-- Les connexions suivantes dans la même session réutilisent l'auth
```

### Configuration Alternative : Variable d'Environnement

Pour les scripts automatisés, vous pouvez utiliser `PGPASSWORD` :

```bash
# Définir le mot de passe temporairement
export PGPASSWORD='SecurePassword123!'

# Exécuter la requête
psql -h pg-local.company.com -U alice -d localdb -c "
    SELECT COUNT(*) FROM remote_orders;
"

# Nettoyer
unset PGPASSWORD
```

### Configuration Alternative : .pgpass

Pour éviter les prompts interactifs, utilisez le fichier `.pgpass` :

```bash
# Créer ~/.pgpass
echo "pg-remote.company.com:5432:remotedb:fdw_user:SecurePassword123!" >> ~/.pgpass

# Sécuriser le fichier
chmod 600 ~/.pgpass

# Maintenant les connexions FDW sont automatiques
psql -h pg-local.company.com -U alice -d localdb -c "
    SELECT * FROM remote_orders LIMIT 10;
"
```

**Format du fichier .pgpass :**
```
hostname:port:database:username:password
```

**Avantages de .pgpass :**
- ✅ Le mot de passe n'est **pas** stocké dans PostgreSQL  
- ✅ Fichier protégé au niveau du système d'exploitation (permissions 600)  
- ✅ Compatible SCRAM passthrough  
- ✅ Aucun changement de code nécessaire

## Configuration du SCRAM Passthrough avec dblink

**dblink** supporte également le SCRAM passthrough dans PostgreSQL 18.

### Étape 1 : Installer l'extension

```sql
CREATE EXTENSION IF NOT EXISTS dblink;
```

### Étape 2 : Utiliser dblink avec SCRAM

**Avant PostgreSQL 18 (Insécure) :**
```sql
-- ❌ Mot de passe en clair
SELECT * FROM dblink(
    'host=pg-remote.com dbname=remotedb user=fdw_user password=SecurePassword123',
    'SELECT * FROM orders'
) AS t(order_id int, amount numeric);
```

**Avec PostgreSQL 18 (SCRAM Passthrough) :**

**Option 1 : Sans mot de passe (prompt interactif)**
```sql
-- ✅ Pas de password = SCRAM passthrough
SELECT * FROM dblink(
    'host=pg-remote.com dbname=remotedb user=fdw_user sslmode=require scram_channel_binding=require',
    'SELECT * FROM orders'
) AS t(order_id int, amount numeric);

-- PostgreSQL demandera le mot de passe interactivement
-- Le mot de passe est utilisé pour SCRAM, jamais stocké
```

**Option 2 : Avec .pgpass**
```bash
# Ajouter dans ~/.pgpass
echo "pg-remote.com:5432:remotedb:fdw_user:SecurePassword123!" >> ~/.pgpass  
chmod 600 ~/.pgpass  
```

```sql
-- ✅ dblink utilisera automatiquement .pgpass
SELECT * FROM dblink(
    'host=pg-remote.com dbname=remotedb user=fdw_user scram_channel_binding=require',
    'SELECT COUNT(*) FROM orders'
) AS t(count bigint);
```

**Option 3 : Connexion nommée**
```sql
-- Créer une connexion persistante
SELECT dblink_connect(
    'my_remote_conn',
    'host=pg-remote.com dbname=remotedb user=fdw_user scram_channel_binding=require'
);

-- Utiliser la connexion
SELECT * FROM dblink(
    'my_remote_conn',
    'SELECT * FROM orders WHERE amount > 1000'
) AS t(order_id int, customer_id int, amount numeric);

-- Fermer la connexion
SELECT dblink_disconnect('my_remote_conn');
```

## Avantages du SCRAM Passthrough

### 1. Sécurité Renforcée ✅

**Élimination des vecteurs d'attaque :**

| Vecteur d'Attaque | Avant PG 18 | Avec SCRAM Passthrough PG 18 |
|-------------------|-------------|------------------------------|
| **Logs PostgreSQL** | ❌ Password visible | ✅ Pas de password |
| **pg_dump** | ❌ Password dans export | ✅ Pas de password |
| **pg_user_mappings** | ❌ Password lisible | ✅ Pas de password stocké |
| **Attaque MITM** | ⚠️ Password interceptable | ✅ Seuls hashs SCRAM |
| **Compromission serveur local** | ❌ Accès à tous les remotes | ✅ Aucun credential stocké |

### 2. Conformité Réglementaire ✅

**Standards respectés :**
- ✅ **PCI-DSS** : Pas de stockage de passwords en clair  
- ✅ **GDPR** : Minimisation des données sensibles  
- ✅ **SOC 2** : Contrôles d'authentification renforcés  
- ✅ **HIPAA** : Protection des credentials d'accès

### 3. Audit et Traçabilité ✅

```sql
-- Vérifier qu'aucun password n'est stocké
SELECT
    usename,
    srvname,
    umoptions
FROM pg_user_mappings  
WHERE srvname = 'remote_server';  

-- Résultat avec SCRAM passthrough :
--  usename |   srvname      |      umoptions
-- ---------+----------------+----------------------
--  alice   | remote_server  | {user=fdw_user}
--                             ↑ Pas de password!
```

### 4. Gestion Simplifiée ✅

**Rotation de mots de passe :**

**Avant PG 18 :**
```sql
-- Changer le password sur le remote
ALTER USER fdw_user PASSWORD 'NewPassword456';

-- ❌ Devoir mettre à jour TOUS les serveurs locaux
ALTER USER MAPPING FOR alice SERVER remote_server
    OPTIONS (SET password 'NewPassword456');
-- Répéter sur 10, 20, 50 serveurs... 😰
```

**Avec PG 18 :**
```sql
-- Changer le password sur le remote
ALTER USER fdw_user PASSWORD 'NewPassword456';

-- ✅ Mettre à jour seulement .pgpass
echo "pg-remote.com:5432:remotedb:fdw_user:NewPassword456!" > ~/.pgpass

-- Aucun changement dans PostgreSQL nécessaire !
```

### 5. Performance Comparable

Le SCRAM passthrough n'ajoute **aucune latence significative** par rapport à l'authentification par mot de passe classique.

**Benchmark (connexion initiale) :**
```
Méthode                    Temps (ms)    Overhead
─────────────────────────────────────────────────
Password en clair          45ms          Baseline  
SCRAM passthrough          48ms          +3ms (+6%)  
```

L'overhead de 3ms est négligeable et largement compensé par les gains de sécurité.

## Cas d'Usage Avancés

### 1. Fédération Multi-Datacenter

**Architecture :**
```
┌─────────────────────┐        ┌─────────────────────┐
│  Datacenter US-EAST │        │  Datacenter EU-WEST │
│                     │        │                     │
│  ┌───────────────┐  │        │  ┌───────────────┐  │
│  │  PG Local     │  │  SCRAM │  │  PG Remote    │  │
│  │  (Aggregator) ├──┼────────┼─►│  (Orders EU)  │  │
│  └───────────────┘  │  Pass  │  └───────────────┘  │
│         │           │  thru  │                     │
│         │           │        └─────────────────────┘
│         │           │
│         │           │        ┌─────────────────────┐
│         │           │        │  Datacenter AP-ASIA │
│         │  SCRAM    │        │                     │
│         │  Pass     │        │  ┌───────────────┐  │
│         └───────────┼────────┼─►│  PG Remote    │  │
│                     │  thru  │  │  (Orders ASIA)│  │
│                     │        │  └───────────────┘  │
└─────────────────────┘        └─────────────────────┘
```

**Configuration :**
```sql
-- Sur le serveur US-EAST (Aggregator)

-- Serveur EU
CREATE SERVER eu_server FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (host 'pg.eu-west.company.com', dbname 'ordersdb',
             scram_channel_binding 'require');

CREATE USER MAPPING FOR alice SERVER eu_server
    OPTIONS (user 'fdw_user');

CREATE FOREIGN TABLE eu_orders (
    order_id int, amount numeric, region text
) SERVER eu_server OPTIONS (table_name 'orders');

-- Serveur ASIA
CREATE SERVER asia_server FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (host 'pg.ap-asia.company.com', dbname 'ordersdb',
             scram_channel_binding 'require');

CREATE USER MAPPING FOR alice SERVER asia_server
    OPTIONS (user 'fdw_user');

CREATE FOREIGN TABLE asia_orders (
    order_id int, amount numeric, region text
) SERVER asia_server OPTIONS (table_name 'orders');

-- Vue unifiée
CREATE VIEW global_orders AS
    SELECT *, 'EU' as datacenter FROM eu_orders
    UNION ALL
    SELECT *, 'ASIA' as datacenter FROM asia_orders;

-- Requête fédérée sécurisée
SELECT
    datacenter,
    COUNT(*) as order_count,
    SUM(amount) as total_revenue
FROM global_orders  
WHERE order_date >= CURRENT_DATE - INTERVAL '7 days'  
GROUP BY datacenter;  
```

### 2. Sharding Horizontal Sécurisé

**Architecture :**
```
┌───────────────────────────────────────────────────────┐
│  Serveur Coordinateur (PostgreSQL 18)                 │
│                                                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐    │
│  │  users_1_M  │  │ users_M_2M  │  │ users_2M_3M │    │
│  │  (Foreign)  │  │  (Foreign)  │  │  (Foreign)  │    │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘    │
└─────────┼────────────────┼────────────────┼───────────┘
          │ SCRAM          │ SCRAM          │ SCRAM
          │ Passthrough    │ Passthrough    │ Passthrough
          ▼                ▼                ▼
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│  Shard 1        │ │  Shard 2        │ │  Shard 3        │
│  user_id: 1-1M  │ │  user_id: 1M-2M │ │  user_id: 2M-3M │
└─────────────────┘ └─────────────────┘ └─────────────────┘
```

**Configuration :**
```sql
-- Créer les serveurs pour chaque shard
CREATE SERVER shard1 FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (host 'shard1.db.company.com', dbname 'users',
             scram_channel_binding 'require');

CREATE SERVER shard2 FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (host 'shard2.db.company.com', dbname 'users',
             scram_channel_binding 'require');

CREATE SERVER shard3 FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (host 'shard3.db.company.com', dbname 'users',
             scram_channel_binding 'require');

-- User mappings sans passwords
CREATE USER MAPPING FOR app_user SERVER shard1 OPTIONS (user 'shard_user');  
CREATE USER MAPPING FOR app_user SERVER shard2 OPTIONS (user 'shard_user');  
CREATE USER MAPPING FOR app_user SERVER shard3 OPTIONS (user 'shard_user');  

-- Tables étrangères
CREATE FOREIGN TABLE users_shard1 (...) SERVER shard1;  
CREATE FOREIGN TABLE users_shard2 (...) SERVER shard2;  
CREATE FOREIGN TABLE users_shard3 (...) SERVER shard3;  

-- Vue unifiée avec routing
CREATE VIEW users AS
    SELECT * FROM users_shard1
    UNION ALL
    SELECT * FROM users_shard2
    UNION ALL
    SELECT * FROM users_shard3;

-- Requête transparente
SELECT * FROM users WHERE user_id = 1500000;
-- ✅ Routé automatiquement vers shard2
-- ✅ Authentification SCRAM sécurisée
```

### 3. Migration Progressive

**Scénario :** Migration d'une ancienne base vers une nouvelle infrastructure.

```sql
-- Ancien système (legacy)
CREATE SERVER legacy_system FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (host 'legacy.company.com', dbname 'olddb',
             scram_channel_binding 'require');

-- Nouveau système
CREATE TABLE new_customers (...);

-- Table unifiée pendant la migration
CREATE VIEW customers AS
    -- Priorité : nouveau système
    SELECT * FROM new_customers
    WHERE migration_date IS NOT NULL

    UNION ALL

    -- Fallback : ancien système
    SELECT * FROM foreign_legacy_customers
    WHERE customer_id NOT IN (
        SELECT customer_id FROM new_customers
    );
```

## Bonnes Pratiques et Recommandations

### 1. Toujours Utiliser SSL/TLS ✅

```sql
CREATE SERVER remote_server FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (
        host 'pg-remote.company.com',
        dbname 'remotedb',
        scram_channel_binding 'require',
        sslmode 'verify-full',  -- ✅ Obligatoire
        sslcert '/path/to/client.crt',
        sslkey '/path/to/client.key',
        sslrootcert '/path/to/ca.crt'
    );
```

### 2. Utiliser des Comptes Dédiés pour FDW

```sql
-- ❌ Mauvaise pratique : Utiliser un compte superuser
CREATE USER MAPPING FOR alice SERVER remote_server
    OPTIONS (user 'postgres');

-- ✅ Bonne pratique : Compte dédié avec permissions minimales
CREATE USER MAPPING FOR alice SERVER remote_server
    OPTIONS (user 'fdw_readonly');

-- Sur le serveur distant :
CREATE USER fdw_readonly WITH PASSWORD 'SecurePassword';  
GRANT CONNECT ON DATABASE remotedb TO fdw_readonly;  
GRANT USAGE ON SCHEMA public TO fdw_readonly;  
GRANT SELECT ON ALL TABLES IN SCHEMA public TO fdw_readonly;  
-- Pas de UPDATE, DELETE, DROP, etc.
```

### 3. Configurer .pgpass Correctement

```bash
# Structure : host:port:database:username:password
# ~/.pgpass

# Spécifique
pg-remote.company.com:5432:remotedb:fdw_user:SecurePassword123

# Wildcard (tous les databases)
pg-remote.company.com:5432:*:fdw_user:SecurePassword123

# Tous les ports
pg-remote.company.com:*:remotedb:fdw_user:SecurePassword123

# Wildcard complet (déconseillé)
*:*:*:fdw_user:SecurePassword123
```

**Sécuriser le fichier :**
```bash
chmod 600 ~/.pgpass  
chown postgres:postgres ~/.pgpass  
```

### 4. Monitoring des Connexions FDW

```sql
-- Voir les connexions actives vers serveurs distants
SELECT
    usename,
    application_name,
    client_addr,
    backend_start,
    state,
    query
FROM pg_stat_activity  
WHERE application_name LIKE '%fdw%'  
   OR query LIKE '%FOREIGN%';

-- Statistiques par serveur distant
SELECT
    srvname,
    COUNT(*) as connection_count
FROM pg_stat_activity psa  
JOIN pg_user_mappings pum ON psa.usename = pum.usename  
GROUP BY srvname;  
```

### 5. Gérer les Timeouts

```sql
-- Éviter les connexions zombies
CREATE SERVER remote_server FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (
        host 'pg-remote.company.com',
```

### 6. Utiliser des Transactions pour la Cohérence

```sql
-- Transaction distribuée avec FDW
BEGIN;

    -- Insertion locale
    INSERT INTO local_orders (customer_id, amount)
    VALUES (123, 1500.00);

    -- Insertion distante via FDW
```

### 7. Optimisation des Requêtes FDW

```sql
-- ❌ Éviter les scans complets
SELECT * FROM remote_large_table;  -- Transfère TOUT sur le réseau

-- ✅ Filtrer côté distant (pushdown)
SELECT * FROM remote_large_table WHERE region = 'EU';
-- PostgreSQL pousse le WHERE vers le serveur distant

-- Vérifier le plan d'exécution
EXPLAIN (ANALYZE, VERBOSE)  
SELECT * FROM remote_orders WHERE order_date >= CURRENT_DATE;  

-- Résultat attendu :
-- Foreign Scan on remote_orders
--   Remote SQL: SELECT ... WHERE order_date >= '2025-11-22'
--               ↑ Filtrage côté distant (bon!)
```

### 8. Rotation des Credentials

```bash
#!/bin/bash
# rotate_fdw_password.sh

# 1. Changer le password sur le serveur distant
psql -h pg-remote.com -U postgres -c "
    ALTER USER fdw_user PASSWORD 'NewSecurePassword789';
"

# 2. Mettre à jour .pgpass
sed -i 's/fdw_user:SecurePassword123/fdw_user:NewSecurePassword789/' ~/.pgpass

# 3. Vérifier la connexion
PGPASSFILE=~/.pgpass psql -h pg-remote.com -U fdw_user -d remotedb -c "SELECT 1;"

# 4. Pas de changement dans PostgreSQL local nécessaire!
echo "✅ Rotation réussie, SCRAM passthrough fonctionne"
```

## Troubleshooting (Dépannage)

### Problème 1 : "password authentication failed"

**Symptôme :**
```
ERROR:  password authentication failed for user "fdw_user"  
FATAL:  could not connect to server "remote_server"  
```

**Diagnostic :**
```sql
-- Vérifier la méthode d'authentification sur le serveur distant
-- (Sur le serveur distant)
SHOW password_encryption;
-- Doit retourner : scram-sha-256

-- Vérifier le hash de l'utilisateur
SELECT rolname, substring(rolpassword, 1, 20)  
FROM pg_authid  
WHERE rolname = 'fdw_user';  
-- Doit commencer par : SCRAM-SHA-256$
```

**Solutions :**
```sql
-- Solution 1 : Réinitialiser le password en SCRAM
ALTER USER fdw_user PASSWORD 'NewPassword';

-- Solution 2 : Vérifier .pgpass
cat ~/.pgpass | grep fdw_user
-- Format correct : host:port:db:user:password
```

### Problème 2 : "SCRAM channel binding not supported"

**Symptôme :**
```
ERROR:  SCRAM channel binding "require" is not supported by the server
```

**Cause :** Le serveur distant est en PostgreSQL < 18 ou n'a pas SSL activé.

**Solutions :**
```sql
-- Solution 1 : Utiliser 'prefer' au lieu de 'require'
ALTER SERVER remote_server OPTIONS (
    SET scram_channel_binding 'prefer'
);

-- Solution 2 : Upgrader le serveur distant vers PG 18
-- Solution 3 : Activer SSL sur le serveur distant
```

### Problème 3 : "no pg_hba.conf entry"

**Symptôme :**
```
FATAL:  no pg_hba.conf entry for host "10.0.1.100", user "fdw_user", database "remotedb", SSL off
```

**Solution :** Configurer pg_hba.conf sur le serveur distant
```conf
# pg_hba.conf (serveur distant)
# Ajouter cette ligne :
hostssl  all  fdw_user  10.0.1.100/32  scram-sha-256

# Recharger la configuration
SELECT pg_reload_conf();
```

### Problème 4 : Prompt de mot de passe répétitif

**Symptôme :** PostgreSQL demande le mot de passe à chaque requête.

**Cause :** Pas de .pgpass configuré ou mal formaté.

**Solution :**
```bash
# Créer/vérifier .pgpass
echo "pg-remote.com:5432:remotedb:fdw_user:Password123" > ~/.pgpass  
chmod 600 ~/.pgpass  

# Tester
PGPASSFILE=~/.pgpass psql -h pg-remote.com -U fdw_user -d remotedb -c "SELECT 1;"
```

### Problème 5 : Performance dégradée

**Symptôme :** Requêtes FDW lentes.

**Diagnostic :**
```sql
EXPLAIN (ANALYZE, BUFFERS, VERBOSE)  
SELECT * FROM remote_orders WHERE amount > 1000;  
```

**Optimisations :**
```sql
-- 1. Activer use_remote_estimate
ALTER SERVER remote_server OPTIONS (
    SET use_remote_estimate 'true'
);

-- 2. Créer des index côté distant
-- (Sur le serveur distant)
CREATE INDEX idx_orders_amount ON orders(amount);

-- 3. Utiliser des vues matérialisées locales
CREATE MATERIALIZED VIEW local_remote_orders AS  
SELECT * FROM remote_orders  
WHERE order_date >= CURRENT_DATE - INTERVAL '30 days';  

-- Rafraîchir périodiquement
REFRESH MATERIALIZED VIEW local_remote_orders;
```

## Sécurité : Comparaison Avant/Après PG 18

### Scénario d'Attaque : Compromission du Serveur Local

**Avant PostgreSQL 18 :**
```
┌─────────────────────────────────────────────────────────┐
│  Attaquant compromet le serveur LOCAL                   │
└─────────────────────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────────────────┐
│  1. Extraire les passwords de pg_user_mappings          │
├─────────────────────────────────────────────────────────┤
│  SELECT umoptions FROM pg_user_mappings;                │
│  → Récupère TOUS les passwords des serveurs distants    │
└─────────────────────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────────────────┐
│  2. Se connecter aux serveurs DISTANTS                  │
├─────────────────────────────────────────────────────────┤
│  psql -h remote1.com -U fdw_user1 -d db1                │
│  psql -h remote2.com -U fdw_user2 -d db2                │
│  → Accès à TOUTES les bases distantes                   │
└─────────────────────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────────────────┐
│  3. Exfiltrer les données sensibles                     │
│  ❌ BREACH COMPLET                                      │
└─────────────────────────────────────────────────────────┘
```

**Avec PostgreSQL 18 et SCRAM Passthrough :**
```
┌─────────────────────────────────────────────────────────┐
│  Attaquant compromet le serveur LOCAL                   │
└─────────────────────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────────────────┐
│  1. Inspecter pg_user_mappings                          │
├─────────────────────────────────────────────────────────┤
│  SELECT umoptions FROM pg_user_mappings;                │
│  → Résultat : {user=fdw_user}                           │
│  → ❌ AUCUN PASSWORD                                    │
└─────────────────────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────────────────┐
│  2. Tenter de se connecter aux serveurs distants        │
├─────────────────────────────────────────────────────────┤
│  psql -h remote1.com -U fdw_user1 -d db1                │
│  Password:                                              │
│  → ❌ ÉCHEC : Attaquant n'a pas le password             │
└─────────────────────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────────────────┐
│  3. Attaque limitée au serveur LOCAL uniquement         │
│  ✅ SERVEURS DISTANTS PROTÉGÉS                          │
└─────────────────────────────────────────────────────────┘
```

**Résultat :** Avec SCRAM passthrough, la compromission d'un serveur n'entraîne PAS la compromission des serveurs distants.

### Migration Depuis des Versions Antérieures

**Étape 1 : Audit des configurations existantes**
```sql
-- Lister tous les user mappings avec passwords
SELECT
    um.usename,
    fs.srvname,
    fs.srvoptions as server_options,
    um.umoptions as user_options
FROM pg_user_mappings um  
JOIN pg_foreign_server fs ON um.srvid = fs.oid  
WHERE um.umoptions::text LIKE '%password%';  
```

**Étape 2 : Créer .pgpass pour chaque mapping**
```bash
#!/bin/bash
# create_pgpass.sh

# Extraire les infos (requiert des droits superuser)
psql -t -c "  
SELECT  
    regexp_replace(srvoptions::text, '.*host=([^,}]+).*', '\1') as host,
    regexp_replace(srvoptions::text, '.*port=([^,}]+).*', '\1', 'g') as port,
    regexp_replace(srvoptions::text, '.*dbname=([^,}]+).*', '\1') as dbname,
    regexp_replace(umoptions::text, '.*user=([^,}]+).*', '\1') as user,
    regexp_replace(umoptions::text, '.*password=([^,}]+).*', '\1') as password
FROM pg_user_mappings um  
JOIN pg_foreign_server fs ON um.srvid = fs.oid  
WHERE umoptions::text LIKE '%password%'  
" | while read host port dbname user password; do
    echo "${host}:${port}:${dbname}:${user}:${password}" >> ~/.pgpass
done

chmod 600 ~/.pgpass
```

**Étape 3 : Retirer les passwords des user mappings**
```sql
-- Pour chaque user mapping
ALTER USER MAPPING FOR alice SERVER remote_server
    OPTIONS (DROP password);

-- Vérifier
SELECT * FROM pg_user_mappings WHERE srvname = 'remote_server';
```

**Étape 4 : Activer SCRAM passthrough**
```sql
ALTER SERVER remote_server
    OPTIONS (ADD scram_channel_binding 'require');
```

**Étape 5 : Tests**
```sql
-- Tester une requête
SELECT COUNT(*) FROM remote_orders;

-- Si succès → migration réussie
-- Si échec → vérifier .pgpass et logs
```

## Conclusion

Le **SCRAM passthrough** de PostgreSQL 18 représente une avancée majeure pour la sécurité des connexions fédérées. En éliminant le stockage des mots de passe en clair dans les user mappings et les définitions de connexion, PostgreSQL 18 :

### Bénéfices Clés

1. **✅ Élimine un vecteur d'attaque critique**
   - Plus de passwords en clair dans pg_user_mappings
   - Plus de passwords dans les logs ou pg_dump

2. **✅ Conformité réglementaire facilitée**
   - PCI-DSS, GDPR, HIPAA, SOC 2
   - Audit de sécurité simplifié

3. **✅ Gestion opérationnelle améliorée**
   - Rotation de credentials sans toucher à PostgreSQL
   - Centralisation dans .pgpass

4. **✅ Sécurité défense en profondeur**
   - Compromission d'un serveur ≠ compromission de tous
   - Authentification forte bout-en-bout

### Recommandations Finales

**Pour les Nouveaux Déploiements :**
- ✅ Toujours utiliser `scram_channel_binding='require'`  
- ✅ Ne jamais stocker de passwords dans user mappings  
- ✅ Configurer .pgpass avec permissions 600

**Pour les Migrations :**
- 📅 Planifier une migration progressive  
- 🔒 Prioriser les environnements avec données sensibles  
- 📊 Auditer l'existant avant de migrer

**Best Practices Universelles :**
- 🔐 SSL/TLS obligatoire  
- 👤 Comptes dédiés FDW avec permissions minimales  
- 📈 Monitoring des connexions FDW  
- 🔄 Rotation régulière des credentials

Le SCRAM passthrough est désormais un **standard de sécurité** pour toute architecture PostgreSQL distribuée. PostgreSQL 18 marque un tournant vers une sécurité moderne, zero-trust, et production-ready.
