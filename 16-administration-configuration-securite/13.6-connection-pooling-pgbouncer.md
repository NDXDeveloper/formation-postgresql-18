# 16.13.6. Connection Pooling avec PgBouncer (Transaction vs Session)

## Introduction

Imaginez un restaurant où chaque client exige sa propre table, même s'il ne commande qu'un café et reste 5 minutes. Avec 1000 clients par jour, vous auriez besoin de 1000 tables ! C'est exactement le problème que PostgreSQL rencontre avec les connexions : chaque connexion consomme des ressources précieuses (mémoire, processus système).

Le **connection pooling** (mutualisation de connexions) est la solution : au lieu de créer une nouvelle connexion pour chaque requête, on réutilise un pool de connexions existantes. **PgBouncer** est l'outil de référence pour implémenter cette stratégie avec PostgreSQL.

Dans cette section, nous allons explorer :
- Pourquoi le connection pooling est essentiel
- Comment fonctionne PgBouncer
- La différence cruciale entre les modes **transaction** et **session**
- Quand utiliser quel mode
- Installation, configuration et monitoring

> **🎯 Objectif** : Comprendre comment gérer efficacement des milliers de connexions applicatives avec seulement quelques dizaines de connexions PostgreSQL réelles.

## Le Problème : Coût des Connexions PostgreSQL

### Architecture PostgreSQL : Un Processus par Connexion

PostgreSQL utilise une architecture **multi-processus** :

```
Application A se connecte → PostgreSQL crée un processus backend dédié  
Application B se connecte → PostgreSQL crée un autre processus backend  
Application C se connecte → PostgreSQL crée encore un autre processus  
...
```

**Schéma** :
```
┌──────────────┐
│ Application A│──────┐
└──────────────┘      │
                      ├──→ ┌─────────────────┐
┌──────────────┐      │    │  Processus A    │
│ Application B│──────┼──→ │  Processus B    │──→ PostgreSQL
└──────────────┘      │    │  Processus C    │    (Shared Memory,
                      ├──→ │  Processus D    │     Disque, etc.)
┌──────────────┐      │    │  ...            │
│ Application C│──────┘    └─────────────────┘
└──────────────┘
```

Chaque processus backend consomme :
- **~10 MB de RAM** minimum (peut monter à 100+ MB avec work_mem)  
- **Descripteurs de fichiers** système  
- **CPU** pour la gestion du processus  
- **Overhead de création/destruction**

### Calcul du Coût

**Exemple : Application web avec 1000 utilisateurs**

```
Scénario sans pooling :
- 1000 utilisateurs simultanés = 1000 connexions PostgreSQL
- 1000 processus × 10 MB = 10 GB RAM minimum
- Création/destruction constante de processus → CPU élevé
- max_connections doit être ≥ 1000 → Configuration complexe
```

**Avec connection pooling** :
```
- 1000 utilisateurs → PgBouncer → 50 connexions PostgreSQL
- 50 processus × 10 MB = 500 MB RAM
- Connexions réutilisées → Pas de création/destruction
- max_connections = 50 → Configuration simple
```

**Gain** : Division par **20** de la consommation RAM et du nombre de processus !

> **💡 Analogie** : C'est comme un parking avec voiturier. Au lieu que chaque client garde sa voiture devant l'entrée (connexion dédiée), le voiturier gare toutes les voitures dans un parking commun (pool) et les ramène quand nécessaire. Vous avez besoin de beaucoup moins de places.

### Le Problème du "Connection Spike"

**Scénario typique** :
```
08h00 : 10 utilisateurs → 10 connexions
09h00 : 100 utilisateurs → 100 connexions
12h00 : 2000 utilisateurs (pic) → 2000 connexions ❌ CRASH
13h00 : 50 utilisateurs → 50 connexions
```

Sans pooling, PostgreSQL peut :
- Atteindre `max_connections` → Refuser nouvelles connexions
- Saturer la RAM → Swap → Performances catastrophiques
- Devenir instable ou crasher

**Avec pooling** :
```
12h00 : 2000 utilisateurs → File d'attente → 100 connexions PostgreSQL
→ Dégradation gracieuse, pas de crash
```

## PgBouncer : Le Connection Pooler de Référence

### Qu'est-ce que PgBouncer ?

**PgBouncer** est un **proxy léger** qui se place entre votre application et PostgreSQL :

```
Application → PgBouncer (Pooling) → PostgreSQL
```

**Caractéristiques** :
- 🚀 **Ultra-léger** : ~2 MB de RAM, écrit en C  
- ⚡ **Très performant** : Overhead < 1 ms par requête  
- 🔄 **Réutilisation intelligente** des connexions  
- 📊 **Monitoring intégré** : Statistiques détaillées  
- 🛡️ **Protection** contre les pics de charge

### Architecture avec PgBouncer

**Sans PgBouncer** :
```
[App 1] ─┐
[App 2] ─┤
[App 3] ─┼──→ [PostgreSQL]
[App 4] ─┤    max_connections = 1000
[App 5] ─┘    (1000 processus backend)
...
```

**Avec PgBouncer** :
```
[App 1] ─┐
[App 2] ─┤
[App 3] ─┼──→ [PgBouncer] ──→ [PostgreSQL]
[App 4] ─┤    (Pool)           max_connections = 50
[App 5] ─┘                     (50 processus backend)
...
```

## Les Trois Modes de Pooling

PgBouncer propose trois modes de fonctionnement :

| Mode | Durée de Vie Connexion | Granularité | Performance | Compatibilité |
|------|------------------------|-------------|-------------|---------------|
| **Session** | Toute la session client | Session complète | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| **Transaction** | Une transaction | Transaction | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| **Statement** | Une requête SQL | Requête unique | ⭐⭐⭐⭐⭐⭐ | ⭐ |

### Aperçu Rapide

**Session Mode** (Le plus compatible)
```
Client se connecte → Obtient connexion PostgreSQL dédiée  
Client exécute 100 requêtes → Même connexion  
Client se déconnecte → Connexion retourne au pool  
```

**Transaction Mode** (Le plus courant)
```
Client BEGIN → Obtient connexion PostgreSQL  
Client exécute requêtes dans transaction → Même connexion  
Client COMMIT → Connexion retourne immédiatement au pool  
Client fait autre requête → Obtient nouvelle connexion du pool  
```

**Statement Mode** (Le plus agressif, rarement utilisé)
```
Client envoie SELECT → Obtient connexion  
Requête terminée → Connexion retourne au pool  
Client envoie UPDATE → Obtient nouvelle connexion  
```

## Mode Session : La Compatibilité Maximale

### Principe de Fonctionnement

En mode **session**, une connexion PostgreSQL est assignée à un client pour **toute la durée de sa session**, du CONNECT au DISCONNECT.

**Timeline** :
```
Client A:
├─ CONNECT à PgBouncer
├─ PgBouncer assigne Connexion PostgreSQL #1
├─ Requête 1 via Connexion #1
├─ Requête 2 via Connexion #1
├─ Requête 3 via Connexion #1
├─ ...
├─ DISCONNECT
└─ Connexion #1 retourne au pool

Client B:
├─ CONNECT à PgBouncer
├─ PgBouncer assigne Connexion PostgreSQL #1 (recyclée)
└─ ...
```

### Avantages du Mode Session

✅ **Compatibilité Maximale**
```sql
-- Toutes ces fonctionnalités fonctionnent en mode session :

-- 1. Tables temporaires
CREATE TEMP TABLE session_data (id INT, value TEXT);  
INSERT INTO session_data VALUES (1, 'test');  
SELECT * FROM session_data;  -- ✅ Fonctionne  

-- 2. Prepared statements
PREPARE get_user (INT) AS SELECT * FROM users WHERE id = $1;  
EXECUTE get_user(123);  -- ✅ Fonctionne  

-- 3. Curseurs (DECLARE CURSOR)
BEGIN;  
DECLARE user_cursor CURSOR FOR SELECT * FROM users;  
FETCH 10 FROM user_cursor;  -- ✅ Fonctionne  
COMMIT;  

-- 4. LISTEN/NOTIFY
LISTEN my_channel;  
NOTIFY my_channel, 'message';  -- ✅ Fonctionne  

-- 5. SET LOCAL / SET SESSION
SET SESSION work_mem = '50MB';  
SET LOCAL statement_timeout = '30s';  -- ✅ Fonctionne  

-- 6. Variables de session personnalisées
SET app.current_user_id = '12345';  
SELECT current_setting('app.current_user_id');  -- ✅ Fonctionne  
```

✅ **Comportement Transparent**
- L'application ne voit aucune différence avec une connexion directe
- Migrations sans changement de code
- Debugging plus simple

✅ **État de Session Préservé**
- Variables de configuration maintenues
- Transactions implicites possibles
- Comportement identique à PostgreSQL natif

### Inconvénients du Mode Session

❌ **Moins de Mutualisation**
```
Si 200 clients sont connectés mais inactifs (idle) :
→ 200 connexions PostgreSQL occupées
→ Peu de gain par rapport à connexion directe
```

❌ **Moins Efficace avec Connexions Longues**
```
Client se connecte à 9h00, fait 1 requête, reste idle jusqu'à 17h00
→ Connexion PostgreSQL bloquée pendant 8 heures
→ Gaspillage de ressources
```

❌ **Scaling Limité**
```
Pool de 50 connexions PostgreSQL
→ Maximum 50 clients simultanés en mode session
→ Clients 51+ doivent attendre
```

### Cas d'Usage Idéaux pour Mode Session

**1. Applications avec Logique Complexe**
```python
# Application utilisant des prepared statements
import psycopg2

conn = pgbouncer_connect()  # Mode session  
cursor = conn.cursor()  

# Préparation (une fois)
cursor.execute("PREPARE get_orders AS SELECT * FROM orders WHERE user_id = $1")

# Réutilisation (multiple)
for user_id in [123, 456, 789]:
    cursor.execute("EXECUTE get_orders(%s)", (user_id,))
    results = cursor.fetchall()
```

**2. Applications Utilisant des Tables Temporaires**
```python
conn = pgbouncer_connect()  # Mode session

# Créer table temporaire
conn.execute("CREATE TEMP TABLE calc_results (id INT, result FLOAT)")

# Utiliser dans plusieurs requêtes
conn.execute("INSERT INTO calc_results SELECT id, calculate(value) FROM data")  
conn.execute("UPDATE calc_results SET result = result * 1.1 WHERE id > 100")  
results = conn.execute("SELECT * FROM calc_results").fetchall()  

# Table temporaire détruite à la déconnexion
```

**3. Systèmes Legacy ou Migrations**
```
Contexte : Migrer une application existante vers PgBouncer  
Contrainte : Pas de temps pour refactoriser le code  
Solution : Mode session → Migration transparente  
```

**4. Applications Desktop/CLI**
```python
# Application desktop avec connexion longue durée
conn = pgbouncer_connect()  # Mode session  
while True:  
    user_action = input("Action: ")
    conn.execute(user_action)
    # Connexion reste ouverte entre les actions
```

## Mode Transaction : L'Équilibre Performance/Compatibilité

### Principe de Fonctionnement

En mode **transaction**, une connexion PostgreSQL est assignée à un client **uniquement pendant une transaction**.

**Timeline** :
```
Client A:
├─ Requête simple (auto-commit)
│  ├─ Obtient Connexion #1
│  ├─ Exécute requête
│  └─ Connexion #1 retourne au pool
│
├─ BEGIN
│  ├─ Obtient Connexion #2
│  ├─ Requête 1
│  ├─ Requête 2
│  ├─ COMMIT
│  └─ Connexion #2 retourne au pool
│
└─ Requête simple
   ├─ Obtient Connexion #3 (peut être #1 ou #2 recyclée)
   └─ ...
```

**Différence clé** : Entre deux transactions, le client n'a **pas** de connexion PostgreSQL assignée.

### Avantages du Mode Transaction

✅ **Mutualisation Maximale**
```
200 clients connectés, mais seulement 20 en transaction active :
→ 20 connexions PostgreSQL utilisées
→ Ratio 10:1 (10× plus efficace que mode session)
```

✅ **Gestion des Pics de Charge**
```
Scénario : Black Friday, 5000 utilisateurs simultanés  
Mode session : 5000 connexions PostgreSQL nécessaires ❌  
Mode transaction : 100-200 connexions PostgreSQL suffisent ✅  
```

✅ **Protection Contre les Connexions Idle**
```
Client ouvre connexion, fait 1 requête, puis oublie de fermer  
Mode session : Connexion PostgreSQL bloquée indéfiniment ❌  
Mode transaction : Connexion retourne au pool après transaction ✅  
```

✅ **Meilleure Utilisation des Ressources**
```
Pool de 50 connexions PostgreSQL peut servir :  
Mode session : ~50 clients maximum  
Mode transaction : ~500-1000 clients (selon pattern d'usage)  
```

### Inconvénients du Mode Transaction

❌ **État de Session Non Préservé**
```sql
-- ❌ Ces fonctionnalités NE FONCTIONNENT PAS :

-- 1. Tables temporaires
CREATE TEMP TABLE data (id INT);
-- Connexion retourne au pool
SELECT * FROM data;  -- ❌ ERREUR : table n'existe plus

-- 2. Prepared statements non nommés
PREPARE my_stmt AS SELECT * FROM users WHERE id = $1;
-- Connexion retourne au pool
EXECUTE my_stmt(123);  -- ❌ ERREUR : prepared statement inconnu

-- 3. Curseurs en dehors de transaction
DECLARE cursor1 CURSOR FOR SELECT * FROM large_table;  
FETCH 10 FROM cursor1;  -- ❌ ERREUR : pas possible hors transaction  

-- 4. LISTEN/NOTIFY
LISTEN events;  -- ❌ Ne fonctionne pas (état session)

-- 5. Variables session personnalisées
SET app.user_id = '123';
-- Connexion retourne au pool
SELECT current_setting('app.user_id');  -- ❌ Variable perdue
```

❌ **Restrictions sur SET**
```sql
-- ❌ SET SESSION ne fonctionne pas
SET SESSION work_mem = '100MB';  -- ❌ Perdu après transaction

-- ✅ SET LOCAL dans transaction fonctionne
BEGIN;  
SET LOCAL work_mem = '100MB';  -- ✅ OK pour cette transaction  
-- ... requêtes ...
COMMIT;
```

❌ **Transactions Implicites Problématiques**
```sql
-- Certains frameworks/ORMs utilisent transactions implicites
-- Peut causer des comportements inattendus en mode transaction
```

### Cas d'Usage Idéaux pour Mode Transaction

**1. Applications Web Modernes (REST APIs)**
```python
# Exemple : API Flask/FastAPI
@app.get("/users/{user_id}")
def get_user(user_id: int):
    # Requête simple (auto-commit)
    conn = pgbouncer_connect()  # Mode transaction
    user = conn.execute("SELECT * FROM users WHERE id = %s", (user_id,)).fetchone()
    # Connexion retourne au pool immédiatement
    return user

@app.post("/orders")
def create_order(order_data: dict):
    conn = pgbouncer_connect()  # Mode transaction
    with conn.begin():  # Transaction explicite
        order_id = conn.execute("INSERT INTO orders (...) RETURNING id").fetchone()[0]
        conn.execute("INSERT INTO order_items (...)")
        conn.execute("UPDATE inventory ...")
    # COMMIT → Connexion retourne au pool
    return {"order_id": order_id}
```

**2. Microservices**
```javascript
// Node.js avec pg
const { Pool } = require('pg');  
const pool = new Pool({ /* PgBouncer config */ });  

async function handleRequest(req, res) {
    // Chaque requête = transaction courte
    const client = await pool.connect();
    try {
        await client.query('BEGIN');
        const result = await client.query('SELECT ...');
        await client.query('INSERT ...');
        await client.query('COMMIT');
        // Connexion libérée immédiatement
        res.json(result.rows);
    } finally {
        client.release();
    }
}
```

**3. Applications Serverless (AWS Lambda, Cloud Functions)**
```python
# AWS Lambda handler
def lambda_handler(event, context):
    # Connexion de courte durée
    conn = get_pgbouncer_connection()  # Mode transaction

    result = conn.execute("SELECT * FROM data WHERE id = %s", (event['id'],))

    # Lambda termine → Connexion retourne au pool
    return {
        'statusCode': 200,
        'body': json.dumps(result)
    }
```

**4. Applications Haute Concurrence**
```
Contexte : E-commerce avec 10,000 requêtes/sec  
Pattern : Requêtes courtes, nombreuses connexions simultanées  
Solution : Mode transaction avec pool de 100 connexions PostgreSQL  
Résultat : 100 connexions servent 10,000 requêtes/sec efficacement  
```

**5. Batch Processing / Workers**
```python
# Worker traitant une queue de messages
while True:
    message = queue.get()
    conn = pgbouncer_connect()  # Mode transaction

    conn.execute("BEGIN")
    process_message(conn, message)
    conn.execute("COMMIT")
    # Connexion libérée pour le prochain worker
```

## Mode Statement : L'Extrême (Rarement Utilisé)

### Principe de Fonctionnement

En mode **statement**, une connexion PostgreSQL est assignée **pour une seule requête SQL**, puis immédiatement retournée au pool.

```
Client:
├─ SELECT * FROM users → Connexion #1 → Retour au pool
├─ UPDATE users SET ... → Connexion #2 → Retour au pool
├─ DELETE FROM ... → Connexion #3 → Retour au pool
```

### Restrictions Majeures

❌ **Pas de Transactions**
```sql
BEGIN;  -- ❌ IMPOSSIBLE en mode statement  
UPDATE ...;  
COMMIT;  
```

❌ **Pas d'État**
- Aucun état préservé entre requêtes
- Chaque requête est complètement isolée

### Cas d'Usage (Très Rares)

**Uniquement pour** :
- Requêtes totalement indépendantes
- Charge extrêmement élevée
- Aucun besoin de transactions

**En pratique** : Presque jamais utilisé. Préférer le mode transaction.

## Comparaison Détaillée : Transaction vs Session

### Tableau Comparatif Complet

| Critère | Mode Session | Mode Transaction |
|---------|--------------|------------------|
| **Durée connexion** | Toute la session client | Durée de la transaction |
| **Compatibilité** | ⭐⭐⭐⭐⭐ Totale | ⭐⭐⭐ Bonne |
| **Performance** | ⭐⭐⭐ Bonne | ⭐⭐⭐⭐⭐ Excellente |
| **Scalabilité** | ⭐⭐ Limitée | ⭐⭐⭐⭐⭐ Excellente |
| **Tables temporaires** | ✅ Oui | ❌ Non |
| **Prepared statements** | ✅ Oui | ❌ Non (sauf nommés) |
| **Curseurs hors transaction** | ✅ Oui | ❌ Non |
| **LISTEN/NOTIFY** | ✅ Oui | ❌ Non |
| **SET SESSION** | ✅ Oui | ❌ Non |
| **SET LOCAL (dans TX)** | ✅ Oui | ✅ Oui |
| **Transactions explicites** | ✅ Oui | ✅ Oui |
| **Ratio clients/connexions** | 1:1 | 10:1 à 50:1 |
| **Gestion connexions idle** | ❌ Bloquées | ✅ Libérées |
| **Complexité migration** | ⭐ Facile | ⭐⭐⭐ Moyenne |

### Fonctionnalités Détaillées

#### Tables Temporaires

**Mode Session** :
```sql
-- ✅ Fonctionne parfaitement
CREATE TEMP TABLE session_cache (key TEXT, value TEXT);  
INSERT INTO session_cache VALUES ('user_id', '123');  

-- Plusieurs requêtes plus tard...
SELECT value FROM session_cache WHERE key = 'user_id';  -- ✅ OK
```

**Mode Transaction** :
```sql
-- ❌ Ne fonctionne pas
CREATE TEMP TABLE session_cache (key TEXT, value TEXT);
-- Connexion retourne au pool

SELECT * FROM session_cache;  -- ❌ ERREUR : relation n'existe pas
```

**Workaround en mode transaction** :
```sql
-- Utiliser des tables permanentes avec cleanup
CREATE TABLE temp_data (
    session_id TEXT,
    key TEXT,
    value TEXT,
    created_at TIMESTAMP DEFAULT now()
);

-- Cleanup régulier (cron job)
DELETE FROM temp_data WHERE created_at < now() - interval '1 hour';
```

#### Prepared Statements

**Mode Session** :
```sql
-- ✅ Prepared statements classiques fonctionnent
PREPARE get_user (INT) AS SELECT * FROM users WHERE id = $1;  
EXECUTE get_user(123);  
EXECUTE get_user(456);  -- ✅ Réutilisable  
```

**Mode Transaction** :
```sql
-- ❌ Prepared statements simples ne fonctionnent pas
PREPARE get_user (INT) AS SELECT * FROM users WHERE id = $1;
-- Connexion retourne au pool
EXECUTE get_user(123);  -- ❌ ERREUR : prepared statement inconnu

-- ✅ Mais les paramètres de requête fonctionnent
-- (gérés par le driver, pas PostgreSQL)
cursor.execute("SELECT * FROM users WHERE id = %s", (123,))  -- ✅ OK
```

#### Curseurs (CURSOR)

**Mode Session** :
```sql
-- ✅ Curseurs fonctionnent
BEGIN;  
DECLARE large_data_cursor CURSOR FOR SELECT * FROM large_table;  
FETCH 100 FROM large_data_cursor;  
-- Peut faire d'autres choses
FETCH 100 FROM large_data_cursor;  
CLOSE large_data_cursor;  
COMMIT;  
```

**Mode Transaction** :
```sql
-- ✅ Curseurs fonctionnent DANS une transaction
BEGIN;  
DECLARE large_data_cursor CURSOR FOR SELECT * FROM large_table;  
FETCH 100 FROM large_data_cursor;  
FETCH 100 FROM large_data_cursor;  
COMMIT;  -- Curseur fermé automatiquement  

-- ❌ Mais curseurs HORS transaction ne fonctionnent pas
DECLARE cursor1 CURSOR FOR SELECT * FROM table1;  -- ❌ ERREUR
```

#### LISTEN/NOTIFY

**Mode Session** :
```sql
-- ✅ Fonctionne
LISTEN order_events;

-- Application attend les notifications
-- (connexion reste assignée)

NOTIFY order_events, 'New order #123';
```

**Mode Transaction** :
```sql
-- ❌ Ne fonctionne pas
LISTEN order_events;
-- Connexion retourne au pool → LISTEN perdu

-- Alternative : Polling ou système externe (Redis Pub/Sub, etc.)
```

#### Variables de Configuration

**Mode Session** :
```sql
-- ✅ SET SESSION fonctionne
SET SESSION work_mem = '100MB';  
SET SESSION statement_timeout = '60s';  

-- Requêtes suivantes utilisent ces valeurs
SELECT * FROM large_table ORDER BY id;  -- Utilise work_mem=100MB
```

**Mode Transaction** :
```sql
-- ❌ SET SESSION ne persiste pas
SET SESSION work_mem = '100MB';
-- Connexion retourne au pool → Valeur perdue

-- ✅ SET LOCAL dans transaction fonctionne
BEGIN;  
SET LOCAL work_mem = '100MB';  
SELECT * FROM large_table ORDER BY id;  -- Utilise work_mem=100MB  
COMMIT;  -- work_mem revient à la valeur par défaut  
```

```bash
docker pull pgbouncer/pgbouncer

docker run -d \
```

### Configuration Basique

#### Fichier pgbouncer.ini

**Localisation** : `/etc/pgbouncer/pgbouncer.ini`

```ini
[databases]
# Format: database_name = host=... port=... dbname=...
myapp = host=127.0.0.1 port=5432 dbname=myapp_production

# Wildcard: toutes les bases du serveur PostgreSQL
* = host=127.0.0.1 port=5432

[pgbouncer]
# === MODE DE POOLING ===
# transaction | session | statement
pool_mode = transaction

# === ÉCOUTE ===
listen_addr = 0.0.0.0  
listen_port = 6432  

# === AUTHENTIFICATION ===
auth_type = md5  
auth_file = /etc/pgbouncer/userlist.txt  

# === LIMITES DE POOL ===
# Nombre max de connexions PostgreSQL par base
default_pool_size = 25

# Nombre min de connexions maintenues
min_pool_size = 5

# Nombre max de connexions PostgreSQL au total
max_db_connections = 100

# Nombre max de clients pouvant se connecter
max_client_conn = 1000

# === TIMEOUTS ===
# Timeout si serveur ne répond pas
server_idle_timeout = 600

# Timeout si client idle
client_idle_timeout = 0

# Timeout connexion au serveur PostgreSQL
server_connect_timeout = 15

# === LOGS ===
logfile = /var/log/postgresql/pgbouncer.log  
pidfile = /var/run/pgbouncer/pgbouncer.pid  

# Niveau de log : 0=quiet, 1=critical, 2=error, 3=warning, 4=info, 5=debug
log_connections = 1  
log_disconnections = 1  
log_pooler_errors = 1  

# === ADMIN ===
admin_users = postgres  
stats_users = monitoring_user  
```

#### Fichier userlist.txt

**Localisation** : `/etc/pgbouncer/userlist.txt`


### Métriques à Surveiller

#### 1. Ratio Clients/Serveurs

```sql
-- Calcul du ratio de pooling
SELECT
    database,
    cl_active,
    sv_active,
    ROUND(cl_active::numeric / NULLIF(sv_active, 0), 2) AS ratio
```

**Interprétation** :
- Ratio > 5 : ✅ Bon pooling
- Ratio 2-5 : 🟡 Pooling modéré

```sql

    maxwait
```

**Action si cl_waiting > 0** :


#### 3. Temps Moyen de Transaction

```sql

    avg_query_time / 1000000.0 AS avg_query_time_sec
```

**Interprétation** :
- < 0.1s : ✅ Excellent
- 0.1-1s : 🟡 Acceptable
- > 1s : 🔴 Problème (requêtes lentes ou pool trop petit)

## Troubleshooting : Problèmes Courants

### Problème 1 : "sorry, too many clients already"

**Symptôme** :
```
ERROR: sorry, too many clients already
```

**Cause** : `max_client_conn` atteint.

**Solution** :
```ini
# Dans pgbouncer.ini
max_client_conn = 2000  # Au lieu de 1000
```

```bash
# Recharger
sudo systemctl reload pgbouncer
```

### Problème 2 : Clients en Attente (cl_waiting > 0)

**Symptôme** :
```sql
SHOW POOLS;
-- cl_waiting = 50
```

**Cause** : Pool saturé, pas assez de connexions PostgreSQL.

**Solutions** :

**Option 1** : Augmenter pool size
```ini
default_pool_size = 50  # Au lieu de 25  
max_db_connections = 200  # Au lieu de 100  
```

**Option 2** : Optimiser requêtes (si transactions trop longues)
```sql
-- Identifier requêtes lentes sur PostgreSQL
SELECT * FROM pg_stat_statements ORDER BY mean_exec_time DESC LIMIT 10;
```

### Problème 3 : "prepared statement does not exist"

**Symptôme** :
```
ERROR: prepared statement "stmt_123" does not exist
```

**Cause** : Mode transaction + utilisation de prepared statements.

**Solution** : Utiliser paramètres de requête au lieu de prepared statements
```python
# ❌ Mauvais (prepared statement)
cursor.execute("PREPARE stmt AS SELECT * FROM users WHERE id = $1")  
cursor.execute("EXECUTE stmt", (123,))  

# ✅ Bon (paramètres de requête)
cursor.execute("SELECT * FROM users WHERE id = %s", (123,))
```

### Problème 4 : "relation does not exist" (table temporaire)

**Symptôme** :
```
ERROR: relation "temp_table" does not exist
```

**Cause** : Mode transaction + table temporaire.

**Solutions** :

**Option 1** : Passer en mode session
```ini
pool_mode = session
```

**Option 2** : Utiliser tables permanentes avec cleanup
```sql
CREATE TABLE app_temp_data (
    session_id TEXT,
    data JSONB,
    created_at TIMESTAMP DEFAULT now()
);

-- Cleanup automatique (trigger ou cron)
DELETE FROM app_temp_data WHERE created_at < now() - interval '1 hour';
```

### Problème 5 : Connexions PostgreSQL Non Libérées

**Symptôme** : `sv_active` reste élevé même quand `cl_active` est faible.

**Cause** : Transactions non fermées, connexions leaked.

**Diagnostic** :
```sql
-- Sur PostgreSQL
SELECT pid, state, state_change, query  
FROM pg_stat_activity  
WHERE datname = 'myapp'  
  AND state IN ('idle in transaction', 'idle in transaction (aborted)')
ORDER BY state_change;
```

**Solution** : Corriger l'application pour toujours fermer les transactions
```python
# ✅ Bon pattern
try:
    conn.begin()
    # ... requêtes ...
    conn.commit()
except Exception as e:
    conn.rollback()  # Important !
finally:
    conn.close()
```

### Problème 6 : Performances Dégradées Après Migration

**Symptôme** : Application plus lente avec PgBouncer qu'en direct.

**Causes possibles** :

1. **Pool trop petit**
```ini
default_pool_size = 10  # Trop petit
```
**Solution** : Augmenter à 25-50

2. **Timeout trop agressif**
```ini
query_timeout = 5  # 5 secondes trop court
```
**Solution** : Augmenter ou désactiver (0)

3. **Mode session au lieu de transaction**
```ini
pool_mode = session  # Peu de bénéfice du pooling
```
**Solution** : Passer en mode transaction si compatible

## Bonnes Pratiques

### ✅ À Faire

1. **Commencer avec Mode Transaction**
   - 90% des applications modernes sont compatibles
   - Meilleures performances et scalabilité

2. **Dimensionner le Pool Correctement**
   ```
   Règle empirique :
   default_pool_size = (Nombre de CPU PostgreSQL) × 2 à 4

   Exemple : Serveur 8 CPUs
   → default_pool_size = 16-32
   ```

3. **Monitorer Régulièrement**
   ```bash
   # Vérifier quotidiennement
   psql -h localhost -p 6432 -U admin pgbouncer -c "SHOW POOLS"
   ```

4. **Configurer Timeouts Appropriés**
   ```ini
   # Production
   server_idle_timeout = 300  # 5 minutes
   query_timeout = 30  # 30 secondes
   query_wait_timeout = 120  # 2 minutes
   ```

5. **Utiliser SSL si Réseau Non Sécurisé**
   ```ini
   client_tls_sslmode = require
   client_tls_ca_file = /path/to/ca.crt
   client_tls_cert_file = /path/to/client.crt
   client_tls_key_file = /path/to/client.key
   ```

6. **Séparer PgBouncer par Environnement**
   ```
   Production : pgbouncer-prod (port 6432)
   Staging : pgbouncer-staging (port 6433)
   Dev : Connexion directe (pas de PgBouncer nécessaire)
   ```

7. **Fermer Proprement les Connexions**
   ```python
   # Toujours dans un finally
   try:
       conn = pool.getconn()
       # ... utilisation ...
   finally:
       pool.putconn(conn)  # Libérer au pool
   ```

### ❌ À Éviter

1. **Ne Pas Utiliser Mode Transaction pour des Cas Incompatibles**
   ```python
   # ❌ En mode transaction
   conn.execute("CREATE TEMP TABLE ...")  # Ne fonctionnera pas
   ```

2. **Pool Trop Grand**
   ```ini
   # ❌ Mauvais
   default_pool_size = 500  # Bien trop élevé, pas de limite PostgreSQL
   ```
   **Risque** : Saturation PostgreSQL

3. **max_client_conn Trop Bas**
   ```ini
   # ❌ Mauvais
   max_client_conn = 50  # Avec 1000 utilisateurs attendus
   ```

4. **Oublier de Recharger Après Changement**
   ```bash
   # Après modification pgbouncer.ini
   sudo systemctl reload pgbouncer  # Ne pas oublier !
   ```

5. **Ignorer les Clients en Attente**
   ```sql
   -- Si cl_waiting > 0 pendant longtemps
   → Action requise (augmenter pool ou optimiser requêtes)
   ```

6. **Connection Pooling Côté Application ET PgBouncer**
   ```python
   # ⚠️ Attention : Double pooling
   app_pool = Pool(min=50, max=200)  # Pool applicatif
   → PgBouncer (25 connexions)
   → PostgreSQL

   # Risque : 200 connexions app pour 25 connexions PostgreSQL
   # → Beaucoup d'attente
   ```
   **Solution** : Soit pool applicatif, soit PgBouncer, pas les deux avec des limites mal calibrées.

## Architectures Avancées

### 1. PgBouncer sur Chaque Serveur d'Application

```
┌─────────────────┐
│ App Server 1    │
│ ├─ Application  │
│ └─ PgBouncer    │──┐

│ ├─ Application  │  │    └──────────────┘
│ └─ PgBouncer    │──┘
└─────────────────┘
```

**Avantages** :
- Latence minimale (PgBouncer local)
- Pas de SPOF (Single Point of Failure)
- Scaling horizontal simple

**Configuration** :
```ini
# Sur chaque serveur app

listen_addr = 127.0.0.1  # Local seulement  
pool_mode = transaction  
default_pool_size = 10  # Petit pool par serveur  
```

```
┌─────────────┐
│ App Server 1│──┐
└─────────────┘  │
                 │    ┌──────────────┐    ┌──────────────┐
┌─────────────┐  ├───→│ PgBouncer    │───→│ PostgreSQL   │
│ App Server 2│──┤    │ (Centralized)│    └──────────────┘
└─────────────┘  │    └──────────────┘
                 │
┌─────────────┐  │

└─────────────┘
```

**Avantages** :
- Configuration centralisée

### 3. PgBouncer avec Réplication

```
                    ┌─────────────────┐
                    │   PgBouncer     │
                    │   (Writes)      │
                    └────────┬────────┘
                             │
┌────────────┐      ┌────────▼────────┐
│ Application├─────→│  PostgreSQL     │
└────────────┘      │  Primary        │
                    └────────┬────────┘
                             │ Replication
                    ┌────────▼────────┐
                    │  PostgreSQL     │
                    │  Replica        │
                    └────────┬────────┘
                             │
                    ┌────────▼────────┐
                    │   PgBouncer     │
                    │   (Reads)       │
                    └─────────────────┘
```

**Configuration** :
```ini
# pgbouncer-write.ini (vers Primary)
[databases]
myapp_write = host=postgres-primary port=5432 dbname=myapp

[pgbouncer]
pool_mode = transaction

# pgbouncer-read.ini (vers Replica)
[databases]
myapp_read = host=postgres-replica port=5432 dbname=myapp

[pgbouncer]
pool_mode = transaction
```

**Application** :
```python
# Séparation lecture/écriture
write_pool = get_pool('pgbouncer-write', 6432)  
read_pool = get_pool('pgbouncer-read', 6433)  

def create_user(name):
    conn = write_pool.getconn()  # Écriture sur Primary
    conn.execute("INSERT INTO users (name) VALUES (%s)", (name,))
    write_pool.putconn(conn)

def get_users():
    conn = read_pool.getconn()  # Lecture sur Replica
    results = conn.execute("SELECT * FROM users").fetchall()
    read_pool.putconn(conn)
    return results
```

## Résumé et Points Clés

### 🎯 L'Essentiel à Retenir

1. **PgBouncer est Essentiel pour les Applications Scalables**
   - Réduit consommation mémoire et CPU
   - Protège contre les pics de charge
   - Améliore utilisation des ressources

2. **Mode Transaction vs Session : Le Choix Crucial**
   ```
   Mode Transaction (90% des cas) :
   ✅ Performance maximale
   ✅ Scalabilité excellente
   ❌ Pas de tables temporaires
   ❌ Pas de prepared statements classiques

   Mode Session (10% des cas) :
   ✅ Compatibilité totale
   ✅ Comportement transparent
   ❌ Moins de mutualisation
   ❌ Scaling limité
   ```

3. **Dimensionnement du Pool**
   ```
   default_pool_size = (CPU PostgreSQL) × 2-4
   max_client_conn = 10-20 × default_pool_size

   Exemple : Serveur 8 CPUs
   → default_pool_size = 25
   → max_client_conn = 500
   ```

4. **Monitoring Continu**
   ```sql
   -- Vérifier quotidiennement
   SHOW POOLS;

   ⚠️ Si cl_waiting > 0 : Pool saturé, augmenter default_pool_size
   ```

5. **Quand Utiliser Quel Mode ?**
   ```
   Mode Transaction :
   → APIs REST/GraphQL
   → Microservices
   → Applications web modernes
   → Serverless (Lambda, Cloud Functions)

   Mode Session :
   → Applications legacy
   → Utilisation de tables temporaires
   → Prepared statements requis
   → Curseurs hors transaction
   → LISTEN/NOTIFY
   ```

6. **Configuration Minimale de Production**
   ```ini
   [pgbouncer]
   pool_mode = transaction
   default_pool_size = 25
   max_db_connections = 100
   max_client_conn = 1000
   server_idle_timeout = 600
   log_connections = 1
   log_disconnections = 1
   ```

PgBouncer est un outil **indispensable** pour toute application PostgreSQL en production gérant plus de 50 connexions simultanées. Il permet de :

- ✅ **Multiplier par 10-50× le nombre de clients gérables** avec la même infrastructure PostgreSQL  
- ✅ **Réduire drastiquement** la consommation mémoire et CPU  
- ✅ **Protéger PostgreSQL** contre les pics de charge et les connection storms  
- ✅ **Améliorer la résilience** globale de votre architecture

Le choix entre **mode transaction** et **mode session** dépend de votre application :
- **Mode transaction** : 90% des cas (applications web/API modernes)  
- **Mode session** : 10% des cas (applications legacy, besoins spécifiques)

Avec une configuration appropriée, un monitoring attentif et une compréhension des limitations de chaque mode, PgBouncer devient un allié de taille pour faire évoluer votre application PostgreSQL de 100 à 10,000+ utilisateurs simultanés sans changement d'infrastructure majeur.
