# 16.11.1. Sauvegardes Logiques (pg_dump, pg_dumpall)

## Introduction aux Sauvegardes Logiques

Les sauvegardes sont l'un des aspects les plus critiques de l'administration d'une base de données. Une sauvegarde logique consiste à extraire les données et la structure de votre base de données sous forme de commandes SQL que vous pourrez rejouer plus tard pour recréer exactement le même état.

### Pourquoi "Logique" ?

On appelle ces sauvegardes "logiques" parce qu'elles sauvegardent la **logique** de reconstruction de la base de données (les commandes SQL), et non pas les fichiers binaires bruts du système PostgreSQL. C'est comme sauvegarder la recette d'un gâteau plutôt que le gâteau lui-même.

### Avantages des Sauvegardes Logiques

✅ **Portabilité** : Le fichier de sauvegarde peut être restauré sur différentes versions de PostgreSQL, différents systèmes d'exploitation, voire même migré vers un autre SGBD (avec quelques adaptations).

✅ **Flexibilité** : Vous pouvez sauvegarder une seule table, un schéma spécifique, ou l'intégralité de la base.

✅ **Lisibilité** : Le fichier de sauvegarde (en format texte) peut être inspecté et édité manuellement si nécessaire.

✅ **Compatibilité** : Excellente compatibilité entre versions PostgreSQL (montée de version facilitée).

### Inconvénients des Sauvegardes Logiques

❌ **Performance** : Plus lentes que les sauvegardes physiques pour les très grandes bases de données.

❌ **Cohérence transactionnelle limitée** : pg_dump capture un instantané cohérent d'une base, mais pg_dumpall peut avoir des incohérences entre bases.

❌ **Taille** : Les fichiers de sauvegarde peuvent être volumineux (bien que compressibles).

❌ **Restauration** : La restauration peut être longue car elle rejoue toutes les commandes SQL.

## pg_dump : Sauvegarde d'une Base de Données

### Qu'est-ce que pg_dump ?

`pg_dump` est l'outil principal pour créer des sauvegardes logiques d'une **seule base de données** PostgreSQL. Il produit un fichier contenant toutes les commandes SQL nécessaires pour recréer la base de données dans l'état exact où elle était au moment de la sauvegarde.

### Principe de Fonctionnement

Lorsque vous exécutez `pg_dump`, PostgreSQL :

1. **Établit une transaction en lecture seule** sur la base de données  
2. **Capture un instantané cohérent** (snapshot) grâce au système MVCC  
3. **Exporte** toutes les définitions d'objets (tables, index, contraintes, etc.)  
4. **Exporte** toutes les données contenues dans les tables  
5. **Garantit la cohérence** : même si d'autres utilisateurs modifient la base pendant le dump, votre sauvegarde reflétera l'état au moment du démarrage

### Syntaxe de Base

```bash
pg_dump [options] nom_base_donnees > fichier_sauvegarde.sql
```

**Exemple simple** :
```bash
pg_dump ma_base > sauvegarde_ma_base.sql
```

Cette commande crée un fichier texte `sauvegarde_ma_base.sql` contenant toutes les commandes SQL pour recréer `ma_base`.

### Formats de Sortie

`pg_dump` propose plusieurs formats de sortie, chacun ayant ses avantages :

#### 1. Format Plain Text (Texte Brut) - Par Défaut

```bash
pg_dump ma_base > sauvegarde.sql
```

**Caractéristiques** :
- Format : Fichier SQL texte lisible
- Extension recommandée : `.sql`
- Avantages : Lisible, éditable, compatible avec psql
- Inconvénients : Pas de restauration sélective, pas de parallélisation

**Contenu typique** :
```sql
-- PostgreSQL database dump
CREATE TABLE clients (
    id integer NOT NULL,
    nom character varying(100),
    email character varying(255)
);

INSERT INTO clients VALUES (1, 'Dupont', 'dupont@example.com');  
INSERT INTO clients VALUES (2, 'Martin', 'martin@example.com');  
```

**Restauration** :
```bash
psql ma_base < sauvegarde.sql
```

#### 2. Format Custom (Personnalisé) - Recommandé

```bash
pg_dump -Fc ma_base > sauvegarde.dump
# Ou avec -f pour spécifier le fichier de sortie
pg_dump -Fc ma_base -f sauvegarde.dump
```

**Caractéristiques** :
- Format : Binaire compressé propriétaire PostgreSQL
- Extension recommandée : `.dump` ou `.backup`
- Option : `-Fc` (F = Format, c = custom)
- Avantages : Compression native, restauration sélective, parallélisation possible
- Inconvénients : Pas lisible directement, nécessite `pg_restore`

**Restauration** :
```bash
pg_restore -d ma_base sauvegarde.dump
```

#### 3. Format Directory (Répertoire)

```bash
pg_dump -Fd ma_base -f /chemin/vers/repertoire_sauvegarde
```

**Caractéristiques** :
- Format : Répertoire contenant plusieurs fichiers
- Option : `-Fd` (F = Format, d = directory)
- Avantages : Parallélisation du dump et de la restauration, restauration sélective
- Inconvénients : Génère plusieurs fichiers (gestion plus complexe)

**Restauration** :
```bash
pg_restore -d ma_base -j 4 /chemin/vers/repertoire_sauvegarde
```

#### 4. Format Tar (Archive)

```bash
pg_dump -Ft ma_base -f sauvegarde.tar
```

**Caractéristiques** :
- Format : Archive TAR
- Option : `-Ft` (F = Format, t = tar)
- Avantages : Portable, restauration sélective
- Inconvénients : Pas de compression native, pas de parallélisation

**Restauration** :
```bash
pg_restore -d ma_base sauvegarde.tar
```

### Options Importantes de pg_dump

#### Options de Connexion

```bash
# Spécifier l'hôte
pg_dump -h localhost ma_base > sauvegarde.sql

# Spécifier le port
pg_dump -p 5432 ma_base > sauvegarde.sql

# Spécifier l'utilisateur
pg_dump -U admin ma_base > sauvegarde.sql

# Combinaison
pg_dump -h localhost -p 5432 -U admin ma_base > sauvegarde.sql
```

#### Options de Contenu

**Sauvegarder uniquement la structure (schéma) sans les données** :
```bash
pg_dump --schema-only ma_base > structure.sql
# Ou
pg_dump -s ma_base > structure.sql
```

**Sauvegarder uniquement les données sans la structure** :
```bash
pg_dump --data-only ma_base > donnees.sql
# Ou
pg_dump -a ma_base > donnees.sql
```

**Inclure les commandes DROP avant CREATE** :
```bash
pg_dump --clean ma_base > sauvegarde.sql
# Ou
pg_dump -c ma_base > sauvegarde.sql
```
Cela génère des commandes comme `DROP TABLE IF EXISTS clients;` avant chaque `CREATE TABLE`.

**Inclure CREATE DATABASE** :
```bash
pg_dump --create ma_base > sauvegarde.sql
# Ou
pg_dump -C ma_base > sauvegarde.sql
```

#### Options de Sélection

**Sauvegarder uniquement une table spécifique** :
```bash
pg_dump -t clients ma_base > clients.sql
```

**Sauvegarder plusieurs tables** :
```bash
pg_dump -t clients -t commandes ma_base > tables.sql
```

**Exclure une table** :
```bash
pg_dump -T logs ma_base > sauvegarde.sql
```

**Sauvegarder uniquement un schéma** :
```bash
pg_dump -n public ma_base > public_schema.sql
```

**Exclure un schéma** :
```bash
pg_dump -N ancien_schema ma_base > sauvegarde.sql
```

#### Options de Compression

**Compression avec format custom (compression intégrée)** :
```bash
pg_dump -Fc ma_base > sauvegarde.dump
```

**Spécifier le niveau de compression (0-9)** :
```bash
pg_dump -Fc -Z 9 ma_base > sauvegarde.dump
# 0 = pas de compression
# 9 = compression maximale (plus lent)
# 6 = défaut (bon compromis)
```

**Compression avec gzip (format plain)** :
```bash
pg_dump ma_base | gzip > sauvegarde.sql.gz
```

#### Options de Performance

**Mode verbose (afficher la progression)** :
```bash
pg_dump -v ma_base > sauvegarde.sql
```

**Parallélisation (uniquement avec format directory)** :
```bash
pg_dump -Fd -j 4 ma_base -f /sauvegarde/
# -j 4 = utiliser 4 processus parallèles
```

### Exemples Pratiques de pg_dump

#### Exemple 1 : Sauvegarde Complète Standard
```bash
pg_dump -U postgres -Fc -v ma_base -f /backups/ma_base_$(date +%Y%m%d).dump
```
- Format custom compressé
- Utilisateur postgres
- Mode verbose
- Nom de fichier avec date

#### Exemple 2 : Sauvegarde Structure Seulement
```bash
pg_dump -U admin --schema-only -Fc production_db -f schema_prod.dump
```
- Sauvegarde uniquement le schéma (DDL)
- Utile pour cloner la structure vers un environnement de test

#### Exemple 3 : Sauvegarde d'un Schéma Spécifique
```bash
pg_dump -n ventes -Fc ecommerce_db -f ventes_schema.dump
```
- Sauvegarde uniquement le schéma "ventes"

#### Exemple 4 : Sauvegarde Haute Performance
```bash
pg_dump -Fd -j 8 -v huge_database -f /backups/huge_db/
```
- Format directory pour permettre la parallélisation
- 8 processus parallèles
- Mode verbose pour suivre la progression

#### Exemple 5 : Sauvegarde avec Exclusions
```bash
pg_dump -T logs -T temp_* -Fc app_db -f app_db_clean.dump
```
- Exclut la table "logs"
- Exclut toutes les tables commençant par "temp_"

## pg_dumpall : Sauvegarde de Toutes les Bases

### Qu'est-ce que pg_dumpall ?

`pg_dumpall` est l'outil pour créer une sauvegarde logique de **toutes les bases de données** d'une instance PostgreSQL (cluster), incluant les objets globaux qui ne sont pas spécifiques à une base de données.

### Différences avec pg_dump

| Aspect | pg_dump | pg_dumpall |
|--------|---------|------------|
| **Portée** | Une seule base de données | Toutes les bases du cluster |
| **Objets globaux** | Non inclus | Inclus (rôles, tablespaces) |
| **Format** | Plain, Custom, Directory, Tar | Plain uniquement |
| **Compression** | Oui (custom, directory) | Non native (utiliser gzip) |
| **Restauration** | psql ou pg_restore | psql uniquement |
| **Parallélisation** | Oui (directory) | Non |

### Objets Globaux Sauvegardés

`pg_dumpall` sauvegarde les objets qui existent au niveau du cluster PostgreSQL :

- **Rôles (utilisateurs et groupes)** : Tous les comptes avec leurs permissions  
- **Tablespaces** : Définitions des emplacements de stockage personnalisés  
- **Configuration globale** : Certains paramètres au niveau cluster  
- **Databases** : La liste et définition de toutes les bases

### Syntaxe de Base

```bash
pg_dumpall [options] > fichier_sauvegarde.sql
```

**Exemple simple** :
```bash
pg_dumpall -U postgres > sauvegarde_complete.sql
```

### Pourquoi Utiliser pg_dumpall ?

**Scénarios typiques** :

1. **Migration complète** : Vous devez migrer toutes vos bases vers un nouveau serveur  
2. **Sauvegarde des rôles** : Vous voulez sauvegarder les utilisateurs et leurs permissions  
3. **Disaster recovery global** : Besoin de restaurer l'intégralité du cluster  
4. **Clone d'environnement** : Répliquer un environnement complet (dev → staging)

### Options Importantes de pg_dumpall

#### Sauvegarder Uniquement les Objets Globaux

```bash
pg_dumpall --globals-only -U postgres > globals.sql
```

**Contenu typique** :
```sql
-- Roles
CREATE ROLE admin LOGIN PASSWORD 'md5...';  
CREATE ROLE readonly NOLOGIN;  

-- Tablespaces
CREATE TABLESPACE fast_storage LOCATION '/mnt/ssd/pg_data';

-- Database creation
CREATE DATABASE production WITH OWNER = admin;
```

**Cas d'usage** : Vous avez déjà des sauvegardes individuelles de chaque base (pg_dump) et vous voulez juste sauvegarder les rôles et tablespaces.

#### Sauvegarder Uniquement les Rôles

```bash
pg_dumpall --roles-only -U postgres > roles.sql
```

**Cas d'usage** : Synchroniser les utilisateurs entre différents environnements.

#### Sauvegarder Uniquement les Tablespaces

```bash
pg_dumpall --tablespaces-only -U postgres > tablespaces.sql
```

#### Options de Connexion

Identiques à `pg_dump` :
```bash
pg_dumpall -h localhost -p 5432 -U postgres > sauvegarde.sql
```

#### Mode Verbose

```bash
pg_dumpall -v -U postgres > sauvegarde.sql
```

### Exemples Pratiques de pg_dumpall

#### Exemple 1 : Sauvegarde Complète du Cluster
```bash
pg_dumpall -U postgres -v | gzip > /backups/cluster_complet_$(date +%Y%m%d).sql.gz
```
- Sauvegarde toutes les bases et objets globaux
- Compression avec gzip
- Nom de fichier avec date

#### Exemple 2 : Sauvegarde des Rôles pour Synchronisation
```bash
pg_dumpall --roles-only -U postgres > /sync/roles_prod.sql
```
- Sauvegarde uniquement les rôles
- Peut être appliqué sur un autre serveur

#### Exemple 3 : Sauvegarde Globaux + Bases Individuelles
```bash
# Étape 1 : Sauvegarder les globaux
pg_dumpall --globals-only -U postgres > globals.sql

# Étape 2 : Sauvegarder chaque base individuellement
pg_dump -Fc db1 -f db1.dump  
pg_dump -Fc db2 -f db2.dump  
pg_dump -Fc db3 -f db3.dump  
```
**Avantage** : Combine la flexibilité de pg_dump (format custom, compression) avec la sauvegarde des objets globaux.

## Restauration des Sauvegardes Logiques

### Restaurer une Sauvegarde pg_dump

#### Format Plain Text

```bash
psql -U postgres -d ma_base < sauvegarde.sql
```

**Créer d'abord la base si nécessaire** :
```bash
createdb ma_base  
psql -U postgres -d ma_base < sauvegarde.sql  
```

**Si la sauvegarde inclut CREATE DATABASE** :
```bash
psql -U postgres < sauvegarde.sql
```

#### Format Custom, Directory ou Tar (pg_restore)

```bash
pg_restore -d ma_base -U postgres sauvegarde.dump
```

**Options utiles de pg_restore** :

**Restauration avec nettoyage préalable** :
```bash
pg_restore -d ma_base --clean sauvegarde.dump
```

**Restauration parallélisée** :
```bash
pg_restore -d ma_base -j 4 repertoire_sauvegarde/
```

**Restauration sélective (une seule table)** :
```bash
pg_restore -d ma_base -t clients sauvegarde.dump
```

**Liste du contenu d'une sauvegarde** :
```bash
pg_restore --list sauvegarde.dump
```

### Restaurer une Sauvegarde pg_dumpall

```bash
psql -U postgres -f sauvegarde_complete.sql
```

**Si compressée avec gzip** :
```bash
gunzip -c sauvegarde_complete.sql.gz | psql -U postgres
```

**Ordre de restauration** :

1. Les objets globaux (rôles, tablespaces) sont créés en premier  
2. Puis chaque base de données est créée  
3. Enfin, le contenu de chaque base est restauré

## Bonnes Pratiques

### 1. Stratégie de Nommage

Utilisez des noms de fichiers explicites avec horodatage :

```bash
# Bonnes pratiques de nommage
ma_base_20251122_0300.dump           # Base + Date + Heure  
production_schema_20251122.sql       # Environnement + Type + Date  
cluster_full_20251122.sql.gz         # Portée + Date  
```

### 2. Automatisation avec Cron

**Script de sauvegarde automatique** :
```bash
#!/bin/bash
# /usr/local/bin/backup_postgres.sh

DATE=$(date +%Y%m%d_%H%M)  
BACKUP_DIR="/backups/postgresql"  
RETENTION_DAYS=7  

# Sauvegarde complète
pg_dump -U postgres -Fc production_db -f "${BACKUP_DIR}/production_${DATE}.dump"

# Nettoyage des anciennes sauvegardes
find ${BACKUP_DIR} -name "production_*.dump" -mtime +${RETENTION_DAYS} -delete

# Log
echo "Backup completed: production_${DATE}.dump" >> ${BACKUP_DIR}/backup.log
```

**Tâche cron (tous les jours à 3h du matin)** :
```bash
0 3 * * * /usr/local/bin/backup_postgres.sh
```

### 3. Test de Restauration

**Ne jamais considérer une sauvegarde valide sans l'avoir testée !**

```bash
# Créer une base de test
createdb test_restore

# Tester la restauration
pg_restore -d test_restore sauvegarde.dump

# Vérifier l'intégrité
psql -d test_restore -c "SELECT count(*) FROM clients;"

# Nettoyer
dropdb test_restore
```

### 4. Vérification d'Intégrité

```bash
# Vérifier qu'un fichier de sauvegarde custom n'est pas corrompu
pg_restore --list sauvegarde.dump > /dev/null
# Si aucune erreur, le fichier est valide
```

### 5. Compression et Espace Disque

**Toujours surveiller l'espace disque disponible** :

```bash
# Vérifier avant sauvegarde
df -h /backups

# Compresser les anciennes sauvegardes plain text
gzip /backups/old_backup.sql
```

### 6. Sécurité des Sauvegardes

**Permissions restrictives** :
```bash
# Les sauvegardes peuvent contenir des données sensibles
chmod 600 sauvegarde.dump  
chown postgres:postgres sauvegarde.dump  
```

**Chiffrement** :
```bash
# Chiffrer une sauvegarde avec GPG
pg_dump ma_base | gpg --encrypt --recipient admin@company.com > sauvegarde.sql.gpg

# Déchiffrer pour restaurer
gpg --decrypt sauvegarde.sql.gpg | psql ma_base
```

### 7. Stratégie Hybride (Recommandée)

Pour un environnement de production, combinez pg_dump et pg_dumpall :

```bash
#!/bin/bash
BACKUP_DIR="/backups/$(date +%Y%m%d)"  
mkdir -p ${BACKUP_DIR}  

# 1. Sauvegarder les objets globaux
pg_dumpall --globals-only > ${BACKUP_DIR}/globals.sql

# 2. Sauvegarder chaque base individuellement en format custom
for DB in $(psql -U postgres -t -c "SELECT datname FROM pg_database WHERE datistemplate = false AND datname != 'postgres'")  
do  
    pg_dump -U postgres -Fc ${DB} -f ${BACKUP_DIR}/${DB}.dump
done

# 3. Compresser le tout
tar czf backups_$(date +%Y%m%d).tar.gz ${BACKUP_DIR}/
```

### 8. Surveillance et Alertes

**Vérifier le succès de la sauvegarde** :
```bash
#!/bin/bash
pg_dump -U postgres -Fc production_db -f /backups/prod.dump

if [ $? -eq 0 ]; then
    echo "Backup successful" | mail -s "Backup OK" admin@company.com
else
    echo "Backup FAILED" | mail -s "BACKUP FAILED" admin@company.com
fi
```

## Comparaison : pg_dump vs pg_dumpall

### Quand Utiliser pg_dump ?

- ✅ Sauvegarde d'une base spécifique  
- ✅ Besoin de compression native  
- ✅ Besoin de restauration parallélisée  
- ✅ Besoin de restauration sélective (tables spécifiques)  
- ✅ Sauvegardes fréquentes de bases individuelles  
- ✅ Migration d'une base vers un autre serveur

### Quand Utiliser pg_dumpall ?

- ✅ Sauvegarde complète du cluster  
- ✅ Besoin de sauvegarder les rôles et permissions  
- ✅ Besoin de sauvegarder les tablespaces  
- ✅ Migration complète d'un serveur  
- ✅ Clonage complet d'un environnement  
- ✅ Disaster recovery global

### Recommandation Générale

Pour la plupart des environnements de production :

1. **Quotidien** : `pg_dump` de chaque base critique en format custom  
2. **Hebdomadaire** : `pg_dumpall --globals-only` pour les objets globaux  
3. **Complémentaire** : Sauvegardes physiques (pg_basebackup) pour PITR

## Limitations et Considérations

### Limitations des Sauvegardes Logiques

1. **Performance** : Pour des bases de plusieurs centaines de Go, les sauvegardes logiques peuvent être très longues (plusieurs heures).

2. **Cohérence entre bases** : `pg_dumpall` ne garantit pas une cohérence transactionnelle entre différentes bases du cluster.

3. **Large Objects** : Les objets de type BLOB (Large Objects) peuvent poser des problèmes de performance.

4. **Downtime pour restauration** : La restauration peut prendre beaucoup de temps (rejeu de toutes les commandes SQL).

5. **Extensions** : Certaines extensions (notamment avec code C) doivent être réinstallées manuellement sur le serveur cible.

### Considérations Techniques

**Taille de la base vs Temps de sauvegarde** :

| Taille Base | Temps pg_dump (approx) | Recommandation |
|-------------|------------------------|----------------|
| < 10 GB     | Minutes               | pg_dump suffisant |
| 10-100 GB   | 30min - 2h            | pg_dump + compression |
| 100-500 GB  | 2h - 8h               | pg_dump -Fd avec parallélisation |
| > 500 GB    | > 8h                  | Considérer sauvegardes physiques |

**Connexions durant la sauvegarde** :

- `pg_dump` n'empêche **pas** les autres connexions  
- `pg_dump` ne bloque **pas** les écritures  
- `pg_dump` peut légèrement impacter les performances (CPU, I/O, mémoire)
- Il est recommandé de faire les sauvegardes durant les heures creuses

**Verrouillage** :

- `pg_dump` acquiert un verrou `ACCESS SHARE` (le moins restrictif)
- Les écritures continuent normalement pendant le dump
- Certaines opérations DDL peuvent être bloquées (ALTER TABLE, DROP TABLE)

## Cas d'Usage Réels

### Cas 1 : Sauvegarde Quotidienne d'une Application Web

**Contexte** : Base de données de 50 GB, application e-commerce.

**Solution** :
```bash
# Script exécuté à 2h du matin via cron
pg_dump -Fc -Z6 -v ecommerce_db -f /backups/ecommerce_$(date +%Y%m%d).dump

# Rotation : garder 7 jours localement
find /backups -name "ecommerce_*.dump" -mtime +7 -delete

# Upload vers S3 pour archivage long terme
aws s3 cp /backups/ecommerce_$(date +%Y%m%d).dump s3://company-backups/postgres/
```

### Cas 2 : Migration de Serveur

**Contexte** : Migration d'un serveur PostgreSQL 15 vers PostgreSQL 18.

**Solution** :
```bash
# Sur l'ancien serveur (PG 15)
pg_dumpall -U postgres > /tmp/full_migration.sql

# Transférer le fichier
scp /tmp/full_migration.sql new_server:/tmp/

# Sur le nouveau serveur (PG 18)
psql -U postgres -f /tmp/full_migration.sql
```

### Cas 3 : Clonage d'Environnement Dev → Staging

**Contexte** : Besoin de rafraîchir l'environnement de staging avec les données de développement.

**Solution** :
```bash
# Sur serveur dev
pg_dump -Fc dev_database -f /tmp/dev_snapshot.dump

# Copier vers staging
scp /tmp/dev_snapshot.dump staging_server:/tmp/

# Sur staging : nettoyer et restaurer
dropdb staging_database  
createdb staging_database  
pg_restore -d staging_database /tmp/dev_snapshot.dump  
```

### Cas 4 : Sauvegarde Avant Modification Majeure

**Contexte** : Déploiement d'une migration de schéma risquée en production.

**Solution** :
```bash
# Sauvegarde de sécurité immédiate
pg_dump -Fc production_db -f /backups/pre_migration_$(date +%Y%m%d_%H%M%S).dump

# Appliquer la migration
psql production_db -f migration_v2.5.sql

# Si problème : rollback
pg_restore -d production_db --clean /backups/pre_migration_*.dump
```

## Points Clés à Retenir

### Pour pg_dump :

- 📌 Sauvegarde **une seule base de données** à la fois  
- 📌 Offre plusieurs **formats** (plain, custom, directory, tar)  
- 📌 Le format **custom est recommandé** (compression, flexibilité)  
- 📌 Permet la **parallélisation** avec le format directory  
- 📌 Capture un **instantané cohérent** grâce à MVCC  
- 📌 **N'inclut pas** les objets globaux (rôles, tablespaces)

### Pour pg_dumpall :

- 📌 Sauvegarde **toutes les bases** du cluster  
- 📌 Inclut les **objets globaux** (rôles, tablespaces)  
- 📌 Format **plain text uniquement**  
- 📌 Utile pour les **migrations complètes**  
- 📌 Peut être utilisé avec `--globals-only` pour sauvegarder uniquement les rôles et tablespaces

### Bonnes Pratiques Universelles :

- ✅ **Testez vos restaurations** régulièrement  
- ✅ **Automatisez** vos sauvegardes avec cron  
- ✅ **Surveillez** l'espace disque  
- ✅ **Compressez** vos sauvegardes

### Commandes de Référence Rapide

```bash
# pg_dump - Formats courants
pg_dump -Fc dbname > backup.dump          # Format custom  
pg_dump -Fd dbname -f backup/             # Format directory  
pg_dump dbname > backup.sql               # Format plain text  

# pg_dumpall - Usage typique
pg_dumpall > full_backup.sql              # Sauvegarde complète  
pg_dumpall --globals-only > globals.sql   # Objets globaux seulement  
pg_dumpall --roles-only > roles.sql       # Rôles seulement  

# Restauration
psql dbname < backup.sql                  # Plain text  
pg_restore -d dbname backup.dump          # Custom/Directory/Tar  
```
