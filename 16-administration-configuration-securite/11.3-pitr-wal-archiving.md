# 16.11.3. Point-In-Time Recovery (PITR) et WAL Archiving

## Introduction au Concept

Imaginez que vous travaillez sur un document important et que votre traitement de texte sauvegarde automatiquement toutes vos modifications, seconde par seconde. Si vous faites une erreur à 14h37, vous pouvez revenir exactement à l'état de 14h36, pas seulement à la dernière sauvegarde de la nuit précédente.

Le **Point-In-Time Recovery (PITR)** fait exactement cela pour votre base de données PostgreSQL : il vous permet de restaurer votre base de données à n'importe quel moment précis dans le passé, à la seconde près.

### Le Problème que PITR Résout

**Scénario sans PITR** :
```
Lundi 23h00 : Sauvegarde complète  
Mardi 14h30 : Un développeur exécute accidentellement "DELETE FROM clients WHERE 1=1"  
Mardi 14h31 : Panique ! 🚨  
```

**Options limitées** :
- ❌ Restaurer la sauvegarde de lundi 23h00 → Perte de 15h30 de données  
- ❌ Essayer de reconstituer les données manuellement → Risqué et long

**Avec PITR** :
- ✅ Restaurer à mardi 14h29:59 (1 seconde avant l'erreur)  
- ✅ Perte de données : 1 seconde seulement  
- ✅ Temps de récupération : Minutes

### Les Composants du PITR

Le PITR repose sur deux éléments essentiels :

1. **Une sauvegarde de base (Base Backup)** : Un snapshot complet du cluster à un moment T  
2. **Les archives WAL (Write-Ahead Logs)** : L'enregistrement continu de toutes les modifications depuis cette sauvegarde

**Analogie** :
- **Sauvegarde de base** = Une photo de votre maison prise il y a 1 mois  
- **Archives WAL** = Une vidéo enregistrant chaque modification depuis (repeindre un mur, déplacer un meuble, etc.)  
- **PITR** = Regarder la vidéo jusqu'à un moment précis pour recréer l'état exact de votre maison à cet instant

## Comprendre le WAL (Write-Ahead Log)

### Qu'est-ce que le WAL ?

Le **Write-Ahead Log** (journal de transactions) est le mécanisme fondamental qui garantit la durabilité et l'intégrité des données dans PostgreSQL.

### Principe de Fonctionnement

**Règle d'or du WAL** : Toute modification de données est **d'abord** écrite dans le WAL **avant** d'être appliquée aux fichiers de données réels.

**Séquence d'une transaction** :

```
1. BEGIN TRANSACTION
   ↓
2. Modification en mémoire (shared buffers)
   ↓
3. ✍️ Écriture dans le WAL (sur disque)
   ↓
4. COMMIT confirmé à l'utilisateur
   ↓
5. (Plus tard) Écriture dans les fichiers de données
```

**Pourquoi cet ordre ?**

Si le serveur crashe entre l'étape 4 et 5 :
- ✅ Le WAL contient la transaction complète  
- ✅ Au redémarrage, PostgreSQL "rejoue" le WAL  
- ✅ Les données sont restaurées correctement

### Structure des Fichiers WAL

Les fichiers WAL sont stockés dans le répertoire `pg_wal/` (anciennement `pg_xlog/` avant PostgreSQL 10).

**Caractéristiques** :
- **Taille fixe** : 16 MB par fichier (par défaut)  
- **Nommage** : 24 caractères hexadécimaux (ex: `000000010000000000000001`)  
- **Rotation** : Création continue de nouveaux fichiers  
- **Recyclage** : Les anciens fichiers sont réutilisés

**Exemple** :
```bash
$ ls -lh /var/lib/postgresql/data/pg_wal/
-rw------- 1 postgres postgres 16M Nov 22 10:15 000000010000000000000042
-rw------- 1 postgres postgres 16M Nov 22 10:20 000000010000000000000043
-rw------- 1 postgres postgres 16M Nov 22 10:25 000000010000000000000044
```

### Contenu du WAL

Le WAL enregistre :
- **Modifications de données** : INSERT, UPDATE, DELETE  
- **Modifications de structure** : CREATE TABLE, ALTER TABLE  
- **Transactions** : BEGIN, COMMIT, ROLLBACK  
- **Checkpoints** : Points de synchronisation

**Format** : Binaire, non lisible directement (utilisez `pg_waldump` pour inspecter).

### Niveaux de WAL (wal_level)

PostgreSQL propose trois niveaux de WAL :

| Niveau | Description | Cas d'usage |
|--------|-------------|-------------|
| **minimal** | Minimum pour crash recovery | Standalone, pas de réplication |
| **replica** | Suffisant pour réplication physique | Production standard |
| **logical** | Inclut infos pour réplication logique | Réplication sélective, CDC |

**Configuration** :
```conf
# postgresql.conf
wal_level = replica  # Recommandé pour PITR
```

## WAL Archiving : Archivage Continu

### Qu'est-ce que le WAL Archiving ?

Le **WAL archiving** consiste à copier automatiquement chaque fichier WAL complété dans un emplacement d'archivage sécurisé **avant** qu'il ne soit recyclé ou supprimé.

**Sans archivage** :
```
WAL créé → WAL utilisé → WAL recyclé → ❌ Perdu
```

**Avec archivage** :
```
WAL créé → WAL utilisé → 📦 WAL copié vers archive → WAL recyclé
```

### Pourquoi Archiver les WAL ?

1. **PITR** : Restauration à n'importe quel moment  
2. **Réplication** : Serveurs standby éloignés  
3. **Disaster Recovery** : Reconstruction complète après sinistre  
4. **Audit** : Traçabilité complète des modifications

### Configuration de l'Archivage

#### Étape 1 : Créer un Répertoire d'Archive

```bash
# Créer le répertoire
sudo mkdir -p /mnt/wal_archive

# Permissions appropriées
sudo chown postgres:postgres /mnt/wal_archive  
sudo chmod 700 /mnt/wal_archive  
```

**Recommandations** :
- Utiliser un disque/partition séparé
- Préférer un montage NFS ou stockage réseau
- Éviter le même disque que le data directory

#### Étape 2 : Configurer postgresql.conf

```conf
# Activer l'archivage
archive_mode = on

# Commande d'archivage (exemples ci-dessous)
archive_command = 'cp %p /mnt/wal_archive/%f'

# Timeout pour l'archivage (optionnel)
archive_timeout = 300  # Forcer archivage toutes les 5 minutes
```

**Paramètres** :
- `%p` : Chemin complet du fichier WAL à archiver  
- `%f` : Nom du fichier WAL seulement

#### Étape 3 : Redémarrer PostgreSQL

```bash
sudo systemctl restart postgresql
```

⚠️ **Important** : Le changement de `archive_mode` nécessite un redémarrage complet (pas juste un reload).

### Commandes d'Archivage Courantes

#### 1. Copie Locale Simple

```conf
archive_command = 'test ! -f /mnt/wal_archive/%f && cp %p /mnt/wal_archive/%f'
```

**Explication** :
- `test ! -f /mnt/wal_archive/%f` : Vérifie que le fichier n'existe pas déjà  
- `&&` : Continue si le test réussit  
- `cp %p /mnt/wal_archive/%f` : Copie le fichier

**Avantages** : Simple, rapide  
**Inconvénients** : Aucune redondance, vulnérable aux pannes disque  

#### 2. Archivage Distant avec rsync

```conf
archive_command = 'rsync -a %p backup-server:/wal_archive/%f'
```

**Avantages** : Stockage hors-site, sécurisé  
**Inconvénients** : Dépend du réseau  

#### 3. Archivage avec Vérification

```conf
archive_command = 'cp %p /mnt/wal_archive/%f && md5sum %p > /mnt/wal_archive/%f.md5'
```

**Avantages** : Vérifie l'intégrité du fichier  
**Inconvénients** : Légèrement plus lent  

#### 4. Archivage Redondant

```bash
#!/bin/bash
# /usr/local/bin/archive_wal.sh
cp "$1" /mnt/wal_archive/"$2"  
rsync -a "$1" backup-server:/wal_archive/"$2"  
aws s3 cp "$1" s3://company-backups/wal/"$2"  
```

```conf
archive_command = '/usr/local/bin/archive_wal.sh %p %f'
```

**Avantages** : Triple redondance (local, distant, cloud)  
**Inconvénients** : Plus lent, plus complexe  

#### 5. Archivage vers S3 (AWS)

```conf
archive_command = 'aws s3 cp %p s3://my-bucket/wal_archive/%f'
```

**Prérequis** : AWS CLI configuré

#### 6. Archivage avec Compression

```conf
archive_command = 'gzip < %p > /mnt/wal_archive/%f.gz'
```

**Avantages** : Économie d'espace disque (~80% compression)  
**Inconvénients** : Plus lent, nécessite décompression à la restauration  

### Vérifier le Fonctionnement de l'Archivage

#### Méthode 1 : Forcer un Switch de WAL

```sql
-- Forcer PostgreSQL à créer un nouveau fichier WAL
SELECT pg_switch_wal();

-- Attendre quelques secondes, puis vérifier
SELECT * FROM pg_stat_archiver;
```

**Colonnes importantes de pg_stat_archiver** :
- `archived_count` : Nombre de WAL archivés avec succès  
- `failed_count` : Nombre d'échecs (devrait être 0)  
- `last_archived_wal` : Dernier WAL archivé  
- `last_archived_time` : Horodatage du dernier archivage  
- `last_failed_wal` : Dernier WAL échoué (si failed_count > 0)

#### Méthode 2 : Vérifier les Logs

```bash
tail -f /var/log/postgresql/postgresql-18-main.log | grep archive
```

**Messages attendus** :
```
archived write-ahead log file "000000010000000000000042"
```

**Messages d'erreur** :
```
could not execute archive command: ...  
archive command failed with exit code 1  
```

#### Méthode 3 : Inspecter le Répertoire d'Archive

```bash
ls -lh /mnt/wal_archive/ | tail
```

Vous devriez voir de nouveaux fichiers WAL apparaître régulièrement.

### Gestion des Archives WAL

#### Taille des Archives

Les archives WAL croissent indéfiniment si non nettoyées.

**Calcul de la croissance** :
```
Taille par jour = (Activité d'écriture) / 16 MB

Exemple : Base très active générant 10 GB/jour de WAL
→ ~640 fichiers WAL par jour
→ ~4.5 TB par an
```

#### Stratégies de Rétention

**Option 1 : Rétention par Durée**
```bash
# Conserver 30 jours de WAL
find /mnt/wal_archive -name "*.wal" -mtime +30 -delete
```

**Option 2 : Rétention par Taille**
```bash
#!/bin/bash
# Garder seulement 500 GB de WAL
TARGET_SIZE=500  # GB  
ARCHIVE_DIR="/mnt/wal_archive"  

CURRENT_SIZE=$(du -s --block-size=1G ${ARCHIVE_DIR} | cut -f1)

if [ ${CURRENT_SIZE} -gt ${TARGET_SIZE} ]; then
    # Supprimer les plus anciens
    ls -t ${ARCHIVE_DIR}/*.wal | tail -n +1000 | xargs rm -f
fi
```

**Option 3 : Rétention Coordonnée avec Sauvegardes**
```bash
# Conserver les WAL uniquement pour les sauvegardes existantes
# (Barman, pgBackRest font cela automatiquement)
```

**Automatisation avec Cron** :
```bash
# Nettoyage quotidien à 4h du matin
0 4 * * * find /mnt/wal_archive -mtime +30 -delete
```

⚠️ **Attention** : Ne supprimez jamais un WAL nécessaire à une restauration PITR !

## Configuration Complète du PITR

### Vue d'Ensemble

Le PITR complet nécessite :

1. **Configuration de l'archivage WAL** (déjà vu)  
2. **Sauvegarde de base régulière** (pg_basebackup)  
3. **Configuration de restauration** (recovery.conf / recovery.signal)  
4. **Restauration et replay des WAL**

### Schéma du Processus

```
┌─────────────────┐
│ Production DB   │
│                 │
│ [WAL archiving] │ ─────> [Archive WAL] (/mnt/wal_archive)
└─────────────────┘              │
        │                        │
        │ pg_basebackup          │
        ↓                        │
┌─────────────────┐              │
│ Base Backup     │              │
│ (Snapshot T0)   │              │
└─────────────────┘              │
        │                        │
        │  En cas d'incident     │
        ↓                        ↓
┌─────────────────────────────────┐
│ Restauration                    │
│ 1. Copier base backup           │
│ 2. Configurer recovery          │
│ 3. Replay WAL jusqu'au point T  │
└─────────────────────────────────┘
```

### Étape par Étape : Configuration Complète

#### Phase 1 : Préparation (À faire UNE FOIS)

**1. Configuration PostgreSQL (postgresql.conf)** :
```conf
# Niveau WAL pour réplication/archivage
wal_level = replica

# Nombre de processus WAL sender
max_wal_senders = 5

# Taille minimale de WAL à conserver
wal_keep_size = 1GB

# Activer l'archivage
archive_mode = on

# Commande d'archivage
archive_command = 'test ! -f /mnt/wal_archive/%f && cp %p /mnt/wal_archive/%f'

# Timeout d'archivage (optionnel)
archive_timeout = 300  # 5 minutes
```

**2. Redémarrer PostgreSQL** :
```bash
sudo systemctl restart postgresql
```

**3. Vérifier la configuration** :
```sql
SHOW wal_level;  
SHOW archive_mode;  
SHOW archive_command;  

-- Vérifier l'état de l'archivage
SELECT * FROM pg_stat_archiver;
```

#### Phase 2 : Sauvegarde de Base (À faire RÉGULIÈREMENT)

**Script de sauvegarde** :
```bash
#!/bin/bash
# /usr/local/bin/pitr_base_backup.sh

DATE=$(date +%Y%m%d_%H%M)  
BACKUP_DIR="/backups/pitr"  
BASE_BACKUP="${BACKUP_DIR}/base_${DATE}"  

# Créer le répertoire
mkdir -p ${BASE_BACKUP}

# Exécuter pg_basebackup
pg_basebackup \
    -D ${BASE_BACKUP} \
    -U backup_user \
    -X stream \
    -c fast \
    -P -v

# Enregistrer le timestamp pour référence
echo ${DATE} > ${BACKUP_DIR}/last_backup_timestamp.txt

echo "Base backup completed: ${BASE_BACKUP}"
```

**Cron (hebdomadaire, dimanche à 1h)** :
```bash
0 1 * * 0 /usr/local/bin/pitr_base_backup.sh
```

#### Phase 3 : Archivage Continu

L'archivage WAL se fait **automatiquement** et **continuellement** grâce à `archive_command`.

**Monitoring recommandé** :
```sql
-- Vérifier toutes les heures
SELECT
    archived_count,
    failed_count,
    last_archived_time,
    now() - last_archived_time AS time_since_last_archive
FROM pg_stat_archiver;
```

**Alerte si** :
- `failed_count` augmente  
- `time_since_last_archive` > 1 heure (sur base active)

## Scénarios de Restauration PITR

### Scénario 1 : Restauration à un Point Précis (Timestamp)

**Contexte** :
- Incident : Suppression accidentelle de données à 14h37
- Objectif : Restaurer à 14h36:59

**Procédure** :

#### Étape 1 : Arrêter PostgreSQL

```bash
sudo systemctl stop postgresql
```

#### Étape 2 : Sauvegarder les Données Actuelles (Sécurité)

```bash
sudo mv /var/lib/postgresql/18/main /var/lib/postgresql/18/main.corrupted
```

#### Étape 3 : Restaurer la Sauvegarde de Base

```bash
# Copier la dernière sauvegarde de base
sudo cp -a /backups/pitr/base_20251122_0100 /var/lib/postgresql/18/main

# Permissions
sudo chown -R postgres:postgres /var/lib/postgresql/18/main  
sudo chmod 700 /var/lib/postgresql/18/main  
```

#### Étape 4 : Créer le Fichier recovery.signal

```bash
# PostgreSQL 12+ utilise recovery.signal
sudo touch /var/lib/postgresql/18/main/recovery.signal
```

#### Étape 5 : Configurer les Paramètres de Récupération

```bash
# Éditer postgresql.conf ou postgresql.auto.conf
sudo tee -a /var/lib/postgresql/18/main/postgresql.auto.conf << EOF

# Configuration PITR
restore_command = 'cp /mnt/wal_archive/%f %p'  
recovery_target_time = '2025-11-22 14:36:59'  
recovery_target_action = 'promote'  
EOF  
```

**Paramètres** :
- `restore_command` : Comment récupérer les WAL archivés  
- `recovery_target_time` : Moment précis de restauration  
- `recovery_target_action` : Que faire après la récupération  
  - `pause` : Arrêter en mode pause (inspection possible)  
  - `promote` : Passer en mode production  
  - `shutdown` : Arrêter PostgreSQL

#### Étape 6 : Permissions et Démarrage

```bash
sudo chown postgres:postgres /var/lib/postgresql/18/main/recovery.signal  
sudo systemctl start postgresql  
```

#### Étape 7 : Observer le Processus de Récupération

```bash
# Logs en temps réel
sudo tail -f /var/log/postgresql/postgresql-18-main.log
```

**Messages attendus** :
```
starting point-in-time recovery to 2025-11-22 14:36:59+00  
restored log file "000000010000000000000042" from archive  
redo starts at 0/42000028  
recovery stopping before commit of transaction 1234, time 2025-11-22 14:37:01+00  
recovery has paused  
database system is ready to accept read-only connections  
```

#### Étape 8 : Vérification

```bash
psql -U postgres -d production

-- Vérifier l'heure de récupération
SELECT pg_last_wal_replay_lsn();

-- Vérifier vos données
SELECT count(*) FROM clients;
```

#### Étape 9 : Promouvoir (si en mode pause)

Si `recovery_target_action = 'pause'` :
```sql
SELECT pg_wal_replay_resume();
-- Ou
SELECT pg_promote();
```

### Scénario 2 : Restauration au Dernier Point Cohérent

**Contexte** : Perte totale du serveur, restaurer le maximum de données.

**Configuration** :
```conf
restore_command = 'cp /mnt/wal_archive/%f %p'
# Pas de recovery_target_* = restaurer jusqu'au dernier WAL disponible
recovery_target_action = 'promote'
```

### Scénario 3 : Restauration Avant une Transaction Spécifique

**Contexte** : Une transaction problématique identifiée (XID).

#### Trouver le XID

```sql
-- Sur la base corrompue (si accessible)
SELECT txid_current();

-- Ou dans les logs
grep "transaction" /var/log/postgresql/*.log
```

**Configuration** :
```conf
restore_command = 'cp /mnt/wal_archive/%f %p'  
recovery_target_xid = '12345678'  # XID à éviter  
recovery_target_inclusive = false  # Exclure cette transaction  
recovery_target_action = 'promote'  
```

### Scénario 4 : Restauration jusqu'à un Point de Sauvegarde Nommé

**Contexte** : Vous avez créé des points de sauvegarde avant des opérations risquées.

#### Créer un Point de Sauvegarde

```sql
-- Avant une opération risquée
BEGIN;  
SAVEPOINT before_migration;  

-- Opération risquée
UPDATE ...;

-- Si tout va bien
COMMIT;
```

Ou au niveau base de données :
```sql
SELECT pg_create_restore_point('before_major_update');
```

**Configuration de restauration** :
```conf
restore_command = 'cp /mnt/wal_archive/%f %p'  
recovery_target_name = 'before_major_update'  
recovery_target_action = 'promote'  
```

### Scénario 5 : Restauration en Mode Pause pour Investigation

**Contexte** : Vous voulez inspecter la base avant de la promouvoir.

**Configuration** :
```conf
restore_command = 'cp /mnt/wal_archive/%f %p'  
recovery_target_time = '2025-11-22 14:36:59'  
recovery_target_action = 'pause'  
```

**Après démarrage** :
```sql
-- Connexion en lecture seule possible
SELECT * FROM clients WHERE id = 123;

-- Si satisfait, promouvoir
SELECT pg_wal_replay_resume();  
SELECT pg_promote();  

-- Si pas satisfait, arrêter et réessayer avec autre timestamp
```

## Paramètres Avancés de Récupération

### Paramètres de Ciblage

```conf
# Restaurer jusqu'à un timestamp
recovery_target_time = '2025-11-22 14:30:00'

# Restaurer jusqu'à une transaction ID
recovery_target_xid = '12345678'

# Restaurer jusqu'à un point nommé
recovery_target_name = 'before_migration'

# Restaurer jusqu'à un LSN (Log Sequence Number)
recovery_target_lsn = '0/42000028'

# Restaurer jusqu'à la fin disponible (défaut si aucun target)
# (Pas de paramètre = restaurer tout ce qui est possible)

# Mode "immediate" : S'arrêter dès que consistant
recovery_target = 'immediate'
```

### Contrôle d'Inclusion

```conf
# Inclure ou exclure la transaction/point cible
recovery_target_inclusive = true   # Inclure (défaut)  
recovery_target_inclusive = false  # Exclure  
```

**Exemple d'usage** :
```conf
# Arrêter AVANT une transaction problématique
recovery_target_xid = '12345678'  
recovery_target_inclusive = false  
```

### Actions Post-Récupération

```conf
# Promouvoir automatiquement
recovery_target_action = 'promote'

# Mettre en pause (inspection possible)
recovery_target_action = 'pause'

# Arrêter PostgreSQL
recovery_target_action = 'shutdown'
```

### Timeline

PostgreSQL utilise le concept de **timeline** pour gérer les divergences après PITR.

```conf
# Suivre une timeline spécifique
recovery_target_timeline = '2'

# Suivre la timeline la plus récente (défaut)
recovery_target_timeline = 'latest'
```

**Explication** :
- Chaque fois que vous faites un PITR, une nouvelle timeline est créée
- Cela évite de "mélanger" les WAL d'une ancienne réalité avec une nouvelle

**Exemple** :
```
Timeline 1 : Production normale
    ├─ Incident à 14h37
    └─ PITR vers 14h36 → Crée Timeline 2
         └─ Nouvelles transactions sur Timeline 2
```

## Outils et Commandes Utiles

### Inspection des WAL

#### pg_waldump : Lire le Contenu d'un WAL

```bash
# Lire un fichier WAL spécifique
pg_waldump /var/lib/postgresql/18/main/pg_wal/000000010000000000000042

# Lire avec filtre sur une table
pg_waldump -r heap 000000010000000000000042

# Chercher des transactions spécifiques
pg_waldump 000000010000000000000042 | grep "COMMIT"

# Statistiques
pg_waldump -z 000000010000000000000042
```

**Exemple de sortie** :
```
rmgr: Heap        len (rec/tot):     59/    59, tx:       1234, lsn: 0/42001A78, prev 0/42001A40, desc: INSERT off 3, blkref #0: rel 1663/16384/16385 blk 0  
rmgr: Transaction len (rec/tot):     34/    34, tx:       1234, lsn: 0/42001AB8, prev 0/42001A78, desc: COMMIT 2025-11-22 14:35:42.123456 CET  
```

#### pg_controldata : Informations de Contrôle

```bash
pg_controldata /var/lib/postgresql/18/main
```

**Informations utiles** :
- Latest checkpoint location (LSN)
- Latest checkpoint's REDO location
- Latest checkpoint's TimeLineID
- Min recovery ending location

### Commandes SQL Utiles

#### Forcer un Switch de WAL

```sql
-- Forcer PostgreSQL à fermer le WAL actuel et en créer un nouveau
SELECT pg_switch_wal();
```

**Usage** : Avant une sauvegarde ou pour tester l'archivage.

#### Créer un Point de Restauration

```sql
SELECT pg_create_restore_point('before_risky_operation');
```

#### Obtenir la Position WAL Actuelle

```sql
-- Position actuelle d'écriture
SELECT pg_current_wal_lsn();

-- Position de replay (sur un standby ou en recovery)
SELECT pg_last_wal_replay_lsn();

-- Différence entre deux LSN
SELECT pg_wal_lsn_diff('0/43000000', '0/42000000');
-- Retourne : 16777216 (16 MB)
```

#### Informations sur l'Archivage

```sql
-- Statistiques d'archivage
SELECT * FROM pg_stat_archiver;

-- Fichiers WAL en attente d'archivage
SELECT * FROM pg_stat_wal;

-- Configuration actuelle
SHOW archive_mode;  
SHOW archive_command;  
```

### Timeline Management

```sql
-- Voir les timelines disponibles
SELECT * FROM pg_controldata_timeline();

-- Informations sur la timeline actuelle
SHOW timeline_id;
```

## Automatisation et Scripts

### Script Complet de Configuration PITR

```bash
#!/bin/bash
# setup_pitr.sh - Configuration complète PITR

set -e  # Arrêt en cas d'erreur

# Configuration
WAL_ARCHIVE_DIR="/mnt/wal_archive"  
BACKUP_DIR="/backups/pitr"  
PG_DATA="/var/lib/postgresql/18/main"  
PG_CONF="${PG_DATA}/postgresql.conf"  

echo "=== Configuration PostgreSQL PITR ==="

# 1. Créer les répertoires
echo "Création des répertoires..."  
sudo mkdir -p ${WAL_ARCHIVE_DIR}  
sudo mkdir -p ${BACKUP_DIR}  

sudo chown postgres:postgres ${WAL_ARCHIVE_DIR}  
sudo chown postgres:postgres ${BACKUP_DIR}  
sudo chmod 700 ${WAL_ARCHIVE_DIR}  
sudo chmod 700 ${BACKUP_DIR}  

# 2. Configurer postgresql.conf
echo "Configuration de postgresql.conf..."  
sudo tee -a ${PG_CONF} << EOF  

# === PITR Configuration ===
wal_level = replica  
max_wal_senders = 5  
wal_keep_size = 1GB  
archive_mode = on  
archive_command = 'test ! -f ${WAL_ARCHIVE_DIR}/%f && cp %p ${WAL_ARCHIVE_DIR}/%f'  
archive_timeout = 300  
EOF  

# 3. Redémarrer PostgreSQL
echo "Redémarrage de PostgreSQL..."  
sudo systemctl restart postgresql  

# 4. Vérifier la configuration
echo "Vérification..."  
sleep 3  
psql -U postgres -c "SHOW wal_level;"  
psql -U postgres -c "SHOW archive_mode;"  
psql -U postgres -c "SELECT pg_switch_wal();"  
sleep 2  
psql -U postgres -c "SELECT * FROM pg_stat_archiver;"  

# 5. Créer la première sauvegarde de base
echo "Création de la première sauvegarde de base..."  
DATE=$(date +%Y%m%d_%H%M)  
pg_basebackup -D ${BACKUP_DIR}/base_${DATE} -U postgres -X stream -c fast -P  

echo "✅ Configuration PITR terminée avec succès !"  
echo "Archive WAL : ${WAL_ARCHIVE_DIR}"  
echo "Sauvegardes : ${BACKUP_DIR}"  
echo "Première sauvegarde : ${BACKUP_DIR}/base_${DATE}"  
```

### Script de Restauration PITR

```bash
#!/bin/bash
# restore_pitr.sh - Restauration PITR automatisée

set -e

# Configuration
PG_DATA="/var/lib/postgresql/18/main"  
WAL_ARCHIVE_DIR="/mnt/wal_archive"  
BACKUP_DIR="/backups/pitr"  
TARGET_TIME=""  

# Aide
usage() {
    echo "Usage: $0 -t 'YYYY-MM-DD HH:MM:SS' [-b backup_path]"
    echo "Exemple: $0 -t '2025-11-22 14:36:59'"
    exit 1
}

# Parse arguments
while getopts "t:b:h" opt; do
    case ${opt} in
        t) TARGET_TIME=$OPTARG ;;
        b) BACKUP_PATH=$OPTARG ;;
        h) usage ;;
        *) usage ;;
    esac
done

if [ -z "${TARGET_TIME}" ]; then
    echo "Erreur: Target time requis"
    usage
fi

# Trouver la dernière sauvegarde si non spécifiée
if [ -z "${BACKUP_PATH}" ]; then
    BACKUP_PATH=$(ls -td ${BACKUP_DIR}/base_* | head -1)
    echo "Utilisation de la dernière sauvegarde: ${BACKUP_PATH}"
fi

if [ ! -d "${BACKUP_PATH}" ]; then
    echo "Erreur: Backup path invalide: ${BACKUP_PATH}"
    exit 1
fi

# Confirmation
echo "=== Restauration PITR ==="  
echo "Backup source: ${BACKUP_PATH}"  
echo "Target time: ${TARGET_TIME}"  
echo "WAL archive: ${WAL_ARCHIVE_DIR}"  
read -p "Continuer? (yes/no): " CONFIRM  

if [ "${CONFIRM}" != "yes" ]; then
    echo "Annulé"
    exit 0
fi

# 1. Arrêter PostgreSQL
echo "Arrêt de PostgreSQL..."  
sudo systemctl stop postgresql  

# 2. Sauvegarder les données actuelles
echo "Sauvegarde des données actuelles..."  
sudo mv ${PG_DATA} ${PG_DATA}.before_pitr_$(date +%Y%m%d_%H%M%S)  

# 3. Restaurer la sauvegarde de base
echo "Restauration de la sauvegarde de base..."  
sudo cp -a ${BACKUP_PATH} ${PG_DATA}  

# 4. Créer recovery.signal
echo "Configuration de la récupération..."  
sudo touch ${PG_DATA}/recovery.signal  

# 5. Configurer les paramètres de récupération
sudo tee ${PG_DATA}/recovery.conf << EOF  
restore_command = 'cp ${WAL_ARCHIVE_DIR}/%f %p'  
recovery_target_time = '${TARGET_TIME}'  
recovery_target_action = 'promote'  
EOF  

# 6. Permissions
sudo chown -R postgres:postgres ${PG_DATA}  
sudo chmod 700 ${PG_DATA}  

# 7. Démarrer PostgreSQL
echo "Démarrage de PostgreSQL en mode recovery..."  
sudo systemctl start postgresql  

# 8. Monitoring
echo "=== Surveillance de la récupération ==="  
echo "Logs: sudo tail -f /var/log/postgresql/postgresql-18-main.log"  
echo "État: psql -U postgres -c \"SELECT pg_is_in_recovery();\""  
```

### Script de Monitoring PITR

```bash
#!/bin/bash
# monitor_pitr.sh - Surveillance continue du système PITR

# Configuration
WAL_ARCHIVE_DIR="/mnt/wal_archive"  
ALERT_EMAIL="admin@company.com"  

# Fonction d'alerte
alert() {
    local MESSAGE=$1
    echo "[ALERT] ${MESSAGE}"
    echo "${MESSAGE}" | mail -s "PostgreSQL PITR Alert" ${ALERT_EMAIL}
}

# 1. Vérifier l'archivage WAL
echo "Vérification de l'archivage WAL..."  
FAILED_COUNT=$(psql -U postgres -t -c "SELECT failed_count FROM pg_stat_archiver;" | tr -d ' ')  
LAST_ARCHIVE=$(psql -U postgres -t -c "SELECT EXTRACT(EPOCH FROM (now() - last_archived_time)) FROM pg_stat_archiver;" | tr -d ' ')  

if [ ${FAILED_COUNT} -gt 0 ]; then
    alert "Archivage WAL échoué: ${FAILED_COUNT} échecs"
fi

if (( $(echo "${LAST_ARCHIVE} > 3600" | bc -l) )); then
    alert "Aucun archivage depuis ${LAST_ARCHIVE} secondes"
fi

# 2. Vérifier l'espace disque archive
echo "Vérification de l'espace disque..."  
DISK_USAGE=$(df ${WAL_ARCHIVE_DIR} | tail -1 | awk '{print $5}' | sed 's/%//')  
if [ ${DISK_USAGE} -gt 80 ]; then  
    alert "Espace disque archive faible: ${DISK_USAGE}%"
fi

# 3. Vérifier l'ancienneté de la dernière sauvegarde
LAST_BACKUP=$(find /backups/pitr -name "base_*" -type d | sort | tail -1)  
if [ -n "${LAST_BACKUP}" ]; then  
    BACKUP_AGE=$(( ($(date +%s) - $(stat -c %Y ${LAST_BACKUP})) / 86400 ))
    if [ ${BACKUP_AGE} -gt 7 ]; then
        alert "Dernière sauvegarde de base trop ancienne: ${BACKUP_AGE} jours"
    fi
fi

# 4. Vérifier le taux de génération WAL
WAL_SIZE=$(du -s --block-size=1M ${WAL_ARCHIVE_DIR} | cut -f1)  
echo "Taille archive WAL: ${WAL_SIZE} MB"  

echo "✅ Monitoring terminé"
```

**Cron (toutes les heures)** :
```bash
0 * * * * /usr/local/bin/monitor_pitr.sh
```

## Bonnes Pratiques et Recommandations

### ✅ Configuration

1. **Toujours utiliser wal_level = replica minimum**
   - Permet réplication et PITR
   - Surcoût négligeable

2. **archive_timeout approprié**
   - Bases très actives : 60-300 secondes
   - Bases peu actives : 600-1800 secondes
   - Balance entre RPO et nombre de fichiers

3. **Commande d'archivage robuste**
   ```bash
   # Mauvais : peut écraser des fichiers
   archive_command = 'cp %p /archive/%f'

   # Bon : vérifie l'existence
   archive_command = 'test ! -f /archive/%f && cp %p /archive/%f'
   ```

4. **Redondance de l'archivage**
   - Local + distant
   - Plusieurs copies (3-2-1 rule)

### ✅ Sauvegardes

1. **Fréquence adaptée**
   - Production critique : Quotidienne
   - Base volumineuse : Hebdomadaire + WAL
   - Compromis : Temps de restauration vs espace

2. **Validation des sauvegardes**
   - Tester la restauration régulièrement
   - Automatiser les tests (environnement de staging)

3. **Documentation**
   - Procédures de restauration claires
   - Runbook accessible 24/7
   - Temps de restauration estimé

### ✅ Monitoring

1. **Surveiller pg_stat_archiver**  
   - `failed_count` doit rester à 0  
   - `last_archived_time` < 1 heure

2. **Alertes essentielles**
   - Échec d'archivage
   - Espace disque faible (archive et backup)
   - Sauvegarde de base trop ancienne

3. **Métriques à suivre**
   - Taux de génération WAL
   - Taille de l'archive
   - Durée des sauvegardes

### ✅ Opérations

1. **Tests réguliers**
   - Restauration complète : Mensuelle
   - Restauration PITR : Trimestrielle
   - Drill de disaster recovery : Annuel

2. **Gestion de l'espace**
   - Nettoyer les anciennes archives
   - Coordonner avec les sauvegardes de base
   - Utiliser pgBackRest/Barman pour automatisation

3. **Documentation des incidents**
   - Enregistrer chaque utilisation de PITR
   - Documenter les leçons apprises
   - Améliorer les procédures

### ❌ Pièges à Éviter

1. **Ne jamais supprimer manuellement des WAL**
   - Utiliser des outils (pgBackRest, Barman)
   - Ou scripts coordonnés avec sauvegardes

2. **Ne pas négliger l'espace disque**
   - WAL archives croissent vite
   - Monitoring proactif essentiel

3. **Ne pas oublier les objets globaux**
   - pg_basebackup ne sauvegarde pas les rôles
   - Faire aussi `pg_dumpall --globals-only`

4. **Ne pas tester uniquement en théorie**
   - La première restauration en production n'est pas le bon moment pour apprendre

5. **Ne pas mélanger les timelines**
   - Comprendre le concept de timeline
   - Utiliser `recovery_target_timeline` correctement

## Dépannage et Erreurs Courantes

### Erreur 1 : Archive Command Failed

```
archive command failed with exit code 1
```

**Causes** :
- Permissions insuffisantes
- Espace disque plein
- Répertoire d'archive inexistant
- Erreur réseau (archivage distant)

**Diagnostic** :
```bash
# Tester la commande manuellement
sudo -u postgres bash -c "cp /var/lib/postgresql/18/main/pg_wal/000000010000000000000001 /mnt/wal_archive/"

# Vérifier les permissions
ls -ld /mnt/wal_archive/  
ls -l /mnt/wal_archive/  

# Vérifier l'espace disque
df -h /mnt/wal_archive/
```

**Solutions** :
```bash
# Corriger les permissions
sudo chown postgres:postgres /mnt/wal_archive/  
sudo chmod 700 /mnt/wal_archive/  

# Libérer de l'espace
sudo find /mnt/wal_archive/ -mtime +30 -delete
```

### Erreur 2 : Could Not Restore File from Archive

Durant la restauration :
```
could not restore file "000000010000000000000042" from archive: return code 1
```

**Causes** :
- Fichier WAL manquant
- Chemin incorrect dans restore_command
- Permissions

**Solutions** :
```bash
# Vérifier la présence du fichier
ls -l /mnt/wal_archive/000000010000000000000042

# Tester restore_command manuellement
sudo -u postgres bash -c "cp /mnt/wal_archive/000000010000000000000042 /tmp/test"

# Vérifier le contenu de recovery.conf/postgresql.conf
cat /var/lib/postgresql/18/main/recovery.conf
```

### Erreur 3 : Timeline Mismatch

```
requested timeline 2 does not contain minimum recovery point
```

**Cause** : Tentative de restauration sur une timeline incompatible.

**Solution** :
```conf
# Spécifier la bonne timeline
recovery_target_timeline = '1'

# Ou utiliser la plus récente
recovery_target_timeline = 'latest'
```

### Erreur 4 : Recovery End Before Minimum Point

```
recovery ended before reaching consistency
```

**Cause** : Fichiers WAL manquants, sauvegarde incomplète.

**Solutions** :
- Utiliser une sauvegarde de base plus récente
- Vérifier que tous les WAL nécessaires sont disponibles
- Utiliser `recovery_target = 'immediate'` si vous acceptez une perte de données

### Erreur 5 : Requested WAL Segment Already Removed

```
requested WAL segment has already been removed
```

**Cause** : `wal_keep_size` trop petit, WAL recyclé avant archivage.

**Solutions** :
```conf
# Augmenter wal_keep_size
wal_keep_size = 2GB

# Utiliser un slot de réplication
SELECT pg_create_physical_replication_slot('backup_slot');
```

### Erreur 6 : Backup Label Not Found

```
backup label not found
```

**Cause** : La sauvegarde de base est incomplète ou corrompue.

**Solution** :
- Utiliser une autre sauvegarde de base
- Refaire une sauvegarde avec pg_basebackup

## Cas d'Usage Avancés

### 1. Restauration en Cascade

**Scénario** : Tester plusieurs points de restauration sans refaire la base backup.

```bash
# Restauration à T1
./restore_pitr.sh -t '2025-11-22 14:00:00'

# Sauvegarder cet état
cp -a /var/lib/postgresql/18/main /backups/test_T1

# Restauration à T2 (à partir de T1)
./restore_pitr.sh -t '2025-11-22 15:00:00'
```

### 2. Clone de Production pour Développement

**Scénario** : Créer un environnement de dev avec données récentes.

```bash
# Sur le serveur de production
pg_basebackup -D /tmp/prod_snapshot -X stream

# Transférer
rsync -avz /tmp/prod_snapshot/ dev-server:/var/lib/postgresql/18/main

# Sur le serveur de dev
cat > /var/lib/postgresql/18/main/recovery.conf << EOF  
restore_command = 'cp /mnt/wal_archive/%f %p'  
recovery_target_time = '2025-11-22 08:00:00'  # Début de journée  
recovery_target_action = 'promote'  
EOF  

touch /var/lib/postgresql/18/main/recovery.signal  
systemctl start postgresql  
```

### 3. Audit Trail Complet

**Scénario** : Investiguer qui a fait quoi et quand.

```bash
# Examiner les WAL pour une période spécifique
pg_waldump /mnt/wal_archive/000000010000000000000042 | \
    grep -A 5 "COMMIT" | \
    grep "2025-11-22 14:3"

# Restaurer à différents points pour comparer
for TIME in "14:35:00" "14:36:00" "14:37:00"; do
    ./restore_pitr.sh -t "2025-11-22 ${TIME}"
    psql -c "SELECT count(*) FROM clients;" > count_${TIME}.txt
done
```

### 4. Fork de Base de Données

**Scénario** : Créer deux versions divergentes pour A/B testing.

```bash
# Sauvegarde commune
pg_basebackup -D /backups/common_base

# Version A
cp -a /backups/common_base /var/lib/postgresql/18/variant_a
# Configuration spécifique...

# Version B
cp -a /backups/common_base /var/lib/postgresql/18/variant_b
# Configuration différente...
```

## Comparaison avec Autres Solutions

### PITR vs Streaming Replication

| Aspect | PITR | Streaming Replication |
|--------|------|----------------------|
| **Usage principal** | Disaster recovery | Haute disponibilité |
| **Récupération** | Minutes à heures | Secondes (promotion) |
| **Complexité** | Moyenne | Moyenne à élevée |
| **Coût** | Stockage archive | Serveur standby |
| **Point de restauration** | N'importe quand | Moment actuel ou dernier checkpoint |
| **Combinaison** | ✅ Recommandé ensemble | ✅ Recommandé ensemble |

**Recommandation** : Utiliser les deux !
- Réplication streaming pour HA (failover rapide)
- PITR pour erreurs logiques (suppression accidentelle)

### PITR vs Logical Replication

| Aspect | PITR | Logical Replication |
|--------|------|---------------------|
| **Granularité** | Cluster entier | Tables sélectionnées |
| **Compatibilité version** | Même majeure | Cross-version possible |
| **Cas d'usage** | Disaster recovery | Migration, consolidation |
| **Complexité** | Moyenne | Élevée |

### PITR vs Outils Tiers (Barman, pgBackRest)

**Avantages des outils tiers** :
- ✅ Gestion centralisée de multiples serveurs  
- ✅ Sauvegardes incrémentales  
- ✅ Compression avancée  
- ✅ Interface de gestion  
- ✅ Automatisation complète

**Avantages PITR natif** :
- ✅ Intégré à PostgreSQL  
- ✅ Pas de dépendance externe  
- ✅ Simplicité pour commencer  
- ✅ Bien documenté

**Recommandation** :
- Débutants / petits environnements : PITR natif
- Environnements complexes : pgBackRest ou Barman

## Conclusion

Le Point-In-Time Recovery (PITR) avec WAL archiving est une capacité essentielle pour toute installation PostgreSQL en production. Il offre :

- ✅ **Protection contre les erreurs logiques** : Suppression accidentelle, mauvaise mise à jour  
- ✅ **Récupération granulaire** : Restauration à la seconde près  
- ✅ **Flexibilité** : Multiples points de restauration possibles  
- ✅ **Complément à la HA** : Fonctionne avec la réplication streaming  
- ✅ **Audit trail** : Historique complet des modifications

**Configuration minimale viable** :
```conf
wal_level = replica  
archive_mode = on  
archive_command = 'test ! -f /archive/%f && cp %p /archive/%f'  
```

**Processus de base** :
1. Configurer l'archivage WAL  
2. Faire des sauvegardes de base régulières (pg_basebackup)  
3. Conserver les archives WAL  
4. En cas de besoin, restaurer base + replay WAL

**Recommandation finale** :

Pour un système de production robuste, combinez :
- **PITR** (ce chapitre) : Protection contre erreurs logiques  
- **Réplication Streaming** : Haute disponibilité  
- **Sauvegardes logiques** (pg_dump) : Portabilité  
- **Monitoring** : Détection proactive des problèmes

Cette approche multi-couches offre la meilleure protection possible contre tous les types de pannes et d'erreurs.
