# 16.13.1. Paramètres Critiques de Configuration PostgreSQL

## Introduction

La configuration de PostgreSQL est l'un des aspects les plus importants pour obtenir des performances optimales. Contrairement à d'autres SGBD qui tentent de s'auto-configurer, PostgreSQL adopte une approche conservatrice avec des valeurs par défaut très prudentes qui fonctionnent sur presque tous les systèmes, mais qui sont rarement optimales.

Dans cette section, nous allons explorer les quatre paramètres les plus critiques qui influencent directement les performances de votre base de données. Ces paramètres se trouvent dans le fichier `postgresql.conf` et peuvent être modifiés pour s'adapter aux ressources de votre serveur.

> **💡 Analogie** : Imaginez PostgreSQL comme un cuisinier dans une cuisine. Les paramètres de configuration définissent la taille de son plan de travail, la quantité d'ustensiles disponibles, et l'espace de stockage. Des paramètres mal configurés, c'est comme demander à un chef de préparer un banquet dans une kitchenette avec un seul couteau !

---

## Vue d'Ensemble des Paramètres

Avant d'entrer dans les détails, voici un tableau récapitulatif des quatre paramètres critiques :

| Paramètre | Rôle Principal | Valeur par Défaut | Impact |
|-----------|---------------|-------------------|--------|
| `shared_buffers` | Cache des données en mémoire partagée | 128 MB | ⭐⭐⭐⭐⭐ Critique |
| `work_mem` | Mémoire pour les opérations de tri/hachage | 4 MB | ⭐⭐⭐⭐ Très important |
| `maintenance_work_mem` | Mémoire pour les opérations de maintenance | 64 MB | ⭐⭐⭐ Important |
| `effective_cache_size` | Estimation du cache système | 4 GB | ⭐⭐⭐ Important |

---

## 1. shared_buffers : Le Cache Central

### Qu'est-ce que c'est ?

`shared_buffers` définit la quantité de mémoire RAM que PostgreSQL va utiliser pour **mettre en cache les données** lues depuis le disque. C'est la mémoire partagée entre tous les processus PostgreSQL.

### Comment ça fonctionne ?

Lorsqu'une requête a besoin de données :
1. PostgreSQL cherche d'abord dans les `shared_buffers` (cache en mémoire)  
2. Si les données sont trouvées → **Cache hit** (rapide ⚡)  
3. Si les données ne sont pas trouvées → **Cache miss** (lecture disque lente 🐌)

> **💡 Analogie** : C'est comme la différence entre prendre un livre sur votre bureau (cache hit) et devoir aller à la bibliothèque municipale pour le chercher (cache miss). L'un prend 2 secondes, l'autre 20 minutes !

### Valeurs Recommandées

| Type de Serveur | RAM Totale | Recommandation | Exemple |
|----------------|------------|----------------|---------|
| Serveur dédié PostgreSQL | 16 GB | 25% de la RAM | 4 GB |
| Serveur dédié PostgreSQL | 64 GB | 25% de la RAM | 16 GB |
| Serveur partagé | 16 GB | 15-20% de la RAM | 2-3 GB |
| Développement local | 8 GB | 512 MB - 1 GB | 1 GB |

**Règle d'or** : Entre **25% de la RAM** pour un serveur dédié, mais rarement au-delà de **32 GB**.

### Pourquoi ne pas mettre 80% de la RAM ?

C'est une erreur courante ! PostgreSQL s'appuie également sur le **cache du système d'exploitation** (page cache Linux). Donner trop de mémoire à `shared_buffers` peut être contre-productif :
- PostgreSQL doit gérer ce cache (overhead)
- Le système d'exploitation fait déjà du cache très efficacement
- Au-delà de 32 GB, les gains deviennent marginaux

### Configuration

```conf
# Dans postgresql.conf
shared_buffers = 4GB
```

**Attention** : Modifier ce paramètre nécessite un **redémarrage** de PostgreSQL.

---

## 2. work_mem : Mémoire de Travail par Opération

### Qu'est-ce que c'est ?

`work_mem` définit la quantité de mémoire maximum qu'une **opération individuelle** peut utiliser avant de devoir utiliser le disque. Ces opérations incluent :
- **Tri (ORDER BY, DISTINCT)**  
- **Hash tables (jointures hash, agrégations)**  
- **Bitmap operations**

### Comment ça fonctionne ?

Prenons l'exemple d'un tri :
1. Si la quantité de données à trier tient dans `work_mem` → **tri en mémoire** (rapide ⚡)  
2. Si elle ne tient pas → **tri sur disque** (utilisation de fichiers temporaires, très lent 🐌)

> **💡 Analogie** : Imaginez que vous devez trier un jeu de cartes. Si vous avez assez de place sur la table (work_mem), vous étalez toutes les cartes et triez rapidement. Sinon, vous devez créer des piles temporaires au sol (disque), c'est beaucoup plus lent !

### Le Piège Critique : Multiplication des work_mem

**⚠️ DANGER** : `work_mem` s'applique **par opération**, et une requête complexe peut utiliser plusieurs opérations simultanément.

Exemple concret :
```sql
SELECT
    category,
    AVG(price) as avg_price
FROM products  
WHERE status = 'active'  
ORDER BY category;  
```

Cette requête simple peut utiliser :
- 1× `work_mem` pour le tri (ORDER BY)
- 1× `work_mem` pour l'agrégation (AVG + GROUP BY implicite)

Si vous avez 100 connexions actives exécutant des requêtes complexes avec 3 opérations chacune :
- Mémoire potentielle = **100 connexions × 3 opérations × work_mem**
- Avec work_mem = 1 GB → **300 GB de RAM** nécessaires ! 💥

### Valeurs Recommandées

| Contexte | Recommandation | Explication |
|----------|----------------|-------------|
| OLTP (Transactions) | 4-16 MB | Requêtes simples, beaucoup de connexions |
| OLAP (Analytique) | 64-256 MB | Requêtes complexes, peu de connexions |
| Mixed Workload | 16-64 MB | Compromis équilibré |
| Data Warehouse | 256 MB - 1 GB | Gros volumes, jointures complexes |

**Formule de sécurité** :
```
work_mem_max = (RAM totale - shared_buffers) / (max_connections × 3)
```

Exemple pour un serveur avec 32 GB RAM, shared_buffers = 8 GB, max_connections = 100 :
```
work_mem_max = (32 GB - 8 GB) / (100 × 3) = 24 GB / 300 = 80 MB
```

### Stratégie Avancée : Ajustement par Session

Il est souvent plus sûr de garder une valeur conservatrice par défaut et d'augmenter `work_mem` pour les requêtes spécifiques qui en ont besoin :

```sql
-- Augmenter work_mem juste pour cette session
SET work_mem = '256MB';

-- Exécuter une grosse requête analytique
SELECT ... FROM huge_table ...;

-- work_mem reviendra à la valeur par défaut à la fin de la session
```

### Configuration

```conf
# Dans postgresql.conf
work_mem = 16MB  # Valeur conservatrice par défaut
```

**Avantage** : Ce paramètre peut être modifié **sans redémarrage** (rechargement de configuration suffit).

## 3. maintenance_work_mem : Mémoire pour la Maintenance

### Qu'est-ce que c'est ?

`maintenance_work_mem` est la mémoire allouée pour les **opérations de maintenance** :
- **CREATE INDEX** : Création d'index  
- **VACUUM** : Nettoyage et récupération d'espace  
- **ALTER TABLE** : Modifications de structure  
- **REINDEX** : Reconstruction d'index  
- **ANALYZE** : Collecte de statistiques

### Différence avec work_mem

| Critère | work_mem | maintenance_work_mem |
|---------|----------|---------------------|
| Usage | Requêtes normales | Opérations de maintenance |
| Fréquence | Très fréquent | Occasionnel |
| Nombre simultané | Potentiellement élevé | Généralement 1-2 |
| Impact si sous-dimensionné | Ralentissement requêtes | Maintenance très lente |

### Pourquoi c'est important ?

Une valeur trop faible peut **dramatiquement ralentir** les opérations de maintenance :
- Un `CREATE INDEX` sur une grosse table peut passer de 5 minutes à 2 heures
- Un `VACUUM` inefficace peut causer du bloat (gonflement des tables)

> **💡 Analogie** : C'est comme faire le ménage d'une maison avec un seul petit sac poubelle vs un grand conteneur. Avec le petit sac, vous devez faire 50 allers-retours, avec le conteneur, vous videz tout d'un coup !

### Valeurs Recommandées

| Contexte | RAM Serveur | Recommandation |
|----------|-------------|----------------|
| Petit serveur | 8 GB | 256 MB - 512 MB |
| Serveur moyen | 16-32 GB | 1 GB - 2 GB |
| Gros serveur | 64+ GB | 2 GB - 4 GB |

**Règle simple** : 5-10% de la RAM, mais rarement au-delà de **4 GB**.

### Cas d'Usage Critique : Autovacuum

PostgreSQL lance automatiquement des processus `autovacuum` en arrière-plan. Par défaut, il y a **3 workers autovacuum** qui peuvent tourner simultanément.

Chaque worker peut utiliser jusqu'à `maintenance_work_mem`, donc :
- Avec maintenance_work_mem = 2 GB et 3 workers → **6 GB** de RAM potentiellement utilisés

C'est pourquoi PostgreSQL a introduit un paramètre spécifique :

```conf
autovacuum_work_mem = 512MB  # Limite spécifique pour autovacuum
```

Si non défini, autovacuum utilise `maintenance_work_mem`.

### Configuration

```conf
# Dans postgresql.conf
maintenance_work_mem = 1GB  
autovacuum_work_mem = 512MB  # Optionnel, pour éviter les surprises  
```

**Avantage** : Peut être modifié **sans redémarrage**.

## 4. effective_cache_size : L'Indice pour le Planificateur

### Qu'est-ce que c'est ?

`effective_cache_size` est **une estimation** de la mémoire totale disponible pour le cache :
- Le cache de PostgreSQL (`shared_buffers`)
- Le cache du système d'exploitation (page cache)

⚠️ **Important** : Ce paramètre **n'alloue PAS de mémoire**. C'est juste un indice donné au planificateur de requêtes.

### Comment ça fonctionne ?

Le planificateur utilise cette valeur pour **décider des stratégies d'exécution** :

**Exemple** : Pour une requête avec un index :
- Si `effective_cache_size` est **faible** → Le planificateur pense que l'index ne sera probablement pas en cache → Préfère un **scan séquentiel**
- Si `effective_cache_size` est **élevé** → Le planificateur pense que l'index est probablement en cache → Préfère un **index scan**

> **💡 Analogie** : C'est comme dire à votre GPS s'il y a des embouteillages ou non. Le GPS (planificateur) va choisir un itinéraire différent en fonction de cette information, même si cette information ne change pas réellement le trafic.

### Impact d'une Mauvaise Valeur

| Si trop bas | Si trop haut |
|-------------|--------------|
| Plans de requête sous-optimaux | Peu d'impact négatif |
| Préférence excessive pour les scans séquentiels | Le planificateur peut parfois sur-estimer |
| Performances dégradées | Généralement sans danger |

**Bonne nouvelle** : Il vaut mieux surestimer que sous-estimer. Une valeur trop haute a peu d'impact négatif.

### Valeurs Recommandées

**Règle d'or** : **50-75% de la RAM totale**

| RAM Serveur | Recommandation | Calcul |
|-------------|----------------|--------|
| 8 GB | 4-6 GB | 50-75% |
| 16 GB | 8-12 GB | 50-75% |
| 32 GB | 16-24 GB | 50-75% |
| 64 GB | 32-48 GB | 50-75% |
| 128+ GB | 64-96 GB | 50-75% |

### Calcul Plus Précis

Si vous voulez être plus précis :
```
effective_cache_size = shared_buffers + (RAM libre pour OS cache)
```

Sur Linux, vous pouvez vérifier la taille du cache OS avec :
```bash
free -h
# Regarder la ligne "available" ou "buff/cache"
```

### Configuration

```conf
# Dans postgresql.conf
effective_cache_size = 12GB  # Pour un serveur avec 16 GB RAM
```

**Avantage** : Peut être modifié **sans redémarrage**.

## Tableau Récapitulatif : Exemple de Configuration

Voici un exemple complet pour différents types de serveurs :

### Serveur de Développement (8 GB RAM)
```conf
shared_buffers = 1GB              # 12.5% de la RAM  
work_mem = 8MB                    # Valeur conservatrice  
maintenance_work_mem = 256MB      # Pour les maintenances légères  
effective_cache_size = 4GB        # 50% de la RAM  
```

### Serveur OLTP Production (32 GB RAM)
```conf
shared_buffers = 8GB              # 25% de la RAM  
work_mem = 16MB                   # Conservateur (100+ connexions)  
maintenance_work_mem = 2GB        # Maintenances efficaces  
effective_cache_size = 20GB       # 62% de la RAM  
```

### Serveur OLAP / Data Warehouse (64 GB RAM)
```conf
shared_buffers = 16GB             # 25% de la RAM  
work_mem = 128MB                  # Requêtes complexes, peu de connexions  
maintenance_work_mem = 4GB        # Gros index à créer  
effective_cache_size = 48GB       # 75% de la RAM  
```

### Serveur Mixed Workload (32 GB RAM)
```conf
shared_buffers = 8GB              # 25% de la RAM  
work_mem = 32MB                   # Compromis équilibré  
maintenance_work_mem = 1GB        # Valeur intermédiaire  
effective_cache_size = 20GB       # 62% de la RAM  
```

## Vérifier l'Impact de vos Paramètres

### 1. Vérifier les Valeurs Actuelles

```sql
-- Afficher les paramètres actuels
SHOW shared_buffers;  
SHOW work_mem;  
SHOW maintenance_work_mem;  
SHOW effective_cache_size;  

-- Ou tous en même temps
SELECT name, setting, unit, source  
FROM pg_settings  
WHERE name IN ('shared_buffers', 'work_mem', 'maintenance_work_mem', 'effective_cache_size');  
```

### 2. Vérifier le Cache Hit Ratio (Taux de Succès du Cache)

Un bon indicateur de l'efficacité de `shared_buffers` :

```sql
SELECT
    sum(heap_blks_read) as heap_read,
    sum(heap_blks_hit)  as heap_hit,
    sum(heap_blks_hit) / (sum(heap_blks_hit) + sum(heap_blks_read)) AS cache_hit_ratio
FROM pg_statio_user_tables;
```

**Interprétation** :
- **> 99%** : Excellent 🟢  
- **95-99%** : Bon 🟡  
- **< 95%** : Problème possible 🔴 (augmenter shared_buffers ou optimiser les requêtes)

### 3. Détecter les Tris sur Disque

Si `work_mem` est trop petit, vous verrez des tris sur disque dans les logs :

```sql
-- Activer le logging des tris temporaires (dans postgresql.conf)
log_temp_files = 0  # Log tous les fichiers temporaires

-- Ou pour une session
SET log_temp_files = 0;
```

Dans les logs, vous verrez :
```
LOG: temporary file: path "base/pgsql_tmp/pgsql_tmp1234.0", size 52428800
```

Si vous voyez beaucoup de ces messages, augmentez `work_mem`.

## Les Erreurs Classiques à Éviter

### ❌ Erreur 1 : shared_buffers trop élevé
```conf
# MAUVAIS pour un serveur avec 16 GB RAM
shared_buffers = 12GB  # 75% de la RAM - TROP !
```
**Problème** : Le système d'exploitation n'a plus assez de RAM pour son propre cache.

**Solution** : Maximum 25-30% de la RAM.

### ❌ Erreur 2 : work_mem trop généreux
```conf
# DANGEREUX avec 200 connexions
work_mem = 512MB
```
**Problème** : 200 connexions × 3 opérations × 512 MB = **307 GB** de RAM potentiels !

**Solution** : Valeur conservatrice par défaut, augmentation par session si nécessaire.

### ❌ Erreur 3 : Ne pas ajuster effective_cache_size
```conf
# Garder la valeur par défaut sur un serveur de 64 GB
effective_cache_size = 4GB  # Beaucoup trop bas !
```
**Problème** : Le planificateur sous-estime le cache et fait de mauvais choix.

**Solution** : Toujours ajuster à 50-75% de la RAM.

### ❌ Erreur 4 : Oublier que shared_buffers nécessite un redémarrage
```conf
# Modifier shared_buffers dans postgresql.conf
shared_buffers = 8GB
```
```bash
# Recharger la configuration
pg_ctl reload  # ❌ NE SUFFIT PAS !
```
**Solution** : Redémarrage complet nécessaire :
```bash
pg_ctl restart
```

## Application des Changements

### Paramètres nécessitant un redémarrage
- ✅ `shared_buffers`

### Paramètres modifiables sans redémarrage (reload)
- ✅ `work_mem`  
- ✅ `maintenance_work_mem`  
- ✅ `effective_cache_size`

### Comment Appliquer les Changements

**Méthode 1 : Éditer postgresql.conf**
```bash
# Éditer le fichier
sudo nano /etc/postgresql/18/main/postgresql.conf

# Recharger (pour work_mem, maintenance_work_mem, effective_cache_size)
sudo systemctl reload postgresql

# Redémarrer (pour shared_buffers)
sudo systemctl restart postgresql
```

**Méthode 2 : Via SQL (temporaire pour la session)**
```sql
-- S'applique uniquement à la session courante
SET work_mem = '128MB';  
SET maintenance_work_mem = '2GB';  
```

**Méthode 3 : ALTER SYSTEM (permanent)**
```sql
-- Modifie postgresql.auto.conf (prioritaire sur postgresql.conf)
ALTER SYSTEM SET work_mem = '64MB';  
SELECT pg_reload_conf();  -- Recharger la configuration  
```

## Points Clés à Retenir

1. **shared_buffers** : Le cache central de PostgreSQL
   - 25% de la RAM pour un serveur dédié
   - Nécessite un redémarrage
   - Maximum 32 GB dans la plupart des cas

2. **work_mem** : Mémoire par opération
   - Attention à la multiplication (connexions × opérations)
   - Valeur conservatrice par défaut (4-16 MB)
   - Augmentation par session pour les grosses requêtes

3. **maintenance_work_mem** : Pour la maintenance
   - 5-10% de la RAM
   - Maximum 4 GB généralement suffisant
   - Impact majeur sur CREATE INDEX et VACUUM

4. **effective_cache_size** : Indice pour le planificateur
   - 50-75% de la RAM totale
   - N'alloue pas de mémoire réelle
   - Mieux vaut surestimer que sous-estimer

> **🎯 Règle d'or finale** : Commencez avec des valeurs conservatrices, surveillez les métriques (cache hit ratio, fichiers temporaires), puis ajustez progressivement en fonction de vos besoins réels.
