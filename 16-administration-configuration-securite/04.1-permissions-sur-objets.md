# 16.4.1. Permissions sur Objets (TABLE, SEQUENCE, FUNCTION)

## Introduction

Dans PostgreSQL, la gestion des permissions sur les objets de la base de données est essentielle pour assurer la sécurité et contrôler l'accès aux données. Après avoir créé des utilisateurs (rôles) et leur avoir donné accès à une base de données, il faut définir précisément ce qu'ils peuvent faire sur chaque objet : lire des données, les modifier, exécuter des fonctions, etc.

Cette section explique comment accorder et révoquer des permissions sur trois types d'objets fondamentaux : les tables, les séquences et les fonctions.

---

## Concept de Base : GRANT et REVOKE

PostgreSQL utilise deux commandes principales pour gérer les permissions :

- **GRANT** : Accorder des permissions à un rôle  
- **REVOKE** : Retirer des permissions à un rôle

### Syntaxe Générale

```sql
-- Accorder une permission
GRANT permission ON objet TO role;

-- Révoquer une permission
REVOKE permission ON objet FROM role;
```

### Principe Important : Les Permissions sont Additives

Par défaut, un rôle n'a **aucune permission** sur les objets qu'il ne possède pas. Il faut explicitement lui accorder chaque permission nécessaire. C'est le principe du **moindre privilège** : donner uniquement les droits nécessaires, pas plus.

---

## 1. Permissions sur les TABLES

Les tables sont les objets les plus courants dans une base de données. PostgreSQL propose plusieurs niveaux de permissions pour contrôler l'accès aux données.

### 1.1. Les Différents Types de Permissions

| Permission | Description | Cas d'usage |
|------------|-------------|-------------|
| **SELECT** | Lire les données (requêtes SELECT) | Utilisateurs en lecture seule, rapports, dashboards |
| **INSERT** | Insérer de nouvelles lignes | Applications qui créent des enregistrements |
| **UPDATE** | Modifier des lignes existantes | Applications qui mettent à jour des données |
| **DELETE** | Supprimer des lignes | Applications qui gèrent la suppression de données |
| **TRUNCATE** | Vider complètement une table | Maintenance, réinitialisation de données |
| **REFERENCES** | Créer des clés étrangères référençant cette table | Modélisation de relations entre tables |
| **TRIGGER** | Créer des triggers sur la table | Automatisation avancée, audit |
| **ALL PRIVILEGES** | Toutes les permissions ci-dessus | Propriétaire ou administrateur de la table |

### 1.2. Exemples Pratiques

#### Exemple 1 : Accorder la Lecture Seule

Imaginons un utilisateur `lecteur` qui doit uniquement consulter les données de la table `clients` :

```sql
-- Créer l'utilisateur
CREATE ROLE lecteur LOGIN PASSWORD 'mot_de_passe_securise';

-- Accorder la permission de lecture
GRANT SELECT ON TABLE clients TO lecteur;
```

Maintenant, `lecteur` peut faire :
```sql
SELECT * FROM clients;  
SELECT nom, email FROM clients WHERE ville = 'Paris';  
```

Mais il **ne peut pas** faire :
```sql
INSERT INTO clients VALUES (...);  -- ❌ Erreur : permission refusée  
UPDATE clients SET email = '...';  -- ❌ Erreur : permission refusée  
DELETE FROM clients WHERE id = 1;  -- ❌ Erreur : permission refusée  
```

#### Exemple 2 : Accorder Lecture et Écriture

Un utilisateur `application` doit pouvoir lire et modifier les données :

```sql
-- Créer l'utilisateur
CREATE ROLE application LOGIN PASSWORD 'mot_de_passe_securise';

-- Accorder lecture, insertion et mise à jour
GRANT SELECT, INSERT, UPDATE ON TABLE clients TO application;
```

Maintenant, `application` peut :
- Lire les données (SELECT)
- Ajouter de nouveaux clients (INSERT)
- Modifier les clients existants (UPDATE)

Mais il **ne peut toujours pas** supprimer de données (DELETE).

#### Exemple 3 : Accorder Toutes les Permissions

Pour un administrateur qui doit avoir un contrôle total :

```sql
-- Créer l'administrateur
CREATE ROLE admin_app LOGIN PASSWORD 'mot_de_passe_securise';

-- Accorder toutes les permissions
GRANT ALL PRIVILEGES ON TABLE clients TO admin_app;
```

#### Exemple 4 : Permissions sur Plusieurs Tables

Accorder des permissions sur plusieurs tables en une seule commande :

```sql
-- Accorder SELECT sur plusieurs tables
GRANT SELECT ON TABLE clients, commandes, produits TO lecteur;

-- Accorder plusieurs permissions sur plusieurs tables
GRANT SELECT, INSERT, UPDATE ON TABLE clients, commandes TO application;
```

#### Exemple 5 : Permissions sur Toutes les Tables d'un Schéma

Pour accorder des permissions sur toutes les tables d'un schéma :

```sql
-- Accorder SELECT sur toutes les tables du schéma public
GRANT SELECT ON ALL TABLES IN SCHEMA public TO lecteur;

-- Accorder plusieurs permissions sur toutes les tables
GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO application;
```

### 1.3. Permissions au Niveau des Colonnes

PostgreSQL permet également de limiter les permissions à certaines colonnes spécifiques d'une table. C'est utile pour protéger des données sensibles.

#### Exemple : Masquer des Colonnes Sensibles

Imaginons une table `employes` avec des informations sensibles :

```sql
CREATE TABLE employes (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100),
    prenom VARCHAR(100),
    email VARCHAR(150),
    salaire NUMERIC(10, 2),  -- Donnée sensible
    numero_securite_sociale VARCHAR(15)  -- Donnée très sensible
);
```

Un utilisateur RH doit voir les noms et emails, mais pas les salaires :

```sql
-- Accorder SELECT uniquement sur certaines colonnes
GRANT SELECT (id, nom, prenom, email) ON TABLE employes TO utilisateur_rh;
```

Maintenant, `utilisateur_rh` peut faire :
```sql
SELECT id, nom, prenom, email FROM employes;  -- ✅ OK
```

Mais il **ne peut pas** faire :
```sql
SELECT salaire FROM employes;  -- ❌ Erreur : permission refusée  
SELECT * FROM employes;  -- ❌ Erreur : permission refusée (inclut des colonnes interdites)  
```

### 1.4. Révoquer des Permissions

Pour retirer des permissions précédemment accordées :

```sql
-- Révoquer une permission spécifique
REVOKE INSERT ON TABLE clients FROM application;

-- Révoquer plusieurs permissions
REVOKE INSERT, UPDATE ON TABLE clients FROM application;

-- Révoquer toutes les permissions
REVOKE ALL PRIVILEGES ON TABLE clients FROM application;
```

#### Option CASCADE

Lorsqu'on révoque une permission qui a été transmise à d'autres rôles (via GRANT OPTION, voir plus loin), on peut utiliser CASCADE :

```sql
-- Révoquer et supprimer les permissions dérivées
REVOKE SELECT ON TABLE clients FROM utilisateur CASCADE;

-- Révoquer sans affecter les permissions dérivées (par défaut)
REVOKE SELECT ON TABLE clients FROM utilisateur RESTRICT;
```

## 2. Permissions sur les SEQUENCES

Les séquences sont des objets qui génèrent des nombres uniques, souvent utilisées pour les colonnes auto-incrémentées (SERIAL, IDENTITY).

### 2.1. Pourquoi Gérer les Permissions sur les Séquences ?

Lorsque vous créez une colonne avec `SERIAL` ou `IDENTITY`, PostgreSQL crée automatiquement une séquence en arrière-plan. Pour qu'un utilisateur puisse insérer des données dans une table avec une colonne auto-incrémentée, il doit avoir la permission **USAGE** sur cette séquence.

### 2.2. Les Permissions Disponibles

| Permission | Description | Cas d'usage |
|------------|-------------|-------------|
| **USAGE** | Utiliser la séquence (nextval, currval) | Nécessaire pour INSERT dans une table avec SERIAL |
| **SELECT** | Lire la valeur actuelle de la séquence | Monitoring, vérification |
| **UPDATE** | Modifier la séquence (setval) | Réinitialisation, maintenance |
| **ALL PRIVILEGES** | Toutes les permissions ci-dessus | Propriétaire ou administrateur |

### 2.3. Exemples Pratiques

#### Exemple 1 : Problème Courant - Permission Manquante

Créons une table avec une colonne SERIAL :

```sql
CREATE TABLE commandes (
    id SERIAL PRIMARY KEY,
    client_id INTEGER,
    montant NUMERIC(10, 2)
);

-- PostgreSQL crée automatiquement une séquence : commandes_id_seq
```

Un utilisateur avec INSERT sur la table, mais sans USAGE sur la séquence :

```sql
GRANT INSERT ON TABLE commandes TO application;

-- L'utilisateur essaie d'insérer
INSERT INTO commandes (client_id, montant) VALUES (1, 100.00);
-- ❌ Erreur : permission refusée pour la séquence commandes_id_seq
```

#### Exemple 2 : Solution - Accorder USAGE sur la Séquence

```sql
-- Accorder USAGE sur la séquence
GRANT USAGE ON SEQUENCE commandes_id_seq TO application;

-- Maintenant l'insertion fonctionne
INSERT INTO commandes (client_id, montant) VALUES (1, 100.00);  -- ✅ OK
```

#### Exemple 3 : Permissions sur Toutes les Séquences

Pour éviter ce problème, accordez USAGE sur toutes les séquences d'un schéma :

```sql
-- Accorder USAGE sur toutes les séquences du schéma
GRANT USAGE ON ALL SEQUENCES IN SCHEMA public TO application;
```

#### Exemple 4 : Réinitialiser une Séquence (UPDATE)

Un administrateur doit pouvoir réinitialiser une séquence :

```sql
-- Accorder UPDATE sur la séquence
GRANT UPDATE ON SEQUENCE commandes_id_seq TO admin_app;

-- L'administrateur peut maintenant réinitialiser
SELECT setval('commandes_id_seq', 1, false);  -- Redémarre à 1
```

### 2.4. Bonne Pratique : Permissions Automatiques

Pour simplifier la gestion, on peut configurer des permissions par défaut sur les nouvelles séquences (voir section sur ALTER DEFAULT PRIVILEGES plus loin).

## 3. Permissions sur les FUNCTIONS

Les fonctions (et procédures stockées) sont des blocs de code SQL ou PL/pgSQL qui peuvent être exécutés par les utilisateurs.

### 3.1. La Permission EXECUTE

Il n'y a qu'**une seule permission** sur les fonctions :

| Permission | Description | Cas d'usage |
|------------|-------------|-------------|
| **EXECUTE** | Exécuter la fonction | Permettre à un utilisateur d'appeler la fonction |

**Important** : Par défaut, PostgreSQL accorde automatiquement EXECUTE à PUBLIC (tous les utilisateurs) sur les nouvelles fonctions. C'est un comportement à surveiller pour la sécurité.

### 3.2. Exemples Pratiques

#### Exemple 1 : Créer et Exécuter une Fonction

Créons une fonction simple :

```sql
CREATE FUNCTION calculer_tva(montant_ht NUMERIC)  
RETURNS NUMERIC AS $$  
BEGIN  
    RETURN montant_ht * 1.20;  -- TVA à 20%
END;
$$ LANGUAGE plpgsql;
```

Par défaut, **tous les utilisateurs** peuvent l'exécuter :

```sql
-- N'importe quel utilisateur peut faire :
SELECT calculer_tva(100);  -- Retourne 120
```

#### Exemple 2 : Restreindre l'Accès à une Fonction

Pour limiter l'exécution à certains utilisateurs seulement :

```sql
-- Révoquer l'accès public
REVOKE EXECUTE ON FUNCTION calculer_tva(NUMERIC) FROM PUBLIC;

-- Accorder l'exécution à un utilisateur spécifique
GRANT EXECUTE ON FUNCTION calculer_tva(NUMERIC) TO application;
```

Maintenant :
- `application` peut exécuter la fonction ✅
- Les autres utilisateurs ne peuvent pas ❌

#### Exemple 3 : Fonction avec Sécurité DEFINER

Les fonctions peuvent être créées avec `SECURITY DEFINER`, ce qui signifie qu'elles s'exécutent avec les permissions de leur créateur, pas de l'appelant. C'est utile pour donner un accès contrôlé à des données sensibles.

```sql
-- Fonction qui accède à des données sensibles
CREATE FUNCTION obtenir_salaire_moyen()  
RETURNS NUMERIC  
SECURITY DEFINER  -- S'exécute avec les droits du créateur  
AS $$  
BEGIN  
    RETURN (SELECT AVG(salaire) FROM employes);
END;
$$ LANGUAGE plpgsql;

-- Révoquer l'accès public
REVOKE EXECUTE ON FUNCTION obtenir_salaire_moyen() FROM PUBLIC;

-- Accorder à un utilisateur spécifique
GRANT EXECUTE ON FUNCTION obtenir_salaire_moyen() TO manager;
```

Ici, `manager` peut obtenir le salaire moyen **sans avoir accès direct** à la table `employes`. La fonction agit comme une "porte contrôlée" vers les données sensibles.

#### Exemple 4 : Permissions sur Toutes les Fonctions

Pour accorder EXECUTE sur toutes les fonctions d'un schéma :

```sql
-- Accorder EXECUTE sur toutes les fonctions
GRANT EXECUTE ON ALL FUNCTIONS IN SCHEMA public TO application;
```

### 3.3. Révoquer des Permissions sur les Fonctions

```sql
-- Révoquer EXECUTE pour un utilisateur
REVOKE EXECUTE ON FUNCTION calculer_tva(NUMERIC) FROM application;

-- Révoquer pour tout le monde
REVOKE EXECUTE ON FUNCTION calculer_tva(NUMERIC) FROM PUBLIC;
```

## 4. L'Option WITH GRANT OPTION

Cette option permet à un utilisateur de **transmettre** les permissions qu'il a reçues à d'autres utilisateurs.

### 4.1. Syntaxe

```sql
GRANT permission ON objet TO role WITH GRANT OPTION;
```

### 4.2. Exemple Pratique

```sql
-- Créer un utilisateur "chef_projet"
CREATE ROLE chef_projet LOGIN PASSWORD 'mot_de_passe';

-- Lui accorder SELECT avec la possibilité de le transmettre
GRANT SELECT ON TABLE clients TO chef_projet WITH GRANT OPTION;

-- Maintenant, chef_projet peut donner SELECT à d'autres
-- (en se connectant avec ce rôle)
GRANT SELECT ON TABLE clients TO developpeur;
```

### 4.3. Cas d'Usage

Cette option est utile dans les organisations hiérarchiques où certains utilisateurs (managers, chefs de projet) doivent pouvoir gérer les permissions de leur équipe sans intervention d'un DBA.

⚠️ **Attention** : Cette option doit être utilisée avec précaution, car elle dilue le contrôle central des permissions.

## 5. Vérifier les Permissions Existantes

### 5.1. Commande \dp dans psql

Dans `psql`, la commande `\dp` (ou `\z`) affiche les permissions sur les objets :

```sql
\dp clients
```

Résultat exemple :
```
                              Access privileges
 Schema |  Name   | Type  |     Access privileges     | Column privileges
--------+---------+-------+---------------------------+-------------------
 public | clients | table | postgres=arwdDxt/postgres+|
        |         |       | lecteur=r/postgres       +|
        |         |       | application=arw/postgres  |
```

Légende des lettres :
- `a` = INSERT  
- `r` = SELECT (read)  
- `w` = UPDATE (write)  
- `d` = DELETE  
- `D` = TRUNCATE  
- `x` = REFERENCES  
- `t` = TRIGGER

### 5.2. Requêtes sur les Vues Système

Pour une analyse plus détaillée, on peut interroger les vues système :

```sql
-- Permissions sur une table spécifique
SELECT
    grantee,
    privilege_type
FROM information_schema.table_privileges  
WHERE table_name = 'clients';  
```

Résultat exemple :
```
  grantee   | privilege_type
------------+----------------
 lecteur    | SELECT
 application| SELECT
 application| INSERT
 application| UPDATE
```

### 5.3. Vérifier les Permissions d'un Rôle Spécifique

```sql
-- Toutes les permissions d'un rôle
SELECT
    table_schema,
    table_name,
    privilege_type
FROM information_schema.table_privileges  
WHERE grantee = 'application';  
```

## 6. Bonnes Pratiques

### 6.1. Principe du Moindre Privilège

Accordez **uniquement** les permissions nécessaires, rien de plus. Si un utilisateur n'a besoin que de lire les données, ne lui donnez que SELECT, pas UPDATE ou DELETE.

❌ **Mauvaise pratique** :
```sql
GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA public TO application;
```

✅ **Bonne pratique** :
```sql
-- Accorder uniquement ce qui est nécessaire
GRANT SELECT, INSERT, UPDATE ON TABLE clients TO application;  
GRANT SELECT ON TABLE produits TO application;  
```

### 6.2. Utiliser des Rôles Groupes

Au lieu d'accorder des permissions à chaque utilisateur individuellement, créez des rôles groupes :

```sql
-- Créer un rôle groupe
CREATE ROLE lecture_seule;

-- Accorder les permissions au groupe
GRANT SELECT ON ALL TABLES IN SCHEMA public TO lecture_seule;

-- Créer des utilisateurs et les ajouter au groupe
CREATE ROLE user1 LOGIN PASSWORD 'pass1';  
CREATE ROLE user2 LOGIN PASSWORD 'pass2';  

GRANT lecture_seule TO user1;  
GRANT lecture_seule TO user2;  
```

Avantages :
- Gestion centralisée
- Cohérence des permissions
- Facilité de maintenance

### 6.3. Révoquer PUBLIC par Défaut

PostgreSQL accorde automatiquement certaines permissions à PUBLIC. Pour une sécurité maximale, révoquez-les :

```sql
-- Révoquer les permissions par défaut sur les fonctions
REVOKE EXECUTE ON ALL FUNCTIONS IN SCHEMA public FROM PUBLIC;

-- Puis accorder sélectivement
GRANT EXECUTE ON FUNCTION ma_fonction() TO application;
```

### 6.4. Documenter les Permissions

Documentez les raisons des permissions accordées, soit dans un fichier de migration, soit en commentaires :

```sql
-- Rôle lecture_seule : Utilisé par les analystes BI pour les rapports
CREATE ROLE lecture_seule;  
GRANT SELECT ON ALL TABLES IN SCHEMA public TO lecture_seule;  

-- Rôle application : Utilisé par l'application web principale
CREATE ROLE application;  
GRANT SELECT, INSERT, UPDATE ON TABLE clients, commandes TO application;  
```

### 6.5. Audit Régulier des Permissions

Vérifiez régulièrement qui a accès à quoi :

```sql
-- Lister tous les utilisateurs et leurs permissions sur une table critique
SELECT
    grantee,
    string_agg(privilege_type, ', ') AS permissions
FROM information_schema.table_privileges  
WHERE table_name = 'donnees_sensibles'  
GROUP BY grantee;  
```

### 6.6. Automatiser avec ALTER DEFAULT PRIVILEGES

Pour que les permissions soient automatiquement accordées sur les nouveaux objets (voir section 16.4.3 pour les détails) :

```sql
-- Les nouvelles tables créées auront automatiquement ces permissions
ALTER DEFAULT PRIVILEGES IN SCHEMA public  
GRANT SELECT ON TABLES TO lecture_seule;  

ALTER DEFAULT PRIVILEGES IN SCHEMA public  
GRANT USAGE ON SEQUENCES TO application;  
```

## 7. Scénarios Pratiques

### Scénario 1 : Application Web Standard

Une application web avec :
- Un frontend qui lit des données
- Un backend qui lit et écrit

```sql
-- Rôle pour le backend
CREATE ROLE app_backend LOGIN PASSWORD 'secure_password';  
GRANT SELECT, INSERT, UPDATE, DELETE ON TABLE clients, commandes, produits TO app_backend;  
GRANT USAGE ON ALL SEQUENCES IN SCHEMA public TO app_backend;  

-- Rôle pour un service de reporting (lecture seule)
CREATE ROLE app_reporting LOGIN PASSWORD 'secure_password';  
GRANT SELECT ON ALL TABLES IN SCHEMA public TO app_reporting;  
```

### Scénario 2 : Environnement Multi-équipes

Une base de données partagée entre plusieurs équipes :

```sql
-- Équipe Marketing (lecture seule)
CREATE ROLE equipe_marketing;  
GRANT SELECT ON TABLE clients, campagnes TO equipe_marketing;  
CREATE ROLE user_marketing LOGIN PASSWORD 'pass';  
GRANT equipe_marketing TO user_marketing;  

-- Équipe Support (lecture + mise à jour limitée)
CREATE ROLE equipe_support;  
GRANT SELECT ON TABLE clients, tickets TO equipe_support;  
GRANT UPDATE (statut, commentaire) ON TABLE tickets TO equipe_support;  
CREATE ROLE user_support LOGIN PASSWORD 'pass';  
GRANT equipe_support TO user_support;  

-- Équipe Dev (accès complet en dev, limité en prod)
CREATE ROLE equipe_dev;
-- En dev : accès complet
GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA public TO equipe_dev;
-- En prod : seulement lecture sur la plupart des tables
-- (à adapter selon les besoins)
```

### Scénario 3 : API avec Fonctions SECURITY DEFINER

Une API qui expose des données via des fonctions, sans accès direct aux tables :

```sql
-- Les utilisateurs de l'API n'ont AUCUN accès direct aux tables
REVOKE ALL ON ALL TABLES IN SCHEMA public FROM app_api;

-- Créer des fonctions qui contrôlent l'accès
CREATE FUNCTION api_get_client(p_client_id INTEGER)  
RETURNS TABLE(id INTEGER, nom VARCHAR, email VARCHAR)  
SECURITY DEFINER  
AS $$  
BEGIN  
    -- Validation, logging, etc.
    RETURN QUERY
    SELECT id, nom, email FROM clients WHERE id = p_client_id;
END;
$$ LANGUAGE plpgsql;

-- Accorder uniquement EXECUTE sur les fonctions
GRANT EXECUTE ON FUNCTION api_get_client(INTEGER) TO app_api;
```

## Conclusion

La gestion des permissions sur les objets PostgreSQL (tables, séquences, fonctions) est un pilier de la sécurité et de la gouvernance des données. Les points clés à retenir :

- ✅ **Principe du moindre privilège** : N'accordez que ce qui est nécessaire  
- ✅ **Utilisez des rôles groupes** : Simplifiez la gestion  
- ✅ **N'oubliez pas les séquences** : Nécessaires pour les colonnes SERIAL  
- ✅ **Soyez prudent avec PUBLIC** : Révoquez les permissions par défaut sur les fonctions  
- ✅ **Auditez régulièrement** : Vérifiez qui a accès à quoi  
- ✅ **Documentez** : Expliquez pourquoi chaque permission existe

La maîtrise de ces concepts vous permettra de construire des systèmes sécurisés, maintenables et conformes aux bonnes pratiques de l'industrie.
