# 16.11.4. Stratégies de Sauvegarde 3-2-1

## Introduction : Pourquoi une Stratégie de Sauvegarde ?

Imaginez ces scénarios catastrophes :

**Scénario 1 : L'incendie** 🔥
```
Lundi : Vos sauvegardes PostgreSQL sont sur un disque externe...
        dans la même salle serveur que votre base de données
Mardi : Un incendie détruit la salle serveur  
Résultat : Base de données ET sauvegardes perdues  
```

**Scénario 2 : La corruption silencieuse** 💾
```
Janvier : Vous faites des sauvegardes régulières sur un seul disque  
Juin : Vous découvrez qu'une corruption s'est produite en février  
Problème : Toutes vos sauvegardes depuis février sont sur le MÊME disque corrompu  
```

**Scénario 3 : Le ransomware** 🦠
```
Vendredi 23h : Un ransomware infecte votre réseau  
Samedi 1h : Vos sauvegardes montées en NFS sont également chiffrées  
Résultat : Base de données ET sauvegardes inutilisables  
```

Ces scénarios ne sont pas de la science-fiction. Ils arrivent régulièrement à des entreprises du monde entier. La stratégie **3-2-1** a été conçue pour vous protéger contre tous ces risques.

---

## Le Concept 3-2-1

La règle **3-2-1** est une stratégie de sauvegarde universelle, simple à retenir et extrêmement efficace :

### Les Trois Règles d'Or

```
┌─────────────────────────────────────┐
│  3  copies de vos données           │
│  2  types de support différents     │
│  1  copie hors-site (off-site)      │
└─────────────────────────────────────┘
```

### Règle 1 : 3 Copies de Vos Données

**3 copies** signifie :
- **1 copie de production** (votre base de données en cours d'utilisation)  
- **2 sauvegardes** (copies de backup)

**Pourquoi 3 ?**
- 1 copie = Aucune protection
- 2 copies = Protection basique (mais une seule sauvegarde)
- 3 copies = Redondance réelle (si une sauvegarde échoue, vous avez encore une autre)

**Analogie** : Vous ne gardez pas un seul exemplaire d'un document important. Vous en faites des copies. La règle 3-2-1 dit : gardez AU MINIMUM 3 exemplaires.

**Exemple PostgreSQL** :
```
Copie 1 : Base de données de production sur serveur A  
Copie 2 : Sauvegarde quotidienne sur disque B  
Copie 3 : Sauvegarde hebdomadaire sur serveur de backup C  
```

### Règle 2 : 2 Types de Support Différents

**2 supports différents** signifie :
- Ne mettez pas toutes vos sauvegardes sur le même type de média
- Diversifiez les technologies de stockage

**Pourquoi 2 types ?**
- Protection contre les défaillances systématiques
- Un type de support peut avoir une vulnérabilité commune
- Diversification du risque

**Exemples de supports différents** :
- Disque dur interne **ET** disque dur externe
- SSD local **ET** HDD réseau (NAS)
- Stockage local **ET** stockage cloud
- Disque physique **ET** bande magnétique

**Mauvais exemples** (même type) :
- ❌ Deux disques durs dans le même serveur (même contrôleur, même alimentation)  
- ❌ Deux volumes sur le même SAN  
- ❌ Deux buckets S3 dans la même région AWS

**Bons exemples** (types différents) :
- ✅ Disque local + NAS distant  
- ✅ Disque local + Amazon S3  
- ✅ SSD serveur + Bande magnétique  
- ✅ Disque physique + Cloud (Azure, GCP)

**Exemple PostgreSQL** :
```
Support 1 : SSD local (sauvegarde pg_basebackup)  
Support 2 : NFS monté (archives WAL)  
Support 3 : Cloud S3 (copies hebdomadaires)  
```

### Règle 3 : 1 Copie Hors-Site (Off-Site)

**Hors-site** signifie :
- Au moins une sauvegarde dans un lieu **géographiquement séparé**
- Protection contre les catastrophes locales

**Pourquoi hors-site ?**
- Incendie, inondation, tremblement de terre
- Panne électrique généralisée
- Catastrophe naturelle
- Coupure réseau/internet locale

**Qu'est-ce qu'on entend par "hors-site" ?**

| Distance | Considéré hors-site ? | Risque |
|----------|----------------------|--------|
| Autre pièce dans le bâtiment | ❌ Non | Incendie, inondation |
| Autre bâtiment, même campus | ⚠️ Limite | Catastrophe naturelle |
| Autre ville (50+ km) | ✅ Oui | Protection adéquate |
| Autre région/pays | ✅✅ Excellent | Protection maximale |
| Cloud dans autre région | ✅✅ Excellent | Protection maximale |

**Exemples hors-site** :
- Datacenter distant (autre ville)
- Cloud public (AWS, Azure, GCP)
- Serveur de backup dans filiale distante
- Coffre-fort bancaire (bandes)
- Maison du directeur IT (pour petites structures)

**Exemple PostgreSQL** :
```
On-site (Paris) :
  - Production PostgreSQL
  - Sauvegarde locale quotidienne
  - Archives WAL locales

Off-site (Lyon) :
  - Réplication serveur standby
  - Archives WAL synchronisées

Off-site (Cloud - Irlande) :
  - Sauvegardes hebdomadaires sur S3
```

## Application du 3-2-1 à PostgreSQL

### Architecture 3-2-1 Basique

Voici comment implémenter la règle 3-2-1 pour PostgreSQL :

```
┌────────────────────────────────────────────────────────────┐
│                    COPIE 1 : PRODUCTION                    │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  PostgreSQL serveur principal (Paris)                │  │
│  │  - Base de données active                            │  │
│  │  - SSD local                                         │  │
│  └──────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────┘
                            │
        ┌───────────────────┼───────────────────┐
        ↓                   ↓                   ↓
┌────────────────┐  ┌────────────────┐  ┌────────────────┐
│   COPIE 2      │  │   COPIE 2      │  │   COPIE 3      │
│   (On-site)    │  │   (On-site)    │  │   (Off-site)   │
├────────────────┤  ├────────────────┤  ├────────────────┤
│ pg_basebackup  │  │ Archives WAL   │  │ Cloud S3       │
│ Quotidien      │  │ Continues      │  │ Hebdomadaire   │
│ Disque local   │  │ NAS/NFS        │  │ Autre région   │
│ Support: HDD   │  │ Support: NAS   │  │ Support: Cloud │
└────────────────┘  └────────────────┘  └────────────────┘
  (Même lieu)        (Même lieu)         (Lieu différent)
```

**Analyse** :
- ✅ **3 copies** : Production + 2 backups + 1 cloud  
- ✅ **2 supports** : Disque local, NAS, Cloud  
- ✅ **1 off-site** : S3 dans une autre région

### Implémentation Étape par Étape

#### Étape 1 : Sauvegarde Locale Quotidienne (Copie 2a)

**Objectif** : Récupération rapide en cas d'erreur

```bash
#!/bin/bash
# /usr/local/bin/backup_local.sh

DATE=$(date +%Y%m%d)  
BACKUP_DIR="/backups/local"  

# Sauvegarde physique avec pg_basebackup
pg_basebackup \
    -D ${BACKUP_DIR}/base_${DATE} \
    -U backup_user \
    -Ft -z -Z 6 \
    -X stream \
    -c fast \
    -P -v

# Rotation : garder 7 jours
find ${BACKUP_DIR} -name "base_*" -mtime +7 -exec rm -rf {} \;

echo "Sauvegarde locale terminée : ${BACKUP_DIR}/base_${DATE}"
```

**Cron : Tous les jours à 2h du matin**
```
0 2 * * * /usr/local/bin/backup_local.sh
```

#### Étape 2 : Archives WAL sur NAS (Copie 2b)

**Objectif** : PITR et récupération granulaire

**Configuration PostgreSQL** :
```conf
# postgresql.conf
archive_mode = on  
archive_command = 'test ! -f /mnt/nas/wal_archive/%f && cp %p /mnt/nas/wal_archive/%f'  
```

**Montage NAS** :
```bash
# /etc/fstab
nas-server:/exports/postgres_wal /mnt/nas nfs defaults,_netdev 0 0

# Monter
sudo mount /mnt/nas  
sudo chown postgres:postgres /mnt/nas/wal_archive  
```

#### Étape 3 : Sauvegarde Cloud Off-Site (Copie 3)

**Objectif** : Protection contre catastrophes locales

**Option A : Vers AWS S3**
```bash
#!/bin/bash
# /usr/local/bin/backup_cloud.sh

DATE=$(date +%Y%m%d)  
BACKUP_LOCAL="/backups/local/base_${DATE}"  
S3_BUCKET="s3://company-backups-postgres"  

# Vérifier que la sauvegarde locale existe
if [ ! -d "${BACKUP_LOCAL}" ]; then
    echo "Erreur : Sauvegarde locale non trouvée"
    exit 1
fi

# Copier vers S3
aws s3 sync ${BACKUP_LOCAL} ${S3_BUCKET}/${DATE}/ \
    --storage-class STANDARD_IA \
    --region eu-west-1

# Copier aussi les WAL récents
aws s3 sync /mnt/nas/wal_archive/ ${S3_BUCKET}/wal_archive/ \
    --region eu-west-1

echo "Sauvegarde cloud terminée : ${S3_BUCKET}/${DATE}"
```

**Cron : Tous les dimanches à 4h du matin**
```
0 4 * * 0 /usr/local/bin/backup_cloud.sh
```

**Option B : Vers Azure Blob Storage**
```bash
#!/bin/bash
# Copier vers Azure

az storage blob upload-batch \
    --account-name companybackups \
    --destination postgres-backups \
    --source /backups/local/base_${DATE} \
    --destination-path ${DATE}
```

**Option C : Vers Google Cloud Storage**
```bash
#!/bin/bash
# Copier vers GCS

gsutil -m rsync -r \
    /backups/local/base_${DATE} \
    gs://company-backups-postgres/${DATE}/
```

## Architectures 3-2-1 Avancées

### Architecture 1 : PME Standard

**Contexte** :
- Base de données < 100 GB
- RTO : 2 heures
- RPO : 1 heure
- Budget modéré

**Solution** :

```
Production (Serveur principal - Paris)
├─ PostgreSQL sur SSD
└─ Archives WAL locales

Copie 2a (Disque local)
├─ pg_basebackup quotidien
├─ Compression activée
└─ Rétention 7 jours

Copie 2b (NAS bureau)
├─ Archives WAL continues
├─ Montage NFS
└─ Nettoyage 30 jours

Copie 3 (Cloud S3 - Irlande)
├─ Sauvegardes hebdomadaires
├─ Classe STANDARD_IA
└─ Lifecycle policy : Glacier après 90 jours
```

**Coût estimé (mensuel)** :
- Disque local : Inclus dans infrastructure
- NAS : ~50€ (appareil + électricité)
- Cloud S3 : ~20€ (100 GB STANDARD_IA)
- **Total : ~70€/mois**

### Architecture 2 : Entreprise avec HA

**Contexte** :
- Base de données 500 GB - 2 TB
- RTO : 15 minutes
- RPO : < 1 minute
- Haute disponibilité requise

**Solution** :

```
Production Site A (Paris)
├─ PostgreSQL Primary
├─ SSD NVMe
└─ Archives WAL vers NAS local

Copie 2a (Standby Site A - Paris)
├─ Réplication streaming synchrone
├─ Hot Standby (lecture seule)
└─ Failover automatique (Patroni)

Copie 2b (Standby Site B - Lyon)
├─ Réplication streaming asynchrone
├─ Distant géographiquement (400 km)
└─ Disaster Recovery

Copie 3a (NAS Site A)
├─ Sauvegardes pg_basebackup depuis Standby
├─ Pas d'impact sur Primary
└─ Rétention 14 jours

Copie 3b (Cloud S3 - Multi-région)
├─ Réplication S3 cross-region
├─ Sauvegardes hebdomadaires complètes
├─ Archives WAL quotidiennes
└─ Lifecycle : STANDARD → IA → Glacier Deep Archive
```

**Avantages** :
- ✅ Failover < 30 secondes (Standby Site A)  
- ✅ Protection géographique (Site B)  
- ✅ PITR disponible (Archives WAL)  
- ✅ Aucun impact production (backup depuis Standby)

### Architecture 3 : Cloud Native

**Contexte** :
- Infrastructure entièrement cloud
- Base de données dans le cloud
- Multi-région requis

**Solution AWS** :

```
Production (us-east-1)
├─ RDS PostgreSQL ou Aurora
├─ Automated backups activés
└─ Point-in-time recovery

Copie 2 (us-east-1)
├─ RDS automated snapshots quotidiens
├─ Rétention 7 jours
└─ Même région

Copie 3a (us-west-2)
├─ Cross-region replica
├─ Read replica fonctionnel
└─ Peut être promu

Copie 3b (eu-west-1)
├─ Snapshots copiés vers autre région
├─ S3 bucket avec versioning
└─ Exports logiques mensuels (pg_dump)
```

**Solution Multi-Cloud** :

```
Production (AWS eu-west-1)
├─ RDS PostgreSQL

Copie 2 (AWS eu-west-1)
├─ Automated backups

Copie 3a (Azure West Europe)
├─ Sauvegardes exportées vers Azure Blob
├─ AzCopy pour synchronisation
└─ Geo-redundant storage

Copie 3b (Google Cloud europe-west1)
├─ Sauvegardes exportées vers GCS
├─ Multi-regional storage class
└─ Diversification fournisseur
```

## Scripts d'Automatisation Complets

### Script Master de Sauvegarde 3-2-1

```bash
#!/bin/bash
# /usr/local/bin/backup_321.sh
# Script complet implémentant stratégie 3-2-1

set -e  # Arrêt en cas d'erreur

# ============================================
# CONFIGURATION
# ============================================

DATE=$(date +%Y%m%d_%H%M)  
LOG_FILE="/var/log/postgresql/backup_321.log"  

# Copie 2a : Local
LOCAL_BACKUP_DIR="/backups/local"  
LOCAL_RETENTION_DAYS=7  

# Copie 2b : NAS
NAS_MOUNT="/mnt/nas"  
NAS_WAL_DIR="${NAS_MOUNT}/wal_archive"  

# Copie 3 : Cloud
CLOUD_PROVIDER="s3"  # Options: s3, azure, gcs  
S3_BUCKET="s3://company-backups-postgres"  
S3_REGION="eu-west-1"  

# Email alertes
ALERT_EMAIL="admin@company.com"

# ============================================
# FONCTIONS
# ============================================

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1" | tee -a ${LOG_FILE}
}

alert() {
    local MESSAGE=$1
    log "ALERT: ${MESSAGE}"
    echo "${MESSAGE}" | mail -s "PostgreSQL Backup 3-2-1 Alert" ${ALERT_EMAIL}
}

check_disk_space() {
    local MOUNT_POINT=$1
    local REQUIRED_GB=$2

    AVAILABLE_GB=$(df -BG ${MOUNT_POINT} | tail -1 | awk '{print $4}' | sed 's/G//')

    if [ ${AVAILABLE_GB} -lt ${REQUIRED_GB} ]; then
        alert "Insufficient disk space on ${MOUNT_POINT}: ${AVAILABLE_GB}GB available, ${REQUIRED_GB}GB required"
        return 1
    fi

    return 0
}

verify_backup() {
    local BACKUP_PATH=$1

    # Vérifier backup_label
    if [ ! -f "${BACKUP_PATH}/backup_label" ]; then
        alert "Backup verification failed: backup_label not found in ${BACKUP_PATH}"
        return 1
    fi

    # Vérifier PG_VERSION
    if [ ! -f "${BACKUP_PATH}/PG_VERSION" ]; then
        alert "Backup verification failed: PG_VERSION not found in ${BACKUP_PATH}"
        return 1
    fi

    log "✓ Backup verification passed: ${BACKUP_PATH}"
    return 0
}

# ============================================
# COPIE 2a : SAUVEGARDE LOCALE
# ============================================

backup_local() {
    log "=== Starting local backup (Copy 2a) ==="

    # Vérifier l'espace disque
    if ! check_disk_space ${LOCAL_BACKUP_DIR} 50; then
        return 1
    fi

    BACKUP_PATH="${LOCAL_BACKUP_DIR}/base_${DATE}"

    # Exécuter pg_basebackup
    log "Running pg_basebackup to ${BACKUP_PATH}"

    pg_basebackup \
        -D ${BACKUP_PATH} \
        -U backup_user \
        -Ft -z -Z 6 \
        -X stream \
        -c fast \
        -P -v \
        2>&1 | tee -a ${LOG_FILE}

    if [ ${PIPESTATUS[0]} -ne 0 ]; then
        alert "Local backup FAILED"
        return 1
    fi

    # Vérifier la sauvegarde
    # Note: Pour format tar, on vérifie juste la présence des fichiers
    if [ ! -f "${BACKUP_PATH}/base.tar.gz" ]; then
        alert "Local backup verification failed: base.tar.gz not found"
        return 1
    fi

    BACKUP_SIZE=$(du -sh ${BACKUP_PATH} | cut -f1)
    log "✓ Local backup completed successfully: ${BACKUP_PATH} (${BACKUP_SIZE})"

    # Rotation
    log "Cleaning local backups older than ${LOCAL_RETENTION_DAYS} days"
    find ${LOCAL_BACKUP_DIR} -name "base_*" -type d -mtime +${LOCAL_RETENTION_DAYS} -exec rm -rf {} \; 2>/dev/null

    echo ${BACKUP_PATH}
}

# ============================================
# COPIE 2b : ARCHIVES WAL SUR NAS
# ============================================

verify_wal_archiving() {
    log "=== Verifying WAL archiving (Copy 2b) ==="

    # Vérifier montage NAS
    if ! mountpoint -q ${NAS_MOUNT}; then
        alert "NAS not mounted at ${NAS_MOUNT}"
        return 1
    fi

    # Vérifier archivage PostgreSQL
    FAILED_COUNT=$(psql -U postgres -t -c "SELECT failed_count FROM pg_stat_archiver;" | tr -d ' ')
    LAST_ARCHIVE_SECONDS=$(psql -U postgres -t -c "SELECT EXTRACT(EPOCH FROM (now() - last_archived_time)) FROM pg_stat_archiver;" | tr -d ' ')

    if [ ${FAILED_COUNT} -gt 0 ]; then
        alert "WAL archiving has ${FAILED_COUNT} failures"
        return 1
    fi

    if (( $(echo "${LAST_ARCHIVE_SECONDS} > 3600" | bc -l) )); then
        alert "No WAL archived in the last hour (${LAST_ARCHIVE_SECONDS} seconds)"
        return 1
    fi

    # Vérifier espace disque NAS
    if ! check_disk_space ${NAS_MOUNT} 20; then
        return 1
    fi

    WAL_COUNT=$(ls -1 ${NAS_WAL_DIR} | wc -l)
    log "✓ WAL archiving operational: ${WAL_COUNT} files in archive"

    return 0
}

# ============================================
# COPIE 3 : SAUVEGARDE CLOUD OFF-SITE
# ============================================

backup_cloud_s3() {
    local SOURCE_PATH=$1

    log "=== Starting cloud backup to S3 (Copy 3) ==="

    if [ ! -d "${SOURCE_PATH}" ]; then
        alert "Source backup not found for cloud upload: ${SOURCE_PATH}"
        return 1
    fi

    CLOUD_PATH="${S3_BUCKET}/backups/${DATE}"

    log "Uploading to ${CLOUD_PATH}"

    aws s3 sync ${SOURCE_PATH} ${CLOUD_PATH}/ \
        --storage-class STANDARD_IA \
        --region ${S3_REGION} \
        2>&1 | tee -a ${LOG_FILE}

    if [ ${PIPESTATUS[0]} -ne 0 ]; then
        alert "Cloud backup to S3 FAILED"
        return 1
    fi

    # Copier aussi les archives WAL récentes (derniers 2 jours)
    log "Syncing recent WAL files to S3"
    find ${NAS_WAL_DIR} -name "*.wal" -mtime -2 -type f | while read WAL_FILE; do
        aws s3 cp ${WAL_FILE} ${S3_BUCKET}/wal_archive/ --region ${S3_REGION}
    done

    log "✓ Cloud backup to S3 completed: ${CLOUD_PATH}"

    return 0
}

backup_cloud_azure() {
    local SOURCE_PATH=$1

    log "=== Starting cloud backup to Azure (Copy 3) ==="

    CONTAINER="postgres-backups"
    BLOB_PREFIX="backups/${DATE}"

    az storage blob upload-batch \
        --account-name companybackups \
        --destination ${CONTAINER} \
        --destination-path ${BLOB_PREFIX} \
        --source ${SOURCE_PATH} \
        2>&1 | tee -a ${LOG_FILE}

    if [ ${PIPESTATUS[0]} -ne 0 ]; then
        alert "Cloud backup to Azure FAILED"
        return 1
    fi

    log "✓ Cloud backup to Azure completed"
    return 0
}

backup_cloud_gcs() {
    local SOURCE_PATH=$1

    log "=== Starting cloud backup to GCS (Copy 3) ==="

    GCS_PATH="gs://company-backups-postgres/backups/${DATE}"

    gsutil -m rsync -r ${SOURCE_PATH} ${GCS_PATH}/ \
        2>&1 | tee -a ${LOG_FILE}

    if [ ${PIPESTATUS[0]} -ne 0 ]; then
        alert "Cloud backup to GCS FAILED"
        return 1
    fi

    log "✓ Cloud backup to GCS completed: ${GCS_PATH}"
    return 0
}

backup_cloud() {
    local SOURCE_PATH=$1

    case ${CLOUD_PROVIDER} in
        s3)
            backup_cloud_s3 ${SOURCE_PATH}
            ;;
        azure)
            backup_cloud_azure ${SOURCE_PATH}
            ;;
        gcs)
            backup_cloud_gcs ${SOURCE_PATH}
            ;;
        *)
            alert "Unknown cloud provider: ${CLOUD_PROVIDER}"
            return 1
            ;;
    esac
}

# ============================================
# ORCHESTRATION
# ============================================

main() {
    log "========================================"
    log "Starting PostgreSQL 3-2-1 Backup Strategy"
    log "========================================"

    # Copie 2a : Sauvegarde locale
    BACKUP_PATH=$(backup_local)
    if [ $? -ne 0 ]; then
        alert "Backup strategy FAILED at local backup stage"
        exit 1
    fi

    # Copie 2b : Vérifier archives WAL
    if ! verify_wal_archiving; then
        alert "WAL archiving verification FAILED"
        # Continue quand même pour la sauvegarde cloud
    fi

    # Copie 3 : Sauvegarde cloud (seulement le dimanche)
    if [ $(date +%u) -eq 7 ]; then
        if ! backup_cloud ${BACKUP_PATH}; then
            alert "Cloud backup FAILED"
            exit 1
        fi
    else
        log "Skipping cloud backup (not Sunday)"
    fi

    log "========================================"
    log "✓✓✓ 3-2-1 Backup Strategy completed successfully"
    log "========================================"

    # Rapport de succès
    echo "PostgreSQL 3-2-1 backup completed successfully on $(date)" | \
        mail -s "Backup Success - 3-2-1" ${ALERT_EMAIL}
}

# Exécution
main
```

**Installation** :

```bash

# Tester

# Cron quotidien à 2h
```

### Script de Vérification 3-2-1

```bash
#!/bin/bash
# /usr/local/bin/verify_321.sh
# Vérifier que la stratégie 3-2-1 est respectée

# Configuration
LOCAL_BACKUP_DIR="/backups/local"  
NAS_WAL_DIR="/mnt/nas/wal_archive"  
S3_BUCKET="s3://company-backups-postgres"  

echo "=== Verification PostgreSQL 3-2-1 Strategy ==="  
echo  

# ============================================
# COPIE 1 : PRODUCTION
# ============================================

echo "1. Production Database (Copy 1)"  
if systemctl is-active --quiet postgresql; then  
    echo "   ✓ PostgreSQL is running"
    PG_VERSION=$(psql -U postgres -t -c "SELECT version();" | head -1 | awk '{print $2}')
    echo "   ✓ Version: ${PG_VERSION}"
    DB_SIZE=$(psql -U postgres -t -c "SELECT pg_size_pretty(pg_database_size('production'));" | tr -d ' ')
    echo "   ✓ Database size: ${DB_SIZE}"
else
    echo "   ✗ PostgreSQL is NOT running"
fi  
echo  

# ============================================
# COPIE 2a : SAUVEGARDE LOCALE
# ============================================

echo "2. Local Backup (Copy 2a)"  
if [ -d "${LOCAL_BACKUP_DIR}" ]; then  
    BACKUP_COUNT=$(find ${LOCAL_BACKUP_DIR} -name "base_*" -type d | wc -l)
    LATEST_BACKUP=$(find ${LOCAL_BACKUP_DIR} -name "base_*" -type d | sort | tail -1)

    if [ ${BACKUP_COUNT} -gt 0 ]; then
        echo "   ✓ ${BACKUP_COUNT} local backups found"
        echo "   ✓ Latest: ${LATEST_BACKUP}"

        BACKUP_AGE=$(( ($(date +%s) - $(stat -c %Y ${LATEST_BACKUP})) / 86400 ))
        echo "   ✓ Age: ${BACKUP_AGE} days"

        if [ ${BACKUP_AGE} -gt 2 ]; then
            echo "   ⚠ WARNING: Latest backup is older than 2 days"
        fi
    else
        echo "   ✗ No local backups found"
    fi
else
    echo "   ✗ Local backup directory does not exist"
fi  
echo  

# ============================================
# COPIE 2b : ARCHIVES WAL
# ============================================
```

### Script de Rotation Intelligent

```bash
#!/bin/bash
# /usr/local/bin/backup_rotation.sh
# Gestion intelligente de la rotation des sauvegardes

BACKUP_DIR="/backups/local"  
DATE=$(date +%Y%m%d)  
DAY_OF_WEEK=$(date +%u)  # 1=Lundi, 7=Dimanche  
DAY_OF_MONTH=$(date +%d)  

# Déterminer le type de sauvegarde à conserver
if [ "${DAY_OF_MONTH}" == "01" ]; then
    # Premier du mois : mensuelle
    BACKUP_TYPE="monthly"
    ln -sf ${BACKUP_DIR}/base_${DATE} ${BACKUP_DIR}/monthly_${DATE}
    echo "Sauvegarde mensuelle créée: monthly_${DATE}"

elif [ "${DAY_OF_WEEK}" == "7" ]; then
    # Dimanche : hebdomadaire
    BACKUP_TYPE="weekly"
    ln -sf ${BACKUP_DIR}/base_${DATE} ${BACKUP_DIR}/weekly_${DATE}
    echo "Sauvegarde hebdomadaire créée: weekly_${DATE}"

else
    # Autres jours : quotidienne
    BACKUP_TYPE="daily"
fi

# Rotation des sauvegardes quotidiennes (garder 7 jours)
find ${BACKUP_DIR} -name "base_*" -type d -mtime +7 -exec rm -rf {} \; 2>/dev/null

# Rotation des sauvegardes hebdomadaires (garder 4 semaines)
find ${BACKUP_DIR} -name "weekly_*" -type l -mtime +28 -delete 2>/dev/null

# Rotation des sauvegardes mensuelles (garder 12 mois)
find ${BACKUP_DIR} -name "monthly_*" -type l -mtime +365 -delete 2>/dev/null

echo "Rotation terminée"
```

## Calcul des Coûts et Dimensionnement

### Estimer les Besoins de Stockage

**Formule** :
```
Stockage total = Taille DB × Facteur compression × Nombre de copies × Rétention
```

**Exemple** :
```
Base de données : 200 GB  
Compression : 0.6 (40% de réduction)  
Copies locales : 7 jours × 120 GB = 840 GB  
Copies cloud : 4 semaines × 120 GB = 480 GB  
Total : ~1.3 TB  
```

### Calcul de Coûts Cloud (Exemple AWS)

```python
#!/usr/bin/env python3
# cost_calculator.py

def calculate_s3_costs(db_size_gb, retention_days, compression_ratio=0.6):
    """
    Calculer les coûts S3 pour stratégie 3-2-1
    """
    # Taille après compression
    compressed_size_gb = db_size_gb * compression_ratio

    # Nombre de sauvegardes complètes
    daily_backups = 7
    weekly_backups = 4
    monthly_backups = 12

    # Storage
    standard_storage = daily_backups * compressed_size_gb
    ia_storage = weekly_backups * compressed_size_gb
    glacier_storage = monthly_backups * compressed_size_gb

    # Prix par GB (us-east-1, novembre 2025)
    standard_price = 0.023  # $/GB
    ia_price = 0.0125       # $/GB
    glacier_price = 0.004   # $/GB

    # Calcul
    cost_standard = standard_storage * standard_price
    cost_ia = ia_storage * ia_price
    cost_glacier = glacier_storage * glacier_price

    total_cost = cost_standard + cost_ia + cost_glacier

    print(f"=== Cost Calculator ===")
    print(f"Database size: {db_size_gb} GB")
    print(f"Compressed size: {compressed_size_gb} GB")
    print(f"")
    print(f"Storage breakdown:")
    print(f"  STANDARD (7 days): {standard_storage:.1f} GB = ${cost_standard:.2f}/month")
    print(f"  STANDARD_IA (4 weeks): {ia_storage:.1f} GB = ${cost_ia:.2f}/month")
    print(f"  GLACIER (12 months): {glacier_storage:.1f} GB = ${cost_glacier:.2f}/month")
    print(f"")
    print(f"TOTAL: ${total_cost:.2f}/month (${total_cost * 12:.2f}/year)")

    return total_cost

# Exemples
print("Small DB (50 GB):")

print("\n" + "="*40 + "\n")
```

## Bonnes Pratiques et Recommandations

### ✅ Configuration et Planification

1. **Documenter votre stratégie**
   - Où sont les sauvegardes ?
   - Quel type pour quel usage ?
   - Qui a accès ?

2. **Automatiser complètement**
   - Aucune intervention manuelle
   - Cron/systemd timers
   - Orchestrateurs (Ansible, Kubernetes CronJob)

3. **Chiffrer les sauvegardes**
   - Au repos (encryption at rest)
   - En transit (TLS/SSL)
   - Clés gérées séparément

4. **Versioning et immuabilité**
   - S3 Versioning activé
   - S3 Object Lock pour protection ransomware
   - Azure Immutable Blob Storage

5. **Diversifier les fournisseurs cloud**
   - Multi-cloud pour haute résilience
   - Éviter le vendor lock-in

### ✅ Monitoring et Alertes

1. **Surveiller TOUTES les copies**
   - Ne pas se concentrer uniquement sur la production
   - Vérifier que chaque copie fonctionne

2. **Alertes critiques**
   - Échec de sauvegarde (quelconque copie)
   - Espace disque < 20%
   - Sauvegarde trop ancienne (> 48h)
   - Erreur de vérification

3. **Dashboard centralisé**
   - Vue d'ensemble du statut 3-2-1
   - Grafana avec métriques personnalisées

4. **Rapports réguliers**
   - Hebdomadaires : statut des sauvegardes
   - Mensuels : tests de restauration
   - Trimestriels : audit complet

### ✅ Tests et Validation

1. **Tester régulièrement**
   - Hebdomadaire : Vérification intégrité
   - Mensuel : Restauration partielle
   - Trimestriel : Restauration complète
   - Annuel : Disaster recovery drill

2. **Automatiser les tests**
   ```bash
   #!/bin/bash
   # Test restauration automatique sur serveur de test

   LATEST_BACKUP=$(aws s3 ls s3://backups/ | tail -1 | awk '{print $4}')
   aws s3 sync s3://backups/${LATEST_BACKUP} /restore/

   # Restaurer sur instance de test
   pg_restore -d test_db /restore/base.dump

   # Vérifier
   psql -d test_db -c "SELECT count(*) FROM clients;"
   ```

3. **Mesurer les performances**
   - RTO réel vs objectif
   - RPO réel vs objectif
   - Temps de restauration par taille

### ❌ Pièges à Éviter

1. **Ne pas avoir de vraie redondance**  
   - ❌ Mauvais : 2 disques dans le même RAID  
   - ✅ Bon : Disques sur serveurs différents

2. **Off-site "pas vraiment off-site"**  
   - ❌ Mauvais : Autre salle dans le même bâtiment  
   - ✅ Bon : Autre ville/région

3. **Ne jamais tester**
   - Une sauvegarde non testée = pas de sauvegarde

4. **Négliger la sécurité**
   - Sauvegardes non chiffrées
   - Accès non restreints
   - Clés de chiffrement avec les données

5. **Oublier les objets globaux**
   - pg_basebackup ne sauvegarde pas les rôles
   - Faire aussi `pg_dumpall --globals-only`

6. **Ignorer les coûts cloud**
   - Lifecycle policies essentielles
   - Monitoring des coûts
   - Classes de stockage appropriées

## Cas d'Usage et Scénarios de Récupération

### Cas 1 : Suppression Accidentelle de Données

**Incident** : Suppression de 10,000 clients à 14h30

**Récupération avec 3-2-1** :

```bash
# Option 1 : PITR depuis copie locale (plus rapide)
# RTO : 30 minutes
# RPO : 0 seconde

# Restaurer base backup local
cp -a /backups/local/base_20251122_0200 /var/lib/postgresql/data

# Configurer PITR avec WAL du NAS
cat > /var/lib/postgresql/data/recovery.conf << EOF  
restore_command = 'cp /mnt/nas/wal_archive/%f %p'  
recovery_target_time = '2025-11-22 14:29:59'  
recovery_target_action = 'promote'  
EOF  

# Démarrer
systemctl start postgresql
```

### Cas 2 : Corruption de la Base de Données

**Incident** : Corruption matérielle détectée

**Récupération** :

```bash
# Utiliser la copie locale la plus récente
# Si corruption récente, utiliser copie cloud plus ancienne

# Test copie locale
pg_restore -d test /backups/local/latest.dump

# Si corruption présente, utiliser cloud
aws s3 cp s3://backups/weekly_20251115/ /restore/ --recursive  
pg_restore -d production /restore/base.dump  
```

### Cas 3 : Destruction Complète du Site (Incendie)

**Incident** : Datacenter détruit, tout perdu

**Récupération** :

```bash
# 1. Provisionner nouveau serveur
# 2. Récupérer depuis cloud (seule copie restante)

aws s3 sync s3://backups/latest/ /var/lib/postgresql/data/

# 3. Récupérer WAL cloud
aws s3 sync s3://backups/wal_archive/ /wal_restore/

# 4. PITR jusqu'au dernier WAL disponible
cat > /var/lib/postgresql/data/recovery.conf << EOF  
restore_command = 'cp /wal_restore/%f %p'  
recovery_target = 'latest'  
EOF  

# 5. Démarrer
systemctl start postgresql
```

**RPO** : Dernière synchronisation WAL (généralement < 5 minutes)  
**RTO** : 2-4 heures (téléchargement depuis cloud + restauration)  

### Cas 4 : Ransomware

**Incident** : Ransomware chiffre production ET sauvegardes locales

**Récupération** :

```bash
# Les copies cloud avec Object Lock sont immuables
# Le ransomware ne peut pas les chiffrer

# 1. Isoler le réseau
# 2. Nettoyer les systèmes
# 3. Restaurer depuis cloud

aws s3 sync s3://backups/pre-ransomware/ /var/lib/postgresql/data/
```

**Pourquoi 3-2-1 sauve la mise** :
- ✅ Copie off-site intacte  
- ✅ Object Lock empêche modification  
- ✅ Versioning permet de retrouver version saine

La stratégie 3-2-1 n'est pas un luxe, c'est une **nécessité absolue** pour toute base de données en production. Elle vous protège contre :

- ✅ **Pannes matérielles** (disque, serveur)  
- ✅ **Erreurs humaines** (suppression accidentelle)  
- ✅ **Catastrophes naturelles** (incendie, inondation)  
- ✅ **Attaques malveillantes** (ransomware, sabotage)  
- ✅ **Corruptions de données** (silencieuses ou brutales)

```
2 = Types de stockage différents
1 = Au moins une copie loin géographiquement
```
