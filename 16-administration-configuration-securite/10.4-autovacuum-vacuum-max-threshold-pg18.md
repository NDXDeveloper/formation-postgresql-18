# 16.10.4. Nouveauté PG 18 : Nouveau paramètre autovacuum_vacuum_max_threshold

## Introduction : Le défi des grandes tables

Imaginez que vous gérez un entrepôt :
- **Petit entrepôt** (100 cartons) : Vous nettoyez quand il y a 20 cartons vides (20%)  
- **Grand entrepôt** (1 million de cartons) : Attendre 200 000 cartons vides (20%) est absurde !

C'est exactement le problème que PostgreSQL rencontrait avec les **grandes tables** avant la version 18. Le nouveau paramètre `autovacuum_vacuum_max_threshold` résout ce problème de manière élégante.

---

## Le fonctionnement classique d'autovacuum (rappel)

### La formule de déclenchement traditionnelle

Depuis toujours, autovacuum utilise cette formule pour décider quand nettoyer une table :

```
Seuil de déclenchement = autovacuum_vacuum_threshold + (autovacuum_vacuum_scale_factor × nombre_de_lignes)
```

**Paramètres par défaut** :
```sql
SHOW autovacuum_vacuum_threshold;      -- 50 lignes  
SHOW autovacuum_vacuum_scale_factor;   -- 0.2 (soit 20%)  
```

### Exemples de calcul

**Petite table (1 000 lignes)** :
```
Seuil = 50 + (0.2 × 1 000) = 50 + 200 = 250 lignes mortes
```
→ Autovacuum se déclenche après **250 suppressions/mises à jour**  
→ ✅ **Raisonnable**

**Table moyenne (100 000 lignes)** :
```
Seuil = 50 + (0.2 × 100 000) = 50 + 20 000 = 20 050 lignes mortes
```
→ Autovacuum se déclenche après **20 050 suppressions/mises à jour**  
→ ✅ **Acceptable**

**Grande table (10 millions de lignes)** :
```
Seuil = 50 + (0.2 × 10 000 000) = 50 + 2 000 000 = 2 000 050 lignes mortes
```
→ Autovacuum se déclenche après **2 millions de suppressions/mises à jour**  
→ ⚠️ **Problématique**

**Très grande table (100 millions de lignes)** :
```
Seuil = 50 + (0.2 × 100 000 000) = 50 + 20 000 000 = 20 000 050 lignes mortes
```
→ Autovacuum se déclenche après **20 millions de suppressions/mises à jour**  
→ ❌ **CATASTROPHIQUE**

---

## Le problème des grandes tables

### 1. Accumulation massive de bloat

**Bloat** = espace disque gaspillé par les lignes mortes.

**Scénario typique** :
- Table `logs` avec 100 millions de lignes
- Chaque jour : 5 millions de nouvelles lignes, 5 millions de suppressions
- Seuil d'autovacuum : 20 millions de lignes mortes

**Chronologie du désastre** :

```
Jour 1 : 5M lignes mortes    (bloat : 5%)  
Jour 2 : 10M lignes mortes   (bloat : 10%)  
Jour 3 : 15M lignes mortes   (bloat : 15%)  
Jour 4 : 20M lignes mortes   (bloat : 20%) ← Autovacuum se déclenche ENFIN  
```

**Problèmes** :
- ❌ La table a **doublé de taille** sur disque pendant 4 jours  
- ❌ Les requêtes sont **2× plus lentes** (plus de données à parcourir)  
- ❌ Les sauvegardes sont **2× plus grosses**  
- ❌ Le vacuum prendra **plusieurs heures** pour nettoyer 20M de lignes

### 2. Vacuum très long et bloquant

Quand autovacuum finit par se déclencher :

```sql
-- Vacuum sur une table avec 20 millions de lignes mortes
VACUUM VERBOSE logs;

-- Output typique :
-- INFO: vacuuming "public.logs"
-- INFO: scanned 250000 pages, removed 20000000 dead tuples
-- INFO: index "logs_pkey" now contains 100000000 tuples in 300000 pages
-- DETAIL: 20000000 index tuples were removed
-- Time: 7200000 ms (2 heures !)
```

**Impact** :
- ⏱️ Vacuum dure **des heures**  
- 🔒 Peut causer des **verrous** et ralentir l'application  
- 💾 Consomme beaucoup d'**I/O** et de **CPU**  
- 🚨 Les utilisateurs voient des **performances dégradées**

### 3. Effet boule de neige

Plus vous attendez, pire c'est :

```
Plus de lignes mortes
    ↓
Vacuum plus long
    ↓
Vacuum monopolise les ressources
    ↓
Application ralentie
    ↓
Plus de nouvelles opérations s'accumulent
    ↓
Encore plus de lignes mortes !
```

### 4. Problèmes spécifiques selon les cas d'usage

**Data Warehouse** :
- Tables de faits avec milliards de lignes
- ETL quotidiens avec mises à jour massives
- Attendre 20% de la table est inacceptable

**SaaS avec tables de logs** :
- Croissance continue
- Purges régulières des anciennes données
- Cycle vicieux : la table grossit, le seuil augmente, le bloat explose

**Tables d'historique** :
- Archivage périodique
- Délétions massives d'anciennes données
- Accumulation de dizaines de millions de lignes mortes

---

## La solution : autovacuum_vacuum_max_threshold

### Qu'est-ce que c'est ?

`autovacuum_vacuum_max_threshold` est un **nouveau paramètre dans PostgreSQL 18** qui définit un **plafond absolu** au nombre de lignes mortes avant le déclenchement d'autovacuum.

```sql
-- Nouveau dans PostgreSQL 18
SHOW autovacuum_vacuum_max_threshold;  -- Valeur par défaut à définir
```

### La nouvelle formule de déclenchement

**Avant PostgreSQL 18** :
```
Seuil = autovacuum_vacuum_threshold + (scale_factor × n_tuples)
```

**Avec PostgreSQL 18** :
```
Seuil = MIN(
    autovacuum_vacuum_threshold + (scale_factor × n_tuples),
    autovacuum_vacuum_max_threshold
)
```

La fonction `MIN()` prend la **plus petite valeur** entre :
1. Le calcul traditionnel  
2. Le plafond défini par `autovacuum_vacuum_max_threshold`

### Exemple concret

Configuration :
```sql
autovacuum_vacuum_threshold = 50  
autovacuum_vacuum_scale_factor = 0.2  
autovacuum_vacuum_max_threshold = 2000000  -- 2 millions (nouveau !)  
```

**Petite table (1 000 lignes)** :
```
Calcul classique : 50 + (0.2 × 1 000) = 250  
Plafond : 2 000 000  

Seuil effectif = MIN(250, 2 000 000) = 250
```
→ Aucun changement, le calcul classique s'applique

**Grande table (100 millions de lignes)** :
```
Calcul classique : 50 + (0.2 × 100 000 000) = 20 000 050  
Plafond : 2 000 000  

Seuil effectif = MIN(20 000 050, 2 000 000) = 2 000 000
```
→ ✅ **Le plafond s'applique, limitant à 2M de lignes mortes maximum !**

## Configuration et tuning

### 1. Valeurs recommandées

#### Base de données généraliste

```sql
-- Configuration équilibrée
autovacuum_vacuum_max_threshold = 2000000;  -- 2 millions
```

**Justification** :
- Protège contre l'accumulation excessive
- Vacuum reste gérable en durée (quelques minutes)
- Balance entre réactivité et overhead

#### Data Warehouse / Analytics

```sql
-- Tables très volumineuses, besoin de contrôle serré
autovacuum_vacuum_max_threshold = 5000000;  -- 5 millions
```

**Justification** :
- Tables de plusieurs milliards de lignes
- ETL massifs quotidiens
- Besoin de fenêtres de maintenance plus larges

#### Application transactionnelle (OLTP)

```sql
-- Tables plus petites, besoin de réactivité
autovacuum_vacuum_max_threshold = 1000000;  -- 1 million
```

**Justification** :
- Performance critique
- Bloat minimal tolérable
- Vacuum fréquents et rapides

#### SaaS avec tables de logs

```sql
-- Forte rotation des données
autovacuum_vacuum_max_threshold = 500000;  -- 500k
```

**Justification** :
- Insertions/suppressions massives continues
- Besoin de nettoyer très fréquemment
- Éviter l'explosion de taille

### 2. Méthode de calcul personnalisée

Pour déterminer la bonne valeur pour votre cas :

**Étape 1 : Identifier vos plus grandes tables**
```sql
SELECT
    schemaname || '.' || tablename AS table_name,
    n_live_tup,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS size
FROM pg_stat_user_tables  
ORDER BY n_live_tup DESC  
LIMIT 10;  
```

**Étape 2 : Estimer le nombre de modifications quotidiennes**
```sql
-- Sur une journée, tracer les modifications
SELECT
    tablename,
    n_tup_ins + n_tup_upd + n_tup_del AS total_modifications
FROM pg_stat_user_tables  
WHERE schemaname = 'public'  
ORDER BY total_modifications DESC;  
```

**Étape 3 : Calculer le seuil souhaité**

Règle empirique :
```
max_threshold = modifications_quotidiennes × 0.5
```

**Exemple** :
- Table `events` : 10 millions de modifications/jour
- Seuil souhaité : 5 millions de lignes mortes
- Configuration : `autovacuum_vacuum_max_threshold = 5000000`

### 3. Configuration globale vs par table

#### Configuration globale

```sql
-- postgresql.conf ou ALTER SYSTEM
ALTER SYSTEM SET autovacuum_vacuum_max_threshold = 2000000;  
SELECT pg_reload_conf();  
```

**Avantage** : S'applique à toutes les tables.

#### Configuration par table

```sql
-- Pour une table spécifique nécessitant un seuil différent
ALTER TABLE logs_volumineuse SET (
    autovacuum_vacuum_scale_factor = 0.05,  -- Plus réactif
    autovacuum_vacuum_threshold = 1000      -- Seuil de base augmenté
);

-- Note : Le max_threshold global s'applique toujours
```

**Important** : `autovacuum_vacuum_max_threshold` est un paramètre **global uniquement**, pas configurable par table. Mais vous pouvez ajuster les autres paramètres par table pour affiner le comportement.

### 4. Interaction avec scale_factor

Le `max_threshold` fonctionne en tandem avec `scale_factor` :

**Stratégie agressive (petites tables réactives, grandes tables plafonnées)** :
```sql
autovacuum_vacuum_scale_factor = 0.1;  -- 10% au lieu de 20%  
autovacuum_vacuum_max_threshold = 1000000;  
```

**Stratégie modérée** :
```sql
autovacuum_vacuum_scale_factor = 0.15;  -- 15%  
autovacuum_vacuum_max_threshold = 2000000;  
```

**Stratégie conservatrice (grosses tables, moins de vacuum)** :
```sql
autovacuum_vacuum_scale_factor = 0.2;  -- 20%  
autovacuum_vacuum_max_threshold = 5000000;  
```

## Cas pratiques détaillés

### Cas 1 : Application SaaS avec table de logs

**Contexte** :
- Table `application_logs` : 200 millions de lignes
- Insertions : 10M/jour
- Rétention : 30 jours (suppression des lignes > 30 jours)
- Suppressions : ~10M/jour

**Problème avant PG 18** :
```
Seuil = 50 + (0.2 × 200 000 000) = 40 000 050
```
→ Attendre 40 millions de lignes mortes = **inacceptable**  
→ La table gonfle de 40M × 1KB ≈ **40 GB de bloat**

**Solution avec PG 18** :
```sql
ALTER SYSTEM SET autovacuum_vacuum_max_threshold = 5000000;  
ALTER SYSTEM SET autovacuum_vacuum_scale_factor = 0.1;  
SELECT pg_reload_conf();  
```

**Nouveau comportement** :
```
Seuil = MIN(50 + (0.1 × 200 000 000), 5 000 000)  
Seuil = MIN(20 000 050, 5 000 000) = 5 000 000  
```
→ Autovacuum se déclenche tous les **5 millions de lignes mortes**  
→ Avec 10M de suppressions/jour, autovacuum tourne **2× par jour**  
→ Bloat maximal : 5M × 1KB ≈ **5 GB** au lieu de 40 GB

**Bénéfices** :
- ✅ Taille table stable  
- ✅ Performances constantes  
- ✅ Sauvegardes 8× plus petites

### Cas 2 : Data Warehouse avec table de faits

**Contexte** :
- Table `sales_fact` : 500 millions de lignes (table de faits)
- ETL nocturne : mise à jour de 50 millions de lignes
- Très peu de suppressions

**Problème avant PG 18** :
```
Seuil = 50 + (0.2 × 500 000 000) = 100 000 050
```
→ Attendre 100 millions de lignes mortes après **chaque ETL** !  
→ Impossible, l'ETL crée "seulement" 50M de lignes mortes

**Paradoxe** : L'autovacuum ne se déclencherait **jamais naturellement** car le seuil (100M) est supérieur aux modifications quotidiennes (50M) !

**Solution avec PG 18** :
```sql
ALTER SYSTEM SET autovacuum_vacuum_max_threshold = 50000000;  
SELECT pg_reload_conf();  
```

**Nouveau comportement** :
```
Seuil = MIN(50 + (0.2 × 500 000 000), 50 000 000)  
Seuil = MIN(100 000 050, 50 000 000) = 50 000 000  
```
→ Autovacuum se déclenche après chaque ETL (50M de lignes mortes)  
→ Nettoyage **quotidien automatique**

**Bénéfices** :
- ✅ Table propre chaque matin  
- ✅ Requêtes analytiques optimales  
- ✅ Pas de VACUUM FULL manuel nécessaire

### Cas 3 : Table d'archivage avec purges périodiques

**Contexte** :
- Table `orders_archive` : 1 milliard de lignes
- Purge trimestrielle : suppression de 250 millions de lignes anciennes
- Sinon, peu d'activité

**Problème avant PG 18** :
```
Seuil = 50 + (0.2 × 1 000 000 000) = 200 000 050
```
→ Après la purge de 250M de lignes, autovacuum se déclenche  
→ Mais le vacuum prendra **des heures** sur 250M de lignes mortes

**Solution avec PG 18** :
```sql
-- Stratégie : plafond + vacuum manuel planifié
ALTER SYSTEM SET autovacuum_vacuum_max_threshold = 10000000;

-- Après chaque purge trimestrielle, vacuum manuel immédiat
DELETE FROM orders_archive WHERE created_at < NOW() - INTERVAL '3 years';  
VACUUM ANALYZE orders_archive;  -- Manuel et contrôlé  
```

**Nouveau comportement** :
- Le plafond de 10M protège contre toute accumulation entre les purges
- Le vacuum manuel post-purge est planifié et contrôlé
- Pas de surprise pendant les heures de production

**Bénéfices** :
- ✅ Contrôle total sur les opérations lourdes  
- ✅ Protection contre l'accumulation accidentelle  
- ✅ Maintenance prévisible

### Cas 4 : Base multi-tenant avec disparité de tailles

**Contexte** :
- 100 clients avec leurs propres tables
- Client A : 10 000 lignes
- Client B : 500 millions de lignes
- Configuration uniforme nécessaire

**Problème avant PG 18** :
- Impossible de satisfaire tout le monde avec un seul `scale_factor`
- Client A : sur-nettoyé
- Client B : sous-nettoyé

**Solution avec PG 18** :
```sql
-- Configuration globale protégeant les gros clients
autovacuum_vacuum_scale_factor = 0.15;  -- Modéré  
autovacuum_vacuum_max_threshold = 3000000;  -- Plafond à 3M  

-- Ajustement pour les petits clients (plus réactif)
ALTER TABLE client_a.orders SET (
    autovacuum_vacuum_scale_factor = 0.1,
    autovacuum_vacuum_threshold = 50
);

-- Les gros clients sont automatiquement protégés par le plafond global
```

**Bénéfices** :
- ✅ Configuration simple et maintenable  
- ✅ Petits clients : nettoyage fréquent  
- ✅ Gros clients : protégés par le plafond  
- ✅ Pas de micro-gestion nécessaire

## Monitoring et observabilité

### 1. Vérifier si le plafond s'applique

```sql
-- Calculer le seuil effectif pour chaque table
WITH thresholds AS (
    SELECT
        schemaname || '.' || tablename AS table_name,
        n_live_tup,
        n_dead_tup,
        -- Calcul classique
        (current_setting('autovacuum_vacuum_threshold')::bigint +
         current_setting('autovacuum_vacuum_scale_factor')::float * n_live_tup)::bigint AS classic_threshold,
        -- Plafond
        current_setting('autovacuum_vacuum_max_threshold')::bigint AS max_threshold
    FROM pg_stat_user_tables
    WHERE n_live_tup > 0
)
SELECT
    table_name,
    n_live_tup,
    n_dead_tup,
    classic_threshold,
    max_threshold,
    -- Seuil effectif (le minimum des deux)
    LEAST(classic_threshold, max_threshold) AS effective_threshold,
    -- Est-ce que le plafond s'applique ?
    CASE
        WHEN classic_threshold > max_threshold THEN '✅ Plafond actif'
        ELSE '⚪ Calcul classique'
    END AS status,
    -- Pourcentage avant déclenchement
    ROUND(n_dead_tup * 100.0 / NULLIF(LEAST(classic_threshold, max_threshold), 0), 2) AS pct_to_trigger
FROM thresholds  
WHERE n_dead_tup > 1000  
ORDER BY n_dead_tup DESC  
LIMIT 20;  
```

**Output exemple** :
```
table_name          | n_live_tup | n_dead_tup | classic_threshold | max_threshold | effective_threshold | status           | pct_to_trigger
--------------------|------------|------------|-------------------|---------------|---------------------|------------------|---------------
public.logs         | 100000000  | 1500000    | 20000050          | 2000000       | 2000000             | ✅ Plafond actif | 75.00  
public.events       | 50000000   | 800000     | 10000050          | 2000000       | 2000000             | ✅ Plafond actif | 40.00  
public.users        | 100000     | 15000      | 20050             | 2000000       | 20050               | ⚪ Calcul classique | 74.81  
```

### 2. Tables proches du plafond

```sql
-- Tables qui vont bientôt déclencher un autovacuum (grâce au plafond)
SELECT
    schemaname || '.' || tablename AS table_name,
    n_live_tup,
    n_dead_tup,
    current_setting('autovacuum_vacuum_max_threshold')::bigint AS max_threshold,
    ROUND(n_dead_tup * 100.0 /
          NULLIF(current_setting('autovacuum_vacuum_max_threshold')::bigint, 0), 2) AS pct_of_max,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS size,
    last_autovacuum
FROM pg_stat_user_tables  
WHERE n_live_tup > 5000000  -- Tables volumineuses uniquement  
  AND n_dead_tup > 100000
ORDER BY pct_of_max DESC  
LIMIT 10;  
```

### 3. Impact du plafond sur la fréquence de vacuum

```sql
-- Comparer la fréquence des autovacuum avant/après le plafond
SELECT
    schemaname || '.' || tablename AS table_name,
    autovacuum_count,
    last_autovacuum,
    EXTRACT(EPOCH FROM (NOW() - last_autovacuum)) / 3600 AS hours_since_last,
    n_live_tup,
    CASE
        WHEN n_live_tup * current_setting('autovacuum_vacuum_scale_factor')::float
             > current_setting('autovacuum_vacuum_max_threshold')::bigint
        THEN 'Protected by max_threshold'
        ELSE 'Classic behavior'
    END AS behavior
FROM pg_stat_user_tables  
WHERE autovacuum_count > 0  
  AND schemaname = 'public'
ORDER BY autovacuum_count DESC  
LIMIT 20;  
```

### 4. Estimer le bloat évité grâce au plafond

```sql
-- Estimer la réduction de bloat grâce au max_threshold
WITH bloat_comparison AS (
    SELECT
        schemaname || '.' || tablename AS table_name,
        n_live_tup,
        -- Seuil sans plafond
        (current_setting('autovacuum_vacuum_threshold')::bigint +
         current_setting('autovacuum_vacuum_scale_factor')::float * n_live_tup)::bigint AS old_threshold,
        -- Seuil avec plafond
        LEAST(
            (current_setting('autovacuum_vacuum_threshold')::bigint +
             current_setting('autovacuum_vacuum_scale_factor')::float * n_live_tup)::bigint,
            current_setting('autovacuum_vacuum_max_threshold')::bigint
        ) AS new_threshold,
        pg_total_relation_size(schemaname||'.'||tablename) / n_live_tup AS avg_tuple_size
    FROM pg_stat_user_tables
    WHERE n_live_tup > 1000000
)
SELECT
    table_name,
    n_live_tup,
    old_threshold,
    new_threshold,
    old_threshold - new_threshold AS dead_tuples_saved,
    pg_size_pretty((old_threshold - new_threshold) * avg_tuple_size) AS bloat_saved
FROM bloat_comparison  
WHERE old_threshold > new_threshold  
ORDER BY (old_threshold - new_threshold) * avg_tuple_size DESC  
LIMIT 15;  
```

**Output exemple** :
```
table_name        | n_live_tup | old_threshold | new_threshold | dead_tuples_saved | bloat_saved
------------------|------------|---------------|---------------|-------------------|-------------
public.logs       | 100000000  | 20000050      | 2000000       | 18000050          | 17 GB  
public.events     | 50000000   | 10000050      | 2000000       | 8000050           | 7.6 GB  
```

### 5. Alertes recommandées

```sql
-- Alerte : Tables dépassant 80% du max_threshold
SELECT
    tablename,
    n_dead_tup,
    current_setting('autovacuum_vacuum_max_threshold')::bigint AS max_threshold,
    ROUND(n_dead_tup * 100.0 /
          current_setting('autovacuum_vacuum_max_threshold')::bigint, 2) AS pct
FROM pg_stat_user_tables  
WHERE n_dead_tup > current_setting('autovacuum_vacuum_max_threshold')::bigint * 0.8  
  AND n_live_tup > 1000000;
```

Intégrer cette requête dans votre système de monitoring (Prometheus, Grafana, DataDog, etc.).

## Migration vers PostgreSQL 18

### Étape 1 : Analyse de l'existant

Avant la migration, analysez vos tables actuelles :

```sql
-- Identifier les tables qui bénéficieraient du plafond
SELECT
    schemaname || '.' || tablename AS table_name,
    n_live_tup,
    (50 + 0.2 * n_live_tup)::bigint AS current_threshold,
    n_dead_tup,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS size,
    CASE
        WHEN (50 + 0.2 * n_live_tup) > 5000000 THEN '🔴 Bénéficierait fortement'
        WHEN (50 + 0.2 * n_live_tup) > 2000000 THEN '🟡 Bénéficierait'
        ELSE '⚪ Peu d''impact'
    END AS impact
FROM pg_stat_user_tables  
WHERE n_live_tup > 500000  
ORDER BY (50 + 0.2 * n_live_tup) DESC  
LIMIT 30;  
```

### Étape 2 : Choisir la valeur initiale

Recommandations selon vos résultats :

**Si vous avez des tables > 100M de lignes** :
```sql
autovacuum_vacuum_max_threshold = 5000000
```

**Si vos plus grandes tables sont 10-100M de lignes** :
```sql
autovacuum_vacuum_max_threshold = 2000000
```

**Si principalement des tables < 10M de lignes** :
```sql
autovacuum_vacuum_max_threshold = 1000000
```

### Étape 3 : Configuration initiale conservative

-- postgresql.conf ou ALTER SYSTEM après migration vers PG 18

### Étape 4 : Monitoring post-migration

Pendant les 2 premières semaines :

```sql
-- Requête quotidienne : fréquence des autovacuum
SELECT
    tablename,
    autovacuum_count,
    last_autovacuum,
    n_dead_tup
FROM pg_stat_user_tables  
WHERE schemaname = 'public'  
  AND n_live_tup > 1000000
ORDER BY last_autovacuum DESC;
```

**Indicateurs de succès** :
- ✅ Autovacuum plus fréquents sur les grosses tables  
- ✅ `n_dead_tup` reste sous le plafond  
- ✅ Taille des tables stable ou en réduction  
- ✅ Pas d'augmentation du temps de vacuum

### Étape 5 : Ajustement progressif

Après 2 semaines, ajustez si nécessaire :

**Si autovacuum tourne trop souvent** :
```sql
ALTER SYSTEM SET autovacuum_vacuum_max_threshold = 7000000;  -- Augmenter
```

**Si les tables accumulent encore du bloat** :
```sql
ALTER SYSTEM SET autovacuum_vacuum_max_threshold = 3000000;  -- Réduire
```

## Erreurs courantes et dépannage

### Erreur 1 : Plafond trop bas

**Symptômes** :
- Autovacuum tourne en permanence sur les mêmes tables
- Logs remplis de messages autovacuum
- Performance I/O dégradée

**Diagnostic** :
```sql
-- Vérifier la fréquence des autovacuum
SELECT
    tablename,
    autovacuum_count,
    EXTRACT(EPOCH FROM (NOW() - stats_reset)) / 3600 AS hours_since_reset,
    ROUND(autovacuum_count / (EXTRACT(EPOCH FROM (NOW() - stats_reset)) / 3600), 2) AS vacuum_per_hour
FROM pg_stat_user_tables  
WHERE schemaname = 'public'  
  AND autovacuum_count > 0
ORDER BY vacuum_per_hour DESC;

-- Si vacuum_per_hour > 2 pour certaines tables → plafond trop bas
```

**Solution** :
```sql
ALTER SYSTEM SET autovacuum_vacuum_max_threshold = 5000000;  -- Doubler  
SELECT pg_reload_conf();  
```

### Erreur 2 : Plafond trop élevé

**Symptômes** :
- Tables continuent à gonfler
- Bloat significatif (> 30%)
- Autovacuum ne se déclenche pas assez

**Diagnostic** :
```sql
-- Vérifier le bloat
SELECT
    tablename,
    n_dead_tup,
    ROUND(n_dead_tup * 100.0 / NULLIF(n_live_tup, 0), 2) AS dead_pct,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS size
FROM pg_stat_user_tables  
WHERE schemaname = 'public'  
  AND n_dead_tup > 100000
ORDER BY dead_pct DESC;

-- Si dead_pct > 20% → plafond trop élevé
```

**Solution** :
```sql
ALTER SYSTEM SET autovacuum_vacuum_max_threshold = 1000000;  -- Réduire  
SELECT pg_reload_conf();  

-- Forcer un vacuum immédiat sur les tables concernées
VACUUM ANALYZE table_problematique;
```

### Erreur 3 : Oublier le plafond après migration

**Symptôme** :
- Vous avez migré vers PG 18
- Mais le comportement n'a pas changé
- Les grandes tables souffrent toujours

**Cause** :
```sql
SHOW autovacuum_vacuum_max_threshold;
-- Résultat : valeur vide ou très élevée par défaut
```

**Solution** :
```sql
-- Définir explicitement le paramètre
ALTER SYSTEM SET autovacuum_vacuum_max_threshold = 2000000;  
SELECT pg_reload_conf();  
```

### Erreur 4 : Confusion avec max_workers

**Erreur courante** :
Penser que `autovacuum_vacuum_max_threshold` contrôle le nombre de workers.

**Clarification** :
- `autovacuum_vacuum_max_threshold` → **Nombre de lignes mortes** (seuil)  
- `autovacuum_worker_slots` → **Nombre de workers** (parallélisme)

Ces deux paramètres sont **indépendants et complémentaires**.

### 1. Toujours définir le paramètre

✅ Dans PostgreSQL 18, **toujours configurer** `autovacuum_vacuum_max_threshold` :

### 2. Commencer conservateur

✅ Lors de la migration ou d'un nouveau projet :

- Ajuster progressivement à la baisse si nécessaire

### 3. Adapter selon votre domaine

✅ Ajuster selon votre cas d'usage :

| Type d'application | Valeur recommandée |
|--------------------|-------------------|

| Analytics / DW | 5-10 millions |

✅ Le `max_threshold` fonctionne mieux avec un `scale_factor` réduit :

```sql
autovacuum_vacuum_scale_factor = 0.1;  -- Au lieu de 0.2

```

→ Petites tables plus réactives, grandes tables protégées

### 5. Documenter votre choix

✅ Documenter pourquoi vous avez choisi cette valeur :

```sql
-- postgresql.conf
# autovacuum_vacuum_max_threshold = 2000000

# Ajusté le 2025-11-22 après analyse de bloat
```

### 6. Alerte de monitoring

✅ Configurer des alertes pour :
- Tables dépassant 80% du `max_threshold`
- Autovacuum échouant sur les mêmes tables
- Bloat > 30% malgré le plafond

### 7. Révision périodique

✅ Réviser la valeur tous les 6 mois :
- Vos données ont-elles changé de volume ?
- De nouveaux patterns d'usage ?
- Performance satisfaisante ?
