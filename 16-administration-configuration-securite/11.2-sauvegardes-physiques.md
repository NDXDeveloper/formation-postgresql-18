# 16.11.2. Sauvegardes Physiques (pg_basebackup)

## Introduction aux Sauvegardes Physiques

Les sauvegardes physiques représentent une approche fondamentalement différente des sauvegardes logiques. Alors qu'une sauvegarde logique extrait les données sous forme de commandes SQL, une sauvegarde physique copie directement les **fichiers binaires** qui constituent la base de données PostgreSQL sur le disque.

### Analogie Simple

Imaginez deux façons de sauvegarder une bibliothèque :

- **Sauvegarde logique (pg_dump)** : Vous recopiez le contenu de chaque livre, page par page, dans de nouveaux cahiers. Vous pouvez lire et modifier ces cahiers, les réorganiser, mais reconstituer la bibliothèque prendra du temps.

- **Sauvegarde physique (pg_basebackup)** : Vous prenez une photo exacte de chaque étagère telle quelle, avec les livres dans leur position exacte. La restauration est rapide (remettre les étagères en place), mais vous ne pouvez pas facilement extraire un seul livre.

### Qu'est-ce qu'une Sauvegarde Physique ?

Une sauvegarde physique copie **l'intégralité du répertoire de données PostgreSQL** (data directory), incluant :

- Les fichiers de tables et d'index
- Les fichiers de configuration
- Les fichiers WAL (Write-Ahead Log)
- Les fichiers de contrôle
- Les fichiers de transaction

Le résultat est une copie **bit à bit** du cluster PostgreSQL entier, prête à être remontée telle quelle.

## Sauvegardes Logiques vs Sauvegardes Physiques

### Comparaison Détaillée

| Critère | Sauvegarde Logique (pg_dump) | Sauvegarde Physique (pg_basebackup) |
|---------|------------------------------|-------------------------------------|
| **Granularité** | Base par base, table par table | Cluster entier seulement |
| **Format** | SQL lisible ou binaire PostgreSQL | Fichiers binaires bruts |
| **Portabilité** | Excellente (versions, OS, architectures) | Limitée (même version majeure, même architecture) |
| **Taille** | Généralement plus petite (données seules) | Plus grande (inclut index, TOAST, etc.) |
| **Vitesse de sauvegarde** | Lente (lecture + export) | Rapide (copie directe) |
| **Vitesse de restauration** | Lente (rejeu SQL) | Très rapide (copie de fichiers) |
| **Point-In-Time Recovery (PITR)** | Non | **Oui** ✅ |
| **Restauration sélective** | Oui (tables spécifiques) | Non (tout ou rien) |
| **Dépendance version** | Flexible | Stricte (majeure identique) |
| **Cohérence transactionnelle** | Snapshot d'une base | Snapshot du cluster entier |
| **Impact performance** | Modéré (CPU, I/O) | Faible à modéré (I/O réseau) |
| **Cas d'usage principal** | Migration, clonage, export | Production, HA, PITR |

### Avantages des Sauvegardes Physiques

✅ **Rapidité** : Sauvegarde et restauration beaucoup plus rapides pour les grandes bases

✅ **Point-In-Time Recovery** : Permet de restaurer à n'importe quel moment précis dans le passé

✅ **Haute Disponibilité** : Base pour la réplication streaming et les serveurs de secours

✅ **Cohérence globale** : Snapshot cohérent de toutes les bases du cluster

✅ **Automatisable** : Intégration facile dans des systèmes de HA (Patroni, Repmgr)

✅ **Efficacité** : Idéal pour les très grandes bases de données (plusieurs To)

### Inconvénients des Sauvegardes Physiques

❌ **Granularité limitée** : Impossible de restaurer une seule base ou une seule table

❌ **Compatibilité version** : Ne fonctionne qu'entre versions mineures (ex: 18.0 → 18.3, mais pas 17 → 18)

❌ **Compatibilité architecture** : Même architecture CPU requise (x86_64 → x86_64)

❌ **Taille** : Plus volumineuse (inclut index, données temporaires, etc.)

❌ **Lisibilité** : Impossible d'inspecter le contenu directement

❌ **Dépendance WAL** : Pour PITR, nécessite une archive continue des fichiers WAL

## pg_basebackup : L'Outil de Sauvegarde Physique

### Qu'est-ce que pg_basebackup ?

`pg_basebackup` est l'outil officiel PostgreSQL pour créer des sauvegardes physiques complètes d'un cluster PostgreSQL en cours d'exécution. Il se connecte au serveur PostgreSQL via le protocole de réplication et copie l'intégralité du répertoire de données.

### Fonctionnement Interne

Lorsque vous lancez `pg_basebackup`, voici ce qui se passe :

1. **Connexion au serveur** : pg_basebackup se connecte au serveur PostgreSQL via une connexion de réplication

2. **Checkpoint de démarrage** : Un checkpoint est déclenché pour garantir que tous les changements en mémoire sont écrits sur disque

3. **Streaming des données** : Les fichiers du répertoire de données sont copiés via le réseau

4. **Copie des WAL** : Les fichiers WAL nécessaires sont inclus ou archivés

5. **Checkpoint de fin** : Un checkpoint final garantit la cohérence

6. **Génération de métadonnées** : Création d'un fichier `backup_label` et `tablespace_map` si nécessaire

### Prérequis Configuration

Avant d'utiliser `pg_basebackup`, vous devez configurer PostgreSQL pour accepter les connexions de réplication.

#### 1. Configuration de postgresql.conf

```conf
# Activer la réplication
wal_level = replica               # Niveau WAL minimum pour réplication  
max_wal_senders = 10              # Nombre max de processus de réplication  
wal_keep_size = 1GB               # Taille min de WAL à conserver (PostgreSQL 13+)  
# Ancienne syntaxe : wal_keep_segments = 64 (avant PG 13)
```

**Explication** :
- `wal_level = replica` : Active la génération des WAL nécessaires pour la réplication  
- `max_wal_senders` : Réserve des slots de connexion pour les processus de réplication (pg_basebackup en utilise 1)  
- `wal_keep_size` : Évite que les WAL nécessaires soient supprimés pendant la sauvegarde

#### 2. Configuration de pg_hba.conf

Autoriser les connexions de réplication :

```conf
# TYPE  DATABASE        USER            ADDRESS                 METHOD
# Réplication locale
local   replication     backup_user                             scram-sha-256

# Réplication réseau
host    replication     backup_user     192.168.1.0/24          scram-sha-256  
host    replication     backup_user     10.0.0.0/8              scram-sha-256  
```

**Explication** :
- `replication` : Mot-clé spécial pour les connexions de réplication  
- `backup_user` : Utilisateur dédié aux sauvegardes (à créer)
- Adresse réseau : Restreindre les IPs autorisées

#### 3. Créer un Utilisateur de Réplication

```sql
-- Créer un rôle dédié aux sauvegardes
CREATE ROLE backup_user WITH REPLICATION LOGIN PASSWORD 'MotDePasseSecurise';

-- Si besoin d'accéder à des tablespaces ou configurations spécifiques
GRANT pg_read_all_settings TO backup_user;  
GRANT pg_read_all_stats TO backup_user;  
```

**Bonnes pratiques** :
- Utilisez un utilisateur dédié (principe du moindre privilège)
- N'accordez pas de droits superutilisateur
- Utilisez un mot de passe fort
- Préférez `scram-sha-256` à `md5`

#### 4. Recharger la Configuration

```bash
# Méthode 1 : Depuis psql
psql -U postgres -c "SELECT pg_reload_conf();"

# Méthode 2 : Depuis le shell
pg_ctl reload -D /var/lib/postgresql/data
```

## Utilisation de pg_basebackup

### Syntaxe de Base

```bash
pg_basebackup [options]
```

**Exemple minimal** :
```bash
pg_basebackup -D /backups/base_backup -U backup_user -h localhost -p 5432
```

### Options Essentielles

#### Options de Destination

**-D, --pgdata=DIRECTORY** : Répertoire de destination

```bash
pg_basebackup -D /backups/my_backup
```

⚠️ Le répertoire doit être **vide** ou ne pas exister.

#### Options de Connexion

Identiques à psql et pg_dump :

```bash
pg_basebackup -D /backups/backup1 \
    -h db-server.company.com \    # Hôte
    -p 5432 \                      # Port
    -U backup_user \               # Utilisateur
    -W                             # Demander mot de passe
```

**Astuce** : Utilisez un `.pgpass` pour éviter de saisir le mot de passe :

```bash
# Fichier ~/.pgpass (permissions 600)
hostname:port:database:username:password  
db-server.company.com:5432:replication:backup_user:MotDePasseSecurise  
```

```bash
chmod 600 ~/.pgpass
```

#### Options de Format

**-F, --format=FORMAT** : Format de sortie (plain ou tar)

**Format Plain (par défaut)** :
```bash
pg_basebackup -D /backups/plain_backup -F p
```
- Copie directe du répertoire de données
- Prêt à être utilisé immédiatement
- Peut être restauré en remplaçant le data directory

**Format Tar** :
```bash
pg_basebackup -D /backups/tar_backup -F t
```
- Crée un ou plusieurs fichiers `.tar`
- Un fichier par tablespace
- Nécessite extraction avant utilisation
- Plus compact et portable

**Format Tar avec Compression** :
```bash
# Compression gzip
pg_basebackup -D /backups/compressed -F t -z

# Spécifier le niveau de compression (1-9)
pg_basebackup -D /backups/compressed -F t -Z 6

# Compression avec autre algorithme (PostgreSQL 15+)
pg_basebackup -D /backups/compressed -F t --compress=lz4  
pg_basebackup -D /backups/compressed -F t --compress=zstd:5  
```

#### Options WAL

**-X, --wal-method=METHOD** : Méthode d'inclusion des WAL

**fetch (par défaut)** :
```bash
pg_basebackup -D /backup -X fetch
```
- Collecte les WAL à la fin de la sauvegarde
- Nécessite `wal_keep_size` suffisant
- Plus simple mais risque de perte de WAL sur bases très actives

**stream (recommandé)** :
```bash
pg_basebackup -D /backup -X stream
```
- Streaming continu des WAL pendant la sauvegarde
- **Recommandé** : aucun risque de perte de WAL
- Nécessite un `max_wal_senders` supplémentaire (utilise 2 connexions)

**none** :
```bash
pg_basebackup -D /backup -X none
```
- Aucun WAL inclus
- Utiliser uniquement si vous avez un archivage WAL séparé

#### Options de Progression et Verbosité

**-v, --verbose** : Mode verbeux

```bash
pg_basebackup -D /backup -v
```
Affiche les détails de l'opération.

**-P, --progress** : Afficher la progression

```bash
pg_basebackup -D /backup -P
```
Affiche une barre de progression avec estimation du temps restant.

**Combinaison recommandée** :
```bash
pg_basebackup -D /backup -P -v
```

#### Options de Checkpoint

**-c, --checkpoint=MODE** : Mode de checkpoint

**fast (recommandé)** :
```bash
pg_basebackup -D /backup -c fast
```
- Force un checkpoint immédiat
- La sauvegarde démarre rapidement
- **Recommandé** pour la plupart des cas

**spread (par défaut)** :
```bash
pg_basebackup -D /backup -c spread
```
- Checkpoint progressif
- Moins d'impact sur les performances
- Sauvegarde démarre plus lentement

#### Options de Slots de Réplication

**-S, --slot=SLOTNAME** : Utiliser un slot de réplication

```bash
# Créer d'abord le slot
psql -U postgres -c "SELECT pg_create_physical_replication_slot('backup_slot');"

# Utiliser le slot pour la sauvegarde
pg_basebackup -D /backup -S backup_slot
```

**Avantages des slots** :
- Garantit que les WAL nécessaires ne seront pas supprimés
- Idéal pour les environnements à forte activité
- Évite les erreurs "requested WAL segment has already been removed"

**--create-slot** : Créer le slot automatiquement

```bash
pg_basebackup -D /backup -S backup_slot --create-slot
```

#### Options de Tablespace

**-T, --tablespace-mapping=OLDDIR=NEWDIR** : Remapper les tablespaces

```bash
pg_basebackup -D /backup \
    -T /mnt/tablespace1=/new/location/tablespace1
```

Utile lors de restauration sur un serveur avec une arborescence différente.

#### Options de Limitation

**--max-rate=RATE** : Limiter la bande passante

```bash
# Limiter à 10 MB/s
pg_basebackup -D /backup --max-rate=10M

# Limiter à 100 KB/s
pg_basebackup -D /backup --max-rate=100k
```

**Cas d'usage** : Éviter de saturer le réseau ou les disques pendant les heures de production.

## Exemples Pratiques

### Exemple 1 : Sauvegarde Locale Simple

```bash
pg_basebackup \
    -D /backups/postgres/$(date +%Y%m%d) \
    -U backup_user \
    -P -v \
    -X stream \
    -c fast
```

**Explications** :
- `-D` : Sauvegarde dans un répertoire daté  
- `-P -v` : Progression et verbosité  
- `-X stream` : Streaming des WAL (recommandé)  
- `-c fast` : Checkpoint immédiat

### Exemple 2 : Sauvegarde Compressée vers Serveur Distant

```bash
pg_basebackup \
    -h production.db.company.com \
    -p 5432 \
    -U backup_user \
    -D /backups/prod_backup \
    -F t \
    -z \
    -Z 6 \
    -P -v \
    -X stream \
    -c fast
```

**Explications** :
- `-h` : Serveur distant  
- `-F t -z -Z 6` : Format tar compressé niveau 6
- Idéal pour économiser espace disque

### Exemple 3 : Sauvegarde avec Slot de Réplication

```bash
# Étape 1 : Créer le slot (une seule fois)
psql -U postgres -h db-server -c "SELECT pg_create_physical_replication_slot('daily_backup_slot');"

# Étape 2 : Sauvegarde utilisant le slot
pg_basebackup \
    -h db-server \
    -U backup_user \
    -D /backups/daily/$(date +%Y%m%d_%H%M) \
    -S daily_backup_slot \
    -P -v \
    -X stream \
    -c fast
```

### Exemple 4 : Sauvegarde Limitée en Bande Passante

```bash
# Limiter à 20 MB/s pour ne pas saturer le réseau
pg_basebackup \
    -h db-server \
    -U backup_user \
    -D /backups/rate_limited \
    --max-rate=20M \
    -P -v \
    -X stream \
    -c fast
```

**Cas d'usage** : Sauvegardes durant les heures de production sur un réseau partagé.

### Exemple 5 : Sauvegarde Plain pour Standby Immédiat

```bash
pg_basebackup \
    -h primary-db \
    -U replicator \
    -D /var/lib/postgresql/14/standby \
    -F p \
    -X stream \
    -c fast \
    -R
```

**-R (--write-recovery-conf)** : Génère automatiquement les fichiers de configuration pour un serveur standby.

## Restauration d'une Sauvegarde Physique

### Restauration Basique (Format Plain)

#### Étape 1 : Arrêter PostgreSQL

```bash
sudo systemctl stop postgresql
# Ou
pg_ctl stop -D /var/lib/postgresql/data
```

#### Étape 2 : Sauvegarder l'Ancien Data Directory (Sécurité)

```bash
mv /var/lib/postgresql/data /var/lib/postgresql/data.old
```

#### Étape 3 : Copier la Sauvegarde

```bash
cp -a /backups/base_backup /var/lib/postgresql/data
```

**Important** : Utilisez `cp -a` pour préserver les permissions et propriétaires.

#### Étape 4 : Ajuster les Permissions

```bash
chown -R postgres:postgres /var/lib/postgresql/data  
chmod 700 /var/lib/postgresql/data  
```

#### Étape 5 : Démarrer PostgreSQL

```bash
sudo systemctl start postgresql
# Ou
pg_ctl start -D /var/lib/postgresql/data
```

### Restauration depuis Format Tar

#### Étape 1 : Arrêter PostgreSQL

```bash
sudo systemctl stop postgresql
```

#### Étape 2 : Créer et Préparer le Répertoire

```bash
rm -rf /var/lib/postgresql/data  
mkdir /var/lib/postgresql/data  
```

#### Étape 3 : Extraire l'Archive Tar

```bash
# Si une seule archive (base.tar)
tar -xzf /backups/base.tar.gz -C /var/lib/postgresql/data

# Si plusieurs archives (tablespaces)
tar -xzf /backups/base.tar.gz -C /var/lib/postgresql/data  
tar -xzf /backups/pg_tblspc_16384.tar.gz -C /mnt/tablespace1  
```

#### Étape 4 : Permissions et Démarrage

```bash
chown -R postgres:postgres /var/lib/postgresql/data  
chmod 700 /var/lib/postgresql/data  
sudo systemctl start postgresql  
```

### Restauration avec Point-In-Time Recovery (PITR)

Le PITR permet de restaurer la base à un moment précis dans le passé. Cela nécessite :
1. Une sauvegarde de base (pg_basebackup)  
2. Une archive continue des fichiers WAL

**Configuration pour PITR** :

#### Étape 1 : Configurer l'Archivage WAL (Sur le Serveur Source)

```conf
# Dans postgresql.conf
wal_level = replica  
archive_mode = on  
archive_command = 'cp %p /mnt/wal_archive/%f'  
# Ou vers un serveur distant
# archive_command = 'rsync -a %p backup-server:/wal_archive/%f'
```

#### Étape 2 : Créer la Sauvegarde de Base

```bash
pg_basebackup \
    -D /backups/pitr_base \
    -X stream \
    -c fast \
    -P -v
```

#### Étape 3 : Restauration avec Récupération

```bash
# Arrêter PostgreSQL
sudo systemctl stop postgresql

# Restaurer la sauvegarde de base
rm -rf /var/lib/postgresql/data  
cp -a /backups/pitr_base /var/lib/postgresql/data  

# Créer un fichier de configuration de récupération
cat > /var/lib/postgresql/data/recovery.signal << EOF
# Fichier vide qui active le mode recovery
EOF

# Configurer les paramètres de récupération
cat >> /var/lib/postgresql/data/postgresql.conf << EOF  
restore_command = 'cp /mnt/wal_archive/%f %p'  
recovery_target_time = '2025-11-22 14:30:00'  
recovery_target_action = 'promote'  
EOF  

# Permissions
chown -R postgres:postgres /var/lib/postgresql/data

# Démarrer PostgreSQL
sudo systemctl start postgresql
```

**Paramètres de récupération** :
- `restore_command` : Commande pour récupérer les WAL archivés  
- `recovery_target_time` : Moment précis de restauration  
- `recovery_target_action` : Action après récupération (promote, pause, shutdown)

**Alternatives de ciblage** :
```conf
# Restaurer jusqu'à une transaction spécifique
recovery_target_xid = '12345678'

# Restaurer jusqu'à un nom de point de sauvegarde
recovery_target_name = 'before_migration'

# Restaurer jusqu'au dernier WAL disponible
recovery_target = 'immediate'
```

## Automatisation et Scripts

### Script de Sauvegarde Automatique

```bash
#!/bin/bash
# /usr/local/bin/pg_physical_backup.sh

# Configuration
BACKUP_DIR="/backups/postgresql"  
RETENTION_DAYS=7  
DB_HOST="localhost"  
DB_USER="backup_user"  
DATE=$(date +%Y%m%d_%H%M%S)  
BACKUP_PATH="${BACKUP_DIR}/${DATE}"  
LOG_FILE="${BACKUP_DIR}/backup.log"  

# Fonction de log
log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1" | tee -a ${LOG_FILE}
}

# Vérifier l'espace disque disponible
AVAILABLE_SPACE=$(df -BG ${BACKUP_DIR} | tail -1 | awk '{print $4}' | sed 's/G//')  
REQUIRED_SPACE=100  # GB  

if [ ${AVAILABLE_SPACE} -lt ${REQUIRED_SPACE} ]; then
    log "ERROR: Insufficient disk space. Available: ${AVAILABLE_SPACE}GB, Required: ${REQUIRED_SPACE}GB"
    exit 1
fi

# Créer le répertoire de sauvegarde
mkdir -p ${BACKUP_PATH}

# Exécuter pg_basebackup
log "Starting physical backup to ${BACKUP_PATH}"

pg_basebackup \
    -h ${DB_HOST} \
    -U ${DB_USER} \
    -D ${BACKUP_PATH} \
    -F t \
    -z \
    -Z 6 \
    -P -v \
    -X stream \
    -c fast \
    2>&1 | tee -a ${LOG_FILE}

# Vérifier le succès
if [ ${PIPESTATUS[0]} -eq 0 ]; then
    log "Backup completed successfully"

    # Calculer la taille
    BACKUP_SIZE=$(du -sh ${BACKUP_PATH} | cut -f1)
    log "Backup size: ${BACKUP_SIZE}"

    # Nettoyer les anciennes sauvegardes
    log "Cleaning backups older than ${RETENTION_DAYS} days"
    find ${BACKUP_DIR} -maxdepth 1 -type d -mtime +${RETENTION_DAYS} -exec rm -rf {} \;

    # Notification de succès
    echo "PostgreSQL backup successful - ${DATE}" | mail -s "Backup Success" admin@company.com

    exit 0
else
    log "ERROR: Backup failed"
    echo "PostgreSQL backup FAILED - ${DATE}" | mail -s "BACKUP FAILED" admin@company.com
    exit 1
fi
```

**Installation** :
```bash
# Rendre le script exécutable
chmod +x /usr/local/bin/pg_physical_backup.sh

# Ajouter à crontab (tous les jours à 2h du matin)
crontab -e
0 2 * * * /usr/local/bin/pg_physical_backup.sh
```

### Script de Vérification de Sauvegarde

```bash
#!/bin/bash
# /usr/local/bin/verify_backup.sh

BACKUP_PATH=$1

if [ -z "${BACKUP_PATH}" ]; then
    echo "Usage: $0 <backup_path>"
    exit 1
fi

echo "Verifying backup at ${BACKUP_PATH}"

# Vérifier la présence du fichier backup_label
if [ ! -f "${BACKUP_PATH}/backup_label" ]; then
    echo "ERROR: backup_label file not found"
    exit 1
fi

echo "✓ backup_label found"

# Vérifier la présence de PG_VERSION
if [ ! -f "${BACKUP_PATH}/PG_VERSION" ]; then
    echo "ERROR: PG_VERSION file not found"
    exit 1
fi

PG_VERSION=$(cat ${BACKUP_PATH}/PG_VERSION)  
echo "✓ PostgreSQL version: ${PG_VERSION}"  

# Vérifier la présence de pg_wal
if [ ! -d "${BACKUP_PATH}/pg_wal" ]; then
    echo "ERROR: pg_wal directory not found"
    exit 1
fi

WAL_COUNT=$(ls -1 ${BACKUP_PATH}/pg_wal | wc -l)  
echo "✓ pg_wal directory present (${WAL_COUNT} files)"  

# Vérifier la taille
BACKUP_SIZE=$(du -sh ${BACKUP_PATH} | cut -f1)  
echo "✓ Backup size: ${BACKUP_SIZE}"  

echo "Backup verification completed successfully"
```

## Surveillance et Monitoring

### Vérifier l'État des Sauvegardes

#### Dernière Sauvegarde Réussie

```sql
-- Vérifier la dernière sauvegarde de base
SELECT
    pg_backup_start_time() AS backup_in_progress_since
FROM pg_stat_activity  
WHERE backend_type = 'walsender'  
  AND state = 'streaming';
```

#### Espace Disque des Sauvegardes

```bash
# Vérifier l'espace utilisé par les sauvegardes
du -sh /backups/postgresql/*

# Surveiller l'espace disque disponible
df -h /backups
```

#### Logs de pg_basebackup

```bash
# Analyser les logs pour détecter des erreurs
grep -i error /backups/postgresql/backup.log  
grep -i failed /backups/postgresql/backup.log  
```

### Alertes Recommandées

1. **Sauvegarde Échouée** : Notification immédiate si pg_basebackup retourne un code d'erreur  
2. **Sauvegarde Absente** : Alerte si aucune sauvegarde n'a été créée dans les dernières 24h  
3. **Espace Disque Faible** : Alerte si < 20% d'espace libre sur le volume de sauvegarde  
4. **Taille Anormale** : Alerte si la taille de sauvegarde varie de > 50% par rapport à la moyenne  
5. **Durée Anormale** : Alerte si la durée de sauvegarde augmente significativement

### Métriques à Surveiller

```sql
-- Taille totale du cluster
SELECT
    pg_size_pretty(sum(pg_database_size(datname))) AS total_size
FROM pg_database;

-- Taille par base
SELECT
    datname,
    pg_size_pretty(pg_database_size(datname)) AS size
FROM pg_database  
ORDER BY pg_database_size(datname) DESC;  

-- WAL générés par heure (indicateur d'activité)
SELECT
    date_trunc('hour', now()) AS hour,
    pg_size_pretty(
        (pg_current_wal_lsn() - pg_wal_lsn_diff(pg_current_wal_lsn(), lag(pg_current_wal_lsn()) OVER (ORDER BY now())))::bigint
    ) AS wal_generated;
```

## Stratégies de Sauvegarde Physique

### Stratégie 1 : Sauvegarde Quotidienne Simple

**Contexte** : PME, base de données < 100 GB, RTO = 1h, RPO = 24h

```bash
# Cron : Tous les jours à 2h
0 2 * * * /usr/local/bin/pg_physical_backup.sh

# Rétention : 7 jours
```

**Coût** : ~700 GB de stockage (100 GB × 7 jours)

### Stratégie 2 : Sauvegarde avec Archivage WAL (PITR)

**Contexte** : Entreprise, base critique, RTO = 30min, RPO = 5min

```bash
# Sauvegarde complète : Dimanche à 1h
0 1 * * 0 pg_basebackup -D /backups/weekly/$(date +%Y%m%d) ...

# Archivage WAL continu
archive_mode = on  
archive_command = 'rsync -a %p backup-server:/wal_archive/%f'  

# Nettoyage des WAL > 30 jours
0 3 * * * find /wal_archive -mtime +30 -delete
```

**Avantage** : Récupération possible à n'importe quel moment dans les 30 derniers jours.

### Stratégie 3 : Sauvegarde Incrémentale (PostgreSQL 17+)

**Contexte** : Très grande base de données (plusieurs To), optimisation de l'espace

```bash
# Sauvegarde complète : 1er du mois
0 1 1 * * pg_basebackup -D /backups/full/$(date +%Y%m) ...

# Sauvegardes incrémentales : Tous les jours
0 2 * * * pg_basebackup -D /backups/incr/$(date +%Y%m%d) --incremental=/backups/full/manifest ...
```

**Avantage** : Économie d'espace disque significative (seuls les blocs modifiés sont sauvegardés).

### Stratégie 4 : Backup + Réplication Streaming

**Contexte** : Haute disponibilité, RTO < 1min, RPO < 1sec

```bash
# Serveur Primary : Archivage WAL
archive_mode = on  
archive_command = 'rsync -a %p backup-server:/wal_archive/%f'  

# Serveur Standby : Réplication en temps réel
standby_mode = on  
restore_command = 'cp /wal_archive/%f %p'  

# Sauvegarde depuis le Standby (pour ne pas impacter le Primary)
0 2 * * * pg_basebackup -h standby-server -D /backups/$(date +%Y%m%d) ...
```

**Avantage** :
- Aucun impact sur la production (sauvegarde depuis standby)
- Récupération quasi-instantanée (promotion du standby)
- PITR disponible via archives WAL

## Considérations de Performance

### Impact sur le Serveur Source

**CPU** :
- Faible : Principalement lecture disque et transfert réseau
- Compression (avec `-z`) augmente légèrement l'usage CPU

**I/O** :
- Modéré à élevé : Lecture complète de tous les fichiers
- Peut affecter les performances des requêtes (contention I/O)

**Réseau** :
- Élevé si sauvegarde distante : tout le cluster transite par le réseau
- Utiliser `--max-rate` pour limiter la bande passante

**Mémoire** :
- Faible : pg_basebackup n'utilise pas beaucoup de RAM

### Optimisations

#### 1. Sauvegarder depuis un Standby

```bash
# Connecter à un serveur standby plutôt que le primary
pg_basebackup -h standby.company.com -D /backup ...
```

**Avantage** : Zéro impact sur le serveur de production.

#### 2. Planifier aux Heures Creuses

```bash
# Exemple : 2h du matin (faible activité)
0 2 * * * /usr/local/bin/pg_physical_backup.sh
```

#### 3. Limiter la Bande Passante

```bash
pg_basebackup -D /backup --max-rate=50M ...
```

#### 4. Utiliser la Compression

```bash
# Réduire la taille et le transfert réseau
pg_basebackup -D /backup -F t -z -Z 6 ...
```

#### 5. Utiliser un Stockage Rapide

- SSD pour les sauvegardes locales
- NVMe si disponible
- Réseau 10 Gigabit pour sauvegardes distantes

## Dépannage et Erreurs Courantes

### Erreur 1 : "could not connect to server"

```
pg_basebackup: error: could not connect to server: could not connect to server: Connection refused
```

**Causes** :
- PostgreSQL non démarré
- Mauvais hôte/port
- Firewall bloque la connexion

**Solutions** :
```bash
# Vérifier que PostgreSQL tourne
systemctl status postgresql

# Tester la connexion
psql -h db-server -U backup_user -c "SELECT version();"

# Vérifier pg_hba.conf
cat /etc/postgresql/18/main/pg_hba.conf | grep replication
```

### Erreur 2 : "must be superuser or replication role"

```
pg_basebackup: error: must be superuser or replication role to start walsender
```

**Cause** : L'utilisateur n'a pas le privilège REPLICATION.

**Solution** :
```sql
ALTER ROLE backup_user WITH REPLICATION;
```

### Erreur 3 : "directory not empty"

```
pg_basebackup: error: directory "/backup" exists but is not empty
```

**Solution** :
```bash
# Utiliser un répertoire vide
rm -rf /backup/*
# Ou un nouveau répertoire
pg_basebackup -D /backup_new
```

### Erreur 4 : "requested WAL segment has already been removed"

```
pg_basebackup: error: could not get write-ahead log end position from server: ERROR: requested WAL segment 000000010000000000000042 has already been removed
```

**Causes** :
- `wal_keep_size` trop petit
- Sauvegarde trop longue (base très volumineuse)
- Forte activité d'écriture

**Solutions** :

**1. Augmenter wal_keep_size** :
```conf
# postgresql.conf
wal_keep_size = 2GB  # Ou plus selon l'activité
```

**2. Utiliser -X stream** :
```bash
pg_basebackup -D /backup -X stream  # Recommandé
```

**3. Utiliser un slot de réplication** :
```bash
pg_basebackup -D /backup -S backup_slot
```

### Erreur 5 : "archive command failed"

Lors de la restauration avec PITR :

```
archive command failed with exit code 1
```

**Causes** :
- Chemin d'archive WAL incorrect
- Permissions insuffisantes
- Fichiers WAL manquants

**Solutions** :
```bash
# Vérifier le chemin
ls -la /mnt/wal_archive/

# Vérifier les permissions
chown -R postgres:postgres /mnt/wal_archive/

# Tester restore_command manuellement
su - postgres -c "cp /mnt/wal_archive/000000010000000000000001 /tmp/test"
```

### Erreur 6 : "insufficient max_wal_senders"

```
pg_basebackup: error: could not send replication command "BASE_BACKUP": ERROR: number of requested standby connections exceeds max_wal_senders
```

**Solution** :
```conf
# postgresql.conf
max_wal_senders = 10  # Augmenter la valeur
```

```bash
# Redémarrer PostgreSQL
sudo systemctl restart postgresql
```

## Sécurité des Sauvegardes Physiques

### 1. Chiffrement des Sauvegardes

**Chiffrement avec GPG** :
```bash
pg_basebackup -D - -F t | gpg --encrypt --recipient admin@company.com > backup.tar.gpg
```

**Déchiffrement** :
```bash
gpg --decrypt backup.tar.gpg | tar -x -C /restore_dir
```

**Chiffrement avec OpenSSL** :
```bash
pg_basebackup -D - -F t | openssl enc -aes-256-cbc -salt -out backup.tar.enc -pass pass:SecurePassword
```

### 2. Stockage Sécurisé

**Permissions restrictives** :
```bash
chmod 700 /backups/postgresql  
chown postgres:postgres /backups/postgresql  
```

**Stockage hors-site** :
- Cloud storage (S3, Azure Blob, Google Cloud Storage)
- Serveur de backup distant (rsync via SSH)
- Support physique externe (pour archives à long terme)

### 3. Vérification d'Intégrité

**Checksums PostgreSQL** :
```bash
# Activer lors de l'initdb (PostgreSQL 18 : activé par défaut)
initdb --data-checksums

# Vérifier l'état
psql -c "SHOW data_checksums;"
```

**Checksum de fichier** :
```bash
# Créer un checksum MD5
md5sum /backups/backup.tar > /backups/backup.tar.md5

# Vérifier
md5sum -c /backups/backup.tar.md5
```

### 4. Rotation et Rétention

**Règle 3-2-1** :
- **3** copies de vos données
- Sur **2** supports différents
- **1** copie hors-site (off-site)

**Exemple d'implémentation** :
```bash
# Copie 1 : Sauvegarde locale
pg_basebackup -D /backups/local/$(date +%Y%m%d)

# Copie 2 : Disque externe monté
rsync -a /backups/local/$(date +%Y%m%d) /mnt/external_drive/

# Copie 3 : Cloud (S3)
aws s3 sync /backups/local/$(date +%Y%m%d) s3://company-backups/postgres/$(date +%Y%m%d)
```

## Comparaison avec Autres Solutions de Sauvegarde

### pg_basebackup vs Sauvegarde Manuelle (rsync)

| Aspect | pg_basebackup | rsync du data directory |
|--------|---------------|-------------------------|
| **Cohérence** | Garantie (snapshot) | Incohérente (sauf arrêt PG) |
| **Facilité** | Simple, une commande | Nécessite arrêt du serveur |
| **Sécurité** | Protocole réplication | Accès direct aux fichiers |
| **Recommandation** | ✅ Recommandé | ❌ À éviter (sauf PG arrêté) |

### pg_basebackup vs Solutions Tierces

**Barman (Backup and Recovery Manager)** :
- Gestion centralisée de multiples serveurs
- Sauvegardes incrémentales
- PITR simplifié
- Monitoring intégré
- Plus complexe à configurer

**pgBackRest** :
- Très performant (parallélisation, compression)
- Sauvegardes incrémentales et différentielles
- Restauration partielle de bases
- Configuration plus complexe

**WAL-G** :
- Optimisé pour le cloud (S3, Azure, GCS)
- Compression efficace (avec Brotli, Zstandard)
- Sauvegardes incrémentales
- Léger et rapide

**Recommandation** :
- **pg_basebackup** : Simple, intégré, parfait pour débuter ou environnements simples  
- **Barman/pgBackRest/WAL-G** : Environnements complexes, grandes bases, besoins avancés

## Bonnes Pratiques en Production

### ✅ DO - À Faire

1. **Automatiser les sauvegardes** : Cron, systemd timers, ou orchestrateur  
2. **Tester les restaurations** : Au moins mensuellement  
3. **Utiliser -X stream** : Plus fiable que fetch  
4. **Monitorer l'espace disque** : Éviter les sauvegardes échouées par manque d'espace  
5. **Documenter la procédure** : Runbook de restauration clair  
6. **Chiffrer les sauvegardes** : Surtout pour le stockage hors-site  
7. **Utiliser des slots de réplication** : Pour bases à forte activité  
8. **Sauvegarder depuis un standby** : Pour ne pas impacter la production  
9. **Conserver plusieurs générations** : Au minimum 7 jours, idéalement 30 jours  
10. **Vérifier les checksums** : Détecter les corruptions

### ❌ DON'T - À Éviter

1. **Ne pas tester** : Une sauvegarde non testée est inutile  
2. **Stocker uniquement localement** : Risque de perte totale (incendie, panne matérielle)  
3. **Ignorer les erreurs** : Toujours monitorer le succès des sauvegardes  
4. **Utiliser rsync sur un PG actif** : Incohérent et dangereux  
5. **Ne pas monitorer l'espace disque** : Échecs silencieux  
6. **Ne pas documenter** : En cas d'incident, chaque minute compte  
7. **Oublier les objets globaux** : Sauvegarder aussi les rôles (pg_dumpall --globals-only)  
8. **Négliger la sécurité** : Permissions, chiffrement, accès restreint

## Conclusion

Les sauvegardes physiques avec `pg_basebackup` sont un pilier essentiel de toute stratégie de sauvegarde PostgreSQL en production. Elles offrent :

- ✅ **Rapidité** : Sauvegarde et restauration bien plus rapides que les sauvegardes logiques  
- ✅ **PITR** : Capacité de restauration à un point précis dans le temps  
- ✅ **Simplicité** : Outil intégré, une seule commande suffit  
- ✅ **Fiabilité** : Garantie de cohérence grâce au système MVCC  
- ✅ **Flexibilité** : Multiples formats et options d'optimisation

**Recommandation finale** : Pour un système de production, combinez :
- **Sauvegardes physiques** (pg_basebackup) : Pour restauration rapide et PITR  
- **Archivage WAL continu** : Pour RPO minimal  
- **Sauvegardes logiques** (pg_dump) : Pour flexibilité et portabilité  
- **Réplication streaming** : Pour haute disponibilité

Cette approche hybride offre la meilleure protection contre tous types de scénarios de perte de données.
