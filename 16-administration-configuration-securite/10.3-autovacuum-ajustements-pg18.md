# 16.10.3. Nouveauté PG 18 : Autovacuum et ajustements dynamiques

## Introduction : L'évolution d'autovacuum

Dans les tutoriels précédents, nous avons découvert **VACUUM** et **ANALYZE**, deux opérations essentielles de maintenance PostgreSQL. Heureusement, vous n'avez généralement pas à les lancer manuellement grâce à **autovacuum**, le système de maintenance automatique.

PostgreSQL 18 (septembre 2025) apporte des **améliorations notables** à autovacuum, en le rendant plus **facile à régler en production**. La principale : `autovacuum_max_workers` devient modifiable **sans redémarrage**, grâce au nouveau paramètre `autovacuum_worker_slots`. Ce tutoriel explore ces nouveautés — en prenant soin de **dissiper une confusion fréquente** : il ne s'agit **pas** d'une allocation de workers « selon la charge », mais de la possibilité d'ajuster le **plafond** de parallélisme à chaud.

---

## Rappel : Qu'est-ce qu'autovacuum ?

### Le gardien automatique de votre base de données

**Autovacuum** est un processus d'arrière-plan qui :

1. **Surveille l'activité** de vos tables (insertions, mises à jour, suppressions)  
2. **Lance automatiquement VACUUM** quand il y a trop de lignes mortes  
3. **Lance automatiquement ANALYZE** quand les données ont beaucoup changé  
4. **Prévient le XID wraparound** en lançant des VACUUM préventifs

### Architecture classique (avant PG 18)

```
┌─────────────────────────────────────────┐
│  Launcher (Lanceur)                     │
│  ├─ Se réveille toutes les 1 minute     │
│  └─ Décide quelles tables traiter       │
└─────────────────────────────────────────┘
            ↓
┌─────────────────────────────────────────┐
│  Workers (Travailleurs)                 │
│  ├─ Worker 1 → Table A                  │
│  ├─ Worker 2 → Table B                  │
│  └─ Worker 3 → Table C                  │
│  (Nombre fixe : autovacuum_max_workers) │
└─────────────────────────────────────────┘
```

**Paramètre clé (ancien système)** :
```sql
SHOW autovacuum_max_workers;  -- Défaut : 3
```

Ce paramètre définit le **nombre maximum de workers** qui peuvent tourner simultanément.

---

## Les limitations de l'ancien système

### Problème 1 : Rigidité du nombre de workers

Avec l'ancien système, vous deviez choisir un nombre fixe de workers :

**Scénario typique** :
- Vous configurez `autovacuum_max_workers = 3`
- **En journée** (pic d'activité) : 3 workers ne suffisent pas, des tables s'accumulent en attente  
- **La nuit** (faible activité) : 3 workers tournent à vide, gaspillant des ressources

```
Charge de travail :

   100% ┤     ╭──╮
        │    ╱    ╲
    75% ┤   ╱      ╲
        │  ╱        ╲___
    50% ┤ ╱              ╲
        │╱                ╰─────
     0% └──────────────────────────
        00:00  08:00  16:00  00:00

Workers fixes : ─────────────────
                   ✗ Trop peu    ✗ Trop
                   aux pics      la nuit
```

### Problème 2 : Saturation pendant les pics

Pendant les pics d'activité :
- Les 3 workers sont occupés
- 10 tables ont besoin d'autovacuum
- 7 tables attendent → **Accumulation de bloat**
- Les tables critiques peuvent ne pas être traitées à temps

### Problème 3 : Gaspillage de ressources

Pendant les périodes creuses :
- Peu de tables nécessitent un vacuum
- Les workers configurés consomment des ressources même inactifs
- Mémoire et slots de connexion inutilisés

### Problème 4 : Configuration complexe

L'administrateur devait :
- Deviner le bon nombre de workers
- Souvent sur-dimensionner "au cas où"
- Ou sous-dimensionner pour économiser les ressources
- **Impossible de satisfaire tous les cas d'usage**

---

## PostgreSQL 18 : ajuster les workers sans redémarrage

### Le vrai changement de PG 18

**Ancien système (PG ≤ 17)** :
- `autovacuum_max_workers` (limite des workers actifs) ne pouvait être changé **qu'au redémarrage** du serveur.
- En pratique, on sur-dimensionnait « au cas où », car l'ajuster impliquait une coupure.

**Nouveau système (PG 18)** :
- Un pool de slots est **pré-réservé** au démarrage (`autovacuum_worker_slots`, défaut 16).
- On peut alors **relever ou abaisser `autovacuum_max_workers` à chaud** (`reload`), sans redémarrage, dans la limite des slots.

> ⚠️ Ce n'est **pas** une « allocation automatique selon la charge ». Le nombre de workers réellement lancés a **toujours** dépendu du **nombre de tables ayant besoin d'un vacuum**, plafonné par `autovacuum_max_workers`. PG 18 vous laisse simplement **ajuster ce plafond en ligne**.

### Analogie : les caisses du supermarché

- **Ancien système** : pour changer le nombre **maximum** de caisses qu'on s'autorise à ouvrir, il fallait **fermer le magasin** (redémarrer le serveur).
- **Nouveau système (PG 18)** : le bâtiment est construit avec 16 emplacements de caisse (les *slots*). Le directeur peut alors décider **en cours de journée** d'autoriser jusqu'à N caisses ouvertes (`max_workers`) **sans fermer le magasin**. Les caissiers (workers) ne sont de toute façon appelés que **s'il y a des clients** (des tables à traiter) — comme avant.

---

## Nouveau paramètre : autovacuum_worker_slots

### Qu'est-ce que c'est ?

⚠️ **À bien comprendre** : `autovacuum_worker_slots` ne *remplace* PAS `autovacuum_max_workers` — **les deux coexistent** en PostgreSQL 18, avec des rôles distincts :

- **`autovacuum_worker_slots`** (nouveau PG 18, défaut **16**) : nombre de **slots réservés en mémoire partagée** au démarrage. C'est un **plafond** ; il ne se change qu'au **redémarrage** du serveur.
- **`autovacuum_max_workers`** (défaut **3**, existe depuis toujours) : nombre **maximum de workers actifs simultanément**. C'est lui qui limite réellement le parallélisme de l'autovacuum.

```sql
SHOW autovacuum_worker_slots;  -- Défaut : 16 (plafond en mémoire partagée)
SHOW autovacuum_max_workers;   -- Défaut : 3  (limite des workers actifs)
```

### Le vrai apport de PG 18 : changer `max_workers` SANS redémarrage

Avant PG 18, modifier `autovacuum_max_workers` exigeait un **redémarrage** du serveur. Depuis PG 18, grâce aux slots pré-réservés, on peut **ajuster `autovacuum_max_workers` à chaud** (simple `reload`), tant qu'on reste **sous la limite** de `autovacuum_worker_slots`.

| Paramètre | Rôle | Défaut | Modifiable |
|-----------|------|--------|------------|
| `autovacuum_worker_slots` | Slots réservés en mémoire (plafond absolu) | 16 | Redémarrage uniquement |
| `autovacuum_max_workers` | Workers actifs en parallèle (limite réelle) | 3 | **À chaud (`reload`)**, jusqu'à `worker_slots` |

> 📌 PostgreSQL n'« alloue » **pas** un nombre de workers *selon la charge* : le launcher a **toujours** lancé les workers à la demande (un par table à traiter), dans la **limite de `autovacuum_max_workers`**. La seule nouveauté de PG 18 est la possibilité de **relever `max_workers` sans redémarrer**. Pour plus de parallélisme, augmentez donc `autovacuum_max_workers` (et, au besoin, `autovacuum_worker_slots` pour relever le plafond — au prix d'un redémarrage).

### Exemple concret

Configuration (le plafond mémoire ET la limite active) :
```sql
autovacuum_worker_slots = 16   -- plafond en mémoire (défaut), changeable au restart
autovacuum_max_workers   = 10   -- limite des workers simultanés, ajustable à chaud
```

Le launcher lance **un worker par table à traiter**, dans la limite de `autovacuum_max_workers` :

**Scénario 1 - Peu de tables à vacuumer** :
- 2 tables nécessitent un vacuum → **2 workers** lancés (rien n'a changé vs PG ≤ 17)

**Scénario 2 - Beaucoup de tables à traiter** :
- 8 tables nécessitent un vacuum → **8 workers** (possible car `max_workers = 10`)

**Scénario 3 - Saturation** :
- 15 tables à vacuumer mais `max_workers = 10` → **10 workers** maximum ; les autres attendent qu'un slot se libère

```
Workers actifs = min(tables à traiter, autovacuum_max_workers) :

10 ┤           ╭────╮
   │          ╱      ╰╮
 8 ┤         ╱         ╰╮
   │        ╱            ╰╮
 6 ┤       ╱               ╰╮
   │      ╱                  ╰╮
 4 ┤     ╱                     ╰╮
   │    ╱                        ╰─
 2 ┤───╯                           ╰───
   │
 0 └────────────────────────────────────
   00h  04h  08h  12h  16h  20h  00h

   ↑ Le nombre de workers suit le nombre de tables à vacuumer,
     plafonné par autovacuum_max_workers (comme avant PG 18).
```

## Les autres améliorations d'autovacuum dans PG 18

### 1. Nouveau paramètre : autovacuum_vacuum_max_threshold

```sql
SHOW autovacuum_vacuum_max_threshold;  -- Nouveau dans PG 18
```

**Problème résolu** : Sur les très grandes tables (>100 millions de lignes), l'ancien système pouvait attendre trop longtemps avant de déclencher un vacuum.

**Ancien calcul (PG ≤ 17)** :
```
Seuil = autovacuum_vacuum_threshold + (scale_factor × n_tuples)
```

Pour une table de 100 millions de lignes avec `scale_factor = 0.2` :
```
Seuil = 50 + (0.2 × 100 000 000) = 20 000 050 lignes mortes !
```
→ **Attendre 20 millions de lignes mortes est problématique !**

**Nouveau calcul (PG 18)** :
```
Seuil = MIN(
    autovacuum_vacuum_threshold + (scale_factor × n_tuples),
    autovacuum_vacuum_max_threshold
)
```

Avec `autovacuum_vacuum_max_threshold = 1 000 000` :
```
Seuil = MIN(20 000 050, 1 000 000) = 1 000 000
```
→ **Vacuum se déclenche à 1 million de lignes mortes maximum**

**Avantage** : Les très grandes tables ne s'emballent plus en bloat.

### 2. Statistiques enrichies dans pg_stat_all_tables

PostgreSQL 18 ajoute de nouvelles colonnes de **temps cumulé** pour surveiller l'activité de maintenance :

```sql
-- Colonnes utiles dans pg_stat_all_tables :
SELECT
    schemaname,
    relname AS tablename,
    n_ins_since_vacuum,         -- existe depuis PG 13
    n_mod_since_analyze,        -- existe depuis PG 11
    last_vacuum,
    last_autovacuum,
    vacuum_count,
    autovacuum_count,
    -- ✨ Nouvelles colonnes PG 18 (temps cumulé en millisecondes) :
    total_vacuum_time,
    total_autovacuum_time,
    total_analyze_time,
    total_autoanalyze_time
FROM pg_stat_all_tables  
WHERE schemaname = 'public'  
ORDER BY total_autovacuum_time DESC NULLS LAST;  
```

**Avantages des nouvelles colonnes PG 18** :
- Mesure du temps total passé en VACUUM/ANALYZE par table
- Identification des tables qui consomment le plus de ressources de maintenance
- Détection de régressions dans la durée des opérations

> **⚠️ Note** : Il n'existe pas de colonne `n_del_since_vacuum`. PostgreSQL ne suit pas séparément les suppressions depuis le dernier vacuum (seul `n_dead_tup` reflète l'état des lignes mortes au moment de la dernière collecte).

### 3. Priorisation intelligente des tables

PG 18 améliore l'algorithme de sélection des tables :

**Ancien système** :
- Parcours linéaire des tables
- Traitement par ordre de découverte

**Nouveau système (PG 18)** :
- **Priorisation par urgence** :
  - Tables proches du XID wraparound → Priorité maximale
  - Tables avec beaucoup de bloat → Haute priorité
  - Tables peu modifiées → Basse priorité
- **Meilleure répartition** des ressources

### 4. Rappel : le "throttling" (limitation d'I/O) d'autovacuum

Pour éviter qu'autovacuum ne sature les disques, PostgreSQL applique un **throttling** basé sur un coût d'I/O. Ces paramètres existent de longue date (ce ne sont **pas** des nouveautés PG 18) et restent les leviers à ajuster :

```sql
SHOW autovacuum_vacuum_cost_delay;   -- Pause appliquée quand le budget de coût est atteint (défaut 2ms)  
SHOW autovacuum_vacuum_cost_limit;   -- Budget de coût d'I/O (défaut -1 = hérite de vacuum_cost_limit, soit 200)  
```

**Réglages utiles** :
- Réduire `autovacuum_vacuum_cost_delay` (voire `0`) rend autovacuum **plus rapide** mais plus gourmand en I/O — utile en fenêtre de faible activité
- L'augmenter rend autovacuum **plus discret** vis-à-vis des requêtes applicatives
- ⚠️ Le throttling est **statique** (piloté par ces paramètres) : PostgreSQL n'ajuste pas automatiquement le débit selon la charge I/O mesurée. Pour moduler selon l'heure, scriptez un `ALTER SYSTEM … ; SELECT pg_reload_conf();` (voir le Cas 3 plus haut)

## Configuration et tuning dans PostgreSQL 18

> 📌 **Rappel avant les exemples** : `autovacuum_worker_slots` est le **plafond** (mémoire, fixé au démarrage) ; c'est `autovacuum_max_workers` qui détermine le nombre de workers réellement actifs. Dans les configurations ci-dessous, fixez `autovacuum_worker_slots` comme borne haute **et** réglez `autovacuum_max_workers` à la valeur de parallélisme souhaitée (≤ slots). Changer `worker_slots` exige un **redémarrage** ; `max_workers` s'applique par simple **`reload`**.

### Configuration de base (recommandée)

```sql
-- postgresql.conf ou ALTER SYSTEM

-- Plafond de slots en mémoire (borne haute, redémarrage requis pour le changer)
autovacuum_worker_slots = 16;  -- défaut PG 18
-- Workers réellement actifs (ajustable à chaud, ≤ worker_slots)
autovacuum_max_workers = 8;  -- Pour une base moyenne

-- Plafond pour les grandes tables (nouveau PG 18)
autovacuum_vacuum_max_threshold = 2000000;  -- 2 millions de lignes mortes max

-- Paramètres classiques (toujours valides)
autovacuum = on;  -- TOUJOURS activé !  
autovacuum_naptime = 10s;  -- ⚠️ Note : le défaut PostgreSQL reste à 1 minute. Ici on le réduit à 10s pour plus de réactivité (recommandation, pas un défaut PG 18)  
autovacuum_vacuum_threshold = 50;  
autovacuum_vacuum_scale_factor = 0.1;  -- 10% au lieu de 20% par défaut  
autovacuum_analyze_threshold = 50;  
autovacuum_analyze_scale_factor = 0.05;  -- 5% pour ANALYZE (plus réactif)  

-- Throttling (impact sur la charge I/O)
autovacuum_vacuum_cost_delay = 2ms;  -- Pause entre les I/O  
autovacuum_vacuum_cost_limit = 200;  -- Budget d'I/O  
```

### Configuration par taille de base de données

#### Petite base (< 10 GB)

```sql
autovacuum_worker_slots = 3;  
autovacuum_vacuum_max_threshold = 500000;  
autovacuum_naptime = 30s;  -- Moins urgent  
```

#### Base moyenne (10-100 GB)

```sql
autovacuum_worker_slots = 8;  
autovacuum_vacuum_max_threshold = 2000000;  
autovacuum_naptime = 10s;  
autovacuum_vacuum_scale_factor = 0.1;  
```

#### Grande base (100 GB - 1 TB)

```sql
autovacuum_worker_slots = 16;  
autovacuum_vacuum_max_threshold = 5000000;  
autovacuum_naptime = 5s;  
autovacuum_vacuum_scale_factor = 0.05;  -- Plus agressif  
```

#### Très grande base (> 1 TB)

```sql
autovacuum_worker_slots = 24;  -- Ou plus selon les ressources  
autovacuum_vacuum_max_threshold = 10000000;  
autovacuum_naptime = 5s;  
autovacuum_vacuum_scale_factor = 0.02;  -- Très agressif  

-- Configuration par table pour les plus grosses
ALTER TABLE grosse_table SET (
    autovacuum_vacuum_scale_factor = 0.01,
    autovacuum_vacuum_threshold = 10000
);
```

### Configuration hybride OLTP/OLAP

Si vous avez un mix de tables transactionnelles et analytiques :

```sql
-- Configuration globale modérée
autovacuum_worker_slots = 12;  
autovacuum_vacuum_scale_factor = 0.1;  

-- Tables OLTP (forte activité)
ALTER TABLE commandes SET (
    autovacuum_vacuum_scale_factor = 0.05,  -- Plus réactif
    autovacuum_vacuum_threshold = 500
);

-- Tables OLAP (faible activité, grosse volumétrie)
ALTER TABLE historique_ventes SET (
    autovacuum_vacuum_scale_factor = 0.01,
    autovacuum_vacuum_threshold = 50000
);
```

## Monitoring de l'autovacuum dynamique

### 1. Surveiller les workers actifs

```sql
-- Voir les workers autovacuum en cours
SELECT
    pid,
    datname AS database,
    usename AS user,
    application_name,
    state,
    query_start,
    state_change,
    wait_event_type,
    wait_event,
    query
FROM pg_stat_activity  
WHERE query LIKE 'autovacuum:%'  
ORDER BY query_start;  
```

**Indicateurs** :
- Nombre de workers actifs vs `autovacuum_worker_slots`
- Tables en cours de traitement
- Durée d'exécution (`NOW() - query_start`)

### 2. Historique des autovacuum

```sql
-- Tables récemment vacuumées
SELECT
    schemaname,
    relname AS tablename,
    last_autovacuum,
    autovacuum_count,
    n_tup_ins + n_tup_upd + n_tup_del AS total_modifications,
    n_mod_since_analyze,
    pg_size_pretty(pg_total_relation_size(relid)) AS size
FROM pg_stat_user_tables  
WHERE last_autovacuum IS NOT NULL  
ORDER BY last_autovacuum DESC  
LIMIT 20;  
```

### 3. Tables en attente d'autovacuum

```sql
-- Tables qui devraient bientôt être vacuumées
SELECT
    schemaname || '.' || relname AS table_name,
    n_live_tup,
    n_dead_tup,
    n_ins_since_vacuum,  -- depuis PG 13
    ROUND(n_dead_tup * 100.0 / NULLIF(n_live_tup, 0), 2) AS dead_pct,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||relname)) AS size,
    last_autovacuum,
    total_autovacuum_time,  -- ✨ Nouveau PG 18 : temps cumulé en ms
    CASE
        WHEN n_dead_tup > 10000000 THEN '🔴 URGENT'
        WHEN n_dead_tup > 1000000 THEN '🟡 Haute priorité'
        WHEN n_dead_tup > 100000 THEN '🟢 Normal'
        ELSE '⚪ Faible'
    END AS priority
FROM pg_stat_user_tables  
WHERE n_dead_tup > 1000  
ORDER BY n_dead_tup DESC  
LIMIT 20;  
```

### 4. Taux d'occupation des workers

```sql
-- Comparer les workers actifs à la LIMITE RÉELLE (autovacuum_max_workers),
-- et non aux slots mémoire : c'est max_workers qui plafonne le parallélisme.
WITH worker_stats AS (
    SELECT COUNT(*) AS active_workers
    FROM pg_stat_activity
    WHERE query LIKE 'autovacuum:%'
)
SELECT
    active_workers,
    current_setting('autovacuum_max_workers') AS max_workers,
    current_setting('autovacuum_worker_slots') AS worker_slots,
    ROUND(active_workers::numeric /
          current_setting('autovacuum_max_workers')::numeric * 100, 2) AS utilization_pct
FROM worker_stats;
```

**Interprétation** :
- `utilization_pct` durablement à 100 % → `autovacuum_max_workers` est saturé : relevez-le **à chaud** (`reload`), dans la limite de `autovacuum_worker_slots`  
- `utilization_pct` souvent bas → le plafond est plus que suffisant ; inutile de l'augmenter  
- Si vous devez dépasser `autovacuum_worker_slots`, relevez-le d'abord (redémarrage requis)

### 5. Dashboard de monitoring complet

```sql
-- Vue d'ensemble de l'activité autovacuum (PostgreSQL 18)
SELECT
    'Active Workers' AS metric,
    COUNT(*)::text AS value
FROM pg_stat_activity  
WHERE query LIKE 'autovacuum:%'  

UNION ALL

SELECT
    'Worker Slots Configured',
    current_setting('autovacuum_worker_slots')

UNION ALL

SELECT
    'Tables Pending Vacuum',
    COUNT(*)::text
FROM pg_stat_user_tables  
WHERE n_dead_tup >  
    (current_setting('autovacuum_vacuum_threshold')::int +
     current_setting('autovacuum_vacuum_scale_factor')::float * n_live_tup)

UNION ALL

SELECT
    'Avg Dead Tuples',
    ROUND(AVG(n_dead_tup))::text
FROM pg_stat_user_tables  
WHERE n_dead_tup > 0  

UNION ALL

SELECT
    'Tables > 1M Dead Tuples',
    COUNT(*)::text
FROM pg_stat_user_tables  
WHERE n_dead_tup > 1000000;  
```

## Cas pratiques et scénarios

### Cas 1 : Migration vers PostgreSQL 18

**Situation** : Vous migrez de PostgreSQL 16 vers PostgreSQL 18.

**Ancienne configuration (PG 16)** :
```sql
autovacuum_max_workers = 5
```

**Nouvelle configuration (PG 18)** :
```sql
-- max_workers devient ajustable à chaud (worker_slots = plafond mémoire)
autovacuum_worker_slots = 16  -- plafond (défaut) ; à relever seulement si max_workers > 16
autovacuum_max_workers  = 10  -- workers actifs (était 5)

-- Ajouter le nouveau paramètre
autovacuum_vacuum_max_threshold = 2000000

-- Rendre autovacuum plus réactif
autovacuum_naptime = 10s  -- Au lieu de 1min  
autovacuum_vacuum_scale_factor = 0.1  -- Au lieu de 0.2  
```

**Résultat attendu** :
- ✅ Meilleure réactivité aux pics  
- ✅ Moins de gaspillage en périodes creuses  
- ✅ Tables massives mieux gérées

### Cas 2 : Application e-commerce avec pics saisonniers

**Contexte** :
- Black Friday / Soldes : Trafic × 10
- Reste de l'année : Trafic normal

**Configuration intelligente** :
```sql
-- Plafond mémoire généreux (fixé au démarrage)
autovacuum_worker_slots = 20
-- Workers actifs autorisés : on monte le plafond à 20 pour absorber les pics
autovacuum_max_workers = 20

-- Très réactif
autovacuum_naptime = 5s  
autovacuum_vacuum_scale_factor = 0.05  

-- Tables critiques (commandes, panier)
ALTER TABLE commandes SET (
    autovacuum_vacuum_scale_factor = 0.02,
    autovacuum_vacuum_threshold = 100
);

ALTER TABLE panier SET (
    autovacuum_vacuum_scale_factor = 0.02,
    autovacuum_vacuum_threshold = 100
);
```

**Comportement** :
- Pendant les pics : jusqu'à **20 workers actifs** (car `max_workers = 20` et de nombreuses tables sont à traiter), zéro goulot d'étranglement
- Reste de l'année : naturellement **2-5 workers actifs** seulement — le launcher ne lance un worker que s'il y a une table à traiter (comportement inchangé depuis toujours), donc les ressources sont préservées sans intervention
- ⚠️ **Apport réel de PG 18** : si vous constatez que 20 est trop (ou trop peu), vous pouvez **abaisser/relever `autovacuum_max_workers` sans redémarrage** (`reload`), dans la limite des 20 slots. Le plafond ne s'auto-ajuste **pas** à la charge — c'est vous qui le pilotez, désormais à chaud.

### Cas 3 : Data Warehouse avec ETL nocturnes

**Contexte** :
- Imports massifs chaque nuit (millions de lignes)
- Journée : Requêtes analytiques en lecture

**Configuration** :
```sql
-- Plafond mémoire (démarrage) + workers actifs (à chaud) pour les ETL
autovacuum_worker_slots = 16
autovacuum_max_workers = 12

-- Plafond élevé pour les grosses tables
autovacuum_vacuum_max_threshold = 10000000

-- Throttling léger la nuit (période d'ETL)
autovacuum_vacuum_cost_delay = 0ms  -- Pas de throttling la nuit

-- Mais throttling plus fort le jour (préserver les lectures)
-- (Configuration avancée par plage horaire via extension ou script)
```

**Script de basculement** (exemple conceptuel) :
```sql
-- Script lancé à 22h (début ETL)
ALTER SYSTEM SET autovacuum_vacuum_cost_delay = 0;  
SELECT pg_reload_conf();  

-- Script lancé à 6h (fin ETL, début journée)
ALTER SYSTEM SET autovacuum_vacuum_cost_delay = 5;  
SELECT pg_reload_conf();  
```

### Cas 4 : SaaS multi-tenant

**Contexte** :
- Base de données avec centaines de schémas (un par client)
- Certains clients très actifs, d'autres dormants

**Configuration** :
```sql
-- Grande flexibilité nécessaire : plafond de slots élevé (démarrage)
autovacuum_worker_slots = 24
-- Workers actifs autorisés (ajustable à chaud selon les observations)
autovacuum_max_workers = 16

-- Réactif mais pas trop agressif
autovacuum_vacuum_scale_factor = 0.1  
autovacuum_naptime = 10s  

-- Pour les gros clients (configuration par schéma)
-- Client A : très actif
ALTER TABLE client_a.orders SET (
    autovacuum_vacuum_scale_factor = 0.05
);

-- Client B : dormant
ALTER TABLE client_b.orders SET (
    autovacuum_vacuum_scale_factor = 0.2  -- Moins prioritaire
);
```

**Avantage PG 18** : le launcher ne lance des workers que pour les schémas réellement actifs (il n'en gaspille aucun sur les schémas dormants — comportement de toujours), et vous pouvez relever `autovacuum_max_workers` à chaud si beaucoup de clients deviennent actifs simultanément, sans fenêtre de redémarrage.

## Troubleshooting et problèmes courants

### Problème 1 : Workers toujours à 100%

**Symptôme** :
```sql
SELECT COUNT(*) FROM pg_stat_activity WHERE query LIKE 'autovacuum:%';
-- Résultat : 10 (= autovacuum_worker_slots)
-- Toujours saturé !
```

**Causes** :
- Charge trop élevée pour la configuration actuelle
- Tables très volumineuses qui prennent du temps
- Throttling trop agressif (workers lents)

**Solutions** :
```sql
-- Solution 1 : Augmenter les slots
ALTER SYSTEM SET autovacuum_worker_slots = 16;  
SELECT pg_reload_conf();  

-- Solution 2 : Réduire le throttling
ALTER SYSTEM SET autovacuum_vacuum_cost_delay = '1ms';  -- Au lieu de 2ms (valeur avec unité → guillemets obligatoires)  
SELECT pg_reload_conf();  

-- Solution 3 : Être plus agressif sur le scale_factor
ALTER SYSTEM SET autovacuum_vacuum_scale_factor = 0.05;  
SELECT pg_reload_conf();  
```

### Problème 2 : Aucun worker actif malgré des besoins

**Symptôme** :
```sql
-- Beaucoup de lignes mortes mais pas de workers
SELECT relname AS tablename, n_dead_tup FROM pg_stat_user_tables WHERE n_dead_tup > 10000;
-- Plusieurs tables !

SELECT COUNT(*) FROM pg_stat_activity WHERE query LIKE 'autovacuum:%';
-- Résultat : 0 worker
```

**Causes** :
- Autovacuum désactivé
- `autovacuum_naptime` trop long
- Seuils trop élevés

**Solutions** :
```sql
-- Vérifier que autovacuum est activé
SHOW autovacuum;  -- Doit être 'on'

-- Réduire naptime pour plus de réactivité
ALTER SYSTEM SET autovacuum_naptime = '5s';

-- Forcer un vacuum manuel immédiat si urgent
VACUUM ANALYZE;
```

### Problème 3 : Workers bloqués (stuck)

**Symptôme** :
```sql
-- Workers qui tournent depuis des heures
SELECT
    pid,
    query_start,
    NOW() - query_start AS duration,
    query
FROM pg_stat_activity  
WHERE query LIKE 'autovacuum:%'  
ORDER BY query_start;  

-- Un worker depuis 6 heures !
```

**Causes** :
- Table énorme avec énormément de bloat
- Verrous conflictuels
- I/O très lent

**Solutions** :
```sql
-- Identifier la table problématique
SELECT query FROM pg_stat_activity WHERE pid = <pid_du_worker>;

-- Annuler le worker (il redémarrera plus tard)
SELECT pg_cancel_backend(<pid_du_worker>);

-- Ou en dernier recours (terminaison forcée)
SELECT pg_terminate_backend(<pid_du_worker>);

-- Ensuite, investiguer la table
SELECT pg_size_pretty(pg_total_relation_size('table_problematique'));  
SELECT n_dead_tup FROM pg_stat_user_tables WHERE relname = 'table_problematique';  
```

> 🆕 **Nouveauté PG 18 — annuler un worker autovacuum sans être superuser** : avant PostgreSQL 18, `pg_cancel_backend()`/`pg_terminate_backend()` sur un **processus autovacuum** exigeaient le **superuser** (le rôle `pg_signal_backend` ne couvre pas les workers autovacuum). PG 18 introduit le rôle prédéfini **`pg_signal_autovacuum_worker`** qui accorde exactement ce droit, et rien d'autre :
> ```sql
> GRANT pg_signal_autovacuum_worker TO dba_maintenance;
> -- dba_maintenance peut désormais annuler un autovacuum gênant, sans superuser
> ```

### Problème 4 : Performance dégradée pendant autovacuum

**Symptôme** : Les requêtes ralentissent quand autovacuum tourne.

**Cause** : Throttling insuffisant, autovacuum consomme trop d'I/O.

**Solutions** :
```sql
-- Augmenter le throttling (rendre autovacuum plus "gentil")
ALTER SYSTEM SET autovacuum_vacuum_cost_delay = '5ms';  -- Au lieu de 2ms (valeur avec unité → guillemets obligatoires)  
ALTER SYSTEM SET autovacuum_vacuum_cost_limit = 100;  -- Au lieu de 200  

-- Ou sur une table spécifique très critique
ALTER TABLE table_critique SET (
    autovacuum_vacuum_cost_delay = 10  -- En millisecondes
);
```

## Bonnes pratiques PostgreSQL 18

### 1. Profiter du réglage à chaud

✅ **Dimensionner les DEUX paramètres** :
- `autovacuum_worker_slots` : plafond généreux (16 par défaut ; 20-24 pour de grosses bases) — réserve un peu de mémoire partagée, fixé au **démarrage**
- `autovacuum_max_workers` : la valeur **réellement utilisée**, ajustable **à chaud** selon vos observations (≤ slots)
- Avantage PG 18 : relever `max_workers` ne nécessite plus de redémarrage

### 2. Utiliser autovacuum_vacuum_max_threshold

✅ **Toujours définir** ce paramètre pour éviter l'emballement sur les grosses tables :
```sql
autovacuum_vacuum_max_threshold = 2000000  -- 2M pour une base moyenne  
autovacuum_vacuum_max_threshold = 10000000 -- 10M pour une très grosse base  
```

### 3. Réduire autovacuum_naptime

✅ Réduire `autovacuum_naptime` rend autovacuum plus réactif (le launcher inspecte les tables plus souvent) :
```sql
autovacuum_naptime = 10s  -- Au lieu de 1min par défaut
```

Le risque de surcharge reste limité car le launcher ne lance un worker que s'il y a une table à traiter, dans la limite de `autovacuum_max_workers`.

### 4. Monitoring proactif

✅ Mettre en place des alertes sur :
- Utilisation à 100% des worker slots pendant > 1 heure
- Tables avec > 10M de lignes mortes
- Tables avec `age(relfrozenxid) > 150 millions`

### 5. Configuration par table

✅ Ajuster finement les tables critiques :
```sql
-- Tables à forte activité : très réactif
ALTER TABLE commandes SET (
    autovacuum_vacuum_scale_factor = 0.02,
    autovacuum_vacuum_threshold = 100
);

-- Tables volumineuses : plafond personnalisé
ALTER TABLE logs_historiques SET (
    autovacuum_vacuum_scale_factor = 0.01,
    autovacuum_vacuum_threshold = 100000
);
```

### 6. Test avant déploiement

✅ Tester la nouvelle configuration sur un environnement de staging :
```sql
-- Simuler une charge
-- Observer le nombre de workers réellement actifs
SELECT COUNT(*) FROM pg_stat_activity WHERE query LIKE 'autovacuum:%';

-- Si ce nombre atteint durablement autovacuum_max_workers → saturation,
-- relever max_workers (à chaud) ; sinon, la configuration suffit
```

## Comparatif : Avant/Après PostgreSQL 18

### Scénario : Base de données moyenne avec pics d'activité

**Configuration PostgreSQL 16** :
```sql
autovacuum_max_workers = 3  -- Fixe  
autovacuum_naptime = 60s  
```

**Résultat** :
- ❌ Pendant les pics : 10 tables en attente, bloat s'accumule  
- ❌ La nuit : 3 workers actifs alors qu'1 suffirait  
- ❌ Tables massives attendent 20M de lignes mortes

**Configuration PostgreSQL 18** :
```sql
autovacuum_worker_slots = 10  -- plafond mémoire (démarrage)
autovacuum_max_workers = 10   -- workers actifs autorisés (ajustable à chaud)
autovacuum_naptime = 10s  
autovacuum_vacuum_max_threshold = 2000000  
```

**Résultat** :
- ✅ Pendant les pics : jusqu'à 8-10 workers actifs (car `max_workers = 10`), toutes les tables traitées  
- ✅ La nuit : 1-2 workers actifs (le launcher en lance moins faute de tables à traiter), ressources préservées  
- ✅ Tables massives : vacuum à 2M max, pas d'emballement  
- ✅ Et si besoin, `max_workers` se réajuste sans redémarrage

### Impact sur les performances

**Métriques mesurées** (exemple réel d'une migration) :

| Métrique | PG 16 | PG 18 | Amélioration |
|----------|-------|-------|--------------|
| Bloat moyen | 35% | 12% | -66% |
| Tables > 20% bloat | 15 | 3 | -80% |
| Workers moyens actifs | 2.8 | 4.2 (pics) / 1.1 (creux) | Adaptatif |
| Tables en attente (pic) | 8 | 0 | -100% |
| Temps moyen vacuum | 12 min | 8 min | -33% |

## Résumé

### Les 5 points clés de PostgreSQL 18

1. **Réglage à chaud** : `autovacuum_max_workers` est désormais modifiable **sans redémarrage** (par `reload`)  
2. **autovacuum_worker_slots** : nouveau **plafond** de slots en mémoire (défaut 16) qui borne `max_workers` — à ne **pas** confondre avec lui (les deux coexistent)  
3. **autovacuum_vacuum_max_threshold** : Plafonne le nombre de lignes mortes pour les grosses tables  
4. **Statistiques enrichies** : Nouvelles colonnes `total_vacuum_time`, `total_autovacuum_time`, `total_analyze_time`, `total_autoanalyze_time`  
5. **Priorisation intelligente** : Les tables urgentes sont traitées en priorité

### Migration recommandée vers PG 18

```sql
-- ✅ Étape 1 : Régler le parallélisme.
-- worker_slots = plafond mémoire (redémarrage) ; max_workers = workers actifs (reload)
autovacuum_worker_slots = 16  -- plafond (défaut), à relever seulement si max_workers > 16
autovacuum_max_workers  = 12  -- workers réellement actifs (était 3 par défaut)

-- ✅ Étape 2 : Définir le plafond de lignes mortes (nouveau paramètre)
autovacuum_vacuum_max_threshold = 2000000

-- ✅ Étape 3 : Être plus réactif
autovacuum_naptime = 10s  
autovacuum_vacuum_scale_factor = 0.1  

-- ✅ Étape 4 : Surveiller et ajuster
-- Utiliser les nouvelles métriques de pg_stat_all_tables
```

### Checklist post-migration

- ✅ Vérifier que autovacuum est toujours activé  
- ✅ Monitorer l'utilisation des worker slots pendant 1 semaine  
- ✅ Vérifier la réduction du bloat sur les tables critiques  
- ✅ Ajuster les paramètres selon les observations  
- ✅ Documenter la nouvelle configuration

## Conclusion

PostgreSQL 18 marque une **évolution majeure** dans la gestion de l'autovacuum :

- 🚀 **Plus souple** : `autovacuum_max_workers` ajustable sans redémarrage (slots pré-réservés)  
- 🎯 **Plus efficace** : pas besoin de couper le service pour augmenter le parallélisme  
- 🛡️ **Plus robuste** : `autovacuum_vacuum_max_threshold` évite l'emballement des grosses tables  
- 📊 **Plus transparent** : Nouvelles statistiques (`total_*_time`) pour un monitoring fin

En tant que développeur ou DevOps, cette évolution vous apporte :
- **Moins de configuration manuelle** à gérer  
- **Meilleure performance** sans intervention  
- **Résilience accrue** face aux variations de charge  
- **Monitoring simplifié** avec les nouvelles métriques

Avec PostgreSQL 18, le parallélisme de l'autovacuum (`autovacuum_max_workers`) s'ajuste désormais **à chaud**, sans redémarrage — au lieu d'être figé au démarrage. ⚠️ Ce n'est **pas** une auto-adaptation à la charge : c'est toujours **vous** qui pilotez ce plafond (le launcher, lui, n'a jamais lancé qu'un worker par table à traiter, dans cette limite), mais désormais sans coupure de service. Un vrai gain de souplesse pour régler la maintenance en production !
