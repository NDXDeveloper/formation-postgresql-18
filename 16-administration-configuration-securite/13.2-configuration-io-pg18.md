# 16.13.2. Nouveauté PostgreSQL 18 : Configuration du Sous-Système I/O

## Introduction

PostgreSQL 18 (septembre 2025) introduit l'une des améliorations de performance les plus significatives de son histoire : le **sous-système I/O asynchrone (AIO)**. Cette nouveauté peut améliorer les performances jusqu'à **3× plus rapides** pour certaines charges de travail, notamment sur les opérations de lecture/écriture intensives.

Dans cette section, nous allons explorer :
- Ce qu'est le sous-système I/O et pourquoi il est crucial
- La différence entre les modes synchrone et asynchrone
- Comment configurer et tirer parti de cette nouveauté
- Les cas d'usage optimaux et les pièges à éviter

> **🆕 Nouveauté PostgreSQL 18** : C'est la première fois que PostgreSQL intègre nativement un système d'I/O asynchrone complet, marquant une évolution majeure dans la gestion des performances.

---

## Comprendre les I/O (Entrées/Sorties)

### Qu'est-ce qu'une opération I/O ?

Une opération **I/O (Input/Output)** est toute interaction entre PostgreSQL et le système de stockage (disque dur, SSD, etc.) pour :
- **Lire** des données depuis le disque vers la mémoire  
- **Écrire** des données depuis la mémoire vers le disque

> **💡 Analogie** : Imaginez que vous travaillez sur un projet et que vous avez besoin de documents stockés dans un classeur. L'opération I/O, c'est le fait d'aller chercher ces documents dans le classeur (lecture) ou d'y ranger de nouveaux documents (écriture).

### Pourquoi les I/O sont-ils critiques ?

Les opérations I/O sont généralement **le goulot d'étranglement n°1** des performances dans une base de données :

| Opération | Temps Approximatif | Vitesse Relative |
|-----------|-------------------|------------------|
| Accès RAM | 100 nanosecondes | ⚡⚡⚡⚡⚡ (Référence) |
| Accès SSD | 0.1 milliseconde | 🐌 1,000× plus lent |
| Accès HDD | 10 millisecondes | 🐌🐌 100,000× plus lent |

**Conclusion** : Même avec un SSD moderne, une opération disque est **1,000 fois plus lente** que la mémoire. C'est pourquoi optimiser les I/O est crucial !

## Le Modèle Historique : I/O Synchrone

### Comment fonctionnait PostgreSQL avant la version 18 ?

Jusqu'à PostgreSQL 17, le système utilisait principalement un modèle **d'I/O synchrone** (bloquant).

### Principe de l'I/O Synchrone

Lorsqu'un processus PostgreSQL doit lire une donnée :

```
1. Processus PostgreSQL : "J'ai besoin du bloc de données #12345"
2. Demande d'I/O au système d'exploitation
3. ⏸️ ATTENTE (le processus est BLOQUÉ)
4. Le disque lit les données
5. ⏸️ ATTENTE (le processus est toujours BLOQUÉ)
6. Les données arrivent en mémoire
7. ✅ Le processus peut continuer
```

> **💡 Analogie** : C'est comme si vous alliez au guichet unique d'une administration. Vous faites la queue, vous attendez (bloqué), le fonctionnaire traite votre demande, et seulement après, vous pouvez partir. Pendant toute cette attente, vous ne pouvez rien faire d'autre.

### Le Problème du Mode Synchrone

**Temps CPU Gaspillé** : Pendant qu'un processus attend une opération I/O, le CPU reste inactif pour ce processus.

**Schéma temporel d'un processus** :
```
[Calcul CPU] → [⏸️ Attente I/O] → [Calcul CPU] → [⏸️ Attente I/O] → [Calcul CPU]
   0.1 ms         10 ms            0.1 ms         10 ms            0.1 ms
```

**Résultat** : Le processus passe **99% de son temps à attendre** ! Le CPU est sous-utilisé.

### Tentatives de Solutions Historiques

Pour compenser ce problème, PostgreSQL utilisait plusieurs stratégies :
- **Préchargement (prefetching)** : Deviner les données nécessaires à l'avance  
- **Parallélisation** : Lancer plusieurs processus  
- **Caching agressif** : Garder un maximum en RAM

Mais aucune ne résolvait le problème fondamental : **l'attente bloquante**.

---

## La Révolution : I/O Asynchrone dans PostgreSQL 18

### Principe de l'I/O Asynchrone

Avec le mode asynchrone, un processus peut **initier une opération I/O et continuer à travailler** pendant que le disque effectue la lecture/écriture.

```
1. Processus PostgreSQL : "J'ai besoin du bloc #12345"
2. Demande d'I/O asynchrone au système
3. ✅ Le processus CONTINUE immédiatement à travailler sur autre chose
4. En arrière-plan : Le disque lit les données
5. Notification : "Les données du bloc #12345 sont prêtes"
6. Le processus utilise les données quand il en a besoin
```

> **💡 Analogie Améliorée** : Au lieu d'attendre au guichet, vous déposez votre demande et vous recevez un numéro. Pendant que votre dossier est traité en arrière-plan, vous pouvez aller faire vos courses, travailler, etc. Quand votre numéro s'affiche, vous revenez chercher le résultat. Vous n'avez pas perdu de temps à attendre !

### Schéma Comparatif

**Mode Synchrone (ancien)** :
```
Processus A: [Calcul] → [⏸️⏸️⏸️ Attente I/O ⏸️⏸️⏸️] → [Calcul]
              0.1ms          10ms                    0.1ms
```

**Mode Asynchrone (PostgreSQL 18)** :
```
Processus A: [Calcul] → [Lancement I/O] → [Calcul sur autre chose] → [Récupération résultat] → [Calcul]
              0.1ms        0.01ms            10ms                      0.01ms                  0.1ms

En parallèle: I/O en arrière-plan [████████████████]
```

**Gain de temps** : Le CPU reste productif pendant l'attente I/O !

## Architecture Technique du Sous-Système AIO

### Comment PostgreSQL 18 Implémente l'I/O Asynchrone

PostgreSQL 18 utilise les APIs natives du système d'exploitation :

| Système d'Exploitation | API Utilisée | Support |
|------------------------|--------------|---------|
| **Linux** | io_uring (kernel 5.1+) | ✅ Optimal |
| **Windows** | Workers dédiés (io_method='worker') | ✅ Multi-plateforme |
| **macOS / BSD** | kqueue + aio | ✅ Supporté |

#### Linux : io_uring (Le Plus Performant)

**io_uring** est une interface moderne du noyau Linux qui permet :
- Soumission batch de plusieurs opérations I/O en une seule syscall
- Réduction drastique du nombre d'appels système (overhead)
- Polling efficace des résultats

> **📊 Résultats de Benchmarks** : Sur Linux avec io_uring, PostgreSQL 18 peut être **2.5× à 3× plus rapide** sur des charges de travail I/O intensives comparé à PostgreSQL 17.

## Configuration : io_method

### Le Nouveau Paramètre : io_method

PostgreSQL 18 introduit le paramètre `io_method` qui contrôle le comportement du sous-système I/O.

```conf
# Dans postgresql.conf
io_method = 'worker'   # Défaut : I/O asynchrone via processus workers dédiés
# ou
io_method = 'io_uring' # Linux uniquement : performances maximales (kernel 5.1+)
# ou
io_method = 'sync'     # Mode synchrone traditionnel (comme PG 17)
```

### Valeurs Possibles

| Valeur | Description | Quand l'Utiliser |
|--------|-------------|------------------|
| `'worker'` | I/O asynchrone via processus workers dédiés | **Défaut PG 18** — fonctionne sur toutes les plateformes |
| `'io_uring'` | I/O asynchrone via io_uring (Linux kernel 5.1+) | **Recommandé sur Linux** pour performances maximales |
| `'sync'` | I/O synchrone traditionnel (comportement PG 17) | Compatibilité, debugging, ou systèmes sans support AIO |

> 💡 Il n'y a que ces 3 valeurs. Le nombre de workers I/O est contrôlé par le paramètre `io_workers` (défaut : 3).

### Valeur par Défaut dans PostgreSQL 18

**Sur toutes les plateformes** :
- Défaut → `io_method = 'worker'` (AIO via processus dédiés, multi-plateforme)
- Linux avec kernel 5.1+ et liburing → possibilité de passer à `'io_uring'` pour de meilleures performances

**Pour revenir au comportement PG 17** :
- Fallback automatique vers `io_method = 'sync'`

> **✅ Bonne nouvelle** : PostgreSQL détecte automatiquement les capacités du système et choisit le meilleur mode. Dans la plupart des cas, vous n'avez **rien à configurer** !

## Comparaison Détaillée : Sync vs Async

### Tableau Comparatif

| Critère | Mode Synchrone (`sync`) | Mode Asynchrone (`async`) |
|---------|------------------------|---------------------------|
| **Performance globale** | Référence (100%) | 150-300% selon charge |
| **Latence I/O** | Bloquante | Non-bloquante |
| **Utilisation CPU** | Sous-optimale (attentes) | Optimale (travail continu) |
| **Throughput (débit)** | Limité | Très élevé |
| **Concurrence** | Chaque connexion attend | Parallélisation effective |
| **Overhead système** | Élevé (nombreux syscalls) | Faible (batch operations) |
| **Compatibilité** | Universelle | Nécessite OS moderne |
| **Complexité** | Simple | Complexe (géré par PG) |
| **Débogage** | Facile | Plus complexe |

### Cas d'Usage Optimaux

#### Quand le Mode Asynchrone Brille ⭐

**1. Workloads I/O Intensifs**
```
Exemples :
- Scans de grosses tables (pas en cache)
- Requêtes analytiques sur data warehouse
- Backups (pg_basebackup)
- VACUUM et REINDEX sur grosses bases
- Chargement massif de données (COPY)
```

**Gains attendus** : 2× à 3× plus rapide

**2. Charges OLTP avec SSD Rapides**
```
Contexte :
- Nombreuses connexions concurrentes
- Mix lecture/écriture
- Base de données plus grande que la RAM
```

**Gains attendus** : 30% à 80% de réduction de latence

**3. Réplication Physique et WAL Shipping**
```
Bénéfices :
- Réplication plus rapide (moins de lag)
- WAL writing optimisé
- Meilleur throughput sur standby servers
```

#### Quand le Mode Synchrone Reste Valide

**1. Systèmes Legacy**
```
- Kernel Linux < 5.10
- Windows Server 2012 ou antérieur
- Systèmes embarqués avec limitations
```

**2. Debugging et Diagnostics**
```
Raison : Le mode synchrone est plus simple à tracer et debugger  
Usage : Isoler un problème de performance spécifique  
```

**3. Bases de Données Entièrement en Cache**
```
Contexte :
- Base < RAM disponible
- Cache hit ratio > 99.9%
- Très peu d'I/O réels
```

**Résultat** : Le gain de l'async est négligeable (car peu d'I/O)

## Configurations Avancées

### Paramètres Complémentaires

PostgreSQL 18 introduit plusieurs paramètres pour fine-tuner le sous-système I/O :

#### 1. io_combine_limit

```conf
io_combine_limit = 128kB  # Défaut
```

**Rôle** : Définit la taille maximum des opérations I/O combinées en mode asynchrone.

**Impact** :
- Valeur plus élevée → Moins d'opérations I/O, mais plus grosses
- Valeur plus basse → Plus d'opérations I/O, mais plus petites

**Recommandations** :
- **SSD NVMe** : 256 kB - 512 kB (profite du parallélisme)  
- **SSD SATA** : 128 kB (défaut optimal)  
- **HDD** : 64 kB - 128 kB (séquentiel préféré)

#### 2. maintenance_io_concurrency

```conf
maintenance_io_concurrency = 10  # Défaut
```

**Rôle** : Nombre d'opérations I/O asynchrones simultanées pour les opérations de maintenance (VACUUM, CREATE INDEX, etc.)

**Recommandations** :
- **SSD Performant** : 10-20  
- **SSD Standard** : 5-10  
- **HDD** : 2-4

#### 3. effective_io_concurrency

```conf
effective_io_concurrency = 1  # Défaut (par table)
```

**Rôle** : Indice pour le planificateur sur le nombre d'opérations I/O concurrentes que le système peut gérer.

**Recommandations** :
- **HDD** : 1-2 (recherches séquentielles)  
- **SSD SATA** : 4-8  
- **SSD NVMe** : 16-32  
- **RAID SSD** : 32-64

### Exemple de Configuration Optimale

#### Pour un Serveur avec SSD NVMe

```conf
# PostgreSQL 18 - Configuration I/O optimisée pour SSD NVMe

# Sous-système I/O asynchrone
io_method = 'worker'                    # Mode asynchrone (auto-détecté)  
io_combine_limit = 256kB               # Combine les opérations  

# Concurrence I/O
effective_io_concurrency = 32          # SSD NVMe très parallèle  
maintenance_io_concurrency = 20        # Maintenances rapides  

# Autres paramètres liés
random_page_cost = 1.1                 # SSD → coût aléatoire faible  
seq_page_cost = 1.0                    # Référence  

# WAL optimisé
wal_sync_method = 'fdatasync'          # Optimal sur Linux  
wal_compression = on                   # Réduit I/O WAL  
```

#### Pour un Serveur avec HDD ou SSD SATA

```conf
# PostgreSQL 18 - Configuration I/O pour stockage standard

# Sous-système I/O asynchrone
io_method = 'worker'                    # Mode asynchrone  
io_combine_limit = 128kB               # Défaut  

# Concurrence I/O
effective_io_concurrency = 4           # SATA moins parallèle  
maintenance_io_concurrency = 10        # Défaut  

# Autres paramètres liés
random_page_cost = 4.0                 # HDD : accès aléatoire coûteux  
seq_page_cost = 1.0                    # Référence  

# WAL
wal_sync_method = 'fdatasync'
```

## Migration et Adoption

### Dois-je Migrer Immédiatement vers PostgreSQL 18 ?

**✅ OUI si** :
- Vous avez des charges de travail I/O intensives
- Votre base dépasse largement la RAM disponible
- Vous utilisez des SSD modernes (surtout NVMe)
- Vous êtes sur Linux avec kernel 5.10+

**⏸️ ATTENDEZ si** :
- Votre système est très ancien (Windows Server 2012, Linux kernel < 5.10)
- Votre base tient entièrement en RAM (cache hit > 99%)
- Vous êtes en phase critique (attendez une fenêtre de maintenance)

### Plan de Migration

#### Phase 1 : Préparation (Avant Upgrade)

```bash
# 1. Vérifier la compatibilité du système
uname -r  # Linux : vérifier kernel >= 5.10

# 2. Benchmarker les performances actuelles
pgbench -c 10 -j 2 -T 300 your_database

# 3. Documenter les métriques I/O actuelles
SELECT * FROM pg_stat_database WHERE datname = 'your_database';
```

#### Phase 2 : Upgrade vers PostgreSQL 18

```bash
# Utiliser pg_upgrade (préserve les stats avec PG 18 !)
pg_upgrade \
  --old-bindir=/usr/lib/postgresql/17/bin \
  --new-bindir=/usr/lib/postgresql/18/bin \
  --old-datadir=/var/lib/postgresql/17/main \
  --new-datadir=/var/lib/postgresql/18/main \
  --jobs=4  # Nouveau : parallélisation des vérifications
```

#### Phase 3 : Vérification et Tuning

```sql
-- Vérifier que le mode async est actif
SHOW io_method;
-- Résultat attendu : 'async' (ou 'io_uring' sur Linux)

-- Vérifier les statistiques I/O (nouveau dans PG 18)
SELECT * FROM pg_stat_io;

-- Surveiller les performances
SELECT
    datname,
    blks_read,
    blks_hit,
    blk_read_time,
    blk_write_time
FROM pg_stat_database;
```

#### Phase 4 : Benchmarking Post-Migration

```bash
# Re-benchmarker avec les mêmes paramètres
pgbench -c 10 -j 2 -T 300 your_database

# Comparer les résultats
# Attendu : 30-80% d'amélioration sur charges I/O intensives
```

## Monitoring et Observabilité

### Nouvelles Vues Système dans PostgreSQL 18

#### pg_stat_io : Statistiques I/O Détaillées

```sql
-- Vue globale des I/O par type d'opération
SELECT
    backend_type,
    object,
    context,
    reads,
    writes,
    read_time,
    write_time
FROM pg_stat_io  
ORDER BY read_time DESC;  
```

**Interprétation** :
- `reads` : Nombre total de lectures  
- `read_time` : Temps total passé en lecture (millisecondes)  
- `backend_type` : Type de processus (client backend, autovacuum, etc.)

#### Statistiques I/O par Backend

```sql
-- Nouveau dans PG 18 : Stats I/O par connexion active
SELECT
    pid,
    usename,
    application_name,
    backend_type,
    query,
    backend_io_reads,
    backend_io_writes,
    backend_io_time
FROM pg_stat_activity  
WHERE state = 'active'  
ORDER BY backend_io_time DESC;  
```

### Métriques Clés à Surveiller

#### 1. Ratio Async vs Sync Ops

```sql
-- Vérifier que les I/O utilisent bien le mode async
SELECT
    io_method,
    COUNT(*) as operations
FROM pg_stat_io  
GROUP BY io_method;  
```

**Objectif** : Voir majoritairement des opérations en mode `async`.

#### 2. Temps d'Attente I/O

```sql
-- Identifier les backends avec le plus d'attente I/O
SELECT
    pid,
    usename,
    wait_event_type,
    wait_event,
    state,
    query_start,
    now() - query_start AS duration
FROM pg_stat_activity  
WHERE wait_event_type = 'IO'  
ORDER BY duration DESC;  
```

#### 3. Comparaison Avant/Après

```sql
-- Créer une baseline AVANT l'activation de async
CREATE TABLE io_baseline AS  
SELECT  
    now() as snapshot_time,
    datname,
    blks_read,
    blks_hit,
    blk_read_time,
    blk_write_time
FROM pg_stat_database;

-- APRÈS activation, comparer
SELECT
    b.datname,
    (a.blk_read_time - b.blk_read_time) / NULLIF(a.blks_read - b.blks_read, 0) AS avg_read_time_ms,
    ((b.blk_read_time - a.blk_read_time) / b.blk_read_time * 100) AS improvement_pct
FROM pg_stat_database a  
JOIN io_baseline b ON a.datname = b.datname  
WHERE a.blks_read > b.blks_read;  
```

## Troubleshooting et Problèmes Courants

### Problème 1 : io_method reste sur 'sync'

**Symptôme** :
```sql
SHOW io_method;
-- Résultat : 'sync' alors que vous attendiez 'async'
```

**Causes Possibles** :

1. **Système d'exploitation trop ancien**
```bash
# Vérifier la version du kernel (Linux)
uname -r
# Besoin : >= 5.10 pour io_uring optimal
```

**Solution** : Upgrader le système d'exploitation ou accepter le mode sync.

2. **io_uring désactivé au niveau kernel**
```bash
# Vérifier si io_uring est disponible
cat /proc/sys/kernel/io_uring_disabled
# 0 = activé, 1 = désactivé
```

**Solution** : Activer io_uring
```bash
sudo sysctl -w kernel.io_uring_disabled=0
```

3. **PostgreSQL compilé sans support async**
```bash
# Vérifier les capacités de votre build PostgreSQL
pg_config --configure
# Chercher : --with-io_uring ou similaire
```

**Solution** : Utiliser un build officiel de PostgreSQL 18 ou recompiler.

### Problème 2 : Performances Dégradées Après Activation

**Symptôme** : Les performances sont **pires** avec `io_method = 'worker'`.

**Causes Possibles** :

1. **Base de données entièrement en cache**
```sql
-- Vérifier le cache hit ratio
SELECT
    sum(heap_blks_hit) / (sum(heap_blks_hit) + sum(heap_blks_read)) AS cache_ratio
FROM pg_statio_user_tables;
-- Si > 0.995 (99.5%), peu d'I/O réels → async inutile
```

**Solution** : Le mode async n'apporte rien si pas d'I/O. Rester en sync est acceptable.

2. **Concurrence I/O trop élevée**
```conf
# Si vous avez mis :
effective_io_concurrency = 200  # TROP ÉLEVÉ !
```

**Solution** : Réduire à des valeurs raisonnables (voir recommandations plus haut).

3. **Système de fichiers non optimisé**
```bash
# Vérifier le système de fichiers
df -T /var/lib/postgresql
# XFS et ext4 sont optimaux pour io_uring
```

**Solution** : Utiliser XFS ou ext4 (éviter ext3, FAT32, etc.).

### Problème 3 : Logs d'Erreurs io_uring

**Symptôme** : Logs PostgreSQL montrant des erreurs io_uring.

```
ERROR: io_uring submission failed: Resource temporarily unavailable
```

**Cause** : Limites kernel trop basses.

**Solution** :
```bash
# Augmenter les limites io_uring
sudo sysctl -w kernel.io_uring_ops_limit=65536  
sudo sysctl -w kernel.io_uring_queues_limit=1024  

# Rendre permanent
echo "kernel.io_uring_ops_limit = 65536" | sudo tee -a /etc/sysctl.conf  
echo "kernel.io_uring_queues_limit = 1024" | sudo tee -a /etc/sysctl.conf  
```

**Après Migration vers PG 18 (async)** :
- Latence P50 : 18 ms (**28% amélioration**)
- Latence P99 : 120 ms (**52% amélioration**)
- Satisfaction client améliorée

**Impact Business** : Réduction de 40% du churn client lié à la performance.

## Bonnes Pratiques et Recommandations

### ✅ À Faire

1. **Utiliser async par défaut sur systèmes modernes**
   - PostgreSQL 18 le détecte automatiquement
   - Profitez-en sans configuration supplémentaire

2. **Monitorer les statistiques I/O**
   ```sql
   -- Ajouter à votre dashboard de monitoring
   SELECT * FROM pg_stat_io;
   ```

3. **Ajuster effective_io_concurrency selon votre stockage**
   - SSD NVMe : 16-32
   - SSD SATA : 4-8
   - HDD : 1-2

4. **Tester en pré-production d'abord**
   - Benchmarks avant/après
   - Vérifier les métriques applicatives

5. **Combiner avec d'autres optimisations**
   ```conf
   # Synergie avec autres paramètres
   io_method = 'worker'
   shared_buffers = 16GB
   effective_cache_size = 48GB
   random_page_cost = 1.1  # SSD
   ```

### ❌ À Éviter

1. **Ne pas forcer 'sync' sur systèmes modernes**
   ```conf
   # MAUVAIS (perte de performance)
   io_method = 'sync'  # Sauf raison spécifique
   ```

2. **Ne pas sur-dimensionner les paramètres de concurrence**
   ```conf
   # MAUVAIS
   effective_io_concurrency = 500  # Trop élevé !
   ```

3. **Ne pas ignorer les logs d'erreurs io_uring**
   - Investiguer immédiatement
   - Vérifier les limites kernel

4. **Ne pas migrer en pleine production sans tests**
   - Toujours tester en dev/staging d'abord

5. **Ne pas s'attendre à des miracles si tout est en cache**
   - Cache hit > 99% → Async inutile
   - Concentrez-vous sur d'autres optimisations

## Questions Fréquentes (FAQ)

### Q1 : Mon PostgreSQL 18 utilise-t-il vraiment le mode async ?

**Réponse** :
```sql
-- Vérifier le mode actuel
SHOW io_method;

-- Vérifier les stats d'utilisation
SELECT backend_type, io_object, io_context, SUM(reads) as total_reads  
FROM pg_stat_io  
WHERE io_method = 'worker'  
GROUP BY backend_type, io_object, io_context;  
```

### Q2 : Puis-je activer async sur PostgreSQL 17 ou antérieur ?

**Réponse** : Non, c'est une fonctionnalité exclusive à PostgreSQL 18+. Les versions antérieures n'ont pas le sous-système AIO.

### Q3 : Le mode async fonctionne-t-il avec la réplication ?

**Réponse** : Oui ! Le mode async améliore également :
- Le WAL shipping (écriture plus rapide)
- La réplication physique (apply plus rapide sur standby)
- La réplication logique (moins de lag)

### Q4 : Y a-t-il des risques de perte de données ?

**Réponse** : Non. Le mode async ne compromet **aucunement** la durabilité ACID. Les écritures utilisent toujours fsync/fdatasync pour garantir la persistance.

### Q5 : Quelle est la consommation mémoire supplémentaire ?

**Réponse** : Minime. Le sous-système AIO utilise des buffers additionnels :
- Environ 1-2 MB par backend actif
- Négligeable comparé aux gains de performance

### Q6 : Dois-je modifier mon code applicatif ?

**Réponse** : **Non**. Le mode async est totalement transparent :
- Aucun changement SQL nécessaire
- Aucune modification de drivers
- Aucune modification de code applicatif

### Q7 : Le mode async aide-t-il si j'ai un RAID ?

**Réponse** : **Oui, énormément** ! Les RAID (surtout RAID 0, 10, 50) bénéficient particulièrement du parallélisme I/O :
```conf
# Configuration optimale pour RAID SSD
effective_io_concurrency = 64  
maintenance_io_concurrency = 32  
```

### Q8 : Puis-je basculer entre sync et async à chaud ?

**Réponse** : Oui, **sans redémarrage** :
```sql
ALTER SYSTEM SET io_method = 'worker';  
SELECT pg_reload_conf();  
```

Cependant, il est recommandé de planifier ce changement lors d'une fenêtre de maintenance pour observer l'impact.

### 🎯 L'Essentiel à Retenir

1. **PostgreSQL 18 introduit l'I/O asynchrone** → Jusqu'à **3× plus rapide** sur workloads I/O intensifs

2. **Le paramètre clé : `io_method = 'worker'`**
   - Détection automatique sur systèmes modernes


   - Monitorer avec `pg_stat_io`
