# 18.7.2. pg_partman : Gestion Automatisée de Partitions dans PostgreSQL

## Introduction

**pg_partman** (PostgreSQL Partition Manager) est une extension PostgreSQL qui automatise et simplifie la gestion des tables partitionnées. Elle prend en charge la création, la maintenance et la suppression automatique de partitions selon des règles définies.

### Qu'est-ce que le partitionnement de tables ?

Avant de parler de pg_partman, il est essentiel de comprendre le concept de **partitionnement**.

Le partitionnement consiste à diviser une grande table en plusieurs sous-tables plus petites appelées **partitions**, tout en conservant une interface unique pour les requêtes.

**Analogie** : Imaginez une bibliothèque avec des millions de livres. Au lieu de tout mettre sur une seule étagère géante (impossible à gérer), vous organisez les livres par catégorie, par année, par auteur, etc. Chaque section devient plus facile à parcourir.

**Exemple concret** :

```
Table "commandes" (100 millions de lignes)
         ↓
Partitionnement par année
         ↓
┌─────────────────────────────────────────┐
│ commandes_2023 (12 millions de lignes)  │
│ commandes_2024 (35 millions de lignes)  │
│ commandes_2025 (53 millions de lignes)  │
└─────────────────────────────────────────┘
```

### Pourquoi partitionner ?

Les avantages du partitionnement :

- ✅ **Performance** : Les requêtes ne scannent que les partitions nécessaires (partition pruning)  
- ✅ **Maintenance** : VACUUM, REINDEX plus rapides sur de petites partitions  
- ✅ **Archivage** : Détacher/supprimer des anciennes partitions facilement  
- ✅ **Gestion de l'espace** : Déplacer des partitions sur différents tablespaces  
- ✅ **Sauvegarde** : Sauvegarder uniquement les partitions actives

**Cas d'usage typiques** :
- Tables de logs avec des millions d'entrées quotidiennes
- Données historiques (commandes, transactions, événements)
- Tables de séries temporelles (IoT, monitoring, métriques)
- Données multi-tenants (un client par partition)

### Le problème sans pg_partman

PostgreSQL supporte nativement le partitionnement déclaratif depuis la version 10, mais la gestion manuelle est **fastidieuse** :

- ❌ Créer manuellement chaque nouvelle partition (chaque mois, chaque jour...)  
- ❌ Surveiller quand créer les prochaines partitions  
- ❌ Détacher et archiver les anciennes partitions  
- ❌ Maintenir les index sur chaque partition  
- ❌ Gérer les contraintes et permissions

**Sans automatisation**, vous devriez écrire et exécuter régulièrement des scripts comme :

```sql
-- Tous les mois, manuellement :
CREATE TABLE commandes_2025_12 PARTITION OF commandes
    FOR VALUES FROM ('2025-12-01') TO ('2026-01-01');

CREATE INDEX idx_commandes_2025_12_date ON commandes_2025_12(date_commande);
-- ... et ainsi de suite pour tous les index

-- Détacher les anciennes partitions
ALTER TABLE commandes DETACH PARTITION commandes_2023_01;
```

### La solution : pg_partman

**pg_partman** automatise tout ce processus :

- ✅ **Création automatique** de nouvelles partitions à l'avance  
- ✅ **Détachement automatique** des anciennes partitions  
- ✅ **Gestion des index** automatique  
- ✅ **Intégration avec pg_cron** pour une automatisation complète  
- ✅ **Maintenance simplifiée** avec des fonctions dédiées  
- ✅ **Support de tous les types** de partitionnement (Range, List, Hash)

## Architecture et Concepts Fondamentaux

### Comment pg_partman fonctionne-t-il ?

pg_partman repose sur trois piliers :

1. **Métadonnées** : Tables de configuration qui décrivent les règles de partitionnement  
2. **Fonctions de maintenance** : Scripts qui créent/détachent les partitions  
3. **Automatisation** : Intégration avec pg_cron pour exécution régulière

```
┌───────────────────────────────────────────────────┐
│           Table Principale (Parent)               │
│              commandes                            │
└───────────────────────────────────────────────────┘
                       │
        ┌──────────────┼──────────────┐
        ↓              ↓              ↓
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ Partition 1  │ │ Partition 2  │ │ Partition 3  │
│ 2024-01      │ │ 2024-02      │ │ 2024-03      │
└──────────────┘ └──────────────┘ └──────────────┘
        ↑              ↑              ↑
        │              │              │
┌───────┴──────────────┴──────────────┴────────────┐
│          pg_partman                              │
│  - Crée automatiquement les partitions futures   │
│  - Détache les anciennes partitions              │
│  - Maintient les index et contraintes            │
└──────────────────────────────────────────────────┘
        ↑
        │ (planification)
┌───────┴────────┐
│    pg_cron     │
└────────────────┘
```

### Types de partitionnement supportés

pg_partman supporte les trois types de partitionnement natifs de PostgreSQL :

#### 1. Range (Plage) - Le plus courant

Partitionnement basé sur des plages de valeurs, typiquement pour des dates.

**Exemple** : Partitionner par mois

```
commandes_2024_01: du 2024-01-01 au 2024-02-01  
commandes_2024_02: du 2024-02-01 au 2024-03-01  
commandes_2024_03: du 2024-03-01 au 2024-04-01  
```

**Cas d'usage** : Tables temporelles (logs, événements, transactions)

#### 2. List (Liste)

Partitionnement basé sur des valeurs discrètes.

**Exemple** : Partitionner par région

```
commandes_europe: région IN ('FR', 'DE', 'IT', 'ES')  
commandes_amerique: région IN ('US', 'CA', 'MX')  
commandes_asie: région IN ('JP', 'CN', 'KR')  
```

**Cas d'usage** : Données multi-tenants, segmentation par catégorie

#### 3. Hash

Partitionnement basé sur un hash, pour distribuer uniformément les données.

**Exemple** : Partitionner par hash de l'ID utilisateur

```
commandes_hash_0: MODULUS 4, REMAINDER 0  
commandes_hash_1: MODULUS 4, REMAINDER 1  
commandes_hash_2: MODULUS 4, REMAINDER 2  
commandes_hash_3: MODULUS 4, REMAINDER 3  
```

**Cas d'usage** : Distribution uniforme pour parallélisation, sharding logique

### Terminologie pg_partman

Quelques termes clés à connaître :

| Terme | Signification |
|-------|---------------|
| **Parent table** | Table principale partitionnée (ex: `commandes`) |
| **Child partition** | Partition individuelle (ex: `commandes_2024_01`) |
| **Premake** | Nombre de partitions à créer à l'avance |
| **Retention** | Durée de conservation avant détachement |
| **Partition interval** | Intervalle de chaque partition (ex: 1 mois, 1 jour) |
| **Template table** | Modèle pour créer de nouvelles partitions |
| **Constraint validation** | Validation des contraintes après détachement |

## Installation et Configuration

### Prérequis

- PostgreSQL 11 ou supérieur (idéalement PostgreSQL 18)
- Droits superutilisateur ou CREATE sur le schéma
- Extension pg_cron (fortement recommandée pour l'automatisation)

### Processus d'installation (théorique)

#### Étape 1 : Installer l'extension

pg_partman doit être disponible dans votre distribution PostgreSQL (package système ou compilation).

```sql
-- Créer l'extension dans la base de données cible
CREATE SCHEMA partman;  
CREATE EXTENSION pg_partman SCHEMA partman;  
```

**Note** : Il est recommandé de créer pg_partman dans son propre schéma pour séparer les objets.

#### Étape 2 : Vérifier l'installation

```sql
-- Lister les fonctions pg_partman
SELECT proname  
FROM pg_proc  
WHERE pronamespace = 'partman'::regnamespace  
ORDER BY proname;  

-- Vérifier les tables de métadonnées
SELECT tablename  
FROM pg_tables  
WHERE schemaname = 'partman'  
ORDER BY tablename;  
```

Les principales tables de métadonnées :
- `partman.part_config` : Configuration des tables partitionnées  
- `partman.part_config_sub` : Configuration pour le partitionnement en cascade (sub-partitions)

#### Étape 3 : Configurer les permissions

```sql
-- Accorder les permissions nécessaires
GRANT USAGE ON SCHEMA partman TO votre_utilisateur;  
GRANT ALL ON ALL TABLES IN SCHEMA partman TO votre_utilisateur;  
GRANT EXECUTE ON ALL FUNCTIONS IN SCHEMA partman TO votre_utilisateur;  
```

## Créer une Table Partitionnée avec pg_partman

### Méthode 1 : Partitionner une nouvelle table (Range/Time)

C'est le cas d'usage le plus courant : partitionner par date.

#### Étape 1 : Créer la table parent

```sql
CREATE TABLE logs (
    id BIGSERIAL,
    timestamp TIMESTAMPTZ NOT NULL,
    user_id INTEGER,
    action VARCHAR(100),
    details JSONB
) PARTITION BY RANGE (timestamp);
```

**Important** :
- La colonne de partitionnement (`timestamp`) doit être dans la clé primaire si vous en définissez une
- Utilisez `PARTITION BY RANGE (colonne)` pour le partitionnement temporel

#### Étape 2 : Créer la table template (optionnel mais recommandé)

La table template définit les index et contraintes qui seront automatiquement appliqués à chaque nouvelle partition.

```sql
CREATE TABLE logs_template (LIKE logs INCLUDING ALL);

-- Ajouter les index désirés
CREATE INDEX idx_logs_user_id ON logs_template(user_id);  
CREATE INDEX idx_logs_action ON logs_template(action);  
```

#### Étape 3 : Configurer pg_partman

```sql
SELECT partman.create_parent(
    p_parent_table := 'public.logs',
    p_control := 'timestamp',
    p_interval := 'daily',  -- 'daily', 'weekly', 'monthly', 'quarterly', 'yearly'
    p_premake := 4,         -- Créer 4 partitions à l'avance
    p_start_partition := '2025-01-01',  -- Première partition
    p_template_table := 'public.logs_template'
);
```

**Paramètres expliqués** :

- `p_parent_table` : Nom complet de la table parent (schéma.table)  
- `p_control` : Colonne de partitionnement  
- `p_interval` : Intervalle de chaque partition  
  - `'daily'` : Une partition par jour  
  - `'weekly'` : Une partition par semaine  
  - `'monthly'` : Une partition par mois  
  - `'quarterly'` : Une partition par trimestre  
  - `'yearly'` : Une partition par an  
- `p_premake` : Nombre de partitions futures à créer d'avance (recommandé: 4-10)  
- `p_start_partition` : Date de début de la première partition  
- `p_template_table` : Table modèle pour les index et contraintes

#### Étape 4 : Vérifier la création

```sql
-- Voir la configuration
SELECT * FROM partman.part_config WHERE parent_table = 'public.logs';

-- Lister les partitions créées
SELECT
    schemaname,
    tablename
FROM pg_tables  
WHERE tablename LIKE 'logs_%'  
ORDER BY tablename;  
```

### Méthode 2 : Partitionner une table existante avec données

Si vous avez déjà une table non partitionnée avec des données, pg_partman peut vous aider.

**Attention** : Cette opération nécessite une maintenance et peut être longue selon le volume de données.

#### Étape 1 : Créer la nouvelle table partitionnée

```sql
CREATE TABLE commandes_partitioned (
    LIKE commandes INCLUDING ALL
) PARTITION BY RANGE (date_commande);
```

#### Étape 2 : Configurer pg_partman

```sql
SELECT partman.create_parent(
    p_parent_table := 'public.commandes_partitioned',
    p_control := 'date_commande',
    p_interval := 'monthly',
    p_premake := 4,
    p_start_partition := '2024-01-01'
);
```

#### Étape 3 : Migrer les données (attention : opération lourde)

```sql
-- Méthode 1 : INSERT SELECT (requiert une fenêtre de maintenance)
INSERT INTO commandes_partitioned  
SELECT * FROM commandes;  

-- Méthode 2 : Batch par batch pour limiter le verrou
-- (à scripter selon vos besoins)
```

#### Étape 4 : Renommer les tables

```sql
BEGIN;
    ALTER TABLE commandes RENAME TO commandes_old;
    ALTER TABLE commandes_partitioned RENAME TO commandes;
    -- Recréer les index, contraintes, etc.
COMMIT;
```

**Note** : Pour de très grandes tables, considérez des stratégies de migration progressive avec pg_partman et réplication logique.

### Méthode 3 : Partitionnement par ID (Range numérique)

Pour des tables où le partitionnement temporel n'est pas pertinent.

```sql
CREATE TABLE utilisateurs (
    id BIGSERIAL,
    nom VARCHAR(100),
    email VARCHAR(255),
    created_at TIMESTAMPTZ
) PARTITION BY RANGE (id);

SELECT partman.create_parent(
    p_parent_table := 'public.utilisateurs',
    p_control := 'id',
    p_type := 'native',
    p_interval := '1000000',  -- Une partition tous les 1 million d'IDs
    p_premake := 4
);
```

**Cas d'usage** : Tables avec auto-incrémentation, sans dimension temporelle forte.

### Méthode 4 : Partitionnement en cascade (Sub-partitioning)

Pour des cas complexes : partitionner par année, puis par mois.

```sql
-- Table parent (partitionnée par année)
CREATE TABLE metriques (
    timestamp TIMESTAMPTZ NOT NULL,
    sensor_id INTEGER,
    valeur NUMERIC
) PARTITION BY RANGE (timestamp);

-- Créer le parent avec pg_partman
SELECT partman.create_parent(
    p_parent_table := 'public.metriques',
    p_control := 'timestamp',
    p_interval := 'yearly',
    p_premake := 2
);

-- Ensuite, sub-partitionner chaque partition annuelle par mois
-- (Nécessite des appels à create_sub_parent pour chaque partition)
```

**Note** : Le sub-partitioning est avancé et moins courant. Utilisez-le seulement si nécessaire.

## Maintenance Automatisée

### La fonction clé : run_maintenance()

`partman.run_maintenance()` est la fonction centrale qui :

1. **Crée** de nouvelles partitions selon le paramètre `premake`  
2. **Détache** les anciennes partitions selon le paramètre `retention`  
3. **Analyse** les partitions (ANALYZE)

**Syntaxe** :

```sql
SELECT partman.run_maintenance(
    p_parent_table := NULL,  -- NULL = toutes les tables configurées
    p_analyze := TRUE,       -- Exécuter ANALYZE sur les nouvelles partitions
    p_jobmon := TRUE         -- Logger dans partman.part_config
);
```

### Automatisation avec pg_cron

pg_partman est conçu pour être exécuté régulièrement via pg_cron.

#### Configuration recommandée

```sql
-- Exécuter la maintenance toutes les heures
SELECT cron.schedule(
    'pg_partman_maintenance',
    '0 * * * *',  -- Toutes les heures à H:00
    'SELECT partman.run_maintenance();'
);
```

**Fréquence recommandée** :
- **Partitionnement daily** : Exécuter 1-2 fois par jour  
- **Partitionnement monthly** : Exécuter quotidiennement ou hebdomadairement  
- **Partitionnement hourly** (rare) : Exécuter toutes les 15-30 minutes

#### Vérifier les exécutions

```sql
-- Voir les logs de maintenance (si jobmon activé)
SELECT * FROM partman.part_config  
WHERE last_partition IS NOT NULL  
ORDER BY last_partition DESC;  

-- Combiner avec l'historique pg_cron
SELECT
    jobid,
    runid,
    status,
    return_message,
    start_time,
    end_time
FROM cron.job_run_details  
WHERE command LIKE '%run_maintenance%'  
ORDER BY start_time DESC  
LIMIT 10;  
```

### Maintenance manuelle

Pour forcer la création de partitions ou tester la configuration :

```sql
-- Exécuter la maintenance pour une table spécifique
SELECT partman.run_maintenance(
    p_parent_table := 'public.logs'
);

-- Vérifier les partitions créées
SELECT
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) as size
FROM pg_tables  
WHERE tablename LIKE 'logs_%'  
ORDER BY tablename;  
```

## Configuration Avancée

### Paramètres de retention (conservation)

Pour détacher automatiquement les anciennes partitions après un certain délai :

```sql
UPDATE partman.part_config  
SET  
    retention = '90 days',           -- Détacher après 90 jours
    retention_keep_table = TRUE,     -- Garder la table détachée
    retention_keep_index = FALSE     -- Supprimer les index après détachement
WHERE parent_table = 'public.logs';
```

**Options de retention** :

| Paramètre | Signification |
|-----------|--------------|
| `retention` | Durée avant détachement (ex: '90 days', '6 months', '2 years') |
| `retention_keep_table` | `TRUE` = garder la table détachée, `FALSE` = supprimer |
| `retention_keep_index` | `TRUE` = garder les index, `FALSE` = supprimer (économiser de l'espace) |
| `retention_schema` | Schéma où déplacer les partitions détachées (si `retention_keep_table = TRUE`) |

**Exemple de workflow d'archivage** :

```sql
-- Configuration pour archivage
UPDATE partman.part_config  
SET  
    retention = '1 year',
    retention_keep_table = TRUE,
    retention_keep_index = FALSE,
    retention_schema = 'archive'
WHERE parent_table = 'public.logs';

-- Créer le schéma d'archive si nécessaire
CREATE SCHEMA IF NOT EXISTS archive;
```

Après un an, les partitions seront automatiquement :
1. Détachées de la table parent  
2. Déplacées dans le schéma `archive`  
3. Leurs index seront supprimés (économie d'espace)

### Paramètres de premake (anticipation)

Contrôler combien de partitions futures créer :

```sql
UPDATE partman.part_config  
SET premake = 10  -- Créer 10 partitions à l'avance  
WHERE parent_table = 'public.logs';  
```

**Recommandations** :
- **Partitionnement daily** : `premake = 7-14` (1-2 semaines à l'avance)  
- **Partitionnement monthly** : `premake = 3-6` (3-6 mois à l'avance)  
- **Partitionnement yearly** : `premake = 2-3` (2-3 ans à l'avance)

**Pourquoi prévoir à l'avance ?**
- Éviter les erreurs d'insertion si une partition manque
- Garantir la continuité du service
- Limiter l'impact de pannes de maintenance

### Gestion des contraintes

pg_partman peut gérer automatiquement les contraintes CHECK sur les partitions :

```sql
UPDATE partman.part_config  
SET  
    constraint_cols = ARRAY['user_id', 'status'],  -- Colonnes pour contraintes CHECK
    optimize_constraint = 10  -- Optimiser après 10 lignes
WHERE parent_table = 'public.logs';
```

**Note** : Avec le partitionnement natif moderne, les contraintes CHECK ne sont généralement pas nécessaires grâce au **partition pruning** automatique.

### Infinite time partitions

Pour créer automatiquement des partitions indéfiniment dans le futur :

```sql
UPDATE partman.part_config  
SET infinite_time_partitions = TRUE  
WHERE parent_table = 'public.logs';  
```

**Effet** : pg_partman créera toujours `premake` partitions dans le futur, sans limite temporelle.

### Gestion des timezones

Spécifier un fuseau horaire pour les partitions temporelles :

```sql
UPDATE partman.part_config  
SET datetime_string = 'YYYY-MM-DD HH24:MI:SS'  
WHERE parent_table = 'public.logs';  

-- Ou définir un timezone spécifique
UPDATE partman.part_config  
SET datetime_string = 'YYYY-MM-DD HH24:MI:SS TZ'  
WHERE parent_table = 'public.logs';  
```

## Surveillance et Monitoring

### Vérifier la configuration active

```sql
-- Vue d'ensemble de toutes les tables partitionnées
SELECT
    parent_table,
    partition_interval,
    premake,
    retention,
    datetime_string,
    last_partition,
    constraint_valid
FROM partman.part_config  
ORDER BY parent_table;  
```

### Identifier les partitions existantes

```sql
-- Fonction utilitaire de pg_partman
SELECT partman.show_partitions(
    'public.logs',     -- Table parent
    'DESC',            -- Ordre (ASC ou DESC)
    FALSE              -- Inclure les tables template ?
);
```

### Vérifier la taille des partitions

```sql
SELECT
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) as total_size,
    pg_size_pretty(pg_relation_size(schemaname||'.'||tablename)) as table_size,
    pg_size_pretty(pg_indexes_size(schemaname||'.'||tablename)) as indexes_size
FROM pg_tables  
WHERE tablename LIKE 'logs_%'  
ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC  
LIMIT 20;  
```

### Vérifier la distribution des données

```sql
-- Nombre de lignes par partition
SELECT
    schemaname,
    tablename,
    n_live_tup as row_count
FROM pg_stat_user_tables  
WHERE tablename LIKE 'logs_%'  
ORDER BY n_live_tup DESC;  
```

### Alerting sur les problèmes potentiels

#### Détection de partitions manquantes dans le futur

```sql
-- Créer une fonction de surveillance
CREATE OR REPLACE FUNCTION partman.check_premake_warning()  
RETURNS TABLE(parent_table TEXT, warning TEXT) AS $$  
BEGIN  
    RETURN QUERY
    SELECT
        pc.parent_table::TEXT,
        'Seulement ' || pc.premake || ' partitions futures' AS warning
    FROM partman.part_config pc
    WHERE pc.premake < 3;
END;
$$ LANGUAGE plpgsql;

-- Planifier une vérification quotidienne avec pg_cron
SELECT cron.schedule(
    'check_partman_premake',
    '0 8 * * *',  -- Tous les jours à 8h
    $$
    SELECT send_alert_if_needed(
        'Partman Warning',
        (SELECT string_agg(parent_table || ': ' || warning, E'\n')
         FROM partman.check_premake_warning())
    )
    WHERE EXISTS (SELECT 1 FROM partman.check_premake_warning())
    $$
);
```

#### Détection de partitions anciennes non détachées

```sql
CREATE OR REPLACE FUNCTION partman.check_old_partitions()  
RETURNS TABLE(partition_name TEXT, age_days INTEGER) AS $$  
BEGIN  
    RETURN QUERY
    SELECT
        schemaname || '.' || tablename AS partition_name,
        EXTRACT(DAY FROM NOW() - last_autovacuum)::INTEGER AS age_days
    FROM pg_stat_user_tables
    WHERE tablename LIKE '%_p%'
      AND last_autovacuum < NOW() - INTERVAL '180 days'
    ORDER BY last_autovacuum;
END;
$$ LANGUAGE plpgsql;
```

### Logs et historique

pg_partman ne possède pas de système de logs intégré sophistiqué, mais vous pouvez :

1. **Utiliser les logs PostgreSQL** : Activez `log_statement = 'ddl'` pour tracer les CREATE/DROP TABLE  
2. **Utiliser pg_stat_user_tables** : Suivre les statistiques d'accès aux partitions  
3. **Créer une table d'audit personnalisée** :

```sql
CREATE TABLE partman.maintenance_log (
    log_id SERIAL PRIMARY KEY,
    execution_time TIMESTAMPTZ DEFAULT NOW(),
    parent_table TEXT,
    action TEXT,  -- 'create', 'detach', 'drop'
    partition_name TEXT,
    details JSONB
);

-- Fonction wrapper pour logger
CREATE OR REPLACE FUNCTION partman.run_maintenance_with_log()  
RETURNS void AS $$  
BEGIN  
    PERFORM partman.run_maintenance();

    INSERT INTO partman.maintenance_log (parent_table, action, details)
    VALUES ('all', 'maintenance_run', jsonb_build_object('timestamp', NOW()));
END;
$$ LANGUAGE plpgsql;
```

## Opérations Avancées

### Détacher manuellement une partition

Pour détacher une partition avant le délai de retention :

```sql
-- Détacher la partition
ALTER TABLE logs DETACH PARTITION logs_p2023_01_01;

-- La partition existe toujours comme table indépendante
-- Vous pouvez l'archiver, la déplacer vers un tablespace différent, etc.

-- Pour la supprimer complètement
DROP TABLE logs_p2023_01_01;
```

### Rattacher une partition

Si vous avez détaché une partition et souhaitez la réintégrer :

```sql
-- Vérifier que les données respectent les contraintes
ALTER TABLE logs ATTACH PARTITION logs_p2023_01_01
    FOR VALUES FROM ('2023-01-01') TO ('2023-02-01');
```

**Important** : PostgreSQL vérifiera que toutes les lignes de la partition détachée respectent bien les bornes définies.

### Répartitionner une table (Repartitioning)

Si vous souhaitez changer l'intervalle de partitionnement (ex: passer de mensuel à hebdomadaire) :

**Méthode 1** : Créer une nouvelle table et migrer

```sql
-- 1. Créer la nouvelle table partitionnée
CREATE TABLE logs_new (LIKE logs INCLUDING ALL)  
PARTITION BY RANGE (timestamp);  

-- 2. Configurer pg_partman avec le nouvel intervalle
SELECT partman.create_parent(
    p_parent_table := 'public.logs_new',
    p_control := 'timestamp',
    p_interval := 'weekly',  -- Nouvel intervalle
    p_premake := 8
);

-- 3. Migrer les données (par batch)
-- 4. Renommer les tables
```

**Méthode 2** : Utiliser des partitions temporaires et consolider

(Plus complexe, à scripter selon vos besoins spécifiques)

### Déplacer des partitions vers un autre tablespace

Pour optimiser le stockage (ex: anciennes partitions sur disque lent) :

```sql
-- Créer un tablespace pour archives
CREATE TABLESPACE archive_ts LOCATION '/mnt/slow_disk/postgres/archive';

-- Déplacer une partition
ALTER TABLE logs_p2023_01_01 SET TABLESPACE archive_ts;

-- Automatiser via pg_partman (après détachement)
UPDATE partman.part_config  
SET retention_schema = 'archive'  
WHERE parent_table = 'public.logs';  
```

### Gestion des partitions par défaut

PostgreSQL permet une partition "default" pour capturer les données hors plages définies :

```sql
-- Créer une partition par défaut
CREATE TABLE logs_default PARTITION OF logs DEFAULT;

-- ATTENTION : pg_partman ne gère pas automatiquement les partitions default
-- Vous devrez surveiller manuellement cette partition
```

**Recommandation** : Utilisez `premake` suffisamment élevé pour éviter d'avoir besoin d'une partition default.

## Stratégies de Partitionnement

### Choisir le bon intervalle

| Intervalle | Volume typique | Cas d'usage |
|------------|----------------|-------------|
| **Hourly** | Millions de lignes/heure | IoT temps réel, logs haute fréquence |
| **Daily** | Centaines de milliers/jour | Logs applicatifs, événements |
| **Weekly** | Millions de lignes/semaine | Données agrégées, analytics |
| **Monthly** | Dizaines de millions/mois | Transactions, commandes, données historiques |
| **Quarterly** | Centaines de millions/trimestre | Archivage long terme |
| **Yearly** | Milliards de lignes/an | Données réglementaires, audit |

**Règle générale** : Une partition ne devrait pas dépasser 100-200 Go pour des performances optimales.

### Partitionnement par date vs par ID

| Critère | Par Date (timestamp) | Par ID (séquence) |
|---------|----------------------|-------------------|
| **Pattern d'accès** | Requêtes récentes fréquentes | Accès uniforme sur tout le dataset |
| **Archivage** | Facile (détacher l'ancien) | Complexe |
| **Performance** | Excellente pour requêtes temporelles | Bonne pour distribution uniforme |
| **Maintenance** | Automatique avec pg_partman | Moins automatisable |

**Conseil** : Privilégiez le partitionnement temporel (par date) sauf cas spécifiques.

### Partitionnement simple vs en cascade

| Approche | Avantages | Inconvénients |
|----------|-----------|---------------|
| **Simple** (1 niveau) | Simple à gérer, performant | Moins flexible |
| **Cascade** (2+ niveaux) | Très granulaire, flexible | Complexe, overhead |

**Exemple cascade** : Partitionner par année, puis chaque année par mois
- `logs_2024` → `logs_2024_01`, `logs_2024_02`, ...  
- `logs_2025` → `logs_2025_01`, `logs_2025_02`, ...

**Recommandation** : Utilisez le partitionnement simple sauf si vous avez des besoins très spécifiques.

## Cas d'Usage Détaillés

### 1. Table de Logs Applicatifs

**Contexte** : Application web avec 50 millions de logs par mois.

**Objectif** : Conserver 3 mois en ligne, archiver l'ancien.

```sql
-- Créer la table
CREATE TABLE app_logs (
    log_id BIGSERIAL,
    timestamp TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    level VARCHAR(20),
    message TEXT,
    user_id INTEGER,
    request_id UUID
) PARTITION BY RANGE (timestamp);

-- Template avec index
CREATE TABLE app_logs_template (LIKE app_logs INCLUDING ALL);  
CREATE INDEX idx_app_logs_user ON app_logs_template(user_id);  
CREATE INDEX idx_app_logs_level ON app_logs_template(level) WHERE level IN ('ERROR', 'CRITICAL');  

-- Configurer pg_partman
SELECT partman.create_parent(
    p_parent_table := 'public.app_logs',
    p_control := 'timestamp',
    p_interval := 'daily',
    p_premake := 7,  -- 1 semaine à l'avance
    p_template_table := 'public.app_logs_template'
);

-- Définir la retention
UPDATE partman.part_config  
SET  
    retention = '90 days',
    retention_keep_table = TRUE,
    retention_keep_index = FALSE,
    retention_schema = 'archive'
WHERE parent_table = 'public.app_logs';

-- Automatiser avec pg_cron
SELECT cron.schedule(
    'maintain_app_logs',
    '0 2 * * *',  -- Tous les jours à 2h
    'SELECT partman.run_maintenance(''public.app_logs'');'
);
```

**Résultat** :
- Nouvelles partitions créées automatiquement chaque jour
- Logs de plus de 90 jours déplacés vers `archive.app_logs_pYYYY_MM_DD`
- Espace disque économisé (index supprimés sur archives)

### 2. Table de Transactions E-commerce

**Contexte** : Site e-commerce avec millions de transactions.

**Objectif** : Partitionnement mensuel, conservation 2 ans.

```sql
CREATE TABLE transactions (
    transaction_id BIGSERIAL,
    date_transaction TIMESTAMPTZ NOT NULL,
    user_id INTEGER NOT NULL,
    montant NUMERIC(10,2),
    statut VARCHAR(20),
    details JSONB
) PARTITION BY RANGE (date_transaction);

-- Template
CREATE TABLE transactions_template (LIKE transactions INCLUDING ALL);  
CREATE INDEX idx_trans_user ON transactions_template(user_id);  
CREATE INDEX idx_trans_status ON transactions_template(statut);  
CREATE INDEX idx_trans_montant ON transactions_template(montant) WHERE montant > 1000;  

-- Configuration pg_partman
SELECT partman.create_parent(
    p_parent_table := 'public.transactions',
    p_control := 'date_transaction',
    p_interval := 'monthly',
    p_premake := 6,  -- 6 mois à l'avance
    p_template_table := 'public.transactions_template'
);

-- Retention 2 ans
UPDATE partman.part_config  
SET  
    retention = '2 years',
    retention_keep_table = FALSE  -- Supprimer après 2 ans
WHERE parent_table = 'public.transactions';
```

### 3. Données IoT / Séries Temporelles

**Contexte** : Capteurs IoT générant 100 millions de mesures par jour.

**Objectif** : Partitionnement horaire, agrégation et archivage.

```sql
CREATE TABLE sensor_data (
    sensor_id INTEGER,
    timestamp TIMESTAMPTZ NOT NULL,
    temperature NUMERIC(5,2),
    humidity NUMERIC(5,2),
    pressure NUMERIC(7,2)
) PARTITION BY RANGE (timestamp);

-- Configuration pour partitionnement horaire
SELECT partman.create_parent(
    p_parent_table := 'public.sensor_data',
    p_control := 'timestamp',
    p_interval := 'hourly',
    p_premake := 48  -- 48h à l'avance
);

-- Retention courte (7 jours en brut)
UPDATE partman.part_config  
SET  
    retention = '7 days',
    retention_keep_table = FALSE
WHERE parent_table = 'public.sensor_data';

-- Créer une table agrégée (quotidienne) pour historique long terme
CREATE TABLE sensor_data_daily (
    sensor_id INTEGER,
    date DATE,
    avg_temperature NUMERIC(5,2),
    avg_humidity NUMERIC(5,2),
    avg_pressure NUMERIC(7,2),
    PRIMARY KEY (sensor_id, date)
);

-- Job d'agrégation quotidien (avec pg_cron)
SELECT cron.schedule(
    'aggregate_sensor_data',
    '0 1 * * *',  -- Tous les jours à 1h
    $$
    INSERT INTO sensor_data_daily (sensor_id, date, avg_temperature, avg_humidity, avg_pressure)
    SELECT
        sensor_id,
        DATE(timestamp) as date,
        AVG(temperature),
        AVG(humidity),
        AVG(pressure)
    FROM sensor_data
    WHERE timestamp >= CURRENT_DATE - 1
      AND timestamp < CURRENT_DATE
    GROUP BY sensor_id, DATE(timestamp)
    ON CONFLICT (sensor_id, date) DO UPDATE
    SET
        avg_temperature = EXCLUDED.avg_temperature,
        avg_humidity = EXCLUDED.avg_humidity,
        avg_pressure = EXCLUDED.avg_pressure;
    $$
);
```

### 4. Table Multi-tenant (Partitionnement par Liste)

**Contexte** : Application SaaS avec séparation stricte par client.

**Objectif** : Partitionnement par tenant_id.

```sql
CREATE TABLE tenant_data (
    id BIGSERIAL,
    tenant_id INTEGER NOT NULL,
    data JSONB,
    created_at TIMESTAMPTZ DEFAULT NOW()
) PARTITION BY LIST (tenant_id);

-- pg_partman supporte aussi le partitionnement par liste
SELECT partman.create_parent(
    p_parent_table := 'public.tenant_data',
    p_control := 'tenant_id',
    p_type := 'native',
    p_interval := 'list'  -- Mode liste
);

-- Création manuelle de partitions par tenant
CREATE TABLE tenant_data_p1 PARTITION OF tenant_data FOR VALUES IN (1);  
CREATE TABLE tenant_data_p2 PARTITION OF tenant_data FOR VALUES IN (2);  
-- ... etc

-- Note : La gestion automatique est moins avancée pour les partitions de type LIST
-- Il faudra souvent créer manuellement les nouvelles partitions
```

## Bonnes Pratiques

### 1. Planification initiale

✅ **Analysez votre workload** :
- Volume de données quotidien/mensuel
- Pattern de requêtes (récentes vs historiques)
- Exigences de retention

✅ **Choisissez le bon intervalle** :
- Trop petit = trop de partitions = overhead de gestion
- Trop grand = partitions volumineuses = mauvaises performances

✅ **Testez sur un environnement de dev** avant la production

### 2. Configuration de premake

✅ **Soyez généreux** : Mieux vaut créer trop de partitions que pas assez  
✅ **Adaptez selon l'intervalle** :
- Daily : 7-14 jours d'avance
- Monthly : 3-6 mois d'avance
- Yearly : 2-3 ans d'avance

❌ Ne jamais avoir `premake < 2` pour éviter les surprises

### 3. Gestion de la retention

✅ **Définissez une politique claire** :
- Combien de temps conserver en ligne ?
- Archiver ou supprimer après ?
- Besoins réglementaires (RGPD, SOX, etc.) ?

✅ **Utilisez un schéma d'archive** :
```sql
CREATE SCHEMA archive;  
UPDATE partman.part_config  
SET  
    retention_schema = 'archive',
    retention_keep_index = FALSE  -- Économiser l'espace
WHERE parent_table = 'public.logs';
```

✅ **Sauvegardez les archives** régulièrement (pg_dump par schéma)

### 4. Surveillance proactive

✅ **Créez des alertes** :
- Partitions manquantes dans le futur
- Échecs de run_maintenance()
- Partitions déséquilibrées (taille anormale)

✅ **Vérifiez régulièrement** `partman.part_config`

✅ **Auditez les performances** :
```sql
-- Identifier les slow queries sur partitions
SELECT * FROM pg_stat_statements  
WHERE query LIKE '%logs_%'  
ORDER BY total_exec_time DESC;  
```

### 5. Template tables

✅ **Utilisez toujours des templates** pour :
- Garantir la cohérence des index
- Appliquer les mêmes contraintes partout
- Simplifier la maintenance

❌ Ne modifiez jamais directement les index sur les partitions individuelles

### 6. Nommage cohérent

✅ **Suivez les conventions pg_partman** :
- Tables parent claires (`logs`, `transactions`)
- Partitions automatiquement nommées (`logs_p2025_01_01`)

✅ **Documentez votre schéma** :
```sql
COMMENT ON TABLE logs IS
'Table de logs applicatifs, partitionnée quotidiennement.
Retention: 90 jours en ligne, archives dans schema "archive".  
Maintenance: automatique via pg_cron (run_maintenance toutes les heures).';  
```

### 7. Tests avant migration

- ✅ **Testez la migration** sur un clone de production  
- ✅ **Mesurez les performances** avant/après  
- ✅ **Préparez un rollback plan**  
- ✅ **Prévoyez une fenêtre de maintenance** pour grandes tables

### 8. Coordination avec les backups

✅ **Excluez les archives** des backups réguliers si possible :
```bash
# pg_dump avec exclusion de schema
pg_dump -Fc -f backup.dump -N archive mydb
```

✅ **Sauvegardez les archives séparément** (moins fréquemment)

✅ **Testez les restaurations** de partitions individuelles

## Dépannage (Troubleshooting)

### Problème 1 : Partitions ne se créent pas automatiquement

**Symptômes** : `run_maintenance()` ne crée pas de nouvelles partitions.

**Diagnostics** :

```sql
-- Vérifier la configuration
SELECT * FROM partman.part_config  
WHERE parent_table = 'public.logs';  

-- Vérifier si premake est suffisant
SELECT
    parent_table,
    premake,
    last_partition
FROM partman.part_config;

-- Vérifier les jobs pg_cron
SELECT * FROM cron.job WHERE command LIKE '%partman%';
```

**Solutions** :
1. Augmenter `premake` si nécessaire  
2. Vérifier que pg_cron s'exécute bien  
3. Exécuter manuellement `run_maintenance()` pour tester

### Problème 2 : Erreur "no partition of relation for row"

**Symptômes** : Erreur lors d'insertion dans une table partitionnée.

**Cause** : Aucune partition ne correspond à la valeur insérée (souvent une date future).

**Solution** :

```sql
-- Créer manuellement la partition manquante
SELECT partman.create_partition_time(
    'public.logs',
    '2026-01-15',  -- Date de la partition nécessaire
    'daily'
);

-- Puis augmenter premake pour éviter le problème
UPDATE partman.part_config  
SET premake = 30  -- Augmenter significativement  
WHERE parent_table = 'public.logs';  

-- Relancer la maintenance
SELECT partman.run_maintenance('public.logs');
```

### Problème 3 : Performances dégradées après partitionnement

**Symptômes** : Requêtes plus lentes qu'avant le partitionnement.

**Diagnostics** :

```sql
-- Vérifier que partition pruning fonctionne
EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM logs  
WHERE timestamp >= '2025-01-01'  
  AND timestamp < '2025-02-01';

-- Doit afficher : "Partitions scanned: 1 of X"
```

**Causes possibles** :
1. **Index manquants** : Vérifiez que les index sont bien créés sur chaque partition  
2. **Partition pruning inactif** : Requêtes sans filtre sur la clé de partitionnement  
3. **Statistiques obsolètes** : Exécuter ANALYZE

**Solutions** :

```sql
-- Recréer les index via template
SELECT partman.reapply_privileges('public.logs');

-- Analyser toutes les partitions
SELECT partman.run_maintenance(
    p_parent_table := 'public.logs',
    p_analyze := TRUE
);

-- Forcer une analyse complète
ANALYZE logs;
```

### Problème 4 : Trop de partitions

**Symptômes** : Des milliers de partitions, lenteur dans les métadonnées (pg_class).

**Cause** : Intervalle trop petit ou retention trop longue.

**Solution** :

```sql
-- Réduire la retention
UPDATE partman.part_config  
SET retention = '30 days'  -- Au lieu de '365 days'  
WHERE parent_table = 'public.logs';  

-- Forcer le nettoyage
SELECT partman.run_maintenance('public.logs');

-- Ou détacher manuellement en masse (avec précaution)
DO $$  
DECLARE  
    part RECORD;
BEGIN
    FOR part IN
        SELECT tablename
        FROM pg_tables
        WHERE tablename LIKE 'logs_p2023%'
    LOOP
        EXECUTE format('ALTER TABLE logs DETACH PARTITION %I', part.tablename);
        EXECUTE format('DROP TABLE %I', part.tablename);
    END LOOP;
END $$;
```

### Problème 5 : Contraintes de clé étrangère

**Symptômes** : Erreur lors de la création de FK vers une table partitionnée.

**Cause** : Les FK sur tables partitionnées ont des limitations.

**Solution** :

```sql
-- Les FK doivent inclure la clé de partitionnement
CREATE TABLE orders (
    order_id BIGSERIAL,
    created_at TIMESTAMPTZ NOT NULL,
    user_id INTEGER
) PARTITION BY RANGE (created_at);

CREATE TABLE order_items (
    item_id BIGSERIAL,
    order_id BIGINT,
    created_at TIMESTAMPTZ NOT NULL,  -- Doit être présent !
    FOREIGN KEY (order_id, created_at)
        REFERENCES orders(order_id, created_at)
) PARTITION BY RANGE (created_at);
```

**Alternative** : Utiliser des triggers pour valider l'intégrité référentielle.

### Problème 6 : pg_partman ne détache pas les anciennes partitions

**Diagnostics** :

```sql
-- Vérifier la configuration de retention
SELECT
    parent_table,
    retention,
    retention_keep_table,
    retention_schema
FROM partman.part_config  
WHERE parent_table = 'public.logs';  

-- Vérifier si run_maintenance est appelé
SELECT * FROM cron.job_run_details  
WHERE command LIKE '%run_maintenance%'  
ORDER BY start_time DESC  
LIMIT 5;  
```

**Solutions** :
1. Vérifier que `retention` est défini (pas NULL)  
2. Vérifier que le schéma de destination existe (`retention_schema`)  
3. Exécuter manuellement pour tester :

```sql
SELECT partman.run_maintenance('public.logs');

-- Vérifier les logs PostgreSQL pour voir les erreurs
```

## Intégration avec l'Écosystème PostgreSQL

### pg_partman + pg_cron : L'automatisation complète

Configuration optimale pour automatisation totale :

```sql
-- Job de maintenance principal (toutes les heures)
SELECT cron.schedule(
    'pg_partman_maintenance',
    '0 * * * *',
    'SELECT partman.run_maintenance();'
);

-- Job de vérification quotidien
SELECT cron.schedule(
    'pg_partman_check',
    '0 9 * * *',
    $$
    SELECT send_alert(
        'Partman Status',
        string_agg(parent_table || ': ' || last_partition, E'\n')
    )
    FROM partman.part_config;
    $$
);

-- Job de nettoyage hebdomadaire des logs
SELECT cron.schedule(
    'pg_partman_cleanup_logs',
    '0 3 * * 0',
    'DELETE FROM partman.maintenance_log WHERE execution_time < NOW() - INTERVAL ''90 days'''
);
```

### pg_partman + Patroni/Replication

Dans une configuration haute disponibilité :

**Important** : pg_partman doit s'exécuter **uniquement sur le serveur primaire**.

```sql
-- Créer un wrapper qui vérifie le rôle
CREATE OR REPLACE FUNCTION partman.run_maintenance_if_primary()  
RETURNS void AS $$  
BEGIN  
    -- Vérifier si c'est le primaire
    IF pg_is_in_recovery() = FALSE THEN
        PERFORM partman.run_maintenance();
    ELSE
        RAISE NOTICE 'Serveur en standby, skip maintenance';
    END IF;
END;
$$ LANGUAGE plpgsql;

-- Job pg_cron utilisant le wrapper
SELECT cron.schedule(
    'pg_partman_maintenance_ha',
    '0 * * * *',
    'SELECT partman.run_maintenance_if_primary();'
);
```

**Failover** : En cas de promotion d'un standby, pg_cron basculera automatiquement et continuera la maintenance sur le nouveau primaire.

### pg_partman + TimescaleDB

pg_partman peut coexister avec TimescaleDB, mais généralement TimescaleDB gère lui-même le partitionnement (hypertables).

**Recommandation** : Utilisez soit pg_partman, soit TimescaleDB, pas les deux ensemble sur les mêmes tables.

### pg_partman + Monitoring (Prometheus/Grafana)

Exporter des métriques de partitionnement :

```sql
-- Vue pour exposition des métriques
CREATE OR REPLACE VIEW partman.metrics AS  
SELECT  
    parent_table,
    EXTRACT(EPOCH FROM COALESCE(retention, '0'::interval)) as retention_seconds,
    premake,
    (SELECT COUNT(*)
     FROM pg_tables
     WHERE tablename LIKE REPLACE(parent_table, 'public.', '') || '_%') as partition_count
FROM partman.part_config;

-- Utiliser postgres_exporter pour exposer cette vue
```

**Dashboard Grafana** recommandé :
- Nombre de partitions par table
- Taille totale par table partitionnée
- Âge de la dernière partition créée
- Échecs de maintenance

## Performances et Optimisations

### Impact du Partitionnement sur les Performances

**Améliorations attendues** :

| Opération | Amélioration | Raison |
|-----------|--------------|--------|
| **SELECT sur plage récente** | 10-100x | Partition pruning |
| **DELETE en masse** | 100-1000x | DROP/DETACH partition vs DELETE |
| **VACUUM** | 10-50x | Opération sur petites partitions |
| **Index rebuild** | 10-50x | Moins de données par partition |
| **Backup sélectif** | Variable | Backup uniquement partitions actives |

**Overhead potentiel** :

| Opération | Impact | Mitigation |
|-----------|--------|------------|
| **Planning de requête** | +10-50ms | Utiliser prepared statements |
| **Scan complet** | Légèrement plus lent | Éviter les queries sans filtre temporel |
| **Métadonnées** | Augmentation pg_class | Limiter le nombre total de partitions |

### Optimisation des Requêtes sur Tables Partitionnées

#### Utiliser le Partition Pruning

**Mauvaise requête** (scan toutes les partitions) :

```sql
-- ❌ Pas de filtre sur clé de partitionnement
SELECT * FROM logs WHERE user_id = 12345;
-- → Scannera TOUTES les partitions !
```

**Bonne requête** (partition pruning actif) :

```sql
-- ✅ Filtre sur la clé de partitionnement
SELECT * FROM logs  
WHERE timestamp >= '2025-11-01'  
  AND timestamp < '2025-12-01'
  AND user_id = 12345;
-- → Scannera uniquement la partition de novembre 2025
```

#### Vérifier le Plan d'Exécution

```sql
EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM logs  
WHERE timestamp >= CURRENT_DATE - INTERVAL '7 days';  

-- Chercher dans le plan :
-- "Partitions scanned: 7 of 100" ← Bon signe !
-- "Partitions scanned: 100 of 100" ← Mauvais signe !
```

#### Index Stratégiques

```sql
-- Template avec index composites intelligents
CREATE TABLE logs_template (LIKE logs INCLUDING ALL);

-- Index composite : clé de partition + colonnes fréquentes
CREATE INDEX idx_logs_timestamp_user ON logs_template(timestamp, user_id);  
CREATE INDEX idx_logs_timestamp_level ON logs_template(timestamp, level);  

-- Éviter les index redondants sur chaque partition
-- Un index sur (user_id) seul est généralement moins utile
```

### Tuning de pg_partman

#### Réduire l'Overhead de run_maintenance()

```sql
-- Désactiver ANALYZE si les partitions sont rarement modifiées
UPDATE partman.part_config  
SET optimize_trigger = 0  -- Désactiver l'analyse automatique  
WHERE parent_table = 'public.logs';  

-- Analyser manuellement à la demande
ANALYZE logs;
```

#### Parallélisation

pg_partman n'a pas de parallélisation native, mais vous pouvez :

```sql
-- Exécuter la maintenance en parallèle pour plusieurs tables
SELECT cron.schedule('maintain_logs', '0 * * * *',
    'SELECT partman.run_maintenance(''public.logs'');');

SELECT cron.schedule('maintain_transactions', '5 * * * *',  -- Décalé de 5min
    'SELECT partman.run_maintenance(''public.transactions'');');
```

## Migrer vers pg_partman depuis une Solution Manuelle

Si vous avez déjà des partitions créées manuellement, pg_partman peut reprendre la gestion.

### Étape 1 : Analyser l'existant

```sql
-- Identifier les partitions actuelles
SELECT
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) as size
FROM pg_tables  
WHERE tablename LIKE 'logs_%'  
ORDER BY tablename;  
```

### Étape 2 : Adapter le schéma si nécessaire

pg_partman attend une nomenclature spécifique :
- Tables parent : `logs`
- Partitions : `logs_pYYYY_MM_DD` (pour daily), `logs_pYYYY_MM` (pour monthly)

Si vos partitions ont une nomenclature différente, vous devrez peut-être les renommer :

```sql
-- Exemple : Renommer logs_202501 en logs_p2025_01
ALTER TABLE logs_202501 RENAME TO logs_p2025_01;
```

### Étape 3 : Configurer pg_partman

```sql
SELECT partman.create_parent(
    p_parent_table := 'public.logs',
    p_control := 'timestamp',
    p_interval := 'monthly',
    p_premake := 4,
    p_start_partition := (SELECT MAX(partition_boundary) FROM existing_partitions)
);
```

### Étape 4 : Tester la maintenance

```sql
-- Exécuter manuellement
SELECT partman.run_maintenance('public.logs');

-- Vérifier que de nouvelles partitions sont créées
SELECT partman.show_partitions('public.logs');
```

### Étape 5 : Automatiser

```sql
SELECT cron.schedule(
    'maintain_migrated_logs',
    '0 2 * * *',
    'SELECT partman.run_maintenance(''public.logs'');'
);
```

| **Performance** | Excellente | Excellente (optimisée pour time-series) |
| **Features avancées** | Basiques | Compression, continuous aggregates |
| **Dépendance** | Extension simple | Nécessite TimescaleDB |

**Verdict** : TimescaleDB pour séries temporelles IoT/métriques, pg_partman pour cas génériques.

### pg_partman vs Citus (Partitionnement Distribué)

| Critère | pg_partman | Citus |
|---------|------------|-------|
| **Architecture** | Single-node | Distributed (multi-nodes) |
| **Scalabilité** | Verticale | Horizontale |
| **Complexité** | Faible | Élevée |
| **Coût** | Gratuit (open-source) | Gratuit + Enterprise |

**Verdict** : pg_partman pour single-node, Citus pour sharding distribué sur plusieurs serveurs.

## Limitations et Considérations

### Limitations de pg_partman

1. **Pas de support pour les vues partitionnées** : Uniquement les tables  
2. **Gestion limitée des partitions LIST** : Moins automatisée que RANGE  
3. **Pas de migration de données automatique** : Doit être scriptée manuellement  
4. **Dépend de pg_cron** : Pour l'automatisation (ou équivalent)  
5. **Pas de gestion de dépendances complexes** : Entre partitions ou avec d'autres tables

### Limitations de PostgreSQL

1. **Foreign Keys** : Les FK sur tables partitionnées doivent inclure la clé de partitionnement  
2. **Triggers BEFORE ROW** : Peuvent être plus lents avec de nombreuses partitions  
3. **Nombre de partitions** : Au-delà de 1000 partitions, overhead sur les métadonnées  
4. **Partition pruning** : Ne fonctionne qu'avec des filtres sur la clé de partitionnement

### Quand NE PAS Utiliser le Partitionnement

❌ **Ne partitionnez pas si** :
- Votre table a moins de 10 millions de lignes
- Vous n'avez pas de dimension temporelle ou catégorielle claire
- Vos requêtes ne filtrent jamais sur une colonne spécifique
- Vous n'avez pas de stratégie d'archivage

Le partitionnement ajoute de la complexité. Il doit résoudre un problème réel (performance, archivage).

## Résumé

**pg_partman** est une extension PostgreSQL indispensable pour gérer efficacement les tables partitionnées à grande échelle.

### Points Clés

- ✅ **Automatisation complète** : Création, maintenance, et suppression de partitions  
- ✅ **Intégration pg_cron** : Planification automatique de la maintenance  
- ✅ **Gestion de retention** : Archivage et nettoyage automatisés  
- ✅ **Flexibilité** : Support Range, List, Hash, avec ou sans template  
- ✅ **Production-ready** : Utilisé par des milliers d'entreprises

### Quand Utiliser pg_partman ?

- Tables avec **10+ millions de lignes**
- Données **temporelles** (logs, événements, transactions)
- Besoin d'**archivage** régulier
- Optimisation des **performances** sur requêtes récentes
- Simplification de la **maintenance** (VACUUM, REINDEX)

### Architecture Recommandée

```
PostgreSQL 18
    ├── pg_partman (gestion de partitions)
    ├── pg_cron (automatisation)
    └── Monitoring (alertes sur échecs)
```

### Workflow Type

1. **Créer** la table partitionnée avec pg_partman  
2. **Configurer** premake, retention, template  
3. **Automatiser** avec pg_cron (run_maintenance)  
4. **Surveiller** régulièrement (logs, alertes)  
5. **Optimiser** selon les besoins (ajuster intervalle, retention)
