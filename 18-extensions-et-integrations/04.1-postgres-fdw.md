# 18.4.1. postgres_fdw : Fédération PostgreSQL

## Introduction

**postgres_fdw** (Foreign Data Wrapper) est une extension PostgreSQL qui permet d'accéder à des données stockées dans d'autres bases PostgreSQL comme si elles étaient locales. C'est un peu comme créer un "pont" entre deux bases de données PostgreSQL pour qu'elles puissent communiquer.

### Pourquoi utiliser postgres_fdw ?

Imaginez que votre entreprise possède :
- Une base de données PostgreSQL pour les **ventes** (sur un serveur A)
- Une base de données PostgreSQL pour les **stocks** (sur un serveur B)

Avec postgres_fdw, vous pouvez interroger les données des deux bases depuis une seule requête SQL, sans avoir à :
- Dupliquer les données
- Créer des exports/imports manuels
- Écrire du code applicatif complexe

## Concepts Fondamentaux

### Qu'est-ce qu'un Foreign Data Wrapper ?

Un **Foreign Data Wrapper (FDW)** est une technologie standardisée (norme SQL/MED) qui permet à PostgreSQL de se connecter à des sources de données externes. PostgreSQL agit comme une "passerelle" vers d'autres systèmes.

**postgres_fdw** est le FDW spécialisé pour se connecter à d'autres bases PostgreSQL.

### Terminologie

| Terme | Définition |
|-------|------------|
| **Serveur local** | La base PostgreSQL depuis laquelle vous lancez vos requêtes |
| **Serveur distant** | La base PostgreSQL qui contient les données que vous voulez interroger |
| **Foreign Server** | La définition de connexion au serveur distant dans PostgreSQL |
| **Foreign Table** | Une table virtuelle sur le serveur local qui représente une table distante |
| **User Mapping** | Les identifiants de connexion pour accéder au serveur distant |

### Schéma d'architecture

```
┌────────────────────────────────────┐
│    Serveur Local (PostgreSQL)      │
│                                    │
│  ┌──────────────────────────────┐  │
│  │  Votre requête SQL           │  │
│  └──────────┬───────────────────┘  │
│             │                      │
│  ┌──────────▼───────────────────┐  │
│  │  Foreign Table (virtuelle)   │  │
│  │  - clients_distants          │  │
│  └──────────┬───────────────────┘  │
│             │                      │
│  ┌──────────▼───────────────────┐  │
│  │  postgres_fdw (extension)    │  │
│  └──────────┬───────────────────┘  │
└─────────────┼──────────────────────┘
              │ Connexion réseau
              │ (protocole PostgreSQL)
┌─────────────▼──────────────────────┐
│   Serveur Distant (PostgreSQL)     │
│                                    │
│  ┌──────────────────────────────┐  │
│  │  Table réelle                │  │
│  │  - clients (données)         │  │
│  └──────────────────────────────┘  │
└────────────────────────────────────┘
```

## Installation et Configuration

### Étape 1 : Installer l'extension

Sur le serveur local, postgres_fdw est généralement inclus dans PostgreSQL. Il suffit de l'activer :

```sql
-- Sur la base de données locale
CREATE EXTENSION postgres_fdw;
```

### Étape 2 : Créer le Foreign Server

Définissez la connexion vers le serveur distant :

```sql
CREATE SERVER serveur_ventes
    FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (
        host 'db-ventes.exemple.com',  -- Adresse du serveur distant
        port '5432',                     -- Port PostgreSQL (5432 par défaut)
        dbname 'ventes_db'               -- Nom de la base distante
    );
```

**Explications** :
- `serveur_ventes` : Nom que vous donnez à cette connexion (peut être n'importe quoi)  
- `host` : Adresse IP ou nom de domaine du serveur distant  
- `port` : Port d'écoute PostgreSQL  
- `dbname` : Nom de la base de données distante

### Étape 3 : Créer le User Mapping

Définissez les identifiants de connexion :

```sql
CREATE USER MAPPING FOR utilisateur_local
    SERVER serveur_ventes
    OPTIONS (
        user 'admin_ventes',      -- Utilisateur sur le serveur distant
        password 'mot_de_passe'   -- Mot de passe (attention : stocké en clair!)
    );
```

**⚠️ Sécurité** : Le mot de passe est stocké dans le catalogue système. En production, utilisez :
- Des certificats SSL
- Des fichiers `.pgpass`
- Des gestionnaires de secrets

### Étape 4 : Créer une Foreign Table

Créez une table virtuelle qui représente la table distante :

```sql
CREATE FOREIGN TABLE clients_distants (
    id INTEGER,
    nom VARCHAR(100),
    email VARCHAR(100),
    date_creation TIMESTAMP
)
SERVER serveur_ventes  
OPTIONS (  
    schema_name 'public',    -- Schéma sur le serveur distant
    table_name 'clients'     -- Nom de la table distante
);
```

**Important** : La structure (colonnes, types) doit correspondre à la table distante.

### Méthode alternative : IMPORT FOREIGN SCHEMA

Au lieu de définir manuellement chaque colonne, vous pouvez importer automatiquement la structure :

```sql
IMPORT FOREIGN SCHEMA public
    LIMIT TO (clients, commandes)  -- Limiter aux tables spécifiques
    FROM SERVER serveur_ventes
    INTO public;                    -- Schéma local de destination
```

Cette commande crée automatiquement les foreign tables correspondantes.

## Utilisation

### Requêtes de lecture

Une fois configurée, une foreign table s'utilise comme une table normale :

```sql
-- Sélectionner tous les clients distants
SELECT * FROM clients_distants;

-- Filtrer
SELECT nom, email  
FROM clients_distants  
WHERE date_creation > '2024-01-01';  

-- Jointure avec une table locale
SELECT
    c.nom,
    l.montant
FROM clients_distants c  
INNER JOIN ventes_locales l ON c.id = l.client_id;  
```

### Requêtes d'écriture

postgres_fdw supporte également les opérations d'écriture :

```sql
-- Insertion
INSERT INTO clients_distants (id, nom, email, date_creation)  
VALUES (100, 'Nouveau Client', 'client@exemple.com', NOW());  

-- Mise à jour
UPDATE clients_distants  
SET email = 'nouveau@exemple.com'  
WHERE id = 100;  

-- Suppression
DELETE FROM clients_distants WHERE id = 100;
```

**Note** : Ces opérations modifient directement les données sur le serveur distant.

## Comment ça fonctionne ?

### Le mécanisme de Query Pushdown

postgres_fdw est **intelligent** : il essaie d'envoyer un maximum de traitement vers le serveur distant pour optimiser les performances.

**Exemple** :

```sql
SELECT nom, email  
FROM clients_distants  
WHERE pays = 'France'  
ORDER BY nom;  
```

**Sans pushdown** (mauvais) :
1. Transférer TOUTES les lignes vers le serveur local  
2. Filtrer `WHERE pays = 'France'` localement  
3. Trier `ORDER BY nom` localement

**Avec pushdown** (bon) :
1. Envoyer la requête complète au serveur distant  
2. Le serveur distant fait le filtrage et le tri  
3. Seules les lignes résultantes sont transférées

### Vérifier le pushdown avec EXPLAIN

```sql
EXPLAIN (VERBOSE)  
SELECT * FROM clients_distants WHERE pays = 'France';  
```

Résultat exemple :

```
Foreign Scan on clients_distants
  Output: id, nom, email, pays
  Remote SQL: SELECT id, nom, email, pays
              FROM public.clients
              WHERE (pays = 'France'::text)
```

Le `Remote SQL` montre que le `WHERE` est bien exécuté à distance.

### Opérations poussées vs rapatriées

| Opération | Poussée ? | Commentaire |
|-----------|-----------|-------------|
| `WHERE` avec opérateurs simples | ✅ Oui | `=`, `<`, `>`, `IN`, etc. |
| `ORDER BY` | ✅ Oui | Tri distant |
| `LIMIT` | ✅ Oui | Limitation à distance |
| `JOIN` entre foreign tables du même serveur | ✅ Oui | Jointure distante |
| `JOIN` entre foreign table et table locale | ❌ Non | Rapatriement des données |
| Agrégations (`SUM`, `COUNT`, etc.) | ✅ Oui | Depuis PostgreSQL 10+ |
| Fonctions complexes ou personnalisées | ❌ Variable | Dépend de la fonction |

## Cas d'usage réels

### 1. Fédération de bases multiples

**Contexte** : Une entreprise avec plusieurs filiales, chacune ayant sa propre base PostgreSQL.

**Solution** : Créer une base "fédérée" centrale qui interroge toutes les bases filiales via postgres_fdw.

```sql
-- Filiale Paris
CREATE FOREIGN TABLE ventes_paris (...) SERVER serveur_paris;

-- Filiale Lyon
CREATE FOREIGN TABLE ventes_lyon (...) SERVER serveur_lyon;

-- Requête consolidée
SELECT 'Paris' AS filiale, SUM(montant)  
FROM ventes_paris  
UNION ALL  
SELECT 'Lyon', SUM(montant)  
FROM ventes_lyon;  
```

### 2. Migration progressive

**Contexte** : Migrer d'une ancienne base vers une nouvelle sans interruption de service.

**Solution** :
1. Nouvelle base avec foreign tables vers l'ancienne  
2. Applications lisent la nouvelle base  
3. Les données non migrées sont accessibles via FDW  
4. Migration progressive table par table

### 3. Reporting et Data Warehouse

**Contexte** : Base de production OLTP + base analytique OLAP.

**Solution** : La base analytique utilise postgres_fdw pour importer des données en temps réel depuis la production sans impacter celle-ci.

```sql
-- Sur le serveur de reporting
CREATE FOREIGN TABLE ventes_production (...) SERVER prod_oltp;

-- ETL simplifié
INSERT INTO ventes_analyse  
SELECT * FROM ventes_production  
WHERE date > (SELECT MAX(date) FROM ventes_analyse);  
```

### 4. Séparation des domaines métier

**Contexte** : Architecture microservices avec une base par service.

**Solution** : Chaque service accède à ses propres données, mais peut lire (lecture seule) les données d'autres services via FDW quand nécessaire.

```
Service Commandes → FDW vers Service Clients (lecture)  
Service Facturation → FDW vers Service Commandes (lecture)  
```

## Performance et Optimisation

### Bonnes pratiques

#### 1. Utiliser des index sur les tables distantes

Les index sur le serveur distant sont utilisés lors du pushdown :

```sql
-- Sur le serveur DISTANT
CREATE INDEX idx_clients_pays ON clients(pays);

-- Sur le serveur LOCAL, ce filtre bénéficiera de l'index
SELECT * FROM clients_distants WHERE pays = 'France';
```

#### 2. Limiter les colonnes sélectionnées

```sql
-- ❌ Mauvais : transfère toutes les colonnes
SELECT * FROM clients_distants WHERE id = 100;

-- ✅ Bon : transfère seulement les colonnes nécessaires
SELECT nom, email FROM clients_distants WHERE id = 100;
```

#### 3. Utiliser des filtres restrictifs

```sql
-- ❌ Mauvais : transfère toute la table
SELECT * FROM clients_distants;

-- ✅ Bon : filtre côté distant
SELECT * FROM clients_distants WHERE date_creation > '2024-01-01';
```

#### 4. Options de performance

```sql
-- Lors de la création du serveur
CREATE SERVER serveur_ventes
    FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (
        host 'db-ventes.exemple.com',
        port '5432',
        dbname 'ventes_db',
        fetch_size '10000',      -- Taille des lots (par défaut 100)
        use_remote_estimate 'true' -- Utiliser les stats distantes
    );
```

**Options importantes** :
- `fetch_size` : Nombre de lignes récupérées par lot (augmenter pour gros volumes)  
- `use_remote_estimate` : Utilise les statistiques du serveur distant pour le plan d'exécution

### Surveillance

#### Vérifier les connexions actives

```sql
-- Sur le serveur DISTANT
SELECT
    datname,
    usename,
    application_name,
    client_addr,
    state
FROM pg_stat_activity  
WHERE application_name = 'postgres_fdw';  
```

#### Analyser les performances

```sql
EXPLAIN (ANALYZE, VERBOSE, BUFFERS)  
SELECT * FROM clients_distants WHERE pays = 'France';  
```

Regardez :
- **Remote SQL** : Que fait le serveur distant ?  
- **Execution Time** : Temps total  
- **Rows** : Nombre de lignes transférées

## Gestion des Transactions

### Comportement transactionnel

postgres_fdw participe aux transactions du serveur local :

```sql
BEGIN;

-- Modification locale
UPDATE ventes_locales SET statut = 'traité' WHERE id = 1;

-- Modification distante
UPDATE clients_distants SET derniere_commande = NOW() WHERE id = 100;

COMMIT;  -- Les DEUX modifications sont validées
```

**Important** : Si `COMMIT` échoue, PostgreSQL essaie de faire un `ROLLBACK` sur le serveur distant, mais ce n'est **pas une transaction distribuée garantie à 100%** (pas de 2PC par défaut).

### Transactions à deux phases (2PC)

Pour des garanties transactionnelles fortes, activez le 2PC :

```sql
-- Sur le Foreign Server
ALTER SERVER serveur_ventes OPTIONS (ADD use_two_phase_commit 'true');
```

Cela garantit que les transactions sont atomiques entre les deux serveurs, au prix d'une latence accrue.

## Sécurité

### Contrôle d'accès

#### Permissions nécessaires

**Sur le serveur LOCAL** :
```sql
-- L'utilisateur doit pouvoir utiliser le foreign server
GRANT USAGE ON FOREIGN SERVER serveur_ventes TO utilisateur;

-- Et accéder à la foreign table
GRANT SELECT ON clients_distants TO utilisateur;
```

**Sur le serveur DISTANT** :
```sql
-- L'utilisateur mappé doit avoir les permissions
GRANT SELECT ON clients TO admin_ventes;
```

### Bonnes pratiques de sécurité

#### 1. Créer des utilisateurs dédiés

```sql
-- Sur le serveur DISTANT
CREATE USER fdw_readonly WITH PASSWORD 'mot_de_passe_fort';  
GRANT CONNECT ON DATABASE ventes_db TO fdw_readonly;  
GRANT USAGE ON SCHEMA public TO fdw_readonly;  
GRANT SELECT ON ALL TABLES IN SCHEMA public TO fdw_readonly;  
```

#### 2. Utiliser SSL/TLS

```sql
CREATE SERVER serveur_ventes
    FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (
        host 'db-ventes.exemple.com',
        port '5432',
        dbname 'ventes_db',
        sslmode 'require'  -- Force SSL
    );
```

#### 3. Fichier .pgpass pour les mots de passe

Au lieu de stocker les mots de passe dans PostgreSQL :

**Sur le serveur local**, créez `~/.pgpass` :
```
db-ventes.exemple.com:5432:ventes_db:admin_ventes:mot_de_passe_fort
```

Permissions : `chmod 600 ~/.pgpass`

Puis créez le mapping sans mot de passe :
```sql
CREATE USER MAPPING FOR utilisateur_local
    SERVER serveur_ventes
    OPTIONS (user 'admin_ventes');  -- Pas de password
```

## Limitations et Considérations

### Limitations techniques

#### 1. Pas de DDL sur les foreign tables

```sql
-- ❌ Impossible
ALTER TABLE clients_distants ADD COLUMN telephone VARCHAR(20);

-- ✅ À faire sur le serveur distant
-- Puis resynchoniser la définition locale
```

#### 2. Pas d'index locaux

Vous ne pouvez pas créer d'index sur une foreign table :

```sql
-- ❌ Erreur
CREATE INDEX idx_nom ON clients_distants(nom);
```

Seuls les index du serveur distant sont utilisés.

#### 3. Certaines opérations ne sont pas poussées

- Fonctions définies par l'utilisateur (UDF)
- Certaines fonctions PostgreSQL spécifiques
- Expressions complexes

#### 4. Latence réseau

Chaque requête traverse le réseau. Pour des volumes importants :
- Privilégiez la réplication physique ou logique
- Utilisez des vues matérialisées pour le cache

### Quand NE PAS utiliser postgres_fdw

❌ **Évitez** postgres_fdw quand :
- Les données doivent être **ultra-rapides** (millisecondes) : Privilégiez la réplication
- Vous avez besoin de **transactions distribuées complexes** : Utilisez une solution dédiée (XA, Saga)
- Les volumes de données transférées sont **très importants** : Préférez ETL batch ou réplication
- La **latence réseau** est élevée (>50ms) : Les performances seront dégradées

✅ **Utilisez** postgres_fdw quand :
- Vous avez besoin d'**accès ponctuel** à des données distantes
- Vous voulez **fédérer** plusieurs bases PostgreSQL
- Vous faites des **migrations progressives**
- Vous construisez un **data warehouse** avec sources multiples

## Dépannage

### Problèmes courants

#### Erreur : "could not connect to server"

```
ERROR:  could not connect to server "serveur_ventes"  
DETAIL:  could not translate host name "db-ventes.exemple.com" to address  
```

**Solutions** :
- Vérifier la connectivité réseau : `ping db-ventes.exemple.com`
- Vérifier le firewall (port 5432 ouvert ?)
- Vérifier `pg_hba.conf` sur le serveur distant

#### Erreur : "password authentication failed"

```
ERROR:  password authentication failed for user "admin_ventes"
```

**Solutions** :
- Vérifier les credentials dans le user mapping
- Vérifier que l'utilisateur existe sur le serveur distant
- Vérifier `pg_hba.conf` autorise la connexion

#### Performances lentes

**Diagnostic** :
```sql
EXPLAIN (ANALYZE, VERBOSE)  
SELECT * FROM clients_distants WHERE condition;  
```

**Solutions** :
- Vérifier que le WHERE est poussé (Remote SQL)
- Ajouter des index sur le serveur distant
- Augmenter `fetch_size`
- Limiter les colonnes sélectionnées

#### Voir les connexions FDW

```sql
-- Sur le serveur distant
SELECT * FROM pg_stat_activity  
WHERE application_name LIKE '%fdw%';  
```

## Outils et Extensions Complémentaires

### postgres_fdw_validator

Valide la configuration avant de l'appliquer :

```sql
SELECT * FROM postgres_fdw_validator(
    '{"host": "db-ventes.exemple.com", "port": "5432", "dbname": "ventes_db"}'::text[]
);
```

### Monitoring avec pg_stat_statements

Sur le serveur distant, activez `pg_stat_statements` pour voir les requêtes FDW :

```sql
CREATE EXTENSION pg_stat_statements;

-- Voir les requêtes FDW
SELECT query, calls, total_exec_time  
FROM pg_stat_statements  
WHERE query LIKE '%clients%'  
ORDER BY total_exec_time DESC;  
```

### Résumé des points clés

✅ **postgres_fdw** permet d'accéder à des données PostgreSQL distantes comme si elles étaient locales

✅ Configuration en 4 étapes : Extension → Foreign Server → User Mapping → Foreign Table

✅ **Query Pushdown** : PostgreSQL optimise en envoyant le maximum de traitement à distance

✅ Cas d'usage principaux : Fédération, migration, reporting, microservices

⚠️ Attention à la latence réseau et aux volumes de données

⚠️ Ne remplace pas la réplication pour les besoins haute performance
