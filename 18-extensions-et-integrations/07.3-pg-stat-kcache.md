# 18.7.3. pg_stat_kcache : Métriques Système pour PostgreSQL

## Introduction

**pg_stat_kcache** est une extension PostgreSQL qui collecte des **métriques au niveau système** (CPU, I/O disque, mémoire) pour chaque requête SQL exécutée. Elle complète l'extension **pg_stat_statements** en ajoutant une dimension système indispensable pour l'analyse de performance.

### Pourquoi les métriques système sont-elles importantes ?

Lorsque vous analysez la performance d'une base de données, connaître uniquement le **temps d'exécution** d'une requête n'est pas suffisant. Vous devez comprendre **où** ce temps est passé :

- ⏱️ **Temps CPU** : La requête calcule beaucoup ? (tri, agrégation, jointures complexes)  
- 💾 **I/O disque** : La requête lit/écrit beaucoup sur disque ? (scan séquentiel, index inefficace)  
- 📊 **Cache hits** : Les données sont-elles en mémoire ou lues depuis le disque ?  
- 🧠 **Mémoire** : Quelle quantité de RAM la requête consomme-t-elle ?

**Exemple concret** :

```
Requête A : 10 secondes d'exécution
├─ 1 seconde de CPU
└─ 9 secondes d'attente I/O disque  ← GOULOT D'ÉTRANGLEMENT

Requête B : 10 secondes d'exécution
├─ 9.5 secondes de CPU  ← GOULOT D'ÉTRANGLEMENT
└─ 0.5 seconde d'I/O disque
```

**Sans pg_stat_kcache** : Vous savez que les deux requêtes prennent 10 secondes, mais vous ne savez pas pourquoi.

**Avec pg_stat_kcache** : Vous identifiez immédiatement que :
- Requête A → Ajouter un index pour réduire l'I/O
- Requête B → Optimiser l'algorithme pour réduire le CPU

### Qu'est-ce que pg_stat_kcache apporte ?

pg_stat_kcache enrichit **pg_stat_statements** avec :

- ✅ **Temps CPU utilisateur** : Temps passé à exécuter du code utilisateur  
- ✅ **Temps CPU système** : Temps passé dans les appels système (kernel)  
- ✅ **Lectures disque** : Nombre de blocs lus depuis le disque  
- ✅ **Écritures disque** : Nombre de blocs écrits sur le disque  
- ✅ **Lectures logiques** : Nombre de blocs lus (cache + disque)  
- ✅ **Faults de mémoire** : Défauts de page mineurs/majeurs

### Architecture : pg_stat_statements + pg_stat_kcache

```
┌─────────────────────────────────────────────────┐
│              Application                        │
└─────────────────┬───────────────────────────────┘
                  │ Exécute des requêtes SQL
                  ↓
┌────────────────────────────────────────────────┐
│           PostgreSQL Server                    │
│                                                │
│  ┌───────────────────────────────────────────┐ │
│  │  Extension: pg_stat_statements            │ │
│  │  (Collecte: temps, appels, lignes)        │ │
│  └───────────────────────────────────────────┘ │
│                  ↓                             │
│  ┌───────────────────────────────────────────┐ │
│  │  Extension: pg_stat_kcache                │ │
│  │  (Collecte: CPU, I/O, mémoire)            │ │
│  │  Via /proc/[pid]/stat (Linux)             │ │
│  └───────────────────────────────────────────┘ │
│                  ↓                             │
│  ┌───────────────────────────────────────────┐ │
│  │  Vue: pg_stat_kcache                      │ │
│  │  (Métriques accessibles via SQL)          │ │
│  └───────────────────────────────────────────┘ │
└────────────────────────────────────────────────┘
```

### Prérequis conceptuels

Pour bien comprendre pg_stat_kcache, quelques notions de base :

#### Qu'est-ce que le CPU ?

Le **CPU (processeur)** exécute les instructions de votre code. Dans PostgreSQL :
- **Calculs** : Agrégations (SUM, AVG), tris (ORDER BY), jointures  
- **Comparaisons** : Filtres WHERE, évaluations de conditions  
- **Traitement** : Décompression, parsing, optimisation de requêtes

**Temps CPU** = Temps où le processeur travaille activement pour votre requête.

#### Qu'est-ce que l'I/O disque ?

L'**I/O (Input/Output)** représente les lectures et écritures sur le disque :
- **Lectures** : Récupérer des données depuis le disque (quand pas en cache)  
- **Écritures** : Persister des données modifiées (INSERT, UPDATE, DELETE)

**Temps I/O** = Temps d'attente pour que le disque réponde.

**Important** : Le disque est **1000× plus lent** que la RAM. C'est souvent le goulot d'étranglement principal.

#### Qu'est-ce que le cache (Buffer Cache) ?

PostgreSQL maintient un **cache en mémoire** (`shared_buffers`) pour éviter de relire le disque :
- **Cache hit** : Donnée trouvée en mémoire (rapide, ~µs)  
- **Cache miss** : Donnée lue depuis le disque (lent, ~ms)

**Objectif** : Maximiser le taux de cache hit (>95% est excellent).

## Installation et Configuration

### Prérequis

- **PostgreSQL 9.4+** (idéalement PostgreSQL 18)  
- **Linux** : pg_stat_kcache utilise `/proc/[pid]/stat` (spécifique à Linux)  
- **pg_stat_statements** : Doit être installé et activé  
- **Droits superutilisateur** : Pour l'installation initiale

**Note** : pg_stat_kcache ne fonctionne que sur Linux. Sur Windows/macOS, certaines métriques ne seront pas disponibles.

### Processus d'installation (théorique)

#### Étape 1 : Installer pg_stat_statements (si pas déjà fait)

pg_stat_kcache dépend de pg_stat_statements.

```ini
# postgresql.conf
shared_preload_libraries = 'pg_stat_statements'

# Configuration recommandée pour pg_stat_statements
pg_stat_statements.max = 10000          # Nombre de requêtes distinctes à tracker  
pg_stat_statements.track = all          # Tracker toutes les requêtes  
pg_stat_statements.track_utility = on   # Inclure les commandes DDL  
```

**Important** : Redémarrer PostgreSQL après modification de `shared_preload_libraries`.

```sql
-- Créer l'extension pg_stat_statements
CREATE EXTENSION pg_stat_statements;
```

#### Étape 2 : Installer pg_stat_kcache

Ajouter pg_stat_kcache aux bibliothèques préchargées :

```ini
# postgresql.conf
shared_preload_libraries = 'pg_stat_statements,pg_stat_kcache'
```

**Important** : Les extensions doivent être séparées par des virgules, sans espaces.

Redémarrer PostgreSQL :

```bash
# Commande système (exemple pour systemd)
sudo systemctl restart postgresql
```

#### Étape 3 : Créer l'extension

```sql
-- Se connecter à la base de données
CREATE EXTENSION pg_stat_kcache;
```

#### Étape 4 : Vérifier l'installation

```sql
-- Vérifier que l'extension est active
SELECT * FROM pg_extension WHERE extname = 'pg_stat_kcache';

-- Vérifier que la vue existe
\d pg_stat_kcache

-- Tester une requête simple
SELECT * FROM pg_stat_kcache LIMIT 5;
```

Si tout fonctionne, vous devriez voir des métriques système pour les requêtes exécutées.

### Configuration Avancée (Optionnelle)

pg_stat_kcache hérite de la configuration de pg_stat_statements. Quelques paramètres utiles :

```ini
# Augmenter le nombre de requêtes trackées
pg_stat_statements.max = 20000

# Réinitialisation automatique en cas de saturation
# (par défaut, les anciennes requêtes sont évincées)

# Activer le tracking des fonctions (coût additionnel)
pg_stat_statements.track = all
```

## Comprendre les Métriques de pg_stat_kcache

### Structure de la vue pg_stat_kcache

```sql
SELECT * FROM pg_stat_kcache LIMIT 1;
```

**Colonnes principales** :

| Colonne | Type | Signification |
|---------|------|---------------|
| `queryid` | bigint | Identifiant unique de la requête (hash) |
| `userid` | oid | Utilisateur ayant exécuté la requête |
| `dbid` | oid | Base de données |
| `user_time` | double precision | Temps CPU utilisateur (secondes) |
| `system_time` | double precision | Temps CPU système/kernel (secondes) |
| `minflts` | bigint | Minor page faults (accès mémoire sans I/O) |
| `majflts` | bigint | Major page faults (accès mémoire avec I/O disque) |
| `nswaps` | bigint | Nombre de swaps (rare, indique manque de RAM) |
| `reads` | bigint | Nombre de lectures I/O (blocs lus depuis disque) |
| `writes` | bigint | Nombre d'écritures I/O (blocs écrits sur disque) |
| `msgsnds` | bigint | Messages IPC envoyés |
| `msgrcvs` | bigint | Messages IPC reçus |
| `nsignals` | bigint | Signaux reçus |
| `nvcsws` | bigint | Context switches volontaires |
| `nivcsws` | bigint | Context switches involontaires |

### Métriques Clés Expliquées

#### 1. user_time (Temps CPU Utilisateur)

**Définition** : Temps passé à exécuter le code PostgreSQL (calculs, traitement de données).

**Interprétation** :
- **Élevé** → La requête fait beaucoup de calculs (jointures, agrégations, tris)  
- **Faible** → La requête est légère en calculs ou passe son temps à attendre (I/O)

**Exemple** :
```sql
SELECT
    query,
    calls,
    total_exec_time,
    user_time,
    ROUND(100.0 * user_time / NULLIF(total_exec_time, 0), 2) as cpu_percent
FROM pg_stat_statements pss  
JOIN pg_stat_kcache psk USING (queryid, userid, dbid)  
WHERE total_exec_time > 0  
ORDER BY user_time DESC  
LIMIT 10;  
```

**Si cpu_percent > 80%** → La requête est CPU-bound (optimiser l'algorithme, ajouter des index pour réduire les calculs).

#### 2. system_time (Temps CPU Système)

**Définition** : Temps passé dans les appels système (kernel) : I/O, gestion de la mémoire, locks.

**Interprétation** :
- **Élevé** → La requête fait beaucoup d'appels système (I/O, locks, network)  
- **Généralement plus faible** que `user_time` pour des requêtes normales

**Ratio typique** : `user_time` / `system_time` ≈ 3:1 à 10:1

**Si system_time > user_time** → Problème potentiel (trop d'I/O, contention, locks).

#### 3. reads / writes (I/O Disque)

**Définition** :
- `reads` : Nombre de blocs de 8 Ko lus depuis le disque (cache miss)  
- `writes` : Nombre de blocs de 8 Ko écrits sur le disque

**Interprétation** :
```sql
SELECT
    query,
    calls,
    reads,
    writes,
    ROUND(reads / NULLIF(calls, 0)::numeric, 2) as avg_reads_per_call,
    ROUND(writes / NULLIF(calls, 0)::numeric, 2) as avg_writes_per_call
FROM pg_stat_statements pss  
JOIN pg_stat_kcache psk USING (queryid, userid, dbid)  
WHERE calls > 100  
ORDER BY reads DESC  
LIMIT 10;  
```

**Critères** :
- **avg_reads_per_call > 1000** → Requête lit beaucoup de données (scan séquentiel ?)  
- **avg_writes_per_call > 100** → Requête écrit beaucoup (bulk insert/update ?)

**Optimisation** :
- Réduire `reads` → Ajouter des index, augmenter `shared_buffers`, optimiser les requêtes
- Réduire `writes` → Batch des opérations, utiliser COPY au lieu de INSERT

#### 4. minflts / majflts (Page Faults)

**Définition** :
- **Minor page fault** : Accès à une page mémoire non chargée mais disponible en RAM (rapide)  
- **Major page fault** : Accès à une page mémoire nécessitant une lecture disque (lent)

**Interprétation** :
```sql
SELECT
    query,
    minflts,
    majflts,
    CASE
        WHEN majflts > minflts * 0.01 THEN 'ALERTE: Trop de major faults'
        ELSE 'OK'
    END as status
FROM pg_stat_statements pss  
JOIN pg_stat_kcache psk USING (queryid, userid, dbid)  
ORDER BY majflts DESC  
LIMIT 10;  
```

**Critères** :
- **majflts élevé** → Manque de RAM, données swappées sur disque  
- **minflts élevé** → Normal pour grandes allocations mémoire

**Action** : Si `majflts` est élevé, augmentez la RAM du serveur ou optimisez les requêtes pour consommer moins de mémoire.

#### 5. nvcsws / nivcsws (Context Switches)

**Définition** :
- **Volontaire (nvcsws)** : Le processus cède volontairement le CPU (attente I/O, lock)  
- **Involontaire (nivcsws)** : Le système force le changement de contexte (préemption)

**Interprétation** :
- **nvcsws élevé** → La requête attend souvent (I/O, locks, network)  
- **nivcsws élevé** → Forte contention CPU (trop de processus concurrents)

**Ratio sain** : `nvcsws` >> `nivcsws` (typiquement 10:1 ou plus)

## Requêtes d'Analyse Essentielles

### 1. Top 10 des requêtes consommant le plus de CPU

```sql
SELECT
    LEFT(query, 100) as query_snippet,
    calls,
    ROUND(total_exec_time::numeric, 2) as total_time_ms,
    ROUND((user_time + system_time)::numeric, 2) as total_cpu_s,
    ROUND(100.0 * (user_time + system_time) / NULLIF(total_exec_time / 1000.0, 0), 2) as cpu_percent,
    ROUND((user_time / NULLIF(calls, 0))::numeric, 4) as avg_user_time_per_call
FROM pg_stat_statements pss  
JOIN pg_stat_kcache psk USING (queryid, userid, dbid)  
WHERE total_exec_time > 0  
ORDER BY (user_time + system_time) DESC  
LIMIT 10;  
```

**Interprétation** :
- `cpu_percent > 80%` → Requête CPU-bound (optimiser l'algorithme)  
- `cpu_percent < 20%` → Requête I/O-bound (optimiser l'I/O)

### 2. Top 10 des requêtes avec le plus d'I/O disque

```sql
SELECT
    LEFT(query, 100) as query_snippet,
    calls,
    reads,
    writes,
    reads + writes as total_io,
    ROUND((reads + writes) / NULLIF(calls, 0)::numeric, 2) as avg_io_per_call,
    pg_size_pretty((reads + writes) * 8192) as total_io_size  -- 8KB par bloc
FROM pg_stat_statements pss  
JOIN pg_stat_kcache psk USING (queryid, userid, dbid)  
WHERE calls > 0  
ORDER BY (reads + writes) DESC  
LIMIT 10;  
```

**Interprétation** :
- `avg_io_per_call > 1000` → Requête lit/écrit beaucoup de données  
- `reads >> writes` → Requête en lecture intensive (SELECT, scan)  
- `writes > reads` → Requête en écriture intensive (INSERT, UPDATE)

### 3. Identifier les requêtes I/O-bound vs CPU-bound

```sql
SELECT
    LEFT(query, 80) as query_snippet,
    calls,
    ROUND(total_exec_time::numeric, 2) as total_time_ms,
    ROUND((user_time + system_time)::numeric, 2) as cpu_time_s,
    ROUND(100.0 * (user_time + system_time) / NULLIF(total_exec_time / 1000.0, 0), 2) as cpu_percent,
    reads + writes as total_io,
    CASE
        WHEN (user_time + system_time) / NULLIF(total_exec_time / 1000.0, 0) > 0.5 THEN 'CPU-bound'
        WHEN (reads + writes) > calls * 100 THEN 'IO-bound'
        ELSE 'Balanced'
    END as bottleneck_type
FROM pg_stat_statements pss  
JOIN pg_stat_kcache psk USING (queryid, userid, dbid)  
WHERE total_exec_time > 1000  -- Au moins 1 seconde au total  
ORDER BY total_exec_time DESC  
LIMIT 20;  
```

**Actions selon le type** :

| Type | Symptômes | Actions |
|------|-----------|---------|
| **CPU-bound** | cpu_percent > 50% | Optimiser l'algorithme, ajouter des index, réduire les jointures |
| **IO-bound** | total_io élevé, cpu_percent < 30% | Ajouter des index, augmenter shared_buffers, optimiser les scans |
| **Balanced** | Mixte | Optimiser les deux aspects |

### 4. Analyse du ratio cache hit

```sql
SELECT
    LEFT(query, 80) as query_snippet,
    calls,
    shared_blks_hit,      -- Blocs trouvés en cache (pg_stat_statements)
    shared_blks_read,     -- Blocs lus depuis disque (pg_stat_statements)
    reads as disk_reads,  -- Confirmation via pg_stat_kcache
    ROUND(100.0 * shared_blks_hit / NULLIF(shared_blks_hit + shared_blks_read, 0), 2) as cache_hit_ratio
FROM pg_stat_statements pss  
JOIN pg_stat_kcache psk USING (queryid, userid, dbid)  
WHERE shared_blks_hit + shared_blks_read > 0  
ORDER BY shared_blks_read DESC  
LIMIT 10;  
```

**Interprétation** :
- **cache_hit_ratio > 95%** : Excellent, la plupart des données sont en cache  
- **cache_hit_ratio < 80%** : Problématique, trop de lectures disque  
  - → Augmenter `shared_buffers`  
  - → Ajouter des index pour réduire les scans séquentiels  
  - → Optimiser les requêtes

### 5. Détecter les requêtes avec trop de context switches

```sql
SELECT
    LEFT(query, 80) as query_snippet,
    calls,
    nvcsws as voluntary_ctx_switches,
    nivcsws as involuntary_ctx_switches,
    ROUND(nvcsws / NULLIF(calls, 0)::numeric, 2) as avg_voluntary_per_call,
    ROUND(nivcsws / NULLIF(calls, 0)::numeric, 2) as avg_involuntary_per_call,
    CASE
        WHEN nivcsws > nvcsws * 0.1 THEN 'High CPU contention'
        WHEN nvcsws > calls * 1000 THEN 'High I/O wait'
        ELSE 'Normal'
    END as status
FROM pg_stat_statements pss  
JOIN pg_stat_kcache psk USING (queryid, userid, dbid)  
WHERE calls > 10  
ORDER BY nivcsws DESC  
LIMIT 10;  
```

**Interprétation** :
- **High CPU contention** : Trop de processus concurrents, envisager de limiter les connexions  
- **High I/O wait** : La requête passe son temps à attendre l'I/O

### 6. Requêtes avec major page faults (swapping)

```sql
SELECT
    LEFT(query, 80) as query_snippet,
    calls,
    majflts as major_faults,
    ROUND(majflts / NULLIF(calls, 0)::numeric, 2) as avg_majflts_per_call
FROM pg_stat_statements pss  
JOIN pg_stat_kcache psk USING (queryid, userid, dbid)  
WHERE majflts > 0  
ORDER BY majflts DESC  
LIMIT 10;  
```

**Action** : Si `majflts` est élevé, c'est un signe critique de manque de RAM. Le système swappe sur disque (très lent).

**Solutions** :
- Augmenter la RAM du serveur
- Réduire `shared_buffers` si trop élevé (paradoxalement)
- Optimiser les requêtes pour consommer moins de mémoire (`work_mem`)

## Cas d'Usage Détaillés

### 1. Identifier une requête lente due à l'I/O

**Scénario** : Une requête prend 30 secondes et ralentit l'application.

```sql
-- Analyser la requête spécifique
SELECT
    query,
    calls,
    total_exec_time / calls as avg_time_ms,
    (user_time + system_time) / calls as avg_cpu_s,
    (reads + writes) / calls as avg_io_per_call,
    100.0 * (user_time + system_time) / (total_exec_time / 1000.0) as cpu_percent
FROM pg_stat_statements pss  
JOIN pg_stat_kcache psk USING (queryid, userid, dbid)  
WHERE query LIKE '%ma_table_problematique%'  
  AND total_exec_time > 10000;
```

**Résultats hypothétiques** :
```
avg_time_ms: 30000  
avg_cpu_s: 2.5  
avg_io_per_call: 50000  
cpu_percent: 8.3%  
```

**Diagnostic** :
- **cpu_percent = 8.3%** → Très peu de temps CPU, donc pas CPU-bound  
- **avg_io_per_call = 50000** → 50 000 blocs lus = 400 Mo par appel !  
- **Conclusion** : La requête est I/O-bound, elle fait probablement un scan séquentiel massif

**Actions** :
```sql
-- 1. Vérifier le plan d'exécution
EXPLAIN (ANALYZE, BUFFERS) SELECT ... FROM ma_table_problematique WHERE ...;

-- 2. Chercher "Seq Scan" dans le plan
-- 3. Ajouter un index sur les colonnes du WHERE

CREATE INDEX idx_ma_table_colonne ON ma_table_problematique(colonne);

-- 4. Ré-analyser après ajout de l'index
ANALYZE ma_table_problematique;
```

### 2. Détecter une requête gourmande en CPU

**Scénario** : Le serveur a une charge CPU élevée (>80%).

```sql
-- Identifier les requêtes CPU-intensives
SELECT
    LEFT(query, 100) as query_snippet,
    calls,
    ROUND((user_time + system_time)::numeric, 2) as total_cpu_s,
    ROUND((user_time + system_time) / calls::numeric, 4) as avg_cpu_per_call,
    ROUND(100.0 * (user_time + system_time) / NULLIF(total_exec_time / 1000.0, 0), 2) as cpu_percent
FROM pg_stat_statements pss  
JOIN pg_stat_kcache psk USING (queryid, userid, dbid)  
WHERE (user_time + system_time) > 10  -- Au moins 10 secondes de CPU total  
ORDER BY (user_time + system_time) DESC  
LIMIT 5;  
```

**Résultats hypothétiques** :
```
query: SELECT ... FROM orders o JOIN products p ... ORDER BY ...  
total_cpu_s: 1250.5  
avg_cpu_per_call: 0.25  
cpu_percent: 92%  
```

**Diagnostic** :
- **cpu_percent = 92%** → Requête CPU-bound
- La requête fait probablement des calculs intensifs (jointures, tris, agrégations)

**Actions** :
```sql
-- 1. Analyser le plan
EXPLAIN (ANALYZE, BUFFERS) SELECT ... FROM orders o JOIN products p ...;

-- 2. Vérifier les coûts des nœuds (Sort, Hash Join, Aggregate)
-- 3. Optimiser :
--    - Ajouter des index pour éviter les tris (ORDER BY)
--    - Préfiltrer avec WHERE pour réduire les jointures
--    - Utiliser des vues matérialisées pour les agrégations

CREATE INDEX idx_orders_date ON orders(order_date)  
WHERE order_date >= CURRENT_DATE - INTERVAL '1 year';  

-- 4. Réécrire la requête si nécessaire
-- Exemple : Remplacer une sous-requête corrélée par une jointure
```

### 3. Analyser une charge mixte (OLTP)

**Scénario** : Application web avec beaucoup de petites requêtes.

```sql
-- Vue d'ensemble de la charge
SELECT
    CASE
        WHEN (user_time + system_time) / NULLIF(total_exec_time / 1000.0, 0) > 0.5 THEN 'CPU-bound'
        WHEN (reads + writes) / NULLIF(calls, 0) > 100 THEN 'IO-bound'
        ELSE 'Balanced'
    END as workload_type,
    COUNT(*) as query_count,
    SUM(calls) as total_calls,
    ROUND(SUM(total_exec_time)::numeric, 2) as total_time_ms,
    ROUND(SUM(user_time + system_time)::numeric, 2) as total_cpu_s,
    SUM(reads + writes) as total_io
FROM pg_stat_statements pss  
JOIN pg_stat_kcache psk USING (queryid, userid, dbid)  
GROUP BY workload_type  
ORDER BY total_calls DESC;  
```

**Résultats hypothétiques** :
```
workload_type | query_count | total_calls | total_time_ms | total_cpu_s | total_io
--------------+-------------+-------------+---------------+-------------+----------
Balanced      | 150         | 5,234,890   | 12,500,000    | 3,450       | 250,000  
CPU-bound     | 25          | 45,230      | 8,750,000     | 6,800       | 5,000  
IO-bound      | 18          | 12,450      | 22,100,000    | 1,200       | 1,500,000  
```

**Diagnostic** :
- Charge majoritairement **Balanced** (bon signe)
- Quelques requêtes **CPU-bound** consomment beaucoup de CPU (focus prioritaire)
- Quelques requêtes **IO-bound** avec beaucoup d'I/O (focus secondaire)

**Stratégie d'optimisation** :
1. Optimiser d'abord les 25 requêtes CPU-bound (impact majeur)  
2. Puis les 18 requêtes IO-bound  
3. Surveiller l'évolution avec pg_stat_kcache

### 4. Diagnostiquer un problème de mémoire (swapping)

**Scénario** : Le serveur est lent, vous soupçonnez un manque de RAM.

```sql
-- Détecter le swapping via major page faults
SELECT
    datname,
    SUM(majflts) as total_major_faults,
    SUM(minflts) as total_minor_faults,
    ROUND(100.0 * SUM(majflts) / NULLIF(SUM(majflts) + SUM(minflts), 0), 2) as major_fault_percent
FROM pg_stat_kcache psk  
JOIN pg_database d ON d.oid = psk.dbid  
GROUP BY datname  
ORDER BY total_major_faults DESC;  
```

**Interprétation** :
- **major_fault_percent > 1%** → Problème de mémoire, le système swappe  
- **major_fault_percent < 0.1%** → Normal, pas de swapping significatif

**Actions si swapping détecté** :

```sql
-- 1. Identifier les requêtes responsables
SELECT
    LEFT(query, 100) as query_snippet,
    calls,
    majflts,
    ROUND(majflts / calls::numeric, 2) as majflts_per_call
FROM pg_stat_statements pss  
JOIN pg_stat_kcache psk USING (queryid, userid, dbid)  
WHERE majflts > 0  
ORDER BY majflts DESC  
LIMIT 10;  

-- 2. Vérifier la configuration mémoire
SHOW shared_buffers;  
SHOW work_mem;  
SHOW maintenance_work_mem;  

-- 3. Actions correctives :
-- a) Augmenter la RAM du serveur (solution long terme)
-- b) Réduire shared_buffers si trop élevé (25% de RAM max recommandé)
-- c) Limiter work_mem pour les requêtes individuelles
-- d) Optimiser les requêtes pour consommer moins de mémoire
```

### pg_stat_kcache + pg_cron (Automatisation)

Automatiser la collecte et l'archivage des statistiques :

```sql
-- Créer une table d'historique
CREATE TABLE monitoring.pg_stat_kcache_history (
    snapshot_time TIMESTAMPTZ DEFAULT NOW(),
    queryid BIGINT,
    query TEXT,
    calls BIGINT,
    total_exec_time DOUBLE PRECISION,
    user_time DOUBLE PRECISION,
    system_time DOUBLE PRECISION,
    reads BIGINT,
    writes BIGINT
);

-- Créer une partition par mois (avec pg_partman)
SELECT partman.create_parent(
    p_parent_table := 'monitoring.pg_stat_kcache_history',
    p_control := 'snapshot_time',
    p_interval := 'monthly',
    p_premake := 3
);

-- Job pour archiver les stats toutes les heures
SELECT cron.schedule(
    'archive_pg_stat_kcache',
    '0 * * * *',  -- Toutes les heures
    $$
    INSERT INTO monitoring.pg_stat_kcache_history
        (queryid, query, calls, total_exec_time, user_time, system_time, reads, writes)
    SELECT
        psk.queryid,
        LEFT(pss.query, 500),
        pss.calls,
        pss.total_exec_time,
        psk.user_time,
        psk.system_time,
        psk.reads,
        psk.writes
    FROM pg_stat_statements pss
    JOIN pg_stat_kcache psk USING (queryid, userid, dbid)
    WHERE pss.calls > 10
      AND pss.total_exec_time > 100;
    $$
);
```

**Avantages de l'archivage** :
- Analyse de tendances sur le long terme
- Détection de régressions de performance
- Corrélation avec des événements (déploiements, pics de charge)

## Bonnes Pratiques

### 1. Activer pg_stat_kcache dès l'installation

✅ **Pourquoi** : Les métriques système sont invisibles sans pg_stat_kcache. Impossible de diagnostiquer finement les problèmes de performance.

✅ **Quand** : Dès le départ, en production et en développement.

```ini
# postgresql.conf (configuration de base)
shared_preload_libraries = 'pg_stat_statements,pg_stat_kcache'  
pg_stat_statements.max = 10000  
pg_stat_statements.track = all  
```

### 2. Réinitialiser les statistiques périodiquement

pg_stat_kcache accumule les statistiques depuis le dernier reset. Pour une analyse claire :

```sql
-- Réinitialiser toutes les stats (attention : perte d'historique)
SELECT pg_stat_statements_reset();

-- Note : pg_stat_kcache se réinitialise automatiquement avec pg_stat_statements
```

**Recommandation** : Archiver les stats avant de reset (voir section pg_cron ci-dessus).

### 3. Analyser les métriques par période

Les métriques cumulatives peuvent être trompeuses. Analysez par période :

```sql
-- Créer une baseline (snapshot initial)
CREATE TEMP TABLE stats_baseline AS  
SELECT * FROM pg_stat_statements;  

CREATE TEMP TABLE kcache_baseline AS  
SELECT * FROM pg_stat_kcache;  

-- Attendre un certain temps (ex: 1 heure)

-- Analyser le delta
SELECT
    pss_now.query,
    pss_now.calls - COALESCE(pss_base.calls, 0) as calls_delta,
    pss_now.total_exec_time - COALESCE(pss_base.total_exec_time, 0) as time_delta,
    psk_now.user_time - COALESCE(psk_base.user_time, 0) as cpu_delta,
    psk_now.reads - COALESCE(psk_base.reads, 0) as reads_delta
FROM pg_stat_statements pss_now  
LEFT JOIN stats_baseline pss_base USING (queryid, userid, dbid)  
JOIN pg_stat_kcache psk_now USING (queryid, userid, dbid)  
LEFT JOIN kcache_baseline psk_base USING (queryid, userid, dbid)  
WHERE pss_now.calls > COALESCE(pss_base.calls, 0)  
ORDER BY time_delta DESC  
LIMIT 20;  
```

### 4. Corréler avec EXPLAIN ANALYZE

pg_stat_kcache donne des métriques agrégées. Pour comprendre **pourquoi**, utilisez EXPLAIN :

```sql
-- 1. Identifier une requête problématique avec pg_stat_kcache
SELECT query FROM pg_stat_statements pss  
JOIN pg_stat_kcache psk USING (queryid)  
WHERE reads > 100000  
LIMIT 1;  

-- 2. Analyser en détail avec EXPLAIN
EXPLAIN (ANALYZE, BUFFERS, VERBOSE)
[copier-coller la requête identifiée];

-- 3. Chercher dans le plan :
-- - Seq Scan sur grandes tables → Ajouter index
-- - Sort avec beaucoup de mémoire → Ajuster work_mem
-- - Nested Loop sur grandes tables → Revoir les jointures
```

### 5. Définir des seuils d'alerte

Configurez des alertes sur les métriques critiques :

```sql
-- Vue pour identifier les requêtes problématiques
CREATE OR REPLACE VIEW monitoring.query_health_check AS  
SELECT  
    LEFT(query, 100) as query_snippet,
    calls,
    ROUND(total_exec_time::numeric, 2) as total_time_ms,
    ROUND((user_time + system_time)::numeric, 2) as total_cpu_s,
    reads + writes as total_io,
    majflts,
    CASE
        WHEN majflts > 100 THEN 'CRITICAL: Memory swapping'
        WHEN (reads + writes) / NULLIF(calls, 0) > 1000 THEN 'WARNING: High I/O'
        WHEN (user_time + system_time) / calls > 1.0 THEN 'WARNING: High CPU per call'
        ELSE 'OK'
    END as health_status
FROM pg_stat_statements pss  
JOIN pg_stat_kcache psk USING (queryid, userid, dbid)  
WHERE calls > 100  
  AND total_exec_time > 10000;

-- Job d'alerte avec pg_cron
SELECT cron.schedule(
    'query_health_alerts',
    '*/15 * * * *',  -- Toutes les 15 minutes
    $$
    SELECT send_alert(
        'Query Health Issues',
        string_agg(query_snippet || ': ' || health_status, E'\n')
    )
    FROM monitoring.query_health_check
    WHERE health_status != 'OK'
    $$
);
```

### 6. Documenter les requêtes critiques

```sql
-- Créer une table de documentation
CREATE TABLE monitoring.query_documentation (
    queryid BIGINT PRIMARY KEY,
    query_description TEXT,
    expected_cpu_s NUMERIC,
    expected_io BIGINT,
    owner TEXT,
    last_reviewed DATE
);

-- Documenter les requêtes importantes
INSERT INTO monitoring.query_documentation  
VALUES (  
    123456789,  -- queryid de la requête
    'Rapport de ventes quotidien - Exécuté chaque nuit à 2h',
    5.0,        -- CPU attendu
    50000,      -- I/O attendu
    'team-analytics',
    CURRENT_DATE
);

-- Détecter les régressions
SELECT
    qd.query_description,
    qd.expected_cpu_s,
    ROUND((psk.user_time + psk.system_time) / pss.calls::numeric, 2) as actual_cpu_per_call,
    CASE
        WHEN (psk.user_time + psk.system_time) / pss.calls > qd.expected_cpu_s * 1.5
        THEN 'REGRESSION'
        ELSE 'OK'
    END as status
FROM monitoring.query_documentation qd  
JOIN pg_stat_statements pss ON pss.queryid = qd.queryid  
JOIN pg_stat_kcache psk USING (queryid, userid, dbid)  
WHERE pss.calls > 0;  
```

### 7. Éviter l'overhead excessif

pg_stat_kcache a un coût (faible mais réel). Quelques recommandations :

- ✅ **Limiter pg_stat_statements.max** : 10 000 à 20 000 max (pas 100 000)  
- ✅ **Filtrer les requêtes triviales** : Analyser uniquement les requêtes > 100ms  
- ✅ **Désactiver sur les standby** : pg_stat_kcache n'est utile que sur le primaire

```sql
-- Désactiver sur standby (si pg_cron pour les stats)
CREATE OR REPLACE FUNCTION monitoring.collect_stats_if_primary()  
RETURNS void AS $$  
BEGIN  
    IF NOT pg_is_in_recovery() THEN
        -- Collecter les stats uniquement sur le primaire
        PERFORM monitoring.archive_stats();
    END IF;
END;
$$ LANGUAGE plpgsql;
```

## Dépannage (Troubleshooting)

### Problème 1 : pg_stat_kcache ne collecte aucune donnée

**Symptômes** : La vue `pg_stat_kcache` est vide ou retourne NULL pour toutes les métriques.

**Diagnostics** :

```sql
-- Vérifier que l'extension est installée
SELECT * FROM pg_extension WHERE extname = 'pg_stat_kcache';

-- Vérifier que shared_preload_libraries est correct
SHOW shared_preload_libraries;  -- Doit inclure 'pg_stat_kcache'

-- Vérifier le système d'exploitation
SELECT version();  -- pg_stat_kcache ne fonctionne que sur Linux
```

**Solutions** :
1. Si `shared_preload_libraries` ne contient pas pg_stat_kcache → Modifier postgresql.conf et redémarrer  
2. Si sur Windows/macOS → pg_stat_kcache ne fonctionnera pas (spécifique à Linux)  
3. Si sur Linux mais toujours NULL → Vérifier les permissions sur `/proc/[pid]/stat`

### Problème 2 : Métriques incohérentes (valeurs négatives, NaN)

**Symptômes** : user_time négatif ou NaN dans les résultats.

**Cause** : Reset de pg_stat_statements sans reset de pg_stat_kcache (désynchronisation).

**Solution** :

```sql
-- Réinitialiser les deux extensions ensemble
SELECT pg_stat_statements_reset();
-- pg_stat_kcache se réinitialise automatiquement avec pg_stat_statements

-- Attendre quelques requêtes, puis vérifier
SELECT * FROM pg_stat_kcache LIMIT 5;
```

### Problème 3 : Performance dégradée après activation

**Symptômes** : Légère augmentation de la latence après activation de pg_stat_kcache.

**Cause** : Overhead de la collecte de métriques système (accès à /proc).

**Diagnostic** :

```sql
-- Mesurer l'overhead (avant/après activation)
-- Sur une charge de test constante

-- Vérifier pg_stat_statements.max
SHOW pg_stat_statements.max;  -- Si > 20000, réduire
```

**Solutions** :
1. Réduire `pg_stat_statements.max` à 10 000  
2. Si l'overhead est inacceptable, désactiver pg_stat_kcache (rare)  
3. S'assurer que le serveur n'est pas déjà saturé (CPU/I/O)

### Problème 4 : Requêtes manquantes dans pg_stat_kcache

**Symptômes** : Certaines requêtes apparaissent dans pg_stat_statements mais pas dans pg_stat_kcache.

**Cause** : Les deux extensions peuvent avoir des limites différentes.

**Diagnostic** :

```sql
-- Comparer les counts
SELECT
    (SELECT COUNT(*) FROM pg_stat_statements) as stmt_count,
    (SELECT COUNT(*) FROM pg_stat_kcache) as kcache_count;
```

**Solution** : Généralement, pg_stat_kcache suit pg_stat_statements. Si désynchronisé, réinitialiser les deux.

### Problème 5 : Valeurs CPU très élevées

**Symptômes** : user_time + system_time > total_exec_time (impossible).

**Cause** : Métriques cumulatives sur plusieurs processus backend (rare).

**Diagnostic** :

```sql
-- Vérifier les requêtes individuelles
SELECT
    queryid,
    calls,
    total_exec_time / 1000.0 as total_s,
    user_time + system_time as cpu_s,
    (user_time + system_time) / (total_exec_time / 1000.0) as ratio
FROM pg_stat_statements pss  
JOIN pg_stat_kcache psk USING (queryid, userid, dbid)  
WHERE total_exec_time > 0  
ORDER BY ratio DESC;  
```

**Solution** : Si `ratio > 1.2` de manière systématique, cela peut indiquer un bug ou une configuration inhabituelle. Vérifier les versions de PostgreSQL et pg_stat_kcache.

## Limitations et Considérations

### Limitations de pg_stat_kcache

1. **Spécifique à Linux** : Ne fonctionne pas sur Windows, macOS, BSD
   - Alternative : pg_stat_statements seul (sans métriques système)

2. **Overhead léger** : ~1-5% de coût additionnel
   - Acceptable pour la plupart des workloads
   - Peut être problématique sur des systèmes déjà saturés

3. **Précision limitée** : Métriques agrégées, pas en temps réel
   - Pour du profiling détaillé, utiliser des outils système (perf, strace)

4. **Pas de métriques par connexion** : Agrégation par requête (queryid)
   - Impossible de savoir quel utilisateur/session consomme le plus

5. **Dépend de /proc** : Nécessite un accès lecture à `/proc/[pid]/stat`
   - Peut ne pas fonctionner dans certains containers (permissions restreintes)

### Limitations de l'Analyse Système

1. **Context switches** : Métriques utiles mais pas toujours actionnables
   - Dépendent fortement de la charge système globale

2. **Major page faults** : Rare sur des systèmes bien configurés
   - Si fréquent, c'est un problème critique (manque de RAM)

3. **Corrélation imparfaite** : Les métriques système ne remplacent pas EXPLAIN
   - Utilisez pg_stat_kcache comme guide, puis EXPLAIN pour comprendre

### Quand NE PAS Utiliser pg_stat_kcache

❌ **Ne pas utiliser si** :
- Système Windows/macOS (ne fonctionnera pas)
- Overhead inacceptable pour votre workload (rare)
- Pas besoin de métriques système (pg_stat_statements suffit pour beaucoup de cas)

## Alternatives et Compléments

### pg_stat_kcache vs Outils Système (top, htop, iostat)

| Critère | pg_stat_kcache | Outils Système |
|---------|----------------|----------------|
| **Granularité** | Par requête SQL | Par processus |
| **Persistance** | Stocké en base | Temps réel uniquement |
| **Accessibilité** | SQL (facile) | CLI (nécessite accès serveur) |
| **Corrélation** | Directe avec requêtes | Manuelle |

**Verdict** : pg_stat_kcache pour analyse SQL, outils système pour diagnostic temps réel.

### pg_stat_kcache vs auto_explain

**auto_explain** : Extension qui log automatiquement les plans des requêtes lentes.

```ini
# postgresql.conf
shared_preload_libraries = 'pg_stat_statements,pg_stat_kcache,auto_explain'

auto_explain.log_min_duration = 1000  # Log si > 1s  
auto_explain.log_analyze = on  
auto_explain.log_buffers = on  
```

**Complémentarité** :
- **pg_stat_kcache** : Vue d'ensemble, métriques agrégées  
- **auto_explain** : Détails spécifiques, plans d'exécution

**Workflow recommandé** :
1. Identifier les requêtes problématiques avec pg_stat_kcache  
2. Activer auto_explain pour capturer les plans de ces requêtes  
3. Optimiser selon les informations combinées

### pg_stat_kcache vs pg_stat_progress_*

PostgreSQL 18 offre des vues `pg_stat_progress_*` pour tracker les opérations longues :
- `pg_stat_progress_vacuum`  
- `pg_stat_progress_create_index`  
- `pg_stat_progress_copy`

**Complémentarité** :
- **pg_stat_kcache** : Performance des requêtes utilisateur  
- **pg_stat_progress_*** : Progression des opérations de maintenance

## Résumé

**pg_stat_kcache** est une extension PostgreSQL essentielle pour comprendre la consommation de ressources système (CPU, I/O, mémoire) au niveau des requêtes SQL.

### Points Clés

- ✅ **Métriques système détaillées** : CPU, I/O disque, page faults, context switches  
- ✅ **Complète pg_stat_statements** : Dimension système indispensable  
- ✅ **Diagnostic précis** : Identifier si une requête est CPU-bound ou I/O-bound  
- ✅ **Production-ready** : Overhead minimal, utilisé par des milliers d'entreprises  
- ✅ **Intégration facile** : Prometheus, Grafana, pgBadger

### Quand Utiliser pg_stat_kcache ?

- **Analyse de performance** : Comprendre où le temps est passé (CPU vs I/O)  
- **Optimisation** : Prioriser les requêtes à optimiser selon leur impact système  
- **Capacity planning** : Dimensionner CPU, RAM, disques selon la charge réelle  
- **Monitoring continu** : Détecter les régressions et anomalies

### Workflow Type

1. **Activer** pg_stat_kcache en production  
2. **Monitorer** avec Prometheus + Grafana (dashboards temps réel)  
3. **Analyser** régulièrement les top consumers (CPU, I/O)  
4. **Optimiser** les requêtes identifiées (index, réécriture)  
5. **Valider** l'amélioration avec les métriques

### Métriques Essentielles à Surveiller

| Métrique | Seuil | Action si dépassé |
|----------|-------|-------------------|
| `cpu_percent > 50%` | CPU-bound | Optimiser algorithme, ajouter index |
| `avg_io_per_call > 1000` | IO-bound | Ajouter index, augmenter cache |
| `majflts > 0` | Swapping | Augmenter RAM, optimiser mémoire |
| `cache_hit_ratio < 90%` | Cache miss | Augmenter shared_buffers |

### Prochaines Étapes

1. Activer pg_stat_kcache sur votre environnement de test  
2. Identifier les 10 requêtes top CPU et top I/O  
3. Analyser avec EXPLAIN ANALYZE  
4. Mettre en place un monitoring Grafana  
5. Établir des baselines et alertes

Pour tout DBA, DevOps ou développeur soucieux de performance, pg_stat_kcache est un outil incontournable.
