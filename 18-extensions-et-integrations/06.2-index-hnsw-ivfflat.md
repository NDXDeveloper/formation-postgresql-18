# 18.6.2. Index HNSW et IVFFlat

## Introduction

Dans le chapitre précédent, nous avons vu comment effectuer des recherches de similarité avec pgvector. Mais il y a un problème : sans index, PostgreSQL doit comparer votre vecteur de recherche avec **TOUS** les vecteurs de la table. C'est comme chercher un livre dans une bibliothèque en regardant tous les livres un par un !

Pour 1 million de vecteurs, une recherche sans index peut prendre **15 secondes**. Avec un bon index, la même recherche prend **0.01 seconde** — soit **1500× plus rapide** ! 🚀

Ce chapitre explique les deux types d'index principaux de pgvector : **IVFFlat** et **HNSW**.

### Analogie : Bibliothèques organisées

**Sans index (scan séquentiel)** :
```
┌─────────────────────────────────────────────────────────┐
│  Bibliothèque désorganisée                              │
│  📚📗📘📙📚📗📘📙📚📗📘📙📚📗📘📙📚📗📘📙             │
│  Vous devez vérifier chaque livre un par un             │
│  Temps : 15 minutes pour 1000 livres                    │
└─────────────────────────────────────────────────────────┘
```

**Avec index IVFFlat** :
```
┌─────────────────────────────────────────────────────────┐
│  Bibliothèque par sections                              │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐                 │
│  │ Fiction  │ │ Science  │ │ Histoire │                 │
│  │ 📚📚📚   │ │ 📗📗📗   │ │ 📘📘📘   │                │
│  └──────────┘ └──────────┘ └──────────┘                 │
│  Vous allez directement dans 1-2 sections               │
│  Temps : 30 secondes pour 1000 livres                   │
└─────────────────────────────────────────────────────────┘
```

**Avec index HNSW** :
```
┌─────────────────────────────────────────────────────────┐
│  Bibliothèque avec index hiérarchique                   │
│  Niveau 1 : Grands thèmes (5 catégories)                │
│  Niveau 2 : Sous-thèmes (50 catégories)                 │
│  Niveau 3 : Étagères précises (500 emplacements)        │
│  Navigation ultra-rapide via liens intelligents         │
│  Temps : 5 secondes pour 1000 livres                    │
└─────────────────────────────────────────────────────────┘
```

## Pourquoi les Index Vectoriels sont Différents

### Index B-Tree classiques : Inefficaces pour les vecteurs

Les index PostgreSQL classiques (B-Tree) fonctionnent bien pour :
- Nombres : `WHERE prix > 100`
- Texte : `WHERE nom = 'Dupont'`
- Dates : `WHERE date_creation > '2024-01-01'`

**Pourquoi ?** Ces données sont **ordonnables** : on peut dire que 5 < 10, "Alice" < "Bob", etc.

**Mais les vecteurs ?** Comment ordonner ces vecteurs ?
```
[0.1, 0.2, 0.3, ..., 0.9]
[0.5, 0.1, 0.8, ..., 0.2]
[0.3, 0.7, 0.2, ..., 0.6]
```

Il n'y a **pas d'ordre naturel** ! Un vecteur n'est ni "plus grand" ni "plus petit" qu'un autre.

### Recherche de proximité vs Recherche exacte

**Recherche classique (exacte)** :
```sql
SELECT * FROM clients WHERE nom = 'Dupont';
-- Retourne EXACTEMENT les Dupont
```

**Recherche vectorielle (proximité)** :
```sql
SELECT * FROM documents  
ORDER BY embedding <=> query_vector  
LIMIT 10;  
-- Retourne les 10 PLUS PROCHES (approximation acceptable)
```

### Le défi : Performance à grande échelle

| Nombre de vecteurs | Sans index | Avec index IVFFlat | Avec index HNSW |
|-------------------|------------|-------------------|-----------------|
| 1 000 | 0.01 s | 0.002 s | 0.001 s |
| 10 000 | 0.1 s | 0.005 s | 0.002 s |
| 100 000 | 1 s | 0.01 s | 0.005 s |
| 1 000 000 | 15 s | 0.05 s | 0.01 s |
| 10 000 000 | 150 s | 0.5 s | 0.05 s |

**Conclusion** : Au-delà de 10 000 vecteurs, les index sont **indispensables**.

## Index IVFFlat

### Qu'est-ce que IVFFlat ?

**IVF** = Inverted File (Fichier inversé)  
**Flat** = Vecteurs stockés tels quels (non compressés)  

**Principe** : Regrouper les vecteurs similaires en **clusters** (groupes), puis chercher uniquement dans les clusters les plus proches.

### Analogie : Villes et quartiers

Imaginez que vous cherchez une pizzeria à Paris :

**Sans index** :
- Vérifier les 15 000 restaurants de Paris un par un
- Temps : Des heures

**Avec IVFFlat** :
1. Paris est divisée en 20 quartiers (clusters)  
2. Votre position : "Je suis dans le Marais"  
3. Chercher uniquement dans le Marais + 1-2 quartiers voisins  
4. Temps : Quelques minutes

**IVFFlat fait pareil avec vos vecteurs !**

### Comment ça fonctionne

#### Étape 1 : Construction de l'index (clustering)

```
Vecteurs dans la table :
• • •    • • •    • • •
• • •    • • •    • • •
• • •    • • •    • • •

     ↓ K-means clustering ↓

Clusters formés (lists) :
┌─────────┐  ┌─────────┐  ┌─────────┐
│Cluster 1│  │Cluster 2│  │Cluster 3│
│  • • •  │  │  • • •  │  │  • • •  │
│  • • •  │  │  • • •  │  │  • • •  │
│  ★      │  │    ★    │  │      ★  │
│centroid │  │centroid │  │centroid │
└─────────┘  └─────────┘  └─────────┘
```

Chaque cluster a un **centroid** (centre) qui le représente.

#### Étape 2 : Recherche

```
1. Requête : Vecteur de recherche Q
             🔍

2. Trouver les clusters les plus proches de Q
   (comparer Q avec les centroids)

   🔍 est proche de Cluster 2 et Cluster 3

3. Scanner uniquement ces clusters

   ┌─────────┐  ┌─────────┐  ┌─────────┐
   │Cluster 1│  │Cluster 2│  │Cluster 3│
   │         │  │  ✓✓✓✓  │  │  ✓✓✓✓  │ ← Scannés
   │ (ignoré)│  │  ✓✓✓✓  │  │  ✓✓✓✓  │
   └─────────┘  └─────────┘  └─────────┘

4. Retourner les k plus proches vecteurs
```

### Création d'un index IVFFlat

#### Syntaxe SQL

```sql
CREATE INDEX nom_index ON table_name  
USING ivfflat (colonne_vector ops_class)  
WITH (lists = nombre_clusters);  
```

**Paramètre clé : `lists`**

Le nombre de clusters (listes) à créer.

**Règle générale** :
```
lists = √(nombre_de_lignes)

Exemples :
- 10 000 lignes    → lists = 100
- 100 000 lignes   → lists = 316
- 1 000 000 lignes → lists = 1000
- 10 000 000 lignes→ lists = 3162
```

#### Exemples concrets

**Pour distance L2 (euclidienne)** :

```sql
CREATE INDEX idx_produits_embedding_l2  
ON produits  
USING ivfflat (embedding vector_l2_ops)  
WITH (lists = 100);  
```

**Pour distance cosine** :

```sql
CREATE INDEX idx_documents_embedding_cosine  
ON documents  
USING ivfflat (embedding vector_cosine_ops)  
WITH (lists = 1000);  
```

**Pour inner product** :

```sql
CREATE INDEX idx_items_embedding_ip  
ON items  
USING ivfflat (embedding vector_ip_ops)  
WITH (lists = 316);  
```

### Configuration de la recherche

#### Paramètre : ivfflat.probes

**`probes`** = Nombre de clusters à scanner lors de la recherche

```sql
-- Par défaut : probes = 1 (très rapide, moins précis)
SET ivfflat.probes = 1;

-- Recommandé : probes = 10-20 (bon équilibre)
SET ivfflat.probes = 10;

-- Maximum précision : probes = lists (équivaut à scan complet)
SET ivfflat.probes = 100;
```

**Trade-off** :

| Probes | Vitesse | Précision | Cas d'usage |
|--------|---------|-----------|-------------|
| 1 | 🟢🟢🟢 Ultra-rapide | 🔴 Faible (~50%) | Exploration rapide |
| 10 | 🟢🟢 Rapide | 🟢🟢 Bon (~95%) | Production (recommandé) |
| 20 | 🟢 Moyen | 🟢🟢🟢 Très bon (~98%) | Haute précision |
| = lists | 🔴 Lent | 🟢🟢🟢 Parfait (100%) | Debug uniquement |

**Exemple** :

```sql
-- Session courante
SET ivfflat.probes = 10;

SELECT
    id,
    nom,
    embedding <=> '[0.1, 0.2, ..., 0.9]'::vector AS distance
FROM produits  
ORDER BY distance  
LIMIT 10;  
```

**Configuration permanente** (postgresql.conf) :

```ini
# postgresql.conf
ivfflat.probes = 10
```

### Avantages d'IVFFlat

✅ **Simple à comprendre** : Concept de clustering intuitif

✅ **Rapide à construire** : Index créé en quelques minutes

✅ **Économe en espace** : Overhead modéré (~20-30% de la taille des données)

✅ **Bon pour datasets moyens** : 10K - 1M vecteurs

✅ **Précision ajustable** : Paramètre `probes` pour trade-off vitesse/précision

### Limitations d'IVFFlat

❌ **Précision limitée** : ~95% de recall avec probes=10 (peut manquer les vrais plus proches)

❌ **Sensible à la distribution** : Si les vecteurs sont mal répartis, certains clusters seront surchargés

❌ **Nécessite des données pour construction** : Il faut avoir des données dans la table avant de créer l'index

❌ **Performance dégradée sur gros volumes** : Au-delà de 10M vecteurs, HNSW est meilleur

### Quand utiliser IVFFlat

✅ **Utilisez IVFFlat quand** :
- Dataset de taille moyenne (10K - 1M vecteurs)
- Contraintes d'espace disque (index plus compact que HNSW)
- Construction rapide nécessaire (prototypage, tests)
- Rappel de ~95% acceptable

❌ **N'utilisez PAS IVFFlat quand** :
- Vous avez besoin de très haute précision (>98%)
- Dataset très volumineux (>10M vecteurs)
- Performance critique (<10ms par requête)

## Index HNSW

### Qu'est-ce que HNSW ?

**HNSW** = Hierarchical Navigable Small World

**Principe** : Créer un graphe multi-niveaux où chaque vecteur est connecté à ses voisins les plus proches, permettant une navigation ultra-rapide.

### Analogie : Réseau autoroutier

**IVFFlat** = Routes départementales avec villages (clusters)
- Vous allez dans le bon village, puis cherchez la maison

**HNSW** = Réseau autoroutier multi-niveaux
- Niveau 1 : Autoroutes (connexions longue distance)
- Niveau 2 : Nationales (connexions moyenne distance)
- Niveau 3 : Départementales (connexions courte distance)
- Niveau 4 : Rues (voisinage immédiat)

Vous naviguez intelligemment de niveau en niveau pour atteindre votre destination ultra-rapidement !

### Comment ça fonctionne

#### Structure hiérarchique

```
Niveau 3 (le plus haut, peu de nœuds)
    •────────────────•
    │                │
    │                │
Niveau 2 (nœuds intermédiaires)
    •────•────•──────•────•────•
    │    │    │      │    │    │
    │    │    │      │    │    │
Niveau 1 (beaucoup de nœuds)
    •──•──•──•──•──•──•──•──•──•──•──•
    │  │  │  │  │  │  │  │  │  │  │  │
    │  │  │  │  │  │  │  │  │  │  │  │
Niveau 0 (tous les vecteurs)
    •••••••••••••••••••••••••••••••••••

Chaque • est un vecteur  
Les liens ── représentent les connexions vers voisins proches  
```

#### Processus de recherche

```
1. Point de départ : Nœud d'entrée au niveau le plus haut

   Niveau 3:  🔍 START
                •

2. Descendre progressivement en suivant les voisins les plus proches

   Niveau 2:    •
              / | \
             •  •  •
                🔍 Descend ici

3. Niveau 1: Navigation plus fine

   Niveau 1:  • • • •
              • •🔍• •
              • • • •

4. Niveau 0: Trouver les k plus proches dans le voisinage final

   Niveau 0:  •••
              •🎯• ← Résultat !
              •••
```

**Complexité** : O(log N) au lieu de O(N) pour un scan complet

### Création d'un index HNSW

#### Syntaxe SQL

```sql
CREATE INDEX nom_index ON table_name  
USING hnsw (colonne_vector ops_class)  
WITH (m = connections, ef_construction = build_param);  
```

**Paramètres clés** :

1. **`m`** : Nombre de connexions par nœud (par niveau)  
2. **`ef_construction`** : Taille de la file de priorité lors de la construction

#### Exemples concrets

**Configuration standard (recommandée)** :

```sql
CREATE INDEX idx_documents_embedding_hnsw  
ON documents  
USING hnsw (embedding vector_cosine_ops)  
WITH (m = 16, ef_construction = 64);  
```

**Configuration haute performance** :

```sql
CREATE INDEX idx_produits_embedding_hnsw  
ON produits  
USING hnsw (embedding vector_l2_ops)  
WITH (m = 32, ef_construction = 128);  
```

**Configuration économe en espace** :

```sql
CREATE INDEX idx_items_embedding_hnsw  
ON items  
USING hnsw (embedding vector_cosine_ops)  
WITH (m = 8, ef_construction = 32);  
```

### Paramètres de l'index

#### Paramètre `m` (connexions)

**Définition** : Nombre maximal de connexions bidirectionnelles par nœud (par couche).

**Impact** :

| Valeur m | Taille index | Vitesse recherche | Précision | Recommandation |
|----------|-------------|------------------|-----------|----------------|
| 8 | Petit | Rapide | Bonne | Datasets compacts |
| 16 | Moyen | Très rapide | Très bonne | **Défaut (recommandé)** |
| 32 | Grand | Ultra-rapide | Excellente | Haute précision |
| 64 | Très grand | Ultra-rapide | Parfaite | Cas critiques |

**Règle générale** :
- Commencez avec `m = 16`
- Augmentez à `m = 32` si vous avez besoin de plus de précision
- Diminuez à `m = 8` si l'espace disque est limité

**Formule espace disque** :
```
Taille index ≈ taille_vecteurs × m × 1.5
```

#### Paramètre `ef_construction` (construction)

**Définition** : Taille de la file de candidats explorés lors de l'insertion d'un nouveau vecteur.

**Impact** :

| Valeur ef_construction | Temps construction | Qualité index | Recommandation |
|------------------------|-------------------|---------------|----------------|
| 32 | Rapide | Acceptable | Prototypage |
| 64 | Moyen | Bonne | **Défaut** |
| 128 | Lent | Très bonne | Production |
| 256 | Très lent | Excellente | Données critiques |

**Règle générale** :
- `ef_construction >= 2 × m` (minimum)
- Standard : `ef_construction = 64` pour `m = 16`
- Haute qualité : `ef_construction = 128` ou plus

**Trade-off** :
```
ef_construction élevé → Construction plus lente, mais index de meilleure qualité  
ef_construction faible → Construction rapide, mais index moins optimal  
```

### Configuration de la recherche

#### Paramètre : hnsw.ef_search

**`ef_search`** = Taille de la liste de candidats explorés lors d'une recherche

```sql
-- Par défaut : ef_search = 40
SET hnsw.ef_search = 40;

-- Haute précision
SET hnsw.ef_search = 100;

-- Vitesse maximale (précision réduite)
SET hnsw.ef_search = 20;
```

**Impact** :

| ef_search | Vitesse | Précision | Cas d'usage |
|-----------|---------|-----------|-------------|
| 20 | 🟢🟢🟢 Très rapide | 🟡 Moyenne (~90%) | Exploration rapide |
| 40 | 🟢🟢 Rapide | 🟢🟢 Bonne (~98%) | **Production (défaut)** |
| 100 | 🟢 Moyen | 🟢🟢🟢 Excellente (~99.5%) | Haute précision |
| 200 | 🟡 Lent | 🟢🟢🟢 Parfaite (~99.9%) | Validation |

**Règle générale** :
```
ef_search >= k (nombre de résultats demandés)

Si LIMIT 10 → ef_search >= 10 (mais 40 recommandé)  
Si LIMIT 100 → ef_search >= 100  
```

**Exemple** :

```sql
-- Recherche haute précision
SET hnsw.ef_search = 100;

SELECT
    id,
    titre,
    embedding <=> '[0.1, 0.2, ..., 0.9]'::vector AS distance
FROM documents  
ORDER BY distance  
LIMIT 10;  
```

### Avantages de HNSW

✅ **Performance exceptionnelle** : Le plus rapide des index vectoriels (~10-50× plus rapide qu'IVFFlat)

✅ **Haute précision** : Recall de ~98-99% par défaut

✅ **Scalabilité** : Fonctionne bien même avec 100M+ vecteurs

✅ **Robuste** : Peu sensible à la distribution des données

✅ **Pas besoin de données d'entraînement** : Peut être créé sur table vide (contrairement à IVFFlat)

✅ **Inserts efficaces** : Ajout de nouveaux vecteurs rapide

### Limitations de HNSW

❌ **Espace disque important** : Index peut être 2-3× la taille des données

❌ **Construction lente** : Peut prendre des heures pour 10M+ vecteurs

❌ **RAM intensive** : Nécessite beaucoup de mémoire pour performances optimales

❌ **Paramètres complexes** : Nécessite tuning (m, ef_construction, ef_search)

### Quand utiliser HNSW

✅ **Utilisez HNSW quand** :
- Dataset volumineux (>1M vecteurs)
- Performance critique (<10ms par requête)
- Haute précision nécessaire (>98% recall)
- Espace disque disponible
- Production à grande échelle

❌ **N'utilisez PAS HNSW quand** :
- Dataset très petit (<10K vecteurs) → Scan séquentiel suffit
- Contraintes d'espace disque sévères → Utilisez IVFFlat
- Prototypage rapide nécessaire → IVFFlat plus rapide à construire

## Configuration et Tuning

### Dimensionnement des ressources

#### RAM nécessaire

**IVFFlat** :
```
RAM minimale = taille_données × 1.5  
RAM recommandée = taille_données × 2  
```

**HNSW** :
```
RAM minimale = taille_données × 2  
RAM recommandée = taille_données × 3  
```

**Exemple** :
- 1M vecteurs de 1536 dimensions = ~6 GB de données
- IVFFlat : 9 GB RAM recommandée
- HNSW : 18 GB RAM recommandée

#### Espace disque

**IVFFlat** :
```
Espace index = taille_données × 0.3
```

**HNSW** :
```
Espace index = taille_données × (1 + m/10)

m=8  → +80% de la taille des données  
m=16 → +160% de la taille des données  
m=32 → +320% de la taille des données  
```

**Exemple** :
- 1M vecteurs de 1536 dimensions = ~6 GB
- IVFFlat : ~1.8 GB index
- HNSW (m=16) : ~9.6 GB index

### Optimisation IVFFlat

#### Choix du nombre de lists

**Formule de base** :
```
lists = √(nombre_de_lignes)
```

**Affinement** :

```sql
-- Pour datasets très grands (>1M)
lists = nombre_de_lignes / 1000

-- Exemple : 10M vecteurs
lists = 10000000 / 1000 = 10000
```

**Test empirique** :

```sql
-- Tester différentes valeurs
CREATE INDEX idx_test_100 ON produits  
USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);  

CREATE INDEX idx_test_316 ON produits  
USING ivfflat (embedding vector_cosine_ops) WITH (lists = 316);  

CREATE INDEX idx_test_1000 ON produits  
USING ivfflat (embedding vector_cosine_ops) WITH (lists = 1000);  

-- Comparer les performances
EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM produits  
ORDER BY embedding <=> '[...]'::vector  
LIMIT 10;  
```

#### Ajustement de probes

**Stratégie progressive** :

```sql
-- Démarrer avec probes faible
SET ivfflat.probes = 1;
-- Mesurer recall et temps

-- Augmenter progressivement
SET ivfflat.probes = 5;  
SET ivfflat.probes = 10;  -- Bon compromis  
SET ivfflat.probes = 20;  -- Haute précision  

-- Trouver le sweet spot pour votre cas d'usage
```

**Règle pratique** :
```
probes = max(10, lists / 100)

Exemples :
- lists = 100  → probes = 10
- lists = 1000 → probes = 10
- lists = 10000 → probes = 100
```

### Optimisation HNSW

#### Choix de m

**Par dataset** :

| Taille dataset | m recommandé | Justification |
|---------------|-------------|---------------|
| < 100K | 8 | Économie d'espace |
| 100K - 1M | 16 | **Défaut optimal** |
| 1M - 10M | 16-24 | Bon équilibre |
| > 10M | 24-32 | Haute précision |

**Par cas d'usage** :

```sql
-- Prototypage / Dev
WITH (m = 8, ef_construction = 32)

-- Production standard
WITH (m = 16, ef_construction = 64)  -- Recommandé

-- Production haute performance
WITH (m = 24, ef_construction = 96)

-- Critique / Financier
WITH (m = 32, ef_construction = 128)
```

#### Choix de ef_construction

**Règle** :
```
ef_construction = 4 × m (minimum)  
ef_construction = 8 × m (recommandé)  

Exemples :
- m=8  → ef_construction = 64
- m=16 → ef_construction = 128
- m=32 → ef_construction = 256
```

#### Choix de ef_search

**Par latence cible** :

```sql
-- < 5ms par requête (vitesse max)
SET hnsw.ef_search = 20;

-- < 10ms (production standard)
SET hnsw.ef_search = 40;  -- Défaut recommandé

-- < 50ms (haute précision)
SET hnsw.ef_search = 100;

-- Précision maximale (peu importe le temps)
SET hnsw.ef_search = 200;
```

## Exemples Pratiques Complets

### Exemple 1 : Startup avec 50K documents

**Contexte** : Base de connaissances avec recherche sémantique

```sql
-- Créer la table
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    titre TEXT,
    contenu TEXT,
    embedding vector(1536)
);

-- Insérer 50 000 documents (via application)
-- ...

-- Choix : IVFFlat (dataset moyen, économie d'espace)
CREATE INDEX idx_documents_embedding  
ON documents  
USING ivfflat (embedding vector_cosine_ops)  
WITH (lists = 224);  -- √50000 ≈ 224  

-- Configuration de recherche
SET ivfflat.probes = 10;

-- Recherche
SELECT
    titre,
    LEFT(contenu, 100) AS apercu,
    embedding <=> '[...]'::vector(1536) AS distance
FROM documents  
ORDER BY distance  
LIMIT 5;  

-- Résultat : ~20ms par requête, recall ~95%
```

### Exemple 2 : E-commerce avec 500K produits

**Contexte** : Recommandation de produits similaires

```sql
-- Créer la table
CREATE TABLE produits (
    id SERIAL PRIMARY KEY,
    nom TEXT,
    description TEXT,
    categorie TEXT,
    prix NUMERIC,
    embedding vector(768)  -- Sentence-BERT
);

-- Insérer 500 000 produits
-- ...

-- Choix : HNSW (haute performance souhaitée)
CREATE INDEX idx_produits_embedding  
ON produits  
USING hnsw (embedding vector_cosine_ops)  
WITH (m = 16, ef_construction = 64);  

-- Construction : ~30 minutes

-- Configuration de recherche
SET hnsw.ef_search = 40;

-- Recherche de produits similaires
SELECT
    p2.id,
    p2.nom,
    p2.prix,
    p1.embedding <=> p2.embedding AS distance
FROM produits p1  
CROSS JOIN produits p2  
WHERE p1.id = 12345  -- Produit de référence  
  AND p2.id != 12345
ORDER BY distance  
LIMIT 10;  

-- Résultat : ~5ms par requête, recall ~98%
```

### Exemple 3 : Archives avec 10M de documents

**Contexte** : Recherche dans archives historiques

```sql
-- Créer la table
CREATE TABLE archives (
    id SERIAL PRIMARY KEY,
    annee INTEGER,
    type_document TEXT,
    texte TEXT,
    embedding vector(1536)
);

-- Insérer 10 millions de documents
-- ...

-- Choix : HNSW haute performance
CREATE INDEX idx_archives_embedding  
ON archives  
USING hnsw (embedding vector_cosine_ops)  
WITH (m = 24, ef_construction = 96);  

-- Construction : ~6 heures (parallélisation recommandée)
SET max_parallel_maintenance_workers = 4;

-- Configuration de recherche optimisée
SET hnsw.ef_search = 60;

-- Recherche avec filtres
SELECT
    id,
    annee,
    type_document,
    LEFT(texte, 150) AS extrait,
    embedding <=> '[...]'::vector(1536) AS distance
FROM archives  
WHERE annee BETWEEN 1990 AND 2000  -- Pre-filtre PostgreSQL  
ORDER BY distance  
LIMIT 20;  

-- Résultat : ~15ms par requête, recall ~98.5%
```

### Exemple 4 : Migration IVFFlat → HNSW

**Contexte** : Croissance du dataset nécessite upgrade

```sql
-- Situation initiale : 100K vecteurs avec IVFFlat
-- Le dataset a grandi à 2M vecteurs → performance dégradée

-- Étape 1 : Créer nouvel index HNSW en parallèle
CREATE INDEX CONCURRENTLY idx_documents_hnsw  
ON documents  
USING hnsw (embedding vector_cosine_ops)  
WITH (m = 16, ef_construction = 64);  

-- Construction : ~1h30 (CONCURRENTLY n'impacte pas les requêtes)

-- Étape 2 : Tester le nouvel index
SET enable_seqscan = off;  -- Forcer utilisation index  
SET hnsw.ef_search = 40;  

EXPLAIN (ANALYZE)  
SELECT * FROM documents  
ORDER BY embedding <=> '[...]'::vector  
LIMIT 10;  

-- Vérifier : "Index Scan using idx_documents_hnsw"

-- Étape 3 : Si OK, supprimer ancien index IVFFlat
DROP INDEX idx_documents_ivfflat;

-- Étape 4 : Configurer par défaut
ALTER DATABASE ma_base SET hnsw.ef_search = 40;
```

## Maintenance des Index

### Monitoring de la santé

#### Taille des index

```sql
SELECT
    schemaname,
    tablename,
    indexname,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
    pg_size_pretty(pg_table_size(schemaname||'.'||tablename)) AS table_size
FROM pg_stat_user_indexes  
WHERE indexname LIKE '%embedding%'  
ORDER BY pg_relation_size(indexrelid) DESC;  
```

**Résultat exemple** :

```
 schemaname | tablename  |       indexname        | index_size | table_size
------------+------------+------------------------+------------+------------
 public     | documents  | idx_documents_hnsw     | 9.2 GB     | 6.1 GB
 public     | produits   | idx_produits_ivfflat   | 1.8 GB     | 5.9 GB
```

#### Utilisation des index

```sql
SELECT
    schemaname,
    tablename,
    indexname,
    idx_scan AS nombre_utilisations,
    idx_tup_read AS lignes_lues,
    idx_tup_fetch AS lignes_recuperees
FROM pg_stat_user_indexes  
WHERE indexname LIKE '%embedding%'  
ORDER BY idx_scan DESC;  
```

### Reconstruction d'index

#### Quand reconstruire ?

**IVFFlat** :
- Après insertion/suppression de >20% des données
- Changement de distribution des données
- Dégradation des performances constatée

**HNSW** :
- Rarement nécessaire (auto-maintenance)
- Après insertion/suppression massive (>50% des données)
- Changement radical de distribution

#### Comment reconstruire

```sql
-- Reconstruction simple (bloque la table)
REINDEX INDEX idx_documents_embedding;

-- Reconstruction sans bloquer les requêtes
REINDEX INDEX CONCURRENTLY idx_documents_embedding;

-- Reconstruire tous les index d'une table
REINDEX TABLE CONCURRENTLY documents;
```

### Vacuum et maintenance

```sql
-- Vacuum régulier (recommandé quotidien/hebdomadaire)
VACUUM ANALYZE documents;

-- Si beaucoup de suppressions, vacuum full
VACUUM FULL ANALYZE documents;
-- Attention : bloque la table, long !

-- Statistiques à jour pour le planificateur
ANALYZE documents;
```

## Troubleshooting

### Problème 1 : Index non utilisé

**Symptôme** : EXPLAIN montre "Seq Scan" au lieu de "Index Scan"

**Diagnostic** :

```sql
EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM documents  
ORDER BY embedding <=> '[...]'::vector  
LIMIT 10;  
```

**Causes possibles** :

1. **Index pas créé ou invalide**

```sql
-- Vérifier les index
SELECT indexname, indexdef  
FROM pg_indexes  
WHERE tablename = 'documents';  
```

2. **Mauvais operator class**

```sql
-- ❌ Mauvais
CREATE INDEX ... USING hnsw (embedding vector_l2_ops);  -- Pour L2
-- Mais requête utilise cosine (<=>)

-- ✅ Bon
CREATE INDEX ... USING hnsw (embedding vector_cosine_ops);  -- Pour cosine
```

3. **Paramètres de recherche non définis**

```sql
-- IVFFlat
SET ivfflat.probes = 10;

-- HNSW
SET hnsw.ef_search = 40;
```

4. **Sequential scan forcé**

```sql
-- Désactiver temporairement
SET enable_seqscan = off;
```

### Problème 2 : Construction d'index échoue

**Symptôme** : ERROR lors de CREATE INDEX

**Erreur : Out of memory**

```sql
-- Solution : Augmenter work_mem temporairement
SET maintenance_work_mem = '4GB';

CREATE INDEX ...;

-- Réinitialiser
RESET maintenance_work_mem;
```

**Erreur : Cannot create IVFFlat index (not enough data)**

```sql
-- IVFFlat nécessite des données pour clustering
-- Solution : Insérer des données d'abord

-- Ou utiliser HNSW qui n'a pas cette limitation
CREATE INDEX ... USING hnsw ...;
```

### Problème 3 : Performances dégradées

**Symptôme** : Recherches de plus en plus lentes

**Diagnostic** :

```sql
-- Vérifier le bloat de la table
SELECT
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS total_size,
    n_live_tup AS lignes_vivantes,
    n_dead_tup AS lignes_mortes,
    round(100.0 * n_dead_tup / nullif(n_live_tup + n_dead_tup, 0), 1) AS pct_dead
FROM pg_stat_user_tables  
WHERE tablename = 'documents';  
```

**Solutions** :

```sql
-- Si pct_dead > 10%
VACUUM ANALYZE documents;

-- Si pct_dead > 30%
REINDEX INDEX CONCURRENTLY idx_documents_embedding;
```

### Problème 4 : Précision insuffisante

**Symptôme** : Résultats ne semblent pas pertinents

**Solutions IVFFlat** :

```sql
-- Augmenter probes
SET ivfflat.probes = 20;  -- Au lieu de 10  
SET ivfflat.probes = 50;  -- Encore plus  

-- Ou recréer avec plus de lists
DROP INDEX idx_documents_embedding;  
CREATE INDEX idx_documents_embedding  
ON documents  
USING ivfflat (embedding vector_cosine_ops)  
WITH (lists = 500);  -- Au lieu de 100  
```

**Solutions HNSW** :

```sql
-- Augmenter ef_search
SET hnsw.ef_search = 100;  -- Au lieu de 40  
SET hnsw.ef_search = 200;  -- Encore plus  

-- Ou recréer avec m plus élevé
DROP INDEX idx_documents_embedding;  
CREATE INDEX idx_documents_embedding  
ON documents  
USING hnsw (embedding vector_cosine_ops)  
WITH (m = 32, ef_construction = 128);  -- Au lieu de m=16  
```

## Checklist de Production

### Avant déploiement

✅ **Dimensionnement** :  
- [ ] RAM suffisante (2-3× la taille des données pour HNSW)  
- [ ] Espace disque (2-3× pour index HNSW)  
- [ ] CPU multi-core pour construction parallèle

✅ **Configuration** :  
- [ ] `shared_buffers` ≥ 25% de la RAM  
- [ ] `effective_cache_size` ≥ 50% de la RAM  
- [ ] `maintenance_work_mem` ≥ 2GB pour construction  
- [ ] `max_parallel_maintenance_workers` = 4-8

✅ **Choix d'index** :  
- [ ] IVFFlat pour <1M vecteurs OU contraintes d'espace  
- [ ] HNSW pour >1M vecteurs OU haute performance

✅ **Paramètres** :  
- [ ] IVFFlat : `lists = √N`, `probes = 10`  
- [ ] HNSW : `m = 16`, `ef_construction = 64`, `ef_search = 40`

### Monitoring continu

✅ **Métriques à surveiller** :  
- [ ] Temps de réponse moyen des recherches  
- [ ] Taux d'utilisation des index (pg_stat_user_indexes)  
- [ ] Taille des index (croissance)  
- [ ] Bloat des tables/index (dead tuples)

✅ **Alertes** :  
- [ ] Temps de recherche > 100ms  
- [ ] Dead tuples > 10%  
- [ ] Index > 200% de la taille des données

### Maintenance régulière

✅ **Quotidien** :  
- [ ] VACUUM ANALYZE automatique (autovacuum activé)

✅ **Hebdomadaire** :  
- [ ] Vérifier les statistiques de performance  
- [ ] Analyser les requêtes lentes (pg_stat_statements)

✅ **Mensuel** :  
- [ ] Vérifier la croissance des index  
- [ ] Tester la qualité des résultats (recall)  
- [ ] Ajuster les paramètres si nécessaire

✅ **Après changements majeurs** :  
- [ ] REINDEX si >20% de données changées (IVFFlat)  
- [ ] ANALYZE après insertions massives  
- [ ] Benchmark des performances

## Conclusion

### Points clés à retenir

✅ **Les index vectoriels sont essentiels** pour la performance au-delà de 10K vecteurs

✅ **IVFFlat** : Simple, rapide à construire, bon pour datasets moyens
- Clustering des vecteurs similaires
- `lists = √N`, `probes = 10` (défaut)  
- ~95% recall, très rapide

✅ **HNSW** : Ultra-performant, précis, idéal pour production
- Graphe multi-niveaux navigable
- `m = 16`, `ef_construction = 64`, `ef_search = 40`  
- ~98-99% recall, ultra-rapide

✅ **Choix de l'index** :  
- < 100K vecteurs : IVFFlat ou pas d'index
- 100K - 1M : IVFFlat (économie) ou HNSW (performance)
- > 1M : HNSW obligatoire

⚠️ **Trade-offs** : Vitesse ↔ Précision ↔ Espace

⚠️ **Tuning essentiel** : Les paramètres par défaut ne sont pas toujours optimaux

🎯 Avec un bon index, recherche de vecteurs = **1500× plus rapide** ! 🚀

### Recommandations finales

**Pour débutants** :
1. Commencez avec HNSW defaults (`m=16, ef_construction=64`)  
2. Ne sur-optimisez pas dès le début  
3. Mesurez avant d'ajuster

**Pour production** :
1. HNSW pour >100K vecteurs  
2. Dimensionner la RAM (3× les données)  
3. Monitorer et ajuster itérativement

**Pour datasets massifs** :
1. HNSW obligatoire  
2. Construction parallèle (`max_parallel_maintenance_workers`)  
3. Partitionnement si >100M vecteurs

### Prochaines étapes

Pour aller plus loin :
1. **Benchmarker** : Tester les deux index sur vos données réelles  
2. **Tuner** : Ajuster les paramètres selon vos métriques  
3. **Monitorer** : Mettre en place monitoring continu  
4. **Optimiser** : Itérer sur la configuration
