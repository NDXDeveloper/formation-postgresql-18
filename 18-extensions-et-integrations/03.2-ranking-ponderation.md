# 18.3.2. Ranking et Pondération : ts_rank et setweight

## Introduction

Imaginez que vous recherchez "base de données PostgreSQL" dans un blog technique. Vous obtenez 500 résultats. Mais comment les trier ? Lequel est le plus pertinent ? Celui qui mentionne PostgreSQL une fois en passant, ou celui qui en parle en profondeur ?

C'est là qu'interviennent le **ranking** (classement par pertinence) et la **pondération** (attribution d'importance aux différentes parties du document).

**Le problème sans ranking** :
```sql
-- Tous les articles qui mentionnent "postgresql"
SELECT titre FROM articles  
WHERE search_vector @@ to_tsquery('french', 'postgresql');  
-- Résultats : pas d'ordre logique, ordre aléatoire ou par ID
```

**La solution avec ranking** :
```sql
-- Articles triés par pertinence
SELECT titre, ts_rank(search_vector, query) AS score  
FROM articles, to_tsquery('french', 'postgresql') query  
WHERE search_vector @@ query  
ORDER BY score DESC;  
-- Résultats : les plus pertinents en premier !
```

Dans ce chapitre, nous allons explorer :
- **ts_rank** : Calcule un score de pertinence  
- **setweight** : Attribue de l'importance aux parties d'un document  
- **Normalisation** : Ajuste les scores pour éviter les biais  
- **Optimisations** : Combine ranking et performance

---

## 1. Qu'est-ce que le Ranking ?

### Le Concept de Pertinence

Le **ranking** (ou classement) consiste à attribuer un **score de pertinence** à chaque document qui correspond à une requête. Plus le score est élevé, plus le document est considéré comme pertinent.

**Facteurs de pertinence** :
1. **Fréquence** : Combien de fois le terme recherché apparaît  
2. **Position** : Où le terme apparaît (titre > contenu)  
3. **Rareté** : Les termes rares sont plus significatifs  
4. **Proximité** : Les termes proches sont plus pertinents

### Exemple Visuel

Recherche : "postgresql"

```
Document A : "PostgreSQL"  
Score : 0.06 (1 occurrence)  

Document B : "PostgreSQL est puissant. PostgreSQL est open-source."  
Score : 0.12 (2 occurrences)  

Document C : "Le guide complet de PostgreSQL. PostgreSQL pour débutants.
              Maîtriser PostgreSQL en profondeur."
Score : 0.30 (3 occurrences + densité élevée)
```

**Ordre de pertinence** : C > B > A

## 2. La Fonction ts_rank

### Signature et Usage de Base

```sql
ts_rank(tsvector, tsquery) → real  
ts_rank(weights, tsvector, tsquery) → real  
ts_rank(weights, tsvector, tsquery, normalization) → real  
```

**Paramètres** :
- `tsvector` : Le document à évaluer  
- `tsquery` : La requête de recherche  
- `weights` : (Optionnel) Tableau de poids par catégorie [D, C, B, A]  
- `normalization` : (Optionnel) Méthode de normalisation (0-32)

### Exemple Simple

```sql
-- Score de base pour un document
SELECT ts_rank(
    to_tsvector('french', 'PostgreSQL est un SGBD puissant'),
    to_tsquery('french', 'postgresql')
) AS score;
-- Résultat : ~0.0607927 (score relatif)
```

### Application Pratique : Trier les Résultats

```sql
-- Table d'articles
CREATE TABLE articles (
    id SERIAL PRIMARY KEY,
    titre VARCHAR(200),
    contenu TEXT,
    search_vector tsvector
);

-- Données d'exemple
INSERT INTO articles (titre, contenu) VALUES
    ('Introduction PostgreSQL', 'PostgreSQL est un SGBD relationnel.'),
    ('Guide avancé PostgreSQL', 'PostgreSQL offre de nombreuses fonctionnalités. PostgreSQL est extensible. PostgreSQL supporte JSON.'),
    ('Bases de données', 'Les bases de données stockent des informations. MySQL et PostgreSQL sont populaires.');

-- Mise à jour du search_vector
UPDATE articles SET search_vector =
    setweight(to_tsvector('french', titre), 'A') ||
    setweight(to_tsvector('french', contenu), 'B');

-- Recherche avec ranking
SELECT
    titre,
    ts_rank(search_vector, query) AS score
FROM
    articles,
    to_tsquery('french', 'postgresql') query
WHERE search_vector @@ query  
ORDER BY score DESC;  
```

**Résultat** :
```
titre                      | score
---------------------------+----------
Guide avancé PostgreSQL    | 0.303964  ← Plus pertinent (3 mentions + titre)  
Introduction PostgreSQL    | 0.151982  ← Moyennement pertinent (titre + 1 mention)  
Bases de données          | 0.075991  ← Moins pertinent (1 mention parmi d'autres)  
```

### Comment ts_rank Calcule le Score

Le score dépend de plusieurs facteurs :

#### 1. Fréquence des Termes

Plus un terme apparaît souvent, plus le score augmente.

```sql
-- Document avec 1 occurrence
SELECT ts_rank(
    to_tsvector('french', 'PostgreSQL est puissant'),
    to_tsquery('french', 'postgresql')
) AS score_1;
-- Résultat : ~0.06

-- Document avec 3 occurrences
SELECT ts_rank(
    to_tsvector('french', 'PostgreSQL est puissant. PostgreSQL est rapide. PostgreSQL est fiable.'),
    to_tsquery('french', 'postgresql')
) AS score_3;
-- Résultat : ~0.30

-- Score × 5 car 5× plus de mentions
```

#### 2. Nombre Total de Mots (Longueur du Document)

Un document plus long dilue la pertinence.

```sql
-- Document court avec 1 mention
SELECT ts_rank(
    to_tsvector('french', 'PostgreSQL'),
    to_tsquery('french', 'postgresql')
) AS score_court;
-- Résultat : ~0.06

-- Document long avec 1 mention
SELECT ts_rank(
    to_tsvector('french', 'PostgreSQL est un système de gestion de base de données relationnel et objet très puissant et performant.'),
    to_tsquery('french', 'postgresql')
) AS score_long;
-- Résultat : ~0.02 (dilué par les autres mots)
```

#### 3. Poids des Lexèmes (A, B, C, D)

Les lexèmes avec un poids élevé comptent plus.

```sql
-- Sans poids (tous D par défaut)
SELECT ts_rank(
    to_tsvector('french', 'PostgreSQL est puissant'),
    to_tsquery('french', 'postgresql')
) AS score_sans_poids;
-- Résultat : ~0.06

-- Avec poids A (titre)
SELECT ts_rank(
    setweight(to_tsvector('french', 'PostgreSQL'), 'A'),
    to_tsquery('french', 'postgresql')
) AS score_avec_poids_A;
-- Résultat : ~0.60 (10× plus élevé !)
```

## 3. Les Poids : setweight

### Qu'est-ce qu'un Poids ?

Un **poids** est une étiquette (A, B, C, ou D) assignée à un lexème pour indiquer son importance dans le document.

**Hiérarchie standard** :
- **A** : Titre, nom (le plus important) - Poids : 1.0  
- **B** : Sous-titre, résumé, description - Poids : 0.4  
- **C** : Contenu principal - Poids : 0.2  
- **D** : Métadonnées, contenu secondaire (défaut) - Poids : 0.1

### Utiliser setweight

```sql
-- Syntaxe
setweight(tsvector, weight) → tsvector
```

**Exemples** :

```sql
-- Assigner le poids A (titre)
SELECT setweight(to_tsvector('french', 'PostgreSQL'), 'A');
-- Résultat : 'postgresql':1A

-- Assigner le poids B (description)
SELECT setweight(to_tsvector('french', 'Base de données performante'), 'B');
-- Résultat : 'base':1B 'donn':3B 'perform':4B

-- Combiner plusieurs poids
SELECT
    setweight(to_tsvector('french', 'PostgreSQL'), 'A') ||
    setweight(to_tsvector('french', 'SGBD puissant'), 'B') ||
    setweight(to_tsvector('french', 'Open source et gratuit'), 'C');
-- Résultat : 'gratuit':9C 'open':7C 'postgresql':1A 'puiss':3B 'sgbd':2B 'sourc':8C
```

### Impact du Poids sur le Score

```sql
-- Sans poids (tous D par défaut)
SELECT ts_rank(
    to_tsvector('french', 'PostgreSQL'),
    to_tsquery('french', 'postgresql')
) AS score_D;
-- Résultat : ~0.06

-- Avec poids C
SELECT ts_rank(
    setweight(to_tsvector('french', 'PostgreSQL'), 'C'),
    to_tsquery('french', 'postgresql')
) AS score_C;
-- Résultat : ~0.12 (2× plus élevé)

-- Avec poids B
SELECT ts_rank(
    setweight(to_tsvector('french', 'PostgreSQL'), 'B'),
    to_tsquery('french', 'postgresql')
) AS score_B;
-- Résultat : ~0.24 (4× plus élevé)

-- Avec poids A
SELECT ts_rank(
    setweight(to_tsvector('french', 'PostgreSQL'), 'A'),
    to_tsquery('french', 'postgresql')
) AS score_A;
-- Résultat : ~0.60 (10× plus élevé)
```

**Ratio de poids par défaut** : A:B:C:D = 1.0:0.4:0.2:0.1

### Cas Pratique : Article de Blog avec Poids

```sql
-- Structure réaliste d'un article
WITH article AS (
    SELECT
        'Guide PostgreSQL' AS titre,
        'Apprenez PostgreSQL facilement' AS description,
        'PostgreSQL est un SGBD open-source. PostgreSQL offre de nombreuses fonctionnalités.' AS contenu,
        'postgresql, base de données, tutoriel' AS tags
)
SELECT
    setweight(to_tsvector('french', titre), 'A') ||           -- Titre : poids A
    setweight(to_tsvector('french', description), 'B') ||     -- Description : poids B
    setweight(to_tsvector('french', contenu), 'C') ||         -- Contenu : poids C
    setweight(to_tsvector('french', tags), 'D') AS search_vector  -- Tags : poids D
FROM article;

-- Résultat :
-- 'base':13D 'donn':15D 'facil':4B 'fonctionnalite':14C 'guid':1A
-- 'nombr':12C 'offr':10C 'open':9C 'open-sourc':9C 'postgresql':2A,3B,7C,8C,13D
-- 'sgbd':8C 'sourc':9C 'tutori':16D
```

### Stratégie de Pondération Typique

```sql
-- Pattern recommandé pour un système de contenu
CREATE OR REPLACE FUNCTION create_search_vector(
    titre TEXT,
    description TEXT,
    contenu TEXT,
    tags TEXT
) RETURNS tsvector AS $$
BEGIN
    RETURN
        setweight(to_tsvector('french', COALESCE(titre, '')), 'A') ||
        setweight(to_tsvector('french', COALESCE(description, '')), 'B') ||
        setweight(to_tsvector('french', COALESCE(contenu, '')), 'C') ||
        setweight(to_tsvector('french', COALESCE(tags, '')), 'D');
END;
$$ LANGUAGE plpgsql IMMUTABLE;

-- Utilisation
UPDATE articles  
SET search_vector = create_search_vector(titre, description, contenu, tags);  
```

## 4. Personnaliser les Poids avec ts_rank

### Tableau de Poids Custom

Par défaut, ts_rank utilise les poids `{0.1, 0.2, 0.4, 1.0}` pour `{D, C, B, A}`.

Vous pouvez les personnaliser :

```sql
-- Syntaxe avec poids personnalisés
ts_rank(ARRAY[weight_D, weight_C, weight_B, weight_A], tsvector, tsquery)
```

#### Exemple : Augmenter l'Importance du Titre

```sql
-- Poids par défaut : {0.1, 0.2, 0.4, 1.0}
SELECT ts_rank(
    setweight(to_tsvector('french', 'PostgreSQL'), 'A'),
    to_tsquery('french', 'postgresql')
) AS score_defaut;
-- Résultat : ~0.60

-- Poids custom : titre 3× plus important
SELECT ts_rank(
    ARRAY[0.1, 0.2, 0.4, 3.0],  -- A = 3.0 au lieu de 1.0
    setweight(to_tsvector('french', 'PostgreSQL'), 'A'),
    to_tsquery('french', 'postgresql')
) AS score_custom;
-- Résultat : ~1.80 (3× plus élevé)
```

#### Exemple : Ignorer le Contenu Secondaire

```sql
-- Poids custom : ignorer complètement le poids D
SELECT ts_rank(
    ARRAY[0.0, 0.2, 0.4, 1.0],  -- D = 0.0 (ignoré)
    to_tsvector('french', 'PostgreSQL est puissant'),  -- Tout en D par défaut
    to_tsquery('french', 'postgresql')
) AS score_ignore_D;
-- Résultat : ~0.00 (ignoré car poids D)

-- Comparaison avec poids A
SELECT ts_rank(
    ARRAY[0.0, 0.2, 0.4, 1.0],
    setweight(to_tsvector('french', 'PostgreSQL est puissant'), 'A'),
    to_tsquery('french', 'postgresql')
) AS score_avec_A;
-- Résultat : ~0.60 (pris en compte)
```

#### Cas d'Usage : E-commerce

Pour un site e-commerce, le nom du produit est crucial :

```sql
-- Poids pour produits : nom > description > spécifications
WITH produit AS (
    SELECT
        setweight(to_tsvector('french', 'Chaussures de sport Nike Air'), 'A') ||
        setweight(to_tsvector('french', 'Chaussures running confortables'), 'B') ||
        setweight(to_tsvector('french', 'Semelle en mousse, taille 42'), 'C') AS search_vector
)
SELECT ts_rank(
    ARRAY[0.0, 0.1, 0.3, 2.0],  -- A=2.0 (nom très important), D=0.0 (ignoré)
    search_vector,
    plainto_tsquery('french', 'chaussures nike')
) AS score
FROM produit;
-- Nom du produit booste significativement le score
```

## 5. Normalisation du Score

### Pourquoi Normaliser ?

Sans normalisation, les documents longs ont tendance à avoir des scores plus élevés simplement parce qu'ils contiennent plus de mots.

**Problème** :
```sql
-- Document court : "PostgreSQL"
Score : 0.06

-- Document long : "PostgreSQL" + 1000 autres mots
Score : 0.08 (légèrement plus élevé à cause de la longueur)
```

**Solution** : Normaliser le score pour compenser la longueur.

### Méthodes de Normalisation

PostgreSQL offre plusieurs méthodes de normalisation via un paramètre entier :

```sql
ts_rank(tsvector, tsquery, normalization) → real
```

**Valeurs de normalisation** (combinables avec `|`) :

| Valeur | Description | Formule |
|--------|-------------|---------|
| 0 | Pas de normalisation (défaut) | score brut |
| 1 | Diviser par (1 + log(longueur)) | score / (1 + log(length)) |
| 2 | Diviser par longueur | score / length |
| 4 | Diviser par moyenne harmonique | formule complexe |
| 8 | Diviser par nombre de mots uniques | score / unique_words |
| 16 | Diviser par (1 + log(nombre uniques)) | score / (1 + log(unique)) |
| 32 | Diviser par lui-même + 1 | score / (score + 1) |

### Exemples de Normalisation

#### Sans Normalisation (0)

```sql
-- Document court
SELECT ts_rank(
    to_tsvector('french', 'PostgreSQL'),
    to_tsquery('french', 'postgresql'),
    0  -- Pas de normalisation
) AS score_court;
-- Résultat : ~0.06

-- Document long (même nombre d'occurrences de "PostgreSQL")
SELECT ts_rank(
    to_tsvector('french', 'PostgreSQL ' || repeat('mot ', 100)),
    to_tsquery('french', 'postgresql'),
    0
) AS score_long;
-- Résultat : ~0.06 (similaire mais légèrement différent)
```

#### Normalisation par Longueur (2)

```sql
-- Document court
SELECT ts_rank(
    to_tsvector('french', 'PostgreSQL'),
    to_tsquery('french', 'postgresql'),
    2  -- Diviser par longueur
) AS score_court;
-- Résultat : ~0.06

-- Document long
SELECT ts_rank(
    to_tsvector('french', 'PostgreSQL ' || repeat('mot ', 100)),
    to_tsquery('french', 'postgresql'),
    2
) AS score_long;
-- Résultat : ~0.0006 (100× plus faible car 100× plus long)
```

#### Normalisation Logarithmique (1) - Recommandée

```sql
-- Normalisation douce (recommandée pour la plupart des cas)
SELECT ts_rank(
    to_tsvector('french', 'PostgreSQL ' || repeat('mot ', 100)),
    to_tsquery('french', 'postgresql'),
    1  -- 1 + log(length)
) AS score_norm;
-- Résultat : ~0.02 (réduit mais pas autant que /2)
```

### Combiner Plusieurs Normalisations

Les valeurs peuvent être combinées avec l'opérateur `|` (OR binaire) :

```sql
-- Normalisation 1 + 2 (logarithmique ET longueur)
SELECT ts_rank(
    to_tsvector('french', 'PostgreSQL ' || repeat('mot ', 100)),
    to_tsquery('french', 'postgresql'),
    1 | 2  -- Combine les deux méthodes
) AS score_combine;
```

### Quelle Normalisation Choisir ?

**Recommandations** :

| Cas d'usage | Normalisation | Raison |
|-------------|---------------|--------|
| **Articles/Blog** | `1` (log) | Documents de longueur variable |
| **Produits e-commerce** | `0` ou `1` | Descriptions courtes, longueur similaire |
| **Documents techniques** | `1` ou `2` | Peut être très long, favoriser la densité |
| **Recherche générale** | `1` (par défaut) | Bon compromis |
| **Tweets/Messages courts** | `0` | Longueur limitée, pas besoin de normaliser |

```sql
-- Pattern recommandé : normalisation logarithmique
SELECT
    titre,
    ts_rank(search_vector, query, 1) AS score  -- Normalisation = 1
FROM articles, plainto_tsquery('french', 'recherche') query  
WHERE search_vector @@ query  
ORDER BY score DESC;  
```

## 6. ts_rank_cd : Cover Density Ranking

### Qu'est-ce que ts_rank_cd ?

**ts_rank_cd** calcule le score basé sur la **densité de couverture** (cover density) : la proximité des termes de recherche dans le document.

**Différence clé** :
- **ts_rank** : Se concentre sur la fréquence et les poids  
- **ts_rank_cd** : Favorise les documents où les termes sont **proches les uns des autres**

```sql
ts_rank_cd(tsvector, tsquery) → real  
ts_rank_cd(weights, tsvector, tsquery, normalization) → real  
```

### Exemple de Différence

```sql
-- Document A : termes espacés
WITH doc_a AS (
    SELECT to_tsvector('french',
        'PostgreSQL est puissant. ' || repeat('Texte sans rapport. ', 50) || 'Base de données performante.'
    ) AS vec
)
SELECT
    ts_rank(vec, query) AS score_rank,
    ts_rank_cd(vec, query) AS score_rank_cd
FROM doc_a, to_tsquery('french', 'postgresql & base') query;
-- ts_rank : ~0.08
-- ts_rank_cd : ~0.02 (pénalisé car termes éloignés)

-- Document B : termes proches
WITH doc_b AS (
    SELECT to_tsvector('french',
        'PostgreSQL est une base de données puissante.'
    ) AS vec
)
SELECT
    ts_rank(vec, query) AS score_rank,
    ts_rank_cd(vec, query) AS score_rank_cd
FROM doc_b, to_tsquery('french', 'postgresql & base') query;
-- ts_rank : ~0.10
-- ts_rank_cd : ~0.15 (récompensé car termes proches)
```

### Quand Utiliser ts_rank_cd ?

**Utilisez ts_rank_cd quand** :
- ✅ La proximité des termes est importante (recherche de phrases)  
- ✅ Vous recherchez plusieurs termes qui doivent être ensemble  
- ✅ Documents longs où les termes peuvent être dispersés

**Utilisez ts_rank quand** :
- ✅ La fréquence globale est plus importante  
- ✅ Documents courts  
- ✅ Cas général (plus rapide)

```sql
-- Comparaison pour une recherche de phrase
SELECT
    titre,
    ts_rank(search_vector, query) AS score_freq,
    ts_rank_cd(search_vector, query) AS score_proximite
FROM articles, phraseto_tsquery('french', 'base de données') query  
WHERE search_vector @@ query  
ORDER BY score_proximite DESC;  -- Privilégier la proximité  
```

## 7. Stratégies de Ranking Avancées

### Combinaison de Plusieurs Facteurs

Dans la pratique, vous voudrez souvent combiner le score textuel avec d'autres critères :

#### Score Hybride : Texte + Popularité

```sql
-- Table avec métadonnées
CREATE TABLE articles (
    id SERIAL PRIMARY KEY,
    titre VARCHAR(200),
    contenu TEXT,
    vues INTEGER DEFAULT 0,
    likes INTEGER DEFAULT 0,
    created_at TIMESTAMP DEFAULT NOW(),
    search_vector tsvector
);

-- Fonction de score hybride
SELECT
    titre,
    -- Score textuel (60%)
    ts_rank(search_vector, query, 1) * 0.6 +
    -- Popularité normalisée (30%)
    (likes::float / NULLIF((SELECT MAX(likes) FROM articles), 0)) * 0.3 +
    -- Fraîcheur (10%)
    (EXTRACT(EPOCH FROM (NOW() - created_at)) / -86400 / 365) * 0.1 AS score_total
FROM articles, plainto_tsquery('french', 'postgresql') query  
WHERE search_vector @@ query  
ORDER BY score_total DESC;  
```

#### Score avec Boost Temporel (Articles Récents)

```sql
-- Favoriser les articles récents
SELECT
    titre,
    created_at,
    ts_rank(search_vector, query, 1) *
    -- Boost exponentiel pour les articles récents
    (1 + EXP(-EXTRACT(EPOCH FROM (NOW() - created_at)) / 2592000)) AS score_boosted
    -- 2592000 = 30 jours en secondes
FROM articles, plainto_tsquery('french', 'postgresql') query  
WHERE search_vector @@ query  
ORDER BY score_boosted DESC;  
```

#### Score avec Catégories Privilégiées

```sql
-- Boost pour certaines catégories
SELECT
    titre,
    categorie,
    ts_rank(search_vector, query, 1) *
    CASE categorie
        WHEN 'Tutoriel' THEN 2.0  -- Boost 2×
        WHEN 'Documentation' THEN 1.5  -- Boost 1.5×
        ELSE 1.0  -- Pas de boost
    END AS score_avec_boost
FROM articles, plainto_tsquery('french', 'postgresql') query  
WHERE search_vector @@ query  
ORDER BY score_avec_boost DESC;  
```

### Multi-Field Search avec Poids Différents

```sql
-- Recherche dans plusieurs champs avec importance variable
SELECT
    titre,
    -- Score titre (poids 3)
    ts_rank(setweight(to_tsvector('french', titre), 'A'), query) * 3 +
    -- Score description (poids 2)
    ts_rank(setweight(to_tsvector('french', description), 'B'), query) * 2 +
    -- Score contenu (poids 1)
    ts_rank(setweight(to_tsvector('french', contenu), 'C'), query) * 1 AS score_total
FROM articles, plainto_tsquery('french', 'postgresql base données') query  
WHERE  
    to_tsvector('french', titre) @@ query OR
    to_tsvector('french', description) @@ query OR
    to_tsvector('french', contenu) @@ query
ORDER BY score_total DESC;
```

## 8. Performance et Optimisation

### Pré-calculer le Ranking vs Calculer à la Volée

#### Option 1 : Calculer à la Volée (Simple mais Plus Lent)

```sql
-- Calcul du score à chaque requête
SELECT
    titre,
    ts_rank(search_vector, query, 1) AS score
FROM articles, plainto_tsquery('french', 'postgresql') query  
WHERE search_vector @@ query  
ORDER BY score DESC  
LIMIT 20;  

-- Impact : +5-10ms par requête pour 100k documents
```

#### Option 2 : Pré-calculer le Score (Complexe mais Plus Rapide)

Cette approche n'est utile que si vous avez des **requêtes fréquentes identiques**.

```sql
-- Table de cache pour requêtes populaires
CREATE TABLE search_cache (
    query_text TEXT PRIMARY KEY,
    article_id INTEGER,
    score REAL,
    updated_at TIMESTAMP DEFAULT NOW()
);

-- Pré-calculer pour les requêtes populaires
INSERT INTO search_cache (query_text, article_id, score)  
SELECT  
    'postgresql' AS query_text,
    a.id,
    ts_rank(a.search_vector, to_tsquery('french', 'postgresql'), 1)
FROM articles a  
WHERE a.search_vector @@ to_tsquery('french', 'postgresql');  

-- Requête ultra-rapide
SELECT a.titre, sc.score  
FROM search_cache sc  
JOIN articles a ON a.id = sc.article_id  
WHERE sc.query_text = 'postgresql'  
ORDER BY sc.score DESC  
LIMIT 20;  

-- Impact : <1ms (lecture simple d'index)
```

### Optimiser les Requêtes avec Ranking

#### ✅ Filtrer PUIS Ranker (Pas l'Inverse)

```sql
-- ✅ BON : Filtrer d'abord avec @@, puis ranker les résultats
SELECT titre, ts_rank(search_vector, query, 1) AS score  
FROM articles, plainto_tsquery('french', 'postgresql') query  
WHERE search_vector @@ query  -- Filtre d'abord (utilise index GIN)  
ORDER BY score DESC  
LIMIT 20;  

-- ❌ MAUVAIS : Ranker tous les documents puis filtrer
SELECT titre, score  
FROM (  
    SELECT titre, ts_rank(search_vector, query, 1) AS score
    FROM articles, plainto_tsquery('french', 'postgresql') query
) t
WHERE score > 0  -- Filtre après ranking (inefficace)  
ORDER BY score DESC;  
```

#### ✅ Limiter les Résultats Tôt

```sql
-- Utiliser LIMIT pour éviter de trier tous les résultats
SELECT titre, ts_rank(search_vector, query, 1) AS score  
FROM articles, plainto_tsquery('french', 'postgresql') query  
WHERE search_vector @@ query  
ORDER BY score DESC  
LIMIT 20;  -- ← Important : ne calcule/trie que le top 20  
```

#### ✅ Utiliser une CTE pour Réutiliser la Query

```sql
-- Éviter de recréer la query plusieurs fois
WITH search AS (
    SELECT plainto_tsquery('french', 'postgresql base données') AS query
)
SELECT
    a.titre,
    ts_rank(a.search_vector, s.query, 1) AS score
FROM articles a, search s  
WHERE a.search_vector @@ s.query  
ORDER BY score DESC  
LIMIT 20;  
```

### Index et Ranking

**Important** : Le ranking lui-même n'est **PAS** accéléré par les index. Seul le **filtrage** (`@@`) bénéficie de l'index GIN.

```sql
-- L'index GIN accélère le WHERE, pas le ORDER BY
CREATE INDEX idx_articles_search ON articles USING GIN(search_vector);

-- Processus :
-- 1. WHERE utilise l'index GIN → Rapide (10ms)
-- 2. ts_rank calcule le score → Coût proportionnel au nombre de résultats (5ms pour 100 docs)
-- 3. ORDER BY trie les scores → Rapide pour un petit nombre de résultats
```

**Best Practice** : Combiner avec LIMIT pour minimiser le nombre de documents à ranker.

## 9. Cas Pratiques Complets

### Cas 1 : Moteur de Recherche de Blog

```sql
-- Structure complète
CREATE TABLE blog_articles (
    id SERIAL PRIMARY KEY,
    titre VARCHAR(200),
    description TEXT,
    contenu TEXT,
    auteur VARCHAR(100),
    categorie VARCHAR(50),
    vues INTEGER DEFAULT 0,
    created_at TIMESTAMP DEFAULT NOW(),
    search_vector tsvector
);

-- Trigger pour maintenir search_vector avec poids
CREATE OR REPLACE FUNCTION blog_search_trigger() RETURNS trigger AS $$  
BEGIN  
    NEW.search_vector :=
        setweight(to_tsvector('french', COALESCE(NEW.titre, '')), 'A') ||
        setweight(to_tsvector('french', COALESCE(NEW.description, '')), 'B') ||
        setweight(to_tsvector('french', COALESCE(NEW.contenu, '')), 'C') ||
        setweight(to_tsvector('french', COALESCE(NEW.auteur, '')), 'D');
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER tsvector_update  
BEFORE INSERT OR UPDATE ON blog_articles  
FOR EACH ROW EXECUTE FUNCTION blog_search_trigger();  

-- Index GIN
CREATE INDEX idx_blog_search ON blog_articles USING GIN(search_vector);

-- Fonction de recherche avec ranking hybride
CREATE OR REPLACE FUNCTION search_blog(
    search_text TEXT,
    max_results INTEGER DEFAULT 20
)
RETURNS TABLE (
    id INTEGER,
    titre VARCHAR,
    description TEXT,
    auteur VARCHAR,
    score REAL
) AS $$
BEGIN
    RETURN QUERY
    SELECT
        a.id,
        a.titre,
        a.description,
        a.auteur,
        -- Score hybride : pertinence + popularité + fraîcheur
        (
            ts_rank(
                ARRAY[0.1, 0.2, 0.4, 1.0],  -- Poids custom
                a.search_vector,
                query,
                1  -- Normalisation logarithmique
            ) * 0.7 +  -- 70% pertinence textuelle
            (a.vues::float / NULLIF((SELECT MAX(vues) FROM blog_articles), 0)) * 0.2 +  -- 20% popularité
            (1.0 / (1.0 + EXTRACT(EPOCH FROM (NOW() - a.created_at)) / 2592000)) * 0.1  -- 10% fraîcheur
        ) AS score
    FROM
        blog_articles a,
        websearch_to_tsquery('french', search_text) query
    WHERE a.search_vector @@ query
    ORDER BY score DESC
    LIMIT max_results;
END;
$$ LANGUAGE plpgsql;

-- Utilisation
SELECT * FROM search_blog('postgresql base données');
```

### Cas 2 : E-commerce avec Ranking de Produits

```sql
-- Table produits
CREATE TABLE produits (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    description TEXT,
    marque VARCHAR(100),
    categorie VARCHAR(50),
    prix NUMERIC(10, 2),
    stock INTEGER,
    note_moyenne NUMERIC(2, 1),

    search_vector tsvector
);

-- Trigger avec poids adaptés e-commerce
CREATE OR REPLACE FUNCTION produits_search_trigger() RETURNS trigger AS $$  
BEGIN  
    NEW.search_vector :=
        -- Nom du produit : très important (poids A, multiplié par 2)
        setweight(to_tsvector('french', COALESCE(NEW.nom, '')), 'A') ||
        setweight(to_tsvector('french', COALESCE(NEW.nom, '')), 'A') ||  -- Dupliqué pour doubler l'importance
        -- Marque : important (poids A)
        setweight(to_tsvector('french', COALESCE(NEW.marque, '')), 'A') ||
        -- Description : moyen (poids B)
        setweight(to_tsvector('french', COALESCE(NEW.description, '')), 'B') ||
        -- Catégorie : faible (poids C)
        setweight(to_tsvector('french', COALESCE(NEW.categorie, '')), 'C');
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER tsvector_update  
BEFORE INSERT OR UPDATE ON produits  
FOR EACH ROW EXECUTE FUNCTION produits_search_trigger();  

-- Index
CREATE INDEX idx_produits_search ON produits USING GIN(search_vector);

-- Recherche avec ranking e-commerce
SELECT
    nom,
    marque,
    prix,
    note_moyenne,
    -- Score composite
    (
        ts_rank(
            ARRAY[0.05, 0.2, 0.4, 2.0],  -- Nom/Marque très importants
            search_vector,
            query,
            1
        ) * 0.5 +  -- 50% pertinence textuelle
        (note_moyenne / 5.0) * 0.2 +  -- 20% note client
        (LEAST(nombre_avis, 1000)::float / 1000) * 0.15 +  -- 15% nombre d'avis
        CASE
            WHEN stock > 10 THEN 0.15  -- 15% disponibilité
            WHEN stock > 0 THEN 0.075
            ELSE 0
        END
    ) AS score_final
FROM produits, websearch_to_tsquery('french', 'chaussures running nike') query  
WHERE search_vector @@ query  
  AND stock > 0  -- Uniquement produits en stock
ORDER BY score_final DESC  
LIMIT 50;  
```

## 11. Résumé et Aide-Mémoire

### ts_rank : Calculer le Score

```sql
-- Basique
ts_rank(tsvector, tsquery) → real

-- Avec poids personnalisés
ts_rank(ARRAY[D, C, B, A], tsvector, tsquery) → real

-- Avec normalisation
ts_rank(tsvector, tsquery, normalization_code) → real

-- Complet
ts_rank(ARRAY[0.1, 0.2, 0.4, 1.0], tsvector, tsquery, 1) → real
```

### setweight : Attribuer un Poids

```sql
-- Syntaxe
setweight(tsvector, 'A'|'B'|'C'|'D') → tsvector

-- Exemple
setweight(to_tsvector('french', 'texte'), 'A')

-- Combiner plusieurs poids
setweight(to_tsvector('french', 'titre'), 'A') ||  
setweight(to_tsvector('french', 'contenu'), 'C')  
```

### Hiérarchie de Poids Standard

| Poids | Usage Typique | Multiplicateur par Défaut |
|-------|---------------|---------------------------|
| **A** | Titre, nom | 1.0 (référence) |
| **B** | Sous-titre, description | 0.4 |
| **C** | Contenu principal | 0.2 |
| **D** | Métadonnées, tags | 0.1 |

### Normalisation

| Code | Description | Cas d'usage |
|------|-------------|-------------|
| `0` | Pas de normalisation | Documents de longueur similaire |
| `1` | Logarithmique (recommandé) | Cas général, documents variables |
| `2` | Par longueur | Favoriser forte densité |
| `1\|2` | Combiné | Maximum de normalisation |

### Pattern Complet

```sql
-- 1. Structure
CREATE TABLE ma_table (
    id SERIAL PRIMARY KEY,
    titre TEXT,
    description TEXT,
    contenu TEXT,
    search_vector tsvector
);

-- 2. Trigger avec poids
CREATE FUNCTION ma_table_search_trigger() RETURNS trigger AS $$  
BEGIN  
    NEW.search_vector :=
        setweight(to_tsvector('french', COALESCE(NEW.titre, '')), 'A') ||
        setweight(to_tsvector('french', COALESCE(NEW.description, '')), 'B') ||
        setweight(to_tsvector('french', COALESCE(NEW.contenu, '')), 'C');
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER tsvector_update BEFORE INSERT OR UPDATE ON ma_table  
FOR EACH ROW EXECUTE FUNCTION ma_table_search_trigger();  

-- 3. Index
CREATE INDEX idx_search ON ma_table USING GIN(search_vector);

-- 4. Recherche avec ranking
SELECT
    titre,
    ts_rank(
        ARRAY[0.1, 0.2, 0.4, 1.0],  -- Poids custom
        search_vector,
        query,
        1  -- Normalisation
    ) AS score
FROM ma_table, websearch_to_tsquery('french', 'recherche') query  
WHERE search_vector @@ query  
ORDER BY score DESC  
LIMIT 20;  
```

## Conclusion

Le **ranking** et la **pondération** sont essentiels pour obtenir des résultats de recherche pertinents et triés intelligemment.

**Points clés à retenir** :

1. **ts_rank** calcule un score de pertinence basé sur :
   - Fréquence des termes
   - Poids des lexèmes (A, B, C, D)
   - Longueur du document

2. **setweight** attribue de l'importance aux parties d'un document :
   - A (titre) > B (description) > C (contenu) > D (métadonnées)
   - Ratio par défaut : 1.0 : 0.4 : 0.2 : 0.1

3. **Normalisation** compense la longueur des documents :  
   - `0` : Pas de normalisation  
   - `1` : Logarithmique (recommandé)  
   - `2` : Par longueur (strict)

4. **Stratégies avancées** combinent :
   - Score textuel (ts_rank)
   - Popularité (vues, likes)
   - Fraîcheur (date)
   - Disponibilité (stock)

5. **Performance** :
   - L'index GIN accélère le filtrage, pas le ranking
   - Utiliser LIMIT pour éviter de calculer trop de scores
   - Pré-calculer pour les requêtes fréquentes

Le ranking transforme une simple recherche en un moteur de recherche intelligent qui comprend l'**intention** de l'utilisateur et retourne les résultats les plus **pertinents** en premier.
