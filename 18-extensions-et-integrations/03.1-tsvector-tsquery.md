# 18.3.1. Full-Text Search : tsvector et tsquery

## Introduction

Imaginez que vous devez construire une fonctionnalité de recherche pour un site e-commerce, un blog, ou une application de gestion documentaire. Vos utilisateurs tapent "chaussures rouges femme" et s'attendent à trouver tous les produits pertinents, même si la description dit "escarpin rouge pour dame".

Les recherches simples avec `LIKE` ou `ILIKE` ne suffisent pas :
- ❌ Pas de gestion du pluriel ("chaussure" vs "chaussures")  
- ❌ Pas de recherche par pertinence (quel résultat est le plus pertinent ?)  
- ❌ Performances médiocres sur de grandes quantités de texte  
- ❌ Pas de gestion des variantes linguistiques ("chaussure", "chaussures", "chausser")

C'est là qu'intervient le **Full-Text Search** de PostgreSQL, un système de recherche textuelle puissant et performant intégré directement dans la base de données.

Au cœur de ce système se trouvent deux types de données fondamentaux :
- **tsvector** : Représente un document préparé pour la recherche (indexé)  
- **tsquery** : Représente une requête de recherche

---

## 1. Qu'est-ce que le Full-Text Search ?

### Le Problème avec LIKE

Prenons un exemple simple :

```sql
-- Table d'articles de blog
CREATE TABLE articles (
    id SERIAL PRIMARY KEY,
    titre VARCHAR(200),
    contenu TEXT
);

INSERT INTO articles (titre, contenu) VALUES
    ('Guide PostgreSQL', 'PostgreSQL est un système de gestion de base de données relationnel puissant.'),
    ('Apprendre SQL', 'SQL est le langage standard pour interroger les bases de données.'),
    ('Tutoriel Bases de Données', 'Les bases de données stockent et organisent vos données.');

-- Recherche avec LIKE
SELECT titre FROM articles  
WHERE contenu LIKE '%base de données%';  
```

**Problèmes** :
1. **Performance** : Scan séquentiel de toute la table (pas d'index possible)  
2. **Pas de pertinence** : Tous les résultats sont égaux  
3. **Pas de stemming** : "bases" ≠ "base", "données" ≠ "donnée"  
4. **Sensible à la casse** : Nécessite ILIKE (encore plus lent)

### La Solution : Full-Text Search

Le Full-Text Search de PostgreSQL :
- ✅ **Indexable** : Recherches ultra-rapides avec index GIN  
- ✅ **Ranking** : Trie par pertinence  
- ✅ **Stemming** : Normalise les mots ("bases" → "base")  
- ✅ **Langues** : Support de 20+ langues (français, anglais, espagnol, etc.)  
- ✅ **Opérateurs riches** : AND, OR, NOT, proximité, phrases exactes

## 2. tsvector : Le Document Indexé

### Qu'est-ce qu'un tsvector ?

Un **tsvector** est une représentation optimisée d'un document texte pour la recherche. Il contient :
- Les **lexèmes** (mots normalisés/stemmed)
- Les **positions** de chaque lexème dans le document
- Les **poids** optionnels (A, B, C, D pour l'importance)

**Transformation** :
```
Texte original :
"PostgreSQL est une base de données puissante. Les bases de données sont essentielles."

↓ Traitement (tokenization + stemming)

tsvector :
'base':4,9 'donn':6,11 'essenti':13 'postgresql':1 'puiss':7
```

### Créer un tsvector

#### Méthode 1 : Conversion Directe (sans langue)

```sql
-- Conversion simple en tsvector
SELECT 'PostgreSQL est une base de données puissante'::tsvector;

-- Résultat :
-- 'PostgreSQL' 'base' 'de' 'données' 'est' 'puissante' 'une'
-- (pas de stemming, mots triés alphabétiquement)
```

**⚠️ Attention** : Cette méthode ne fait **PAS** de stemming et garde tous les mots tels quels.

#### Méthode 2 : to_tsvector() - La Méthode Recommandée

```sql
-- Avec stemming et langue française
SELECT to_tsvector('french', 'PostgreSQL est une base de données puissante');

-- Résultat :
-- 'base':4 'donn':6 'postgresql':1 'puiss':7
-- (stemming appliqué, positions conservées, mots vides supprimés)
```

**Explication** :
- `'french'` : Configuration linguistique (français)  
- **Stemming** : "données" → "donn", "puissante" → "puiss"  
- **Stop words** : "est", "une", "de" supprimés (mots vides)  
- **Positions** : Les chiffres indiquent la position d'origine

#### Langues Disponibles

```sql
-- Lister les configurations disponibles
SELECT cfgname FROM pg_ts_config;

-- Résultat typique :
-- simple      (pas de stemming)
-- arabic
-- danish
-- dutch
-- english
-- finnish
-- french      ← Pour le français
-- german
-- hungarian
-- italian
-- norwegian
-- portuguese
-- romanian
-- russian
-- spanish
-- swedish
-- turkish
```

### Anatomie d'un tsvector

```sql
-- Exemple détaillé
SELECT to_tsvector('french',
    'Les bases de données PostgreSQL sont très puissantes. ' ||
    'PostgreSQL gère les données efficacement.'
);

-- Résultat :
-- 'base':2 'donn':4,11 'efficac':12 'ger':10 'postgresql':5,9 'puiss':8 'tres':7
```

**Décomposition** :
```
'base':2           → Lexème "base" à la position 2
'donn':4,11        → Lexème "donn" aux positions 4 et 11 (données répété)
'postgresql':5,9   → "PostgreSQL" apparaît 2 fois
'puiss':8          → "puissantes" → stemmed en "puiss"
```

### Opérations sur tsvector

#### Concaténation

```sql
-- Combiner plusieurs tsvector
SELECT
    to_tsvector('french', 'PostgreSQL est puissant') ||
    to_tsvector('french', 'et performant');

-- Résultat : 'perform':6 'postgresql':1 'puiss':3'
```

#### Extraction d'Information

```sql
-- Nombre de lexèmes uniques
SELECT length(to_tsvector('french', 'PostgreSQL est une base de données PostgreSQL'));
-- Résultat : 4 (postgresql, base, donn, puiss)

-- Extraire tous les lexèmes
SELECT unnest(to_tsvector('french', 'PostgreSQL et les bases de données'));
-- Résultat :
-- base
-- donn
-- postgresql
```

### tsvector avec Poids

Les lexèmes peuvent avoir des **poids** (A, B, C, D) pour indiquer leur importance.

```sql
-- Créer un tsvector avec poids
SELECT
    setweight(to_tsvector('french', 'PostgreSQL'), 'A') ||  -- Titre (poids A)
    setweight(to_tsvector('french', 'Base de données performante'), 'B');  -- Description (poids B)

-- Résultat : 'base':3B 'donn':5B 'perform':6B 'postgresql':1A
```

**Poids standard** :
- **A** : Titre (le plus important)  
- **B** : Sous-titre ou résumé  
- **C** : Contenu principal  
- **D** : Métadonnées ou contenu secondaire (défaut)

## 3. tsquery : La Requête de Recherche

### Qu'est-ce qu'un tsquery ?

Un **tsquery** représente une requête de recherche avec :
- Les termes à rechercher (normalisés/stemmed)
- Les opérateurs logiques (AND, OR, NOT)
- Les opérateurs de proximité

### Créer un tsquery

#### Méthode 1 : to_tsquery() - Requête Structurée

```sql
-- Recherche simple
SELECT to_tsquery('french', 'postgresql');
-- Résultat : 'postgresql'

-- Recherche avec AND
SELECT to_tsquery('french', 'postgresql & base');
-- Résultat : 'postgresql' & 'base'

-- Recherche avec OR
SELECT to_tsquery('french', 'postgresql | mysql');
-- Résultat : 'postgresql' | 'mysql'

-- Recherche avec NOT
SELECT to_tsquery('french', 'postgresql & !oracle');
-- Résultat : 'postgresql' & !'oracle'
```

**Opérateurs disponibles** :
- `&` : AND (intersection)  
- `|` : OR (union)  
- `!` : NOT (négation)  
- `<->` : Suivi de (mots adjacents)  
- `<N>` : Proximité (N mots d'écart maximum)

**⚠️ Important** : `to_tsquery()` exige une syntaxe stricte (opérateurs explicites).

#### Méthode 2 : plainto_tsquery() - Requête en Texte Libre

```sql
-- Convertit du texte libre en tsquery (plus tolérant)
SELECT plainto_tsquery('french', 'postgresql base données');
-- Résultat : 'postgresql' & 'base' & 'donn'
-- (tous les mots avec AND implicite)

-- Gère les mots vides automatiquement
SELECT plainto_tsquery('french', 'le postgresql et les bases de données');
-- Résultat : 'postgresql' & 'base' & 'donn'
-- ("le", "et", "les", "de" ignorés)
```

**Quand utiliser ?** : Pour les recherches utilisateur en texte libre (comme Google).

#### Méthode 3 : phraseto_tsquery() - Phrase Exacte

```sql
-- Recherche de phrase exacte
SELECT phraseto_tsquery('french', 'base de données');
-- Résultat : 'base' <-> 'donn'
-- (cherche "base" immédiatement suivi de "donn")

-- Équivalent à :
SELECT to_tsquery('french', 'base <-> données');
```

**Usage** : Recherche de citations ou expressions exactes.

#### Méthode 4 : websearch_to_tsquery() - Style Google (PostgreSQL 11+)

```sql
-- Syntaxe intuitive façon Google
SELECT websearch_to_tsquery('french', 'postgresql "base de données" -oracle');
-- Résultat : 'postgresql' & 'base' <-> 'donn' & !'oracle'
```

**Syntaxe** :
- Guillemets `"..."` : Phrase exacte
- Mot seul : Recherche du mot
- `-mot` : Exclusion (NOT)  
- `OR` : Union explicite
- AND implicite entre mots

**Recommandation** : C'est la méthode **la plus user-friendly** pour les interfaces de recherche.

## 4. L'Opérateur de Correspondance : @@

### Le Cœur du Full-Text Search

L'opérateur `@@` teste si un `tsvector` correspond à un `tsquery`.

**Syntaxe** :
```sql
tsvector @@ tsquery  → boolean
```

### Exemples Simples

```sql
-- Est-ce que le document contient "postgresql" ?
SELECT
    to_tsvector('french', 'PostgreSQL est puissant') @@
    to_tsquery('french', 'postgresql');
-- Résultat : true

-- Est-ce que le document contient "mysql" ?
SELECT
    to_tsvector('french', 'PostgreSQL est puissant') @@
    to_tsquery('french', 'mysql');
-- Résultat : false
```

### Requêtes avec Opérateurs

#### AND : Tous les Termes Doivent Être Présents

```sql
-- Document contenant "postgresql" ET "base"
SELECT
    to_tsvector('french', 'PostgreSQL est une base de données') @@
    to_tsquery('french', 'postgresql & base');
-- Résultat : true

SELECT
    to_tsvector('french', 'PostgreSQL est performant') @@
    to_tsquery('french', 'postgresql & base');
-- Résultat : false (manque "base")
```

#### OR : Au Moins Un Terme Doit Être Présent

```sql
-- Document contenant "postgresql" OU "mysql"
SELECT
    to_tsvector('french', 'PostgreSQL est open-source') @@
    to_tsquery('french', 'postgresql | mysql');
-- Résultat : true (contient postgresql)

SELECT
    to_tsvector('french', 'Oracle est propriétaire') @@
    to_tsquery('french', 'postgresql | mysql');
-- Résultat : false (ni postgresql ni mysql)
```

#### NOT : Exclusion

```sql
-- Document contenant "base" mais PAS "oracle"
SELECT
    to_tsvector('french', 'Base de données PostgreSQL') @@
    to_tsquery('french', 'base & !oracle');
-- Résultat : true

SELECT
    to_tsvector('french', 'Base de données Oracle') @@
    to_tsquery('french', 'base & !oracle');
-- Résultat : false (contient "oracle")
```

#### Proximité : Mots Proches

```sql
-- "base" immédiatement suivi de "données"
SELECT
    to_tsvector('french', 'une base de données') @@
    phraseto_tsquery('french', 'base données');
-- Résultat : true

-- "base" et "données" séparés par d'autres mots
SELECT
    to_tsvector('french', 'une base relationnelle de données') @@
    phraseto_tsquery('french', 'base données');
-- Résultat : false (pas adjacents)
```

### Ordre des Opérandes

L'opérateur `@@` est **commutatif** :

```sql
-- Ces deux requêtes sont équivalentes
SELECT to_tsvector('french', 'texte') @@ to_tsquery('french', 'mot');  
SELECT to_tsquery('french', 'mot') @@ to_tsvector('french', 'texte');  
```

## 5. Applications Pratiques

### Cas 1 : Table de Produits avec Recherche

```sql
-- Créer une table de produits
CREATE TABLE produits (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    description TEXT,
    categorie VARCHAR(100),
    prix NUMERIC(10, 2)
);

-- Insérer des données
INSERT INTO produits (nom, description, categorie, prix) VALUES
    ('Chaussures de sport Nike', 'Chaussures running légères et confortables', 'Sport', 89.99),
    ('Escarpin rouge Gucci', 'Escarpins élégants pour femme, couleur rouge', 'Mode', 450.00),
    ('Baskets Adidas blanches', 'Baskets casual pour homme et femme', 'Sport', 75.50),
    ('Sandales d''été', 'Sandales confortables pour l''été', 'Mode', 35.00);

-- Recherche avec Full-Text Search
SELECT
    nom,
    description,
    prix
FROM produits  
WHERE  
    to_tsvector('french', nom || ' ' || description) @@
    plainto_tsquery('french', 'chaussure rouge femme');
```

**Résultat** :
```
nom                        | description                                    | prix
---------------------------+-----------------------------------------------+-------
Escarpin rouge Gucci       | Escarpins élégants pour femme, couleur rouge  | 450.00  
Baskets Adidas blanches    | Baskets casual pour homme et femme            | 75.50  
```

### Cas 2 : Blog avec Recherche d'Articles

```sql
-- Table d'articles
CREATE TABLE articles (
    id SERIAL PRIMARY KEY,
    titre VARCHAR(200),
    contenu TEXT,
    auteur VARCHAR(100),
    publie_le DATE
);

INSERT INTO articles (titre, contenu, auteur, publie_le) VALUES
    ('Introduction à PostgreSQL',
     'PostgreSQL est un système de gestion de base de données relationnel et objet. ' ||
     'Il est connu pour sa fiabilité et ses fonctionnalités avancées.',
     'Jean Dupont',
     '2024-01-15'),
    ('Guide du Full-Text Search',
     'Le Full-Text Search permet de rechercher efficacement dans de grandes quantités de texte. ' ||
     'PostgreSQL offre des fonctionnalités puissantes pour la recherche textuelle.',
     'Marie Martin',
     '2024-02-20'),
    ('Optimisation des requêtes SQL',
     'L''optimisation des requêtes est essentielle pour de bonnes performances. ' ||
     'Les index et l''analyse du plan d''exécution sont des outils clés.',

-- Recherche d'articles
SELECT

    LEFT(contenu, 100) || '...' AS extrait
FROM articles  
WHERE

    websearch_to_tsquery('french', 'postgresql recherche');
```

**Résultat** :
```
titre                        | auteur       | extrait
-----------------------------+--------------+----------------------------------------
Introduction à PostgreSQL    | Jean Dupont  | PostgreSQL est un système de gestion...  
Guide du Full-Text Search    | Marie Martin | Le Full-Text Search permet de rechercher...  
```

### Cas 3 : Colonne tsvector Pré-calculée (Optimisation)

Pour éviter de recalculer le tsvector à chaque requête, on peut le stocker dans une colonne dédiée.

```sql
-- Ajouter une colonne tsvector
ALTER TABLE articles  
ADD COLUMN search_vector tsvector;  

-- Remplir la colonne avec les données existantes
UPDATE articles  
SET search_vector =  
    setweight(to_tsvector('french', COALESCE(titre, '')), 'A') ||
    setweight(to_tsvector('french', COALESCE(contenu, '')), 'B');

-- Créer un index GIN pour la performance
CREATE INDEX idx_articles_search ON articles USING GIN(search_vector);

-- Maintenir automatiquement avec un trigger
CREATE FUNCTION articles_search_trigger() RETURNS trigger AS $$  
BEGIN  
    NEW.search_vector :=
        setweight(to_tsvector('french', COALESCE(NEW.titre, '')), 'A') ||
        setweight(to_tsvector('french', COALESCE(NEW.contenu, '')), 'B');
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER tsvector_update  
BEFORE INSERT OR UPDATE ON articles  
FOR EACH ROW EXECUTE FUNCTION articles_search_trigger();  

-- Recherche ultra-rapide (utilise l'index)
SELECT titre, auteur  
FROM articles  
WHERE search_vector @@ websearch_to_tsquery('french', 'postgresql base données');  
```

**Avantages** :
- ✅ Pas de recalcul à chaque recherche  
- ✅ Index GIN pour des performances optimales  
- ✅ Mise à jour automatique via trigger  
- ✅ Poids distincts (titre vs contenu)

## 6. Gestion du Stemming et des Variantes

### Comment Fonctionne le Stemming ?

Le **stemming** réduit les mots à leur racine (stem/lemme).

```sql
-- Exemples de stemming français
SELECT to_tsvector('french', 'chanter chantons chantaient chantera');
-- Résultat : 'chant':1,2,3,4
-- (tous réduits à "chant")

SELECT to_tsvector('french', 'base bases données donnée');
-- Résultat : 'base':1,2 'donn':3,4
-- "bases" → "base", "données/donnée" → "donn"

SELECT to_tsvector('french', 'heureux heureuse heureusement');
-- Résultat : 'heur':1,2,3
-- (tous réduits à "heur")
```

### Impact sur les Recherches

```sql
-- Recherche de "chaussure" trouve aussi "chaussures"
SELECT
    'Les chaussures de sport sont confortables'::text AS texte,
    to_tsvector('french', 'Les chaussures de sport sont confortables') @@
    plainto_tsquery('french', 'chaussure') AS trouve;
-- Résultat : true

-- Recherche de "données" trouve aussi "donnée"
SELECT
    'La donnée est importante'::text AS texte,
    to_tsvector('french', 'La donnée est importante') @@
    plainto_tsquery('french', 'données') AS trouve;
-- Résultat : true
```

### Stop Words (Mots Vides)

Les **stop words** sont des mots très courants qui sont automatiquement ignorés.

```sql
-- Mots vides français ignorés
SELECT to_tsvector('french', 'le la les de des un une et ou');
-- Résultat : '' (vide, tous les mots sont des stop words)

-- Impact sur les recherches
SELECT to_tsvector('french', 'Le PostgreSQL et les bases de données');
-- Résultat : 'base':5 'donn':7 'postgresql':2
-- ("le", "et", "les", "de" ignorés)
```

**Liste partielle des stop words français** :
- Articles : le, la, les, un, une, des
- Prépositions : de, à, en, par, pour, dans
- Conjonctions : et, ou, mais
- Pronoms : je, tu, il, nous, vous, ils

### Configuration Simple (sans langue)

Pour désactiver le stemming et les stop words :

```sql
-- Configuration "simple" : pas de stemming
SELECT to_tsvector('simple', 'Les chaussures sont rouges');
-- Résultat : 'chaussures':2 'les':1 'rouges':4 'sont':3
-- (tous les mots conservés, pas de stemming)
```

## 7. Debugging et Comprendre les Résultats

### Visualiser le Traitement

```sql
-- Voir comment un texte est décomposé
SELECT * FROM ts_debug('french', 'PostgreSQL est une base de données puissante');
```

**Résultat** (extrait) :
```
alias      | description          | token      | dictionaries  | dictionary | lexemes
-----------+---------------------+------------+---------------+------------+---------
asciiword  | Word, all ASCII     | PostgreSQL | {french_stem} | french_stem| {postgresql}  
blank      | Space symbols       |            | {}            |            |  
asciiword  | Word, all ASCII     | est        | {french_stem} | french_stem| {}  
blank      | Space symbols       |            | {}            |            |  
asciiword  | Word, all ASCII     | une        | {french_stem} | french_stem| {}  
...
```

**Colonnes importantes** :
- `token` : Mot original  
- `dictionary` : Dictionnaire utilisé  
- `lexemes` : Forme normalisée (vide si stop word)

### Comprendre Pourquoi une Recherche Échoue

```sql
-- Texte à rechercher
WITH doc AS (
    SELECT 'PostgreSQL est rapide' AS texte
)
-- Analyser le document
SELECT
    texte,
    to_tsvector('french', texte) AS doc_vector,
    to_tsquery('french', 'performant') AS requete,
    to_tsvector('french', texte) @@ to_tsquery('french', 'performant') AS trouve
FROM doc;
```

**Résultat** :
```
texte                  | doc_vector             | requete      | trouve
-----------------------+------------------------+-------------+--------
PostgreSQL est rapide  | 'postgresql':1 'rapid':3 | 'perform':1 | false
```

**Explication** : "rapide" (stemmed: "rapid") ≠ "performant" (stemmed: "perform")

### Tester le Stemming d'un Mot

```sql
-- Voir le stem d'un mot spécifique
SELECT to_tsvector('french', 'performant');
-- Résultat : 'perform':1

SELECT to_tsvector('french', 'performance');
-- Résultat : 'perform':1

-- Ces deux mots ont le même stem, donc se trouvent mutuellement
SELECT
    to_tsvector('french', 'système performant') @@
    to_tsquery('french', 'performance');
-- Résultat : true
```

## 8. Opérateurs et Fonctions Avancés

### Opérateurs de Proximité

#### Adjacence Stricte (<->)

```sql
-- "base" immédiatement suivi de "données"
SELECT
    to_tsvector('french', 'base de données') @@
    to_tsquery('french', 'base <-> données');
-- Résultat : true (après suppression de "de", ils sont adjacents)

SELECT
    to_tsvector('french', 'base relationnelle de données') @@
    to_tsquery('french', 'base <-> données');
-- Résultat : false ("relationnelle" entre les deux)
```

#### Proximité avec Distance (<N>)

```sql
-- "base" et "données" séparés par max 2 mots
SELECT
    to_tsvector('french', 'base relationnelle de données') @@
    to_tsquery('french', 'base <2> données');
-- Résultat : true

-- Distance trop grande
SELECT
    to_tsvector('french', 'base très grande et complexe de données') @@
    to_tsquery('french', 'base <2> données');
-- Résultat : false (trop de mots entre)
```

### Préfixes (Recherche par Début de Mot)

```sql
-- Chercher tous les mots commençant par "post"
SELECT
    to_tsvector('french', 'PostgreSQL PostGIS poster') @@
    to_tsquery('french', 'post:*');
-- Résultat : true

-- Utile pour l'autocomplétion
SELECT
    to_tsvector('french', 'développement développeur développé') @@
    to_tsquery('french', 'devel:*');
-- Résultat : true (tous commencent par "devel")
```

### Fonctions Utiles

#### ts_rank : Calculer la Pertinence

```sql
-- Calculer le score de pertinence
WITH doc AS (
    SELECT
        'PostgreSQL' AS id,
        'PostgreSQL est un SGBD. PostgreSQL est open-source.' AS texte
)
SELECT
    id,
    ts_rank(
        to_tsvector('french', texte),
        to_tsquery('french', 'postgresql')
    ) AS score
FROM doc;
-- Résultat : score ≈ 0.06 (plus le score est élevé, plus pertinent)
```

#### ts_headline : Extraire des Extraits

```sql
-- Générer un extrait avec les mots trouvés mis en évidence
SELECT ts_headline('french',
    'PostgreSQL est un système de gestion de base de données. ' ||
    'PostgreSQL offre de nombreuses fonctionnalités avancées.',
    to_tsquery('french', 'postgresql & base')
);

-- Résultat :
-- "<b>PostgreSQL</b> est un système de gestion de <b>base</b> de données..."
```

#### ts_rewrite : Modifier une Requête

```sql
-- Réécrire une requête (synonymes)
SELECT ts_rewrite(
    to_tsquery('french', 'postgresql'),
    to_tsquery('french', 'postgresql'),
    to_tsquery('french', 'postgres | postgresql')
);
-- Résultat : 'postgres' | 'postgresql'
```

## 9. Bonnes Pratiques

### ✅ Utiliser websearch_to_tsquery() pour les Entrées Utilisateur

```sql
-- ✅ BON : Tolérant, syntaxe intuitive
SELECT * FROM articles  
WHERE search_vector @@ websearch_to_tsquery('french', 'postgresql "base données"');  

-- ❌ À ÉVITER : Nécessite syntaxe stricte, erreurs faciles
SELECT * FROM articles  
WHERE search_vector @@ to_tsquery('french', 'postgresql & base <-> données');  
```

### ✅ Créer des Colonnes tsvector Dédiées

```sql
-- ✅ BON : Performance optimale
ALTER TABLE produits ADD COLUMN search_vector tsvector;  
CREATE INDEX idx_produits_search ON produits USING GIN(search_vector);  

-- ❌ LENT : Recalcul à chaque requête
SELECT * FROM produits  
WHERE to_tsvector('french', nom || ' ' || description) @@ ...;  
```

### ✅ Utiliser des Poids pour Hiérarchiser

```sql
-- ✅ BON : Titre plus important que contenu
UPDATE articles  
SET search_vector =  
    setweight(to_tsvector('french', titre), 'A') ||
    setweight(to_tsvector('french', contenu), 'B');

-- Recherche avec ranking
SELECT titre, ts_rank(search_vector, query) AS score  
FROM articles, to_tsquery('french', 'postgresql') query  
WHERE search_vector @@ query  
ORDER BY score DESC;  
```

### ✅ Gérer les Valeurs NULL

```sql
-- ✅ BON : Protection contre NULL
to_tsvector('french', COALESCE(colonne, ''))

-- ❌ ERREUR : NULL génère une erreur
to_tsvector('french', colonne)  -- si colonne est NULL
```

### ✅ Choisir la Bonne Configuration Linguistique

```sql
-- ✅ BON : Langue appropriée
to_tsvector('french', texte_francais)  
to_tsvector('english', english_text)  

-- ❌ MAUVAIS : Mauvais stemming
to_tsvector('english', texte_francais)  -- Stemming anglais sur français
```

### ✅ Combiner Full-Text avec Filtres Classiques

```sql
-- ✅ BON : Combiner recherche texte + filtres
SELECT * FROM produits  
WHERE  
    search_vector @@ plainto_tsquery('french', 'chaussure')
    AND prix BETWEEN 50 AND 200
    AND categorie = 'Sport';
```

## 10. Performance et Index GIN

### Créer un Index GIN

```sql
-- Index GIN sur tsvector (requis pour la performance)
CREATE INDEX idx_articles_search  
ON articles  
USING GIN(search_vector);  
```

**Sans index GIN** :
```
Seq Scan on articles  (cost=0..150 rows=10)
  Filter: (search_vector @@ to_tsquery(...))
Planning Time: 0.5ms  
Execution Time: 45ms  
```

**Avec index GIN** :
```
Bitmap Index Scan on idx_articles_search  (cost=0..5 rows=10)
  Index Cond: (search_vector @@ to_tsquery(...))
Planning Time: 0.3ms  
Execution Time: 2ms  
```

**Gain** : 20× plus rapide !

### Options d'Index GIN

```sql
-- Index GIN standard
CREATE INDEX idx_standard ON articles USING GIN(search_vector);

-- Index GIN avec fastupdate (mises à jour plus rapides)
CREATE INDEX idx_fastupdate ON articles  
USING GIN(search_vector)  
WITH (fastupdate = on);  

-- Index GIN avec maintenance réduite
CREATE INDEX idx_low_maintenance ON articles  
USING GIN(search_vector)  
WITH (fastupdate = on, gin_pending_list_limit = 4096);  
```

### Maintenance des Index GIN

```sql
-- Statistiques de l'index
SELECT
    schemaname,
    tablename,
    indexname,
    idx_scan,
    idx_tup_read,
    idx_tup_fetch
FROM pg_stat_user_indexes  
WHERE indexname = 'idx_articles_search';  

-- Reconstruire l'index si nécessaire
REINDEX INDEX idx_articles_search;
```

## 11. Cas d'Usage Avancé : Moteur de Recherche Complet

```sql
-- Table complète avec recherche full-text
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    titre VARCHAR(200),
    description TEXT,
    contenu TEXT,
    auteur VARCHAR(100),
    categorie VARCHAR(50),
    tags TEXT[],
    created_at TIMESTAMP DEFAULT NOW(),
    search_vector tsvector
);

-- Trigger pour maintenir search_vector
CREATE FUNCTION documents_search_trigger() RETURNS trigger AS $$  
BEGIN  
    NEW.search_vector :=
        setweight(to_tsvector('french', COALESCE(NEW.titre, '')), 'A') ||
        setweight(to_tsvector('french', COALESCE(NEW.description, '')), 'B') ||
        setweight(to_tsvector('french', COALESCE(NEW.contenu, '')), 'C') ||
        setweight(to_tsvector('french', COALESCE(NEW.auteur, '')), 'D') ||
        setweight(to_tsvector('french', COALESCE(array_to_string(NEW.tags, ' '), '')), 'B');
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER tsvector_update  
BEFORE INSERT OR UPDATE ON documents  
FOR EACH ROW EXECUTE FUNCTION documents_search_trigger();  

-- Index GIN
CREATE INDEX idx_documents_search ON documents USING GIN(search_vector);

-- Fonction de recherche complète avec ranking et highlights
CREATE OR REPLACE FUNCTION search_documents(search_query TEXT)  
RETURNS TABLE (  
    id INT,
    titre VARCHAR,
    description TEXT,
    score REAL,
    extrait TEXT
) AS $$
BEGIN
    RETURN QUERY
    SELECT
        d.id,
        d.titre,
        d.description,
        ts_rank(d.search_vector, query) AS score,
        ts_headline('french', d.contenu, query, 'MaxWords=50, MinWords=25') AS extrait
    FROM
        documents d,
        websearch_to_tsquery('french', search_query) query
    WHERE d.search_vector @@ query
    ORDER BY score DESC
    LIMIT 20;
END;
$$ LANGUAGE plpgsql;

-- Utilisation
SELECT * FROM search_documents('postgresql base données');
```

## 12. Comparaison : LIKE vs Full-Text Search

### Performance

```sql
-- Test avec 100,000 articles

-- Méthode 1 : LIKE (LENT)
EXPLAIN ANALYZE  
SELECT * FROM articles  
WHERE contenu LIKE '%postgresql%';  
-- Seq Scan: 2500ms

-- Méthode 2 : ILIKE (ENCORE PLUS LENT)
EXPLAIN ANALYZE  
SELECT * FROM articles  
WHERE contenu ILIKE '%postgresql%';  
-- Seq Scan: 3200ms

-- Méthode 3 : Full-Text Search (RAPIDE)
EXPLAIN ANALYZE  
SELECT * FROM articles  
WHERE search_vector @@ to_tsquery('french', 'postgresql');  
-- Bitmap Index Scan: 12ms

-- Gain: 200× plus rapide !
```

### Fonctionnalités

| Aspect | LIKE/ILIKE | Full-Text Search |
|--------|------------|------------------|
| **Performance** | ❌ Lent (scan séquentiel) | ✅ Rapide (index GIN) |
| **Indexable** | ⚠️ Partiel (trigram) | ✅ Oui (GIN) |
| **Stemming** | ❌ Non | ✅ Oui |
| **Langues** | ❌ Non | ✅ 20+ langues |
| **Ranking** | ❌ Non | ✅ Oui (pertinence) |
| **Phrases** | ⚠️ Basique | ✅ Avancé (proximité) |
| **Stop words** | ❌ Non | ✅ Oui |
| **Opérateurs** | ⚠️ Basique (%, _) | ✅ Riche (&, \|, !, <->) |
| **Highlights** | ❌ Non | ✅ Oui (ts_headline) |

**Conclusion** : Full-Text Search est supérieur dans presque tous les cas.

## 13. Résumé et Aide-Mémoire

### tsvector : Document Indexé

```sql
-- Créer un tsvector
to_tsvector('french', 'texte à indexer')

-- Avec poids
setweight(to_tsvector('french', 'titre'), 'A')

-- Concaténation
tsvector1 || tsvector2
```

### tsquery : Requête

```sql
-- Requête structurée (opérateurs explicites)
to_tsquery('french', 'mot1 & mot2 | mot3')

-- Texte libre (recommandé pour utilisateurs)
plainto_tsquery('french', 'mots en texte libre')  
websearch_to_tsquery('french', 'syntaxe "google style" -exclusion')  

-- Phrase exacte
phraseto_tsquery('french', 'phrase exacte')
```

### Opérateur de Recherche

```sql
-- Correspondance
tsvector @@ tsquery

-- Exemple complet
to_tsvector('french', 'texte') @@ plainto_tsquery('french', 'recherche')
```

### Opérateurs tsquery

| Opérateur | Signification | Exemple |
|-----------|---------------|---------|
| `&` | AND (ET) | `mot1 & mot2` |
| `\|` | OR (OU) | `mot1 \| mot2` |
| `!` | NOT (NON) | `mot1 & !mot2` |
| `<->` | Suivi immédiat | `mot1 <-> mot2` |
| `<N>` | Proximité (N mots) | `mot1 <2> mot2` |
| `:*` | Préfixe | `post:*` |

### Fonctions Utiles

```sql
-- Ranking (pertinence)
ts_rank(tsvector, tsquery)

-- Extraits avec mots en évidence
ts_headline('french', texte, tsquery)

-- Debug
ts_debug('french', texte)
```

### Pattern Complet

```sql
-- 1. Créer la colonne
ALTER TABLE ma_table ADD COLUMN search_vector tsvector;

-- 2. Remplir
UPDATE ma_table SET search_vector =
    setweight(to_tsvector('french', COALESCE(titre, '')), 'A') ||
    setweight(to_tsvector('french', COALESCE(contenu, '')), 'B');

-- 3. Index GIN
CREATE INDEX idx_search ON ma_table USING GIN(search_vector);

-- 4. Trigger auto-update
CREATE TRIGGER tsvector_update BEFORE INSERT OR UPDATE ON ma_table  
FOR EACH ROW EXECUTE FUNCTION ma_table_search_trigger();  

-- 5. Recherche
SELECT * FROM ma_table  
WHERE search_vector @@ websearch_to_tsquery('french', 'recherche utilisateur')  
ORDER BY ts_rank(search_vector, websearch_to_tsquery('french', 'recherche utilisateur')) DESC;  
```

## Conclusion

Les types **tsvector** et **tsquery** sont au cœur du système Full-Text Search de PostgreSQL. Ils permettent de construire des moteurs de recherche puissants et performants directement dans la base de données, sans dépendre d'outils externes comme Elasticsearch.

**Points clés à retenir** :

1. **tsvector** représente un document préparé pour la recherche
   - Lexèmes normalisés (stemming)
   - Positions conservées
   - Poids optionnels (A, B, C, D)

2. **tsquery** représente une requête de recherche
   - Opérateurs logiques (&, |, !)
   - Opérateurs de proximité (<->, <N>)
   - Préfixes (:*)

3. **Opérateur @@** teste la correspondance  
   - `tsvector @@ tsquery` → boolean

4. **Performance** avec index GIN
   - Requis pour la rapidité
   - 100× à 1000× plus rapide que LIKE

5. **Bonnes pratiques**
   - Colonne tsvector dédiée + trigger
   - websearch_to_tsquery() pour entrées utilisateur
   - Poids pour hiérarchiser (titre > contenu)
   - Index GIN obligatoire

Le Full-Text Search PostgreSQL est une alternative sérieuse aux solutions externes (Elasticsearch, Solr) pour la plupart des cas d'usage, avec l'avantage de la simplicité (tout en SQL) et de la cohérence transactionnelle.
