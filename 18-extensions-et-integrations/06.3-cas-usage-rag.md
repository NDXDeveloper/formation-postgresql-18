# 18.6.3. Cas d'Usage : RAG, Semantic Search

## Introduction

Vous avez probablement déjà utilisé des moteurs de recherche traditionnels qui cherchent des **mots-clés exacts**. Mais que se passe-t-il si vous tapez "comment réparer une fuite d'eau" et que les meilleurs résultats utilisent les termes "colmater canalisation" ou "urgence plomberie" ? Un moteur classique les manquera.

C'est là qu'interviennent deux technologies révolutionnaires rendues possibles par **pgvector** :

1. **Semantic Search (Recherche Sémantique)** : Comprendre le *sens* de votre recherche, pas seulement les mots  
2. **RAG (Retrieval-Augmented Generation)** : Permettre aux IA (comme ChatGPT) d'accéder à vos données privées pour des réponses précises

Ce chapitre explique ces deux cas d'usage majeurs qui transforment PostgreSQL en moteur de recherche intelligent.

### Analogie : Le bibliothécaire expert

**Recherche traditionnelle (mots-clés)** :
```
Vous : "Je cherche un livre sur les dauphins"  
Bibliothécaire robot : *Scan des titres*  
                       "Voici les livres avec 'dauphin' dans le titre"
                       ❌ Manque les livres sur "cétacés" ou "mammifères marins"
```

**Semantic Search (recherche sémantique)** :
```
Vous : "Je cherche un livre sur les dauphins"  
Bibliothécaire expert : *Comprend le sujet*  
                        "Voici des livres sur les dauphins, mais aussi
                         sur les cétacés, la vie marine, et l'intelligence animale"
                        ✅ Comprend l'intention et les concepts liés
```

**RAG (ChatGPT + vos données)** :
```
Vous : "Quelle est notre politique de retour ?"  
ChatGPT seul : "Je ne connais pas votre entreprise spécifique..."  

ChatGPT + RAG : 1. Cherche dans votre base de connaissances interne
                2. Trouve les documents pertinents
                3. "Selon votre politique (v2.3), les retours sont
                   acceptés sous 30 jours avec facture..."
                ✅ Réponse précise basée sur VOS données
```

## Partie 1 : Semantic Search (Recherche Sémantique)

### Qu'est-ce que la Recherche Sémantique ?

La **recherche sémantique** est la capacité de trouver des documents en comprenant leur **signification** plutôt qu'en cherchant des mots-clés exacts.

**Concept clé** : Deux phrases peuvent avoir des mots différents mais le même sens.

```
"J'adore les chats"
≈ "J'aime beaucoup les félins"
≈ "Les chats sont mes animaux préférés"

Ces trois phrases sont sémantiquement similaires,  
même si elles ne partagent que peu de mots.  
```

### Différence avec Recherche Traditionnelle

#### Recherche par mots-clés (Full-Text Search)

```sql
-- PostgreSQL Full-Text Search classique
SELECT * FROM articles  
WHERE to_tsvector('french', contenu) @@ to_tsquery('french', 'Python & programmation');  
```

**Avantages** :
- ✅ Très rapide  
- ✅ Exact (trouve les mots spécifiés)

**Limitations** :
- ❌ Synonymes non compris ("coder" vs "programmer")  
- ❌ Variations linguistiques ("programmation" vs "développement logiciel")  
- ❌ Concepts liés non détectés ("Python" ne trouve pas "langage interprété")

#### Recherche sémantique (pgvector)

```sql
-- Recherche sémantique avec pgvector
SELECT * FROM articles  
ORDER BY embedding <=> query_embedding  
LIMIT 10;  
```

**Avantages** :
- ✅ Comprend les synonymes  
- ✅ Capture les concepts liés  
- ✅ Fonctionne entre langues (multilingue)  
- ✅ Comprend l'intention de recherche

**Limitations** :
- ⚠️ Nécessite génération d'embeddings (API/modèle)  
- ⚠️ Légèrement plus lent que full-text  
- ⚠️ Moins précis pour recherche exacte de mots spécifiques

### Tableau comparatif

| Critère | Full-Text Search | Semantic Search |
|---------|------------------|-----------------|
| **Requête** | "voiture électrique" | "voiture électrique" |
| **Trouve** | Documents avec "voiture" ET "électrique" | Concepts : véhicule électrique, Tesla, mobilité verte, auto écologique |
| **Synonymes** | ❌ Non (sauf dictionnaire) | ✅ Oui (automatique) |
| **Fautes de frappe** | ⚠️ Avec fuzzy search | ✅ Robuste |
| **Multilingue** | ❌ Une langue par index | ✅ Fonctionne entre langues |
| **Vitesse** | 🟢🟢🟢 Ultra-rapide | 🟢🟢 Rapide |
| **Setup** | 🟢 Simple (natif PG) | 🟡 Nécessite modèle embeddings |

#### Étape 3 : Fonction de recherche sémantique

```python
def recherche_semantique(requete, limit=10, seuil_similarite=None):
    """
    Recherche sémantique dans les articles

    Args:
        requete: Question ou mots-clés de recherche
        limit: Nombre de résultats à retourner
        seuil_similarite: Similarité minimale (0-1), optionnel

    Returns:
        Liste de tuples (id, titre, contenu, similarite)
    """

    # Générer l'embedding de la requête
    query_embedding = generer_embedding(requete)

    # Construire la requête SQL
    sql = """
        SELECT
            id,
            titre,
            contenu,
            categorie,
            auteur,
            1 - (embedding <=> %s::vector) AS similarite
        FROM articles
    """

    params = [query_embedding]

    # Ajouter filtre de similarité si spécifié
    if seuil_similarite:
        sql += " WHERE 1 - (embedding <=> %s::vector) >= %s"
        params.append(query_embedding)
        params.append(seuil_similarite)

    sql += """
        ORDER BY embedding <=> %s::vector
        LIMIT %s
    """
    params.extend([query_embedding, limit])

    # Exécuter la requête
    with conn.cursor() as cur:
        cur.execute(sql, params)
        resultats = cur.fetchall()

    return resultats

# Exemple d'utilisation
resultats = recherche_semantique(
    requete="comment optimiser les performances d'une base de données",
    limit=5,
    seuil_similarite=0.7  # Seulement résultats >70% similaires
)

for id, titre, contenu, categorie, auteur, similarite in resultats:
    print(f"\n📄 {titre} (Similarité: {similarite:.2%})")
    print(f"   Auteur: {auteur} | Catégorie: {categorie}")
    print(f"   Aperçu: {contenu[:150]}...")
```

```sql
-- Créer une fonction PostgreSQL pour recherche directe
CREATE OR REPLACE FUNCTION rechercher_kb(
    p_requete TEXT,
    p_limit INTEGER DEFAULT 5
)
RETURNS TABLE (
    titre TEXT,
    contenu TEXT,
    categorie TEXT,
    pertinence NUMERIC
) AS $$
DECLARE
    v_query_embedding vector(1536);
BEGIN
    -- En production, générer l'embedding via extension pl/python
    -- ou récupérer depuis l'application

    RETURN QUERY
    SELECT
        a.titre,
        LEFT(a.contenu, 300) AS contenu,
        a.categorie,
        ROUND((1 - (a.embedding <=> v_query_embedding))::numeric, 3) AS pertinence
    FROM articles a
    ORDER BY a.embedding <=> v_query_embedding
    LIMIT p_limit;
END;
$$ LANGUAGE plpgsql;

-- Utilisation (si embedding pré-calculé)
```

#### 3. Système de tickets / Support

```python
def classifier_ticket(description_ticket):
    """
    Classe automatiquement un ticket de support
    en trouvant les articles de KB les plus similaires
    """

    # Rechercher articles similaires
    resultats = recherche_semantique(description_ticket, limit=3)

    # Extraire les catégories
    categories = [r[3] for r in resultats]  # categorie
    similarites = [r[5] for r in resultats]  # similarite

    # Catégorie la plus probable (pondérée par similarité)
    from collections import Counter
    categorie_ponderee = Counter()
    for cat, sim in zip(categories, similarites):
        categorie_ponderee[cat] += sim

    categorie_suggeree = categorie_ponderee.most_common(1)[0][0]
    confiance = categorie_ponderee[categorie_suggeree] / sum(similarites)

    return {
        'categorie': categorie_suggeree,
        'confiance': confiance,
        'articles_similaires': resultats[:3]
    }

# Exemple
ticket = "Impossible d'insérer des données, erreur de contrainte foreign key"  
classification = classifier_ticket(ticket)  

print(f"Catégorie suggérée: {classification['categorie']}")  
print(f"Confiance: {classification['confiance']:.1%}")  
```

```python
def detecter_doublons(seuil=0.95):
    """Trouve les articles très similaires (possibles doublons)"""

    with conn.cursor() as cur:
        cur.execute("""
            SELECT
                a1.id AS id1,
                a1.titre AS titre1,
                a2.id AS id2,
                a2.titre AS titre2,
                1 - (a1.embedding <=> a2.embedding) AS similarite
            FROM articles a1
            CROSS JOIN articles a2
            WHERE a1.id < a2.id  -- Éviter duplicatas (a1-a2 et a2-a1)
              AND 1 - (a1.embedding <=> a2.embedding) >= %s
            ORDER BY similarite DESC
        """, (seuil,))

        doublons = cur.fetchall()

    return doublons

# Trouver doublons
```

## Partie 2 : RAG (Retrieval-Augmented Generation)

### Qu'est-ce que RAG ?

**RAG** = Retrieval-Augmented Generation (Génération Augmentée par Récupération)

**Définition simple** : Permettre à un modèle d'IA (comme ChatGPT) d'accéder à des informations externes **avant** de générer sa réponse.

### Le Problème sans RAG

**ChatGPT seul** :

```
User: Quelle est notre politique de congés pour 2025 ?

ChatGPT: Je n'ai pas accès aux politiques internes de votre entreprise.
         Je ne peux pas vous aider avec des informations spécifiques
         à votre organisation.
```

**Limitations** :
- ❌ Pas d'accès aux données privées  
- ❌ Connaissances limitées à sa date d'entraînement  
- ❌ Hallucinations possibles (invente des réponses)

### La Solution avec RAG

**ChatGPT + RAG** :

```
1. User: "Quelle est notre politique de congés pour 2025 ?"

2. Système RAG:
   - Génère embedding de la question
   - Recherche dans la base de docs internes (pgvector)
   - Trouve : "Politique RH 2025 - Congés.pdf"

3. Contexte récupéré:
   "Politique de congés 2025 : Chaque employé bénéficie de 25 jours
    de congés payés par an, plus 11 jours fériés..."

4. Prompt à ChatGPT:
   Contexte: [Document ci-dessus]
   Question: Quelle est notre politique de congés pour 2025 ?

5. ChatGPT (avec contexte):
   "Selon votre politique RH 2025, les employés bénéficient de
    25 jours de congés payés annuels, auxquels s'ajoutent 11 jours
    fériés. Les congés peuvent être fractionnés..."
```

**Avantages** :
- ✅ Réponses basées sur VOS données  
- ✅ Toujours à jour (base de données en temps réel)  
- ✅ Moins d'hallucinations (contexte factuel)  
- ✅ Citabilité (sources identifiables)

### Implémentation Complète d'un RAG

#### Étape 1 : Schéma de base de données

```sql
-- Table pour stocker les chunks de documents
CREATE TABLE knowledge_base (
    id SERIAL PRIMARY KEY,
    document_id INTEGER,  -- ID du document source
    document_titre TEXT,
    document_source TEXT,  -- Chemin fichier, URL, etc.
    chunk_index INTEGER,   -- Position du chunk dans le document
    chunk_text TEXT NOT NULL,
    chunk_tokens INTEGER,  -- Nombre de tokens
    embedding vector(1536),
    metadata JSONB,  -- Métadonnées flexibles
    created_at TIMESTAMP DEFAULT NOW()
);

-- Index pour recherche rapide
CREATE INDEX idx_kb_embedding  
ON knowledge_base  
USING hnsw (embedding vector_cosine_ops)  
WITH (m = 16, ef_construction = 64);  

-- Index sur métadonnées pour filtres
CREATE INDEX idx_kb_metadata ON knowledge_base USING gin (metadata);

-- Table pour tracker les documents sources
CREATE TABLE documents_source (
    id SERIAL PRIMARY KEY,
    titre TEXT NOT NULL,
    source_path TEXT NOT NULL,
    type_mime TEXT,
    taille_bytes BIGINT,
    date_ajout TIMESTAMP DEFAULT NOW(),
    derniere_indexation TIMESTAMP,
    statut TEXT DEFAULT 'indexed',  -- indexed, processing, error
```

#### Étape 2 : Chunking et Indexation

```python
import tiktoken  
from typing import List, Dict  

def compter_tokens(texte: str, model="gpt-3.5-turbo") -> int:
    """Compte le nombre de tokens dans un texte"""
    encoding = tiktoken.encoding_for_model(model)
    return len(encoding.encode(texte))

def decouper_en_chunks(
    texte: str,
    taille_chunk: int = 500,
    overlap: int = 50
) -> List[Dict]:
    """
    Découpe un texte en chunks avec overlap

    Args:
        texte: Texte à découper
        taille_chunk: Nombre de tokens par chunk
        overlap: Nombre de tokens de chevauchement entre chunks

    Returns:
        Liste de dictionnaires contenant le texte et métadonnées
    """
    encoding = tiktoken.encoding_for_model("gpt-3.5-turbo")
    tokens = encoding.encode(texte)

    chunks = []
    start = 0
    chunk_index = 0

    while start < len(tokens):
        # Extraire un chunk
        end = min(start + taille_chunk, len(tokens))
        chunk_tokens = tokens[start:end]
        chunk_text = encoding.decode(chunk_tokens)

        chunks.append({
            'text': chunk_text,
            'tokens': len(chunk_tokens),
            'chunk_index': chunk_index,
            'start_token': start,
            'end_token': end
        })

        chunk_index += 1
        start += taille_chunk - overlap  # Avancer avec overlap

    return chunks

def indexer_document(
    titre: str,
    texte: str,
    source: str,
    metadata: Dict = None
) -> int:
    """Indexe un document complet dans la knowledge base"""

    # 1. Enregistrer le document source
    with conn.cursor() as cur:
        cur.execute("""
            INSERT INTO documents_source
            (titre, source_path, metadata, statut)
            VALUES (%s, %s, %s, 'processing')
            RETURNING id
        """, (titre, source, metadata))
        doc_id = cur.fetchone()[0]
    conn.commit()

    try:
        # 2. Découper en chunks
        chunks = decouper_en_chunks(texte, taille_chunk=500, overlap=50)

        # 3. Générer embeddings et insérer
        for chunk in chunks:
            embedding = generer_embedding(chunk['text'])

            with conn.cursor() as cur:
                cur.execute("""
                    INSERT INTO knowledge_base
                    (document_id, document_titre, document_source,
                     chunk_index, chunk_text, chunk_tokens, embedding, metadata)
                    VALUES (%s, %s, %s, %s, %s, %s, %s, %s)
                """, (
                    doc_id,
                    titre,
                    source,
                    chunk['chunk_index'],
                    chunk['text'],
                    chunk['tokens'],
                    embedding,
                    metadata
                ))

        conn.commit()

        # 4. Mettre à jour le statut
        with conn.cursor() as cur:
            cur.execute("""
                UPDATE documents_source
                SET statut = 'indexed',
                    derniere_indexation = NOW()
                WHERE id = %s
            """, (doc_id,))
        conn.commit()

        print(f"✅ Document '{titre}' indexé : {len(chunks)} chunks")
        return doc_id

    except Exception as e:
        # En cas d'erreur, marquer comme erreur
        with conn.cursor() as cur:
            cur.execute("""

                SET statut = 'error'
                WHERE id = %s
            """, (doc_id,))
```

#### Étape 3 : Fonction de Récupération (Retrieval)

```python
def recuperer_contexte(
    question: str,
    k: int = 3,
    seuil_similarite: float = 0.7,
    filtres_metadata: Dict = None
) -> List[Dict]:
    """
    Récupère les chunks les plus pertinents pour une question

    Args:
        question: Question de l'utilisateur
        k: Nombre de chunks à récupérer
        seuil_similarite: Similarité minimale (0-1)
        filtres_metadata: Filtres optionnels (ex: {'categorie': 'Database'})

    Returns:
        Liste de chunks avec métadonnées
    """

    # Générer embedding de la question
    question_embedding = generer_embedding(question)

    # Construire la requête SQL
    sql = """
        SELECT
            id,
            document_titre,
            document_source,
            chunk_text,
            chunk_index,
            metadata,
            1 - (embedding <=> %s::vector) AS similarite
        FROM knowledge_base
        WHERE 1 - (embedding <=> %s::vector) >= %s
    """

    params = [question_embedding, question_embedding, seuil_similarite]

    # Ajouter filtres sur métadonnées
    if filtres_metadata:
        for key, value in filtres_metadata.items():
            sql += f" AND metadata->>{key!r} = %s"
            params.append(value)

    sql += """
        ORDER BY embedding <=> %s::vector
        LIMIT %s
    """
    params.extend([question_embedding, k])

    # Exécuter
    with conn.cursor() as cur:
        cur.execute(sql, params)
        resultats = cur.fetchall()

    # Formater les résultats
    contexte = []
    for row in resultats:
        contexte.append({
            'id': row[0],
            'document': row[1],
            'source': row[2],
            'texte': row[3],
            'chunk_index': row[4],
            'metadata': row[5],
            'similarite': row[6]
        })

    return contexte
```

```python
def generer_reponse_rag(
    question: str,
    k: int = 3,
    modele: str = "gpt-4"
) -> Dict:
    """
    Pipeline RAG complet : Récupération + Génération

    Returns:
        Dict avec réponse, sources et métadonnées
    """

    # 1. RETRIEVAL : Récupérer le contexte pertinent
    contexte = recuperer_contexte(question, k=k)

    if not contexte:
        return {
            'reponse': "Je n'ai pas trouvé d'information pertinente dans la base de connaissances.",
            'sources': [],
            'confiance': 0.0
        }

    # 2. AUGMENTATION : Construire le prompt avec contexte
    contexte_texte = "\n\n---\n\n".join([
        f"Source: {c['document']} (Similarité: {c['similarite']:.1%})\n{c['texte']}"
        for c in contexte
    ])

    prompt = f"""Vous êtes un assistant qui répond aux questions en utilisant UNIQUEMENT le contexte fourni ci-dessous. Si la réponse n'est pas dans le contexte, dites-le clairement.

Contexte:
{contexte_texte}

Question: {question}

Réponse (basée sur le contexte ci-dessus):"""

    # 3. GENERATION : Appeler ChatGPT
    response = openai.chat.completions.create(
        model=modele,
        messages=[
            {
                "role": "system",
                "content": "Tu es un assistant qui répond uniquement en te basant sur le contexte fourni. Tu cites tes sources."
            },
            {
                "role": "user",
                "content": prompt
            }
        ],
        temperature=0.3,  # Réponses plus factuelles
        max_tokens=500
    )

    reponse_text = response.choices[0].message.content

    # 4. Formater le résultat avec sources
    sources_uniques = list({c['document'] for c in contexte})
    similarite_moyenne = sum(c['similarite'] for c in contexte) / len(contexte)

    return {
        'reponse': reponse_text,
        'sources': sources_uniques,
        'contexte': contexte,
        'confiance': similarite_moyenne,
        'nb_chunks_utilises': len(contexte)
    }
```

### 1. Hybrid Search (Combiné Full-Text + Semantic)

```sql
-- Créer index full-text en plus de pgvector
ALTER TABLE knowledge_base ADD COLUMN texte_tsvector tsvector;

UPDATE knowledge_base  
SET texte_tsvector = to_tsvector('french', chunk_text);  

CREATE INDEX idx_kb_fulltext ON knowledge_base USING gin(texte_tsvector);

-- Fonction de recherche hybride
CREATE OR REPLACE FUNCTION recherche_hybride(
    p_query TEXT,
    p_query_embedding vector(1536),
    p_limit INTEGER DEFAULT 10,
    p_weight_semantic FLOAT DEFAULT 0.7,
    p_weight_fulltext FLOAT DEFAULT 0.3
)
RETURNS TABLE (
    id INTEGER,
    chunk_text TEXT,
    score_final FLOAT
) AS $$
BEGIN
    RETURN QUERY
    WITH semantic AS (
        SELECT
            kb.id,
            kb.chunk_text,
            (1 - (kb.embedding <=> p_query_embedding)) AS score_semantic
        FROM knowledge_base kb
        ORDER BY kb.embedding <=> p_query_embedding
        LIMIT p_limit * 2
    ),
    fulltext AS (
        SELECT
            kb.id,
            kb.chunk_text,
            ts_rank(kb.texte_tsvector, to_tsquery('french', p_query)) AS score_fulltext
        FROM knowledge_base kb
        WHERE kb.texte_tsvector @@ to_tsquery('french', p_query)
        ORDER BY score_fulltext DESC
        LIMIT p_limit * 2
    ),
    combined AS (
        SELECT
            COALESCE(s.id, f.id) AS id,
            COALESCE(s.chunk_text, f.chunk_text) AS chunk_text,
            COALESCE(s.score_semantic, 0) * p_weight_semantic +
            COALESCE(f.score_fulltext, 0) * p_weight_fulltext AS score_final
        FROM semantic s
        FULL OUTER JOIN fulltext f ON s.id = f.id
    )
    SELECT
        combined.id,
        combined.chunk_text,
        combined.score_final
    FROM combined
    ORDER BY score_final DESC
    LIMIT p_limit;
END;
$$ LANGUAGE plpgsql;
```

**Avantage** : Combine précision sémantique + correspondance mots-clés exacts

```python
from sentence_transformers import CrossEncoder

# Charger un modèle de re-ranking
reranker = CrossEncoder('cross-encoder/ms-marco-MiniLM-L-6-v2')

def recherche_avec_reranking(question, k_initial=20, k_final=3):
    """
    1. Récupère k_initial candidats avec recherche sémantique (rapide)
    2. Re-rank avec modèle plus précis (lent mais précis)
    3. Retourne top k_final
    """

    # Étape 1 : Récupération rapide
    candidats = recuperer_contexte(question, k=k_initial)

    # Étape 2 : Re-ranking précis
    pairs = [[question, c['texte']] for c in candidats]
    scores = reranker.predict(pairs)

    # Ajouter scores et trier
    for i, candidat in enumerate(candidats):
        candidat['rerank_score'] = float(scores[i])

    candidats_tries = sorted(
        candidats,
        key=lambda x: x['rerank_score'],
        reverse=True
    )

    # Étape 3 : Retourner top k_final
    return candidats_tries[:k_final]
```

```python
import hashlib  
import json  

# Table de cache
"""
CREATE TABLE embedding_cache (
    query_hash TEXT PRIMARY KEY,
    query_text TEXT NOT NULL,
    embedding vector(1536) NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_cache_created ON embedding_cache(created_at);
"""

def generer_embedding_avec_cache(texte):
    """Génère ou récupère un embedding depuis le cache"""

    # Hash du texte pour clé de cache
    query_hash = hashlib.md5(texte.encode()).hexdigest()

    # Vérifier cache
    with conn.cursor() as cur:
        cur.execute(
            "SELECT embedding FROM embedding_cache WHERE query_hash = %s",
            (query_hash,)
        )
        result = cur.fetchone()

        if result:
            # Cache hit
            return result[0]

    # Cache miss : générer
    embedding = generer_embedding(texte)

    # Stocker en cache
    with conn.cursor() as cur:
        cur.execute(
            """
            INSERT INTO embedding_cache (query_hash, query_text, embedding)
            VALUES (%s, %s, %s)
            ON CONFLICT (query_hash) DO NOTHING
            """,
            (query_hash, texte, embedding)
        )
    conn.commit()

    return embedding
```

```python
def generer_reponse_rag_stream(question, k=3):
    """RAG avec streaming de la réponse pour UX temps réel"""

    # Récupération (rapide)
    contexte = recuperer_contexte(question, k=k)

        return

        stream=True,
        temperature=0.3

    for chunk in stream:
```

## Comparaison des Approches

### RAG vs Fine-tuning

| Critère | RAG | Fine-tuning LLM |
|---------|-----|----------------|
| **Coût** | 🟢 Faible (API embeddings + LLM) | 🔴 Élevé (entraînement GPU) |
| **Mise à jour données** | 🟢 Temps réel (ajouter docs) | 🔴 Réentraînement complet |
| **Données sensibles** | 🟢 Restent en DB privée | 🔴 Intégrées au modèle |
| **Transparence** | 🟢 Sources citables | 🟡 Boîte noire |
| **Précision** | 🟢 Excellente (si bons docs) | 🟢 Excellente |
| **Latence** | 🟡 ~1-3s (recherche + génération) | 🟢 ~0.5s (génération seule) |
| **Cas d'usage** | FAQ, docs, support | Ton, style, domaine spécialisé |

**Recommandation** : RAG pour la plupart des cas, fine-tuning pour personnalisation style/ton.

### RAG vs Embeddings seuls (sans LLM)

| Critère | RAG (Embeddings + LLM) | Embeddings seuls |
|---------|----------------------|-----------------|
| **Réponse** | ✅ Texte généré, synthèse | ❌ Chunks bruts |
| **Compréhension** | ✅ Reformulation, clarification | ❌ Texte source tel quel |
| **Citations** | ✅ Sources + explication | ✅ Sources uniquement |
| **Coût** | 🟡 Moyen ($0.002/requête) | 🟢 Faible ($0.0001/requête) |
| **Latence** | 🟡 1-3s | 🟢 0.1-0.5s |
| **Cas d'usage** | Chatbot conversationnel | Recherche documentaire pure |

**Recommandation** :
- Embeddings seuls pour recherche rapide sans génération
- RAG pour interface conversationnelle avec synthèse

## Conclusion

### Points clés à retenir

✅ **Semantic Search** : Comprend le sens, pas seulement les mots-clés
- Utilise embeddings + pgvector
- Trouve synonymes et concepts liés
- Idéal pour : documentation, FAQ, recherche intelligente

✅ **RAG** : ChatGPT + vos données privées
- Architecture : Récupération (pgvector) → Augmentation (prompt) → Génération (LLM)
- Chunking optimal : 500-800 tokens avec overlap
- k = 3-5 chunks pour la plupart des cas

✅ **Avantages RAG** :
- Données toujours à jour (temps réel)
- Transparence (sources citables)
- Coût raisonnable vs fine-tuning
- Pas de hallucinations sur vos données

⚠️ **Considérations** :
- Qualité des embeddings cruciale
- Chunking impacte directement qualité
- Monitoring et évaluation continues nécessaires

🎯 **RAG + pgvector = Chatbot intelligent sur VOS données !** 🚀
