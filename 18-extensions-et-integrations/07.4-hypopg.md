# 18.7.4. HypoPG : Indexation Hypothétique dans PostgreSQL

## Introduction

**HypoPG** (Hypothetical Indexes for PostgreSQL) est une extension PostgreSQL qui permet de créer des **index virtuels** (hypothétiques) pour tester leur impact sur les plans d'exécution des requêtes, **sans réellement créer ces index** sur le disque.

### Le Problème : Tester des Index est Coûteux

Dans PostgreSQL, créer un index est une opération qui :

- ❌ **Prend du temps** : De quelques secondes à plusieurs heures selon la taille de la table  
- ❌ **Bloque les écritures** : Sauf avec `CONCURRENT` (plus lent)  
- ❌ **Consomme de l'espace** : L'index occupe de l'espace disque  
- ❌ **Impacte les performances** : Les écritures (INSERT/UPDATE/DELETE) sont ralenties  
- ❌ **Peut être inutile** : Si l'index n'améliore pas vraiment les performances

**Scénario typique** :

```
Développeur : "Cette requête est lente. Je vais créer un index sur la colonne X."
↓
CREATE INDEX ... (30 minutes sur une grande table)
↓
EXPLAIN ANALYZE ... (vérification)
↓
Résultat : L'index n'est pas utilisé par le planificateur ! 😱
↓
DROP INDEX ... (encore 10 minutes)
↓
Recommencer avec un autre index...
```

**Ce processus est long, risqué et coûteux, surtout en production.**

### La Solution : HypoPG

HypoPG permet de :

- ✅ **Créer des index virtuels** : En quelques millisecondes  
- ✅ **Tester leur impact** : Le planificateur les voit comme de vrais index  
- ✅ **Zéro impact disque** : Aucun espace consommé  
- ✅ **Zéro impact performance** : Pas de ralentissement des écritures  
- ✅ **Tester plusieurs scénarios** : Créer/supprimer instantanément  
- ✅ **Valider avant création** : Créer seulement les index vraiment utiles

**Avec HypoPG** :

```
Développeur : "Cette requête est lente. Je vais TESTER un index avec HypoPG."
↓
SELECT hypopg_create_index(...) (instantané)
↓
EXPLAIN ... (le planificateur voit l'index virtuel)
↓
Résultat : L'index n'est pas utilisé → Pas besoin de le créer ! ✅
↓
Ou : L'index améliore le plan → Le créer réellement
```

### Analogie Simple

Imaginez que vous voulez réorganiser les meubles dans votre maison :

- **Sans HypoPG** : Vous déplacez physiquement tous les meubles (lourd, fatiguant), puis vous réalisez que ça ne marche pas. Vous devez tout remettre et recommencer.

- **Avec HypoPG** : Vous utilisez un logiciel 3D pour tester différentes configurations virtuellement, puis vous ne déplacez physiquement les meubles qu'une seule fois, quand vous êtes sûr de la configuration optimale.

## Concepts Fondamentaux

### Qu'est-ce qu'un Index Hypothétique ?

Un **index hypothétique** est une structure de données **virtuelle** qui :

1. **N'existe que dans les métadonnées** : Stocké en mémoire, pas sur disque  
2. **Est visible par le planificateur** : PostgreSQL le considère dans ses calculs de coût  
3. **N'est PAS utilisable pour les requêtes réelles** : Seulement pour EXPLAIN, pas pour SELECT  
4. **Disparaît à la déconnexion** : Les index virtuels sont liés à la session

**Schéma conceptuel** :

```
┌─────────────────────────────────────────────────┐
│           Session PostgreSQL                    │
│                                                 │
│  ┌───────────────────────────────────────────┐  │
│  │  Tables Réelles                           │  │
│  │  - customers (1M lignes)                  │  │
│  │  - orders (10M lignes)                    │  │
│  └───────────────────────────────────────────┘  │
│              ↓                                  │
│  ┌───────────────────────────────────────────┐  │
│  │  Index Réels (sur disque)                 │  │
│  │  - idx_customers_pk                       │  │
│  │  - idx_orders_customer_id                 │  │
│  └───────────────────────────────────────────┘  │
│              +                                  │
│  ┌───────────────────────────────────────────┐  │
│  │  Index Hypothétiques (en mémoire)         │  │
│  │  - hypoidx_1: orders(order_date)          │  │
│  │  - hypoidx_2: orders(status, amount)      │  │
│  └───────────────────────────────────────────┘  │
│                                                 │
│  ┌───────────────────────────────────────────┐  │
│  │  Planificateur de Requêtes                │  │
│  │  (voit les index réels ET hypothétiques)  │  │
│  └───────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

### Comment le Planificateur Voit les Index Hypothétiques

Le planificateur PostgreSQL utilise des **statistiques** pour estimer le coût des différents plans d'exécution :

- **Nombre de lignes** dans la table  
- **Taille des données** (pages, tuples)  
- **Sélectivité** des colonnes (distribution des valeurs)  
- **Index disponibles** et leur coût d'utilisation

HypoPG **injecte des métadonnées fictives** qui font croire au planificateur qu'un index existe réellement :

```
Planificateur : "Je dois exécuter SELECT * FROM orders WHERE order_date = '2025-01-15'"
                ↓
                "Quels sont mes options ?"
                ↓
┌───────────────────────────────────────────────────────────────┐
│ Option 1: Seq Scan (scan séquentiel de toute la table)        │
│ Coût estimé: 100,000                                          │
└───────────────────────────────────────────────────────────────┘
┌───────────────────────────────────────────────────────────────┐
│ Option 2: Index Scan sur hypoidx_1 (order_date)               │
│ Coût estimé: 150  ← BIEN MOINS CHER !                         │
└───────────────────────────────────────────────────────────────┘
                ↓
                "Je choisis l'Option 2"
```

**Important** : Le planificateur peut voir et choisir l'index hypothétique dans EXPLAIN, mais si vous essayez d'exécuter réellement la requête (SELECT), elle échouera car l'index n'existe pas physiquement.

## Installation et Configuration

### Prérequis

- **PostgreSQL 9.2+** (idéalement PostgreSQL 18)  
- **Droits superutilisateur** : Pour l'installation initiale  
- **Extension disponible** : Selon votre distribution PostgreSQL

### Processus d'Installation (Théorique)

#### Étape 1 : Installer l'extension

HypoPG ne nécessite **pas** d'être ajouté à `shared_preload_libraries` (contrairement à pg_stat_statements).

```sql
-- Se connecter à la base de données cible
CREATE EXTENSION hypopg;
```

C'est tout ! Pas besoin de redémarrage PostgreSQL.

#### Étape 2 : Vérifier l'installation

```sql
-- Vérifier que l'extension est active
SELECT * FROM pg_extension WHERE extname = 'hypopg';

-- Lister les fonctions disponibles
\df hypopg*

-- Fonctions principales :
-- hypopg_create_index()     : Créer un index hypothétique
-- hypopg_drop_index()       : Supprimer un index hypothétique
-- hypopg_list_indexes()     : Lister les index hypothétiques
-- hypopg_reset()            : Supprimer tous les index hypothétiques
-- hypopg_get_indexdef()     : Obtenir la définition d'un index
```

#### Étape 3 : Test rapide

```sql
-- Créer un index hypothétique simple
SELECT hypopg_create_index('CREATE INDEX ON customers(email)');

-- Résultat : Retourne l'OID de l'index virtuel créé
-- Exemple : 12345

-- Vérifier qu'il existe
SELECT * FROM hypopg_list_indexes();

-- Le voir dans un plan d'exécution
EXPLAIN SELECT * FROM customers WHERE email = 'user@example.com';
-- Doit mentionner l'index hypothétique si utilisé
```

### Permissions et Sécurité

HypoPG nécessite les mêmes permissions que pour lire les métadonnées de la base :

```sql
-- Permissions minimales pour utiliser HypoPG
GRANT USAGE ON SCHEMA public TO user_dev;  
GRANT SELECT ON ALL TABLES IN SCHEMA public TO user_dev;  

-- Pour créer l'extension (une seule fois)
-- Nécessite superutilisateur ou droits CREATE sur la base
```

**Note** : Les index hypothétiques sont **par session**. Chaque utilisateur connecté a ses propres index virtuels qui ne sont pas visibles par les autres sessions.

## Utilisation de Base

### Créer un Index Hypothétique

La fonction principale est `hypopg_create_index()`.

**Syntaxe** :

```sql
SELECT hypopg_create_index('définition_index');
```

**Exemples** :

```sql
-- Index simple sur une colonne
SELECT hypopg_create_index('CREATE INDEX ON orders(customer_id)');

-- Index composite (multi-colonnes)
SELECT hypopg_create_index('CREATE INDEX ON orders(customer_id, order_date)');

-- Index avec nom personnalisé
SELECT hypopg_create_index('CREATE INDEX idx_orders_status ON orders(status)');

-- Index partiel (avec WHERE)
SELECT hypopg_create_index('CREATE INDEX ON orders(order_date) WHERE status = ''pending''');

-- Index sur expression
SELECT hypopg_create_index('CREATE INDEX ON customers(LOWER(email))');

-- Index UNIQUE (hypothétique)
SELECT hypopg_create_index('CREATE UNIQUE INDEX ON customers(email)');

-- Index de type spécifique (GIN, GiST, etc.)
SELECT hypopg_create_index('CREATE INDEX ON documents USING GIN(content)');
```

**Retour** : La fonction retourne l'**OID** (identifiant) de l'index hypothétique créé.

```sql
SELECT hypopg_create_index('CREATE INDEX ON orders(order_date)');
-- Retour : 123456
```

### Lister les Index Hypothétiques

```sql
SELECT * FROM hypopg_list_indexes();
```

**Colonnes retournées** :

| Colonne | Type | Description |
|---------|------|-------------|
| `indexrelid` | oid | OID de l'index hypothétique |
| `index_name` | text | Nom de l'index (auto-généré si non spécifié) |
| `schema_name` | text | Schéma de la table |
| `table_name` | text | Nom de la table |
| `am_name` | text | Type d'index (btree, gin, gist, etc.) |

**Exemple de résultat** :

```
indexrelid | index_name              | schema_name | table_name | am_name
-----------+-------------------------+-------------+------------+--------
123456     | <123456>btree_orders    | public      | orders     | btree
123457     | idx_orders_status       | public      | orders     | btree
```

### Obtenir la Définition d'un Index

```sql
SELECT hypopg_get_indexdef(123456);
-- Retour : 'CREATE INDEX ON public.orders USING btree (order_date)'
```

### Tester l'Impact avec EXPLAIN

Une fois un index hypothétique créé, utilisez `EXPLAIN` pour voir si le planificateur l'utilise :

```sql
-- Créer l'index hypothétique
SELECT hypopg_create_index('CREATE INDEX ON orders(order_date)');

-- Tester la requête
EXPLAIN SELECT * FROM orders WHERE order_date = '2025-01-15';
```

**Résultat possible** :

```
Index Scan using <123456>btree_orders on orders
  Index Cond: (order_date = '2025-01-15'::date)
```

**Interprétation** : Le planificateur **utilise** l'index hypothétique `<123456>btree_orders`. Cela signifie que créer cet index réellement **améliorerait** les performances de cette requête.

**Si l'index n'est pas utilisé** :

```
Seq Scan on orders
  Filter: (order_date = '2025-01-15'::date)
```

**Interprétation** : Le planificateur **n'utilise pas** l'index hypothétique. Créer cet index serait **inutile** pour cette requête.

### Supprimer un Index Hypothétique

```sql
-- Supprimer par OID
SELECT hypopg_drop_index(123456);

-- Supprimer TOUS les index hypothétiques de la session
SELECT hypopg_reset();
```

**Important** : Les index hypothétiques sont automatiquement supprimés à la **déconnexion** de la session.

## Workflow Typique d'Utilisation

### Scénario : Optimiser une Requête Lente

**Étape 1 : Identifier une requête problématique**

```sql
-- Via pg_stat_statements ou logs
SELECT query, calls, mean_exec_time  
FROM pg_stat_statements  
ORDER BY mean_exec_time DESC  
LIMIT 10;  

-- Identifier une requête cible
-- Exemple : "SELECT * FROM orders WHERE customer_id = $1 AND status = $2"
```

**Étape 2 : Analyser le plan actuel**

```sql
EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM orders  
WHERE customer_id = 12345  
  AND status = 'pending';
```

**Résultat hypothétique** :

```
Seq Scan on orders  (cost=0.00..500000.00 rows=100 width=200) (actual time=120.5..3500.2 rows=95 loops=1)
  Filter: ((customer_id = 12345) AND (status = 'pending'::text))
  Rows Removed by Filter: 9999905
Planning Time: 0.5 ms  
Execution Time: 3500.8 ms  
```

**Diagnostic** : Scan séquentiel de 10 millions de lignes, très lent (3.5 secondes).

**Étape 3 : Créer un index hypothétique**

```sql
-- Hypothèse : Un index sur (customer_id, status) pourrait aider
SELECT hypopg_create_index('CREATE INDEX ON orders(customer_id, status)');
-- Retour : 123456
```

**Étape 4 : Tester avec l'index hypothétique**

```sql
EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM orders  
WHERE customer_id = 12345  
  AND status = 'pending';
```

**Résultat avec index hypothétique** :

```
Index Scan using <123456>btree_orders on orders  (cost=0.56..150.32 rows=100 width=200) (actual time=0.05..0.25 rows=95 loops=1)
  Index Cond: ((customer_id = 12345) AND (status = 'pending'::text))
Planning Time: 0.6 ms  
Execution Time: 0.3 ms  
```

**Diagnostic** : Le planificateur utilise l'index hypothétique et le temps d'exécution estimé passe de **3500ms à 0.3ms** (amélioration de 99.99%) !

**Étape 5 : Décision**

L'index hypothétique améliore drastiquement les performances → **Le créer réellement**.

```sql
-- Créer l'index réel
CREATE INDEX idx_orders_customer_status ON orders(customer_id, status);
```

**Étape 6 : Valider**

```sql
-- Vérifier que l'index réel est bien utilisé
EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM orders  
WHERE customer_id = 12345  
  AND status = 'pending';

-- Doit montrer l'utilisation de idx_orders_customer_status
```

**Étape 7 : Nettoyer**

```sql
-- Supprimer l'index hypothétique (optionnel, sera supprimé à la déconnexion)
SELECT hypopg_reset();
```

## Cas d'Usage Détaillés

### 1. Comparer Plusieurs Stratégies d'Indexation

**Problème** : Vous ne savez pas quel index créer entre plusieurs options.

```sql
-- Requête problématique
SELECT * FROM orders  
WHERE customer_id = 12345  
  AND order_date >= '2024-01-01'
  AND status = 'shipped';

-- Option 1 : Index sur customer_id seul
SELECT hypopg_create_index('CREATE INDEX idx1 ON orders(customer_id)');

EXPLAIN SELECT * FROM orders  
WHERE customer_id = 12345  
  AND order_date >= '2024-01-01'
  AND status = 'shipped';
-- Analyser le coût estimé : cost=0.56..1500.32

-- Nettoyer
SELECT hypopg_reset();

-- Option 2 : Index composite (customer_id, order_date)
SELECT hypopg_create_index('CREATE INDEX idx2 ON orders(customer_id, order_date)');

EXPLAIN SELECT * FROM orders  
WHERE customer_id = 12345  
  AND order_date >= '2024-01-01'
  AND status = 'shipped';
-- Coût : cost=0.56..850.12 (meilleur !)

SELECT hypopg_reset();

-- Option 3 : Index composite (customer_id, status)
SELECT hypopg_create_index('CREATE INDEX idx3 ON orders(customer_id, status)');

EXPLAIN SELECT * FROM orders  
WHERE customer_id = 12345  
  AND order_date >= '2024-01-01'
  AND status = 'shipped';
-- Coût : cost=0.56..950.45

SELECT hypopg_reset();

-- Option 4 : Index composite complet (customer_id, order_date, status)
SELECT hypopg_create_index('CREATE INDEX idx4 ON orders(customer_id, order_date, status)');

EXPLAIN SELECT * FROM orders  
WHERE customer_id = 12345  
  AND order_date >= '2024-01-01'
  AND status = 'shipped';
-- Coût : cost=0.56..120.22 (le meilleur !)
```

**Conclusion** : L'index composite `(customer_id, order_date, status)` offre le meilleur plan. **Le créer réellement**.

### 2. Tester des Index Partiels

Les **index partiels** (avec `WHERE`) ne couvrent qu'un sous-ensemble de lignes, économisant de l'espace.

```sql
-- Requête fréquente : Chercher les commandes en attente
SELECT * FROM orders WHERE status = 'pending';

-- Hypothèse : Les commandes 'pending' sont rares (1% de la table)
-- Index partiel pour économiser l'espace

SELECT hypopg_create_index('CREATE INDEX ON orders(customer_id) WHERE status = ''pending''');

EXPLAIN SELECT * FROM orders  
WHERE customer_id = 12345  
  AND status = 'pending';

-- Le planificateur utilise-t-il l'index partiel ?
-- Si oui : Index Scan using <123456>btree_orders...
-- Si non : Seq Scan (l'index partiel n'est pas avantageux)
```

**Avantages de l'index partiel** :
- Plus petit (seulement 1% des lignes)
- Maintenance plus rapide (INSERT/UPDATE/DELETE)
- Toujours efficace pour les requêtes ciblées

### 3. Tester des Index sur Expressions

```sql
-- Requête avec recherche insensible à la casse
SELECT * FROM customers WHERE LOWER(email) = 'user@example.com';

-- Sans index : Seq Scan avec calcul de LOWER() sur chaque ligne (lent)

-- Tester un index sur l'expression
SELECT hypopg_create_index('CREATE INDEX ON customers(LOWER(email))');

EXPLAIN SELECT * FROM customers WHERE LOWER(email) = 'user@example.com';

-- Si le planificateur utilise l'index : Amélioration majeure
```

### 4. Tester des Index Couvrants (INCLUDE)

PostgreSQL 11+ supporte les **index couvrants** avec `INCLUDE` (covering index).

```sql
-- Requête qui sélectionne plusieurs colonnes
SELECT customer_id, email, created_at  
FROM customers  
WHERE customer_id = 12345;  

-- Option 1 : Index simple
SELECT hypopg_create_index('CREATE INDEX ON customers(customer_id)');  
EXPLAIN SELECT customer_id, email, created_at FROM customers WHERE customer_id = 12345;  
-- Résultat : Index Scan + récupération des données de la table (heap fetch)

SELECT hypopg_reset();

-- Option 2 : Index couvrant (inclut les colonnes dans l'index)
SELECT hypopg_create_index('CREATE INDEX ON customers(customer_id) INCLUDE (email, created_at)');  
EXPLAIN SELECT customer_id, email, created_at FROM customers WHERE customer_id = 12345;  
-- Résultat : Index Only Scan (plus rapide, pas de heap fetch)
```

**Index Only Scan** : Le planificateur peut récupérer toutes les données nécessaires directement depuis l'index, sans accéder à la table. **Très performant.**

### 5. Analyser l'Impact d'un Index sur Plusieurs Requêtes

HypoPG permet de tester un même index sur plusieurs requêtes différentes.

```sql
-- Créer un index hypothétique
SELECT hypopg_create_index('CREATE INDEX ON orders(order_date)');

-- Tester sur plusieurs requêtes
EXPLAIN SELECT * FROM orders WHERE order_date = '2025-01-15';
-- Utilise l'index ?

EXPLAIN SELECT * FROM orders WHERE order_date >= '2025-01-01' AND order_date < '2025-02-01';
-- Utilise l'index ?

EXPLAIN SELECT COUNT(*) FROM orders WHERE order_date > '2024-01-01';
-- Utilise l'index ?

EXPLAIN SELECT * FROM orders ORDER BY order_date DESC LIMIT 10;
-- Utilise l'index pour éviter le tri ?

-- Si l'index améliore plusieurs requêtes fréquentes : Le créer
-- Si l'index n'améliore qu'une seule requête rare : Reconsidérer
```

### 6. Détecter les Index Redondants

Avant de créer un nouvel index, vérifier qu'il n'est pas **redondant** avec un index existant.

```sql
-- Index existant : idx_orders_customer_id sur (customer_id)

-- Tester un nouvel index composite
SELECT hypopg_create_index('CREATE INDEX ON orders(customer_id, order_date)');

-- Pour une requête filtrant uniquement sur customer_id
EXPLAIN SELECT * FROM orders WHERE customer_id = 12345;

-- Le planificateur choisit-il le nouvel index ou l'ancien ?
-- Si l'ancien suffit : Le nouvel index est inutile pour cette requête
```

**Règle** : Un index composite `(A, B)` peut remplacer un index simple `(A)`, mais l'inverse n'est pas vrai.

## Automatisation et Intégration

### Script d'Analyse Automatisée

Créer un script qui teste automatiquement plusieurs index hypothétiques :

```sql
-- Fonction d'analyse d'index
CREATE OR REPLACE FUNCTION analyze_index_candidates(
    p_table_name TEXT,
    p_columns TEXT[],
    p_query TEXT
)
RETURNS TABLE(
    index_def TEXT,
    estimated_cost NUMERIC,
    uses_index BOOLEAN
) AS $$
DECLARE
    v_index_def TEXT;
    v_plan TEXT;
    v_cost NUMERIC;
BEGIN
    -- Tester chaque colonne individuellement
    FOREACH v_index_def IN ARRAY p_columns LOOP
        -- Créer l'index hypothétique
        PERFORM hypopg_create_index(
            format('CREATE INDEX ON %s(%s)', p_table_name, v_index_def)
        );

        -- Obtenir le plan
        EXECUTE format('EXPLAIN (FORMAT JSON) %s', p_query) INTO v_plan;

        -- Extraire le coût (simplifié)
        v_cost := (v_plan::json->0->'Plan'->>'Total Cost')::numeric;

        -- Retourner le résultat
        RETURN QUERY SELECT
            format('CREATE INDEX ON %s(%s)', p_table_name, v_index_def),
            v_cost,
            v_plan LIKE '%<1234567%';  -- Détection basique d'utilisation d'index hypo

        -- Nettoyer
        PERFORM hypopg_reset();
    END LOOP;
END;
$$ LANGUAGE plpgsql;

-- Utilisation
SELECT * FROM analyze_index_candidates(
    'orders',
    ARRAY['customer_id', 'order_date', 'status', 'customer_id, order_date'],
    'SELECT * FROM orders WHERE customer_id = 12345 AND order_date >= ''2024-01-01'''
)
ORDER BY estimated_cost;
```

### Intégration avec pg_stat_statements

Combiner HypoPG avec pg_stat_statements pour analyser les requêtes lentes :

```sql
-- Créer une vue des requêtes candidates à l'optimisation
CREATE OR REPLACE VIEW optimization_candidates AS  
SELECT  
    queryid,
    LEFT(query, 100) as query_snippet,
    calls,
    mean_exec_time,
    total_exec_time
FROM pg_stat_statements  
WHERE mean_exec_time > 100  -- Plus de 100ms en moyenne  
  AND calls > 100           -- Appelée au moins 100 fois
ORDER BY total_exec_time DESC  
LIMIT 20;  

-- Tester des index hypothétiques sur ces requêtes
-- (processus manuel ou via script d'automatisation)
```

### Générer des Recommandations d'Index

```sql
-- Fonction simple pour recommander des index
CREATE OR REPLACE FUNCTION recommend_indexes(p_query TEXT)  
RETURNS TABLE(recommendation TEXT, reason TEXT) AS $$  
DECLARE  
    v_plan TEXT;
BEGIN
    -- Obtenir le plan actuel
    EXECUTE format('EXPLAIN %s', p_query) INTO v_plan;

    -- Analyser le plan (simplifié)
    IF v_plan LIKE '%Seq Scan%' THEN
        RETURN QUERY SELECT
            'Index recommandé sur les colonnes du WHERE'::TEXT,
            'Scan séquentiel détecté'::TEXT;
    END IF;

    IF v_plan LIKE '%Sort%' THEN
        RETURN QUERY SELECT
            'Index recommandé sur les colonnes du ORDER BY'::TEXT,
            'Tri en mémoire détecté'::TEXT;
    END IF;

    RETURN;
END;
$$ LANGUAGE plpgsql;
```

### Workflow de Test Continu

Intégrer HypoPG dans un pipeline de CI/CD :

```bash
#!/bin/bash
# Script de test d'index hypothétiques (conceptuel)

# 1. Exporter les requêtes lentes depuis production
psql -c "COPY (SELECT query FROM pg_stat_statements WHERE mean_exec_time > 1000) TO '/tmp/slow_queries.txt'"

# 2. Pour chaque requête, tester des index hypothétiques
while read query; do
    psql -c "SELECT hypopg_create_index('...');"
    psql -c "EXPLAIN $query;"
    psql -c "SELECT hypopg_reset();"
done < /tmp/slow_queries.txt

# 3. Générer un rapport de recommandations
# 4. Valider avec l'équipe avant création réelle des index
```

### 1. Toujours Tester Avant de Créer

✅ **Principe** : Ne JAMAIS créer un index en production sans l'avoir testé avec HypoPG.

**Workflow recommandé** :

```
Requête lente identifiée
    ↓
Test avec HypoPG (dev/staging)
    ↓
L'index améliore-t-il le plan ?
    ↓ OUI
Créer l'index réel (d'abord en staging)
    ↓
Valider les performances réelles
    ↓
Déployer en production
```

### 2. Tester sur des Données Réalistes

Les statistiques du planificateur dépendent des **données réelles**.

❌ **Mauvais** : Tester sur une base vide ou avec peu de données  
✅ **Bon** : Tester sur une base avec volume et distribution similaires à la production

**Solution** : Utiliser `pg_dump` / `pg_restore` pour cloner la production en staging, ou utiliser des données synthétiques réalistes.

### 3. Analyser le Coût Total, Pas Seulement l'Utilisation

Un index peut être **utilisé** mais pas forcément **optimal**.

```sql
-- Scénario : Index utilisé mais coûteux
EXPLAIN SELECT * FROM orders WHERE status = 'pending';

-- Avec index hypothétique :
-- Index Scan using <123456>btree_orders (cost=0.56..95000.00 rows=1000000)

-- Sans index :
-- Seq Scan on orders (cost=0.00..50000.00 rows=1000000)
```

**Interprétation** : L'index est **utilisé** mais son coût (95000) est **plus élevé** que le scan séquentiel (50000). Ne pas créer cet index !

**Raison** : La colonne `status` a probablement une faible sélectivité (ex: 50% des lignes sont 'pending'), rendant l'index inefficace.

### 4. Considérer l'Ordre des Colonnes dans les Index Composites

L'ordre des colonnes dans un index composite est **crucial**.

```sql
-- Tester les deux ordres
SELECT hypopg_create_index('CREATE INDEX ON orders(customer_id, order_date)');  
EXPLAIN SELECT * FROM orders WHERE customer_id = 12345 AND order_date >= '2024-01-01';  
-- Coût : 150

SELECT hypopg_reset();

SELECT hypopg_create_index('CREATE INDEX ON orders(order_date, customer_id)');  
EXPLAIN SELECT * FROM orders WHERE customer_id = 12345 AND order_date >= '2024-01-01';  
-- Coût : 450 (moins bon)
```

**Règle générale** : Placer en premier la colonne avec la **meilleure sélectivité** (qui filtre le plus de lignes).

### 5. Ne Pas Oublier l'Impact des Écritures

HypoPG teste uniquement les **lectures** (SELECT). Un index ralentit les **écritures** (INSERT/UPDATE/DELETE).

**Questions à se poser** :

- Cette table est-elle principalement en **lecture** (OLAP) ou en **écriture** (OLTP) ?
- Combien d'écritures par seconde ?
- Le gain en lecture justifie-t-il le coût en écriture ?

**Exemple** :

```
Table : 1 million d'INSERT par jour  
Requête lente : Exécutée 10 fois par jour  

→ L'index ralentira 1 million d'écritures pour accélérer 10 requêtes
→ Probablement pas un bon compromis
→ Envisager plutôt une vue matérialisée ou une table agrégée
```

### 6. Tester les Index UNIQUE

HypoPG permet de tester des index UNIQUE, mais attention :

```sql
SELECT hypopg_create_index('CREATE UNIQUE INDEX ON customers(email)');

EXPLAIN SELECT * FROM customers WHERE email = 'user@example.com';
```

**Important** : HypoPG ne vérifie PAS que les valeurs sont uniques. C'est juste pour tester le plan.

### 7. Nettoyer Régulièrement

Les index hypothétiques consomment de la mémoire (peu, mais quand même).

```sql
-- Supprimer tous les index hypothétiques après chaque session de test
SELECT hypopg_reset();
```

**Note** : Ils sont automatiquement supprimés à la déconnexion.

### 8. Documenter les Décisions

```sql
-- Créer une table de décisions d'indexation
CREATE TABLE monitoring.index_decisions (
    decision_id SERIAL PRIMARY KEY,
    table_name TEXT,
    index_definition TEXT,
    tested_date DATE DEFAULT CURRENT_DATE,
    cost_without_index NUMERIC,

    decision TEXT,  -- 'created', 'rejected', 'pending'
    reason TEXT,
    created_by TEXT DEFAULT CURRENT_USER

-- Documenter chaque décision
```

## Limitations et Considérations

### Limitations de HypoPG

1. **Estimation, pas exécution réelle**
   - HypoPG montre ce que le planificateur **pense** faire
   - L'exécution réelle peut différer (cache, I/O, etc.)
   - Toujours valider avec EXPLAIN ANALYZE après création réelle

2. **Pas de validation des contraintes**
   - Un index UNIQUE hypothétique ne vérifie pas l'unicité
   - Un index partiel ne vérifie pas la condition WHERE

3. **Statistiques limitées**
   - HypoPG génère des statistiques fictives basées sur les statistiques de la table
   - Si les statistiques sont obsolètes (manque d'ANALYZE), HypoPG sera imprécis

4. **Pas de test de maintenance**
   - Impossible de mesurer l'impact réel sur les INSERT/UPDATE/DELETE
   - Estimation théorique uniquement

5. **Session-specific**
   - Les index hypothétiques ne sont visibles que dans la session actuelle
   - Impossible de partager entre plusieurs connexions

6. **Pas de test de fragmentation**
   - Impossible de prédire la fragmentation de l'index dans le temps
   - Impossible de tester le comportement après VACUUM

### Situations où HypoPG est Moins Fiable

1. **Tables sans statistiques récentes**

```sql
-- Vérifier la date du dernier ANALYZE
SELECT
    schemaname,
    tablename,
    last_analyze,
    last_autoanalyze
FROM pg_stat_user_tables  
WHERE tablename = 'orders';  

-- Si last_analyze est ancien ou NULL : Analyser d'abord
ANALYZE orders;
```

2. **Requêtes avec paramètres**

Le planificateur peut choisir différents plans selon les valeurs réelles des paramètres.

```sql
-- Générique
EXPLAIN SELECT * FROM orders WHERE customer_id = $1;

-- Avec valeur spécifique
EXPLAIN SELECT * FROM orders WHERE customer_id = 12345;
```

**Solution** : Tester avec plusieurs valeurs représentatives.

3. **Workload mixte (OLTP + OLAP)**

HypoPG optimise pour une requête spécifique, mais peut dégrader d'autres requêtes.

**Solution** : Tester l'index sur un **panel représentatif** de requêtes.

## Dépannage (Troubleshooting)

### Problème 1 : L'Index Hypothétique n'est Pas Utilisé

**Symptômes** : Le plan EXPLAIN ne mentionne pas l'index hypothétique.

**Causes possibles** :

1. **Index inapproprié**
   - La requête ne filtre pas sur les colonnes de l'index
   - L'ordre des colonnes est incorrect (index composite)

2. **Statistiques obsolètes**

```sql
-- Mettre à jour les statistiques
ANALYZE orders;

-- Recréer l'index hypothétique et retester
SELECT hypopg_reset();  
SELECT hypopg_create_index('...');  
EXPLAIN ...;  
```

3. **Coût estimé de l'index > Seq Scan**

Le planificateur choisit le scan séquentiel car moins coûteux.

```sql
-- Vérifier le coût estimé
EXPLAIN (FORMAT JSON) SELECT * FROM orders WHERE status = 'pending';

-- Si le coût du Seq Scan est plus bas : L'index n'est pas avantageux
```

4. **Type d'index inadapté**

```sql
-- Mauvais : B-tree pour LIKE '%pattern%'
SELECT hypopg_create_index('CREATE INDEX ON products(name)');  
EXPLAIN SELECT * FROM products WHERE name LIKE '%widget%';  
-- N'utilisera pas l'index (scan infixe)

-- Bon : GIN avec pg_trgm pour LIKE '%pattern%'
CREATE EXTENSION pg_trgm;  
SELECT hypopg_create_index('CREATE INDEX ON products USING GIN(name gin_trgm_ops)');  
EXPLAIN SELECT * FROM products WHERE name LIKE '%widget%';  
```

### Problème 2 : Résultats Contradictoires (HypoPG vs Réalité)

**Symptômes** : HypoPG prédit un gain, mais l'index réel n'améliore pas les performances.

**Causes** :

1. **Statistiques incorrectes**

```sql
-- Augmenter les cibles statistiques pour plus de précision
ALTER TABLE orders ALTER COLUMN customer_id SET STATISTICS 1000;  
ANALYZE orders;  
```

2. **Cache effects**

HypoPG ne modélise pas précisément le cache. Un index peut être moins utile si les données sont déjà en cache.

3. **I/O réel vs estimé**

Le planificateur estime l'I/O, mais le comportement réel dépend du hardware (SSD vs HDD).

**Solution** : Toujours valider avec `EXPLAIN (ANALYZE, BUFFERS)` après création réelle.

### Problème 3 : HypoPG Ralentit les EXPLAIN

**Symptômes** : Les commandes EXPLAIN sont lentes avec beaucoup d'index hypothétiques.

**Cause** : Trop d'index hypothétiques créés (> 50).

**Solution** :

```sql
-- Lister et compter
SELECT COUNT(*) FROM hypopg_list_indexes();

-- Nettoyer
SELECT hypopg_reset();
```

### Problème 4 : Extension Non Disponible

**Symptômes** : `ERROR: extension "hypopg" is not available`.

**Cause** : HypoPG n'est pas installé sur le serveur.

**Solution** :

- **Distributions** : Installer le package (ex: `postgresql-18-hypopg` sur Debian/Ubuntu)  
- **Compilation** : Compiler depuis les sources (GitHub)  
- **Cloud managé** : Vérifier la disponibilité (RDS, Azure, GCP)

**Note** : HypoPG n'est pas toujours disponible sur les services cloud managés.

## Comparaison avec d'Autres Approches

### HypoPG vs Créer/Tester/Supprimer Réellement

| Critère | HypoPG | Création Réelle |
|---------|--------|----------------|
| **Temps** | Millisecondes | Minutes à heures |
| **Espace disque** | Aucun | Consomme de l'espace |
| **Impact production** | Aucun | Risque (locks, I/O) |
| **Précision** | Estimation | Résultat réel |
| **Test écritures** | Non | Oui |
| **Multiples tests** | Rapide et facile | Lent et risqué |

**Verdict** : HypoPG pour les tests, création réelle pour la validation finale.

### HypoPG vs Index Advisors (outils externes)

Certains outils proposent des recommandations d'index automatiques :

- **pgAdmin** : Index advisor intégré  
- **pgTune** : Suggestions de configuration  
- **Outils commerciaux** : Datadog, New Relic, etc.

| Critère | HypoPG | Index Advisors |
|---------|--------|----------------|
| **Contrôle** | Total (manuel) | Automatique |
| **Personnalisation** | Complète | Limitée |
| **Complexité** | Moyenne | Faible |
| **Coût** | Gratuit (open-source) | Variable (souvent payant) |

**Verdict** : HypoPG pour contrôle fin, Index Advisors pour recommandations rapides.

### HypoPG vs Auto-Indexing (future feature ?)

PostgreSQL n'a pas (encore) d'auto-indexing natif, mais des discussions existent dans la communauté.

**Hypothétique Auto-Indexing** :
- Le système crée/supprime automatiquement des index selon la charge
- Utiliserait HypoPG en arrière-plan pour tester

**Pour l'instant** : HypoPG reste un outil manuel, nécessitant intervention humaine.

## Cas d'Usage Avancés

### 1. Tester des Stratégies d'Index pour du Sharding

Avant de partitionner/sharding une table, tester les index appropriés.

```sql
-- Table avant partitionnement
CREATE TABLE events (
    event_id BIGSERIAL,
    event_date DATE,
    user_id INTEGER,
    event_type VARCHAR(50),
    data JSONB
);

-- Stratégie 1 : Index global sur event_date
SELECT hypopg_create_index('CREATE INDEX ON events(event_date)');  
EXPLAIN SELECT * FROM events WHERE event_date = '2025-01-15';  

-- Stratégie 2 : Index composite (event_date, user_id)
SELECT hypopg_reset();  
SELECT hypopg_create_index('CREATE INDEX ON events(event_date, user_id)');  
EXPLAIN SELECT * FROM events WHERE event_date = '2025-01-15' AND user_id = 12345;  

-- Choisir la stratégie avant de partitionner
```

### 2. Optimiser des Requêtes Full-Text Search

```sql
-- Requête de recherche textuelle
SELECT * FROM articles WHERE to_tsvector('english', content) @@ to_tsquery('postgresql & indexing');

-- Tester un index GIN
SELECT hypopg_create_index('CREATE INDEX ON articles USING GIN(to_tsvector(''english'', content))');

EXPLAIN SELECT * FROM articles WHERE to_tsvector('english', content) @@ to_tsquery('postgresql & indexing');
```

### 3. Analyser l'Impact sur des Requêtes de Reporting

```sql
-- Requête d'agrégation complexe
SELECT
    DATE_TRUNC('month', order_date) as month,
    customer_id,
    SUM(amount) as total
FROM orders  
WHERE order_date >= '2024-01-01'  
GROUP BY DATE_TRUNC('month', order_date), customer_id;  

-- Tester un index pour cette requête
SELECT hypopg_create_index('CREATE INDEX ON orders(order_date, customer_id, amount)');

EXPLAIN SELECT ...;
```

## Intégration avec l'Écosystème PostgreSQL

### HypoPG + pg_stat_statements

```sql
-- Vue combinée : Requêtes lentes + Recommandations d'index
CREATE OR REPLACE VIEW slow_queries_with_recommendations AS  
SELECT  
    queryid,
    LEFT(query, 100) as query_snippet,
    calls,
    mean_exec_time,
    -- Extraire les colonnes du WHERE (simplifié)
    regexp_matches(query, 'WHERE\s+(\w+)', 'g') as potential_index_columns
FROM pg_stat_statements  
WHERE mean_exec_time > 100  
ORDER BY total_exec_time DESC  
LIMIT 20;  
```

### HypoPG + auto_explain

Combiner avec auto_explain pour capturer automatiquement les plans des requêtes lentes.

```ini
# postgresql.conf
shared_preload_libraries = 'auto_explain'  
auto_explain.log_min_duration = 1000  
auto_explain.log_analyze = on  
auto_explain.log_buffers = on  
```

Ensuite, analyser les logs et tester des index hypothétiques sur les requêtes identifiées.

### HypoPG + pgBadger

pgBadger peut extraire des recommandations d'index depuis les logs. Utiliser HypoPG pour valider ces recommandations.

```bash
# Générer un rapport pgBadger
pgbadger /var/log/postgresql/*.log -o report.html

# Identifier les "Queries that could benefit from indexes"
# Tester ces index avec HypoPG
```

## Résumé

**HypoPG** est un outil indispensable pour optimiser les performances PostgreSQL en testant des index **sans les créer physiquement**.

### Points Clés

- ✅ **Test rapide** : Créer/supprimer des index virtuels en millisecondes  
- ✅ **Zéro impact** : Pas d'espace disque, pas de ralentissement  
- ✅ **Décisions éclairées** : Créer uniquement les index réellement utiles  
- ✅ **Comparaison facile** : Tester plusieurs stratégies d'indexation  
- ✅ **Production-safe** : Tester sans risque

### Quand Utiliser HypoPG ?

- **Optimisation de requêtes lentes** : Identifier le meilleur index  
- **Comparaison de stratégies** : Tester plusieurs options d'indexation  
- **Validation avant création** : S'assurer qu'un index sera utilisé  
- **Analyse d'impact** : Comprendre comment un index modifie le plan  
- **Formation et apprentissage** : Comprendre le comportement du planificateur

### Workflow Recommandé

```
1. Identifier une requête lente (pg_stat_statements, logs)
2. Analyser le plan actuel (EXPLAIN)
3. Créer un index hypothétique (hypopg_create_index)
4. Tester le nouveau plan (EXPLAIN avec index virtuel)
5. Comparer les coûts (avec/sans index)
6. Créer l'index réel si bénéfique
7. Valider avec EXPLAIN ANALYZE
8. Monitorer dans le temps
```

### Métriques de Succès

Un index hypothétique est **candidat à la création réelle** si :

- ✅ **Coût réduit** : Le coût estimé diminue significativement (> 50%)  
- ✅ **Index utilisé** : Le planificateur choisit l'index dans le plan  
- ✅ **Sélectivité suffisante** : L'index filtre efficacement (pas de low selectivity)  
- ✅ **Multiple requêtes** : Améliore plusieurs requêtes fréquentes  
- ✅ **Balance lecture/écriture** : Le gain en lecture justifie le coût en écriture

3. Tester des index hypothétiques pour chaque requête  
4. Comparer les plans et les coûts  
5. Créer les index réellement bénéfiques  
6. Documenter vos décisions d'indexation
