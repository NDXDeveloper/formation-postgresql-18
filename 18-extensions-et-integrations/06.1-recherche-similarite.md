# 18.6.1. Recherche de Similarité (cosine, L2)

## Introduction

Imaginez que vous voulez trouver des produits "similaires" à un produit donné, ou des documents qui "ressemblent" à une recherche, ou encore recommander des films qu'un utilisateur pourrait aimer. Comment faire ?

La **recherche de similarité** (similarity search) est la capacité de trouver des éléments "proches" ou "similaires" à un élément de référence, en se basant sur leurs caractéristiques.

Avec **pgvector**, PostgreSQL peut stocker et rechercher des **vecteurs** (représentations mathématiques d'objets) et calculer leur similarité ultra-rapidement.

### Analogie : La bibliothèque magique

Imaginez une bibliothèque où chaque livre est représenté par sa "position" dans un espace imaginaire :
- Les livres de science-fiction sont regroupés dans une zone
- Les romans d'amour dans une autre zone
- Les livres techniques dans encore une autre

Quand vous demandez "trouve-moi des livres comme celui-ci", le bibliothécaire mesure la **distance** entre votre livre et tous les autres, puis vous donne les plus proches.

**pgvector** fait exactement ça, mais avec des mathématiques précises et à grande vitesse ! 🚀

## Qu'est-ce qu'un Vecteur ?

### Définition simple

Un **vecteur** est une liste ordonnée de nombres qui représente un objet dans un espace mathématique.

**Exemples** :

```
Vecteur 2D (2 dimensions) : [3.5, 2.1]  
Vecteur 3D (3 dimensions) : [1.0, 4.2, -2.5]  
Vecteur 1536D (1536 dimensions) : [0.023, -0.891, 0.445, ..., 0.234]  
                                    ↑ Typique pour OpenAI embeddings
```

### Visualisation (2D et 3D)

**Vecteurs 2D** (faciles à visualiser) :

```
        Y
        ↑
      5 |           • (4, 4)
      4 |         /
      3 |       /
      2 |     • (2, 2)
      1 |   /
      0 +--+--+--+--+--→ X
        0  1  2  3  4  5

Point A = [2, 2]  
Point B = [4, 4]  

Ces deux points sont "proches" dans l'espace 2D.
```

**Vecteurs 3D** :

```
      Z
      ↑
      |     • (3, 4, 2)
      |    /|
      |   / |
      |  /  • (1, 2, 1)
      | /   |
      +-----+----→ Y
     /
    /
   ↓
   X
```

**En réalité, nous travaillons souvent avec des vecteurs à 100, 768, 1536 dimensions** (impossibles à visualiser, mais le principe est le même).

### Embeddings : De texte à vecteur

Un **embedding** est la transformation d'un objet (texte, image, son) en vecteur numérique.

**Exemple avec du texte** :

```
Phrase : "J'aime les chats"
         ↓ Modèle d'embedding (par ex. OpenAI text-embedding-3-small)
Vecteur : [0.023, -0.154, 0.387, ..., 0.892]  (1536 dimensions)
```

**Magie** : Des phrases similaires auront des vecteurs proches !

```
"J'aime les chats"        → [0.023, -0.154, 0.387, ...]
"J'adore les félins"      → [0.029, -0.148, 0.391, ...]  ← Proche !
"Le temps est nuageux"    → [-0.892, 0.445, -0.234, ...]  ← Éloigné
```

## Extension pgvector

### Qu'est-ce que pgvector ?

**pgvector** est une extension PostgreSQL open-source qui ajoute :
- Un type de données `vector` pour stocker des vecteurs
- Des opérateurs de calcul de distance/similarité
- Des index spécialisés pour recherche rapide

**Développée par** : Ankane (Andrew Kane)  
**Licence** : Open-source (PostgreSQL License)  
**Popularité** : Utilisée par Supabase, Neon, et des milliers d'applications IA

## Métriques de Distance et Similarité

### Concept : Distance vs Similarité

**Distance** : Mesure de l'éloignement entre deux vecteurs
- Plus la distance est **petite**, plus les vecteurs sont **proches/similaires**
- Exemple : Distance = 0.1 (très proches), Distance = 5.0 (très éloignés)

**Similarité** : Mesure de la ressemblance entre deux vecteurs
- Plus la similarité est **élevée**, plus les vecteurs sont **similaires**
- Exemple : Similarité = 0.95 (très similaires), Similarité = 0.2 (peu similaires)

### Les 3 principales métriques

pgvector supporte trois métriques principales :

| Métrique | Opérateur pgvector | Type | Plage | Usage principal |
|----------|-------------------|------|-------|----------------|
| **Distance L2 (euclidienne)** | `<->` | Distance | [0, ∞) | Général, images |
| **Distance cosine** | `<=>` | Distance | [0, 2] | Texte, embeddings |
| **Produit scalaire (inner product)** | `<#>` | Similarité | (-∞, ∞) | Recommandation |

## 1. Distance L2 (Euclidienne)

### Définition

La **distance euclidienne** (ou distance L2) est la distance "en ligne droite" entre deux points dans l'espace.

**Formule mathématique** :

```
Pour deux vecteurs A = [a₁, a₂, ..., aₙ] et B = [b₁, b₂, ..., bₙ]

Distance L2(A, B) = √[(a₁-b₁)² + (a₂-b₂)² + ... + (aₙ-bₙ)²]
```

### Visualisation 2D

```
        Y
        ↑
      5 |           • B (4, 4)
      4 |         /|
      3 |       / |
      2 |     /   |
      1 |   • A   |
        |  (2,1)  |
      0 +--+------+----→ X
        0  2      4

Distance = √[(4-2)² + (4-1)²]
        = √[2² + 3²]
        = √[4 + 9]
        = √13
        ≈ 3.61
```

### Propriétés

✅ **Intuitive** : Correspond à notre notion de "distance physique"

✅ **Symétrique** : `distance(A, B) = distance(B, A)`

✅ **Triangle inequality** : `distance(A, C) ≤ distance(A, B) + distance(B, C)`

⚠️ **Sensible à la magnitude** : Les vecteurs de grande magnitude ont plus d'influence

### Utilisation avec pgvector

**Opérateur** : `<->`

```sql
-- Trouver les 5 produits les plus proches du produit #1
SELECT
    id,
    nom,
    embedding <-> (SELECT embedding FROM produits WHERE id = 1) AS distance
FROM produits  
WHERE id != 1  -- Exclure le produit lui-même  
ORDER BY distance ASC  
LIMIT 5;  
```

**Résultat exemple** :

```
 id |        nom         | distance
----+--------------------+----------
  3 | Clavier mécanique  |   0.234
  7 | Écran 27 pouces    |   0.456
 12 | Webcam HD          |   0.589
  5 | Hub USB            |   0.634
 18 | Casque audio       |   0.712
```

### Exemple complet

```sql
-- Créer une table de films avec embeddings
CREATE TABLE films (
    id SERIAL PRIMARY KEY,
    titre TEXT,
    genre TEXT,
    embedding vector(128)  -- 128 dimensions pour l'exemple
);

-- Insérer des films (embeddings générés par un modèle IA)
INSERT INTO films (titre, genre, embedding) VALUES
    ('Inception', 'Sci-Fi', '[0.1, 0.8, 0.3, ..., 0.5]'),
    ('Interstellar', 'Sci-Fi', '[0.12, 0.79, 0.31, ..., 0.52]'),
    ('Le Seigneur des Anneaux', 'Fantasy', '[0.9, 0.2, 0.1, ..., 0.3]'),
    ('Harry Potter', 'Fantasy', '[0.88, 0.21, 0.12, ..., 0.31]'),
    ('Titanic', 'Romance', '[0.2, 0.1, 0.9, ..., 0.8]');

-- Rechercher les films similaires à "Inception"
SELECT
    titre,
    genre,
    embedding <-> (SELECT embedding FROM films WHERE titre = 'Inception') AS distance
FROM films  
WHERE titre != 'Inception'  
ORDER BY distance ASC  
LIMIT 3;  
```

**Résultat** :

```
       titre        |  genre  | distance
--------------------+---------+----------
 Interstellar       | Sci-Fi  |   0.087   ← Très proche (même genre)
 Le Seigneur...     | Fantasy |   1.234
 Harry Potter       | Fantasy |   1.289
```

### Cas d'usage de L2

✅ **Images** : Recherche d'images similaires (embeddings CNN)

✅ **Données numériques** : Comparaison de profils utilisateurs avec features numériques

✅ **Géospatial** : Recherche de points géographiques proches (coordonnées)

✅ **Général** : Quand la magnitude des vecteurs est importante

## 2. Distance Cosine

### Définition

La **distance cosine** mesure l'angle entre deux vecteurs, **indépendamment de leur magnitude**.

**Concept clé** : Deux vecteurs pointant dans la même direction sont considérés comme similaires, même si l'un est "plus long" que l'autre.

### Visualisation 2D

```
        Y
        ↑
      8 |         • B [4, 8]
      7 |        /
      6 |       /
      5 |      /
      4 |     /   • A [2, 4]
      3 |    /   /
      2 |   /   /
      1 |  /   /
      0 +--+--+----------→ X
        0  2  4

A = [2, 4]  
B = [4, 8] = 2 × A  

Angle entre A et B = 0° (même direction)
→ Similarité cosine = 1.0 (parfaitement similaires)
→ Distance cosine = 0.0
```

**Même direction, magnitude différente = similaires pour cosine !**

### Formule mathématique

**Similarité cosine** :

```
similarity_cosine(A, B) = (A · B) / (||A|| × ||B||)

Où :
- A · B = produit scalaire (somme des produits des composantes)
- ||A|| = norme (longueur) du vecteur A
- Résultat : -1 (opposés) à +1 (identiques)
```

**Distance cosine** (utilisée par pgvector) :

```
distance_cosine(A, B) = 1 - similarity_cosine(A, B)

Résultat : 0 (identiques) à 2 (opposés)
```

### Exemple de calcul

**Vecteurs** :
- A = [1, 2, 3]
- B = [2, 4, 6] (= 2 × A)

**Calcul** :

```
1. Produit scalaire : A · B = (1×2) + (2×4) + (3×6) = 2 + 8 + 18 = 28

2. Normes :
   ||A|| = √(1² + 2² + 3²) = √14 ≈ 3.742
   ||B|| = √(2² + 4² + 6²) = √56 ≈ 7.483

3. Similarité cosine :
   cos(θ) = 28 / (3.742 × 7.483) = 28 / 28 = 1.0

4. Distance cosine :
   distance = 1 - 1.0 = 0.0  ← Parfaitement similaires
```

### Propriétés

✅ **Indépendant de la magnitude** : Seule la direction compte

✅ **Normalisé** : Toujours entre 0 et 2

✅ **Idéal pour texte** : Les embeddings de texte bénéficient de cette propriété

⚠️ **Moins intuitif** : La notion de "distance" est moins évidente

### Utilisation avec pgvector

**Opérateur** : `<=>`

```sql
-- Recherche de similarité sémantique de documents
SELECT
    id,
    titre,
    contenu,
    embedding <=> (SELECT embedding FROM documents WHERE id = 1) AS distance_cosine
FROM documents  
WHERE id != 1  
ORDER BY distance_cosine ASC  
LIMIT 5;  
```

### Exemple complet : Recherche sémantique

```sql
-- Créer une table de documents avec embeddings
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    titre TEXT,
    contenu TEXT,
    embedding vector(1536)  -- OpenAI text-embedding-3-small
);

-- Fonction pour générer embedding (fictive, en réalité via API OpenAI)
-- En pratique, vous utiliseriez l'API OpenAI ou un modèle local

-- Insérer des documents
INSERT INTO documents (titre, contenu, embedding) VALUES
    (
        'Python pour débutants',
        'Apprendre Python est facile. Ce langage est idéal pour commencer la programmation.',
        '[0.023, -0.154, 0.387, ..., 0.892]'  -- Embedding réel via OpenAI
    ),
    (
        'Introduction à Python',
        'Python est un langage de programmation accessible aux débutants.',
        '[0.029, -0.148, 0.391, ..., 0.885]'  -- Similaire au premier
    ),
    (
        'Recette de cookies',
        'Pour faire des cookies, mélanger beurre, sucre et farine.',
        '[-0.892, 0.445, -0.234, ..., 0.123]'  -- Très différent
    );

-- Requête : "Trouver des documents similaires à une recherche"
-- Supposons que nous avons l'embedding de la recherche "tutoriel programmation Python"

WITH recherche AS (
    SELECT '[0.025, -0.152, 0.389, ..., 0.888]'::vector(1536) AS query_embedding
)
SELECT
    d.id,
    d.titre,
    LEFT(d.contenu, 80) AS apercu,
    d.embedding <=> recherche.query_embedding AS distance
FROM documents d, recherche  
ORDER BY distance ASC  
LIMIT 3;  
```

**Résultat** :

```
 id |         titre          |                      apercu                       | distance
----+------------------------+---------------------------------------------------+----------
  1 | Python pour débutants  | Apprendre Python est facile. Ce langage est...    |   0.012
  2 | Introduction à Python  | Python est un langage de programmation access... |   0.015
  3 | Recette de cookies     | Pour faire des cookies, mélanger beurre...        |   0.892
```

Les deux premiers documents (sur Python) sont très proches sémantiquement de la recherche !

### Pourquoi cosine pour le texte ?

**Problème avec L2 sur du texte** :

```
Document A : "chat"
  → Embedding : [0.5, 0.3]
  → Norme : 0.58

Document B : "chat chat chat chat chat"  (5 fois "chat")
  → Embedding : [2.5, 1.5]  (5× plus grand)
  → Norme : 2.92

Distance L2 : Très grande (à cause de la magnitude)
→ Considérés comme DIFFÉRENTS

Distance Cosine : Proche de 0 (même direction)
→ Considérés comme SIMILAIRES ✅
```

**Conclusion** : Pour le texte, la fréquence des mots (magnitude) ne devrait pas influencer la similarité sémantique.

### Cas d'usage de cosine

✅ **Recherche sémantique** : Documents, articles, questions-réponses

✅ **Embeddings de texte** : OpenAI, Sentence Transformers, etc.

✅ **Recommandation** : Profils utilisateurs, préférences

✅ **Classification** : Trouver la catégorie la plus proche

✅ **RAG (Retrieval-Augmented Generation)** : Récupération de contexte pour LLM

## 3. Produit Scalaire (Inner Product)

### Définition

Le **produit scalaire** (dot product ou inner product) mesure à quel point deux vecteurs "pointent dans la même direction" ET ont une grande magnitude.

**Formule** :

```
A · B = a₁×b₁ + a₂×b₂ + ... + aₙ×bₙ
```

### Différence avec cosine

- **Cosine** : Ignore la magnitude (normalise)  
- **Inner product** : Prend en compte la magnitude

**Analogie** :
- Cosine : "Ces deux personnes ont les mêmes goûts ?"
- Inner product : "Ces deux personnes ont les mêmes goûts ET sont très passionnées ?"

### Utilisation avec pgvector

**Opérateur** : `<#>`

**Note** : pgvector retourne `-1 * inner_product` (négation) pour que l'ordre ASC donne les plus similaires.

```sql
-- Recommandation de produits
SELECT
    id,
    nom,
    embedding <#> '[0.1, 0.2, 0.3, ..., 0.9]'::vector AS neg_inner_product
FROM produits  
ORDER BY neg_inner_product ASC  -- Les plus négatifs = plus grands inner products = plus similaires  
LIMIT 10;  
```

### Cas d'usage

✅ **Systèmes de recommandation** : Combinaison de préférences + intensité

✅ **Ranking** : Classer par pertinence avec poids

⚠️ Moins utilisé que L2 et cosine pour la recherche de similarité standard

## Comparaison des Métriques

### Tableau récapitulatif

| Critère | L2 (Euclidienne) | Cosine | Inner Product |
|---------|------------------|--------|---------------|
| **Opérateur pgvector** | `<->` | `<=>` | `<#>` |
| **Sensibilité magnitude** | ✅ Oui | ❌ Non | ✅ Oui |
| **Plage de valeurs** | [0, ∞) | [0, 2] | (-∞, ∞) |
| **Normalisé** | ❌ Non | ✅ Oui | ❌ Non |
| **Idéal pour texte** | ⚠️ Moyen | ✅✅ Excellent | ⚠️ Moyen |
| **Idéal pour images** | ✅✅ Excellent | ✅ Bon | ⚠️ Moyen |
| **Rapidité calcul** | 🟡 Moyen | 🟢 Rapide | 🟢 Rapide |
| **Usage principal** | Général | Embeddings texte | Recommandation |

### Choix de la métrique : Arbre de décision

```
Vous travaillez avec des embeddings de texte ?
│
├─ Oui → Utilisez COSINE (<=>)
│         ✅ OpenAI, Sentence Transformers, etc.
│
└─ Non
    │
    ├─ Images / CNN embeddings ?
    │   │
    │   └─ Oui → Utilisez L2 (<->)
    │
    └─ Système de recommandation avec poids ?
        │
        └─ Oui → Considérez INNER PRODUCT (<#>)
             Sinon → Utilisez L2 (<->)
```

### Benchmark comparatif

**Scénario** : 1 million de vecteurs de 1536 dimensions

| Métrique | Temps de recherche | Précision | Cas d'usage |
|----------|-------------------|-----------|-------------|
| **L2** | 45 ms | Haute | Images, données numériques |
| **Cosine** | 38 ms | Haute | Texte, sémantique |
| **Inner Product** | 35 ms | Moyenne | Recommandation avec scores |

## Exemples Pratiques Complets

### Exemple 1 : Recherche sémantique de produits

```sql
-- Créer une table de produits e-commerce
CREATE TABLE produits_ecommerce (
    id SERIAL PRIMARY KEY,
    nom TEXT NOT NULL,
    description TEXT,
    categorie TEXT,
    prix NUMERIC(10, 2),
    embedding vector(1536)  -- Embedding de la description
);

-- Insérer des produits (embeddings générés via OpenAI API)
INSERT INTO produits_ecommerce (nom, description, categorie, prix, embedding) VALUES
    (
        'MacBook Pro 16"',
        'Ordinateur portable haute performance pour professionnels créatifs',
        'Informatique',
        2499.99,
        '[0.023, -0.154, 0.387, ..., 0.892]'
    ),
    (
        'Dell XPS 15',
        'PC portable puissant pour développeurs et designers',
        'Informatique',
        1899.99,
        '[0.029, -0.148, 0.391, ..., 0.885]'
    ),
    (
        'Chaise ergonomique',
        'Siège de bureau confortable pour longues sessions de travail',
        'Mobilier',
        349.99,
        '[-0.234, 0.567, -0.123, ..., 0.445]'
    );

-- Fonction de recherche (supposons que nous avons l'embedding de la recherche)
-- En production, vous appelleriez l'API OpenAI pour obtenir l'embedding

-- Recherche : "ordinateur portable pour développeur"
WITH recherche AS (
    SELECT '[0.025, -0.152, 0.389, ..., 0.888]'::vector(1536) AS query_emb
)
SELECT
    p.id,
    p.nom,
    p.description,
    p.categorie,
    p.prix,
    (p.embedding <=> recherche.query_emb) AS distance_cosine,
    (1 - (p.embedding <=> recherche.query_emb)) AS similarite_cosine
FROM produits_ecommerce p, recherche  
ORDER BY distance_cosine ASC  
LIMIT 5;  
```

```

  1 | MacBook Pro 16"| Ordinateur portable haute performance...   | Informatique| 2499.99 |     0.023      |      0.977
  3 | Chaise ergo... | Siège de bureau confortable...             | Mobilier   |  349.99 |     0.892      |      0.108
```

        'Compte',

    ),

    ),
    (
        'Quels sont les modes de paiement acceptés ?',
        'Nous acceptons les cartes bancaires (Visa, Mastercard), PayPal et virement bancaire.',
        'Paiement',
        '[-0.892, 0.445, -0.234, ..., 0.567]'
    );

-- Fonction de recherche dans la FAQ
CREATE OR REPLACE FUNCTION rechercher_faq(
    question_utilisateur TEXT,
    limite INTEGER DEFAULT 3
)
RETURNS TABLE (
    question TEXT,
    reponse TEXT,
    categorie TEXT,
    pertinence NUMERIC
) AS $$
DECLARE
    query_embedding vector(1536);
BEGIN
    -- En production, appel API pour obtenir l'embedding
    -- Pour cet exemple, simulons un embedding
    query_embedding := '[0.124, -0.454, 0.790, ..., 0.236]'::vector(1536);

    RETURN QUERY
    SELECT
        f.question,
        f.reponse,
        f.categorie,
        ROUND((1 - (f.embedding <=> query_embedding))::numeric, 3) AS pertinence
    FROM faq f
    ORDER BY f.embedding <=> query_embedding ASC
    LIMIT limite;
END;
$$ LANGUAGE plpgsql;

-- Utilisation
SELECT * FROM rechercher_faq('mot de passe perdu');
```

```

### Exemple 3 : Détection de doublons

```sql
-- Détecter des articles en double (même contenu, titres différents)
WITH doublons AS (
    SELECT
        a1.id AS id1,
        a1.titre AS titre1,
        a2.id AS id2,
        a2.titre AS titre2,
        a1.embedding <=> a2.embedding AS distance
    FROM articles a1
    CROSS JOIN articles a2
    WHERE a1.id < a2.id  -- Éviter les duplicatas (a1-a2 et a2-a1)
      AND a1.embedding <=> a2.embedding < 0.05  -- Seuil de similarité très élevé
)
SELECT * FROM doublons  
ORDER BY distance ASC;  
```

**Résultat** :

```
 id1 |         titre1          | id2 |         titre2          | distance
-----+-------------------------+-----+-------------------------+----------
  12 | PostgreSQL Guide        |  34 | Guide complet PostgreSQL|   0.012
  45 | Python Tips             |  78 | Astuces Python          |   0.018
  89 | Docker Introduction     | 123 | Intro à Docker          |   0.023
```

Articles potentiellement en double détectés !

### Exemple 4 : Recommandation de contenu

```sql
-- Recommander des articles à un utilisateur basé sur son historique
CREATE TABLE lectures_utilisateur (
    utilisateur_id INTEGER,
    article_id INTEGER,
    date_lecture TIMESTAMP DEFAULT NOW()
);

-- Recommander des articles similaires à ceux qu'un utilisateur a lus
WITH articles_lus AS (
    SELECT a.embedding
    FROM lectures_utilisateur lu
    INNER JOIN articles a ON lu.article_id = a.id
    WHERE lu.utilisateur_id = 42
      AND lu.date_lecture > NOW() - INTERVAL '30 days'
),
vecteur_moyen AS (
    -- Calculer le vecteur moyen des préférences de l'utilisateur
    SELECT AVG(embedding) AS profil_utilisateur
    FROM articles_lus
)
SELECT
    a.id,
    a.titre,
    a.auteur,
    a.embedding <=> vm.profil_utilisateur AS distance
FROM articles a, vecteur_moyen vm  
WHERE a.id NOT IN (  
    SELECT article_id FROM lectures_utilisateur WHERE utilisateur_id = 42
)
ORDER BY distance ASC  
LIMIT 10;  
```

**Résultat** : Les 10 articles les plus pertinents pour l'utilisateur 42.

## Optimisation des Performances

### 1. Sans index : Scan complet (lent)

```sql
-- Recherche sans index
SELECT * FROM documents  
ORDER BY embedding <=> '[0.1, 0.2, ..., 0.9]'::vector(1536)  
LIMIT 10;  

-- EXPLAIN montre : Seq Scan sur documents
-- Temps : 15 secondes pour 1M de vecteurs
```

**Problème** : PostgreSQL doit calculer la distance avec TOUS les vecteurs.

### 2. Index IVFFlat (rapide)

```sql
-- Créer un index IVFFlat
CREATE INDEX idx_documents_embedding_ivfflat  
ON documents  
USING ivfflat (embedding vector_cosine_ops)  
WITH (lists = 100);  -- Nombre de clusters  

-- Après création de l'index
SET ivfflat.probes = 10;  -- Nombre de clusters à rechercher

-- Même recherche
SELECT * FROM documents  
ORDER BY embedding <=> '[0.1, 0.2, ..., 0.9]'::vector(1536)  
LIMIT 10;  

-- EXPLAIN montre : Index Scan using idx_documents_embedding_ivfflat
-- Temps : 0.05 secondes pour 1M de vecteurs
```

**Amélioration** : 300× plus rapide ! 🚀

### 3. Index HNSW (le plus rapide)

```sql
-- Créer un index HNSW (Hierarchical Navigable Small World)
CREATE INDEX idx_documents_embedding_hnsw  
ON documents  
USING hnsw (embedding vector_cosine_ops)  
WITH (m = 16, ef_construction = 64);  

-- Paramètres de requête
SET hnsw.ef_search = 40;

-- Recherche ultra-rapide
SELECT * FROM documents  
ORDER BY embedding <=> '[0.1, 0.2, ..., 0.9]'::vector(1536)  
LIMIT 10;  

-- Temps : 0.01 secondes pour 1M de vecteurs
```

**Amélioration** : 1500× plus rapide ! 🚀🚀

### Comparaison des index

| Méthode | Temps recherche (1M vecteurs) | Précision | Espace disque | Construction |
|---------|------------------------------|-----------|---------------|--------------|
| **Sans index** | 15 s | 100% | 0 | Immédiat |
| **IVFFlat** | 0.05 s | ~95% | Moyen | Rapide |
| **HNSW** | 0.01 s | ~98% | Élevé | Lent |

**Recommandation** :
- **HNSW** pour production (meilleur compromis vitesse/précision)  
- **IVFFlat** si contraintes d'espace ou construction rapide nécessaire

WITH nouveau_ticket AS (

## Intégration avec l'IA

### Workflow typique avec OpenAI

```
┌──────────────────────────────────────────────────────────────┐
│  1. Générer embeddings (Application)                         │
│     - Python, Node.js, etc.                                  │
│     - Appel API OpenAI text-embedding-3-small                │
│     - Résultat : vecteur de 1536 dimensions                  │
└───────────────┬──────────────────────────────────────────────┘
                │
                ↓ INSERT INTO documents (texte, embedding)
┌───────────────▼──────────────────────────────────────────────┐
│  2. Stocker dans PostgreSQL (pgvector)                       │
│     - Colonne type vector(1536)                              │
│     - Index HNSW pour recherche rapide                       │
└───────────────┬──────────────────────────────────────────────┘
                │
                ↓ Question utilisateur
┌───────────────▼──────────────────────────────────────────────┐
│  3. Rechercher (Application)                                 │
│     - Générer embedding de la question (OpenAI)              │
│     - SELECT ... ORDER BY embedding <=> question_emb         │
│     - Récupérer top K documents                              │
└───────────────┬──────────────────────────────────────────────┘
                │
                ↓ Contexte + Question
┌───────────────▼──────────────────────────────────────────────┐
│  4. Générer réponse (ChatGPT)                                │
│     - Prompt : Contexte + Question                           │
│     - API OpenAI Chat Completions                            │
│     - Réponse finale à l'utilisateur                         │
└──────────────────────────────────────────────────────────────┘
```

### Exemple de code Python

```python
import openai  
import psycopg2  
from psycopg2.extras import execute_values  

# Configuration
openai.api_key = "votre_clé_api"  
conn = psycopg2.connect("postgresql://user:pass@localhost/dbname")  

# 1. Générer embedding d'un texte
def get_embedding(text, model="text-embedding-3-small"):
    response = openai.embeddings.create(
        input=text,
        model=model
    )
    return response.data[0].embedding

# 2. Stocker dans PostgreSQL
def ajouter_document(titre, contenu):
    embedding = get_embedding(contenu)

    with conn.cursor() as cur:
        cur.execute(
            """
            INSERT INTO documents (titre, contenu, embedding)
            VALUES (%s, %s, %s)
            """,
            (titre, contenu, embedding)
        )
    conn.commit()

# 3. Rechercher des documents similaires
def rechercher(question, limit=3):
    question_emb = get_embedding(question)

    with conn.cursor() as cur:
        cur.execute(
            """
            SELECT
                titre,
                contenu,
                1 - (embedding <=> %s::vector) AS similarite
            FROM documents
            ORDER BY embedding <=> %s::vector
            LIMIT %s
            """,
            (question_emb, question_emb, limit)
        )
        return cur.fetchall()

# 4. RAG : Récupération + Génération
def chatbot_rag(question):
    # Récupérer contexte pertinent
    documents = rechercher(question, limit=3)

    # Construire le contexte
    contexte = "\n\n".join([doc[1] for doc in documents])

    # Générer réponse avec ChatGPT
    response = openai.chat.completions.create(
        model="gpt-4",
        messages=[
            {"role": "system", "content": "Tu es un assistant qui répond en utilisant le contexte fourni."},
            {"role": "user", "content": f"Contexte:\n{contexte}\n\nQuestion: {question}"}
        ]
    )

    return response.choices[0].message.content

# Utilisation
reponse = chatbot_rag("Comment fonctionne pgvector ?")  
print(reponse)  
```

## Limitations et Considérations

### 1. Taille des vecteurs

⚠️ **Impact mémoire et stockage** :

```
Table avec 1M de vecteurs de 1536 dimensions :
- Stockage : ~6 Go (sans compression)
- Index HNSW : ~12-15 Go supplémentaires
- RAM nécessaire : 20-30 Go pour performances optimales
```

### 2. Précision vs Performance

⚠️ **Trade-off avec les index** :

- **IVFFlat** : ~95% de précision, très rapide  
- **HNSW** : ~98% de précision, ultra-rapide

Vous ne récupérerez pas toujours les résultats théoriquement les plus proches (approximation).

### 3. Coût des embeddings

⚠️ **API OpenAI** :

- text-embedding-3-small : $0.02 / 1M tokens
- text-embedding-3-large : $0.13 / 1M tokens

Pour 100 000 documents de 500 mots → ~$1-$7

**Alternative** : Modèles open-source locaux (Sentence Transformers, etc.)

### 4. Mises à jour d'index

⚠️ **Reconstruction partielle** :

Les index HNSW et IVFFlat peuvent nécessiter une reconstruction périodique pour maintenir les performances après de nombreuses insertions/suppressions.

```sql
-- Reconstruire un index
REINDEX INDEX idx_documents_embedding_hnsw;
```

## Conclusion

### Points clés à retenir

✅ **pgvector** permet la recherche de similarité directement dans PostgreSQL

✅ Trois métriques principales :  
- **L2 (euclidienne)** `<->` : Images, données numériques  
- **Cosine** `<=>` : Texte, embeddings sémantiques (le plus utilisé)  
- **Inner product** `<#>` : Recommandation avec poids

✅ La **distance cosine** est idéale pour les embeddings de texte (OpenAI, etc.)

✅ Les **index HNSW** offrent les meilleures performances (1000× plus rapide que sans index)

✅ Cas d'usage principaux : Recherche sémantique, RAG, recommandation, classification

⚠️ Choisir la bonne métrique selon vos données (texte → cosine, images → L2)

⚠️ Dimensionner correctement (RAM, stockage) pour les gros volumes

🎯 **pgvector** transforme PostgreSQL en base de données vectorielle performante pour l'IA !
