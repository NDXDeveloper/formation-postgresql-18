# 18.4.2. file_fdw, oracle_fdw, mysql_fdw

## Introduction

Dans le chapitre précédent, nous avons découvert **postgres_fdw** qui permet de connecter PostgreSQL à d'autres bases PostgreSQL. Mais PostgreSQL peut aussi se connecter à d'autres types de sources de données grâce aux **Foreign Data Wrappers (FDW)** spécialisés.

Ce chapitre couvre trois FDW importants :
- **file_fdw** : Lire des fichiers CSV, texte ou autres formats de fichiers  
- **oracle_fdw** : Connecter PostgreSQL à des bases de données Oracle  
- **mysql_fdw** : Connecter PostgreSQL à des bases de données MySQL/MariaDB

### Pourquoi ces FDW sont-ils utiles ?

Imaginez ces scénarios :

📁 **file_fdw** : Vous recevez quotidiennement des fichiers CSV de vos partenaires et vous voulez les interroger directement avec SQL sans les importer.

🔶 **oracle_fdw** : Votre entreprise migre d'Oracle vers PostgreSQL, mais certaines applications utilisent encore Oracle. Vous voulez accéder aux données Oracle depuis PostgreSQL.

🐬 **mysql_fdw** : Vous consolidez des données depuis plusieurs systèmes MySQL vers un data warehouse PostgreSQL.

## Partie 1 : file_fdw - Interroger des Fichiers

### Qu'est-ce que file_fdw ?

**file_fdw** est un FDW fourni nativement avec PostgreSQL qui permet de lire des fichiers comme s'ils étaient des tables PostgreSQL. C'est particulièrement utile pour :
- Fichiers CSV
- Fichiers texte délimités (TSV, pipes, etc.)
- Logs structurés
- Exports de données

### Schéma conceptuel

```
┌────────────────────────────────────┐
│    PostgreSQL Server               │
│                                    │
│  ┌──────────────────────────────┐  │
│  │  SELECT * FROM               │  │
│  │  fichier_ventes;             │  │
│  └──────────┬───────────────────┘  │
│             │                      │
│  ┌──────────▼───────────────────┐  │
│  │  Foreign Table               │  │
│  │  (définition de structure)   │  │
│  └──────────┬───────────────────┘  │
│             │                      │
│  ┌──────────▼───────────────────┐  │
│  │  file_fdw                    │  │
│  └──────────┬───────────────────┘  │
└─────────────┼──────────────────────┘
              │ Lecture système de fichiers
              │
┌─────────────▼───────────────────────┐
│  /data/ventes.csv                   │
│  id,produit,quantite,prix           │
│  1,Laptop,2,1200.00                 │
│  2,Souris,10,25.00                  │
└─────────────────────────────────────┘
```

### Installation et Configuration

#### Étape 1 : Activer l'extension

```sql
CREATE EXTENSION file_fdw;
```

#### Étape 2 : Créer un Foreign Server

Même pour les fichiers locaux, il faut déclarer un "serveur" :

```sql
CREATE SERVER serveur_fichiers
    FOREIGN DATA WRAPPER file_fdw;
```

**Note** : Contrairement à postgres_fdw, il n'y a pas d'options de connexion car les fichiers sont locaux.

#### Étape 3 : Créer une Foreign Table pointant vers un fichier

**Exemple : Fichier CSV simple**

Supposons que vous avez un fichier `/data/ventes.csv` :
```csv
id,produit,quantite,prix
1,Laptop,2,1200.00
2,Souris,10,25.00
3,Clavier,5,75.00
```

Créez la foreign table :

```sql
CREATE FOREIGN TABLE ventes_csv (
    id INTEGER,
    produit TEXT,
    quantite INTEGER,
    prix NUMERIC(10, 2)
)
SERVER serveur_fichiers  
OPTIONS (  
    filename '/data/ventes.csv',
    format 'csv',
    header 'true',           -- Le fichier a une ligne d'en-tête
    delimiter ',',           -- Séparateur (virgule)
    null ''                  -- Représentation des NULL (chaîne vide)
);
```

#### Étape 4 : Interroger le fichier

```sql
-- Lire toutes les données
SELECT * FROM ventes_csv;

-- Filtrer
SELECT produit, prix  
FROM ventes_csv  
WHERE quantite > 5;  

-- Agréger
SELECT SUM(quantite * prix) AS chiffre_affaires  
FROM ventes_csv;  
```

### Options de Configuration

#### Options principales de file_fdw

| Option | Description | Exemple |
|--------|-------------|---------|
| `filename` | Chemin absolu du fichier | `/data/ventes.csv` |
| `format` | Format du fichier | `csv`, `text` |
| `header` | Le fichier a une ligne d'en-tête | `true`, `false` |
| `delimiter` | Caractère séparateur | `,` (CSV), `\t` (TSV), `|` (pipe) |
| `quote` | Caractère d'échappement des chaînes | `"` (guillemet double) |
| `escape` | Caractère d'échappement | `\` (backslash) |
| `null` | Représentation des valeurs NULL | `''` (vide), `NULL`, `\N` |
| `encoding` | Encodage du fichier | `UTF8`, `LATIN1`, `WIN1252` |

#### Exemples de configurations

**Fichier TSV (Tab-Separated Values)** :

```sql
CREATE FOREIGN TABLE logs_tsv (
    timestamp TIMESTAMP,
    niveau TEXT,
    message TEXT
)
SERVER serveur_fichiers  
OPTIONS (  
    filename '/var/log/application.tsv',
    format 'text',
    delimiter E'\t',    -- Tabulation (\t avec E pour escape)
    null 'NULL'
);
```

**Fichier avec séparateur pipe** :

```sql
CREATE FOREIGN TABLE donnees_pipe (
    id INTEGER,
    nom TEXT,
    valeur NUMERIC
)
SERVER serveur_fichiers  
OPTIONS (  
    filename '/data/export.txt',
    format 'text',
    delimiter '|',
    header 'false'
);
```

**Fichier avec encodage spécifique** :

```sql
CREATE FOREIGN TABLE donnees_latines (
    id INTEGER,
    nom TEXT
)
SERVER serveur_fichiers  
OPTIONS (  
    filename '/data/legacy.csv',
    format 'csv',
    encoding 'LATIN1'    -- Pour les anciens systèmes
);
```

### Cas d'usage réels

#### 1. Intégration de fichiers externes

**Contexte** : Vos partenaires vous envoient quotidiennement des fichiers CSV de commandes.

**Solution** :

```sql
-- Créer la foreign table
CREATE FOREIGN TABLE commandes_partenaire (
    commande_id TEXT,
    date_commande DATE,
    montant NUMERIC
)
SERVER serveur_fichiers  
OPTIONS (  
    filename '/imports/commandes_2025-11-23.csv',
    format 'csv',
    header 'true'
);

-- Importer dans la base locale
INSERT INTO commandes_locales (commande_id, date_commande, montant)  
SELECT * FROM commandes_partenaire  
WHERE date_commande >= CURRENT_DATE - INTERVAL '7 days';  
```

**Script automatisé** : Vous pourriez créer une fonction PL/pgSQL qui :
1. Détecte les nouveaux fichiers  
2. Crée dynamiquement la foreign table  
3. Importe les données  
4. Archive le fichier

#### 2. Analyse de logs

**Contexte** : Analyser des logs d'application directement avec SQL.

**Fichier de log** (`/var/log/app.log`) :
```
2025-11-23 10:15:32|INFO|Utilisateur connecté: user123
2025-11-23 10:16:45|ERROR|Échec connexion base de données
2025-11-23 10:17:01|INFO|Transaction validée: 89234
```

```sql
CREATE FOREIGN TABLE logs_app (
    timestamp TIMESTAMP,
    niveau TEXT,
    message TEXT
)
SERVER serveur_fichiers  
OPTIONS (  
    filename '/var/log/app.log',
    format 'text',
    delimiter '|'
);

-- Compter les erreurs par heure
SELECT
    DATE_TRUNC('hour', timestamp) AS heure,
    COUNT(*) AS nb_erreurs
FROM logs_app  
WHERE niveau = 'ERROR'  
GROUP BY heure  
ORDER BY heure DESC;  
```

#### 3. Validation de données avant import

```sql
-- Vérifier la qualité des données
SELECT
    COUNT(*) AS total_lignes,
    COUNT(DISTINCT commande_id) AS commandes_uniques,
    COUNT(*) FILTER (WHERE montant IS NULL) AS montants_null,
    MIN(montant) AS montant_min,
    MAX(montant) AS montant_max
FROM commandes_partenaire;

-- Détecter les anomalies
SELECT *  
FROM commandes_partenaire  
WHERE montant < 0 OR montant > 100000;  
```

### Limitations de file_fdw

❌ **Lecture seule** : file_fdw est en **lecture seule**. Vous ne pouvez pas faire de `INSERT`, `UPDATE` ou `DELETE`.

```sql
-- ❌ Erreur
INSERT INTO ventes_csv VALUES (4, 'Écran', 3, 450.00);
-- ERROR: cannot insert into foreign table "ventes_csv"
```

❌ **Fichiers locaux uniquement** : Les fichiers doivent être accessibles sur le serveur PostgreSQL (pas sur la machine cliente).

❌ **Permissions** : L'utilisateur PostgreSQL (souvent `postgres`) doit avoir les droits de lecture sur le fichier.

```bash
# Donner les permissions de lecture
chmod 644 /data/ventes.csv  
chown postgres:postgres /data/ventes.csv  
```

❌ **Pas d'index** : Les requêtes sur foreign tables file_fdw font toujours un scan séquentiel (pas d'optimisation par index).

❌ **Performance** : Pour de gros fichiers (>100 Mo), les performances peuvent être médiocres. Privilégiez `COPY` pour l'import.

### Comparaison : file_fdw vs COPY

| Critère | file_fdw | COPY |
|---------|----------|------|
| Vitesse | ⚠️ Moyen | ✅ Très rapide |
| Persistance | ❌ Données non stockées | ✅ Données importées dans PostgreSQL |
| Filtrage SQL avant import | ✅ Oui | ❌ Non (import tout) |
| Transformations | ✅ Avec SELECT | ⚠️ Nécessite post-traitement |
| Cas d'usage | Analyse ponctuelle, validation | Import massif de données |

**Recommandation** :
- Utilisez **file_fdw** pour analyser ou valider avant import
- Utilisez **COPY** pour des imports volumineux en production

## Partie 2 : oracle_fdw - Connexion à Oracle

### Qu'est-ce que oracle_fdw ?

**oracle_fdw** est un FDW tiers qui permet à PostgreSQL de se connecter à des bases de données Oracle et d'interroger leurs tables comme si elles étaient locales.

### Pourquoi utiliser oracle_fdw ?

🔄 **Migration progressive** : Migrer d'Oracle vers PostgreSQL sans interruption de service

📊 **Reporting consolidé** : Agréger des données depuis Oracle et PostgreSQL dans un seul rapport

🔗 **Intégration hybride** : Applications utilisant PostgreSQL qui doivent accéder à des données Oracle legacy

### Prérequis

Avant d'utiliser oracle_fdw, vous devez installer :

1. **Oracle Instant Client** : Bibliothèques Oracle pour se connecter à Oracle  
2. **oracle_fdw** : L'extension PostgreSQL (non fournie par défaut)

**Installation sur Linux (exemple Ubuntu)** :

```bash
# 1. Télécharger Oracle Instant Client depuis le site Oracle
# (nécessite un compte Oracle gratuit)

# 2. Installer les packages
sudo apt-get install libaio1  
sudo dpkg -i oracle-instantclient-basic_*.deb  
sudo dpkg -i oracle-instantclient-sqlplus_*.deb  
sudo dpkg -i oracle-instantclient-devel_*.deb  

# 3. Configurer les variables d'environnement
export ORACLE_HOME=/usr/lib/oracle/21/client64  
export LD_LIBRARY_PATH=$ORACLE_HOME/lib  

# 4. Installer oracle_fdw (depuis les sources ou package)
sudo apt-get install postgresql-16-oracle-fdw
```

### Configuration

#### Étape 1 : Créer l'extension

```sql
CREATE EXTENSION oracle_fdw;
```

#### Étape 2 : Créer le Foreign Server

```sql
CREATE SERVER serveur_oracle
    FOREIGN DATA WRAPPER oracle_fdw
    OPTIONS (
        dbserver '//oracle-prod.entreprise.com:1521/ORCL'  -- Connexion Oracle (TNS ou Easy Connect)
    );
```

**Format de dbserver** :
- **Easy Connect** : `//host:port/service_name`  
- **TNS** : `tns_alias` (défini dans tnsnames.ora)

Exemple avec TNS :
```sql
OPTIONS (dbserver 'PROD_DB')  -- Référence à l'alias dans tnsnames.ora
```

#### Étape 3 : Créer le User Mapping

```sql
CREATE USER MAPPING FOR utilisateur_postgresql
    SERVER serveur_oracle
    OPTIONS (
        user 'admin_oracle',        -- Utilisateur Oracle
        password 'mot_de_passe'     -- Mot de passe Oracle
    );
```

#### Étape 4 : Créer les Foreign Tables

**Méthode manuelle** :

```sql
CREATE FOREIGN TABLE clients_oracle (
    client_id NUMBER,
    nom VARCHAR2(100),
    email VARCHAR2(100),
    date_creation DATE
)
SERVER serveur_oracle  
OPTIONS (  
    schema 'SALES',         -- Schéma Oracle
    table 'CUSTOMERS'       -- Table Oracle
);
```

**Méthode automatique (IMPORT)** :

```sql
-- Importer toutes les tables d'un schéma Oracle
IMPORT FOREIGN SCHEMA "SALES"
    FROM SERVER serveur_oracle
    INTO public;

-- Importer uniquement certaines tables
IMPORT FOREIGN SCHEMA "SALES"
    LIMIT TO (CUSTOMERS, ORDERS)
    FROM SERVER serveur_oracle
    INTO public;
```

**⚠️ Note** : Les noms de schéma Oracle doivent être en **MAJUSCULES** et entre guillemets doubles.

### Utilisation

#### Requêtes de lecture

```sql
-- Sélection simple
SELECT * FROM clients_oracle;

-- Jointure avec table PostgreSQL locale
SELECT
    c.nom,
    p.montant
FROM clients_oracle c  
INNER JOIN paiements_postgresql p ON c.client_id = p.client_id;  

-- Agrégation
SELECT
    COUNT(*) AS nb_clients,
    AVG(montant_total) AS moyenne
FROM clients_oracle  
WHERE date_creation >= DATE '2024-01-01';  
```

#### Requêtes d'écriture

oracle_fdw supporte les écritures :

```sql
-- Insertion
INSERT INTO clients_oracle (client_id, nom, email, date_creation)  
VALUES (1001, 'Nouveau Client', 'client@exemple.com', SYSDATE);  

-- Mise à jour
UPDATE clients_oracle  
SET email = 'nouveau@exemple.com'  
WHERE client_id = 1001;  

-- Suppression
DELETE FROM clients_oracle WHERE client_id = 1001;
```

### Mapping des Types de Données

PostgreSQL et Oracle ont des types de données différents. oracle_fdw fait la conversion automatiquement :

| Type Oracle | Type PostgreSQL | Notes |
|-------------|-----------------|-------|
| `NUMBER` | `NUMERIC` | Conversion exacte |
| `NUMBER(10)` | `INTEGER` ou `BIGINT` | Selon la précision |
| `VARCHAR2(n)` | `VARCHAR(n)` | Taille préservée |
| `CHAR(n)` | `CHAR(n)` | Taille fixe |
| `DATE` | `TIMESTAMP` | Oracle DATE inclut l'heure |
| `TIMESTAMP` | `TIMESTAMP` | Conversion directe |
| `CLOB` | `TEXT` | Texte long |
| `BLOB` | `BYTEA` | Données binaires |
| `RAW` | `BYTEA` | Binaire de taille limitée |

### Options de Performance

```sql
-- Options au niveau du serveur
ALTER SERVER serveur_oracle OPTIONS (
    ADD nls_lang 'AMERICAN_AMERICA.UTF8',  -- Encodage Oracle
    ADD prefetch '200'                      -- Nombre de lignes à précharger
);

-- Options au niveau de la foreign table
ALTER FOREIGN TABLE clients_oracle OPTIONS (
    ADD prefetch '500',                     -- Surcharge pour cette table
    ADD readonly 'true'                     -- Lecture seule (plus rapide)
);
```

### Cas d'usage : Migration d'Oracle vers PostgreSQL

**Scénario** : Migration progressive d'une application Oracle vers PostgreSQL.

**Phase 1 : Préparation**
```sql
-- Dans PostgreSQL, importer la structure Oracle
IMPORT FOREIGN SCHEMA "APP_PROD"
    FROM SERVER serveur_oracle
    INTO oracle_legacy;
```

**Phase 2 : Tests de compatibilité**
```sql
-- Tester les requêtes applicatives
SELECT client_id, nom  
FROM oracle_legacy.customers  
WHERE status = 'ACTIVE';  

-- Comparer les performances
EXPLAIN ANALYZE SELECT * FROM oracle_legacy.orders WHERE order_date > SYSDATE - 30;
```

**Phase 3 : Migration progressive table par table**
```sql
-- Migrer une table
CREATE TABLE customers_pg (LIKE oracle_legacy.customers INCLUDING ALL);

-- Copier les données
INSERT INTO customers_pg  
SELECT * FROM oracle_legacy.customers;  

-- Vérifier la cohérence
SELECT
    (SELECT COUNT(*) FROM oracle_legacy.customers) AS oracle_count,
    (SELECT COUNT(*) FROM customers_pg) AS pg_count;
```

**Phase 4 : Basculement**
- Rediriger l'application vers les tables PostgreSQL natives
- Garder oracle_fdw en lecture seule pour consultation historique

### Limitations et Considérations

#### Limitations de oracle_fdw

❌ **Pas de support pour tous les types Oracle** :
- Types objets (OBJECT)
- Collections (VARRAY, nested tables)
- Types XML complexes
- Spatial (SDO_GEOMETRY) nécessite des conversions

❌ **Différences SQL** :
```sql
-- Oracle
SELECT SYSDATE FROM DUAL;

-- PostgreSQL (via oracle_fdw, fonctionne mais...)
SELECT CURRENT_DATE FROM dual;  -- 'dual' doit être mappé
```

❌ **Fonctions Oracle propriétaires** : Les fonctions Oracle (comme `NVL`, `DECODE`) ne sont pas directement utilisables dans PostgreSQL. Utilisez les équivalents :

| Oracle | PostgreSQL Équivalent |
|--------|----------------------|
| `NVL(col, 'N/A')` | `COALESCE(col, 'N/A')` |
| `DECODE(col, 1, 'Un', 2, 'Deux')` | `CASE WHEN col = 1 THEN 'Un' WHEN col = 2 THEN 'Deux' END` |
| `SYSDATE` | `CURRENT_TIMESTAMP` |
| `ROWNUM` | `ROW_NUMBER()` |

#### Performance

⚠️ **Latence réseau** : Chaque requête traverse le réseau vers Oracle. Pour des volumes importants, préférez :
- La réplication (Oracle GoldenGate vers PostgreSQL)
- Des imports batch réguliers

⚠️ **Pushdown limité** : oracle_fdw pousse certaines opérations vers Oracle, mais moins efficacement que postgres_fdw :  
- `WHERE` : ✅ Poussé  
- `ORDER BY` : ✅ Poussé  
- `LIMIT` : ✅ Poussé  
- `JOIN` entre foreign tables Oracle : ✅ Poussé  
- `JOIN` entre Oracle et PostgreSQL : ❌ Rapatrié

### Sécurité

```sql
-- Créer un utilisateur Oracle dédié en lecture seule
-- (côté Oracle)
CREATE USER fdw_readonly IDENTIFIED BY "mot_de_passe_fort";  
GRANT CONNECT TO fdw_readonly;  
GRANT SELECT ON schema.table TO fdw_readonly;  

-- Utiliser SSL/TLS (si Oracle est configuré pour)
CREATE SERVER serveur_oracle
    FOREIGN DATA WRAPPER oracle_fdw
    OPTIONS (
        dbserver '//oracle-prod:1521/ORCL',
        nls_lang 'AMERICAN_AMERICA.UTF8'
    );
```

## Partie 3 : mysql_fdw - Connexion à MySQL/MariaDB

### Qu'est-ce que mysql_fdw ?

**mysql_fdw** est un FDW tiers qui permet à PostgreSQL de se connecter à des bases MySQL ou MariaDB.

### Pourquoi utiliser mysql_fdw ?

🔗 **Intégration multi-SGBD** : Votre entreprise utilise MySQL pour certaines applications et PostgreSQL pour d'autres

📊 **Data Warehouse** : Consolider des données depuis plusieurs sources MySQL vers PostgreSQL

🔄 **Migration** : Migrer progressivement de MySQL vers PostgreSQL

### Prérequis

**Installation sur Linux (exemple Ubuntu)** :

```bash
# 1. Installer les bibliothèques MySQL client
sudo apt-get install libmysqlclient-dev

# 2. Installer mysql_fdw
sudo apt-get install postgresql-16-mysql-fdw

# Ou compiler depuis les sources
git clone https://github.com/EnterpriseDB/mysql_fdw.git  
cd mysql_fdw  
make USE_PGXS=1  
sudo make USE_PGXS=1 install  
```

### Configuration

#### Étape 1 : Créer l'extension

```sql
CREATE EXTENSION mysql_fdw;
```

#### Étape 2 : Créer le Foreign Server

```sql
CREATE SERVER serveur_mysql
    FOREIGN DATA WRAPPER mysql_fdw
    OPTIONS (
        host 'mysql-prod.entreprise.com',   -- Serveur MySQL
        port '3306'                          -- Port MySQL (3306 par défaut)
    );
```

#### Étape 3 : Créer le User Mapping

```sql
CREATE USER MAPPING FOR utilisateur_postgresql
    SERVER serveur_mysql
    OPTIONS (
        username 'admin_mysql',   -- Utilisateur MySQL
        password 'mot_de_passe'   -- Mot de passe MySQL
    );
```

#### Étape 4 : Créer les Foreign Tables

**Méthode manuelle** :

```sql
CREATE FOREIGN TABLE produits_mysql (
    produit_id INT,
    nom VARCHAR(100),
    prix DECIMAL(10, 2),
    stock INT
)
SERVER serveur_mysql  
OPTIONS (  
    dbname 'boutique',      -- Nom de la base MySQL
    table_name 'products'   -- Nom de la table MySQL
);
```

**Méthode automatique (IMPORT)** :

```sql
-- Importer toutes les tables d'une base MySQL
IMPORT FOREIGN SCHEMA 'boutique'
    FROM SERVER serveur_mysql
    INTO public;

-- Limiter à certaines tables
IMPORT FOREIGN SCHEMA 'boutique'
    LIMIT TO (products, categories)
    FROM SERVER serveur_mysql
    INTO public;
```

### Utilisation

#### Requêtes de lecture

```sql
-- Sélection
SELECT * FROM produits_mysql;

-- Jointure avec PostgreSQL
SELECT
    p.nom AS produit,
    c.quantite
FROM produits_mysql p  
INNER JOIN commandes_postgresql c ON p.produit_id = c.produit_id;  

-- Agrégation
SELECT
    COUNT(*) AS nb_produits,
    SUM(stock) AS stock_total
FROM produits_mysql  
WHERE prix > 100;  
```

#### Requêtes d'écriture

mysql_fdw supporte les écritures :

```sql
-- Insertion
INSERT INTO produits_mysql (produit_id, nom, prix, stock)  
VALUES (101, 'Nouveau produit', 49.99, 50);  

-- Mise à jour
UPDATE produits_mysql  
SET stock = stock - 1  
WHERE produit_id = 101;  

-- Suppression
DELETE FROM produits_mysql WHERE produit_id = 101;
```

### Mapping des Types de Données

| Type MySQL | Type PostgreSQL | Notes |
|------------|-----------------|-------|
| `INT`, `INTEGER` | `INTEGER` | Entiers standards |
| `BIGINT` | `BIGINT` | Grands entiers |
| `DECIMAL(p,s)` | `NUMERIC(p,s)` | Nombres décimaux |
| `FLOAT`, `DOUBLE` | `REAL`, `DOUBLE PRECISION` | Nombres flottants |
| `VARCHAR(n)` | `VARCHAR(n)` | Chaînes variables |
| `TEXT` | `TEXT` | Texte long |
| `DATE` | `DATE` | Dates |
| `DATETIME` | `TIMESTAMP` | Date + heure |
| `TIMESTAMP` | `TIMESTAMP` | Horodatage |
| `BLOB` | `BYTEA` | Données binaires |
| `ENUM` | `TEXT` | Converti en texte |
| `SET` | `TEXT` | Converti en texte |

### Options de Performance

```sql
-- Options au niveau du serveur
ALTER SERVER serveur_mysql OPTIONS (
    ADD fetch_size '1000',           -- Taille des lots
    ADD use_remote_estimate 'true'   -- Utiliser stats MySQL
);

-- Options au niveau de la table
ALTER FOREIGN TABLE produits_mysql OPTIONS (
    ADD fetch_size '5000'            -- Surcharge pour cette table
);
```

### Différences MySQL vs PostgreSQL

Lors de l'utilisation de mysql_fdw, soyez conscient des différences SQL :

#### 1. Syntaxe des chaînes

```sql
-- MySQL accepte les guillemets simples ET doubles
SELECT * FROM produits WHERE nom = "Laptop";

-- PostgreSQL préfère les guillemets simples
SELECT * FROM produits_mysql WHERE nom = 'Laptop';
```

#### 2. Auto-increment vs SERIAL

```sql
-- MySQL
CREATE TABLE products (
    id INT AUTO_INCREMENT PRIMARY KEY,
    nom VARCHAR(100)
);

-- PostgreSQL équivalent
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100)
);
```

#### 3. Fonctions spécifiques

| MySQL | PostgreSQL Équivalent |
|-------|----------------------|
| `IFNULL(col, 'N/A')` | `COALESCE(col, 'N/A')` |
| `CONCAT(a, b)` | `a || b` ou `CONCAT(a, b)` |
| `NOW()` | `NOW()` (identique) |
| `DATE_FORMAT()` | `TO_CHAR()` |
| `LIMIT 10 OFFSET 20` | `LIMIT 10 OFFSET 20` (identique) |

#### 4. Comportement du NULL

MySQL a des comportements particuliers avec NULL et les chaînes vides :

```sql
-- MySQL : '' et NULL peuvent être traités différemment selon sql_mode
-- PostgreSQL : '' et NULL sont TOUJOURS distincts

-- Vérifier dans PostgreSQL
SELECT * FROM produits_mysql WHERE description IS NULL;  
SELECT * FROM produits_mysql WHERE description = '';  -- Différent!  
```

### Cas d'usage : Consolidation multi-SGBD

**Scénario** : Créer un data warehouse PostgreSQL qui consolide des données depuis plusieurs bases MySQL.

```sql
-- Serveur MySQL 1 (boutique en ligne)
CREATE SERVER mysql_boutique
    FOREIGN DATA WRAPPER mysql_fdw
    OPTIONS (host 'mysql1.exemple.com', port '3306');

-- Serveur MySQL 2 (CRM)
CREATE SERVER mysql_crm
    FOREIGN DATA WRAPPER mysql_fdw
    OPTIONS (host 'mysql2.exemple.com', port '3306');

-- Importer les données
IMPORT FOREIGN SCHEMA 'shop'
    LIMIT TO (orders, order_items)
    FROM SERVER mysql_boutique INTO shop_data;

IMPORT FOREIGN SCHEMA 'crm'
    LIMIT TO (customers, interactions)
    FROM SERVER mysql_crm INTO crm_data;

-- Créer une vue consolidée
CREATE VIEW ventes_consolidees AS  
SELECT  
    c.customer_id,
    c.nom,
    c.email,
    o.order_id,
    o.montant,
    o.date_commande
FROM crm_data.customers c  
INNER JOIN shop_data.orders o ON c.customer_id = o.customer_id;  

-- Matérialiser pour les performances
CREATE MATERIALIZED VIEW ventes_consolidees_mv AS  
SELECT * FROM ventes_consolidees;  

-- Rafraîchir quotidiennement
REFRESH MATERIALIZED VIEW ventes_consolidees_mv;
```

### Cas d'usage : Migration MySQL vers PostgreSQL

**Étape 1 : Analyse de la structure**

```sql
-- Importer toutes les tables
IMPORT FOREIGN SCHEMA 'app_prod'
    FROM SERVER serveur_mysql
    INTO mysql_legacy;

-- Analyser les types de données
SELECT
    table_name,
    column_name,
    data_type,
    character_maximum_length
FROM information_schema.columns  
WHERE table_schema = 'mysql_legacy'  
ORDER BY table_name, ordinal_position;  
```

**Étape 2 : Créer le schéma PostgreSQL**

```sql
-- Recréer les tables avec types PostgreSQL optimisés
CREATE TABLE users_pg (
    id SERIAL PRIMARY KEY,                    -- Au lieu de AUTO_INCREMENT
    username VARCHAR(50) UNIQUE NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()        -- Au lieu de DEFAULT CURRENT_TIMESTAMP
);

-- Créer les index
CREATE INDEX idx_users_email ON users_pg(email);
```

**Étape 3 : Migrer les données**

```sql
-- Migration initiale
INSERT INTO users_pg (id, username, email, created_at)  
SELECT id, username, email, created_at  
FROM mysql_legacy.users;  

-- Mettre à jour la séquence
SELECT setval('users_pg_id_seq', (SELECT MAX(id) FROM users_pg));
```

**Étape 4 : Synchronisation incrémentale**

```sql
-- Migrer uniquement les nouveaux enregistrements
INSERT INTO users_pg (id, username, email, created_at)  
SELECT id, username, email, created_at  
FROM mysql_legacy.users  
WHERE id > (SELECT COALESCE(MAX(id), 0) FROM users_pg);  
```

### Performance et Optimisation

#### Bonnes pratiques

✅ **Utiliser des index sur MySQL** : Les index MySQL sont utilisés lors du pushdown.

✅ **Limiter les transferts** :
```sql
-- ❌ Mauvais : transfère toute la table
SELECT * FROM produits_mysql;

-- ✅ Bon : filtre à distance
SELECT * FROM produits_mysql WHERE categorie = 'Électronique';
```

✅ **Matérialiser pour les requêtes fréquentes** :
```sql
-- Au lieu d'interroger MySQL à chaque fois
CREATE MATERIALIZED VIEW produits_cache AS  
SELECT * FROM produits_mysql;  

-- Rafraîchir périodiquement
REFRESH MATERIALIZED VIEW produits_cache;
```

#### Monitoring

```sql
-- Sur PostgreSQL, voir les requêtes vers MySQL
EXPLAIN (VERBOSE, ANALYZE)  
SELECT * FROM produits_mysql WHERE prix > 100;  

-- Sur MySQL, activer le slow query log
-- (fichier de config MySQL)
slow_query_log = 1  
slow_query_log_file = /var/log/mysql/slow.log  
long_query_time = 1  
```

### Limitations

❌ **Transactions distribuées** : mysql_fdw ne garantit pas l'atomicité parfaite entre PostgreSQL et MySQL.

```sql
BEGIN;  
UPDATE table_postgresql SET statut = 'traité';  
UPDATE produits_mysql SET stock = stock - 1;  
COMMIT;  -- Pas de garantie 2PC  
```

❌ **Pushdown limité** : Moins efficace que postgres_fdw. Certaines opérations ne sont pas poussées vers MySQL.

❌ **Différences sémantiques** : Les comportements subtils (NULL, comparaisons de chaînes, etc.) peuvent différer.

❌ **Types complexes** : Les types PostgreSQL avancés (JSONB, ARRAY, hstore) ne peuvent pas être écrits vers MySQL.

## Comparaison des FDW

### Tableau récapitulatif

| Critère | file_fdw | oracle_fdw | mysql_fdw |
|---------|----------|------------|-----------|
| **Source** | Fichiers locaux | Oracle Database | MySQL/MariaDB |
| **Inclus par défaut** | ✅ Oui | ❌ Non (tierce partie) | ❌ Non (tierce partie) |
| **Lecture** | ✅ | ✅ | ✅ |
| **Écriture** | ❌ Lecture seule | ✅ | ✅ |
| **Performance** | ⚠️ Moyenne | ⚠️⚠️ Dépend réseau | ⚠️⚠️ Dépend réseau |
| **Pushdown** | ❌ Aucun | ✅✅ Bon | ✅ Partiel |
| **Transactions** | N/A | ⚠️ Pas 2PC garantie | ⚠️ Pas 2PC garantie |
| **Complexité installation** | ✅ Facile | ⚠️⚠️ Oracle Client requis | ⚠️ Bibliothèques MySQL requises |
| **Cas d'usage principal** | Import/Analyse fichiers | Migration Oracle | Migration/Intégration MySQL |

### Quand utiliser quel FDW ?

#### file_fdw

✅ **Utilisez quand** :
- Vous devez analyser des fichiers CSV/TSV avant import
- Données externes non structurées en base
- Validation de données avant intégration
- Logs et fichiers texte structurés

❌ **N'utilisez PAS quand** :
- Vous avez besoin de performances élevées (utilisez COPY)
- Les fichiers sont distants (utilisez des outils de transfert d'abord)
- Vous voulez modifier les fichiers (lecture seule)

#### oracle_fdw

✅ **Utilisez quand** :
- Migration d'Oracle vers PostgreSQL
- Accès ponctuel à des données Oracle legacy
- Reporting consolidé Oracle + PostgreSQL
- Prototypage d'applications avant migration

❌ **N'utilisez PAS quand** :
- Les performances sont critiques (préférez la réplication)
- Vous avez besoin de transactions distribuées strictes
- La latence réseau est élevée

#### mysql_fdw

✅ **Utilisez quand** :
- Migration de MySQL vers PostgreSQL
- Data warehouse multi-SGBD
- Intégration d'applications MySQL + PostgreSQL
- Consolidation de plusieurs bases MySQL

❌ **N'utilisez PAS quand** :
- Vous avez besoin de types PostgreSQL avancés (JSONB, etc.)
- Les performances temps réel sont critiques
- Vous voulez des garanties transactionnelles fortes

## Sécurité Commune aux FDW

### Principes généraux

#### 1. Principe du moindre privilège

```sql
-- Créer des utilisateurs dédiés avec permissions minimales
-- Sur la source (Oracle/MySQL)
CREATE USER fdw_readonly IDENTIFIED BY 'mot_de_passe';  
GRANT CONNECT ON DATABASE TO fdw_readonly;  
GRANT SELECT ON schema.table TO fdw_readonly;  -- Lecture seule  
```

#### 2. Stockage des mots de passe

❌ **Évitez** :
```sql
CREATE USER MAPPING FOR user
    SERVER serveur_distant
    OPTIONS (password 'mot_de_passe_en_clair');  -- Visible dans pg_catalog
```

✅ **Préférez** :
- Fichiers `.pgpass`
- Gestionnaires de secrets (HashiCorp Vault, AWS Secrets Manager)
- Certificats SSL/TLS pour l'authentification

#### 3. Chiffrement des connexions

```sql
-- Forcer SSL pour oracle_fdw
-- (nécessite configuration Oracle SSL)

-- Pour mysql_fdw
ALTER SERVER serveur_mysql OPTIONS (
    ADD ssl_ca '/etc/ssl/mysql-ca.pem',
    ADD ssl_cert '/etc/ssl/mysql-cert.pem',
    ADD ssl_key '/etc/ssl/mysql-key.pem'
);
```

#### 4. Audit et monitoring

```sql
-- Surveiller les connexions FDW
SELECT
    datname,
    usename,
    application_name,
    client_addr,
    state,
    query
FROM pg_stat_activity  
WHERE application_name LIKE '%fdw%';  
```

## Dépannage Commun

### Problèmes fréquents et solutions

#### Erreur : "extension not found"

```
ERROR: could not open extension control file
```

**Solution** : Installer le package correspondant :
```bash
sudo apt-get install postgresql-16-mysql-fdw  
sudo apt-get install postgresql-16-oracle-fdw  
```

#### Erreur de connexion réseau

```
ERROR: could not connect to server
```

**Checklist** :
1. Vérifier la connectivité : `telnet host port`  
2. Vérifier le firewall : Port ouvert ?  
3. Vérifier les permissions de connexion distante (my.cnf pour MySQL, listener.ora pour Oracle)  
4. Tester avec les outils natifs (mysql client, sqlplus)

#### Erreur d'authentification

```
ERROR: authentication failed
```

**Solutions** :
1. Vérifier les credentials dans le user mapping  
2. Vérifier que l'utilisateur existe côté distant  
3. Vérifier les permissions (GRANT)

#### Performances lentes

**Diagnostic** :
```sql
EXPLAIN (ANALYZE, VERBOSE, BUFFERS)  
SELECT * FROM foreign_table WHERE condition;  
```

**Solutions** :
- Vérifier le pushdown (Remote SQL)
- Ajouter des index côté distant
- Augmenter `fetch_size`
- Utiliser une vue matérialisée pour le cache

## Conclusion

### Points clés à retenir

✅ **file_fdw** : Idéal pour lire et analyser des fichiers CSV/texte sans les importer

✅ **oracle_fdw** : Permet de connecter PostgreSQL à Oracle, utile pour les migrations et l'intégration

✅ **mysql_fdw** : Connecte PostgreSQL à MySQL/MariaDB, essentiel pour les environnements multi-SGBD

⚠️ **Performance** : Les FDW vers bases distantes (Oracle, MySQL) ont une latence réseau à considérer

⚠️ **Transactions** : Pas de garantie 2PC stricte par défaut

🔒 **Sécurité** : Toujours utiliser des comptes dédiés avec privilèges minimaux et chiffrement
