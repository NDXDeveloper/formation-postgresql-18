# 18.7.1. pg_cron : Planification de Tâches dans PostgreSQL

## Introduction

**pg_cron** est une extension PostgreSQL qui permet de planifier et d'exécuter automatiquement des tâches (jobs) directement au sein de votre base de données, de manière similaire au système Unix/Linux `cron`.

### Qu'est-ce qu'une tâche planifiée ?

Une tâche planifiée est une opération qui s'exécute automatiquement à des moments prédéfinis, sans intervention manuelle. Par exemple :
- Nettoyer des données obsolètes tous les jours à minuit
- Générer des rapports toutes les heures
- Rafraîchir des vues matérialisées chaque lundi matin
- Effectuer des sauvegardes automatiques

### Pourquoi utiliser pg_cron ?

Avant pg_cron, pour automatiser des tâches PostgreSQL, il fallait :
- Utiliser le cron système Unix/Linux (externe à PostgreSQL)
- Créer des scripts shell complexes
- Gérer l'authentification et les connexions manuellement
- Maintenir une infrastructure parallèle

**Avec pg_cron**, tout est intégré dans PostgreSQL :
- ✅ Gestion centralisée dans la base de données  
- ✅ Pas besoin d'accès système (serveur)  
- ✅ Configuration simple via SQL  
- ✅ Logs et historique intégrés  
- ✅ Fonctionne sur tous les systèmes (Windows, Linux, macOS)

## Architecture et Fonctionnement

### Comment pg_cron fonctionne-t-il ?

pg_cron est un **processus d'arrière-plan (background worker)** qui :

1. **Démarre avec PostgreSQL** : Il se lance automatiquement au démarrage de l'instance  
2. **Vérifie le planning** : Toutes les secondes (par défaut), il consulte sa table de jobs  
3. **Exécute les tâches** : Lorsqu'une tâche doit s'exécuter, pg_cron lance une connexion PostgreSQL et exécute le SQL défini  
4. **Enregistre les résultats** : Chaque exécution est tracée (succès/échec, durée, messages)

### Architecture technique

```
┌────────────────────────────────────────┐
│         Instance PostgreSQL            │
│                                        │
│  ┌───────────────────────────────────┐ │
│  │   Extension pg_cron               │ │
│  │   (Background Worker)             │ │
│  │                                   │ │
│  │   ┌─────────────────────────┐     │ │
│  │   │  Scheduler Thread       │     │ │
│  │   │  (vérifie les jobs)     │     │ │
│  │   └─────────────────────────┘     │ │
│  │            ↓                      │ │
│  │   ┌─────────────────────────┐     │ │
│  │   │  Executor Threads       │     │ │
│  │   │  (exécute le SQL)       │     │ │
│  │   └─────────────────────────┘     │ │
│  └───────────────────────────────────┘ │
│                                        │
│  ┌───────────────────────────────────┐ │
│  │   Tables pg_cron                  │ │
│  │   - cron.job (définitions)        │ │
│  │   - cron.job_run_details (logs)   │ │
│  └───────────────────────────────────┘ │
└────────────────────────────────────────┘
```

## Installation et Configuration

### Prérequis

- PostgreSQL 10 ou supérieur (idéalement PostgreSQL 18)
- Droits superutilisateur pour l'installation initiale
- Extension disponible (selon votre distribution PostgreSQL)

### Processus d'installation (théorique)

#### Étape 1 : Modifier postgresql.conf

Pour que pg_cron se charge au démarrage, il faut l'ajouter dans les bibliothèques préchargées :

```ini
shared_preload_libraries = 'pg_cron'
```

**Important** : Ce paramètre nécessite un redémarrage complet de PostgreSQL.

#### Étape 2 : Configurer la base de données cible

pg_cron doit savoir dans quelle base de données stocker ses métadonnées :

```ini
cron.database_name = 'postgres'
```

Cette base contiendra les tables `cron.job` et `cron.job_run_details`.

#### Étape 3 : Créer l'extension

Une fois PostgreSQL redémarré, se connecter à la base de données spécifiée et créer l'extension :

```sql
CREATE EXTENSION pg_cron;
```

Cette commande crée :
- Le schéma `cron`
- La table `cron.job` (définitions des tâches)
- La table `cron.job_run_details` (historique d'exécution)
- Les fonctions de gestion

## Syntaxe de Planification (Cron Syntax)

### Format général

pg_cron utilise la syntaxe **cron standard** avec 5 champs :

```
 ┌───────────── minute (0 - 59)
 │ ┌───────────── heure (0 - 23)
 │ │ ┌───────────── jour du mois (1 - 31)
 │ │ │ ┌───────────── mois (1 - 12)
 │ │ │ │ ┌───────────── jour de la semaine (0 - 7) (0 et 7 = dimanche)
 │ │ │ │ │
 * * * * *
```

### Opérateurs spéciaux

| Symbole | Signification | Exemple |
|---------|--------------|---------|
| `*` | Toutes les valeurs | `* * * * *` = chaque minute |
| `,` | Liste de valeurs | `0,30 * * * *` = à 0 et 30 minutes |
| `-` | Plage de valeurs | `0 9-17 * * *` = toutes les heures de 9h à 17h |
| `/` | Intervalle | `*/15 * * * *` = toutes les 15 minutes |

### Exemples de syntaxe courante

```
'0 0 * * *'          → Tous les jours à minuit (00:00)
'30 2 * * *'         → Tous les jours à 2h30
'0 9-17 * * *'       → Toutes les heures entre 9h et 17h
'*/15 * * * *'       → Toutes les 15 minutes
'0 0 * * 0'          → Tous les dimanches à minuit
'0 9 1 * *'          → Le 1er de chaque mois à 9h
'0 0 1 1 *'          → Le 1er janvier à minuit
'0 22 * * 1-5'       → Du lundi au vendredi à 22h
'*/5 9-17 * * 1-5'   → Toutes les 5 min en semaine, pendant les heures de bureau
```

### Raccourcis pratiques

pg_cron supporte également des raccourcis textuels :

```sql
'@hourly'    → Équivalent à '0 * * * *'
'@daily'     → Équivalent à '0 0 * * *'
'@weekly'    → Équivalent à '0 0 * * 0'
'@monthly'   → Équivalent à '0 0 1 * *'
'@yearly'    → Équivalent à '0 0 1 1 *'
```

## Gestion des Tâches

### Créer une tâche

Pour créer une nouvelle tâche, on utilise la fonction `cron.schedule()` :

```sql
SELECT cron.schedule(
    'nom_du_job',           -- Nom descriptif de la tâche
    '0 0 * * *',            -- Planning cron
    'DELETE FROM logs WHERE created_at < NOW() - INTERVAL ''30 days'''  -- SQL à exécuter
);
```

**Exemple concret** : Nettoyer les anciens logs tous les jours à minuit

```sql
SELECT cron.schedule(
    'nettoyer_logs_anciens',
    '0 0 * * *',
    'DELETE FROM logs WHERE created_at < NOW() - INTERVAL ''30 days'''
);
```

**Retour** : La fonction retourne un `jobid` (identifiant unique du job), par exemple `42`.

### Créer une tâche sans nom

Si vous ne spécifiez pas de nom, pg_cron génère un nom automatique basé sur le jobid :

```sql
SELECT cron.schedule(
    '*/10 * * * *',
    'REFRESH MATERIALIZED VIEW CONCURRENTLY stats_view'
);
```

### Créer une tâche dans une autre base de données

Par défaut, les jobs s'exécutent dans la base de données où ils sont définis. Pour exécuter dans une autre base :

```sql
SELECT cron.schedule_in_database(
    'backup_analytics',
    '0 2 * * *',
    'SELECT pg_dump_function()',
    'analytics_db'  -- Nom de la base cible
);
```

### Lister les tâches actives

Pour voir toutes les tâches définies :

```sql
SELECT * FROM cron.job;
```

**Colonnes principales** :
- `jobid` : Identifiant unique  
- `schedule` : Expression cron  
- `command` : Commande SQL à exécuter  
- `nodename` : Serveur d'exécution (pour les configurations distribuées)  
- `nodeport` : Port du serveur  
- `database` : Base de données cible  
- `username` : Utilisateur qui exécute la tâche  
- `active` : Tâche active ou non

### Désactiver temporairement une tâche

Pour suspendre une tâche sans la supprimer :

```sql
UPDATE cron.job SET active = false WHERE jobid = 42;
```

Pour la réactiver :

```sql
UPDATE cron.job SET active = true WHERE jobid = 42;
```

### Modifier une tâche existante

#### Modifier le planning

```sql
SELECT cron.alter_job(
    42,                    -- jobid
    schedule := '0 3 * * *'  -- Nouveau planning
);
```

#### Modifier la commande

```sql
SELECT cron.alter_job(
    42,
    command := 'DELETE FROM logs WHERE created_at < NOW() - INTERVAL ''60 days'''
);
```

#### Modifier plusieurs propriétés

```sql
SELECT cron.alter_job(
    42,
    schedule := '0 4 * * *',
    command := 'VACUUM ANALYZE logs',
    database := 'production'
);
```

### Supprimer une tâche

Pour supprimer définitivement une tâche :

```sql
SELECT cron.unschedule(42);  -- Par jobid
```

Ou par nom :

```sql
SELECT cron.unschedule('nettoyer_logs_anciens');
```

## Surveillance et Historique

### Table d'historique : cron.job_run_details

Chaque exécution de tâche est enregistrée dans la table `cron.job_run_details` :

```sql
SELECT * FROM cron.job_run_details  
ORDER BY start_time DESC  
LIMIT 10;  
```

**Colonnes principales** :
- `jobid` : Identifiant de la tâche  
- `runid` : Identifiant unique de l'exécution  
- `job_pid` : PID du processus PostgreSQL  
- `database` : Base de données d'exécution  
- `username` : Utilisateur ayant exécuté  
- `command` : Commande SQL exécutée  
- `status` : Statut (succeeded, failed, running)  
- `return_message` : Message d'erreur en cas d'échec  
- `start_time` : Heure de début  
- `end_time` : Heure de fin

### Exemples de requêtes de surveillance

#### Tâches ayant échoué dans les dernières 24h

```sql
SELECT
    jobid,
    command,
    status,
    return_message,
    start_time
FROM cron.job_run_details  
WHERE status = 'failed'  
  AND start_time > NOW() - INTERVAL '24 hours'
ORDER BY start_time DESC;
```

#### Durée moyenne d'exécution par tâche

```sql
SELECT
    jobid,
    COUNT(*) as executions,
    AVG(EXTRACT(EPOCH FROM (end_time - start_time))) as avg_duration_seconds,
    MAX(EXTRACT(EPOCH FROM (end_time - start_time))) as max_duration_seconds
FROM cron.job_run_details  
WHERE status = 'succeeded'  
  AND start_time > NOW() - INTERVAL '7 days'
GROUP BY jobid  
ORDER BY avg_duration_seconds DESC;  
```

#### Taux de succès par tâche

```sql
SELECT
    j.jobid,
    j.schedule,
    j.command,
    COUNT(*) as total_runs,
    COUNT(*) FILTER (WHERE jrd.status = 'succeeded') as successes,
    COUNT(*) FILTER (WHERE jrd.status = 'failed') as failures,
    ROUND(100.0 * COUNT(*) FILTER (WHERE jrd.status = 'succeeded') / COUNT(*), 2) as success_rate
FROM cron.job j  
JOIN cron.job_run_details jrd ON j.jobid = jrd.jobid  
WHERE jrd.start_time > NOW() - INTERVAL '30 days'  
GROUP BY j.jobid, j.schedule, j.command  
ORDER BY success_rate;  
```

### Nettoyer l'historique

Pour éviter que la table `job_run_details` ne devienne trop volumineuse, il est recommandé de la nettoyer régulièrement :

```sql
-- Supprimer l'historique de plus de 30 jours
DELETE FROM cron.job_run_details  
WHERE start_time < NOW() - INTERVAL '30 days';  
```

**Astuce** : On peut automatiser ce nettoyage avec... pg_cron lui-même ! 🔄

```sql
SELECT cron.schedule(
    'nettoyer_historique_cron',
    '0 3 * * 0',  -- Tous les dimanches à 3h
    'DELETE FROM cron.job_run_details WHERE start_time < NOW() - INTERVAL ''30 days'''
);
```

## Cas d'Usage Courants

### 1. Maintenance de base de données

#### Nettoyer les données obsolètes

```sql
SELECT cron.schedule(
    'purge_sessions_expirees',
    '*/30 * * * *',  -- Toutes les 30 minutes
    'DELETE FROM user_sessions WHERE expires_at < NOW()'
);
```

#### VACUUM et ANALYZE réguliers

```sql
SELECT cron.schedule(
    'vacuum_hebdomadaire',
    '0 2 * * 0',  -- Dimanches à 2h
    'VACUUM ANALYZE'
);
```

#### Rafraîchir les vues matérialisées

```sql
SELECT cron.schedule(
    'refresh_mv_stats',
    '0 * * * *',  -- Toutes les heures
    'REFRESH MATERIALIZED VIEW CONCURRENTLY ventes_mensuelles'
);
```

### 2. Traitement de données

#### Agrégation quotidienne

```sql
SELECT cron.schedule(
    'agreger_stats_jour',
    '5 0 * * *',  -- Tous les jours à 00:05
    $$
    INSERT INTO stats_quotidiennes (date, total_ventes, nb_commandes)
    SELECT
        CURRENT_DATE - 1,
        SUM(montant),
        COUNT(*)
    FROM commandes
    WHERE created_at >= CURRENT_DATE - 1
      AND created_at < CURRENT_DATE
    $$
);
```

#### Import de données externes

```sql
SELECT cron.schedule(
    'import_taux_change',
    '0 9 * * 1-5',  -- Tous les jours ouvrés à 9h
    'SELECT import_exchange_rates_from_api()'
);
```

### 3. Notifications et alertes

#### Vérifier les anomalies

```sql
SELECT cron.schedule(
    'detecter_anomalies',
    '*/5 * * * *',  -- Toutes les 5 minutes
    $$
    SELECT send_alert('Anomalie détectée')
    WHERE EXISTS (
        SELECT 1 FROM transactions
        WHERE montant > 10000
          AND created_at > NOW() - INTERVAL '5 minutes'
    )
    $$
);
```

#### Envoyer des rapports

```sql
SELECT cron.schedule(
    'rapport_hebdomadaire',
    '0 8 * * 1',  -- Lundis à 8h
    'SELECT generate_and_email_weekly_report()'
);
```

### 4. Gestion des partitions

#### Créer des partitions futures

```sql
SELECT cron.schedule(
    'creer_partitions_mensuelles',
    '0 0 25 * *',  -- Le 25 de chaque mois
    $$
    SELECT create_next_month_partition('logs')
    $$
);
```

#### Détacher et archiver des anciennes partitions

```sql
SELECT cron.schedule(
    'archiver_anciennes_partitions',
    '0 3 1 * *',  -- Le 1er de chaque mois à 3h
    $$
    SELECT detach_and_archive_old_partitions('logs', 6)  -- Garder 6 mois
    $$
);
```

### 5. Sauvegardes logiques

#### Export régulier de tables

```sql
SELECT cron.schedule(
    'backup_referentiels',
    '0 1 * * *',  -- Tous les jours à 1h
    $$
    COPY (SELECT * FROM produits) TO '/backups/produits.csv' WITH CSV HEADER
    $$
);
```

**Note** : Nécessite les permissions appropriées sur le système de fichiers.

## Bonnes Pratiques

### 1. Nommage des jobs

- ✅ Utilisez des noms descriptifs et explicites  
- ✅ Incluez l'action et la fréquence si pertinent  
- ❌ Évitez les noms génériques comme "job1", "task_backup"

**Exemples** :
```
✅ 'nettoyer_sessions_expirees_horaire'
✅ 'agreger_ventes_quotidien'
✅ 'backup_complet_dimanche'
❌ 'job1'
❌ 'maintenance'
```

### 2. Gestion des erreurs

Les erreurs dans les tâches pg_cron ne bloquent pas l'exécution des autres jobs, mais :

- **Surveillez** : Consultez régulièrement `cron.job_run_details`  
- **Alertez** : Mettez en place des alertes sur les échecs  
- **Testez** : Validez vos commandes SQL avant de les planifier

### 3. Utiliser des fonctions stockées

Au lieu d'écrire du SQL complexe directement dans `cron.schedule()`, créez des fonctions :

```sql
-- Créer la fonction
CREATE OR REPLACE FUNCTION maintenance_quotidienne()  
RETURNS void AS $$  
BEGIN  
    DELETE FROM logs WHERE created_at < NOW() - INTERVAL '30 days';
    DELETE FROM temp_data WHERE processed = true;
    VACUUM ANALYZE logs;
END;
$$ LANGUAGE plpgsql;

-- Planifier la fonction
SELECT cron.schedule(
    'maintenance_quotidienne',
    '0 3 * * *',
    'SELECT maintenance_quotidienne()'
);
```

**Avantages** :
- Code plus lisible
- Meilleure maintenabilité
- Gestion d'erreurs améliorée
- Réutilisabilité

### 4. Éviter les chevauchements

Si une tâche peut prendre du temps, évitez qu'elle ne se chevauche avec elle-même :

```sql
-- Utiliser un advisory lock
CREATE OR REPLACE FUNCTION process_long_running_task()  
RETURNS void AS $$  
BEGIN  
    -- Tentative de verrouillage (non-bloquant)
    IF pg_try_advisory_lock(123456) THEN
        BEGIN
            -- Votre traitement long ici
            PERFORM traitement_long();

            -- Libérer le verrou
            PERFORM pg_advisory_unlock(123456);
        EXCEPTION
            WHEN OTHERS THEN
                PERFORM pg_advisory_unlock(123456);
                RAISE;
        END;
    ELSE
        RAISE NOTICE 'Tâche déjà en cours, skip';
    END IF;
END;
$$ LANGUAGE plpgsql;
```

### 5. Documenter les tâches

Ajoutez des commentaires sur vos jobs :

```sql
COMMENT ON FUNCTION maintenance_quotidienne() IS
'Exécute la maintenance quotidienne :
- Supprime les logs de plus de 30 jours
- Nettoie les données temporaires
- Lance VACUUM ANALYZE
Planifié pour s''exécuter tous les jours à 3h du matin';
```

### 6. Limiter les ressources

Les tâches pg_cron s'exécutent dans des connexions PostgreSQL normales. Pour éviter la surcharge :

```sql
-- Limiter le temps d'exécution
SET statement_timeout = '5min';

-- Limiter la mémoire utilisée
SET work_mem = '64MB';
```

### 7. Planification intelligente

- **Heures creuses** : Planifiez les tâches lourdes pendant les heures de faible activité  
- **Étaler les tâches** : Évitez de lancer tous les jobs à minuit pile  
- **Tenant compte des sauvegardes** : Ne planifiez pas de maintenance pendant les fenêtres de backup

**Exemple** :
```
✅ Job A : 02:00 - Backup
✅ Job B : 03:00 - VACUUM
✅ Job C : 03:30 - Agrégation
✅ Job D : 04:00 - Nettoyage

❌ Job A, B, C, D tous à 00:00 → Surcharge !
```

### 8. Gestion des fuseaux horaires

pg_cron utilise le fuseau horaire du serveur PostgreSQL. Soyez explicite si nécessaire :

```sql
-- Vérifier le fuseau horaire
SHOW timezone;

-- Définir si nécessaire
SET timezone = 'Europe/Paris';
```

## Limitations et Considérations

### 1. Précision temporelle

- pg_cron vérifie les jobs **toutes les secondes** par défaut
- La granularité minimale est donc d'**une minute** (comme cron Unix)
- Pour des tâches nécessitant une précision à la seconde, pg_cron n'est pas adapté

### 2. Dépendances entre jobs

pg_cron n'a **pas de gestion native des dépendances** entre jobs :
- Impossible de dire "Job B démarre seulement si Job A a réussi"
- Pas de workflows complexes

**Solution** : Implémenter la logique dans une fonction PL/pgSQL qui appelle d'autres fonctions séquentiellement.

### 3. Performances et scalabilité

- Chaque job s'exécute dans sa **propre connexion PostgreSQL**
- Attention au nombre de jobs simultanés vs `max_connections`
- Les jobs longs peuvent monopoliser des connexions

### 4. Sécurité et permissions

- Les jobs s'exécutent avec les **permissions de l'utilisateur qui les a créés**
- Soyez prudent avec les superutilisateurs
- Utilisez le **principe du moindre privilège**

**Exemple** :
```sql
-- Créer un rôle dédié aux jobs cron
CREATE ROLE cron_worker;  
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO cron_worker;  

-- Ensuite, créer les jobs en tant que cet utilisateur
```

### 5. Pas de garantie d'exécution

- Si PostgreSQL est arrêté au moment de l'exécution, **le job est manqué**
- pg_cron ne rattrape pas les exécutions manquées au redémarrage
- Il n'y a pas de "catch-up" comme dans certains orchestrateurs

### 6. Haute disponibilité

Dans une configuration avec réplication :
- pg_cron s'exécute **uniquement sur le serveur primaire**
- En cas de failover, les jobs basculent sur le nouveau primaire
- Vérifiez la configuration après une promotion

### 7. Pas de parallélisation automatique

- pg_cron exécute les jobs **séquentiellement** par défaut
- Plusieurs jobs au même horaire sont exécutés dans l'ordre de leur `jobid`
- Pour du parallélisme, créez des jobs à des horaires légèrement décalés

## Comparaison avec d'Autres Solutions

### pg_cron vs Cron Système (Unix/Linux)

| Critère | pg_cron | Cron Système |
|---------|---------|--------------|
| **Intégration** | Natif PostgreSQL | Externe (OS) |
| **Configuration** | SQL (simple) | Fichiers crontab |
| **Portabilité** | Tous OS (Windows inclus) | Unix/Linux uniquement |
| **Logs** | Table PostgreSQL | Fichiers syslog/journalctl |
| **Accès nécessaire** | Base de données | Accès serveur (root/user) |
| **Transactions** | Support natif | Non applicable |
| **HA** | Suit le primaire | Configuration manuelle |

**Verdict** : pg_cron est idéal pour les tâches purement PostgreSQL, tandis que le cron système convient mieux pour des orchestrations complexes impliquant plusieurs systèmes.

### pg_cron vs pg_agent

**pg_agent** est une autre extension de planification, plus ancienne :

| Critère | pg_cron | pg_agent |
|---------|---------|----------|
| **Maintenance** | Actif, moderne | Peu maintenu |
| **Complexité** | Simple | Plus complexe |
| **Interface** | SQL pur | Nécessite pgAdmin |
| **Dépendances** | Aucune | pgAdmin requis |
| **Logs** | Natifs SQL | Interface graphique |

**Verdict** : pg_cron est devenu le standard de facto pour la planification dans PostgreSQL moderne.

### pg_cron vs Orchestrateurs externes (Airflow, Prefect, Temporal)

| Critère | pg_cron | Orchestrateurs |
|---------|---------|----------------|
| **Workflows** | Simples | Complexes, DAGs |
| **Dépendances** | Non | Oui (natif) |
| **Monitoring** | Basique | Avancé (dashboards) |
| **Scalabilité** | Limitée | Élevée |
| **Infrastructure** | Aucune | Serveurs dédiés |
| **Courbe d'apprentissage** | Faible | Moyenne à élevée |

**Verdict** : pg_cron pour des tâches simples et locales, les orchestrateurs pour des pipelines de données complexes multi-systèmes.

## Dépannage (Troubleshooting)

### Problème 1 : pg_cron ne démarre pas

**Symptômes** : Après installation, les jobs ne s'exécutent pas.

**Causes possibles** :
1. `shared_preload_libraries` non configuré  
2. `cron.database_name` absent ou incorrect  
3. PostgreSQL non redémarré après configuration

**Solutions** :
```sql
-- Vérifier que l'extension est chargée
SELECT * FROM pg_extension WHERE extname = 'pg_cron';

-- Vérifier les background workers
SELECT * FROM pg_stat_activity WHERE backend_type = 'pg_cron launcher';
```

### Problème 2 : Jobs ne s'exécutent pas

**Symptômes** : Les jobs sont créés mais aucune trace dans `job_run_details`.

**Vérifications** :
```sql
-- Le job est-il actif ?
SELECT jobid, active, schedule, command FROM cron.job WHERE jobid = 42;

-- Y a-t-il des erreurs de syntaxe cron ?
-- Testez votre expression sur https://crontab.guru

-- Le prochain lancement est-il dans le futur ?
-- pg_cron ne rattrape pas les exécutions passées
```

### Problème 3 : Jobs échouent systématiquement

**Diagnostic** :
```sql
SELECT
    jobid,
    status,
    return_message,
    command
FROM cron.job_run_details  
WHERE status = 'failed'  
  AND jobid = 42
ORDER BY start_time DESC  
LIMIT 5;  
```

**Causes fréquentes** :
- Erreur SQL dans la commande
- Permissions insuffisantes
- Objets manquants (tables, fonctions)
- Timeout

**Solution** : Testez la commande manuellement avant de la planifier.

### Problème 4 : Jobs trop lents

**Symptômes** : Les jobs prennent plus de temps que prévu.

**Analyse** :
```sql
SELECT
    jobid,
    command,
    EXTRACT(EPOCH FROM (end_time - start_time)) as duration_seconds
FROM cron.job_run_details  
WHERE jobid = 42  
  AND status = 'succeeded'
ORDER BY start_time DESC  
LIMIT 10;  
```

**Solutions** :
- Optimiser la requête SQL (ajouter des index, EXPLAIN ANALYZE)
- Diviser le job en sous-tâches plus petites
- Exécuter aux heures creuses
- Utiliser `CONCURRENTLY` pour les opérations qui le supportent

### Problème 5 : Saturation des connexions

**Symptômes** : Erreur "FATAL: sorry, too many clients already".

**Cause** : Trop de jobs simultanés pour `max_connections`.

**Solutions** :
```sql
-- Vérifier le nombre de connexions
SELECT COUNT(*) FROM pg_stat_activity;

-- Vérifier max_connections
SHOW max_connections;

-- Étaler les jobs dans le temps
-- Au lieu de tous les jobs à 00:00, les répartir :
-- Job A : 00:00
-- Job B : 00:05
-- Job C : 00:10
```

### Surveillance et alerting

#### Script de monitoring des échecs

```sql
-- Vue pour simplifier la surveillance
CREATE OR REPLACE VIEW cron.failed_jobs_last_24h AS  
SELECT  
    j.jobid,
    j.schedule,
    j.command,
    jrd.status,
    jrd.return_message,
    jrd.start_time,
    jrd.end_time
FROM cron.job j  
JOIN cron.job_run_details jrd ON j.jobid = jrd.jobid  
WHERE jrd.status = 'failed'  
  AND jrd.start_time > NOW() - INTERVAL '24 hours'
ORDER BY jrd.start_time DESC;

-- Job de notification en cas d'échec
SELECT cron.schedule(
    'alerter_echecs_cron',
    '*/30 * * * *',  -- Toutes les 30 minutes
    $$
    SELECT send_alert_email(
        'admin@company.com',
        'Échecs pg_cron détectés',
        'Détails: ' ||
        (SELECT string_agg(jobid::text || ': ' || return_message, E'\n')
         FROM cron.failed_jobs_last_24h)
    )
    WHERE EXISTS (SELECT 1 FROM cron.failed_jobs_last_24h)
    $$
);
```

## Résumé

**pg_cron** est une extension PostgreSQL puissante qui permet de :

- ✅ **Planifier** des tâches récurrentes directement dans PostgreSQL  
- ✅ **Simplifier** l'administration sans dépendre du cron système  
- ✅ **Automatiser** la maintenance, les agrégations, les nettoyages  
- ✅ **Surveiller** l'exécution via des tables SQL standards  
- ✅ **Intégrer** facilement dans des architectures cloud et Kubernetes

**Cas d'usage typiques** :
- Maintenance de base de données (VACUUM, ANALYZE)
- Nettoyage de données obsolètes
- Agrégations et rapports réguliers
- Rafraîchissement de vues matérialisées
- Gestion automatisée de partitions

**Points d'attention** :
- Nécessite un redémarrage PostgreSQL initial
- Pas de gestion native des dépendances entre jobs
- Surveillance et alerting à mettre en place
- Attention aux chevauchements et à la saturation des connexions

**Alternative à considérer si** :
- Vous avez besoin de workflows complexes avec dépendances → Airflow, Prefect
- Vous orchestrez plusieurs systèmes externes → Orchestrateurs dédiés
- Vous êtes sur Cloud SQL (GCP) où pg_cron n'est pas disponible → Cloud Scheduler

Pour un développeur ou DevOps travaillant avec PostgreSQL, **pg_cron est un outil indispensable** qui simplifie grandement l'automatisation tout en restant dans l'écosystème PostgreSQL.

3. Mettre en place une surveillance des échecs  
4. Documenter et standardiser les patterns de jobs dans votre organisation  
5. Intégrer pg_cron dans vos runbooks et procédures d'exploitation
