# 11.4.1. Stratégies de partitionnement

## Introduction

Le **partitionnement** est une technique qui consiste à diviser une grande table en plusieurs tables plus petites appelées **partitions**. Chaque partition contient un sous-ensemble des données selon des critères définis. Pour l'utilisateur et l'application, la table partitionnée reste transparente : on continue d'interroger la table principale, et PostgreSQL se charge automatiquement de diriger les requêtes vers les bonnes partitions.

### Pourquoi Partitionner ?

Imaginez une table avec **100 millions de lignes**. Sans partitionnement :
- Les recherches scannent potentiellement toute la table
- Les index deviennent énormes et moins efficaces
- La maintenance (VACUUM, ANALYZE) prend énormément de temps
- Les sauvegardes et restaurations sont lentes

Avec le partitionnement :
- ✅ Les requêtes ne scannent que les partitions pertinentes  
- ✅ Les index sont plus petits et plus efficaces  
- ✅ La maintenance peut être faite partition par partition  
- ✅ On peut archiver ou supprimer des partitions entières rapidement  
- ✅ Les performances d'insertion peuvent être améliorées

### Exemple Visuel

```
Sans partitionnement :
┌─────────────────────────────────────────┐
│         ventes (100M lignes)            │
│  Toutes les données mélangées           │
└─────────────────────────────────────────┘

Avec partitionnement par année :
┌─────────────────────────────────────────┐
│         ventes (table parent)           │  ← Interface unifiée
└─────────────────────────────────────────┘
           ↓ partitionnée en
    ┌──────┴───────┬───────────┐
    ↓              ↓           ↓
┌─────────┐  ┌─────────┐  ┌─────────┐
│ventes   │  │ventes   │  │ventes   │
│2023     │  │2024     │  │2025     │
│(30M)    │  │(35M)    │  │(35M)    │
└─────────┘  └─────────┘  └─────────┘
```

**Avantage :** Une requête sur 2025 ne touche qu'une partition de 35M lignes au lieu de 100M.

---

## Les Trois Stratégies de Partitionnement

PostgreSQL propose trois stratégies principales de partitionnement, chacune adaptée à des cas d'usage spécifiques :

| Stratégie | Description | Cas d'usage typique |
|-----------|-------------|---------------------|
| **RANGE** | Partitionner par plage de valeurs | Dates, timestamps, séquences numériques |
| **LIST** | Partitionner par valeurs discrètes | Pays, statuts, catégories fixes |
| **HASH** | Partitionner par hash de la clé | Distribution uniforme, pas de critère naturel |

Chaque stratégie a ses forces et ses cas d'usage optimaux. Voyons-les en détail.

---

## 1. Partitionnement par RANGE (Plage)

### Principe

Le partitionnement **RANGE** divise les données en fonction de **plages de valeurs** d'une colonne. C'est la stratégie la plus courante et la plus intuitive.

**Exemple conceptuel :**
```
Colonne date_commande :
├─ Partition 1 : [2023-01-01 → 2024-01-01)
├─ Partition 2 : [2024-01-01 → 2025-01-01)
└─ Partition 3 : [2025-01-01 → 2026-01-01)
```

Chaque ligne est dirigée vers la partition dont la plage contient sa valeur de date.

### Syntaxe

```sql
-- Créer la table parent partitionnée
CREATE TABLE ventes (
    vente_id BIGSERIAL,
    date_vente DATE NOT NULL,
    client_id INTEGER,
    montant NUMERIC(10,2),
    produit_id INTEGER
) PARTITION BY RANGE (date_vente);
```

**Explication :**
- `PARTITION BY RANGE (date_vente)` : On partitionne selon la colonne `date_vente`
- La table `ventes` devient une table "parent" virtuelle
- Elle ne contient pas de données elle-même
- Les données seront stockées dans les partitions

### Créer les Partitions

```sql
-- Partition pour 2023
CREATE TABLE ventes_2023 PARTITION OF ventes
    FOR VALUES FROM ('2023-01-01') TO ('2024-01-01');

-- Partition pour 2024
CREATE TABLE ventes_2024 PARTITION OF ventes
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

-- Partition pour 2025
CREATE TABLE ventes_2025 PARTITION OF ventes
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');
```

**Points importants :**
- `FOR VALUES FROM (min) TO (max)` définit la plage **[min, max)**
- La borne inférieure est **inclusive** (>=)
- La borne supérieure est **exclusive** (<)
- Les plages **ne doivent pas se chevaucher**
- Il peut y avoir des "trous" (plages non couvertes)

### Insertion et Routage Automatique

```sql
-- Insertion : PostgreSQL route automatiquement vers la bonne partition
INSERT INTO ventes (date_vente, client_id, montant, produit_id) VALUES
    ('2023-06-15', 1001, 150.00, 5),   -- → ventes_2023
    ('2024-03-20', 1002, 200.00, 8),   -- → ventes_2024
    ('2025-11-10', 1003, 75.50, 12);   -- → ventes_2025

-- Vérification : les données sont bien dans les bonnes partitions
SELECT COUNT(*) FROM ventes_2023;  -- 1  
SELECT COUNT(*) FROM ventes_2024;  -- 1  
SELECT COUNT(*) FROM ventes_2025;  -- 1  
```

**Magie :** Vous insérez dans `ventes`, PostgreSQL détermine automatiquement la partition cible.

### Requêtes et Partition Pruning

Le **partition pruning** (élagage de partitions) est l'optimisation clé du partitionnement.

```sql
-- Requête avec filtre sur la colonne de partitionnement
SELECT * FROM ventes WHERE date_vente = '2024-05-10';
```

**Plan d'exécution :**
```
Seq Scan on ventes_2024
  Filter: (date_vente = '2024-05-10'::date)
```

PostgreSQL **scanne uniquement** `ventes_2024`, ignorant complètement les autres partitions. Pas besoin de toucher aux 100M de lignes totales, juste aux 35M de 2024.

```sql
-- Requête sur une plage
EXPLAIN SELECT * FROM ventes  
WHERE date_vente >= '2024-01-01' AND date_vente < '2024-07-01';  
```

**Plan d'exécution :**
```
Append
  ->  Seq Scan on ventes_2024
        Filter: (date_vente >= '2024-01-01' AND date_vente < '2024-07-01')
```

Encore une fois, seule `ventes_2024` est scannée.

### Cas d'Usage Typiques de RANGE

#### 1. Partitionnement par Date/Timestamp

**Le cas le plus fréquent** : données chronologiques.

```sql
-- Logs d'application par mois
CREATE TABLE logs (
    log_id BIGSERIAL,
    timestamp TIMESTAMPTZ NOT NULL,
    niveau VARCHAR(10),
    message TEXT
) PARTITION BY RANGE (timestamp);

-- Créer une partition par mois
CREATE TABLE logs_2025_01 PARTITION OF logs
    FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');

CREATE TABLE logs_2025_02 PARTITION OF logs
    FOR VALUES FROM ('2025-02-01') TO ('2025-03-01');

-- Et ainsi de suite...
```

**Avantages :**
- Archivage facile : supprimer `logs_2024_01` supprime janvier 2024 instantanément
- Maintenance par partition : `VACUUM logs_2025_11` pour novembre seulement
- Requêtes temporelles ultra-rapides

#### 2. Partitionnement par Séquence Numérique

```sql
-- Données IoT avec ID séquentiel
CREATE TABLE mesures_capteurs (
    mesure_id BIGSERIAL,
    capteur_id INTEGER,
    valeur NUMERIC(10,2),
    timestamp TIMESTAMPTZ
) PARTITION BY RANGE (mesure_id);

-- Partitions de 10 millions d'enregistrements
CREATE TABLE mesures_0_10m PARTITION OF mesures_capteurs
    FOR VALUES FROM (0) TO (10000000);

CREATE TABLE mesures_10m_20m PARTITION OF mesures_capteurs
    FOR VALUES FROM (10000000) TO (20000000);

CREATE TABLE mesures_20m_30m PARTITION OF mesures_capteurs
    FOR VALUES FROM (20000000) TO (30000000);
```

#### 3. Partitionnement par Score ou Valeur

```sql
-- Données de scoring client
CREATE TABLE clients (
    client_id SERIAL,
    nom VARCHAR(100),
    score_credit INTEGER CHECK (score_credit BETWEEN 300 AND 850)
) PARTITION BY RANGE (score_credit);

-- Partitions par catégorie de score
CREATE TABLE clients_risque_eleve PARTITION OF clients
    FOR VALUES FROM (300) TO (580);  -- Mauvais score

CREATE TABLE clients_risque_moyen PARTITION OF clients
    FOR VALUES FROM (580) TO (670);  -- Score moyen

CREATE TABLE clients_risque_faible PARTITION OF clients
    FOR VALUES FROM (670) TO (850);  -- Bon score
```

### Gestion de la Partition par Défaut

Que se passe-t-il si vous insérez une valeur hors de toutes les plages définies ?

```sql
-- Sans partition par défaut
INSERT INTO ventes (date_vente, montant) VALUES ('2026-05-10', 100.00);
-- ERREUR : no partition of relation "ventes" found for row
```

**Solution : Créer une partition DEFAULT**

```sql
-- Partition fourre-tout pour les valeurs non couvertes
CREATE TABLE ventes_default PARTITION OF ventes DEFAULT;

-- Maintenant ça fonctionne
INSERT INTO ventes (date_vente, montant) VALUES ('2026-05-10', 100.00);
-- → Ira dans ventes_default

INSERT INTO ventes (date_vente, montant) VALUES ('2022-03-15', 50.00);
-- → Ira aussi dans ventes_default
```

**Usage :** Utile en développement ou pour capturer les données inattendues, mais **à éviter en production** car elle peut devenir un goulot d'étranglement.

## 2. Partitionnement par LIST (Liste)

### Principe

Le partitionnement **LIST** divise les données en fonction de **valeurs discrètes spécifiques** d'une colonne. Chaque partition contient un ensemble explicite de valeurs.

**Exemple conceptuel :**
```
Colonne pays :
├─ Partition Europe : ['FR', 'DE', 'IT', 'ES', 'UK']
├─ Partition Amérique : ['US', 'CA', 'MX', 'BR']
└─ Partition Asie : ['JP', 'CN', 'IN', 'KR']
```

### Syntaxe

```sql
-- Table parent
CREATE TABLE commandes (
    commande_id BIGSERIAL,
    client_id INTEGER,
    pays_livraison VARCHAR(2),  -- Code pays ISO
    montant NUMERIC(10,2),
    date_commande DATE
) PARTITION BY LIST (pays_livraison);
```

### Créer les Partitions

```sql
-- Partition pour l'Europe
CREATE TABLE commandes_europe PARTITION OF commandes
    FOR VALUES IN ('FR', 'DE', 'IT', 'ES', 'UK', 'BE', 'NL', 'PT');

-- Partition pour l'Amérique du Nord
CREATE TABLE commandes_amerique_nord PARTITION OF commandes
    FOR VALUES IN ('US', 'CA', 'MX');

-- Partition pour l'Asie
CREATE TABLE commandes_asie PARTITION OF commandes
    FOR VALUES IN ('JP', 'CN', 'IN', 'KR', 'TH', 'VN');

-- Partition pour l'Océanie
CREATE TABLE commandes_oceanie PARTITION OF commandes
    FOR VALUES IN ('AU', 'NZ');
```

**Points clés :**
- `FOR VALUES IN (...)` : Liste explicite des valeurs
- Une valeur ne peut appartenir qu'à **une seule partition**
- Les valeurs peuvent être des chaînes, entiers, etc.

### Insertion et Routage

```sql
-- Insertions automatiquement routées
INSERT INTO commandes (client_id, pays_livraison, montant) VALUES
    (1001, 'FR', 150.00),  -- → commandes_europe
    (1002, 'US', 200.00),  -- → commandes_amerique_nord
    (1003, 'JP', 350.00),  -- → commandes_asie
    (1004, 'AU', 125.00);  -- → commandes_oceanie
```

### Requêtes avec Partition Pruning

```sql
-- Requête sur la France
SELECT * FROM commandes WHERE pays_livraison = 'FR';
-- Scanne UNIQUEMENT commandes_europe

-- Requête sur l'Amérique du Nord
SELECT * FROM commandes WHERE pays_livraison IN ('US', 'CA');
-- Scanne UNIQUEMENT commandes_amerique_nord
```

### Cas d'Usage Typiques de LIST

#### 1. Partitionnement Géographique

```sql
-- Données utilisateurs par région
CREATE TABLE utilisateurs (
    utilisateur_id SERIAL,
    nom VARCHAR(100),
    email VARCHAR(255),
    region VARCHAR(20)
) PARTITION BY LIST (region);

CREATE TABLE utilisateurs_emea PARTITION OF utilisateurs
    FOR VALUES IN ('Europe', 'Moyen-Orient', 'Afrique');

CREATE TABLE utilisateurs_americas PARTITION OF utilisateurs
    FOR VALUES IN ('Amerique_Nord', 'Amerique_Sud');

CREATE TABLE utilisateurs_apac PARTITION OF utilisateurs
    FOR VALUES IN ('Asie_Pacifique');
```

**Avantages :**
- Conformité RGPD : données européennes physiquement séparées
- Backup par région
- Performance : queries régionales ultra-rapides

#### 2. Partitionnement par Statut

```sql
-- Commandes par statut
CREATE TABLE commandes (
    commande_id BIGSERIAL,
    statut VARCHAR(20),
    montant NUMERIC(10,2),
    date_creation TIMESTAMPTZ
) PARTITION BY LIST (statut);

CREATE TABLE commandes_en_cours PARTITION OF commandes
    FOR VALUES IN ('en_attente', 'en_preparation', 'en_livraison');

CREATE TABLE commandes_terminees PARTITION OF commandes
    FOR VALUES IN ('livree', 'completee');

CREATE TABLE commandes_problemes PARTITION OF commandes
    FOR VALUES IN ('annulee', 'retournee', 'remboursee');
```

**Avantages :**
- Hot data (en cours) vs Cold data (terminées) séparées
- Index optimisés par type de statut
- Archivage facile des commandes terminées

#### 3. Partitionnement par Catégorie

```sql
-- Produits par catégorie
CREATE TABLE produits (
    produit_id SERIAL,
    nom VARCHAR(255),
    categorie VARCHAR(50),
    prix NUMERIC(10,2)
) PARTITION BY LIST (categorie);

CREATE TABLE produits_electronique PARTITION OF produits
    FOR VALUES IN ('ordinateurs', 'smartphones', 'tablettes', 'accessoires_tech');

CREATE TABLE produits_mode PARTITION OF produits
    FOR VALUES IN ('vetements', 'chaussures', 'accessoires_mode');

CREATE TABLE produits_maison PARTITION OF produits
    FOR VALUES IN ('meubles', 'decoration', 'electromenager');
```

#### 4. Partitionnement par Type de Client

```sql
-- Transactions par type de client
CREATE TABLE transactions (
    transaction_id BIGSERIAL,
    client_id INTEGER,
    type_client VARCHAR(20),
    montant NUMERIC(10,2)
) PARTITION BY LIST (type_client);

CREATE TABLE transactions_particuliers PARTITION OF transactions
    FOR VALUES IN ('particulier', 'individuel');

CREATE TABLE transactions_professionnels PARTITION OF transactions
    FOR VALUES IN ('pme', 'entreprise', 'grand_compte');

CREATE TABLE transactions_associations PARTITION OF transactions
    FOR VALUES IN ('association', 'ong', 'collectivite');
```

### Partition DEFAULT pour LIST

Comme avec RANGE, vous pouvez créer une partition par défaut :

```sql
CREATE TABLE commandes_autres PARTITION OF commandes DEFAULT;

-- Toute commande avec un pays non listé ira dans commandes_autres
INSERT INTO commandes (client_id, pays_livraison, montant) VALUES
    (1005, 'BR', 75.00);  -- → commandes_autres (Brésil non listé)
```

## 3. Partitionnement par HASH

### Principe

Le partitionnement **HASH** utilise une fonction de hachage sur une colonne pour distribuer les données **uniformément** entre les partitions. Contrairement à RANGE et LIST, la distribution n'est pas basée sur la valeur réelle mais sur son hash.

**Exemple conceptuel :**
```
Colonne user_id :  
Hash(user_id) modulo 4 :  
├─ Partition 0 : hash % 4 = 0
├─ Partition 1 : hash % 4 = 1
├─ Partition 2 : hash % 4 = 2
└─ Partition 3 : hash % 4 = 3
```

### Syntaxe

```sql
-- Table parent
CREATE TABLE evenements (
    evenement_id BIGSERIAL,
    utilisateur_id INTEGER,
    type_evenement VARCHAR(50),
    timestamp TIMESTAMPTZ,
    donnees JSONB
) PARTITION BY HASH (utilisateur_id);
```

### Créer les Partitions

```sql
-- 4 partitions avec distribution hash
CREATE TABLE evenements_p0 PARTITION OF evenements
    FOR VALUES WITH (MODULUS 4, REMAINDER 0);

CREATE TABLE evenements_p1 PARTITION OF evenements
    FOR VALUES WITH (MODULUS 4, REMAINDER 1);

CREATE TABLE evenements_p2 PARTITION OF evenements
    FOR VALUES WITH (MODULUS 4, REMAINDER 2);

CREATE TABLE evenements_p3 PARTITION OF evenements
    FOR VALUES WITH (MODULUS 4, REMAINDER 3);
```

**Explication :**
- `MODULUS 4` : Nombre total de partitions  
- `REMAINDER 0,1,2,3` : Reste de la division du hash par 4
- Chaque partition reçoit environ 25% des données

### Distribution Automatique

```sql
-- Insertions distribuées automatiquement par hash
INSERT INTO evenements (utilisateur_id, type_evenement) VALUES
    (1001, 'connexion'),    -- Hash(1001) % 4 → Partition X
    (1002, 'achat'),        -- Hash(1002) % 4 → Partition Y
    (1003, 'deconnexion'),  -- Hash(1003) % 4 → Partition Z
    (1004, 'consultation'); -- Hash(1004) % 4 → Partition W
```

La distribution est **imprévisible** pour l'humain mais **uniforme** statistiquement.

### Caractéristiques Importantes

#### 1. Distribution Uniforme

```sql
-- Vérifier la distribution
SELECT
    tableoid::regclass AS partition,
    COUNT(*) AS nb_lignes
FROM evenements  
GROUP BY tableoid::regclass  
ORDER BY partition;  
```

**Résultat typique (après insertion de beaucoup de données) :**
```
partition        | nb_lignes
-----------------+-----------
evenements_p0    | 251234  
evenements_p1    | 249876  
evenements_p2    | 250555  
evenements_p3    | 248335  
```

La distribution est équilibrée (~25% chacune).

#### 2. Pas de Partition Pruning par Plage

```sql
-- Cette requête scanne TOUTES les partitions
SELECT * FROM evenements WHERE utilisateur_id = 1001;
```

PostgreSQL doit calculer le hash de 1001 et ne peut pas éliminer de partitions par plage. Seule la partition contenant hash(1001) % 4 est scannée, mais il faut d'abord déterminer laquelle.

**Limitation :** Les requêtes par plage (`utilisateur_id BETWEEN 1000 AND 2000`) scannent toutes les partitions.

#### 3. Avantage : Parallélisation

Avec HASH, les données sont **naturellement distribuées**, ce qui permet :
- Parallélisation des écritures (chaque partition est indépendante)
- Load balancing entre partitions
- Pas de "hot partition" (partition surchargée)

### Cas d'Usage Typiques de HASH

#### 1. Distribution de Charge (Load Balancing)

```sql
-- Sessions utilisateur avec forte concurrence
CREATE TABLE sessions (
    session_id UUID PRIMARY KEY,
    utilisateur_id INTEGER,
    data JSONB,
    created_at TIMESTAMPTZ DEFAULT NOW()
) PARTITION BY HASH (utilisateur_id);

-- 8 partitions pour distribuer la charge
CREATE TABLE sessions_p0 PARTITION OF sessions
    FOR VALUES WITH (MODULUS 8, REMAINDER 0);
-- ... jusqu'à p7
```

**Avantage :** Les écritures sont distribuées uniformément, réduisant la contention sur les index.

#### 2. Pas de Critère de Partitionnement Naturel

```sql
-- Données sans dimension temporelle ou géographique claire
CREATE TABLE metrics (
    metric_id BIGSERIAL,
    device_id VARCHAR(100),
    metric_name VARCHAR(50),
    value NUMERIC,
    timestamp TIMESTAMPTZ
) PARTITION BY HASH (device_id);

-- 16 partitions
CREATE TABLE metrics_p0 PARTITION OF metrics
    FOR VALUES WITH (MODULUS 16, REMAINDER 0);
-- ... jusqu'à p15
```

**Cas d'usage :** IoT avec des milliards de devices sans structure prévisible.

#### 3. Éviter les Hot Spots

Dans certains systèmes, certaines valeurs (ex: user_id=1 pour l'admin) sont beaucoup plus fréquentes. HASH distribue ces "hot values" de manière équilibrée.

```sql
-- Audit trail avec beaucoup d'actions de certains utilisateurs
CREATE TABLE audit_logs (
    log_id BIGSERIAL,
    user_id INTEGER,
    action VARCHAR(100),
    timestamp TIMESTAMPTZ
) PARTITION BY HASH (user_id);

-- Distribution hash empêche qu'un user_id populaire surcharge une partition
```

### Inconvénients de HASH

❌ **Pas de partition pruning par plage** : `WHERE user_id BETWEEN x AND y` scanne tout  
❌ **Distribution opaque** : Difficile de savoir quelle partition contient quoi  
❌ **Archivage complexe** : Pas de notion temporelle directe

## Comparaison des Stratégies

### Tableau Récapitulatif

| Critère | RANGE | LIST | HASH |
|---------|-------|------|------|
| **Cas d'usage principal** | Dates, séquences | Catégories, régions | Distribution uniforme |
| **Partition pruning** | ✅ Excellent | ✅ Excellent | ⚠️ Limité |
| **Distribution des données** | ⚠️ Variable | ⚠️ Variable | ✅ Uniforme |
| **Archivage** | ✅ Facile (par période) | ✅ Facile (par catégorie) | ❌ Difficile |
| **Maintenance** | ✅ Par période | ✅ Par catégorie | ⚠️ Toutes partitions |
| **Ajout de partitions** | ✅ Simple | ✅ Simple | ⚠️ Complexe |
| **Requêtes par plage** | ✅ Optimales | ❌ Scan complet | ❌ Scan complet |
| **Requêtes par égalité** | ✅ Rapide | ✅ Très rapide | ✅ Rapide |
| **Hot spots** | ⚠️ Possible | ⚠️ Possible | ✅ Évités |
| **Complexité** | ⭐⭐ Moyenne | ⭐ Simple | ⭐⭐⭐ Complexe |

### Arbre de Décision

```
Quel type de colonne utilisez-vous ?
│
├─ Date ou Timestamp ?
│  └─→ RANGE (par jour/mois/année)
│
├─ Catégories fixes (pays, statut, type) ?
│  └─→ LIST
│
├─ Valeurs numériques séquentielles (ID, score) ?
│  └─→ RANGE
│
├─ Pas de critère naturel, besoin de distribution uniforme ?
│  └─→ HASH
│
└─ Données à forte concurrence ?
   └─→ HASH
```

## Sous-Partitionnement (Partitionnement Multi-Niveaux)

PostgreSQL permet de **combiner** les stratégies en créant des **sous-partitions**.

### Exemple : RANGE puis LIST

**Cas d'usage :** Commandes partitionnées par année, puis par région dans chaque année.

```sql
-- Table parent : partitionnement par RANGE sur l'année
CREATE TABLE commandes (
    commande_id BIGSERIAL,
    date_commande DATE NOT NULL,
    pays VARCHAR(2),
    montant NUMERIC(10,2)
) PARTITION BY RANGE (date_commande);

-- Partition 2024 (elle-même partitionnée)
CREATE TABLE commandes_2024 PARTITION OF commandes
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01')
    PARTITION BY LIST (pays);

-- Sous-partitions de 2024 par région
CREATE TABLE commandes_2024_europe PARTITION OF commandes_2024
    FOR VALUES IN ('FR', 'DE', 'IT', 'ES', 'UK');

CREATE TABLE commandes_2024_amerique PARTITION OF commandes_2024
    FOR VALUES IN ('US', 'CA', 'MX');

CREATE TABLE commandes_2024_asie PARTITION OF commandes_2024
    FOR VALUES IN ('JP', 'CN', 'IN', 'KR');

-- Partition 2025 (pareil)
CREATE TABLE commandes_2025 PARTITION OF commandes
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01')
    PARTITION BY LIST (pays);

CREATE TABLE commandes_2025_europe PARTITION OF commandes_2025
    FOR VALUES IN ('FR', 'DE', 'IT', 'ES', 'UK');

CREATE TABLE commandes_2025_amerique PARTITION OF commandes_2025
    FOR VALUES IN ('US', 'CA', 'MX');

CREATE TABLE commandes_2025_asie PARTITION OF commandes_2025
    FOR VALUES IN ('JP', 'CN', 'IN', 'KR');
```

**Hiérarchie résultante :**
```
commandes (parent)
├─ commandes_2024
│  ├─ commandes_2024_europe
│  ├─ commandes_2024_amerique
│  └─ commandes_2024_asie
└─ commandes_2025
   ├─ commandes_2025_europe
   ├─ commandes_2025_amerique
   └─ commandes_2025_asie
```

### Requêtes avec Sous-Partitionnement

```sql
-- Requête sur 2024 en Europe
SELECT * FROM commandes  
WHERE date_commande >= '2024-01-01'  
  AND date_commande < '2025-01-01'
  AND pays = 'FR';
```

**Partition pruning :**
- Élimine commandes_2025 (date hors plage)
- Dans commandes_2024, élimine amerique et asie (pays hors liste)
- Scanne UNIQUEMENT commandes_2024_europe

**Performance optimale !**

### Exemple : RANGE puis HASH

**Cas d'usage :** Logs partitionnés par mois, puis distribués par hash pour équilibrer.

```sql
-- Table parent : RANGE sur le mois
CREATE TABLE logs (
    log_id BIGSERIAL,
    timestamp TIMESTAMPTZ NOT NULL,
    user_id INTEGER,
    message TEXT
) PARTITION BY RANGE (timestamp);

-- Partition novembre 2025 (sous-partitionnée par HASH)
CREATE TABLE logs_2025_11 PARTITION OF logs
    FOR VALUES FROM ('2025-11-01') TO ('2025-12-01')
    PARTITION BY HASH (user_id);

-- Sous-partitions hash dans novembre 2025
CREATE TABLE logs_2025_11_h0 PARTITION OF logs_2025_11
    FOR VALUES WITH (MODULUS 4, REMAINDER 0);

CREATE TABLE logs_2025_11_h1 PARTITION OF logs_2025_11
    FOR VALUES WITH (MODULUS 4, REMAINDER 1);

CREATE TABLE logs_2025_11_h2 PARTITION OF logs_2025_11
    FOR VALUES WITH (MODULUS 4, REMAINDER 2);

CREATE TABLE logs_2025_11_h3 PARTITION OF logs_2025_11
    FOR VALUES WITH (MODULUS 4, REMAINDER 3);
```

**Avantages :**
- Archivage facile par mois (RANGE)
- Distribution uniforme dans chaque mois (HASH)
- Parallélisation des écritures

## Stratégies Avancées et Considérations

### 1. Taille Optimale des Partitions

**Règle générale :** Visez des partitions de **quelques gigaoctets à quelques dizaines de gigaoctets**.

❌ **Trop petites** (< 1 GB) :
- Trop de partitions → overhead du planificateur
- Gestion complexe
- Bénéfices limités

❌ **Trop grandes** (> 100 GB) :
- Maintenance longue
- Peu de bénéfice du partitionnement

✅ **Idéal** (5-50 GB) :
- Bon équilibre performance/gestion
- Index de taille raisonnable
- Maintenance rapide

### 2. Nombre de Partitions

**RANGE et LIST :**
- Quelques dizaines à quelques centaines : OK
- Au-delà de 1000 partitions : attention aux performances du planificateur

**HASH :**
- Choisir un nombre qui divise bien votre charge
- Puissances de 2 recommandées : 4, 8, 16, 32, 64
- Ne pas dépasser 256 partitions sauf cas extrême

### 3. Choix de la Colonne de Partitionnement

✅ **Bonne colonne :**
- Utilisée fréquemment dans les WHERE
- Distribution équilibrée (pour RANGE/LIST)
- Immuable (ne change pas après insertion)

❌ **Mauvaise colonne :**
- Rarement utilisée dans les filtres
- Distribution très déséquilibrée
- Peut changer (ex: statut qui évolue)

### 4. Partitionnement et Performance d'Écriture

**Impact positif :**
- ✅ Réduction de la contention sur les index  
- ✅ Parallélisation possible

**Impact négatif potentiel :**
- ⚠️ Overhead de routage (généralement négligeable)  
- ⚠️ Plus d'index à maintenir globalement

**En pratique :** L'impact sur les écritures est généralement neutre ou légèrement positif.

## Exemples Pratiques par Secteur

### E-Commerce

```sql
-- Commandes : RANGE par date
CREATE TABLE commandes (
    commande_id BIGSERIAL,
    date_commande TIMESTAMPTZ NOT NULL,
    client_id INTEGER,
    montant NUMERIC(10,2)
) PARTITION BY RANGE (date_commande);

-- Partition par mois (conservation 2 ans)
CREATE TABLE commandes_2025_11 PARTITION OF commandes
    FOR VALUES FROM ('2025-11-01') TO ('2025-12-01');
```

### Finance

```sql
-- Transactions : RANGE par date avec sous-partitionnement LIST par type
CREATE TABLE transactions (
    transaction_id BIGSERIAL,
    date_transaction DATE NOT NULL,
    type_transaction VARCHAR(20),
    montant NUMERIC(15,2)
) PARTITION BY RANGE (date_transaction);

CREATE TABLE transactions_2025_q4 PARTITION OF transactions
    FOR VALUES FROM ('2025-10-01') TO ('2026-01-01')
    PARTITION BY LIST (type_transaction);

CREATE TABLE transactions_2025_q4_debit PARTITION OF transactions_2025_q4
    FOR VALUES IN ('debit', 'retrait');

CREATE TABLE transactions_2025_q4_credit PARTITION OF transactions_2025_q4
    FOR VALUES IN ('credit', 'depot');
```

### Télécommunications

```sql
-- CDR (Call Detail Records) : HASH pour distribution
CREATE TABLE cdr (
    cdr_id BIGSERIAL,
    phone_number VARCHAR(20),
    call_timestamp TIMESTAMPTZ,
    duration INTEGER,
    cost NUMERIC(8,4)
) PARTITION BY HASH (phone_number);

-- 32 partitions pour 100M+ appels/jour
CREATE TABLE cdr_p0 PARTITION OF cdr
    FOR VALUES WITH (MODULUS 32, REMAINDER 0);
-- ... jusqu'à p31
```

### IoT / Capteurs

```sql
-- Mesures : RANGE par timestamp
CREATE TABLE mesures (
    mesure_id BIGSERIAL,
    capteur_id INTEGER,
    timestamp TIMESTAMPTZ NOT NULL,
    valeur NUMERIC(10,4)
) PARTITION BY RANGE (timestamp);

-- Partition par jour avec rétention 90 jours
CREATE TABLE mesures_2025_11_20 PARTITION OF mesures
    FOR VALUES FROM ('2025-11-20 00:00:00') TO ('2025-11-21 00:00:00');
```

### SaaS Multi-Tenant

```sql
-- Données tenant : LIST par tenant_id ou HASH
CREATE TABLE tenant_data (
    data_id BIGSERIAL,
    tenant_id INTEGER,
    data JSONB,
    created_at TIMESTAMPTZ
) PARTITION BY LIST (tenant_id);

-- Un tenant important dans sa propre partition
CREATE TABLE tenant_data_1001 PARTITION OF tenant_data
    FOR VALUES IN (1001);

-- Petits tenants groupés
CREATE TABLE tenant_data_small PARTITION OF tenant_data
    FOR VALUES IN (2001, 2002, 2003, 2004, 2005);
```

## Checklist de Décision

Avant de partitionner, répondez à ces questions :

### 1. Ai-je Besoin de Partitionner ?

- [ ] Ma table dépasse 10 GB ?  
- [ ] J'ai des problèmes de performance mesurés ?  
- [ ] Je veux archiver/supprimer des données régulièrement ?  
- [ ] J'ai besoin de maintenance granulaire ?

Si **NON** à tout → Pas besoin de partitionner

### 2. Quelle Stratégie ?

- [ ] Mes données ont une dimension temporelle ? → **RANGE**  
- [ ] Mes données ont des catégories fixes ? → **LIST**  
- [ ] Je veux distribuer la charge uniformément ? → **HASH**  
- [ ] J'ai deux dimensions (temps + région) ? → **Sous-partitionnement**

### 3. Configuration

- [ ] Taille cible par partition : 5-50 GB  
- [ ] Nombre de partitions : < 500 idéalement  
- [ ] Colonne immuable et fréquemment filtrée  
- [ ] Plan de création/archivage des partitions

---

## Conclusion

### Points Clés à Retenir

1. **RANGE** : Pour les données temporelles ou séquentielles  
   - ✅ Cas le plus fréquent  
   - ✅ Archivage facile  
   - ✅ Partition pruning excellent

2. **LIST** : Pour les catégories et valeurs discrètes  
   - ✅ Distribution par région, statut, type  
   - ✅ Séparation logique claire  
   - ✅ Gestion par métier

3. **HASH** : Pour la distribution uniforme  
   - ✅ Pas de critère naturel  
   - ✅ Équilibrage de charge  
   - ⚠️ Moins de partition pruning

4. **Sous-partitionnement** : Combiner les stratégies  
   - ✅ Maximum de flexibilité  
   - ⚠️ Complexité accrue

### Recommandations Générales

**Pour débuter :**
- Commencez par **RANGE sur une date** (le plus simple et fréquent)
- Créez vos partitions au fur et à mesure (pas besoin de toutes les créer d'avance)
- Testez avec des données réelles avant la production

**Pour optimiser :**
- Mesurez l'impact avec `EXPLAIN ANALYZE`
- Surveillez la distribution des données
- Ajustez la granularité des partitions selon les besoins

**Pour maintenir :**
- Automatisez la création de partitions (voir chapitre 11.4.3)
- Planifiez l'archivage/suppression
- Documentez votre stratégie
