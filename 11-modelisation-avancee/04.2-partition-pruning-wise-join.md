# 11.4.2. Partition Pruning et Partition-wise Join

## Introduction

Le **partitionnement** est une technique puissante pour gérer de grandes tables, mais son véritable intérêt réside dans les **optimisations** que PostgreSQL applique automatiquement lors de l'exécution des requêtes. Les deux optimisations majeures sont :

1. **Partition Pruning** (Élagage de partitions) : PostgreSQL élimine les partitions non nécessaires  
2. **Partition-wise Join** (Jointure par partition) : PostgreSQL joint les partitions correspondantes directement

Ces optimisations peuvent **transformer des requêtes qui prendraient des minutes en requêtes qui s'exécutent en secondes**.

### Analogie Simple

Imaginez une bibliothèque avec 1 million de livres :

**Sans partitionnement :**
```
Chercher un livre de 2024 = Parcourir 1 million de livres
```

**Avec partitionnement mais sans pruning :**
```
Livres organisés par année, mais on cherche quand même partout
```

**Avec partitionnement ET pruning :**
```
Livres organisés par année → Aller directement à l'étagère 2024  
Chercher parmi seulement 50 000 livres au lieu de 1 million  
```

C'est exactement ce que fait le partition pruning : **éliminer intelligemment les partitions inutiles**.

## Partie 1 : Partition Pruning (Élagage de Partitions)

### Qu'est-ce que le Partition Pruning ?

Le **partition pruning** est le processus par lequel PostgreSQL analyse votre requête et détermine quelles partitions il peut **ignorer complètement** parce qu'elles ne peuvent pas contenir de données pertinentes.

### Exemple de Base

```sql
-- Table partitionnée par année
CREATE TABLE ventes (
    vente_id BIGSERIAL,
    date_vente DATE NOT NULL,
    client_id INTEGER,
    montant NUMERIC(10,2)
) PARTITION BY RANGE (date_vente);

-- Créer 3 partitions
CREATE TABLE ventes_2023 PARTITION OF ventes
    FOR VALUES FROM ('2023-01-01') TO ('2024-01-01');

CREATE TABLE ventes_2024 PARTITION OF ventes
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

CREATE TABLE ventes_2025 PARTITION OF ventes
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');

-- Insérer des données
INSERT INTO ventes (date_vente, client_id, montant)  
SELECT  
    '2023-01-01'::date + (random() * 365)::integer,
    (random() * 10000)::integer,
    (random() * 1000)::numeric(10,2)
FROM generate_series(1, 1000000);  -- 1M de lignes en 2023

INSERT INTO ventes (date_vente, client_id, montant)  
SELECT  
    '2024-01-01'::date + (random() * 365)::integer,
    (random() * 10000)::integer,
    (random() * 1000)::numeric(10,2)
FROM generate_series(1, 1000000);  -- 1M de lignes en 2024

INSERT INTO ventes (date_vente, client_id, montant)  
SELECT  
    '2025-01-01'::date + (random() * 365)::integer,
    (random() * 10000)::integer,
    (random() * 1000)::numeric(10,2)
FROM generate_series(1, 1000000);  -- 1M de lignes en 2025
```

Nous avons maintenant **3 millions de lignes** réparties sur 3 partitions.

### Requête Avec Partition Pruning

```sql
-- Chercher les ventes de juin 2024
EXPLAIN SELECT * FROM ventes  
WHERE date_vente BETWEEN '2024-06-01' AND '2024-06-30';  
```

**Plan d'exécution :**
```
Seq Scan on ventes_2024 ventes
  Filter: (date_vente >= '2024-06-01'::date AND date_vente <= '2024-06-30'::date)
```

**Analyse :**
- PostgreSQL scanne **UNIQUEMENT** `ventes_2024`
- Les partitions `ventes_2023` et `ventes_2025` sont **complètement ignorées**
- Au lieu de scanner 3M de lignes, on scanne seulement 1M
- **Performance multipliée par 3** grâce au pruning !

### Visualisation du Pruning

```
Requête : WHERE date_vente BETWEEN '2024-06-01' AND '2024-06-30'

Sans Partition Pruning :
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│ ventes_2023 │  │ ventes_2024 │  │ ventes_2025 │
│   (1M)      │  │   (1M)      │  │   (1M)      │
└─────────────┘  └─────────────┘  └─────────────┘
      ↓                 ↓                 ↓
   Scanner          Scanner          Scanner
   (inutile)        (utile)         (inutile)

Avec Partition Pruning :
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│ ventes_2023 │  │ ventes_2024 │  │ ventes_2025 │
│   (1M)      │  │   (1M)      │  │   (1M)      │
└─────────────┘  └─────────────┘  └─────────────┘
      ✗                 ↓                 ✗
   Ignorée          Scanner           Ignorée
                    (utile)
```

## Comment Fonctionne le Partition Pruning ?

### Phase de Planification

Le partition pruning peut se produire à **deux moments** :

1. **Plan-time pruning** : Pendant la phase de planification (avant exécution)  
2. **Run-time pruning** : Pendant l'exécution (pour les paramètres dynamiques)

### 1. Plan-Time Pruning (Pruning à la Planification)

C'est le cas le plus courant et le plus efficace.

```sql
-- Requête avec valeurs constantes
EXPLAIN ANALYZE SELECT * FROM ventes WHERE date_vente = '2024-05-15';
```

**Plan d'exécution :**
```
Seq Scan on ventes_2024 ventes (actual time=0.015..25.367 rows=3 loops=1)
  Filter: (date_vente = '2024-05-15'::date)
  Rows Removed by Filter: 999997
Planning Time: 0.234 ms  
Execution Time: 25.389 ms  
```

**Ce qui s'est passé :**
1. PostgreSQL analyse la condition `date_vente = '2024-05-15'`  
2. Il calcule que seule `ventes_2024` peut contenir cette date  
3. Le plan généré **ne mentionne même pas** les autres partitions  
4. Elles sont éliminées **avant l'exécution**

### 2. Run-Time Pruning (Pruning à l'Exécution)

Nécessaire quand les valeurs de filtre ne sont pas connues à la planification.

```sql
-- Requête préparée avec paramètre
PREPARE ventes_mois(date) AS
    SELECT * FROM ventes WHERE date_vente = $1;

-- Exécution avec différentes dates
EXECUTE ventes_mois('2024-06-15');  
EXECUTE ventes_mois('2023-03-20');  
EXECUTE ventes_mois('2025-09-10');  
```

**Fonctionnement :**
1. Le plan est créé **une fois** sans connaître les valeurs  
2. À chaque exécution, PostgreSQL élimine les partitions **dynamiquement**  
3. Légèrement moins efficace que plan-time pruning, mais toujours très performant

## Cas où le Partition Pruning Fonctionne

### ✅ 1. Égalité Simple

```sql
-- Fonctionne parfaitement
SELECT * FROM ventes WHERE date_vente = '2024-05-15';
```

### ✅ 2. Comparaisons de Plage

```sql
-- Fonctionne
SELECT * FROM ventes WHERE date_vente >= '2024-01-01';

-- Fonctionne
SELECT * FROM ventes WHERE date_vente < '2024-07-01';

-- Fonctionne
SELECT * FROM ventes WHERE date_vente BETWEEN '2024-03-01' AND '2024-03-31';
```

### ✅ 3. Opérateurs IN

```sql
-- Fonctionne
SELECT * FROM ventes WHERE date_vente IN ('2024-01-15', '2024-02-20', '2024-03-10');
```

### ✅ 4. Fonctions Immuables

```sql
-- Fonctionne (CURRENT_DATE est évalué une fois)
SELECT * FROM ventes WHERE date_vente = CURRENT_DATE;

-- Fonctionne
SELECT * FROM ventes WHERE date_vente = DATE_TRUNC('month', CURRENT_DATE);
```

### ✅ 5. Combinaisons avec AND

```sql
-- Fonctionne : pruning sur date_vente
SELECT * FROM ventes  
WHERE date_vente = '2024-05-15'  
  AND montant > 100;
```

---

## Cas où le Partition Pruning NE Fonctionne PAS (ou Mal)

### ❌ 1. OR sur Différentes Colonnes

```sql
-- Pruning impossible : conditions sur différentes colonnes
SELECT * FROM ventes  
WHERE date_vente = '2024-05-15' OR montant > 1000;  
```

**Résultat :** Toutes les partitions sont scannées car `montant > 1000` peut être vrai n'importe où.

### ❌ 2. Fonctions Non-Immuables sur la Colonne de Partitionnement

```sql
-- Ne fonctionne PAS bien
SELECT * FROM ventes WHERE EXTRACT(YEAR FROM date_vente) = 2024;
```

**Problème :** PostgreSQL ne peut pas facilement déterminer quelles partitions correspondent.

**Solution :** Réécrire la requête :
```sql
-- ✅ Fonctionne : pruning actif
SELECT * FROM ventes  
WHERE date_vente >= '2024-01-01' AND date_vente < '2025-01-01';  
```

### ❌ 3. Colonnes Non-Partitionnées dans le Filtre Principal

```sql
-- Pruning impossible
SELECT * FROM ventes WHERE client_id = 1001;
```

Si vous partitionnez par `date_vente` mais filtrez sur `client_id`, aucune partition ne peut être éliminée.

### ⚠️ 4. OR avec Plages qui Se Chevauchent

```sql
-- Pruning partiel seulement
SELECT * FROM ventes  
WHERE date_vente = '2024-05-15' OR date_vente = '2025-06-20';  
```

**Résultat :** Deux partitions scannées (`ventes_2024` et `ventes_2025`), ce qui est correct.

## Vérifier le Partition Pruning avec EXPLAIN

### Méthode 1 : EXPLAIN Simple

```sql
EXPLAIN SELECT * FROM ventes WHERE date_vente = '2024-06-15';
```

**Ce qu'il faut chercher :**
- Le nombre de partitions mentionnées dans le plan
- Si seule `ventes_2024` apparaît → Pruning réussi

### Méthode 2 : EXPLAIN ANALYZE pour Confirmation

```sql
EXPLAIN ANALYZE SELECT * FROM ventes WHERE date_vente = '2024-06-15';
```

**Exemple de sortie :**
```
Seq Scan on ventes_2024 ventes (actual time=0.012..18.456 rows=2 loops=1)
  Filter: (date_vente = '2024-06-15'::date)
Planning Time: 0.123 ms  
Execution Time: 18.478 ms  
```

**Indicateurs de pruning efficace :**
- Une seule partition mentionnée
- Temps de planification faible
- Temps d'exécution réduit

### Méthode 3 : Comparer avec ONLY

```sql
-- Avec pruning automatique
EXPLAIN ANALYZE  
SELECT COUNT(*) FROM ventes WHERE date_vente BETWEEN '2024-01-01' AND '2024-12-31';  

-- Sans pruning (forcer le scan de la table parent uniquement)
EXPLAIN ANALYZE  
SELECT COUNT(*) FROM ONLY ventes WHERE date_vente BETWEEN '2024-01-01' AND '2024-12-31';  
```

La première requête devrait être **beaucoup plus rapide**.

## Configuration du Partition Pruning

### Paramètre enable_partition_pruning

```sql
-- Vérifier le statut (devrait être ON par défaut)
SHOW enable_partition_pruning;
-- Résultat: on

-- Désactiver temporairement (pour tester)
SET enable_partition_pruning = off;

-- Réactiver
SET enable_partition_pruning = on;
```

**Note :** Il est **très rare** de vouloir désactiver cette fonctionnalité. Elle est presque toujours bénéfique.

### Paramètre constraint_exclusion (Legacy)

Avant PostgreSQL 11, on utilisait `constraint_exclusion` pour un effet similaire.

```sql
-- Vérification
SHOW constraint_exclusion;
-- Valeurs possibles: on, off, partition (recommandé)

-- Configuration recommandée
SET constraint_exclusion = partition;
```

**Recommandation :** Laisser à `partition` (par défaut depuis PG 11+).

## Partie 2 : Partition-wise Join

### Qu'est-ce qu'un Partition-wise Join ?

Un **partition-wise join** se produit quand PostgreSQL joint deux tables partitionnées **partition par partition** au lieu de joindre les tables entières.

### Analogie

Imaginez deux bibliothèques organisées par année :

**Join classique :**
```
Comparer chaque livre de la bibliothèque A  
avec chaque livre de la bibliothèque B  
= Beaucoup de comparaisons inutiles
```

**Partition-wise join :**
```
Joindre les livres de 2024 de A avec les livres de 2024 de B  
Joindre les livres de 2023 de A avec les livres de 2023 de B  
etc.  
= Comparaisons ciblées et efficaces
```

### Exemple de Configuration

```sql
-- Tables partitionnées de la même manière
CREATE TABLE commandes (
    commande_id BIGSERIAL,
    date_commande DATE NOT NULL,
    client_id INTEGER,
    montant NUMERIC(10,2)
) PARTITION BY RANGE (date_commande);

CREATE TABLE paiements (
    paiement_id BIGSERIAL,
    commande_id BIGINT,
    date_paiement DATE NOT NULL,
    montant_paye NUMERIC(10,2)
) PARTITION BY RANGE (date_paiement);

-- Partitions pour commandes
CREATE TABLE commandes_2024 PARTITION OF commandes
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

CREATE TABLE commandes_2025 PARTITION OF commandes
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');

-- Partitions pour paiements (même structure)
CREATE TABLE paiements_2024 PARTITION OF paiements
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

CREATE TABLE paiements_2025 PARTITION OF paiements
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');
```

### Requête avec Partition-wise Join

```sql
-- Activer partition-wise join
SET enable_partitionwise_join = on;

-- Jointure
EXPLAIN SELECT
    c.commande_id,
    c.date_commande,
    p.montant_paye
FROM commandes c  
JOIN paiements p ON c.date_commande = p.date_paiement  
WHERE c.date_commande >= '2024-01-01' AND c.date_commande < '2025-01-01';  
```

**Plan d'exécution (simplifié) :**
```
Append
  -> Hash Join
       Hash Cond: (c.date_commande = p.date_paiement)
       -> Seq Scan on commandes_2024 c
       -> Hash
            -> Seq Scan on paiements_2024 p
```

**Ce qui s'est passé :**
1. Partition pruning élimine les partitions 2025  
2. Partition-wise join joint directement `commandes_2024` avec `paiements_2024`  
3. Pas besoin de traiter les données 2025 du tout

### Visualisation

```
Sans Partition-wise Join :
┌───────────────┐     ┌───────────────┐
│  commandes    │     │  paiements    │
│  (toutes les  │     │  (toutes les  │
│   partitions) │ →→→ │   partitions) │
└───────────────┘     └───────────────┘
        ↓                     ↓
    Joindre tout avec tout
    (coûteux)

Avec Partition-wise Join :
┌───────────────┐     ┌───────────────┐
│ commandes_2024│ →→→ │ paiements_2024│
└───────────────┘     └───────────────┘
        ↓                     ↓
    Join ciblé (rapide)

┌───────────────┐     ┌───────────────┐
│ commandes_2025│ →→→ │ paiements_2025│
└───────────────┘     └───────────────┘
        ↓                     ↓
    Join ciblé (rapide)
```

## Conditions pour Partition-wise Join

Pour qu'un partition-wise join soit possible, plusieurs conditions doivent être réunies :

### ✅ 1. Tables Partitionnées de Manière Compatible

Les deux tables doivent être partitionnées avec :
- La même stratégie (RANGE, LIST ou HASH)
- Le même nombre de partitions
- Les mêmes bornes/valeurs de partition

```sql
-- Compatible : même partitionnement
CREATE TABLE table_a (...) PARTITION BY RANGE (date_col);  
CREATE TABLE table_b (...) PARTITION BY RANGE (date_col);  

CREATE TABLE table_a_2024 PARTITION OF table_a
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');
CREATE TABLE table_b_2024 PARTITION OF table_b
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');
```

### ✅ 2. Jointure sur la Colonne de Partitionnement

```sql
-- ✅ Fonctionne
SELECT * FROM table_a a  
JOIN table_b b ON a.date_col = b.date_col;  

-- ❌ Ne fonctionne pas (join sur autre colonne)
SELECT * FROM table_a a  
JOIN table_b b ON a.id = b.id;  
```

### ✅ 3. enable_partitionwise_join Activé

```sql
-- Vérifier
SHOW enable_partitionwise_join;

-- Activer si nécessaire
SET enable_partitionwise_join = on;
```

**Note :** Désactivé par défaut pour des raisons de compatibilité, mais **recommandé de l'activer** si vous utilisez le partitionnement.

---

## Avantages du Partition-wise Join

### 1. Réduction de la Mémoire

**Sans partition-wise join :**
- La table complète (toutes partitions) doit être chargée en mémoire pour la jointure
- Peut dépasser la RAM disponible

**Avec partition-wise join :**
- Seule une partition à la fois est traitée
- Empreinte mémoire beaucoup plus faible

### 2. Parallélisation Possible

Chaque partition peut être jointe en parallèle :

```sql
-- Configuration
SET max_parallel_workers_per_gather = 4;  
SET enable_partitionwise_join = on;  

-- Jointure parallèle sur partitions
EXPLAIN SELECT * FROM commandes c  
JOIN paiements p ON c.date_commande = p.date_paiement;  
```

**Plan d'exécution :**
```
Gather
  Workers Planned: 4
  -> Parallel Append
       -> Parallel Hash Join
            -> Parallel Seq Scan on commandes_2024 c
            -> Parallel Hash
                 -> Parallel Seq Scan on paiements_2024 p
       -> Parallel Hash Join
            -> Parallel Seq Scan on commandes_2025 c_1
            -> Parallel Hash
                 -> Parallel Seq Scan on paiements_2025 p_1
```

Les partitions sont traitées **en parallèle** par différents workers.

### 3. Amélioration des Performances

**Benchmark typique :**
```
Sans partition-wise join : 45 secondes  
Avec partition-wise join : 12 secondes  
Gain : ~73% plus rapide  
```

## Partition-wise Aggregate

PostgreSQL supporte aussi les **agrégations par partition**.

### Concept

Au lieu d'agréger toutes les données puis grouper, PostgreSQL agrège chaque partition indépendamment puis combine les résultats.

### Exemple

```sql
-- Activer
SET enable_partitionwise_aggregate = on;

-- Agrégation
EXPLAIN SELECT
    DATE_TRUNC('month', date_vente) as mois,
    COUNT(*) as nb_ventes,
    SUM(montant) as total
FROM ventes  
WHERE date_vente >= '2024-01-01' AND date_vente < '2025-01-01'  
GROUP BY DATE_TRUNC('month', date_vente);  
```

**Plan d'exécution optimisé :**
```
Finalize GroupAggregate
  Group Key: (date_trunc('month', date_vente))
  -> Gather Merge
       -> Partial GroupAggregate
            Group Key: (date_trunc('month', date_vente))
            -> Seq Scan on ventes_2024
```

Chaque partition calcule ses agrégats localement, puis ils sont fusionnés.

## Configuration Optimale

### Paramètres à Activer

```sql
-- Dans postgresql.conf ou via SET

-- Partition pruning (activé par défaut)
enable_partition_pruning = on

-- Partition-wise join (désactivé par défaut)
enable_partitionwise_join = on

-- Partition-wise aggregate (désactivé par défaut)
enable_partitionwise_aggregate = on

-- Constraint exclusion (pour compatibilité legacy)
constraint_exclusion = partition
```

### Configuration Recommandée pour Production

```sql
-- postgresql.conf
enable_partition_pruning = on  
enable_partitionwise_join = on  
enable_partitionwise_aggregate = on  
constraint_exclusion = partition  

-- Parallélisme (ajuster selon votre hardware)
max_parallel_workers_per_gather = 4  
max_parallel_workers = 8  
```

## Exemples Pratiques Complets

### Exemple 1 : E-Commerce avec Commandes et Lignes

```sql
-- Tables partitionnées
CREATE TABLE commandes (
    commande_id BIGSERIAL,
    date_commande DATE NOT NULL,
    client_id INTEGER
) PARTITION BY RANGE (date_commande);

CREATE TABLE lignes_commande (
    ligne_id BIGSERIAL,
    commande_id BIGINT,
    date_commande DATE NOT NULL,  -- Dénormalisé pour partitionnement
    produit_id INTEGER,
    quantite INTEGER
) PARTITION BY RANGE (date_commande);

-- Partitions 2024
CREATE TABLE commandes_2024 PARTITION OF commandes
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

CREATE TABLE lignes_commande_2024 PARTITION OF lignes_commande
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

-- Partitions 2025
CREATE TABLE commandes_2025 PARTITION OF commandes
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');

CREATE TABLE lignes_commande_2025 PARTITION OF lignes_commande
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');

-- Activer les optimisations
SET enable_partition_pruning = on;  
SET enable_partitionwise_join = on;  

-- Requête optimisée
EXPLAIN ANALYZE  
SELECT  
    c.commande_id,
    c.client_id,
    COUNT(l.ligne_id) as nb_produits,
    SUM(l.quantite) as quantite_totale
FROM commandes c  
JOIN lignes_commande l ON c.commande_id = l.commande_id  
    AND c.date_commande = l.date_commande  -- Important pour partition-wise join
WHERE c.date_commande >= '2024-06-01' AND c.date_commande < '2024-07-01'  
GROUP BY c.commande_id, c.client_id;  
```

**Optimisations appliquées :**
1. ✅ Partition pruning élimine toutes les partitions sauf `commandes_2024` et `lignes_2024`  
2. ✅ Partition-wise join entre les deux partitions 2024  
3. ✅ Agrégation sur partition unique

### Exemple 2 : Logs avec Métriques

```sql
-- Table de logs partitionnée
CREATE TABLE logs_app (
    log_id BIGSERIAL,
    timestamp TIMESTAMPTZ NOT NULL,
    user_id INTEGER,
    action VARCHAR(50),
    duree_ms INTEGER
) PARTITION BY RANGE (timestamp);

-- Partitions par semaine
CREATE TABLE logs_2025_w46 PARTITION OF logs_app
    FOR VALUES FROM ('2025-11-10') TO ('2025-11-17');

CREATE TABLE logs_2025_w47 PARTITION OF logs_app
    FOR VALUES FROM ('2025-11-17') TO ('2025-11-24');

-- Table métriques partitionnée pareil
CREATE TABLE metriques_app (
    metrique_id BIGSERIAL,
    timestamp TIMESTAMPTZ NOT NULL,
    cpu_percent NUMERIC(5,2),
    memory_mb INTEGER
) PARTITION BY RANGE (timestamp);

CREATE TABLE metriques_2025_w46 PARTITION OF metriques_app
    FOR VALUES FROM ('2025-11-10') TO ('2025-11-17');

CREATE TABLE metriques_2025_w47 PARTITION OF metriques_app
    FOR VALUES FROM ('2025-11-17') TO ('2025-11-24');

-- Requête corrélée
SET enable_partition_pruning = on;  
SET enable_partitionwise_join = on;  

EXPLAIN ANALYZE  
SELECT  
    DATE_TRUNC('hour', l.timestamp) as heure,
    COUNT(l.log_id) as nb_actions,
    AVG(m.cpu_percent) as cpu_moyen
FROM logs_app l  
JOIN metriques_app m ON DATE_TRUNC('minute', l.timestamp) = DATE_TRUNC('minute', m.timestamp)  
WHERE l.timestamp >= '2025-11-15' AND l.timestamp < '2025-11-16'  
GROUP BY DATE_TRUNC('hour', l.timestamp)  
ORDER BY heure;  
```

## Diagnostic et Troubleshooting

### Problème 1 : Partition Pruning Ne Fonctionne Pas

**Symptômes :**
```sql
EXPLAIN SELECT * FROM ventes WHERE date_vente = '2024-05-15';
-- Affiche Append avec toutes les partitions
```

**Solutions :**

1. Vérifier que `enable_partition_pruning` est ON :
```sql
SHOW enable_partition_pruning;  
SET enable_partition_pruning = on;  
```

2. Vérifier la requête (fonctions sur la colonne) :
```sql
-- ❌ Mauvais
WHERE EXTRACT(YEAR FROM date_vente) = 2024

-- ✅ Bon
WHERE date_vente >= '2024-01-01' AND date_vente < '2025-01-01'
```

3. Vérifier les statistiques :
```sql
ANALYZE ventes;  
ANALYZE ventes_2024;  
```

### Problème 2 : Partition-wise Join Ne S'Active Pas

**Symptômes :**
```sql
EXPLAIN montre un Hash Join global au lieu de joins par partition
```

**Solutions :**

1. Activer le paramètre :
```sql
SET enable_partitionwise_join = on;
```

2. Vérifier que les partitions correspondent :
```sql
-- Lister les partitions de chaque table
SELECT
    parent.relname as table_parent,
    child.relname as partition,
    pg_get_expr(child.relpartbound, child.oid) as partition_bounds
FROM pg_inherits  
JOIN pg_class parent ON pg_inherits.inhparent = parent.oid  
JOIN pg_class child ON pg_inherits.inhrelid = child.oid  
WHERE parent.relname IN ('commandes', 'paiements')  
ORDER BY parent.relname, partition_bounds;  
```

3. Vérifier la jointure sur colonne de partitionnement :
```sql
-- ✅ Bon
ON table_a.date_col = table_b.date_col

-- ❌ Mauvais (pas la colonne de partitionnement)
ON table_a.id = table_b.id
```

**Symptômes :**
```
Les requêtes sont lentes malgré le partitionnement
```

**Checklist :**


```sql
SHOW shared_buffers;  
SHOW work_mem;  
-- Augmenter si nécessaire
```

## Monitoring du Partition Pruning

### Requête pour Vérifier l'Utilisation des Partitions

```sql
-- Nombre de lignes par partition
SELECT
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) as taille,
    n_live_tup as lignes_actives,
    n_dead_tup as lignes_mortes,
    last_vacuum,
    last_autovacuum
FROM pg_stat_user_tables  
WHERE tablename LIKE 'ventes_%'  
ORDER BY tablename;  
```

### Statistiques de Scan

```sql
-- Identifier les partitions les plus scannées
SELECT
    schemaname,
    tablename,
    seq_scan,
    seq_tup_read,
    idx_scan,
    idx_tup_fetch
FROM pg_stat_user_tables  
WHERE tablename LIKE 'ventes_%'  
ORDER BY seq_scan DESC;  
```

**Interprétation :**
- Partitions anciennes peu scannées → Bon signe (données archivées)
- Partition récente très scannée → Normal (données actives)
- Toutes les partitions également scannées → Problème de pruning

## Bonnes Pratiques

### 1. Toujours Utiliser la Colonne de Partitionnement dans WHERE

```sql
-- ✅ Bon
SELECT * FROM ventes  
WHERE date_vente >= '2024-01-01'  
  AND client_id = 1001;

-- ⚠️ Moins bon (pruning impossible)
SELECT * FROM ventes WHERE client_id = 1001;
```

### 2. Activer les Optimisations Partition-wise

```sql
-- Mettre dans postgresql.conf
enable_partitionwise_join = on  
enable_partitionwise_aggregate = on  
```

### 3. Créer des Index sur Chaque Partition

```sql
-- Index sur table parent = index sur toutes les partitions
CREATE INDEX idx_ventes_client ON ventes(client_id);  
CREATE INDEX idx_ventes_montant ON ventes(montant);  
```

### 4. Maintenir les Statistiques

```sql
-- Après insertions massives
ANALYZE ventes;

-- Ou configurer autovacuum agressif
ALTER TABLE ventes SET (autovacuum_analyze_scale_factor = 0.05);
```

### 5. Documenter la Stratégie

```sql
COMMENT ON TABLE ventes IS
'Table partitionnée par RANGE sur date_vente.
Partitions mensuelles avec rétention de 2 ans.  
Partition pruning actif pour requêtes avec filtre sur date_vente.  
Partition-wise join disponible avec table paiements.';  
```

### 6. Surveiller les Plans d'Exécution

```sql
-- Créer une fonction de monitoring
CREATE OR REPLACE FUNCTION check_partition_usage(table_name TEXT)  
RETURNS TABLE(  
    partition_name TEXT,
    scan_count BIGINT,
    rows_scanned BIGINT
) AS $$
BEGIN
    RETURN QUERY
    SELECT
        tablename::TEXT,
        seq_scan + idx_scan as scan_count,
        seq_tup_read + idx_tup_fetch as rows_scanned
    FROM pg_stat_user_tables
    WHERE tablename LIKE table_name || '_%'
    ORDER BY scan_count DESC;
END;
$$ LANGUAGE plpgsql;

-- Utilisation
SELECT * FROM check_partition_usage('ventes');
```

## Cas d'Usage Avancés

### 1. Partition Pruning avec Sous-requêtes

```sql
-- Pruning fonctionne aussi dans les sous-requêtes
SELECT client_id, SUM(montant) as total  
FROM ventes  
WHERE date_vente IN (  
    SELECT DISTINCT date_commande
    FROM commandes
    WHERE statut = 'validee'
    AND date_commande >= '2024-01-01'
)
GROUP BY client_id;
```

Le pruning s'applique à la fois sur `ventes` et `commandes`.

### 2. Partition-wise Join avec Agrégation

```sql
SET enable_partitionwise_join = on;  
SET enable_partitionwise_aggregate = on;  

EXPLAIN ANALYZE  
SELECT  
    c.client_id,
    DATE_TRUNC('month', c.date_commande) as mois,
    COUNT(DISTINCT c.commande_id) as nb_commandes,
    SUM(l.quantite) as quantite_totale
FROM commandes c  
JOIN lignes_commande l ON c.commande_id = l.commande_id  
    AND c.date_commande = l.date_commande
WHERE c.date_commande >= '2024-01-01'  
GROUP BY c.client_id, DATE_TRUNC('month', c.date_commande);  
```

**Optimisations combinées :**
1. Partition pruning sur la période  
2. Partition-wise join  
3. Partition-wise aggregate  
4. Possibilité de parallélisation

### 3. CTEs avec Partitionnement

```sql
WITH ventes_recentes AS (
    SELECT * FROM ventes
    WHERE date_vente >= CURRENT_DATE - INTERVAL '30 days'
)
SELECT
    DATE_TRUNC('day', date_vente) as jour,
    COUNT(*) as nb_ventes,
    SUM(montant) as total
FROM ventes_recentes  
GROUP BY DATE_TRUNC('day', date_vente)  
ORDER BY jour;  
```

Le pruning fonctionne dans la CTE.

## Conclusion

### Points Clés à Retenir

1. **Partition Pruning** : PostgreSQL élimine automatiquement les partitions inutiles  
   - ✅ Activé par défaut depuis PostgreSQL 11  
   - ✅ Fonctionne avec égalité, plages, IN  
   - ⚠️ Éviter les fonctions sur la colonne de partitionnement

2. **Partition-wise Join** : Jointures partition par partition  
   - ⚠️ Désactivé par défaut → Activer avec `enable_partitionwise_join = on`  
   - ✅ Requiert des partitions compatibles  
   - ✅ Gains majeurs sur grandes tables

3. **Partition-wise Aggregate** : Agrégations par partition  
   - ⚠️ Désactivé par défaut → Activer avec `enable_partitionwise_aggregate = on`  
   - ✅ Permet la parallélisation  
   - ✅ Réduit l'empreinte mémoire

4. **Monitoring** : Utilisez `EXPLAIN ANALYZE` pour vérifier
   - Nombre de partitions scannées
   - Type de join utilisé
   - Temps d'exécution

### Configuration Recommandée

```sql
-- postgresql.conf pour production avec partitionnement
enable_partition_pruning = on            # ON par défaut  
enable_partitionwise_join = on           # OFF par défaut → ACTIVER  
enable_partitionwise_aggregate = on      # OFF par défaut → ACTIVER  
constraint_exclusion = partition         # Compatibilité  

# Parallélisme (ajuster selon CPU)
max_parallel_workers_per_gather = 4  
max_parallel_workers = 8  
```
