# 11.4.3. Détachement et attachement de partitions

## Introduction

L'une des fonctionnalités les plus puissantes du partitionnement dans PostgreSQL est la capacité de **détacher** (DETACH) et **attacher** (ATTACH) des partitions dynamiquement, sans interrompre le service. Cette flexibilité permet de gérer efficacement le cycle de vie des données : archivage, purge, migration, maintenance, et bien plus encore.

### Analogie Simple

Imaginez une bibliothèque avec des étagères mobiles :

```
Bibliothèque principale (table partitionnée)
├── Étagère 2023 (partition)
├── Étagère 2024 (partition)
└── Étagère 2025 (partition)

Détacher une étagère :
- Retirer physiquement l'étagère 2023 de la bibliothèque
- Elle existe toujours comme table indépendante
- Les livres sont intacts, juste "déconnectés"

Attacher une étagère :
- Ajouter une nouvelle étagère (ou remettre l'ancienne)
- Elle devient instantanément partie de la bibliothèque
- Accès immédiat aux livres qu'elle contient
```

### Pourquoi Détacher/Attacher ?

**Cas d'usage typiques :**

1. **Archivage** : Détacher les anciennes partitions pour les archiver  
2. **Purge rapide** : Supprimer des données anciennes instantanément  
3. **Maintenance** : Opérations lourdes sur une partition isolée  
4. **Migration** : Déplacer des données vers un autre tablespace  
5. **Réorganisation** : Changer la structure d'une partition  
6. **Import/Export** : Intégrer des données externes comme partition

---

## Partie 1 : Détacher une Partition (DETACH)

### Concept

**Détacher** une partition signifie la **déconnecter** de la table partitionnée parente tout en **conservant** la table et ses données. La partition devient une table ordinaire indépendante.

### Syntaxe de Base

```sql
ALTER TABLE table_parent DETACH PARTITION partition_name;
```

### Exemple Simple

```sql
-- Table partitionnée
CREATE TABLE ventes (
    vente_id BIGSERIAL,
    date_vente DATE NOT NULL,
    montant NUMERIC(10,2)
) PARTITION BY RANGE (date_vente);

-- Créer des partitions
CREATE TABLE ventes_2023 PARTITION OF ventes
    FOR VALUES FROM ('2023-01-01') TO ('2024-01-01');

CREATE TABLE ventes_2024 PARTITION OF ventes
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

CREATE TABLE ventes_2025 PARTITION OF ventes
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');

-- Insérer des données
INSERT INTO ventes (date_vente, montant)  
SELECT  
    '2023-01-01'::date + (random() * 365)::integer,
    (random() * 1000)::numeric(10,2)
FROM generate_series(1, 100000);  -- 100k lignes en 2023

-- Vérifier les partitions
SELECT
    parent.relname AS table_parent,
    child.relname AS partition
FROM pg_inherits  
JOIN pg_class parent ON pg_inherits.inhparent = parent.oid  
JOIN pg_class child ON pg_inherits.inhrelid = child.oid  
WHERE parent.relname = 'ventes';  
```

**Résultat :**
```
table_parent | partition
-------------+--------------
ventes       | ventes_2023  
ventes       | ventes_2024  
ventes       | ventes_2025  
```

### Détacher la Partition 2023

```sql
-- Détacher ventes_2023
ALTER TABLE ventes DETACH PARTITION ventes_2023;
```

**Ce qui se passe :**
1. `ventes_2023` n'est plus une partition de `ventes`  
2. `ventes_2023` devient une table ordinaire indépendante  
3. Les données restent intactes dans `ventes_2023`  
4. Les requêtes sur `ventes` n'incluent plus `ventes_2023`

### Vérification Après Détachement

```sql
-- Vérifier les partitions restantes
SELECT
    parent.relname AS table_parent,
    child.relname AS partition
FROM pg_inherits  
JOIN pg_class parent ON pg_inherits.inhparent = parent.oid  
JOIN pg_class child ON pg_inherits.inhrelid = child.oid  
WHERE parent.relname = 'ventes';  
```

**Résultat :**
```
table_parent | partition
-------------+--------------
ventes       | ventes_2024  
ventes       | ventes_2025  
```

`ventes_2023` n'apparaît plus !

```sql
-- Mais la table existe toujours
SELECT COUNT(*) FROM ventes_2023;
-- Résultat: 100000 (les données sont là)

-- Requête sur ventes n'inclut plus 2023
SELECT COUNT(*) FROM ventes;
-- Résultat: 0 (seules 2024 et 2025, qui sont vides pour l'instant)
```

## Détachement CONCURRENT (PostgreSQL 14+)

### Problème du Détachement Standard

Le détachement standard (`DETACH PARTITION`) prend un **verrou exclusif** (ACCESS EXCLUSIVE) sur la table parent, ce qui **bloque** toutes les opérations (lecture et écriture) pendant l'opération.

**Impact :**
- Sur une grosse table : plusieurs secondes de blocage
- Inacceptable en production avec fort trafic

### Solution : DETACH PARTITION CONCURRENTLY

Depuis PostgreSQL 14, vous pouvez détacher une partition **sans bloquer** les opérations.

```sql
-- Détachement concurrent (non-bloquant)
ALTER TABLE ventes DETACH PARTITION ventes_2023 CONCURRENTLY;
```

**Avantages :**
- ✅ Pas de blocage des lectures  
- ✅ Pas de blocage des écritures  
- ✅ Opération transparente pour l'application

**Processus en deux phases :**

1. **Phase 1** : Marquer la partition comme "en cours de détachement"
   - Verrou léger
   - Les requêtes continuent de fonctionner

2. **Phase 2** : Finaliser le détachement après validation
   - Attendre que toutes les transactions en cours se terminent
   - Détacher définitivement

### Exemple Complet

```sql
-- Démarrer le détachement concurrent
ALTER TABLE ventes DETACH PARTITION ventes_2024 CONCURRENTLY;

-- L'opération peut prendre du temps si des transactions sont en cours
-- Pendant ce temps, la table ventes reste accessible

-- Vérifier l'état
SELECT
    parent.relname AS table_parent,
    child.relname AS partition,
    child.relispartition AS est_partition
FROM pg_inherits  
JOIN pg_class parent ON pg_inherits.inhparent = parent.oid  
JOIN pg_class child ON pg_inherits.inhrelid = child.oid  
WHERE parent.relname = 'ventes';  
```

**Note :** Si l'opération est interrompue (crash, annulation), la partition reste dans un état "partiellement détachée". Vous devrez finaliser l'opération :

```sql
-- Finaliser un détachement interrompu
ALTER TABLE ventes DETACH PARTITION ventes_2024 FINALIZE;
```

## Partie 2 : Attacher une Partition (ATTACH)

### Concept

**Attacher** une partition signifie transformer une table ordinaire en partition d'une table partitionnée. Les données de la table deviennent immédiatement accessibles via la table parent.

### Syntaxe de Base

```sql
-- Pour RANGE
ALTER TABLE table_parent ATTACH PARTITION table_name
    FOR VALUES FROM (min_value) TO (max_value);

-- Pour LIST
ALTER TABLE table_parent ATTACH PARTITION table_name
    FOR VALUES IN (value1, value2, ...);

-- Pour HASH
ALTER TABLE table_parent ATTACH PARTITION table_name
    FOR VALUES WITH (MODULUS x, REMAINDER y);
```

### Exemple : Attacher une Table Existante

```sql
-- Créer une table ordinaire avec des données
CREATE TABLE nouvelles_ventes_2026 (
    vente_id BIGSERIAL,
    date_vente DATE NOT NULL,
    montant NUMERIC(10,2)
);

-- Insérer des données
INSERT INTO nouvelles_ventes_2026 (date_vente, montant)  
SELECT  
    '2026-01-01'::date + (random() * 365)::integer,
    (random() * 1000)::numeric(10,2)
FROM generate_series(1, 50000);

-- Attacher comme partition
ALTER TABLE ventes ATTACH PARTITION nouvelles_ventes_2026
    FOR VALUES FROM ('2026-01-01') TO ('2027-01-01');
```

**Ce qui se passe :**
1. `nouvelles_ventes_2026` devient une partition de `ventes`  
2. Les 50 000 lignes sont immédiatement accessibles via `SELECT * FROM ventes`  
3. La structure doit être compatible (colonnes, types)

### Vérification

```sql
-- Vérifier les partitions
SELECT
    parent.relname AS table_parent,
    child.relname AS partition
FROM pg_inherits  
JOIN pg_class parent ON pg_inherits.inhparent = parent.oid  
JOIN pg_class child ON pg_inherits.inhrelid = child.oid  
WHERE parent.relname = 'ventes'  
ORDER BY child.relname;  

-- Compter les lignes totales
SELECT COUNT(*) FROM ventes;
-- Inclut maintenant les lignes de nouvelles_ventes_2026
```

## Validation des Données lors de l'Attachement

### Le Problème

Lorsque vous attachez une table existante, PostgreSQL doit **vérifier** que toutes les données respectent les contraintes de la partition (plage de valeurs).

**Par défaut :**
- PostgreSQL scanne **toutes les lignes** de la table
- Vérifie que chaque ligne respecte les bornes de partition
- Opération **bloquante** et potentiellement longue

### Exemple de Validation

```sql
-- Table avec 10 millions de lignes
CREATE TABLE archive_ventes_2022 (
    vente_id BIGSERIAL,
    date_vente DATE NOT NULL,
    montant NUMERIC(10,2)
);

-- Insérer 10M de lignes (simulation)
-- INSERT INTO archive_ventes_2022 ...

-- Attacher : PostgreSQL va scanner les 10M de lignes
ALTER TABLE ventes ATTACH PARTITION archive_ventes_2022
    FOR VALUES FROM ('2022-01-01') TO ('2023-01-01');
-- Cette opération peut prendre plusieurs minutes et BLOQUE la table
```

### Solution : Contrainte CHECK Pré-Validée

Pour **éviter le scan complet**, ajoutez une contrainte CHECK **avant** l'attachement :

```sql
-- Étape 1 : Ajouter une contrainte CHECK sur la table à attacher
ALTER TABLE archive_ventes_2022  
ADD CONSTRAINT check_date_range  
CHECK (date_vente >= '2022-01-01' AND date_vente < '2023-01-01');  

-- PostgreSQL scanne la table UNE FOIS pour valider la contrainte
-- Cela peut se faire en période creuse

-- Étape 2 : Attacher avec la contrainte validée
ALTER TABLE ventes ATTACH PARTITION archive_ventes_2022
    FOR VALUES FROM ('2022-01-01') TO ('2023-01-01');
-- INSTANTANÉ : PostgreSQL voit que la contrainte existe déjà, pas besoin de re-scanner !
```

**Avantages :**
- ✅ Scan de validation fait en avance (pendant période creuse)  
- ✅ Attachement quasi-instantané  
- ✅ Pas de blocage prolongé

## Cas d'Usage Pratiques

### 1. Archivage de Données Anciennes

**Scénario :** Archiver les données de 2022 sur un stockage froid.

```sql
-- Étape 1 : Détacher la partition 2022
ALTER TABLE ventes DETACH PARTITION ventes_2022 CONCURRENTLY;

-- Étape 2 : Sauvegarder les données
-- pg_dump -t ventes_2022 > archive_ventes_2022.sql

-- Étape 3 : Déplacer vers un tablespace froid (optionnel)
ALTER TABLE ventes_2022 SET TABLESPACE tablespace_archive;

-- Étape 4 : Supprimer de la base principale (si vraiment nécessaire)
DROP TABLE ventes_2022;

-- Les requêtes sur ventes ne touchent plus les données 2022
-- Mais elles sont sauvegardées et peuvent être restaurées si besoin
```

### 2. Purge Rapide de Données

**Scénario :** Supprimer toutes les données de 2020 (millions de lignes).

**Méthode 1 : DELETE (LENT) :**
```sql
-- ❌ Très lent sur grosse table
DELETE FROM ventes WHERE date_vente >= '2020-01-01' AND date_vente < '2021-01-01';
-- Peut prendre des heures, génère beaucoup de WAL, bloque les autres opérations
```

**Méthode 2 : DETACH + DROP (RAPIDE) :**
```sql
-- ✅ Quasi-instantané
-- Étape 1 : Détacher
ALTER TABLE ventes DETACH PARTITION ventes_2020 CONCURRENTLY;

-- Étape 2 : Supprimer la table
DROP TABLE ventes_2020;
-- Instantané : pas de scan, pas de WAL massif
```

**Gain de temps :** De plusieurs heures à quelques secondes !

### 3. Maintenance Lourde sur une Partition

**Scénario :** Réorganiser une partition pour réduire le bloat.

```sql
-- Étape 1 : Détacher la partition
ALTER TABLE ventes DETACH PARTITION ventes_2023 CONCURRENTLY;

-- Étape 2 : Opérations de maintenance sans impacter ventes
VACUUM FULL ventes_2023;  -- Réorganisation complète  
REINDEX TABLE ventes_2023;  -- Reconstruction des index  
ANALYZE ventes_2023;  -- Mise à jour des statistiques  

-- Étape 3 : Réattacher
ALTER TABLE ventes ATTACH PARTITION ventes_2023
    FOR VALUES FROM ('2023-01-01') TO ('2024-01-01');

-- Avantage : Les requêtes sur ventes (2024, 2025) continuent normalement
```

### 4. Migration vers Nouveau Schéma

**Scénario :** Ajouter une colonne à une table partitionnée massivement.

```sql
-- Approche classique (très lent sur grosse table)
-- ALTER TABLE ventes ADD COLUMN region VARCHAR(50);

-- Approche par partition (plus rapide et progressif)

-- Étape 1 : Détacher chaque partition
ALTER TABLE ventes DETACH PARTITION ventes_2023 CONCURRENTLY;

-- Étape 2 : Ajouter la colonne à la partition détachée
ALTER TABLE ventes_2023 ADD COLUMN region VARCHAR(50) DEFAULT 'EU';

-- Étape 3 : Mettre à jour la valeur (sur partition isolée)
UPDATE ventes_2023 SET region = 'US' WHERE montant > 500;

-- Étape 4 : Modifier la table parent pour avoir la nouvelle colonne
ALTER TABLE ventes ADD COLUMN region VARCHAR(50);

-- Étape 5 : Réattacher
ALTER TABLE ventes ATTACH PARTITION ventes_2023
    FOR VALUES FROM ('2023-01-01') TO ('2024-01-01');

-- Répéter pour chaque partition
-- Permet de gérer la migration progressivement, partition par partition
```

### 5. Import de Données Externes

**Scénario :** Importer un fichier CSV de plusieurs Go comme nouvelle partition.

```sql
-- Étape 1 : Créer une table temporaire
CREATE TABLE import_ventes_janvier_2025 (
    vente_id BIGINT,
    date_vente DATE,
    montant NUMERIC(10,2)
);

-- Étape 2 : Importer les données (rapide car table non partitionnée)
COPY import_ventes_janvier_2025 FROM '/data/ventes_janvier_2025.csv' CSV HEADER;

-- Étape 3 : Ajouter la contrainte pour éviter le scan à l'attachement
ALTER TABLE import_ventes_janvier_2025  
ADD CONSTRAINT check_date_janvier_2025  
CHECK (date_vente >= '2025-01-01' AND date_vente < '2025-02-01');  

-- Étape 4 : Créer les index nécessaires
CREATE INDEX idx_import_ventes_date ON import_ventes_janvier_2025(date_vente);  
CREATE INDEX idx_import_ventes_montant ON import_ventes_janvier_2025(montant);  

-- Étape 5 : Attacher comme partition (instantané grâce à la contrainte)
ALTER TABLE ventes ATTACH PARTITION import_ventes_janvier_2025
    FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');

-- Les données sont immédiatement disponibles dans ventes
```

### 6. Échange de Partitions (Swap)

**Scénario :** Remplacer une partition par une nouvelle version.

```sql
-- Situation : ventes_2024 contient des données corrompues

-- Étape 1 : Créer une nouvelle partition avec données correctes
CREATE TABLE ventes_2024_nouveau (
    vente_id BIGSERIAL,
    date_vente DATE NOT NULL,
    montant NUMERIC(10,2)
);

-- Insérer les données corrigées
INSERT INTO ventes_2024_nouveau (date_vente, montant)  
SELECT date_vente, montant FROM source_donnees_correctes;  

-- Ajouter la contrainte
ALTER TABLE ventes_2024_nouveau  
ADD CONSTRAINT check_date_2024  
CHECK (date_vente >= '2024-01-01' AND date_vente < '2025-01-01');  

-- Étape 2 : Transaction atomique pour l'échange
BEGIN;
    -- Détacher l'ancienne
    ALTER TABLE ventes DETACH PARTITION ventes_2024;

    -- Attacher la nouvelle
    ALTER TABLE ventes ATTACH PARTITION ventes_2024_nouveau
        FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

    -- Renommer pour cohérence
    ALTER TABLE ventes_2024 RENAME TO ventes_2024_backup;
    ALTER TABLE ventes_2024_nouveau RENAME TO ventes_2024;
COMMIT;

-- Étape 3 : Vérifier puis supprimer le backup
-- DROP TABLE ventes_2024_backup;
```

## Précautions et Contraintes

### 1. Compatibilité de Structure

La table à attacher doit avoir **exactement la même structure** que la table parent.

```sql
-- ❌ Erreur : colonnes différentes
CREATE TABLE mauvaise_table (
    vente_id BIGINT,  -- Type différent (BIGINT vs BIGSERIAL)
    date_vente DATE,
    montant NUMERIC(10,2)
);

ALTER TABLE ventes ATTACH PARTITION mauvaise_table
    FOR VALUES FROM ('2027-01-01') TO ('2028-01-01');
-- ERREUR : table "mauvaise_table" has different column definitions

-- ✅ Solution : correspondance exacte
CREATE TABLE bonne_table (
    vente_id BIGSERIAL,  -- Même type que parent
    date_vente DATE NOT NULL,
    montant NUMERIC(10,2)
);
```

### 2. Données Hors Bornes

Toutes les données de la table à attacher doivent respecter les bornes de partition.

```sql
-- Table contenant des dates hors de la plage 2026
CREATE TABLE donnees_mixtes (
    vente_id BIGSERIAL,
    date_vente DATE NOT NULL,
    montant NUMERIC(10,2)
);

INSERT INTO donnees_mixtes (date_vente, montant) VALUES
    ('2026-05-15', 100.00),  -- OK
    ('2025-12-31', 50.00),   -- Hors plage !
    ('2026-11-20', 75.00);   -- OK

-- Tentative d'attachement
ALTER TABLE ventes ATTACH PARTITION donnees_mixtes
    FOR VALUES FROM ('2026-01-01') TO ('2027-01-01');
-- ERREUR : partition constraint is violated by some row
```

**Solution :** Nettoyer les données avant :
```sql
-- Supprimer les lignes hors plage
DELETE FROM donnees_mixtes  
WHERE date_vente < '2026-01-01' OR date_vente >= '2027-01-01';  

-- Maintenant l'attachement fonctionne
ALTER TABLE ventes ATTACH PARTITION donnees_mixtes
    FOR VALUES FROM ('2026-01-01') TO ('2027-01-01');
```

### 3. Contraintes et Index

**Contraintes :**
- Les contraintes NOT NULL, CHECK de la table parent sont héritées
- Les contraintes supplémentaires sur la partition sont permises

**Index :**
- Les index de la table parent ne sont **pas automatiquement créés** sur une table existante attachée
- Il faut créer les index manuellement **avant** l'attachement

```sql
-- Bonnes pratiques avant attachement
CREATE TABLE nouvelle_partition (
    vente_id BIGSERIAL,
    date_vente DATE NOT NULL,
    montant NUMERIC(10,2)
);

-- Créer les mêmes index que sur les autres partitions
CREATE INDEX idx_nouvelle_partition_date ON nouvelle_partition(date_vente);  
CREATE INDEX idx_nouvelle_partition_montant ON nouvelle_partition(montant);  

-- Puis attacher
ALTER TABLE ventes ATTACH PARTITION nouvelle_partition
    FOR VALUES FROM ('2028-01-01') TO ('2029-01-01');
```

### 4. Transactions et Verrous

**Détachement standard :**
- Prend un verrou ACCESS EXCLUSIVE (bloque tout)
- À éviter en production

**Détachement concurrent :**
- Verrous plus légers
- Peut prendre du temps si transactions longues en cours
- Recommandé en production

**Attachement :**
- Prend un verrou ACCESS EXCLUSIVE (bloque tout)
- Durée dépend de la validation des données
- Utilisez la contrainte CHECK pour accélérer

## Automatisation de la Gestion des Partitions

### Script de Création Automatique de Partitions Futures

```sql
-- Fonction pour créer les partitions des 12 prochains mois
CREATE OR REPLACE FUNCTION create_monthly_partitions(
    parent_table TEXT,
    start_date DATE,
    num_months INTEGER
)
RETURNS TEXT AS $$  
DECLARE  
    partition_name TEXT;
    start_range DATE;
    end_range DATE;
    result TEXT := '';
    i INTEGER;
BEGIN
    FOR i IN 0..(num_months-1) LOOP
        start_range := start_date + (i || ' months')::INTERVAL;
        end_range := start_range + '1 month'::INTERVAL;

        partition_name := parent_table || '_' || TO_CHAR(start_range, 'YYYY_MM');

        -- Vérifier si la partition existe déjà
        IF NOT EXISTS (
            SELECT 1 FROM pg_class WHERE relname = partition_name
        ) THEN
            EXECUTE format(
                'CREATE TABLE %I PARTITION OF %I FOR VALUES FROM (%L) TO (%L)',
                partition_name, parent_table, start_range, end_range
            );
            result := result || 'Créé: ' || partition_name || E'\n';
        ELSE
            result := result || 'Existe déjà: ' || partition_name || E'\n';
        END IF;
    END LOOP;

    RETURN result;
END;
$$ LANGUAGE plpgsql;

-- Utilisation
SELECT create_monthly_partitions('ventes', '2025-01-01', 12);
```

### Script d'Archivage Automatique

```sql
-- Fonction pour archiver les partitions anciennes
CREATE OR REPLACE FUNCTION archive_old_partitions(
    parent_table TEXT,
    retention_months INTEGER
)
RETURNS TEXT AS $$  
DECLARE  
    partition_record RECORD;
    cutoff_date DATE;
    result TEXT := '';
BEGIN
    cutoff_date := CURRENT_DATE - (retention_months || ' months')::INTERVAL;

    FOR partition_record IN
        SELECT
            child.relname AS partition_name,
            pg_get_expr(child.relpartbound, child.oid) AS bounds
        FROM pg_inherits
        JOIN pg_class parent ON pg_inherits.inhparent = parent.oid
        JOIN pg_class child ON pg_inherits.inhrelid = child.oid
        WHERE parent.relname = parent_table
    LOOP
        -- Logique simplifiée : détacher si le nom contient une année ancienne
        IF partition_record.partition_name ~ '\d{4}' THEN
            DECLARE
                partition_year INTEGER;
                cutoff_year INTEGER;
            BEGIN
                partition_year := (regexp_match(partition_record.partition_name, '(\d{4})'))[1]::INTEGER;
                cutoff_year := EXTRACT(YEAR FROM cutoff_date)::INTEGER;

                IF partition_year < cutoff_year THEN
                    -- Détacher
                    EXECUTE format(
                        'ALTER TABLE %I DETACH PARTITION %I CONCURRENTLY',
                        parent_table, partition_record.partition_name
                    );

                    result := result || 'Détaché: ' || partition_record.partition_name || E'\n';

                    -- Optionnel : Sauvegarder puis supprimer
                    -- EXECUTE format('DROP TABLE %I', partition_record.partition_name);
                END IF;
            END;
        END IF;
    END LOOP;

    RETURN result;
END;
$$ LANGUAGE plpgsql;

-- Utilisation : archiver les partitions > 24 mois
SELECT archive_old_partitions('ventes', 24);
```

### Extension pg_partman

Pour une gestion automatisée complète, utilisez l'extension **pg_partman**.

```sql
-- Installation
CREATE EXTENSION pg_partman;

-- Configuration pour création automatique
SELECT partman.create_parent(
    p_parent_table := 'public.ventes',
    p_control := 'date_vente',
    p_type := 'native',
    p_interval := 'monthly',
    p_premake := 3  -- Créer 3 mois à l'avance
);

-- Fonction de maintenance (à exécuter via cron)
SELECT partman.run_maintenance('public.ventes');

-- Configuration de la rétention (détacher automatiquement)
UPDATE partman.part_config  
SET retention = '12 months',  
    retention_keep_table = true  -- Garder les tables détachées
WHERE parent_table = 'public.ventes';
```

## Monitoring et Gestion

### Lister Toutes les Partitions

```sql
-- Vue complète des partitions
CREATE OR REPLACE VIEW v_partitions AS  
SELECT  
    nmsp_parent.nspname AS schema_parent,
    parent.relname AS table_parent,
    nmsp_child.nspname AS schema_partition,
    child.relname AS partition_name,
    pg_get_expr(child.relpartbound, child.oid) AS partition_bounds,
    pg_size_pretty(pg_total_relation_size(child.oid)) AS taille,
    pg_stat_get_live_tuples(child.oid) AS nb_lignes_estimees
FROM pg_inherits  
JOIN pg_class parent ON pg_inherits.inhparent = parent.oid  
JOIN pg_class child ON pg_inherits.inhrelid = child.oid  
JOIN pg_namespace nmsp_parent ON parent.relnamespace = nmsp_parent.oid  
JOIN pg_namespace nmsp_child ON child.relnamespace = nmsp_child.oid  
ORDER BY parent.relname, partition_bounds;  

-- Utilisation
SELECT * FROM v_partitions WHERE table_parent = 'ventes';
```

**Résultat exemple :**
```
schema_parent | table_parent | partition_name | partition_bounds                        | taille  | nb_lignes
--------------+--------------+----------------+-----------------------------------------+---------+-----------
public        | ventes       | ventes_2023    | FOR VALUES FROM ('2023-01-01') TO ...   | 125 MB  | 1000000  
public        | ventes       | ventes_2024    | FOR VALUES FROM ('2024-01-01') TO ...   | 130 MB  | 1050000  
public        | ventes       | ventes_2025    | FOR VALUES FROM ('2025-01-01') TO ...   | 15 MB   | 120000  
```

### Identifier les Partitions à Archiver

```sql
-- Partitions anciennes candidates à l'archivage
SELECT
    partition_name,
    partition_bounds,
    taille,
    nb_lignes_estimees,
    AGE(NOW(), (regexp_match(partition_name, '(\d{4})'))[1]::INTEGER || '-01-01') AS age
FROM v_partitions  
WHERE table_parent = 'ventes'  
  AND partition_name ~ '\d{4}'
  AND EXTRACT(YEAR FROM NOW())::INTEGER - (regexp_match(partition_name, '(\d{4})'))[1]::INTEGER > 2
ORDER BY partition_name;
```

### Statistiques d'Utilisation

```sql
-- Activité par partition
SELECT
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS taille,
    n_tup_ins AS insertions,
    n_tup_upd AS mises_a_jour,
    n_tup_del AS suppressions,
    seq_scan AS scans_sequentiels,
    idx_scan AS scans_index,
    last_vacuum,
    last_autovacuum
FROM pg_stat_user_tables  
WHERE tablename LIKE 'ventes_%'  
ORDER BY tablename;  
```

## Stratégies de Maintenance par Cycle de Vie

### Cycle de Vie Typique d'une Partition

```
1. Création (T-1 mois)
   ↓
2. Données actives (mois courant)
   ↓
3. Données récentes (1-6 mois)
   ↓
4. Données historiques (6-24 mois)
   ↓
5. Archivage (détachement)
   ↓
6. Suppression ou stockage froid
```

### Implémentation

```sql
-- 1. CRÉATION : Automatique via cron
-- Script cron quotidien :
-- 0 2 * * * psql -U user -d db -c "SELECT create_monthly_partitions('ventes', CURRENT_DATE, 3);"

-- 2. DONNÉES ACTIVES : Aucune action nécessaire

-- 3. DONNÉES RÉCENTES : Maintenance légère
-- Vacuum automatique suffit généralement

-- 4. DONNÉES HISTORIQUES : Optimisation
-- Mensuel : ANALYZE sur partitions anciennes
DO $$  
DECLARE  
    partition TEXT;
BEGIN
    FOR partition IN
        SELECT relname FROM pg_class
        WHERE relname LIKE 'ventes_202%'
        AND relname < 'ventes_' || TO_CHAR(NOW() - INTERVAL '6 months', 'YYYY_MM')
    LOOP
        EXECUTE 'VACUUM ANALYZE ' || partition;
    END LOOP;
END $$;

-- 5. ARCHIVAGE : Script mensuel
-- 0 3 1 * * psql -U user -d db -c "SELECT archive_old_partitions('ventes', 24);"

-- 6. SUPPRESSION : Manuel ou automatique après sauvegarde
-- Script qui vérifie que la sauvegarde existe avant de supprimer
```

## Checklist de Production

### Avant de Détacher une Partition

- [ ] Vérifier qu'aucune transaction longue n'est en cours  
- [ ] Utiliser DETACH ... CONCURRENTLY pour éviter les blocages  
- [ ] Sauvegarder les données si nécessaire (pg_dump)  
- [ ] Documenter l'opération (raison, date, responsable)  
- [ ] Prévoir une procédure de rollback

### Avant d'Attacher une Partition

- [ ] Vérifier la compatibilité de structure  
- [ ] Ajouter une contrainte CHECK pour éviter le scan complet  
- [ ] Créer les index nécessaires  
- [ ] Valider que toutes les données respectent les bornes  
- [ ] Estimer le temps d'attachement (surtout si pas de contrainte)  
- [ ] Planifier pendant une période creuse si attachement long

### Après Détachement/Attachement

- [ ] Vérifier que l'opération a réussi  
- [ ] Exécuter ANALYZE sur la table parent  
- [ ] Tester une requête pour vérifier l'accès aux données  
- [ ] Mettre à jour la documentation  
- [ ] Surveiller les performances

## Dépannage et Problèmes Courants

### Problème 1 : DETACH CONCURRENTLY Bloqué

**Symptôme :**
```sql
ALTER TABLE ventes DETACH PARTITION ventes_2023 CONCURRENTLY;
-- L'opération ne se termine pas
```

**Cause :** Des transactions longues maintiennent des verrous.

**Solution :**
```sql
-- Identifier les transactions bloquantes
SELECT
    pid,
    usename,
    application_name,
    state,
    age(clock_timestamp(), query_start) AS age,
    query
FROM pg_stat_activity  
WHERE state != 'idle'  
  AND pid != pg_backend_pid()
ORDER BY age DESC;

-- Si nécessaire, terminer les transactions problématiques
SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE pid = <problem_pid>;

-- Ou attendre qu'elles se terminent naturellement
```

### Problème 2 : Erreur lors de l'Attachement

**Erreur :**
```
ERROR: partition constraint is violated by some row
```

**Solution :**
```sql
-- Identifier les lignes problématiques
SELECT * FROM table_a_attacher  
WHERE NOT (date_vente >= '2026-01-01' AND date_vente < '2027-01-01');  

-- Corriger ou supprimer ces lignes
DELETE FROM table_a_attacher  
WHERE NOT (date_vente >= '2026-01-01' AND date_vente < '2027-01-01');  

-- Réessayer l'attachement
ALTER TABLE ventes ATTACH PARTITION table_a_attacher
    FOR VALUES FROM ('2026-01-01') TO ('2027-01-01');
```

### Problème 3 : Partition Partiellement Détachée

**Symptôme :** Après un crash pendant DETACH CONCURRENTLY.

**Solution :**
```sql
-- Finaliser le détachement
ALTER TABLE ventes DETACH PARTITION ventes_2023 FINALIZE;

-- Si erreur, forcer en mode normal (bloquant)
ALTER TABLE ventes DETACH PARTITION ventes_2023;
```

### Problème 4 : Index Manquants Après Attachement

**Symptôme :** Requêtes lentes sur la nouvelle partition.

**Solution :**
```sql
-- Vérifier les index sur les autres partitions
SELECT
    schemaname,
    tablename,
    indexname,
    indexdef
FROM pg_indexes  
WHERE tablename = 'ventes_2024';  

-- Créer les mêmes index sur la nouvelle partition
-- (en dehors des heures de pointe)
CREATE INDEX idx_ventes_2026_date ON ventes_2026(date_vente);  
CREATE INDEX idx_ventes_2026_montant ON ventes_2026(montant);  
```

## Bonnes Pratiques

### 1. Planification

- Créer les partitions **à l'avance** (3-6 mois)
- Définir une politique de rétention claire
- Automatiser la création/archivage
- Documenter la stratégie

### 2. Exécution

- Utiliser DETACH ... CONCURRENTLY en production
- Ajouter les contraintes CHECK avant attachement
- Effectuer les opérations pendant les périodes creuses
- Tester sur un environnement de staging

### 3. Monitoring

- Surveiller la taille des partitions
- Alerter si une partition devient trop grosse
- Monitorer l'utilisation des partitions anciennes
- Vérifier régulièrement les partitions orphelines

### 4. Sauvegarde

- Sauvegarder avant détachement
- Vérifier la sauvegarde avant suppression
- Conserver les archives selon la politique de l'entreprise
- Documenter l'emplacement des archives

## Conclusion

### Points Clés à Retenir

1. **Détachement** : Déconnecter une partition sans perdre les données
   - DETACH PARTITION standard : bloquant
   - DETACH PARTITION CONCURRENTLY : non-bloquant (PG 14+)

2. **Attachement** : Transformer une table en partition
   - Nécessite compatibilité de structure
   - Ajoutez une contrainte CHECK pour éviter le scan
   - Créez les index avant attachement

3. **Cas d'usage** :
   - Archivage et purge rapide
   - Maintenance sans impact
   - Import de données massives
   - Migration progressive

4. **Automatisation** : Indispensable en production
   - Scripts de création anticipée
   - Scripts d'archivage automatique
   - Extension pg_partman pour gestion complète

### Configuration Recommandée

```sql
-- Dans votre application ou scripts de maintenance

-- 1. Création automatique des partitions futures

-- 2. Archivage automatique
-- Cron mensuel : 0 3 1 * *

-- 3. Maintenance
-- Cron hebdomadaire : 0 4 * * 0
VACUUM ANALYZE ventes;

-- 4. Monitoring
-- Vérification quotidienne
SELECT * FROM v_partitions WHERE table_parent = 'ventes';
```
