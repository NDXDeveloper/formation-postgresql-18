# 17.5.1. Promotion Manuelle (pg_ctl promote)

## Introduction

Dans une architecture de haute disponibilité PostgreSQL, vous disposez généralement d'un serveur **Primary** (principal) qui accepte les lectures et écritures, et d'un ou plusieurs serveurs **Standby** (secondaires) qui répliquent les données du Primary en temps réel.

La **promotion** est l'opération qui consiste à transformer un serveur Standby en serveur Primary. Cette opération est nécessaire lorsque :

- Le serveur Primary tombe en panne (failover)
- Vous devez effectuer une maintenance planifiée sur le Primary
- Vous souhaitez basculer vers un autre datacenter
- Vous effectuez un test de reprise après sinistre (disaster recovery)

La promotion manuelle avec `pg_ctl promote` est la méthode la plus simple et la plus directe pour effectuer cette opération.

---

## Concept de Base : Primary et Standby

### Le Serveur Primary (Principal)

Le serveur Primary est le serveur PostgreSQL qui :
- Accepte les connexions en **lecture ET écriture**
- Génère les fichiers WAL (Write-Ahead Logs) contenant toutes les modifications
- Est la source de vérité de vos données

### Le Serveur Standby (Secondaire)

Le serveur Standby est un serveur PostgreSQL qui :
- Est en mode **réplication** (lecture seule par défaut)
- Reçoit continuellement les fichiers WAL du Primary
- Rejoue ces WAL pour maintenir une copie à jour des données
- Peut accepter des requêtes en lecture (Hot Standby)

### La Promotion : Transformation Standby → Primary

Lorsque vous **promouvez** un Standby :
1. Le Standby arrête de recevoir les WAL du Primary  
2. Il termine de rejouer tous les WAL qu'il a en attente  
3. Il passe en mode lecture-écriture  
4. Il devient le nouveau Primary et commence à générer ses propres WAL

---

## La Commande `pg_ctl promote`

### Syntaxe de Base

```bash
pg_ctl promote -D /chemin/vers/data/directory
```

### Paramètres Principaux

| Paramètre | Description | Exemple |
|-----------|-------------|---------|
| `-D` ou `--pgdata` | Répertoire de données du Standby | `-D /var/lib/postgresql/data` |
| `-t` ou `--timeout` | Temps d'attente maximal (secondes) | `-t 60` |
| `-w` | Attendre que la promotion soit terminée | `-w` |
| `-W` | Ne pas attendre (mode asynchrone) | `-W` |

### Exemple Complet

```bash
# Promotion avec attente de confirmation
pg_ctl promote -D /var/lib/postgresql/14/main -w -t 60

# Message de sortie attendu :
# waiting for server to promote.... done
# server promoted
```

## Prérequis Avant la Promotion

### 1. Vérifier l'État du Standby

Avant de promouvoir, assurez-vous que le Standby est :

**a) Bien en mode réplication**

Connectez-vous au Standby avec `psql` et exécutez :

```sql
SELECT pg_is_in_recovery();
```

- **Résultat attendu** : `t` (true) → Le serveur est en mode recovery/standby
- Si `f` (false) → Le serveur est déjà un Primary !

**b) À jour avec le Primary**

Vérifiez le retard de réplication :

```sql
SELECT
    pg_last_wal_receive_lsn() AS derniere_reception,
    pg_last_wal_replay_lsn() AS derniere_application,
    pg_last_wal_receive_lsn() = pg_last_wal_replay_lsn() AS est_a_jour;
```

- `est_a_jour = true` → Tous les WAL reçus ont été appliqués
- Un petit retard est normal, mais un retard important peut indiquer un problème

### 2. S'Assurer que le Primary est Vraiment Hors Service

⚠️ **ATTENTION** : Ne jamais promouvoir un Standby si le Primary est encore actif !

**Conséquence** : Vous auriez alors **deux Primary** (split-brain), ce qui mènerait à une **corruption des données**.

**Vérifications à effectuer** :
- Le Primary ne répond plus aux connexions réseau
- Le Primary est physiquement arrêté ou isolé du réseau
- Les applications ne peuvent plus se connecter au Primary

### 3. Prévenir les Applications

Avant la promotion :
- Informer les utilisateurs d'une interruption de service imminente
- Mettre les applications en mode maintenance si possible
- Noter les applications qui devront reconfigurer leur connexion après promotion

## Procédure Complète de Promotion Manuelle

### Étape 1 : Vérifications Préliminaires

```bash
# 1. Connectez-vous au serveur Standby (SSH)
ssh postgres@serveur-standby

# 2. Devenez l'utilisateur postgres
sudo su - postgres

# 3. Vérifiez que le serveur est bien en mode standby
psql -c "SELECT pg_is_in_recovery();"
# Résultat attendu : t
```

### Étape 2 : Vérifier le Retard de Réplication

```bash
psql -c "  
SELECT  
    CASE
        WHEN pg_last_wal_receive_lsn() = pg_last_wal_replay_lsn()
        THEN 'Standby à jour'
        ELSE 'Standby en retard'
    END AS statut,
    pg_wal_lsn_diff(pg_last_wal_receive_lsn(), pg_last_wal_replay_lsn()) AS retard_bytes;
"
```

### Étape 3 : Lancer la Promotion

```bash
# Promotion avec attente de confirmation (recommandé)
pg_ctl promote -D /var/lib/postgresql/14/main -w

# OU avec timeout explicite
pg_ctl promote -D /var/lib/postgresql/14/main -w -t 60
```

**Que se passe-t-il pendant la promotion ?**

1. PostgreSQL termine de rejouer tous les WAL en attente  
2. PostgreSQL écrit un fichier spécial signalant la fin du mode recovery  
3. PostgreSQL démarre en mode lecture-écriture  
4. PostgreSQL est maintenant un Primary capable d'accepter des écritures

### Étape 4 : Vérifier la Promotion

```bash
# Vérifier que le serveur n'est plus en recovery
psql -c "SELECT pg_is_in_recovery();"
# Résultat attendu : f (false)

# Vérifier que le serveur accepte les écritures
psql -c "CREATE TABLE test_promotion (id serial primary key, donnee text);"  
psql -c "INSERT INTO test_promotion (donnee) VALUES ('Test réussi');"  
psql -c "SELECT * FROM test_promotion;"  
psql -c "DROP TABLE test_promotion;"  
```

### Étape 5 : Reconfigurer les Applications

Une fois la promotion réussie :
- Mettez à jour les chaînes de connexion des applications pour pointer vers le nouveau Primary
- Si vous utilisez un système de basculement (HAProxy, PgBouncer), mettez à jour la configuration
- Testez la connectivité depuis les applications

## Détails Techniques de la Promotion

### Fichiers Impliqués

Pendant la promotion, PostgreSQL manipule plusieurs fichiers :

**1. Le fichier `standby.signal`**
- **Présent sur un Standby** : Indique que le serveur est en mode réplication  
- **Supprimé lors de la promotion** : PostgreSQL le supprime automatiquement
- Emplacement : `$PGDATA/standby.signal`

**2. Le fichier `recovery.conf` (PostgreSQL < 12)**
- Ancienne méthode de configuration de la réplication
- Renommé en `recovery.done` après promotion

**3. Le fichier `postgresql.auto.conf`**
- Peut contenir des paramètres de réplication
- Non modifié automatiquement lors de la promotion

### Que se Passe-t-il en Arrière-Plan ?

```
1. pg_ctl envoie un signal SIGUSR1 au processus postmaster
2. Le postmaster détecte la demande de promotion
3. Le Startup Process termine de rejouer les WAL
4. PostgreSQL supprime standby.signal
5. PostgreSQL démarre les processus en mode Primary :
   - WAL Writer (écriture des WAL)
   - Background Writer
   - Autovacuum
6. Le serveur accepte maintenant les connexions en écriture
```

### Temps de Promotion

Le temps nécessaire pour une promotion dépend de :

| Facteur | Impact |
|---------|--------|
| **Retard de réplication** | Plus il y a de WAL à rejouer, plus c'est long |
| **Taille des WAL** | Des transactions volumineuses prennent du temps |
| **Performance disque** | Les I/O influencent la vitesse de replay |
| **Configuration** | `recovery_target_action` peut imposer des pauses |

**Durée typique** : De quelques secondes à quelques minutes

## Scénarios d'Usage

### Scénario 1 : Panne du Primary (Failover Non Planifié)

**Contexte** : Le serveur Primary est tombé en panne (hardware, réseau, corruption)

**Procédure** :
1. Confirmer que le Primary est inaccessible  
2. Vérifier l'état du Standby (retard de réplication)  
3. Promouvoir le Standby avec `pg_ctl promote`  
4. Reconfigurer les applications  
5. Enquêter sur la cause de la panne du Primary

**Considération** : Le Primary doit être complètement isolé pour éviter un split-brain

---

### Scénario 2 : Maintenance Planifiée (Switchover)

**Contexte** : Vous devez effectuer une maintenance sur le Primary (mise à jour OS, migration)

**Procédure** :
1. Annoncer une fenêtre de maintenance  
2. Passer les applications en lecture seule (optionnel)  
3. Attendre que le Standby soit complètement à jour  
4. Arrêter proprement le Primary : `pg_ctl stop -m fast`  
5. Promouvoir le Standby : `pg_ctl promote`  
6. Basculer les applications vers le nouveau Primary  
7. (Optionnel) Transformer l'ancien Primary en nouveau Standby

**Avantage** : Contrôle total et minimisation de la perte de données

---

### Scénario 3 : Test de Disaster Recovery

**Contexte** : Vous testez votre plan de reprise après sinistre

**Procédure** :
1. Isoler un Standby de test du Primary (réseau séparé)  
2. Promouvoir ce Standby de test  
3. Vérifier l'intégrité des données  
4. Tester la connectivité applicative  
5. Détruire le Standby de test après validation

**Important** : Ne jamais promouvoir un Standby de production pendant un test !

## Gestion des Erreurs et Troubleshooting

### Erreur 1 : "server is not in standby mode"

**Message complet** :
```
pg_ctl: cannot promote server; server is not in standby mode
```

**Cause** : Le serveur est déjà un Primary

**Solution** :
```bash
# Vérifier le mode du serveur
psql -c "SELECT pg_is_in_recovery();"
# Si false (f), c'est déjà un Primary
```

---

### Erreur 2 : "promotion timed out"

**Message** :
```
pg_ctl: promotion timed out
```

**Cause** : La promotion prend plus de temps que le timeout configuré

**Solution** :
```bash
# Augmenter le timeout
pg_ctl promote -D /var/lib/postgresql/14/main -w -t 300

# Vérifier les logs pour comprendre le délai
tail -f /var/log/postgresql/postgresql-14-main.log
```

**Raisons possibles** :
- Beaucoup de WAL à rejouer
- Disque lent ou saturé
- Corruption de WAL (rare)

---

### Erreur 3 : "could not connect to server"

**Message** :
```
pg_ctl: could not connect to server
```

**Cause** : PostgreSQL n'est pas démarré ou ne répond pas

**Solution** :
```bash
# Vérifier si PostgreSQL est en cours d'exécution
pg_ctl status -D /var/lib/postgresql/14/main

# Si arrêté, le démarrer d'abord
pg_ctl start -D /var/lib/postgresql/14/main
```

---

### Erreur 4 : Split-Brain (Deux Primary actifs)

**Symptôme** : Après promotion, l'ancien Primary redémarre et accepte aussi des écritures

**Conséquence** : **CORRUPTION DE DONNÉES GARANTIE**

**Prévention** :
- Utiliser des mécanismes de fencing (STONITH, IPMI, watchdogs)
- Arrêter physiquement le Primary avant promotion
- Utiliser Patroni ou Repmgr pour gérer automatiquement le fencing

**Détection** :
```bash
# Sur chaque serveur, vérifier le mode
psql -c "SELECT pg_is_in_recovery();"
# Si les deux renvoient 'f' (false), vous avez un split-brain !
```

**Résolution** :
1. Identifier le serveur avec les données les plus récentes  
2. Arrêter immédiatement l'autre serveur  
3. Reconstruire complètement le Standby depuis le Primary valide

## Alternatives et Outils Complémentaires

### 1. Utilisation de `pg_ctl` vs Signal Direct

**Méthode pg_ctl** (recommandée) :
```bash
pg_ctl promote -D /var/lib/postgresql/14/main
```

**Méthode par signal** (niveau expert) :
```bash
# Trouver le PID du postmaster
PID=$(head -1 /var/lib/postgresql/14/main/postmaster.pid)

# Envoyer le signal SIGUSR1
kill -SIGUSR1 $PID
```

Les deux méthodes ont le même effet, mais `pg_ctl promote` est plus sûr et plus clair.

---

### 2. Fichier `promote` (Méthode Trigger)

**Alternative historique** : Créer un fichier `promote` dans `$PGDATA`

```bash
# Méthode par fichier trigger
touch /var/lib/postgresql/14/main/promote

# PostgreSQL détectera ce fichier au prochain cycle et se promouvra
```

**Note** : Cette méthode est moins utilisée depuis l'existence de `pg_ctl promote`

---

### 3. Fonction SQL `pg_promote()` (PostgreSQL 12+)

**Nouveauté PostgreSQL 12** : Vous pouvez promouvoir depuis une connexion SQL

```sql
-- Depuis une session psql connectée au Standby
SELECT pg_promote();
```

**Avantage** : Permet d'automatiser la promotion via des scripts SQL

**Cas d'usage** : Intégration dans des outils de monitoring ou d'orchestration

## Considérations de Haute Disponibilité

### 1. Cascade de Standby

Si vous aviez plusieurs Standby en cascade :

```
Primary → Standby A → Standby B
```

Après promotion de Standby A :
- Standby B doit être reconfiguré pour se connecter au nouveau Primary (ancien Standby A)
- Sinon, Standby B restera en état "déconnecté"

**Reconfiguration de Standby B** :
```bash
# Éditer postgresql.auto.conf ou primary_conninfo
# Changer l'IP du Primary pour pointer vers Standby A promu

# Redémarrer Standby B
pg_ctl restart -D /var/lib/postgresql/14/main
```

---

### 2. Réplication Synchrone

Si le Primary utilisait la réplication **synchrone** :

```sql
-- Configuration sur l'ancien Primary
synchronous_standby_names = 'standby1';
```

**Après promotion**, le nouveau Primary hérite de cette configuration, mais :
- Il n'y a plus de Standby synchrone immédiatement
- Les transactions vont **attendre** un Standby qui n'existe pas
- **Impact** : Les écritures seront bloquées !

**Solution** : Désactiver temporairement la réplication synchrone

```sql
-- Sur le nouveau Primary
ALTER SYSTEM SET synchronous_standby_names = '';  
SELECT pg_reload_conf();  
```

Vous pourrez réactiver la réplication synchrone une fois qu'un nouveau Standby sera configuré.

---

### 3. Slots de Réplication

Si le Primary utilisait des **slots de réplication** :

```sql
-- Liste des slots sur le Primary
SELECT slot_name, active FROM pg_replication_slots;
```

Après promotion :
- Les slots sont préservés, mais ne sont plus actifs
- Si vous reconstruisez l'ancien Primary en Standby, vous devrez recréer les slots

## Automatisation vs Manuelle

### Quand Utiliser la Promotion Manuelle ?

- ✅ **Environnements de développement ou test**  
- ✅ **Clusters de petite taille** (1 Primary + 1 Standby)  
- ✅ **Maintenance planifiée** avec contrôle total  
- ✅ **Apprentissage et compréhension des mécanismes**

### Quand Utiliser des Outils d'Automatisation ?

- ✅ **Production critique** (99.99% uptime)  
- ✅ **Clusters complexes** (plusieurs Standby)  
- ✅ **Failover automatique requis** (RTO < 2 minutes)  
- ✅ **Répartition géographique** (multi-datacenter)

**Outils recommandés** :
- **Patroni** : HA automatisé avec consensus (etcd, Consul, ZooKeeper)  
- **Repmgr** : Gestion de réplication simplifiée  
- **pg_auto_failover** : Solution simple de Citus Data

## Checklist de Promotion Manuelle

Avant de promouvoir un Standby en production, suivez cette checklist :

### Phase de Préparation

- [ ] Le Primary est confirmé hors service (panne ou arrêt volontaire)  
- [ ] Le Standby est en bonne santé (logs, connectivité)  
- [ ] Le retard de réplication est acceptable (<1 minute)  
- [ ] Les applications sont prévenues d'une indisponibilité  
- [ ] Un backup récent est disponible (au cas où)  
- [ ] La procédure de rollback est documentée

### Phase de Promotion

- [ ] Connexion SSH au serveur Standby établie  
- [ ] Utilisateur postgres actif  
- [ ] Commande `pg_ctl promote` préparée  
- [ ] Logs PostgreSQL surveillés en temps réel  
- [ ] Commande exécutée : `pg_ctl promote -D /path -w -t 60`

### Phase de Vérification

- [ ] La promotion a réussi (message "server promoted")  
- [ ] `pg_is_in_recovery()` retourne `f` (false)  
- [ ] Le serveur accepte les connexions en écriture  
- [ ] Test d'écriture réussi (INSERT/UPDATE)  
- [ ] Les autres Standby sont reconfigurés (si applicable)

### Phase de Bascule Applicative

- [ ] Chaînes de connexion applicatives mises à jour  
- [ ] Tests de connectivité depuis les applications  
- [ ] Vérification des logs applicatifs (pas d'erreurs)  
- [ ] Monitoring activé sur le nouveau Primary  
- [ ] Communication aux utilisateurs (fin de maintenance)

### Phase Post-Promotion

- [ ] Surveillance intensive du nouveau Primary (24-48h)  
- [ ] Vérification des performances (pas de dégradation)  
- [ ] Planification de la reconstruction d'un nouveau Standby  
- [ ] Documentation de l'incident (si failover non planifié)  
- [ ] Retour d'expérience (REX) avec l'équipe

## Résumé des Concepts Clés

### Ce qu'il Faut Retenir

1. **Promotion = Transformation Standby → Primary**
   - Le Standby devient autonome et accepte les écritures

2. **Commande de Base**
   ```bash
   pg_ctl promote -D /chemin/vers/data -w
   ```

3. **Prérequis Essentiels**
   - Le Primary doit être hors service (éviter le split-brain)
   - Le Standby doit être à jour (retard minimal)
   - Les applications doivent être reconfigurées après promotion

4. **Vérifications Post-Promotion**  
   - `pg_is_in_recovery()` → doit retourner `false`
   - Test d'écriture manuel pour confirmer le mode lecture-écriture

5. **Risque Principal : Split-Brain**
   - Deux Primary actifs = corruption garantie
   - Toujours s'assurer que l'ancien Primary est isolé

La maîtrise de cette technique est une étape essentielle vers l'expertise PostgreSQL en production.
