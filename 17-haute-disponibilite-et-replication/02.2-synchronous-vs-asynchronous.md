# 17.2.2. Synchronous vs Asynchronous - Modes de Réplication PostgreSQL 18

## Introduction

Lorsque vous configurez une réplication Primary/Standby dans PostgreSQL, vous devez faire un choix crucial : utiliser une réplication **synchrone** ou **asynchrone**. Ce choix impacte directement la cohérence des données, la performance, et la disponibilité de votre système.

Ce chapitre explique en profondeur ces deux modes de réplication, leurs différences, leurs avantages et inconvénients, et comment choisir le mode le plus adapté à votre cas d'usage.

---

## Concepts Fondamentaux

### Qu'est-ce que la réplication synchrone ?

Dans une **réplication synchrone**, le serveur primary attend la confirmation du (ou des) standby avant de considérer une transaction comme validée (committed).

**Le processus :**
1. L'application envoie une commande COMMIT au primary  
2. Le primary écrit les WAL sur son disque local  
3. Le primary **attend** que le standby confirme avoir reçu et écrit les WAL  
4. Une fois la confirmation reçue, le primary répond à l'application que la transaction est validée  
5. L'application peut continuer

**Analogie du monde réel :**
Imaginez que vous envoyez une lettre recommandée avec accusé de réception. Vous attendez la confirmation que le destinataire a bien reçu la lettre avant de considérer votre envoi comme réussi.

### Qu'est-ce que la réplication asynchrone ?

Dans une **réplication asynchrone**, le serveur primary valide la transaction immédiatement après l'avoir écrite sur son propre disque, **sans attendre** la confirmation du standby.

**Le processus :**
1. L'application envoie une commande COMMIT au primary  
2. Le primary écrit les WAL sur son disque local  
3. Le primary répond **immédiatement** à l'application que la transaction est validée  
4. L'application peut continuer  
5. En arrière-plan, les WAL sont envoyés au standby qui les applique

**Analogie du monde réel :**
C'est comme envoyer une lettre normale par la poste. Vous la déposez dans la boîte et vous continuez votre journée sans attendre la confirmation qu'elle est arrivée.

## Comparaison Visuelle

### Réplication Asynchrone (par défaut)

```
Application                Primary                 Standby
    │                         │                       │
    │──── COMMIT ────────────>│                       │
    │                         │                       │
    │                    [Écrit WAL]                  │
    │                         │                       │
    │<──── SUCCESS ───────────│                       │
    │                         │                       │
    │   (continue)            │──── WAL ─────────────>│
    │                         │                       │
    │                         │                  [Reçoit WAL]
    │                         │                       │
    │                         │                  [Applique WAL]
    │                         │                       │

Temps écoulé : ~1-2 ms
```

**Caractéristiques :**
- ✅ Très rapide (latence faible)  
- ✅ Le primary n'est pas affecté par les performances du standby  
- ⚠️ Risque de perte de données si le primary tombe avant l'envoi des WAL

### Réplication Synchrone

```
Application                Primary                 Standby
    │                         │                       │
    │──── COMMIT ────────────>│                       │
    │                         │                       │
    │                    [Écrit WAL]                  │
    │                         │                       │
    │                         │──── WAL ─────────────>│
    │                         │                       │
    │       ATTEND            │                  [Reçoit WAL]
    │                         │                       │
    │                         │                  [Écrit WAL]
    │                         │                       │
    │                         │<──── ACK ─────────────│
    │                         │                       │
    │<──── SUCCESS ───────────│                       │
    │                         │                       │
    │   (continue)            │                  [Applique WAL]
    │                         │                       │

Temps écoulé : ~5-50 ms (dépend du réseau)
```

**Caractéristiques :**
- ✅ Garantie de durabilité des données  
- ✅ Aucune perte de données en cas de panne du primary  
- ⚠️ Latence plus élevée (dépend du réseau)  
- ⚠️ Le primary peut être bloqué si le standby est lent ou indisponible

## Avantages et Inconvénients

### Réplication Asynchrone

#### Avantages ✅

**1. Performances maximales**
- Latence de commit très faible (1-2 ms typiquement)
- Débit d'écriture optimal sur le primary
- Pas d'impact des performances réseau sur les transactions

**2. Résilience aux problèmes de standby**
- Si le standby tombe en panne, le primary continue à fonctionner normalement
- Si le réseau entre primary et standby est lent, aucun impact sur le primary
- Idéal pour les réplications géographiquement distantes (cross-region)

**3. Simplicité opérationnelle**
- Configuration par défaut de PostgreSQL
- Moins de risques de blocage du primary
- Plus facile à diagnostiquer

#### Inconvénients ⚠️

**1. Risque de perte de données**
- Si le primary tombe en panne brutalement (crash serveur, coupure électrique), les transactions validées mais non encore répliquées sont perdues
- Le "window of data loss" peut aller de quelques millisecondes à plusieurs secondes selon la charge

**2. Inconsistance temporaire**
- Le standby peut être en retard par rapport au primary
- Les lectures sur le standby peuvent retourner des données légèrement obsolètes

**3. Pas adapté aux applications critiques**
- Pour des données financières, médicales, ou toute donnée où la perte est inacceptable

### Réplication Synchrone

#### Avantages ✅

**1. Garantie de durabilité (Zero Data Loss)**
- Aucune transaction validée ne peut être perdue
- Si le primary tombe, le standby possède **toutes** les données validées
- Conforme aux exigences de haute criticité (finance, santé, etc.)

**2. Cohérence forte**
- Le standby est toujours à jour (à quelques microsecondes près)
- Les lectures sur le standby sont quasi synchronisées avec le primary

**3. Conformité réglementaire**
- Répond aux normes de type ACID strict
- Facilite les audits de conformité (GDPR, SOC2, PCI-DSS)

#### Inconvénients ⚠️

**1. Impact sur les performances**
- Latence de commit augmentée (5-50 ms selon le réseau)
- Chaque transaction doit attendre l'aller-retour réseau
- Débit d'écriture réduit de 20-50% par rapport à l'asynchrone

**2. Dépendance au standby**
- Si le standby tombe, le primary peut être configuré pour :
  - **Bloquer** toutes les écritures (mode strict) → Indisponibilité  
  - **Revenir en mode asynchrone** (mode dégradé) → Risque de perte de données
- Si le réseau est lent, les transactions sont impactées

**3. Complexité opérationnelle**
- Monitoring plus complexe (surveillance du lag synchrone)
- Nécessite une planification réseau rigoureuse
- Gestion des scenarios de failover plus délicate

## Tableau Comparatif Récapitulatif

| Critère | Asynchrone | Synchrone |
|---------|------------|-----------|
| **Latence de commit** | 1-2 ms | 5-50 ms |
| **Débit d'écriture** | Maximum | Réduit (20-50%) |
| **Risque de perte de données** | Oui (fenêtre de quelques ms à quelques secondes) | Non (zero data loss) |
| **Impact panne standby** | Aucun sur le primary | Peut bloquer le primary |
| **Impact réseau lent** | Aucun | Ralentit les transactions |
| **Cohérence primary/standby** | Quelques secondes de retard possible | Quasi temps réel |
| **Complexité opérationnelle** | Faible | Moyenne à élevée |
| **Cas d'usage typique** | Blogs, applications web, réplication cross-region | Finance, santé, données critiques |
| **Coût d'infrastructure** | Standard | Nécessite réseau de qualité |

## Configuration de la Réplication Synchrone

### Étape 1 : Configurer postgresql.conf sur le Primary

La réplication synchrone se configure principalement sur le serveur primary via le paramètre `synchronous_standby_names`.

**Éditer le fichier postgresql.conf :**

```ini
# ────────────────────────────────────────────────
# Configuration de la réplication synchrone
# ────────────────────────────────────────────────

# Liste des standbys considérés comme synchrones
# Format : 'FIRST num (standby1, standby2, ...)'
synchronous_standby_names = 'FIRST 1 (standby1, standby2)'

# Ou pour un seul standby :
# synchronous_standby_names = 'standby1'

# Niveau de synchronisation (à partir de PG 18)
# Options : off, local, remote_write, remote_apply, on (défaut)
synchronous_commit = on
```

**Explications détaillées :**

#### synchronous_standby_names

Ce paramètre définit quels standbys doivent confirmer les transactions pour qu'elles soient considérées comme validées.

**Syntaxe générale :**
```ini
synchronous_standby_names = 'FIRST num (standby1, standby2, ...)'
```

**Exemples de configuration :**

**Exemple 1 : Un seul standby synchrone**
```ini
synchronous_standby_names = 'standby1'
```
- Le primary attend la confirmation de `standby1` pour chaque commit
- Si `standby1` est indisponible, le primary **bloque** (ou bascule en async selon `synchronous_commit`)

**Exemple 2 : Plusieurs standbys, 1 suffit (FIRST 1)**
```ini
synchronous_standby_names = 'FIRST 1 (standby1, standby2)'
```
- Le primary attend la confirmation d'**au moins 1** standby parmi `standby1` et `standby2`
- Si `standby1` répond en premier, la transaction est validée immédiatement
- Si `standby1` est en panne, `standby2` prend le relais automatiquement
- ✅ **Configuration recommandée** pour allier performance et résilience

**Exemple 3 : Plusieurs standbys, tous doivent confirmer (ANY)**
```ini
synchronous_standby_names = 'ANY 2 (standby1, standby2, standby3)'
```
- Le primary attend la confirmation d'**au moins 2** standbys parmi les 3
- Plus résilient mais plus lent
- Utilisé pour des données ultra-critiques

**Exemple 4 : Plusieurs standbys, ordre de priorité**
```ini
synchronous_standby_names = 'standby1, standby2'
```
- Le primary essaie d'abord `standby1`
- Si `standby1` est indisponible, bascule automatiquement sur `standby2`
- Syntaxe sans `FIRST` (équivalent à `FIRST 1`)

**Note importante :** Le nom doit correspondre exactement à l'`application_name` configuré dans le `primary_conninfo` du standby.

#### synchronous_commit

Ce paramètre contrôle le **niveau de garantie** de synchronisation.

**Valeurs possibles :**

| Valeur | Description | Garantie | Performance |
|--------|-------------|----------|-------------|
| `off` | Asynchrone total (pas d'attente) | ❌ Aucune | ⚡ Maximale |
| `local` | Attend écriture locale (sur le primary) | ⚠️ Locale uniquement | ⚡ Élevée |
| `remote_write` | Attend écriture sur le standby (OS cache) | ⚠️ Partielle | 🔶 Moyenne |
| `on` (défaut) | Attend flush disque sur le standby | ✅ Forte | 🔶 Moyenne |
| `remote_apply` | Attend application des WAL sur le standby | ✅ Maximale | 🐌 Faible |

**Explications détaillées :**

**`off` - Asynchrone complet**
- Aucune attente de confirmation
- Le primary valide dès l'écriture en mémoire
- ⚠️ Risque de perte même en cas de crash du primary seul

**`local` - Garantie locale**
- Le primary attend que les WAL soient écrits sur son propre disque
- Pas d'attente du standby
- Protection contre un crash du processus PostgreSQL, mais pas contre une panne disque

**`remote_write` - Écriture distante**
- Le standby confirme avoir reçu les WAL et les avoir écrits en mémoire (cache OS)
- Pas encore flushés sur disque
- ⚠️ Risque de perte si le standby crash immédiatement après (données en cache non persistées)  
- **Usage :** Bon compromis performance/sécurité si le standby a un système de cache sécurisé (battery-backed)

**`on` - Flush disque distant (par défaut)**
- Le standby confirme avoir écrit les WAL sur disque physique (fsync)
- ✅ **Configuration recommandée** pour la réplication synchrone
- Garantie de durabilité même en cas de crash du standby

**`remote_apply` - Application distante**
- Le standby confirme avoir non seulement écrit les WAL sur disque, mais aussi les avoir **appliqués** à la base de données
- Garantie la plus forte : les données sont disponibles immédiatement sur le standby
- **Usage :** Lecture sur le standby avec garantie de cohérence absolue (read-after-write)  
- ⚠️ Plus lent car nécessite le temps d'application des WAL

**Comparaison des temps de latence (approximatifs) :**
- `off` : 0.5-1 ms  
- `local` : 1-2 ms  
- `remote_write` : 3-10 ms (latence réseau)  
- `on` : 5-20 ms (latence réseau + fsync)  
- `remote_apply` : 10-50 ms (latence réseau + fsync + application)

### Étape 2 : Configurer l'application_name sur le Standby

Le standby doit s'identifier avec un nom que le primary pourra reconnaître.

**Éditer postgresql.conf sur le Standby :**

```ini
# ────────────────────────────────────────────────
# Configuration de connexion au Primary
# ────────────────────────────────────────────────

primary_conninfo = 'host=192.168.1.10 port=5432 user=replicator password=*** application_name=standby1'
```

**Le paramètre `application_name=standby1` est crucial :** c'est ce nom qui sera référencé dans `synchronous_standby_names` sur le primary.

### Étape 3 : Recharger la configuration

**Sur le Primary :**
```bash
# Recharger la configuration sans redémarrage
pg_ctl reload -D /var/lib/postgresql/18/main

# Ou avec systemd
sudo systemctl reload postgresql
```

**Vérification :**
```sql
-- Sur le primary
SHOW synchronous_standby_names;  
SHOW synchronous_commit;  
```

### Étape 4 : Vérifier l'état de synchronisation

**Sur le Primary :**
```sql
SELECT
    application_name,
    client_addr,
    state,
    sync_state,          -- 'sync' = synchrone, 'async' = asynchrone, 'potential' = peut devenir sync
    sync_priority,       -- Priorité (1 = le plus prioritaire)
    replay_lag
FROM pg_stat_replication  
ORDER BY sync_priority;  
```

**Résultat attendu pour une réplication synchrone :**
```
 application_name | client_addr   | state     | sync_state | sync_priority | replay_lag
------------------+---------------+-----------+------------+---------------+------------
 standby1         | 192.168.1.20  | streaming | sync       | 1             | 00:00:00.05
 standby2         | 192.168.1.21  | streaming | potential  | 2             | 00:00:00.08
```

**Interprétation :**
- **sync_state = 'sync'** : Ce standby est actuellement utilisé pour la réplication synchrone  
- **sync_state = 'potential'** : Ce standby peut devenir synchrone si le standby sync actuel tombe  
- **sync_state = 'async'** : Ce standby est en mode asynchrone  
- **sync_priority** : Ordre de priorité (1 = plus haute priorité)

## Modes Hybrides et Stratégies Avancées

### Mode FIRST N : Quorum de Synchronisation

Le mode `FIRST N` permet d'attendre la confirmation de N standbys parmi une liste.

**Configuration :**
```ini
synchronous_standby_names = 'FIRST 2 (standby1, standby2, standby3)'
```

**Fonctionnement :**
- Le primary attend **au moins 2** confirmations parmi les 3 standbys
- Si `standby1` et `standby2` répondent rapidement, la transaction est validée
- Si `standby1` est en panne, le primary attend `standby2` et `standby3`

**Avantages :**
- ✅ Tolérance aux pannes : 1 standby peut tomber sans bloquer le primary  
- ✅ Garantie de durabilité forte : les données sont sur 3 serveurs

**Inconvénients :**
- ⚠️ Latence augmentée (attente du 2ème standby le plus lent)  
- ⚠️ Plus complexe à monitorer

**Cas d'usage :**
- Données ultra-critiques (finance, santé)
- Architectures multi-datacenter
- Réglementations strictes (ex: 3 copies de données minimum)

### Mode ANY N : N parmi M

Le mode `ANY N` est similaire à `FIRST N` mais sans notion de priorité.

**Configuration :**
```ini
synchronous_standby_names = 'ANY 2 (standby1, standby2, standby3)'
```

**Différence avec FIRST :**
- `FIRST 2` : Choisit les 2 premiers qui se connectent (priorité implicite)  
- `ANY 2` : N'importe quels 2 standbys parmi les 3 (pas de priorité)

En pratique, la différence est minime et `FIRST N` est plus couramment utilisé.

### Réplication Synchrone à Plusieurs Niveaux

Pour des architectures complexes avec plusieurs régions géographiques.

**Architecture :**
```
Primary (Paris)
    ├─> Standby1 (Paris) [SYNC]
    └─> Standby2 (Londres) [ASYNC]
            └─> Standby3 (New York) [ASYNC via cascading]
```

**Configuration sur le Primary (Paris) :**
```ini
synchronous_standby_names = 'standby1'  # Uniquement le standby local
```

**Avantages :**
- ✅ Faible latence (standby synchrone dans le même datacenter)  
- ✅ Protection géographique (standbys asynchrones à distance)  
- ✅ Bon équilibre performance/résilience

**Inconvénients :**
- ⚠️ Perte de données possible si le datacenter Paris complet tombe

## Gestion des Scénarios de Panne

### Scénario 1 : Le standby synchrone tombe

**Configuration :**
```ini
synchronous_standby_names = 'standby1'
```

**Que se passe-t-il ?**
1. Le standby synchrone (`standby1`) devient indisponible (panne serveur, réseau)  
2. Le primary détecte la perte de connexion (après `wal_sender_timeout`)  
3. **Comportement par défaut :** Le primary **bloque** toutes les transactions en attente de commit

**Conséquences :**
- ❌ Le primary devient **indisponible en écriture** (les applications reçoivent des timeouts)  
- ✅ Garantie de zero data loss maintenue

**Solutions :**

**Option 1 : Configuration FIRST 1 avec fallback**
```ini
synchronous_standby_names = 'FIRST 1 (standby1, standby2)'
```
- Si `standby1` tombe, le primary bascule automatiquement sur `standby2`
- ✅ **Recommandé** pour la production

**Option 2 : Désactiver temporairement la réplication synchrone**
```sql
-- En urgence, pour débloquer le primary
ALTER SYSTEM SET synchronous_standby_names = '';  
SELECT pg_reload_conf();  
```
- ⚠️ Le primary rebascule en mode asynchrone  
- ⚠️ Risque de perte de données pendant cette période

**Option 3 : Timeout de synchronisation (PostgreSQL 18+)**
PostgreSQL 18 n'a pas de timeout natif, mais on peut simuler avec des paramètres de connexion :
```ini
# Sur le primary
wal_sender_timeout = 60s  # Timeout après 60 secondes
```

### Scénario 2 : Réseau lent entre Primary et Standby

**Symptôme :** Les transactions sont lentes (plusieurs secondes de latence).

**Diagnostic :**
```sql
SELECT
    application_name,
    write_lag,
    flush_lag,
    replay_lag
FROM pg_stat_replication;
```

Si `flush_lag` > 100ms, le réseau est probablement saturé ou lent.

**Solutions :**

**Solution 1 : Passer temporairement en asynchrone**
```sql
ALTER SYSTEM SET synchronous_commit = 'local';  
SELECT pg_reload_conf();  
```

**Solution 2 : Augmenter la bande passante réseau**
- Upgrade du lien réseau (1 Gbps → 10 Gbps)
- Utiliser un réseau dédié à la réplication

**Solution 3 : Changer de mode de synchronisation**
```sql
-- Passer de 'on' à 'remote_write' pour réduire la latence
ALTER SYSTEM SET synchronous_commit = 'remote_write';  
SELECT pg_reload_conf();  
```

### Scénario 3 : Failover avec réplication synchrone

**Situation :** Le primary tombe et on doit promouvoir le standby.

**Avantages de la réplication synchrone :**
- ✅ Le standby possède **toutes** les données validées  
- ✅ Aucune perte de données  
- ✅ RTO (Recovery Time Objective) rapide

**Procédure de promotion :**
```bash
# Sur le standby
pg_ctl promote -D /var/lib/postgresql/18/main

# Ou créer un trigger file (si configuré)
touch /tmp/postgresql.trigger.5432
```

**Après la promotion :**
1. Le standby devient le nouveau primary  
2. Vérifier que les applications se reconnectent au nouveau primary  
3. Reconfigurer l'ancien primary en standby (quand il revient)

## Monitoring et Observabilité

### Métriques clés pour la réplication synchrone

**1. État de synchronisation**
```sql
SELECT
    application_name,
    sync_state,
    sync_priority
FROM pg_stat_replication;
```

**Alertes à configurer :**
- ⚠️ Aucun standby en mode 'sync'  
- ⚠️ Changement inattendu de sync_state (sync → potential)

**2. Latence de réplication**
```sql
SELECT
    application_name,
    extract(epoch from write_lag) * 1000 AS write_lag_ms,
    extract(epoch from flush_lag) * 1000 AS flush_lag_ms,
    extract(epoch from replay_lag) * 1000 AS replay_lag_ms
FROM pg_stat_replication;
```

**Seuils recommandés :**
- `flush_lag_ms < 10ms` : Excellent  
- `flush_lag_ms < 50ms` : Acceptable  
- `flush_lag_ms > 100ms` : Problème (réseau lent, standby surchargé)

**3. Transactions en attente**
```sql
SELECT count(*)  
FROM pg_stat_activity  
WHERE wait_event_type = 'IPC'  
AND wait_event = 'SyncRep';  
```

Si ce nombre est > 0 de manière prolongée, cela signifie que des transactions attendent la confirmation du standby.

**4. Impact sur le débit**

Comparer le débit d'écriture avant/après activation de la réplication synchrone :

```sql
-- Transactions par seconde
SELECT
    datname,
    xact_commit + xact_rollback AS tps
FROM pg_stat_database  
WHERE datname = 'ma_base';  
```

Exécuter cette requête régulièrement et comparer. Une baisse de 20-50% est normale.

### Dashboard Grafana recommandé

**Métriques essentielles à afficher :**
- État de synchronisation (sync/async/potential)
- Latence de réplication (write_lag, flush_lag, replay_lag)
- Nombre de standbys actifs
- Débit de transactions (TPS)
- Temps de commit moyen
- Lag bytes (octets de retard)

## Choix du Mode : Arbre de Décision

### Questions à se poser

**1. Quelle est la criticité des données ?**

| Criticité | Mode recommandé |
|-----------|-----------------|
| ⭐ Faible (blogs, logs, cache) | Asynchrone |
| ⭐⭐ Moyenne (e-commerce, CRM) | Asynchrone avec backup régulier |
| ⭐⭐⭐ Élevée (finance, santé, inventaire) | Synchrone (FIRST 1) |
| ⭐⭐⭐⭐ Critique (banking, données médicales) | Synchrone (FIRST 2 ou ANY 2) |

**2. Quelle est la tolérance à la latence ?**

| Latence acceptable | Mode recommandé |
|--------------------|-----------------|
| < 5 ms | Asynchrone |
| < 20 ms | Synchrone (local datacenter) |
| < 100 ms | Asynchrone ou remote_write |
| > 100 ms | Asynchrone uniquement |

**3. Quelle est l'architecture réseau ?**

| Configuration réseau | Mode recommandé |
|----------------------|-----------------|
| Primary et Standby dans le même datacenter (< 1ms latence) | Synchrone (on) |
| Primary et Standby dans la même région (< 5ms latence) | Synchrone (on ou remote_write) |
| Primary et Standby dans des régions différentes (> 10ms latence) | Asynchrone |
| Multi-datacenter avec plusieurs standbys | Synchrone local + Asynchrone distant |

**4. Quel est le volume d'écritures ?**

| Volume d'écritures | Mode recommandé |
|--------------------|-----------------|
| < 1000 TPS | Synchrone ou Asynchrone |
| 1000-10000 TPS | Asynchrone ou Synchrone avec réseau rapide |
| > 10000 TPS | Asynchrone (réplication synchrone difficile à maintenir) |

### Arbre de décision visuel

```
Est-ce que la perte de données est acceptable ?
├─ OUI → Asynchrone
└─ NON →
    │
    Le standby est dans le même datacenter ?
    ├─ OUI → Synchrone (FIRST 1 avec fallback)
    └─ NON →
        │
        Latence réseau < 20ms ?
        ├─ OUI → Synchrone (considérer remote_write)
        └─ NON → Asynchrone (+ PITR pour récupération)
```

## Exemples de Configuration par Cas d'Usage

### Cas 1 : Application Web E-commerce

**Contexte :**
- Données importantes mais perte tolérable de quelques transactions
- Priorité : Performance et disponibilité
- Volume : 5000 TPS
- Architecture : 1 primary + 2 standbys (même datacenter)

**Configuration recommandée :**
```ini
# Sur le primary
synchronous_commit = local           # Garantie locale uniquement  
synchronous_standby_names = ''       # Asynchrone  
max_wal_senders = 5  
```

**Stratégie de backup :**
- Sauvegardes quotidiennes (pg_dump)
- Archivage WAL activé
- Tests de restauration mensuels

### Cas 2 : Application Financière (Banque)

**Contexte :**
- Zero data loss impératif
- Conformité réglementaire stricte
- Volume : 2000 TPS
- Architecture : 1 primary + 3 standbys (2 locaux, 1 distant)

**Configuration recommandée :**
```ini
# Sur le primary
synchronous_commit = on                                    # Garantie forte  
synchronous_standby_names = 'FIRST 2 (standby1, standby2, standby3)'  # Quorum de 2  
```

**Architecture :**
- Primary (Paris)
- Standby1 (Paris datacenter A) [SYNC]
- Standby2 (Paris datacenter B) [SYNC]
- Standby3 (Londres) [ASYNC via cascading depuis Standby1]

**Avantages :**
- ✅ Protection contre la panne de 1 datacenter complet  
- ✅ Zero data loss garanti (2 copies synchrones)  
- ✅ Réplication géographique pour disaster recovery

### Cas 3 : Plateforme Analytics (Data Warehouse)

**Contexte :**
- Charges de lecture intensives
- Écritures batch (nuit)
- Priorité : Séparation lecture/écriture
- Volume : 500 TPS

**Configuration recommandée :**
```ini
# Sur le primary
synchronous_commit = remote_apply                # Garantie forte + application  
synchronous_standby_names = 'standby_analytics'  # 1 standby dédié lectures  
```

**Architecture :**
- Primary : Écritures uniquement
- Standby analytics : Lectures (reporting, BI)
- `remote_apply` garantit que les données sont immédiatement visibles sur le standby

**Avantages :**
- ✅ Zéro délai entre écriture et lecture (remote_apply)  
- ✅ Isolation des charges (OLTP vs OLAP)

### Cas 4 : Application SaaS Multi-tenant

**Contexte :**
- Plusieurs clients avec criticités différentes
- Nécessité de flexibilité
- Volume : Variable (100-5000 TPS par tenant)

**Configuration recommandée :**
```ini
# Sur le primary
synchronous_commit = local                        # Par défaut : asynchrone  
synchronous_standby_names = 'FIRST 1 (standby1, standby2)'  
```

**Stratégie applicative :**
Utiliser `synchronous_commit` au niveau de la session pour les clients critiques :

```sql
-- Pour un client premium nécessitant zero data loss
SET SESSION synchronous_commit = on;

-- Pour un client standard
SET SESSION synchronous_commit = local;
```

**Avantages :**
- ✅ Flexibilité par client  
- ✅ Performance optimale pour la majorité  
- ✅ Garantie forte pour les clients critiques

## Nouveautés PostgreSQL 18

PostgreSQL 18 apporte quelques améliorations à la réplication synchrone :

### 1. Métriques de réplication améliorées

Nouvelles colonnes dans `pg_stat_replication` :
```sql
SELECT
    application_name,
    sync_state,
    -- Nouvelles métriques PG 18
    wal_sender_pid,
    wal_sender_state,
    last_msg_send_time,
    last_msg_receipt_time
FROM pg_stat_replication;
```

### 2. Observabilité du lag améliorée

PostgreSQL 18 affiche automatiquement les buffers dans EXPLAIN, facilitant le diagnostic des problèmes de réplication :

```sql
EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM ma_table WHERE id = 123;  
```

### 3. Optimisations du planificateur

Les améliorations du planificateur (auto-élimination des self-joins, optimisation des OR-clauses) réduisent la charge sur le primary, ce qui améliore indirectement les performances de réplication.

## Bonnes Pratiques de Production

### 1. Toujours avoir un fallback

**❌ Mauvais :**
```ini
synchronous_standby_names = 'standby1'
```
Si `standby1` tombe, le primary bloque.

**✅ Bon :**
```ini
synchronous_standby_names = 'FIRST 1 (standby1, standby2)'
```
Si `standby1` tombe, `standby2` prend le relais automatiquement.

### 2. Monitorer en continu

**Alertes essentielles :**
- Aucun standby en mode sync pendant > 5 minutes
- flush_lag > 50ms pendant > 10 minutes
- Changement de sync_state (sync → potential)
- Augmentation soudaine du temps de commit

### 3. Tester les scénarios de panne

**Tests réguliers (trimestriels) :**
- Arrêter le standby sync et vérifier le failover vers le standby potential
- Simuler une latence réseau élevée (avec `tc` sur Linux)
- Promouvoir un standby et vérifier l'intégrité des données

### 4. Documenter la configuration

**Documentation minimale :**
- Diagramme d'architecture (primary, standbys, réseau)
- Configuration de `synchronous_standby_names` et justification
- Procédures de failover et rollback
- Contacts et escalade en cas d'incident
- RPO (Recovery Point Objective) et RTO (Recovery Time Objective) cibles

### 5. Équilibrer performance et durabilité

**Ne pas tomber dans les extrêmes :**
- Trop asynchrone → Risque de perte de données
- Trop synchrone → Performances dégradées, complexité accrue

**Approche pragmatique :**
1. Commencer en asynchrone  
2. Mesurer le volume d'écritures et la latence acceptable  
3. Activer la réplication synchrone de manière progressive  
4. Monitorer l'impact et ajuster

### 6. Prévoir la croissance

**Questions à anticiper :**
- Comment la configuration évoluera avec 10× le volume d'écritures actuel ?
- Est-ce que la bande passante réseau est suffisante pour 5 ans ?
- Peut-on ajouter un 3ème standby sans refonte complète ?

## Résumé et Recommandations

### Quand utiliser l'Asynchrone ?

✅ **Utilisez l'asynchrone si :**
- Les performances sont la priorité absolue
- La perte de quelques transactions est acceptable
- Le standby est géographiquement distant (latence > 20ms)
- Vous avez une stratégie de backup robuste (PITR)
- Le volume d'écritures est très élevé (> 10 000 TPS)

**Cas d'usage typiques :**
- Applications web grand public
- Systèmes de logs et d'analytics
- Réplication cross-region pour disaster recovery
- Caches distribués

### Quand utiliser le Synchrone ?

✅ **Utilisez le synchrone si :**
- Zero data loss est requis (finance, santé, données légales)
- Les standbys sont dans le même datacenter (latence < 5ms)
- Le volume d'écritures est modéré (< 5000 TPS)
- Vous êtes soumis à des réglementations strictes (GDPR, SOC2, PCI-DSS)

**Cas d'usage typiques :**
- Applications bancaires et financières
- Systèmes de santé (dossiers médicaux)
- E-commerce avec transactions critiques
- Gestion d'inventaire en temps réel

### Configuration recommandée par défaut

**Pour la plupart des applications de production :**

```ini
# Sur le primary
synchronous_commit = on  
synchronous_standby_names = 'FIRST 1 (standby1, standby2)'  
wal_keep_size = 2GB  
max_wal_senders = 5  
```

**Cette configuration offre :**
- ✅ Protection contre la perte de données (synchrone)  
- ✅ Résilience aux pannes (FIRST 1 avec fallback)  
- ✅ Performance acceptable (< 20ms de latence si réseau local)

## Conclusion

Le choix entre réplication synchrone et asynchrone est un **trade-off fondamental** entre :
- **Durabilité des données** ↔ **Performance**  
- **Garanties fortes** ↔ **Simplicité opérationnelle**  
- **Zero data loss** ↔ **Haute disponibilité**

**Points clés à retenir :**

1. **L'asynchrone est le mode par défaut** et convient à la majorité des cas d'usage  
2. **Le synchrone est indispensable** pour les données critiques où la perte est inacceptable  
3. **La configuration `FIRST 1` avec fallback** est le meilleur compromis pour la production  
4. **Le monitoring est crucial** pour détecter les dégradations de performance  
5. **Il n'y a pas de solution universelle** : adaptez votre configuration à vos besoins réels

**Prochaines étapes :**
- Mettre en place un monitoring de la réplication (Prometheus, Grafana)
- Tester les scénarios de failover en environnement de staging
- Mesurer l'impact réel sur les performances de votre application
- Explorer la réplication en cascade (17.2.3) pour les architectures multi-régions

La réplication synchrone est un outil puissant, mais elle doit être utilisée avec discernement. Une configuration bien pensée et adaptée à votre contexte vous permettra d'atteindre le juste équilibre entre fiabilité et performance.
