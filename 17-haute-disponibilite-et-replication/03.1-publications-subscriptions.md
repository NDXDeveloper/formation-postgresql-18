# 17.3.1. Publications et Subscriptions - Réplication Logique PostgreSQL 18

## Introduction

La réplication logique dans PostgreSQL repose sur deux concepts fondamentaux : les **publications** et les **subscriptions**. Contrairement à la réplication physique qui copie les blocs de données bruts, la réplication logique reproduit les changements de données au niveau logique (lignes, colonnes, tables), offrant ainsi une flexibilité bien plus grande.

### Analogie simple

Imaginez deux journaux d'entreprise :
- **La publication** : C'est comme un journal qui décide quels articles publier (quelles tables partager)  
- **La subscription** : C'est comme un abonné qui reçoit ce journal et lit les articles qui l'intéressent

## Qu'est-ce qu'une Publication ?

Une **publication** est un objet créé sur le serveur **source** (appelé publisher) qui détermine **quelles données** seront répliquées. C'est une sorte de "catalogue" qui dit : "Voici les tables et les changements que je rends disponibles pour la réplication".

### Caractéristiques d'une Publication

- **Sélectivité** : Vous pouvez publier toutes les tables d'une base ou seulement certaines tables spécifiques  
- **Filtrage d'opérations** : Vous pouvez choisir de publier uniquement les INSERT, ou les UPDATE, ou les DELETE, ou toute combinaison  
- **Granularité fine** : Possibilité de publier seulement certaines colonnes d'une table (depuis PostgreSQL 15)  
- **Multiple publications** : Une même table peut appartenir à plusieurs publications différentes

### Syntaxe de base - CREATE PUBLICATION

```sql
-- Publication simple : publier une table
CREATE PUBLICATION ma_publication FOR TABLE utilisateurs;

-- Publication de plusieurs tables
CREATE PUBLICATION ma_publication FOR TABLE utilisateurs, commandes, produits;

-- Publication de toutes les tables de la base
CREATE PUBLICATION ma_publication_complete FOR ALL TABLES;

-- Publication avec filtre d'opérations (seulement INSERT et UPDATE)
CREATE PUBLICATION ma_publication  
FOR TABLE utilisateurs  
WITH (publish = 'insert, update');  

-- Publication avec filtrage de colonnes (PostgreSQL 15+)
CREATE PUBLICATION ma_publication  
FOR TABLE utilisateurs (id, nom, email);  -- seulement ces colonnes  

-- Publication avec clause WHERE (PostgreSQL 15+)
CREATE PUBLICATION ma_publication  
FOR TABLE utilisateurs WHERE (pays = 'France');  
```

### Options de Publication

| Option | Valeurs possibles | Description |
|--------|-------------------|-------------|
| `publish` | `'insert'`, `'update'`, `'delete'`, `'truncate'` | Définit quelles opérations sont répliquées (par défaut : toutes) |
| `publish_via_partition_root` | `true` / `false` | Pour les tables partitionnées : publier via la table parente ou les partitions |

## Qu'est-ce qu'une Subscription ?

Une **subscription** est un objet créé sur le serveur **destination** (appelé subscriber) qui se connecte à une publication et **reçoit** les changements de données. C'est l'abonné qui "tire" les données depuis le publisher.

### Caractéristiques d'une Subscription

- **Connexion active** : La subscription établit une connexion permanente vers le publisher  
- **Réplication en temps quasi-réel** : Les changements sont transmis au fur et à mesure  
- **Snapshot initial** : Par défaut, une copie initiale des données est effectuée  
- **Gestion automatique** : PostgreSQL gère automatiquement le suivi des changements (via WAL)

### Syntaxe de base - CREATE SUBSCRIPTION

```sql
-- Subscription simple
CREATE SUBSCRIPTION ma_souscription  
CONNECTION 'host=serveur-source.example.com port=5432 dbname=production user=replicateur password=secret'  
PUBLICATION ma_publication;  

-- Subscription sans copie initiale
CREATE SUBSCRIPTION ma_souscription  
CONNECTION 'host=serveur-source.example.com port=5432 dbname=production user=replicateur'  
PUBLICATION ma_publication  
WITH (copy_data = false);  

-- Subscription pour plusieurs publications
CREATE SUBSCRIPTION ma_souscription  
CONNECTION 'host=serveur-source.example.com port=5432 dbname=production user=replicateur'  
PUBLICATION publication1, publication2;  

-- Subscription désactivée (à activer manuellement plus tard)
CREATE SUBSCRIPTION ma_souscription  
CONNECTION 'host=serveur-source.example.com port=5432 dbname=production user=replicateur'  
PUBLICATION ma_publication  
WITH (enabled = false);  
```

### Options de Subscription

| Option | Valeurs possibles | Description |
|--------|-------------------|-------------|
| `copy_data` | `true` / `false` | Copier les données existantes au démarrage (défaut : `true`) |
| `enabled` | `true` / `false` | Activer immédiatement la subscription (défaut : `true`) |
| `create_slot` | `true` / `false` | Créer automatiquement un slot de réplication (défaut : `true`) |
| `slot_name` | nom personnalisé | Nom du slot de réplication à utiliser |
| `synchronous_commit` | `off`, `local`, `remote_write`, `on` | Niveau de commit synchrone |

## Comment ça fonctionne : Le Flux de Réplication

### Étape par Étape

1. **Création de la publication** (sur le serveur source)
   - L'administrateur décide quelles tables publier
   - PostgreSQL enregistre cette configuration

2. **Création de la subscription** (sur le serveur destination)
   - Le serveur destination se connecte au serveur source
   - Un **slot de réplication** est créé sur le source (pour suivre la progression)

3. **Snapshot initial** (si `copy_data = true`)
   - Toutes les données existantes des tables publiées sont copiées
   - Cette opération peut prendre du temps pour de grandes tables

4. **Réplication continue**
   - Le serveur source écrit tous les changements dans le WAL
   - Le décodeur logique (`pgoutput`) lit le WAL et transforme les changements en format logique
   - Ces changements sont envoyés au subscriber via le slot de réplication
   - Le subscriber applique les changements sur ses tables locales

### Schéma conceptuel

```
[SERVEUR SOURCE - Publisher]
    │
    ├─ Base de données : production
    │   ├─ Table : utilisateurs  ──┐
    │   ├─ Table : commandes      ─┤
    │   └─ Table : produits       ─┤
    │                               │
    └─ PUBLICATION "ma_pub" ───────┘
         │
         │ (Connexion réseau)
         │
         ▼
    ┌─ SUBSCRIPTION "ma_sub"
    │
[SERVEUR DESTINATION - Subscriber]
    │
    └─ Base de données : replication
        ├─ Table : utilisateurs   (répliquée)
        ├─ Table : commandes      (répliquée)
        └─ Table : produits       (répliquée)
```

## Slots de Réplication : Le Mécanisme de Suivi

Un **slot de réplication** est un objet sur le serveur source qui :
- **Mémorise** jusqu'où le subscriber a lu les changements  
- **Préserve** les données WAL nécessaires même si elles devraient être supprimées  
- **Garantit** qu'aucun changement ne sera perdu si le subscriber est temporairement déconnecté

### ⚠️ Attention aux Slots

Si un subscriber est arrêté longtemps et que le slot est actif :
- Le WAL continue de s'accumuler sur le serveur source
- **Risque de saturation disque** sur le publisher
- Solution : surveiller la taille des slots et les supprimer si nécessaire

```sql
-- Voir les slots de réplication actifs
SELECT * FROM pg_replication_slots;

-- Supprimer un slot manuellement (si nécessaire)
SELECT pg_drop_replication_slot('nom_du_slot');
```

## Configuration Requise

### Sur le Serveur Source (Publisher)

1. **Paramètres PostgreSQL** à configurer dans `postgresql.conf` :

```ini
# Niveau de WAL requis pour la réplication logique
wal_level = logical

# Nombre maximum de workers de réplication
max_replication_slots = 10  # Au moins 1 par subscription

# Nombre de processus WAL sender
max_wal_senders = 10        # Au moins 1 par subscription
```

2. **Authentification** dans `pg_hba.conf` :

```
# Autoriser la connexion pour la réplication
host    production    replicateur    192.168.1.0/24    scram-sha-256
```

3. **Redémarrage requis** après modification de `wal_level`

### Sur le Serveur Destination (Subscriber)

1. **Paramètres PostgreSQL** :

```ini
# Nombre maximum de workers pour appliquer les changements
max_logical_replication_workers = 4

# Nombre de workers par subscription
max_sync_workers_per_subscription = 2
```

2. **Les tables doivent exister** avant de créer la subscription
   - Même structure (colonnes, types)
   - Pas nécessairement les mêmes contraintes ou index

## Gestion des Publications et Subscriptions

### Commandes de Gestion - Publications

```sql
-- Voir toutes les publications
SELECT * FROM pg_publication;

-- Voir les tables d'une publication
SELECT * FROM pg_publication_tables WHERE pubname = 'ma_publication';

-- Ajouter une table à une publication existante
ALTER PUBLICATION ma_publication ADD TABLE nouvelle_table;

-- Retirer une table d'une publication
ALTER PUBLICATION ma_publication DROP TABLE ancienne_table;

-- Modifier les opérations publiées
ALTER PUBLICATION ma_publication SET (publish = 'insert, update');

-- Supprimer une publication
DROP PUBLICATION ma_publication;
```

### Commandes de Gestion - Subscriptions

```sql
-- Voir toutes les subscriptions
SELECT * FROM pg_subscription;

-- Voir l'état de réplication
SELECT * FROM pg_stat_subscription;

-- Désactiver temporairement une subscription
ALTER SUBSCRIPTION ma_souscription DISABLE;

-- Réactiver une subscription
ALTER SUBSCRIPTION ma_souscription ENABLE;

-- Rafraîchir la subscription (recharger les changements de publication)
ALTER SUBSCRIPTION ma_souscription REFRESH PUBLICATION;

-- Supprimer une subscription
DROP SUBSCRIPTION ma_souscription;
```

### Surveillance et Monitoring

```sql
-- État global des subscriptions
SELECT
    subname AS subscription_name,
    pid,
    received_lsn,
    latest_end_lsn,
    last_msg_send_time,
    last_msg_receipt_time,
    latest_end_time
FROM pg_stat_subscription;

-- Décalage de réplication (lag)
SELECT
    subname,
    pg_size_pretty(
        pg_wal_lsn_diff(sent_lsn, write_lsn)
    ) AS replication_lag
FROM pg_stat_subscription;

-- Slots de réplication actifs et leur consommation
SELECT
    slot_name,
    database,
    active,
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)
    ) AS retained_wal
FROM pg_replication_slots  
WHERE slot_type = 'logical';  
```

## Cas d'Usage Typiques

### 1. Migration de Base de Données en Douceur

**Scénario** : Vous devez migrer votre base de production vers un nouveau serveur sans interruption de service.

**Solution** :
- Créer une publication sur l'ancien serveur
- Créer une subscription sur le nouveau serveur
- Laisser la réplication se synchroniser
- Basculer les applications vers le nouveau serveur
- Supprimer la réplication une fois la migration validée

**Avantage** : Zéro downtime, possibilité de tester le nouveau serveur avant la bascule complète.

### 2. Réplication Sélective entre Environnements

**Scénario** : Vous voulez répliquer uniquement les données de référence (produits, catégories) de production vers développement, mais pas les données sensibles (utilisateurs, commandes).

**Solution** :
```sql
-- Sur production
CREATE PUBLICATION pub_referentiel  
FOR TABLE produits, categories, fournisseurs;  

-- Sur développement
CREATE SUBSCRIPTION sub_referentiel  
CONNECTION 'host=prod.example.com dbname=production user=repl'  
PUBLICATION pub_referentiel;  
```

**Avantage** : Les développeurs ont des données réalistes sans accès aux données sensibles.

### 3. Agrégation Multi-Sources

**Scénario** : Vous avez plusieurs bases PostgreSQL régionales et vous voulez agréger certaines données dans une base centrale pour l'analytics.

**Solution** :
- Chaque région a une publication pour les données à partager
- Le serveur central a une subscription pour chaque région
- Les tables destinations peuvent avoir des noms différents

**Avantage** : Centralisation des données pour reporting sans impacter les bases régionales.

### 4. Réplication Géographique Partielle

**Scénario** : Vous avez des bureaux dans plusieurs pays, chaque pays doit avoir une copie locale des données qui le concernent.

**Solution** :
```sql
-- Sur le serveur central
CREATE PUBLICATION pub_france  
FOR TABLE commandes  
WHERE (pays = 'France');  

CREATE PUBLICATION pub_allemagne  
FOR TABLE commandes  
WHERE (pays = 'Allemagne');  

-- Sur chaque serveur régional
CREATE SUBSCRIPTION sub_local  
CONNECTION 'host=central.example.com ...'  
PUBLICATION pub_france;  -- ou pub_allemagne selon la région  
```

**Avantage** : Réduit la latence et améliore la performance locale.

## Différences avec la Réplication Physique

| Aspect | Réplication Logique | Réplication Physique |
|--------|---------------------|----------------------|
| **Granularité** | Table par table, colonnes filtrables | Instance complète |
| **Versions PostgreSQL** | Peut répliquer entre versions différentes | Même version majeure requise |
| **Direction** | Unidirectionnelle, mais multi-sources possibles | Unidirectionnelle stricte |
| **Écriture sur replica** | ✅ Possible sur les tables non répliquées | ❌ Lecture seule stricte |
| **Schéma** | Doit être créé manuellement | Copié automatiquement |
| **DDL** | ❌ Non répliqué | ✅ Répliqué automatiquement |
| **Overhead** | Plus élevé (décodage logique) | Plus faible |
| **Flexibilité** | ⭐⭐⭐⭐⭐ | ⭐⭐ |

## Limitations et Considérations

### Ce qui N'est PAS Répliqué

❌ **Les modifications de schéma (DDL)** :  
- `ALTER TABLE`, `CREATE INDEX`, etc. ne sont pas répliqués
- Vous devez les appliquer manuellement sur le subscriber

❌ **Les séquences** :
- Les valeurs de séquences ne sont pas synchronisées automatiquement
- Risque de conflits si le subscriber génère aussi des ID

❌ **Les Large Objects** :
- Les BLOB stockés via `pg_largeobject` ne sont pas supportés

❌ **TRUNCATE** :
- Par défaut non répliqué (sauf si `publish = 'truncate'`)

### Gestion des Conflits

Si une même ligne est modifiée sur les deux serveurs (dans des subscriptions bidirectionnelles), PostgreSQL applique la **dernière écriture gagne** par défaut, mais :
- Pas de résolution automatique de conflits sophistiquée
- Il faut concevoir l'architecture pour éviter les écritures concurrentes
- Utiliser des triggers ou logique applicative si nécessaire

### Contraintes d'Intégrité

- Les **clés primaires sont obligatoires** sur les tables répliquées
- Sans PK, PostgreSQL ne peut pas identifier de manière unique les lignes à mettre à jour ou supprimer
- Les contraintes de clés étrangères peuvent causer des erreurs si l'ordre d'application n'est pas respecté

### Performance

- Le décodage logique consomme du **CPU** sur le publisher
- La taille du **WAL** augmente avec le volume de réplication
- Le délai de réplication dépend de la bande passante réseau et du volume de transactions

## Bonnes Pratiques

### ✅ DO (À Faire)

1. **Toujours avoir une clé primaire** sur les tables répliquées  
2. **Surveiller les slots de réplication** pour éviter la saturation disque  
3. **Tester la réplication** en environnement de staging avant production  
4. **Documenter** quelles tables sont répliquées et pourquoi  
5. **Utiliser `copy_data = false`** si les données existent déjà côté subscriber  
6. **Monitorer le lag de réplication** avec `pg_stat_subscription`  
7. **Planifier les changements de schéma** et les appliquer manuellement des deux côtés

### ❌ DON'T (À Éviter)

1. **Ne pas oublier de supprimer** les subscriptions et slots inutilisés  
2. **Ne pas répliquer** les tables système ou temporaires  
3. **Ne pas compter** sur la réplication des séquences  
4. **Ne pas ignorer** les erreurs dans les logs de réplication  
5. **Ne pas dépasser** la capacité réseau avec trop de publications volumineuses  
6. **Ne pas créer** de boucles de réplication (A → B → A)

## Exemple Complet : Mise en Place Étape par Étape

### Contexte
- **Serveur source** : `prod.example.com` (base : `ecommerce`)  
- **Serveur destination** : `analytics.example.com` (base : `reporting`)  
- **Objectif** : Répliquer les tables `commandes` et `produits`

### Étape 1 : Configuration du Serveur Source

```sql
-- 1.1. Se connecter au serveur source
psql -h prod.example.com -U postgres -d ecommerce

-- 1.2. Vérifier la configuration WAL
SHOW wal_level;  -- Doit être 'logical'

-- Si ce n'est pas le cas, éditer postgresql.conf et redémarrer
-- wal_level = logical
-- max_replication_slots = 10
-- max_wal_senders = 10

-- 1.3. Créer un utilisateur pour la réplication
CREATE ROLE replicateur WITH LOGIN PASSWORD 'motdepasse_securise' REPLICATION;

-- 1.4. Donner les permissions nécessaires
GRANT SELECT ON commandes TO replicateur;  
GRANT SELECT ON produits TO replicateur;  

-- 1.5. Créer la publication
CREATE PUBLICATION pub_ecommerce  
FOR TABLE commandes, produits;  

-- 1.6. Vérifier la création
SELECT * FROM pg_publication WHERE pubname = 'pub_ecommerce';  
SELECT * FROM pg_publication_tables WHERE pubname = 'pub_ecommerce';  
```

### Étape 2 : Configuration du Serveur Destination

```sql
-- 2.1. Se connecter au serveur destination
psql -h analytics.example.com -U postgres -d reporting

-- 2.2. Créer les tables avec la même structure
CREATE TABLE commandes (
    id SERIAL PRIMARY KEY,
    client_id INT NOT NULL,
    date_commande TIMESTAMP NOT NULL,
    montant NUMERIC(10,2) NOT NULL
);

CREATE TABLE produits (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200) NOT NULL,
    prix NUMERIC(10,2) NOT NULL,
    stock INT DEFAULT 0
);

-- 2.3. Créer la subscription
CREATE SUBSCRIPTION sub_ecommerce  
CONNECTION 'host=prod.example.com port=5432 dbname=ecommerce user=replicateur password=motdepasse_securise'  
PUBLICATION pub_ecommerce;  

-- 2.4. Vérifier la création et l'état
SELECT * FROM pg_subscription WHERE subname = 'sub_ecommerce';  
SELECT * FROM pg_stat_subscription WHERE subname = 'sub_ecommerce';  
```

### Étape 3 : Vérification et Monitoring

```sql
-- 3.1. Sur le serveur source : vérifier le slot de réplication
SELECT * FROM pg_replication_slots;

-- 3.2. Sur le serveur destination : vérifier l'état de réplication
SELECT
    subname,
    received_lsn,
    last_msg_receipt_time,
    latest_end_time
FROM pg_stat_subscription;

-- 3.3. Tester : insérer des données sur le source
INSERT INTO commandes (client_id, date_commande, montant)  
VALUES (123, NOW(), 99.99);  

-- 3.4. Vérifier sur la destination (après quelques secondes)
SELECT * FROM commandes ORDER BY id DESC LIMIT 5;
```

## Nouveautés PostgreSQL 18 pour la Réplication Logique

PostgreSQL 18 (septembre 2025) apporte quelques améliorations notables :

### 1. Meilleure Performance du Décodeur Logique
- Optimisation du processus de décodage WAL
- Réduction de la latence de réplication

### 2. Support Amélioré des Opérations DDL
- Bien que toujours non automatique, PostgreSQL 18 facilite la synchronisation des schémas
- Nouvelles fonctions système pour comparer les structures

### 3. Statistiques Étendues
- Nouvelles colonnes dans `pg_stat_subscription` pour un monitoring plus fin
- Meilleure visibilité sur les erreurs et retards

### 4. Gestion Améliorée des Conflits
- Nouvelles options pour gérer les conflits de manière plus granulaire
- Logs plus détaillés en cas d'erreur de réplication

## Résumé des Points Clés

- 📌 **Publications** = Quelles données partager (côté source)  
- 📌 **Subscriptions** = Recevoir ces données (côté destination)  
- 📌 **Flexibilité** = Réplication sélective, filtrage, multi-versions  
- 📌 **Asynchrone** = Léger délai de réplication (quasi temps-réel)  
- 📌 **PK obligatoires** = Identifiant unique nécessaire  
- 📌 **DDL manuel** = Les changements de schéma ne sont pas automatiques  
- 📌 **Monitoring** = Surveiller les slots et le lag de réplication

## Conclusion

La réplication logique via Publications et Subscriptions est un outil puissant et flexible pour :
- Migrer des bases de données sans downtime
- Répliquer sélectivement des données
- Agréger des informations depuis plusieurs sources
- Créer des architectures distribuées sophistiquées

Contrairement à la réplication physique "tout ou rien", la réplication logique vous donne un contrôle fin sur **quoi**, **comment** et **où** vos données sont répliquées. C'est un choix idéal pour les architectures modernes nécessitant de la flexibilité.
