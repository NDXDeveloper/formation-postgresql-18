# 17.5.2. Patroni : HA automatisé avec consensus (etcd, Consul)

## Introduction

Dans la section précédente (17.5.1), nous avons vu comment promouvoir manuellement un Standby avec `pg_ctl promote`. Cette approche fonctionne bien pour des environnements simples ou lors de maintenances planifiées. Cependant, elle présente des limites importantes :

- **Intervention humaine requise** : Quelqu'un doit détecter la panne et lancer la promotion  
- **Temps de réaction lent** : Plusieurs minutes peuvent s'écouler avant l'intervention  
- **Risque d'erreur** : Un administrateur peut faire une mauvaise manipulation  
- **Disponibilité nocturne** : Qui intervient à 3h du matin ?  
- **Split-brain possible** : Sans coordination, deux Primary peuvent coexister

**Patroni** résout tous ces problèmes en automatisant complètement le processus de haute disponibilité.

---

## Qu'est-ce que Patroni ?

### Définition

**Patroni** est un outil open-source développé par **Zalando** (la plateforme e-commerce européenne) pour gérer automatiquement la haute disponibilité de PostgreSQL.

**En résumé** : Patroni transforme votre cluster PostgreSQL en un système auto-réparant capable de détecter les pannes et de basculer automatiquement vers un Standby sans intervention humaine.

### Philosophie de Patroni

Patroni repose sur trois principes fondamentaux :

1. **Consensus distribué** : Utilisation d'un système de consensus (etcd, Consul, ZooKeeper) pour décider collectivement qui est le Primary  
2. **Heartbeat constant** : Surveillance continue de la santé des serveurs  
3. **Élection automatique** : En cas de panne du Primary, élection automatique d'un nouveau leader

### Analogie Simple

Imaginez une équipe de footballeurs :
- **Sans Patroni** : Si le capitaine tombe, personne ne sait qui doit prendre le bras de capitaine. Le jeu s'arrête.  
- **Avec Patroni** : Si le capitaine tombe, l'équipe vote immédiatement pour élire un nouveau capitaine. Le jeu continue sans interruption.

## Architecture Patroni : Vue d'Ensemble

### Les Composants

Un cluster Patroni est composé de :

```
┌─────────────────────────────────────────────────────┐
│              Système de Consensus                   │
│         (etcd, Consul, ou ZooKeeper)                │
│                                                     │
│  Stocke : Qui est le Primary ? Quelle est la        │
│           configuration du cluster ?                │
└─────────────────────────────────────────────────────┘
                        ▲
                        │ Heartbeat / Lock
                        │
        ┌───────────────┼───────────────┐
        │               │               │
        ▼               ▼               ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│  Patroni 1   │ │  Patroni 2   │ │  Patroni 3   │
│  (Primary)   │ │  (Standby)   │ │  (Standby)   │
│              │ │              │ │              │
│ PostgreSQL   │ │ PostgreSQL   │ │ PostgreSQL   │
│   Instance   │ │   Instance   │ │   Instance   │
└──────────────┘ └──────────────┘ └──────────────┘
```

### Flux de Fonctionnement

1. **Chaque instance PostgreSQL** a un agent Patroni qui la supervise  
2. **Chaque agent Patroni** communique avec le système de consensus  
3. **Le système de consensus** stocke l'état du cluster (qui est Primary, configuration, etc.)  
4. **Un seul Patroni** détient le "leader lock" (verrou de leadership) et son PostgreSQL est le Primary  
5. **En cas de panne** du Primary, les autres Patroni se battent pour obtenir le lock et promouvoir leur PostgreSQL

## Le Concept de Consensus Distribué

### Qu'est-ce que le Consensus ?

Le **consensus distribué** est un mécanisme qui permet à plusieurs ordinateurs de s'accorder sur une vérité commune, même en cas de pannes partielles.

**Exemple concret** :
- 3 serveurs doivent décider : "Qui est le Primary ?"
- Serveur A dit : "C'est moi !"
- Serveur B dit : "Non, c'est moi !"
- Serveur C dit : "Je vote pour A"

**Résultat** : Le consensus est atteint. A est le Primary parce qu'il a le support de C (majorité 2/3).

### Pourquoi le Consensus est Indispensable ?

Sans consensus, vous pourriez avoir :
- **Split-brain** : Deux serveurs se proclament Primary en même temps  
- **Incohérence** : Les serveurs ne savent pas qui est le Primary  
- **Perte de données** : Écritures concurrentes sur deux Primary

**Avec le consensus** :
- **Une seule vérité** : Tous les serveurs s'accordent sur qui est le Primary  
- **Élection automatique** : En cas de panne, un nouveau Primary est élu démocratiquement  
- **Protection contre le split-brain** : Impossible d'avoir deux Primary

### Le Concept de Quorum

Le **quorum** est le nombre minimum de serveurs nécessaires pour prendre une décision.

**Règle générale** : Quorum = (N / 2) + 1

| Nombre de serveurs | Quorum requis | Pannes tolérées |
|--------------------|---------------|-----------------|
| 3                  | 2             | 1               |
| 5                  | 3             | 2               |
| 7                  | 4             | 3               |

**Exemple avec 3 serveurs** :
- Si 2 serveurs sont actifs → Décision possible (quorum atteint)
- Si 1 seul serveur est actif → Pas de décision (quorum non atteint)

**Important** : Utilisez toujours un **nombre impair** de serveurs de consensus pour éviter les situations de "vote égal".

## Les Systèmes de Consensus : etcd, Consul, ZooKeeper

Patroni supporte trois systèmes de consensus. Voici leur présentation.

### 1. etcd (Recommandé)

**etcd** est un système de stockage clé-valeur distribué développé par CoreOS (maintenant Red Hat).

**Caractéristiques** :
- ✅ **Simplicité** : Facile à installer et configurer  
- ✅ **Performance** : Très rapide pour les opérations de lecture/écriture  
- ✅ **Intégration Kubernetes** : Utilisé nativement par Kubernetes  
- ✅ **API HTTP** : Interface REST simple  
- ⚠️ **Mémoire** : Consomme plus de RAM que Consul

**Quand utiliser etcd ?**
- Vous déployez dans Kubernetes
- Vous voulez la solution la plus simple
- Vous n'avez pas besoin de Service Discovery avancé

**Architecture typique** :
```
┌─────────┐  ┌─────────┐  ┌─────────┐
│  etcd 1 │  │  etcd 2 │  │  etcd 3 │
│ (Leader)│◄─┤         │◄─┤         │
└─────────┘  └─────────┘  └─────────┘
     ▲            ▲            ▲
     └────────────┴────────────┘
              Patroni
```

### 2. Consul

**Consul** est un système de Service Discovery et de consensus développé par HashiCorp.

**Caractéristiques** :
- ✅ **Service Discovery** : Découverte automatique des services  
- ✅ **Health Checks** : Vérifications de santé intégrées  
- ✅ **Multi-datacenter** : Support natif du multi-DC  
- ✅ **UI Web** : Interface graphique pour visualiser le cluster  
- ⚠️ **Complexité** : Plus complexe à configurer qu'etcd

**Quand utiliser Consul ?**
- Vous avez déjà une infrastructure HashiCorp (Vault, Nomad)
- Vous avez besoin de Service Discovery
- Vous gérez plusieurs datacenters

**Architecture typique** :
```
┌──────────┐  ┌──────────┐  ┌──────────┐
│ Consul 1 │  │ Consul 2 │  │ Consul 3 │
│ (Leader) │◄─┤          │◄─┤          │
└──────────┘  └──────────┘  └──────────┘
     ▲             ▲             ▲
     └─────────────┴─────────────┘
               Patroni
```

### 3. ZooKeeper

**ZooKeeper** est un système de consensus développé par Apache, historiquement utilisé par Hadoop.

**Caractéristiques** :
- ✅ **Mature** : Existe depuis 2008, très stable  
- ✅ **Éprouvé** : Utilisé massivement dans le Big Data (Kafka, Hadoop)  
- ⚠️ **Complexité** : Configuration complexe  
- ⚠️ **Java** : Requiert la JVM (overhead mémoire)  
- ⚠️ **Moins moderne** : Moins actif que etcd/Consul

**Quand utiliser ZooKeeper ?**
- Vous avez déjà ZooKeeper dans votre infrastructure (Kafka, HBase)
- Vous voulez la solution la plus mature

**Note** : etcd et Consul sont généralement préférés pour les nouvelles installations.

### Comparaison Rapide

| Critère | etcd | Consul | ZooKeeper |
|---------|------|--------|-----------|
| **Simplicité** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ |
| **Performance** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **Fonctionnalités** | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| **Popularité (2025)** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| **Kubernetes** | Natif | Plugin | Plugin |

**Recommandation** : Utilisez **etcd** pour la simplicité, **Consul** pour les fonctionnalités avancées.

## Comment Patroni Fonctionne : Le Cycle de Vie

### 1. Démarrage Initial (Bootstrap)

**Scénario** : Vous démarrez Patroni pour la première fois.

```
1. Patroni démarre sur les 3 serveurs
2. Chaque Patroni contacte etcd/Consul
3. Patroni constate qu'aucun cluster n'existe
4. Le premier Patroni à obtenir le "initialize lock" :
   - Initialise PostgreSQL (initdb)
   - Se déclare Primary
   - Enregistre cette information dans etcd/Consul
5. Les autres Patroni :
   - Voient qu'un Primary existe
   - Se configurent automatiquement en Standby
   - Démarrent la réplication depuis le Primary
```

**Résultat** : Un cluster PostgreSQL HA est créé automatiquement !

### 2. Fonctionnement Normal (Heartbeat)

**Scénario** : Le cluster fonctionne normalement.

```
┌─────────────────────────────────────────┐
│ Toutes les 10 secondes (par défaut) :   │
└─────────────────────────────────────────┘
                   │
                   ▼
        ┌──────────────────────┐
        │ Patroni Primary :    │
        │ 1. Vérifie que PG    │
        │    fonctionne        │
        │ 2. Renouvelle le     │
        │    leader lock       │
        │ 3. Met à jour les    │
        │    métriques         │
        └──────────────────────┘
                   │
                   ▼
        ┌──────────────────────┐
        │ Patroni Standby :    │
        │ 1. Vérifie que PG    │
        │    fonctionne        │
        │ 2. Vérifie la        │
        │    réplication       │
        │ 3. Surveille le      │
        │    leader lock       │
        └──────────────────────┘
```

**Concept clé : Le Leader Lock**

Le Primary détient un **verrou temporel** (TTL = Time To Live) dans etcd/Consul :
- Le Primary renouvelle ce verrou toutes les 10 secondes
- Si le verrou expire (Primary en panne), il devient disponible
- Les Standby surveillent ce verrou et tentent de l'acquérir s'il expire

### 3. Détection de Panne

**Scénario** : Le Primary tombe en panne.

```
T+0s  : Primary fonctionne normalement
        Leader lock TTL = 30 secondes

T+10s : Primary renouvelle le lock
        Leader lock TTL = 30 secondes

T+15s : Primary tombe en panne ❌
        (ne peut plus renouveler le lock)

T+30s : Leader lock expire dans etcd/Consul
        Les Standby détectent que le lock est disponible
```

**Que surveille Patroni ?**

Sur le Primary :
- PostgreSQL répond-il aux connexions ?
- Les processus PostgreSQL sont-ils actifs ?
- Le disque est-il en lecture/écriture ?

Sur les Standby :
- PostgreSQL répond-il ?
- La réplication est-elle active ?
- Le retard de réplication est-il acceptable ?

### 4. Élection Automatique (Failover)

**Scénario** : Le leader lock a expiré. Il faut élire un nouveau Primary.

```
┌─────────────────────────────────────────────┐
│ Phase 1 : Course au Lock                    │
└─────────────────────────────────────────────┘
        │
        ▼
Standby 1 : "Je veux être Primary !"  
Standby 2 : "Moi aussi !"  
        │
        ▼
etcd/Consul : "Le premier qui m'envoie une
               requête obtient le lock"
        │
        ▼
┌─────────────────────────────────────────────┐
│ Phase 2 : Sélection du Meilleur Candidat    │
└─────────────────────────────────────────────┘
        │
        ▼
Patroni compare les Standby selon :
1. Timeline PostgreSQL (plus récente = mieux)
2. LSN (Log Sequence Number - plus haut = mieux)
3. Score de réplication (lag, état)
        │
        ▼
Le meilleur candidat obtient le lock
        │
        ▼
┌─────────────────────────────────────────────┐
│ Phase 3 : Promotion                         │
└─────────────────────────────────────────────┘
        │
        ▼
1. Patroni exécute "pg_ctl promote"
2. PostgreSQL passe en mode lecture-écriture
3. Patroni enregistre le nouveau Primary dans etcd
4. Les autres Standby se reconfigurent automatiquement
```

**Durée typique** : 15-30 secondes (configurable)

### 5. Retour à la Normale

**Scénario** : L'ancien Primary revient en ligne.

```
1. L'ancien Primary démarre
2. Patroni détecte qu'il était Primary avant
3. Patroni interroge etcd/Consul
4. etcd répond : "Un nouveau Primary existe déjà"
5. Patroni reconfigure automatiquement l'ancien Primary en Standby
6. L'ancien Primary se synchronise avec le nouveau Primary
7. Le cluster a maintenant 3 serveurs opérationnels
```

**Important** : Patroni empêche automatiquement le split-brain !

## Algorithme de Sélection du Leader

### Critères de Sélection

Lorsqu'il faut élire un nouveau Primary, Patroni évalue chaque Standby selon :

#### 1. Timeline PostgreSQL

PostgreSQL utilise des **timelines** pour suivre l'historique des promotions.
- Timeline 1 : Le Primary original
- Timeline 2 : Après une première promotion
- Timeline 3 : Après une deuxième promotion

**Règle** : On choisit le Standby avec la timeline la plus élevée.

#### 2. LSN (Log Sequence Number)

Le **LSN** représente la position dans les WAL.
- Plus le LSN est élevé, plus le Standby a de données récentes

**Exemple** :
- Standby A : LSN = 0/3000000
- Standby B : LSN = 0/2FFFFFF

**Gagnant** : Standby A (plus de données)

#### 3. État de Réplication

Patroni vérifie :
- Le Standby est-il en état "streaming" ?
- Le Standby a-t-il un retard de réplication ?
- Le Standby est-il "in sync" (réplication synchrone) ?

#### 4. Scores Personnalisés

Vous pouvez configurer des **priorités** manuelles :

```yaml
# Dans la configuration Patroni
tags:
  nofailover: false     # Ce serveur peut devenir Primary
  noloadbalance: false  # Ce serveur peut recevoir des lectures
  clonefrom: true       # Ce serveur peut servir de source pour les clones
  priority: 100         # Score manuel (plus haut = prioritaire)
```

**Exemple** :
- Standby A : priority = 100
- Standby B : priority = 50

→ Standby A sera préféré (toutes choses égales par ailleurs)

### Cas Particuliers

**Tous les Standby sont en retard ?**
- Patroni choisit celui avec le plus petit retard
- Si le retard est trop important (configurable), Patroni peut refuser la promotion

**Aucun Standby disponible ?**
- Patroni attend qu'un Standby devienne disponible
- Pas de Primary = Cluster en lecture seule

## Configuration de Patroni

### Structure du Fichier de Configuration

Patroni utilise un fichier YAML (par défaut : `/etc/patroni/patroni.yml`)

Voici une configuration simplifiée pour comprendre les concepts :

```yaml
# ============================================
# Section 1 : Identification du Nœud
# ============================================
scope: postgres-cluster    # Nom du cluster  
namespace: /service/       # Namespace dans etcd  
name: postgres1            # Nom unique de ce nœud  

# ============================================
# Section 2 : API REST de Patroni
# ============================================
restapi:
  listen: 0.0.0.0:8008     # Port pour l'API Patroni
  connect_address: 192.168.1.10:8008

# ============================================
# Section 3 : Connexion au Système de Consensus
# ============================================
etcd:
  hosts:
    - 192.168.1.11:2379    # etcd serveur 1
    - 192.168.1.12:2379    # etcd serveur 2
    - 192.168.1.13:2379    # etcd serveur 3

# ============================================
# Section 4 : Configuration Bootstrap
# ============================================
bootstrap:
  dcs:
    ttl: 30                      # Durée du leader lock
    loop_wait: 10                # Intervalle de heartbeat
    retry_timeout: 10            # Timeout de retry
    maximum_lag_on_failover: 1048576  # Lag max pour être éligible (1MB)

    postgresql:
      use_pg_rewind: true        # Utiliser pg_rewind pour resynchroniser
      parameters:
        max_connections: 100
        shared_buffers: 256MB

  initdb:
    - encoding: UTF8
    - data-checksums             # Activer les checksums (PostgreSQL 18 par défaut)

  pg_hba:
    - host replication replicator 0.0.0.0/0 md5
    - host all all 0.0.0.0/0 md5

# ============================================
# Section 5 : Configuration PostgreSQL
# ============================================
postgresql:
  listen: 0.0.0.0:5432
  connect_address: 192.168.1.10:5432
  data_dir: /var/lib/postgresql/14/main
  bin_dir: /usr/lib/postgresql/14/bin

  authentication:
    replication:
      username: replicator
      password: secret123
    superuser:
      username: postgres
      password: admin123

  parameters:
    unix_socket_directories: '/var/run/postgresql'

# ============================================
# Section 6 : Tags et Priorités
# ============================================
tags:
  nofailover: false
  noloadbalance: false
  clonefrom: true
  priority: 100
```

### Explication des Paramètres Clés

#### 1. `scope`
Le nom logique de votre cluster. Tous les nœuds du même cluster doivent avoir le même `scope`.

#### 2. `ttl` (Time To Live)
Durée de validité du leader lock.
- **Valeur par défaut** : 30 secondes  
- **Trop court** : Risque de faux positifs (failover inutiles)  
- **Trop long** : Temps de détection de panne rallongé

**Recommandation** : 30 secondes pour la production

#### 3. `loop_wait`
Intervalle entre chaque vérification de santé.
- **Valeur par défaut** : 10 secondes  
- **Impact** : Temps de détection de panne = `ttl` - `loop_wait`

#### 4. `maximum_lag_on_failover`
Retard de réplication maximum accepté pour qu'un Standby soit éligible comme Primary.
- **Valeur par défaut** : 1 MB  
- **Utilité** : Empêche la promotion d'un Standby très en retard

#### 5. `use_pg_rewind`
Active l'utilisation de `pg_rewind` pour resynchroniser un ancien Primary sans reconstruction complète.
- **Valeur recommandée** : `true`  
- **Avantage** : Réintégration rapide de l'ancien Primary

## Patroni en Action : Exemples de Scénarios

### Scénario 1 : Failover Automatique

**Contexte** : Le Primary tombe en panne (crash système, perte réseau)

```
T+0s   : Primary actif, 2 Standby en réplication  
T+15s  : Primary crash ❌  
T+15s  : Patroni du Primary ne peut plus renouveler le lock  
T+30s  : Leader lock expire dans etcd  
T+31s  : Standby 1 détecte l'expiration  
T+32s  : Standby 1 et 2 tentent d'acquérir le lock  
T+33s  : Standby 1 obtient le lock (meilleur LSN)  
T+34s  : Patroni exécute "pg_ctl promote" sur Standby 1  
T+40s  : PostgreSQL Standby 1 devient Primary ✅  
T+41s  : Standby 2 détecte le nouveau Primary  
T+42s  : Standby 2 se reconfigure automatiquement  
T+45s  : Cluster opérationnel avec 1 Primary + 1 Standby  
```

**RTO (Recovery Time Objective)** : ~30 secondes

**RPO (Recovery Point Objective)** :
- Réplication asynchrone : Quelques transactions perdues
- Réplication synchrone : 0 perte

### Scénario 2 : Maintenance Planifiée (Switchover)

**Contexte** : Vous voulez faire une maintenance sur le Primary actuel

**Commande Patroni** :
```bash
# Basculer manuellement vers le meilleur Standby
patronictl switchover postgres-cluster

# Patroni demande :
# Master [postgres1]:
# Candidate [postgres2]: postgres2
# When should the switchover take place (e.g. 2024-11-22T10:00 )  [now]:
```

**Ce qui se passe** :
1. Patroni met le Primary actuel en mode lecture seule  
2. Attend que les Standby soient à jour  
3. Promeut le Standby cible  
4. Reconfigure l'ancien Primary en Standby  
5. Tout ça en **0 perte de données**

**RTO** : ~5-10 secondes (configurable avec `--scheduled`)

### Scénario 3 : Retour du Primary Original

**Contexte** : L'ancien Primary (qui avait crashé) redémarre

```
T+0m   : Ancien Primary démarre  
T+0m   : Patroni démarre et interroge etcd  
T+0m   : etcd répond : "postgres2 est le Primary maintenant"  
T+1m   : Patroni compare les timelines  
T+1m   : Timeline ancien Primary = 1, Timeline nouveau Primary = 2  
T+2m   : Patroni exécute "pg_rewind" pour resynchroniser  
T+5m   : Ancien Primary devient Standby ✅  
T+5m   : Cluster a maintenant 1 Primary + 2 Standby  
```

**Important** : Aucune intervention manuelle nécessaire !

### Scénario 4 : Split-Brain Prévention

**Contexte** : Perte réseau temporaire (network partition)

```
Situation :
- Primary perd la connexion à etcd (problème réseau)
- Les Standby perdent aussi la connexion au Primary
- Mais les Standby peuvent toujours accéder à etcd

Que fait Patroni ?

Sur le Primary :
1. Patroni ne peut plus renouveler le leader lock
2. Au bout de TTL secondes, Patroni détecte la perte du lock
3. Patroni met automatiquement PostgreSQL en PAUSE (lecture seule)
4. Patroni attend de retrouver la connexion à etcd

Sur les Standby :
1. Standby détectent l'expiration du leader lock
2. Un Standby acquiert le lock
3. Ce Standby est promu en Primary

Résultat :
✅ Un seul Primary à tout moment
✅ Pas de split-brain
✅ L'ancien Primary ne peut plus accepter d'écritures
```

**Mécanisme de protection** : **Fencing automatique**

## Commandes Patroni (patronictl)

Patroni fournit l'utilitaire `patronictl` pour interagir avec le cluster.

### Voir l'État du Cluster

```bash
patronictl -c /etc/patroni/patroni.yml list postgres-cluster

# Sortie :
+ Cluster: postgres-cluster ------+----+-----------+
| Member    | Host         | Role   | State     | TL | Lag in MB |
+-----------+--------------+--------+-----------+----+-----------+
| postgres1 | 192.168.1.10 | Leader | running   |  2 |           |
| postgres2 | 192.168.1.11 | Replica| streaming |  2 |         0 |
| postgres3 | 192.168.1.12 | Replica| streaming |  2 |         0 |
+-----------+--------------+--------+-----------+----+-----------+
```

**Interprétation** :
- **Leader** : Le Primary actuel  
- **Replica** : Les Standby  
- **TL** : Timeline PostgreSQL  
- **Lag in MB** : Retard de réplication

### Basculer Manuellement (Switchover)

```bash
# Switchover planifié vers postgres2
patronictl switchover postgres-cluster --master postgres1 --candidate postgres2
```

### Forcer un Failover (Urgence)

```bash
# En cas d'urgence, forcer le failover immédiatement
patronictl failover postgres-cluster --candidate postgres2 --force
```

### Redémarrer un Nœud

```bash
# Redémarrer PostgreSQL via Patroni
patronictl restart postgres-cluster postgres1
```

### Réintégrer un Nœud (Reinitialize)

```bash
# Si un nœud est désynchronisé, le reconstruire
patronictl reinit postgres-cluster postgres1
```

### Mettre en Pause (Maintenance Mode)

```bash
# Désactiver temporairement l'automatisation Patroni
patronictl pause postgres-cluster

# Le cluster reste en l'état actuel, mais Patroni n'intervient plus
```

```bash
# Réactiver l'automatisation
patronictl resume postgres-cluster
```

### Voir la Configuration DCS

```bash
# Voir la configuration stockée dans etcd/Consul
patronictl show-config postgres-cluster
```

## Patroni et HAProxy : Load Balancing

Patroni seul ne gère pas la redirection des connexions applicatives. Vous devez utiliser un **proxy** ou un **load balancer**.

### Architecture avec HAProxy

```
    Applications
         │
         ▼
    ┌──────────┐
    │ HAProxy  │
    │ Port 5000│ → Connexions WRITE → Primary (port 5432)
    │ Port 5001│ → Connexions READ  → Standby 1 ou 2 (port 5432)
    └──────────┘
         │
    ┌────┴────┬────────────┐
    ▼         ▼            ▼
postgres1  postgres2   postgres3
(Primary)  (Standby)   (Standby)
```

### Configuration HAProxy

HAProxy interroge l'**API REST de Patroni** pour savoir qui est le Primary :

```
# /etc/haproxy/haproxy.cfg

frontend postgres_write
    bind *:5000
    default_backend postgres_primary

backend postgres_primary
    option httpchk GET /primary   # Patroni REST API endpoint
    http-check expect status 200
    server postgres1 192.168.1.10:5432 check port 8008
    server postgres2 192.168.1.11:5432 check port 8008
    server postgres3 192.168.1.12:5432 check port 8008

frontend postgres_read
    bind *:5001
    default_backend postgres_replicas

backend postgres_replicas
    option httpchk GET /replica   # Patroni REST API endpoint
    http-check expect status 200
    balance roundrobin
    server postgres2 192.168.1.11:5432 check port 8008
    server postgres3 192.168.1.12:5432 check port 8008
```

### Endpoints Patroni REST API

| Endpoint | Rôle | Retour |
|----------|------|--------|
| `/primary` | Vérifie si le nœud est Primary | 200 si Primary, 503 sinon |
| `/replica` | Vérifie si le nœud est Standby | 200 si Standby, 503 sinon |
| `/health` | Vérifie si PostgreSQL est en bonne santé | 200 si OK |
| `/read-only` | Vérifie si le nœud accepte les lectures | 200 si OK |

**Avantage** : HAProxy bascule automatiquement les connexions après un failover !

## Avantages et Limites de Patroni

### ✅ Avantages

**1. Automatisation Complète**
- Pas d'intervention humaine requise 24/7
- Failover en 15-30 secondes

**2. Prévention du Split-Brain**
- Protection garantie par le consensus
- Fencing automatique

**3. Flexibilité**
- Switchover planifié sans perte de données
- Mode pause pour maintenance

**4. Observabilité**
- API REST pour monitoring
- Intégration facile avec Prometheus/Grafana

**5. Écosystème Mature**
- Utilisé en production par des milliers d'entreprises
- Communauté active
- Compatible Kubernetes (Zalando Operator)

### ⚠️ Limites et Considérations

**1. Complexité Accrue**
- Nécessite d'installer et de maintenir etcd/Consul
- Plus de composants = plus de points de défaillance potentiels

**2. Dépendance au Système de Consensus**
- Si etcd/Consul tombe, Patroni ne peut plus fonctionner
- Important d'avoir un etcd HA (3+ nœuds)

**3. Latence Réseau**
- Le failover dépend de la latence réseau entre les nœuds
- Réseaux lents = temps de failover plus longs

**4. Courbe d'Apprentissage**
- Plus complexe qu'une simple réplication PostgreSQL
- Nécessite de comprendre les concepts de consensus

**5. Pas de Magie**
- Patroni ne résout pas les problèmes de performance PostgreSQL
- Patroni ne remplace pas les backups

| **Communauté** | Très active | Active | Moyenne |

### Quand Utiliser Chaque Solution ?

**Patroni** :
- ✅ Production critique (99.99% uptime)  
- ✅ Déploiement Kubernetes  
- ✅ Clusters complexes (5+ nœuds)  
- ✅ Multi-datacenter

**Repmgr** :
- ✅ Clusters simples (1 Primary + 2 Standby)  
- ✅ Équipes préférant la simplicité

**pg_auto_failover** :
- ✅ Alternative simple à Patroni

## Monitoring et Troubleshooting Patroni

### Métriques Clés à Surveiller

#### 1. État du Leader Lock

```bash
# Vérifier qui détient le leader lock
curl http://192.168.1.10:8008/patroni | jq .

# Réponse :
{
  "state": "running",
  "role": "master",
  "timeline": 2,
  "xlog": {
    "location": 67108864
  }
}
```

#### 2. Retard de Réplication

```bash
patronictl -c /etc/patroni/patroni.yml list postgres-cluster
```

**Alerte** si `Lag in MB` > 100 MB

#### 3. Nombre de Failovers

```bash
# Dans etcd, vérifier l'historique
etcdctl get /service/postgres-cluster/history --prefix
```

**Alerte** si failovers fréquents (> 1 par jour) → Problème sous-jacent

### Logs Patroni

Les logs Patroni sont essentiels pour diagnostiquer :

```bash
# Logs systemd
journalctl -u patroni -f

# Ou logs fichier
tail -f /var/log/patroni/patroni.log
```

**Événements importants à chercher** :
- `Lock owner`  
- `promoted self to leader`  
- `demoted self`  
- `no healthy members found`

### Problèmes Courants

#### Problème 1 : Flapping (Failover Répétés)

**Symptôme** : Le Primary change constamment

**Causes possibles** :
- TTL trop court
- Latence réseau élevée
- etcd/Consul surchargé

**Solution** :
```yaml
# Augmenter le TTL
bootstrap:
  dcs:
    ttl: 60  # Au lieu de 30
    loop_wait: 10
```

#### Problème 2 : Standby Ne Se Synchronise Pas

**Symptôme** : Un Standby reste désynchronisé

**Diagnostic** :
```bash
# Sur le Standby
patronictl reinit postgres-cluster postgres2
```

**Causes** : Corruption, pg_rewind échoué, manque d'espace disque

#### Problème 3 : etcd Inaccessible

**Symptôme** : Patroni ne peut plus contacter etcd

**Impact** :
- Le Primary se met automatiquement en pause (lecture seule)
- Pas de failover possible

**Solution** :
1. Vérifier la connexion réseau vers etcd  
2. Vérifier les logs etcd  
3. Redémarrer etcd si nécessaire

### Checklist de Déploiement

#### Phase 1 : Préparation

#### Phase 2 : Installation

- [ ] Tester la connexion à etcd

#### Phase 3 : Bootstrap

- [ ] Démarrer Patroni sur le premier nœud

- [ ] Démarrer Patroni sur les nœuds suivants  
- [ ] Vérifier la réplication automatique

#### Phase 4 : Load Balancer

## Résumé des Concepts Clés

### Ce qu'il Faut Retenir

**1. Patroni = HA Automatisé**
- Détection automatique de panne
- Élection automatique du nouveau Primary
- Prévention du split-brain

**2. Consensus Distribué**
- etcd/Consul/ZooKeeper : Un seul point de vérité
- Leader lock : Mécanisme de coordination
- Quorum : Décision à la majorité

**3. Cycle de Vie Patroni**
- Bootstrap → Fonctionnement normal → Panne → Failover → Réintégration

**4. RTO & RPO**
- **RTO** : 15-30 secondes (temps de failover)  
- **RPO** : 0 (synchrone) ou quelques transactions (asynchrone)

**5. Limites**
- Nécessite etcd/Consul (dépendance externe)
- Plus complexe qu'une simple réplication
- Nécessite une bonne architecture réseau
