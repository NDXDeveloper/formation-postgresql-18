# 17.5.3. Repmgr : Gestion de réplication simplifiée

## Introduction

Dans les sections précédentes, nous avons vu :
- **17.5.1** : La promotion manuelle avec `pg_ctl promote` (intervention humaine)  
- **17.5.2** : Patroni avec son automatisation complète (mais complexe avec etcd/Consul)

**Repmgr** se positionne entre ces deux approches : il simplifie grandement la gestion de la réplication PostgreSQL tout en offrant une automatisation partielle, sans nécessiter de système de consensus externe.

### Qu'est-ce que Repmgr ?

**Repmgr** (Replication Manager) est un outil open-source développé par **2ndQuadrant** (maintenant EDB) pour :
- Configurer et gérer la réplication PostgreSQL de manière simple
- Monitorer l'état des nœuds du cluster
- Faciliter le failover (automatique ou manuel)
- Cloner rapidement de nouveaux Standby

**Philosophie** : "Keep it simple" - Repmgr privilégie la simplicité sur l'automatisation totale.

---

## Repmgr vs Patroni vs Manuel : Quelle Différence ?

### Comparaison Rapide

| Aspect | Manuel (pg_ctl) | Repmgr | Patroni |
|--------|-----------------|--------|---------|
| **Complexité** | Simple | Moyenne | Élevée |
| **Infrastructure requise** | Aucune | Aucune (optionnel: witness) | etcd/Consul/ZK |
| **Failover automatique** | ❌ Non | ⚠️ Optionnel | ✅ Oui |
| **Split-brain protection** | ❌ Non | ⚠️ Partielle | ✅ Excellente |
| **Maintenance** | Faible | Faible | Moyenne |
| **Courbe d'apprentissage** | Courte | Courte | Longue |
| **Cas d'usage** | Dev/Test | Petits clusters | Production critique |

### Positionnement de Repmgr

**Repmgr est idéal pour** :
- ✅ Équipes qui veulent plus qu'une réplication manuelle  
- ✅ Clusters de petite à moyenne taille (2-5 nœuds)  
- ✅ Environnements où la simplicité prime  
- ✅ Budgets limités (pas d'infrastructure additionnelle)  
- ✅ Failover semi-automatique acceptable (avec intervention humaine)

**Repmgr n'est PAS adapté pour** :
- ❌ Production ultra-critique (99.99% uptime requis)  
- ❌ Clusters très complexes (10+ nœuds)  
- ❌ Failover automatique sans supervision humaine obligatoire  
- ❌ Architectures multi-datacenter complexes

## Architecture de Repmgr

### Vue d'Ensemble

Contrairement à Patroni qui nécessite etcd/Consul, Repmgr stocke toutes ses métadonnées **directement dans PostgreSQL**.

```
┌─────────────────────────────────────────────┐
│         Base de Données "repmgr"            │
│    (métadonnées du cluster stockées ici)    │
└─────────────────────────────────────────────┘
                     ▲
                     │ Lecture/Écriture
                     │
        ┌────────────┼────────────┐
        │            │            │
        ▼            ▼            ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│  repmgrd 1   │ │  repmgrd 2   │ │  repmgrd 3   │
│  (daemon)    │ │  (daemon)    │ │  (daemon)    │
│      │       │ │      │       │ │      │       │
│      ▼       │ │      ▼       │ │      ▼       │
│ PostgreSQL 1 │ │ PostgreSQL 2 │ │ PostgreSQL 3 │
│  (Primary)   │ │  (Standby)   │ │  (Standby)   │
└──────────────┘ └──────────────┘ └──────────────┘
```

### Composants de Repmgr

#### 1. L'Outil `repmgr`

C'est l'utilitaire en ligne de commande pour :
- Enregistrer les nœuds dans le cluster
- Cloner de nouveaux Standby
- Promouvoir manuellement un Standby
- Afficher l'état du cluster

**Analogie** : `repmgr` est comme `patronictl` pour Patroni.

#### 2. Le Daemon `repmgrd`

C'est un processus d'arrière-plan qui :
- Surveille l'état de PostgreSQL
- Détecte les pannes (heartbeat)
- Peut déclencher un failover automatique (si configuré)
- Envoie des notifications

**Analogie** : `repmgrd` est comme l'agent Patroni.

#### 3. La Base de Données `repmgr`

Une base PostgreSQL dédiée qui stocke :
- La liste des nœuds du cluster
- L'historique des événements (failover, switchover, etc.)
- La configuration du cluster

**Important** : Cette base est répliquée automatiquement avec le reste de PostgreSQL.

#### 4. Le Witness (Optionnel)

Un serveur léger qui sert d'**arbitre** en cas de perte de connexion réseau.

**Rôle** : Éviter les faux positifs et améliorer la détection des pannes.

## Concepts Fondamentaux de Repmgr

### 1. Enregistrement des Nœuds

Chaque nœud PostgreSQL doit être **enregistré** dans le cluster Repmgr.

**Types de nœuds** :
- **Primary** : Le serveur maître (lecture/écriture)  
- **Standby** : Les réplicas (lecture seule)  
- **Witness** : Arbitre sans données (optionnel)

**Registre** : La table `repmgr.nodes` contient la liste de tous les nœuds :

```sql
SELECT node_id, node_name, type, active  
FROM repmgr.nodes;  

 node_id | node_name  |   type   | active
---------+------------+----------+--------
       1 | postgres1  | primary  | t
       2 | postgres2  | standby  | t
       3 | postgres3  | standby  | t
       4 | witness1   | witness  | t
```

### 2. Node ID et Priorité

Chaque nœud a :
- **Node ID** : Identifiant unique (entier)  
- **Priorité** : Score pour l'élection (plus haut = préféré)

**Exemple** :
```ini
node_id=1  
node_name='postgres1'  
priority=100  
```

**Utilisation** : En cas de failover, Repmgr préfère le nœud avec la priorité la plus élevée.

### 3. Événements et Historique

Repmgr enregistre tous les événements importants dans `repmgr.events` :

```sql
SELECT event_timestamp, event, node_name, successful  
FROM repmgr.events  
ORDER BY event_timestamp DESC  
LIMIT 5;  

    event_timestamp     |        event        | node_name | successful
------------------------+---------------------+-----------+------------
 2025-11-22 10:15:32+00 | standby_promote     | postgres2 | t
 2025-11-22 10:15:20+00 | repmgrd_failover    | postgres2 | t
 2025-11-22 09:00:00+00 | standby_clone       | postgres3 | t
```

**Avantage** : Traçabilité complète de l'historique du cluster.

### 4. Le Témoin (Witness Node)

Le **witness** est un serveur PostgreSQL spécial :
- ✅ Ne contient **aucune donnée utilisateur**  
- ✅ Participe aux **décisions de failover**  
- ✅ Très **léger en ressources**

**Pourquoi utiliser un witness ?**

Imaginez ce scénario :
```
Situation : 1 Primary + 1 Standby dans le même datacenter

Primary tombe en panne
→ Le Standby détecte la panne
→ Mais comment être sûr que c'est bien une panne ?
→ Et pas juste une perte de connexion réseau ?
```

**Avec un witness** (dans un datacenter différent) :
```
Le Standby vérifie auprès du witness :
"Est-ce que tu vois le Primary ?"

Si le witness répond "Non" → C'est bien une panne du Primary  
Si le witness répond "Oui" → C'est juste une coupure réseau locale  
```

**Architecture recommandée** :
```
┌──────────────┐         ┌──────────────┐
│ Datacenter A │         │ Datacenter B │
│              │         │              │
│  Primary     │◄────────┤  Standby     │
│  Standby     │         │  Witness     │
└──────────────┘         └──────────────┘
```

## Configuration de Repmgr

### 1. Installation

**Sur Debian/Ubuntu** :
```bash
# Ajouter le dépôt PostgreSQL officiel
sudo apt install postgresql-contrib-14 repmgr
```

**Sur Red Hat/CentOS** :
```bash
sudo yum install repmgr14
```

### 2. Fichier de Configuration : `repmgr.conf`

Chaque nœud a son propre fichier `/etc/repmgr/14/repmgr.conf`.

#### Configuration du Primary (Node 1)

```ini
# ==============================================================
# IDENTIFICATION DU NŒUD
# ==============================================================
node_id=1  
node_name='postgres1'  
conninfo='host=192.168.1.10 user=repmgr dbname=repmgr connect_timeout=2'  
data_directory='/var/lib/postgresql/14/main'  

# ==============================================================
# REPLICATION
# ==============================================================
replication_user='repmgr'

# ==============================================================
# FAILOVER ET PRIORITÉ
# ==============================================================
priority=100                    # Priorité élevée  
failover=automatic             # Failover automatique activé  
promote_command='repmgr standby promote -f /etc/repmgr/14/repmgr.conf --log-to-file'  
follow_command='repmgr standby follow -f /etc/repmgr/14/repmgr.conf --log-to-file --upstream-node-id=%n'  

# ==============================================================
# MONITORING (repmgrd)
# ==============================================================
monitoring_history=yes         # Enregistrer l'historique  
monitor_interval_secs=5        # Vérifier toutes les 5 secondes  
reconnect_attempts=6           # Essayer 6 fois avant de déclarer un nœud mort  
reconnect_interval=10          # Attendre 10 secondes entre chaque essai  

# ==============================================================
# NOTIFICATIONS
# ==============================================================
event_notification_command='/usr/local/bin/repmgr_notify.sh %n %e %s "%t" "%d"'

# ==============================================================
# LOGGING
# ==============================================================
log_file='/var/log/postgresql/repmgr.log'  
log_level=INFO  
log_status_interval=300        # Log de statut toutes les 5 minutes  
```

#### Configuration du Standby (Node 2)

```ini
node_id=2  
node_name='postgres2'  
conninfo='host=192.168.1.11 user=repmgr dbname=repmgr connect_timeout=2'  
data_directory='/var/lib/postgresql/14/main'  

replication_user='repmgr'

priority=90                    # Priorité légèrement plus faible  
failover=automatic  
promote_command='repmgr standby promote -f /etc/repmgr/14/repmgr.conf --log-to-file'  
follow_command='repmgr standby follow -f /etc/repmgr/14/repmgr.conf --log-to-file --upstream-node-id=%n'  

monitoring_history=yes  
monitor_interval_secs=5  
reconnect_attempts=6  
reconnect_interval=10  

log_file='/var/log/postgresql/repmgr.log'  
log_level=INFO  
```

### 3. Configuration PostgreSQL

**Paramètres requis dans `postgresql.conf`** :

```ini
# Replication settings
wal_level = replica  
max_wal_senders = 10  
max_replication_slots = 10  
hot_standby = on  

# Shared preload libraries
shared_preload_libraries = 'repmgr'
```

**Configuration de `pg_hba.conf`** :

```
# Repmgr connections
local   replication   repmgr                              trust  
host    replication   repmgr      127.0.0.1/32            trust  
host    replication   repmgr      192.168.1.0/24          scram-sha-256  

local   repmgr        repmgr                              trust  
host    repmgr        repmgr      127.0.0.1/32            trust  
host    repmgr        repmgr      192.168.1.0/24          scram-sha-256  
```

### 4. Création de l'Utilisateur Repmgr

```bash
# Sur le Primary
sudo -u postgres createuser -s repmgr  
sudo -u postgres createdb repmgr -O repmgr  
```

## Mise en Place d'un Cluster Repmgr : Étape par Étape

### Étape 1 : Enregistrer le Primary

Sur le serveur Primary (postgres1) :

```bash
# Enregistrer le nœud Primary
sudo -u postgres repmgr -f /etc/repmgr/14/repmgr.conf primary register

# Sortie attendue :
# INFO: connecting to primary database...
# NOTICE: attempting to install extension "repmgr"
# NOTICE: "repmgr" extension successfully installed
# NOTICE: primary node record (ID: 1) registered
```

**Vérification** :

```bash
# Voir les nœuds enregistrés
sudo -u postgres repmgr -f /etc/repmgr/14/repmgr.conf cluster show

# Sortie :
#  ID | Name      | Role    | Status    | Upstream | Location | Priority | Timeline | Connection string
# ----+-----------+---------+-----------+----------+----------+----------+----------+--------------------
#  1  | postgres1 | primary | * running |          | default  | 100      | 1        | host=192.168.1.10...
```

### Étape 2 : Cloner et Enregistrer le Premier Standby

Sur le serveur Standby (postgres2) :

**a) Arrêter PostgreSQL s'il tourne** :

```bash
sudo systemctl stop postgresql
```

**b) Vider le répertoire de données** (si existant) :

```bash
sudo -u postgres rm -rf /var/lib/postgresql/14/main/*
```

**c) Cloner depuis le Primary** :

```bash
# Dry-run pour vérifier la configuration
sudo -u postgres repmgr -h 192.168.1.10 -U repmgr -d repmgr \
  -f /etc/repmgr/14/repmgr.conf standby clone --dry-run

# Si OK, cloner réellement
sudo -u postgres repmgr -h 192.168.1.10 -U repmgr -d repmgr \
  -f /etc/repmgr/14/repmgr.conf standby clone

# Sortie :
# NOTICE: destination directory "/var/lib/postgresql/14/main" provided
# INFO: connecting to source node
# NOTICE: checking for available walsenders on the source node (2 required)
# INFO: sufficient walsenders available on the source node
# INFO: successfully connected to source node
# DETAIL: current installation size is 30 MB
# NOTICE: standby clone (using pg_basebackup) complete
# NOTICE: you can now start your PostgreSQL server
```

**d) Démarrer PostgreSQL** :

```bash
sudo systemctl start postgresql
```

**e) Enregistrer le Standby** :

```bash
sudo -u postgres repmgr -f /etc/repmgr/14/repmgr.conf standby register

# Sortie :
# INFO: connecting to local node "postgres2" (ID: 2)
# INFO: connecting to primary database
# NOTICE: standby node "postgres2" (ID: 2) successfully registered
```

**Vérification** :

```bash
sudo -u postgres repmgr -f /etc/repmgr/14/repmgr.conf cluster show

# Sortie :
#  ID | Name      | Role    | Status    | Upstream  | Priority | Timeline
# ----+-----------+---------+-----------+-----------+----------+----------
#  1  | postgres1 | primary | * running |           | 100      | 1
#  2  | postgres2 | standby |   running | postgres1 | 90       | 1
```

### Étape 3 : Ajouter un Second Standby (postgres3)

Répéter les mêmes étapes que pour postgres2 :

```bash
# Sur postgres3
sudo systemctl stop postgresql  
sudo -u postgres rm -rf /var/lib/postgresql/14/main/*  
sudo -u postgres repmgr -h 192.168.1.10 -U repmgr -d repmgr \  
  -f /etc/repmgr/14/repmgr.conf standby clone
sudo systemctl start postgresql  
sudo -u postgres repmgr -f /etc/repmgr/14/repmgr.conf standby register  
```

**Cluster final** :

```bash
sudo -u postgres repmgr -f /etc/repmgr/14/repmgr.conf cluster show

#  ID | Name      | Role    | Status    | Upstream  | Priority | Timeline
# ----+-----------+---------+-----------+-----------+----------+----------
#  1  | postgres1 | primary | * running |           | 100      | 1
#  2  | postgres2 | standby |   running | postgres1 | 90       | 1
#  3  | postgres3 | standby |   running | postgres1 | 80       | 1
```

### Étape 4 : Démarrer le Daemon `repmgrd` (Optionnel mais Recommandé)

Pour activer le failover automatique, démarrer `repmgrd` sur tous les nœuds :

```bash
# Sur chaque nœud (postgres1, postgres2, postgres3)
sudo systemctl enable repmgrd  
sudo systemctl start repmgrd  
sudo systemctl status repmgrd  

# Vérifier les logs
tail -f /var/log/postgresql/repmgr.log
```

**Ce que fait `repmgrd`** :
- Surveille l'état de PostgreSQL toutes les 5 secondes (configurable)
- Détecte les pannes du Primary
- Déclenche automatiquement le failover (si `failover=automatic`)

## Commandes Repmgr Essentielles

### 1. Afficher l'État du Cluster

```bash
# Vue d'ensemble du cluster
repmgr -f /etc/repmgr/14/repmgr.conf cluster show

# Vue détaillée avec informations supplémentaires
repmgr -f /etc/repmgr/14/repmgr.conf cluster show --verbose
```

### 2. Voir l'Historique des Événements

```bash
# 10 derniers événements
repmgr -f /etc/repmgr/14/repmgr.conf cluster event --limit 10

# Événements d'un nœud spécifique
repmgr -f /etc/repmgr/14/repmgr.conf cluster event --node-id=2
```

### 3. Promouvoir Manuellement un Standby

```bash
# Sur le Standby que vous voulez promouvoir
repmgr -f /etc/repmgr/14/repmgr.conf standby promote

# Sortie :
# NOTICE: promoting standby to primary
# DETAIL: promoting server "postgres2" (ID: 2) using "pg_ctl promote"
# NOTICE: waiting up to 60 seconds (parameter "promote_check_timeout") for promotion to complete
# INFO: standby promoted successfully
# NOTICE: STANDBY PROMOTE successful
```

### 4. Reconfigurer un Standby (Follow)

Si un Standby réplique depuis le mauvais Primary :

```bash
# Reconfigurer pour suivre le nouveau Primary (node ID 2)
repmgr -f /etc/repmgr/14/repmgr.conf standby follow

# Ou spécifier explicitement le nouveau Primary
repmgr -f /etc/repmgr/14/repmgr.conf standby follow --upstream-node-id=2
```

### 5. Switchover Planifié

Pour basculer manuellement vers un Standby sans interruption :

```bash
# Sur le Standby que vous voulez promouvoir (postgres2)
repmgr -f /etc/repmgr/14/repmgr.conf standby switchover

# Repmgr va :
# 1. Arrêter le Primary actuel proprement
# 2. Promouvoir le Standby
# 3. Reconfigurer l'ancien Primary en Standby
```

**Options utiles** :

```bash
# Switchover avec confirmation
repmgr -f /etc/repmgr/14/repmgr.conf standby switchover --force

# Dry-run (simulation)
repmgr -f /etc/repmgr/14/repmgr.conf standby switchover --dry-run
```

### 6. Tester la Configuration

```bash
# Vérifier la configuration avant de faire des opérations
repmgr -f /etc/repmgr/14/repmgr.conf node check

# Sortie :
# Node "postgres2":
#     Server role:                  OK (node is standby)
#     Replication lag:              OK (0 seconds)
#     WAL archiving:                OK (0 pending archive ready files)
#     Downstream servers:           OK (0 downstream nodes attached)
#     Replication slots:            OK (node has no replication slots)
#     Data directory:               OK (configured location "/var/lib/postgresql/14/main")
```

### 7. Réenregistrer un Nœud

Si vous devez reconstruire un nœud :

```bash
# Désenregistrer
repmgr -f /etc/repmgr/14/repmgr.conf standby unregister --node-id=3

# Cloner à nouveau
repmgr -h 192.168.1.10 -U repmgr -d repmgr \
  -f /etc/repmgr/14/repmgr.conf standby clone --force

# Réenregistrer
repmgr -f /etc/repmgr/14/repmgr.conf standby register --force
```

## Failover avec Repmgr

### Failover Automatique (avec repmgrd)

**Prérequis** :
- `repmgrd` actif sur tous les nœuds  
- `failover=automatic` dans `repmgr.conf`  
- `promote_command` et `follow_command` configurés

**Scénario** : Le Primary (postgres1) tombe en panne

```
T+0s   : Primary fonctionne normalement
         repmgrd surveille (heartbeat toutes les 5s)

T+15s  : Primary crash ❌

T+20s  : repmgrd sur postgres2 détecte la panne
         (monitor_interval_secs = 5, reconnect_attempts = 6)

T+20s  : repmgrd vérifie si d'autres nœuds voient le Primary
         (en interrogeant postgres3 et witness si présent)

T+21s  : Consensus : Le Primary est vraiment en panne

T+22s  : repmgrd sur postgres2 consulte la priorité des Standby
         postgres2 (priorité 90) vs postgres3 (priorité 80)
         → postgres2 est élu

T+23s  : repmgrd exécute promote_command
         → pg_ctl promote sur postgres2

T+30s  : postgres2 devient Primary ✅

T+31s  : postgres3 détecte le nouveau Primary
         repmgrd exécute follow_command
         → postgres3 se reconfigure pour répliquer depuis postgres2

T+35s  : Cluster opérationnel : postgres2 (Primary) + postgres3 (Standby)
```

**RTO** : 30-60 secondes (selon la configuration)

### Failover Manuel

Si `repmgrd` n'est pas actif ou si vous préférez contrôler :

**Sur le Standby que vous choisissez (postgres2)** :

```bash
# 1. Promouvoir le Standby
sudo -u postgres repmgr -f /etc/repmgr/14/repmgr.conf standby promote

# 2. Sur les autres Standby (postgres3), les reconfigurer
sudo -u postgres repmgr -f /etc/repmgr/14/repmgr.conf standby follow
```

### Protection contre le Split-Brain

Repmgr utilise plusieurs mécanismes :

#### 1. Witness Node

Le witness sert d'arbitre :

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│ Primary  │◄────┤ Standby  │◄────┤ Witness  │
│ (panne)  │  ✗  │          │  ✓  │          │
└──────────┘     └──────────┘     └──────────┘

Le Standby demande au Witness :
"Est-ce que tu vois le Primary ?"

Witness : "Non" → Le Primary est vraiment en panne → OK pour promouvoir  
Witness : "Oui" → C'est juste une perte de connexion locale → NE PAS promouvoir  
```

#### 2. Vérification de l'État du Primary

Avant de promouvoir, Repmgr vérifie :
- Le Primary répond-il aux connexions TCP ?
- Le Primary répond-il aux requêtes PostgreSQL ?
- Le Primary a-t-il été vu récemment par d'autres nœuds ?

#### 3. Désactivation de l'Ancien Primary (Fencing)

Repmgr **ne fait PAS** de fencing automatique comme Patroni.

**Important** : Vous devez vous assurer que l'ancien Primary est bien arrêté avant de promouvoir un Standby.

**Options pour le fencing manuel** :
- Script personnalisé via `event_notification_command`
- Utilisation de STONITH (Shoot The Other Node In The Head)
- Isolation réseau (déconnecter physiquement)

**Exemple de script de notification** :

```bash
#!/bin/bash
# /usr/local/bin/repmgr_notify.sh

NODE_ID=$1  
EVENT=$2  
SUCCESS=$3  
TIMESTAMP=$4  
DETAILS=$5  

if [ "$EVENT" = "repmgrd_failover_promote" ]; then
    echo "ALERTE: Failover en cours - Node $NODE_ID promu en Primary"
    # Envoyer une notification Slack/Email
    # Isoler l'ancien Primary via IPMI ou autre
fi
```

## Monitoring et Observabilité

### 1. Surveillance via `repmgr cluster show`

```bash
# Script de monitoring simple
#!/bin/bash
while true; do
    clear
    date
    repmgr -f /etc/repmgr/14/repmgr.conf cluster show
    sleep 5
done
```

### 2. Analyse des Logs

**Log de repmgr** : `/var/log/postgresql/repmgr.log`

**Événements importants à surveiller** :
- `standby promote` : Promotion d'un Standby  
- `repmgrd_failover_promote` : Failover automatique déclenché  
- `unable to connect` : Perte de connexion  
- `reconnecting` : Tentative de reconnexion

**Exemple de log lors d'un failover** :

```
[2025-11-22 10:15:20] [WARNING] unable to connect to node "postgres1" (ID: 1)
[2025-11-22 10:15:30] [NOTICE] attempting to promote node "postgres2" (ID: 2)
[2025-11-22 10:15:32] [INFO] executing: pg_ctl promote
[2025-11-22 10:15:40] [NOTICE] node "postgres2" (ID: 2) promoted to primary
[2025-11-22 10:15:41] [INFO] notifying other nodes of new primary
```

### 3. Métriques à Monitorer

| Métrique | Requête SQL | Seuil d'Alerte |
|----------|-------------|----------------|
| **Retard de réplication** | `SELECT EXTRACT(EPOCH FROM (now() - pg_last_xact_replay_timestamp())) AS lag_seconds;` | > 60 secondes |
| **Nombre de failovers** | `SELECT COUNT(*) FROM repmgr.events WHERE event = 'repmgrd_failover_promote' AND event_timestamp > now() - interval '24 hours';` | > 1 par jour |
| **État des nœuds** | `SELECT node_name, active FROM repmgr.nodes WHERE active = false;` | Si active = false |

### 4. Intégration avec Prometheus

Repmgr n'a pas d'exporter Prometheus natif, mais vous pouvez :

**Option 1 : Utiliser `postgres_exporter`**

```yaml
# Ajouter des requêtes personnalisées
queries:
  - name: repmgr_nodes
    query: "SELECT node_id, node_name, type, active, priority FROM repmgr.nodes"
    metrics:
      - node_id:
          usage: "LABEL"
          description: "Node ID"
      - active:
          usage: "GAUGE"
          description: "Is node active"
```

**Option 2 : Script shell + node_exporter**

```bash
#!/bin/bash
# Exporter des métriques Repmgr pour node_exporter

# Nombre de nœuds actifs
ACTIVE_NODES=$(psql -U repmgr -d repmgr -tAc "SELECT COUNT(*) FROM repmgr.nodes WHERE active = true")  
echo "repmgr_active_nodes $ACTIVE_NODES"  

# Retard de réplication (si Standby)
if [ "$(psql -tAc 'SELECT pg_is_in_recovery()')" = "t" ]; then
    LAG=$(psql -tAc "SELECT COALESCE(EXTRACT(EPOCH FROM (now() - pg_last_xact_replay_timestamp())), 0)")
    echo "repmgr_replication_lag_seconds $LAG"
fi
```

## Scénarios d'Usage de Repmgr

### Scénario 1 : Cluster Simple (1 Primary + 1 Standby)

**Configuration minimale** :

```
┌──────────────┐         ┌──────────────┐
│  postgres1   │────────►│  postgres2   │
│  (Primary)   │         │  (Standby)   │
│  Priority:100│         │  Priority:90 │
└──────────────┘         └──────────────┘
```

**Cas d'usage** :
- Petite application
- Backup à chaud (hot standby pour les lectures)
- Disaster recovery simple

**Failover** : Manuel recommandé (car pas de witness)

### Scénario 2 : Cluster avec Witness (Recommandé)

**Configuration** :

```
┌──────────────┐         ┌──────────────┐
│  postgres1   │────────►│  postgres2   │
│  (Primary)   │         │  (Standby)   │
│  Priority:100│         │  Priority:90 │
└──────────────┘         └──────────────┘
        │                        │
        └────────┬───────────────┘
                 ▼
          ┌──────────────┐
          │   witness1   │
          │   (Witness)  │
          └──────────────┘
```

**Avantage** : Meilleure détection des pannes, réduction des faux positifs

**Failover** : Automatique possible avec `repmgrd`

### Scénario 3 : Multi-Standby avec Cascading

**Configuration** :

```
     ┌──────────────┐
     │  postgres1   │
     │  (Primary)   │
     └──────────────┘
            │
            ├───────────────────┐
            ▼                   ▼
     ┌──────────────┐    ┌──────────────┐
     │  postgres2   │    │  postgres3   │
     │  (Standby)   │    │  (Standby)   │
     └──────────────┘    └──────────────┘
            │
            ▼
     ┌──────────────┐
     │  postgres4   │
     │  (Standby)   │ → Cascade depuis postgres2
     └──────────────┘
```

**Avantage** : Réduire la charge sur le Primary (postgres4 réplique depuis postgres2)

**Configuration** :

```bash
# Sur postgres4, cloner depuis postgres2
repmgr -h postgres2 -U repmgr -d repmgr \
  -f /etc/repmgr/14/repmgr.conf standby clone --upstream-node-id=2
```

## Maintenance et Opérations Courantes

### 1. Ajouter un Nouveau Standby

```bash
# Sur le nouveau serveur (postgres4)
sudo -u postgres repmgr -h postgres1 -U repmgr -d repmgr \
  -f /etc/repmgr/14/repmgr.conf standby clone
sudo systemctl start postgresql  
sudo -u postgres repmgr -f /etc/repmgr/14/repmgr.conf standby register  
sudo systemctl start repmgrd  
```

### 2. Retirer un Standby du Cluster

```bash
# Arrêter repmgrd et PostgreSQL
sudo systemctl stop repmgrd  
sudo systemctl stop postgresql  

# Désenregistrer du cluster
sudo -u postgres repmgr -f /etc/repmgr/14/repmgr.conf standby unregister --node-id=4
```

### 3. Reconstruire un Standby Désynchronisé

Si un Standby est trop en retard ou corrompu :

```bash
# Sur le Standby à reconstruire
sudo systemctl stop postgresql  
sudo -u postgres rm -rf /var/lib/postgresql/14/main/*  
sudo -u postgres repmgr -h postgres1 -U repmgr -d repmgr \  
  -f /etc/repmgr/14/repmgr.conf standby clone --force
sudo systemctl start postgresql
```

### 4. Switchover Planifié (Maintenance du Primary)

```bash
# Sur le Standby cible (postgres2)
sudo -u postgres repmgr -f /etc/repmgr/14/repmgr.conf standby switchover --dry-run

# Si OK, exécuter réellement
sudo -u postgres repmgr -f /etc/repmgr/14/repmgr.conf standby switchover

# Repmgr va :
# 1. Vérifier que le switchover est possible
# 2. Arrêter le Primary actuel
# 3. Promouvoir postgres2
# 4. Reconfigurer l'ancien Primary en Standby
```

**RTO** : ~10-30 secondes avec 0 perte de données

### 5. Réintégrer l'Ancien Primary après Panne

Après un failover, l'ancien Primary doit être réintégré :

**Option 1 : Reconstruction Complète** (sûr mais lent)

```bash
# Sur l'ancien Primary (postgres1)
sudo systemctl stop postgresql  
sudo -u postgres rm -rf /var/lib/postgresql/14/main/*  
sudo -u postgres repmgr -h postgres2 -U repmgr -d repmgr \  
  -f /etc/repmgr/14/repmgr.conf standby clone
sudo systemctl start postgresql  
sudo -u postgres repmgr -f /etc/repmgr/14/repmgr.conf standby register --force  
```

**Option 2 : pg_rewind** (rapide mais nécessite pg_rewind activé)

```bash
# Sur l'ancien Primary
sudo systemctl stop postgresql

# Exécuter pg_rewind pour resynchroniser
sudo -u postgres pg_rewind \
  --target-pgdata=/var/lib/postgresql/14/main \
  --source-server="host=postgres2 port=5432 user=repmgr dbname=postgres" \
  --progress

# Reconfigurer en Standby
sudo -u postgres repmgr -f /etc/repmgr/14/repmgr.conf standby register --force  
sudo systemctl start postgresql  
```

## Troubleshooting Repmgr

### Problème 1 : "unable to connect to primary"

**Symptôme** : Standby ne peut pas se connecter au Primary

**Diagnostic** :

```bash
# Tester la connexion PostgreSQL
psql -h postgres1 -U repmgr -d repmgr

# Vérifier pg_hba.conf sur le Primary
# Doit contenir :
# host repmgr repmgr 192.168.1.0/24 scram-sha-256
```

**Causes courantes** :
- Firewall bloquant le port 5432
- pg_hba.conf incorrect
- Mot de passe incorrect

### Problème 2 : "insufficient walsenders available"

**Symptôme** : Impossible de cloner un nouveau Standby

**Cause** : `max_wal_senders` trop faible dans `postgresql.conf`

**Solution** :

```ini
# postgresql.conf
max_wal_senders = 10  # Au moins égal au nombre de Standby + 2
```

```bash
# Recharger la configuration
sudo systemctl reload postgresql
```

### Problème 3 : Failover Automatique Ne Se Déclenche Pas

**Diagnostic** :

```bash
# Vérifier que repmgrd tourne
sudo systemctl status repmgrd

# Vérifier les logs
tail -f /var/log/postgresql/repmgr.log
```

**Causes courantes** :
- `failover=automatic` non défini  
- `promote_command` mal configuré
- Pas assez de nœuds actifs pour atteindre le quorum
- Witness indisponible

### Problème 4 : Split-Brain Détecté

**Symptôme** : Deux nœuds se considèrent Primary

**Détection** :

```bash
# Sur chaque nœud
psql -c "SELECT pg_is_in_recovery();"

# Si les deux retournent 'f' (false) → Split-brain !
```

**Résolution** :

1. **Identifier le bon Primary** (celui avec les données les plus récentes)  
2. **Arrêter immédiatement le faux Primary**  
3. **Reconstruire le faux Primary en Standby**

```bash
# Sur le faux Primary
sudo systemctl stop postgresql  
sudo -u postgres rm -rf /var/lib/postgresql/14/main/*  
sudo -u postgres repmgr -h <bon_primary> -U repmgr -d repmgr \  
  -f /etc/repmgr/14/repmgr.conf standby clone
sudo systemctl start postgresql
```

## Repmgr et Load Balancing

Repmgr ne gère pas lui-même le routage des connexions. Utilisez :

### Option 1 : HAProxy

Similaire à la configuration avec Patroni, mais en interrogeant directement PostgreSQL :

```
# /etc/haproxy/haproxy.cfg

backend postgres_primary
    option pgsql-check user repmgr
    server postgres1 192.168.1.10:5432 check
    server postgres2 192.168.1.11:5432 check backup
    server postgres3 192.168.1.12:5432 check backup

backend postgres_replicas
    option pgsql-check user repmgr
    balance roundrobin
    server postgres2 192.168.1.11:5432 check
    server postgres3 192.168.1.12:5432 check
```

### Option 2 : PgBouncer + Scripts

Utiliser un script qui interroge `repmgr cluster show` pour déterminer le Primary :

```bash
#!/bin/bash
# get_primary.sh

PRIMARY=$(repmgr -f /etc/repmgr/14/repmgr.conf cluster show --csv \
  | grep primary | cut -d',' -f2)

echo $PRIMARY
```

Reconfigurer PgBouncer dynamiquement en fonction du résultat.

### Option 3 : DNS Dynamique

Utiliser un script qui met à jour un enregistrement DNS en cas de failover :

```bash
# Script déclenché par event_notification_command

if [ "$EVENT" = "repmgrd_failover_promote" ]; then
    # Mettre à jour le DNS pour pointer vers le nouveau Primary
    aws route53 change-resource-record-sets ...
fi
```

## Avantages et Limites de Repmgr

### ✅ Avantages

**1. Simplicité**
- Pas de dépendance externe (etcd, Consul)
- Configuration plus simple que Patroni
- Courbe d'apprentissage courte

**2. Légèreté**
- Faible overhead (daemon léger)
- Pas d'infrastructure additionnelle à maintenir

**3. Flexibilité**
- Failover automatique ou manuel au choix
- Compatible avec PostgreSQL vanilla (pas de fork)

**4. Intégration PostgreSQL**
- Métadonnées stockées dans PostgreSQL
- Utilise les mécanismes natifs de réplication

**5. Maturité**
- Existe depuis 2010
- Utilisé en production par de nombreuses entreprises

### ⚠️ Limites et Considérations

**1. Protection Split-Brain Partielle**
- Moins robuste que Patroni avec consensus
- Nécessite un witness pour être fiable
- Pas de fencing automatique

**2. Failover Moins Rapide**
- RTO de 30-60 secondes (vs 15-30s pour Patroni)
- Dépend de la configuration `reconnect_attempts`

**3. Automatisation Limitée**
- Pas de reconfiguration automatique des applications
- Nécessite HAProxy ou script pour le routage

**4. Moins Adapté aux Grandes Échelles**
- Clusters 10+ nœuds : Patroni recommandé
- Multi-datacenter complexe : Patroni préférable

**5. Moins Kubernetes-Friendly**
- Pas d'Operator natif comme Patroni (Zalando)
- Nécessite plus de travail pour l'intégration K8s

## Résumé des Concepts Clés

### Ce qu'il Faut Retenir

**1. Repmgr = Simplicité et Efficacité**
- Pas de dépendance externe (etcd/Consul)
- Métadonnées stockées dans PostgreSQL
- Outil idéal pour les clusters de petite à moyenne taille

**2. Composants Principaux**
- **repmgr** : CLI pour la gestion  
- **repmgrd** : Daemon pour la surveillance et le failover automatique

- Automatique possible avec `repmgrd`

**5. Limites**

- Nécessite supervision humaine pour les grandes échelles

## Conclusion

Repmgr est une excellente solution pour gérer la haute disponibilité PostgreSQL de manière simple et efficace. Sa philosophie "Keep it simple" en fait un choix idéal pour :

- Les équipes qui débutent avec la HA PostgreSQL
- Les clusters de petite à moyenne taille
- Les environnements où la simplicité prime sur l'automatisation totale

**Points forts** :
- ✅ Installation et configuration simples  
- ✅ Pas d'infrastructure externe à maintenir  
- ✅ Excellent pour cloner et gérer des Standby  
- ✅ Switchover planifié très simple  
- ✅ Coût total de possession faible

**Quand préférer Patroni ?**
- Production ultra-critique (99.99%+ uptime)
- Clusters très complexes (10+ nœuds)
- Déploiements Kubernetes
- Nécessité de fencing automatique garanti

Repmgr reste une solution mature, stable et largement utilisée en production. Pour de nombreux cas d'usage, sa simplicité est un atout majeur qui compense largement ses limites par rapport à des solutions plus complexes comme Patroni.
