# 17.2.1. Configuration Primary/Standby - Réplication Physique PostgreSQL 18

## Introduction

La réplication Primary/Standby (aussi appelée Master/Slave ou Leader/Follower) est une architecture fondamentale de haute disponibilité dans PostgreSQL. Elle permet de maintenir une ou plusieurs copies de votre base de données sur des serveurs différents, garantissant ainsi la continuité de service en cas de panne.

### Pourquoi mettre en place une réplication Primary/Standby ?

**Les objectifs principaux :**
- **Haute disponibilité (HA)** : Si le serveur primaire tombe, un serveur secondaire peut prendre le relais  
- **Répartition de charge en lecture** : Les serveurs standby peuvent traiter les requêtes de lecture (SELECT), soulageant ainsi le serveur primaire  
- **Sauvegardes sans impact** : Effectuer les sauvegardes sur un standby plutôt que sur le serveur primaire  
- **Disaster Recovery** : Avoir une copie des données dans un autre datacenter ou région géographique

### Concepts clés à comprendre

Avant de configurer la réplication, il est essentiel de comprendre quelques concepts :

**Le serveur Primary (Primaire)** :
- C'est le serveur "maître" qui accepte les écritures (INSERT, UPDATE, DELETE)
- Il génère les WAL (Write-Ahead Logs) qui contiennent toutes les modifications
- Il envoie ces WAL aux serveurs standby en temps réel

**Le serveur Standby (Secondaire)** :
- C'est une copie du serveur primaire, maintenue à jour en continu
- Il reçoit et rejoue les WAL envoyés par le primary
- En mode par défaut, il est en lecture seule (read-only)
- Il peut devenir primary en cas de promotion (failover)

**Les WAL (Write-Ahead Logs)** :
- Fichiers journaux contenant toutes les modifications de la base de données
- Écrits avant que les données ne soient réellement modifiées sur disque
- Utilisés pour la récupération en cas de crash et pour la réplication
- Situés dans le répertoire `pg_wal/` de l'installation PostgreSQL

**Le Streaming Replication** :
- Mécanisme par lequel les WAL sont envoyés en continu (en "streaming") du primary au standby
- Utilise une connexion réseau dédiée
- Permet une latence minimale entre primary et standby (quelques millisecondes)

## Architecture de la Réplication Physical

### Vue d'ensemble

```
┌─────────────────────────┐
│   Serveur Primary       │
│   (Lecture/Écriture)    │
│                         │
│   ┌─────────────────┐   │
│   │ Base de données │   │
│   └─────────────────┘   │
│          │              │
│          ↓              │
│   ┌─────────────────┐   │
│   │   WAL Sender    │   │  ← Processus qui envoie les WAL
│   └────────┬────────┘   │
└────────────┼────────────┘
             │
             │ Streaming (TCP/IP)
             │ Port 5432 (ou personnalisé)
             │
             ↓
┌─────────────────────────┐
│   Serveur Standby       │
│   (Lecture seule)       │
│                         │
│   ┌─────────────────┐   │
│   │  WAL Receiver   │   │  ← Processus qui reçoit les WAL
│   └────────┬────────┘   │
│            ↓            │
│   ┌─────────────────┐   │
│   │ Startup Process │   │  ← Rejoue les WAL
│   └────────┬────────┘   │
│            ↓            │
│   ┌─────────────────┐   │
│   │ Base de données │   │  ← Copie identique du primary
│   └─────────────────┘   │
└─────────────────────────┘
```

### Le cycle de vie de la réplication

1. **Initialisation** : Le standby est créé à partir d'une copie physique (backup) du primary  
2. **Connexion** : Le standby se connecte au primary via une connexion réseau authentifiée  
3. **Streaming** : Le primary envoie en continu les WAL au fur et à mesure de leur génération  
4. **Replay** : Le standby rejoue ces WAL pour maintenir sa copie à jour  
5. **Monitoring** : Les deux serveurs échangent des informations sur leur état (lag, position WAL, etc.)

## Prérequis et Planification

### Configuration réseau

**Connectivité requise :**
- Le serveur standby doit pouvoir se connecter au serveur primary sur le port PostgreSQL (par défaut 5432)
- Prévoir une bande passante suffisante pour le transfert des WAL (dépend du volume d'écritures)
- Latence réseau faible recommandée (< 10ms pour une réplication synchrone)

**Considérations de sécurité :**
- Utiliser SSL/TLS pour chiffrer les connexions de réplication
- Configurer des règles firewall strictes
- Créer un utilisateur dédié à la réplication avec des privilèges minimaux

### Configuration matérielle

**Pour le serveur Standby :**
- Idéalement, configuration matérielle identique ou supérieure au primary
- Même version de PostgreSQL (obligatoire pour la réplication physique)
- Même système d'exploitation et architecture (x86_64, ARM, etc.)
- Espace disque suffisant pour stocker la base de données + WAL

**Calcul de l'espace disque pour les WAL :**
- Les WAL s'accumulent sur le primary si le standby est indisponible
- Prévoir un espace tampon : au minimum 2-3× la taille de `max_wal_size`
- Pour une production critique : 50-100 Go dédiés aux WAL

### Planification de la topologie

**Options courantes :**
1. **Primary → Standby simple** : Architecture minimale (1 primary + 1 standby)  
2. **Primary → Multiple Standbys** : Un primary peut alimenter plusieurs standbys  
3. **Cascading Replication** : Primary → Standby1 → Standby2 (réduction de charge sur le primary)

## Configuration du Serveur Primary

### Étape 1 : Créer un utilisateur de réplication

Le primary a besoin d'un utilisateur dédié que le standby utilisera pour se connecter.

**Concept :** Un utilisateur avec le privilège `REPLICATION` peut initier une connexion de réplication et recevoir les WAL.

**Création de l'utilisateur :**
```sql
-- Se connecter en tant que superutilisateur (postgres)
CREATE ROLE replicator WITH REPLICATION LOGIN PASSWORD 'mot_de_passe_fort';
```

**Explications :**
- `REPLICATION` : Privilège spécial autorisant la réplication  
- `LOGIN` : Permet à cet utilisateur de se connecter  
- `PASSWORD` : Définit un mot de passe sécurisé (à remplacer par un mot de passe fort)

**Bonnes pratiques :**
- Ne pas utiliser le compte `postgres` pour la réplication
- Générer un mot de passe long et complexe (20+ caractères)
- Stocker ce mot de passe de manière sécurisée (coffre-fort de mots de passe)

### Étape 2 : Configurer postgresql.conf sur le Primary

Le fichier `postgresql.conf` contient les paramètres de configuration de PostgreSQL. Il est généralement situé dans le répertoire de données PostgreSQL (par exemple `/var/lib/postgresql/18/main/`).

**Paramètres essentiels à configurer :**

```ini
# ────────────────────────────────────────────────
# Configuration WAL
# ────────────────────────────────────────────────

# Niveau de détail des WAL (requis pour la réplication)
wal_level = replica

# Taille maximale des WAL entre deux checkpoints
# Plus cette valeur est élevée, moins il y a de checkpoints (mais plus de WAL à rejouer en cas de crash)
max_wal_size = 2GB

# Nombre de segments WAL à conserver
# Utile si le standby est temporairement déconnecté
wal_keep_size = 1GB


# ────────────────────────────────────────────────
# Configuration de la réplication
# ────────────────────────────────────────────────

# Nombre maximum de connexions WAL sender (1 par standby + quelques extras)
# Pour 2 standbys : mettre au moins 4-5
max_wal_senders = 5

# Active l'archivage des WAL (optionnel mais recommandé pour PITR)
archive_mode = on

# Commande d'archivage (exemple : copie vers un répertoire de sauvegarde)
# À adapter selon votre infrastructure
archive_command = 'test ! -f /archive/pg_wal/%f && cp %p /archive/pg_wal/%f'

# Timeout pour l'envoi des WAL (en millisecondes)
# Si le standby ne répond pas pendant ce délai, la connexion est fermée
wal_sender_timeout = 60s


# ────────────────────────────────────────────────
# Configuration des slots de réplication (recommandé)
# ────────────────────────────────────────────────

# Nombre de slots de réplication (1 par standby recommandé)
max_replication_slots = 5

# Permet aux standbys de signaler leur progression
# Nécessaire pour le monitoring
hot_standby_feedback = on
```

**Explications détaillées des paramètres clés :**

**`wal_level = replica`**
- `minimal` : WAL minimum (pas de réplication possible)  
- `replica` : WAL suffisant pour la réplication physique (valeur standard)  
- `logical` : WAL suffisant pour la réplication logique (plus verbeux)

**`max_wal_senders`**
- Chaque standby nécessite 1 wal_sender
- Ajouter 2-3 connexions supplémentaires pour les outils (pg_basebackup, monitoring)
- Formule : `nombre_de_standbys + 3`

**`wal_keep_size`**
- WAL minimum à conserver sur le primary même s'ils ne sont plus nécessaires localement
- Protection contre la perte de WAL si le standby est temporairement déconnecté
- Valeur typique : 1-2 Go (dépend du volume d'écritures)

**`archive_command`**
- Commande shell exécutée pour chaque segment WAL à archiver
- `%p` : chemin complet du fichier WAL  
- `%f` : nom du fichier WAL
- Le test `! -f` évite d'écraser un fichier existant (sécurité)

**Nouveautés PostgreSQL 18 :**
Dans PostgreSQL 18, le sous-système I/O asynchrone améliore les performances de réplication. Si vous souhaitez l'activer :

```ini
# Activer l'I/O asynchrone (PG 18+)
io_method = 'worker'  # ou 'io_uring' sur Linux
```

### Étape 3 : Configurer pg_hba.conf sur le Primary

Le fichier `pg_hba.conf` (Host-Based Authentication) contrôle qui peut se connecter à PostgreSQL et comment.

**Ajouter une ligne pour autoriser la réplication :**

```
# TYPE  DATABASE        USER            ADDRESS                 METHOD

# Autoriser la réplication depuis le serveur standby
# Remplacer 192.168.1.20/32 par l'adresse IP réelle du standby
host    replication     replicator      192.168.1.20/32         scram-sha-256
```

**Explications des colonnes :**
- **TYPE** : `host` (connexion TCP/IP), `replication` (connexion de réplication)  
- **DATABASE** : `replication` (pseudo-base pour les connexions de réplication)  
- **USER** : `replicator` (l'utilisateur créé à l'étape 1)  
- **ADDRESS** : Adresse IP du standby (notation CIDR)  
  - `/32` : Une seule adresse IP  
  - `/24` : Tout un sous-réseau (ex: 192.168.1.0/24 = 192.168.1.0 à 192.168.1.255)  
- **METHOD** : `scram-sha-256` (méthode d'authentification moderne et sécurisée)

**Pour plusieurs standbys :**
```
host    replication     replicator      192.168.1.20/32         scram-sha-256  
host    replication     replicator      192.168.1.21/32         scram-sha-256  
host    replication     replicator      192.168.1.22/32         scram-sha-256  
```

**Ou pour un sous-réseau entier (moins sécurisé) :**
```
host    replication     replicator      192.168.1.0/24          scram-sha-256
```

**Note importante sur SSL/TLS (recommandé en production) :**
Pour forcer l'utilisation de SSL, remplacer `host` par `hostssl` :
```
hostssl replication     replicator      192.168.1.20/32         scram-sha-256
```

### Étape 4 : Créer un slot de réplication (recommandé)

Un **slot de réplication** garantit que le primary conserve les WAL nécessaires tant que le standby ne les a pas reçus, même si `wal_keep_size` est dépassé.

**Création du slot :**
```sql
-- Se connecter au primary en tant que superutilisateur
SELECT pg_create_physical_replication_slot('standby_slot_1');
```

**Explications :**
- `standby_slot_1` : Nom du slot (choisir un nom descriptif, ex: `standby_paris_slot`)
- Le slot est **persistant** : il survit aux redémarrages
- Le primary conservera **tous** les WAL nécessaires pour ce slot

**Avantages des slots :**
- ✅ Protection contre la perte de WAL si le standby est en panne prolongée  
- ✅ Pas besoin de calculer précisément `wal_keep_size`  
- ✅ Monitoring facile de l'état de réplication

**Inconvénients :**
- ⚠️ Les WAL s'accumulent indéfiniment si le standby ne se reconnecte jamais  
- ⚠️ Peut remplir le disque du primary si on ne surveille pas  
- ⚠️ Nécessite une suppression manuelle du slot si le standby est définitivement retiré

**Surveillance d'un slot :**
```sql
-- Vérifier l'état des slots
SELECT slot_name, active, restart_lsn, confirmed_flush_lsn  
FROM pg_replication_slots;  
```

### Étape 5 : Redémarrer le serveur Primary

Après avoir modifié `postgresql.conf` et `pg_hba.conf`, il faut redémarrer PostgreSQL :

```bash
# Sur Linux avec systemd
sudo systemctl restart postgresql

# Ou avec pg_ctl
pg_ctl -D /var/lib/postgresql/18/main restart
```

**Vérification après redémarrage :**
```sql
-- Vérifier que wal_level est bien configuré
SHOW wal_level;  -- Doit retourner 'replica' ou 'logical'

-- Vérifier max_wal_senders
SHOW max_wal_senders;  -- Doit être > 0

-- Vérifier que le slot existe
SELECT * FROM pg_replication_slots;
```

## Configuration du Serveur Standby

### Étape 1 : Créer une copie initiale du Primary (Base Backup)

Le standby doit démarrer avec une copie physique exacte du serveur primary. PostgreSQL fournit l'outil `pg_basebackup` pour cela.

**Sur le serveur Standby, exécuter :**

```bash
# Arrêter PostgreSQL sur le standby (si déjà en cours d'exécution)
sudo systemctl stop postgresql

# Sauvegarder l'ancien répertoire de données (par précaution)
sudo mv /var/lib/postgresql/18/main /var/lib/postgresql/18/main.old

# Créer une copie du primary
pg_basebackup \
    -h 192.168.1.10 \              # Adresse IP du serveur primary
    -D /var/lib/postgresql/18/main \  # Répertoire de données du standby
    -U replicator \                # Utilisateur de réplication
    -P \                           # Afficher la progression
    -v \                           # Mode verbose
    -R \                           # Créer automatiquement standby.signal
    -X stream \                    # Inclure les WAL nécessaires
    -C \                           # Créer automatiquement le slot (si configuré)
    -S standby_slot_1              # Nom du slot de réplication
```

**Explications des options :**

- `-h 192.168.1.10` : Adresse IP du serveur primary  
- `-D /var/lib/postgresql/18/main` : Répertoire de destination (sera créé)  
- `-U replicator` : Utilisateur PostgreSQL pour la connexion  
- `-P` : Affiche une barre de progression (utile pour les grandes bases)  
- `-v` : Mode verbose (affiche les détails de l'opération)  
- `-R` : Crée automatiquement le fichier `standby.signal` (marque le serveur comme standby)  
- `-X stream` : Inclut les WAL générés pendant le backup (évite les trous)  
- `-C` : Crée le slot de réplication automatiquement (si `-S` est utilisé)  
- `-S standby_slot_1` : Nom du slot à créer/utiliser

**Le processus va :**
1. Se connecter au primary avec l'utilisateur `replicator`  
2. Initier un backup cohérent de la base de données  
3. Copier tous les fichiers de données vers le standby  
4. Créer automatiquement les fichiers de configuration pour le standby

**Durée estimée :**
Dépend de la taille de la base et de la bande passante réseau. Pour une base de 100 Go sur un réseau à 1 Gbps : environ 15-20 minutes.

### Étape 2 : Configurer postgresql.conf sur le Standby

Le standby nécessite quelques paramètres spécifiques, bien que beaucoup soient hérités du primary via le basebackup.

**Paramètres spécifiques au Standby :**

```ini
# ────────────────────────────────────────────────
# Configuration Hot Standby (lectures sur le standby)
# ────────────────────────────────────────────────

# Active les connexions en lecture seule sur le standby
hot_standby = on

# Feedback du standby vers le primary (évite les conflits de requêtes)
hot_standby_feedback = on

# Délai maximal d'attente pour résoudre un conflit de requête
max_standby_streaming_delay = 30s


# ────────────────────────────────────────────────
# Configuration de la connexion au Primary
# ────────────────────────────────────────────────

# Chaîne de connexion au serveur primary
# Format : libpq connection string
primary_conninfo = 'host=192.168.1.10 port=5432 user=replicator password=mot_de_passe_fort application_name=standby1'

# Nom du slot de réplication à utiliser
primary_slot_name = 'standby_slot_1'


# ────────────────────────────────────────────────
# Configuration des WAL
# ────────────────────────────────────────────────

# Répertoire de restauration des WAL archivés (optionnel, pour PITR)
restore_command = 'cp /archive/pg_wal/%f %p'

# Trigger file pour promouvoir le standby en primary
promote_trigger_file = '/tmp/postgresql.trigger.5432'
```

**Explications des paramètres clés :**

**`hot_standby = on`**
- Permet aux utilisateurs de se connecter en lecture seule au standby
- Idéal pour répartir les charges de lecture (reporting, analytics)
- Si `off`, le standby est inaccessible aux connexions clients

**`hot_standby_feedback = on`**
- Le standby informe le primary des requêtes en cours
- Évite que le primary ne supprime des lignes encore utilisées par le standby
- Peut légèrement augmenter le bloat sur le primary (acceptable dans la plupart des cas)

**`max_standby_streaming_delay`**
- Temps maximum qu'une requête sur le standby peut bloquer le replay des WAL
- Si une requête en lecture entre en conflit avec un WAL (ex: DELETE), PostgreSQL attendra
- Après ce délai, la requête est annulée pour permettre au standby de rattraper son retard
- Valeurs typiques : 30s à 300s (selon tolérance)

**`primary_conninfo`**
- Chaîne de connexion au serveur primary (format libpq)
- **host** : IP du primary  
- **user** : utilisateur de réplication  
- **password** : mot de passe (ou utiliser un `.pgpass` plus sécurisé)  
- **application_name** : nom pour identifier ce standby dans les vues de monitoring

**Format alternatif avec SSL :**
```ini
primary_conninfo = 'host=192.168.1.10 port=5432 user=replicator password=mot_de_passe_fort sslmode=require application_name=standby1'
```

**`primary_slot_name`**
- Nom du slot de réplication créé sur le primary
- Si configuré, le standby utilisera ce slot pour garantir la conservation des WAL

**`restore_command`**
- Commande pour restaurer des WAL archivés (utilisé si le streaming échoue)
- Fonctionne comme un filet de sécurité
- `%f` : nom du fichier WAL demandé  
- `%p` : chemin où PostgreSQL attend le fichier

### Étape 3 : Créer le fichier standby.signal

Ce fichier est un marqueur qui indique à PostgreSQL que ce serveur doit démarrer en mode standby.

**Création :**

Si vous avez utilisé l'option `-R` avec `pg_basebackup`, ce fichier est créé automatiquement. Sinon :

```bash
# Sur le serveur standby
touch /var/lib/postgresql/18/main/standby.signal
```

**Ce fichier :**
- Est un fichier vide (seule sa présence compte)
- Doit être dans le répertoire de données PostgreSQL (`PGDATA`)
- Sera automatiquement supprimé lors d'une promotion en primary

### Étape 4 : Ajuster les permissions (important)

Le répertoire de données doit appartenir à l'utilisateur PostgreSQL :

```bash
# Sur le serveur standby
sudo chown -R postgres:postgres /var/lib/postgresql/18/main  
sudo chmod 700 /var/lib/postgresql/18/main  
```

**Explications :**
- PostgreSQL refuse de démarrer si les permissions sont incorrectes (sécurité)
- `700` : Seul le propriétaire (postgres) peut lire/écrire/exécuter
- Cela protège les données contre les accès non autorisés

### Étape 5 : Démarrer le serveur Standby

```bash
# Sur le serveur standby
sudo systemctl start postgresql

# Vérifier le statut
sudo systemctl status postgresql
```

**Le standby va :**
1. Détecter le fichier `standby.signal`  
2. Se connecter au primary avec les paramètres de `primary_conninfo`  
3. Commencer à recevoir et rejouer les WAL  
4. Devenir accessible en lecture seule (si `hot_standby = on`)

## Vérification et Monitoring de la Réplication

### Sur le serveur Primary

**1. Vérifier les connexions de réplication actives :**

```sql
SELECT
    pid,                          -- ID du processus wal sender
    usename,                      -- Utilisateur de réplication
    application_name,             -- Nom du standby
    client_addr,                  -- Adresse IP du standby
    state,                        -- État (streaming, catchup, etc.)
    sent_lsn,                     -- Position WAL envoyée
    write_lsn,                    -- Position WAL écrite par le standby
    flush_lsn,                    -- Position WAL flushée par le standby
    replay_lsn,                   -- Position WAL rejouée par le standby
    sync_state                    -- État de synchronisation (async/sync)
FROM pg_stat_replication;
```

**Résultat attendu :**
```
 pid  | usename    | application_name | client_addr   | state     | sent_lsn  | ...
------+------------+------------------+---------------+-----------+-----------+-----
 1234 | replicator | standby1         | 192.168.1.20  | streaming | 0/3000148 | ...
```

**Interprétation :**
- **state = 'streaming'** : La réplication fonctionne correctement  
- **state = 'catchup'** : Le standby est en train de rattraper un retard  
- **state = 'backup'** : Un basebackup est en cours

**2. Calculer le lag (retard) de réplication :**

```sql
SELECT
    application_name,
    client_addr,
    pg_wal_lsn_diff(sent_lsn, replay_lsn) AS lag_bytes,
    pg_wal_lsn_diff(sent_lsn, replay_lsn) / 1024 / 1024 AS lag_mb,
    write_lag,
    flush_lag,
    replay_lag
FROM pg_stat_replication;
```

**Explication des métriques de lag :**
- **lag_bytes** : Nombre d'octets de WAL non encore rejoués par le standby  
- **write_lag** : Temps entre l'écriture sur le primary et la confirmation d'écriture par le standby  
- **flush_lag** : Temps pour que les WAL soient flushés sur disque sur le standby  
- **replay_lag** : Temps pour que les WAL soient rejoués (appliqués) sur le standby

**Valeurs acceptables :**
- **lag_bytes < 16 MB** : Excellent  
- **lag_bytes < 100 MB** : Acceptable  
- **lag_bytes > 1 GB** : Problème (réseau lent, standby sous-dimensionné, ou problème de performance)

**3. Vérifier l'état des slots de réplication :**

```sql
SELECT
    slot_name,
    slot_type,
    active,
    restart_lsn,
    pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) AS retained_bytes
FROM pg_replication_slots;
```

**Interprétation :**
- **active = true** : Le standby est connecté et utilise ce slot  
- **active = false** : Le slot existe mais aucun standby n'est connecté (⚠️ WAL s'accumulent)  
- **retained_bytes** : Quantité de WAL conservés pour ce slot

### Sur le serveur Standby

**1. Vérifier que le serveur est bien en mode recovery (standby) :**

```sql
SELECT pg_is_in_recovery();
```

**Résultat attendu :** `true`

Si `false`, le serveur pense être un primary (problème de configuration).

**2. Afficher les informations de réplication :**

```sql
SELECT
    now() - pg_last_xact_replay_timestamp() AS replication_lag_time,
    pg_last_wal_receive_lsn(),
    pg_last_wal_replay_lsn(),
    pg_last_xact_replay_timestamp();
```

**Explications :**
- **replication_lag_time** : Temps écoulé depuis la dernière transaction rejouée (en secondes)  
- **pg_last_wal_receive_lsn()** : Dernière position WAL reçue du primary  
- **pg_last_wal_replay_lsn()** : Dernière position WAL rejouée localement  
- **pg_last_xact_replay_timestamp()** : Timestamp de la dernière transaction rejouée

**Valeurs attendues :**
- **replication_lag_time < 5 secondes** : Excellent  
- **replication_lag_time < 30 secondes** : Acceptable  
- **replication_lag_time > 60 secondes** : Problème

**3. Tester une connexion en lecture seule :**

```sql
-- Cette requête doit fonctionner
SELECT count(*) FROM ma_table;

-- Cette requête doit échouer avec une erreur "cannot execute INSERT in a read-only transaction"
INSERT INTO ma_table VALUES (1);
```

### Tests de bout en bout

**Scénario de test complet :**

**Sur le Primary :**
```sql
-- Créer une table de test
CREATE TABLE test_replication (
    id SERIAL PRIMARY KEY,
    created_at TIMESTAMP DEFAULT now(),
    message TEXT
);

-- Insérer une ligne
INSERT INTO test_replication (message) VALUES ('Test depuis le primary');
```

**Sur le Standby (attendre 1-2 secondes) :**
```sql
-- Vérifier que la ligne est visible
SELECT * FROM test_replication;
```

**Résultat attendu :** La ligne insérée sur le primary doit apparaître sur le standby quasi instantanément (< 1 seconde).

## Fichiers de configuration : pgpass pour sécuriser les mots de passe

Plutôt que de stocker le mot de passe en clair dans `primary_conninfo`, vous pouvez utiliser un fichier `.pgpass`.

**Sur le serveur Standby, créer `/var/lib/postgresql/.pgpass` :**

```
# Format: hostname:port:database:username:password
192.168.1.10:5432:replication:replicator:mot_de_passe_fort
```

**Configurer les permissions (obligatoire) :**
```bash
chmod 600 /var/lib/postgresql/.pgpass  
chown postgres:postgres /var/lib/postgresql/.pgpass  
```

**Modifier `primary_conninfo` pour ne plus inclure le mot de passe :**
```ini
primary_conninfo = 'host=192.168.1.10 port=5432 user=replicator application_name=standby1'
```

PostgreSQL lira automatiquement le mot de passe depuis `.pgpass`.

## Problèmes courants et dépannage

### Problème 1 : "FATAL: no pg_hba.conf entry for replication"

**Symptôme :** Le standby ne peut pas se connecter au primary.

**Cause :** Le fichier `pg_hba.conf` du primary n'autorise pas la connexion de réplication depuis l'IP du standby.

**Solution :**
1. Vérifier l'adresse IP réelle du standby : `ip addr show` ou `hostname -I`  
2. Ajouter une ligne dans `pg_hba.conf` du primary :
   ```
   host    replication     replicator      <IP_STANDBY>/32         scram-sha-256
   ```
3. Recharger la configuration : `pg_ctl reload` ou `systemctl reload postgresql`

### Problème 2 : "FATAL: could not connect to the primary server"

**Symptôme :** Le standby ne peut pas joindre le primary via le réseau.

**Causes possibles :**
- Firewall bloquant le port 5432
- Adresse IP incorrecte dans `primary_conninfo`
- Le primary n'est pas démarré
- Problème DNS

**Solutions :**
1. Tester la connectivité réseau : `telnet 192.168.1.10 5432` ou `nc -zv 192.168.1.10 5432`  
2. Vérifier les règles firewall :
   ```bash
   # Sur le primary
   sudo firewall-cmd --list-all  # (Fedora/CentOS)
   sudo ufw status               # (Ubuntu/Debian)
   ```
3. Autoriser le port 5432 :
   ```bash
   # Firewalld (CentOS/Fedora)
   sudo firewall-cmd --permanent --add-service=postgresql
   sudo firewall-cmd --reload

   # UFW (Ubuntu/Debian)
   sudo ufw allow from 192.168.1.20 to any port 5432
   ```

### Problème 3 : Lag de réplication élevé (> 100 MB)

**Symptôme :** Le standby accumule un retard important par rapport au primary.

**Causes possibles :**
- Réseau lent ou saturé
- Standby sous-dimensionné (CPU, RAM, I/O)
- Volume d'écritures très élevé sur le primary
- Le standby est occupé par des requêtes en lecture lourdes

**Diagnostic :**
```sql
-- Sur le primary
SELECT
    application_name,
    pg_wal_lsn_diff(sent_lsn, replay_lsn) / 1024 / 1024 AS lag_mb,
    write_lag,
    flush_lag,
    replay_lag
FROM pg_stat_replication;
```

**Solutions :**
1. **Réseau :** Vérifier la bande passante et la latence
   ```bash
   # Tester la bande passante avec iperf3
   # Sur le standby :
   iperf3 -s
   # Sur le primary :
   iperf3 -c 192.168.1.20
   ```

2. **Standby sous-dimensionné :**
   - Augmenter `max_wal_size` sur le primary
   - Augmenter `maintenance_work_mem` sur le standby
   - Améliorer les I/O disques (SSD, RAID)

3. **Requêtes en lecture sur le standby :**
   - Augmenter `max_standby_streaming_delay`
   - Créer un deuxième standby dédié aux lectures
   - Utiliser un pool de connexion pour limiter les connexions

### Problème 4 : "requested WAL segment has already been removed"

**Symptôme :** Le standby ne peut pas rejoindre le primary car les WAL nécessaires ont été supprimés.

**Cause :** Le standby était déconnecté trop longtemps et les WAL ont été recyclés sur le primary.

**Solutions :**

**Option 1 : Utiliser un slot de réplication (prévention)**
```sql
-- Sur le primary
SELECT pg_create_physical_replication_slot('standby_slot_1');
```

**Option 2 : Reconstruire le standby**
Si le problème est déjà arrivé, il faut refaire un `pg_basebackup` complet.

**Option 3 : Restaurer depuis l'archive WAL (si configuré)**
Si vous avez configuré `archive_command` et `restore_command`, le standby peut récupérer les WAL manquants depuis l'archive.

### Problème 5 : Le slot de réplication accumule des WAL (disque plein)

**Symptôme :** Le répertoire `pg_wal/` sur le primary grandit indéfiniment.

**Cause :** Un slot de réplication existe mais le standby n'est pas connecté (ou ne progresse pas).

**Diagnostic :**
```sql
SELECT
    slot_name,
    active,
    pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) / 1024 / 1024 AS wal_retained_mb
FROM pg_replication_slots;
```

Si `active = false` et `wal_retained_mb` est élevé (> 10 GB), c'est un problème.

**Solution temporaire :** Supprimer le slot (attention : cela empêchera le standby de se reconnecter)
```sql
SELECT pg_drop_replication_slot('standby_slot_1');
```

**Solution permanente :**
1. Diagnostiquer pourquoi le standby ne se connecte pas  
2. Reconstruire le standby avec un nouveau basebackup  
3. Recréer le slot

## Bonnes pratiques de production

### 1. Monitoring et alerting

**Métriques critiques à surveiller :**
- **Lag de réplication** (objectif : < 10 secondes)  
- **État de connexion** (pg_stat_replication : state = 'streaming')  
- **Utilisation disque des WAL** (pg_wal/ sur le primary)  
- **État des slots** (active = true)

**Outils recommandés :**
- **pg_stat_statements** : Extension de monitoring des requêtes  
- **Prometheus + postgres_exporter** : Collecte de métriques  
- **Grafana** : Visualisation de tableaux de bord  
- **pgBadger** : Analyse de logs  
- **Check_postgres** : Scripts Nagios/Icinga pour monitoring

**Alertes à configurer :**
- Lag de réplication > 30 secondes
- État de connexion != 'streaming' pendant > 5 minutes
- Disque WAL > 80% de capacité
- Slot de réplication inactif

### 2. Sauvegardes régulières

La réplication n'est **pas** une sauvegarde !

**Raisons :**
- Une suppression accidentelle (DROP TABLE) sera répliquée instantanément sur tous les standbys
- Corruption de données sera propagée
- Pas de point de restauration dans le temps (sauf avec PITR)

**Stratégie recommandée :**
- Sauvegardes logiques quotidiennes (pg_dump)
- Sauvegardes physiques hebdomadaires (pg_basebackup)
- Archivage des WAL (pour PITR)
- Tests de restauration mensuels
- Stockage hors site (autre région, cloud)

### 3. Tests de failover

Planifier des tests réguliers de promotion du standby :

**Procédure de test (en environnement non-production) :**
1. Arrêter le primary : `systemctl stop postgresql`  
2. Promouvoir le standby : `pg_ctl promote -D /var/lib/postgresql/18/main`  
3. Vérifier que le standby accepte les écritures  
4. Reconfigurer l'ancien primary en nouveau standby  
5. Documenter les problèmes rencontrés

**Fréquence recommandée :** Trimestrielle (tous les 3 mois)

### 4. Documentation

Maintenir une documentation à jour :
- Architecture réseau (IPs, ports, DNS)
- Procédures de failover
- Contacts d'escalade
- Runbooks pour incidents courants
- Historique des changements

### 5. Sécurité

**Checkliste de sécurité :**
- ✅ Utiliser SSL/TLS pour les connexions de réplication  
- ✅ Authentification SCRAM-SHA-256 (pas MD5)  
- ✅ Firewall limitant les IPs autorisées  
- ✅ Mots de passe forts (20+ caractères)  
- ✅ Fichier .pgpass avec permissions 600  
- ✅ Principe du moindre privilège (utilisateur dédié à la réplication)  
- ✅ Logs d'audit activés  
- ✅ Surveillance des tentatives de connexion échouées

### 6. Dimensionnement

**Règles générales :**
- **Standby = Primary** (configuration matérielle identique)  
- **Bande passante réseau** : Au moins 10 Mbps dédié, idéalement 100 Mbps+  
- **Latence réseau** : < 10ms pour réplication synchrone, < 50ms pour asynchrone  
- **Disque WAL** : 50-100 Go sur le primary (sécurité)

## Résumé : Checklist de mise en place

### Sur le Primary

- [ ] Créer l'utilisateur de réplication : `CREATE ROLE replicator WITH REPLICATION LOGIN PASSWORD '...'`  
- [ ] Configurer `postgresql.conf` :  
  - [ ] `wal_level = replica`  
  - [ ] `max_wal_senders = 5`  
  - [ ] `wal_keep_size = 1GB`  
  - [ ] `max_replication_slots = 5`  
- [ ] Configurer `pg_hba.conf` : ajouter ligne pour connexion de réplication  
- [ ] Créer un slot de réplication : `SELECT pg_create_physical_replication_slot('standby_slot_1')`  
- [ ] Redémarrer PostgreSQL : `systemctl restart postgresql`  
- [ ] Vérifier la configuration : `SHOW wal_level;`

### Sur le Standby

- [ ] Arrêter PostgreSQL (si en cours) : `systemctl stop postgresql`  
- [ ] Effectuer un basebackup : `pg_basebackup -h <PRIMARY_IP> -D <DATA_DIR> -U replicator -P -v -R -X stream -S standby_slot_1`  
- [ ] Configurer `postgresql.conf` :  
  - [ ] `hot_standby = on`  
  - [ ] `primary_conninfo = '...'`  
  - [ ] `primary_slot_name = 'standby_slot_1'`  
- [ ] Vérifier que `standby.signal` existe  
- [ ] Ajuster les permissions : `chown -R postgres:postgres <DATA_DIR> && chmod 700 <DATA_DIR>`  
- [ ] Démarrer PostgreSQL : `systemctl start postgresql`  
- [ ] Vérifier : `SELECT pg_is_in_recovery();` (doit retourner `true`)

### Vérifications finales

- [ ] Sur le primary : `SELECT * FROM pg_stat_replication;` (doit montrer le standby connecté)  
- [ ] Sur le primary : Insérer une ligne de test  
- [ ] Sur le standby : Vérifier que la ligne apparaît  
- [ ] Vérifier le lag : `SELECT pg_wal_lsn_diff(sent_lsn, replay_lsn) FROM pg_stat_replication;`

## Conclusion

La configuration Primary/Standby est la pierre angulaire de la haute disponibilité dans PostgreSQL. Cette architecture offre :

- ✅ **Redondance** : Protection contre les pannes matérielles  
- ✅ **Répartition de charge** : Lectures sur le standby  
- ✅ **Reprise rapide** : Promotion en quelques secondes  
- ✅ **Flexibilité** : Supporte plusieurs standbys et cascading replication

**Points clés à retenir :**
- La réplication physique copie les WAL (Write-Ahead Logs) du primary au standby
- Le standby est une copie bit-à-bit du primary, maintenue à jour en temps réel
- Les slots de réplication garantissent la conservation des WAL
- Le monitoring du lag est crucial pour détecter les problèmes
- La réplication n'est pas une sauvegarde : continuez à faire des backups !

**Prochaines étapes :**
- Configurer une réplication synchrone (17.2.2)
- Mettre en place le cascading replication (17.2.3)
- Automatiser le failover avec Patroni
- Explorer la réplication logique pour des cas d'usage spécifiques

La maîtrise de cette architecture est essentielle pour tout administrateur ou développeur PostgreSQL travaillant sur des systèmes critiques en production.
