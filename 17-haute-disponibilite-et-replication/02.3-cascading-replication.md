# 17.2.3. Cascading Replication - Réplication en Cascade PostgreSQL 18

## Introduction

La **réplication en cascade** (Cascading Replication) est une fonctionnalité avancée de PostgreSQL qui permet à un serveur standby de servir lui-même de source de réplication pour d'autres standbys. Autrement dit, au lieu que tous les standbys se connectent directement au primary, certains standbys peuvent se connecter à d'autres standbys pour recevoir les WAL.

Cette architecture permet de réduire la charge sur le serveur primary, d'améliorer la scalabilité, et de construire des topologies de réplication géographiquement distribuées plus efficaces.

### Concept de base

**Architecture traditionnelle (sans cascade) :**
```
Primary
 ├─> Standby 1
 ├─> Standby 2
 └─> Standby 3
```

**Architecture en cascade :**
```
Primary
 ├─> Standby 1
 └─> Standby 2
      └─> Standby 3
```

Dans le second cas, `Standby 3` reçoit les WAL de `Standby 2` au lieu du `Primary`. On dit que `Standby 2` est un **standby intermédiaire** et `Standby 3` est un **standby en cascade**.

---

## Pourquoi Utiliser la Réplication en Cascade ?

### 1. Réduction de la charge sur le Primary

**Problème :** Chaque standby connecté au primary consomme des ressources :
- Un processus WAL sender par standby
- Bande passante réseau
- CPU pour l'encodage et l'envoi des WAL
- Mémoire pour maintenir les buffers de réplication

**Solution avec la cascade :**
Si vous avez 10 standbys, au lieu de les connecter tous au primary, vous pouvez avoir :
- Primary → 2 standbys intermédiaires
- Chaque standby intermédiaire → 4 standbys en cascade

**Avantages :**
- Le primary ne gère que 2 connexions WAL sender (au lieu de 10)
- Réduction de 80% de la charge réseau sur le primary
- CPU et mémoire libérés pour traiter les requêtes applicatives

### 2. Réplication géographiquement distribuée

**Scénario :** Vous avez des bureaux dans plusieurs régions et souhaitez un standby dans chaque région.

**Architecture sans cascade :**
```
Primary (Paris)
 ├─> Standby (Londres)    [Latence: 10ms, 2000 km]
 ├─> Standby (New York)   [Latence: 80ms, 6000 km]
 ├─> Standby (Tokyo)      [Latence: 200ms, 10000 km]
 └─> Standby (Sydney)     [Latence: 250ms, 17000 km]
```

**Problèmes :**
- Latence élevée pour les standbys distants
- Coûts de bande passante inter-continentale élevés
- Le primary doit envoyer les WAL 4 fois sur de longues distances

**Architecture avec cascade :**
```
Primary (Paris)
 ├─> Standby (Londres)    [Latence: 10ms]
 └─> Standby (New York)   [Latence: 80ms]
      ├─> Standby (Tokyo)      [Latence depuis NY: 150ms]
      └─> Standby (Sydney)     [Latence depuis NY: 180ms]
```

**Avantages :**
- Le primary n'envoie les WAL qu'à 2 destinations (Europe et Amérique)
- Les standbys asiatiques reçoivent les WAL depuis New York (plus proche)
- Réduction des coûts de bande passante intercontinentale
- Meilleure répartition de la charge réseau

### 3. Hiérarchie de criticité

**Scénario :** Vous avez des standbys avec différents niveaux de criticité.

**Architecture :**
```
Primary (Production)
 ├─> Standby Hot (HA immédiate)     [Criticité: Haute]
 └─> Standby Warm (HA secondaire)   [Criticité: Moyenne]
      ├─> Standby Reporting          [Criticité: Faible - Lectures]
      └─> Standby Dev/Test            [Criticité: Faible - Développement]
```

**Avantages :**
- Les standbys critiques (HA) reçoivent les WAL directement du primary (latence minimale)
- Les standbys non critiques (reporting, dev) sont en cascade (ne surchargent pas le primary)
- En cas de panne du primary, seuls les standbys directs sont candidats à la promotion

### 4. Limitation de bande passante

**Scénario :** Le primary a une bande passante réseau limitée (ex: 100 Mbps).

**Calcul de bande passante :**
- Volume de WAL généré : 50 MB/s
- Avec 5 standbys : 5 × 50 MB/s = 250 MB/s requis
- Bande passante disponible : 100 MB/s → **Saturation !**

**Solution avec cascade :**
```
Primary (50 MB/s WAL)
 ├─> Standby 1 (50 MB/s) → 2 standbys en cascade
 └─> Standby 2 (50 MB/s) → 2 standbys en cascade
```

Total utilisé sur le primary : 100 MB/s (au lieu de 250 MB/s)

## Architecture et Fonctionnement

### Architecture de base en cascade

```
┌───────────────────────┐
│   Primary             │
│   (Read/Write)        │
│                       │
│   ┌───────────────┐   │
│   │ WAL Sender 1  │   │
│   └───────┬───────┘   │
└───────────┼───────────┘
            │
            │ Stream WAL (50 MB/s)
            ↓
┌───────────────────────┐
│   Standby Intermediate│
│   (Read-Only)         │
│                       │
│   ┌───────────────┐   │
│   │ WAL Receiver  │   │  ← Reçoit du Primary
│   └───────┬───────┘   │
│           ↓           │
│   ┌───────────────┐   │
│   │ Startup Proc  │   │  ← Applique les WAL
│   └───────┬───────┘   │
│           ↓           │
│   ┌───────────────┐   │
│   │ WAL Sender 2  │   │  ← Renvoie aux cascades
│   └───────┬───────┘   │
└───────────┼───────────┘
            │
            │ Stream WAL (50 MB/s)
            ↓
┌───────────────────────┐
│   Standby Cascade     │
│   (Read-Only)         │
│                       │
│   ┌───────────────┐   │
│   │ WAL Receiver  │   │  ← Reçoit du Standby
│   └───────┬───────┘   │
│           ↓           │
│   ┌───────────────┐   │
│   │ Startup Proc  │   │  ← Applique les WAL
│   └───────────────┘   │
└───────────────────────┘
```

### Flux de données détaillé

**Étape par étape :**

1. **Primary génère des WAL**
   - Les transactions sont validées sur le primary
   - Les WAL sont écrits dans `pg_wal/`

2. **Primary → Standby Intermédiaire**
   - Le WAL sender du primary envoie les WAL au standby intermédiaire
   - Le standby intermédiaire les reçoit via son WAL receiver
   - Le standby intermédiaire écrit ces WAL dans son propre `pg_wal/`

3. **Standby Intermédiaire applique les WAL**
   - Le processus de startup rejoue les WAL
   - La base de données du standby intermédiaire est mise à jour

4. **Standby Intermédiaire → Standby Cascade**  
   - **Important :** Le standby intermédiaire agit comme un "mini-primary" pour les standbys en cascade
   - Son WAL sender envoie les WAL (qu'il a reçus du primary) aux standbys en cascade
   - Les standbys en cascade reçoivent et appliquent ces WAL

**Points clés à comprendre :**
- Le standby intermédiaire **ne modifie pas** les WAL, il les retransmet
- Les standbys en cascade reçoivent exactement les mêmes WAL que ceux générés par le primary
- Il peut y avoir un **délai cumulatif** : le standby cascade est légèrement plus en retard que le standby intermédiaire

### Délai de réplication (lag)

**Sans cascade :**
```
Primary → Standby : Lag = 0.01s
```

**Avec cascade :**
```
Primary → Standby Intermédiaire : Lag 1 = 0.01s  
Standby Intermédiaire → Standby Cascade : Lag 2 = 0.01s  
Total : 0.02s  
```

Le lag se **cumule** à chaque niveau de cascade. Pour une architecture à 3 niveaux :
```
Primary → Standby1 → Standby2 → Standby3  
Lag total = Lag1 + Lag2 + Lag3  
```

**Recommandation :** Limiter à 2-3 niveaux maximum pour éviter un lag excessif.

## Configuration de la Réplication en Cascade

### Prérequis

Avant de configurer une réplication en cascade, vous devez avoir :
- ✅ Un serveur Primary fonctionnel  
- ✅ Au moins un Standby Intermédiaire déjà configuré et opérationnel  
- ✅ Comprendre les bases de la réplication Primary/Standby (voir chapitre 17.2.1)

### Étape 1 : Configurer le Standby Intermédiaire

Le standby intermédiaire doit être configuré pour **accepter des connexions de réplication** des standbys en cascade.

#### 1.1. Configuration de postgresql.conf sur le Standby Intermédiaire

**Éditer `/var/lib/postgresql/18/main/postgresql.conf` :**

```ini
# ────────────────────────────────────────────────
# Configuration pour accepter les cascades
# ────────────────────────────────────────────────

# Nombre maximum de connexions WAL sender
# Doit être > 0 pour permettre les cascades
max_wal_senders = 5

# Niveau WAL (hérité du primary, mais on le confirme)
wal_level = replica

# Conserver suffisamment de WAL pour les cascades
wal_keep_size = 1GB

# Slots de réplication pour les cascades (recommandé)
max_replication_slots = 5

# Hot standby doit être activé
hot_standby = on
```

**Explications :**

**`max_wal_senders = 5`**
- **Crucial !** Un standby avec `max_wal_senders = 0` ne peut pas servir de source de réplication
- Nombre de standbys en cascade qui peuvent se connecter à ce standby intermédiaire
- Valeur typique : nombre de cascades prévues + 2-3 pour les outils de monitoring

**`wal_keep_size = 1GB`**
- Garantit que les WAL nécessaires aux cascades sont conservés
- Important si un standby cascade se déconnecte temporairement
- Dimensionner selon le volume d'écritures et la tolérance aux déconnexions

**`max_replication_slots = 5`**
- Permet de créer des slots de réplication pour les standbys en cascade
- Garantit que les WAL ne seront pas supprimés tant que les cascades ne les ont pas reçus
- **Recommandé** pour éviter les erreurs "WAL segment has been removed"

#### 1.2. Configuration de pg_hba.conf sur le Standby Intermédiaire

Le standby intermédiaire doit autoriser les connexions de réplication des standbys en cascade.

**Éditer `/var/lib/postgresql/18/main/pg_hba.conf` :**

```
# TYPE  DATABASE        USER            ADDRESS                 METHOD

# Autoriser la réplication depuis les standbys en cascade
# Remplacer par les IPs réelles de vos standbys cascade
host    replication     replicator      192.168.1.30/32         scram-sha-256  
host    replication     replicator      192.168.1.31/32         scram-sha-256  

# Ou pour un sous-réseau entier (moins sécurisé)
host    replication     replicator      192.168.1.0/24          scram-sha-256
```

**Note :** C'est le même utilisateur de réplication (`replicator`) que celui utilisé par le primary.

#### 1.3. Créer des slots de réplication sur le Standby Intermédiaire (recommandé)

**Se connecter au standby intermédiaire :**

```sql
-- Créer un slot pour chaque standby cascade
SELECT pg_create_physical_replication_slot('cascade_standby1_slot');  
SELECT pg_create_physical_replication_slot('cascade_standby2_slot');  
```

**Vérification :**
```sql
SELECT slot_name, active, restart_lsn  
FROM pg_replication_slots;  
```

#### 1.4. Recharger/Redémarrer le Standby Intermédiaire

```bash
# Recharger la configuration (pour pg_hba.conf)
sudo systemctl reload postgresql

# Si max_wal_senders a été modifié, redémarrage nécessaire
sudo systemctl restart postgresql
```

**Vérification :**
```sql
-- Se connecter au standby intermédiaire
SHOW max_wal_senders;  -- Doit être > 0
```

### Étape 2 : Créer le Standby en Cascade

Le standby en cascade se configure exactement comme un standby normal, sauf qu'il se connecte au standby intermédiaire au lieu du primary.

#### 2.1. Effectuer un Base Backup depuis le Standby Intermédiaire

**Sur le serveur du futur standby cascade :**

```bash
# Arrêter PostgreSQL (si déjà en cours)
sudo systemctl stop postgresql

# Sauvegarder l'ancien répertoire (précaution)
sudo mv /var/lib/postgresql/18/main /var/lib/postgresql/18/main.old

# Effectuer un basebackup DEPUIS le standby intermédiaire
pg_basebackup \
    -h 192.168.1.20 \                      # IP du STANDBY INTERMÉDIAIRE (pas du primary)
    -D /var/lib/postgresql/18/main \       # Répertoire de destination
    -U replicator \                        # Utilisateur de réplication
    -P \                                   # Afficher la progression
    -v \                                   # Mode verbose
    -R \                                   # Créer automatiquement standby.signal
    -X stream \                            # Inclure les WAL nécessaires
    -C \                                   # Créer le slot automatiquement
    -S cascade_standby1_slot               # Nom du slot (doit exister sur le standby intermédiaire)
```

**Point crucial :** L'option `-h` pointe vers le **standby intermédiaire**, pas vers le primary.

#### 2.2. Configurer postgresql.conf sur le Standby Cascade

**Éditer `/var/lib/postgresql/18/main/postgresql.conf` :**

```ini
# ────────────────────────────────────────────────
# Configuration Hot Standby
# ────────────────────────────────────────────────

hot_standby = on  
hot_standby_feedback = on  

# ────────────────────────────────────────────────
# Configuration de connexion au Standby Intermédiaire
# ────────────────────────────────────────────────

# Connexion au STANDBY INTERMÉDIAIRE (pas au primary !)
primary_conninfo = 'host=192.168.1.20 port=5432 user=replicator password=*** application_name=cascade_standby1'

# Utiliser le slot créé sur le standby intermédiaire
primary_slot_name = 'cascade_standby1_slot'

# ────────────────────────────────────────────────
# Options additionnelles
# ────────────────────────────────────────────────

# Commande de restauration WAL (optionnel)
restore_command = 'cp /archive/pg_wal/%f %p'

# Fichier trigger pour promotion
promote_trigger_file = '/tmp/postgresql.trigger.5432'
```

**Points importants :**

**`primary_conninfo`**
- **host=192.168.1.20** : C'est l'IP du **standby intermédiaire**, pas du primary  
- **application_name=cascade_standby1** : Nom pour identifier ce standby dans les vues de monitoring

**`primary_slot_name`**
- Doit correspondre au slot créé sur le standby intermédiaire
- Garantit que le standby intermédiaire conservera les WAL nécessaires

#### 2.3. Vérifier le fichier standby.signal

Si vous avez utilisé l'option `-R` avec `pg_basebackup`, ce fichier est créé automatiquement :

```bash
ls -la /var/lib/postgresql/18/main/standby.signal
```

Sinon, créez-le :
```bash
touch /var/lib/postgresql/18/main/standby.signal
```

#### 2.4. Ajuster les permissions

```bash
sudo chown -R postgres:postgres /var/lib/postgresql/18/main  
sudo chmod 700 /var/lib/postgresql/18/main  
```

#### 2.5. Démarrer le Standby Cascade

```bash
sudo systemctl start postgresql  
sudo systemctl status postgresql  
```

    application_name,

## Topologies Avancées

### Cascade à 2 niveaux

**Architecture :**
```
Primary
 ├─> Standby A
 │    ├─> Standby A1 (cascade)
 │    └─> Standby A2 (cascade)
 └─> Standby B
      ├─> Standby B1 (cascade)
      └─> Standby B2 (cascade)
```

**Cas d'usage :**
- Répartition géographique : 2 régions avec plusieurs standbys par région
- Répartition de charge : Standby A pour les lectures OLTP, Standby B pour les lectures analytics

**Configuration :**
- Primary : `max_wal_senders = 5` (pour A et B + réserve)
- Standby A : `max_wal_senders = 5` (pour A1 et A2 + réserve)
- Standby B : `max_wal_senders = 5` (pour B1 et B2 + réserve)

### Cascade à 3 niveaux

**Architecture :**
```
Primary (Paris)
 └─> Standby 1 (Londres)
      └─> Standby 2 (New York)
           └─> Standby 3 (Tokyo)
```

**Avantages :**
- Réduction maximale de la charge réseau sur le primary
- Adapté aux architectures vraiment globales

**Inconvénients :**
- Lag cumulatif élevé (peut atteindre 500ms ou plus)
- Complexité de troubleshooting accrue
- Risque en cascade : si Standby 2 tombe, Standby 3 est déconnecté

**Recommandation :** À utiliser uniquement si absolument nécessaire. Privilégier 2 niveaux maximum.

### Cascade mixte (synchrone + asynchrone)

**Architecture :**
```
Primary
 ├─> Standby Hot (synchrone)
 └─> Standby Warm (asynchrone)
      ├─> Standby Reporting (cascade)
      └─> Standby Dev/Test (cascade)
```

**Configuration sur le Primary :**
```ini
synchronous_standby_names = 'standby_hot'
```

**Avantages :**
- Zero data loss pour le standby Hot (synchrone)
- Standbys non critiques (reporting, dev) ne surchargent pas le primary
- Isolation des charges : lectures intensives sur les cascades

**Cas d'usage typique :** Production avec environnements de reporting et développement.

### Multi-datacenter avec cascade

**Architecture :**
```
Datacenter Paris (Primary)
    ├─> Standby Local 1 (Paris)
    └─> Standby Local 2 (Paris)

Datacenter Londres
    └─> Standby Intermédiaire (Londres)
         ├─> Standby Cascade 1 (Londres)
         └─> Standby Cascade 2 (Londres)

Datacenter New York
    └─> Standby Intermédiaire (New York)
         ├─> Standby Cascade 1 (New York)
         └─> Standby Cascade 2 (New York)
```

**Flux de données :**
- Primary (Paris) → Standby Intermédiaires (Londres, New York)
- Chaque standby intermédiaire alimente ses cascades locales

**Avantages :**
- Réduction drastique du trafic inter-datacenter
- Chaque région a plusieurs standbys pour la HA locale
- Coûts de bande passante optimisés

**Configuration du trafic :**
- Primary → Londres : 50 MB/s
- Primary → New York : 50 MB/s
- Total sur le primary : 100 MB/s (au lieu de 50 MB/s × 6 standbys = 300 MB/s)

## Monitoring et Observabilité

### Vue d'ensemble de la topologie

Pour visualiser la topologie complète de réplication :

```sql
-- Sur n'importe quel serveur
WITH RECURSIVE replication_tree AS (
    -- Niveau 0 : Primary
    SELECT
        NULL::text AS upstream_host,
        inet_server_addr()::text AS host,
        current_database() AS database,
        0 AS level
    WHERE NOT pg_is_in_recovery()

    UNION ALL

    -- Niveaux suivants : Standbys
    SELECT
        regexp_replace(conninfo, '.*host=([^ ]+).*', '\1') AS upstream_host,
        application_name AS host,
        current_database() AS database,
        rt.level + 1 AS level
    FROM pg_stat_replication
    JOIN replication_tree rt ON true
)
SELECT
    level,
    REPEAT('  ', level) || host AS hierarchy,
    upstream_host
FROM replication_tree  
ORDER BY level, host;  
```

**Exemple de résultat :**
```
 level | hierarchy               | upstream_host
-------+-------------------------+---------------
 0     | primary-db              | NULL
 1     |   standby-intermediate  | primary-db
 2     |     cascade-standby1    | standby-intermediate
 2     |     cascade-standby2    | standby-intermediate
```

### Monitoring du lag par niveau

**Sur le Primary :**
```sql
SELECT
    application_name,
    client_addr,
    state,
    pg_wal_lsn_diff(sent_lsn, replay_lsn) / 1024 / 1024 AS lag_mb,
    write_lag,
    flush_lag,
    replay_lag
FROM pg_stat_replication  
ORDER BY application_name;  
```

**Sur les Standbys Intermédiaires :**
```sql
-- Lag depuis le primary
SELECT
    now() - pg_last_xact_replay_timestamp() AS replication_lag_time
FROM pg_stat_database LIMIT 1;

-- Standbys en cascade connectés
SELECT
    application_name,
    client_addr,
    state,
    pg_wal_lsn_diff(sent_lsn, replay_lsn) / 1024 / 1024 AS lag_mb
FROM pg_stat_replication;
```

### Métriques à surveiller

**1. Lag cumulatif total**

Pour calculer le lag total d'un standby cascade :
```
Lag Total = Lag(Primary → Intermédiaire) + Lag(Intermédiaire → Cascade)
```

**Seuils recommandés :**
- Lag niveau 1 (Primary → Intermédiaire) : < 10 secondes
- Lag niveau 2 (Intermédiaire → Cascade) : < 10 secondes
- **Lag total : < 20 secondes**

**2. État des connexions**

```sql
-- Sur chaque niveau, vérifier que state = 'streaming'
SELECT application_name, state FROM pg_stat_replication;
```

**3. Saturation des WAL senders**

```sql
-- Vérifier qu'on ne dépasse pas max_wal_senders
SELECT
    count(*) AS active_senders,
    current_setting('max_wal_senders')::int AS max_senders
FROM pg_stat_replication;
```

Si `active_senders` approche `max_senders`, augmenter `max_wal_senders`.

**4. Accumulation de WAL**

```sql
-- Sur les standbys intermédiaires
SELECT
    slot_name,
    active,
    pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) / 1024 / 1024 AS retained_mb
FROM pg_replication_slots;
```

Si `retained_mb` > 10 GB et `active = false`, un standby cascade est déconnecté depuis longtemps.

### Alertes recommandées

**Alertes critiques :**
- Aucun standby en cascade connecté à un standby intermédiaire (alors que prévu)
- Lag total > 60 secondes sur un standby cascade
- État de réplication != 'streaming' pendant > 5 minutes
- Accumulation de WAL sur standby intermédiaire > 20 GB

**Alertes warning :**
- Lag > 20 secondes sur n'importe quel niveau
- Utilisation de `max_wal_senders` > 80%

## Cas d'Usage Détaillés

### Cas 1 : E-commerce avec Plusieurs Environnements

**Contexte :**
- Production : 1 primary + 1 standby HA
- Reporting : 2 standbys pour BI et analytics
- Développement : 2 standbys pour dev/test

**Architecture :**
```
Primary (Production)
 ├─> Standby HA (Production)
 └─> Standby Reporting (Production)
      ├─> Standby BI (Cascade - lectures intensives)
      └─> Standby Analytics (Cascade - requêtes lourdes)
```

**Configuration :**
```ini
# Primary
max_wal_senders = 5  
synchronous_standby_names = 'standby_ha'  

# Standby Reporting (intermédiaire)
max_wal_senders = 5
```

**Avantages :**
- Standby HA protège la production (sync)
- Standbys reporting/analytics ne surchargent pas le primary
- Isolation des requêtes lourdes (analytics) qui n'impactent pas la production

### Cas 2 : SaaS Multi-région avec DR

**Contexte :**
- 3 régions géographiques (EU, US, ASIA)
- Chaque région nécessite 2 standbys (HA + lectures)
- Disaster Recovery cross-region

**Architecture :**
```
Primary (EU - Paris)
 ├─> Standby HA (EU - Paris)
 ├─> Standby Intermédiaire (US - New York)
 │    ├─> Standby HA (US - New York)
 │    └─> Standby Read (US - New York)
 └─> Standby Intermédiaire (ASIA - Singapore)
      ├─> Standby HA (ASIA - Singapore)
      └─> Standby Read (ASIA - Singapore)
```

**Flux de données :**
- Primary → 3 destinations : EU local, US, ASIA
- Chaque région a 2 standbys locaux (1 HA, 1 read)

**Avantages :**
- Réplication géographique pour DR
- Chaque région peut lire localement (latence faible)
- Trafic inter-région optimisé (3 flux au lieu de 7)

**Coûts de bande passante :**
- Sans cascade : Primary → 7 standbys
- Avec cascade : Primary → 3 standbys intermédiaires

**Réduction de coûts :** ~60% sur la bande passante sortante du primary

### Cas 3 : Plateforme de Données avec Lacs de Données

**Contexte :**
- Base de données transactionnelle (OLTP)
- Besoin d'extraire les données vers un data lake
- Plusieurs pipelines ETL avec des charges différentes

**Architecture :**
```
Primary (OLTP)
 └─> Standby ETL (Intermédiaire)
      ├─> Standby Data Lake 1 (Cascade - export continu)
      ├─> Standby Data Lake 2 (Cascade - export batch)
      └─> Standby CDC (Cascade - Change Data Capture)
```

**Configuration :**
- Standby ETL : Dédié à l'extraction de données (pas de lectures utilisateur)
- Cascades : Chacune connectée via un outil spécifique (Debezium, Airbyte, etc.)

**Avantages :**
- Le primary n'est pas impacté par les pipelines ETL
- Isolation des différents processus d'extraction
- Scalabilité : ajout de nouveaux pipelines sans surcharge du primary

### Cas 4 : Banque avec Exigences Réglementaires

**Contexte :**
- Données critiques (zero data loss)
- Obligation de garder 3 copies des données
- Audit trail complet

**Architecture :**
```
Primary (Datacenter A)
 ├─> Standby Sync 1 (Datacenter A) [SYNC]
 ├─> Standby Sync 2 (Datacenter B) [SYNC]
 └─> Standby Intermediate (Datacenter C) [ASYNC]
      ├─> Standby Audit (Datacenter C) [CASCADE]
      └─> Standby Compliance (Datacenter C) [CASCADE]
```

**Configuration :**
```ini
# Primary
synchronous_standby_names = 'FIRST 2 (standby_sync1, standby_sync2)'
```

**Explication :**
- 2 standbys synchrones (zero data loss)
- 1 standby asynchrone + 2 cascades pour audit/compliance
- Total : 5 copies des données (conforme aux exigences)

**Avantages :**
- Zero data loss garanti (réplication synchrone)
- Conformité réglementaire (3+ copies)
- Standbys audit/compliance ne ralentissent pas le primary

## Scénarios de Panne et Gestion

### Scénario 1 : Le Standby Intermédiaire tombe

**Situation :**
```
Primary
 └─> Standby Intermediate [PANNE]
      ├─> Cascade 1 [DÉCONNECTÉ]
      └─> Cascade 2 [DÉCONNECTÉ]
```

**Conséquences :**
- Les standbys en cascade perdent leur source de réplication
- Ils restent opérationnels en lecture seule avec les données actuelles
- Lag croissant tant que le standby intermédiaire est down

**Actions à entreprendre :**

**Option 1 : Réparer le standby intermédiaire**
```bash
# Diagnostiquer le problème
sudo systemctl status postgresql  
sudo tail -f /var/log/postgresql/postgresql-18-main.log  

# Redémarrer si nécessaire
sudo systemctl restart postgresql
```

Les cascades se reconnecteront automatiquement.

**Option 2 : Reconfigurer temporairement les cascades vers le primary**

```bash
# Sur chaque standby cascade
sudo -u postgres psql

# Modifier temporairement la source
ALTER SYSTEM SET primary_conninfo = 'host=<PRIMARY_IP> port=5432 user=replicator password=*** application_name=cascade_standby1';

# Recharger
SELECT pg_reload_conf();
```

**Option 3 : Promouvoir un standby cascade en intermédiaire**

Si le standby intermédiaire est définitivement perdu, promouvoir un des standbys cascade :

```bash
# Sur Cascade 1 (futur intermédiaire)
pg_ctl promote -D /var/lib/postgresql/18/main
```

Puis reconfigurer Cascade 2 pour se connecter à Cascade 1.

### Scénario 2 : Le Primary tombe

**Situation :**
```
Primary [PANNE]
 └─> Standby Intermediate
      ├─> Cascade 1
      └─> Cascade 2
```

**Questions critiques :**
1. Quel serveur doit devenir le nouveau primary ?  
2. Les standbys en cascade doivent-ils être reconfigurés ?

**Réponse :** Promouvoir le **standby intermédiaire** en primary (pas un standby cascade).

**Raisons :**
- Le standby intermédiaire a le moins de lag par rapport à l'ancien primary
- Il est déjà configuré avec `max_wal_senders` et peut alimenter les cascades
- Transition transparente pour les standbys cascade (pas de reconfiguration nécessaire)

**Procédure de promotion :**

**1. Promouvoir le standby intermédiaire**
```bash
# Sur le standby intermédiaire
pg_ctl promote -D /var/lib/postgresql/18/main
```

**2. Vérifier qu'il accepte les écritures**
```sql
SELECT pg_is_in_recovery();  -- Doit retourner 'false'

-- Tester une insertion
INSERT INTO test_table VALUES (1, 'test');
```

**3. Reconfigurer les applications**
Pointer les applications vers le nouveau primary (ancien standby intermédiaire).

**4. Vérifier que les cascades sont toujours connectées**
```sql
-- Sur le nouveau primary
SELECT application_name, state FROM pg_stat_replication;
```

Les standbys cascade devraient toujours être connectés (aucune reconfiguration nécessaire).

**5. Reconfigurer l'ancien primary en standby (quand il revient)**

Si l'ancien primary revient, le transformer en standby :

```bash
# Sur l'ancien primary
sudo systemctl stop postgresql

# Sauvegarder les données actuelles
sudo mv /var/lib/postgresql/18/main /var/lib/postgresql/18/main.old

# Créer un nouveau basebackup depuis le nouveau primary
pg_basebackup -h <NOUVEAU_PRIMARY_IP> -D /var/lib/postgresql/18/main -U replicator -P -v -R -X stream

# Démarrer en tant que standby
sudo systemctl start postgresql
```

### Scénario 3 : Lag excessif sur les cascades

**Symptôme :** Le standby cascade a 5 minutes de retard.

**Diagnostic :**

**Sur le primary :**
```sql
SELECT
    application_name,
    pg_wal_lsn_diff(sent_lsn, replay_lsn) / 1024 / 1024 AS lag_mb
FROM pg_stat_replication;
```

**Sur le standby intermédiaire :**
```sql
SELECT
    application_name,
    pg_wal_lsn_diff(sent_lsn, replay_lsn) / 1024 / 1024 AS lag_mb
FROM pg_stat_replication;
```

**Causes possibles :**

**1. Réseau lent entre intermédiaire et cascade**
```bash
# Tester la latence
ping <CASCADE_IP>

# Tester la bande passante
iperf3 -s  # Sur le cascade  
iperf3 -c <CASCADE_IP>  # Sur l'intermédiaire  
```

**Solutions :**
- Upgrade du lien réseau
- Compression des WAL (paramètre `wal_compression`)
- Si impossible : accepter le lag ou supprimer la cascade

**2. Standby cascade sous-dimensionné (CPU, I/O)**

```sql
-- Sur le cascade, vérifier les I/O waits
SELECT
    wait_event_type,
    wait_event,
    count(*)
FROM pg_stat_activity  
WHERE state = 'active'  
GROUP BY wait_event_type, wait_event  
ORDER BY count(*) DESC;  
```

Si beaucoup de `wait_event_type = 'IO'`, le disque est saturé.

**Solutions :**
- Améliorer les I/O (SSD, augmenter IOPS)
- Augmenter `maintenance_work_mem`
- Réduire les lectures sur le cascade (redirections vers un autre standby)

**3. Volume de WAL très élevé sur le primary**

Si le primary génère 200 MB/s de WAL, les cascades peuvent avoir du mal à suivre.

**Solutions :**
- Augmenter `max_wal_size` sur le primary
- Optimiser les requêtes pour réduire les écritures
- Ajouter un standby intermédiaire supplémentaire pour répartir la charge

## Bonnes Pratiques

### 1. Limiter la profondeur de cascade à 2-3 niveaux maximum

**❌ À éviter :**
```
Primary → Standby1 → Standby2 → Standby3 → Standby4 → Standby5
```

**Problèmes :**
- Lag cumulatif très élevé
- Complexité de debugging
- Effet domino en cas de panne intermédiaire

**✅ Recommandé :**
```
Primary
 ├─> Standby1
 │    ├─> Standby1a
 │    └─> Standby1b
 └─> Standby2
      ├─> Standby2a
      └─> Standby2b
```

### 2. Utiliser des slots de réplication pour les cascades

Les slots garantissent que les WAL ne sont pas supprimés prématurément.

**Créer un slot pour chaque cascade :**
```sql
-- Sur le standby intermédiaire
SELECT pg_create_physical_replication_slot('cascade1_slot');  
SELECT pg_create_physical_replication_slot('cascade2_slot');  
```

**Configurer les cascades pour utiliser ces slots :**
```ini
# Sur chaque cascade
primary_slot_name = 'cascade1_slot'
```

### 3. Monitorer chaque niveau de réplication

Ne pas se contenter de monitorer le primary. Chaque niveau doit être surveillé :
- Primary → Intermédiaires
- Intermédiaires → Cascades

**Dashboard recommandé (Grafana) :**
- Graphique 1 : Lag du primary vers tous les standbys directs
- Graphique 2 : Lag de chaque intermédiaire vers ses cascades
- Graphique 3 : Lag total (cumulatif) de chaque cascade

### 4. Documenter la topologie

Maintenir un schéma à jour de votre topologie de réplication :

**Documentation minimale :**
- Schéma visuel (diagramme)
- Adresses IP de chaque serveur
- Rôle de chaque serveur (primary, intermédiaire, cascade)
- Configuration de `synchronous_standby_names`
- Procédures de failover spécifiques à votre topologie

### 5. Tester les scénarios de panne

**Tests trimestriels recommandés :**
- Panne du standby intermédiaire : vérifier que les cascades gèrent la déconnexion
- Panne du primary : promouvoir l'intermédiaire et vérifier que les cascades continuent
- Panne d'un cascade : vérifier qu'il rattrape son retard après redémarrage

### 6. Prévoir la bande passante

**Calcul de bande passante :**
```
Bande passante totale = Volume WAL/s × (Nombre de standbys directs + overhead)
```

**Exemple :**
- Primary génère 50 MB/s de WAL
- 2 standbys directs du primary
- Bande passante nécessaire : 50 MB/s × 2 = 100 MB/s

Si vous avez 100 Mbps disponibles, vous êtes limite. Avec la cascade, vous pouvez répartir :
- Primary → Intermédiaire 1 : 50 MB/s
- Primary → Intermédiaire 2 : 50 MB/s
- Intermédiaire 1 → 3 cascades : gérées localement
- Intermédiaire 2 → 3 cascades : gérées localement

### 7. Utiliser la cascade pour les environnements non-production

**Principe :** Isoler les environnements de dev/test/staging du primary de production.

**Architecture recommandée :**
```
Primary (Production)
 ├─> Standby HA (Production)
 └─> Standby Staging (Intermédiaire)
      ├─> Standby Dev
      ├─> Standby Test
      └─> Standby QA
```

**Avantages :**
- Production isolée des charges non-production
- Facile d'ajouter/supprimer des environnements de dev sans impacter la prod
- Possibilité de tester des migrations sur les cascades avant la prod

## Configuration pour les Nouveautés PostgreSQL 18

PostgreSQL 18 apporte des améliorations qui bénéficient à la réplication en cascade :

### 1. I/O Asynchrone (AIO)

Activer l'I/O asynchrone améliore les performances de réplication :

```ini
# Sur tous les serveurs (primary, intermédiaires, cascades)
io_method = 'worker'  # ou 'io_uring' sur Linux
```

**Gains attendus :**
- Réduction du lag de 20-30%
- Meilleure utilisation du réseau
- Idéal pour les cascades avec beaucoup de standbys

### 2. Statistiques de réplication améliorées

PostgreSQL 18 expose de nouvelles métriques :

```sql
-- Métriques I/O par backend (incluant WAL sender)
SELECT
    backend_type,
    reads,
    writes,
    read_time,
    write_time
FROM pg_stat_io  
WHERE backend_type = 'walsender';  
```

**Utilité :**
- Identifier les goulots d'étranglement I/O sur les standbys intermédiaires
- Optimiser la configuration des disques

### 3. SCRAM Passthrough

PostgreSQL 18 permet l'authentification SCRAM via FDW et dblink, ce qui simplifie la gestion des mots de passe en cascade :

```ini
# Les standbys peuvent utiliser l'authentification SCRAM de manière transparente
# Pas de configuration supplémentaire nécessaire
```

## Résumé : Checklist de Configuration

### Configuration du Standby Intermédiaire

- [ ] Configurer `postgresql.conf` :  
  - [ ] `max_wal_senders = 5` (ou plus selon le nombre de cascades)  
  - [ ] `wal_keep_size = 1GB`  
  - [ ] `max_replication_slots = 5`  
  - [ ] `hot_standby = on`  
- [ ] Configurer `pg_hba.conf` : autoriser les connexions depuis les cascades  
- [ ] Créer des slots de réplication pour chaque cascade  
- [ ] Redémarrer PostgreSQL : `systemctl restart postgresql`  
- [ ] Vérifier : `SHOW max_wal_senders;` (doit être > 0)

### Configuration du Standby Cascade

- [ ] Effectuer un `pg_basebackup` **depuis le standby intermédiaire** (pas le primary)  
- [ ] Configurer `postgresql.conf` :  
  - [ ] `hot_standby = on`  
  - [ ] `primary_conninfo` pointant vers le standby intermédiaire  
  - [ ] `primary_slot_name` (si slot créé sur l'intermédiaire)  
- [ ] Vérifier que `standby.signal` existe  
- [ ] Ajuster les permissions : `chown -R postgres:postgres ...`  
- [ ] Démarrer PostgreSQL : `systemctl start postgresql`  
- [ ] Vérifier : `SELECT pg_is_in_recovery();` (doit retourner `true`)

### Vérifications Post-Configuration

- [ ] Sur le primary : Voir le standby intermédiaire dans `pg_stat_replication`  
- [ ] Sur l'intermédiaire : Voir les cascades dans `pg_stat_replication`  
- [ ] Sur les cascades : Vérifier `pg_stat_wal_receiver` pointe vers l'intermédiaire  
- [ ] Test de bout en bout : INSERT sur primary → visible sur cascades  
- [ ] Configurer le monitoring : métriques de lag pour chaque niveau  
- [ ] Documenter la topologie : schéma + IPs + procédures de failover

## Conclusion

La réplication en cascade est une fonctionnalité puissante qui permet de construire des architectures de réplication scalables et géographiquement distribuées. Elle offre :

- ✅ **Réduction de la charge sur le primary** : Moins de connexions WAL sender, bande passante économisée  
- ✅ **Scalabilité** : Possibilité d'avoir des dizaines de standbys sans surcharger le primary  
- ✅ **Flexibilité géographique** : Réplication multi-région optimisée  
- ✅ **Isolation** : Séparer les environnements production/non-production

**Points clés à retenir :**

1. **Le standby intermédiaire** doit avoir `max_wal_senders > 0` et être configuré comme un "mini-primary"  
2. **Le lag se cumule** à chaque niveau : limiter à 2-3 niveaux maximum  
3. **Utiliser des slots de réplication** pour éviter la perte de WAL  
4. **Monitorer chaque niveau** de réplication, pas seulement le primary  
5. **Documenter la topologie** pour faciliter le troubleshooting et les failovers  
6. **Tester les scénarios de panne** régulièrement

**Quand utiliser la cascade :**
- Vous avez > 5 standbys à gérer
- Vous déployez sur plusieurs régions géographiques
- Vous voulez isoler des environnements (prod vs non-prod)
- Vous avez des contraintes de bande passante sur le primary

**Quand ne pas utiliser la cascade :**
- Vous avez seulement 1-2 standbys (overhead inutile)
- Le lag cumulatif est inacceptable (< 1 seconde requis)
- Vous préférez la simplicité à la scalabilité

La réplication en cascade est un outil d'architecture avancé qui nécessite une planification et un monitoring rigoureux, mais qui offre une scalabilité et une flexibilité inégalées pour les déploiements PostgreSQL de grande envergure.
