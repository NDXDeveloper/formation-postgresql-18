# 17.6.2. Quorum-based Commit

## Introduction

Dans la section précédente, nous avons vu que la réplication synchrone offre une protection maximale contre la perte de données, mais présente un risque majeur : **si votre unique serveur Standby synchrone tombe en panne, toutes vos écritures sont bloquées**. C'est ce qu'on appelle un "point de défaillance unique" (Single Point of Failure - SPOF).

Le **Quorum-based commit** (commit basé sur le quorum) est une solution élégante à ce problème. Au lieu d'attendre un seul serveur spécifique, PostgreSQL attend la confirmation d'un **nombre minimum de serveurs parmi un groupe**. C'est comme dire : "Je veux que mes données soient présentes sur au moins 2 serveurs parmi mes 3 Standby, peu importe lesquels".

Cette approche offre le meilleur équilibre entre fiabilité, disponibilité et tolérance aux pannes.

---

## Qu'est-ce qu'un Quorum ? Analogie Simple

Imaginez que vous organisez une réunion importante et que vous voulez prendre une décision :

**Sans quorum (réplication synchrone classique) :**
- "La décision n'est validée QUE si Jean est présent"
- Problème : Si Jean est malade, aucune décision ne peut être prise

**Avec quorum :**
- "La décision est validée si au moins 2 personnes parmi Jean, Marie et Paul sont présentes"
- Avantage : Si Jean est malade, Marie et Paul peuvent valider la décision
- La réunion peut continuer même si quelqu'un est absent

C'est exactement le même principe avec PostgreSQL : au lieu de dépendre d'un serveur unique, vous attendez qu'un **nombre minimum de serveurs** (le quorum) confirment avoir reçu les données.

---

## Le Problème de la Réplication Synchrone Classique

### Scénario de défaillance

Reprenons l'architecture synchrone simple :

```
Primary → Standby1 (synchrone)
```

**Que se passe-t-il si Standby1 tombe en panne ?**

1. Le Primary tente d'envoyer les modifications à Standby1  
2. Standby1 ne répond pas  
3. Le Primary **attend indéfiniment** (comportement par défaut)  
4. **Toutes les écritures sont bloquées** ❌  
5. Votre application est en panne

**Options de mitigation classiques :**

- **Intervention manuelle** : Un administrateur doit reconfigurer le Primary pour retirer Standby1 de la liste synchrone  
- **Timeout** : Configurer un timeout après lequel le Primary abandonne (mais cela peut entraîner une perte de données)  
- **Basculer en asynchrone** : Modifier `synchronous_commit` (mais on perd la garantie de réplication)

Aucune de ces solutions n'est idéale : elles nécessitent soit une intervention humaine, soit un compromis sur la fiabilité.

---

## Le Quorum-Based Commit : La Solution

### Principe de fonctionnement

Avec le quorum, vous configurez PostgreSQL ainsi :

```
Primary → Standby1
       → Standby2
       → Standby3

Configuration : "ANY 2 (Standby1, Standby2, Standby3)"
```

**Signification :** Le Primary attend la confirmation d'**au moins 2 serveurs parmi les 3**, peu importe lesquels.

**Scénarios de fonctionnement :**

| État des Standby | Résultat |
|------------------|----------|
| Standby1 ✅, Standby2 ✅, Standby3 ✅ | Commit réussi (3 confirmations) |
| Standby1 ✅, Standby2 ✅, Standby3 ❌ | Commit réussi (2 confirmations) ✅ |
| Standby1 ✅, Standby2 ❌, Standby3 ✅ | Commit réussi (2 confirmations) ✅ |
| Standby1 ❌, Standby2 ✅, Standby3 ✅ | Commit réussi (2 confirmations) ✅ |
| Standby1 ✅, Standby2 ❌, Standby3 ❌ | Commit bloqué (1 seule confirmation) ❌ |

**Avantage majeur :** Un seul Standby en panne n'empêche plus les écritures !

---

## Syntaxe de Configuration

PostgreSQL propose deux syntaxes pour définir un quorum :

### 1. Syntaxe ANY (Quorum souple)

```conf
synchronous_standby_names = 'ANY N (liste_de_serveurs)'
```

**N** = Nombre minimum de serveurs qui doivent confirmer

**Exemple :**
```conf
synchronous_standby_names = 'ANY 2 (standby1, standby2, standby3)'
```
- Attend **au moins 2** confirmations parmi les 3 serveurs
- Peu importe lesquels

**Cas d'usage :** Haute disponibilité avec tolérance aux pannes

### 2. Syntaxe FIRST (Priorité stricte)

```conf
synchronous_standby_names = 'FIRST N (liste_de_serveurs)'
```

**N** = Nombre de serveurs dans l'ordre de priorité

**Exemple :**
```conf
synchronous_standby_names = 'FIRST 2 (standby1, standby2, standby3)'
```
- Attend les confirmations de **standby1 ET standby2** (les 2 premiers)
- standby3 n'est utilisé que si standby1 ou standby2 tombe

**Cas d'usage :** Priorisation géographique ou matérielle

### 3. Syntaxe legacy (un seul serveur)

```conf
synchronous_standby_names = 'standby1'
```
- Attend uniquement standby1 (comportement classique)
- **Ne protège pas contre la panne du Standby**

---

## Comparaison des Syntaxes

### Exemple avec 3 Standby

```
Primary → Standby1 (local, SSD rapide)
       → Standby2 (local, SSD rapide)
       → Standby3 (distant, HDD)
```

**Configuration ANY 2 :**
```conf
synchronous_standby_names = 'ANY 2 (standby1, standby2, standby3)'
```

**Comportement :**
- Si Standby1 et Standby2 répondent vite → Commit rapide ✅
- Si Standby1 est lent → Primary utilise Standby2 et Standby3 ✅
- Même si Standby3 est lent, ça ne bloque pas si Standby1 et Standby2 répondent

**Avantage :** Utilise toujours les serveurs les plus rapides disponibles

---

**Configuration FIRST 2 :**
```conf
synchronous_standby_names = 'FIRST 2 (standby1, standby2, standby3)'
```

**Comportement :**
- Primary attend **TOUJOURS** Standby1 ET Standby2
- Standby3 n'est jamais utilisé tant que les deux premiers fonctionnent
- Si Standby1 tombe → Primary attend Standby2 ET Standby3

**Avantage :** Contrôle précis de quels serveurs doivent être synchrones (utile pour des raisons géographiques)

---

## Calcul du Quorum : Trouver le Bon Nombre

### Règle de base pour `synchronous_standby_names`

Le quorum dans `synchronous_standby_names` (`ANY N` ou `FIRST N`) porte **uniquement sur les standbys** : `N` est le nombre de standbys qui doivent confirmer pour valider un COMMIT côté primary. Le primary lui-même **n'est pas dans la liste** (c'est lui qui attend).

> ⚠️ À ne pas confondre avec le **quorum d'un système de consensus** (etcd, Patroni) qui, lui, raisonne en majorité parmi N nœuds incluant le primary. Ce sont deux notions distinctes.

**Choix typiques de `N`** :

| Configuration | `N` recommandé | Confirmations attendues | Tolérance aux pannes de standby |
|--------------|----------------|-------------------------|--------------------------------|
| 1 Primary + 1 Standby | `1` | 1 | 0 (si le standby tombe, COMMIT bloqué) |
| 1 Primary + 2 Standbys | `1` (ANY 1) | au moins 1/2 | 1 standby |
| 1 Primary + 3 Standbys | `2` (ANY 2) | au moins 2/3 | 1 standby |
| 1 Primary + 4 Standbys | `2` ou `3` | selon le besoin | 2 ou 1 standby |
| 1 Primary + 5 Standbys | `2` ou `3` | selon le besoin | 3 ou 2 standbys |

**Compromis durabilité ↔ disponibilité** :
- `N` élevé → plus de copies confirmées, mais plus de risque de blocage si trop de standbys tombent
- `N` bas → moins de copies confirmées, mais service écriture maintenu plus longtemps

### Cas pratiques de dimensionnement

**Cas 1 : Architecture minimale haute disponibilité**
```
1 Primary + 2 Standby
Quorum = 1 (ANY 1)  
Tolérance : 1 standby en panne  
```
✅ Configuration minimale recommandée pour la production

> ⚠️ Avec 2 standbys, **`ANY 2` exigerait les DEUX standbys** (tolérance 0). Pour tolérer une panne, il faut `ANY 1`. Si vous voulez garantir 2 copies synchrones tout en tolérant une panne, il faut **3 standbys** avec `ANY 2` (voir Cas 2).

**Cas 2 : Architecture multi-datacenter**
```
1 Primary (DC1) + 2 Standby (DC1) + 1 Standby (DC2)
Total : 3 standbys  
Quorum = 2 (ANY 2)  
Tolérance : 1 standby en panne (parmi les 3, sans distinction de DC)  
```
✅ Protection géographique avec performance préservée. La perte complète de DC2 équivaut à 1 standby perdu (DC2 ne contient qu'un standby) → encore tolérée.

**Cas 3 : Architecture haute disponibilité avancée**
```
1 Primary + 4 Standby
Quorum = 2 (ANY 2) → tolérance 2 standbys en panne  
ou  
Quorum = 3 (ANY 3) → tolérance 1 standby en panne (mais 3 copies garanties)  
```
✅ Niveau de protection maximum (pour applications critiques). Choisir `ANY 2` pour la résilience, `ANY 3` pour la durabilité (3 copies synchrones simultanées).

## Configuration Complète : Exemple Pas à Pas

### Architecture cible

```
Primary (production)
  ├── Standby1 (même datacenter)
  ├── Standby2 (même datacenter)
  └── Standby3 (datacenter distant)

Objectif : Quorum ANY 2 — au moins 2 standbys parmi les 3 confirment chaque COMMIT.
```

> ℹ️ Avec `ANY 2 (standby1, standby2, standby3)`, **les 3 standbys sont éligibles au quorum** : le primary attend les 2 qui répondent le plus vite. En pratique, ce seront presque toujours les 2 standbys du datacenter local (réseau plus rapide), mais Standby3 servira de "filet de sécurité" si un local tombe.  
>  
> Si vous voulez **figer** Standby3 en asynchrone et n'avoir le quorum que sur les standbys locaux, écrivez plutôt :  
>  
> ```ini  
> synchronous_standby_names = 'ANY 2 (standby1, standby2)'  
> # Standby3 ne figure pas dans la liste → il restera 'async'.  
> ```

### Étape 1 : Configuration sur le Primary

Éditez `postgresql.conf` sur le Primary :

```conf
# Activer la réplication
wal_level = replica  
max_wal_senders = 10  
wal_keep_size = 1GB  

# Configuration du quorum synchrone
synchronous_standby_names = 'ANY 2 (standby1, standby2, standby3)'  
synchronous_commit = on  

# Paramètres de sécurité
max_replication_slots = 10
```

**Explication :**
- `synchronous_standby_names` : Définit le quorum (2 confirmations minimum)  
- `synchronous_commit = on` : Active le mode synchrone
- Les 3 Standby sont listés, mais seuls 2 doivent confirmer

### Étape 2 : Configuration sur les Standby

Sur **Standby1**, dans `postgresql.conf` :

```conf
# Mode Standby
hot_standby = on  
primary_conninfo = 'host=primary_ip port=5432 user=replicator password=xxx application_name=standby1'  
```

Sur **Standby2**, dans `postgresql.conf` :

```conf
hot_standby = on  
primary_conninfo = 'host=primary_ip port=5432 user=replicator password=xxx application_name=standby2'  
```

Sur **Standby3**, dans `postgresql.conf` :

```conf
hot_standby = on  
primary_conninfo = 'host=primary_ip port=5432 user=replicator password=xxx application_name=standby3'  
```

**Point crucial :** `application_name` doit correspondre aux noms dans `synchronous_standby_names` !

### Étape 3 : Rechargement et vérification

> 💡 **Quels paramètres nécessitent un redémarrage ?**  
> - `wal_level`, `max_wal_senders`, `max_replication_slots` → **restart obligatoire** (paramètres `postmaster`)  
> - `synchronous_standby_names`, `synchronous_commit` → **simple reload suffit** (paramètres `sighup`)  
>  
> Si vous changez **uniquement** `synchronous_standby_names` (cas le plus fréquent en exploitation), un reload suffit :

```bash
# Sur le Primary — reload (pas de coupure)
sudo systemctl reload postgresql
# OU via SQL : SELECT pg_reload_conf();

# Si vous avez modifié wal_level/max_wal_senders/max_replication_slots, redémarrer :
# sudo systemctl restart postgresql

# Vérifier la configuration
psql -U postgres -c "SHOW synchronous_standby_names;"
```

**Résultat attendu :**
```
 synchronous_standby_names
---------------------------
 ANY 2 (standby1, standby2, standby3)
```

### Étape 4 : Vérifier l'état de la réplication

```sql
SELECT
    application_name,
    state,
    sync_state,
    sync_priority
FROM pg_stat_replication  
ORDER BY sync_priority;  
```

**Résultat attendu :**

| application_name | state | sync_state | sync_priority |
|------------------|-------|------------|---------------|
| standby1 | streaming | quorum | 1 |
| standby2 | streaming | quorum | 1 |
| standby3 | streaming | quorum | 1 |

**Note :** Avec `ANY`, tous les Standby ont la même priorité (1). Le Primary choisit dynamiquement les 2 premiers qui répondent.

## Surveillance et Métriques

### Vérifier le statut du quorum en temps réel

```sql
-- Vue complète de l'état de réplication
SELECT
    application_name,
    client_addr,
    state,
    sync_state,
    sync_priority,
    pg_wal_lsn_diff(pg_current_wal_lsn(), sent_lsn) AS send_lag_bytes,
    pg_wal_lsn_diff(sent_lsn, flush_lsn) AS flush_lag_bytes,
    pg_wal_lsn_diff(flush_lsn, replay_lsn) AS replay_lag_bytes,
    write_lag,
    flush_lag,
    replay_lag
FROM pg_stat_replication  
ORDER BY sync_priority, application_name;  
```

**Interprétation :**

- **sync_state = 'quorum'** : Le Standby fait partie du quorum  
- **sync_state = 'async'** : Le Standby est en mode asynchrone (si plus de serveurs que requis)  
- **state = 'streaming'** : Le Standby reçoit activement les données  
- **lag_* faibles** : La réplication est à jour

### Alertes essentielles à configurer

**Alerte 1 : Nombre de Standby synchrones insuffisant**

```sql
SELECT COUNT(*) AS standby_count  
FROM pg_stat_replication  
WHERE sync_state IN ('sync', 'quorum') AND state = 'streaming';  
```

**Seuil d'alerte :** Si `standby_count < 2` (dans notre exemple avec quorum de 2)

**Action :** Intervention immédiate - risque de blocage des écritures si un autre Standby tombe

---

**Alerte 2 : Lag de réplication élevé**

```sql
SELECT
    application_name,
    EXTRACT(EPOCH FROM write_lag) AS write_lag_seconds
FROM pg_stat_replication  
WHERE sync_state = 'quorum'  
    AND write_lag > INTERVAL '5 seconds';
```

**Seuil d'alerte :** `write_lag > 5 secondes`

**Action :** Vérifier la performance réseau et la charge du Standby

---

**Alerte 3 : Déconnexion d'un Standby quorum**

```sql
-- À exécuter régulièrement (via un monitoring externe)
SELECT COUNT(*) AS connected_quorum_standbys  
FROM pg_stat_replication  
WHERE sync_state = 'quorum';  
```

**Seuil d'alerte :** Si le nombre descend en dessous du quorum configuré

---

### Script de monitoring complet

Voici un script SQL à exécuter périodiquement (ex: toutes les minutes) :

```sql
-- Monitoring complet du quorum
WITH quorum_config AS (
    SELECT current_setting('synchronous_standby_names') AS config
),
expected_quorum AS (
    -- Extraire le nombre du quorum (simplifié, suppose format ANY N)
    SELECT
        CASE
            WHEN config ~ 'ANY (\d+)'
            THEN (regexp_match(config, 'ANY (\d+)'))[1]::int
            ELSE 1
        END AS required_standbys
    FROM quorum_config
),
current_status AS (
    SELECT
        COUNT(*) FILTER (WHERE sync_state = 'quorum' AND state = 'streaming') AS active_quorum_standbys,
        COUNT(*) AS total_standbys,
        MAX(EXTRACT(EPOCH FROM write_lag)) AS max_lag_seconds
    FROM pg_stat_replication
)
SELECT
    cs.active_quorum_standbys,
    eq.required_standbys,
    cs.active_quorum_standbys >= eq.required_standbys AS quorum_healthy,
    cs.total_standbys,
    COALESCE(cs.max_lag_seconds, 0) AS max_lag_seconds,
    CASE
        WHEN cs.active_quorum_standbys < eq.required_standbys THEN 'CRITICAL - Quorum not met'
        WHEN cs.max_lag_seconds > 10 THEN 'WARNING - High replication lag'
        ELSE 'OK'
    END AS health_status
FROM current_status cs, expected_quorum eq;
```

**Exemple de résultat :**

| active_quorum_standbys | required_standbys | quorum_healthy | total_standbys | max_lag_seconds | health_status |
|------------------------|-------------------|----------------|----------------|-----------------|---------------|
| 2 | 2 | true | 3 | 0.003 | OK |

## Scénarios de Panne et Comportement

### Scénario 1 : Un Standby tombe (le plus courant)

**Situation :**
```
Primary ✅  
Standby1 ❌ (panne)  
Standby2 ✅  
Standby3 ✅  

Configuration : ANY 2
```

**Comportement :**
1. Le Primary détecte que Standby1 ne répond plus  
2. Il continue d'attendre Standby2 et Standby3  
3. Le quorum est toujours atteint (2 serveurs fonctionnels)  
4. ✅ Les écritures continuent normalement  
5. Pas d'intervention nécessaire

**Impact :** Aucun (c'est tout l'intérêt du quorum !)

---

### Scénario 2 : Deux Standby tombent simultanément

**Situation :**
```
Primary ✅  
Standby1 ❌  
Standby2 ❌  
Standby3 ✅  

Configuration : ANY 2
```

**Comportement :**
1. Le Primary ne peut plus atteindre le quorum de 2  
2. ⚠️ Toutes les écritures se bloquent  
3. Le Primary attend indéfiniment (comportement par défaut)  
4. Intervention humaine nécessaire

**Actions possibles :**
- **Réparer rapidement** un des Standby en panne  
- **Réduire temporairement le quorum** :
  ```sql
  ALTER SYSTEM SET synchronous_standby_names = 'ANY 1 (standby1, standby2, standby3)';
  SELECT pg_reload_conf();
  ```
- **Passer en mode asynchrone temporairement** :
  ```sql
  ALTER SYSTEM SET synchronous_standby_names = '';
  SELECT pg_reload_conf();
  ```

---

### Scénario 3 : Lag important sur un Standby

**Situation :**
```
Primary ✅  
Standby1 ✅ (lag: 2ms)  
Standby2 ✅ (lag: 2ms)  
Standby3 ✅ (lag: 500ms - réseau saturé)  

Configuration : ANY 2
```

**Comportement :**
1. Le Primary envoie les données aux 3 Standby  
2. Standby1 et Standby2 répondent en 2ms  
3. Le quorum est atteint sans attendre Standby3  
4. ✅ Les écritures ne sont pas ralenties par Standby3

**Avantage du ANY :** Le Primary utilise toujours les serveurs les plus rapides pour satisfaire le quorum

---

### Scénario 4 : Maintenance planifiée

**Objectif :** Mettre Standby1 hors ligne pour maintenance

**Procédure :**

1. **Vérifier que le quorum peut être maintenu après arrêt de Standby1 :**
   ```sql
   -- Avant d'arrêter Standby1, il faut que les 3 standbys soient en streaming
   -- pour pouvoir continuer avec les 2 restants (quorum ANY 2).
   SELECT COUNT(*) FROM pg_stat_replication WHERE state = 'streaming';
   -- Doit retourner 3 (les 3 standbys actuels sont OK)
   -- Après arrêt de Standby1, il en restera 2 — pile le quorum requis.
   ```

2. **Arrêter Standby1 :**
   ```bash
   # Sur Standby1
   sudo systemctl stop postgresql
   ```

3. **Vérifier que le quorum est toujours satisfait :**
   ```sql
   -- Sur le Primary
   SELECT * FROM pg_stat_replication;
   -- Doit afficher Standby2 et Standby3 en état 'quorum'
   ```

4. **Effectuer la maintenance sur Standby1**

5. **Redémarrer Standby1 :**
   ```bash
   sudo systemctl start postgresql
   ```

6. **Vérifier la réintégration :**
   ```sql
   SELECT * FROM pg_stat_replication;
   -- Standby1 doit réapparaître en état 'quorum'
   ```

**Impact sur les utilisateurs :** Aucun, grâce au quorum !

## Avantages du Quorum-Based Commit

### ✅ 1. Haute disponibilité

- Tolérance à la panne d'un ou plusieurs Standby (selon le dimensionnement)
- Pas de point de défaillance unique
- Maintenance sans interruption de service

### ✅ 2. Garantie de durabilité

- Les données validées sont présentes sur au moins N serveurs
- Protection contre la perte de données même en cas de pannes multiples
- RPO = 0 (aucune perte de transaction validée)

### ✅ 3. Performance optimale

- Utilisation automatique des serveurs les plus rapides
- Pas de ralentissement dû à un Standby lent
- Latence = latence du Nième serveur le plus rapide (et non du plus lent)

### ✅ 4. Flexibilité opérationnelle

- Ajout/retrait de Standby sans interruption
- Maintenance simplifiée
- Scaling horizontal naturel

### ✅ 5. Résilience au réseau

- Un problème réseau vers un Standby n'impacte pas les autres
- Tolérance aux micro-coupures
- Adaptation automatique aux conditions réseau

---

## Inconvénients et Limitations

### ⚠️ 1. Complexité opérationnelle accrue

- Configuration plus complexe qu'une réplication simple
- Nécessite une surveillance plus sophistiquée
- Plus de serveurs à gérer

**Mitigation :** Utiliser des outils d'automatisation (Patroni, Ansible)

### ⚠️ 2. Coût matériel

- Nécessite au minimum 3 serveurs (1 Primary + 2 Standby)
- Recommandation : 4 serveurs pour tolérer 1 panne (1 Primary + 3 Standby)

**Justification :** Le coût est compensé par la disponibilité et la protection des données

### ⚠️ 3. Impact performance selon le quorum

- Plus le quorum est élevé, plus l'impact sur les performances est important
- Latence = celle du Nième serveur le plus rapide (N = taille du quorum)

**Exemple :**
- Quorum de 2 avec lags [2ms, 3ms, 50ms] → Latence = 3ms
- Quorum de 3 avec lags [2ms, 3ms, 50ms] → Latence = 50ms

### ⚠️ 4. Pas de protection contre les erreurs logiques

- Si vous supprimez accidentellement des données, elles seront supprimées partout
- Le quorum ne protège que contre les pannes matérielles
- Nécessite des sauvegardes régulières (backups)

---

## Cas d'Usage Recommandés

### ✅ Cas 1 : Applications critiques multi-datacenter

**Architecture :**
```
DC1: 1 Primary + 2 Standby  
DC2: 1 Standby  
Configuration : ANY 2  
```

**Avantages :**
- Protection locale contre les pannes (quorum dans DC1)
- Protection géographique (Standby dans DC2)
- Performance optimale (quorum local)

---

### ✅ Cas 2 : SaaS avec SLA élevé

**Architecture :**
```
1 Primary + 3 Standby (même région)
Configuration : ANY 2
```

**Avantages :**
- Garantie de disponibilité 99.99%
- Maintenance sans interruption
- Tolérance à 1 panne serveur

---

### ✅ Cas 3 : Finance / E-commerce

**Architecture :**
```
1 Primary + 4 Standby
Configuration : ANY 2  (pour la résilience max)  
ou  
Configuration : ANY 3  (pour garantir 3 copies confirmées)  
```

**Avantages :**
- Protection maximale (RPO = 0)
- Avec `ANY 2` : Tolérance à **2 pannes simultanées** (4 - 2 = 2)
- Avec `ANY 3` : Tolérance à **1 panne**, mais 3 copies sync à chaque commit
- Conformité réglementaire

---

### ❌ Cas où le quorum n'est PAS recommandé

**1. Petite application / Budget limité**
- Le coût de 3+ serveurs n'est pas justifié
- Alternative : Réplication asynchrone simple

**2. Infrastructure géographiquement distribuée**
- Latence réseau trop élevée entre datacenters
- Alternative : Synchrone local + Asynchrone distant

**3. Très fort volume d'écritures**
- L'impact performance du quorum peut être problématique
- Alternative : Réplication asynchrone avec monitoring renforcé

---

## Bonnes Pratiques

### 🎯 Dimensionnement optimal

**Règle d'or :** Si vous voulez du quorum **avec tolérance à une panne**, il faut **au minimum 4 serveurs** (1 Primary + 3 Standby) avec `ANY 2` → tolérance d'1 standby + 2 copies sync garanties. Avec seulement 2 standbys, vous devez choisir entre tolérance (`ANY 1`) et 2 copies sync (`ANY 2`, sans tolérance).

**Recommandations par niveau de criticité :**

Règle de calcul : avec N standbys et quorum K, la **tolérance** est N − K standbys.

| Criticité | Configuration minimale | Quorum | Tolérance | Copies sync minimum |
|-----------|------------------------|--------|-----------|---------------------|
| Standard | 1 Primary + 2 Standby | ANY 1 | 1 panne | 1 |
| Élevée | 1 Primary + 3 Standby | ANY 2 | 1 panne | 2 |
| Critique | 1 Primary + 4 Standby | ANY 2 | 2 pannes | 2 |
| Critique++ | 1 Primary + 4 Standby | ANY 3 | 1 panne | 3 |

### 🎯 Placement géographique

**Architecture hybride recommandée :**
```
DC1 (Local):
  - Primary
  - Standby1
  - Standby2

DC2 (Distant):
  - Standby3

Configuration : ANY 2 (standby1, standby2, standby3)
```

**Astuce :** Les 3 standbys sont éligibles au quorum, mais comme `ANY 2` attend les **2 plus rapides à confirmer**, ce seront en pratique presque toujours Standby1 et Standby2 (locaux, latence < 1 ms). Standby3 (distant) sert de filet de sécurité s'il en manque un.

**Si vous voulez forcer Standby3 à être strictement asynchrone** (jamais dans le quorum), retirez-le de la liste :
```ini
synchronous_standby_names = 'ANY 2 (standby1, standby2)'
# Standby3 reste connecté en réplication mais 'async'.
```

### 🎯 Surveillance proactive

Configurez des alertes pour :
- Nombre de Standby actifs < quorum + 1
- Lag de réplication > seuil acceptable
- Déconnexion d'un Standby
- Espace disque faible (WAL accumulation)

### 🎯 Tests réguliers

**Testez mensuellement :**
1. Arrêt d'un Standby → Vérifier que le service continue  
2. Arrêt de deux Standby → Vérifier le comportement de blocage  
3. Temps de récupération après réintégration d'un Standby

### 🎯 Documentation

Documentez :
- La topologie complète de réplication
- La procédure de réduction temporaire du quorum (urgence)
- Les contacts et procédures d'escalade
- Les runbooks de panne

## Migration vers le Quorum : Guide Pratique

### Étape 1 : Ajouter des Standby supplémentaires

Si vous avez actuellement 1 Primary + 1 Standby :

```bash
# Créer une deuxième réplica
pg_basebackup -h primary_ip -D /var/lib/postgresql/18/main -U replicator -P -v -R -X stream
```

### Étape 2 : Vérifier la réplication

```sql
SELECT * FROM pg_stat_replication;
```

Attendez que tous les Standby soient en état `streaming` et à jour.

### Étape 3 : Modifier la configuration

```sql
-- Sur le Primary
ALTER SYSTEM SET synchronous_standby_names = 'ANY 2 (standby1, standby2)';  
SELECT pg_reload_conf();  
```

### Étape 4 : Vérifier la nouvelle configuration

```sql
SHOW synchronous_standby_names;  
SELECT application_name, sync_state FROM pg_stat_replication;  
```

### Étape 5 : Tester

```sql
-- Simuler une écriture
INSERT INTO test_table VALUES (NOW());

-- Vérifier qu'elle est bien répliquée
-- (sur les Standby)
SELECT * FROM test_table ORDER BY 1 DESC LIMIT 1;
```

### Étape 6 : Tester la résilience

Arrêtez un Standby et vérifiez que les écritures continuent.

## Conclusion

Le **Quorum-based commit** est une évolution majeure de la réplication synchrone dans PostgreSQL. Il offre le meilleur équilibre entre :

- **Fiabilité** : Aucune perte de données (RPO = 0)  
- **Disponibilité** : Tolérance aux pannes de serveurs  
- **Performance** : Utilisation des serveurs les plus rapides

**Points clés à retenir :**

1. ✅ **ANY N** est généralement le meilleur choix (utilise les serveurs les plus rapides)  
2. ✅ **Minimum 3 serveurs** (1 Primary + 2 Standby) pour bénéficier du quorum  
3. ✅ **Surveillance essentielle** : Alerter si le nombre de Standby < quorum + 1  
4. ✅ **Testez régulièrement** les scénarios de panne  
5. ✅ **Documentez** votre architecture et vos procédures

Le quorum transforme la réplication synchrone d'une solution fragile (avec point de défaillance unique) en une architecture véritablement hautement disponible. C'est aujourd'hui la **configuration recommandée** pour toute application critique nécessitant une protection maximale contre la perte de données.
