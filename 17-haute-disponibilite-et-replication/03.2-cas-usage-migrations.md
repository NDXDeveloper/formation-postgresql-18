# 17.3.2. Cas d'Usage : Migrations et Réplication Sélective

## Introduction

La réplication logique PostgreSQL est un outil exceptionnellement polyvalent qui va bien au-delà de la simple copie de données. Dans cette section, nous allons explorer deux cas d'usage majeurs qui démontrent toute la puissance et la flexibilité de ce mécanisme :

1. **Les migrations de bases de données** : Comment déplacer vos données d'un serveur à un autre sans interruption de service  
2. **La réplication sélective** : Comment ne répliquer que les données pertinentes selon vos besoins métier

Ces deux scénarios sont parmi les plus fréquents en production et illustrent parfaitement pourquoi la réplication logique est devenue un standard de l'industrie.

---

## Partie 1 : Migrations de Bases de Données

### Qu'est-ce qu'une Migration de Base de Données ?

Une **migration de base de données** consiste à transférer une base de données d'un environnement à un autre. Cela peut être :
- D'un serveur physique vers le cloud
- D'une version PostgreSQL ancienne vers une version plus récente
- D'un data center vers un autre
- D'un fournisseur cloud vers un autre
- D'une architecture on-premise vers un SaaS managé

### Le Défi des Migrations Traditionnelles

#### Approche "Stop-and-Copy" (Ancienne Méthode)

```
┌─────────────────────────────────────────────┐
│ 1. Arrêter l'application                    │  ❌ Downtime
│ 2. Faire un dump de la base (pg_dump)       │  ⏱️ Plusieurs heures
│ 3. Transférer le dump vers le nouveau       │  ⏱️ Dépend de la bande passante
│ 4. Restaurer sur le nouveau serveur         │  ⏱️ Plusieurs heures
│ 5. Vérifier l'intégrité                     │  ⏱️ Quelques minutes
│ 6. Redémarrer l'application                 │  ✅ Service restauré
└─────────────────────────────────────────────┘

Total downtime : De quelques heures à plusieurs jours pour de très grandes bases
```

**Problèmes** :
- ❌ Interruption de service prolongée  
- ❌ Risque élevé en cas d'échec (retour en arrière complexe)  
- ❌ Stress pour les équipes (migration en "big bang")  
- ❌ Perte de revenus pendant l'interruption  
- ❌ Expérience utilisateur dégradée

### La Solution : Migration avec Réplication Logique (Zero-Downtime)

La réplication logique permet une approche progressive et sans interruption de service :

```
┌─────────────────────────────────────────────┐
│ Phase 1 : Préparation (serveur opérationnel)│
│ - Configurer la réplication                 │
│ - Copie initiale des données                │
│                                             │
│ Phase 2 : Synchronisation (serveur opér.)   │
│ - Réplication continue                      │
│ - Les deux serveurs restent synchronisés    │
│                                             │
│ Phase 3 : Bascule (downtime : secondes)     │
│ - Rediriger l'application                   │
│ - Vérification rapide                       │
│                                             │
│ Phase 4 : Validation et nettoyage           │
│ - Surveillance du nouveau serveur           │
│ - Suppression de la réplication             │
└─────────────────────────────────────────────┘

Total downtime : Quelques secondes à quelques minutes maximum
```

#### Étape par Étape : Migration Complète

##### Étape 1 : Préparation du Serveur Source (On-Premise)

```sql
-- 1.1. Connexion au serveur source
psql -h serveur-onpremise.local -U postgres -d ecommerce

-- 1.2. Vérification et ajustement de la configuration
-- Dans postgresql.conf (redémarrage requis si modifications nécessaires)
ALTER SYSTEM SET wal_level = 'logical';  
ALTER SYSTEM SET max_replication_slots = 5;  
ALTER SYSTEM SET max_wal_senders = 5;  

-- Redémarrer PostgreSQL si nécessaire
-- sudo systemctl restart postgresql

-- 1.3. Création d'un utilisateur dédié à la réplication
CREATE ROLE replication_user WITH LOGIN PASSWORD 'secure_password_2024' REPLICATION;

-- 1.4. Accorder les permissions sur toutes les tables nécessaires
GRANT SELECT ON ALL TABLES IN SCHEMA public TO replication_user;  
GRANT USAGE ON SCHEMA public TO replication_user;  

-- 1.5. Configurer pg_hba.conf pour autoriser la connexion depuis AWS
-- Ajouter une ligne (adapter l'IP en fonction de votre VPN/VPC)
-- host    ecommerce    replication_user    10.0.0.0/8    scram-sha-256

-- 1.6. Recharger la configuration
SELECT pg_reload_conf();

-- 1.7. Créer la publication pour toutes les tables
CREATE PUBLICATION pub_migration FOR ALL TABLES;

-- Alternative : publication sélective si toutes les tables ne sont pas nécessaires
-- CREATE PUBLICATION pub_migration FOR TABLE commandes, produits, clients, inventaire;

-- 1.8. Vérification
SELECT * FROM pg_publication WHERE pubname = 'pub_migration';  
SELECT * FROM pg_publication_tables WHERE pubname = 'pub_migration';  
```

##### Étape 2 : Préparation du Serveur Destination (AWS RDS)

```sql
-- 2.1. Connexion au serveur AWS RDS
psql -h ecommerce.abc123.us-east-1.rds.amazonaws.com -U postgres -d ecommerce

-- 2.2. Créer la structure de base (schéma)
-- Option A : Restaurer un dump sans données
-- pg_dump -h serveur-onpremise.local -U postgres -d ecommerce --schema-only > schema.sql
-- psql -h ecommerce.abc123.us-east-1.rds.amazonaws.com -U postgres -d ecommerce -f schema.sql

-- Option B : Utiliser un outil de migration de schéma
-- Option C : Scripts DDL manuels

-- 2.3. Vérifier que toutes les tables ont des clés primaires
SELECT
    schemaname,
    tablename
FROM pg_tables  
WHERE schemaname = 'public'  
AND tablename NOT IN (  
    SELECT tablename
    FROM pg_indexes
    WHERE indexdef LIKE '%PRIMARY KEY%'
);

-- Si des tables n'ont pas de PK, en ajouter :
-- ALTER TABLE ma_table ADD PRIMARY KEY (id);

-- 2.4. Ajuster les paramètres pour optimiser la réplication initiale
ALTER SYSTEM SET max_logical_replication_workers = 8;  
ALTER SYSTEM SET max_sync_workers_per_subscription = 4;  

-- Sur RDS, appliquer immédiatement :
SELECT pg_reload_conf();
```

##### Étape 3 : Création de la Subscription et Copie Initiale

```sql
-- 3.1. Toujours sur le serveur AWS RDS
-- Créer la subscription (la copie initiale va démarrer automatiquement)
CREATE SUBSCRIPTION sub_migration  
CONNECTION 'host=serveur-onpremise.local port=5432 dbname=ecommerce user=replication_user password=secure_password_2024 sslmode=require'  
PUBLICATION pub_migration  
WITH (  
    copy_data = true,           -- Copier les données existantes
    create_slot = true,         -- Créer automatiquement le slot
    enabled = true,             -- Activer immédiatement
    slot_name = 'sub_migration_slot'
);

-- 3.2. Surveiller la progression de la copie initiale
SELECT
    subname,
    pid,
    relid::regclass AS table_name,
    phase,
    received_lsn,
    last_msg_receipt_time
FROM pg_stat_subscription  
JOIN pg_subscription_rel ON pg_subscription.oid = pg_subscription_rel.srsubid  
WHERE subname = 'sub_migration';  

-- 3.3. Vérifier l'état global
SELECT
    subname,
    pid,
    received_lsn,
    latest_end_lsn,
    last_msg_receipt_time
FROM pg_stat_subscription;

-- Phase 'i' = initializing (copie initiale en cours)
-- Phase 'r' = replicating (réplication continue active)
-- Phase 'd' = data is being copied (copie de table spécifique)
-- Phase 's' = synchronized (synchronisé)
```

##### Étape 4 : Surveillance de la Synchronisation (Phase Cruciale)

```sql
-- 4.1. Surveiller le lag de réplication (sur AWS RDS)
SELECT
    subname,
    pg_size_pretty(
        pg_wal_lsn_diff(latest_end_lsn, received_lsn)
    ) AS replication_lag_bytes,
    latest_end_time - last_msg_receipt_time AS time_lag
FROM pg_stat_subscription  
WHERE subname = 'sub_migration';  

-- 4.2. Vérifier le slot de réplication (sur serveur source)
SELECT
    slot_name,
    database,
    active,
    restart_lsn,
    confirmed_flush_lsn,
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)
    ) AS retained_wal_size
FROM pg_replication_slots  
WHERE slot_name = 'sub_migration_slot';  

-- 4.3. Compter les lignes pour vérification (les deux serveurs)
-- Sur source :
SELECT 'commandes' AS table_name, COUNT(*) FROM commandes  
UNION ALL  
SELECT 'produits', COUNT(*) FROM produits  
UNION ALL  
SELECT 'clients', COUNT(*) FROM clients;  

-- Sur destination (même requête) :
-- Les nombres doivent converger et devenir identiques

-- 4.4. Créer un script de monitoring automatique (exemple bash)
-- Sauvegarder dans monitor_replication.sh :
```

```bash
#!/bin/bash
# monitor_replication.sh - Surveiller le lag de réplication

DEST_HOST="ecommerce.abc123.us-east-1.rds.amazonaws.com"  
DEST_USER="postgres"  
DEST_DB="ecommerce"  

while true; do
    echo "=== $(date) ==="

    psql -h $DEST_HOST -U $DEST_USER -d $DEST_DB -t -c "
        SELECT
            'Lag: ' || COALESCE(
                pg_size_pretty(pg_wal_lsn_diff(latest_end_lsn, received_lsn)),
                '0 bytes'
            ) AS lag,
            'Last received: ' || last_msg_receipt_time
        FROM pg_stat_subscription
        WHERE subname = 'sub_migration';
    "

    sleep 10
done
```

```sql
-- 4.5. Critère de validation : lag < 1 MB et stable
-- Une fois ce critère atteint, la migration peut être planifiée
```

##### Étape 5 : Planification de la Bascule (Cutover)

**Checklist Pre-Bascule** :

```
□ Le lag de réplication est inférieur à 1 MB
□ La réplication est stable depuis au moins 24 heures
□ Les comptages de lignes sont identiques
□ Les équipes sont informées et disponibles
□ Un plan de rollback est documenté
□ Les scripts de bascule sont testés en staging
□ Une fenêtre de maintenance est planifiée (même si courte)
□ Les sauvegardes sont à jour sur les deux serveurs
□ Les endpoints DNS/Load Balancer sont préparés
□ Les paramètres de connexion applicatifs sont prêts
```

##### Étape 6 : Exécution de la Bascule (D-Day)

**Timeline de Bascule (Exemple : Fenêtre de 23h00 à 23h15)** :

```
T-15 min (22h45) : Notification aux utilisateurs
├─ "Maintenance planifiée dans 15 minutes"
├─ Dernière vérification du lag

T-5 min (22h55) : Préparation finale
├─ Mode lecture seule sur l'application (optionnel)
├─ Dernière vérification lag < 100 KB

T-0 (23h00) : DÉBUT DE LA BASCULE
├─ 23h00:00 → Passer l'application en mode maintenance
├─ 23h00:30 → Attendre que les transactions en cours se terminent
├─ 23h01:00 → Vérifier que le lag est à 0

T+1 min (23h01) : Arrêt de la réplication
├─ Sur AWS RDS : ALTER SUBSCRIPTION sub_migration DISABLE;
├─ Sur Source : Vérifier qu'aucune nouvelle transaction n'arrive

T+2 min (23h02) : Validation finale des données
├─ Comptage des lignes (automatisé)
├─ Vérification d'intégrité référentielle
├─ Tests de requêtes critiques

T+3 min (23h03) : Reconfiguration applicative
├─ Changer la chaîne de connexion vers AWS RDS
├─ Redémarrer les services applicatifs

T+5 min (23h05) : Tests de validation
├─ Test de connexion
├─ Test d'écriture (INSERT test)
├─ Test de lecture (SELECT test)
├─ Test des APIs critiques

T+7 min (23h07) : Mise en production
├─ Retirer le mode maintenance
├─ Activer le monitoring renforcé
├─ Surveillance des logs applicatifs

T+10 min (23h10) : Validation post-bascule
├─ Vérifier les métriques (connexions, CPU, I/O)
├─ Tester les fonctionnalités critiques
├─ Confirmer avec les équipes métier

T+15 min (23h15) : FIN DE LA FENÊTRE
└─ Migration terminée avec succès
```

**Scripts SQL de Bascule** :

```sql
-- SCRIPT 1 : Sur AWS RDS - Désactivation de la subscription
-- À exécuter à T+1 min
ALTER SUBSCRIPTION sub_migration DISABLE;

-- SCRIPT 2 : Validation des données (sur AWS RDS)
-- Comptage automatisé
DO $$  
DECLARE  
    source_count BIGINT;
    dest_count BIGINT;
    table_rec RECORD;
BEGIN
    FOR table_rec IN
        SELECT tablename
        FROM pg_tables
        WHERE schemaname = 'public'
    LOOP
        EXECUTE format('SELECT COUNT(*) FROM %I', table_rec.tablename) INTO dest_count;
        RAISE NOTICE 'Table % : % rows', table_rec.tablename, dest_count;
    END LOOP;
END $$;

-- SCRIPT 3 : Test d'écriture (sur AWS RDS)
-- Vérifier que les écritures fonctionnent
CREATE TABLE test_write_check (
    id SERIAL PRIMARY KEY,
    test_time TIMESTAMP DEFAULT NOW(),
    test_data TEXT
);

INSERT INTO test_write_check (test_data) VALUES ('Migration test - écriture OK');  
SELECT * FROM test_write_check;  
DROP TABLE test_write_check;  

-- SCRIPT 4 : Vérification de l'intégrité référentielle
SELECT
    conname AS constraint_name,
    conrelid::regclass AS table_name,
    confrelid::regclass AS referenced_table
FROM pg_constraint  
WHERE contype = 'f';  
-- Toutes les contraintes FK doivent être intactes
```

##### Étape 7 : Surveillance Post-Bascule

```sql
-- 7.1. Monitoring des performances (premières 24 heures)
-- Sur AWS RDS

-- Activité des connexions
SELECT
    count(*) AS active_connections,
    state,
    wait_event_type
FROM pg_stat_activity  
WHERE datname = 'ecommerce'  
GROUP BY state, wait_event_type;  

-- Cache hit ratio (doit être > 95%)
SELECT
    sum(heap_blks_hit) / nullif(sum(heap_blks_hit + heap_blks_read), 0) * 100 AS cache_hit_ratio
FROM pg_statio_user_tables;

-- Requêtes lentes (nécessite pg_stat_statements)
SELECT
    query,
    calls,
    mean_exec_time,
    max_exec_time
FROM pg_stat_statements  
ORDER BY mean_exec_time DESC  
LIMIT 10;  

-- 7.2. Alertes à configurer
-- - CPU > 70% pendant > 5 minutes
-- - Connexions actives > 80% de max_connections
-- - Cache hit ratio < 90%
-- - Slow queries > 1 seconde
-- - Disk space < 20%
```

##### Étape 8 : Nettoyage et Désactivation de l'Ancien Serveur

```sql
-- 8.1. Après validation complète (J+7), nettoyer la réplication

-- Sur AWS RDS (destination) :
DROP SUBSCRIPTION sub_migration;

-- Sur serveur source (on-premise) :
DROP PUBLICATION pub_migration;

-- Supprimer le slot de réplication (si pas auto-supprimé)
SELECT pg_drop_replication_slot('sub_migration_slot');

-- 8.2. Archivage de l'ancien serveur
-- - Conserver pendant 30 jours minimum
-- - Faire un backup final complet
-- - Documenter pour audit
-- - Planifier désactivation définitive
```

##### Plan de Rollback (En Cas de Problème)

**Si la bascule échoue pendant la fenêtre** :

```sql

```

## Partie 2 : Réplication Sélective

### Qu'est-ce que la Réplication Sélective ?

La **réplication sélective** consiste à ne répliquer qu'une partie des données d'une base vers une autre, selon des critères précis :
- Certaines tables seulement
- Certaines colonnes d'une table
- Certaines lignes selon un filtre WHERE
- Certaines opérations (INSERT mais pas DELETE)

Cette approche est essentielle pour :
- Respecter les contraintes de conformité (RGPD, HIPAA, etc.)
- Optimiser les coûts (ne stocker que ce qui est nécessaire)
- Améliorer les performances (réduire le volume de données)
- Créer des environnements spécialisés (analytics, reporting, dev)

### Cas d'Usage 3 : Séparation OLTP / OLAP

#### Scénario : Base de Reporting Dédiée

**Contexte** :
- Base OLTP : PostgreSQL en production avec transactions intensives
- Besoin : Une base OLAP pour analytics et Business Intelligence
- Contrainte : Ne pas impacter les performances de production

**Architecture** :

```
┌───────────────────────────────────────────────────────────┐
│                    BASE DE PRODUCTION                     │
│                        (OLTP)                             │
│   - Transactions en temps réel                            │
│   - Écritures fréquentes                                  │
│   - Données opérationnelles                               │
│                                                           │
│   Tables :                                                │
│   ├─ commandes (toutes colonnes)                          │
│   ├─ clients (toutes colonnes)                            │
│   ├─ produits (toutes colonnes)                           │
│   ├─ logs_audit (NON répliqué)                            │
│   ├─ sessions_temp (NON répliqué)                         │
│   └─ cache_redis (NON répliqué)                           │
│                                                           │
│        │                                                  │
│        │ Publication Sélective                            │
│        │ (tables opérationnelles uniquement)              │
│        ▼                                                  │
│                                                           │
│   ┌────────────────────────────────────────┐              │
│   │        Réplication Logique             │              │
│   │  (async, sans impact sur production)   │              │
│   └────────────────────────────────────────┘              │
│                                                           │
│        ▼                                                  │
│                                                           │
│                  BASE DE REPORTING                        │
│                       (OLAP)                              │
│   - Agrégations lourdes                                   │
│   - Lectures complexes                                    │
│   - Historisation longue                                  │
│                                                           │
│   Tables :                                                │
│   ├─ commandes (répliquées)                               │
│   ├─ clients (répliquées)                                 │
│   ├─ produits (répliquées)                                │
│   ├─ vues_materialisees (locales)                         │
│   ├─ agregations (locales)                                │
│   └─ cubes_olap (locaux)                                  │
└───────────────────────────────────────────────────────────┘
```

**Implémentation** :

```sql
-- ===== SUR LA BASE DE PRODUCTION (OLTP) =====

-- 1. Créer une publication sélective (seulement les tables métier)
CREATE PUBLICATION pub_reporting  
FOR TABLE  
    commandes,
    clients,
    produits,
    categories,
    paiements
WITH (publish = 'insert, update');  -- Pas de DELETE pour conserver l'historique

-- Note : Les tables logs_audit, sessions_temp ne sont PAS dans la publication

-- 2. Vérifier la publication
SELECT * FROM pg_publication_tables WHERE pubname = 'pub_reporting';


-- ===== SUR LA BASE DE REPORTING (OLAP) =====

-- 1. Créer le même schéma (seulement les tables nécessaires)
-- Les tables peuvent avoir des index différents, optimisés pour le reporting

CREATE TABLE commandes (
    id SERIAL PRIMARY KEY,
    client_id INT NOT NULL,
    date_commande TIMESTAMP NOT NULL,
    montant_total NUMERIC(10,2),
    statut VARCHAR(50),
    -- Index optimisé pour analytics
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_commandes_date ON commandes(date_commande);  
CREATE INDEX idx_commandes_client ON commandes(client_id);  
CREATE INDEX idx_commandes_created ON commandes(created_at);  

-- Répéter pour autres tables...

-- 2. Créer la subscription
CREATE SUBSCRIPTION sub_reporting  
CONNECTION 'host=prod-db.example.com port=5432 dbname=production user=replication_user password=xxx sslmode=require'  
PUBLICATION pub_reporting  
WITH (  
    copy_data = true,
    synchronous_commit = 'off'  -- Async pour ne pas ralentir la prod
);

-- 3. Créer des vues matérialisées pour l'analytics (local uniquement)
CREATE MATERIALIZED VIEW mv_ventes_par_jour AS  
SELECT  
    DATE(date_commande) AS jour,
    COUNT(*) AS nombre_commandes,
    SUM(montant_total) AS ca_jour,
    AVG(montant_total) AS panier_moyen
FROM commandes  
WHERE statut = 'payee'  
GROUP BY DATE(date_commande);  

CREATE INDEX ON mv_ventes_par_jour(jour);

-- Rafraîchir périodiquement (via cron ou pg_cron)
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_ventes_par_jour;
```

**Avantages de cette Architecture** :

- ✅ **Isolation des charges** : Les requêtes analytics n'impactent pas la production  
- ✅ **Optimisation différenciée** : Index et structures adaptés à chaque usage  
- ✅ **Flexibilité** : Possibilité d'agréger, transformer, archiver côté OLAP  
- ✅ **Coût optimisé** : Le serveur OLAP peut être dimensionné différemment

### Cas d'Usage 4 : Réplication Géographique avec Filtrage

#### Scénario : Application Multi-Régions

**Contexte** :
- Application SaaS mondiale
- Chaque région ne doit voir que ses propres données (conformité RGPD)
- Base centrale + bases régionales

**Architecture** :

```
                    ┌─────────────────────┐
                    │   BASE CENTRALE     │
                    │   (Toutes régions)  │
                    │                     │
                    │  Table: commandes   │
                    │  - id               │
                    │  - client_id        │
                    │  - region: 'EU'/'US'│
                    │  - montant          │
                    │  - ...              │
                    └─────────────────────┘
                             │
                             │ Publications avec WHERE
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
     ┌────────────┐  ┌────────────┐  ┌────────────┐
     │  BASE EU   │  │  BASE US   │  │  BASE APAC │
     │  (Europe)  │  │ (Amérique) │  │   (Asie)   │
     │            │  │            │  │            │
     │ Seulement  │  │ Seulement  │  │ Seulement  │
     │ region='EU'│  │ region='US'│  │region='APAC│
     └────────────┘  └────────────┘  └────────────┘
```

**Implémentation avec Row-Level Filtering** :

```sql
-- ===== SUR LA BASE CENTRALE =====

-- 1. Publication pour l'Europe (avec filtre WHERE)
CREATE PUBLICATION pub_europe  
FOR TABLE commandes WHERE (region = 'EU'),  
         clients WHERE (region = 'EU'),
         produits;  -- Tous les produits dans toutes les régions

-- Vérification

    pubname,
    schemaname,

    rowfilter
FROM pg_publication_tables  
WHERE pubname LIKE 'pub_%';  


-- ===== SUR CHAQUE BASE RÉGIONALE =====
```

### Cas d'Usage 5 : Réplication Partielle pour Environnement de Développement

#### Scénario : Environnement de Dev avec Données Réelles Anonymisées

**Contexte** :
- Les développeurs ont besoin de données réalistes
- Impossibilité de copier toute la production (trop volumineuse + données sensibles)
- Solution : Réplication sélective + anonymisation

**Implémentation** :

```sql
-- ===== SUR LA BASE DE PRODUCTION =====

-- 1. Créer une publication sans colonnes sensibles
CREATE PUBLICATION pub_dev_safe  
FOR TABLE  
    commandes (id, date_commande, montant_total, statut),  -- Sans adresse livraison
    produits,  -- Table complète (pas de données sensibles)
    categories;

-- 2. Exclure les tables sensibles
-- clients, paiements, logs_audit ne sont PAS dans la publication


-- ===== SUR LA BASE DE DÉVELOPPEMENT =====

-- 1. Créer les tables avec structure identique
CREATE TABLE commandes (
    id INT PRIMARY KEY,
    date_commande TIMESTAMP,
    montant_total NUMERIC(10,2),
    statut VARCHAR(50),
    -- colonnes sensibles absentes
    created_at TIMESTAMP DEFAULT NOW()
);

-- 2. Créer la subscription avec limitation du volume
CREATE SUBSCRIPTION sub_dev  
CONNECTION 'host=prod-db.example.com ...'  
PUBLICATION pub_dev_safe  
WITH (  
    copy_data = true,
    synchronous_commit = 'off'
);

-- 3. Anonymiser les données après copie initiale (si nécessaire)
-- Exemple : Remplacer les emails par des fakes
UPDATE clients  
SET email = 'user' || id || '@example.com'  
WHERE email NOT LIKE '%@example.com';  

-- 4. Limiter la rétention des données (purge périodique)
-- Via un CRON job ou pg_cron
DELETE FROM commandes  
WHERE date_commande < NOW() - INTERVAL '90 days';  
```

### Cas d'Usage 6 : Réplication avec Transformation de Colonnes

#### Scénario : Anonymisation à la Volée

**Problématique** :
- Besoin de répliquer des données pour analytics
- Certaines colonnes doivent être anonymisées (RGPD)

**Solution : Utiliser des Triggers Côté Destination** :

```sql
-- ===== SUR LA BASE DE DESTINATION (ANALYTICS) =====

-- 1. Tables créées avec colonnes identiques
CREATE TABLE clients (
    id INT PRIMARY KEY,
    nom VARCHAR(100),
    prenom VARCHAR(100),
    email VARCHAR(200),
    telephone VARCHAR(20),
    adresse TEXT,
    ville VARCHAR(100),
    code_postal VARCHAR(10),
    created_at TIMESTAMP
);

-- 2. Créer un trigger d'anonymisation
CREATE OR REPLACE FUNCTION anonymize_pii()  
RETURNS TRIGGER AS $$  
BEGIN  
    -- Anonymiser les données personnelles
    NEW.nom := 'CLIENT_' || NEW.id;
    NEW.prenom := 'ANONYME';
    NEW.email := 'anonyme' || NEW.id || '@example.com';
    NEW.telephone := '0600000000';
    NEW.adresse := 'Adresse anonymisée';

    -- Conserver ville et code postal pour analytics géographiques
    -- (pas de données personnelles identifiables)

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- 3. Attacher le trigger
CREATE TRIGGER trg_anonymize_clients  
BEFORE INSERT OR UPDATE ON clients  
FOR EACH ROW  
EXECUTE FUNCTION anonymize_pii();  

-- 4. Créer la subscription (les données seront auto-anonymisées)
CREATE SUBSCRIPTION sub_analytics_anonymized  
CONNECTION 'host=prod-db.example.com ...'  
PUBLICATION pub_clients;  

-- Test : Les données insérées sont automatiquement anonymisées
SELECT id, nom, prenom, email, ville FROM clients LIMIT 5;
-- Résultat :
--  id  |    nom     |  prenom  |          email           |   ville
-- -----+------------+----------+--------------------------+-----------
--  1   | CLIENT_1   | ANONYME  | anonyme1@example.com     | Paris
--  2   | CLIENT_2   | ANONYME  | anonyme2@example.com     | Lyon
```

### Cas d'Usage 7 : Réplication Multi-Sources (Consolidation)

#### Scénario : Agrégation de Données depuis Plusieurs Bases

**Contexte** :
- Plusieurs filiales avec leurs propres bases PostgreSQL
- Besoin d'agréger toutes les données dans une base centrale pour reporting

**Architecture** :

```
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│  FILIALE A   │   │  FILIALE B   │   │  FILIALE C   │
│  (Region 1)  │   │  (Region 2)  │   │  (Region 3)  │
│              │   │              │   │              │
│  Commandes   │   │  Commandes   │   │  Commandes   │
│  Clients     │   │  Clients     │   │  Clients     │
└──────────────┘   └──────────────┘   └──────────────┘
       │                  │                  │
       │ Publication      │ Publication      │ Publication
       │                  │                  │
       └──────────────────┼──────────────────┘
                          ▼
                 ┌────────────────┐
                 │  BASE CENTRALE │
                 │   (Reporting)  │
                 │                │
                 │  3 Subscriptions
                 │                │
                 │  commandes_a   │
                 │  commandes_b   │
                 │  commandes_c   │
                 └────────────────┘
```

**Implémentation** :

```sql
-- ===== SUR CHAQUE BASE FILIALE (A, B, C) =====

-- Filiale A :
CREATE PUBLICATION pub_filiale_a FOR TABLE commandes, clients;

-- Filiale B :
CREATE PUBLICATION pub_filiale_b FOR TABLE commandes, clients;

-- Filiale C :
CREATE PUBLICATION pub_filiale_c FOR TABLE commandes, clients;


-- ===== SUR LA BASE CENTRALE =====

-- 1. Créer des tables distinctes par source (pour éviter les conflits de PK)
CREATE TABLE commandes_filiale_a (
    LIKE commandes INCLUDING ALL,
    source_filiale VARCHAR(10) DEFAULT 'A'
);

CREATE TABLE commandes_filiale_b (
    LIKE commandes INCLUDING ALL,
    source_filiale VARCHAR(10) DEFAULT 'B'
);

CREATE TABLE commandes_filiale_c (
    LIKE commandes INCLUDING ALL,
    source_filiale VARCHAR(10) DEFAULT 'C'
);

-- 2. Créer les subscriptions
CREATE SUBSCRIPTION sub_filiale_a  
CONNECTION 'host=filiale-a.example.com ...'  
PUBLICATION pub_filiale_a;  

CREATE SUBSCRIPTION sub_filiale_b  
CONNECTION 'host=filiale-b.example.com ...'  
PUBLICATION pub_filiale_b;  

CREATE SUBSCRIPTION sub_filiale_c  
CONNECTION 'host=filiale-c.example.com ...'  
PUBLICATION pub_filiale_c;  

-- 3. Créer une vue unifiée pour le reporting
CREATE VIEW commandes_toutes_filiales AS  
SELECT *, 'A' AS filiale FROM commandes_filiale_a  
UNION ALL  
SELECT *, 'B' AS filiale FROM commandes_filiale_b  
UNION ALL  
SELECT *, 'C' AS filiale FROM commandes_filiale_c;  

-- 4. Requêtes analytics sur la vue unifiée
SELECT
    filiale,
    COUNT(*) AS nombre_commandes,
    SUM(montant_total) AS ca_total
FROM commandes_toutes_filiales  
GROUP BY filiale;  
```

## Partie 3 : Bonnes Pratiques et Pièges à Éviter

### Bonnes Pratiques pour les Migrations

#### 1. Planification

✅ **Faire un dry-run complet** :
- Tester la migration sur une copie de production
- Chronométrer chaque étape
- Documenter les problèmes rencontrés

✅ **Préparer un runbook détaillé** :
- Séquence exacte des commandes
- Scripts SQL prêts à l'exécution
- Numéros de téléphone des personnes-clés
- Plan de rollback clair

✅ **Communiquer largement** :
- Informer les parties prenantes
- Prévoir une fenêtre de maintenance (même si courte)
- Avoir une personne de chaque équipe disponible

#### 2. Validation

✅ **Valider la copie initiale** :
```sql
-- Script de validation automatique
DO $$  
DECLARE  
    table_rec RECORD;
    source_count BIGINT;
    dest_count BIGINT;
BEGIN
    FOR table_rec IN
        SELECT tablename
        FROM pg_tables
        WHERE schemaname = 'public'
    LOOP
        -- Compter sur source (via dblink ou script externe)
        -- Compter sur destination
        EXECUTE format('SELECT COUNT(*) FROM %I', table_rec.tablename) INTO dest_count;

        -- Comparer et logger
        RAISE NOTICE 'Table %: % rows', table_rec.tablename, dest_count;
    END LOOP;
END $$;
```

✅ **Tester les performances** :
```sql
-- Comparer les plans d'exécution critiques
EXPLAIN (ANALYZE, BUFFERS)  
SELECT ... FROM ... WHERE ...;  

-- Exécuter sur les deux serveurs et comparer
```

#### 3. Monitoring

✅ **Surveiller pendant et après** :
- Lag de réplication
- Utilisation CPU/RAM/Disque
- Connexions actives
- Slow queries
- Erreurs dans les logs

### Pièges à Éviter

#### ❌ Piège 1 : Oublier les Séquences

**Problème** :
Les séquences ne sont pas répliquées automatiquement. Si le serveur destination génère des IDs, vous aurez des conflits.

**Solution** :
```sql
-- Après la bascule, synchroniser les séquences
SELECT setval('commandes_id_seq', (SELECT MAX(id) FROM commandes));  
SELECT setval('clients_id_seq', (SELECT MAX(id) FROM clients));  
-- Répéter pour toutes les séquences
```

#### ❌ Piège 2 : DDL Non Répliqués

**Problème** :
Les changements de schéma (ALTER TABLE, CREATE INDEX) ne sont pas répliqués.

**Solution** :
- Appliquer manuellement les DDL sur les deux serveurs
- Documenter tous les changements de schéma
- Utiliser un outil de migration de schéma (Flyway, Liquibase)

```sql
-- Procédure recommandée pour un ALTER TABLE en production :
-- 1. Appliquer sur le serveur destination AVANT la bascule
-- 2. Appliquer sur le serveur source
-- 3. Attendre la synchronisation
-- 4. Procéder à la bascule
```

#### ❌ Piège 3 : Slots de Réplication qui Grossissent

**Problème** :
Si une subscription est arrêtée, le slot continue de retenir le WAL, risquant de saturer le disque.

**Solution** :
```sql
-- Surveiller régulièrement
SELECT
    slot_name,
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)
    ) AS retained_wal
FROM pg_replication_slots;

-- Supprimer les slots inutilisés
SELECT pg_drop_replication_slot('slot_name');
```

#### ❌ Piège 4 : Conflits de Clés Primaires

**Problème** :
Si les deux serveurs génèrent des ID simultanément, vous aurez des conflits.

**Solution** :
```sql
-- Option 1 : Utiliser des plages d'ID distinctes
-- Serveur A : 1-1000000000
ALTER SEQUENCE commandes_id_seq RESTART WITH 1 INCREMENT BY 2;

-- Serveur B : 2-1000000001
ALTER SEQUENCE commandes_id_seq RESTART WITH 2 INCREMENT BY 2;

-- Option 2 : Utiliser UUID au lieu d'ID séquentiels
CREATE TABLE commandes (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    ...
);

-- Option 3 (PostgreSQL 18) : Utiliser UUIDv7 (trié temporellement)
CREATE TABLE commandes (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    ...
);
```

#### ❌ Piège 5 : Négliger les Extensions

**Problème** :
Les extensions doivent être installées sur le serveur destination AVANT la réplication.

**Solution** :
```sql
-- Lister les extensions sur le source
SELECT * FROM pg_extension;

-- Installer sur la destination (avant la subscription)
CREATE EXTENSION IF NOT EXISTS postgis;  
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;  
CREATE EXTENSION IF NOT EXISTS pg_trgm;  
```

### Checklist de Migration (Récapitulatif)

**Phase Préparation** :
```
□ Analyser la base source (taille, volume, structure)
□ Dimensionner le serveur destination
□ Tester la migration sur un environnement de staging
□ Préparer les scripts SQL
□ Documenter le runbook
□ Configurer le monitoring
□ Former les équipes
□ Planifier une fenêtre de maintenance
```

**Phase Exécution** :
```
□ Créer la publication sur le source
□ Créer les structures sur la destination
□ Créer la subscription
□ Surveiller la copie initiale
□ Attendre la synchronisation (lag < 1 MB)
□ Valider les données
□ Effectuer la bascule
□ Tester les fonctionnalités critiques
□ Activer le monitoring renforcé
```

**Phase Post-Migration** :
```
□ Surveiller les performances (24-48h)
□ Valider avec les équipes métier
□ Documenter les incidents/résolutions
□ Nettoyer la réplication (après validation)
□ Archiver l'ancien serveur
□ Mise à jour de la documentation
```

## Conclusion

La réplication logique PostgreSQL, grâce aux mécanismes de **publications** et **subscriptions**, offre une flexibilité inégalée pour :

1. **Migrer** des bases de données sans interruption de service  
2. **Répliquer sélectivement** exactement les données nécessaires  
3. **Créer des architectures distribuées** adaptées à chaque besoin métier  
4. **Respecter les contraintes réglementaires** (RGPD, localisation des données)  
5. **Optimiser les coûts** en ne répliquant que l'essentiel

Les cas d'usage présentés montrent que la réplication logique n'est pas qu'un outil de haute disponibilité : c'est une **brique fondamentale** des architectures modernes, permettant de concilier performance, flexibilité et conformité.

### Points Clés à Retenir

- 📌 **Migrations Zero-Downtime** = La réplication logique permet des migrations avec seulement quelques secondes d'interruption  
- 📌 **Sélectivité Fine** = Filtrage par table, colonne, ligne (WHERE), opération  
- 📌 **Compatibilité Multi-Versions** = Possible entre PostgreSQL 13 et 18  
- 📌 **Planification Essentielle** = Une migration réussie repose sur une préparation minutieuse  
- 📌 **Monitoring Continu** = Surveiller le lag, les slots, les performances  
- 📌 **Rollback Préparé** = Toujours avoir un plan B documenté
