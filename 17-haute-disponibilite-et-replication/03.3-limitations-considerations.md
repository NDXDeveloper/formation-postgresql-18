# 17.3.3. Limitations et Considérations de la Réplication Logique

## Introduction

La réplication logique PostgreSQL est un outil puissant et flexible, mais comme toute technologie, elle a ses **limitations** et nécessite certaines **considérations** importantes. Comprendre ces contraintes est essentiel pour :

- **Éviter les surprises** en production  
- **Concevoir une architecture** adaptée à vos besoins  
- **Prendre des décisions éclairées** entre réplication logique et physique  
- **Anticiper les problèmes** avant qu'ils ne surviennent

Cette section est cruciale car elle vous permettra d'identifier si la réplication logique est la bonne solution pour votre cas d'usage, et comment contourner ses limitations quand c'est possible.

---

## Partie 1 : Ce Qui N'est PAS Répliqué (Limitations Techniques)

### 1.1. Les Modifications de Schéma (DDL)

#### Le Problème

**Les opérations DDL (Data Definition Language) ne sont PAS répliquées automatiquement** par la réplication logique.

Cela inclut :
- `CREATE TABLE`, `ALTER TABLE`, `DROP TABLE`  
- `CREATE INDEX`, `DROP INDEX`  
- `ALTER COLUMN` (ajout, suppression, modification de type)  
- `ADD CONSTRAINT`, `DROP CONSTRAINT`  
- `CREATE SEQUENCE`, `ALTER SEQUENCE`  
- `CREATE FUNCTION`, `CREATE TRIGGER`  
- `GRANT`, `REVOKE` (permissions)

**Exemple de scénario problématique** :

```sql
-- ===== SUR LE SERVEUR SOURCE =====
-- Jour 1 : Tout fonctionne
CREATE PUBLICATION pub_prod FOR ALL TABLES;

-- Jour 30 : Ajout d'une colonne
ALTER TABLE commandes ADD COLUMN tracking_number VARCHAR(50);

-- Insertion avec la nouvelle colonne
INSERT INTO commandes (client_id, montant, tracking_number)  
VALUES (123, 99.99, 'TRACK123456');  


-- ===== SUR LE SERVEUR DESTINATION =====
-- La table n'a toujours que les anciennes colonnes !
-- L'ALTER TABLE n'a pas été répliqué

-- Résultat : ERREUR DE RÉPLICATION
-- ERROR: logical replication target relation "commandes" is missing column "tracking_number"
```

#### Solutions et Contournements

**Solution 1 : Gestion Manuelle (Recommandée)**

```sql
-- Procédure sécurisée pour ajouter une colonne :

-- ÉTAPE 1 : Sur le serveur DESTINATION d'abord
ALTER TABLE commandes ADD COLUMN tracking_number VARCHAR(50);

-- ÉTAPE 2 : Sur le serveur SOURCE
ALTER TABLE commandes ADD COLUMN tracking_number VARCHAR(50);

-- ÉTAPE 3 : Vérifier que la réplication fonctionne toujours
SELECT * FROM pg_stat_subscription;
```

**Solution 2 : Scripts de Synchronisation**

```bash
#!/bin/bash
# sync_ddl.sh - Appliquer un changement DDL sur tous les serveurs

DDL_SCRIPT=$1

# Liste des serveurs
SERVERS=("prod-source.example.com" "replica1.example.com" "replica2.example.com")

for server in "${SERVERS[@]}"; do
    echo "Applying DDL on $server..."
    psql -h $server -U admin -d production -f $DDL_SCRIPT

    if [ $? -eq 0 ]; then
        echo "✓ Success on $server"
    else
        echo "✗ FAILED on $server"
        exit 1
    fi
done

echo "DDL synchronized on all servers"
```

**Solution 3 : Outils de Migration de Schéma**

Utiliser des outils comme :
- **Flyway** : Gestion de versions de schéma SQL  
- **Liquibase** : Suivi des changements de base de données  
- **Alembic** : Migration pour Python/SQLAlchemy

```sql
-- Exemple Flyway : V1__add_tracking_column.sql
-- Ce script sera appliqué sur tous les serveurs de manière contrôlée

ALTER TABLE commandes ADD COLUMN tracking_number VARCHAR(50);

-- Flyway garantit que le script est exécuté une seule fois
-- et sur tous les environnements de manière cohérente
```

**⚠️ Implications Importantes**

```
□ Documenter TOUS les changements de schéma
□ Tester les changements DDL en staging d'abord
□ Avoir une procédure claire pour appliquer les DDL en production
□ Utiliser un système de gestion de versions de schéma
□ Former les équipes sur cette limitation
```

### 1.2. Les Séquences (SEQUENCE)

#### Le Problème

**Les valeurs de séquences ne sont PAS synchronisées** entre le serveur source et destination.

```sql
-- ===== SUR LE SERVEUR SOURCE =====
SELECT currval('commandes_id_seq');  -- Valeur : 1000

-- ===== SUR LE SERVEUR DESTINATION =====
SELECT currval('commandes_id_seq');  -- Valeur : 1 (valeur initiale!)

-- Si le serveur destination génère aussi des IDs :
INSERT INTO commandes (...) VALUES (...);  -- Génère id = 1
-- CONFLIT avec les données répliquées qui ont déjà id = 1 !
```

**Scénario de Conflit** :

```
┌─────────────────────────────────────────────────────────┐
│ Serveur SOURCE                                          │
│ Séquence commandes_id_seq : current value = 5000        │
│                                                         │
│ INSERT INTO commandes → id = 5001 ────┐                 │
│                              (nouveau) │                │
└────────────────────────────────────────┼────────────────┘
                                         │
                                         │ Réplication
                                         ▼
┌─────────────────────────────────────────────────────────┐
│ Serveur DESTINATION                                     │
│ Séquence commandes_id_seq : current value = 1 (initial) │
│                                                         │
│ Reçoit : id = 5001 → OK (insertion réussie)             │
│                                                         │
│ Si application génère un ID localement :                │
│ INSERT INTO commandes → id = 1 ────> ERREUR !           │
│                              (conflit avec données      │
│                               répliquées précédemment)  │
└─────────────────────────────────────────────────────────┘
```

#### Solutions et Contournements

**Solution 1 : Synchronisation Manuelle Après Migration**

```sql
-- Après avoir basculé vers le serveur destination,
-- synchroniser toutes les séquences :

-- Méthode automatique pour toutes les séquences
DO $$  
DECLARE  
    seq_record RECORD;
    max_val BIGINT;
BEGIN
    FOR seq_record IN
        SELECT
            sequence_schema,
            sequence_name,
            -- Extraire le nom de la table et colonne depuis la séquence
            REPLACE(sequence_name, '_id_seq', '') AS table_name
        FROM information_schema.sequences
        WHERE sequence_schema = 'public'
    LOOP
        -- Récupérer la valeur maximale dans la table
        EXECUTE format(
            'SELECT COALESCE(MAX(id), 0) FROM %I.%I',
            seq_record.sequence_schema,
            seq_record.table_name
        ) INTO max_val;

        -- Ajuster la séquence
        EXECUTE format(
            'SELECT setval(%L, %s)',
            seq_record.sequence_schema || '.' || seq_record.sequence_name,
            max_val
        );

        RAISE NOTICE 'Séquence % ajustée à %',
            seq_record.sequence_name, max_val;
    END LOOP;
END $$;
```

**Solution 2 : Utiliser UUID au Lieu d'ID Séquentiels**

```sql
-- Remplacer les SERIAL par UUID
-- Avantage : Pas de conflit possible entre serveurs

-- Ancienne méthode (avec conflit potentiel)
CREATE TABLE commandes (
    id SERIAL PRIMARY KEY,  -- Peut causer des conflits
    ...
);

-- Nouvelle méthode (sans conflit)
CREATE TABLE commandes (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    ...
);

-- PostgreSQL 18 : UUIDv7 (trié temporellement, meilleur pour les index)
CREATE TABLE commandes (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    ...
);
```

**Comparaison UUID vs SERIAL** :

| Aspect | SERIAL/BIGSERIAL | UUID | UUIDv7 (PG18) |
|--------|------------------|------|---------------|
| Taille | 4 ou 8 bytes | 16 bytes | 16 bytes |
| Lisibilité | ✅ Facile (1, 2, 3...) | ❌ Difficile | ❌ Difficile |
| Tri naturel | ✅ Oui | ❌ Non | ✅ Oui (temps) |
| Conflits multi-serveurs | ❌ Risque élevé | ✅ Quasi impossible | ✅ Quasi impossible |
| Performance index | ✅ Excellent | ⚠️ Moyen (aléatoire) | ✅ Très bon |
| Recommandé pour | Single-master | Multi-master | Multi-master (PG18+) |

**Solution 3 : Plages d'ID Distinctes (Si SERIAL Requis)**

```sql
-- Serveur SOURCE : IDs impairs
ALTER SEQUENCE commandes_id_seq RESTART WITH 1 INCREMENT BY 2;

-- Serveur DESTINATION : IDs pairs
ALTER SEQUENCE commandes_id_seq RESTART WITH 2 INCREMENT BY 2;

-- Résultat :
-- Source génère : 1, 3, 5, 7, 9, 11...
-- Destination génère : 2, 4, 6, 8, 10, 12...
-- ✅ Aucun conflit possible
```

### 1.3. Les Large Objects (BLOB)

#### Le Problème

**Les Large Objects (LO) stockés via l'interface `pg_largeobject` ne sont PAS répliqués.**

```sql
-- Cette approche NE FONCTIONNE PAS avec la réplication logique
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    fichier_oid OID  -- Large Object ID → NON répliqué !
);

-- Insertion d'un fichier
SELECT lo_import('/path/to/file.pdf');  -- Retourne OID : 16385

INSERT INTO documents (nom, fichier_oid) VALUES ('contrat.pdf', 16385);

-- Sur le serveur destination :
-- L'ID 16385 est répliqué dans la table
-- MAIS le contenu du fichier (pg_largeobject) n'est PAS répliqué !
-- Résultat : OID invalide sur le replica
```

#### Solutions et Contournements

**Solution 1 : Utiliser BYTEA (Recommandée)**

```sql
-- Stocker les fichiers en BYTEA (byte array)
-- ✅ Répliqué automatiquement

CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    mime_type VARCHAR(100),
    contenu BYTEA  -- Fichier stocké directement
);

-- Insertion d'un fichier
INSERT INTO documents (nom, mime_type, contenu)  
VALUES (  
    'contrat.pdf',
    'application/pdf',
    pg_read_binary_file('/path/to/file.pdf')
);

-- Ou depuis une application (Python exemple)
-- import psycopg2
-- with open('file.pdf', 'rb') as f:
--     data = f.read()
--     cursor.execute(
--         "INSERT INTO documents (nom, contenu) VALUES (%s, %s)",
--         ('contrat.pdf', data)
--     )
```

**Limitations de BYTEA** :
- ⚠️ Limite : 1 GB par champ (mais suffisant pour la plupart des fichiers)  
- ⚠️ Les fichiers sont chargés en mémoire lors de la récupération  
- ✅ Avantage : Sauvegarde/réplication automatique

**Solution 2 : Stockage Externe (S3, Object Storage)**

```sql
-- Stocker seulement les métadonnées et un pointeur vers le stockage externe
CREATE TABLE documents (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    nom VARCHAR(200),
    mime_type VARCHAR(100),
    s3_bucket VARCHAR(100),
    s3_key VARCHAR(500),  -- Chemin du fichier dans S3
    taille_bytes BIGINT,
    created_at TIMESTAMP DEFAULT NOW()
);

-- Insertion
INSERT INTO documents (nom, mime_type, s3_bucket, s3_key, taille_bytes)  
VALUES (  
    'contrat.pdf',
    'application/pdf',
    'mon-bucket-prod',
    'documents/2025/11/contrat-123.pdf',
    524288
);

-- ✅ Les métadonnées sont répliquées
-- ✅ Le fichier est accessible depuis tous les serveurs via S3
-- ✅ Pas de limite de taille
-- ✅ Architecture cloud-native
```

### 1.4. Les Tables Système et Temporaires

#### Le Problème

**Certaines tables ne peuvent pas être répliquées** :

- ❌ **Tables système** (`pg_catalog.*`, `information_schema.*`)  
- ❌ **Tables temporaires** (`CREATE TEMP TABLE`)  
- ❌ **Tables non journalisées** (`CREATE UNLOGGED TABLE`)  
- ❌ **Tables sans clé primaire** (limitation importante)

```sql
-- Ces tables NE PEUVENT PAS être publiées :

-- Table temporaire
CREATE TEMP TABLE session_data (
    session_id VARCHAR(50),
    data JSONB
);
-- ❌ Non réplicable par nature

-- Table non journalisée (pour performance)
CREATE UNLOGGED TABLE cache_applicatif (
    key VARCHAR(100) PRIMARY KEY,
    value TEXT,
    expires_at TIMESTAMP
);
-- ❌ Non réplicable (pas de WAL)

-- Table sans clé primaire
CREATE TABLE logs_bruts (
    timestamp TIMESTAMP,
    message TEXT,
    level VARCHAR(10)
);
-- ❌ Non réplicable sans PK
```

#### Solutions et Contournements

**Pour les Tables Sans Clé Primaire** :

```sql
-- Solution 1 : Ajouter une clé primaire
ALTER TABLE logs_bruts ADD COLUMN id BIGSERIAL PRIMARY KEY;

-- Solution 2 : Utiliser REPLICA IDENTITY FULL (attention aux performances)
ALTER TABLE logs_bruts REPLICA IDENTITY FULL;

-- Avec REPLICA IDENTITY FULL :
-- - Toute la ligne est comparée pour les UPDATE/DELETE
-- - Impact performance important
-- - À utiliser seulement si ajout de PK impossible
```

**Pour les Tables Non Journalisées** :

```sql
-- Si la table DOIT être répliquée, la convertir en table normale
ALTER TABLE cache_applicatif SET LOGGED;

-- Ou recréer la table :
CREATE TABLE cache_applicatif_logged (LIKE cache_applicatif INCLUDING ALL);  
INSERT INTO cache_applicatif_logged SELECT * FROM cache_applicatif;  
DROP TABLE cache_applicatif;  
ALTER TABLE cache_applicatif_logged RENAME TO cache_applicatif;  
```

### 1.5. TRUNCATE dans la Réplication Logique

#### Comportement depuis PostgreSQL 11

**Depuis PostgreSQL 11, `TRUNCATE` EST répliqué par défaut.** La publication par défaut inclut `insert, update, delete, truncate`.

```sql
-- Vérifier les opérations publiées
SELECT pubname, puballtables, pubinsert, pubupdate, pubdelete, pubtruncate  
FROM pg_publication;  
-- pubtruncate = true par défaut depuis PG 11
```

#### Cas où TRUNCATE peut poser problème

Si vous avez explicitement créé une publication sans `truncate` :

```sql
-- Publication sans truncate (configuration manuelle)
CREATE PUBLICATION pub_sans_truncate  
FOR ALL TABLES  
WITH (publish = 'insert, update, delete');  
-- ⚠️ TRUNCATE ne sera PAS répliqué

-- Pour ajouter truncate à une publication existante :
ALTER PUBLICATION pub_sans_truncate SET (publish = 'insert, update, delete, truncate');

-- Ou modifier une publication existante
ALTER PUBLICATION pub_prod SET (publish = 'insert, update, delete, truncate');

-- Vérifier
SELECT pubname, pubtruncate FROM pg_publication;
```

## Partie 2 : Limitations de Performance et Scalabilité

### 2.1. Overhead du Décodage Logique

#### Le Problème

Le décodage logique du WAL consomme des ressources significatives sur le serveur source :

**Impact CPU** :
```
┌─────────────────────────────────────────────┐
│ Sans Réplication Logique                    │
│ CPU: ████████░░░░░░░░░░░░ (40%)             │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│ Avec Réplication Logique (3 subscriptions)  │
│ CPU: ████████████████░░░░ (70%)             │
│                ▲                            │
│                │                            │
│           +30% pour le décodage logique     │
└─────────────────────────────────────────────┘
```

**Coût par Opération** :

| Opération | Overhead Réplication Physique | Overhead Réplication Logique |
|-----------|-------------------------------|------------------------------|
| INSERT | ~1-2% | ~5-10% |
| UPDATE | ~1-2% | ~10-20% |
| DELETE | ~1-2% | ~5-15% |
| Bulk INSERT | ~1% | ~3-5% |

#### Solutions et Considérations

**1. Dimensionnement Approprié**

```sql
-- Surveiller l'utilisation CPU liée à la réplication
SELECT
    pid,
    usename,
    application_name,
    state,
    query
FROM pg_stat_activity  
WHERE application_name LIKE 'walreceiver%'  
   OR backend_type = 'walsender';

-- Métriques système (nécessite pg_stat_kcache)
SELECT
    queryid,
    query,
    user_time,  -- Temps CPU utilisateur
    system_time  -- Temps CPU système
FROM pg_stat_kcache  
ORDER BY (user_time + system_time) DESC  
LIMIT 10;  
```

**2. Limiter le Nombre de Publications**

```sql
-- ❌ Mauvais : Beaucoup de publications redondantes
CREATE PUBLICATION pub1 FOR TABLE table1;  
CREATE PUBLICATION pub2 FOR TABLE table2;  
CREATE PUBLICATION pub3 FOR TABLE table3;  
-- ... 50 publications

-- ✅ Bon : Une publication consolidée
CREATE PUBLICATION pub_all FOR TABLE table1, table2, table3, ...;

-- Ou grouper par thématique
CREATE PUBLICATION pub_metier FOR TABLE commandes, clients, produits;  
CREATE PUBLICATION pub_analytics FOR TABLE stats, logs, events;  
```

### 2.2. Consommation de WAL et Stockage

#### Le Problème

Les slots de réplication **conservent le WAL** jusqu'à ce que le subscriber l'ait consommé.

**Scénario Problématique** :

```
T+0h : Slot créé, WAL = 1 GB  
T+2h : Subscriber arrêté pour maintenance  
T+6h : WAL accumulé = 50 GB (haute activité)  
T+10h : WAL accumulé = 120 GB  
T+12h : ALERTE : Disque à 95% !  
```

#### Solutions et Prévention

**1. Monitoring des Slots de Réplication**

```sql
-- Surveiller la taille du WAL retenu par slot
SELECT
    slot_name,
    slot_type,
    database,
    active,
    restart_lsn,
    confirmed_flush_lsn,
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)
    ) AS wal_retained,
    pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) AS wal_retained_bytes
FROM pg_replication_slots  
WHERE slot_type = 'logical'  
ORDER BY wal_retained_bytes DESC;  

-- Alerter si WAL > 10 GB
CREATE OR REPLACE FUNCTION check_wal_retention()  
RETURNS TABLE(slot_name TEXT, wal_gb NUMERIC, alert TEXT) AS $$  
BEGIN  
    RETURN QUERY
    SELECT
        rs.slot_name::TEXT,
        ROUND(
            pg_wal_lsn_diff(pg_current_wal_lsn(), rs.restart_lsn)
            / 1024.0 / 1024.0 / 1024.0,
            2
        ) AS wal_gb,
        CASE
            WHEN pg_wal_lsn_diff(pg_current_wal_lsn(), rs.restart_lsn) > 10737418240
            THEN '⚠️ CRITICAL: WAL > 10 GB'
            WHEN pg_wal_lsn_diff(pg_current_wal_lsn(), rs.restart_lsn) > 5368709120
            THEN '⚠️ WARNING: WAL > 5 GB'
            ELSE '✅ OK'
        END AS alert
    FROM pg_replication_slots rs
    WHERE rs.slot_type = 'logical';
END;
$$ LANGUAGE plpgsql;

-- Exécuter régulièrement
SELECT * FROM check_wal_retention();
```

**2. Configuration de Limites**

```sql
-- Dans postgresql.conf

-- Limite de taille des slots (PostgreSQL 13+)
max_slot_wal_keep_size = '50GB'  -- Limite la rétention WAL

-- Si cette limite est atteinte, le slot devient "invalidated"
-- et la réplication doit être recréée (resync complet)

-- Paramètres WAL recommandés
wal_keep_size = '10GB'  -- WAL minimum à garder  
max_wal_size = '5GB'    -- Déclenche checkpoint  

-- Vérifier les slots invalidés
SELECT slot_name, slot_type, active, invalidation_reason  
FROM pg_replication_slots  
WHERE slot_type = 'logical' AND invalidation_reason IS NOT NULL;  
```

**3. Procédure d'Urgence (Disque Saturé)**

```sql
-- ATTENTION : À utiliser seulement en cas d'urgence

-- 1. Identifier le slot problématique
SELECT slot_name, pg_size_pretty(
    pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)
) AS wal_retained
FROM pg_replication_slots  
ORDER BY pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) DESC;  

-- 2. Option A : Supprimer le slot (la réplication devra être recréée)
SELECT pg_drop_replication_slot('slot_problematique');

-- 3. Option B : Désactiver temporairement la subscription (côté subscriber)
-- puis supprimer le slot
ALTER SUBSCRIPTION sub_problematique DISABLE;  
ALTER SUBSCRIPTION sub_problematique SET (slot_name = NONE);  
-- Sur le publisher :
SELECT pg_drop_replication_slot('slot_problematique');
```

### 2.3. Latence de Réplication

#### Le Problème

La réplication logique introduit une **latence** (lag) entre source et destination.

**Facteurs influençant la latence** :
- Volume de transactions
- Bande passante réseau
- Charge du serveur source (décodage)
- Charge du serveur destination (application)
- Distance géographique

**Mesure de la Latence** :

```sql
-- Sur le serveur destination
SELECT
    subname,
    received_lsn,
    latest_end_lsn,
    -- Différence en bytes
    pg_wal_lsn_diff(latest_end_lsn, received_lsn) AS lag_bytes,
    pg_size_pretty(pg_wal_lsn_diff(latest_end_lsn, received_lsn)) AS lag_pretty,
    -- Différence temporelle
    EXTRACT(EPOCH FROM (now() - last_msg_receipt_time)) AS lag_seconds
FROM pg_stat_subscription;

-- Résultat type :
--  subname  | lag_pretty | lag_seconds
-- ----------+------------+-------------
--  sub_prod | 2567 kB    | 3.45
```

**Latences Typiques** :

| Configuration | Latence Attendue | Usage |
|---------------|------------------|-------|
| LAN, charges faibles | < 1 seconde | ✅ Acceptable pour la plupart des cas |
| LAN, charges élevées | 1-5 secondes | ⚠️ À surveiller |
| WAN, inter-continents | 5-30 secondes | ⚠️ Selon le cas d'usage |
| Problème de réseau | > 1 minute | ❌ Investigation requise |

#### Améliorer les Performances

**1. Optimisation Réseau**

```sql
-- Utiliser la compression (PostgreSQL 14+)
ALTER SUBSCRIPTION sub_prod SET (streaming = 'parallel');

-- Désactiver le commit synchrone côté subscriber
ALTER SUBSCRIPTION sub_prod SET (synchronous_commit = 'off');
```

**2. Parallélisation (PostgreSQL 16+)**

```sql
-- Dans postgresql.conf (subscriber)
max_parallel_apply_workers_per_subscription = 4

-- Permet d'appliquer les changements en parallèle
-- Amélioration possible : 2-4x plus rapide
```

**3. Tuning des Paramètres**

```sql
-- Côté publisher (dans postgresql.conf)
wal_sender_timeout = 60s  -- Augmenter si réseau instable  
max_wal_senders = 10      -- Suffisant pour plusieurs subscribers  

-- Côté subscriber
wal_receiver_timeout = 60s  
max_logical_replication_workers = 8  
```

## Partie 3 : Gestion des Conflits

### 3.1. Types de Conflits

#### Conflit d'INSERT (Violation de Clé Primaire)

```sql
-- SCÉNARIO :
-- Serveur SOURCE : INSERT INTO clients VALUES (1, 'Alice');
-- Serveur DESTINATION : INSERT INTO clients VALUES (1, 'Bob');
-- → CONFLIT : La clé primaire 1 existe déjà

-- Erreur dans les logs :
-- ERROR: duplicate key value violates unique constraint "clients_pkey"
-- DETAIL: Key (id)=(1) already exists.
```

**Résultat** : La réplication s'**arrête** jusqu'à résolution manuelle.

#### Conflit d'UPDATE (Ligne Modifiée ou Absente)

```sql
-- SCÉNARIO 1 : Ligne modifiée différemment
-- SOURCE : UPDATE clients SET solde = 100 WHERE id = 1;
-- DESTINATION a déjà fait : UPDATE clients SET solde = 200 WHERE id = 1;
-- → Quelle valeur doit prévaloir ?

-- SCÉNARIO 2 : Ligne supprimée
-- SOURCE : UPDATE clients SET email = 'new@example.com' WHERE id = 1;
-- DESTINATION a déjà fait : DELETE FROM clients WHERE id = 1;
-- → Impossible de mettre à jour une ligne inexistante
```

#### Conflit de DELETE

```sql
-- SCÉNARIO :
-- SOURCE : DELETE FROM clients WHERE id = 1;
-- DESTINATION a déjà fait : DELETE FROM clients WHERE id = 1;
-- → DELETE d'une ligne déjà supprimée (généralement bénin)
```

### 3.2. Stratégies de Résolution de Conflits

#### Stratégie Par Défaut : "Last Write Wins"

Par défaut, PostgreSQL applique le **dernier changement reçu** :

```sql
-- Pas de gestion automatique sophistiquée
-- Si un conflit survient, la réplication s'arrête avec une erreur

-- Vérifier l'état
SELECT
    subname,
    pid,
    relid::regclass AS table_name,
    received_lsn,
    last_msg_receipt_time
FROM pg_stat_subscription;

-- En cas d'erreur, le pid sera NULL
```

#### Résolution Manuelle des Conflits

**Étape 1 : Identifier le Conflit**

```sql
-- Consulter les logs PostgreSQL
-- Chercher les erreurs de type :
-- ERROR: duplicate key value violates unique constraint
-- ERROR: could not find row to be updated
-- ERROR: could not find row to be deleted

-- Identifier la transaction problématique
SELECT * FROM pg_stat_subscription WHERE pid IS NULL;
```

**Étape 2 : Résoudre le Conflit**

```sql
-- Option A : Supprimer la ligne en conflit sur le subscriber
DELETE FROM clients WHERE id = 1;

-- Option B : Modifier la donnée conflictuelle
UPDATE clients SET id = 999999 WHERE id = 1;  -- Temporairement

-- Option C : Autoriser le conflit pour cette transaction
-- (Avancer manuellement le slot de réplication)
-- ⚠️ DANGEREUX : Peut créer des incohérences

-- Redémarrer la subscription
ALTER SUBSCRIPTION sub_prod ENABLE;
```

**Étape 3 : Prévention Future**

```sql
-- Mettre en place des contraintes pour éviter les conflits

-- 1. Utiliser UUID au lieu de SERIAL
ALTER TABLE clients ADD COLUMN uuid UUID DEFAULT gen_random_uuid();  
ALTER TABLE clients DROP CONSTRAINT clients_pkey;  
ALTER TABLE clients ADD PRIMARY KEY (uuid);  

-- 2. Désigner clairement un "master" pour chaque table
-- Ne permettre les écritures QUE sur le master

-- 3. Utiliser des triggers pour détecter les écritures non autorisées
CREATE OR REPLACE FUNCTION prevent_writes_on_replica()  
RETURNS TRIGGER AS $$  
BEGIN  
    RAISE EXCEPTION 'Écritures non autorisées sur ce serveur';
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER no_writes_clients  
BEFORE INSERT OR UPDATE OR DELETE ON clients  
FOR EACH ROW  
EXECUTE FUNCTION prevent_writes_on_replica();  

-- Désactiver sur le master, activer sur les replicas
```

### 3.3. Architecture pour Éviter les Conflits

#### Architecture "Master-Replica" (Recommandée)

```
┌──────────────────────────────────────────┐
│         SERVEUR MASTER (Source)          │
│  ✅ Écritures autorisées                 │
│  ✅ Lectures autorisées                  │
│                                          │
│  Application → Toutes opérations         │
└──────────────────────────────────────────┘
                    │
                    │ Réplication unidirectionnelle
                    ▼
┌──────────────────────────────────────────┐
│        SERVEUR REPLICA (Destination)     │
│  ❌ Écritures INTERDITES                 │
│  ✅ Lectures autorisées                  │
│                                          │
│  Analytics, Reporting → SELECT only      │
└──────────────────────────────────────────┘

✅ Aucun conflit possible
```

#### Architecture "Multi-Master" (Complexe, Déconseillée)

```
┌─────────────┐ ←─── Réplication ───→ ┌─────────────┐
│  SERVEUR A  │                        │  SERVEUR B  │
│  Écritures  │ ←─── bidirectionnelle  │  Écritures  │
│  autorisées │                        │  autorisées │
└─────────────┘                        └─────────────┘

⚠️ Risques de conflits élevés
⚠️ Complexité de gestion
⚠️ Nécessite une logique métier pour éviter les conflits
```

**Si Multi-Master Nécessaire** :

```sql
-- Utiliser des stratégies de partitionnement des données

-- Exemple : Chaque serveur gère une région
-- Serveur A : region = 'EU'
-- Serveur B : region = 'US'

CREATE PUBLICATION pub_to_b  
FOR TABLE commandes WHERE (region = 'US'),  -- Envoie US vers B  
         produits;  -- Produits partagés

-- Sur B :
CREATE PUBLICATION pub_to_a  
FOR TABLE commandes WHERE (region = 'EU'),  -- Envoie EU vers A  
         produits;  -- Produits partagés

-- Règle métier : Chaque serveur écrit SEULEMENT dans sa région
-- → Pas de conflit possible
```

## Partie 4 : Considérations Opérationnelles

### 4.1. Impact sur les Sauvegardes

#### Le Problème

Les slots de réplication peuvent **interférer avec les sauvegardes** :

```sql
-- Scénario problématique :
-- 1. Une subscription est créée
-- 2. Le subscriber est arrêté
-- 3. Le slot conserve le WAL
-- 4. Tentative de VACUUM sur de grandes tables → Bloqué
-- 5. pg_dump prend beaucoup plus de temps (données + WAL)
```

#### Solutions

**1. Surveillance des Slots Avant Sauvegarde**

```bash
#!/bin/bash
# pre_backup_check.sh

# Vérifier que tous les slots sont actifs et à jour
psql -U postgres -d production -t -c "  
SELECT  
    slot_name,
    active,
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)
    ) AS lag
FROM pg_replication_slots  
WHERE slot_type = 'logical'  
  AND (NOT active OR pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) > 1073741824);
" | while read slot active lag; do
    if [ ! -z "$slot" ]; then
        echo "⚠️ WARNING: Slot $slot is problematic (active=$active, lag=$lag)"
        # Envoyer une alerte
    fi
done
```

**2. Stratégie de Sauvegarde Adaptée**

```sql
-- Option 1 : Sauvegardes physiques (pg_basebackup)
-- Non affectées par les slots logiques
pg_basebackup -h localhost -U replication -D /backup/physical -Fp -Xs -P

-- Option 2 : Exclure les slots logiques des sauvegardes
-- (ils seront recréés lors de la restauration)
pg_dump --exclude-table=pg_replication_slots ...

-- Option 3 : Pause temporaire de la réplication pendant la sauvegarde
-- (si possible)
```

### 4.2. Maintenance (VACUUM, ANALYZE)

#### Le Problème

Les **transactions longues** liées à la réplication peuvent bloquer VACUUM.

```sql
-- VACUUM ne peut pas nettoyer les lignes mortes
-- tant qu'elles pourraient être lues par une transaction active

-- Vérifier les transactions longues
SELECT
    pid,
    usename,
    application_name,
    state,
    now() - xact_start AS duration,
    query
FROM pg_stat_activity  
WHERE state = 'active'  
  AND now() - xact_start > interval '1 hour'
ORDER BY xact_start;
```

#### Solutions

**1. Configurer un Timeout**

```sql
-- Dans postgresql.conf
idle_in_transaction_session_timeout = '1h'  -- Tuer les transactions idle  
statement_timeout = '2h'  -- Timeout global des requêtes  

-- Pour la réplication spécifiquement
wal_sender_timeout = 60s  
wal_receiver_timeout = 60s  
```

**2. VACUUM Agressif en Dehors des Heures de Pointe**

```sql
-- Script de maintenance nocturne
-- maintenance.sh (via cron : 0 2 * * *)

#!/bin/bash
psql -U postgres -d production <<EOF
-- Tables critiques avec beaucoup de modifications
VACUUM (VERBOSE, ANALYZE) commandes;  
VACUUM (VERBOSE, ANALYZE) clients;  
VACUUM (VERBOSE, ANALYZE) logs;  

-- Analyser les statistiques
ANALYZE;  
EOF  
```

### 4.3. Versions et Compatibilité

#### Compatibilité entre Versions PostgreSQL

La réplication logique permet de **répliquer entre versions différentes** (avantage majeur) :

| Source | Destination | Compatible ? | Notes |
|--------|-------------|--------------|-------|
| PG 13 | PG 13 | ✅ | Idéal |
| PG 13 | PG 14 | ✅ | Recommandé pour upgrade |
| PG 13 | PG 18 | ✅ | Possible (tester) |
| PG 18 | PG 13 | ⚠️ | Risqué (fonctionnalités manquantes) |
| PG 14 | PG 13 | ❌ | Déconseillé |

#### Points d'Attention

```sql
-- 1. Vérifier les types de données
-- PostgreSQL 18 a de nouveaux types (UUIDv7, etc.)

-- 2. Vérifier les fonctions utilisées dans les contraintes
-- Une fonction disponible en PG 18 peut ne pas exister en PG 13

-- 3. Tester la compatibilité en staging
-- Créer un environnement de test avec les deux versions

-- 4. Consulter les release notes
-- Chaque version documente les breaking changes
```

### 4.4. Sécurité et Contrôle d'Accès

#### Permissions Requises

**Sur le Serveur Source (Publisher)** :

```sql
-- L'utilisateur de réplication doit avoir :
-- 1. Droit de REPLICATION
CREATE ROLE replicateur WITH LOGIN REPLICATION PASSWORD 'secure_password';

-- 2. SELECT sur toutes les tables publiées
GRANT SELECT ON ALL TABLES IN SCHEMA public TO replicateur;  
GRANT USAGE ON SCHEMA public TO replicateur;  

-- 3. Pour les futures tables
ALTER DEFAULT PRIVILEGES IN SCHEMA public  
GRANT SELECT ON TABLES TO replicateur;  
```

**Sur le Serveur Destination (Subscriber)** :

```sql
-- L'utilisateur doit pouvoir :
-- 1. Créer la subscription
-- Nécessite superuser OU rôle avec CREATE sur la database

-- 2. Écrire dans les tables
GRANT INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO replicateur;

-- 3. Gérer les séquences (si applicable)
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO replicateur;
```

#### Sécurité des Connexions

```sql
-- pg_hba.conf sur le source
-- Autoriser seulement les IPs des subscribers
hostssl    production    replicateur    10.0.1.0/24    scram-sha-256

-- Utiliser SSL/TLS obligatoirement
-- Dans la connection string de la subscription
CREATE SUBSCRIPTION sub_secure  
CONNECTION 'host=prod.example.com port=5432 dbname=production user=replicateur password=xxx sslmode=require'  
PUBLICATION pub_prod;  

-- PostgreSQL 18 : Authentification OAuth 2.0
-- Nouvelle méthode d'authentification plus sécurisée
-- Voir documentation PostgreSQL 18 pour configuration
```

## Partie 5 : Comparaison Réplication Logique vs Physique

### Tableau Comparatif Détaillé

| Critère | Réplication Logique | Réplication Physique (Streaming) |
|---------|---------------------|----------------------------------|
| **Granularité** | ✅ Table par table, colonnes filtrables | ❌ Instance complète (tout ou rien) |
| **Versions PostgreSQL** | ✅ Différentes versions (ex: 13→18) | ❌ Même version majeure requise |
| **Schémas différents** | ✅ Possible (tables peuvent différer) | ❌ Schéma identique obligatoire |
| **Écriture sur replica** | ✅ Possible (tables non répliquées) | ❌ Lecture seule stricte |
| **DDL automatique** | ❌ Manuel | ✅ Automatique |
| **Séquences** | ❌ Non synchronisées | ✅ Automatique |
| **Large Objects** | ❌ Non supportés | ✅ Supportés |
| **Overhead CPU** | ⚠️ Moyen-élevé (10-20%) | ✅ Faible (1-2%) |
| **Overhead I/O** | ⚠️ Moyen | ✅ Faible |
| **Latence typique** | ⚠️ Quelques secondes | ✅ < 1 seconde |
| **Complexité gestion** | ⚠️ Moyenne | ✅ Simple |
| **Cas d'usage** | Migration, Analytics, Multi-sources | Haute disponibilité, Failover |
| **Gestion conflits** | ❌ Complexe (si multi-master) | ✅ N/A (read-only replica) |
| **PostgreSQL 18 perf** | ⚠️ Optimisations continues | ✅ Très performant |

### Quand Utiliser Chaque Type ?

#### ✅ Utiliser la Réplication Logique Quand :

- Migration entre versions PostgreSQL
- Réplication sélective (certaines tables seulement)
- Filtrage de données (WHERE clause, colonnes)
- Agrégation multi-sources
- Nécessité d'écrire sur le replica (tables non répliquées)
- Architecture microservices (replication partielle)
- Conformité RGPD (filtrage géographique)

#### ✅ Utiliser la Réplication Physique Quand :

- Haute disponibilité (HA) critique
- Failover automatique requis
- Latence minimale exigée (< 100ms)
- Réplication complète nécessaire
- Simplicité de gestion prioritaire
- Charge élevée (overhead minimal requis)
- Disaster Recovery (DR)

## Partie 7 : Dépannage (Troubleshooting)

### Problèmes Courants et Solutions

#### Problème 1 : "could not create replication slot"

**Erreur** :
```
ERROR: could not create replication slot "sub_prod_slot":  
ERROR: all replication slots are in use  
```

**Cause** : `max_replication_slots` trop faible.

**Solution** :
```sql
-- Vérifier l'utilisation actuelle
SELECT count(*) FROM pg_replication_slots;

-- Dans postgresql.conf
max_replication_slots = 10  -- Augmenter

-- Redémarrer PostgreSQL
sudo systemctl restart postgresql
```

#### Problème 2 : Lag de Réplication Élevé

**Symptôme** :
```sql
SELECT
    pg_size_pretty(pg_wal_lsn_diff(latest_end_lsn, received_lsn)) AS lag
FROM pg_stat_subscription;
-- Résultat : lag = 500 MB (problématique)
```

**Causes Possibles et Solutions** :

```sql
-- 1. Charge élevée sur le subscriber
-- Solution : Augmenter max_logical_replication_workers
ALTER SYSTEM SET max_logical_replication_workers = 8;  
SELECT pg_reload_conf();  

-- 2. Réseau saturé
-- Solution : Vérifier avec iperf, netstat
-- Activer la compression (PG 14+)

-- 3. Tables sans index
-- Solution : Ajouter des index sur les colonnes fréquemment filtrées
CREATE INDEX idx_commandes_client ON commandes(client_id);

-- 4. Requêtes lentes sur le subscriber
-- Solution : Analyser avec pg_stat_statements
SELECT query, mean_exec_time  
FROM pg_stat_statements  
ORDER BY mean_exec_time DESC LIMIT 10;  
```

#### Problème 3 : Slot Invalide (Trop de WAL)

**Erreur** :
```
WARNING: replication slot "sub_prod_slot" has been invalidated  
DETAIL: This slot has been invalidated because it exceeded max_slot_wal_keep_size.  
```

**Solution** :
```sql
-- La subscription doit être recréée complètement

-- 1. Sur le subscriber : Supprimer la subscription
DROP SUBSCRIPTION sub_prod;

-- 2. Sur le publisher : Le slot est déjà supprimé
SELECT * FROM pg_replication_slots;  -- Vérifier

-- 3. Recréer la subscription (copie complète)
CREATE SUBSCRIPTION sub_prod  
CONNECTION '...'  
PUBLICATION pub_prod  
WITH (copy_data = true);  -- Re-copie toutes les données  
```

#### Problème 4 : Conflits de Données

**Erreur** :
```
ERROR: duplicate key value violates unique constraint "clients_pkey"
```

**Solution** :
```sql
-- 1. Identifier la ligne en conflit (dans les logs)
-- ERROR: Key (id)=(12345) already exists.

-- 2. Résoudre manuellement
DELETE FROM clients WHERE id = 12345;  -- Ou UPDATE

-- 3. Réactiver la subscription
ALTER SUBSCRIPTION sub_prod ENABLE;

-- 4. Prévenir : Utiliser UUID
ALTER TABLE clients ADD COLUMN uuid UUID DEFAULT gen_random_uuid();
-- Puis migrer vers UUID comme PK
```

## Conclusion et Recommandations

### Résumé des Limitations Principales

La réplication logique PostgreSQL est puissante mais présente des contraintes qu'il est essentiel de comprendre :

#### ❌ Limitations Techniques
- DDL non répliqué automatiquement
- Séquences non synchronisées
- Large Objects non supportés
- Tables sans PK non réplicables
- TRUNCATE répliqué par défaut (depuis PG 11), mais vérifiez la configuration de votre publication

#### ⚠️ Limitations de Performance
- Overhead CPU de 10-20%
- Consommation WAL accrue
- Latence de quelques secondes
- Complexité de gestion

#### 🔧 Limitations Opérationnelles
- Gestion manuelle des conflits
- Surveillance active requise
- Formation des équipes nécessaire
- Complexité si multi-master

### Quand la Réplication Logique Est le Bon Choix

✅ **Cas d'usage idéaux** :
- Migration entre versions
- Réplication sélective
- Analytics/Reporting
- Multi-sources
- Conformité (RGPD)

✅ **Si vous pouvez accepter** :
- Quelques secondes de latence
- Gestion manuelle des DDL
- Overhead modéré
- Architecture master-replica

### Quand Envisager des Alternatives

❌ **Réplication physique plutôt si** :
- HA critique (failover < 1s)
- Réplication complète requise
- Simplicité prioritaire
- Latence minimale exigée

❌ **Solutions externes plutôt si** :
- Multi-master complexe requis
- Résolution automatique de conflits
- Synchronisation bi-directionnelle
- (Considérer: BDR, Patroni avec Logical, ou solutions comme CockroachDB)

### Recommandations Finales

1. **Toujours tester en staging** avant production  
2. **Documenter** la procédure DDL  
3. **Monitorer** activement (lag, slots, WAL)  
4. **Former** les équipes sur les limitations  
5. **Prévoir** un plan de rollback  
6. **Utiliser UUID** pour éviter conflits de PK  
7. **Automatiser** la surveillance avec alertes  
8. **Planifier** la maintenance régulière
