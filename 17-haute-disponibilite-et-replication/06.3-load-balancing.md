# 17.6.3. Load Balancing avec HAProxy ou PgPool-II

## Introduction

Vous avez maintenant une infrastructure PostgreSQL hautement disponible avec un serveur Primary et plusieurs Standby en réplication. Vos données sont protégées, mais un nouveau défi apparaît : **comment distribuer intelligemment les requêtes entre vos différents serveurs ?**

Le **load balancing** (répartition de charge) est la solution. C'est un composant qui se place entre vos applications et vos serveurs PostgreSQL pour :
- Diriger les écritures vers le Primary
- Distribuer les lectures entre les Standby
- Détecter les pannes et rediriger le trafic automatiquement
- Améliorer les performances globales de votre infrastructure

Dans ce chapitre, nous allons explorer les deux solutions les plus populaires pour PostgreSQL : **HAProxy** et **PgPool-II**.

---

## Comprendre le Load Balancing : Analogie Simple

### Le problème sans load balancing

Imaginez un restaurant avec :
- **Une cuisine principale** (le Primary) qui prend les commandes ET prépare les plats  
- **Deux cuisines auxiliaires** (les Standby) qui peuvent lire les recettes mais ne peuvent que préparer les plats déjà commandés

Sans load balancer, chaque client doit savoir :
- Quelle cuisine peut prendre sa commande (écriture)
- Quelle cuisine consulter pour voir le menu (lecture)
- Si une cuisine ferme, il doit manuellement chercher une autre cuisine

**Chaos total !** 😵

### Avec un load balancer

Maintenant, imaginez un **maître d'hôtel** (le load balancer) :
- Les clients parlent uniquement au maître d'hôtel
- Le maître d'hôtel connaît l'état de chaque cuisine
- Il envoie les commandes à la cuisine principale
- Il dirige les consultations de menu vers les cuisines les moins occupées
- Si une cuisine ferme, il redirige automatiquement vers une autre

**Organisation parfaite !** ✅

## Pourquoi le Load Balancing pour PostgreSQL ?

### 1. Simplification applicative

**Sans load balancer :**
```
Application
  ├── Connexion Primary (écritures)
  ├── Connexion Standby1 (lectures)
  ├── Connexion Standby2 (lectures)
  └── Logique de failover dans chaque application
```

Chaque application doit gérer :
- Les différentes adresses des serveurs
- La détection de panne
- Le basculement automatique
- L'équilibrage des lectures

**Avec load balancer :**
```
Application
  └── Connexion unique au load balancer
        ├── Port 5432 (écritures) → Primary
        └── Port 5433 (lectures) → Standby1, Standby2 (rotation)
```

L'application voit **un seul point d'accès**. Toute la complexité est gérée par le load balancer.

### 2. Distribution optimale des lectures

Avec plusieurs Standby, le load balancer peut :
- **Round-robin** : Distribuer les lectures équitablement  
- **Least connections** : Envoyer vers le serveur le moins chargé  
- **Weighted** : Favoriser les serveurs plus puissants

### 3. Haute disponibilité automatique

Le load balancer surveille en permanence la santé des serveurs :
- Détection de panne (health checks)
- Retrait automatique des serveurs défaillants
- Réintégration automatique après récupération
- Pas d'intervention manuelle nécessaire

### 4. Amélioration des performances

En distribuant les lectures sur plusieurs serveurs :
- Réduction de la charge CPU sur le Primary
- Meilleure utilisation des ressources
- Temps de réponse améliorés
- Capacité de traiter plus de requêtes simultanées

## HAProxy : Le Couteau Suisse du Load Balancing

### Présentation

**HAProxy** (High Availability Proxy) est un load balancer généraliste, extrêmement populaire et mature :

**Caractéristiques :**
- ✅ Open-source (GPLv2)  
- ✅ Très performant (dizaines de milliers de connexions)  
- ✅ Léger et rapide (écrit en C)  
- ✅ Protocole agnostique (HTTP, TCP, PostgreSQL, etc.)  
- ✅ Monitoring intégré (interface web)  
- ✅ Large communauté et documentation

**Philosophie :** HAProxy est un **proxy TCP/HTTP générique**. Il ne comprend pas le protocole PostgreSQL en profondeur, mais c'est aussi sa force : simplicité et performance.

### Architecture avec HAProxy

```
                     HAProxy
                        |
        +---------------+---------------+
        |                               |
    Port 5432                       Port 5433
  (Write - Primary)              (Read - Standby)
        |                               |
        |                      +--------+--------+
        |                      |                 |
    PostgreSQL             PostgreSQL       PostgreSQL
     Primary               Standby1          Standby2
   (écritures)            (lectures)        (lectures)
```

**Principe :**
- HAProxy écoute sur deux ports (ou deux IP)
- **Port 5432** : Toutes les connexions vont au Primary  
- **Port 5433** : Les connexions sont distribuées entre les Standby

### Configuration de Base

#### Étape 1 : Installation

```bash
# Ubuntu/Debian
sudo apt update  
sudo apt install haproxy  

# CentOS/RHEL
sudo yum install haproxy

# Vérifier la version
haproxy -v
```

#### Étape 2 : Configuration (`/etc/haproxy/haproxy.cfg`)

```conf
#---------------------------------------------------------------------
# Configuration globale
#---------------------------------------------------------------------
global
    log /dev/log local0
    maxconn 10000
    user haproxy
    group haproxy
    daemon

#---------------------------------------------------------------------
# Configuration par défaut
#---------------------------------------------------------------------
defaults
    mode tcp
    log global
    option tcplog
    option dontlognull
    timeout connect 5000ms
    timeout client 50000ms
    timeout server 50000ms

#---------------------------------------------------------------------
# Interface de monitoring (optionnel)
#---------------------------------------------------------------------
listen stats
    mode http
    bind *:8404
    stats enable
    stats uri /stats
    stats refresh 30s
    stats admin if TRUE

#---------------------------------------------------------------------
# Frontend pour les ÉCRITURES (Primary)
#---------------------------------------------------------------------
frontend postgres_write
    bind *:5432
    default_backend postgres_primary

backend postgres_primary
    option pgsql-check user haproxy_check
    server primary1 192.168.1.10:5432 check port 5432 inter 2000 rise 2 fall 3

#---------------------------------------------------------------------
# Frontend pour les LECTURES (Standby)
#---------------------------------------------------------------------
frontend postgres_read
    bind *:5433
    default_backend postgres_replicas

backend postgres_replicas
    balance roundrobin
    option pgsql-check user haproxy_check
    server replica1 192.168.1.11:5432 check port 5432 inter 2000 rise 2 fall 3
    server replica2 192.168.1.12:5432 check port 5432 inter 2000 rise 2 fall 3
    server replica3 192.168.1.13:5432 check port 5432 inter 2000 rise 2 fall 3 backup
```

**Explications des paramètres :**

| Paramètre | Signification |
|-----------|---------------|
| `mode tcp` | Mode TCP transparent (pas de parsing HTTP) |
| `bind *:5432` | Écouter sur toutes les interfaces, port 5432 |
| `balance roundrobin` | Distribution équitable des connexions |
| `option pgsql-check` | Health check spécifique PostgreSQL |
| `check port 5432` | Port à vérifier pour le health check |
| `inter 2000` | Intervalle entre les checks (2 secondes) |
| `rise 2` | Nombre de checks réussis avant marquage "UP" |
| `fall 3` | Nombre de checks échoués avant marquage "DOWN" |
| `backup` | Serveur de secours (utilisé si les autres tombent) |

#### Étape 3 : Créer l'utilisateur de health check

Sur **chaque serveur PostgreSQL** :


### Utilisation depuis l'application

```python
# Python avec psycopg2
import psycopg2

# Connexion pour ÉCRITURES (port 5432 = Primary)
conn_write = psycopg2.connect(
    host="haproxy-ip",
    port=5432,
    database="mydb",
    user="app_user",
    password="password"
)

# Connexion pour LECTURES (port 5433 = Standby)
conn_read = psycopg2.connect(
    host="haproxy-ip",
    port=5433,
    database="mydb",
    user="app_user",
    password="password"
)

# Écriture
cursor_write = conn_write.cursor()  
cursor_write.execute("INSERT INTO users (name) VALUES ('Alice')")  
conn_write.commit()  

# Lecture
cursor_read = conn_read.cursor()  
cursor_read.execute("SELECT * FROM users")  
users = cursor_read.fetchall()  
```

**Important :** L'application doit gérer **deux connexions** (une pour écrire, une pour lire).

### Configuration Avancée : Détection Automatique du Primary

Pour détecter automatiquement quel serveur est le Primary (en cas de failover) :

```conf
backend postgres_primary
    option pgsql-check user haproxy_check

    # Script de vérification personnalisé
    option httpchk
    http-check send meth GET uri /primary

    server pg1 192.168.1.10:5432 check port 8008
    server pg2 192.168.1.11:5432 check port 8008 backup
    server pg3 192.168.1.12:5432 check port 8008 backup
```

Sur chaque serveur PostgreSQL, créez un endpoint HTTP (port 8008) qui retourne :
- **HTTP 200** si le serveur est Primary  
- **HTTP 503** si le serveur est Standby

Exemple avec Python :

```python
# health_check.py
from http.server import BaseHTTPRequestHandler, HTTPServer  
import psycopg2  

class HealthCheck(BaseHTTPRequestHandler):
    def do_GET(self):
        if self.path == '/primary':
            # Vérifier si c'est le Primary
            conn = psycopg2.connect("dbname=postgres user=postgres")
            cur = conn.cursor()
            cur.execute("SELECT pg_is_in_recovery()")
            is_standby = cur.fetchone()[0]

            if not is_standby:  # C'est le Primary
                self.send_response(200)
            else:  # C'est un Standby
                self.send_response(503)

            self.end_headers()

HTTPServer(('', 8008), HealthCheck).serve_forever()
```

### Avantages de HAProxy

| Avantage | Description |
|----------|-------------|
| 🚀 **Performance** | Extrêmement rapide, optimisé pour le TCP |
| 💪 **Fiabilité** | Utilisé en production par des millions de sites |
| 📊 **Monitoring** | Interface web intégrée très complète |
| 🔧 **Flexibilité** | Configuration simple et puissante |
| 🌐 **Polyvalence** | Fonctionne pour tout protocole TCP (pas que PostgreSQL) |
| 💰 **Gratuit** | Open-source sans limitations |
| 📚 **Communauté** | Documentation extensive et large adoption |

### Inconvénients de HAProxy

| Inconvénient | Description |
|--------------|-------------|
| ⚙️ **Pas de connection pooling** | HAProxy ne fait que du routage, pas de gestion de pool |
| 🔌 **Deux connexions nécessaires** | L'app doit gérer read vs write manuellement |
| 🧩 **Pas de split automatique** | HAProxy ne parse pas le SQL (SELECT vs INSERT) |
| 🔍 **Pas de query routing** | Pas d'analyse des requêtes pour routage intelligent |
| ⚠️ **Point de défaillance** | Si HAProxy tombe, tout est inaccessible (sauf HA HAProxy) |

## PgPool-II : Le Middleware PostgreSQL Spécialisé

### Présentation

**PgPool-II** est un middleware **spécialement conçu pour PostgreSQL** :

**Caractéristiques :**
- ✅ Connection pooling intégré  
- ✅ Query routing automatique (SELECT → Standby, INSERT/UPDATE → Primary)  
- ✅ Load balancing intelligent  
- ✅ Failover automatique  
- ✅ Cache de requêtes  
- ✅ Réécriture de requêtes (query rewrite)  
- ✅ Parallel query execution

**Philosophie :** PgPool-II est un **middleware PostgreSQL aware**. Il comprend le protocole et peut analyser les requêtes SQL pour des fonctionnalités avancées.

### Architecture avec PgPool-II

```
    Applications
         |
    PgPool-II (port 9999)
         |
    +----+----+----+
    |    |    |    |
   PG1  PG2  PG3  PG4
(Primary)(Standby)(Standby)(Standby)
```

**Principe :**
- Les applications se connectent à PgPool-II (un seul endpoint)
- PgPool-II analyse **chaque requête SQL**
- Il route automatiquement :
  - **SELECT** → Standby (load balanced)  
  - **INSERT/UPDATE/DELETE** → Primary  
  - **Transactions** → Primary (cohérence)
- Il gère un **pool de connexions** vers chaque serveur

### Configuration de Base

#### Étape 1 : Installation

```bash
# Ubuntu/Debian
sudo apt install pgpool2

# CentOS/RHEL
sudo yum install pgpool-II

# Vérifier la version
pgpool -v
```

#### Étape 2 : Configuration (`/etc/pgpool2/pgpool.conf`)

Fichier de configuration principal (version simplifiée) :

```conf
#---------------------------------------------------------------------
# Configuration réseau
#---------------------------------------------------------------------
listen_addresses = '*'  
port = 9999  
socket_dir = '/var/run/postgresql'  

#---------------------------------------------------------------------
# Connection pooling
#---------------------------------------------------------------------
num_init_children = 32                    # Nombre de processus enfants  
max_pool = 4                              # Connexions par enfant vers chaque backend  
child_life_time = 300                     # Durée de vie d'un enfant (secondes)  
child_max_connections = 0                 # 0 = illimité  
connection_life_time = 0                  # 0 = connexions persistantes  
client_idle_limit = 0                     # 0 = pas de timeout  

#---------------------------------------------------------------------
# Load balancing
#---------------------------------------------------------------------
load_balance_mode = on                    # Activer le load balancing  
ignore_leading_white_space = on           # Ignorer espaces en début de requête  
white_function_list = ''                  # Fonctions considérées comme read-only  
black_function_list = 'nextval,setval'    # Fonctions considérées comme write  

#---------------------------------------------------------------------
# Configuration des backends (serveurs PostgreSQL)
#---------------------------------------------------------------------
# Backend 0 (Primary)
backend_hostname0 = '192.168.1.10'  
backend_port0 = 5432  
backend_weight0 = 0                       # Poids 0 = pas de lectures  
backend_data_directory0 = '/var/lib/postgresql/14/main'  
backend_flag0 = 'ALWAYS_PRIMARY'  

# Backend 1 (Standby)
backend_hostname1 = '192.168.1.11'  
backend_port1 = 5432  
backend_weight1 = 1                       # Poids pour load balancing  
backend_data_directory1 = '/var/lib/postgresql/14/main'  
backend_flag1 = 'DISALLOW_TO_FAILOVER'  

# Backend 2 (Standby)
backend_hostname2 = '192.168.1.12'  
backend_port2 = 5432  
backend_weight2 = 1  
backend_data_directory2 = '/var/lib/postgresql/14/main'  
backend_flag2 = 'DISALLOW_TO_FAILOVER'  

#---------------------------------------------------------------------
# Health check
#---------------------------------------------------------------------
health_check_period = 10                  # Intervalle (secondes)  
health_check_timeout = 20                 # Timeout  
health_check_user = 'pgpool_check'        # Utilisateur pour health check  
health_check_password = ''                # Mot de passe (ou utiliser pool_passwd)  
health_check_database = 'postgres'  
health_check_max_retries = 3  
health_check_retry_delay = 1  

#---------------------------------------------------------------------
# Failover
#---------------------------------------------------------------------
failover_on_backend_error = off           # off pour éviter failover auto sur erreur simple  
failover_command = ''                     # Script personnalisé de failover  
failback_command = ''                     # Script de réintégration  

#---------------------------------------------------------------------
# Authentification
#---------------------------------------------------------------------
enable_pool_hba = on  
pool_passwd = 'pool_passwd'               # Fichier de mots de passe  

#---------------------------------------------------------------------
# Logging
#---------------------------------------------------------------------
log_destination = 'stderr'  
log_line_prefix = '%t: pid %p: '  
log_connections = on  
log_hostname = on  
log_statement = off                       # off pour éviter de logger toutes les requêtes  
log_per_node_statement = off  
log_client_messages = off  
```

**Explications des paramètres clés :**

| Paramètre | Signification |
|-----------|---------------|
| `num_init_children` | Nombre de processus enfants PgPool |
| `max_pool` | Connexions maintenues vers chaque backend par enfant |
| `backend_weight0 = 0` | Primary ne reçoit pas de lectures |
| `backend_weight1 = 1` | Standby reçoit des lectures (poids égal) |
| `ALWAYS_PRIMARY` | Ce backend est toujours considéré comme Primary |
| `load_balance_mode = on` | Activer la répartition des lectures |

### Fonctionnalités Avancées

#### 1. Query Routing Intelligent

PgPool-II analyse chaque requête :

```sql
-- Routée vers un Standby (lecture)
SELECT * FROM users WHERE id = 1;

-- Routée vers le Primary (écriture)
INSERT INTO users (name) VALUES ('Bob');

-- Routée vers le Primary (transaction)
BEGIN;  
SELECT * FROM users WHERE id = 1 FOR UPDATE;  
UPDATE users SET name = 'Robert' WHERE id = 1;  
COMMIT;  
```

**Règles de routage :**
- **SELECT sans transaction** → Standby (load balanced)  
- **INSERT/UPDATE/DELETE** → Primary  
- **Toute requête dans une transaction** → Primary (cohérence)  
- **SELECT ... FOR UPDATE** → Primary  
- **Appel de fonctions "write"** (nextval, setval) → Primary

#### 2. Connection Pooling

PgPool-II maintient un pool de connexions persistantes :

```
Application (100 clients) → PgPool-II → PostgreSQL (10 connexions max par backend)
```

**Avantages :**
- Réduction de la latence (pas de reconnexion)
- Économie de ressources sur PostgreSQL
- Support de plus de clients que de connexions backend

#### 3. Failover Automatique

Si PgPool-II détecte qu'un backend est down :
- Il le marque comme "down" automatiquement
- Il redirige le trafic vers les autres backends
- Il peut exécuter un script personnalisé (promotion d'un Standby)

Configuration du failover :

```conf
failover_on_backend_error = on  
failover_command = '/usr/local/bin/failover.sh %d %h %p %D %m %H %M %P %r %R'  
```

Exemple de script `/usr/local/bin/failover.sh` :

```bash
#!/bin/bash
# Paramètres passés par PgPool-II
NODE_ID=$1  
NODE_HOST=$2  
NODE_PORT=$3  
# ... autres paramètres

# Logger l'événement
echo "$(date): Failover triggered for node $NODE_ID ($NODE_HOST:$NODE_PORT)" >> /var/log/pgpool_failover.log

# Promouvoir un Standby (exemple simplifié)
if [ $NODE_ID -eq 0 ]; then
    # Le Primary est tombé, promouvoir Standby1
    ssh standby1.domain.com "pg_ctl promote -D /var/lib/postgresql/14/main"
fi
```

#### 4. Online Recovery

PgPool-II peut réintégrer automatiquement un serveur après réparation sans interruption de service.

## Architectures Recommandées

### Architecture 1 : HAProxy pour la Simplicité

```
          Applications
               |
           HAProxy
        /            \
   Port 5432      Port 5433
   (Write)         (Read)
       |              |
   Primary    +-------+-------+
             Standby1      Standby2
```

**Cas d'usage :**
- Application capable de gérer 2 connexions (read/write)
- Performance critique (latence minimale)
- Infrastructure déjà familière avec HAProxy
- Besoin de monitoring HTTP

**Avantages :**
- ✅ Performance maximale  
- ✅ Configuration simple  
- ✅ Stable et éprouvé

**Inconvénients :**
- ⚠️ L'app doit gérer read vs write  
- ⚠️ Pas de connection pooling

---

### Architecture 2 : PgPool-II pour l'Intelligence

```
       Applications
            |
       PgPool-II
            |
    +-------+-------+
    |       |       |
 Primary Standby1 Standby2
```

**Cas d'usage :**
- Application avec connexion unique souhaitée
- Besoin de routing automatique
- Besoin de connection pooling
- Équipe familière avec PostgreSQL

**Avantages :**
- ✅ Transparence pour l'application  
- ✅ Routing intelligent  
- ✅ Connection pooling intégré

**Inconvénients :**
- ⚠️ Overhead de latence (~1-2ms)  
- ⚠️ Configuration plus complexe

---

### Architecture 3 : Hybride (Best of Both Worlds)

```
       Applications
            |
       PgPool-II (connection pooling + routing)
            |
         HAProxy (failover + monitoring)
            |
    +-------+-------+
    |       |       |
 Primary Standby1 Standby2
```

**Cas d'usage :**
- Infrastructure critique
- Besoin de toutes les fonctionnalités
- Budget disponible pour la complexité

**Avantages :**
- ✅ Connection pooling (PgPool-II)  
- ✅ Routing intelligent (PgPool-II)  
- ✅ Monitoring avancé (HAProxy)  
- ✅ Détection de panne robuste (HAProxy)

**Inconvénients :**
- ⚠️ Complexité opérationnelle élevée  
- ⚠️ Deux couches = deux points de défaillance  
- ⚠️ Overhead cumulé

**Note :** Cette architecture est rarement nécessaire. Préférez HAProxy OU PgPool-II.

---

### Architecture 4 : HAProxy en Haute Disponibilité

```
           Applications
                |
         Virtual IP (keepalived)
           /         \
    HAProxy-1      HAProxy-2
    (Primary)      (Backup)
         |             |
    +----+----+--------+
    |         |        |
 Primary  Standby1  Standby2
```

**Cas d'usage :**
- HAProxy ne doit pas être un SPOF
- Infrastructure critique
- Besoin de 99.99% de disponibilité

**Configuration :**
- **Keepalived** : Gère une IP virtuelle (VIP)
- Si HAProxy-1 tombe, la VIP bascule vers HAProxy-2
- Les applications pointent toujours vers la VIP

## Bonnes Pratiques

### 🎯 Choix de la Solution

**Choisissez HAProxy si :**
- ✅ Votre application peut gérer 2 connexions (read/write)  
- ✅ Vous privilégiez la performance pure  
- ✅ Vous avez déjà une expertise HAProxy  
- ✅ Vous voulez une solution simple et éprouvée

**Choisissez PgPool-II si :**
- ✅ Vous voulez une connexion unique depuis l'application  
- ✅ Vous avez besoin de connection pooling  
- ✅ Le routing automatique est important  
- ✅ Vous êtes prêt à investir dans la configuration

**Ne choisissez pas les deux en même temps** (sauf cas très spécifiques).

### 🎯 Dimensionnement

**HAProxy :**
```conf
# Configuration minimaliste
maxconn 10000                  # Adapté à la plupart des cas  
timeout connect 5s  
timeout client 50s  
timeout server 50s  
```

**PgPool-II :**
```conf
# Règle empirique
num_init_children = (Nombre de connexions clients max) / max_pool  
max_pool = 4                   # 4 connexions persistantes par backend par processus  

# Exemple : 1000 clients, 4 connexions par backend
# num_init_children = 1000 / 4 = 250
```

### 🎯 Health Checks

**Fréquence :**
- HAProxy : `inter 2000` (toutes les 2 secondes)
- PgPool-II : `health_check_period = 10` (toutes les 10 secondes)

**Tolérance :**
- HAProxy : `rise 2 fall 3` (2 succès pour UP, 3 échecs pour DOWN)
- PgPool-II : `health_check_max_retries = 3`

**Principe :** Trouver le bon équilibre entre réactivité et faux positifs.

### 🎯 Monitoring

**HAProxy :**
```bash
# Stats interface
curl http://haproxy-ip:8404/stats

# Via socket (plus détaillé)
echo "show stat" | sudo socat stdio /var/lib/haproxy/stats
```

**PgPool-II :**
```sql
-- État des backends
SHOW POOL_NODES;

-- Processus actifs
SHOW POOL_PROCESSES;

-- Statistiques
SHOW POOL_VERSION;
```

### 🎯 Sécurité

**HAProxy :**
- Utiliser `bind` avec des IP spécifiques (pas `*`)
- Activer `stats auth` pour l'interface de monitoring
- Limiter les connexions par IP (`tcp-request connection reject if { src_conn_rate gt 100 }`)

**PgPool-II :**
- Configurer `pool_hba.conf` de manière restrictive
- Utiliser `enable_pool_hba = on`
- Chiffrer les mots de passe dans `pool_passwd`
- Activer SSL entre PgPool et PostgreSQL

### 🎯 Tests de Charge

Avant la mise en production, testez :

```bash
# pgbench via HAProxy ou PgPool-II
pgbench -h load-balancer-ip -p 5432 -U test_user -d testdb -c 50 -j 4 -T 60

# Mesurer les performances
# - Transactions per second (TPS)
# - Latence moyenne et P99
# - Taux d'erreur
```

## Troubleshooting

### Problème 1 : "Connection refused" sur HAProxy

**Symptômes :**
```
psql: error: connection to server at "haproxy-ip", port 5432 failed: Connection refused
```

**Causes possibles :**
1. HAProxy n'est pas démarré  
2. HAProxy n'écoute pas sur `*` ou la bonne IP  
3. Firewall bloque le port

**Solutions :**
```bash
# Vérifier le statut
sudo systemctl status haproxy

# Vérifier les ports
sudo netstat -tlnp | grep haproxy

# Vérifier les logs
sudo journalctl -u haproxy -f
```

---

### Problème 2 : Tous les serveurs sont "DOWN" dans HAProxy

**Symptômes :**
```
# Dans les stats HAProxy
Backend: postgres_replicas  
All servers DOWN  
```

**Causes possibles :**
1. L'utilisateur `haproxy_check` n'existe pas sur PostgreSQL  
2. `pg_hba.conf` n'autorise pas `haproxy_check`  
3. Les serveurs PostgreSQL sont réellement down

**Solutions :**
```sql
-- Sur chaque serveur PostgreSQL
CREATE USER haproxy_check;

-- Dans pg_hba.conf
host postgres haproxy_check haproxy-ip/32 trust

-- Recharger la configuration
SELECT pg_reload_conf();
```

---

### Problème 3 : PgPool-II ne route pas vers les Standby

**Symptômes :**
```sql
-- Toutes les requêtes vont au Primary
SHOW POOL_NODES;
-- select_cnt élevé sur Primary, 0 sur Standby
```

**Causes possibles :**
1. `load_balance_mode = off`  
2. `backend_weight` des Standby est 0  
3. Les requêtes sont dans une transaction

**Solutions :**
```conf
# Dans pgpool.conf
load_balance_mode = on  
backend_weight1 = 1    # Standby 1  
backend_weight2 = 1    # Standby 2  

# Redémarrer PgPool-II
sudo systemctl restart pgpool2
```

**Note :** Les requêtes dans une transaction vont TOUJOURS au Primary (c'est normal).

---

### Problème 4 : Latence élevée avec PgPool-II

**Symptômes :**
```
Requêtes 2-3ms plus lentes qu'en direct
```

**Causes :**
- Overhead normal de parsing SQL par PgPool-II

**Solutions :**
1. **Connection pooling optimisé :**
   ```conf
   max_pool = 4              # Plus de connexions persistantes
   child_life_time = 0       # Pas de rotation
   ```

2. **Cache de requêtes** (pour SELECT répétitifs) :
   ```conf
   memory_cache_enabled = on
   memqcache_method = 'memcached'
   ```

3. **Si performance critique :** Envisager HAProxy à la place

## Cas Pratiques

### Cas 1 : Startup avec Budget Limité

**Besoin :** Simple, performant, peu coûteux

**Recommandation :** **HAProxy**

```
    App
     |
  HAProxy
  /     \
Write  Read
  |      |
Primary Standby
```

**Justification :**
- 2 serveurs seulement (Primary + 1 Standby)
- HAProxy léger et rapide
- L'app gère read/write (acceptable pour une petite équipe)

---

### Cas 2 : SaaS Multi-Tenant

**Besoin :** Scaling, pooling, routing automatique

**Recommandation :** **PgPool-II**

```
   Applications (100-1000 clients)
              |
         PgPool-II (pooling)
              |
      +-------+-------+
      |       |       |
   Primary Standby1 Standby2
```

**Justification :**
- Connection pooling essentiel (économie de connexions)
- Routing automatique simplifie le code applicatif
- Nombreux tenants = nombreuses connexions

---

### Cas 3 : E-Commerce avec Pics de Trafic

**Besoin :** Haute disponibilité, monitoring, performance

**Recommandation :** **HAProxy + HA HAProxy**

```
      Applications
           |
    Virtual IP (keepalived)
       /        \
  HAProxy-1  HAProxy-2
       |        |
   +---+---+----+
   |       |    |
Primary Standby1 Standby2
```

**Justification :**
- Performance maximale (HAProxy)
- Résilience (HA HAProxy)
- Monitoring temps réel essentiel (interface web)
- Pics de Black Friday supportés

---

## Conclusion

Le load balancing est un composant **essentiel** d'une architecture PostgreSQL hautement disponible. Le choix entre HAProxy et PgPool-II dépend de vos priorités :

**HAProxy : Performance et Simplicité**
- ✅ Si vous privilégiez la vitesse pure  
- ✅ Si votre application peut gérer 2 connexions  
- ✅ Si vous voulez une solution éprouvée et stable  
- ✅ Si vous avez déjà l'expertise HAProxy

**PgPool-II : Intelligence et Intégration**
- ✅ Si vous voulez une connexion unique  
- ✅ Si le connection pooling est important  
- ✅ Si le routing automatique simplifie votre architecture  
- ✅ Si vous acceptez un overhead de 1-2ms

**Points clés à retenir :**

1. 🎯 **Le load balancer n'est pas optionnel** en production  
2. 🎯 **Testez avant de choisir** : Vos besoins sont uniques  
3. 🎯 **Monitoring essentiel** : Surveillez la santé des backends  
4. 🎯 **HA du load balancer** : Ne créez pas un nouveau SPOF  
5. 🎯 **Documentation** : Documentez votre configuration

Quelle que soit votre choix, un load balancer bien configuré transforme votre infrastructure PostgreSQL en un système véritablement résilient et performant.
