# 15.4.2. Triggers par instruction (FOR EACH STATEMENT)

## Introduction

Les **triggers par instruction** (ou *statement-level triggers*) sont des déclencheurs qui s'exécutent **une seule fois par opération SQL**, quel que soit le nombre de lignes affectées. Contrairement aux triggers par ligne (FOR EACH ROW), ils ne s'exécutent pas pour chaque enregistrement individuel, mais une seule fois pour l'instruction complète.

### Analogie pour comprendre la différence

Reprenons l'analogie du magasin :

| Type de trigger | Analogie | Quand s'exécute-t-il ? |
|-----------------|----------|------------------------|
| **FOR EACH ROW** | Portique de sécurité qui scanne chaque personne individuellement | 1 fois par personne (par ligne) |
| **FOR EACH STATEMENT** | Gardien qui vérifie le groupe entier en une fois | 1 fois pour tout le groupe (par instruction) |

### Exemple concret

```sql
-- Si cette requête affecte 1000 lignes :
UPDATE employes SET salaire = salaire * 1.10;

-- FOR EACH ROW : s'exécute 1000 fois (une fois par ligne)
-- FOR EACH STATEMENT : s'exécute 1 seule fois
```

## Syntaxe de base

```sql
CREATE TRIGGER nom_du_trigger
  { BEFORE | AFTER } { INSERT | UPDATE | DELETE }
  ON nom_table
  FOR EACH STATEMENT
  EXECUTE FUNCTION nom_fonction_trigger();
```

### Différence clé dans la syntaxe

La seule différence syntaxique avec un trigger par ligne est la clause :

- **FOR EACH ROW** → Trigger par ligne  
- **FOR EACH STATEMENT** → Trigger par instruction

## Caractéristiques fondamentales

### 1. Pas d'accès à NEW et OLD

C'est la différence majeure avec les triggers par ligne :

❌ **Impossible d'accéder aux données individuelles**
```sql
CREATE OR REPLACE FUNCTION trigger_statement()  
RETURNS TRIGGER AS $$  
BEGIN  
  -- ❌ ERREUR : NEW et OLD ne sont pas disponibles
  RAISE NOTICE 'Nouvelle valeur : %', NEW.colonne;

  RETURN NULL;
END;
$$ LANGUAGE plpgsql;
```

✅ **On peut seulement savoir qu'une opération a eu lieu**
```sql
CREATE OR REPLACE FUNCTION trigger_statement()  
RETURNS TRIGGER AS $$  
BEGIN  
  -- ✅ OK : On sait qu'une opération s'est produite
  RAISE NOTICE 'Opération % sur la table %', TG_OP, TG_TABLE_NAME;

  RETURN NULL;
END;
$$ LANGUAGE plpgsql;
```

### 2. Performance optimale pour les opérations en masse

**Avantage principal :** Idéal pour les opérations affectant de nombreuses lignes

```sql
-- Cette opération affecte 50 000 lignes

-- Avec FOR EACH ROW : 50 000 exécutions du trigger
-- Temps : ~5 secondes

-- Avec FOR EACH STATEMENT : 1 seule exécution
-- Temps : ~0.01 secondes
```

### 3. Valeur de retour

La valeur de retour est **toujours ignorée** (contrairement aux triggers BEFORE ROW).

```sql
CREATE OR REPLACE FUNCTION trigger_statement()  
RETURNS TRIGGER AS $$  
BEGIN  
  -- La valeur de retour n'a aucun impact
  RETURN NULL;  -- Acceptable
END;
$$ LANGUAGE plpgsql;
```

Par convention, on retourne toujours `NULL` dans les triggers par instruction.

## BEFORE vs AFTER Statement

### BEFORE STATEMENT

**S'exécute avant que l'opération soit effectuée**

**Cas d'usage typiques :**
1. **Vérifications préliminaires** avant traitement  
2. **Préparation de tables temporaires**  
3. **Validation de conditions globales**  
4. **Verrouillage de ressources**

**Exemple - Vérification de l'heure**
```sql
CREATE OR REPLACE FUNCTION verifier_horaire_modification()  
RETURNS TRIGGER AS $$  
BEGIN  
  -- Interdire les modifications en dehors des heures de bureau
  IF EXTRACT(HOUR FROM now()) < 8 OR EXTRACT(HOUR FROM now()) > 18 THEN
    RAISE EXCEPTION 'Modifications interdites en dehors des heures de bureau (8h-18h)';
  END IF;

  -- Interdire les modifications le weekend
  IF EXTRACT(DOW FROM now()) IN (0, 6) THEN  -- 0=dimanche, 6=samedi
    RAISE EXCEPTION 'Modifications interdites le weekend';
  END IF;

  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_horaire
  BEFORE UPDATE OR DELETE ON donnees_critiques
  FOR EACH STATEMENT
  EXECUTE FUNCTION verifier_horaire_modification();
```

**Explication :** Avant toute tentative de modification de la table `donnees_critiques`, le trigger vérifie que nous sommes bien en heures de bureau. Si ce n'est pas le cas, l'opération est bloquée avec un message d'erreur explicite.

### AFTER STATEMENT

**S'exécute après que l'opération ait été effectuée**

**Cas d'usage typiques :**
1. **Journalisation d'opérations en masse**  
2. **Mise à jour de statistiques ou compteurs**  
3. **Notifications d'événements**  
4. **Refresh de vues matérialisées**  
5. **Nettoyage ou maintenance**

**Exemple - Journalisation des opérations en masse**
```sql
-- Table de log
CREATE TABLE operations_log (
  id SERIAL PRIMARY KEY,
  table_name TEXT,
  operation TEXT,
  nb_lignes_estimees INTEGER,
  executee_par TEXT,
  executee_le TIMESTAMP DEFAULT now()
);

-- Fonction trigger
CREATE OR REPLACE FUNCTION logger_operation_masse()  
RETURNS TRIGGER AS $$  
BEGIN  
  INSERT INTO operations_log (
    table_name,
    operation,
    executee_par
  ) VALUES (
    TG_TABLE_NAME,
    TG_OP,
    current_user
  );

  RAISE NOTICE 'Opération % enregistrée sur %', TG_OP, TG_TABLE_NAME;

  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

-- Trigger
CREATE TRIGGER trigger_log_masse
  AFTER INSERT OR UPDATE OR DELETE ON produits
  FOR EACH STATEMENT
  EXECUTE FUNCTION logger_operation_masse();
```

**Scénario d'utilisation :**
```sql
-- Cette opération massive ne génère qu'une seule entrée de log
UPDATE produits SET prix = prix * 1.05 WHERE categorie = 'electronique';

-- Résultat dans operations_log :
-- | table_name | operation | executee_par | executee_le |
-- | produits   | UPDATE    | admin        | 2025-11-22... |
```

## Variables contextuelles disponibles

Bien que NEW et OLD ne soient pas disponibles, plusieurs variables contextuelles restent accessibles :

### TG_OP
Type d'opération : 'INSERT', 'UPDATE', ou 'DELETE'

```sql
IF TG_OP = 'DELETE' THEN
  RAISE WARNING 'Suppression massive détectée';
END IF;
```

### TG_TABLE_NAME
Nom de la table concernée

```sql
RAISE NOTICE 'Opération sur la table : %', TG_TABLE_NAME;
```

### TG_TABLE_SCHEMA
Schéma de la table

```sql
IF TG_TABLE_SCHEMA = 'production' THEN
  -- Logique spécifique pour le schéma de production
END IF;
```

### TG_WHEN
Moment du trigger : 'BEFORE' ou 'AFTER'

```sql
RAISE DEBUG 'Trigger % en cours', TG_WHEN;
```

### TG_LEVEL
Niveau du trigger : toujours 'STATEMENT' ici

```sql
-- Toujours 'STATEMENT' dans ce contexte
ASSERT TG_LEVEL = 'STATEMENT';
```

## Exemples pratiques détaillés

### Exemple 1 : Mise à jour d'un compteur de modifications

**Contexte :** Suivre le nombre total d'opérations sur une table sans détailler chaque ligne

```sql
-- Table de statistiques
CREATE TABLE table_stats (
  table_name TEXT PRIMARY KEY,
  nb_insertions INTEGER DEFAULT 0,
  nb_updates INTEGER DEFAULT 0,
  nb_deletions INTEGER DEFAULT 0,
  derniere_modif TIMESTAMP
);

-- Initialiser pour la table produits
INSERT INTO table_stats (table_name) VALUES ('produits');

-- Fonction trigger
CREATE OR REPLACE FUNCTION maj_statistiques()  
RETURNS TRIGGER AS $$  
BEGIN  
  -- Mettre à jour les statistiques selon l'opération
  IF TG_OP = 'INSERT' THEN
    UPDATE table_stats
    SET nb_insertions = nb_insertions + 1,
        derniere_modif = now()
    WHERE table_name = TG_TABLE_NAME;

  ELSIF TG_OP = 'UPDATE' THEN
    UPDATE table_stats
    SET nb_updates = nb_updates + 1,
        derniere_modif = now()
    WHERE table_name = TG_TABLE_NAME;

  ELSIF TG_OP = 'DELETE' THEN
    UPDATE table_stats
    SET nb_deletions = nb_deletions + 1,
        derniere_modif = now()
    WHERE table_name = TG_TABLE_NAME;
  END IF;

  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

-- Création du trigger
CREATE TRIGGER trigger_stats
  AFTER INSERT OR UPDATE OR DELETE ON produits
  FOR EACH STATEMENT
  EXECUTE FUNCTION maj_statistiques();
```

**Utilisation :**
```sql
-- Effectuer des opérations
INSERT INTO produits (nom, prix) VALUES ('Laptop', 999.99);  
UPDATE produits SET prix = prix * 1.10 WHERE categorie = 'tech';  
DELETE FROM produits WHERE stock = 0;  

-- Consulter les statistiques
SELECT * FROM table_stats WHERE table_name = 'produits';
-- | table_name | nb_insertions | nb_updates | nb_deletions | derniere_modif |
-- | produits   | 1             | 1          | 1            | 2025-11-22...  |
```

### Exemple 2 : Refresh automatique de vue matérialisée

**Contexte :** Actualiser une vue matérialisée après chaque modification d'une table source

```sql
-- Vue matérialisée (calcul coûteux)
CREATE MATERIALIZED VIEW ventes_mensuelles AS  
SELECT  
  DATE_TRUNC('month', date_vente) AS mois,
  SUM(montant) AS total_ventes,
  COUNT(*) AS nb_ventes
FROM ventes  
GROUP BY DATE_TRUNC('month', date_vente);  

-- Index pour performance
CREATE INDEX idx_ventes_mensuelles_mois ON ventes_mensuelles(mois);

-- Fonction de refresh
CREATE OR REPLACE FUNCTION refresh_ventes_mensuelles()  
RETURNS TRIGGER AS $$  
BEGIN  
  -- Rafraîchir la vue matérialisée
  REFRESH MATERIALIZED VIEW CONCURRENTLY ventes_mensuelles;

  RAISE NOTICE 'Vue matérialisée ventes_mensuelles rafraîchie';

  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

-- Trigger qui rafraîchit après chaque batch de modifications
CREATE TRIGGER trigger_refresh_stats
  AFTER INSERT OR UPDATE OR DELETE ON ventes
  FOR EACH STATEMENT
  EXECUTE FUNCTION refresh_ventes_mensuelles();
```

**Avantage :** Une seule opération de refresh pour un batch de 1000 insertions, au lieu de 1000 refresh avec FOR EACH ROW.

### Exemple 3 : Notification d'opération massive

**Contexte :** Alerter les administrateurs lors d'opérations massives

```sql
-- Table de notifications
CREATE TABLE alertes_admin (
  id SERIAL PRIMARY KEY,
  message TEXT,
  severite TEXT,
  cree_le TIMESTAMP DEFAULT now()
);

-- Fonction de notification
CREATE OR REPLACE FUNCTION notifier_operation_masse()  
RETURNS TRIGGER AS $$  
DECLARE  
  message_alerte TEXT;
BEGIN
  -- Construire le message d'alerte
  message_alerte := format(
    'Opération %s effectuée sur la table %s par %s',
    TG_OP,
    TG_TABLE_NAME,
    current_user
  );

  -- Enregistrer l'alerte
  INSERT INTO alertes_admin (message, severite)
  VALUES (message_alerte, 'INFO');

  -- Notification PostgreSQL (NOTIFY/LISTEN)
  PERFORM pg_notify('operations_masse', message_alerte);

  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

-- Trigger sur les tables sensibles
CREATE TRIGGER trigger_alerte_employes
  AFTER UPDATE OR DELETE ON employes
  FOR EACH STATEMENT
  EXECUTE FUNCTION notifier_operation_masse();
```

**Utilisation :**
```sql
-- Côté application : écouter les notifications
LISTEN operations_masse;

-- Côté base : effectuer une opération
UPDATE employes SET salaire = salaire * 1.05;

-- L'application reçoit immédiatement :
-- "Opération UPDATE effectuée sur la table employes par admin"
```

### Exemple 4 : Validation de contrainte globale

**Contexte :** Vérifier qu'une contrainte métier globale est respectée après une modification

```sql
-- Règle métier : Le total des pourcentages de répartition doit égaler 100%
CREATE TABLE repartition_budget (
  departement TEXT PRIMARY KEY,
  pourcentage NUMERIC(5,2) CHECK (pourcentage >= 0 AND pourcentage <= 100)
);

-- Fonction de validation globale
CREATE OR REPLACE FUNCTION valider_total_pourcentage()  
RETURNS TRIGGER AS $$  
DECLARE  
  total_pct NUMERIC;
BEGIN
  -- Calculer le total des pourcentages
  SELECT SUM(pourcentage) INTO total_pct
  FROM repartition_budget;

  -- Vérifier que le total est 100%
  IF total_pct != 100 THEN
    RAISE EXCEPTION
      'Le total des pourcentages doit être égal à 100%% (actuellement : %%)',
      total_pct
      USING HINT = 'Ajustez les répartitions pour atteindre 100%';
  END IF;

  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

-- Trigger de validation (AFTER pour avoir les valeurs finales)
CREATE TRIGGER trigger_validation_total
  AFTER INSERT OR UPDATE OR DELETE ON repartition_budget
  FOR EACH STATEMENT
  EXECUTE FUNCTION valider_total_pourcentage();
```

**Scénario d'utilisation :**
```sql
-- ✅ OK : Total = 100%
BEGIN;
  INSERT INTO repartition_budget VALUES ('IT', 40.00);
  INSERT INTO repartition_budget VALUES ('Marketing', 35.00);
  INSERT INTO repartition_budget VALUES ('RH', 25.00);
COMMIT;

-- ❌ ERREUR : Total != 100%
BEGIN;
  INSERT INTO repartition_budget VALUES ('IT', 50.00);
  INSERT INTO repartition_budget VALUES ('Marketing', 30.00);
  -- Trigger déclenché : ERREUR car total = 80% seulement
COMMIT;
```

### Exemple 5 : Maintenance automatique après opérations massives

**Contexte :** Déclencher des tâches de maintenance après des modifications importantes

```sql
CREATE OR REPLACE FUNCTION maintenance_apres_maj()  
RETURNS TRIGGER AS $$  
BEGIN  
  -- Pour les opérations DELETE massives, nettoyer
  IF TG_OP = 'DELETE' THEN
    -- Nettoyer les références orphelines dans d'autres tables
    DELETE FROM cache_produits
    WHERE produit_id NOT IN (SELECT id FROM produits);

    RAISE NOTICE 'Nettoyage du cache effectué';
  END IF;

  -- Pour les UPDATE massifs, mettre à jour les index
  IF TG_OP = 'UPDATE' THEN
    -- Analyser la table pour mettre à jour les statistiques
    EXECUTE 'ANALYZE ' || TG_TABLE_NAME;

    RAISE NOTICE 'Statistiques de % mises à jour', TG_TABLE_NAME;
  END IF;

  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_maintenance
  AFTER UPDATE OR DELETE ON produits
  FOR EACH STATEMENT
  EXECUTE FUNCTION maintenance_apres_maj();
```

## Accéder aux lignes affectées (solution de contournement)

Bien que NEW et OLD ne soient pas directement accessibles, il existe des techniques pour accéder aux données modifiées :

### Technique 1 : Tables de transition (PostgreSQL 10+)

PostgreSQL permet de créer des tables temporaires contenant les lignes affectées :

```sql
CREATE OR REPLACE FUNCTION analyser_modifications()  
RETURNS TRIGGER AS $$  
DECLARE  
  nb_lignes INTEGER;
BEGIN
  -- Compter les lignes dans la table de transition
  -- 'new_table' contient les nouvelles valeurs (INSERT/UPDATE)
  -- 'old_table' contient les anciennes valeurs (UPDATE/DELETE)

  IF TG_OP = 'INSERT' THEN
    SELECT COUNT(*) INTO nb_lignes FROM new_table;
    RAISE NOTICE '% lignes insérées', nb_lignes;

  ELSIF TG_OP = 'UPDATE' THEN
    SELECT COUNT(*) INTO nb_lignes FROM new_table;
    RAISE NOTICE '% lignes mises à jour', nb_lignes;

  ELSIF TG_OP = 'DELETE' THEN
    SELECT COUNT(*) INTO nb_lignes FROM old_table;
    RAISE NOTICE '% lignes supprimées', nb_lignes;
  END IF;

  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

-- Créer le trigger avec tables de transition
CREATE TRIGGER trigger_analyse
  AFTER INSERT OR UPDATE OR DELETE ON produits
  REFERENCING
    NEW TABLE AS new_table
    OLD TABLE AS old_table
  FOR EACH STATEMENT
  EXECUTE FUNCTION analyser_modifications();
```

**Exemple avancé avec tables de transition :**
```sql
CREATE OR REPLACE FUNCTION auditer_changements_prix()  
RETURNS TRIGGER AS $$  
DECLARE  
  augmentation_moyenne NUMERIC;
  nb_augmentations INTEGER;
BEGIN
  -- Analyser les changements de prix
  SELECT
    AVG(n.prix - o.prix) AS aug_moyenne,
    COUNT(*) AS nb_changes
  INTO
    augmentation_moyenne,
    nb_augmentations
  FROM new_table n
  JOIN old_table o ON n.id = o.id
  WHERE n.prix != o.prix;

  -- Logger si des prix ont changé
  IF nb_augmentations > 0 THEN
    INSERT INTO audit_prix (
      nb_produits_modifies,
      augmentation_moyenne,
      date_modification
    ) VALUES (
      nb_augmentations,
      augmentation_moyenne,
      now()
    );

    RAISE NOTICE 'Prix modifiés : % produits, augmentation moyenne : %',
      nb_augmentations, augmentation_moyenne;
  END IF;

  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_audit_prix
  AFTER UPDATE ON produits
  REFERENCING
    NEW TABLE AS new_table
    OLD TABLE AS old_table
  FOR EACH STATEMENT
  EXECUTE FUNCTION auditer_changements_prix();
```

### Technique 2 : Table temporaire manuelle

Pour les versions antérieures ou des cas spécifiques :

```sql
-- Table temporaire pour stocker les IDs modifiés
CREATE TEMPORARY TABLE IF NOT EXISTS temp_ids_modifies (
  id INTEGER
);

-- Trigger ROW pour capturer les IDs
CREATE OR REPLACE FUNCTION capturer_ids()  
RETURNS TRIGGER AS $$  
BEGIN  
  INSERT INTO temp_ids_modifies VALUES (NEW.id);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Trigger STATEMENT pour traiter en masse
CREATE OR REPLACE FUNCTION traiter_modifications()  
RETURNS TRIGGER AS $$  
DECLARE  
  nb_modifs INTEGER;
BEGIN
  SELECT COUNT(*) INTO nb_modifs FROM temp_ids_modifies;

  RAISE NOTICE '% lignes modifiées', nb_modifs;

  -- Nettoyer la table temporaire
  DELETE FROM temp_ids_modifies;

  RETURN NULL;
END;
$$ LANGUAGE plpgsql;
```

## FOR EACH ROW vs FOR EACH STATEMENT : Tableau comparatif

| Critère | FOR EACH ROW | FOR EACH STATEMENT |
|---------|--------------|-------------------|
| **Exécutions** | 1 fois par ligne affectée | 1 fois par instruction SQL |
| **Accès à NEW/OLD** | ✅ Oui | ❌ Non (sauf via tables de transition) |
| **Modification des données** | ✅ Possible (BEFORE) | ❌ Impossible |
| **Performance (1 ligne)** | Rapide | Très rapide |
| **Performance (10 000 lignes)** | Lent (10 000 exécutions) | Très rapide (1 exécution) |
| **Validation par ligne** | ✅ Idéal | ❌ Impossible |
| **Statistiques globales** | ❌ Inefficace | ✅ Parfait |
| **Audit détaillé** | ✅ Précis | ❌ Limité |
| **Complexité** | Moyenne | Faible |

## Quand utiliser FOR EACH STATEMENT ?

### ✅ Cas d'usage recommandés

1. **Opérations en masse**
   - Import de données (COPY, INSERT massif)
   - Mises à jour groupées (UPDATE avec WHERE large)
   - Suppressions massives

2. **Statistiques et compteurs globaux**
   - Compter le nombre d'opérations
   - Suivre la fréquence des modifications
   - Métriques de performance

3. **Notifications d'événements**
   - Alerter d'une modification (sans détail par ligne)
   - Déclencher des processus externes
   - Système de pub/sub (NOTIFY/LISTEN)

4. **Maintenance et optimisation**
   - Refresh de vues matérialisées
   - ANALYZE automatique
   - Nettoyage de tables liées

5. **Validation de contraintes globales**
   - Vérifier des totaux (sommes, pourcentages)
   - Contraintes inter-lignes
   - Règles métier globales

6. **Journalisation simplifiée**
   - Logger qu'une opération a eu lieu
   - Audit de haut niveau
   - Conformité réglementaire basique

### ❌ Cas où FOR EACH STATEMENT n'est PAS adapté

1. **Transformation de données par ligne**
   - Calculer une colonne à partir d'autres colonnes
   - Normaliser des valeurs individuelles

2. **Validation détaillée**
   - Vérifier chaque enregistrement individuellement
   - Règles métier spécifiques par ligne

3. **Audit détaillé**
   - Tracer les modifications ligne par ligne
   - Historiser les anciennes et nouvelles valeurs

4. **Synchronisation précise**
   - Réplication sélective
   - Mise à jour de tables liées basée sur des valeurs spécifiques

## Combinaison de triggers ROW et STATEMENT

Il est possible et parfois utile d'avoir les deux types sur la même table :

```sql
-- Trigger par ligne : validation détaillée
CREATE OR REPLACE FUNCTION valider_ligne()  
RETURNS TRIGGER AS $$  
BEGIN  
  IF NEW.prix < 0 THEN
    RAISE EXCEPTION 'Prix négatif interdit';
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_validation_ligne
  BEFORE INSERT OR UPDATE ON produits
  FOR EACH ROW
  EXECUTE FUNCTION valider_ligne();

-- Trigger par instruction : statistiques globales
CREATE OR REPLACE FUNCTION maj_stats_globales()  
RETURNS TRIGGER AS $$  
BEGIN  
  UPDATE stats_globales
  SET derniere_modif_produits = now(),
      nb_operations = nb_operations + 1;
  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_stats_globales
  AFTER INSERT OR UPDATE ON produits
  FOR EACH STATEMENT
  EXECUTE FUNCTION maj_stats_globales();
```

**Ordre d'exécution :**
1. BEFORE STATEMENT (si existe)  
2. BEFORE ROW pour chaque ligne  
3. L'opération SQL est effectuée  
4. AFTER ROW pour chaque ligne  
5. AFTER STATEMENT (si existe)

## Bonnes pratiques

### 1. Préférer STATEMENT pour les opérations volumineuses

```sql
-- ❌ Mauvais : FOR EACH ROW sur table volumineuse
CREATE TRIGGER mauvais_trigger
  AFTER INSERT ON logs  -- Table avec millions de lignes/jour
  FOR EACH ROW
  EXECUTE FUNCTION fonction_lourde();

-- ✅ Bon : FOR EACH STATEMENT
CREATE TRIGGER bon_trigger
  AFTER INSERT ON logs
  FOR EACH STATEMENT
  EXECUTE FUNCTION fonction_legere();
```

### 2. Utiliser STATEMENT pour les refresh de vues matérialisées

```sql
-- ✅ Refresh une seule fois après un batch complet
CREATE TRIGGER refresh_vue
  AFTER INSERT OR UPDATE ON table_source
  FOR EACH STATEMENT
  EXECUTE FUNCTION refresh_materialized_view();
```

### 3. Documenter clairement l'absence de NEW/OLD

```sql
CREATE OR REPLACE FUNCTION ma_fonction()  
RETURNS TRIGGER AS $$  
/**
 * Trigger de type STATEMENT
 * NOTE : NEW et OLD ne sont PAS disponibles
 * Objectif : Logger les opérations en masse sur la table
 */
BEGIN
  -- Logique qui ne nécessite pas NEW/OLD
  RETURN NULL;
END;
$$ LANGUAGE plpgsql;
```

### 4. Utiliser les tables de transition quand nécessaire

```sql
-- ✅ Accéder aux données via tables de transition
CREATE TRIGGER trigger_avec_donnees
  AFTER UPDATE ON produits
  REFERENCING
    NEW TABLE AS new_table
    OLD TABLE AS old_table
  FOR EACH STATEMENT
  EXECUTE FUNCTION analyser_avec_donnees();
```

### 5. Gérer proprement les erreurs

```sql
CREATE OR REPLACE FUNCTION fonction_statement()  
RETURNS TRIGGER AS $$  
BEGIN  
  -- Logique métier
  PERFORM ma_logique();

  RETURN NULL;
EXCEPTION
  WHEN OTHERS THEN
    RAISE WARNING 'Erreur dans trigger statement : %', SQLERRM;
    -- Décider si on propage l'erreur ou non
    RETURN NULL;  -- Ne pas bloquer l'opération
    -- ou RAISE; pour propager
END;
$$ LANGUAGE plpgsql;
```

## Impact sur les performances : Benchmark

Comparaison sur une table avec 100 000 lignes :

### Test 1 : UPDATE massif

```sql
-- Scénario : Augmenter tous les prix de 10%

-- Avec FOR EACH ROW (trigger simple)
UPDATE produits SET prix = prix * 1.10;
-- Temps : 15.2 secondes (100 000 exécutions du trigger)

-- Avec FOR EACH STATEMENT (trigger simple)
UPDATE produits SET prix = prix * 1.10;
-- Temps : 1.8 secondes (1 seule exécution du trigger)

-- Gain : 8.4× plus rapide
```

### Test 2 : INSERT massif

```sql
-- Scénario : Insérer 50 000 lignes

-- Avec FOR EACH ROW
INSERT INTO produits SELECT * FROM produits_import;
-- Temps : 8.5 secondes

-- Avec FOR EACH STATEMENT
INSERT INTO produits SELECT * FROM produits_import;
-- Temps : 0.9 secondes

-- Gain : 9.4× plus rapide
```

### Recommandations de performance

| Nombre de lignes | Recommandation |
|------------------|----------------|
| < 10 lignes | FOR EACH ROW acceptable |
| 10-100 lignes | Évaluer cas par cas |
| 100-1000 lignes | Préférer FOR EACH STATEMENT si possible |
| > 1000 lignes | Impératif FOR EACH STATEMENT |

### Mesurer le temps d'exécution

```sql
CREATE OR REPLACE FUNCTION mesurer_temps_statement()  
RETURNS TRIGGER AS $$  
DECLARE  
  debut TIMESTAMP;
  fin TIMESTAMP;
  duree INTERVAL;
BEGIN
  debut := clock_timestamp();

  -- Votre logique ici
  PERFORM pg_sleep(0.1);  -- Simuler un traitement

  fin := clock_timestamp();
  duree := fin - debut;

  RAISE NOTICE 'Durée du trigger: %', duree;

  RETURN NULL;
END;
$$ LANGUAGE plpgsql;
```

### Lister les triggers statement d'une table

```sql
SELECT
  tgname AS nom_trigger,
  tgtype & 1 = 1 AS is_row_trigger,
  CASE
    WHEN tgtype & 2 = 2 THEN 'BEFORE'
    ELSE 'AFTER'
  END AS timing,
  pg_get_triggerdef(oid) AS definition
FROM pg_trigger  
WHERE tgrelid = 'ma_table'::regclass  
  AND tgisinternal = false
  AND tgtype & 1 = 0  -- 0 = STATEMENT, 1 = ROW
ORDER BY tgname;
```

## Cas d'usage avancés

### 1. Rate Limiting

Limiter la fréquence des opérations :

```sql
CREATE TABLE derniere_operation (
  table_name TEXT PRIMARY KEY,
  derniere_exec TIMESTAMP
);

CREATE OR REPLACE FUNCTION limiter_frequence()  
RETURNS TRIGGER AS $$  
DECLARE  
  derniere TIMESTAMP;
  delai_minimum INTERVAL := '1 minute';
BEGIN
  SELECT derniere_exec INTO derniere
  FROM derniere_operation
  WHERE table_name = TG_TABLE_NAME;

  IF derniere IS NOT NULL AND (now() - derniere) < delai_minimum THEN
    RAISE EXCEPTION
      'Opération trop fréquente. Veuillez attendre % secondes',
      EXTRACT(EPOCH FROM (delai_minimum - (now() - derniere)));
  END IF;

  INSERT INTO derniere_operation (table_name, derniere_exec)
  VALUES (TG_TABLE_NAME, now())
  ON CONFLICT (table_name)
  DO UPDATE SET derniere_exec = now();

  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_rate_limit
  BEFORE UPDATE OR DELETE ON donnees_critiques
  FOR EACH STATEMENT
  EXECUTE FUNCTION limiter_frequence();
```

### 2. Circuit Breaker Pattern

Désactiver temporairement des opérations si trop d'erreurs :

```sql
CREATE TABLE circuit_breaker (
  table_name TEXT PRIMARY KEY,
  nb_erreurs INTEGER DEFAULT 0,
  etat TEXT DEFAULT 'FERME',  -- FERME, OUVERT, SEMI_OUVERT
  derniere_erreur TIMESTAMP
);

CREATE OR REPLACE FUNCTION verifier_circuit()  
RETURNS TRIGGER AS $$  
DECLARE  
  etat_circuit TEXT;
  nb_err INTEGER;
BEGIN
  SELECT etat, nb_erreurs INTO etat_circuit, nb_err
  FROM circuit_breaker
  WHERE table_name = TG_TABLE_NAME;

  IF etat_circuit = 'OUVERT' THEN
    RAISE EXCEPTION 'Circuit breaker OUVERT : opérations suspendues sur %',
      TG_TABLE_NAME;
  END IF;

  RETURN NULL;
END;
$$ LANGUAGE plpgsql;
```

### 3. Webhook déclenché par opération

```sql
CREATE OR REPLACE FUNCTION notifier_webhook()  
RETURNS TRIGGER AS $$  
DECLARE  
  payload JSONB;
BEGIN
  payload := jsonb_build_object(
    'table', TG_TABLE_NAME,
    'operation', TG_OP,
    'timestamp', now(),
    'user', current_user
  );

  -- Utiliser pg_notify pour un worker externe
  PERFORM pg_notify('webhook_channel', payload::text);

  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_webhook
  AFTER INSERT OR UPDATE OR DELETE ON commandes
  FOR EACH STATEMENT
  EXECUTE FUNCTION notifier_webhook();
```

## Résumé des points clés

### Caractéristiques essentielles

✅ **FOR EACH STATEMENT** s'exécute **une seule fois** par instruction SQL

✅ **Pas d'accès à NEW/OLD** (sauf via tables de transition)

✅ **Performance optimale** pour les opérations en masse

✅ **Idéal pour les statistiques** et compteurs globaux

✅ **Parfait pour les notifications** d'événements

### Quand utiliser STATEMENT vs ROW ?

| Besoin | Utiliser |
|--------|----------|
| Valider/modifier chaque ligne | FOR EACH ROW |
| Compteur d'opérations | FOR EACH STATEMENT |
| Audit détaillé (anciennes/nouvelles valeurs) | FOR EACH ROW |
| Refresh vue matérialisée | FOR EACH STATEMENT |
| Calculs par ligne | FOR EACH ROW |
| Notification simple | FOR EACH STATEMENT |
| Opération en masse (>1000 lignes) | FOR EACH STATEMENT |
| Règle métier par enregistrement | FOR EACH ROW |

### Points d'attention

⚠️ Impossible de modifier les données avec STATEMENT

⚠️ Pas de NEW/OLD sans tables de transition

⚠️ Nécessite PostgreSQL 10+ pour tables de transition

⚠️ Toujours retourner NULL par convention

## Conclusion

Les triggers par instruction (**FOR EACH STATEMENT**) sont un outil puissant pour les opérations qui concernent l'ensemble d'une modification plutôt que chaque ligne individuellement. Leur principal avantage est la **performance** sur les opérations volumineuses, en s'exécutant une seule fois au lieu de milliers de fois.

**Principe directeur :** Utilisez FOR EACH STATEMENT quand vous vous souciez du **fait** qu'une opération a eu lieu, et FOR EACH ROW quand vous avez besoin du **détail** de chaque modification.

Les deux types de triggers sont complémentaires et peuvent coexister sur la même table pour répondre à des besoins différents.
