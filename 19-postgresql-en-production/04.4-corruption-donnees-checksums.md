# 19.4.4. Corruption de données et Checksums

## Introduction : Le cauchemar silencieux

La **corruption de données** est le pire scénario qui puisse arriver à une base de données. Contrairement aux autres problèmes (verrous, saturation de ressources, wraparound), la corruption peut entraîner une **perte définitive de données**.

Imaginez que votre disque dur est comme un livre gigantesque. La corruption, c'est comme si des pages de ce livre devenaient illisibles, des mots disparaissaient, ou pire, changeaient sans raison. Vous pourriez lire "Solde : 1000€" alors que la valeur réelle a été corrompue en "Solde : #@$%".

**La bonne nouvelle** :
- ✅ La corruption est **rare** avec du matériel de qualité  
- ✅ PostgreSQL 18 active les **checksums par défaut** pour détecter la corruption  
- ✅ Des stratégies de prévention et récupération existent  
- ✅ La majorité des corruptions sont détectables avant de causer des dégâts

Ce chapitre va vous expliquer ce qu'est la corruption, comment la détecter, et surtout comment vous en **protéger**.

---

## Qu'est-ce que la corruption de données ?

### Définition simple

La corruption de données se produit quand le **contenu physique** d'une base de données ne correspond plus à ce qui a été écrit initialement. Les bits stockés sur le disque sont modifiés, perdus ou mélangés.

### Types de corruption

#### 1. Corruption matérielle (la plus fréquente)

Causée par des défaillances physiques :

- **Disque dur défaillant** : Secteurs défectueux qui retournent des données erronées  
- **RAM défectueuse** : Erreurs lors du traitement en mémoire (bit flips)  
- **Câbles SATA/SAS défectueux** : Données corrompues pendant le transfert  
- **Contrôleur RAID bugué** : Erreurs dans la logique de redondance  
- **Coupure de courant** : Écriture partielle sur disque

**Analogie** : C'est comme un stylo qui bave sur un document important, rendant certains mots illisibles.

#### 2. Corruption logicielle (rare)

Causée par des bugs logiciels :

- **Bug dans PostgreSQL** (extrêmement rare, mais possible)  
- **Bug dans le système de fichiers** (ext4, XFS, ZFS)  
- **Bug dans les drivers de stockage**  
- **Crash pendant une écriture critique**

#### 3. Corruption induite par l'opérateur (humaine)

- **Arrêt brutal du serveur** (`kill -9` sur postgres)  
- **Modification manuelle des fichiers de données** (JAMAIS faire ça !)  
- **Clonage de disque pendant que PostgreSQL tourne**  
- **Restauration partielle ou incorrecte de sauvegarde**

## Les symptômes de la corruption

### Symptômes immédiats (erreurs)

```sql
-- Exemples d'erreurs typiques de corruption

ERROR:  invalid page in block 12345 of relation base/16384/16385  
ERROR:  could not read block 67890 in file "base/16384/16385": read only 0 of 8192 bytes  
ERROR:  invalid page header in block 456 of relation "users"  
ERROR:  checksum failure on block 789 in file "base/16384/16385"  
WARNING:  page verification failed, calculated checksum 12345 but expected 67890  
PANIC:  corrupted page pointers  
```

### Symptômes silencieux (plus dangereux)

- **Données incohérentes** : Totaux qui ne correspondent pas  
- **Lignes qui disparaissent** : SELECT retourne moins de résultats qu'attendu  
- **Valeurs bizarres** : Nombres négatifs impossibles, dates dans le futur  
- **Crashes aléatoires** : PostgreSQL redémarre sans raison apparente

### Vérifier les logs PostgreSQL

```bash
# Rechercher des erreurs de corruption dans les logs
sudo tail -f /var/log/postgresql/postgresql-*.log | grep -i "invalid\|corrupt\|checksum\|verification failed"

# Ou consulter les logs complets
sudo grep -i "invalid page\|corrupt\|checksum" /var/log/postgresql/postgresql-*.log
```

## Les Checksums : Votre première ligne de défense

### Qu'est-ce qu'un checksum ?

Un **checksum** (ou somme de contrôle) est comme une **empreinte digitale** pour chaque bloc de données. PostgreSQL calcule cette empreinte quand il écrit des données, et la vérifie quand il les relit.

**Analogie** : Imaginez que vous mettez un cadenas numérique sur chaque page d'un livre. Si quelqu'un modifie la page, le code du cadenas ne correspondra plus, et vous saurez immédiatement que la page a été altérée.

### Comment fonctionne un checksum ?

```
1. ÉCRITURE de données
   ├─ PostgreSQL écrit un bloc de 8192 bytes
   ├─ Calcule un checksum (hash) de ce bloc
   └─ Stocke le checksum dans l'en-tête du bloc

2. LECTURE de données
   ├─ PostgreSQL lit le bloc depuis le disque
   ├─ Recalcule le checksum à partir des données lues
   ├─ Compare avec le checksum stocké
   └─ Si différent → CORRUPTION DÉTECTÉE !
```

### Exemple visuel

```
┌─────────────────────────────────────┐
│  BLOC DE DONNÉES (8192 bytes)       │
├─────────────────────────────────────┤
│ En-tête (header)                    │
│ ├─ Checksum: 0xABCD1234 ✅           │
│ └─ Métadonnées diverses             │
├─────────────────────────────────────┤
│ Données réelles (lignes de table)   │
│ Row 1: id=1, name="Alice"           │
│ Row 2: id=2, name="Bob"             │
│ ...                                 │
└─────────────────────────────────────┘

À la lecture :
1. Lire le bloc entier
2. Recalculer checksum → 0xABCD1234
3. Comparer avec stocké (0xABCD1234)
4. ✅ Match → Données intègres

Si corruption :
1. Lire le bloc (données corrompues)
2. Recalculer checksum → 0x9876FEDC
3. Comparer avec stocké (0xABCD1234)
4. ❌ Mismatch → CORRUPTION DÉTECTÉE !
```

## Nouveauté PostgreSQL 18 : Checksums activés par défaut

### Historique

- **PostgreSQL ≤ 17** : Checksums **désactivés** par défaut (pour compatibilité et légère performance)  
- **PostgreSQL 18** : Checksums **activés** par défaut ! 🎉

### Pourquoi ce changement ?

1. **Matériel moderne** : Le coût CPU des checksums est négligeable sur processeurs récents  
2. **Protection essentielle** : Détection précoce vaut mieux que perte de données  
3. **Meilleures pratiques** : La majorité des DBAs les activaient manuellement

### Impact sur les performances

**Overhead typique** :
- Écriture : +1-3% de CPU
- Lecture : +0.5-2% de CPU

Sur matériel moderne (2020+), cet overhead est **imperceptible** et largement compensé par la sécurité apportée.

### Vérifier si les checksums sont activés

```sql
-- Méthode 1 : Via SQL
SHOW data_checksums;
-- Résultat : on (PG 18) ou off (versions antérieures)

-- Méthode 2 : Via pg_controldata
pg_controldata /var/lib/postgresql/data | grep checksum
-- Résultat : Data page checksum version:           1
```

### Activer les checksums sur une instance existante

Si vous migrez depuis une version antérieure où checksums était désactivé :

```bash
# ⚠️ ATTENTION : Nécessite arrêt de la base et peut prendre des heures !

# 1. Arrêter PostgreSQL
sudo systemctl stop postgresql

# 2. Activer les checksums (scan complet de la base)
sudo -u postgres pg_checksums --enable -D /var/lib/postgresql/data

# Exemple de sortie :
# Checksum operation completed
# Files scanned:  1234567
# Blocks scanned: 98765432
# pg_checksums: syncing data directory
# pg_checksums: checksums enabled in cluster

# 3. Redémarrer PostgreSQL
sudo systemctl start postgresql
```

**Durée estimée** :
- 100 GB : ~10-30 minutes (SSD)
- 1 TB : ~1-3 heures (SSD)
- 10 TB : ~10-30 heures (SSD)

**Planifier pendant une fenêtre de maintenance !**

### Désactiver les checksums (non recommandé)

```bash
# Lors de l'initialisation d'une nouvelle instance
initdb --no-data-checksums -D /var/lib/postgresql/data

# Ou pour désactiver sur instance existante
sudo systemctl stop postgresql  
sudo -u postgres pg_checksums --disable -D /var/lib/postgresql/data  
sudo systemctl start postgresql  
```

⚠️ **À éviter** : Sauf raison très spécifique (hardware extrêmement limité), gardez les checksums activés.

## Détecter la corruption

### Méthode 1 : Attendre une erreur (réactif)

La méthode la plus simple mais la moins souhaitable : attendre qu'une requête échoue.

```sql
-- Si corruption, vous verrez :
SELECT * FROM users WHERE id = 12345;

ERROR:  invalid page in block 789 of relation base/16384/16385  
HINT:  This error can be caused by corrupt data or hardware failure.  
```

### Méthode 2 : Vérification proactive avec pg_verifybackup

```bash
# Vérifier une sauvegarde avec pg_basebackup
pg_verifybackup /path/to/backup

# Résultat si OK :
# backup successfully verified
# 0 files verified, 0 checksums verified

# Si corruption détectée :
# error: checksum mismatch in file "base/16384/16385"
```

### Méthode 3 : Extension amcheck (vérification d'index)

```sql
-- Installer l'extension
CREATE EXTENSION amcheck;

-- Vérifier l'intégrité d'un index
SELECT bt_index_check('users_pkey');
-- Résultat si OK : (pas de résultat, c'est bon)

-- Vérification plus approfondie (plus lente)
SELECT bt_index_parent_check('users_pkey');

-- Vérifier tous les index d'une table
SELECT
    indexrelid::regclass AS index_name,
    bt_index_check(indexrelid)
FROM pg_index  
WHERE indrelid = 'users'::regclass;  
```

Si corruption détectée :
```
ERROR:  heap tuple (0,52) from table "users" lacks matching index tuple within index "users_pkey"
```

### Méthode 4 : Extension pg_verify_checksums (versions < 18)

Pour PostgreSQL < 18 sans checksums natifs :

```bash
# Installer l'extension
CREATE EXTENSION pg_verify_checksums;

# Vérifier toute la base
SELECT pg_verify_checksums();
```

### Méthode 5 : Vérification manuelle des fichiers

```bash
# Utiliser l'outil pg_checksums en lecture seule
sudo -u postgres pg_checksums --check -D /var/lib/postgresql/data

# Résultat si OK :
# Checksum operation completed
# Files scanned:  12345
# Blocks scanned: 9876543
# Bad checksums:  0

# Si corruption :
# pg_checksums: error: checksum verification failed in file "base/16384/16385", block 789
# Blocks scanned:  9876543
# Bad checksums:  3
```

## Diagnostic : Identifier l'étendue de la corruption

### Étape 1 : Identifier les fichiers corrompus

```sql
-- Noter le numéro de fichier dans l'erreur
-- ERROR:  checksum failure on block 789 in file "base/16384/16385"
--                                                    ^^^^^^^^^^^^^^

-- Identifier la table correspondante
SELECT
    c.relname AS table_name,
    n.nspname AS schema_name,
    c.relfilenode,
    pg_relation_filepath(c.oid) AS file_path
FROM pg_class c  
JOIN pg_namespace n ON n.oid = c.relnamespace  
WHERE c.relfilenode = 16385;  -- Numéro du fichier  
```

### Étape 2 : Tenter de lire autour de la corruption

```sql
-- Si la table est connue, essayer de lire les données valides
-- en évitant les blocs corrompus

-- Utiliser LIMIT pour éviter de scanner toute la table
SELECT * FROM users LIMIT 100;

-- Si ça échoue, essayer par portions
SELECT * FROM users WHERE id < 1000;  
SELECT * FROM users WHERE id >= 1000 AND id < 2000;  
-- Continue jusqu'à trouver le bloc problématique
```

### Étape 3 : Utiliser l'extension pageinspect

```sql
-- Installer l'extension
CREATE EXTENSION pageinspect;

-- Examiner un bloc spécifique
SELECT * FROM page_header(get_raw_page('users', 0));

-- Résultat normal :
--  lsn        | checksum | flags | lower | upper | special | pagesize | version | prune_xid
-- ------------+----------+-------+-------+-------+---------+----------+---------+-----------
--  0/12345678 |    12345 |     0 |    48 |  8064 |    8192 |     8192 |       4 |         0

-- Si corruption, get_raw_page échouera :
-- ERROR:  invalid page in block 789
```

### Étape 4 : Créer un rapport de corruption

```sql
-- Script pour documenter la corruption
DO $$  
DECLARE  
    corrupted_table TEXT;
    corrupted_block INTEGER;
BEGIN
    -- Capturer les informations
    corrupted_table := 'users';
    corrupted_block := 789;

    RAISE NOTICE 'CORRUPTION REPORT';
    RAISE NOTICE '==================';
    RAISE NOTICE 'Date: %', now();
    RAISE NOTICE 'Table: %', corrupted_table;
    RAISE NOTICE 'Block: %', corrupted_block;
    RAISE NOTICE 'File: %', pg_relation_filepath(corrupted_table::regclass);
END $$;
```

## Récupération : Que faire face à une corruption ?

### Stratégie de récupération par niveau de gravité

```
┌─────────────────────────────────────────────────────┐
│  NIVEAU 1 : Corruption mineure (quelques blocs)     │
│  → Restaurer depuis sauvegarde                      │
│  → Ou reconstruire les index                        │
└─────────────────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────────────┐
│  NIVEAU 2 : Corruption modérée (table entière)      │
│  → Restaurer la table depuis sauvegarde             │
│  → Ou reconstruire depuis réplica                   │
└─────────────────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────────────┐
│  NIVEAU 3 : Corruption majeure (plusieurs tables)   │
│  → Restauration complète depuis sauvegarde          │
│  → Point-In-Time Recovery (PITR)                    │
└─────────────────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────────────┐
│  NIVEAU 4 : Corruption catastrophique               │
│  → Dernière chance : zero_damaged_pages             │
│  → Accepter la perte de données                     │
└─────────────────────────────────────────────────────┘
```

### Niveau 1 : Corruption d'index

Si seuls les index sont corrompus (le plus courant et le moins grave) :

```sql
-- 1. Identifier l'index corrompu
SELECT indexrelid::regclass, indrelid::regclass  
FROM pg_index  
WHERE indexrelid = 'users_pkey'::regclass;  

-- 2. Supprimer l'index corrompu
DROP INDEX CONCURRENTLY users_pkey;

-- 3. Recréer l'index
CREATE UNIQUE INDEX CONCURRENTLY users_pkey ON users(id);

-- 4. Réaffecter comme clé primaire si nécessaire
ALTER TABLE users ADD PRIMARY KEY USING INDEX users_pkey;
```

**Avantage** : Pas de perte de données, opération possible en production avec CONCURRENTLY.

### Niveau 2 : Corruption de table - Option A (avec sauvegarde)

```bash
# 1. Identifier la table corrompue
# ERROR in table "users"

# 2. Créer une nouvelle table temporaire
psql -d mydb -c "CREATE TABLE users_backup (LIKE users INCLUDING ALL);"

# 3. Restaurer depuis pg_dump (sauvegarde logique)
pg_restore -d mydb -t users --data-only backup.dump -a users_backup

# 4. Vérifier les données restaurées
psql -d mydb -c "SELECT count(*) FROM users_backup;"

# 5. Remplacer la table (ATTENTION : Perte des données depuis la sauvegarde)
psql -d mydb <<EOF  
BEGIN;  
DROP TABLE users CASCADE;  
ALTER TABLE users_backup RENAME TO users;  
COMMIT;  
EOF  

# 6. Recréer les index, contraintes, etc.
```

### Niveau 2 : Corruption de table - Option B (extraction partielle)

Si vous devez sauver ce qui peut l'être :

```sql
-- 1. Créer une nouvelle table
CREATE TABLE users_recovered (LIKE users);

-- 2. Copier les lignes lisibles (ignorer les blocs corrompus)
-- Utiliser zero_damaged_pages (DANGER : Perte de données)

-- Activer temporairement (à vos risques !)
SET zero_damaged_pages = on;

-- Copier les données valides
INSERT INTO users_recovered  
SELECT * FROM users;  -- Ignore silencieusement les blocs corrompus  

-- Désactiver immédiatement
SET zero_damaged_pages = off;

-- 3. Vérifier ce qui a été sauvé
SELECT count(*) FROM users;          -- Avant (avec corruption)  
SELECT count(*) FROM users_recovered; -- Après (données valides)  

-- 4. Remplacer la table
BEGIN;  
DROP TABLE users CASCADE;  
ALTER TABLE users_recovered RENAME TO users;  
COMMIT;  
```

⚠️ **DANGER** : `zero_damaged_pages` **supprime silencieusement les données corrompues**. À utiliser uniquement en dernier recours.

### Niveau 3 : Restauration complète depuis sauvegarde

#### Option A : Sauvegarde logique (pg_dump)

```bash
# 1. Arrêter l'application (éviter nouvelles transactions)
# ...

# 2. Créer une nouvelle base
createdb mydb_restored

# 3. Restaurer depuis pg_dump
pg_restore -d mydb_restored -v backup.dump

# Ou si format SQL :
psql -d mydb_restored -f backup.sql

# 4. Vérifier l'intégrité
psql -d mydb_restored -c "SELECT count(*) FROM users;"

# 5. Basculer l'application vers la nouvelle base
# Modifier la configuration de connexion
```

#### Option B : Sauvegarde physique (pg_basebackup + PITR)

```bash
# 1. Arrêter PostgreSQL
sudo systemctl stop postgresql

# 2. Sauvegarder l'instance corrompue (au cas où)
sudo mv /var/lib/postgresql/data /var/lib/postgresql/data.corrupted

# 3. Restaurer depuis pg_basebackup
sudo -u postgres cp -a /path/to/backup/base /var/lib/postgresql/data

# 4. Configurer la récupération (recovery.signal)
sudo -u postgres touch /var/lib/postgresql/data/recovery.signal

# Éditer postgresql.conf
restore_command = 'cp /path/to/wal_archive/%f %p'  
recovery_target_time = '2024-11-23 10:30:00'  # Point avant corruption  

# 5. Démarrer PostgreSQL (mode recovery)
sudo systemctl start postgresql

# 6. Vérifier dans les logs que recovery s'est bien passé
sudo tail -f /var/log/postgresql/postgresql-*.log

# 7. Promouvoir si OK
psql -c "SELECT pg_wal_replay_resume();"
```

### Niveau 4 : Dernier recours - Accepter la perte partielle

Si aucune sauvegarde n'existe (😱) :

```sql
-- Configuration globale (TRÈS DANGEREUX)
-- Dans postgresql.conf
zero_damaged_pages = on  
ignore_checksum_failure = on  -- Ignore les erreurs de checksum  

-- Redémarrer PostgreSQL
-- sudo systemctl restart postgresql

-- Extraire ce qui peut l'être
CREATE TABLE rescued_data AS  
SELECT * FROM corrupted_table;  -- Ignore les blocs illisibles  

-- Restaurer les paramètres normaux immédiatement
-- zero_damaged_pages = off
-- ignore_checksum_failure = off
```

⚠️ **NE JAMAIS laisser ces paramètres activés en production !**

## Prévention : Éviter la corruption

### 1. Hardware de qualité

#### Disques

- ✅ **SSD d'entreprise** avec protection contre les coupures de courant (PLP - Power Loss Protection)  
- ✅ **RAID 10 ou RAID 6** pour redondance  
- ❌ **Éviter** : Disques grand public, USB, NFS non redondant

#### RAM

- ✅ **RAM ECC** (Error Correcting Code) : Détecte et corrige les erreurs mémoire  
- ❌ **Éviter** : RAM standard pour serveurs de production

```bash
# Vérifier si vous avez RAM ECC
sudo dmidecode -t memory | grep -i ecc

# Résultat souhaité :
# Error Correction Type: Multi-bit ECC
```

#### Alimentation

- ✅ **UPS (onduleur)** : Protection contre coupures de courant  
- ✅ **Double alimentation** : Redondance alimentation

### 2. Système de fichiers approprié

```bash
# XFS ou ext4 sont recommandés
# Vérifier le système de fichiers actuel
df -T /var/lib/postgresql/data

# Résultat exemple :
# Filesystem     Type  Size  Used Avail Use% Mounted on
# /dev/sda1      ext4  100G   45G   50G  48% /var/lib/postgresql/data
```

**Recommandations** :
- ✅ **ext4** : Solide, éprouvé, bonne performance  
- ✅ **XFS** : Excellent pour très grands volumes  
- ✅ **ZFS** : Protection intégrée contre corruption (checksums natifs)  
- ⚠️ **Btrfs** : Prometteur mais moins mature  
- ❌ **NFS** : Éviter pour data, OK pour archives

### 3. Configuration PostgreSQL anti-corruption

```sql
-- Dans postgresql.conf

# Activer checksums (défaut PG 18)
data_checksums = on

# Synchronisation stricte (évite corruption en cas de crash)
fsync = on                          # JAMAIS désactiver  
synchronous_commit = on             # Pour transactions critiques  
full_page_writes = on               # Protection après checkpoint  
wal_sync_method = fdatasync         # Méthode la plus sûre (Linux)  

# Journalisation
wal_level = replica                 # Minimum pour PITR  
archive_mode = on                   # Archiver les WAL  
archive_command = 'cp %p /path/to/wal_archive/%f'  

# Logging des corruptions
log_checkpoints = on  
log_autovacuum_min_duration = 0  
```

### 4. Sauvegardes régulières et testées

```bash
# Script de sauvegarde quotidienne
#!/bin/bash
# backup_daily.sh

DATE=$(date +%Y%m%d)  
BACKUP_DIR="/backup/postgresql"  
PGDATABASE="mydb"  

# Sauvegarde logique
pg_dump -Fc $PGDATABASE > $BACKUP_DIR/mydb_$DATE.dump

# Sauvegarde physique (base backup)
pg_basebackup -D $BACKUP_DIR/basebackup_$DATE -Ft -z -P

# Vérifier l'intégrité de la sauvegarde
pg_verifybackup $BACKUP_DIR/basebackup_$DATE

# Rotation (garder 7 jours)
find $BACKUP_DIR -name "*.dump" -mtime +7 -delete  
find $BACKUP_DIR -name "basebackup_*" -mtime +7 -exec rm -rf {} \;  

echo "Backup completed: $DATE"
```

**Planifier avec cron** :

```bash
# Tous les jours à 2h du matin
0 2 * * * /usr/local/bin/backup_daily.sh >> /var/log/postgresql_backup.log 2>&1
```

**CRITIQUE** : Testez vos sauvegardes régulièrement !

```bash
# Test de restauration mensuel
#!/bin/bash
# test_restore.sh

BACKUP="/backup/postgresql/mydb_latest.dump"  
TESTDB="mydb_test"  

# Créer DB de test
dropdb --if-exists $TESTDB  
createdb $TESTDB  

# Restaurer
pg_restore -d $TESTDB $BACKUP

# Vérifier
psql -d $TESTDB -c "SELECT count(*) FROM users;"

# Nettoyer
dropdb $TESTDB

echo "Restore test successful!"
```

### 5. Monitoring de l'intégrité

#### Surveillance automatisée avec pg_cron

```sql
-- Installer pg_cron
CREATE EXTENSION pg_cron;

-- Vérifier les index chaque dimanche à 3h
SELECT cron.schedule(
    'check-indexes',
    '0 3 * * 0',
    $$SELECT bt_index_check(indexrelid::regclass)
      FROM pg_index
      WHERE indrelid IN (
        SELECT oid FROM pg_class
        WHERE relname IN ('users', 'orders', 'products')
      )$$
);

-- Notifier si problème
CREATE OR REPLACE FUNCTION check_critical_indexes()  
RETURNS void AS $$  
DECLARE  
    v_error TEXT;
BEGIN
    -- Vérifier chaque index critique
    PERFORM bt_index_check('users_pkey');
    PERFORM bt_index_check('orders_pkey');
EXCEPTION WHEN OTHERS THEN
    -- Logger l'erreur
    INSERT INTO corruption_alerts (detected_at, error_message)
    VALUES (now(), SQLERRM);

    -- Notifier (via pg_notify ou autre)
    PERFORM pg_notify('corruption_alert', SQLERRM);
END;
$$ LANGUAGE plpgsql;
```

#### Script de surveillance externe

```bash
#!/bin/bash
# check_corruption.sh

LOGFILE="/var/log/corruption_check.log"  
DATE=$(date +"%Y-%m-%d %H:%M:%S")  

# Vérifier les checksums
RESULT=$(sudo -u postgres pg_checksums --check -D /var/lib/postgresql/data 2>&1)

if echo "$RESULT" | grep -q "Bad checksums: 0"; then
    echo "[$DATE] OK - No corruption detected" >> $LOGFILE
    exit 0
else
    echo "[$DATE] ERROR - Corruption detected!" >> $LOGFILE
    echo "$RESULT" >> $LOGFILE

    # Alerter (email, Slack, PagerDuty, etc.)
    echo "CORRUPTION DETECTED: $RESULT" | mail -s "PostgreSQL Corruption Alert" admin@example.com

    exit 1
fi
```

**Planifier avec cron** :

```bash
# Tous les jours à 4h
0 4 * * * /usr/local/bin/check_corruption.sh
```

## Cas d'étude : Récupération réelle

### Scénario : Corruption détectée en production

**Contexte** :
- Production, 500 utilisateurs simultanés
- Erreur détectée : `checksum failure on block 789 in file base/16384/16385`
- Table identifiée : `orders` (critique)

**Plan d'action** :

#### Phase 1 : Évaluation (5 minutes)

```sql
-- 1. Identifier la table
SELECT relname, relfilenode  
FROM pg_class  
WHERE relfilenode = 16385;  
-- Résultat : orders

-- 2. Évaluer l'impact
SELECT count(*) FROM orders;
-- Si erreur → corruption majeure
-- Si réussi → corruption mineure (quelques blocs)

-- 3. Vérifier l'âge de la dernière sauvegarde
SELECT pg_size_pretty(pg_database_size('mydb')),
       now() - (SELECT modification FROM pg_stat_file('base/16384/16385'));
```

#### Phase 2 : Isolement (10 minutes)

```sql
-- 1. Basculer application en mode lecture seule
-- (dans l'application, bloquer les INSERT/UPDATE/DELETE sur orders)

-- 2. Créer une table temporaire avec les données valides
CREATE TABLE orders_safe AS  
SELECT * FROM orders WHERE order_id < 10000;  -- Avant le bloc corrompu  

-- 3. Vérifier les données sauvées
SELECT count(*) FROM orders_safe;
```

#### Phase 3 : Récupération (30 minutes)

```sql
-- Option A : Restauration depuis sauvegarde
-- 1. Restaurer dans nouvelle table
CREATE TABLE orders_restored (LIKE orders INCLUDING ALL);

-- 2. Importer depuis backup (commande externe)
-- pg_restore -d mydb -t orders --data-only backup.dump -a orders_restored

-- 3. Comparer
SELECT count(*) FROM orders_safe;     -- Données depuis production  
SELECT count(*) FROM orders_restored; -- Données depuis backup  

-- 4. Merger les deux sources
BEGIN;
-- Supprimer la table corrompue
DROP TABLE orders CASCADE;
-- Restaurer depuis backup
ALTER TABLE orders_restored RENAME TO orders;
-- Ajouter les commandes récentes depuis orders_safe si nécessaire
COMMIT;
```

#### Phase 4 : Vérification (15 minutes)

```sql
-- 1. Vérifier l'intégrité de la nouvelle table
SELECT bt_index_check('orders_pkey');

-- 2. Vérifier la cohérence des données
SELECT count(*), min(order_date), max(order_date)  
FROM orders;  

-- 3. Tester quelques requêtes critiques
SELECT * FROM orders WHERE user_id = 12345 LIMIT 10;  
SELECT sum(amount) FROM orders WHERE order_date = current_date;  
```

#### Phase 5 : Reprise (5 minutes)

```sql
-- 1. Remettre application en mode lecture-écriture
-- (configuration applicative)

-- 2. Surveiller les logs
-- sudo tail -f /var/log/postgresql/postgresql-*.log

-- 3. Monitorer les performances
SELECT * FROM pg_stat_activity WHERE state = 'active';
```

#### Phase 6 : Post-mortem (dès que possible)

```bash

# 2. Consulter les logs système

# - Symptômes
# - Actions prises
# - Durée d'interruption

# - Cause identifiée
# - Actions préventives

# - Remplacer matériel défaillant
# - Améliorer monitoring

```

## Mythes et idées reçues

### ❌ Mythe 1 : "Les checksums ralentissent trop la base"

**Réalité** : Impact négligeable sur matériel moderne (+1-3% CPU). Le coût d'une corruption non détectée est infiniment plus élevé.

### ❌ Mythe 2 : "Si j'ai du RAID, je n'ai pas besoin de checksums"

**Réalité** : RAID protège contre défaillance matérielle, pas contre corruption silencieuse (silent data corruption). Les deux sont complémentaires.

### ❌ Mythe 3 : "La corruption est extrêmement rare"

**Réalité** : Une étude de Google (2016) a montré que 2-4% des disques subissent au moins une corruption silencieuse par an. Sur un datacenter de 10,000 disques, c'est 200-400 corruptions/an !

### ❌ Mythe 4 : "Les sauvegardes suffisent"

**Réalité** : Les checksums détectent **immédiatement** la corruption. Les sauvegardes vous sauvent **après coup**, mais vous perdez potentiellement des heures/jours de données.

### ❌ Mythe 5 : "fsync = off c'est OK pour plus de performance"

**Réalité** : **JAMAIS**. `fsync = off` est une recette pour la corruption en cas de crash. Les gains de performance ne valent jamais la perte de données.

## Résumé des points clés

✅ **Corruption** : Modification involontaire des données sur disque, potentiellement catastrophique

✅ **Checksums** : Empreintes digitales pour chaque bloc, détectent corruption immédiatement

✅ **PostgreSQL 18** : Checksums activés par défaut (enfin !)

✅ **RAM ECC** : Protection contre erreurs mémoire (bit flips)

✅ **Matériel de qualité** : SSD entreprise, RAID, UPS indispensables en production

✅ **Configuration** : `fsync = on`, `full_page_writes = on`, `synchronous_commit = on`

✅ **Sauvegardes** : Régulières (quotidiennes), testées (mensuelles), vérifiées (pg_verifybackup)

✅ **Monitoring** : Checksums automatiques, SMART disks, logs PostgreSQL

✅ **Récupération** : Index → Restauration partielle → Restauration complète → Dernier recours (zero_damaged_pages)

✅ **Prévention** : 95% du problème. Investir dans hardware et monitoring

## Conclusion

La corruption de données est le scénario le plus effrayant pour tout administrateur de base de données. Contrairement aux autres problèmes qui peuvent ralentir votre système, la corruption peut entraîner une **perte définitive de données**.

**La bonne approche** :

1. **Prévention** : Hardware de qualité, checksums activés, configuration appropriée  
2. **Détection** : Monitoring automatisé, vérifications régulières  
3. **Réaction** : Sauvegardes testées, procédures de récupération documentées

Avec PostgreSQL 18 activant les checksums par défaut, la communauté PostgreSQL fait un pas majeur vers la protection de vos données. Mais les checksums ne sont qu'une partie de la solution : matériel de qualité, sauvegardes régulières et tests de restauration sont tout aussi critiques.

**Rappelez-vous** :
- Checksums activés = détection précoce ✅
- Sauvegardes régulières = filet de sécurité ✅
- Tests de restauration = assurance que le filet fonctionne ✅
- Hardware de qualité = réduction drastique du risque ✅

La corruption est rare avec des pratiques appropriées, mais quand elle survient, être préparé fait toute la différence entre une interruption mineure et une catastrophe.
