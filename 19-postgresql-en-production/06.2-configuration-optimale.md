# 19.6.2. Configuration Optimale

## Introduction

PostgreSQL est livré avec des **paramètres par défaut très conservateurs**, conçus pour fonctionner sur des machines modestes. En production, ces valeurs sont souvent **inadaptées** et peuvent limiter considérablement les performances.

La configuration optimale de PostgreSQL consiste à ajuster les paramètres en fonction de :
- Vos **ressources matérielles** (RAM, CPU, disques)
- Votre **type de charge** (OLTP, OLAP, mixte)
- Vos **besoins applicatifs** (latence, débit, concurrence)

Cette section vous guide à travers les paramètres essentiels et les stratégies de tuning pour maximiser les performances de votre base de données.

> **Avertissement** : La configuration optimale n'est **pas universelle**. Les valeurs proposées ici sont des **points de départ** à ajuster selon votre contexte spécifique. Testez toujours vos modifications en environnement de staging avant la production.

## Comprendre le Fichier de Configuration

### Localisation de postgresql.conf

Le fichier principal de configuration est `postgresql.conf` :

```bash
# Trouver le fichier de configuration
sudo -u postgres psql -c "SHOW config_file;"
# Résultat typique : /var/lib/pgsql/18/data/postgresql.conf

# Ou via psql
psql -U postgres -c "SHOW config_file;"
```

### Recharger la Configuration

Après modification, rechargez la configuration :

```bash
# Méthode 1 : Via pg_ctl
sudo -u postgres pg_ctl reload -D /var/lib/pgsql/18/data

# Méthode 2 : Via SQL
psql -U postgres -c "SELECT pg_reload_conf();"

# Méthode 3 : Via systemd
sudo systemctl reload postgresql-18
```

> **Note** : Certains paramètres nécessitent un **redémarrage complet** (marqués par `# restart required`).

### Vérifier les Paramètres Actifs

```sql
-- Voir un paramètre spécifique
SHOW shared_buffers;

-- Voir tous les paramètres modifiés par rapport aux défauts
SELECT name, setting, unit, source  
FROM pg_settings  
WHERE source != 'default'  
ORDER BY name;  

-- Voir les paramètres nécessitant un restart
SELECT name, setting, pending_restart  
FROM pg_settings  
WHERE pending_restart = true;  
```

## 1. Paramètres Mémoire Critiques

### 1.1. shared_buffers : Le Cache Central

**Rôle** : Mémoire partagée pour mettre en cache les pages de données. C'est le **paramètre le plus important**.

**Valeur par défaut** : 128 MB (beaucoup trop faible)

**Recommandation générale** :
- **25% de la RAM** pour serveurs dédiés
- Maximum efficace : **8-16 GB** (au-delà, le bénéfice diminue)

**Exemples** :

```conf
# Serveur avec 16 GB de RAM
shared_buffers = 4GB

# Serveur avec 64 GB de RAM
shared_buffers = 16GB

# Serveur avec 128 GB de RAM
shared_buffers = 16GB  # Pas plus, le reste pour le cache OS
```

**Pourquoi ne pas mettre 50% ou plus ?**
- PostgreSQL s'appuie aussi sur le **cache du système d'exploitation** (page cache)
- Un équilibre optimal : PostgreSQL cache + OS cache = meilleures performances

**Nécessite un restart** : ✅ Oui

### 1.2. effective_cache_size : Indicateur pour le Planificateur

**Rôle** : Informe le planificateur de la quantité de mémoire disponible pour le cache (PostgreSQL + OS).

**Valeur par défaut** : 4 GB

**Recommandation** :
- **50-75% de la RAM totale** (sur serveur dédié)

**Exemples** :

```conf
# Serveur avec 16 GB de RAM
effective_cache_size = 12GB

# Serveur avec 64 GB de RAM
effective_cache_size = 48GB

# Serveur avec 128 GB de RAM
effective_cache_size = 96GB
```

**Impact** : N'alloue PAS de mémoire, juste un **hint** pour le planificateur. Peut influencer le choix entre scan séquentiel et index scan.

**Nécessite un restart** : ❌ Non (reload suffit)

### 1.3. work_mem : Mémoire par Opération de Tri/Hash

**Rôle** : Mémoire allouée à **chaque opération** de tri, hash join, agrégation, etc.

**Valeur par défaut** : 4 MB (souvent insuffisant)

**Piège** : Cette mémoire est allouée **par opération ET par connexion**. Une requête complexe peut utiliser `work_mem × nombre d'opérations × nombre de connexions`.

**Formule de dimensionnement** :

```
work_mem = (RAM totale × 0.25) / max_connections / 10
```

**Exemples** :

```conf
# Serveur 16 GB, 100 connexions max
# (16 GB × 0.25) / 100 / 10 = 4 MB → Augmenter
work_mem = 32MB

# Serveur 64 GB, 50 connexions (OLAP)
work_mem = 128MB

# Serveur 16 GB, 500 connexions (OLTP haute concurrence)
work_mem = 8MB
```

**Stratégie avancée** : Configurer au niveau session pour les requêtes analytiques :

```sql
-- Avant une requête lourde
SET work_mem = '256MB';  
SELECT ... FROM ... JOIN ... GROUP BY ...;  
RESET work_mem;  
```

**Symptôme de work_mem insuffisant** : Logs indiquant `temporary file` créés sur disque.

```
LOG:  temporary file: path "base/pgsql_tmp/pgsql_tmp12345.0", size 52428800
```

**Nécessite un restart** : ❌ Non (reload suffit)

### 1.4. maintenance_work_mem : Mémoire pour Maintenance

**Rôle** : Mémoire pour opérations de maintenance (VACUUM, CREATE INDEX, ALTER TABLE, REINDEX).

**Valeur par défaut** : 64 MB

**Recommandation** :
- **5-10% de la RAM**, avec un maximum de **2 GB**
- Pour bases volumineuses : Jusqu'à **8-16 GB** peut être bénéfique

**Exemples** :

```conf
# Serveur 16 GB
maintenance_work_mem = 1GB

# Serveur 64 GB
maintenance_work_mem = 4GB

# Serveur 128 GB
maintenance_work_mem = 8GB
```

**Impact** : Accélère les créations d'index, VACUUM, et réorganisations de tables.

**Nécessite un restart** : ❌ Non (reload suffit)

| 16 GB       | 4 GB           | 12 GB                | 16-32 MB        | 64-128 MB       | 1 GB                 |

| 128 GB      | 16 GB          | 96 GB                | 32-64 MB        | 256-512 MB      | 8 GB                 |

## 2. Configuration des Connexions

### 2.1. max_connections : Nombre Maximum de Connexions

**Rôle** : Limite le nombre de connexions simultanées.

**Valeur par défaut** : 100

**Réflexion importante** : PostgreSQL utilise un modèle **process-per-connection**. Chaque connexion consomme :
- Environ **10 MB de RAM** minimum
- Un processus système

**Piège courant** : Augmenter `max_connections` sans Connection Pooling = catastrophe.

**Recommandations** :

```conf
# OLTP avec Connection Pooler (PgBouncer)
max_connections = 100-200

# OLTP sans Connection Pooler (petit trafic)
max_connections = 50-100

# OLAP / Data Warehouse
max_connections = 20-50

# ❌ À éviter absolument
max_connections = 1000  # Surcharge le serveur
```

**Solution** : Utiliser un **Connection Pooler** (PgBouncer, PgPool-II) pour multiplexer les connexions.

**Architecture recommandée** :

```
Application (1000 connexions logiques)
         ↓
    PgBouncer (pool de 100)
         ↓
    PostgreSQL (max_connections = 100)
```

**Nécessite un restart** : ✅ Oui

### 2.2. superuser_reserved_connections

**Rôle** : Réserve des connexions pour les superutilisateurs (dépannage).

**Valeur par défaut** : 3

**Recommandation** :

```conf
# Garder quelques slots pour l'admin
superuser_reserved_connections = 5
```

**Scénario** : Si `max_connections` est saturé, vous pouvez toujours vous connecter en superutilisateur pour diagnostiquer.

**Nécessite un restart** : ✅ Oui

## 3. Configuration WAL (Write-Ahead Log)

Le WAL est essentiel pour :
- **Durabilité** (ACID)  
- **Réplication**  
- **Point-In-Time Recovery (PITR)**

### 3.1. wal_level : Niveau de Détail du WAL

**Rôle** : Détermine la quantité d'informations écrites dans le WAL.

**Valeurs possibles** :
- `minimal` : Minimum (pas de réplication)  
- `replica` : Réplication physique (défaut)  
- `logical` : Réplication logique

**Recommandation production** :

```conf
# Pour réplication physique ou logique
wal_level = replica

# Si réplication logique nécessaire
wal_level = logical
```

**Nécessite un restart** : ✅ Oui

### 3.2. max_wal_size : Taille Maximum du WAL

**Rôle** : Déclenche un checkpoint quand le WAL atteint cette taille.

**Valeur par défaut** : 1 GB (souvent insuffisant)

**Recommandation** :

```conf
# Serveur haute activité (OLTP)
max_wal_size = 4GB

# Data Warehouse (OLAP, gros bulk loads)
max_wal_size = 16GB

# Serveur très haute activité
max_wal_size = 32GB
```

**Avantage** : Réduit la fréquence des checkpoints, améliore les performances d'écriture.

**Nécessite un restart** : ❌ Non (reload suffit)

### 3.3. min_wal_size : Taille Minimum du WAL

**Rôle** : Conserve au moins cette quantité de WAL (évite de recréer des fichiers).

**Valeur par défaut** : 80 MB

**Recommandation** :

```conf
# Proportionnel à max_wal_size (environ 25%)
min_wal_size = 1GB  # Si max_wal_size = 4GB  
min_wal_size = 4GB  # Si max_wal_size = 16GB  
```

**Nécessite un restart** : ❌ Non (reload suffit)

### 3.4. checkpoint_timeout : Intervalle entre Checkpoints

**Rôle** : Force un checkpoint au moins toutes les X secondes.

**Valeur par défaut** : 5 min (300s)

**Recommandation** :

```conf
# Réduire la fréquence des checkpoints (IO burst)
checkpoint_timeout = 15min  # 900s

# OLAP / Data Warehouse
checkpoint_timeout = 30min  # 1800s
```

**Trade-off** : Plus long = moins d'IO, mais recovery plus long en cas de crash.

**Nécessite un restart** : ❌ Non (reload suffit)

### 3.5. checkpoint_completion_target : Étaler les Écritures

**Rôle** : Étale les écritures du checkpoint sur cette fraction de `checkpoint_timeout`.

**Valeur par défaut** : 0.9 (90%)

**Recommandation** :

```conf
# Étaler au maximum (réduire les pics d'IO)
checkpoint_completion_target = 0.9
```

**Effet** : Lisse les écritures disque, évite les I/O burst.

**Nécessite un restart** : ❌ Non (reload suffit)

### 3.6. wal_buffers : Buffer WAL en Mémoire

**Rôle** : Buffer mémoire avant écriture sur disque.

**Valeur par défaut** : -1 (auto, 1/32 de shared_buffers, min 64KB, max 16MB)

**Recommandation** :

```conf
# Laisser en auto (performant dans la plupart des cas)
wal_buffers = -1

# Ou augmenter pour charge très élevée
wal_buffers = 32MB
```

**Nécessite un restart** : ✅ Oui

## 4. Configuration I/O (Nouveauté PostgreSQL 18)

### 4.1. io_method : Méthode d'I/O Asynchrone

**Nouveauté majeure PostgreSQL 18** : Support des **I/O asynchrones** (AIO) pour des gains de performance jusqu'à **3× plus rapide** sur certaines charges.

**Valeurs possibles** :
- `sync` : I/O synchrones (classique)  
- `async` : I/O asynchrones (nouveau)

**Recommandation** :

```conf
# Activer les I/O asynchrones (PG 18+)
io_method = 'worker'
```

**Prérequis** :
- PostgreSQL 18+
- Système d'exploitation supportant `io_uring` (Linux 5.1+) ou équivalent

**Vérification** :

```sql
SHOW io_method;
-- Résultat attendu : async
```

**Impact** : Améliore drastiquement les performances sur :
- Scans de tables volumineuses
- Bulk loads
- Opérations de tri et agrégation massives

**Nécessite un restart** : ✅ Oui

### 4.2. Vérifier le Support AIO

```bash
# Vérifier si io_uring est disponible (Linux)
cat /proc/sys/kernel/io_uring_disabled
# 0 = activé, 1 = désactivé
```

Si désactivé, activer :

```bash
sudo sysctl -w kernel.io_uring_disabled=0  
echo "kernel.io_uring_disabled=0" | sudo tee -a /etc/sysctl.conf  
```

## 5. Configuration Autovacuum (Nouveauté PostgreSQL 18)

L'**autovacuum** est critique pour maintenir les performances. PostgreSQL 18 apporte des améliorations significatives.

### 5.1. autovacuum : Activer l'Autovacuum

**Rôle** : Active le vacuum automatique.

**Valeur par défaut** : `on`

**Recommandation** :

```conf
# ✅ TOUJOURS activé en production
autovacuum = on

# ❌ Désactiver = catastrophe garantie
autovacuum = off  # Ne JAMAIS faire ça !
```

**Nécessite un restart** : ❌ Non (reload suffit)

### 5.2. autovacuum_max_workers : Parallélisation

**Rôle** : Nombre de workers autovacuum simultanés.

**Valeur par défaut** : 3

**Nouveauté PostgreSQL 18** : Ajustement dynamique avec `autovacuum_worker_slots`.

**Recommandation** :

```conf
# Serveur modeste (4-8 cores)
autovacuum_max_workers = 3

# Serveur puissant (16+ cores)
autovacuum_max_workers = 6-8

# Data Warehouse (tables massives)
autovacuum_max_workers = 10
```

**Limite** : Augmenter consomme plus de CPU et I/O.

**Nécessite un restart** : ✅ Oui

### 5.3. autovacuum_vacuum_scale_factor : Seuil de Déclenchement

**Rôle** : Déclenche un vacuum quand % de lignes modifiées.

**Valeur par défaut** : 0.2 (20%)

**Problème** : Sur tables massives, 20% = énorme délai avant vacuum.

**Recommandation** :

```conf
# Réduire pour tables volumineuses
autovacuum_vacuum_scale_factor = 0.05  # 5%

# Ou configurer par table (pour tables critiques)
ALTER TABLE ma_grande_table  
SET (autovacuum_vacuum_scale_factor = 0.01);  # 1%  
```

**Nécessite un restart** : ❌ Non (reload suffit)

### 5.4. autovacuum_vacuum_max_threshold (Nouveauté PG 18)

**Nouveauté PostgreSQL 18** : Nouveau paramètre pour limiter le nombre de lignes à traiter par vacuum.

**Rôle** : Plafond maximum de lignes mortes avant vacuum.

**Valeur par défaut** : Aucune (illimité)

**Recommandation** :

```conf
# Limiter pour éviter les vacuums trop longs
autovacuum_vacuum_max_threshold = 10000000  # 10M de lignes
```

**Cas d'usage** : Éviter qu'un vacuum monopolise les ressources trop longtemps.

**Nécessite un restart** : ❌ Non (reload suffit)

### 5.5. autovacuum_vacuum_cost_delay : Limiter l'Impact

**Rôle** : Temps de pause entre opérations I/O (throttling).

**Valeur par défaut** : 2 ms

**Recommandation** :

```conf
# Réduire pour vacuum plus agressif (si I/O disponibles)
autovacuum_vacuum_cost_delay = 0  # Pas de throttling

# Augmenter pour vacuum moins intrusif
autovacuum_vacuum_cost_delay = 10ms
```

**Nécessite un restart** : ❌ Non (reload suffit)

### 5.6. Tableau Récapitulatif Autovacuum

| Paramètre                          | Défaut | OLTP           | OLAP         |
|------------------------------------|--------|----------------|--------------|
| autovacuum                         | on     | on             | on           |
| autovacuum_max_workers             | 3      | 4-6            | 8-10         |
| autovacuum_vacuum_scale_factor     | 0.2    | 0.05-0.1       | 0.02         |
| autovacuum_vacuum_max_threshold    | ∞      | 10M            | 50M          |
| autovacuum_vacuum_cost_delay       | 2ms    | 0-2ms          | 5ms          |

## 6. Configuration du Planificateur

### 6.1. random_page_cost : Coût d'I/O Aléatoire

**Rôle** : Informe le planificateur du coût relatif d'un accès disque aléatoire vs séquentiel.

**Valeur par défaut** : 4.0 (pour disques mécaniques)

**Recommandation** :

```conf
# SSD / NVMe
random_page_cost = 1.1

# HDD (disques mécaniques)
random_page_cost = 4.0

# Stockage cloud (latence variable)
random_page_cost = 1.5-2.0
```

**Impact** : Influence le choix entre index scan et sequential scan.

**Nécessite un restart** : ❌ Non (reload suffit)

### 6.2. effective_io_concurrency : Parallélisation I/O

**Rôle** : Nombre d'opérations I/O simultanées que le système peut gérer.

**Valeur par défaut** : 1

**Recommandation** :

```conf
# SSD / NVMe
effective_io_concurrency = 200

# RAID array
effective_io_concurrency = 100-300

# HDD seul
effective_io_concurrency = 2
```

**Impact** : Améliore les performances de Bitmap Heap Scan.

**Nécessite un restart** : ❌ Non (reload suffit)

### 6.3. default_statistics_target : Précision des Statistiques

**Rôle** : Nombre d'échantillons pour les statistiques du planificateur.

**Valeur par défaut** : 100

**Recommandation** :

```conf
# Augmenter pour requêtes complexes (joins multiples)
default_statistics_target = 200

# Data Warehouse
default_statistics_target = 500
```

**Trade-off** : Plus élevé = meilleur plan, mais ANALYZE plus lent.

**Nécessite un restart** : ❌ Non (reload suffit)

## 7. Configuration par Type de Charge

### 7.1. OLTP (Online Transaction Processing)

**Caractéristiques** :
- Haute concurrence (nombreuses connexions)
- Transactions courtes
- Latence critique
- Writes et reads fréquents

**Configuration recommandée** :

```conf
# Mémoire
shared_buffers = 4GB              # 25% RAM (16GB serveur)  
effective_cache_size = 12GB       # 75% RAM  
work_mem = 16MB                   # Faible (haute concurrence)  
maintenance_work_mem = 1GB  

# Connexions
max_connections = 200             # Avec PgBouncer  
superuser_reserved_connections = 5  

# WAL
wal_level = replica  
max_wal_size = 4GB  
checkpoint_timeout = 10min  
checkpoint_completion_target = 0.9  

# I/O (PG 18)
io_method = 'worker'  
random_page_cost = 1.1            # SSD  
effective_io_concurrency = 200  

# Autovacuum
autovacuum = on  
autovacuum_max_workers = 4  
autovacuum_vacuum_scale_factor = 0.05  
autovacuum_vacuum_cost_delay = 2ms  

# Planificateur
default_statistics_target = 100
```

### 7.2. OLAP (Online Analytical Processing)

**Caractéristiques** :
- Faible concurrence
- Requêtes longues et complexes
- Débit (throughput) prioritaire
- Beaucoup d'agrégations et jointures

**Configuration recommandée** :

```conf
# Mémoire
shared_buffers = 16GB             # 25% RAM (64GB serveur)  
effective_cache_size = 48GB       # 75% RAM  
work_mem = 256MB                  # Élevé (peu de connexions)  
maintenance_work_mem = 4GB  

# Connexions
max_connections = 50              # Peu de connexions  
superuser_reserved_connections = 5  

# WAL
wal_level = replica  
max_wal_size = 16GB               # Gros bulk loads  
checkpoint_timeout = 30min  
checkpoint_completion_target = 0.9  

# I/O (PG 18)
io_method = 'worker'  
random_page_cost = 1.1            # SSD  
effective_io_concurrency = 300  

# Autovacuum
autovacuum = on  
autovacuum_max_workers = 8  
autovacuum_vacuum_scale_factor = 0.02  
autovacuum_vacuum_cost_delay = 0ms  # Pas de throttling  

# Planificateur
default_statistics_target = 500   # Statistiques précises
```

### 7.3. Mixed Workload (OLTP + OLAP)

**Compromis** entre les deux profils :

```conf
# Mémoire
shared_buffers = 8GB  
effective_cache_size = 24GB  
work_mem = 64MB  
maintenance_work_mem = 2GB  

# Connexions
max_connections = 100

# WAL
max_wal_size = 8GB  
checkpoint_timeout = 15min  

# I/O
io_method = 'worker'  
random_page_cost = 1.1  
effective_io_concurrency = 200  

# Autovacuum
autovacuum_max_workers = 6  
autovacuum_vacuum_scale_factor = 0.05  

# Planificateur
default_statistics_target = 200
```

## 9. Connection Pooling avec PgBouncer

PostgreSQL ne scale pas bien avec des milliers de connexions directes. Solution : **PgBouncer**.

### 9.1. Pourquoi PgBouncer ?

**Problème** : Chaque connexion PostgreSQL = 1 processus (~10 MB RAM).

- 1000 connexions = 10 GB RAM + overhead CPU
- Context switching élevé
- Performances dégradées

**Solution** : PgBouncer multiplexe les connexions.

**Architecture** :

```
Application: 1000 connexions logiques
         ↓
    PgBouncer (pool)
         ↓
PostgreSQL: 100 connexions physiques
```

### 9.2. Installation

```bash
# RHEL/CentOS
sudo dnf install pgbouncer

# Debian/Ubuntu
sudo apt install pgbouncer
```

### 9.3. Configuration de Base

**Fichier** : `/etc/pgbouncer/pgbouncer.ini`

```ini
[databases]
myapp = host=localhost port=5432 dbname=myapp

[pgbouncer]
# Mode de pooling
pool_mode = transaction

# Connexions PostgreSQL
default_pool_size = 25  
max_client_conn = 1000  

# Authentification
auth_type = scram-sha-256  
auth_file = /etc/pgbouncer/userlist.txt  

# Écoute
listen_addr = 0.0.0.0  
listen_port = 6432  

# Logs
logfile = /var/log/pgbouncer/pgbouncer.log  
pidfile = /var/run/pgbouncer/pgbouncer.pid  
```

### 9.4. Modes de Pooling

**1. Transaction Mode** (recommandé pour la plupart des cas)

```ini
pool_mode = transaction
```

- Connexion rendue au pool après chaque **transaction**
- Ratio élevé (1:100)
- Compatible avec la plupart des applications

**Limitations** :
- Pas de `SET` persistant entre transactions
- Pas de cursors/prepared statements partagés

**2. Session Mode**

```ini
pool_mode = session
```

- Connexion rendue après **déconnexion du client**
- Ratio faible (1:10)
- Compatible avec toutes les fonctionnalités PostgreSQL

**3. Statement Mode**

```ini
pool_mode = statement
```

- Connexion rendue après chaque **requête**
- Ratio très élevé, mais très limitatif

### 9.5. Dimensionnement

**Formule** :

```
default_pool_size = max_connections PostgreSQL / nombre de bases poolées
```

**Exemple** :

- PostgreSQL : `max_connections = 100`
- 2 bases : myapp, analytics
- PgBouncer : `default_pool_size = 50` (50 pour myapp, 50 pour analytics)

### 9.6. Configuration PostgreSQL Ajustée

Avec PgBouncer, réduire `max_connections` :

```conf
# postgresql.conf
max_connections = 100  # Au lieu de 1000

# PgBouncer gère la concurrence applicative
```

### 10.2. Monitoring des Ressources

```sql
-- Cache hit ratio (doit être > 95%)
SELECT
  sum(heap_blks_read) as heap_read,
  sum(heap_blks_hit) as heap_hit,
  sum(heap_blks_hit) / (sum(heap_blks_hit) + sum(heap_blks_read)) as ratio
FROM pg_statio_user_tables;

-- Connexions actives
SELECT count(*) FROM pg_stat_activity WHERE state = 'active';

-- Temporary files (symptôme de work_mem insuffisant)
SELECT
  datname,
  temp_files,
  pg_size_pretty(temp_bytes) as temp_size
FROM pg_stat_database  
WHERE temp_files > 0  
ORDER BY temp_bytes DESC;  
```

## Conclusion

La configuration optimale de PostgreSQL est un **équilibre** entre :
- **Ressources matérielles** disponibles  
- **Type de charge** (OLTP, OLAP, mixte)  
- **Besoins applicatifs** (latence, débit, concurrence)

**Points clés à retenir** :

1. **Mémoire** : `shared_buffers` (25% RAM), `effective_cache_size` (75% RAM), `work_mem` (ajusté selon concurrence)  
2. **Connexions** : Utiliser PgBouncer au-delà de 100-200 connexions  
3. **WAL** : Augmenter `max_wal_size` pour réduire les checkpoints  
4. **I/O** : Activer `io_method = 'worker'` (PG 18) pour gains massifs  
5. **Autovacuum** : Toujours activé, ajuster selon le volume  
6. **Monitoring** : Valider et ajuster en continu

**La configuration n'est jamais "terminée"** - c'est un processus continu d'observation, d'ajustement et de validation. Commencez avec les valeurs recommandées ici, puis affinez selon vos observations terrain.
