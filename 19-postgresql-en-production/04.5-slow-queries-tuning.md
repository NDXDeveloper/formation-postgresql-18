# 19.4.5. Slow queries et Query tuning

## Introduction : La quête de la performance

Une **requête lente** (slow query) est comme un embouteillage sur l'autoroute de vos données : elle bloque tout le trafic derrière elle, frustre vos utilisateurs et peut mettre à genoux votre application entière.

Imaginez que vous cherchez un livre dans une bibliothèque :
- **Requête rapide** : Vous utilisez le catalogue informatisé, allez directement à la bonne étagère → 30 secondes  
- **Requête lente** : Vous parcourez chaque étagère, chaque livre, une par une → 3 heures

Le **Query tuning** (optimisation de requêtes) est l'art de transformer ces 3 heures en 30 secondes. C'est souvent le moyen le plus **efficace** et le moins **coûteux** d'améliorer drastiquement les performances d'une application.

**Dans ce chapitre, vous apprendrez** :
- Comment identifier les requêtes qui ralentissent votre base
- Comment lire et comprendre un plan d'exécution (EXPLAIN)
- Les techniques pour optimiser vos requêtes
- Comment créer les bons index
- Les pièges courants et comment les éviter

## Partie 1 : Identifier les requêtes lentes

### Qu'est-ce qu'une "requête lente" ?

Il n'y a pas de définition absolue, mais voici des repères :

| Type de requête | Acceptable | Lent | Très lent |
|-----------------|------------|------|-----------|
| SELECT simple (avec index) | < 10ms | 50-100ms | > 500ms |
| SELECT avec JOIN (petites tables) | < 50ms | 100-500ms | > 1s |
| SELECT avec JOIN (grandes tables) | < 200ms | 500ms-2s | > 5s |
| INSERT/UPDATE/DELETE | < 20ms | 50-100ms | > 500ms |
| Agrégations complexes | < 500ms | 1-5s | > 10s |

**Contexte important** : Une requête "lente" dépend aussi de :
- Fréquence d'exécution (1000×/seconde vs 1×/jour)
- Attentes utilisateur (batch nocturne vs API temps réel)
- Volume de données traité

### Méthode 1 : Via pg_stat_statements (recommandé)

C'est l'extension **indispensable** pour tout DBA PostgreSQL.

#### Installation et configuration

```sql
-- 1. Activer l'extension
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- 2. Configurer dans postgresql.conf
-- shared_preload_libraries = 'pg_stat_statements'
-- pg_stat_statements.track = all
-- pg_stat_statements.max = 10000

-- Puis redémarrer PostgreSQL
-- sudo systemctl restart postgresql

-- 3. Vérifier que ça fonctionne
SELECT count(*) FROM pg_stat_statements;
```

#### Trouver les requêtes les plus lentes (temps total)

```sql
SELECT
    query,
    calls,
    total_exec_time,
    mean_exec_time,
    max_exec_time,
    stddev_exec_time,
    rows
FROM pg_stat_statements  
ORDER BY total_exec_time DESC  
LIMIT 10;  
```

**Colonnes importantes** :
- `total_exec_time` : Temps cumulé total (ms) - **Impact global**  
- `mean_exec_time` : Temps moyen par exécution (ms)  
- `calls` : Nombre d'exécutions  
- `max_exec_time` : Pire cas observé

**Interpréter les résultats** :

```
 query                                      | calls  | total_exec_time | mean_exec_time | max_exec_time
--------------------------------------------+--------+-----------------+----------------+---------------
 SELECT * FROM orders WHERE user_id = $1   | 500000 | 250000.00       | 0.50           | 150.00
 SELECT * FROM logs WHERE created_at > $1  | 100    | 180000.00       | 1800.00        | 3500.00
```

**Analyse** :
- **Requête 1** : Très fréquente (500k appels), rapide individuellement (0.5ms), mais impact global énorme (250s cumulés)  
  → **Optimiser pour réduire les appels** ou améliorer la vitesse

- **Requête 2** : Peu fréquente (100 appels), mais très lente (1.8s en moyenne)  
  → **Optimiser la requête elle-même**

#### Trouver les requêtes les plus lentes (temps moyen)

```sql
SELECT
    query,
    calls,
    mean_exec_time,
    max_exec_time,
    total_exec_time,
    rows
FROM pg_stat_statements  
WHERE calls > 10  -- Ignorer les one-off queries  
ORDER BY mean_exec_time DESC  
LIMIT 10;  
```

#### Requêtes avec variance élevée (comportement erratique)

```sql
SELECT
    query,
    calls,
    mean_exec_time,
    stddev_exec_time,
    max_exec_time,
    min_exec_time,
    -- Coefficient de variation
    CASE
        WHEN mean_exec_time > 0
        THEN round((stddev_exec_time / mean_exec_time * 100)::numeric, 2)
        ELSE 0
    END AS cv_percent
FROM pg_stat_statements  
WHERE calls > 100  
ORDER BY stddev_exec_time DESC  
LIMIT 10;  
```

**Interprétation** : `cv_percent` > 100% indique un comportement très variable (parfois rapide, parfois très lent).

### Méthode 2 : Via les logs PostgreSQL

#### Configuration du logging

```sql
-- Dans postgresql.conf

# Logger toutes les requêtes > 1 seconde
log_min_duration_statement = 1000  # en millisecondes

# Ajouter le temps d'exécution dans les logs
log_duration = off  # Déjà inclus avec log_min_duration_statement  
log_line_prefix = '%t [%p]: user=%u,db=%d,app=%a '  

# Activer auto_explain pour voir le plan des requêtes lentes
shared_preload_libraries = 'pg_stat_statements,auto_explain'  
auto_explain.log_min_duration = 1000  # Log EXPLAIN pour requêtes > 1s  
auto_explain.log_analyze = on  
auto_explain.log_buffers = on  
auto_explain.log_timing = on  
```

#### Consulter les logs

```bash
# Voir les requêtes lentes en temps réel
sudo tail -f /var/log/postgresql/postgresql-*.log | grep "duration:"

# Exemple de ligne de log :
# 2024-11-23 10:30:15.123 [12345]: user=app_user,db=mydb,app=MyApp LOG: duration: 2341.567 ms  statement: SELECT * FROM orders WHERE status = 'pending';

# Extraire les requêtes > 5 secondes
sudo grep "duration: [5-9][0-9][0-9][0-9]" /var/log/postgresql/postgresql-*.log
```

#### Analyser avec pgBadger

```bash
# Installer pgBadger
sudo apt install pgbadger

# Analyser les logs
pgbadger /var/log/postgresql/postgresql-*.log -o /tmp/report.html

# Ouvrir le rapport
firefox /tmp/report.html
```

**pgBadger vous montre** :
- Top requêtes par temps cumulé
- Top requêtes par temps moyen
- Graphiques temporels
- Répartition par type de requête

### Méthode 3 : Monitoring en temps réel

```sql
-- Voir les requêtes actuellement actives et lentes
SELECT
    pid,
    usename,
    application_name,
    client_addr,
    state,
    query_start,
    now() - query_start AS duration,
    wait_event_type,
    wait_event,
    query
FROM pg_stat_activity  
WHERE state = 'active'  
  AND query NOT LIKE '%pg_stat_activity%'
  AND now() - query_start > interval '5 seconds'
ORDER BY duration DESC;
```

**Colonnes importantes** :
- `duration` : Depuis combien de temps la requête tourne  
- `wait_event_type` / `wait_event` : Ce qui bloque la requête  
  - `IO` : Attente disque  
  - `Lock` : Attente d'un verrou  
  - `Client` : Attente du client (réseau)

## Partie 2 : EXPLAIN - Comprendre le plan d'exécution

### Qu'est-ce qu'un plan d'exécution ?

Quand vous donnez une requête SQL à PostgreSQL, il doit décider **comment** récupérer les données. Le **plan d'exécution** est la stratégie choisie par le planificateur (query planner).

**Analogie GPS** : Vous demandez un itinéraire de Paris à Lyon. Le GPS (planificateur) peut choisir :
- Autoroute A6 (rapide mais péages) → Index Scan
- Routes nationales (plus lent mais gratuit) → Sequential Scan
- Combinaison des deux → Mix de stratégies

### EXPLAIN de base

```sql
EXPLAIN  
SELECT * FROM users WHERE email = 'alice@example.com';  
```

**Résultat** :
```
                                QUERY PLAN
--------------------------------------------------------------------------
 Index Scan using users_email_idx on users  (cost=0.42..8.44 rows=1 width=100)
   Index Cond: (email = 'alice@example.com'::text)
```

**Lecture du plan** :
- `Index Scan` : Type d'opération (bon signe, utilise un index)  
- `users_email_idx` : Nom de l'index utilisé  
- `cost=0.42..8.44` : Coût estimé (unités arbitraires PostgreSQL)  
  - `0.42` : Coût pour obtenir la première ligne  
  - `8.44` : Coût total estimé  
- `rows=1` : Nombre de lignes estimé  
- `width=100` : Taille moyenne d'une ligne (bytes)

### EXPLAIN ANALYZE (avec exécution réelle)

```sql
EXPLAIN ANALYZE  
SELECT * FROM users WHERE email = 'alice@example.com';  
```

**Résultat** :
```
                                QUERY PLAN
--------------------------------------------------------------------------
 Index Scan using users_email_idx on users  (cost=0.42..8.44 rows=1 width=100)
   (actual time=0.023..0.024 rows=1 loops=1)
   Index Cond: (email = 'alice@example.com'::text)
 Planning Time: 0.156 ms
 Execution Time: 0.045 ms
```

**Nouvelles informations** :
- `actual time=0.023..0.024` : Temps réel (ms) pour première ligne..dernière ligne  
- `rows=1` (réel) vs `rows=1` (estimé) : Ici, estimation parfaite  
- `loops=1` : Nombre de fois que ce nœud a été exécuté  
- `Planning Time` : Temps pour créer le plan  
- `Execution Time` : Temps d'exécution réel

### EXPLAIN options complètes

```sql
EXPLAIN (ANALYZE, BUFFERS, VERBOSE, FORMAT JSON)  
SELECT * FROM users WHERE email = 'alice@example.com';  
```

**Options importantes** :

| Option | Description |
|--------|-------------|
| `ANALYZE` | Exécute réellement la requête et montre les temps réels |
| `BUFFERS` | Montre l'utilisation du cache (shared buffers) |
| `VERBOSE` | Affiche plus de détails sur chaque nœud |
| `COSTS` | Affiche les coûts estimés (activé par défaut) |
| `TIMING` | Mesure le temps de chaque nœud (activé par défaut avec ANALYZE) |
| `FORMAT TEXT\|JSON\|XML\|YAML` | Format de sortie |

**Exemple avec BUFFERS** :
```
Index Scan using users_email_idx on users (actual time=0.023..0.024 rows=1 loops=1)
  Index Cond: (email = 'alice@example.com'::text)
  Buffers: shared hit=4
Planning Time: 0.156 ms  
Execution Time: 0.045 ms  
```

**Interprétation BUFFERS** :
- `shared hit=4` : 4 blocs lus depuis le cache (pas de disque) ✅  
- `shared read=4` : 4 blocs lus depuis le disque ⚠️
- Plus de `hit` que de `read` = bon signe !

## Partie 3 : Les types de Scan

### 1. Sequential Scan (Scan séquentiel)

**Description** : PostgreSQL lit **toute la table** de bout en bout.

```sql
EXPLAIN ANALYZE  
SELECT * FROM orders WHERE status = 'pending';  

Seq Scan on orders  (cost=0.00..18457.34 rows=123456 width=200)
  (actual time=0.012..145.234 rows=123456 loops=1)
  Filter: (status = 'pending'::text)
  Rows Removed by Filter: 876544
```

**Quand est-ce utilisé ?**
- Pas d'index disponible
- Petite table (< 10,000 lignes)
- Requête renvoie > 5-10% des lignes

**Bon ou mauvais ?**
- ✅ **Bon** si petite table ou beaucoup de lignes à retourner  
- ❌ **Mauvais** si grande table et peu de lignes à retourner

**Optimisation** :
```sql
-- Créer un index
CREATE INDEX idx_orders_status ON orders(status);
```

### 2. Index Scan (Scan d'index)

**Description** : PostgreSQL utilise un index pour trouver les lignes.

```sql
EXPLAIN ANALYZE  
SELECT * FROM orders WHERE order_id = 12345;  

Index Scan using orders_pkey on orders  (cost=0.42..8.44 rows=1 width=200)
  (actual time=0.023..0.024 rows=1 loops=1)
  Index Cond: (order_id = 12345)
```

**Avantages** :
- Très rapide pour retrouver peu de lignes
- Évite de lire toute la table

**Inconvénient** :
- Si beaucoup de lignes à retourner, peut être plus lent que Seq Scan

### 3. Index Only Scan (Scan d'index seul)

**Description** : Les données sont entièrement dans l'index, pas besoin d'aller à la table.

```sql
-- Créer un index couvrant
CREATE INDEX idx_orders_user_status ON orders(user_id, status);

EXPLAIN ANALYZE  
SELECT user_id, status FROM orders WHERE user_id = 123;  

Index Only Scan using idx_orders_user_status on orders
  (cost=0.42..4.44 rows=5 width=8) (actual time=0.015..0.017 rows=5 loops=1)
  Index Cond: (user_id = 123)
  Heap Fetches: 0  -- ✅ Excellent ! Aucun accès à la table
```

**Avantage énorme** : Jusqu'à **10× plus rapide** qu'Index Scan normal.

**Comment l'obtenir ?**
- L'index contient **toutes** les colonnes du SELECT
- Utiliser `INCLUDE` pour ajouter des colonnes à l'index

```sql
-- Index avec colonnes supplémentaires
CREATE INDEX idx_orders_user_include  
ON orders(user_id) INCLUDE (status, amount, created_at);  

-- Maintenant cette requête utilise Index Only Scan
SELECT user_id, status, amount, created_at  
FROM orders  
WHERE user_id = 123;  
```

### 4. Bitmap Scan

**Description** : Utilisé quand plusieurs index peuvent être combinés.

```sql
EXPLAIN ANALYZE  
SELECT * FROM orders  
WHERE user_id = 123  
  AND status = 'pending'
  AND amount > 100;

Bitmap Heap Scan on orders  (cost=12.34..56.78 rows=10 width=200)
  Recheck Cond: ((user_id = 123) AND (status = 'pending') AND (amount > 100))
  ->  BitmapAnd
        ->  Bitmap Index Scan on idx_orders_user
              Index Cond: (user_id = 123)
        ->  Bitmap Index Scan on idx_orders_status
              Index Cond: (status = 'pending')
        ->  Bitmap Index Scan on idx_orders_amount
              Index Cond: (amount > 100)
```

**Fonctionnement** :
1. Scan chaque index pour créer une bitmap (carte de bits) des lignes  
2. Combiner les bitmaps (AND/OR)  
3. Lire les lignes identifiées

**Avantage** : Peut utiliser plusieurs index ensemble.

## Partie 4 : Les types de JOIN

### 1. Nested Loop (Boucle imbriquée)

**Description** : Pour chaque ligne de la table externe, scan la table interne.

```sql
EXPLAIN ANALYZE  
SELECT * FROM users u  
JOIN orders o ON o.user_id = u.id  
WHERE u.id = 123;  

Nested Loop  (cost=0.85..24.31 rows=5 width=400)
  ->  Index Scan using users_pkey on users u  (cost=0.42..8.44 rows=1)
        Index Cond: (id = 123)
  ->  Index Scan using orders_user_id_idx on orders o  (cost=0.42..15.82 rows=5)
        Index Cond: (user_id = 123)
```

**Analogie** : Deux boucles for imbriquées.

**Quand est-ce utilisé ?**
- Petite table externe (peu de lignes)
- Index disponible sur la jointure

**Performance** :
- ✅ **Excellent** pour petits datasets (< 100 lignes)  
- ❌ **Mauvais** pour grandes jointures (complexité O(n×m))

### 2. Hash Join

**Description** : Crée une table de hachage en mémoire, puis la sonde.

```sql
EXPLAIN ANALYZE  
SELECT * FROM users u  
JOIN orders o ON o.user_id = u.id;  

Hash Join  (cost=1234.56..5678.90 rows=50000 width=400)
  Hash Cond: (o.user_id = u.id)
  ->  Seq Scan on orders o  (cost=0.00..3456.78 rows=100000)
  ->  Hash  (cost=890.12..890.12 rows=10000 width=200)
        ->  Seq Scan on users u  (cost=0.00..890.12 rows=10000)
```

**Fonctionnement** :
1. Scan la table la plus petite (users)  
2. Créer une table de hachage en mémoire  
3. Scan la table la plus grande (orders)  
4. Sonder la table de hachage pour chaque ligne

**Performance** :
- ✅ **Très bon** pour grandes jointures (complexité O(n+m))  
- ⚠️ Nécessite assez de mémoire (`work_mem`)

### 3. Merge Join

**Description** : Trie les deux tables, puis les fusionne.

```sql
EXPLAIN ANALYZE  
SELECT * FROM users u  
JOIN orders o ON o.user_id = u.id  
ORDER BY u.id;  

Merge Join  (cost=123.45..678.90 rows=50000 width=400)
  Merge Cond: (u.id = o.user_id)
  ->  Index Scan using users_pkey on users u
  ->  Index Scan using orders_user_id_idx on orders o
```

**Fonctionnement** :
1. Trier les deux tables (ou utiliser index déjà trié)  
2. Fusionner les deux flux triés

**Performance** :
- ✅ **Optimal** si les deux côtés sont déjà triés (par index)  
- ⚠️ Lent si besoin de trier (coûteux en mémoire)

## Partie 5 : Optimisation des requêtes

### Technique 1 : Ajouter un index approprié

#### Exemple de problème

```sql
-- Requête lente (Seq Scan)
EXPLAIN ANALYZE  
SELECT * FROM users WHERE email = 'alice@example.com';  

Seq Scan on users  (cost=0.00..18457.34 rows=1 width=100)
  (actual time=52.341..145.678 rows=1 loops=1)
  Filter: (email = 'alice@example.com'::text)
  Rows Removed by Filter: 999999
Execution Time: 145.723 ms
```

**Problème** : Scan de 1 million de lignes pour en retourner 1 seule !

#### Solution : Index

```sql
-- Créer l'index
CREATE INDEX idx_users_email ON users(email);

-- Même requête après index
EXPLAIN ANALYZE  
SELECT * FROM users WHERE email = 'alice@example.com';  

Index Scan using idx_users_email on users  (cost=0.42..8.44 rows=1 width=100)
  (actual time=0.023..0.024 rows=1 loops=1)
Execution Time: 0.045 ms
```

**Amélioration** : 145ms → 0.045ms = **3200× plus rapide !** 🚀

### Technique 2 : Index multi-colonnes

#### Exemple

```sql
-- Requête avec deux filtres
SELECT * FROM orders  
WHERE user_id = 123  
  AND status = 'pending';
```

#### Mauvaise approche : Un index par colonne

```sql
CREATE INDEX idx_orders_user ON orders(user_id);  
CREATE INDEX idx_orders_status ON orders(status);  

-- PostgreSQL utilisera Bitmap Scan (pas optimal)
```

#### Bonne approche : Index multi-colonnes

```sql
-- Index composé (ordre important !)
CREATE INDEX idx_orders_user_status ON orders(user_id, status);

-- Maintenant : Index Scan direct
EXPLAIN ANALYZE  
SELECT * FROM orders WHERE user_id = 123 AND status = 'pending';  

Index Scan using idx_orders_user_status on orders
  Index Cond: ((user_id = 123) AND (status = 'pending'))
```

**Règle d'ordre** : Colonne avec **égalité** d'abord, puis **range/inequality**.

```sql
-- ✅ BON
CREATE INDEX idx_orders_user_date ON orders(user_id, created_at);  
SELECT * FROM orders WHERE user_id = 123 AND created_at > '2024-01-01';  

-- ❌ MOINS BON (order inversé)
CREATE INDEX idx_orders_date_user ON orders(created_at, user_id);
-- Ne peut pas utiliser l'index efficacement pour user_id = 123
```

### Technique 3 : Index partiels

Pour indexer seulement un sous-ensemble de lignes.

```sql
-- Index uniquement les commandes non payées
CREATE INDEX idx_orders_pending ON orders(user_id)  
WHERE status = 'pending';  

-- Cette requête utilise l'index partiel (plus petit, plus rapide)
SELECT * FROM orders  
WHERE user_id = 123  
  AND status = 'pending';
```

**Avantages** :
- Index plus petit (moins d'espace disque)
- Plus rapide à maintenir (moins de mises à jour)
- Cache plus efficace

### Technique 4 : Réécrire la requête

#### Exemple : NOT IN vs NOT EXISTS

```sql
-- ❌ LENT : NOT IN avec sous-requête
SELECT * FROM users  
WHERE id NOT IN (  
    SELECT user_id FROM orders WHERE amount > 1000
);

-- ✅ RAPIDE : NOT EXISTS (mieux optimisé)
SELECT * FROM users u  
WHERE NOT EXISTS (  
    SELECT 1 FROM orders o
    WHERE o.user_id = u.id
      AND o.amount > 1000
);
```

**Raison** : `NOT EXISTS` peut s'arrêter dès qu'il trouve une correspondance. `NOT IN` doit évaluer toute la sous-requête.

#### Exemple : OR vs UNION

```sql
-- ❌ LENT : OR peut empêcher l'utilisation d'index
SELECT * FROM orders  
WHERE user_id = 123 OR customer_email = 'alice@example.com';  

-- ✅ RAPIDE : UNION ALL avec deux index
SELECT * FROM orders WHERE user_id = 123  
UNION ALL  
SELECT * FROM orders WHERE customer_email = 'alice@example.com'  
  AND user_id != 123;  -- Éviter les doublons
```

**Nouveauté PG 18** : PostgreSQL 18 optimise automatiquement certaines OR-clauses en ANY, donc cette optimisation est moins nécessaire qu'avant !

### Technique 5 : Éviter SELECT *

```sql
-- ❌ LENT : Récupère toutes les colonnes
SELECT * FROM users WHERE id = 123;

-- ✅ RAPIDE : Seulement les colonnes nécessaires
SELECT id, username, email FROM users WHERE id = 123;
```

**Pourquoi ?**
- Moins de données à transférer
- Peut utiliser Index Only Scan si les colonnes sont dans l'index
- Moins de bande passante réseau

### Technique 6 : LIMIT avec ORDER BY

```sql
-- ❌ LENT : Trie tout puis limite
SELECT * FROM orders  
ORDER BY created_at DESC  
LIMIT 10;  

-- Si pas d'index sur created_at : Sort de toute la table !
```

**Solution** :

```sql
-- Créer un index sur la colonne de tri
CREATE INDEX idx_orders_created_desc ON orders(created_at DESC);

-- Maintenant : Index Scan avec Limit (très rapide)
EXPLAIN ANALYZE  
SELECT * FROM orders ORDER BY created_at DESC LIMIT 10;  

Limit  (cost=0.42..1.23 rows=10 width=200)
  ->  Index Scan Backward using idx_orders_created_desc on orders
        (cost=0.42..8457.34 rows=100000 width=200)
```

**Amélioration** : PostgreSQL s'arrête après avoir lu 10 lignes, pas besoin de trier !

### Technique 7 : Utiliser les CTE judicieusement

```sql
-- ❌ POTENTIELLEMENT LENT : CTE non matérialisée
WITH recent_orders AS (
    SELECT * FROM orders WHERE created_at > now() - interval '7 days'
)
SELECT u.username, count(o.*)  
FROM users u  
JOIN recent_orders o ON o.user_id = u.id  
GROUP BY u.username;  

-- CTE peut être recalculé plusieurs fois ou empêcher optimisations
```

**Solutions** :

```sql
-- Option 1 : Forcer la matérialisation (cache le résultat)
WITH recent_orders AS MATERIALIZED (
    SELECT * FROM orders WHERE created_at > now() - interval '7 days'
)
SELECT u.username, count(o.*)  
FROM users u  
JOIN recent_orders o ON o.user_id = u.id  
GROUP BY u.username;  

-- Option 2 : Réécrire sans CTE (peut être mieux optimisé)
SELECT u.username, count(o.*)  
FROM users u  
JOIN orders o ON o.user_id = u.id  
WHERE o.created_at > now() - interval '7 days'  
GROUP BY u.username;  
```

**Astuce** : Testez les deux versions avec EXPLAIN ANALYZE !

## Partie 6 : Optimisations avancées

### Nouveauté PostgreSQL 18 : Skip Scan

PostgreSQL 18 peut maintenant utiliser des index multi-colonnes même si la première colonne n'est pas dans la clause WHERE.

```sql
-- Index multi-colonnes
CREATE INDEX idx_orders_status_date ON orders(status, created_at);

-- Avant PG 18 : Seq Scan (ne pouvait pas utiliser l'index)
-- PG 18 : Skip Scan (utilise l'index !)
SELECT * FROM orders WHERE created_at > '2024-01-01';

Index Scan using idx_orders_status_date on orders
  Filter: (created_at > '2024-01-01')
```

**Avantage** : Moins d'index nécessaires, meilleure utilisation des index existants.

### Nouveauté PostgreSQL 18 : Auto-élimination des self-joins

```sql
-- Avant PG 18 : Join inutile non détecté
SELECT DISTINCT o1.id  
FROM orders o1  
JOIN orders o2 ON o1.id = o2.id;  

-- PG 18 détecte et élimine le self-join
-- Transformé en :
SELECT DISTINCT id FROM orders;
```

### Nouveauté PostgreSQL 18 : Optimisation IN (VALUES ...)

```sql
-- Avant : Lent
SELECT * FROM users WHERE id IN (1, 2, 3, 4, 5);

-- PG 18 optimise automatiquement en :
SELECT * FROM users WHERE id = ANY(ARRAY[1, 2, 3, 4, 5]);
```

### Partitioning pour performances

```sql
-- Partitionner une grande table par date
CREATE TABLE logs (
    id BIGSERIAL,
    message TEXT,
    created_at TIMESTAMP
) PARTITION BY RANGE (created_at);

-- Créer des partitions mensuelles
CREATE TABLE logs_2024_01 PARTITION OF logs
    FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

CREATE TABLE logs_2024_02 PARTITION OF logs
    FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');

-- Requête ne scanne que la partition pertinente
SELECT * FROM logs WHERE created_at >= '2024-01-15' AND created_at < '2024-01-20';

-- EXPLAIN montre : Partition Pruning
-- Scan uniquement logs_2024_01
```

**Avantages** :
- Scan de moins de données
- Maintenance par partition (VACUUM, index)
- Suppression rapide (DROP PARTITION)

## Partie 7 : Configuration PostgreSQL pour performances

### Paramètres critiques

```sql
-- work_mem : Mémoire pour tris, hash tables
-- Par défaut : 4MB (souvent trop faible)
-- Recommandé : 64MB - 256MB
SET work_mem = '128MB';

-- Effet visible dans EXPLAIN
EXPLAIN ANALYZE  
SELECT * FROM orders ORDER BY created_at;  

-- Avant (work_mem = 4MB) :
-- Sort Method: external merge  Disk: 123456kB  -- ❌ Trie sur disque !

-- Après (work_mem = 128MB) :
-- Sort Method: quicksort  Memory: 45678kB      -- ✅ Trie en mémoire !
```

**Attention** : `work_mem` peut être utilisé **plusieurs fois** par requête. Formule :

```
Mémoire max = work_mem × max_connections × 2 à 4
```

### effective_cache_size

Indique au planificateur combien de RAM le système a pour cacher les données.

```sql
-- Valeur recommandée : 50-75% de la RAM totale
-- Serveur avec 32GB RAM :
effective_cache_size = '24GB'
```

**Impact** : Influence les décisions du planificateur (Index Scan vs Seq Scan).

### random_page_cost

Coût relatif d'une lecture aléatoire vs séquentielle.

```sql
-- HDD traditionnel (défaut)
random_page_cost = 4.0

-- SSD (recommandé)
random_page_cost = 1.1

-- NVMe (très rapide)
random_page_cost = 1.0
```

**Impact** : Valeur plus basse favorise l'utilisation d'index.

### Configuration optimale pour performances

```sql
-- Dans postgresql.conf

# MÉMOIRE
shared_buffers = 8GB                # 25% de RAM  
work_mem = 128MB                    # Pour tris, hash  
maintenance_work_mem = 2GB          # Pour VACUUM, CREATE INDEX  
effective_cache_size = 24GB         # 75% de RAM  

# QUERY PLANNER
random_page_cost = 1.1              # Pour SSD  
effective_io_concurrency = 200      # Opérations I/O parallèles (SSD)  
default_statistics_target = 100     # Statistiques plus précises  

# PARALLÉLISATION
max_parallel_workers_per_gather = 4  
max_parallel_workers = 8  
max_worker_processes = 8  

# LOGGING (pour debugging)
log_min_duration_statement = 1000   # Log requêtes > 1s  
log_line_prefix = '%t [%p]: '  
shared_preload_libraries = 'pg_stat_statements'  
```

## Partie 8 : Maintenance pour performances

### ANALYZE : Mettre à jour les statistiques

Le planificateur se base sur des **statistiques** pour choisir le meilleur plan.

```sql
-- Statistiques obsolètes → Mauvais plans
-- Exemple : Table a 1M lignes, mais stats disent 1000 lignes
-- Planificateur choisit Nested Loop → Très lent !

-- Mettre à jour les statistiques
ANALYZE users;  
ANALYZE orders;  

-- Ou toute la base
ANALYZE;
```

**Quand faire ANALYZE ?**
- Après gros import de données
- Après DELETE/UPDATE massif
- Périodiquement (autovacuum le fait automatiquement)

### VACUUM : Nettoyer les lignes mortes

Les UPDATE/DELETE laissent des "lignes mortes" qui ralentissent les scans.

```sql
-- Voir les lignes mortes
SELECT
    schemaname,
    tablename,
    n_live_tup,    -- Lignes vivantes
    n_dead_tup,    -- Lignes mortes
    last_vacuum,
    last_autovacuum
FROM pg_stat_user_tables  
ORDER BY n_dead_tup DESC;  

-- Si n_dead_tup élevé :
VACUUM ANALYZE users;
```

### REINDEX : Reconstruire les index

Les index peuvent devenir "gonflés" (bloat) avec le temps.

```sql
-- Vérifier la taille des index
SELECT
    indexrelname,
    pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes  
ORDER BY pg_relation_size(indexrelid) DESC;  

-- Reconstruire un index gonflé
REINDEX INDEX CONCURRENTLY users_email_idx;

-- Ou toute une table
REINDEX TABLE CONCURRENTLY users;
```

**Note** : `CONCURRENTLY` permet de faire l'opération sans bloquer les lectures/écritures.

## Partie 10 : Anti-patterns et pièges courants

### Piège 1 : N+1 queries

**Problème** :

```python
# ❌ MAUVAIS : 1 requête pour users + N requêtes pour orders
users = db.query("SELECT * FROM users LIMIT 100")  
for user in users:  
    orders = db.query("SELECT * FROM orders WHERE user_id = %s", user.id)
    print(f"{user.name} has {len(orders)} orders")

# Total : 101 requêtes !
```

**Solution** :

```python
# ✅ BON : 2 requêtes seulement
users = db.query("SELECT * FROM users LIMIT 100")  
user_ids = [u.id for u in users]  
orders = db.query("""  
    SELECT user_id, count(*) as order_count
    FROM orders
    WHERE user_id = ANY(%s)
    GROUP BY user_id
""", user_ids)

# Créer un dict pour lookup rapide
order_counts = {o.user_id: o.order_count for o in orders}  
for user in users:  
    print(f"{user.name} has {order_counts.get(user.id, 0)} orders")
```

### Piège 2 : Fonctions dans WHERE empêchent index

```sql
-- ❌ MAUVAIS : Fonction empêche l'utilisation d'index
SELECT * FROM users WHERE LOWER(email) = 'alice@example.com';

-- ✅ BON : Index fonctionnel
CREATE INDEX idx_users_email_lower ON users(LOWER(email));

-- Ou mieux : normaliser les données en amont
UPDATE users SET email = LOWER(email);  
CREATE INDEX idx_users_email ON users(email);  
```

### Piège 3 : Type mismatch

```sql
-- ❌ MAUVAIS : user_id est INTEGER, mais '123' est TEXT
-- PostgreSQL ne peut pas utiliser l'index !
SELECT * FROM orders WHERE user_id = '123';

-- ✅ BON : Même type
SELECT * FROM orders WHERE user_id = 123;
```

### Piège 4 : OFFSET élevé

```sql
-- ❌ MAUVAIS : Pour page 1000 (OFFSET 10000)
-- PostgreSQL doit scanner 10000 lignes puis les ignorer !
SELECT * FROM orders ORDER BY created_at OFFSET 10000 LIMIT 10;

-- ✅ BON : Pagination par curseur
SELECT * FROM orders  
WHERE created_at < '2024-01-01 10:30:00'  -- Last créé de page précédente  
ORDER BY created_at DESC  
LIMIT 10;  
```

### Piège 5 : LIKE avec wildcard au début

```sql
-- ❌ MAUVAIS : Ne peut pas utiliser l'index B-Tree
SELECT * FROM users WHERE email LIKE '%@example.com';

-- ✅ Solutions :
-- 1. Créer un index GIN avec trigram
CREATE EXTENSION pg_trgm;  
CREATE INDEX idx_users_email_trgm ON users USING gin(email gin_trgm_ops);  

-- 2. Ou utiliser full-text search
CREATE INDEX idx_users_email_fts ON users USING gin(to_tsvector('english', email));  
SELECT * FROM users WHERE to_tsvector('english', email) @@ to_tsquery('example.com');  

-- 3. Ou ajouter une colonne domain
ALTER TABLE users ADD COLUMN email_domain TEXT;  
UPDATE users SET email_domain = split_part(email, '@', 2);  
CREATE INDEX idx_users_email_domain ON users(email_domain);  
SELECT * FROM users WHERE email_domain = 'example.com';  
```

## Cas d'étude : Optimisation réelle

### Problème : Dashboard lent

**Contexte** :
- Dashboard affiche statistiques utilisateur
- Temps de chargement : 8-12 secondes
- Requête principale identifiée via pg_stat_statements

### Étape 1 : Requête originale

```sql
SELECT
    u.id,
    u.username,
    u.email,
    COUNT(DISTINCT o.id) as order_count,
    SUM(o.amount) as total_spent,
    MAX(o.created_at) as last_order_date,
    (
        SELECT COUNT(*) FROM reviews r
        WHERE r.user_id = u.id
    ) as review_count
FROM users u  
LEFT JOIN orders o ON o.user_id = u.id  
WHERE u.created_at > now() - interval '30 days'  
GROUP BY u.id  
ORDER BY total_spent DESC  
LIMIT 20;  
```

### Étape 2 : EXPLAIN ANALYZE (avant)

```
Hash Join  (cost=45678.90..123456.78 rows=1000 width=200)
  (actual time=4567.123..11234.567 rows=20 loops=1)
  ->  Seq Scan on users u  (cost=0.00..12345.67 rows=5000 width=100)
        Filter: (created_at > (now() - '30 days'))
        Rows Removed by Filter: 995000
  ->  Hash  (cost=23456.78..23456.78 rows=100000 width=50)
        ->  Seq Scan on orders o  (cost=0.00..23456.78 rows=1000000 width=50)
  SubPlan 1
    ->  Aggregate  (cost=45.67..45.68 rows=1 width=8)
          ->  Seq Scan on reviews r  (cost=0.00..45.67 rows=5 width=0)
                Filter: (user_id = u.id)

Planning Time: 12.345 ms  
Execution Time: 11234.567 ms  
```

**Problèmes identifiés** :
1. Seq Scan sur `users` avec filtre (995k lignes supprimées)  
2. Seq Scan sur `orders` (1M lignes)  
3. Sous-requête corrélée exécutée pour chaque ligne  
4. Pas d'index sur `users.created_at`

### Étape 3 : Optimisations appliquées

```sql
-- 1. Index sur created_at pour filtrer rapidement
CREATE INDEX idx_users_created ON users(created_at);

-- 2. Index pour les jointures
CREATE INDEX idx_orders_user_amount ON orders(user_id, amount, created_at);

-- 3. Index pour les reviews
CREATE INDEX idx_reviews_user ON reviews(user_id);

-- 4. Réécrire la requête sans sous-requête corrélée
SELECT
    u.id,
    u.username,
    u.email,
    COUNT(DISTINCT o.id) as order_count,
    SUM(o.amount) as total_spent,
    MAX(o.created_at) as last_order_date,
    COALESCE(r.review_count, 0) as review_count
FROM users u  
LEFT JOIN orders o ON o.user_id = u.id  
LEFT JOIN (  
    SELECT user_id, COUNT(*) as review_count
    FROM reviews
    GROUP BY user_id
) r ON r.user_id = u.id
WHERE u.created_at > now() - interval '30 days'  
GROUP BY u.id, r.review_count  
ORDER BY total_spent DESC NULLS LAST  
LIMIT 20;  
```

### Étape 4 : EXPLAIN ANALYZE (après)

```
Limit  (cost=234.56..345.67 rows=20 width=200)
  (actual time=5.123..12.456 rows=20 loops=1)
  ->  Sort  (cost=234.56..245.67 rows=1000 width=200)
        Sort Key: (sum(o.amount)) DESC NULLS LAST
        ->  Hash Left Join  (cost=123.45..234.56 rows=1000 width=200)
              ->  Index Scan using idx_users_created on users u
                    (cost=0.42..45.67 rows=5000 width=100)
                    Index Cond: (created_at > (now() - '30 days'))
              ->  Hash  (cost=100.00..100.00 rows=1000 width=50)
                    ->  Index Scan using idx_orders_user_amount on orders o
                          (cost=0.42..100.00 rows=50000 width=50)
              ->  Hash  (cost=23.45..23.45 rows=500 width=16)
                    ->  HashAggregate  (cost=20.00..23.45 rows=500 width=16)
                          ->  Index Only Scan using idx_reviews_user on reviews
                                (cost=0.42..15.00 rows=5000 width=8)

Planning Time: 2.345 ms  
Execution Time: 12.567 ms  
```

### Résultat

| Métrique | Avant | Après | Amélioration |
|----------|-------|-------|--------------|
| Execution Time | 11,234 ms | 12.5 ms | **898× plus rapide** |
| Planning Time | 12.3 ms | 2.3 ms | 5× plus rapide |
| Rows Scanned | ~2,000,000 | ~10,000 | 200× moins |

**Actions** :
- 3 index créés
- Requête réécrite (sans sous-requête corrélée)
- Temps de chargement dashboard : **12 secondes → 0.01 seconde**

## Résumé des points clés

✅ **pg_stat_statements** : Extension indispensable pour identifier les slow queries

✅ **EXPLAIN ANALYZE** : Outil principal pour comprendre pourquoi une requête est lente

✅ **Index appropriés** : Peuvent améliorer les performances de 100× à 10,000×

✅ **Index multi-colonnes** : Ordre important (égalité avant range)

✅ **Index Only Scan** : Le plus rapide, utiliser `INCLUDE` pour l'obtenir

✅ **Réécriture de requêtes** : NOT EXISTS > NOT IN, UNION peut être meilleur que OR

✅ **Configuration** : work_mem, effective_cache_size, random_page_cost pour SSD

✅ **Maintenance** : ANALYZE pour statistiques, VACUUM pour performance

✅ **Anti-patterns** : Éviter N+1, fonctions dans WHERE, OFFSET élevé

✅ **Nouveautés PG 18** : Skip Scan, auto-élimination self-joins, optimisations OR/IN

## Conclusion

L'optimisation des requêtes est un **processus itératif** :

1. **Mesurer** : Identifier les requêtes lentes avec pg_stat_statements  
2. **Analyser** : Comprendre le plan avec EXPLAIN ANALYZE  
3. **Optimiser** : Ajouter index, réécrire requêtes, ajuster configuration  
4. **Valider** : Vérifier l'amélioration  
5. **Surveiller** : Monitoring continu pour nouveaux problèmes

**Règles d'or** :
- Un bon index peut être 1000× plus efficace qu'une requête complexe
- EXPLAIN ANALYZE est votre meilleur ami
- Mesurez toujours avant et après l'optimisation
- Optimisez d'abord les requêtes à fort impact (fréquentes OU très lentes)

Avec les techniques de ce chapitre, vous pouvez transformer une base de données lente en une machine de course performante. La majorité des problèmes de performance proviennent de **requêtes mal optimisées** plutôt que de limitations matérielles. Investir du temps dans le query tuning est donc **toujours** rentable.
