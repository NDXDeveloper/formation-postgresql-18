# 19.1.2. Virtual Machines (VM)

## Introduction à la Virtualisation

### Qu'est-ce qu'une Machine Virtuelle ?

Une **machine virtuelle (VM)** est un ordinateur logiciel qui s'exécute au-dessus d'un ordinateur physique. C'est comme avoir plusieurs ordinateurs indépendants fonctionnant simultanément sur un seul serveur physique.

**Analogie simple :**

Imaginez un grand immeuble (serveur physique) divisé en plusieurs appartements (VMs). Chaque appartement :
- A ses propres murs (isolation)
- Partage les infrastructures communes (électricité, eau = CPU, RAM)
- Est indépendant des autres (un problème dans l'appartement A n'affecte pas B)
- Peut être agrandi ou réduit selon les besoins (ressources ajustables)

### Vocabulaire de Base

| Terme | Définition | Exemple |
|-------|-----------|---------|
| **Hyperviseur** | Logiciel qui crée et gère les VMs | VMware ESXi, Proxmox, KVM |
| **Hôte (Host)** | Serveur physique hébergeant les VMs | Serveur Dell R750 avec ESXi |
| **Invité (Guest)** | Système d'exploitation dans la VM | Ubuntu Server dans une VM |
| **vCPU** | CPU virtuel alloué à une VM | 8 vCPU = 8 cœurs virtuels |
| **Snapshot** | Copie instantanée de l'état d'une VM | Backup avant mise à jour |
| **Live Migration** | Déplacement VM sans interruption | Déplacer VM entre serveurs |

### Pourquoi Utiliser des VMs pour PostgreSQL ?

**Avantages clés :**

1. **Consolidation des ressources**
   - Un serveur physique peut héberger 10-20 VMs
   - Meilleure utilisation du matériel (taux d'occupation 70-80% vs 20-30% bare metal)
   - Réduction des coûts matériels

2. **Flexibilité**
   - Ajustement des ressources à chaud (CPU, RAM)
   - Création rapide de nouveaux environnements (minutes vs jours)
   - Clonage facile pour test/développement

3. **Haute disponibilité simplifiée**
   - Live Migration : déplacer une VM sans arrêt
   - Failover automatique en cas de panne matérielle
   - Snapshots pour restauration rapide

4. **Isolation**
   - Environnements séparés (production, staging, développement)
   - Sécurité renforcée (isolation réseau et stockage)
   - Pas d'impact entre VMs en cas de problème

### Overhead de Virtualisation : Comprendre l'Impact

**Qu'est-ce que l'overhead ?**

L'overhead (surcharge) représente les ressources consommées par la couche de virtualisation elle-même. C'est le "coût" de la flexibilité.

**Décomposition de l'overhead typique :**

```
Serveur physique : 100% de ressources disponibles
├─ Hyperviseur : 5-8% (gestion VMs, ordonnancement)
├─ Virtualisation CPU : 2-3% (instructions CPU virtualisées)
├─ Virtualisation I/O : 3-5% (disque et réseau)
└─ Disponible pour PostgreSQL : 85-90%
```

**En pratique pour PostgreSQL :**
- **CPU :** 2-5% d'overhead (négligeable avec technologies modernes)  
- **RAM :** 0-2% (très faible impact)  
- **I/O disque :** 5-15% (le plus impactant, amélioration avec paravirtualisation)  
- **Réseau :** 2-5% (négligeable avec SR-IOV)

**Conclusion :** Sur hardware moderne avec hyperviseur optimisé, l'overhead est < 10% pour PostgreSQL, ce qui est un excellent compromis pour la flexibilité obtenue.

Les hyperviseurs de **Type 1** s'installent directement sur le matériel, sans système d'exploitation intermédiaire.

**Présentation :**
- Leader du marché enterprise

**Avantages :**

```bash

```

**Prix indicatif :** 0 € (gratuit)

## Architecture Matérielle pour VMs PostgreSQL

### Dimensionnement du Serveur Hôte

#### Règles de Consolidation

**Ratio de consolidation :**
Le nombre de VMs que vous pouvez héberger dépend de leur taille et charge.

**Exemple concret :**

```
Serveur physique :
├─ CPU : 2× AMD EPYC 7543 = 64 cœurs / 128 threads
├─ RAM : 512 GB
├─ Disque : 8× NVMe 2TB en RAID 10 = 8 TB utile
└─ Réseau : 2× 25 GbE

Peut héberger :
├─ Option 1 (OLTP) : 8 VMs PostgreSQL
│   └─ 8 vCPU, 64 GB RAM, 500 GB disque chacune
├─ Option 2 (Mixte) : 4 VMs grosses + 8 VMs petites
│   ├─ 4× (16 vCPU, 96 GB, 1 TB)
│   └─ 8× (4 vCPU, 16 GB, 200 GB)
└─ Ressources réservées hyperviseur : 10%
```

#### Règles d'Or du Dimensionnement

**1. CPU : Éviter l'overcommit pour PostgreSQL**

```
Overcommit ratio = vCPU totaux alloués / Cœurs physiques

✅ Ratio 1:1 (recommandé OLTP)
   └─ 64 cœurs physiques = max 64 vCPU allouées total

✅ Ratio 2:1 (acceptable mixte)
   └─ 64 cœurs physiques = max 128 vCPU allouées

⚠️ Ratio 4:1 (risqué pour PostgreSQL)
   └─ Contentions CPU possibles

❌ Ratio >4:1 (à éviter)
   └─ Performances dégradées garanties
```

**Pourquoi ?** PostgreSQL est sensible au CPU. Des contentions ralentissent significativement les requêtes.

**2. RAM : Jamais d'overcommit**

```
❌ INTERDIT : Allouer plus de RAM virtuelle que RAM physique

Exemple :  
Serveur avec 512 GB RAM  
├─ Hyperviseur : 25 GB réservés
├─ Disponible VMs : 487 GB maximum
└─ ⚠️ Ne JAMAIS dépasser cette limite

Conséquence overcommit RAM :
└─ Swap utilisé → PostgreSQL ralentit de 100-1000×
```

**Best practice :** Réserver 5-10% de RAM pour l'hyperviseur, ne jamais allouer plus que le reste.

**3. Disque : Privilégier performance à capacité**

```
Stockage VM PostgreSQL :
├─ Jamais de stockage réseau lent (NFS, iSCSI standard)
├─ Privilégier :
│   ├─ Stockage local NVMe (meilleur)
│   ├─ SAN Flash haute performance
│   ├─ vSAN VMware (bon compromis)
│   └─ Ceph avec réseaux dédiés (open source)
└─ Éviter :
    ├─ NAS entry-level
    ├─ Stockage distant avec latence >2ms
    └─ iSCSI sur réseau partagé
```

### Configuration CPU Optimale

#### Types de vCPU

**1. CPU Passthrough (Recommandé Production)**

```yaml
Configuration Proxmox :
  cpu: host
  # Expose toutes les instructions CPU natives à la VM
  # Performances optimales, pas d'émulation
```

```xml
Configuration KVM/libvirt :
  <cpu mode='host-passthrough'>
    <topology sockets='2' cores='8' threads='1'/>
  </cpu>
```

**Avantages :**
- Performance maximale (overhead < 2%)
- Support instructions modernes (AVX-512, etc.)
- PostgreSQL voit le CPU réel

**Inconvénient :**
- Migration limitée entre CPU différents

**2. CPU Émulé (Compatible Migration)**

```yaml
Configuration Proxmox :
  cpu: x86-64-v3  # ou kvm64
  # CPU générique pour compatibilité migration
```

**Avantages :**
- Live Migration entre hôtes différents
- Compatibilité maximale

**Inconvénient :**
- Overhead 3-5%
- Instructions modernes désactivées

**Recommandation PostgreSQL :**
- **Production critique :** host-passthrough  
- **Environnement multi-hôtes hétérogènes :** CPU émulé compatible

#### Topologie CPU : Sockets vs Cores

**Exemple :**
Une VM avec 16 vCPU peut être configurée ainsi :

```
Option 1 : 1 socket × 16 cœurs  
Option 2 : 2 sockets × 8 cœurs  
Option 3 : 4 sockets × 4 cœurs  
```

**Impact sur PostgreSQL :**

PostgreSQL n'est pas NUMA-aware par défaut. Privilégiez :

```yaml
Recommandé :
  sockets: 1
  cores: 16
  threads: 1  # Désactiver hyperthreading si possible
```

**Pourquoi ?**
- Évite la complexité NUMA (Non-Uniform Memory Access)
- Meilleure localité mémoire
- Plus simple à gérer

**Exception :** Si >32 vCPU, suivez la topologie de l'hôte physique.

#### CPU Pinning (Épinglage)

**Qu'est-ce que le CPU pinning ?**

Lier les vCPU d'une VM à des cœurs physiques spécifiques pour éviter la migration entre cœurs.

```yaml
Configuration Proxmox (affinity) :
  taskset: 0-15  # vCPU liés aux cœurs physiques 0 à 15
```

**Avantages :**
- Performances prédictibles
- Pas de contention entre VMs
- Meilleur pour benchmark

**Inconvénients :**
- Perte de flexibilité
- Moins de consolidation possible
- Gestion complexe

**Quand utiliser pour PostgreSQL ?**
- ✅ VM critique unique sur hôte  
- ✅ Besoins de performances maximales et constantes  
- ❌ Environnement avec >5 VMs par hôte

### Configuration RAM Optimale

#### Réservation Mémoire (Memory Reservation)

**Concept :** Garantir qu'une quantité de RAM physique soit toujours disponible pour la VM.

```yaml
VM PostgreSQL Production :
  memory: 64 GB
  min_memory: 64 GB  # Réservation 100%
  # ⚠️ Désactive ballooning et swap
```

**Pourquoi c'est critique pour PostgreSQL ?**

PostgreSQL gère sa propre mémoire (shared_buffers, work_mem). Si la RAM physique n'est pas garantie :
1. Hyperviseur utilise ballooning ou swap  
2. Performances PostgreSQL s'effondrent (×10-1000 plus lent)  
3. Timeouts applicatifs en cascade

**Règle d'or :** Pour PostgreSQL production, **toujours réserver 100% de la RAM**.

#### Ballooning : À Désactiver !

**Qu'est-ce que le ballooning ?**

Mécanisme permettant à l'hyperviseur de "reprendre" de la RAM aux VMs si nécessaire.

```yaml
# VMware
sched.mem.maxmemctl: 0  # Désactive ballooning

# Proxmox / KVM
balloon: 0  # Désactive ballooning
```

**Pourquoi désactiver pour PostgreSQL ?**
- PostgreSQL utilise toute sa RAM allouée
- Le ballooning cause des ralentissements imprévisibles
- Incompatible avec performances stables

#### Huge Pages (Pages Mémoire Géantes)

**Qu'est-ce que les Huge Pages ?**

Pages mémoire de 2 MB au lieu de 4 KB standard. Réduit le nombre d'entrées TLB (Translation Lookaside Buffer).

**Configuration dans la VM :**

```bash
# /etc/sysctl.conf dans la VM
vm.nr_hugepages = 33792  # Pour 64 GB (64*1024/2)

# PostgreSQL postgresql.conf
huge_pages = on
```

**Gains pour PostgreSQL :**
- Réduction overhead TLB : 2-5% de performance CPU
- Mémoire non swappable
- Recommandé pour VMs >32 GB RAM

### Configuration Stockage pour VMs

#### Types de Disques Virtuels

**1. Thin Provisioning vs Thick Provisioning**

```
Thick Provisioning (Recommandé PostgreSQL) :
┌────────────────────────────────────────┐
│ Disque VM : 500 GB alloués             │
│ ↓                                      │
│ Datastore : 500 GB réservés            │ ← Espace garanti
└────────────────────────────────────────┘
Avantage : Performance stable, pas de surprise

Thin Provisioning (À éviter production) :
┌────────────────────────────────────────┐
│ Disque VM : 500 GB virtuels            │
│ ↓                                      │
│ Datastore : 50 GB réellement utilisés  │ ← Grandit au besoin
└────────────────────────────────────────┘
Risque : Saturation datastore inattendue, performances variables
```

**Recommandation PostgreSQL :** **Thick Provisioning Eager Zeroed** (VMware) ou équivalent.

**2. Format de Disque Virtuel**

| Format | Description | Performance | Usage |
|--------|-------------|-------------|--------|
| **Raw** | Fichier brut 1:1 | ⭐⭐⭐⭐⭐ | Production |
| **qcow2** | QEMU Copy-On-Write | ⭐⭐⭐⭐ | Snapshots, dev |
| **VMDK** | VMware natif | ⭐⭐⭐⭐⭐ | VMware |
| **VHD/VHDX** | Hyper-V | ⭐⭐⭐⭐ | Hyper-V |

**Recommandation :**
- **Production :** Raw (KVM/Proxmox) ou VMDK (VMware)  
- **Développement :** qcow2 pour snapshots faciles

#### Drivers de Disque Paravirtualisés

**Qu'est-ce que la paravirtualisation ?**

Le guest OS "sait" qu'il est virtualisé et utilise des drivers optimisés au lieu d'émuler du matériel.

```
Émulation classique (lent) :
VM → Émulation SATA → Hyperviseur → Disque physique  
Overhead : 10-20%  

Paravirtualisation (rapide) :  
VM → Driver VirtIO → Hyperviseur → Disque physique  
Overhead : 2-5%  
```

**Configuration recommandée :**

```yaml
# Proxmox / KVM
disk:
  bus: virtio
  driver: virtio-blk
  cache: none      # Désactive cache pour garantir fsync
  aio: native      # I/O asynchrone natif
  discard: on      # Support TRIM/DISCARD

# VMware
scsi_controller: pvscsi  # Paravirtual SCSI  
disk_type: thin  # ou thick eager zeroed  
```

#### Cache Policies (Politiques de Cache)

**Options de cache :**

| Mode | Description | Usage PostgreSQL |
|------|-------------|------------------|
| **none** | Pas de cache, I/O direct | ✅ **Production (recommandé)** |
| **writethrough** | Cache lecture seulement | ✅ Acceptable |
| **writeback** | Cache lecture/écriture | ❌ **Dangereux** (perte données) |
| **directsync** | Sync total | ⚠️ Très lent |

**Pourquoi `cache=none` ?**

PostgreSQL gère son propre cache (shared_buffers) et utilise fsync() pour garantir la durabilité. Un cache hyperviseur intermédiaire :
- Risque de perte de données si crash
- Double cache inefficace
- Masque les vraies performances I/O

**Configuration recommandée :**

```bash
# Proxmox
qm set 100 --scsi0 local-lvm:vm-100-disk-0,cache=none,aio=native
```

#### Séparation des Disques Virtuels

**Architecture recommandée :**

```
VM PostgreSQL
├─ vDisk 1 (50 GB) : OS (/boot, /)
│   └─ Stockage : Datastore standard
├─ vDisk 2 (500 GB) : PGDATA (/var/lib/postgresql)
│   └─ Stockage : Flash/NVMe haute performance
├─ vDisk 3 (100 GB) : WAL (/pg_wal)
│   └─ Stockage : Flash/NVMe ultra rapide
└─ vDisk 4 (1 TB) : Backups (/backup)
    └─ Stockage : Capacity (SATA)
```

**Avantages :**
- Optimisation coût/performance par usage
- Isolation des I/O (WAL séparé améliore performances)
- Snapshots sélectifs (snapshot OS sans data)

### Configuration Réseau pour VMs

#### Types d'Interfaces Réseau Virtuelles

**1. Émulation (e1000, rtl8139) - À Éviter**

```
Performance : 100-200 MB/s  
Overhead : 15-20%  
Usage : Compatibilité legacy uniquement  
```

**2. VirtIO (Paravirtualisé) - Recommandé**

```yaml
# Proxmox / KVM
network:
  model: virtio
  bridge: vmbr0
  mtu: 9000  # Jumbo frames

Performance : 1-10 GB/s  
Overhead : 2-5%  
Usage : Production Linux  
```

**3. vmxnet3 (VMware Paravirtualisé) - Recommandé VMware**

```
Performance : 1-10 GB/s  
Overhead : 2-5%  
Usage : Production VMware  
```

**4. SR-IOV (Single Root I/O Virtualization) - Performance Maximale**

```
Performance : 10-25 GB/s (quasi-native)  
Overhead : <1%  
Complexité : Élevée (nécessite carte réseau compatible)  
Usage : Très haute performance, latence critique  
```

**Recommandation PostgreSQL :**
- **Standard :** VirtIO (KVM) ou vmxnet3 (VMware)  
- **Haute performance :** SR-IOV si infrastructure le permet

#### Segmentation Réseau

**Architecture multi-réseaux :**

```
VM PostgreSQL Production
├─ vNIC 1 (eth0) : Réseau production (VLAN 100)
│   └─ Connexions clients applicatifs
├─ vNIC 2 (eth1) : Réseau réplication (VLAN 200)
│   └─ Streaming replication vers standby
├─ vNIC 3 (eth2) : Réseau backup (VLAN 300)
│   └─ pg_basebackup, transferts sauvegardes
└─ vNIC 4 (eth3) : Réseau admin (VLAN 900)
    └─ SSH, monitoring, gestion
```

**Avantages :**
- Isolation trafic (pas d'impact backup sur production)
- Sécurité renforcée (firewall par VLAN)
- QoS (Quality of Service) par réseau
- Troubleshooting simplifié

#### Jumbo Frames

**Activer sur réseau 10 GbE+ :**

```bash
# Dans la VM
ip link set eth0 mtu 9000

# Persistant (Ubuntu)
# /etc/netplan/01-netcfg.yaml
network:
  ethernets:
    eth0:
      mtu: 9000
```

**Gain :** 10-20% de débit réseau, moins d'interruptions CPU.

**⚠️ Important :** Tous les équipements du chemin réseau doivent supporter MTU 9000.

## Configuration PostgreSQL pour Environnement Virtualisé

### Adaptations Spécifiques aux VMs

#### 1. Configuration Mémoire

```ini
# postgresql.conf pour VM 64 GB RAM

# Shared Buffers : Légèrement réduit vs bare metal
shared_buffers = 16GB  # 25% de RAM (vs 30-40% bare metal)
# Raison : Hyperviseur + guest OS consomment RAM

# Work Memory
work_mem = 64MB
# Identique bare metal

# Maintenance Work Memory
maintenance_work_mem = 2GB
# Identique bare metal

# Effective Cache Size : Tenir compte overhead VM
effective_cache_size = 48GB  # 75% de RAM VM (vs 85% bare metal)
# Le guest OS + buffers hyperviseur réduisent cache disponible

# Huge Pages : Toujours recommandé en VM
huge_pages = on
# Configuration hôte ET guest nécessaire
```

**Calcul simplifié :**

```
VM avec 64 GB RAM allouée :
├─ Guest OS : 2-3 GB
├─ Overhead hyperviseur : 0.5 GB (transparent pour guest)
├─ Disponible PostgreSQL : ~61 GB
└─ Configuration :
    ├─ shared_buffers : 16 GB (25%)
    ├─ effective_cache_size : 48 GB (75%)
    └─ Marge pour work_mem × connexions : ~45 GB
```

#### 2. Configuration I/O

```ini
# WAL et Checkpoints
wal_level = replica  
max_wal_size = 16GB  
min_wal_size = 4GB  
wal_compression = zstd  # PostgreSQL 18  
checkpoint_timeout = 15min  
checkpoint_completion_target = 0.9  

# I/O Asynchrone (PostgreSQL 18) - Adapté VM
io_method = 'worker'  # Bénéficie même en VM  
io_async_workers = 8  # Adapter au nombre vCPU  
io_async_queue_depth = 128  # Réduit vs bare metal (256)  

# Raison : L'hyperviseur ajoute sa propre couche I/O asynchrone
# Éviter de saturer la queue
```

#### 3. Paramètres Spécifiques Virtualisation

```ini
# Logging étendu pour troubleshooting
log_line_prefix = '%t [%p-%l] %q%u@%d VM=%h '  
log_lock_waits = on  
log_temp_files = 10MB  # Log fichiers temporaires > 10 MB  

# Statistiques
shared_preload_libraries = 'pg_stat_statements'  
track_io_timing = on  # Important : détecte latences I/O anormales en VM  

# Autovacuum adapté
autovacuum_max_workers = 4  # Réduit vs bare metal  
autovacuum_vacuum_cost_delay = 5ms  # Augmenté (moins agressif)  
# Raison : En VM, autovacuum trop agressif peut impacter VMs voisines
```

## Snapshots et Gestion des VMs

### Snapshots : Usage et Précautions

#### Qu'est-ce qu'un Snapshot ?

Un **snapshot** (instantané) capture l'état complet d'une VM à un instant T :
- Disques virtuels
- RAM (optionnel)
- Configuration

**Analogie :** C'est comme une photo de votre ordinateur. Vous pouvez "revenir dans le temps" à cette photo.

#### Snapshots avec PostgreSQL : Bonnes Pratiques

**⚠️ Attention :** Les snapshots ne sont PAS des backups PostgreSQL valides !

**Problème :**

```
Snapshot classique (dangereux) :
14h00 : Snapshot pris
  ├─ Fichiers disque figés
  ├─ Transactions en cours NON cohérentes
  └─ WAL potentiellement désynchronisé

Restauration snapshot → Base potentiellement corrompue
```

**Solution 1 : Snapshot avec arrêt PostgreSQL**

```bash
# Arrêt propre
sudo systemctl stop postgresql

# Prise snapshot (Proxmox)
qm snapshot 100 before-upgrade

# Redémarrage
sudo systemctl start postgresql
```

- ✅ **Avantages :** Snapshot cohérent, rapide (downtime 1-2 min)  
- ❌ **Inconvénient :** Interruption de service

**Solution 2 : Snapshot avec pg_start_backup() (Obsolète PG 15+)**

```sql
-- Avant PostgreSQL 15
SELECT pg_start_backup('snapshot-label', false, false);
-- Prendre snapshot hyperviseur
SELECT pg_stop_backup();
```

⚠️ **Déprécié** : Remplacé par `pg_backup_start()` mais complexe en VM.

**Solution 3 : Snapshot + WAL archiving (Recommandé)**

```ini
# postgresql.conf
archive_mode = on  
archive_command = 'test ! -f /mnt/wal_archive/%f && cp %p /mnt/wal_archive/%f'  
```

```bash
# Snapshot VM
qm snapshot 100 pitr-base

# Restauration PITR possible depuis snapshot + WAL
```

- ✅ **Avantage :** Point-In-Time Recovery possible  
- ❌ **Complexité :** Configuration avancée

**Recommandation générale :**

Pour PostgreSQL, privilégier :
1. **pg_basebackup** pour backups (pas snapshots)  
2. **Snapshots** uniquement pour :
   - Avant upgrade OS/PostgreSQL (avec arrêt propre)
   - Tests/dev (cohérence non critique)
   - Infrastructure as Code (VM template)

### Clonage de VMs PostgreSQL

**Use cases :**

1. **Environnement de développement :**
   ```bash
   # Proxmox : Cloner VM production vers dev
   qm clone 100 200 --name postgresql-dev
   ```

2. **Tests de migration :**
   ```bash
   # Test upgrade PostgreSQL 17 → 18 sur clone
   qm clone 100 150 --name pg18-test
   ```

**⚠️ Précautions après clonage :**

```bash
# 1. Changer nom d'hôte
sudo hostnamectl set-hostname postgresql-dev

# 2. Modifier configuration réseau (nouvelle IP)

# 3. Régénérer identifiants système
sudo rm -f /var/lib/dbus/machine-id  
sudo dbus-uuidgen --ensure  

# 4. PostgreSQL : Modifier cluster_name
# postgresql.conf
cluster_name = 'postgresql-dev'
```

## Haute Disponibilité avec VMs

### Live Migration (vMotion / Migration à Chaud)

**Qu'est-ce que la Live Migration ?**

Déplacer une VM d'un hôte physique à un autre **sans interruption de service**.

```
Avant Migration :  
Host-1 [VM-PostgreSQL] ──┐  
Host-2 [Vide]          ──┘  

Pendant Migration (invisible pour clients) :  
Host-1 [VM-PostgreSQL] ──┐  
          ↓              │ État transféré
Host-2 [VM-PostgreSQL] ──┘

Après Migration :  
Host-1 [Vide]          ──┐  
Host-2 [VM-PostgreSQL] ──┘  
```

**Technologies :**
- **VMware :** vMotion  
- **Proxmox / KVM :** Live Migration  
- **Hyper-V :** Live Migration

**Durée typique :** 10-30 secondes (downtime < 1 seconde perçu)

**Cas d'usage PostgreSQL :**

1. **Maintenance sans interruption :**
   - Mise à jour firmware hôte
   - Changement matériel
   - Répartition de charge

2. **Optimisation ressources :**
   - DRS (Distributed Resource Scheduler) VMware
   - Déplacer VMs vers hôtes moins chargés

**Prérequis pour Live Migration :**

```
✅ Stockage partagé (SAN, vSAN, Ceph)
✅ Réseau rapide entre hôtes (10 GbE minimum)
✅ CPU compatibles (même famille, ou émulation)
✅ Configuration réseau identique
```

**Impact sur PostgreSQL :**

```
Pendant migration :
├─ Connexions maintenues (du point de vue client)
├─ Légère augmentation latence (~50-100ms pendant 1-2s)
├─ Aucune perte de transaction
└─ Logs PostgreSQL : RAS (transparent)

Surveillance :
└─ pg_stat_activity montre augmentation momentanée wait_event
```

**Limitation :** Ne remplace PAS la réplication PostgreSQL native pour la HA applicative !

---

### HA au Niveau Hyperviseur

#### VMware HA (High Availability)

**Fonctionnement :**

```
Cluster VMware (3 hôtes minimum) :
├─ Host-1 : VM-PostgreSQL-Primary
├─ Host-2 : Standby
├─ Host-3 : Standby

Si Host-1 crash :
└─ vSphere HA redémarre automatiquement VM-PostgreSQL sur Host-2
   └─ Downtime : 2-5 minutes (reboot VM)
```

**Configuration :**

```yaml
HA Settings :
  admission_control: enabled  # Réserve ressources
  restart_priority: highest   # PostgreSQL prioritaire
  host_isolation_response: power_off  # VM éteinte si hôte isolé
  monitoring_sensitivity: high
```

**Avantages :**
- ✅ Automatique  
- ✅ Pas de configuration PostgreSQL spécifique

**Limites :**
- ⚠️ Downtime 2-5 minutes (reboot)  
- ⚠️ Perte transactions en vol (non-committed)  
- ⚠️ Nécessite IP virtuelle ou load balancer

#### Proxmox HA

**Fonctionnement similaire VMware :**

```bash
# Activer HA sur VM PostgreSQL
ha-manager add vm:100 --state started --max_restart 3 --max_relocate 3
```

**Configuration Fencing (STONITH) :**

```bash
# Proxmox nécessite quorum (3 nœuds minimum)
pvecm status  # Vérifier cluster

# Watchdog pour détection panne
echo "softdog" >> /etc/modules  
systemctl enable watchdog-mux.service  
```

### Combinaison HA Hyperviseur + Réplication PostgreSQL

**Architecture optimale :**

```
Datacenter Principal
├─ Host-1 : VM-PostgreSQL-Primary (production)
│   └─ Protected by HA Cluster
├─ Host-2 : VM-PostgreSQL-Standby-1 (local)
│   └─ Streaming Replication async
└─ Host-3 : Standby disponible pour HA

Datacenter Secondaire (DR)
└─ Host-4 : VM-PostgreSQL-Standby-2 (distant)
    └─ Streaming Replication async

Niveaux de protection :
1. HA Hyperviseur : Panne hôte → Redémarrage sur Host-2/3 (2-5 min)
2. Réplication PostgreSQL : Failover standby → Promotion (30s-2min)
3. DR : Disaster → Basculement datacenter secondaire (15-30 min)
```

**Avantages combinés :**
- HA matérielle (panne hôte)
- HA logicielle (corruption PostgreSQL)
- Disaster Recovery (incendie, coupure longue)

## Solutions de Stockage pour VMs

### Stockage Local vs Partagé

#### Stockage Local

**Configuration :**

```
Host-1
├─ NVMe 1-2 : RAID 1 (Hyperviseur)
└─ NVMe 3-8 : RAID 10 (VMs locales)
    ├─ VM-PostgreSQL : 500 GB
    ├─ VM-App : 200 GB
    └─ VM-Web : 100 GB
```

**Avantages :**
- ✅ Performance maximale (latence < 100µs)  
- ✅ Simplicité (pas de SAN)  
- ✅ Coût réduit  
- ✅ Pas de réseau stockage

**Inconvénients :**
- ❌ Pas de Live Migration (sauf avec Ceph/DRBD)  
- ❌ Snapshots limités à l'hôte  
- ❌ Perte données si panne hôte (sans réplication)

**Usage PostgreSQL :**
- VM unique sans besoin mobilité
- Performance critique (trading, analytics temps réel)
- Budget limité

#### SAN (Storage Area Network)

**Architecture :**

```
Cluster Hyperviseur (3 hôtes)
├─ Host-1 ─┐
├─ Host-2 ─┼─ Réseau Fiber Channel / iSCSI ──→ SAN
└─ Host-3 ─┘                                    ├─ Baie All-Flash
                                                 └─ LUNs partagées
```

**Avantages :**
- ✅ Live Migration possible  
- ✅ Snapshots centralisés  
- ✅ Haute disponibilité stockage  
- ✅ Gestion centralisée

**Inconvénients :**
- ❌ Coût élevé (50 000 - 500 000€+)  
- ❌ Complexité (expertise SAN)  
- ❌ Latence supérieure local (+100-500µs)  
- ❌ Point de contention (bande passante partagée)

**Types de SAN :**

| Type | Latence | Débit | Complexité | Coût |
|------|---------|-------|------------|------|
| **Fiber Channel** | <200µs | 16-32 Gb/s | Élevée | €€€€€ |
| **iSCSI (10GbE)** | <500µs | 10 Gb/s | Moyenne | €€€ |
| **iSCSI (25GbE+)** | <300µs | 25-100 Gb/s | Moyenne | €€€€ |
| **NVMe-oF** | <100µs | 25-100 Gb/s | Élevée | €€€€€ |

**Recommandation PostgreSQL :**
- **SAN All-Flash** obligatoire (pas de HDD)  
- **Fiber Channel** ou **NVMe-oF** pour haute performance  
- **iSCSI 25GbE+** en compromis coût/perf

#### vSAN (VMware Virtual SAN)

**Concept :**

```
Cluster VMware (4 hôtes) :  
Chaque hôte contribue ses disques locaux  
├─ Host-1 : 4× NVMe ─┐
├─ Host-2 : 4× NVMe ─┼─ Pool vSAN distribué
├─ Host-3 : 4× NVMe ─┤  (Software-Defined Storage)
└─ Host-4 : 4× NVMe ─┘

Données répliquées entre hôtes (RAID logiciel)
```

**Avantages :**
- ✅ Pas de SAN externe (économie)  
- ✅ Performance excellente (local + distribué)  
- ✅ Scalabilité linéaire (ajout hôtes)  
- ✅ Live Migration supportée

**Inconvénients :**
- ❌ Réseau 10GbE+ obligatoire  
- ❌ Licence VMware coûteuse  
- ❌ Minimum 3-4 hôtes  
- ❌ Overhead CPU/réseau

**Configuration vSAN pour PostgreSQL :**

```yaml
Storage Policy :
  failures_to_tolerate: 1  # Réplication 2 copies
  stripe_width: 2          # Performance lecture
  object_space_reservation: 100%  # Thick provisioning
  disable_site_read_locality: false  # Read local
```

**Performance typique :**
- Latence : 200-500µs
- IOPS : 100 000 - 500 000+
- Débit : 2-10 GB/s par hôte

#### Ceph (Open Source)

**Architecture :**

```
Cluster Ceph (5 serveurs dédiés stockage) :
├─ Node-1 : 8× NVMe ─┐
├─ Node-2 : 8× NVMe ─┤
├─ Node-3 : 8× NVMe ─┼─ Pool RBD (images block)
├─ Node-4 : 8× NVMe ─┤  Réplication 3×
└─ Node-5 : 8× NVMe ─┘

Cluster Proxmox (VMs) :
├─ Host-1 ─┐
├─ Host-2 ─┼─ Réseau 25GbE ──→ Cluster Ceph
└─ Host-3 ─┘
```

**Avantages :**
- ✅ Open source et gratuit  
- ✅ Scalabilité massive (pétaoctets)  
- ✅ Haute disponibilité native  
- ✅ Intégration Proxmox native

**Inconvénients :**
- ❌ Complexité importante  
- ❌ Overhead réseau significatif  
- ❌ Performances variables (dépend config)  
- ❌ Nécessite réseau dédié 25GbE+

**Configuration Ceph pour PostgreSQL :**

```bash
# Pool RBD optimisé PostgreSQL
ceph osd pool create postgresql 128 128  
ceph osd pool set postgresql size 3  # Réplication 3×  
ceph osd pool set postgresql min_size 2  # Minimum 2 copies  
ceph osd pool set postgresql pg_autoscale_mode on  

# Tuning client
rbd_cache = false  # Désactiver cache (PostgreSQL gère)
```

**Performance typique (cluster bien configuré) :**
- Latence : 500-2000µs (3-10× local)
- IOPS : 50 000 - 200 000
- Débit : 5-20 GB/s (total cluster)

**Recommandation :** Excellent pour infrastructure >10 hôtes, nécessite expertise.

## Best Practices PostgreSQL sur VMs

### Checklist Configuration Optimale

#### Niveau Hyperviseur

**CPU :**
- [ ] Type CPU : host-passthrough (ou émulé pour migration)  
- [ ] Topologie : 1 socket × N cœurs  
- [ ] Pas d'overcommit (ratio ≤ 2:1)  
- [ ] Hyper-Threading désactivé si possible  
- [ ] CPU pinning pour VMs critiques uniquement

**RAM :**
- [ ] Réservation mémoire : 100%  
- [ ] Ballooning : désactivé  
- [ ] Swap : désactivé dans la VM  
- [ ] Huge Pages : activé (VMs >16GB)

**Disque :**
- [ ] Type : Thick provisioning  
- [ ] Bus : VirtIO (KVM) ou PVSCSI (VMware)  
- [ ] Cache : none  
- [ ] AIO : native  
- [ ] Discard : activé (TRIM support)  
- [ ] Séparation : OS / PGDATA / WAL / Backup

**Réseau :**
- [ ] Interface : VirtIO (KVM) ou vmxnet3 (VMware)  
- [ ] MTU : 9000 (jumbo frames si 10GbE+)  
- [ ] Multi-NICs : séparation traffic (prod/repli/backup/admin)

#### Niveau OS Guest (dans la VM)

- [ ] OS : Ubuntu Server LTS / Rocky Linux  
- [ ] Kernel : Paramètres sysctl optimisés
  ```bash
  vm.swappiness = 1
  vm.dirty_ratio = 10
  vm.dirty_background_ratio = 3
  ```
- [ ] Huge Pages : configuré
  ```bash
  vm.nr_hugepages = <calculé selon RAM>
  ```
- [ ] Limits : nofile et nproc augmentés  
- [ ] Filesystem : ext4 ou XFS avec noatime

#### Niveau PostgreSQL

- [ ] shared_buffers : 25% RAM VM  
- [ ] effective_cache_size : 75% RAM VM  
- [ ] work_mem : adapté au nombre connexions  
- [ ] maintenance_work_mem : 2-4 GB  
- [ ] io_method : 'worker' ou 'io_uring' (PostgreSQL 18)  
- [ ] huge_pages : on  
- [ ] track_io_timing : on (détecter latence I/O)  
- [ ] log_lock_waits : on  
- [ ] pg_stat_statements : activé

#### Monitoring Spécifique VM

**Métriques hyperviseur à surveiller :**

```sql
-- CPU Ready Time (VMware)
-- % temps où vCPU attend CPU physique
-- Target : < 5%
-- Si > 10% : overcommit CPU ou contention

-- Memory Ballooning
-- Target : 0 MB balloonné
-- Si > 0 : overcommit RAM → problème

-- Disk Latency
-- Target : < 5ms (SSD/Flash)
-- Si > 10ms : problème stockage ou réseau SAN

-- Network Drops
-- Target : 0 packets dropped
-- Si > 0 : saturation réseau ou configuration
```

**Dans PostgreSQL :**

```sql
-- Détecter problèmes I/O (latence anormale en VM)
SELECT
  relname,
  heap_blks_read,
  heap_blks_hit,
  idx_blks_read,
  idx_blks_hit
FROM pg_statio_user_tables  
WHERE heap_blks_read > 100000  
ORDER BY heap_blks_read DESC;  

-- Si ratio blks_hit/blks_read < 99% : cache insuffisant ou I/O lent
```

## Troubleshooting : Problèmes Courants

### Performance Dégradée Soudaine

**Symptôme :** PostgreSQL lent sans raison apparente.

**Diagnostics :**

1. **Vérifier overcommit CPU :**
   ```bash
   # Sur l'hôte
   # VMware
   esxtop  # Regarder %RDY (CPU Ready)

   # Proxmox / KVM
   top  # Regarder %steal (temps volé aux VMs)
   ```

   **Solution :** Réduire nombre vCPU ou ajouter hôtes au cluster.

2. **Vérifier ballooning mémoire :**
   ```bash
   # Dans la VM
   free -h  # Regarder utilisation swap
   vmstat 1  # Colonnes si/so (swap in/out)
   ```

   **Solution :** Désactiver ballooning, réserver mémoire.

3. **Vérifier latence stockage :**
   ```bash
   # Dans la VM
   iostat -x 1
   # Regarder await et svctm
   # Si await > 10ms : problème stockage
   ```

   **Solution :** Vérifier saturation SAN ou réseau stockage.

4. **Vérifier "noisy neighbors" :**
   ```bash
   # Sur l'hôte : identifier VMs consommant trop
   # Proxmox
   pvesh get /cluster/resources --type vm

   # Migrer VMs ou ajouter limites CPU/IO
   ```

### Corruption Après Snapshot

**Symptôme :** Base corrompue après restauration snapshot.

**Cause :** Snapshot pris pendant écritures actives.

**Prévention :**
1. Toujours arrêter PostgreSQL avant snapshot  
2. Ou utiliser pg_basebackup au lieu de snapshots  
3. Configurer WAL archiving pour PITR

**Récupération :**
```bash
# Si corruption détectée
pg_resetwal -f /var/lib/postgresql/18/main
# ⚠️ Perte potentielle de transactions

# Restaurer depuis backup sain
pg_restore -d dbname /backup/last_good_backup.dump
```

### Live Migration Échoue

**Causes fréquentes :**

1. **Stockage non partagé :**
   - Solution : Migrer vers SAN/vSAN/Ceph

2. **CPUs incompatibles :**
   - Solution : Utiliser émulation CPU au lieu de passthrough

3. **RAM insuffisante sur cible :**
   - Solution : Libérer RAM sur hôte de destination

4. **Réseau saturé :**
   - Solution : Utiliser réseau dédié 10GbE+ pour migrations

- Snapshots pour rollback rapide

✅ **Environnements multiples**
- Production + Staging + Dev sur même hardware
- Isolation sécurisée
- Clonage rapide

✅ **Croissance incertaine**
- Ajout ressources à chaud
- Scalabilité progressive
- Pas de sur-dimensionnement initial

✅ **Équipe technique moyenne**
- Outils d'administration graphiques
- Moins d'expertise bare metal nécessaire
- Automatisation facilitée

### Éviter les VMs Si

❌ **Performance absolue requise**
- Trading haute fréquence
- Analytics temps réel extrême
- Overhead inacceptable

❌ **Infrastructure très petite (1-2 serveurs)**
- Coût overhead non justifié
- Complexité inutile

❌ **Conformité interdisant virtualisation**
- Certifications spécifiques
- Réglementations secteur bancaire strict

### Points Clés à Retenir

1. **Performance :** VMs modernes offrent 90-95% des performances bare metal avec hyperviseur optimisé.

2. **Configuration :** Les réglages hyperviseur (CPU pinning, memory reservation, cache policies) sont **critiques** pour PostgreSQL.

3. **Stockage :** Le choix stockage (local, SAN, vSAN, Ceph) impacte davantage que la virtualisation elle-même.

4. **HA :** Les VMs simplifient radicalement la haute disponibilité vs bare metal.

5. **Coût :** Virtualisé devient rentable à partir de 5+ serveurs, grâce à la consolidation.

6. **PostgreSQL 18 :** Les améliorations I/O asynchrones réduisent l'écart de performance avec bare metal.

**Recommandation générale :**

Pour la majorité des cas d'usage PostgreSQL en entreprise, **la virtualisation avec Proxmox (open source) ou VMware vSphere (enterprise)** offre le meilleur compromis flexibilité/performance/coût.

Réservez le bare metal aux charges ultra-critiques ou infrastructures <3 serveurs.
