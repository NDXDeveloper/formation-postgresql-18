# 19.2.3. Google Cloud SQL et AlloyDB

## Introduction

**Google Cloud Platform (GCP)** propose deux solutions distinctes pour héberger PostgreSQL dans le cloud : **Cloud SQL for PostgreSQL** et **AlloyDB for PostgreSQL**. Cette dualité reflète une stratégie similaire à celle d'AWS (RDS vs Aurora), mais avec des différences importantes dans l'approche technique.

Dans cette section, nous allons explorer ces deux offres, comprendre leurs architectures respectives, leurs différences fondamentales et comment choisir entre elles selon vos besoins.

---

## Vue d'ensemble de l'Offre Google Cloud

Google Cloud propose **deux options principales** pour PostgreSQL :

1. **Cloud SQL for PostgreSQL** : Service managé standard, mature et éprouvé  
2. **AlloyDB for PostgreSQL** : Service nouvelle génération, ultra-performant (lancé en 2022)

## Google Cloud SQL for PostgreSQL

### Qu'est-ce que Cloud SQL ?

**Cloud SQL** est le service de base de données relationnelle managé de Google Cloud, supportant PostgreSQL, MySQL et SQL Server. C'est une solution mature, fiable et largement utilisée.

### Concepts Clés Google Cloud

Avant d'approfondir, voici quelques concepts GCP importants :

#### Project (Projet)
- **Unité d'organisation** de base dans GCP
- Toutes les ressources appartiennent à un projet
- Facturation et quotas par projet

#### VPC (Virtual Private Cloud)
- **Réseau privé virtuel** global (s'étend sur plusieurs régions)
- Sous-réseaux régionaux
- Firewall rules pour contrôle d'accès

#### Zone et Region
- **Region** : Localisation géographique (ex: europe-west1 = Belgique)  
- **Zone** : Datacenter isolé dans une région (ex: europe-west1-b)
- Cloud SQL peut utiliser plusieurs zones pour HA

### Caractéristiques Principales de Cloud SQL PostgreSQL

#### 1. Versions PostgreSQL Supportées

Cloud SQL supporte activement :
- PostgreSQL **11** (support étendu)
- PostgreSQL **12**
- PostgreSQL **13**
- PostgreSQL **14**
- PostgreSQL **15**
- PostgreSQL **16**
- PostgreSQL **17** (selon disponibilité)

**Particularité Google :** Mises à jour mineures automatiques par défaut (configurable)

#### 2. Tiers de Machine (Types d'Instances)

Cloud SQL propose différentes configurations :

##### Shared-core (Partagé)
- **db-f1-micro** : 0,6 GB RAM (1 vCPU partagé)  
- **db-g1-small** : 1,7 GB RAM (1 vCPU partagé)  
- **Cas d'usage** : Développement, tests, très petites charges  
- **Limitation** : Performance non garantie (partagé avec d'autres)

##### Dedicated-core (Dédié)
Plusieurs familles disponibles :

**Standard (N1, N2, N2D)**
- **n1-standard-1** : 1 vCPU, 3,75 GB RAM  
- **n1-standard-2** : 2 vCPU, 7,5 GB RAM  
- **n1-standard-4** : 4 vCPU, 15 GB RAM
- Jusqu'à **n1-standard-96** : 96 vCPU, 360 GB RAM

**High-Memory (N1, N2, N2D)**
- **n1-highmem-2** : 2 vCPU, 13 GB RAM  
- **n1-highmem-4** : 4 vCPU, 26 GB RAM
- Jusqu'à **n1-highmem-96** : 96 vCPU, 624 GB RAM
- Ratio : 1 vCPU pour ~6,5 GB RAM

**High-CPU (N1, N2)**
- **n1-highcpu-2** : 2 vCPU, 1,8 GB RAM
- Pour charges CPU-intensives
- Moins courant pour PostgreSQL

**Exemple de dimensionnement :**

| Charge | Machine Type | Specs | Prix approximatif |
|--------|--------------|-------|-------------------|
| Dev/Test | db-g1-small | 1 vCPU partagé, 1,7 GB | ~25€/mois |
| Production Petite | n1-standard-2 | 2 vCPU, 7,5 GB | ~100€/mois |
| Production Moyenne | n1-standard-4 | 4 vCPU, 15 GB | ~200€/mois |
| Production Intensive | n1-highmem-8 | 8 vCPU, 52 GB | ~550€/mois |

#### 3. Stockage

Cloud SQL propose deux types de stockage :

##### SSD (Solid State Drive) - Recommandé
- **Performance** : Jusqu'à 60 000 IOPS et 1 200 MB/s  
- **Latence** : Très faible (millisecondes)  
- **Taille** : 10 GB à 64 TB  
- **Auto-scaling** : Croissance automatique possible  
- **Cas d'usage** : Production, charges I/O intensives

##### HDD (Hard Disk Drive) - Legacy
- **Performance** : Jusqu'à 1 200 IOPS  
- **Latence** : Plus élevée  
- **Taille** : 10 GB à 64 TB  
- **Coût** : ~50% moins cher que SSD  
- **Cas d'usage** : Développement, données froides, backup

**Important :** Le type de stockage ne peut pas être changé après création. Il faut migrer vers une nouvelle instance.

**Performance Stockage SSD :**

| Taille Stockage | IOPS Lecture | IOPS Écriture | Débit |
|-----------------|--------------|---------------|-------|
| 10-500 GB | 3000 | 3000 | 624 MB/s |
| 500-1000 GB | 6000 | 6000 | 624 MB/s |
| 1-2 TB | 12000 | 12000 | 624 MB/s |
| 2-4 TB | 24000 | 24000 | 1200 MB/s |
| 4+ TB | 60000 | 60000 | 1200 MB/s |

#### 4. Haute Disponibilité (HA)

Cloud SQL propose une configuration **Regional HA** :

##### Configuration HA
```
┌───────────────────────────────────────────────────┐
│         Cloud SQL Regional Configuration          │
│                                                   │
│  ┌────────────────┐         ┌────────────────┐    │
│  │  Zone A        │         │  Zone B        │    │
│  │                │         │                │    │
│  │  ┌──────────┐  │         │  ┌──────────┐  │    │
│  │  │ Primary  │  │ Sync    │  │ Standby  │  │    │
│  │  │ Instance │◄─┼────────►│  │ Instance │  │    │
│  │  └────┬─────┘  │  Repl   │  └──────────┘  │    │
│  │       │        │         │                │    │
│  │       ▼        │         │                │    │
│  │  ┌──────────┐  │         │  ┌──────────┐  │    │
│  │  │Regional  │  │         │  │Regional  │  │    │
│  │  │Persistent│◄─┼────────►│  │Persistent│  │    │
│  │  │  Disk    │  │  Mirror │  │  Disk    │  │    │
│  │  └──────────┘  │         │  └──────────┘  │    │
│  └────────────────┘         └────────────────┘    │
│                                                   │
│  En cas de panne Zone A : Failover vers Zone B    │
│  Durée : 30-60 secondes                           │
└───────────────────────────────────────────────────┘
```

**Caractéristiques HA :**
- Réplication **synchrone** entre zones
- **Failover automatique** en 30-60 secondes  
- **RPO = 0** (aucune perte de données)  
- **RTO < 1 minute**
- Regional Persistent Disk : répliqué automatiquement
- Pas d'impact sur les performances

**Coût HA :**
- Compute : 2× le coût de l'instance
- Stockage : Inclus dans le prix (Regional Disk)

**Sans HA :**
- Instance **Zonal** : Dans une seule zone
- Moins cher (pas de doublement compute)
- Vulnérable aux pannes de zone
- **Non recommandé pour production**

#### 5. Read Replicas (Réplicas de Lecture)

Cloud SQL supporte plusieurs types de réplicas :

##### Read Replica Standard
- Réplication **asynchrone** depuis l'instance primaire
- Peut être dans la **même région** ou **cross-region**
- Jusqu'à **10 Read Replicas** par instance primaire
- Utilisé pour scalabilité en lecture

##### Cross-Region Read Replica
- Réplica dans une **région différente**
- Utile pour :
  - Géo-distribution (latence réduite)
  - Disaster Recovery
  - Conformité réglementaire (données dans pays spécifique)

##### Cascading Read Replica
- Réplica qui se réplique depuis un autre **Read Replica**
- Permet de créer des chaînes de réplication
- Réduit la charge sur l'instance primaire

**Lag de Réplication :**
- Même région : Quelques secondes
- Cross-region : Variable (réseau, charge)
- Visible via métriques Cloud Monitoring

#### 6. Sauvegardes et Restauration

##### Sauvegardes Automatiques
- **Fréquence** : Quotidienne pendant fenêtre configurable  
- **Rétention** : 1 à 365 jours (7 jours par défaut)  
- **Type** : Snapshot complet + logs binaires  
- **Stockage** : Multi-régional automatiquement (résilience)  
- **Impact** : Minimal sur performances (I/O légèrement augmentées)

##### Point-in-Time Recovery (PITR)
- Restauration à **n'importe quelle seconde** durant période de rétention
- Basé sur sauvegardes + logs de transactions
- Crée une **nouvelle instance** (l'originale reste inchangée)
- Granularité : À la seconde près

##### Sauvegardes On-Demand
- Snapshots manuels à tout moment
- Conservation indéfinie (jusqu'à suppression manuelle)
- Utile avant migrations, upgrades, changements majeurs

##### Export vers Cloud Storage
- Export SQL (pg_dump) ou CSV
- Stocké dans Google Cloud Storage (GCS)
- Permet des restaurations cross-project ou cross-cloud

**Processus de Restauration :**
```
1. Sélection du point de restauration
   ↓
2. Cloud SQL crée une nouvelle instance
   ↓
3. Restauration des données (peut prendre du temps)
   ↓
4. Validation de l'intégrité
   ↓
5. Basculement de l'application
   ↓
6. (Optionnel) Suppression de l'ancienne instance
```

#### 7. Options de Connectivité

Cloud SQL propose plusieurs méthodes de connexion :

##### Public IP (Adresse IP Publique)
- Instance accessible via Internet
- Sécurisé par **Authorized Networks** (whitelist IP)
- SSL/TLS obligatoire
- **Moins sécurisé**, mais plus simple

##### Private IP (VPC Peering)
- Instance déployée dans votre **VPC**
- Accessible uniquement depuis le VPC (ou VPCs appairés)
- Pas d'exposition Internet
- **Recommandé pour production**

##### Cloud SQL Proxy
- Outil client Google pour connexion sécurisée
- Chiffrement automatique
- Pas besoin de gérer SSL/TLS manuellement
- Authentification via IAM
- Fonctionne avec Public ou Private IP

**Architecture Private IP :**

```
│  │   │          │   │  Engine  │          │ │
```


#### 8. Sécurité

##### Authentification
- **PostgreSQL native** : Utilisateur/mot de passe  
- **Cloud SQL IAM Authentication** :
  - Connexion via IAM (sans mot de passe PostgreSQL)
  - Intégration avec Google Cloud Identity
  - Tokens temporaires

**Exemple IAM Authentication :**
```bash
# L'utilisateur utilise son compte Google Cloud
gcloud sql connect my-instance --user=user@example.com
```

##### Chiffrement
- **En transit** : SSL/TLS obligatoire par défaut
  - Certificats gérés automatiquement
  - TLS 1.2+ uniquement
- **Au repos** : Chiffrement **automatique** et **transparent**
  - Utilise Google-managed encryption keys (par défaut)
  - **Customer-managed encryption keys (CMEK)** via Cloud KMS
  - Chiffrement du stockage, backups, réplicas

##### Audit Logging
- **Cloud Audit Logs** : Enregistre toutes les opérations administratives  
- **Database Audit Plugin** : Audit des connexions et requêtes SQL
  - pgaudit extension disponible
  - Export vers Cloud Logging

##### Firewall et Réseau
- **Authorized Networks** : Whitelist d'adresses IP (Public IP)  
- **VPC Firewall Rules** : Contrôle d'accès réseau (Private IP)  
- **Private Service Connection** : Isolation complète

##### Compliance
- Certifications : ISO 27001, SOC 2/3, PCI DSS, HIPAA, GDPR
- Data residency : Choix de la région pour conformité

## AlloyDB for PostgreSQL

### Qu'est-ce qu'AlloyDB ?

**AlloyDB** est le service de base de données **nouvelle génération** de Google Cloud, lancé en 2022. Il s'agit d'un moteur **propriétaire** développé par Google, compatible avec PostgreSQL, et optimisé pour des performances exceptionnelles.

### Philosophie d'AlloyDB

AlloyDB est conçu pour être **4× plus rapide** que Cloud SQL (selon Google) pour les charges transactionnelles, et **100× plus rapide** pour les requêtes analytiques grâce à son moteur de colonnes intégré.

### Architecture d'AlloyDB

AlloyDB repose sur une architecture **séparation compute/storage** similaire à Aurora AWS :

```
┌───────────────────────────────────────────────────────────┐
│                 AlloyDB Cluster Architecture              │
│                                                           │
│  ┌──────────────────────────────────────────────────────┐ │
│  │              Cluster (Unité logique)                 │ │
│  │                                                      │ │
│  │  ┌────────────────┐       ┌────────────────┐         │ │
│  │  │   Primary      │       │   Read Pool    │         │ │
│  │  │   Instance     │       │   Instance 1   │         │ │
│  │  │   (Lecture +   │       │   (Lecture     │         │ │
│  │  │   Écriture)    │       │   seule)       │         │ │
│  │  └───────┬────────┘       └───────┬────────┘         │ │
│  │          │                        │                  │ │
│  │          │      ┌────────────────┐│                  │ │
│  │          │      │   Read Pool    ││                  │ │
│  │          │      │   Instance 2   ││                  │ │
│  │          │      │   (Analytique) ││                  │ │
│  │          │      └───────┬────────┘│                  │ │
│  │          └──────────┬───┴─────────┘                  │ │
│  │                     │                                │ │
│  └─────────────────────┼────────────────────────────────┘ │
│                        │                                  │
│  ┌─────────────────────▼────────────────────────────────┐ │
│  │       Shared Distributed Storage Layer               │ │
│  │                                                      │ │
│  │  • Réplication automatique (3 copies minimum)        │ │
│  │  • Stockage log-structured (LSM)                     │ │
│  │  • Cache distribué intelligent                       │ │
│  │  • Columnar engine intégré (analytique)              │ │
│  │  • Capacité : Jusqu'à 64 TB auto-scaling             │ │
│  └──────────────────────────────────────────────────────┘ │
└───────────────────────────────────────────────────────────┘
```

### Différences Fondamentales avec Cloud SQL

| Aspect | Cloud SQL | AlloyDB |
|--------|-----------|---------|
| **Moteur** | PostgreSQL standard | Moteur propriétaire Google |
| **Architecture** | Instance + Persistent Disk | Compute/Storage séparés |
| **Réplication** | Streaming replication | Log-based, storage-level |
| **Performances OLTP** | Baseline | 4× plus rapide (selon Google) |
| **Performances OLAP** | Standard | 100× plus rapide (columnar) |
| **Scaling Lecture** | Jusqu'à 10 réplicas | Read Pool illimité |
| **Stockage Max** | 64 TB | 64 TB (auto-scaling) |
| **Compatibilité** | 100% PostgreSQL | ~100% (quelques extensions manquantes) |
| **Coût** | Modéré | 2-3× plus cher |

### Caractéristiques Principales d'AlloyDB

#### 1. Versions PostgreSQL Supportées

AlloyDB supporte actuellement :
- PostgreSQL **14**
- PostgreSQL **15**
- PostgreSQL **16**

**Note :** Support de versions plus récent que PostgreSQL standard (Google ajoute rapidement les nouvelles versions).

#### 2. Types d'Instances

##### Primary Instance (Instance Primaire)
- **Lecture + Écriture**
- Machine types disponibles :
  - **2 vCPU, 16 GB RAM** (minimum)  
  - **4 vCPU, 32 GB RAM**  
  - **8 vCPU, 64 GB RAM**
  - Jusqu'à **128 vCPU, 864 GB RAM**

**Ratio fixe** : 1 vCPU = 8 GB RAM (optimisé mémoire)

##### Read Pool Instances (Instances en Lecture)
- **Lecture seule**
- Réplication **quasi-synchrone** (lag < 10ms)
- **Nombre illimité** d'instances (scaling horizontal)
- Même choix de machine types

##### Configuration HA
- **Instance primaire** dans une zone  
- **Standby synchrone** dans une autre zone (automatique)
- Failover **automatique** en ~10-20 secondes
- **HA toujours activée** (pas d'option pour la désactiver)

#### 3. Columnar Engine (Moteur Colonnes)

**Innovation majeure d'AlloyDB :**

##### Fonctionnement
- Données stockées **en colonnes** en plus du format ligne traditionnel
- **Transparent** : Aucune modification applicative nécessaire  
- **Optimiseur intelligent** : Choisit automatiquement le format optimal  
- **Synchronisation automatique** : Conversion ligne→colonne en arrière-plan

##### Avantages
- **100× plus rapide** pour requêtes analytiques (agrégations, scans)
- Compression supérieure
- Idéal pour :
  - Rapports et dashboards
  - Analytics en temps réel
  - OLTP + OLAP mixte (HTAP)

**Exemple de Gain :**
```sql
-- Requête analytique sur 1 milliard de lignes
SELECT product_category, SUM(sales_amount)  
FROM sales  
WHERE sale_date BETWEEN '2024-01-01' AND '2024-12-31'  
GROUP BY product_category;  

Cloud SQL :    45 secondes  
AlloyDB :      0.5 secondes (90× plus rapide)  
```

##### Activation
- **Automatique** par défaut
- Peut être désactivé par table si nécessaire
- Monitoring du ratio ligne/colonne via métriques

#### 4. Intelligent Query Planner

AlloyDB intègre un **planificateur de requêtes avancé** :

##### Machine Learning Intégré
- **Auto-tuning** : Apprend des patterns de requêtes  
- **Prédiction de coût** : Meilleure estimation des plans d'exécution  
- **Adaptation dynamique** : Ajuste les stratégies selon la charge

##### Optimisations Spécifiques
- **Join optimization** : Réordonnancement intelligent  
- **Index hints** : Suggestions automatiques  
- **Parallel query execution** : Parallélisation agressive

#### 5. Caching Intelligent

##### Multi-Level Cache
- **Shared cache** : Partagé entre toutes les instances du cluster  
- **Local cache** : Par instance read pool  
- **Préchargement intelligent** : Anticipe les données nécessaires

##### Résultat
- **Hit ratio élevé** : Moins d'I/O sur le stockage  
- **Latence réduite** : Réponses sub-milliseconde  
- **Scalabilité** : Ajout d'instances read pool améliore le cache global

#### 6. Continuous Protection (Protection Continue)

##### Point-in-Time Recovery Amélioré
- **Granularité** : À la seconde près  
- **Rétention** : Jusqu'à 35 jours  
- **Restauration rapide** : Minutes (vs heures pour dump/restore)

##### Backup Continu
- Sauvegardes **incrémentales continues**
- Stockage dans Cloud Storage multi-régional
- **Aucun impact** sur les performances
- Restore vers n'importe quel point dans la période de rétention

##### Clone de Base de Données
- **Instant cloning** : Création en quelques secondes  
- **Copy-on-write** : Pas de duplication initiale
- Idéal pour :
  - Tests de migration
  - Développement
  - Analyse de données

#### 7. Networking et Connectivité

##### Private Services Access (Obligatoire)
- AlloyDB **nécessite** un VPC avec Private Services Access
- Pas d'option Public IP
- **Sécurité maximale** : Aucune exposition Internet

##### Connection Pooling Intégré
- **PgBouncer intégré** dans le cluster
- Gestion automatique des connexions
- Réduit la charge sur l'instance primaire

##### Endpoints
- **Primary endpoint** : Toujours pointe vers l'instance primaire  
- **Read pool endpoint** : Load balancing automatique entre read instances
- Basculement transparent en cas de failover

#### 8. Sécurité AlloyDB

##### Authentification
- **IAM Database Authentication** (recommandé)
- PostgreSQL native (utilisateur/mot de passe)
- Intégration Google Cloud Identity

##### Chiffrement
- **En transit** : TLS 1.2+ obligatoire  
- **Au repos** : Chiffrement automatique
  - Google-managed keys (par défaut)
  - Customer-managed keys (CMEK) via Cloud KMS
- Chiffrement des sauvegardes et clones

##### Audit et Compliance
- Cloud Audit Logs pour opérations administratives
- pgaudit extension pour audit SQL
- Certifications : ISO, SOC, PCI DSS, HIPAA, GDPR

#### 9. Database Flags

Comme Cloud SQL, AlloyDB permet de configurer PostgreSQL via flags :

##### Flags Supportés
- Majorité des paramètres PostgreSQL standard
- Quelques flags spécifiques AlloyDB (columnar engine, caching)

##### Limitations
- Certains paramètres liés au stockage ne sont pas modifiables
- Paramètres HA gérés automatiquement

#### 10. Extensions PostgreSQL

AlloyDB supporte la plupart des extensions PostgreSQL :

##### Extensions Disponibles

```
Core :
- pg_stat_statements
- pgcrypto
- uuid-ossp
- citext
- hstore

PostGIS : Complet

IA/ML :
- pgvector (vecteurs pour ML)

Autres :
- pg_trgm
- fuzzystrmatch
- btree_gin, btree_gist
```

**Limitation :** Certaines extensions tierces peuvent ne pas être supportées.

#### 11. Monitoring AlloyDB

##### Métriques Avancées
- **Métriques standards** : CPU, RAM, IOPS, Connections  
- **Métriques AlloyDB spécifiques** :
  - Columnar engine hit ratio
  - Cache efficiency
  - Replication lag (< 10ms typiquement)
  - Query performance insights

##### Query Insights
- **Analyse automatique** des requêtes lentes
- Recommandations d'optimisation
- Détection d'anomalies

##### Integration Cloud Monitoring
- Dashboards préconfigurés
- Alertes personnalisables
- Export vers BigQuery pour analyse avancée

- Validation automatique
- Monitoring intégré
- Gratuit pour migrations vers GCP

#### 2. pg_dump / pg_restore

Méthode traditionnelle pour petites bases :

```bash
# Export depuis source
```

**Cas d'usage :**
- Bases < 50 GB
- Migration simple
- Pas de contrainte de downtime strict
