# 4.4.5. Types Géométriques, hstore et ltree

## Introduction

PostgreSQL offre des types de données spécialisés pour gérer des cas d'usage avancés : formes géométriques, paires clé-valeur et structures hiérarchiques. Ces types sont moins courants dans les applications quotidiennes mais extrêmement puissants pour des besoins spécifiques.

Dans cette section, nous allons explorer :
- **Types géométriques** : Stocker et manipuler des formes géométriques simples  
- **hstore** : Stocker des paires clé-valeur (alternative légère à JSONB)  
- **ltree** : Représenter et requêter des hiérarchies et structures arborescentes

**Note importante :** Pour la plupart des applications géospatiales modernes, vous utiliserez l'extension **PostGIS** plutôt que les types géométriques natifs. PostGIS sera couvert dans une section ultérieure.

---

## 1. Types Géométriques

### Vue d'Ensemble

PostgreSQL fournit des types natifs pour représenter des objets géométriques en 2D. Ces types sont simples et intégrés directement dans le moteur.

| Type | Description | Exemple |
|------|-------------|---------|
| `POINT` | Un point (x, y) | `(1, 2)` |
| `LINE` | Ligne infinie (Ax + By + C = 0) | `{A, B, C}` |
| `LSEG` | Segment de ligne (deux points) | `[(0,0),(1,1)]` |
| `BOX` | Rectangle (deux points opposés) | `(1,1),(3,3)` |
| `PATH` | Chemin (suite de points) | `[(0,0),(1,1),(2,0)]` |
| `POLYGON` | Polygone fermé | `((0,0),(1,0),(1,1),(0,1))` |
| `CIRCLE` | Cercle (centre + rayon) | `<(0,0),5>` |

### POINT (Point)

Un **POINT** représente une position en 2D avec des coordonnées (x, y).

```sql
-- Créer une table avec des points
CREATE TABLE lieux (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    position POINT
);

-- Insérer des points (syntaxe : (x, y))
INSERT INTO lieux (nom, position) VALUES
    ('Tour Eiffel', POINT(48.858844, 2.294351)),
    ('Louvre', POINT(48.860611, 2.337644)),
    ('Notre-Dame', POINT(48.852968, 2.349902));

-- Ou avec notation texte
INSERT INTO lieux (nom, position) VALUES
    ('Arc de Triomphe', '(48.873792, 2.295028)');

SELECT * FROM lieux;
```

#### Opérations sur les Points

```sql
-- Accéder aux coordonnées
SELECT
    nom,
    position,
    position[0] AS longitude,  -- x
    position[1] AS latitude    -- y
FROM lieux;

-- Distance entre deux points (distance euclidienne)
SELECT
    l1.nom AS lieu1,
    l2.nom AS lieu2,
    l1.position <-> l2.position AS distance
FROM lieux l1, lieux l2  
WHERE l1.id < l2.id;  

-- Point le plus proche
SELECT nom, position  
FROM lieux  
WHERE nom != 'Tour Eiffel'  
ORDER BY position <-> POINT(48.858844, 2.294351)  
LIMIT 1;  
```

### BOX (Rectangle)

Un **BOX** représente un rectangle défini par deux coins opposés.

```sql
CREATE TABLE zones (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100),
    perimetre BOX
);

-- Insérer des rectangles
-- Syntaxe : (coin_haut_droit, coin_bas_gauche)
INSERT INTO zones (nom, perimetre) VALUES
    ('Zone A', BOX(POINT(0, 0), POINT(10, 10))),
    ('Zone B', '((5, 5), (15, 15))');

SELECT * FROM zones;
```

#### Opérations sur les BOX

```sql
-- Aire du rectangle
SELECT
    nom,
    area(perimetre) AS aire
FROM zones;

-- Vérifier si un point est dans un rectangle
SELECT nom  
FROM zones  
WHERE perimetre @> POINT(7, 7);  -- Le point (7,7) est-il dans la zone ?  

-- Vérifier si deux rectangles se chevauchent
SELECT z1.nom, z2.nom  
FROM zones z1, zones z2  
WHERE z1.id < z2.id  
  AND z1.perimetre && z2.perimetre;  -- Chevauchement
```

### CIRCLE (Cercle)

Un **CIRCLE** est défini par un centre (point) et un rayon.

```sql
CREATE TABLE zones_circulaires (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100),
    zone CIRCLE
);

-- Insérer des cercles
-- Syntaxe : <(x, y), rayon>
INSERT INTO zones_circulaires (nom, zone) VALUES
    ('Zone Centre', CIRCLE(POINT(0, 0), 5)),
    ('Zone Nord', '<(0, 10), 3>');

SELECT * FROM zones_circulaires;
```

#### Opérations sur les CIRCLE

```sql
-- Aire du cercle
SELECT
    nom,
    area(zone) AS aire,
    radius(zone) AS rayon,
    center(zone) AS centre
FROM zones_circulaires;

-- Vérifier si un point est dans un cercle
SELECT nom  
FROM zones_circulaires  
WHERE zone @> POINT(2, 2);  

-- Distance entre centres de deux cercles
SELECT
    z1.nom,
    z2.nom,
    center(z1.zone) <-> center(z2.zone) AS distance_centres
FROM zones_circulaires z1, zones_circulaires z2  
WHERE z1.id < z2.id;  
```

### POLYGON (Polygone)

Un **POLYGON** est une forme fermée définie par une suite de points.

```sql
CREATE TABLE parcelles (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100),
    forme POLYGON
);

-- Insérer des polygones
-- Triangle
INSERT INTO parcelles (nom, forme) VALUES
    ('Parcelle Triangle', '((0,0), (4,0), (2,3))');

-- Carré
INSERT INTO parcelles (nom, forme) VALUES
    ('Parcelle Carrée', '((0,0), (5,0), (5,5), (0,5))');

-- Pentagone irrégulier
INSERT INTO parcelles (nom, forme) VALUES
    ('Parcelle Complexe', '((0,0), (5,0), (6,3), (3,6), (-1,3))');

SELECT * FROM parcelles;
```

#### Opérations sur les POLYGON

```sql
-- Aire du polygone
SELECT
    nom,
    area(forme) AS aire,
    npoints(forme) AS nombre_points
FROM parcelles;

-- Vérifier si un point est dans un polygone
SELECT nom  
FROM parcelles  
WHERE forme @> POINT(2, 2);  

-- Vérifier si deux polygones se chevauchent
SELECT p1.nom, p2.nom  
FROM parcelles p1, parcelles p2  
WHERE p1.id < p2.id  
  AND p1.forme && p2.forme;
```

### PATH (Chemin)

Un **PATH** représente un chemin (suite de points reliés). Il peut être ouvert ou fermé.

```sql
CREATE TABLE trajets (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100),
    parcours PATH
);

-- Chemin ouvert : [(x1,y1), (x2,y2), ...]
INSERT INTO trajets (nom, parcours) VALUES
    ('Route A', '[(0,0), (1,1), (2,1), (3,2)]');

-- Chemin fermé : ((x1,y1), (x2,y2), ...)
INSERT INTO trajets (nom, parcours) VALUES
    ('Circuit B', '((0,0), (2,0), (2,2), (0,2))');

SELECT * FROM trajets;
```

### Fonctions et Opérateurs Géométriques

```sql
-- Distance entre objets
SELECT point1 <-> point2 AS distance;

-- Contient (est à l'intérieur de)
SELECT box @> point;  -- Le point est-il dans le rectangle ?  
SELECT circle @> point;  -- Le point est-il dans le cercle ?  

-- Chevauche (intersecte)
SELECT box1 && box2;  -- Les rectangles se chevauchent-ils ?

-- À gauche de / à droite de
SELECT point1 << point2;  -- point1 est strictement à gauche de point2  
SELECT point1 >> point2;  -- point1 est strictement à droite de point2  

-- Au-dessus / en-dessous
SELECT point1 |<< point2;  -- point1 est strictement en dessous de point2  
SELECT point1 |>> point2;  -- point1 est strictement au-dessus de point2  

-- Centre d'un objet
SELECT center(circle);  
SELECT center(box);  

-- Aire
SELECT area(circle);  
SELECT area(box);  
SELECT area(polygon);  

-- Rayon
SELECT radius(circle);

-- Nombre de points
SELECT npoints(path);  
SELECT npoints(polygon);  
```

### Cas d'Usage des Types Géométriques

#### 1. Application de Cartographie Simple

```sql
CREATE TABLE batiments (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    emplacement POINT,
    emprise BOX,  -- Rectangle du bâtiment
    type VARCHAR(50)
);

INSERT INTO batiments (nom, emplacement, emprise, type) VALUES
    ('Bibliothèque', POINT(10, 10), BOX(POINT(9, 9), POINT(11, 11)), 'Public'),
    ('École', POINT(20, 20), BOX(POINT(18, 18), POINT(22, 22)), 'Éducation');

-- Trouver les bâtiments dans une zone
SELECT nom  
FROM batiments  
WHERE emprise && BOX(POINT(0, 0), POINT(15, 15));  

-- Bâtiments les plus proches d'un point
SELECT nom, type  
FROM batiments  
ORDER BY emplacement <-> POINT(12, 12)  
LIMIT 3;  
```

#### 2. Zones de Livraison

```sql
CREATE TABLE zones_livraison (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100),
    zone CIRCLE,
    frais_livraison NUMERIC(5, 2)
);

INSERT INTO zones_livraison (nom, zone, frais_livraison) VALUES
    ('Centre-ville', CIRCLE(POINT(0, 0), 5), 5.00),
    ('Banlieue proche', CIRCLE(POINT(0, 0), 10), 10.00),
    ('Banlieue éloignée', CIRCLE(POINT(0, 0), 20), 15.00);

-- Vérifier les frais de livraison pour une adresse
SELECT nom, frais_livraison  
FROM zones_livraison  
WHERE zone @> POINT(3, 4)  
ORDER BY frais_livraison  
LIMIT 1;  
```

### Limites des Types Géométriques Natifs

**⚠️ Important :** Les types géométriques natifs de PostgreSQL sont limités :
- ❌ Pas de support des projections cartographiques  
- ❌ Pas de géodésie (calculs sur sphère/ellipsoïde)  
- ❌ Fonctionnalités limitées comparé à PostGIS  
- ❌ Pas de support des formats standards (WKT, GeoJSON)

**Recommandation :** Pour des applications géospatiales sérieuses, utilisez **PostGIS** (extension PostgreSQL dédiée).

## 2. hstore (Paires Clé-Valeur)

### Qu'est-ce que hstore ?

**hstore** est un type de données qui stocke des ensembles de paires **clé-valeur** (similaire à un dictionnaire ou un hash). C'est une alternative plus légère à JSONB pour des données simples.

### Différences : hstore vs JSONB

| Aspect | hstore | JSONB |
|--------|--------|-------|
| Structure | Plat (clé-valeur simple) | Hiérarchique (objets imbriqués) |
| Valeurs | Texte uniquement | Tout type JSON (nombres, bool, tableaux, objets) |
| Performance | Légèrement plus rapide | Très bon |
| Taille | Plus compact | Plus gros |
| **Recommandation moderne** | Legacy (héritage) | **Préférer JSONB** |

**Note :** hstore est historique. Pour les nouveaux projets, **JSONB est généralement préféré**.

### Activer l'Extension hstore

```sql
-- Activer l'extension hstore
CREATE EXTENSION IF NOT EXISTS hstore;
```

### Création et Insertion

```sql
CREATE TABLE produits_hstore (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    attributs hstore  -- Paires clé-valeur
);

-- Insérer des données hstore
INSERT INTO produits_hstore (nom, attributs) VALUES
    ('Ordinateur',
     'marque => "Dell", processeur => "Intel i7", ram => "16GB", couleur => "Noir"'),
    ('Téléphone',
     'marque => "Samsung", stockage => "128GB", couleur => "Bleu", ecran => "6.5 pouces"'),
    ('Tablette',
     'marque => "Apple", stockage => "256GB", couleur => "Argent"');

SELECT * FROM produits_hstore;
```

### Accéder aux Données hstore

```sql
-- Accéder à une valeur par clé
SELECT
    nom,
    attributs -> 'marque' AS marque,
    attributs -> 'couleur' AS couleur
FROM produits_hstore;

-- Vérifier si une clé existe
SELECT nom  
FROM produits_hstore  
WHERE attributs ? 'processeur';  -- Possède la clé 'processeur'  

-- Vérifier si contient plusieurs clés
SELECT nom  
FROM produits_hstore  
WHERE attributs ?& ARRAY['marque', 'couleur'];  -- Possède toutes ces clés  

-- Vérifier si contient au moins une clé
SELECT nom  
FROM produits_hstore  
WHERE attributs ?| ARRAY['processeur', 'stockage'];  -- Possède au moins une  

-- Filtrer par valeur
SELECT nom  
FROM produits_hstore  
WHERE attributs -> 'marque' = 'Dell';  

-- Recherche de paire clé-valeur
SELECT nom  
FROM produits_hstore  
WHERE attributs @> 'couleur => "Noir"'::hstore;  
```

### Modifier des Données hstore

```sql
-- Ajouter ou mettre à jour une paire
UPDATE produits_hstore  
SET attributs = attributs || 'garantie => "2 ans"'::hstore  
WHERE nom = 'Ordinateur';  

-- Supprimer une clé
UPDATE produits_hstore  
SET attributs = delete(attributs, 'couleur')  
WHERE nom = 'Téléphone';  

-- Supprimer plusieurs clés
UPDATE produits_hstore  
SET attributs = delete(attributs, ARRAY['couleur', 'ecran'])  
WHERE nom = 'Téléphone';  
```

### Fonctions hstore

```sql
-- akeys : Obtenir toutes les clés
SELECT akeys(attributs) FROM produits_hstore;

-- avals : Obtenir toutes les valeurs
SELECT avals(attributs) FROM produits_hstore;

-- each : Décomposer en lignes (clé, valeur)
SELECT nom, (each(attributs)).*  
FROM produits_hstore  
WHERE nom = 'Ordinateur';  

-- hstore_to_json : Convertir en JSON
SELECT
    nom,
    hstore_to_json(attributs) AS attributs_json
FROM produits_hstore;

-- Nombre de paires
SELECT
    nom,
    array_length(akeys(attributs), 1) AS nombre_attributs
FROM produits_hstore;
```

### Indexation hstore

```sql
-- Index GIN pour recherches
CREATE INDEX idx_produits_attributs ON produits_hstore USING GIN (attributs);

-- Requêtes optimisées
SELECT * FROM produits_hstore  
WHERE attributs @> 'marque => "Dell"'::hstore;  -- Utilise l'index  
```

### Cas d'Usage de hstore

#### 1. Attributs Produits Variables

```sql
CREATE TABLE catalogue (
    id SERIAL PRIMARY KEY,
    categorie VARCHAR(100),
    nom VARCHAR(200),
    prix NUMERIC(10, 2),
    specifications hstore
);

-- Électronique
INSERT INTO catalogue (categorie, nom, prix, specifications) VALUES
    ('Électronique', 'Laptop Pro', 1299.99,
     'cpu => "i7", ram => "32GB", ssd => "1TB", gpu => "RTX 3060"');

-- Vêtements
INSERT INTO catalogue (categorie, nom, prix, specifications) VALUES
    ('Vêtements', 'T-Shirt', 29.99,
     'taille => "M", couleur => "Bleu", matiere => "Coton"');

-- Livres
INSERT INTO catalogue (categorie, nom, prix, specifications) VALUES
    ('Livres', 'PostgreSQL Guide', 49.99,
     'auteur => "Expert DB", pages => "500", langue => "FR"');
```

#### 2. Paramètres Utilisateurs

```sql
CREATE TABLE parametres_utilisateurs (
    user_id INTEGER PRIMARY KEY,
    preferences hstore
);

INSERT INTO parametres_utilisateurs VALUES
    (1, 'theme => "dark", langue => "fr", notifications => "true", timezone => "Europe/Paris"'),
    (2, 'theme => "light", langue => "en", notifications => "false"');

-- Récupérer les préférences
SELECT
    user_id,
    preferences -> 'theme' AS theme,
    preferences -> 'langue' AS langue
FROM parametres_utilisateurs;
```

### hstore vs JSONB : Quand Utiliser Quoi ?

```sql
-- Exemple identique avec JSONB (RECOMMANDÉ)
CREATE TABLE produits_jsonb (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    attributs JSONB
);

INSERT INTO produits_jsonb (nom, attributs) VALUES
    ('Ordinateur',
     '{"marque": "Dell", "processeur": "Intel i7", "ram": "16GB", "prix": 1200}'::JSONB);

-- JSONB supporte des structures complexes
INSERT INTO produits_jsonb (nom, attributs) VALUES
    ('Smartphone',
     '{
         "marque": "Samsung",
         "prix": 799,
         "specifications": {
             "ecran": "6.5 pouces",
             "batterie": 5000
         },
         "couleurs_disponibles": ["Noir", "Bleu", "Blanc"]
     }'::JSONB);
```

**Conclusion :** Pour les nouveaux projets, **utilisez JSONB** au lieu de hstore.

## 3. ltree (Structures Hiérarchiques)

### Qu'est-ce que ltree ?

**ltree** est un type de données spécialisé pour représenter et manipuler des **structures hiérarchiques** ou **arborescentes** (comme des catégories, organigrammes, chemins de fichiers, taxonomies).

### Activer l'Extension ltree

```sql
-- Activer l'extension ltree
CREATE EXTENSION IF NOT EXISTS ltree;
```

### Format et Syntaxe

Un **ltree** est un chemin composé de labels séparés par des points (`.`), similaire à un chemin de fichier.

```
Format : label1.label2.label3.label4  
Exemple : France.Ile_de_France.Paris.75001  
Exemple : Entreprise.IT.Backend.Database  
```

**Règles :**
- Labels : lettres, chiffres et underscores (pas d'espaces)
- Séparateur : point (`.`)
- Maximum : 65 535 caractères total

### Création et Insertion

```sql
CREATE TABLE categories (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    chemin ltree UNIQUE NOT NULL
);

-- Hiérarchie de catégories e-commerce
INSERT INTO categories (nom, chemin) VALUES
    ('Électronique', 'Electronique'),
    ('Ordinateurs', 'Electronique.Ordinateurs'),
    ('Laptops', 'Electronique.Ordinateurs.Laptops'),
    ('Gaming', 'Electronique.Ordinateurs.Laptops.Gaming'),
    ('Bureautique', 'Electronique.Ordinateurs.Laptops.Bureautique'),
    ('Composants', 'Electronique.Ordinateurs.Composants'),
    ('Téléphones', 'Electronique.Telephones'),
    ('Smartphones', 'Electronique.Telephones.Smartphones');

-- Hiérarchie géographique
INSERT INTO categories (nom, chemin) VALUES
    ('France', 'Monde.Europe.France'),
    ('Île-de-France', 'Monde.Europe.France.Ile_de_France'),
    ('Paris', 'Monde.Europe.France.Ile_de_France.Paris');

SELECT * FROM categories ORDER BY chemin;
```

### Opérateurs ltree

#### Ancêtre et Descendant

```sql
-- @ : Est un descendant (ou égal)
SELECT nom, chemin  
FROM categories  
WHERE chemin @ 'Electronique.Ordinateurs';  
-- Retourne : Ordinateurs, Laptops, Gaming, Bureautique, Composants

-- ~ : Correspond à un pattern (ancêtre)
SELECT nom, chemin  
FROM categories  
WHERE chemin ~ 'Electronique.*';  
-- Retourne tous les descendants d'Électronique

-- ? : Correspond à un pattern lquery
SELECT nom, chemin  
FROM categories  
WHERE chemin ~ 'Electronique.*.Gaming';  
-- Retourne les chemins contenant Electronique...Gaming
```

#### Ancêtre Commun

```sql
-- Trouver l'ancêtre commun de deux chemins
SELECT
    'Electronique.Ordinateurs.Laptops.Gaming'::ltree <@
    'Electronique.Ordinateurs.Laptops.Bureautique'::ltree AS est_ancetre;

-- LCA : Lowest Common Ancestor (ancêtre commun le plus proche)
SELECT lca(
    'Electronique.Ordinateurs.Laptops.Gaming'::ltree,
    'Electronique.Ordinateurs.Composants'::ltree
);
-- Résultat : Electronique.Ordinateurs
```

#### Sous-arbre et Niveau

```sql
-- Obtenir le premier niveau (enfants directs)
SELECT nom, chemin  
FROM categories  
WHERE chemin ~ 'Electronique.*{1}';  
-- {1} = exactement 1 niveau en dessous

-- Obtenir tous les descendants jusqu'à 2 niveaux
SELECT nom, chemin  
FROM categories  
WHERE chemin ~ 'Electronique.*{1,2}';  

-- Niveau de profondeur
SELECT
    nom,
    chemin,
    nlevel(chemin) AS profondeur
FROM categories  
ORDER BY profondeur;  
```

### Fonctions ltree

```sql
-- nlevel : Nombre de niveaux
SELECT
    chemin,
    nlevel(chemin) AS profondeur
FROM categories;

-- subpath : Extraire une partie du chemin
SELECT
    chemin,
    subpath(chemin, 0, 2) AS deux_premiers_niveaux,
    subpath(chemin, 1) AS sans_racine
FROM categories  
WHERE chemin ~ 'Electronique.*';  

-- subltree : Extraire sous-arbre
SELECT
    chemin,
    subltree(chemin, 0, 2) AS sous_arbre
FROM categories;

-- index : Position d'un sous-chemin
SELECT
    chemin,
    index(chemin, 'Ordinateurs') AS position
FROM categories  
WHERE chemin ~ '*.Ordinateurs.*';  

-- text2ltree : Convertir texte vers ltree
SELECT text2ltree('France.Paris.75001');

-- ltree2text : Convertir ltree vers texte
SELECT ltree2text('France.Paris.75001'::ltree);
```

### Indexation ltree

```sql
-- Index GiST pour requêtes ltree
CREATE INDEX idx_categories_chemin ON categories USING GIST (chemin);

-- Index sur ancêtres (pour requêtes de descendants)
CREATE INDEX idx_categories_chemin_ancetre ON categories USING BTREE (chemin);

-- Requêtes optimisées
SELECT * FROM categories  
WHERE chemin @ 'Electronique.Ordinateurs';  -- Utilise l'index  
```

### Cas d'Usage de ltree

#### 1. Catégories de Produits

```sql
CREATE TABLE produits_ltree (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    categorie ltree,
    prix NUMERIC(10, 2)
);

INSERT INTO produits_ltree (nom, categorie, prix) VALUES
    ('Dell XPS 15', 'Electronique.Ordinateurs.Laptops.Bureautique', 1299.99),
    ('ASUS ROG', 'Electronique.Ordinateurs.Laptops.Gaming', 1799.99),
    ('iPhone 15', 'Electronique.Telephones.Smartphones', 999.99),
    ('Processeur i9', 'Electronique.Ordinateurs.Composants', 499.99);

-- Tous les produits dans une catégorie et ses sous-catégories
SELECT nom, prix  
FROM produits_ltree  
WHERE categorie @ 'Electronique.Ordinateurs';  

-- Produits d'une catégorie spécifique uniquement
SELECT nom, prix  
FROM produits_ltree  
WHERE categorie = 'Electronique.Ordinateurs.Laptops.Gaming';  
```

#### 2. Organigramme d'Entreprise

```sql
CREATE TABLE employes_org (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100),
    poste VARCHAR(100),
    chemin_hierarchique ltree
);

INSERT INTO employes_org (nom, poste, chemin_hierarchique) VALUES
    ('Alice CEO', 'CEO', 'Entreprise'),
    ('Bob CTO', 'CTO', 'Entreprise.CTO'),
    ('Charlie Dev Manager', 'Dev Manager', 'Entreprise.CTO.Dev'),
    ('Dave Dev', 'Developer', 'Entreprise.CTO.Dev.Backend'),
    ('Eve Dev', 'Developer', 'Entreprise.CTO.Dev.Frontend'),
    ('Frank CFO', 'CFO', 'Entreprise.CFO'),
    ('Grace Comptable', 'Comptable', 'Entreprise.CFO.Compta');

-- Tous les employés sous le CTO
SELECT nom, poste  
FROM employes_org  
WHERE chemin_hierarchique @ 'Entreprise.CTO';  

-- Manager direct (parent immédiat)
SELECT nom, poste  
FROM employes_org  
WHERE chemin_hierarchique = subpath(  
    'Entreprise.CTO.Dev.Backend'::ltree,
    0,
    nlevel('Entreprise.CTO.Dev.Backend'::ltree) - 1
);

-- Nombre de subordonnés directs
SELECT
    e.nom,
    e.poste,
    COUNT(sub.id) AS nombre_subordonnes_directs
FROM employes_org e  
LEFT JOIN employes_org sub ON sub.chemin_hierarchique ~ (e.chemin_hierarchique::text || '.*{1}')::lquery  
GROUP BY e.id, e.nom, e.poste;  
```

#### 3. Système de Fichiers

```sql
CREATE TABLE fichiers (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    chemin ltree,
    type VARCHAR(20),  -- 'dossier' ou 'fichier'
    taille_ko INTEGER
);

INSERT INTO fichiers (nom, chemin, type, taille_ko) VALUES
    ('home', 'home', 'dossier', NULL),
    ('alice', 'home.alice', 'dossier', NULL),
    ('Documents', 'home.alice.Documents', 'dossier', NULL),
    ('rapport.pdf', 'home.alice.Documents.rapport_pdf', 'fichier', 1024),
    ('photos', 'home.alice.photos', 'dossier', NULL),
    ('vacances.jpg', 'home.alice.photos.vacances_jpg', 'fichier', 2048);

-- Lister le contenu d'un dossier (enfants directs)
SELECT nom, type, taille_ko  
FROM fichiers  
WHERE chemin ~ 'home.alice.*{1}'  
ORDER BY type DESC, nom;  

-- Taille totale d'un dossier (récursif)
SELECT
    SUM(taille_ko) AS taille_totale_ko
FROM fichiers  
WHERE chemin @ 'home.alice'  
  AND type = 'fichier';
```

### ltree vs Approches Alternatives

#### Alternative 1 : Adjacency List (Parent ID)

```sql
-- Approche classique avec parent_id
CREATE TABLE categories_parent (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    parent_id INTEGER REFERENCES categories_parent(id)
);

-- Problème : Requêtes récursives complexes pour obtenir toute la hiérarchie
-- Nécessite des CTEs récursives
```

**Avantages de ltree :**
- ✅ Requêtes de hiérarchie simples et rapides  
- ✅ Pas besoin de récursion  
- ✅ Indexation efficace

**Inconvénients de ltree :**
- ❌ Difficile de déplacer des sous-arbres entiers  
- ❌ Changement de parent nécessite mise à jour de tous les descendants

#### Alternative 2 : Nested Sets

```sql
-- Nested Sets (left, right)
CREATE TABLE categories_nested (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    lft INTEGER,
    rgt INTEGER
);
```

**Comparaison :**
- ltree est plus intuitif et lisible
- Nested Sets est complexe à maintenir
- ltree préféré dans la plupart des cas

### Quand Utiliser Quoi ?

| Besoin | Type Recommandé | Alternative |
|--------|----------------|-------------|
| Coordonnées géographiques simples | `POINT` | PostGIS (recommandé) |

## Bonnes Pratiques

### 1. Types Géométriques : Considérez PostGIS

```sql
-- ❌ Pour une vraie application géospatiale
CREATE TABLE lieux (position POINT);

-- ✅ Utilisez PostGIS
CREATE EXTENSION postgis;  
CREATE TABLE lieux (position GEOMETRY(Point, 4326));  -- WGS84  
```

### 2. hstore : Préférez JSONB pour Nouveaux Projets

```sql
-- ❌ Nouveau projet avec hstore
CREATE TABLE produits (attributs hstore);

-- ✅ Nouveau projet avec JSONB
CREATE TABLE produits (attributs JSONB);
```

### 3. ltree : Indexez pour Performance

```sql
-- ✅ Toujours indexer les colonnes ltree
CREATE INDEX idx_categories_chemin ON categories USING GIST (chemin);

-- Pour recherches de descendants
CREATE INDEX idx_categories_ancetre ON categories USING BTREE (chemin);
```

### 4. ltree : Convention de Nommage Claire

```sql
-- ✅ BON : Noms descriptifs et cohérents
'Electronique.Ordinateurs.Laptops.Gaming'

-- ❌ MAUVAIS : Noms incohérents
'elec.ordi.lap.gam'  -- Trop abrégé
'Électronique.Ordi.laptops.GAMING'  -- Incohérent
```

### 5. ltree : Limiter la Profondeur

```sql
-- ✅ BON : Profondeur raisonnable (3-5 niveaux)
'Entreprise.IT.Backend.Database'

-- ❌ ÉVITER : Hiérarchie trop profonde
'A.B.C.D.E.F.G.H.I.J.K'  -- Difficile à maintenir
```

## Récapitulatif

### Types Spécialisés PostgreSQL

1. **Types Géométriques** : Formes 2D basiques
   - POINT, BOX, CIRCLE, POLYGON
   - Opérateurs : `<->`, `@>`, `&&`
   - Limité : préférer PostGIS pour géospatial

2. **hstore** : Paires clé-valeur plates
   - Legacy, remplacé par JSONB
   - Toujours texte uniquement
   - Extensions nécessaire

3. **ltree** : Structures hiérarchiques
   - Format : `label.sublabel.subsublabel`
   - Opérateurs : `@`, `~`, `?`
   - Excellent pour catégories, organigrammes

### Commandes Essentielles

```sql
-- Types géométriques
SELECT point1 <-> point2;  -- Distance  
SELECT box @> point;  -- Contient  

-- hstore
CREATE EXTENSION hstore;  
SELECT data -> 'key';  -- Accès  
SELECT data ? 'key';  -- Existe  

-- ltree
CREATE EXTENSION ltree;  
SELECT path @ 'root.child';  -- Descendant  
SELECT nlevel(path);  -- Profondeur  
```

## Conclusion

Ces types spécialisés étendent PostgreSQL pour des cas d'usage spécifiques :

- **Types géométriques** : Utiles pour des besoins géométriques simples, mais PostGIS est recommandé pour du géospatial sérieux  
- **hstore** : Type historique pour clé-valeur, maintenant supplanté par JSONB dans la plupart des cas  
- **ltree** : Excellent choix pour représenter et requêter des hiérarchies, plus simple et performant que les CTEs récursives

**ltree** est le plus utile de ces trois types dans les applications modernes, particulièrement pour les catégories, organigrammes et structures arborescentes.
