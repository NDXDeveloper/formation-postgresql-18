# 4.4.1. Types Numériques (INTEGER, SERIAL/IDENTITY, NUMERIC, FLOAT)

## Introduction

Les types numériques sont fondamentaux dans toute base de données. PostgreSQL offre une grande variété de types numériques adaptés à différents besoins : nombres entiers, nombres décimaux précis, nombres à virgule flottante, et identifiants auto-incrémentés.

Dans cette section, nous allons explorer en détail :
- Les **nombres entiers** (INTEGER et ses variantes)
- Les **identifiants auto-incrémentés** (SERIAL hérité et IDENTITY recommandé)
- Les **nombres décimaux précis** (NUMERIC et DECIMAL)
- Les **nombres à virgule flottante** (REAL et DOUBLE PRECISION)

> 💡 **Note d'orientation** : pour les nouveaux projets, privilégiez `GENERATED ALWAYS AS IDENTITY` (standard SQL:2003) plutôt que `SERIAL`. Pour les systèmes distribués, considérez également `UUID` avec `uuidv7()` (nouveauté PostgreSQL 18, classable temporellement). Ces alternatives sont détaillées plus bas et dans la section [4.4 Types spécifiques PostgreSQL](/04-objets-de-la-base-de-donnees/04.4-types-specifiques-postgresql.md).

## Vue d'Ensemble des Types Numériques

Voici un tableau récapitulatif des types numériques PostgreSQL :

| Type | Taille | Plage de Valeurs | Utilisation Typique |
|------|--------|------------------|---------------------|
| `SMALLINT` | 2 octets | -32 768 à +32 767 | Petits entiers |
| `INTEGER` / `INT` | 4 octets | -2 milliards à +2 milliards | Entiers standards |
| `BIGINT` | 8 octets | -9 × 10¹⁸ à +9 × 10¹⁸ | Très grands entiers |
| `SERIAL` | 4 octets | 1 à 2 milliards | ID auto-incrémentés |
| `BIGSERIAL` | 8 octets | 1 à 9 × 10¹⁸ | ID auto-incrémentés (grande table) |
| `NUMERIC(p,s)` | Variable | Précision arbitraire | Montants financiers, calculs précis |
| `DECIMAL(p,s)` | Variable | Identique à NUMERIC | Alias de NUMERIC |
| `REAL` | 4 octets | ~6 chiffres significatifs (IEEE 754 simple) | Nombres flottants simples |
| `DOUBLE PRECISION` | 8 octets | ~15 chiffres significatifs (IEEE 754 double) | Nombres flottants doubles |

## 1. Les Nombres Entiers (INTEGER)

### Types d'entiers disponibles

PostgreSQL propose trois types d'entiers de tailles différentes :

#### SMALLINT (Petit Entier)

```sql
-- Définition
SMALLINT

-- Plage : -32 768 à +32 767
-- Stockage : 2 octets (16 bits)
```

**Utilisation :**
- Âges, notes, quantités limitées
- Lorsque vous êtes **certain** que les valeurs resteront petites

**Exemple :**

```sql
CREATE TABLE etudiants (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100),
    age SMALLINT,  -- L'âge ne dépassera jamais 32 767
    note SMALLINT CHECK (note BETWEEN 0 AND 20)
);

INSERT INTO etudiants (nom, age, note) VALUES ('Alice', 22, 18);  
INSERT INTO etudiants (nom, age, note) VALUES ('Bob', 19, 15);  

-- Tentative de dépassement (erreur)
INSERT INTO etudiants (nom, age, note) VALUES ('Charlie', 40000, 10);
-- ERROR: smallint out of range
```

#### INTEGER ou INT (Entier Standard)

```sql
-- Définition
INTEGER  -- ou INT (alias)

-- Plage : -2 147 483 648 à +2 147 483 647 (environ -2 milliards à +2 milliards)
-- Stockage : 4 octets (32 bits)
```

**Utilisation :**
- **Type d'entier par défaut** le plus courant
- IDs, compteurs, quantités standards
- Bon équilibre entre plage et taille

**Exemple :**

```sql
CREATE TABLE produits (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    quantite_stock INTEGER,  -- Stock de produits
    nombre_ventes INTEGER DEFAULT 0
);

INSERT INTO produits (nom, quantite_stock) VALUES ('Ordinateur', 150);  
INSERT INTO produits (nom, quantite_stock) VALUES ('Souris', 5000);  
INSERT INTO produits (nom, quantite_stock) VALUES ('Clavier', 2500);  

SELECT * FROM produits;
```

#### BIGINT (Grand Entier)

```sql
-- Définition
BIGINT

-- Plage : -9 223 372 036 854 775 808 à +9 223 372 036 854 775 807
--         (environ -9 × 10^18 à +9 × 10^18)
-- Stockage : 8 octets (64 bits)
```

**Utilisation :**
- Très grandes quantités (compteurs de vues, transactions bancaires)
- Timestamps en millisecondes
- Identifiants pour très grandes tables (plusieurs milliards de lignes)

**Exemple :**

```sql
CREATE TABLE statistiques_site (
    id BIGSERIAL PRIMARY KEY,
    page VARCHAR(500),
    nombre_vues BIGINT DEFAULT 0,  -- Peut atteindre des milliards
    timestamp_ms BIGINT  -- Timestamp en millisecondes depuis epoch
);

INSERT INTO statistiques_site (page, nombre_vues, timestamp_ms)  
VALUES ('/home', 15000000000, 1700400000000);  

-- Afficher les données
SELECT
    page,
    nombre_vues,
    to_timestamp(timestamp_ms / 1000.0) AS date_heure
FROM statistiques_site;
```

### Comparaison des Types Entiers

| Type | Taille | Plage Approximative | Quand l'Utiliser |
|------|--------|---------------------|------------------|
| `SMALLINT` | 2 octets | -32K à +32K | Petites valeurs garanties |
| `INTEGER` | 4 octets | -2B à +2B | **Usage général** (recommandé) |
| `BIGINT` | 8 octets | -9 × 10¹⁸ à +9 × 10¹⁸ | Très grandes valeurs |

**Règle pratique** : utilisez `INTEGER` par défaut, sauf si :
- Vous avez besoin d'économiser de l'espace → `SMALLINT`
- Vous prévoyez des valeurs > 2 milliards → `BIGINT`

### Opérations Arithmétiques

PostgreSQL supporte toutes les opérations arithmétiques standards :

```sql
-- Créer une table de démonstration
CREATE TABLE calculs (
    a INTEGER,
    b INTEGER
);

INSERT INTO calculs VALUES (10, 3), (100, 25), (7, 2);

-- Addition
SELECT a, b, a + b AS addition FROM calculs;

-- Soustraction
SELECT a, b, a - b AS soustraction FROM calculs;

-- Multiplication
SELECT a, b, a * b AS multiplication FROM calculs;

-- Division entière (tronque les décimales)
SELECT a, b, a / b AS division_entiere FROM calculs;
-- Résultat : 10/3 = 3 (pas 3.33)

-- Modulo (reste de la division)
SELECT a, b, a % b AS modulo FROM calculs;
-- Résultat : 10 % 3 = 1

-- Puissance
SELECT a, b, a ^ b AS puissance FROM calculs;
-- Ou avec power()
SELECT a, b, power(a, b) AS puissance FROM calculs;
```

**Attention à la division entière :**

```sql
-- Division entre deux entiers = résultat entier (tronqué)
SELECT 10 / 3;  -- Résultat : 3 (pas 3.333...)

-- Pour obtenir un résultat décimal, convertir l'un des nombres
SELECT 10.0 / 3;        -- Résultat : 3.333...  
SELECT 10 / 3.0;        -- Résultat : 3.333...  
SELECT 10::NUMERIC / 3; -- Résultat : 3.333...  
SELECT CAST(10 AS NUMERIC) / 3; -- Résultat : 3.333...  
```

## 2. Les Identifiants Auto-Incrémentés (SERIAL et IDENTITY)

### SERIAL, BIGSERIAL, SMALLSERIAL

Les types `SERIAL` ne sont **pas de vrais types** PostgreSQL. C'est un raccourci qui crée automatiquement :
1. Une séquence  
2. Une colonne INTEGER avec une valeur par défaut provenant de cette séquence

#### SERIAL (Auto-Increment Standard)

```sql
-- Définition
SERIAL

-- Équivalent à :
INTEGER NOT NULL DEFAULT nextval('sequence_name')
```

**Exemple de base :**

```sql
CREATE TABLE utilisateurs (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100),
    email VARCHAR(255)
);

-- Insérer sans spécifier l'ID
INSERT INTO utilisateurs (nom, email) VALUES ('Alice', 'alice@example.com');  
INSERT INTO utilisateurs (nom, email) VALUES ('Bob', 'bob@example.com');  
INSERT INTO utilisateurs (nom, email) VALUES ('Charlie', 'charlie@example.com');  

-- PostgreSQL génère automatiquement les IDs : 1, 2, 3
SELECT * FROM utilisateurs;
```

Résultat :
```
 id |   nom   |        email
----+---------+---------------------
  1 | Alice   | alice@example.com
  2 | Bob     | bob@example.com
  3 | Charlie | charlie@example.com
```

#### BIGSERIAL (Auto-Increment pour Grandes Tables)

```sql
-- Définition
BIGSERIAL

-- Équivalent à :
BIGINT NOT NULL DEFAULT nextval('sequence_name')

-- Plage : 1 à 9 223 372 036 854 775 807
```

**Utilisation :**
- Tables qui auront plus de 2 milliards de lignes
- Logs système, événements, données de capteurs

**Exemple :**

```sql
CREATE TABLE logs_application (
    id BIGSERIAL PRIMARY KEY,  -- Peut atteindre des milliards
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    niveau VARCHAR(20),
    message TEXT
);

INSERT INTO logs_application (niveau, message)  
VALUES ('INFO', 'Application démarrée');  
```

#### SMALLSERIAL (Auto-Increment Limité)

```sql
-- Définition
SMALLSERIAL

-- Équivalent à :
SMALLINT NOT NULL DEFAULT nextval('sequence_name')

-- Plage : 1 à 32 767
```

**Utilisation** : rarement utilisé (32 767 lignes max).

### Comparaison SERIAL vs BIGSERIAL

| Type | Type Réel | Plage | Utilisation |
|------|-----------|-------|-------------|
| `SMALLSERIAL` | SMALLINT | 1 à 32K | Très petites tables |
| `SERIAL` | INTEGER | 1 à 2 milliards | **Standard** (tables normales) |
| `BIGSERIAL` | BIGINT | 1 à 9 × 10¹⁸ | Grandes tables (logs, analytics) |

### Comprendre les Séquences

Sous le capot, `SERIAL` crée une **séquence** :

```sql
-- Créer une table avec SERIAL
CREATE TABLE test_serial (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(50)
);

-- PostgreSQL crée automatiquement une séquence nommée :
-- test_serial_id_seq

-- Voir la séquence
\ds

-- Voir la valeur actuelle de la séquence
SELECT currval('test_serial_id_seq');  -- Erreur si jamais utilisée

-- Voir la prochaine valeur (et l'incrémenter)
SELECT nextval('test_serial_id_seq');  -- Retourne 1, puis 2, puis 3...

-- Réinitialiser la séquence
ALTER SEQUENCE test_serial_id_seq RESTART WITH 100;

-- Les prochaines insertions commenceront à 100
INSERT INTO test_serial (nom) VALUES ('Test');  -- id = 100
```

### Comportement Important de SERIAL

#### 1. Les valeurs ne sont PAS réutilisées

```sql
CREATE TABLE demo_serial (
    id SERIAL PRIMARY KEY,
    valeur VARCHAR(50)
);

INSERT INTO demo_serial (valeur) VALUES ('A');  -- id = 1  
INSERT INTO demo_serial (valeur) VALUES ('B');  -- id = 2  
INSERT INTO demo_serial (valeur) VALUES ('C');  -- id = 3  

-- Supprimer la ligne avec id = 2
DELETE FROM demo_serial WHERE id = 2;

-- Insérer une nouvelle ligne
INSERT INTO demo_serial (valeur) VALUES ('D');  -- id = 4 (pas 2 !)

SELECT * FROM demo_serial;
-- Résultat :
-- id | valeur
-- ---+--------
-- 1  | A
-- 3  | C
-- 4  | D
```

**Pourquoi ?** Les séquences ne regardent pas les valeurs existantes, elles incrémentent toujours.

#### 2. Les trous dans les IDs sont normaux

```sql
-- Début de transaction
BEGIN;  
INSERT INTO demo_serial (valeur) VALUES ('E');  -- id = 5 (séquence incrémentée)  
ROLLBACK;  -- Annulation de la transaction  

-- La séquence N'EST PAS annulée
INSERT INTO demo_serial (valeur) VALUES ('F');  -- id = 6 (pas 5)
```

**Résultat** : il peut y avoir des trous dans la numérotation. **C'est normal et acceptable.**

#### 3. On peut spécifier manuellement l'ID

```sql
-- Insérer avec un ID spécifique
INSERT INTO demo_serial (id, valeur) VALUES (100, 'Manuel');

-- La séquence n'est PAS mise à jour automatiquement !
INSERT INTO demo_serial (valeur) VALUES ('Auto');  -- Pourrait créer un conflit

-- Il faut mettre à jour la séquence manuellement
SELECT setval('demo_serial_id_seq', (SELECT MAX(id) FROM demo_serial));
```

### IDENTITY (Standard SQL — **RECOMMANDÉ**)

Depuis PostgreSQL 10, le standard SQL `IDENTITY` (SQL:2003) est supporté. C'est l'**alternative moderne et recommandée** à `SERIAL` :

```sql
-- Syntaxe IDENTITY (recommandée pour tout nouveau code)
CREATE TABLE produits (
    id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    nom VARCHAR(200)
);

-- Ou avec BY DEFAULT (permet l'insertion manuelle)
CREATE TABLE produits2 (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    nom VARCHAR(200)
);

-- Avec BIGINT pour les grandes tables
CREATE TABLE evenements (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    payload JSONB
);
```

#### Différences entre SERIAL et IDENTITY

| Aspect | SERIAL | IDENTITY |
|--------|--------|----------|
| Standard SQL | Non (spécifique PostgreSQL) | Oui (SQL:2003) |
| Insertion manuelle | Toujours possible (risque de désync) | `BY DEFAULT` : oui, `ALWAYS` : non |
| Portabilité | Faible | Meilleure (Oracle, DB2, SQL Server) |
| Permissions | La séquence est un objet **séparé** : GRANT à gérer | Séquence **liée** à la colonne (héritage automatique) |
| Suppression | La séquence reste si la table est `DROP`pée par mégarde | Séquence supprimée automatiquement avec la colonne |
| Modification de la séquence | Via `ALTER SEQUENCE` (objet séparé) | Via `ALTER TABLE … ALTER COLUMN … RESTART` |
| `pg_dump` | Ordre de restauration parfois délicat | Plus propre |
| Recommandation officielle | Hérité (toujours supporté) | **Préféré** pour tout nouveau code |

**Exemple avec GENERATED ALWAYS :**

```sql
CREATE TABLE commandes (
    id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    numero VARCHAR(50)
);

-- Insertion normale (OK)
INSERT INTO commandes (numero) VALUES ('CMD-001');  -- id = 1

-- Tentative d'insertion manuelle (ERREUR)
INSERT INTO commandes (id, numero) VALUES (100, 'CMD-002');
-- ERROR: cannot insert into column "id"

-- Pour forcer quand même (rare)
INSERT INTO commandes (id, numero) OVERRIDING SYSTEM VALUE VALUES (100, 'CMD-002');
```

**Exemple avec GENERATED BY DEFAULT :**

```sql
CREATE TABLE articles (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    titre VARCHAR(300)
);

-- Insertion normale
INSERT INTO articles (titre) VALUES ('Article 1');  -- id = 1

-- Insertion manuelle (OK avec BY DEFAULT)
INSERT INTO articles (id, titre) VALUES (100, 'Article 100');

-- Prochaine insertion automatique
INSERT INTO articles (titre) VALUES ('Article 2');  -- id = 2 (pas 101 !)
```

> ⚠️ **Piège classique** : avec `BY DEFAULT`, après une insertion manuelle d'un ID élevé (ici 100), la séquence n'est pas automatiquement avancée. L'insertion suivante reprend la séquence interne (id = 2). Si vous insérez `id = 100` puis continuez avec des inserts automatiques, vous finirez par produire un conflit de clé primaire quand la séquence rattrapera 100. Pour resynchroniser, utilisez `pg_catalog.pg_get_serial_sequence` :  
>  
> ```sql  
> SELECT setval(pg_get_serial_sequence('articles', 'id'),  
>               (SELECT MAX(id) FROM articles));  
> ```

#### Migration SERIAL → IDENTITY

Sur un schéma existant utilisant `SERIAL`, vous pouvez basculer vers `IDENTITY` sans réécrire la table :

```sql
-- Table existante avec SERIAL
CREATE TABLE clients (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100)
);

-- 1) Retirer le DEFAULT lié à la séquence et l'attachement
ALTER TABLE clients ALTER COLUMN id DROP DEFAULT;

-- 2) Détacher la séquence existante (elle restera un objet séparé)
ALTER SEQUENCE clients_id_seq OWNED BY NONE;

-- 3) Transformer la colonne en IDENTITY
ALTER TABLE clients ALTER COLUMN id ADD GENERATED BY DEFAULT AS IDENTITY (
    RESTART WITH 1000  -- ou la valeur actuelle de la séquence
);

-- 4) Optionnel : supprimer l'ancienne séquence devenue inutile
DROP SEQUENCE clients_id_seq;
```

### Alternative moderne : UUID avec uuidv7() (PostgreSQL 18 🆕)

Pour les systèmes distribués, les API publiques, ou pour éviter de divulguer le volume métier via des IDs séquentiels, `UUID` est une excellente alternative. PostgreSQL 18 introduit la fonction native `uuidv7()` qui génère des UUIDs **classables temporellement** (la première moitié encode un timestamp ms) :

```sql
-- PostgreSQL 18 (nouveauté)
CREATE TABLE commandes_distribuees (
    id UUID PRIMARY KEY DEFAULT uuidv7(),
    numero VARCHAR(50),
    date_creation TIMESTAMPTZ DEFAULT now()
);

INSERT INTO commandes_distribuees (numero) VALUES ('CMD-001');
-- id = 0192a4b5-c7e3-7a9c-bd1e-... (préfixe temporel)

-- Versions antérieures (UUID v4 aléatoire, moins efficace en B-tree)
CREATE TABLE commandes_v4 (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    numero VARCHAR(50)
);
```

| Critère | IDENTITY (INT/BIGINT) | UUID v4 (`gen_random_uuid`) | UUID v7 (`uuidv7()` — PG 18) |
|---------|----------------------|------------------------------|------------------------------|
| Taille | 4 ou 8 octets | 16 octets | 16 octets |
| Classable temporellement | Oui (croissant) | Non (aléatoire) | **Oui** (préfixe timestamp) |
| Performance B-tree (insert) | Optimale | Dégradée (insertions aléatoires) | Bonne (insertions séquentielles) |
| Génération distribuée sans coordination | Non | Oui | Oui |
| Divulgue le volume métier | Oui (compteur visible) | Non | Partiellement (timestamp visible) |
| Lisibilité humaine | Bonne | Faible | Faible |

> 📚 La section [4.4 Types spécifiques PostgreSQL](/04-objets-de-la-base-de-donnees/04.4-types-specifiques-postgresql.md) approfondit l'usage des UUIDs.

## 3. Les Nombres Décimaux Précis (NUMERIC et DECIMAL)

### NUMERIC / DECIMAL (Exactitude Mathématique)

Les types `NUMERIC` et `DECIMAL` sont **identiques** dans PostgreSQL. Ils stockent les nombres avec une **précision exacte**, sans erreur d'arrondi.

```sql
-- Syntaxe
NUMERIC(precision, scale)  
DECIMAL(precision, scale)  

-- Où :
-- precision = nombre total de chiffres significatifs
-- scale = nombre de chiffres après la virgule
```

#### Exemples de Déclaration

```sql
-- NUMERIC(10, 2) : 10 chiffres au total, 2 après la virgule
-- Exemple de valeurs valides : 12345678.90, -1234567.89, 0.50

CREATE TABLE finances (
    id SERIAL PRIMARY KEY,
    montant NUMERIC(10, 2),  -- 99 999 999.99 max
    taux_interet NUMERIC(5, 4)  -- 9.9999 max
);

-- Exemples d'insertions
INSERT INTO finances (montant, taux_interet) VALUES (1500.50, 0.0475);  
INSERT INTO finances (montant, taux_interet) VALUES (999999.99, 5.5000);  
INSERT INTO finances (montant, taux_interet) VALUES (0.01, 0.0001);  

SELECT * FROM finances;
```

#### Précision et Scale en Détail

```sql
-- NUMERIC(5, 2)
-- precision = 5 (total de chiffres)
-- scale = 2 (chiffres après la virgule)
-- Plage : -999.99 à 999.99

CREATE TABLE test_numeric (
    valeur NUMERIC(5, 2)
);

-- Insertions valides
INSERT INTO test_numeric VALUES (123.45);   -- OK  
INSERT INTO test_numeric VALUES (999.99);   -- OK (valeur max)  
INSERT INTO test_numeric VALUES (-999.99);  -- OK  
INSERT INTO test_numeric VALUES (0.01);     -- OK  

-- Insertion refusée : trop de chiffres AVANT la virgule (overflow)
INSERT INTO test_numeric VALUES (1234.56);  -- ERROR: numeric field overflow

-- ⚠️ En revanche, trop de chiffres APRÈS la virgule ne provoque PAS d'erreur :
-- la valeur est simplement ARRONDIE au scale défini.
INSERT INTO test_numeric VALUES (123.456);  -- OK → stocké 123.46 (arrondi)

-- …sauf si l'arrondi fait dépasser la précision totale :
INSERT INTO test_numeric VALUES (999.999);  -- ERROR: overflow (arrondi → 1000.00)
```

#### NUMERIC sans Précision

Si vous omettez la précision et le scale, PostgreSQL accepte une précision quasi-arbitraire :

- **Jusqu'à 131 072 chiffres avant la virgule**
- **Jusqu'à 16 383 chiffres après la virgule**

```sql
CREATE TABLE calculs_precis (
    id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    resultat NUMERIC  -- Précision quasi-arbitraire
);

-- Peut stocker des nombres très précis
INSERT INTO calculs_precis (resultat)  
VALUES (123456789012345678901234567890.123456789);  

SELECT * FROM calculs_precis;
```

> ⚠️ **Attention** : un `NUMERIC` sans précision occupe **plus d'espace** et est plus lent à manipuler que `NUMERIC(p, s)`. Spécifiez une précision dès que les limites métier le permettent (`NUMERIC(15, 2)` pour de l'argent, `NUMERIC(5, 4)` pour un taux d'intérêt, etc.). Ne réservez `NUMERIC` sans paramètre qu'aux cas véritablement à précision variable (calculs scientifiques exotiques).

### Cas d'Usage de NUMERIC

#### 1. Montants Financiers (Le Plus Important)

```sql
-- TOUJOURS utiliser NUMERIC pour l'argent !
CREATE TABLE transactions (
    id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    compte_source VARCHAR(50),
    compte_destination VARCHAR(50),
    montant NUMERIC(15, 2) NOT NULL CHECK (montant > 0),
    devise CHAR(3) DEFAULT 'EUR',  -- ISO 4217 : exactement 3 lettres
    date_transaction TIMESTAMPTZ DEFAULT NOW()
);

-- Insertion
INSERT INTO transactions (compte_source, compte_destination, montant, devise)  
VALUES ('FR123', 'FR456', 1250.75, 'EUR');  

-- Calculs financiers (sans erreur d'arrondi)
SELECT
    compte_source,
    SUM(montant) AS total_debite,
    AVG(montant) AS montant_moyen
FROM transactions  
GROUP BY compte_source;  
```

> 💡 **Astuce pour les calculs financiers en masse** : `SUM(numeric)` retourne un `numeric` sans précision fixe, qui peut donc grossir au-delà des `NUMERIC(15, 2)` source. Cela évite les overflows lors des agrégations. Mais attention quand vous **stockez** un agrégat dans une colonne : prévoyez une précision suffisante.  
>  
> ```sql  
> -- Stocker un total mensuel : utiliser une précision plus large que les lignes  
> CREATE TABLE bilan_mensuel (  
>     mois DATE PRIMARY KEY,  
>     total_chiffre_affaires NUMERIC(18, 2)  -- 16 chiffres entiers max ≈ 10 quadrillions  
> );  
> ```

❌ **JAMAIS UTILISER FLOAT POUR L'ARGENT !**

```sql
-- MAUVAISE PRATIQUE
CREATE TABLE mauvais_exemple (
    montant FLOAT  -- NON ! Erreurs d'arrondi garanties
);

-- Problème avec FLOAT (il faut forcer le calcul en flottant pour le constater)
SELECT 0.1::float8 + 0.2::float8;  -- Résultat en FLOAT : 0.30000000000000004 (!)  
SELECT 0.1 + 0.2;  -- En NUMERIC (type par défaut des littéraux décimaux) : 0.3 (exact)  
```

#### 2. Pourcentages et Taux

```sql
CREATE TABLE produits_promo (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    prix_original NUMERIC(10, 2),
    taux_remise NUMERIC(5, 2),  -- De 0.00 à 100.00 (pourcentage)
    prix_final NUMERIC(10, 2) GENERATED ALWAYS AS (
        prix_original * (1 - taux_remise / 100)
    ) STORED
);

INSERT INTO produits_promo (nom, prix_original, taux_remise)  
VALUES ('Laptop', 1200.00, 15.50);  

SELECT
    nom,
    prix_original,
    taux_remise || '%' AS remise,
    prix_final
FROM produits_promo;
```

```sql
CREATE TABLE mesures_laboratoire (
```

### Opérations avec NUMERIC

```sql
-- Créer une table de test
CREATE TABLE operations_numeric (
    a NUMERIC(10, 2),
    b NUMERIC(10, 2)
);

INSERT INTO operations_numeric VALUES (100.50, 25.25);

-- Addition et soustraction (précision préservée)
SELECT a + b AS addition, a - b AS soustraction  
FROM operations_numeric;  

-- Multiplication (précision augmente)
SELECT a * b AS multiplication  
FROM operations_numeric;  
-- Résultat : 2537.6250 (4 décimales)

-- Division (précision configurable)
SELECT a / b AS division  
FROM operations_numeric;  
-- Résultat : 3.98019801980198... (très précis)

-- Arrondir le résultat (a/b ≈ 3.9801980198…)
SELECT ROUND(a / b, 2) AS division_arrondie  
FROM operations_numeric;  
-- Résultat : 3.98 (la 3e décimale est 0, l'arrondi ne change rien)

-- Tronquer (enlever les décimales sans arrondir)
SELECT TRUNC(a / b, 2) AS division_tronquee  
FROM operations_numeric;  
-- Résultat : 3.98 (même valeur ici par coïncidence)

-- Différence visible sur un cas plus parlant :
SELECT ROUND(3.999::numeric, 2) AS arrondi,   -- 4.00 (arrondi supérieur)
       TRUNC(3.999::numeric, 2) AS tronque;   -- 3.99 (troncature)

-- Plafond et plancher
SELECT CEIL(a / b) AS ceil, FLOOR(a / b) AS floor  
FROM operations_numeric;  
-- Résultat : ceil = 4, floor = 3
```

### Fonctions Utiles avec NUMERIC

```sql
-- ROUND : Arrondir
SELECT ROUND(123.456, 2);  -- 123.46  
SELECT ROUND(123.456, 0);  -- 123  
SELECT ROUND(123.456, -1); -- 120  

-- TRUNC : Tronquer (pas d'arrondi)
SELECT TRUNC(123.456, 2);  -- 123.45  
SELECT TRUNC(123.456, 0);  -- 123  

-- CEIL : Arrondir vers le haut
SELECT CEIL(123.1);  -- 124  
SELECT CEIL(-123.9); -- -123  

-- FLOOR : Arrondir vers le bas
SELECT FLOOR(123.9);  -- 123  
SELECT FLOOR(-123.1); -- -124  

-- ABS : Valeur absolue
SELECT ABS(-123.45);  -- 123.45

-- SIGN : Signe (-1, 0, 1)
SELECT SIGN(123.45);  -- 1  
SELECT SIGN(-123.45); -- -1  
SELECT SIGN(0);       -- 0  
```

## 4. Les Nombres à Virgule Flottante (REAL et DOUBLE PRECISION)

### Types à Virgule Flottante

Les types à virgule flottante stockent des **approximations** de nombres réels. Ils sont basés sur le standard IEEE 754.

```sql
-- REAL (simple précision)
REAL  -- ou FLOAT4
-- Stockage : 4 octets
-- Précision : ~6 chiffres significatifs (24 bits de mantisse)

-- DOUBLE PRECISION (double précision)
DOUBLE PRECISION  -- ou FLOAT8 ou FLOAT
-- Stockage : 8 octets
-- Précision : ~15 chiffres significatifs (53 bits de mantisse)
```

> 💡 **Précision « significative »** : il s'agit du nombre de chiffres décimaux fiables, pas du nombre de décimales après la virgule. Pour `REAL`, `12345.67` (7 chiffres significatifs) est déjà au bord de la limite ; `123456789` (9 chiffres) ne peut pas être stocké exactement. Pour stocker un grand entier exactement, utilisez `INTEGER`/`BIGINT` ou `NUMERIC`.

### Quand Utiliser les Flottants ?

✅ **Utiliser FLOAT pour :**
- Calculs scientifiques où une approximation est acceptable
- Mesures physiques (température, distance)
- Graphiques et visualisations
- Performances critiques avec grandes quantités de données

❌ **NE JAMAIS utiliser FLOAT pour :**
- Montants financiers
- Tout calcul nécessitant une exactitude parfaite

### REAL (Simple Précision)

```sql
CREATE TABLE mesures_temperature (
    id SERIAL PRIMARY KEY,
    capteur VARCHAR(50),
    temperature REAL,  -- Suffisant pour une température
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

INSERT INTO mesures_temperature (capteur, temperature)  
VALUES ('CAPTEUR-01', 23.456);  

INSERT INTO mesures_temperature (capteur, temperature)  
VALUES ('CAPTEUR-02', -15.789);  

SELECT * FROM mesures_temperature;
```

### DOUBLE PRECISION (Double Précision)

```sql
CREATE TABLE coordonnees_gps (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100),
    latitude DOUBLE PRECISION,  -- Précision importante pour GPS
    longitude DOUBLE PRECISION,
    altitude REAL  -- Moins critique
);

INSERT INTO coordonnees_gps (nom, latitude, longitude, altitude)  
VALUES ('Tour Eiffel', 48.858844, 2.294351, 324.0);  

INSERT INTO coordonnees_gps (nom, latitude, longitude, altitude)  
VALUES ('Statue de la Liberté', 40.689247, -74.044502, 93.0);  

SELECT * FROM coordonnees_gps;
```

### Le Problème des Erreurs d'Arrondi

Les nombres flottants ne peuvent pas représenter exactement tous les nombres décimaux :

```sql
-- Démonstration du problème
CREATE TABLE demo_float (
    id SERIAL PRIMARY KEY,
    valeur_float REAL,
    valeur_numeric NUMERIC(10, 2)
);

-- Insérer la même valeur dans les deux colonnes
INSERT INTO demo_float (valeur_float, valeur_numeric)  
VALUES (0.1, 0.1);  

-- Multiplier par 10 révèle l'imprécision du REAL
SELECT
    valeur_float * 10 AS resultat_float,
    valeur_numeric * 10 AS resultat_numeric
FROM demo_float;
-- Float (REAL) : 1.0000000149011612 (imprécis)
-- Numeric      : 1.00 (exact)

-- Exemple classique en DOUBLE PRECISION (l'erreur d'arrondi devient visible) :
SELECT 0.1::float8 + 0.2::float8;    -- 0.30000000000000004 (!)  
SELECT 0.1::NUMERIC + 0.2::NUMERIC;  -- 0.3 (exact)  
-- (En REAL/float4, la précision est si faible que 0.1 + 0.2 s'arrondit à 0.3 :
--  c'est le DOUBLE PRECISION qui expose l'erreur.)
```

### Valeurs Spéciales

Les flottants (ainsi que `NUMERIC`) supportent des valeurs spéciales définies par le standard IEEE 754 :

```sql
-- Infini positif
SELECT 'Infinity'::REAL;
-- 1.0 / 0.0 lève une erreur "division by zero" en PostgreSQL,
-- mais 'Infinity' peut être stocké et utilisé explicitement.

-- Infini négatif
SELECT '-Infinity'::REAL;

-- Not a Number (pas un nombre)
SELECT 'NaN'::REAL;

-- Tester ces valeurs
SELECT
    'Infinity'::REAL > 999999999 AS inf_est_plus_grand,  -- true
    'NaN'::REAL = 'NaN'::REAL AS nan_egal_nan;           -- true (!)
```

> ⚠️ **Particularité PostgreSQL** : contrairement au standard IEEE 754 (où `NaN ≠ NaN`), PostgreSQL traite `NaN` comme **égal à lui-même** et **supérieur à toute autre valeur numérique**, y compris `Infinity`. Cette décision permet le tri et l'indexation B-tree de colonnes contenant des `NaN`. Le comportement diffère donc de la plupart des langages de programmation (Python, JavaScript, C…).

```sql
-- Démonstration de l'ordre PostgreSQL
SELECT unnest(ARRAY['NaN', 'Infinity', '1.0', '-Infinity']::float8[]) AS valeur  
ORDER BY valeur;  
-- Résultat : -Infinity, 1.0, Infinity, NaN  (NaN en dernier)
```

> 💡 **Division par zéro** : contrairement à certaines implémentations, PostgreSQL lève une erreur (`division_by_zero`, SQLSTATE `22012`) sur `1.0 / 0.0`. Pour produire `Infinity`, utilisez explicitement le littéral `'Infinity'::float8`.

### Fonctions Mathématiques avec Flottants

PostgreSQL offre de nombreuses fonctions mathématiques :

```sql
-- Fonctions trigonométriques (angles en radians)
SELECT
    SIN(PI() / 2) AS sinus_90_deg,  -- 1
    COS(PI()) AS cosinus_180_deg,   -- -1
    TAN(PI() / 4) AS tangente_45_deg; -- 1

-- Puissances et racines
SELECT
    POWER(2, 10) AS deux_puissance_10,  -- 1024
    SQRT(16) AS racine_16,              -- 4
    CBRT(27) AS racine_cubique_27;      -- 3

-- Exponentielles et logarithmes

    EXP(1) AS e,                    -- 2.718...
    LN(2.718281828) AS ln_e,        -- 1
    LOG(100) AS log10_100;          -- 2

-- Arrondir
SELECT
    ROUND(3.14159::NUMERIC, 2) AS pi_arrondi;  -- 3.14
```

## Comparaison : NUMERIC vs FLOAT

Voici un tableau comparatif pour vous aider à choisir :

| Aspect | NUMERIC / DECIMAL | REAL / DOUBLE PRECISION |
|--------|-------------------|-------------------------|
| **Exactitude** | Exacte (pas d'erreur d'arrondi) | Approximative |
| **Précision** | Arbitraire (définie par vous) | Limitée (6 ou 15 chiffres) |
| **Performances** | Plus lent | Plus rapide |
| **Stockage** | Variable (plus d'espace) | Fixe (4 ou 8 octets) |
| **Argent** | ✅ OUI (obligatoire) | ❌ NON (jamais) |
| **Calculs scientifiques** | Possible mais lent | ✅ OUI (recommandé) |
| **Valeurs spéciales** | Non | Infinity, -Infinity, NaN |

### Exemple Comparatif

```sql
CREATE TABLE comparaison_types (
    id SERIAL PRIMARY KEY,
    description VARCHAR(100),
    valeur_numeric NUMERIC(15, 2),
    valeur_float DOUBLE PRECISION
);

-- Insérer la même valeur
INSERT INTO comparaison_types (description, valeur_numeric, valeur_float)  
VALUES ('Test 1', 1234567.89, 1234567.89);  

-- Calculs
SELECT
    description,
    valeur_numeric * 1.15 AS numeric_avec_tva,
    valeur_float * 1.15 AS float_avec_tva,
    (valeur_numeric * 1.15) = (valeur_float * 1.15) AS sont_egaux
FROM comparaison_types;
-- Résultat : sont_egaux = false (à cause des erreurs d'arrondi en float)
```

## Conversions entre Types Numériques

### Cast Explicite

```sql
-- Conversion avec ::
SELECT
    123::SMALLINT AS small,
    123::INTEGER AS int,
    123::BIGINT AS big,
    123::NUMERIC(5,2) AS num,
    123::REAL AS real,
    123::DOUBLE PRECISION AS double;

-- Conversion avec CAST()
SELECT
    CAST(123.456 AS INTEGER) AS int_tronque,  -- 123
    CAST('123' AS INTEGER) AS text_to_int,    -- 123
    CAST(123 AS VARCHAR) AS int_to_text;      -- '123'
```

### Précautions lors des Conversions

```sql
-- Conversion qui tronque
SELECT 123.789::INTEGER;  -- 123 (décimales perdues)

-- Conversion qui peut échouer
SELECT 50000::SMALLINT;
-- ERROR:  smallint out of range  (SQLSTATE 22003)

-- Conversion NUMERIC vers INTEGER (arrondit)
SELECT
    123.4::INTEGER AS arrondi_bas,   -- 123
    123.5::INTEGER AS arrondi_haut,  -- 124
    123.9::INTEGER AS arrondi;       -- 124

-- Conversion texte vers nombre
SELECT '123.45'::NUMERIC;  -- 123.45  
SELECT '123abc'::INTEGER;  -- ERREUR : format invalide  
```

## Bonnes Pratiques avec les Types Numériques

### 1. Choisir le Bon Type

```sql
-- ✅ BON : Types appropriés
CREATE TABLE commandes (
    id SERIAL PRIMARY KEY,              -- Auto-increment standard
    numero_commande VARCHAR(50),
    quantite INTEGER,                   -- Entier simple
    prix_unitaire NUMERIC(10, 2),      -- Argent : NUMERIC !
    total NUMERIC(12, 2),              -- Argent : NUMERIC !
    taux_tva NUMERIC(4, 2),            -- Pourcentage
    poids_kg REAL                       -- Mesure physique : FLOAT OK
);

-- ❌ MAUVAIS : Types inadaptés
CREATE TABLE commandes_mal (
    id VARCHAR(50),                     -- ID en texte ? Non !
    quantite NUMERIC(10, 2),           -- Quantité avec décimales ? Bizarre
    prix_unitaire REAL,                 -- Argent en FLOAT ? JAMAIS !
    total VARCHAR(20)                   -- Total en texte ? Non !
);
```

### 2. Toujours NUMERIC pour l'Argent

```sql
-- ✅ CORRECT
CREATE TABLE factures (
    id SERIAL PRIMARY KEY,
    montant_ht NUMERIC(12, 2) NOT NULL,
    montant_tva NUMERIC(12, 2) NOT NULL,
    montant_ttc NUMERIC(12, 2) NOT NULL,
    CHECK (montant_ttc = montant_ht + montant_tva)
);

-- ❌ INCORRECT
CREATE TABLE factures_mal (
    montant FLOAT  -- NON ! Erreurs d'arrondi sur l'argent
);
```

### 3. Spécifier la Précision

```sql
-- ✅ BON : Précision définie
CREATE TABLE produits (
    prix NUMERIC(10, 2),  -- Max 99 999 999.99
    poids NUMERIC(8, 3)   -- Max 99 999.999 kg
);

-- ❓ ACCEPTABLE mais moins optimal
CREATE TABLE produits_vague (
    prix NUMERIC,  -- Précision arbitraire (plus lourd)
    poids NUMERIC
);
```

### 4. Attention aux Dépassements

```sql
-- Utiliser CHECK pour les limites
CREATE TABLE stock (
    id SERIAL PRIMARY KEY,
    quantite INTEGER CHECK (quantite >= 0 AND quantite <= 100000)
);

-- Ou choisir le type approprié
CREATE TABLE stock_petit (
    quantite SMALLINT CHECK (quantite >= 0)  -- Si quantité < 32K toujours
);
```

### 5. Documenter les Unités

```sql
-- Nommer explicitement avec l'unité
CREATE TABLE mesures (
    temperature_celsius REAL,
    distance_metres NUMERIC(10, 2),
    poids_kg NUMERIC(8, 3),
    prix_euros NUMERIC(10, 2)
);

-- Ou utiliser des commentaires
CREATE TABLE produits (
    poids NUMERIC(8, 3)  -- en kilogrammes
);

COMMENT ON COLUMN produits.poids IS 'Poids du produit en kilogrammes';
```

| Besoin | Type Recommandé | Exemple |

### Commandes SQL Utiles

```sql
-- Voir le type d'une colonne
\d nom_table

-- Conversion de type
SELECT valeur::nouveau_type FROM table;

SELECT ROUND(valeur, 2) FROM table;
```

## Conclusion

Les types numériques PostgreSQL offrent une grande flexibilité pour stocker différents types de nombres :

- **INTEGER** : Le choix par défaut pour les entiers
- **IDENTITY** (`GENERATED AS IDENTITY`) : Pour les identifiants auto-incrémentés — **recommandé** pour tout nouveau code
- **SERIAL** : Alternative historique, toujours supportée mais moins propre que IDENTITY
- **UUID** avec `uuidv7()` (PG 18) : Pour les identifiants distribués classables temporellement
- **NUMERIC** : **Obligatoire** pour l'argent et les calculs précis
- **FLOAT** : Pour les mesures physiques et calculs scientifiques

**Règle d'or :**
> Utilisez **NUMERIC pour l'argent**, **INTEGER pour les compteurs**, **IDENTITY pour les clés primaires auto-incrémentées**, et **FLOAT seulement** pour les calculs scientifiques où l'approximation est acceptable.
