# 13.9.2. Réorganisation Automatique des Colonnes DISTINCT (PostgreSQL 18)

## Introduction

PostgreSQL 18 introduit une optimisation intelligente du planificateur : **la réorganisation automatique des colonnes DISTINCT**. Cette fonctionnalité permet au moteur de réorganiser l'ordre des colonnes dans une clause `DISTINCT` pour améliorer les performances, sans changer le résultat de la requête.

Cette optimisation est particulièrement utile lorsque vous travaillez avec des requêtes qui éliminent les doublons sur plusieurs colonnes.

---

## Rappel : Qu'est-ce que DISTINCT ?

Avant d'aborder l'optimisation, rappelons ce qu'est `DISTINCT`.

### Le Problème des Doublons

Lorsqu'on interroge une base de données, on peut obtenir des lignes en double :

```sql
-- Table des commandes
CREATE TABLE commandes (
    id SERIAL PRIMARY KEY,
    client_id INTEGER,
    produit_id INTEGER,
    date_commande DATE,
    montant NUMERIC
);
```

Si on veut la liste des clients qui ont passé commande :

```sql
SELECT client_id FROM commandes;
```

**Résultat possible :**
```
client_id
---------
101
102
101
103
101
102
```

On a des **doublons** ! Le client 101 apparaît 3 fois, le client 102 deux fois.

### La Solution : DISTINCT

Le mot-clé `DISTINCT` élimine les doublons :

```sql
SELECT DISTINCT client_id FROM commandes;
```

**Résultat :**
```
client_id
---------
101
102
103
```

Chaque client n'apparaît qu'**une seule fois**.

### DISTINCT sur Plusieurs Colonnes

On peut utiliser `DISTINCT` sur plusieurs colonnes. Dans ce cas, PostgreSQL élimine les lignes où **la combinaison** de toutes les colonnes est identique :

```sql
SELECT DISTINCT client_id, produit_id FROM commandes;
```

Cela retourne toutes les **combinaisons uniques** de client et produit.

---

## Le Problème de Performance : L'Ordre des Colonnes

### Comment PostgreSQL Traite DISTINCT ?

Pour éliminer les doublons, PostgreSQL doit **trier ou grouper** les données. L'algorithme le plus courant utilise le **tri** (Sort) suivi d'une élimination des valeurs consécutives identiques.

### L'Impact de l'Ordre des Colonnes

Voici le point crucial : **l'ordre dans lequel les colonnes sont triées peut avoir un impact énorme sur les performances**.

#### Exemple avec Données Réelles

Imaginons une table avec 1 million de commandes :

```sql
-- 1 million de commandes
-- 1000 clients différents
-- 10000 produits différents

SELECT DISTINCT client_id, produit_id FROM commandes;
```

**Scénario 1 : Tri par (client_id, produit_id)**
- PostgreSQL trie d'abord par `client_id`, puis par `produit_id`
- Pour éliminer les doublons, il compare ligne par ligne
- Cardinalité de `client_id` : **1000 valeurs distinctes**
- Cardinalité de `produit_id` : **10000 valeurs distinctes**

**Scénario 2 : Tri par (produit_id, client_id)**
- PostgreSQL trie d'abord par `produit_id`, puis par `client_id`
- L'ordre **inverse** du précédent — même résultat (mêmes combinaisons uniques)

Lequel des deux est « le meilleur » ? La réponse est contre-intuitive :

### Le rôle réel de l'ordre des colonnes

> ⚠️ **Précision importante (vérifiée sur PostgreSQL 18.4).** Pour un `DISTINCT`
> résolu par un **tri**, l'ordre des colonnes ne change **pas** grand-chose à lui
> seul : deux lignes strictement identiques (mêmes valeurs sur **toutes** les
> colonnes du `DISTINCT`) se retrouvent adjacentes après le tri **quel que soit**
> l'ordre des clés. L'idée répandue selon laquelle « trier d'abord par la colonne la
> plus cardinale rend les doublons plus adjacents » est donc **inexacte** — c'est
> d'ailleurs visible sur l'illustration ci-dessous : les trois lignes `(1,100)` sont
> contiguës dans les **deux** ordres de tri.

```
Données :  (1,100) (1,200) (2,100) (1,100) (2,200) (1,100)

Tri (client_id, produit_id)        Tri (produit_id, client_id)
   1 | 100  ┐                          100 | 1  ┐
   1 | 100  ├ 3 doublons contigus      100 | 1  ├ 3 doublons contigus
   1 | 100  ┘                          100 | 1  ┘
   1 | 200                             100 | 2
   2 | 100                             200 | 1
   2 | 200                             200 | 2
```

Dans les deux cas, les doublons sont adjacents : l'ordre des colonnes **n'accélère
pas** l'élimination par lui-même.

Là où l'ordre des colonnes compte vraiment, c'est lorsqu'il permet de **réutiliser un
ordre de tri déjà disponible** :

1. Un **index** ordonné sur `(produit_id, client_id)` peut alimenter directement le
   `DISTINCT` si ses clés sont réorganisées dans ce même ordre → plus aucun tri
   (`Index Only Scan` + `Unique`) ;
2. Un **`ORDER BY`** présent dans la requête peut partager son tri avec celui du
   `DISTINCT`, au lieu d'imposer deux tris successifs.

C'est exactement ce que fait l'optimisation PostgreSQL 18 décrite plus bas : elle ne
« trie pas par cardinalité », elle **aligne les clés du `DISTINCT` sur un ordre déjà
exploitable**. En l'absence d'un tel ordre, le planificateur préfère le plus souvent
un **`HashAggregate`** (aucun tri), où l'ordre des colonnes est sans effet.

---

## Le Problème Avant PostgreSQL 18

### Ordre Fixe = Performances Suboptimales

Avant PostgreSQL 18, le planificateur utilisait **l'ordre des colonnes tel qu'écrit** dans la requête :

```sql
-- PostgreSQL 17 et antérieurs
SELECT DISTINCT client_id, produit_id FROM commandes;
```

Le planificateur trie par `(client_id, produit_id)` dans **cet ordre précis** — même
s'il existe un **index** sur `(produit_id, client_id)` qui, réutilisé, éviterait
complètement l'étape de tri.

### La Responsabilité du Développeur

Le développeur devait **manuellement** réorganiser les colonnes pour faire coïncider
le `DISTINCT` avec un index existant (et ainsi supprimer le tri) :

```sql
-- Réécriture manuelle pour matcher un index sur (produit_id, client_id)
SELECT DISTINCT produit_id, client_id FROM commandes;
```

Mais cela pose plusieurs problèmes :

1. **Nécessite de connaître les statistiques** de chaque colonne  
2. **Change l'ordre des colonnes dans le résultat** (ce qui peut casser le code applicatif)  
3. **Difficile à maintenir** si les données évoluent  
4. **Fastidieux** pour les requêtes complexes avec beaucoup de colonnes

---

## La Solution PostgreSQL 18 : Réorganisation Automatique

### Une Optimisation Intelligente

> ⚠️ **Mise au point importante (vérifié sur PostgreSQL 18.4).** L'objectif réel de
> cette optimisation n'est **pas** de « trier d'abord par la colonne la plus
> cardinale ». Son but est de **réorganiser les colonnes du `DISTINCT` (ou du
> `GROUP BY`) pour réutiliser un ordre de tri DÉJÀ disponible** — typiquement
> l'ordre d'un **index existant** ou celui d'un **`ORDER BY`** présent dans la
> requête — afin d'**éviter un tri** (voire de permettre un `Index Only Scan`).
> Quand aucun ordre n'est réutilisable, le planificateur choisit le plus souvent un
> **`HashAggregate`** (qui ne trie pas du tout), et l'ordre des clés n'a alors aucun
> impact. Les sections ci-dessous illustrent le principe avec la cardinalité, mais
> retenez que le déclencheur concret est la **réutilisation d'un ordre de tri**.

PostgreSQL 18 peut donc **réorganiser** l'ordre des clés de regroupement pour
correspondre à un ordre de tri exploitable, tout en **conservant l'ordre original
des colonnes dans le résultat**.

### Comment Cela Fonctionne ?

1. **Recherche d'un ordre de tri réutilisable** : index multi-colonnes couvrant les
   colonnes du `DISTINCT`, ou `pathkeys` d'un `ORDER BY` de la requête

2. **Réorganisation des clés** : le planificateur permute l'ordre des colonnes du
   `DISTINCT`/`GROUP BY` pour qu'il corresponde à cet ordre déjà disponible

3. **Élimination du tri** : si la correspondance réussit, le `Sort` explicite
   disparaît (réutilisation de l'index → `Index Only Scan`, ou tri unique partagé
   avec l'`ORDER BY`)

4. **Projection finale** : le résultat reste renvoyé dans l'ordre des colonnes
   spécifié par la requête originale

### Exemple Concret

```sql
-- Index existant exploitable
CREATE INDEX idx_pc ON commandes(produit_id, client_id);

-- Requête écrite par le développeur (ordre "inversé" par rapport à l'index)
SELECT DISTINCT client_id, produit_id FROM commandes;
```

**Ce que fait PostgreSQL 18 en interne :**

1. Repère l'index `idx_pc` ordonné sur `(produit_id, client_id)`
2. Réorganise les clés du `DISTINCT` en `(produit_id, client_id)` pour matcher l'index
3. Exécute un **`Index Only Scan` + `Unique`** — **aucun `Sort`** (vérifié par `EXPLAIN`)
4. Renvoie les colonnes dans l'ordre demandé `(client_id, produit_id)`

**Résultat :** le tri est complètement évité grâce à l'index, et l'ordre des colonnes
du résultat est respecté.

---

## Exemples Détaillés

### Exemple 1 : Deux Colonnes — l'effet d'un index réutilisable

**Configuration :**
```sql
CREATE TABLE ventes (
    id SERIAL PRIMARY KEY,
    pays VARCHAR(50),
    code_postal VARCHAR(10),
    montant NUMERIC
);
-- Insertion de 1 million de lignes
```

**Requête :**
```sql
SELECT DISTINCT pays, code_postal FROM ventes;
```

**Sans index** sur ces colonnes — PG 17 **et** PG 18 résolvent ce `DISTINCT` par un
**`HashAggregate`** (aucun tri ; l'ordre des colonnes est sans effet). Il n'y a ici
**rien** à réorganiser.

**Avec un index sur `(code_postal, pays)`** — PG 18 réorganise les clés du `DISTINCT`
pour réutiliser cet index et produire le résultat **sans tri** (`Index Only Scan` +
`Unique`), là où PG 17 aurait ignoré l'index (clés non réorganisées) et trié. Le gain
vient de **la réutilisation de l'index**, pas d'un « tri par cardinalité ».

### Exemple 2 : Trois Colonnes

```sql
CREATE TABLE events (
    id SERIAL PRIMARY KEY,
    categorie VARCHAR(20),
    ville VARCHAR(100),
    utilisateur_id INTEGER
);
```

**Requête :**
```sql
SELECT DISTINCT categorie, ville, utilisateur_id FROM events;
```

Sans index couvrant ces trois colonnes, le plan est un `HashAggregate` (sans tri).
S'il existe un index sur, par exemple, `(utilisateur_id, ville, categorie)`, PG 18 peut
réorganiser les clés du `DISTINCT` dans **cet ordre** pour le réutiliser et éviter le
tri. La réorganisation suit donc l'**ordre d'un index disponible**, jamais la
cardinalité.

### Exemple 3 : Quand l'Optimisation N'Apporte Rien

```sql
CREATE TABLE produits (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100),
    reference VARCHAR(50)
);

SELECT DISTINCT nom, reference FROM produits;
```

Sans index réutilisable sur ces colonnes, PG 18 fait un `HashAggregate` : l'ordre des
clés n'a aucune incidence et il n'y a rien à réorganiser. L'optimisation
`enable_distinct_reordering` ne change le plan que lorsqu'un ordre de tri exploitable
(index ou `ORDER BY`) existe.

---

## Vérifier l'Optimisation avec EXPLAIN

Vous pouvez utiliser `EXPLAIN` pour observer comment PostgreSQL réorganise les colonnes.

### Avant PostgreSQL 18

```sql
EXPLAIN (VERBOSE)  
SELECT DISTINCT client_id, produit_id FROM commandes;  
```

**Résultat (PostgreSQL 17) :**
```
Unique
  Output: client_id, produit_id
  ->  Sort
        Sort Key: commandes.client_id, commandes.produit_id
        ->  Seq Scan on commandes
```

L'ordre de tri suit **exactement** l'ordre de la requête.

### Avec PostgreSQL 18 — sans index réutilisable

```sql
EXPLAIN (VERBOSE)  
SELECT DISTINCT client_id, produit_id FROM commandes;  
```

**Résultat réel (PostgreSQL 18.4, aucun index sur ces colonnes) :**
```
HashAggregate
  Output: client_id, produit_id
  Group Key: commandes.client_id, commandes.produit_id
  ->  Seq Scan on public.commandes
```

⚠️ Ici **aucun tri** n'est effectué : le planificateur choisit un `HashAggregate`.
L'ordre des clés est donc sans importance et la « réorganisation » n'a rien à
réutiliser. C'est le cas le plus courant pour un `DISTINCT` simple.

### Avec PostgreSQL 18 — quand un index est réutilisable

C'est là que `enable_distinct_reordering` agit réellement : s'il existe un index
ordonné sur `(produit_id, client_id)`, PG 18 permute les clés du `DISTINCT` pour le
réutiliser.

```sql
CREATE INDEX idx_pc ON commandes(produit_id, client_id);

EXPLAIN (VERBOSE)  
SELECT DISTINCT client_id, produit_id FROM commandes;  
```

**Résultat réel (PostgreSQL 18.4) :**
```
Unique
  Output: client_id, produit_id
  ->  Index Only Scan using idx_pc on public.commandes
        Output: client_id, produit_id
```

→ **Plus aucun `Sort`** : les clés ont été réorganisées en `(produit_id, client_id)`
pour réutiliser l'index, et l'ordre des colonnes du résultat reste celui demandé.

Avec `SET enable_distinct_reordering = off`, PG ignore cet index et retombe sur un
`Sort` explicite (`Sort Key: client_id, produit_id`) — c'est exactement ce que fait
PostgreSQL 17 et antérieurs.

## Impact sur les Performances

### Gains Typiques

Les gains de performance dépendent de plusieurs facteurs :

| Situation | Gain |
|-----------|------|
| **Index réutilisable** couvrant les colonnes du `DISTINCT`, dans un ordre exploitable | **Élevé** : le `Sort` disparaît (`Index Only Scan` + `Unique`) |
| **`ORDER BY`** partageable avec le tri du `DISTINCT` | **Modéré** : un seul tri au lieu de deux |
| **Aucun ordre réutilisable** (le plan est un `HashAggregate`) | **Nul** : il n'y a rien à réorganiser |

### Quand les Gains sont Maximaux

1. **Un index existe** sur (un sur-ensemble de) les colonnes du `DISTINCT`, dans un ordre que la réorganisation peut réutiliser — c'est **la** condition déterminante  
2. La **table est volumineuse** : le tri ainsi évité aurait été coûteux  
3. La requête combine `DISTINCT` et `ORDER BY` dont les tris peuvent être partagés

---

## Considérations et Limites

### 1. Nécessite des Statistiques à Jour

Pour que l'optimisation fonctionne, PostgreSQL doit avoir des **statistiques précises** :

```sql
-- Mettre à jour les statistiques
ANALYZE commandes;
```

Si les statistiques sont obsolètes, le planificateur peut faire un mauvais choix.

### 2. Automatique = Pas de Contrôle Direct

L'optimisation est **entièrement automatique**. Vous ne pouvez pas forcer un ordre de tri spécifique (sauf à désactiver l'optimisation, ce qui est rarement utile).

### 3. Overhead de Réorganisation

Il y a un **léger coût** pour réorganiser les colonnes dans le résultat final, mais ce coût est **négligeable** comparé au gain sur le tri.

### 4. Compatibilité

Cette optimisation est **transparente** et ne change pas le comportement observable de la requête. Le résultat est **strictement identique**.

---

## Bonnes Pratiques

### 1. Maintenir les Statistiques à Jour

Assurez-vous que l'autovacuum est actif ou exécutez `ANALYZE` régulièrement :

```sql
-- Sur une table spécifique
ANALYZE commandes;

-- Sur toute la base
ANALYZE;
```

### 2. Vérifier avec EXPLAIN sur les Requêtes Critiques

Pour les requêtes de production importantes, vérifiez que l'optimisation est appliquée :

```sql
EXPLAIN (ANALYZE, VERBOSE, BUFFERS)  
SELECT DISTINCT col1, col2, col3 FROM ma_table;  
```

### 3. Ne Pas Sur-Optimiser Manuellement

Avec PostgreSQL 18, **ne réorganisez plus manuellement** les colonnes DISTINCT pour optimiser. Laissez le planificateur faire son travail :

```sql
-- ❌ Plus nécessaire dans PostgreSQL 18
SELECT DISTINCT produit_id, client_id FROM commandes; -- ordre "optimisé" manuellement

-- ✅ Écrivez naturellement
SELECT DISTINCT client_id, produit_id FROM commandes; -- PG 18 optimise automatiquement
```

### 4. Augmenter les Statistiques si Nécessaire

Pour des colonnes avec beaucoup de valeurs distinctes, augmentez le niveau de statistiques :

```sql
ALTER TABLE commandes  
ALTER COLUMN produit_id SET STATISTICS 1000;  

ANALYZE commandes;
```

Cela donne au planificateur des informations plus précises.

## Cas d'Usage Réels

### Cas 1 : Logs et Événements

```sql
-- Table de logs avec millions d'entrées
CREATE TABLE logs (
    timestamp TIMESTAMPTZ,
    user_id INTEGER,
    action VARCHAR(50),
    ip_address INET
);

-- Quelles combinaisons utilisateur/action existent ?
SELECT DISTINCT user_id, action FROM logs;
```

- `user_id` : haute cardinalité (~100k utilisateurs)  
- `action` : basse cardinalité (~20 actions)

Sans index ni `ORDER BY`, PostgreSQL 18 résout ce `DISTINCT` par un **`HashAggregate`**
(aucun tri — l'ordre des colonnes est sans effet). La réorganisation des clés
n'intervient que s'il existe un **index** à réutiliser (p. ex. sur `(user_id, action)`),
permettant alors un `Index Only Scan` sans tri.

### Cas 2 : E-commerce

```sql
-- Quelles combinaisons produit/catégorie ont été vendues ?
SELECT DISTINCT categorie, sous_categorie, produit_id  
FROM ventes;  
```

- `categorie` : 10 valeurs  
- `sous_categorie` : 50 valeurs  
- `produit_id` : 10000 valeurs

S'il existe un index couvrant ces trois colonnes, PostgreSQL 18 réorganise les clés du
`DISTINCT` pour le réutiliser et **éviter le tri**. Sans index, c'est un `HashAggregate`
(l'ordre des colonnes n'a aucune incidence).

### Cas 3 : Analyse Géographique

```sql
-- Quelles combinaisons pays/région/ville ont des clients ?
SELECT DISTINCT pays, region, ville  
FROM clients;  
```

Hiérarchie naturelle : pays < région < ville (cardinalité croissante)

Si un index `(ville, region, pays)` existe, PostgreSQL 18 aligne les clés du `DISTINCT`
sur cet ordre pour le réutiliser (tri évité). Sinon, `HashAggregate` — sans tri.

## Interaction avec D'autres Optimisations

### DISTINCT vs GROUP BY

Une alternative à `DISTINCT` est `GROUP BY` sans agrégation :

```sql
-- Ces deux requêtes sont équivalentes
SELECT DISTINCT client_id, produit_id FROM commandes;  
SELECT client_id, produit_id FROM commandes GROUP BY client_id, produit_id;  
```

PostgreSQL 18 applique **également** la réorganisation automatique avec `GROUP BY` !

### Combinaison avec l'Indexation

Si un index existe sur les colonnes DISTINCT, PostgreSQL peut utiliser un **Index-Only Scan** :

```sql
-- Créer un index
CREATE INDEX idx_commandes_client_produit  
ON commandes(client_id, produit_id);  

-- La requête peut utiliser l'index
SELECT DISTINCT client_id, produit_id FROM commandes;
```

PostgreSQL 18 choisit automatiquement entre :
1. Utiliser l'index (si efficace)  
2. Scanner la table et réorganiser le tri

---

## Différence avec DISTINCT ON

### DISTINCT (simple)

```sql
SELECT DISTINCT col1, col2 FROM ma_table;
-- Retourne toutes les combinaisons uniques
```

### DISTINCT ON (PostgreSQL spécifique)

```sql
SELECT DISTINCT ON (col1) col1, col2 FROM ma_table ORDER BY col1, col2;
-- Retourne une seule ligne par valeur distincte de col1
```

**Important :** La réorganisation automatique de PostgreSQL 18 s'applique à `DISTINCT`, mais **pas à `DISTINCT ON`** car l'ordre des colonnes dans `DISTINCT ON` a une **signification sémantique** différente.

## Monitoring et Débogage

### Vérifier les Statistiques

```sql
-- Voir les statistiques d'une colonne
SELECT
    tablename,
    attname,
    n_distinct,
    correlation
FROM pg_stats  
WHERE tablename = 'commandes'  
  AND attname IN ('client_id', 'produit_id');
```

**Résultat exemple :**
```
tablename  | attname     | n_distinct | correlation
-----------|-------------|------------|------------
commandes  | client_id   |       1000 |        0.85  
commandes  | produit_id  |      10000 |        0.12  
```

Plus `n_distinct` est élevé, plus la cardinalité est haute.

### Comparer les Plans d'Exécution

```sql
-- Forcer l'ancien comportement (désactiver l'optimisation)
SET enable_distinct_reordering = off;  
EXPLAIN ANALYZE SELECT DISTINCT client_id, produit_id FROM commandes;  

-- Réactiver et comparer
SET enable_distinct_reordering = on;  
EXPLAIN ANALYZE SELECT DISTINCT client_id, produit_id FROM commandes;  
```

> Le nom exact du GUC est bien `enable_distinct_reordering` (suffixe `-ing`).  
> Le GUC équivalent pour `GROUP BY` s'appelle `enable_group_by_reordering`.  
> Les deux sont activés à `on` par défaut en PG 18.

---

## Conclusion

La **réorganisation automatique des colonnes DISTINCT** dans PostgreSQL 18 est une optimisation puissante qui :

- ✅ Améliore automatiquement les performances des requêtes DISTINCT  
- ✅ Élimine le besoin de réorganiser manuellement les colonnes  
- ✅ S'adapte dynamiquement aux données réelles  
- ✅ Est totalement transparente pour le développeur  
- ✅ Ne change pas le résultat de la requête

Cette fonctionnalité s'inscrit dans la philosophie de PostgreSQL : **un planificateur intelligent qui optimise automatiquement vos requêtes**, vous permettant de vous concentrer sur la logique métier plutôt que sur les détails de performance.

**Message clé pour les débutants :** Écrivez vos requêtes DISTINCT de manière naturelle et logique. PostgreSQL 18 se charge automatiquement de les optimiser en réorganisant l'ordre de tri en fonction des statistiques réelles de vos données.
