# 13.5.1. Index Partiels (WHERE clause)

## Introduction

Un **index partiel** est un index qui ne porte que sur un sous-ensemble des lignes d'une table, défini par une condition WHERE. Contrairement à un index classique qui indexe toutes les lignes de la table, un index partiel n'indexe que les lignes qui satisfont un prédicat spécifique.

Cette technique permet d'optimiser à la fois la performance des requêtes et l'utilisation de l'espace disque, en ne créant un index que là où il est réellement nécessaire.

---

## Concept Fondamental

### Index Classique vs Index Partiel

**Index classique :**
```sql
CREATE INDEX idx_commandes_statut ON commandes(statut);
```
→ Indexe **toutes** les lignes de la table `commandes`, quel que soit leur statut.

**Index partiel :**
```sql
CREATE INDEX idx_commandes_statut_en_cours  
ON commandes(statut)  
WHERE statut = 'en_cours';  
```
→ N'indexe **que** les commandes dont le statut est 'en_cours'.

---

## Pourquoi Utiliser des Index Partiels ?

### 1. **Réduction de la Taille de l'Index**

Un index partiel occupe moins d'espace disque car il contient moins d'entrées.

**Exemple concret :**
```sql
-- Table avec 1 million de commandes
-- 95% sont livrées, 5% sont en cours (50 000 lignes)

-- Index classique : indexe 1 000 000 de lignes
CREATE INDEX idx_all ON commandes(statut);

-- Index partiel : indexe seulement 50 000 lignes
CREATE INDEX idx_partial ON commandes(statut)  
WHERE statut IN ('en_cours', 'en_preparation');  
```

**Avantage :** L'index partiel sera environ 20 fois plus petit !

### 2. **Amélioration de la Performance**

Un index plus petit :
- Tient mieux en mémoire (cache)
- Se parcourt plus rapidement
- Nécessite moins de pages disque à lire

### 3. **Maintenance Plus Légère**

Lors des opérations d'écriture (INSERT, UPDATE, DELETE), PostgreSQL doit maintenir les index. Un index partiel :
- Est mis à jour uniquement si la ligne satisfait la condition WHERE
- Réduit le coût des écritures sur les lignes non indexées

---

## Syntaxe et Utilisation

### Syntaxe Générale

```sql
CREATE INDEX nom_index  
ON nom_table(colonne1, colonne2, ...)  
WHERE condition;  
```

### Règles Importantes

1. **La condition WHERE doit être simple et stable**
   - Éviter les fonctions volatiles (RANDOM(), NOW(), etc.)
   - Privilégier les comparaisons sur des valeurs constantes

2. **PostgreSQL utilise l'index partiel uniquement si la requête contient la même condition**
   - La condition de la requête doit être **identique ou plus restrictive** que celle de l'index

---

## Exemples Pratiques

### Exemple 1 : Indexer les Lignes Actives

**Contexte :** Une table d'utilisateurs où 99% sont actifs et 1% sont supprimés (soft delete).

```sql
-- Table utilisateurs
CREATE TABLE utilisateurs (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100),
    email VARCHAR(255),
    est_actif BOOLEAN DEFAULT TRUE,
    date_creation TIMESTAMP DEFAULT NOW()
);

-- Index partiel : uniquement les utilisateurs actifs
CREATE INDEX idx_utilisateurs_actifs  
ON utilisateurs(email)  
WHERE est_actif = TRUE;  
```

**Requête qui utilisera l'index :**
```sql
-- ✅ Utilise l'index partiel
SELECT * FROM utilisateurs  
WHERE email = 'user@example.com'  
AND est_actif = TRUE;  
```

**Requête qui N'utilisera PAS l'index :**
```sql
-- ❌ Ne peut pas utiliser l'index partiel (pas de condition sur est_actif)
SELECT * FROM utilisateurs  
WHERE email = 'user@example.com';  
```

### Exemple 2 : Indexer les Données Récentes

**Contexte :** Une table de logs où on interroge principalement les 30 derniers jours.

```sql
-- Table de logs
CREATE TABLE logs (
    id BIGSERIAL PRIMARY KEY,
    niveau VARCHAR(20),
    message TEXT,
    date_creation TIMESTAMP DEFAULT NOW()
);

-- Index partiel : logs récents (30 derniers jours)
CREATE INDEX idx_logs_recents  
ON logs(niveau, date_creation)  
WHERE date_creation > NOW() - INTERVAL '30 days';  
```

**Problème :** Cette approche a une limite ! La condition `NOW() - INTERVAL '30 days'` change chaque jour.

**Solution alternative :**
```sql
-- Meilleure approche : partitionner la table par date
-- Ou utiliser un index partiel avec une date fixe et le recréer périodiquement
```

### Exemple 3 : Indexer les Valeurs Non-NULL

**Contexte :** Une colonne optionnelle qui est NULL dans 90% des cas.

```sql
-- Table de produits
CREATE TABLE produits (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    code_promo VARCHAR(50), -- NULL dans 90% des cas
    prix NUMERIC(10,2)
);

-- Index partiel : uniquement les produits avec promo
CREATE INDEX idx_produits_avec_promo  
ON produits(code_promo)  
WHERE code_promo IS NOT NULL;  
```

**Requête optimisée :**
```sql
-- ✅ Utilise l'index partiel
SELECT * FROM produits  
WHERE code_promo = 'NOEL2025';  
```

### Exemple 4 : Combiner Plusieurs Conditions

```sql
-- Table de commandes
CREATE TABLE commandes (
    id SERIAL PRIMARY KEY,
    client_id INTEGER,
    statut VARCHAR(50),
    montant NUMERIC(10,2),
    date_commande TIMESTAMP DEFAULT NOW()
);

-- Index partiel : commandes à traiter (non livrées et montant > 100€)
CREATE INDEX idx_commandes_a_traiter  
ON commandes(client_id, date_commande)  
WHERE statut IN ('en_cours', 'en_preparation', 'expedie')  
AND montant > 100;  
```

**Requête qui utilise l'index :**
```sql
-- ✅ Conditions compatibles avec l'index
SELECT * FROM commandes  
WHERE client_id = 12345  
AND statut = 'en_cours'  
AND montant > 100  
ORDER BY date_commande DESC;  
```

## Cas d'Usage Typiques

### 1. **Soft Delete (Suppression Logique)**

```sql
CREATE INDEX idx_actifs ON utilisateurs(email)  
WHERE est_supprime = FALSE;  
```

**Scénario :** 95% des lignes sont actives, 5% sont supprimées.

### 2. **États Temporaires**

```sql
CREATE INDEX idx_en_attente ON taches(priorite, date_creation)  
WHERE statut = 'en_attente';  
```

**Scénario :** On interroge principalement les tâches en attente.

### 3. **Données Anormales ou Exceptionnelles**

```sql
CREATE INDEX idx_erreurs ON transactions(date_creation)  
WHERE statut = 'erreur';  
```

**Scénario :** Seulement 0.1% des transactions échouent, mais on doit les retrouver rapidement.

### 4. **Valeurs Spécifiques Fréquemment Recherchées**

```sql
CREATE INDEX idx_admin ON utilisateurs(date_derniere_connexion)  
WHERE role = 'admin';  
```

**Scénario :** Les administrateurs représentent 1% des utilisateurs mais font 50% des requêtes.

### 5. **Exclusion de Valeurs Communes**

```sql
CREATE INDEX idx_prioritaire ON tickets(date_creation)  
WHERE priorite != 'basse';  
```

**Scénario :** 80% des tickets sont de priorité basse et rarement consultés.

## Vérifier l'Utilisation de l'Index

Pour vérifier si PostgreSQL utilise bien votre index partiel, utilisez `EXPLAIN` :

```sql
EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM commandes  
WHERE statut = 'en_cours'  
AND client_id = 12345;  
```

**Sortie attendue :**
```
Index Scan using idx_commandes_statut_en_cours on commandes
  Index Cond: (client_id = 12345)
  Filter: (statut = 'en_cours'::text)
```

Si vous voyez `Seq Scan` au lieu d'`Index Scan`, l'index n'est pas utilisé !

## Pièges et Limitations

### ❌ Piège 1 : Condition de Requête Incompatible

```sql
-- Index partiel
CREATE INDEX idx_actifs ON utilisateurs(email)  
WHERE est_actif = TRUE;  

-- ❌ Cette requête N'utilisera PAS l'index
SELECT * FROM utilisateurs WHERE email = 'test@example.com';
-- Il manque la condition : AND est_actif = TRUE
```

**Solution :** Toujours inclure la condition de l'index partiel dans vos requêtes.

### ❌ Piège 2 : Fonctions Volatiles dans la Condition

```sql
-- ❌ MAUVAIS : NOW() change à chaque appel
CREATE INDEX idx_recent ON logs(niveau)  
WHERE date_creation > NOW() - INTERVAL '7 days';  
```

**Solution :** Utiliser des valeurs fixes ou recréer l'index périodiquement.

### ❌ Piège 3 : Index Partiel Trop Large

```sql
-- ❌ Cet index partiel indexe 90% de la table → Peu d'intérêt
CREATE INDEX idx_presque_tout ON commandes(client_id)  
WHERE montant > 1;  -- Presque toutes les commandes sont > 1€  
```

**Règle générale :** Un index partiel est pertinent quand il indexe **moins de 20-30%** de la table.

### ❌ Piège 4 : Condition Complexe

```sql
-- ❌ Condition trop complexe, difficile à utiliser par le planificateur
CREATE INDEX idx_complexe ON produits(nom)  
WHERE (prix > 100 AND categorie = 'Electronique')  
OR (prix > 500 AND categorie = 'Mobilier');  
```

**Solution :** Privilégier des conditions simples avec AND.

## Maintenance et Monitoring

### Lister les Index Partiels

```sql
SELECT
    schemaname,
    tablename,
    indexname,
    indexdef
FROM pg_indexes  
WHERE indexdef LIKE '%WHERE%'  
ORDER BY tablename, indexname;  
```

### Vérifier la Taille d'un Index

```sql
SELECT
    pg_size_pretty(pg_relation_size('nom_index')) AS taille_index;
```

### Comparer Index Classique vs Partiel

```sql
-- Index classique
SELECT pg_size_pretty(pg_relation_size('idx_classique'));
-- Résultat : 50 MB

-- Index partiel
SELECT pg_size_pretty(pg_relation_size('idx_partiel'));
-- Résultat : 5 MB  → 10× plus petit !
```

## Quand Utiliser un Index Partiel ?

### ✅ Utilisez un Index Partiel Si :

1. **Vous interrogez fréquemment un sous-ensemble spécifique de données**
   - Exemple : Commandes en cours, utilisateurs actifs, tickets non résolus

2. **La condition filtre une petite portion de la table (< 20-30%)**
   - Plus le filtre est sélectif, plus l'index partiel est avantageux

3. **Vous connaissez à l'avance les conditions de filtrage**
   - Les conditions doivent être prévisibles et stables

4. **La table est volumineuse et la colonne contient peu de valeurs distinctes**
   - Exemple : Statut (5 valeurs), Type (10 valeurs)

### ❌ N'Utilisez PAS un Index Partiel Si :

1. **Les conditions de filtrage varient constamment**
   - Impossible de prédire les requêtes futures

2. **La condition couvre > 50% de la table**
   - Un index classique serait plus simple et presque aussi efficace

3. **Les requêtes n'incluent pas systématiquement la condition WHERE**
   - L'index ne serait jamais utilisé

---

## Comparaison : Index Partiel vs Alternatives

| Technique | Cas d'Usage | Avantages | Inconvénients |
|-----------|-------------|-----------|---------------|
| **Index Partiel** | Filtrage sur valeurs spécifiques | Compact, rapide | Requiert condition exacte |
| **Index Classique** | Toutes les valeurs | Simple, universel | Plus volumineux |
| **Partitionnement** | Archivage, données temporelles | Isolation des données | Complexité de gestion |
| **Index sur Expression** | Transformations (LOWER, etc.) | Flexibilité | Calcul à l'indexation |

## Exemple Complet : Optimisation d'une Table de Commandes

### Contexte

```sql
-- Table avec 10 millions de commandes
CREATE TABLE commandes (
    id BIGSERIAL PRIMARY KEY,
    client_id INTEGER,
    statut VARCHAR(50), -- 'brouillon', 'en_cours', 'livree', 'annulee'
    montant NUMERIC(10,2),
    date_commande TIMESTAMP DEFAULT NOW()
);

-- Distribution des statuts :
-- brouillon : 5% (500 000)
-- en_cours : 3% (300 000)
-- livree : 90% (9 000 000)
-- annulee : 2% (200 000)
```

### Stratégie d'Indexation

```sql
-- ❌ MAUVAIS : Index classique sur statut
CREATE INDEX idx_statut ON commandes(statut);
-- Problème : Indexe 10M de lignes, mais on interroge surtout 'en_cours'

-- ✅ BON : Index partiel sur commandes actives
CREATE INDEX idx_commandes_actives  
ON commandes(client_id, date_commande)  
WHERE statut IN ('brouillon', 'en_cours');  
-- Avantage : Indexe seulement 800 000 lignes (8% de la table)

-- ✅ BON : Index partiel sur commandes livrées récentes
CREATE INDEX idx_commandes_livrees_recentes  
ON commandes(client_id, date_commande DESC)  
WHERE statut = 'livree'  
AND date_commande > '2025-01-01';  
-- Avantage : Optimise les recherches sur l'historique récent
```

### Requêtes Optimisées

```sql
-- Requête 1 : Commandes actives d'un client
-- ✅ Utilise idx_commandes_actives
SELECT * FROM commandes  
WHERE client_id = 12345  
AND statut = 'en_cours'  
ORDER BY date_commande DESC;  

-- Requête 2 : Historique récent d'un client
-- ✅ Utilise idx_commandes_livrees_recentes
SELECT * FROM commandes  
WHERE client_id = 12345  
AND statut = 'livree'  
AND date_commande > '2025-01-01'  
ORDER BY date_commande DESC;  
```

### Gains Obtenus

- **Taille des index réduite de 80%**  
- **Performance des requêtes améliórée de 3-5×**  
- **Coût de maintenance (INSERT/UPDATE) réduit**

---

## Bonnes Pratiques

### ✅ DO (À Faire)

1. **Nommer explicitement vos index partiels**
   ```sql
   -- ✅ Nom clair et descriptif
   CREATE INDEX idx_utilisateurs_actifs ON utilisateurs(email)
   WHERE est_actif = TRUE;
   ```

2. **Documenter la logique de l'index**
   ```sql
   COMMENT ON INDEX idx_commandes_actives IS
   'Index partiel pour requêtes sur commandes non livrées (8% de la table)';
   ```

3. **Tester avec EXPLAIN avant de créer l'index**
   ```sql
   EXPLAIN SELECT * FROM commandes
   WHERE statut = 'en_cours' AND client_id = 12345;
   ```

4. **Monitorer l'utilisation de vos index**
   ```sql
   SELECT * FROM pg_stat_user_indexes
   WHERE indexrelname = 'idx_commandes_actives';
   ```

### ❌ DON'T (À Éviter)

1. **Ne créez pas d'index partiels sur des conditions changeantes**
   ```sql
   -- ❌ NOW() change chaque jour
   WHERE date_creation > NOW() - INTERVAL '7 days'
   ```

2. **N'oubliez pas la condition WHERE dans vos requêtes**
   ```sql
   -- ❌ L'index ne sera pas utilisé
   SELECT * FROM utilisateurs WHERE email = 'test@example.com';
   -- Il manque : AND est_actif = TRUE
   ```

3. **Ne créez pas trop d'index partiels concurrents**
   ```sql
   -- ❌ Trop d'index = confusion et coût de maintenance
   CREATE INDEX idx1 ON table(col) WHERE statut = 'A';
   CREATE INDEX idx2 ON table(col) WHERE statut = 'B';
   CREATE INDEX idx3 ON table(col) WHERE statut = 'C';
   -- Mieux : Un index classique ou partitionner la table
   ```

---

## Résumé

### Points Clés

1. **Un index partiel n'indexe qu'un sous-ensemble de lignes défini par une condition WHERE**

2. **Avantages :**
   - Réduit la taille de l'index (espace disque)
   - Améliore la performance (cache, parcours)
   - Diminue le coût de maintenance (écritures)

3. **Condition d'utilisation :**
   - La requête doit contenir la **même condition** que l'index partiel

4. **Cas d'usage idéaux :**
   - Soft delete (lignes actives)
   - États temporaires (en_cours, en_attente)
   - Valeurs exceptionnelles (erreurs, priorités hautes)
   - Données récentes ou spécifiques

5. **Règle d'or :**
   - Un index partiel est pertinent quand il indexe **< 20-30%** de la table
