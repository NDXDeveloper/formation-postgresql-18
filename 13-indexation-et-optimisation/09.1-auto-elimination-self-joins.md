# 13.9.1. Auto-élimination des Self-Joins (PostgreSQL 18)

## Introduction

PostgreSQL 18 introduit une optimisation intelligente du planificateur de requêtes : **l'auto-élimination des self-joins**. Cette fonctionnalité permet au moteur de base de données de détecter et d'éliminer automatiquement certaines jointures redondantes d'une table avec elle-même, améliorant ainsi significativement les performances sans modification du code SQL.

Pour comprendre cette nouveauté, commençons par les bases.

---

## Qu'est-ce qu'un Self-Join ?

Un **self-join** (auto-jointure en français) est une opération où une table est jointe **avec elle-même**. Cela peut sembler étrange au premier abord, mais c'est une technique très utile dans certains scénarios.

### Exemple Simple

Imaginons une table `employes` qui contient des employés et leurs managers :

```sql
CREATE TABLE employes (
    id INTEGER PRIMARY KEY,
    nom VARCHAR(100),
    manager_id INTEGER REFERENCES employes(id)
);
```

Pour afficher chaque employé avec le nom de son manager, on doit joindre la table avec elle-même :

```sql
SELECT
    e.nom AS employe,
    m.nom AS manager
FROM employes e  
LEFT JOIN employes m ON e.manager_id = m.id;  
```

Ici, `employes` apparaît deux fois dans la requête :
- Une fois avec l'alias `e` (pour l'employé)
- Une fois avec l'alias `m` (pour le manager)

C'est un self-join légitime et nécessaire.

---

## Le Problème : Self-Joins Redondants

Dans certaines situations, le SQL généré par des ORM (Object-Relational Mapping) ou des outils de reporting peut créer des **self-joins inutiles**. Ces jointures redondantes n'apportent aucune information supplémentaire mais consomment des ressources.

### Exemple de Self-Join Redondant

```sql
SELECT e1.nom, e1.salaire  
FROM employes e1  
INNER JOIN employes e2 ON e1.id = e2.id  
WHERE e1.departement = 'IT';  
```

Dans cette requête, la jointure entre `e1` et `e2` est **totalement inutile** :
- On joint `employes` avec lui-même sur `id = id`
- La clé primaire garantit une correspondance 1:1
- Aucune colonne de `e2` n'est utilisée
- Cette jointure n'ajoute aucune valeur

Cette requête devrait simplement être :

```sql
SELECT nom, salaire  
FROM employes  
WHERE departement = 'IT';  
```

---

## Pourquoi Cela Arrive-t-il ?

Les self-joins redondants apparaissent fréquemment dans plusieurs contextes :

### 1. Génération Automatique de Requêtes par des ORM

Les frameworks ORM (comme Hibernate, Entity Framework, SQLAlchemy) génèrent parfois du SQL non optimal lorsqu'ils gèrent des relations complexes :

```python
# Code Python avec SQLAlchemy (exemple simplifié)
query = session.query(Employee.name, Employee.salary)\
    .join(Employee, Employee.id == Employee.id)\
    .filter(Employee.department == 'IT')
```

Ce code peut générer un self-join inutile.

### 2. Outils de Business Intelligence

Les outils BI construisant des requêtes dynamiquement peuvent créer des jointures superflues lors de la composition de multiples filtres ou vues.

### 3. Vues Imbriquées

Des vues SQL construites les unes sur les autres peuvent accumuler des self-joins redondants :

```sql
CREATE VIEW vue_employes_base AS  
SELECT * FROM employes e1  
INNER JOIN employes e2 ON e1.id = e2.id;  

-- Puis une autre vue utilisant la première...
CREATE VIEW vue_employes_it AS  
SELECT * FROM vue_employes_base  
WHERE departement = 'IT';  
```

---

## La Solution PostgreSQL 18 : Auto-élimination

Avant PostgreSQL 18, ces self-joins redondants étaient **exécutés** tels quels, même s'ils étaient inutiles. Le développeur devait identifier et corriger manuellement ces problèmes.

### Comment Fonctionne l'Optimisation ?

Le planificateur de requêtes de PostgreSQL 18 analyse désormais la structure des requêtes et détecte automatiquement les self-joins qui respectent certains critères :

**Critères de Détection :**

1. **Jointure sur clé primaire ou unique** : La jointure doit se faire sur une colonne garantissant l'unicité (clé primaire, contrainte UNIQUE)

2. **Relation 1:1 garantie** : La contrainte d'unicité assure qu'une ligne de la première occurrence correspond à exactement une ligne de la seconde occurrence

3. **Égalité prouvant l'identité des lignes** : la condition de jointure contient une égalité (`e1.col = e2.col`) sur la/les colonne(s) UNIQUE/PRIMARY KEY. ⚠️ **Idée fausse répandue** : il n'est PAS nécessaire que les colonnes de la seconde occurrence soient inutilisées. Le planificateur **réécrit** simplement les références `e2.colonne` vers `e1.colonne` (puisque les deux désignent la même ligne), puis supprime la jointure.

4. **Type de jointure compatible** : INNER JOIN ou LEFT JOIN dont la condition se réduit à cette égalité sur la clé unique

### Que Fait PostgreSQL 18 ?

Lorsque ces conditions sont remplies, PostgreSQL 18 **élimine complètement le self-join** du plan d'exécution, transformant efficacement la requête en une simple lecture de table.

---

## Exemples Concrets

### Exemple 1 : Self-Join Simple Éliminé

**Requête Originale :**
```sql
SELECT e1.nom, e1.email  
FROM employes e1  
INNER JOIN employes e2 ON e1.id = e2.id  
WHERE e1.salaire > 50000;  
```

**Ce Que PostgreSQL 18 Exécute Réellement :**
```sql
-- Le planificateur transforme automatiquement en :
SELECT nom, email  
FROM employes  
WHERE salaire > 50000;  
```

Le self-join a été éliminé car :
- ✅ Jointure sur `id` (clé primaire) → `e1` et `e2` sont prouvés identiques  
- ✅ Relation 1:1 garantie  
- ✅ INNER JOIN  
- ℹ️ Ici `e2` n'est pas référencée — mais ce n'est **pas** une condition de l'élimination (cf. Exemple 3, où `e2.salaire` est utilisée et le self-join est tout de même supprimé)

### Exemple 2 : Avec Contrainte UNIQUE

```sql
CREATE TABLE produits (
    code_produit VARCHAR(20) UNIQUE,
    nom VARCHAR(100),
    prix NUMERIC
);

SELECT p1.nom, p1.prix  
FROM produits p1  
INNER JOIN produits p2 ON p1.code_produit = p2.code_produit  
WHERE p1.prix > 100;  
```

Même avec une contrainte UNIQUE (et non PRIMARY KEY), PostgreSQL 18 peut éliminer le self-join.

### Exemple 3 : Self-Join Éliminé MÊME en Utilisant la Seconde Table

```sql
SELECT e1.nom, e2.salaire  
FROM employes e1  
INNER JOIN employes e2 ON e1.id = e2.id  
WHERE e1.departement = 'IT';  
```

✅ **Ce self-join EST quand même éliminé** par PostgreSQL 18. Comme la jointure se fait sur la clé primaire `id`, le planificateur prouve que `e1` et `e2` désignent **la même ligne** : il réécrit donc `e2.salaire` en `e1.salaire`, puis supprime la jointure. `EXPLAIN` ne montre alors qu'**un seul** `Seq Scan` (vérifié sur PostgreSQL 18.4). Utiliser une colonne de `e2` n'empêche donc **pas** l'optimisation — c'est une idée fausse fréquente.

### Exemple 4 : Self-Join NON Éliminé (Pas de Contrainte Unique)

```sql
SELECT c1.nom, c1.email  
FROM clients c1  
INNER JOIN clients c2 ON c1.email = c2.email  
WHERE c1.ville = 'Paris';  
```

❌ **Ce self-join NE SERA PAS éliminé** si `email` n'a pas de contrainte UNIQUE. Plusieurs clients pourraient avoir le même email, créant une relation 1:N.

---

## Bénéfices de Cette Optimisation

### 1. **Performances Améliorées Automatiquement**

Sans modification du code applicatif, les requêtes s'exécutent plus rapidement :
- Moins d'opérations de lecture
- Moins de consommation mémoire
- Moins de cycles CPU

### 2. **Réduction de la Dette Technique**

Vous n'avez plus besoin de refactoriser immédiatement le code généré par les ORM qui produit des self-joins inutiles. PostgreSQL 18 compense automatiquement.

### 3. **Compatibilité Descendante**

Cette optimisation est **totalement transparente** :
- Aucune modification de syntaxe SQL requise
- Les anciennes requêtes fonctionnent exactement comme avant
- Le comportement est activé par défaut

### 4. **Gain Particulièrement Important sur Grandes Tables**

Sur des tables de plusieurs millions de lignes, éliminer un self-join peut réduire le temps d'exécution de **50% ou plus**, car on évite un scan complet supplémentaire.

---

## Vérifier l'Optimisation avec EXPLAIN

Vous pouvez utiliser la commande `EXPLAIN` pour vérifier si PostgreSQL 18 a éliminé un self-join :

### Avant PostgreSQL 18 (ou sur une requête non optimisable)

```sql
EXPLAIN  
SELECT e1.nom, e1.email  
FROM employes e1  
INNER JOIN employes e2 ON e1.id = e2.id  
WHERE e1.salaire > 50000;  
```

**Résultat (PostgreSQL 17 et antérieurs) :**
```
Hash Join
  Hash Cond: (e1.id = e2.id)
  -> Seq Scan on employes e1
       Filter: (salaire > 50000)
  -> Hash
       -> Seq Scan on employes e2
```

On voit bien les **deux scans** de la table `employes`.

### Avec PostgreSQL 18 (self-join éliminé)

```sql
EXPLAIN  
SELECT e1.nom, e1.email  
FROM employes e1  
INNER JOIN employes e2 ON e1.id = e2.id  
WHERE e1.salaire > 50000;  
```

**Résultat (PostgreSQL 18) :**
```
Seq Scan on employes e1
  Filter: (salaire > 50000)
```

Le self-join a **disparu** ! Un seul scan de table.

---

## Cas Limites et Considérations

### Quand l'Optimisation NE S'Applique PAS

1. **Pas de contrainte d'unicité sur la colonne de jointure** (cas le plus fréquent)
   ```sql
   -- client_id n'est PAS unique → relation 1:N possible : la jointure est conservée
   SELECT c1.id
   FROM commandes c1
   INNER JOIN commandes c2 ON c1.client_id = c2.client_id;
   ```
   Sans PRIMARY KEY/UNIQUE garantissant le 1:1, le planificateur ne peut pas prouver
   l'identité des lignes et conserve donc le `Hash Join` (deux scans).

2. **La condition de jointure n'est pas une égalité sur la clé unique**
   ```sql
   -- Jointure sur une inégalité : aucune identité de ligne prouvable
   SELECT e1.nom
   FROM employes e1
   INNER JOIN employes e2 ON e1.id > e2.id;
   ```

> ⚠️ En revanche, **utiliser des colonnes de la seconde occurrence** (`e2.colonne`),
> ou ajouter une condition comme `LEFT JOIN … AND e2.actif = true` sans projeter de
> colonne de `e2`, n'empêche PAS l'élimination : PostgreSQL 18 réécrit les références
> vers la première occurrence. C'est une idée fausse répandue (vérifié sur PG 18.4).

### Impact sur les Statistiques

L'élimination du self-join est une **transformation du plan d'exécution**, pas une réécriture de la requête. Les statistiques collectées (`pg_stat_statements`) montreront toujours la requête originale avec le self-join.

---

## Bonnes Pratiques

### 1. Laissez le Planificateur Faire Son Travail

Ne vous précipitez pas pour réécrire toutes vos requêtes. PostgreSQL 18 gère automatiquement de nombreux cas.

### 2. Utilisez EXPLAIN pour Vérifier

En cas de doute sur une requête critique, utilisez `EXPLAIN ANALYZE` pour confirmer que l'optimisation est appliquée :

```sql
EXPLAIN ANALYZE  
SELECT e1.nom  
FROM employes e1  
INNER JOIN employes e2 ON e1.id = e2.id  
WHERE e1.actif = true;  
```

### 3. Déclarez Vos Contraintes !

Pour que cette optimisation fonctionne, il est **crucial** d'avoir des contraintes PRIMARY KEY ou UNIQUE correctement définies :

```sql
-- ✅ BON : Permet l'optimisation
ALTER TABLE employes ADD PRIMARY KEY (id);

-- ✅ BON : Permet aussi l'optimisation
ALTER TABLE produits ADD CONSTRAINT unique_code UNIQUE (code_produit);
```

### 4. Restez Vigilant avec les ORM

Même si PostgreSQL 18 compense, il reste préférable de configurer correctement vos ORM pour éviter de générer des self-joins inutiles en premier lieu.

---

## Conclusion

L'**auto-élimination des self-joins** dans PostgreSQL 18 est une amélioration intelligente qui :

- ✅ Améliore automatiquement les performances de requêtes sous-optimales  
- ✅ Réduit la charge sur le serveur de base de données  
- ✅ Ne nécessite aucune modification de code  
- ✅ Est totalement transparente et rétrocompatible  
- ✅ Fonctionne particulièrement bien avec du code généré automatiquement

Cette fonctionnalité fait partie d'une série d'optimisations du planificateur introduites dans PostgreSQL 18, témoignant de la maturité croissante du moteur et de son intelligence dans la réécriture et l'optimisation des requêtes.

**Message clé pour les débutants :** Vous n'avez rien à faire de spécial pour bénéficier de cette optimisation ! Assurez-vous simplement d'avoir des contraintes PRIMARY KEY et UNIQUE correctement définies sur vos tables, et PostgreSQL 18 s'occupera du reste.
