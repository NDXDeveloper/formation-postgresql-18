# 13.4.2. GiST (Generalized Search Tree) : Géométrie, Texte, ltree

## Introduction

Les index GiST (Generalized Search Tree) sont un type d'index polyvalent de PostgreSQL conçu pour des **données complexes et multidimensionnelles**. Contrairement aux index B-Tree qui fonctionnent bien avec des données unidimensionnelles (nombres, textes), les index GiST excellent avec des données qui ont des relations spatiales ou hiérarchiques.

### Qu'est-ce qu'un arbre de recherche généralisé ?

Le terme "Generalized Search Tree" signifie que GiST est un **framework d'indexation** adaptable à différents types de données. C'est comme un squelette d'index que l'on peut personnaliser pour différents besoins :
- **Géométrie** : Points, lignes, polygones, cercles  
- **Texte** : Recherche full-text (alternative à GIN)  
- **Hiérarchies** : Arbres et chemins (avec ltree)  
- **Ranges** : Intervalles et périodes

### Philosophie de GiST

GiST organise les données en **régions qui peuvent se chevaucher**, à l'inverse d'un B-Tree où les valeurs sont strictement ordonnées. Cette caractéristique est idéale pour :
- Trouver des objets dans une zone géographique
- Rechercher des éléments proches ou voisins
- Naviguer dans des hiérarchies arborescentes

### Avantages des index GiST

- ✅ **Polyvalence** : Supporte de nombreux types de données  
- ✅ **Multidimensionnel** : Excellent pour données spatiales  
- ✅ **Extensible** : Peut être adapté à de nouveaux types  
- ✅ **Recherche de proximité** : Trouve les "voisins" efficacement  
- ✅ **Équilibré** : Maintient automatiquement l'équilibre de l'arbre

### Limitations à connaître

- ⚠️ **Moins rapide que GIN** pour certains cas (arrays, JSONB)  
- ⚠️ **Plus rapide en écriture** que GIN mais plus lent en lecture  
- ⚠️ **Taille variable** : Peut être volumineux selon les données  
- ⚠️ **Complexité** : Configuration plus subtile que B-Tree

## 1. GiST pour la Géométrie (Types Natifs PostgreSQL)

PostgreSQL inclut nativement des types géométriques simples qui peuvent être indexés avec GiST.

### Les types géométriques natifs

PostgreSQL propose plusieurs types géométriques de base :

| Type | Description | Exemple |
|------|-------------|---------|
| `point` | Un point (x, y) | `point '(1, 2)'` |
| `line` | Une ligne infinie | `line '{1, 2, 3}'` |
| `lseg` | Un segment de ligne | `lseg '[(0,0), (1,1)]'` |
| `box` | Un rectangle | `box '((0,0), (1,1))'` |
| `path` | Une séquence de points | `path '[(0,0), (1,1), (2,0)]'` |
| `polygon` | Un polygone fermé | `polygon '((0,0), (1,0), (1,1), (0,1))'` |
| `circle` | Un cercle | `circle '((0,0), 5)'` |

### Création d'un index GiST sur des données géométriques

```sql
-- Création d'une table de lieux
CREATE TABLE lieux (
    id SERIAL PRIMARY KEY,
    nom TEXT NOT NULL,
    position point NOT NULL
);

-- Insertion de données (coordonnées approximatives)
INSERT INTO lieux (nom, position) VALUES
    ('Tour Eiffel', point '(2.2945, 48.8584)'),
    ('Louvre', point '(2.3376, 48.8606)'),
    ('Notre-Dame', point '(2.3522, 48.8530)'),
    ('Arc de Triomphe', point '(2.2950, 48.8738)'),
    ('Sacré-Cœur', point '(2.3431, 48.8867)'),
    ('Panthéon', point '(2.3464, 48.8462)');

-- Création de l'index GiST
CREATE INDEX idx_lieux_position_gist ON lieux USING GiST (position);
```

### Opérateurs géométriques avec GiST

PostgreSQL propose de nombreux opérateurs pour travailler avec la géométrie :

#### Distance et proximité

```sql
-- Trouver le lieu le plus proche d'un point donné
-- Utilise l'opérateur <-> pour la distance
SELECT nom, position  
FROM lieux  
ORDER BY position <-> point '(2.35, 48.86)'  
LIMIT 3;  

-- Résultat (ordonnés par distance croissante) :
-- Notre-Dame
-- Louvre
-- Sacré-Cœur
```

#### Recherche dans une zone (box)

```sql
-- Créer une table avec des zones rectangulaires
CREATE TABLE zones (
    id SERIAL PRIMARY KEY,
    nom TEXT NOT NULL,
    zone box NOT NULL
);

-- Insertion de zones (rectangles délimités par deux points)
INSERT INTO zones (nom, zone) VALUES
    ('Centre Paris', box '((2.25, 48.82), (2.40, 48.90))'),
    ('La Défense', box '((2.22, 48.88), (2.26, 48.90))'),
    ('Banlieue Est', box '((2.40, 48.82), (2.50, 48.90))');

-- Index GiST sur les zones
CREATE INDEX idx_zones_gist ON zones USING GiST (zone);

-- Trouver tous les lieux dans une zone spécifique
-- Utilise l'opérateur <@ (contenu dans)
SELECT l.nom, l.position  
FROM lieux l, zones z  
WHERE z.nom = 'Centre Paris'  
  AND l.position <@ z.zone;

-- Résultat : tous les monuments de Paris intra-muros
```

#### Chevauchement et intersection

```sql
-- Créer une table de cercles (zones circulaires)
CREATE TABLE zones_circulaires (
    id SERIAL PRIMARY KEY,
    nom TEXT NOT NULL,
    zone circle NOT NULL
);

-- Insertion de zones circulaires (centre + rayon)
-- Format : ((x, y), rayon)
INSERT INTO zones_circulaires (nom, zone) VALUES
    ('Zone Tour Eiffel', circle '((2.2945, 48.8584), 0.02)'),
    ('Zone Louvre', circle '((2.3376, 48.8606), 0.015)');

-- Index GiST
CREATE INDEX idx_zones_circ_gist ON zones_circulaires USING GiST (zone);

-- Trouver les lieux dans un rayon donné
-- Utilise l'opérateur <@ (point contenu dans cercle)
SELECT l.nom  
FROM lieux l, zones_circulaires z  
WHERE z.nom = 'Zone Tour Eiffel'  
  AND l.position <@ z.zone;
```

### Opérateurs géométriques principaux

| Opérateur | Signification | Exemple | Indexé par GiST ? |
|-----------|---------------|---------|-------------------|
| `<->` | Distance | `position <-> point '(0,0)'` | ✅ Oui (ordering) |
| `<@` | Contenu dans | `point <@ box` | ✅ Oui |
| `@>` | Contient | `box @> point` | ✅ Oui |
| `&&` | Chevauche | `box && box` | ✅ Oui |
| `<<` | Strictement à gauche | `box << box` | ✅ Oui |
| `>>` | Strictement à droite | `box >> box` | ✅ Oui |
| `&<` | N'étend pas à droite | `box &< box` | ✅ Oui |
| `&>` | N'étend pas à gauche | `box &> box` | ✅ Oui |
| `~=` | Identique | `point ~= point` | ✅ Oui |

### Fonctions géométriques utiles

```sql
-- Calculer la distance entre deux points
SELECT
    l1.nom AS lieu1,
    l2.nom AS lieu2,
    l1.position <-> l2.position AS distance
FROM lieux l1, lieux l2  
WHERE l1.id < l2.id  
ORDER BY distance  
LIMIT 5;  

-- Créer un box (rectangle) autour d'un point
SELECT
    nom,
    box(point(position[0] - 0.01, position[1] - 0.01),
        point(position[0] + 0.01, position[1] + 0.01)) AS zone_autour
FROM lieux;

-- Vérifier si un point est dans un polygone
CREATE TABLE quartiers (
    id SERIAL PRIMARY KEY,
    nom TEXT NOT NULL,
    frontiere polygon NOT NULL
);

INSERT INTO quartiers (nom, frontiere) VALUES
    ('Marais', polygon '((2.35, 48.85), (2.37, 48.85), (2.37, 48.87), (2.35, 48.87))');

CREATE INDEX idx_quartiers_gist ON quartiers USING GiST (frontiere);

-- Trouver dans quel quartier se trouve un lieu
SELECT l.nom AS lieu, q.nom AS quartier  
FROM lieux l  
JOIN quartiers q ON l.position <@ q.frontiere;  
```

## 2. GiST pour PostGIS (Géométrie Avancée)

PostGIS est l'extension géospatiale de référence pour PostgreSQL. Elle transforme PostgreSQL en une véritable base de données géographique professionnelle.

### Installation et activation de PostGIS

```sql
-- Vérifier si PostGIS est disponible
SELECT * FROM pg_available_extensions WHERE name = 'postgis';

-- Installer PostGIS (à faire une seule fois par base de données)
CREATE EXTENSION IF NOT EXISTS postgis;

-- Vérifier la version installée
SELECT PostGIS_Version();
```

### Types géométriques PostGIS

PostGIS offre des types géométriques beaucoup plus riches et conformes aux standards OGC (Open Geospatial Consortium) :

| Type | Description | Exemple |
|------|-------------|---------|
| `geometry(Point)` | Point 2D/3D | `ST_GeomFromText('POINT(2.35 48.86)')` |
| `geometry(LineString)` | Ligne | `ST_GeomFromText('LINESTRING(0 0, 1 1, 2 0)')` |
| `geometry(Polygon)` | Polygone | `ST_GeomFromText('POLYGON((0 0, 4 0, 4 4, 0 4, 0 0))')` |
| `geometry(MultiPoint)` | Plusieurs points | `ST_GeomFromText('MULTIPOINT((0 0), (1 1))')` |
| `geometry(MultiLineString)` | Plusieurs lignes | Multiple lignes |
| `geometry(MultiPolygon)` | Plusieurs polygones | Multiple polygones |
| `geometry(GeometryCollection)` | Collection mixte | Mélange de types |
| `geography` | Coordonnées sphériques | Calculs précis sur Terre |

### Différence entre geometry et geography

- **`geometry`** : Coordonnées dans un plan (plus rapide, moins précis sur longues distances)  
- **`geography`** : Coordonnées sur une sphère (plus lent, très précis pour distances réelles)

```sql
-- geometry : utilise un plan cartésien
-- geography : utilise une sphère (la Terre)

-- La distance sera différente !
SELECT
    ST_Distance(
        ST_GeomFromText('POINT(0 0)'),
        ST_GeomFromText('POINT(1 1)')
    ) AS distance_geometry,
    ST_Distance(
        ST_GeogFromText('POINT(0 0)'),
        ST_GeogFromText('POINT(1 1)')
    ) AS distance_geography_metres;

-- distance_geometry : ~1.41 (unités du plan)
-- distance_geography_metres : ~156898 mètres (distance réelle sur Terre)
```

### Création d'une table avec PostGIS

```sql
-- Table de points d'intérêt (POI)
CREATE TABLE pois (
    id SERIAL PRIMARY KEY,
    nom TEXT NOT NULL,
    categorie TEXT,
    geom geometry(Point, 4326),  -- SRID 4326 = WGS84 (GPS)
    date_creation TIMESTAMPTZ DEFAULT NOW()
);

-- Insertion de données (format WKT - Well-Known Text)
INSERT INTO pois (nom, categorie, geom) VALUES
    ('Tour Eiffel', 'monument', ST_GeomFromText('POINT(2.2945 48.8584)', 4326)),
    ('Louvre', 'musée', ST_GeomFromText('POINT(2.3376 48.8606)', 4326)),
    ('Gare du Nord', 'transport', ST_GeomFromText('POINT(2.3550 48.8809)', 4326)),
    ('Parc Monceau', 'parc', ST_GeomFromText('POINT(2.3089 48.8799)', 4326));

-- Création de l'index GiST spatial
CREATE INDEX idx_pois_geom_gist ON pois USING GiST (geom);
```

### Qu'est-ce que le SRID ?

Le **SRID (Spatial Reference System Identifier)** définit le système de coordonnées utilisé :
- **4326** : WGS84 (système GPS standard, longitude/latitude)  
- **3857** : Web Mercator (utilisé par Google Maps, OpenStreetMap)  
- **2154** : Lambert 93 (système français officiel)

### Requêtes spatiales avec PostGIS et GiST

#### Recherche par distance (KNN - K Nearest Neighbors)

```sql
-- Trouver les 3 POIs les plus proches d'un point
SELECT
    nom,
    categorie,
    ST_Distance(geom, ST_GeomFromText('POINT(2.35 48.86)', 4326)) AS distance_degrees,
    ST_Distance(
        geom::geography,
        ST_GeogFromText('POINT(2.35 48.86)')
    ) AS distance_metres
FROM pois  
ORDER BY geom <-> ST_GeomFromText('POINT(2.35 48.86)', 4326)  
LIMIT 3;  

-- L'opérateur <-> utilise l'index GiST pour un tri efficace
```

#### Recherche dans un rayon (buffer)

```sql
-- Trouver tous les POIs dans un rayon de 1 km autour d'un point
SELECT nom, categorie  
FROM pois  
WHERE ST_DWithin(  
    geom::geography,
    ST_GeogFromText('POINT(2.35 48.86)'),
    1000  -- 1000 mètres = 1 km
);

-- ST_DWithin utilise l'index GiST pour optimiser la recherche
```

#### Recherche dans un polygone

```sql
-- Créer une table de zones (quartiers, arrondissements)
CREATE TABLE quartiers_paris (
    id SERIAL PRIMARY KEY,
    nom TEXT NOT NULL,
    arrondissement INTEGER,
    geom geometry(Polygon, 4326)
);

-- Exemple : définir le 1er arrondissement (simplifié)
INSERT INTO quartiers_paris (nom, arrondissement, geom) VALUES
    ('Louvre', 1, ST_GeomFromText('POLYGON((
        2.33 48.86,
        2.35 48.86,
        2.35 48.87,
        2.33 48.87,
        2.33 48.86
    ))', 4326));

-- Index GiST sur les polygones
CREATE INDEX idx_quartiers_geom_gist ON quartiers_paris USING GiST (geom);

-- Trouver dans quel quartier se trouve chaque POI
SELECT
    p.nom AS poi,
    q.nom AS quartier,
    q.arrondissement
FROM pois p  
JOIN quartiers_paris q ON ST_Within(p.geom, q.geom);  

-- ST_Within utilise l'index GiST
```

#### Intersection de géométries

```sql
-- Créer une table de lignes (routes, rivières)
CREATE TABLE routes (
    id SERIAL PRIMARY KEY,
    nom TEXT NOT NULL,
    geom geometry(LineString, 4326)
);

INSERT INTO routes (nom, geom) VALUES
    ('Avenue des Champs-Élysées', ST_GeomFromText('LINESTRING(2.31 48.87, 2.30 48.87)', 4326)),
    ('Seine', ST_GeomFromText('LINESTRING(2.25 48.86, 2.35 48.85, 2.40 48.83)', 4326));

CREATE INDEX idx_routes_geom_gist ON routes USING GiST (geom);

-- Trouver les quartiers traversés par une route
SELECT DISTINCT
    r.nom AS route,
    q.nom AS quartier_traverse
FROM routes r  
JOIN quartiers_paris q ON ST_Intersects(r.geom, q.geom);  
```

### Fonctions PostGIS essentielles

#### Mesures et calculs

```sql
-- Distance entre deux géométries
SELECT ST_Distance(geom1, geom2);  -- En degrés (geometry)  
SELECT ST_Distance(geom1::geography, geom2::geography);  -- En mètres (geography)  

-- Longueur d'une ligne
SELECT ST_Length(geom);  -- Pour LineString

-- Aire d'un polygone
SELECT ST_Area(geom);  -- Pour Polygon

-- Périmètre
SELECT ST_Perimeter(geom);
```

#### Relations spatiales

```sql
-- Teste si A contient B
SELECT ST_Contains(polygon_geom, point_geom);

-- Teste si A est dans B
SELECT ST_Within(point_geom, polygon_geom);

-- Teste si A et B se touchent
SELECT ST_Touches(geom1, geom2);

-- Teste si A et B se chevauchent
SELECT ST_Overlaps(geom1, geom2);

-- Teste si A et B s'intersectent
SELECT ST_Intersects(geom1, geom2);

-- Teste si A et B sont disjoints
SELECT ST_Disjoint(geom1, geom2);
```

#### Transformations

```sql
-- Créer un buffer (zone tampon) autour d'un point
SELECT ST_Buffer(geom, 0.01);  -- 0.01 degré (geometry)  
SELECT ST_Buffer(geom::geography, 1000);  -- 1000 mètres (geography)  

-- Centroïde d'une géométrie
SELECT ST_Centroid(geom);

-- Enveloppe (bounding box)
SELECT ST_Envelope(geom);

-- Simplifier une géométrie complexe
SELECT ST_Simplify(geom, 0.001);

-- Union de plusieurs géométries
SELECT ST_Union(geom1, geom2);

-- Intersection de deux géométries
SELECT ST_Intersection(geom1, geom2);

-- Différence entre deux géométries
SELECT ST_Difference(geom1, geom2);
```

### Exemple concret : Application de livraison

```sql
-- Table des restaurants
CREATE TABLE restaurants (
    id SERIAL PRIMARY KEY,
    nom TEXT NOT NULL,
    adresse TEXT,
    geom geometry(Point, 4326),
    zone_livraison geometry(Polygon, 4326)  -- Zone de livraison
);

-- Index spatial
CREATE INDEX idx_restaurants_geom ON restaurants USING GiST (geom);  
CREATE INDEX idx_restaurants_zone ON restaurants USING GiST (zone_livraison);  

-- Insertion d'un restaurant avec sa zone de livraison (3 km de rayon)
INSERT INTO restaurants (nom, adresse, geom, zone_livraison)  
VALUES (  
    'Pizza Roma',
    '10 Rue de la Paix',
    ST_GeomFromText('POINT(2.35 48.86)', 4326),
    ST_Buffer(
        ST_GeomFromText('POINT(2.35 48.86)', 4326)::geography,
        3000  -- 3 km
    )::geometry
);

-- Un client commande : est-il dans la zone de livraison ?
SELECT
    r.nom,
    ST_Distance(
        r.geom::geography,
        ST_GeogFromText('POINT(2.36 48.87)')
    ) AS distance_metres,
    ST_Within(
        ST_GeomFromText('POINT(2.36 48.87)', 4326),
        r.zone_livraison
    ) AS peut_livrer
FROM restaurants r;

-- Trouver tous les restaurants qui livrent à une adresse
SELECT nom, adresse  
FROM restaurants  
WHERE ST_Within(  
    ST_GeomFromText('POINT(2.36 48.87)', 4326),
    zone_livraison
);
```

## 3. GiST pour Full-Text Search (Alternative à GIN)

GiST peut également être utilisé pour la recherche full-text, bien que GIN soit généralement préféré pour ce cas d'usage.

### Quand utiliser GiST au lieu de GIN pour le texte ?

**Utilisez GIN si** :
- ✅ Les lectures (recherches) sont prioritaires  
- ✅ Les données changent peu

**Utilisez GiST si** :
- ✅ Les écritures (insertions/mises à jour) sont fréquentes  
- ✅ L'index doit rester compact  
- ✅ Vous avez déjà un index GiST sur la table

### Création d'un index GiST pour Full-Text

```sql
-- Table de documents
CREATE TABLE articles (
    id SERIAL PRIMARY KEY,
    titre TEXT NOT NULL,
    contenu TEXT NOT NULL,
    tsv tsvector
);

-- Remplir la colonne tsvector
UPDATE articles  
SET tsv = to_tsvector('french', titre || ' ' || contenu);  

-- Index GiST pour full-text
CREATE INDEX idx_articles_tsv_gist ON articles USING GiST (tsv);

-- Comparaison avec GIN
-- CREATE INDEX idx_articles_tsv_gin ON articles USING GIN (tsv);
```

### Recherche avec GiST

```sql
-- La syntaxe est identique à GIN
SELECT titre, contenu  
FROM articles  
WHERE tsv @@ to_tsquery('french', 'postgresql & performance');  

-- L'opérateur @@ utilise l'index GiST
```

### Comparaison GIN vs GiST pour Full-Text

| Critère | GIN | GiST |
|---------|-----|------|
| **Vitesse de recherche** | ⚡ Plus rapide | Légèrement plus lent |
| **Vitesse d'écriture** | Plus lente | ⚡ Plus rapide |
| **Taille de l'index** | Plus volumineux | Plus compact |
| **Cas d'usage** | Données stables, recherches fréquentes | Données volatiles, écritures fréquentes |

### Exemple : Blog avec mises à jour fréquentes

```sql
-- Blog où les articles sont souvent modifiés
CREATE TABLE blog_posts (
    id SERIAL PRIMARY KEY,
    titre TEXT NOT NULL,
    contenu TEXT NOT NULL,
    auteur TEXT,
    derniere_modification TIMESTAMPTZ DEFAULT NOW(),
    tsv tsvector
);

-- Trigger pour maintenir tsvector à jour
CREATE OR REPLACE FUNCTION blog_tsv_trigger() RETURNS trigger AS $$  
BEGIN  
    NEW.tsv := to_tsvector('french', NEW.titre || ' ' || NEW.contenu);
    NEW.derniere_modification := NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER tsv_update  
BEFORE INSERT OR UPDATE ON blog_posts  
FOR EACH ROW EXECUTE FUNCTION blog_tsv_trigger();  

-- Index GiST (meilleur pour mises à jour fréquentes)
CREATE INDEX idx_blog_tsv_gist ON blog_posts USING GiST (tsv);

-- Recherche
SELECT titre, auteur  
FROM blog_posts  
WHERE tsv @@ plainto_tsquery('french', 'postgresql tutoriel');  
```

## 4. GiST pour ltree (Hiérarchies et Arbres)

L'extension `ltree` permet de stocker et interroger efficacement des structures hiérarchiques (arbres). GiST est l'index idéal pour ce type de données.

### Qu'est-ce que ltree ?

`ltree` (Label Tree) est un type de données PostgreSQL qui représente un **chemin hiérarchique** sous forme de labels séparés par des points, similaire à un chemin de fichier ou une URL.

**Exemples de chemins ltree** :
- `'entreprise.it.developpement'`  
- `'france.ile_de_france.paris.75001'`  
- `'biologie.animaux.mammiferes.carnivores.felides'`

### Installation de ltree

```sql
-- Vérifier la disponibilité
SELECT * FROM pg_available_extensions WHERE name = 'ltree';

-- Installer l'extension
CREATE EXTENSION IF NOT EXISTS ltree;
```

### Création d'une table avec ltree

```sql
-- Table d'organisation hiérarchique
CREATE TABLE organisation (
    id SERIAL PRIMARY KEY,
    nom TEXT NOT NULL,
    chemin ltree NOT NULL,
    responsable TEXT,
    nb_employes INTEGER
);

-- Insertion de données hiérarchiques
INSERT INTO organisation (nom, chemin, responsable, nb_employes) VALUES
    ('Entreprise', 'entreprise', 'PDG Martin', 500),
    ('IT', 'entreprise.it', 'CTO Sophie', 150),
    ('RH', 'entreprise.rh', 'DRH Pierre', 20),
    ('Ventes', 'entreprise.ventes', 'Directeur Commercial Julie', 100),
    ('Développement', 'entreprise.it.dev', 'Manager Dev Alice', 80),
    ('Infrastructure', 'entreprise.it.infra', 'Manager Infra Bob', 40),
    ('Backend', 'entreprise.it.dev.backend', 'Lead Backend Charlie', 30),
    ('Frontend', 'entreprise.it.dev.frontend', 'Lead Frontend Diana', 25),
    ('DevOps', 'entreprise.it.dev.devops', 'Lead DevOps Eve', 15),
    ('Réseau', 'entreprise.it.infra.reseau', 'Lead Réseau Frank', 10),
    ('Sécurité', 'entreprise.it.infra.securite', 'Lead Sécu George', 20);

-- Création de l'index GiST sur ltree
CREATE INDEX idx_organisation_chemin_gist ON organisation USING GiST (chemin);
```

### Opérateurs ltree avec GiST

#### Recherche d'ancêtres et descendants

```sql
-- Opérateur @ : "est un descendant de" (est sous)
-- Trouver tous les départements sous IT
SELECT nom, chemin  
FROM organisation  
WHERE chemin @ 'entreprise.it'  
  AND chemin != 'entreprise.it';  -- Exclure IT lui-même

-- Résultat :
-- Développement, Infrastructure, Backend, Frontend, DevOps, Réseau, Sécurité

-- Opérateur ~ : "correspond au pattern"
-- Trouver tous les nœuds "dev" dans la hiérarchie
SELECT nom, chemin  
FROM organisation  
WHERE chemin ~ '*.dev'::lquery;  

-- Opérateur ? : "correspond à un tableau de patterns"
-- Trouver tous les départements tech ou ventes
SELECT nom, chemin  
FROM organisation  
WHERE chemin ? ARRAY['*.it.*', '*.ventes'];  
```

#### Recherche d'ancêtres directs (parents)

```sql
-- Opérateur <@ : "est un ancêtre de" (contient)
-- Trouver les ancêtres de "Backend"
SELECT nom, chemin  
FROM organisation  
WHERE 'entreprise.it.dev.backend' @ chemin;  

-- Résultat (du plus haut au plus bas) :
-- Entreprise (entreprise)
-- IT (entreprise.it)
-- Développement (entreprise.it.dev)
```

#### Recherche de profondeur

```sql
-- Fonction nlevel() : retourne le niveau de profondeur
-- Racine = 1, enfants directs = 2, etc.

-- Trouver tous les départements de niveau 3
SELECT nom, chemin, nlevel(chemin) AS niveau  
FROM organisation  
WHERE nlevel(chemin) = 3;  

-- Résultat :
-- Développement (entreprise.it.dev), Infrastructure (entreprise.it.infra)

-- Tous les nœuds de niveau 2 et inférieur
SELECT nom, chemin, nlevel(chemin) AS niveau  
FROM organisation  
WHERE nlevel(chemin) <= 2  
ORDER BY chemin;  
```

#### Recherche par pattern avec lquery

```sql
-- lquery permet des patterns plus complexes
-- * : un label quelconque
-- *{n} : exactement n labels
-- *{n,} : au moins n labels
-- *{,m} : au plus m labels
-- *{n,m} : entre n et m labels

-- Trouver tous les "lead" (4e niveau sous entreprise.it.dev)
SELECT nom, chemin  
FROM organisation  
WHERE chemin ~ 'entreprise.it.dev.*'::lquery  
  AND nlevel(chemin) = 4;

-- Trouver toutes les équipes directement sous "dev"
SELECT nom, chemin  
FROM organisation  
WHERE chemin ~ 'entreprise.it.dev.*{1}'::lquery;  

-- Résultat : Backend, Frontend, DevOps (exactement un label après dev)
```

#### Opérations sur les chemins

```sql
-- subpath() : extraire une portion du chemin
SELECT
    nom,
    chemin,
    subpath(chemin, 0, 2) AS racine,  -- 2 premiers labels
    subpath(chemin, 1) AS sans_racine  -- À partir du 2e label
FROM organisation  
WHERE chemin ~ 'entreprise.it.dev.*';  

-- subltree() : extraire une sous-hiérarchie
SELECT
    nom,
    chemin,
    subltree(chemin, 2, 10) AS sous_chemin
FROM organisation  
WHERE chemin @ 'entreprise.it.dev';  

-- index() : position d'un sous-chemin
SELECT
    nom,
    index(chemin, 'it') AS position_it
FROM organisation  
WHERE chemin @ 'entreprise.it';  
```

### Agrégation dans une hiérarchie

```sql
-- Calculer le nombre total d'employés sous IT (récursif)
WITH RECURSIVE it_tree AS (
    -- Nœud de départ
    SELECT id, nom, chemin, nb_employes
    FROM organisation
    WHERE chemin = 'entreprise.it'

    UNION ALL

    -- Récursion : tous les descendants
    SELECT o.id, o.nom, o.chemin, o.nb_employes
    FROM organisation o
    JOIN it_tree t ON o.chemin @ t.chemin AND o.chemin != t.chemin
)
SELECT
    'IT (Total)' AS departement,
    SUM(nb_employes) AS total_employes
FROM it_tree;

-- Alternative plus simple avec ltree
SELECT
    'IT (Total)' AS departement,
    SUM(nb_employes) AS total_employes
FROM organisation  
WHERE chemin @ 'entreprise.it';  
```

### Opérateurs ltree principaux

| Opérateur | Signification | Exemple | Indexé par GiST ? |
|-----------|---------------|---------|-------------------|
| `@>` | Est un ancêtre de (contient) | `'a.b' @> 'a.b.c'` | ✅ Oui |
| `<@` | Est un descendant de (dans) | `'a.b.c' <@ 'a.b'` | ✅ Oui |
| `@` | Alias de `<@` | `'a.b.c' @ 'a.b'` | ✅ Oui |
| `~` | Correspond au pattern (lquery) | `chemin ~ '*.it.*'` | ✅ Oui |
| `?` | Correspond à un pattern (ltxtquery) | `chemin ? '@.it.*'` | ✅ Oui |

## 5. Comparaison et Choix d'Index

### GiST vs GIN : Quand utiliser quoi ?

| Critère | GiST | GIN |
|---------|------|-----|
| **Performance lecture** | Bon | ⚡ Excellent |
| **Performance écriture** | ⚡ Rapide | Plus lent |
| **Taille index** | Plus compact | Plus volumineux |
| **Types supportés** | Géométrie, ltree, texte, ranges | Arrays, JSONB, texte |
| **Cas d'usage principal** | Données spatiales, hiérarchies | Recherche dans collections |
| **Maintenance** | Moins exigeant | Plus exigeant |

### GiST vs B-Tree

| Critère | GiST | B-Tree |
|---------|------|--------|
| **Type de données** | Multidimensionnelles, complexes | Scalaires, simples |
| **Opérations** | Proximité, containment, hiérarchies | Égalité, ranges simples |
| **Performances** | Excellent pour requêtes spatiales | Excellent pour requêtes simples |
| **Complexité** | Plus complexe | Simple et éprouvé |

### Quand utiliser GiST ?

✅ **Utilisez GiST pour :**
- Données **géospatiales** (points, polygones, lignes)
- Structures **hiérarchiques** (ltree)
- **Ranges** et intervalles (int4range, tstzrange)
- Full-text avec **mises à jour fréquentes**
- Données **multidimensionnelles**

❌ **N'utilisez PAS GiST pour :**
- Colonnes scalaires simples (préférez B-Tree)
- Arrays ou JSONB (préférez GIN)
- Égalité stricte simple (B-Tree est plus efficace)

### Tableau récapitulatif des types et index recommandés

| Type de données | Index recommandé | Pourquoi ? |
|----------------|------------------|------------|
| `INTEGER`, `TEXT`, `DATE` | B-Tree | Simple et efficace |
| `TEXT[]`, `INTEGER[]` | GIN | Recherche dans collections |
| `JSONB` | GIN (jsonb_path_ops) | Recherche par valeurs |
| `geometry`, `geography` | GiST | Requêtes spatiales |
| `ltree` | GiST | Hiérarchies |
| `tsvector` (lecture) | GIN | Recherche texte rapide |
| `tsvector` (écriture) | GiST | Mises à jour fréquentes |
| `int4range`, `tstzrange` | GiST | Opérations sur intervalles |

## 6. Performance et Optimisations

### Configuration des index GiST

```sql
-- Paramètres lors de la création
CREATE INDEX idx_pois_geom ON pois USING GiST (geom)  
WITH (fillfactor = 90, buffering = auto);  
```

#### Paramètre `fillfactor`
- **Par défaut** : 90  
- **Description** : Pourcentage de remplissage des pages d'index  
- **Recommandation** :
  - 90 (par défaut) pour données volatiles
  - 100 pour données stables (lecture seule)

#### Paramètre `buffering` (spécifique à PostGIS)
- **Par défaut** : `auto`  
- **Options** : `on`, `off`, `auto`  
- **Description** : Optimise la construction d'index en bufferisant les insertions

### Maintenance des index GiST

```sql
-- Analyser les statistiques (crucial pour le planificateur)
ANALYZE pois;

-- Reconstruire l'index si fragmenté ou corrompu
REINDEX INDEX idx_pois_geom_gist;

-- Reconstruire TOUS les index d'une table
REINDEX TABLE pois;

-- Vacuum pour nettoyer l'espace mort
VACUUM pois;

-- Vacuum complet (plus lourd, bloque la table)
VACUUM FULL pois;
```

### Surveiller la performance

```sql
-- Taille des index
SELECT
    schemaname,
    tablename,
    indexname,
    pg_size_pretty(pg_relation_size(indexrelid)) AS taille
FROM pg_stat_user_indexes  
WHERE indexname LIKE '%gist%'  
ORDER BY pg_relation_size(indexrelid) DESC;  

-- Utilisation des index
SELECT
    schemaname,
    tablename,
    indexname,
    idx_scan AS nb_utilisations,
    idx_tup_read AS lignes_lues,
    idx_tup_fetch AS lignes_retournees
FROM pg_stat_user_indexes  
WHERE indexname LIKE '%gist%'  
ORDER BY idx_scan DESC;  

-- Index inutilisés (potentiellement à supprimer)
SELECT
    schemaname,
    tablename,
    indexname
FROM pg_stat_user_indexes  
WHERE idx_scan = 0  
  AND indexname LIKE '%gist%'
  AND schemaname NOT IN ('pg_catalog', 'information_schema');
```

### Optimiser les requêtes spatiales

```sql
-- ✅ BON : Utilise l'index GiST avec opérateur de distance
EXPLAIN ANALYZE  
SELECT nom  
FROM pois  
ORDER BY geom <-> ST_GeomFromText('POINT(2.35 48.86)', 4326)  
LIMIT 10;  

-- Plan attendu : Index Scan using idx_pois_geom_gist

-- ❌ MAUVAIS : Calcul de distance sans index
EXPLAIN ANALYZE  
SELECT nom  
FROM pois  
ORDER BY ST_Distance(geom, ST_GeomFromText('POINT(2.35 48.86)', 4326))  
LIMIT 10;  

-- Plan : Seq Scan + Sort (très lent sur grandes tables)

-- ✅ BON : Filtrage avec bounding box puis vérification précise
SELECT nom  
FROM pois  
WHERE ST_DWithin(geom::geography, ST_GeogFromText('POINT(2.35 48.86)'), 1000)  
ORDER BY geom <-> ST_GeomFromText('POINT(2.35 48.86)', 4326);  

-- L'index GiST est utilisé pour les deux opérations
```

### Astuces d'optimisation

#### 1. Utiliser des bounding boxes pour pré-filtrer

```sql
-- Au lieu de calculer des intersections coûteuses directement
-- Utiliser d'abord l'opérateur && (bounding box overlap)

-- ✅ Optimisé avec bounding box
SELECT p.nom  
FROM pois p, quartiers_paris q  
WHERE q.nom = 'Louvre'  
  AND p.geom && q.geom  -- Bounding box overlap (rapide avec index)
  AND ST_Within(p.geom, q.geom);  -- Vérification précise

-- L'index est utilisé pour &&, puis vérification exacte seulement sur candidats
```

#### 2. Choisir entre geometry et geography selon le besoin

```sql
-- geometry : Plus rapide, moins précis (plan cartésien)
SELECT ST_Distance(geom1, geom2);  -- Résultat en degrés

-- geography : Plus lent, très précis (sphère terrestre)
SELECT ST_Distance(geom1::geography, geom2::geography);  -- Résultat en mètres

-- Règle : Utilisez geometry pour surfaces < 1000 km², geography sinon
```

#### 3. Simplifier les géométries complexes

```sql
-- Simplifier un polygone avec beaucoup de points
UPDATE quartiers_paris  
SET geom = ST_Simplify(geom, 0.0001)  -- Tolérance en degrés  
WHERE ST_NPoints(geom) > 1000;  

-- Réduire la précision pour les index
CREATE INDEX idx_quartiers_simple ON quartiers_paris  
USING GiST (ST_Simplify(geom, 0.001));  
```

#### 4. Utiliser les statistiques spatiales

```sql
-- Analyser la distribution spatiale
SELECT
    tablename,
    attname,
    n_distinct,
    correlation
FROM pg_stats  
WHERE tablename = 'pois'  
  AND attname = 'geom';

-- Forcer la collecte de statistiques étendues
ALTER TABLE pois ALTER COLUMN geom SET STATISTICS 1000;  
ANALYZE pois;  
```

```sql
CREATE TABLE permissions_system (
    id SERIAL PRIMARY KEY,
    ressource ltree NOT NULL,
    utilisateur_id INTEGER,
    niveau TEXT CHECK (niveau IN ('read', 'write', 'admin'))
);

CREATE INDEX idx_perms_ressource ON permissions_system USING GiST (ressource);

-- Accorder des permissions
INSERT INTO permissions_system (ressource, utilisateur_id, niveau) VALUES
    ('app.users', 101, 'admin'),              -- Admin sur tous les users
    ('app.users.profile', 102, 'write'),      -- Write sur les profils
    ('app.users.profile.avatar', 103, 'read'), -- Read sur les avatars
    ('app.reports', 101, 'read');             -- Read sur les rapports

-- Vérifier si un utilisateur a accès à une ressource (avec héritage)
CREATE OR REPLACE FUNCTION check_permission(
    p_utilisateur_id INTEGER,
    p_ressource TEXT,
    p_niveau_requis TEXT
) RETURNS BOOLEAN AS $$
BEGIN
    RETURN EXISTS (
        SELECT 1
        FROM permissions_system
        WHERE utilisateur_id = p_utilisateur_id
          AND p_ressource::ltree <@ ressource  -- La ressource demandée est sous une permission
          AND (
              niveau = 'admin' OR  -- Admin a tous les droits
              (p_niveau_requis = 'read' AND niveau IN ('read', 'write', 'admin')) OR
              (p_niveau_requis = 'write' AND niveau IN ('write', 'admin'))
          )
    );
END;
$$ LANGUAGE plpgsql;

-- Tester
SELECT check_permission(101, 'app.users.profile.settings', 'write');  -- TRUE (admin sur app.users)  
SELECT check_permission(102, 'app.users.profile.avatar', 'write');     -- TRUE (write sur app.users.profile)  
SELECT check_permission(103, 'app.users.profile.avatar', 'write');     -- FALSE (seulement read)  
```

```sql
    nom TEXT NOT NULL,
    type TEXT,  -- 'ecole', 'transport', 'commerce'

);

CREATE INDEX idx_zones_geom ON zones_interet USING GiST (geom);

-- Recherche multi-critères avec proximité

    b.titre,
    b.prix,
    b.surface,
    COUNT(z.id) FILTER (WHERE z.type = 'ecole') AS ecoles_proches,
    COUNT(z.id) FILTER (WHERE z.type = 'transport') AS transports_proches,
    COUNT(z.id) FILTER (WHERE z.type = 'commerce') AS commerces_proches

LEFT JOIN zones_interet z ON ST_DWithin(b.position::geography, z.geom::geography, 1000)  
WHERE b.prix <= 500000  
  AND b.chambres >= 3
  AND EXISTS (
      -- Au moins un transport à moins de 500m
      SELECT 1 FROM zones_interet z2
      WHERE z2.type = 'transport'
        AND ST_DWithin(b.position::geography, z2.geom::geography, 500)
  )
GROUP BY b.id, b.titre, b.prix, b.surface  
HAVING COUNT(z.id) FILTER (WHERE z.type = 'ecole') >= 2  
ORDER BY b.prix;
```

## 8. Pièges Courants et Solutions

### Piège 1 : Mauvais SRID avec PostGIS

```sql
-- ❌ PROBLÈME : Mélanger différents SRID
CREATE TABLE lieux_bad (geom geometry(Point, 4326));  
INSERT INTO lieux_bad VALUES (ST_GeomFromText('POINT(0 0)', 3857));  -- ERREUR  

-- ✅ SOLUTION : Toujours utiliser le même SRID ou transformer
INSERT INTO lieux_bad VALUES (
    ST_Transform(ST_GeomFromText('POINT(0 0)', 3857), 4326)
);
```

### Piège 2 : Confusion geometry vs geography

```sql
-- ❌ PROBLÈME : Distance en degrés (peu utile)
SELECT ST_Distance(geom1, geom2);  -- Résultat : 0.05 degrés

-- ✅ SOLUTION : Utiliser geography pour distances réelles
SELECT ST_Distance(geom1::geography, geom2::geography);  -- Résultat : 5547 mètres
```

### Piège 3 : Oublier l'index sur ltree

```sql
-- ❌ PROBLÈME : Requêtes lentes sans index
SELECT * FROM organisation WHERE chemin @ 'entreprise.it';  -- Seq Scan

-- ✅ SOLUTION : Créer l'index GiST
CREATE INDEX idx_org_chemin ON organisation USING GiST (chemin);
-- Maintenant : Index Scan using idx_org_chemin
```

### Piège 4 : Mauvais opérateur spatial

```sql
-- ❌ PROBLÈME : ST_Distance dans WHERE (n'utilise pas l'index)
SELECT * FROM pois WHERE ST_Distance(geom, point) < 1000;

-- ✅ SOLUTION : Utiliser ST_DWithin (optimisé avec index)
SELECT * FROM pois WHERE ST_DWithin(geom::geography, point, 1000);
```

### Piège 5 : Géométries invalides

```sql
-- ❌ PROBLÈME : Créer des géométries invalides
INSERT INTO quartiers VALUES (ST_GeomFromText('POLYGON((0 0, 1 1, 1 0, 0 1, 0 0))'));
-- Polygone auto-intersectant !

-- ✅ SOLUTION : Valider et corriger
SELECT ST_IsValid(geom) FROM quartiers;  -- FALSE

-- Corriger automatiquement
UPDATE quartiers SET geom = ST_MakeValid(geom) WHERE NOT ST_IsValid(geom);
```

### Piège 6 : Mauvais pattern ltree

```sql
-- ❌ PROBLÈME : Pattern qui ne matche rien
SELECT * FROM organisation WHERE chemin ~ 'entreprise.IT';  -- Sensible à la casse !

-- ✅ SOLUTION : Utiliser le bon pattern
SELECT * FROM organisation WHERE chemin ~ 'entreprise.it';
```

- [ ] Surveiller les statistiques d'utilisation  
- [ ] `REINDEX` si fragmentation importante

### ✅ Requêtes

- [ ] Utiliser les bons opérateurs (`@>`, `<@`, `&&`, `<->`)  
- [ ] Préférer `ST_DWithin()` à `ST_Distance() < x`  
- [ ] Utiliser `ORDER BY geom <->` pour KNN (plus proches voisins)  
- [ ] Caster en geography pour distances en mètres  
- [ ] Valider les géométries avant requêtes complexes

## Conclusion

Les index GiST sont un outil essentiel et polyvalent pour gérer des **données multidimensionnelles et complexes** dans PostgreSQL. Que ce soit pour des requêtes spatiales avec PostGIS, la navigation dans des hiérarchies avec ltree, ou la gestion d'intervalles, GiST offre des performances excellentes et une grande flexibilité.

### Points clés à retenir

1. **GiST = Arbre équilibré adaptable** : Framework d'indexation pour données complexes  
2. **Trois domaines principaux** : Géométrie (PostGIS), Hiérarchies (ltree), Ranges  
3. **Trade-off intelligent** : Plus rapide en écriture que GIN, légèrement plus lent en lecture  
4. **PostGIS est incontournable** : Pour toute application avec composante géospatiale  
5. **ltree pour hiérarchies** : Solution élégante pour arbres et taxonomies

### Quand utiliser GiST ?

- ✅ **OUI** : Géométrie, hiérarchies, ranges, données spatiales  
- ❌ **NON** : Colonnes simples (B-Tree), collections (GIN)

### Cas d'usage typiques

- 🗺️ **Applications cartographiques** : Recherche de proximité, zones d'intérêt  
- 🏢 **Systèmes d'organisation** : Organigrammes, structures d'entreprise  
- 🛒 **E-commerce** : Catégories de produits, taxonomies  
- 🚗 **Mobilité** : Covoiturage, livraison, transport  
- 🏠 **Immobilier** : Recherche par localisation et critères  
- 🔐 **Permissions** : Contrôle d'accès hiérarchique

Avec une bonne compréhension des index GiST, vous pouvez concevoir des systèmes PostgreSQL capables de gérer efficacement des requêtes spatiales complexes et des structures hiérarchiques à grande échelle !
