# 13.4.4. Hash : Égalité stricte

## Introduction

Les index **Hash** sont un type d'index spécialisé de PostgreSQL conçu exclusivement pour les **comparaisons d'égalité stricte**. Contrairement aux index B-Tree qui peuvent gérer les égalités ET les comparaisons (>, <, BETWEEN), les index Hash ne savent faire qu'une seule chose : répondre à la question "cette valeur existe-t-elle ?".

### L'analogie du casier postal

Imaginez un bureau de poste avec des casiers numérotés :
- Vous avez un nom (la valeur à chercher)
- Une formule mathématique transforme ce nom en numéro de casier (fonction de hachage)
- Vous allez directement au bon casier sans chercher ailleurs

C'est exactement ce que fait un index Hash : il transforme une valeur en position, permettant un accès direct et ultra-rapide.

### Principe de fonctionnement

```
Valeur → [Fonction de Hachage] → Position dans l'index → Données

Exemple :
'alice@example.com' → hash() → 42857 → Ligne dans la table
'bob@example.com'   → hash() → 18923 → Ligne dans la table
'alice@example.com' → hash() → 42857 → Même position = trouvé !
```

### Caractéristiques essentielles

- ✅ **Ultra-rapide** pour recherche d'égalité exacte (théoriquement O(1))  
- ✅ **Compact** : Taille similaire ou légèrement inférieure à B-Tree  
- ✅ **Simple** : Une seule opération supportée (=)  
- ✅ **Déterministe** : Même valeur = même hash = même position

- ❌ **Aucune comparaison** : Ne supporte pas >, <, >=, <=, BETWEEN  
- ❌ **Pas d'ordre** : Ne peut pas être utilisé pour ORDER BY  
- ❌ **Pas de LIKE** : Aucune recherche partielle  
- ❌ **Collisions** : Plusieurs valeurs peuvent avoir le même hash

### Quand utiliser un index Hash ?

✅ **À utiliser pour :**
- Recherches d'égalité stricte uniquement (WHERE colonne = valeur)
- Colonnes avec beaucoup de valeurs uniques (haute cardinalité)
- Jointures par égalité (JOIN ... ON a = b)
- Clés primaires ou uniques (théoriquement)

❌ **À éviter pour :**
- Toute comparaison autre que l'égalité
- ORDER BY ou GROUP BY
- Colonnes avec peu de valeurs distinctes
- LIKE, IN avec plages, BETWEEN

## 1. Fonctions de Hachage : Le Cœur du Système

### Qu'est-ce qu'une fonction de hachage ?

Une **fonction de hachage** (hash function) est un algorithme mathématique qui transforme une valeur (de n'importe quelle taille) en un nombre entier de taille fixe, appelé "hash code" ou "empreinte".

### Propriétés importantes

1. **Déterministe** : Même entrée → Toujours même sortie
   ```
   hash('alice') = 42857 (toujours)
   hash('alice') = 42857 (toujours)
   ```

2. **Distribution uniforme** : Les hash sont répartis uniformément
   ```
   hash('alice') = 42857
   hash('alicf') = 91234  (très différent malgré 1 caractère de différence)
   hash('bob')   = 15678
   ```

3. **Rapide à calculer** : L'algorithme doit être ultra-performant

4. **Sensible aux changements** : Petite modification → Hash complètement différent
   ```
   hash('Alice')  = 42857
   hash('alice')  = 88291  (sensible à la casse)
   hash('alice ') = 63442  (espace en plus)
   ```

### Visualisation simplifiée

```
Table : users (1 million de lignes)  
Colonne : email (avec index Hash)  

Sans index Hash :  
WHERE email = 'alice@example.com'  
→ Scan séquentiel : vérifier 1 million de lignes (LENT)

Avec index Hash :  
WHERE email = 'alice@example.com'  
→ hash('alice@example.com') = 42857
→ Aller directement au bucket 42857
→ Vérifier seulement les lignes dans ce bucket (RAPIDE)
```

### Gestion des collisions

Une **collision** se produit quand deux valeurs différentes produisent le même hash.

```
hash('alice@example.com') = 42857  
hash('xyz@domain.org')    = 42857  ← Collision !  

PostgreSQL utilise le chaînage :  
Bucket 42857 → [alice@example.com] → [xyz@domain.org]  
                     ↓                       ↓
                  Ligne 100              Ligne 58392
```

Lorsqu'une collision se produit, PostgreSQL stocke plusieurs valeurs dans le même "bucket" et les vérifie toutes lors de la recherche.

## 2. Création et Syntaxe des Index Hash

### Syntaxe de base

```sql
CREATE INDEX nom_index ON table USING HASH (colonne);
```

### Exemple simple

```sql
-- Créer une table d'utilisateurs
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    email TEXT UNIQUE NOT NULL,
    username TEXT UNIQUE NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Insérer des données de test
INSERT INTO users (email, username)  
SELECT  
    'user' || generate_series || '@example.com',
    'user' || generate_series
FROM generate_series(1, 1000000);

-- Créer un index Hash sur email
CREATE INDEX idx_users_email_hash ON users USING HASH (email);

-- Analyser la table
ANALYZE users;
```

### Comparaison avec B-Tree (syntaxe)

```sql
-- Index Hash (égalité uniquement)
CREATE INDEX idx_email_hash ON users USING HASH (email);

-- Index B-Tree (défaut, toutes opérations)
CREATE INDEX idx_email_btree ON users USING BTREE (email);
-- Ou simplement :
CREATE INDEX idx_email_btree ON users (email);  -- B-Tree par défaut
```

### Index Hash sur différents types de données

```sql
-- Sur INTEGER
CREATE INDEX idx_user_id_hash ON users USING HASH (id);

-- Sur TEXT
CREATE INDEX idx_username_hash ON users USING HASH (username);

-- Sur UUID
CREATE TABLE sessions (
    session_id UUID PRIMARY KEY,
    user_id INTEGER,
    data JSONB
);
CREATE INDEX idx_session_id_hash ON sessions USING HASH (session_id);

-- Sur INET (adresse IP)
CREATE TABLE access_logs (
    id BIGSERIAL PRIMARY KEY,
    ip_address INET NOT NULL,
    timestamp TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_ip_hash ON access_logs USING HASH (ip_address);
```

### Types supportés par les index Hash

PostgreSQL 18 supporte les index Hash pour la plupart des types de données :

| Catégorie | Types supportés |
|-----------|-----------------|
| **Numériques** | INTEGER, BIGINT, SMALLINT, NUMERIC, REAL, DOUBLE PRECISION |
| **Texte** | TEXT, VARCHAR, CHAR, NAME |
| **Binaire** | BYTEA |
| **Réseau** | INET, CIDR, MACADDR |
| **UUID** | UUID |
| **Temporels** | DATE, TIMESTAMP, TIMESTAMPTZ, TIME |
| **Booléen** | BOOLEAN |
| **Autres** | OID, JSON (comme TEXT) |

### Index Hash multi-colonnes

**Important** : Contrairement à B-Tree, les index Hash **ne supportent PAS** les index multi-colonnes dans PostgreSQL.

```sql
-- ❌ ERREUR : Index Hash ne peut avoir qu'une seule colonne
CREATE INDEX idx_multi_hash ON users USING HASH (email, username);
-- ERROR: access method "hash" does not support multicolumn indexes

-- ✅ Solution : Créer des index séparés
CREATE INDEX idx_email_hash ON users USING HASH (email);  
CREATE INDEX idx_username_hash ON users USING HASH (username);  
```

## 3. Requêtes Optimisées par les Index Hash

### Recherche d'égalité stricte

```sql
-- ✅ PARFAIT : Égalité exacte (utilise l'index Hash)
SELECT * FROM users WHERE email = 'user123456@example.com';

-- Vérifier avec EXPLAIN
EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM users WHERE email = 'user123456@example.com';  

-- Résultat attendu :
-- Index Scan using idx_users_email_hash on users
-- Time: 0.05 ms
```

### Jointures par égalité

```sql
-- Table de commandes
CREATE TABLE orders (
    id BIGSERIAL PRIMARY KEY,
    user_id INTEGER NOT NULL,
    total NUMERIC(10,2),
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Index Hash sur user_id
CREATE INDEX idx_orders_user_id_hash ON orders USING HASH (user_id);

-- ✅ Jointure optimisée par Hash
SELECT
    u.username,
    o.id AS order_id,
    o.total
FROM users u  
JOIN orders o ON u.id = o.user_id  
WHERE u.email = 'user123456@example.com';  

-- PostgreSQL peut utiliser Hash Join avec l'index Hash
```

### Clause IN avec valeurs explicites

```sql
-- ✅ Peut utiliser l'index Hash (une égalité par valeur)
SELECT * FROM users  
WHERE email IN ('user1@example.com', 'user2@example.com', 'user3@example.com');  

-- PostgreSQL transforme en :
-- WHERE email = 'user1@example.com' OR email = 'user2@example.com' OR ...
```

### Opérateurs supportés

Les index Hash ne supportent **QUE** l'opérateur d'égalité :

| Opérateur | Supporté ? | Exemple |
|-----------|------------|---------|
| `=` | ✅ Oui | `WHERE email = 'alice@example.com'` |
| `!=` ou `<>` | ❌ Non | `WHERE email != 'alice@example.com'` |
| `<` | ❌ Non | `WHERE id < 1000` |
| `>` | ❌ Non | `WHERE id > 1000` |
| `<=` | ❌ Non | `WHERE id <= 1000` |
| `>=` | ❌ Non | `WHERE id >= 1000` |
| `BETWEEN` | ❌ Non | `WHERE id BETWEEN 100 AND 1000` |
| `LIKE` | ❌ Non | `WHERE email LIKE '%@example.com'` |
| `IN` | ✅ Partiel | `WHERE email IN (...)` (transformé en égalités) |
| `IS NULL` | ❌ Non | `WHERE email IS NULL` |

### Cas où l'index Hash N'EST PAS utilisé

```sql
-- ❌ Comparaison (nécessite B-Tree)
SELECT * FROM users WHERE id > 50000;

-- ❌ Plage (nécessite B-Tree)
SELECT * FROM users WHERE id BETWEEN 1000 AND 2000;

-- ❌ Tri (nécessite B-Tree)
SELECT * FROM users ORDER BY email;

-- ❌ LIKE (nécessite B-Tree ou GIN/GiST)
SELECT * FROM users WHERE email LIKE '%@example.com';

-- ❌ Pattern matching
SELECT * FROM users WHERE email ~ '^user[0-9]+';

-- ❌ IS NULL / IS NOT NULL
SELECT * FROM users WHERE email IS NULL;

-- Dans tous ces cas, PostgreSQL :
-- - Ignore l'index Hash
-- - Utilise un autre index (B-Tree) si disponible
-- - Fait un Seq Scan sinon
```

## 4. Comparaison : Hash vs B-Tree

### Tableau comparatif détaillé

| Critère | Hash | B-Tree | Recommandation |
|---------|------|--------|----------------|
| **Égalité (=)** | ⭐⭐⭐⭐⭐ Ultra-rapide | ⭐⭐⭐⭐ Très rapide | Hash légèrement meilleur |
| **Comparaison (>, <)** | ❌ Non supporté | ⭐⭐⭐⭐⭐ Excellent | B-Tree obligatoire |
| **BETWEEN** | ❌ Non supporté | ⭐⭐⭐⭐⭐ Excellent | B-Tree obligatoire |
| **ORDER BY** | ❌ Non supporté | ⭐⭐⭐⭐⭐ Excellent | B-Tree obligatoire |
| **Taille index** | ⭐⭐⭐⭐ Compact | ⭐⭐⭐⭐ Compact | Similaire |
| **Insertion** | ⭐⭐⭐⭐ Rapide | ⭐⭐⭐⭐ Rapide | Similaire |
| **Maintenance** | ⭐⭐⭐ Correcte | ⭐⭐⭐⭐ Bonne | B-Tree plus mature |
| **WAL logging** | ⭐⭐⭐⭐ Depuis PG 10 | ⭐⭐⭐⭐⭐ Complet | B-Tree plus fiable |
| **Multi-colonnes** | ❌ Non supporté | ✅ Supporté | B-Tree obligatoire |
| **Clé étrangère** | ❌ Non utilisable | ✅ Utilisable | B-Tree obligatoire |

### Benchmark de performance

```sql
-- Créer une table de test
CREATE TABLE benchmark (
    id INTEGER PRIMARY KEY,
    code TEXT NOT NULL UNIQUE,
    data TEXT
);

-- Insérer 1 million de lignes
INSERT INTO benchmark  
SELECT  
    generate_series,
    'CODE_' || lpad(generate_series::TEXT, 10, '0'),
    'data'
FROM generate_series(1, 1000000);

-- Créer les deux types d'index
CREATE INDEX idx_code_hash ON benchmark USING HASH (code);  
CREATE INDEX idx_code_btree ON benchmark USING BTREE (code);  

ANALYZE benchmark;

-- Test 1 : Recherche d'égalité exacte
\timing on

-- Avec Hash
SET enable_indexscan = off;  -- Forcer Bitmap Scan  
EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM benchmark WHERE code = 'CODE_0000500000';  

-- Avec B-Tree
DROP INDEX idx_code_hash;  
EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM benchmark WHERE code = 'CODE_0000500000';  

-- Résultats typiques :
-- Hash :   Index Scan → 0.03 ms
-- B-Tree : Index Scan → 0.04 ms
-- Différence : Négligeable en pratique
```

### Taille comparative

```sql
-- Comparer la taille des index
SELECT
    indexname,
    pg_size_pretty(pg_relation_size(indexname::regclass)) AS taille
FROM pg_indexes  
WHERE tablename = 'benchmark'  
  AND indexname LIKE 'idx_code%';

-- Résultat typique sur 1M lignes :
-- idx_code_hash  : 22 MB
-- idx_code_btree : 21 MB
-- Taille : Quasi identique
```

### Quand préférer Hash à B-Tree ?

En réalité, dans PostgreSQL moderne (depuis la version 10 avec le WAL logging), **B-Tree est généralement préférable** car :

1. **Performance similaire** : B-Tree est presque aussi rapide pour l'égalité  
2. **Plus polyvalent** : Supporte toutes les opérations  
3. **Plus mature** : Meilleure optimisation et maintenance  
4. **Support d'unicité** : Peut être utilisé pour UNIQUE et PRIMARY KEY  
5. **Multi-colonnes** : Supporte les index composites

**Utilisez Hash uniquement si** :
- Vous êtes **absolument certain** de ne jamais avoir besoin de comparaisons
- Vous avez des mesures montrant un gain réel
- La colonne a une **très haute cardinalité** (beaucoup de valeurs uniques)

## 5. Cas d'Usage Spécifiques

### Cas 1 : Recherche par UUID

```sql
-- Sessions avec UUID
CREATE TABLE user_sessions (
    session_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id INTEGER NOT NULL,
    expires_at TIMESTAMPTZ NOT NULL,
    data JSONB
);

-- Index Hash sur session_id (recherche d'égalité uniquement)
CREATE INDEX idx_sessions_id_hash ON user_sessions USING HASH (session_id);

-- Requête typique : vérifier si session existe
SELECT * FROM user_sessions  
WHERE session_id = 'a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11';  

-- Alternative avec B-Tree (recommandé)
CREATE INDEX idx_sessions_id_btree ON user_sessions (session_id);
```

### Cas 2 : Lookup par clé externe

```sql
-- Table de traduction de codes
CREATE TABLE code_mapping (
    external_code TEXT PRIMARY KEY,
    internal_id INTEGER NOT NULL,
    description TEXT
);

-- Index Hash pour lookup rapide
CREATE INDEX idx_mapping_code_hash ON code_mapping USING HASH (external_code);

-- Requête de traduction
SELECT internal_id  
FROM code_mapping  
WHERE external_code = 'EXT_CODE_12345';  

-- Utilisé dans les jointures
SELECT
    o.order_id,
    m.internal_id,
    m.description
FROM orders o  
JOIN code_mapping m ON o.external_reference = m.external_code;  
```

### Cas 3 : Déduplication par hash

```sql
-- Table avec hash du contenu pour déduplication
CREATE TABLE documents (
    id BIGSERIAL PRIMARY KEY,
    title TEXT,
    content TEXT,
    content_hash TEXT NOT NULL,  -- SHA256 du contenu
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Index Hash pour détecter doublons rapidement
CREATE INDEX idx_docs_hash ON documents USING HASH (content_hash);

-- Vérifier si document existe déjà
SELECT id  
FROM documents  
WHERE content_hash = 'abc123...def';  

-- Éviter les doublons à l'insertion
INSERT INTO documents (title, content, content_hash)  
SELECT 'Mon document', 'Contenu...', 'abc123...def'  
WHERE NOT EXISTS (  
    SELECT 1 FROM documents WHERE content_hash = 'abc123...def'
);
```

### Cas 4 : Cache lookup

```sql
-- Table de cache simple
CREATE TABLE app_cache (
    cache_key TEXT PRIMARY KEY,
    cache_value JSONB NOT NULL,
    expires_at TIMESTAMPTZ NOT NULL
);

-- Index Hash sur cache_key
CREATE INDEX idx_cache_key_hash ON app_cache USING HASH (cache_key);

-- Récupération depuis le cache
SELECT cache_value  
FROM app_cache  
WHERE cache_key = 'user:profile:12345'  
  AND expires_at > NOW();

-- Invalidation
DELETE FROM app_cache WHERE cache_key = 'user:profile:12345';
```

### Cas 5 : Vérification d'existence

```sql
-- Whitelist/Blacklist
CREATE TABLE email_whitelist (
    email TEXT PRIMARY KEY,
    added_at TIMESTAMPTZ DEFAULT NOW(),
    added_by TEXT
);

CREATE INDEX idx_whitelist_email_hash ON email_whitelist USING HASH (email);

-- Vérifier rapidement si email est autorisé
SELECT EXISTS (
    SELECT 1 FROM email_whitelist WHERE email = 'user@example.com'
) AS is_whitelisted;

-- Dans une fonction
CREATE OR REPLACE FUNCTION is_email_allowed(p_email TEXT)  
RETURNS BOOLEAN AS $$  
BEGIN  
    RETURN EXISTS (
        SELECT 1 FROM email_whitelist WHERE email = p_email
    );
END;
$$ LANGUAGE plpgsql STABLE;
```

## 6. Limitations et Contraintes

### Limitation 1 : Pas de support UNIQUE

```sql
-- ❌ IMPOSSIBLE : Index Hash ne peut pas être UNIQUE
CREATE UNIQUE INDEX idx_email_unique_hash ON users USING HASH (email);
-- ERROR: access method "hash" does not support unique indexes

-- ✅ Solution : Utiliser B-Tree pour UNIQUE
CREATE UNIQUE INDEX idx_email_unique_btree ON users (email);
```

### Limitation 2 : Pas de support PRIMARY KEY

```sql
-- ❌ IMPOSSIBLE : PRIMARY KEY utilise toujours B-Tree
CREATE TABLE test (
    id INTEGER PRIMARY KEY USING HASH  -- Syntaxe invalide
);

-- PostgreSQL crée automatiquement un index B-Tree pour PK
CREATE TABLE test (
    id INTEGER PRIMARY KEY  -- Index B-Tree automatique
);
```

### Limitation 3 : Pas de support FOREIGN KEY

```sql
-- Les FOREIGN KEY nécessitent B-Tree (pour vérifications de range)
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id)  -- Nécessite B-Tree sur users(id)
);

-- L'index Hash ne peut pas être utilisé pour FK
```

### Limitation 4 : Pas d'index multi-colonnes

```sql
-- ❌ IMPOSSIBLE
CREATE INDEX idx_multi_hash ON users USING HASH (email, username);
-- ERROR: access method "hash" does not support multicolumn indexes

-- ✅ Solution : Index séparés ou B-Tree multi-colonnes
CREATE INDEX idx_email_hash ON users USING HASH (email);  
CREATE INDEX idx_username_hash ON users USING HASH (username);  
-- Ou
CREATE INDEX idx_multi_btree ON users (email, username);
```

### Limitation 5 : Pas de REINDEX CONCURRENTLY avant PG 12

```sql
-- Dans PostgreSQL 11 et antérieurs :
-- ❌ REINDEX CONCURRENTLY ne fonctionnait pas avec Hash

-- Dans PostgreSQL 12+ (dont PG 18) :
-- ✅ Supporté
REINDEX INDEX CONCURRENTLY idx_users_email_hash;
```

## 7. Maintenance et Monitoring

### Vérifier l'utilisation d'un index Hash

```sql
-- Statistiques d'utilisation
SELECT
    schemaname,
    tablename,
    indexname,
    idx_scan AS utilisations,
    idx_tup_read AS lignes_lues,
    idx_tup_fetch AS lignes_retournees,
    pg_size_pretty(pg_relation_size(indexrelid)) AS taille
FROM pg_stat_user_indexes  
WHERE indexname = 'idx_users_email_hash';  

-- Si idx_scan = 0 après un certain temps, l'index n'est jamais utilisé
```

### Vérifier qu'une requête utilise l'index Hash

```sql
EXPLAIN (ANALYZE, BUFFERS, VERBOSE)  
SELECT * FROM users WHERE email = 'user12345@example.com';  

-- Cherchez dans le plan :
-- "Index Scan using idx_users_email_hash on users"
-- ou
-- "Bitmap Index Scan on idx_users_email_hash"
```

### Maintenance des index Hash

```sql
-- Reconstruire l'index (si problèmes ou fragmentation)
REINDEX INDEX idx_users_email_hash;

-- Avec PostgreSQL 12+ : reconstruction sans verrouillage
REINDEX INDEX CONCURRENTLY idx_users_email_hash;

-- Reconstruire tous les index d'une table
REINDEX TABLE users;

-- Analyser les statistiques
ANALYZE users;
```

### Surveiller la taille et la fragmentation

```sql
-- Taille de l'index
SELECT
    indexname,
    pg_size_pretty(pg_relation_size(indexrelid)) AS taille,
    pg_size_pretty(pg_table_size(tablename::regclass)) AS taille_table,
    round(100.0 * pg_relation_size(indexrelid) /
          NULLIF(pg_table_size(tablename::regclass), 0), 2) AS pct_taille
FROM pg_stat_user_indexes  
WHERE indexname = 'idx_users_email_hash';  

-- Vérifier le bloat (fragmentation)
-- Nécessite l'extension pgstattuple
CREATE EXTENSION IF NOT EXISTS pgstattuple;

SELECT * FROM pgstatindex('idx_users_email_hash');
-- Regarder : avg_leaf_density, leaf_fragmentation
```

### Quand reconstruire un index Hash ?

Reconstruisez l'index si :
- ✅ Beaucoup de suppressions/mises à jour (bloat élevé)  
- ✅ Performance dégradée au fil du temps  
- ✅ Après une opération massive (COPY, bulk INSERT)  
- ✅ Migration de version PostgreSQL majeure

## 8. Pièges Courants et Solutions

### Piège 1 : Choisir Hash alors que B-Tree suffit

```sql
-- ❌ PROBLÈME : Index Hash pour une colonne qui sera parfois filtrée par range
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    sku TEXT UNIQUE,
    price NUMERIC(10,2)
);

CREATE INDEX idx_sku_hash ON products USING HASH (sku);

-- Requêtes actuelles (égalité) :
SELECT * FROM products WHERE sku = 'PROD-12345';  -- ✅ OK avec Hash

-- Mais plus tard, besoin de :
SELECT * FROM products WHERE sku LIKE 'PROD-%';  -- ❌ Hash inutile

-- ✅ SOLUTION : Utiliser B-Tree par défaut (plus flexible)
DROP INDEX idx_sku_hash;  
CREATE INDEX idx_sku_btree ON products (sku);  
```

### Piège 2 : Oublier que Hash ne supporte pas NULL

```sql
-- ❌ PROBLÈME : Rechercher les valeurs NULL
SELECT * FROM users WHERE email IS NULL;
-- L'index Hash ne sera PAS utilisé

-- ✅ SOLUTION :
-- 1. Utiliser B-Tree si NULL est fréquent
-- 2. Ou ajouter une contrainte NOT NULL
ALTER TABLE users ALTER COLUMN email SET NOT NULL;
```

### Piège 3 : Index Hash sur colonne de clé étrangère

```sql
-- ❌ PROBLÈME : Index Hash sur FK
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id)
);

CREATE INDEX idx_orders_user_hash ON orders USING HASH (user_id);

-- Problèmes :
-- 1. FK nécessite B-Tree sur users(id)
-- 2. Souvent besoin de ranges : WHERE user_id IN (...)
-- 3. Peut-être ORDER BY user_id

-- ✅ SOLUTION : B-Tree sur les FK
CREATE INDEX idx_orders_user_btree ON orders (user_id);
```

### Piège 4 : Mélanger sensibilité à la casse

```sql
-- ❌ PROBLÈME : Hash est sensible à la casse
CREATE INDEX idx_username_hash ON users USING HASH (username);

-- Ces deux requêtes produisent des hash différents :
SELECT * FROM users WHERE username = 'Alice';  -- hash('Alice')  
SELECT * FROM users WHERE username = 'alice';  -- hash('alice') ≠ hash('Alice')  

-- ✅ SOLUTION 1 : Index sur LOWER()
CREATE INDEX idx_username_lower_hash ON users USING HASH (LOWER(username));  
SELECT * FROM users WHERE LOWER(username) = LOWER('Alice');  

-- ✅ SOLUTION 2 : Utiliser CITEXT
CREATE EXTENSION IF NOT EXISTS citext;  
ALTER TABLE users ALTER COLUMN username TYPE CITEXT;  
CREATE INDEX idx_username_hash ON users USING HASH (username);  
```

### Piège 5 : Ne pas vérifier l'utilisation réelle

```sql
-- ❌ PROBLÈME : Créer un index Hash "au cas où"
CREATE INDEX idx_some_col_hash ON big_table USING HASH (some_col);

-- Après 6 mois, l'index n'a jamais été utilisé
SELECT idx_scan FROM pg_stat_user_indexes  
WHERE indexname = 'idx_some_col_hash';  
-- idx_scan = 0

-- ✅ SOLUTION : Supprimer les index inutilisés
DROP INDEX idx_some_col_hash;
```

### Piège 6 : Index Hash sur colonne avec peu de valeurs distinctes

```sql
-- ❌ PROBLÈME : Index Hash sur colonne avec faible cardinalité
CREATE TABLE orders (
    id BIGSERIAL PRIMARY KEY,
    status TEXT  -- Seulement 5 valeurs : pending, processing, shipped, delivered, cancelled
);

CREATE INDEX idx_status_hash ON orders USING HASH (status);

-- Problèmes :
-- 1. Beaucoup de collisions (5 buckets différents)
-- 2. B-Tree ou Bitmap scan serait mieux
-- 3. Pas de bénéfice réel

-- ✅ SOLUTION : Pas d'index ou B-Tree
-- Pour faible cardinalité, souvent pas besoin d'index du tout
```

---

## 9. Hash vs B-Tree : Arbre de Décision

```
Avez-vous UNIQUEMENT besoin de recherches d'égalité stricte (=) ?
├─ NON (besoin de >, <, BETWEEN, ORDER BY, LIKE) → B-Tree
└─ OUI → Continuez

Avez-vous besoin de UNIQUE ou PRIMARY KEY ?
├─ OUI → B-Tree (Hash ne supporte pas)
└─ NON → Continuez

Avez-vous besoin d'index multi-colonnes ?
├─ OUI → B-Tree (Hash ne supporte pas)
└─ NON → Continuez

La colonne contient-elle beaucoup de valeurs distinctes (haute cardinalité) ?
├─ NON (peu de valeurs) → B-Tree ou pas d'index
└─ OUI → Continuez

Avez-vous des benchmarks montrant que Hash est plus rapide ?
├─ NON → B-Tree (par défaut, plus sûr)
└─ OUI → Hash peut être envisagé

→ Dans 95% des cas : B-Tree est le meilleur choix
→ Hash uniquement pour cas très spécifiques
```

---

## 10. Bonnes Pratiques et Recommandations

### ✅ Checklist avant de créer un index Hash

- [ ] Ai-je **uniquement** besoin d'égalité stricte ?  
- [ ] La colonne a-t-elle une **haute cardinalité** ?  
- [ ] Suis-je **certain** de ne jamais avoir besoin de comparaisons ?  
- [ ] Ai-je **testé** avec B-Tree et Hash ?  
- [ ] L'index Hash apporte-t-il un **gain mesurable** ?

### ✅ Recommandations générales

1. **Par défaut : B-Tree**
   - Plus polyvalent, performances similaires
   - Supporté par toutes les contraintes
   - Mieux optimisé dans le temps

2. **Hash uniquement si** :
   - Besoin prouvé par benchmarks
   - Cas d'usage très spécifique (UUID lookup, cache key)
   - Haute cardinalité confirmée

3. **Surveiller l'utilisation** :
   - Vérifier `pg_stat_user_indexes` régulièrement
   - Supprimer les index inutilisés

4. **Documenter le choix** :
   ```sql
   COMMENT ON INDEX idx_users_email_hash IS
   'Index Hash pour lookup UUID uniquement (égalité stricte)';
   ```

5. **Maintenir régulièrement** :
   - REINDEX après bulk operations
   - ANALYZE pour statistiques à jour

## 11. Histoire et Évolution des Index Hash

### Évolution dans PostgreSQL

| Version | Changement majeur |
|---------|-------------------|
| **PG 8.x et avant** | Hash disponibles mais non recommandés (pas de WAL logging) |
| **PG 10 (2017)** | 🎉 **WAL logging ajouté** → Hash enfin fiables pour production |
| **PG 12 (2019)** | Support REINDEX CONCURRENTLY |
| **PG 13 (2020)** | Améliorations de performance |
| **PG 18 (2025)** | Optimisations continues, mais B-Tree reste dominant |

### Pourquoi B-Tree a gagné ?

Malgré les améliorations, B-Tree reste préféré car :
1. **Polyvalence** : Supporte tous les opérateurs  
2. **Maturité** : 40+ ans d'optimisations  
3. **Performance** : Quasi égal à Hash pour l'égalité  
4. **Fiabilité** : Historique plus long en production

## 12. Exemples Concrets : Quand Hash EST Justifié

### Exemple 1 : API Token Verification

```sql
-- Table de tokens d'API
CREATE TABLE api_tokens (
    token TEXT PRIMARY KEY,  -- SHA256 hash
    user_id INTEGER NOT NULL,
    scopes TEXT[],
    expires_at TIMESTAMPTZ NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Index Hash justifié : uniquement vérification d'égalité
CREATE INDEX idx_api_tokens_hash ON api_tokens USING HASH (token);

-- Requête typique : vérifier token
SELECT user_id, scopes, expires_at  
FROM api_tokens  
WHERE token = 'abc123...xyz'  -- Hash direct  
  AND expires_at > NOW();

-- Jamais besoin de : LIKE, >, <, ORDER BY sur token
```

### Exemple 2 : Distributed Cache Key Lookup

```sql
-- Table de cache distribué
CREATE TABLE distributed_cache (
    cache_key TEXT PRIMARY KEY,
    cache_value BYTEA NOT NULL,
    ttl INTEGER NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Hash optimal : lookup ultra-rapide par clé exacte
CREATE INDEX idx_cache_key_hash ON distributed_cache USING HASH (cache_key);

-- Pattern d'utilisation :
-- GET /cache/{key}
SELECT cache_value FROM distributed_cache WHERE cache_key = $1;

-- SET /cache/{key}
INSERT INTO distributed_cache (cache_key, cache_value, ttl)  
VALUES ($1, $2, $3)  
ON CONFLICT (cache_key) DO UPDATE SET cache_value = EXCLUDED.cache_value;  

-- DELETE /cache/{key}
DELETE FROM distributed_cache WHERE cache_key = $1;
```

### Exemple 3 : Fingerprint Deduplication

```sql
-- Détection de doublons par empreinte
CREATE TABLE file_uploads (
    id BIGSERIAL PRIMARY KEY,
    filename TEXT NOT NULL,
    content_fingerprint TEXT NOT NULL,  -- MD5/SHA256
    file_size BIGINT,
    upload_date TIMESTAMPTZ DEFAULT NOW()
);

-- Hash pour déduplication rapide
CREATE INDEX idx_uploads_fingerprint_hash ON file_uploads  
USING HASH (content_fingerprint);  

-- Avant upload : vérifier si fichier existe déjà
SELECT id, filename  
FROM file_uploads  
WHERE content_fingerprint = 'abc123def456...';  

-- Si EXISTS → fichier déjà présent, ne pas uploader
```

## 13. Conclusion

### Points clés à retenir

1. **Hash = Égalité uniquement** : Aucune autre opération supportée  
2. **B-Tree généralement meilleur** : Performance similaire + polyvalence  
3. **Hash rarement justifié** : Cas très spécifiques uniquement  
4. **Depuis PG 10 : fiable** : WAL logging résout les anciens problèmes  
5. **Simple mais limité** : Conception minimaliste, usage restreint

### Quand utiliser Hash (récapitulatif)

✅ **Utilisez Hash si** :
- Égalité stricte **uniquement**
- Haute cardinalité (UUID, tokens, hashes)
- Benchmarks montrent un gain
- Jamais besoin de comparaisons

❌ **N'utilisez PAS Hash si** :
- Besoin de >, <, BETWEEN
- Besoin de ORDER BY
- UNIQUE ou PRIMARY KEY
- Index multi-colonnes
- FOREIGN KEY
- Doute sur usage futur

### Recommandation finale

**Dans 95% des cas : utilisez B-Tree**

B-Tree est :
- ✅ Aussi rapide pour l'égalité  
- ✅ Supporte toutes les opérations  
- ✅ Plus mature et optimisé  
- ✅ Standard de l'industrie

Hash n'est utile que pour des cas très spécifiques où vous avez :
- Besoin prouvé par benchmarks
- Contraintes d'usage strictes (égalité uniquement)
- Haute cardinalité confirmée

### L'avenir des index Hash

Les index Hash resteront probablement un **outil de niche** dans PostgreSQL. Le B-Tree a tellement évolué qu'il est devenu le couteau suisse universel, laissant Hash pour des micro-optimisations spécifiques.

**En production** : Commencez toujours par B-Tree, ne passez à Hash que si vous avez une raison mesurée et documentée.
