# 13.4.5. SP-GiST (Space-Partitioned GiST) : Structures non-équilibrées

## Introduction

Les index **SP-GiST** (Space-Partitioned Generalized Search Tree) sont un type d'index avancé de PostgreSQL conçu pour des **structures de données partitionnées dans l'espace**. Contrairement à GiST qui crée des arbres équilibrés, SP-GiST supporte des structures **non-équilibrées** qui reflètent mieux certains types de données.

### L'analogie de l'arbre généalogique vs l'organigramme

Pour comprendre la différence :

- **GiST (équilibré)** : C'est comme un organigramme d'entreprise où tous les niveaux sont bien définis et équilibrés (direction → départements → équipes)  
- **SP-GiST (non-équilibré)** : C'est comme un arbre généalogique où certaines branches peuvent être très longues (arrière-arrière-grands-parents) et d'autres courtes

### Qu'est-ce que "non-équilibré" signifie ?

Un arbre **équilibré** (comme B-Tree ou GiST) maintient une profondeur similaire pour toutes les branches :
```
        Racine
       /  |  \
      A   B   C     ← Niveau 1
     /|   |   |\
    D E   F   G H   ← Niveau 2
```

Un arbre **non-équilibré** (SP-GiST) peut avoir des branches de profondeurs différentes :
```
        Racine
       /  |  \
      A   B   C     ← Niveau 1
     /    |
    D     E          ← Niveau 2
   /     /|\
  F     G H I        ← Niveau 3
 /
J                    ← Niveau 4
```

Cette flexibilité est idéale pour certaines structures de données naturellement déséquilibrées.

### Caractéristiques essentielles

- ✅ **Structures non-équilibrées** : Arbres naturellement déséquilibrés  
- ✅ **Partitionnement spatial** : Divise l'espace en régions  
- ✅ **Quadtrees** : Pour données géospatiales (2D)  
- ✅ **K-d trees** : Pour recherche multidimensionnelle  
- ✅ **Radix trees** : Pour préfixes de texte  
- ✅ **Performance prévisible** : Pas de rééquilibrage coûteux

### Quand utiliser SP-GiST ?

✅ **À utiliser pour :**
- Recherche de **préfixes** dans du texte (tel%, email@%)
- Données **géospatiales** avec quadtree
- Points dans un **espace multidimensionnel** (k-d tree)
- Adresses **IP** et **CIDR** (routage réseau)
- Structures naturellement **hiérarchiques** et déséquilibrées

❌ **À éviter pour :**
- Données purement unidimensionnelles (préférer B-Tree)
- Recherches full-text complètes (préférer GIN)
- Données sans structure spatiale ou hiérarchique

## 1. Comprendre les Structures de Données SP-GiST

### Structure 1 : Quadtree (Arbre Quaternaire)

Un **quadtree** divise récursivement un espace 2D en quatre quadrants.

#### Visualisation d'un quadtree

```
Espace 2D divisé en quadrants :

    +-------+-------+
    |  NW   |  NE   |  Nord-Ouest | Nord-Est
    |   •   |       |
    +-------+-------+
    |  SW   |  SE   |  Sud-Ouest  | Sud-Est
    |       |   •   |
    +-------+-------+

Si trop de points dans un quadrant → subdivision :

    +---+---+-------+
    |NW1|NW2|  NE   |
    +---+---+       |
    |NW3|NW4|       |
    +-------+-------+
    |  SW   |  SE   |
    |       |   •   |
    +-------+-------+
```

**Cas d'usage** : Points GPS, coordonnées spatiales, données cartographiques

### Structure 2 : K-d Tree (K-dimensional Tree)

Un **k-d tree** partitionne l'espace en séparant alternativement selon différentes dimensions.

#### Exemple avec des points 2D (x, y)

```
Points : A(2,3), B(5,4), C(9,6), D(4,7), E(8,1), F(7,2)

Division alternée X puis Y puis X...

Niveau 1 (X) : diviser par x=5
    x < 5          |     x >= 5
  A(2,3), D(4,7)   |   C(9,6), E(8,1), F(7,2), B(5,4)

Niveau 2 (Y) : diviser par y
    y < 5          |     y >= 5
  A(2,3), E(8,1)   |   D(4,7), C(9,6)
  F(7,2), B(5,4)
```

**Cas d'usage** : Recherche de voisins les plus proches, clustering spatial

### Structure 3 : Radix Tree (Prefix Tree)

Un **radix tree** stocke des chaînes en partageant les préfixes communs.

#### Visualisation d'un radix tree

```
Mots : "test", "tea", "ted", "to", "inn", "in"

        [root]
       /      \
    "te"      "in"
    / \         \
  "a" "d"      "n"
  /   /
"st" ""

Résultat :
- "tea"  : te → a
- "ted"  : te → d
- "test" : te → st
- "to"   : to
- "in"   : in
- "inn"  : in → n
```

**Cas d'usage** : Autocomplétion, recherche de préfixes, suggestions de recherche

### Structure 4 : Suffix Tree

Un **suffix tree** stocke tous les suffixes d'une chaîne pour recherche rapide.

**Cas d'usage** : Recherche de motifs, bioinformatique (séquences ADN)

## 2. Différence entre GiST et SP-GiST

### Comparaison conceptuelle

| Critère | GiST | SP-GiST |
|---------|------|---------|
| **Structure** | Arbre équilibré | Arbre non-équilibré |
| **Profondeur** | Uniforme pour toutes les branches | Variable selon les données |
| **Partitionnement** | Régions qui se chevauchent | Partitionnement strict (disjoints) |
| **Rééquilibrage** | Coûteux lors d'insertions | Pas de rééquilibrage nécessaire |
| **Cas d'usage** | Données multidimensionnelles générales | Données avec structure hiérarchique naturelle |

### Visualisation : Partitionnement GiST vs SP-GiST

#### GiST : Régions qui se chevauchent
```
    +-------+
    | Zone1 |
    |   +-------+
    +---|Zone2  |
        |       |
        +-------+

Les zones peuvent se chevaucher
→ Recherche peut vérifier plusieurs zones
```

#### SP-GiST : Partitionnement strict
```
    +-------+-------+
    | Zone1 | Zone2 |
    +-------+-------+
    | Zone3 | Zone4 |
    +-------+-------+

Les zones sont disjointes
→ Recherche va directement à la bonne zone
```

### Quand préférer SP-GiST à GiST ?

**Choisissez SP-GiST si** :
- ✅ Données avec **structure naturelle** (préfixes, quadrants)  
- ✅ Besoin de **partitionnement strict** (pas de chevauchement)  
- ✅ Recherches de **préfixes** ou **suffixes**  
- ✅ Points dans un **espace multidimensionnel** régulier

**Choisissez GiST si** :
- ✅ Données **géométriques complexes** (polygones, lignes)  
- ✅ Chevauchement de **régions inévitable**  
- ✅ Hiérarchies avec **ltree**

## 3. Création et Syntaxe des Index SP-GiST

### Syntaxe de base

```sql
CREATE INDEX nom_index ON table USING SPGIST (colonne);
```

### Classes d'opérateurs SP-GiST

SP-GiST supporte différentes classes d'opérateurs selon le type de données :

| Type de données | Classe d'opérateur | Structure utilisée | Cas d'usage |
|----------------|-------------------|-------------------|-------------|
| `TEXT`, `VARCHAR` | `text_ops` | Radix tree | Recherche de préfixes |
| `INET`, `CIDR` | `inet_ops` | Radix tree | Adresses réseau |
| `geometry(Point)` | `kd_point_ops` | K-d tree | Points 2D |
| `box` | `quad_box_ops` | Quadtree | Rectangles 2D |
| `point` | `quad_point_ops` | Quadtree | Points natifs PostgreSQL |
| `range types` | `range_ops` | Radix tree | Intervalles |

## 4. SP-GiST pour la Recherche de Préfixes (Texte)

### Principe

SP-GiST avec un radix tree est **excellent** pour les recherches de préfixes car il partage les préfixes communs.

### Création d'un index SP-GiST sur TEXT

```sql
-- Table d'utilisateurs avec emails
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    email TEXT NOT NULL,
    username TEXT NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Insérer des données de test
INSERT INTO users (email, username)  
SELECT  
    'user' || generate_series || '@example.com',
    'user' || generate_series
FROM generate_series(1, 100000);

-- Ajouter des emails avec préfixes variés
INSERT INTO users (email, username) VALUES
    ('admin@example.com', 'admin'),
    ('admin.john@example.com', 'admin_john'),
    ('admin.jane@example.com', 'admin_jane'),
    ('support@example.com', 'support'),
    ('support.tech@example.com', 'support_tech');

-- Créer un index SP-GiST pour recherche de préfixes
CREATE INDEX idx_users_email_spgist ON users USING SPGIST (email);

ANALYZE users;
```

### Requêtes optimisées par SP-GiST

#### Recherche de préfixe avec LIKE

```sql
-- ✅ EXCELLENT : Préfixe au début (utilise SP-GiST)
SELECT email, username  
FROM users  
WHERE email LIKE 'admin%';  

-- Plan d'exécution :
-- Index Scan using idx_users_email_spgist on users

-- Résultat :
-- admin@example.com
-- admin.john@example.com
-- admin.jane@example.com
```

#### Recherche de préfixe avec opérateurs texte

```sql
-- ✅ Opérateur ^@ (starts with) - spécifique à SP-GiST
SELECT email  
FROM users  
WHERE email ^@ 'support';  

-- Équivalent à LIKE 'support%'
```

#### Comparaison avec B-Tree et GiST

```sql
-- Créer les trois types d'index pour comparaison
CREATE INDEX idx_email_btree ON users USING BTREE (email);  
CREATE INDEX idx_email_gist ON users USING GIST (email);  
CREATE INDEX idx_email_spgist ON users USING SPGIST (email);  

-- Test de performance : recherche de préfixe
EXPLAIN (ANALYZE, BUFFERS)  
SELECT COUNT(*) FROM users WHERE email LIKE 'user1%';  

-- Résultats typiques :
-- SP-GiST : ~2 ms   (optimal pour préfixes)
-- B-Tree :  ~3 ms   (bon aussi)
-- GiST :    ~15 ms  (moins efficace)
```

### Cas d'usage : Autocomplétion

```sql
-- Fonction d'autocomplétion
CREATE OR REPLACE FUNCTION autocomplete_email(prefix TEXT, max_results INTEGER DEFAULT 10)  
RETURNS TABLE(email TEXT, username TEXT) AS $$  
BEGIN  
    RETURN QUERY
    SELECT u.email, u.username
    FROM users u
    WHERE u.email LIKE prefix || '%'
    ORDER BY u.email
    LIMIT max_results;
END;
$$ LANGUAGE plpgsql STABLE;

-- Utilisation
SELECT * FROM autocomplete_email('admin');

-- Résultat :
-- admin.jane@example.com | admin_jane
-- admin.john@example.com | admin_john
-- admin@example.com      | admin
```

### Limitations importantes

```sql
-- ✅ SP-GiST est utilisé
SELECT * FROM users WHERE email LIKE 'admin%';

-- ❌ SP-GiST N'EST PAS utilisé (préfixe manquant)
SELECT * FROM users WHERE email LIKE '%admin%';  -- Scan séquentiel  
SELECT * FROM users WHERE email LIKE '%@example.com';  -- Scan séquentiel  

-- ✅ Possible mais moins optimal
SELECT * FROM users WHERE email LIKE 'admin.%';  -- Utilise SP-GiST
```

## 5. SP-GiST pour les Adresses Réseau (INET/CIDR)

### Principe

Les adresses IP et les plages CIDR ont une structure hiérarchique naturelle (préfixes de bits), parfaite pour SP-GiST.

### Création d'un index SP-GiST sur INET

```sql
-- Table de logs réseau
CREATE TABLE network_logs (
    id BIGSERIAL PRIMARY KEY,
    source_ip INET NOT NULL,
    destination_ip INET NOT NULL,
    timestamp TIMESTAMPTZ DEFAULT NOW(),
    bytes_transferred BIGINT,
    protocol TEXT
);

-- Insérer des données de test
INSERT INTO network_logs (source_ip, destination_ip, bytes_transferred, protocol)  
SELECT  
    ('192.168.' || (random() * 255)::INTEGER || '.' || (random() * 255)::INTEGER)::INET,
    ('10.0.' || (random() * 255)::INTEGER || '.' || (random() * 255)::INTEGER)::INET,
    (random() * 10000000)::BIGINT,
    (ARRAY['TCP', 'UDP', 'ICMP'])[floor(random() * 3 + 1)]
FROM generate_series(1, 1000000);

-- Index SP-GiST sur source_ip
CREATE INDEX idx_network_source_spgist ON network_logs USING SPGIST (source_ip);

ANALYZE network_logs;
```

### Table de routage avec CIDR

```sql
-- Table de routage réseau
CREATE TABLE routing_table (
    id SERIAL PRIMARY KEY,
    network CIDR NOT NULL,
    gateway INET NOT NULL,
    interface TEXT,
    metric INTEGER
);

-- Insérer des routes
INSERT INTO routing_table (network, gateway, interface, metric) VALUES
    ('192.168.0.0/16', '10.0.0.1', 'eth0', 10),
    ('192.168.1.0/24', '10.0.0.2', 'eth0', 20),
    ('192.168.1.0/25', '10.0.0.3', 'eth1', 15),
    ('10.0.0.0/8', '172.16.0.1', 'eth2', 50),
    ('172.16.0.0/12', '192.168.1.1', 'eth0', 30);

-- Index SP-GiST pour recherche de route
CREATE INDEX idx_routing_network_spgist ON routing_table USING SPGIST (network);
```

### Requêtes réseau optimisées

#### Recherche de containment (IP dans réseau)

```sql
-- Trouver à quel réseau appartient une IP
SELECT network, gateway, interface  
FROM routing_table  
WHERE network >>= '192.168.1.50'::INET  -- Opérateur "contient ou égal"  
ORDER BY masklen(network) DESC  -- Plus spécifique d'abord  
LIMIT 1;  

-- Résultat : 192.168.1.0/25 (plus spécifique que /24 ou /16)
```

#### Opérateurs réseau supportés par SP-GiST

```sql
-- << : strictement contenu dans
SELECT * FROM routing_table WHERE '192.168.1.0/25'::CIDR << '192.168.0.0/16'::CIDR;

-- <<= : contenu dans ou égal
SELECT * FROM routing_table WHERE network <<= '192.168.0.0/16'::CIDR;

-- >> : strictement contient
SELECT * FROM routing_table WHERE '192.168.0.0/16'::CIDR >> network;

-- >>= : contient ou égal
SELECT * FROM routing_table WHERE '192.168.0.0/16'::CIDR >>= network;

-- && : chevauche
SELECT * FROM routing_table WHERE network && '192.168.1.0/24'::CIDR;
```

#### Cas d'usage : Filtrage de logs par subnet

```sql
-- Tous les logs d'un subnet spécifique
SELECT
    source_ip,
    destination_ip,
    bytes_transferred,
    timestamp
FROM network_logs  
WHERE source_ip <<= '192.168.1.0/24'::CIDR  -- IPs dans ce subnet  
ORDER BY timestamp DESC  
LIMIT 100;  

-- SP-GiST parcourt efficacement l'arbre de préfixes
```

### Exemple : Géolocalisation IP

```sql
-- Table de géolocalisation IP
CREATE TABLE ip_geolocation (
    id SERIAL PRIMARY KEY,
    ip_range CIDR NOT NULL,
    country TEXT,
    city TEXT,
    latitude NUMERIC(9,6),
    longitude NUMERIC(9,6)
);

-- Index SP-GiST
CREATE INDEX idx_ipgeo_range_spgist ON ip_geolocation USING SPGIST (ip_range);

-- Données exemple
INSERT INTO ip_geolocation (ip_range, country, city, latitude, longitude) VALUES
    ('8.8.8.0/24', 'US', 'Mountain View', 37.386, -122.084),
    ('1.1.1.0/24', 'AU', 'Sydney', -33.868, 151.207),
    ('208.67.222.0/24', 'US', 'San Francisco', 37.775, -122.419);

-- Lookup géographique
CREATE OR REPLACE FUNCTION geolocate_ip(p_ip INET)  
RETURNS TABLE(country TEXT, city TEXT, latitude NUMERIC, longitude NUMERIC) AS $$  
BEGIN  
    RETURN QUERY
    SELECT g.country, g.city, g.latitude, g.longitude
    FROM ip_geolocation g
    WHERE g.ip_range >>= p_ip
    ORDER BY masklen(g.ip_range) DESC
    LIMIT 1;
END;
$$ LANGUAGE plpgsql STABLE;

-- Utilisation
SELECT * FROM geolocate_ip('8.8.8.8');
-- Résultat : US | Mountain View | 37.386000 | -122.084000
```

## 6. SP-GiST pour les Données Géométriques (Points)

### Quadtree pour points 2D

SP-GiST peut indexer des points géométriques avec une structure de quadtree, particulièrement efficace pour des données distribuées uniformément.

```sql
-- Table de points d'intérêt
CREATE TABLE poi_points (
    id SERIAL PRIMARY KEY,
    nom TEXT NOT NULL,
    location point NOT NULL,  -- Type natif PostgreSQL
    categorie TEXT
);

-- Insérer des points aléatoires
INSERT INTO poi_points (nom, location, categorie)  
SELECT  
    'POI_' || generate_series,
    point(random() * 100, random() * 100),
    (ARRAY['restaurant', 'hotel', 'musee', 'parc'])[floor(random() * 4 + 1)]
FROM generate_series(1, 100000);

-- Index SP-GiST avec quadtree
CREATE INDEX idx_poi_location_spgist ON poi_points USING SPGIST (location);

ANALYZE poi_points;
```

### Requêtes spatiales avec SP-GiST

```sql
-- Recherche dans une boîte (rectangle)
SELECT nom, location  
FROM poi_points  
WHERE location <@ box '((10,10),(20,20))';  -- Points dans la boîte  

-- Recherche de proximité (moins efficace que GiST pour cela)
SELECT nom, location  
FROM poi_points  
ORDER BY location <-> point '(50, 50)'  -- Distance au point  
LIMIT 10;  
```

### K-d Tree pour points (avec PostGIS)

```sql
-- Avec PostGIS (plus puissant que le type natif)
CREATE EXTENSION IF NOT EXISTS postgis;

CREATE TABLE poi_postgis (
    id SERIAL PRIMARY KEY,
    nom TEXT NOT NULL,
    geom geometry(Point, 4326)  -- WGS84
);

-- Insérer des points GPS (Paris)
INSERT INTO poi_postgis (nom, geom) VALUES
    ('Tour Eiffel', ST_Point(2.2945, 48.8584, 4326)),
    ('Louvre', ST_Point(2.3376, 48.8606, 4326)),
    ('Arc de Triomphe', ST_Point(2.2950, 48.8738, 4326));

-- Index SP-GiST avec k-d tree (disponible avec PostGIS)
CREATE INDEX idx_poi_geom_spgist ON poi_postgis USING SPGIST (geom);

-- Note : Pour PostGIS, GiST reste généralement préféré
-- SP-GiST est utile pour des distributions spécifiques de points
```

## 7. SP-GiST pour les Types Range

### Principe

Les types range (intervalles) ont une structure hiérarchique naturelle, adaptée à SP-GiST.

```sql
-- Table de réservations avec ranges temporels
CREATE TABLE reservations (
    id SERIAL PRIMARY KEY,
    ressource_id INTEGER NOT NULL,
    periode tstzrange NOT NULL,  -- Range de timestamps
    utilisateur TEXT,
    EXCLUDE USING GIST (ressource_id WITH =, periode WITH &&)
);

-- Index SP-GiST sur la période
CREATE INDEX idx_reservations_periode_spgist ON reservations USING SPGIST (periode);

-- Insérer des réservations
INSERT INTO reservations (ressource_id, periode, utilisateur) VALUES
    (1, tstzrange('2024-01-15 09:00', '2024-01-15 11:00'), 'Alice'),
    (1, tstzrange('2024-01-15 14:00', '2024-01-15 16:00'), 'Bob'),
    (2, tstzrange('2024-01-15 10:00', '2024-01-15 12:00'), 'Charlie'),
    (1, tstzrange('2024-01-16 09:00', '2024-01-16 11:00'), 'Diana');
```

### Requêtes sur ranges

```sql
-- Trouver les réservations qui chevauchent une période
SELECT *  
FROM reservations  
WHERE periode && tstzrange('2024-01-15 10:00', '2024-01-15 15:00');  

-- Trouver les réservations contenues dans une période
SELECT *  
FROM reservations  
WHERE periode <@ tstzrange('2024-01-15 00:00', '2024-01-16 00:00');  

-- Disponibilités (ranges qui ne chevauchent pas)
SELECT *  
FROM reservations  
WHERE NOT (periode && tstzrange('2024-01-15 12:00', '2024-01-15 13:00'));  
```

### Autres types range

```sql
-- Ranges numériques
CREATE TABLE promotions (
    id SERIAL PRIMARY KEY,
    nom TEXT,
    prix_range numrange  -- Prix entre X et Y
);

CREATE INDEX idx_promo_range_spgist ON promotions USING SPGIST (prix_range);

-- Ranges de dates
CREATE TABLE events (
    id SERIAL PRIMARY KEY,
    titre TEXT,
    periode daterange  -- Du DD/MM/YYYY au DD/MM/YYYY
);

CREATE INDEX idx_events_periode_spgist ON events USING SPGIST (periode);
```

## 8. Performance et Comparaisons

### Comparaison : SP-GiST vs B-Tree vs GiST

#### Pour recherche de préfixes (texte)

```sql
CREATE TABLE test_text (
    id SERIAL PRIMARY KEY,
    email TEXT NOT NULL
);

INSERT INTO test_text (email)  
SELECT 'user' || generate_series || '@' ||  
       (ARRAY['gmail.com', 'yahoo.com', 'hotmail.com'])[floor(random() * 3 + 1)]
FROM generate_series(1, 1000000);

-- Créer les trois index
CREATE INDEX idx_text_btree ON test_text (email);  
CREATE INDEX idx_text_gist ON test_text USING GIST (email);  
CREATE INDEX idx_text_spgist ON test_text USING SPGIST (email);  

ANALYZE test_text;

-- Benchmark : recherche de préfixe
\timing on

-- SP-GiST (optimal)
EXPLAIN ANALYZE SELECT COUNT(*) FROM test_text WHERE email LIKE 'user12%';
-- ~3 ms

-- B-Tree (bon aussi)
EXPLAIN ANALYZE SELECT COUNT(*) FROM test_text WHERE email LIKE 'user12%';
-- ~4 ms

-- GiST (moins bon)
EXPLAIN ANALYZE SELECT COUNT(*) FROM test_text WHERE email LIKE 'user12%';
-- ~20 ms
```

#### Pour adresses réseau (INET)

```sql
-- SP-GiST : Excellent
-- Recherche de containment hiérarchique optimale

-- GiST : Bon
-- Mais structure plus lourde

-- B-Tree : Inadapté
-- Ne supporte pas les opérateurs réseau
```

### Taille comparative

```sql
SELECT
    indexname,
    pg_size_pretty(pg_relation_size(indexrelid)) AS taille,
    idx_scan AS utilisations
FROM pg_stat_user_indexes  
WHERE tablename = 'test_text'  
ORDER BY pg_relation_size(indexrelid) DESC;  

-- Résultats typiques (1M lignes) :
-- idx_text_btree  : 21 MB  (plus compact)
-- idx_text_spgist : 24 MB  (légèrement plus gros)
-- idx_text_gist   : 35 MB  (plus volumineux)
```

## 9. Cas d'Usage Concrets

### Cas 1 : Système d'autocomplétion

```sql
-- Table de recherche avec suggestions
CREATE TABLE search_suggestions (
    id SERIAL PRIMARY KEY,
    term TEXT NOT NULL,
    popularity INTEGER DEFAULT 0,
    category TEXT
);

-- Index SP-GiST pour préfixes
CREATE INDEX idx_suggestions_term_spgist ON search_suggestions USING SPGIST (term);

-- Insérer des suggestions
INSERT INTO search_suggestions (term, popularity, category) VALUES
    ('postgresql tutorial', 1000, 'database'),
    ('postgresql index', 800, 'database'),
    ('postgresql performance', 600, 'database'),
    ('postgres vs mysql', 400, 'comparison'),
    ('python tutorial', 1200, 'programming'),
    ('python flask', 900, 'programming');

-- Fonction d'autocomplétion
CREATE OR REPLACE FUNCTION suggest_search(prefix TEXT, limit_results INTEGER DEFAULT 5)  
RETURNS TABLE(term TEXT, popularity INTEGER) AS $$  
BEGIN  
    RETURN QUERY
    SELECT s.term, s.popularity
    FROM search_suggestions s
    WHERE s.term LIKE prefix || '%'
    ORDER BY s.popularity DESC
    LIMIT limit_results;
END;
$$ LANGUAGE plpgsql STABLE;

-- Utilisation
SELECT * FROM suggest_search('post');

-- Résultat :
-- postgresql tutorial    | 1000
-- postgresql index       | 800
-- postgresql performance | 600
-- postgres vs mysql      | 400
```

### Cas 2 : Routage IP dynamique

```sql
-- Table de routage dynamique
CREATE TABLE dynamic_routing (
    id SERIAL PRIMARY KEY,
    destination_network CIDR NOT NULL,
    next_hop INET NOT NULL,
    interface TEXT,
    metric INTEGER,
    last_updated TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_routing_dest_spgist ON dynamic_routing USING SPGIST (destination_network);

-- Fonction de routage (longest prefix match)
CREATE OR REPLACE FUNCTION find_route(dest_ip INET)  
RETURNS TABLE(  
    network CIDR,
    next_hop INET,
    interface TEXT,
    metric INTEGER
) AS $$
BEGIN
    RETURN QUERY
    SELECT
        dr.destination_network,
        dr.next_hop,
        dr.interface,
        dr.metric
    FROM dynamic_routing dr
    WHERE dr.destination_network >>= dest_ip
    ORDER BY masklen(dr.destination_network) DESC
    LIMIT 1;
END;
$$ LANGUAGE plpgsql STABLE;

-- Test
SELECT * FROM find_route('192.168.1.50');
```

### Cas 3 : Gestion de calendrier avec exclusions

```sql
-- Table de disponibilités
CREATE TABLE disponibilites (
    id SERIAL PRIMARY KEY,
    employe_id INTEGER NOT NULL,
    periode tstzrange NOT NULL,
    type TEXT CHECK (type IN ('disponible', 'occupe', 'conges')),
    EXCLUDE USING GIST (employe_id WITH =, periode WITH &&)
);

-- Index SP-GiST pour recherches temporelles
CREATE INDEX idx_dispo_periode_spgist ON disponibilites USING SPGIST (periode);

-- Trouver les employés disponibles dans une plage
CREATE OR REPLACE FUNCTION employes_disponibles(
    debut TIMESTAMPTZ,
    fin TIMESTAMPTZ
)
RETURNS TABLE(employe_id INTEGER) AS $$  
BEGIN  
    RETURN QUERY
    SELECT DISTINCT d.employe_id
    FROM disponibilites d
    WHERE d.type = 'disponible'
      AND d.periode @> tstzrange(debut, fin)  -- Contient toute la période
    EXCEPT
    SELECT d.employe_id
    FROM disponibilites d
    WHERE d.type IN ('occupe', 'conges')
      AND d.periode && tstzrange(debut, fin);  -- Chevauche
END;
$$ LANGUAGE plpgsql STABLE;
```

### Cas 4 : Système de cache hiérarchique

-- Cache avec clés hiérarchiques
CREATE TABLE hierarchical_cache (
    id SERIAL PRIMARY KEY,
    cache_key TEXT NOT NULL,
    cache_value JSONB,
    expires_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ DEFAULT NOW()

-- Index SP-GiST pour recherche par préfixe
CREATE INDEX idx_hcache_key_spgist ON hierarchical_cache USING SPGIST (cache_key);

-- Clés de type "app:module:feature:id"
INSERT INTO hierarchical_cache (cache_key, cache_value, expires_at) VALUES
    ('app:users:profile:123', '{"name":"Alice"}', NOW() + INTERVAL '1 hour'),
    ('app:users:profile:456', '{"name":"Bob"}', NOW() + INTERVAL '1 hour'),

-- Invalider tout le cache d'un module
DELETE FROM hierarchical_cache  
WHERE cache_key LIKE 'app:users:%'

FROM hierarchical_cache  
WHERE cache_key LIKE 'app:users:profile:%'  
  AND expires_at > NOW();

## 10. Pièges Courants et Solutions

### Piège 1 : Utiliser SP-GiST pour recherche full-text

```sql
-- ❌ PROBLÈME : SP-GiST pour recherche dans le texte
CREATE INDEX idx_content_spgist ON articles USING SPGIST (content);

-- Requêtes non optimisées :
SELECT * FROM articles WHERE content LIKE '%postgresql%';  -- Scan séquentiel  
SELECT * FROM articles WHERE content ~* 'performance';     -- Scan séquentiel  

-- ✅ SOLUTION : GIN pour full-text search
CREATE INDEX idx_content_gin ON articles USING GIN (to_tsvector('french', content));

SELECT * FROM articles  
WHERE to_tsvector('french', content) @@ to_tsquery('french', 'postgresql');  
```

### Piège 2 : SP-GiST sur données sans structure

```sql
-- ❌ PROBLÈME : SP-GiST sur UUID aléatoires
CREATE TABLE sessions (
    session_id UUID PRIMARY KEY DEFAULT gen_random_uuid()
);

CREATE INDEX idx_session_spgist ON sessions USING SPGIST (session_id);

-- Pas de bénéfice : UUIDs sont aléatoires, pas de structure hiérarchique

-- ✅ SOLUTION : B-Tree ou Hash pour UUID
CREATE INDEX idx_session_btree ON sessions (session_id);
```

### Piège 3 : Oublier la limitation de préfixes

```sql
-- ❌ PROBLÈME : Recherche sans préfixe
SELECT * FROM users WHERE email LIKE '%@gmail.com';  -- SP-GiST inutile

-- ✅ SOLUTION : Réorganiser la requête ou utiliser GIN
-- Option 1 : Si souvent recherche par domaine, stocker séparément
ALTER TABLE users ADD COLUMN email_domain TEXT;  
UPDATE users SET email_domain = split_part(email, '@', 2);  
CREATE INDEX idx_domain_btree ON users (email_domain);  

-- Option 2 : GIN avec trigrams
CREATE EXTENSION IF NOT EXISTS pg_trgm;  
CREATE INDEX idx_email_trgm ON users USING GIN (email gin_trgm_ops);  
SELECT * FROM users WHERE email LIKE '%@gmail.com';  -- Utilise GIN  
```

### Piège 4 : SP-GiST pour données géométriques complexes

```sql
-- ❌ PROBLÈME : SP-GiST pour polygones complexes
CREATE TABLE zones (
    id SERIAL PRIMARY KEY,
    frontiere geometry(Polygon, 4326)
);

CREATE INDEX idx_zones_spgist ON zones USING SPGIST (frontiere);

-- SP-GiST quadtree n'est pas optimal pour polygones

-- ✅ SOLUTION : GiST pour géométries complexes
CREATE INDEX idx_zones_gist ON zones USING GIST (frontiere);
```

### Piège 5 : Ne pas analyser après création

```sql
-- ❌ PROBLÈME : Oublier ANALYZE
CREATE INDEX idx_email_spgist ON users USING SPGIST (email);

-- Statistiques non à jour → planificateur peut ne pas l'utiliser

-- ✅ SOLUTION : Toujours ANALYZE
CREATE INDEX idx_email_spgist ON users USING SPGIST (email);  
ANALYZE users;  
```

## 11. Maintenance et Monitoring

### Vérifier l'utilisation d'un index SP-GiST

```sql
-- Statistiques d'utilisation
SELECT
    schemaname,
    tablename,
    indexname,
    idx_scan AS utilisations,
    idx_tup_read AS lignes_lues,
    idx_tup_fetch AS lignes_retournees,
    pg_size_pretty(pg_relation_size(indexrelid)) AS taille
FROM pg_stat_user_indexes  
WHERE indexname LIKE '%spgist%';  
```

### Vérifier qu'une requête utilise SP-GiST

```sql
EXPLAIN (ANALYZE, BUFFERS, VERBOSE)  
SELECT * FROM users WHERE email LIKE 'admin%';  

-- Chercher :
-- "Index Scan using idx_users_email_spgist"
-- ou
-- "Bitmap Index Scan on idx_users_email_spgist"
```

### Maintenance des index SP-GiST

```sql
-- Reconstruire l'index
REINDEX INDEX idx_users_email_spgist;

-- Avec PostgreSQL 12+ : sans verrouillage
REINDEX INDEX CONCURRENTLY idx_users_email_spgist;

-- Analyser les statistiques
ANALYZE users;

-- Vacuum pour nettoyage
VACUUM users;
```

### Surveiller la santé de l'index

```sql
-- Extension pgstattuple pour détails
CREATE EXTENSION IF NOT EXISTS pgstattuple;

SELECT * FROM pgstatindex('idx_users_email_spgist');

-- Regarder :
-- - avg_leaf_density : densité des feuilles
-- - leaf_fragmentation : fragmentation
```

## Conclusion

Les index **SP-GiST** sont un outil spécialisé et puissant pour des structures de données **partitionnées dans l'espace** et **naturellement non-équilibrées**. Bien que moins universels que B-Tree, ils excellent dans des niches spécifiques.

### Points clés à retenir

1. **SP-GiST = Structures non-équilibrées** : Radix trees, quadtrees, k-d trees  
2. **Trois domaines principaux** : Préfixes texte, réseaux IP, points 2D  
3. **Partitionnement strict** : Pas de chevauchement (contrairement à GiST)  
4. **Performance prévisible** : Pas de rééquilibrage coûteux  
5. **Spécialisé mais efficace** : Excellent pour cas d'usage ciblés

### Cas d'usage idéaux

- 🔍 **Autocomplétion** : Recherche de préfixes (emails, noms, termes)  
- 🌐 **Routage réseau** : CIDR matching, géolocalisation IP  
- 📍 **Données spatiales** : Points uniformément distribués (quadtree)  
- 📅 **Ranges temporels** : Réservations, disponibilités  
- 🔑 **Clés hiérarchiques** : Cache structuré, chemins

### Quand utiliser SP-GiST ?

```
Avez-vous des données avec une structure hiérarchique naturelle ?
├─ NON → B-Tree, GiST ou GIN selon le cas
└─ OUI → SP-GiST peut être optimal

Type de hiérarchie :
├─ Préfixes de texte → SP-GiST excellent
├─ Réseaux IP/CIDR → SP-GiST excellent
├─ Points 2D uniformes → SP-GiST ou GiST
├─ Géométries complexes → GiST préférable
└─ Ranges hiérarchiques → SP-GiST bon

→ Toujours comparer avec B-Tree et GiST via EXPLAIN ANALYZE
```

Avec SP-GiST, PostgreSQL offre une solution élégante pour des structures de données qui ne s'accommodent pas bien des arbres équilibrés traditionnels. C'est un outil de niche, mais redoutablement efficace quand le cas d'usage correspond ! 🌳
