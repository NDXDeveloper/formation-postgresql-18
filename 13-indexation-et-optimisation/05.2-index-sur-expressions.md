# 13.5.2. Index sur Expressions (CREATE INDEX ON table((lower(col))))

## Introduction

Un **index sur expression** (également appelé **index fonctionnel**) est un index qui porte non pas directement sur une colonne, mais sur le **résultat d'une expression ou d'une fonction** appliquée à une ou plusieurs colonnes.

Contrairement à un index classique qui indexe les valeurs brutes d'une colonne, un index sur expression indexe les valeurs **transformées** par une fonction ou une expression.

Cette technique est particulièrement utile lorsque vos requêtes filtrent ou trient systématiquement sur des transformations de colonnes.

---

## Concept Fondamental

### Index Classique vs Index sur Expression

**Index classique :**
```sql
CREATE INDEX idx_email ON utilisateurs(email);
```
→ Indexe les valeurs brutes : `'John@Example.COM'`, `'alice@test.com'`

**Problème :** Cette requête ne peut pas utiliser l'index :
```sql
SELECT * FROM utilisateurs WHERE LOWER(email) = 'john@example.com';
```
→ PostgreSQL doit calculer `LOWER(email)` pour chaque ligne (Seq Scan)

**Index sur expression :**
```sql
CREATE INDEX idx_email_lower ON utilisateurs(LOWER(email));
```
→ Indexe les valeurs transformées : `'john@example.com'`, `'alice@test.com'`

**Maintenant, cette requête utilise l'index :**
```sql
SELECT * FROM utilisateurs WHERE LOWER(email) = 'john@example.com';
```
→ PostgreSQL utilise directement l'index sur `LOWER(email)` (Index Scan)

---

## Pourquoi Utiliser des Index sur Expressions ?

### 1. **Recherches Insensibles à la Casse**

Sans index sur expression :
```sql
-- ❌ Scan séquentiel (lent sur grandes tables)
SELECT * FROM utilisateurs WHERE LOWER(email) = 'john@example.com';
```

Avec index sur expression :
```sql
CREATE INDEX idx_email_lower ON utilisateurs(LOWER(email));

-- ✅ Utilise l'index (rapide)
SELECT * FROM utilisateurs WHERE LOWER(email) = 'john@example.com';
```

### 2. **Recherches sur Données Normalisées**

Exemple : Recherche sur numéros de téléphone sans espaces ni tirets
```sql
-- Créer un index sur la version nettoyée
CREATE INDEX idx_tel_clean ON clients(REPLACE(REPLACE(telephone, ' ', ''), '-', ''));

-- Requête optimisée
SELECT * FROM clients  
WHERE REPLACE(REPLACE(telephone, ' ', ''), '-', '') = '0612345678';  
```

### 3. **Calculs Fréquents**

Exemple : Indexer une colonne JSON extraite
```sql
CREATE INDEX idx_user_name ON commandes((data->>'user_name'));

-- Requête rapide sur un champ JSON
SELECT * FROM commandes WHERE data->>'user_name' = 'John Doe';
```

### 4. **Tri sur Expressions**

```sql
CREATE INDEX idx_nom_prenom ON personnes((nom || ' ' || prenom));

-- Tri optimisé
SELECT * FROM personnes ORDER BY nom || ' ' || prenom;
```

## Syntaxe et Utilisation

### Syntaxe Générale

```sql
CREATE INDEX nom_index ON nom_table((expression));
```

**Note importante :** Les **doubles parenthèses** `(( ))` sont **obligatoires** pour les expressions !

### Exemples de Syntaxes

```sql
-- ✅ Correct : doubles parenthèses
CREATE INDEX idx_lower_email ON users((LOWER(email)));

-- ❌ Incorrect : parenthèses simples
CREATE INDEX idx_lower_email ON users(LOWER(email));
-- Erreur : syntax error
```

### Expressions Supportées

PostgreSQL supporte pratiquement toutes les expressions **immutables** :

1. **Fonctions de transformation de texte**
   ```sql
   LOWER(colonne)
   UPPER(colonne)
   TRIM(colonne)
   SUBSTRING(colonne, 1, 10)
   ```

2. **Opérations mathématiques**
   ```sql
   (prix * 1.20)  -- Prix TTC
   (longueur * largeur)  -- Surface
   ABS(valeur)
   ROUND(montant, 2)
   ```

3. **Extraction de données**
   ```sql
   DATE(timestamp_col)
   EXTRACT(YEAR FROM date_col)
   data->>'field'  -- JSON
   ```

4. **Concaténation**
   ```sql
   (nom || ' ' || prenom)
   CONCAT(code_pays, '-', code_postal)
   ```

5. **Expressions conditionnelles simples**
   ```sql
   CASE WHEN statut = 'actif' THEN 1 ELSE 0 END
   COALESCE(colonne, 'défaut')
   ```

## Exemples Pratiques

### Exemple 1 : Recherche Insensible à la Casse

**Contexte :** Table d'utilisateurs avec emails en casse mixte.

```sql
-- Table utilisateurs
CREATE TABLE utilisateurs (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100),
    email VARCHAR(255) UNIQUE,
    date_inscription TIMESTAMP DEFAULT NOW()
);

-- Données exemples
INSERT INTO utilisateurs (nom, email) VALUES
    ('John Doe', 'John.Doe@Example.COM'),
    ('Alice Smith', 'alice.smith@test.com'),
    ('Bob Martin', 'BOB.MARTIN@COMPANY.FR');
```

**Sans index sur expression :**
```sql
-- ❌ Scan séquentiel
EXPLAIN ANALYZE  
SELECT * FROM utilisateurs WHERE LOWER(email) = 'john.doe@example.com';  

-- Résultat :
-- Seq Scan on utilisateurs  (cost=0.00..25.88 rows=6 width=...)
--   Filter: (lower((email)::text) = 'john.doe@example.com'::text)
```

**Avec index sur expression :**
```sql
-- Créer l'index
CREATE INDEX idx_email_lower ON utilisateurs(LOWER(email));

-- ✅ Index Scan
EXPLAIN ANALYZE  
SELECT * FROM utilisateurs WHERE LOWER(email) = 'john.doe@example.com';  

-- Résultat :
-- Index Scan using idx_email_lower on utilisateurs  (cost=0.15..8.17 rows=1 width=...)
--   Index Cond: (lower((email)::text) = 'john.doe@example.com'::text)
```

**Gain de performance :** 3× à 100× plus rapide selon la taille de la table.

### Exemple 2 : Extraction de Date depuis un Timestamp

**Contexte :** Requêtes fréquentes groupées par jour.

```sql
-- Table de logs
CREATE TABLE logs (
    id BIGSERIAL PRIMARY KEY,
    message TEXT,
    niveau VARCHAR(20),
    created_at TIMESTAMP DEFAULT NOW()
);

-- Requête typique : logs d'une journée spécifique
SELECT * FROM logs  
WHERE DATE(created_at) = '2025-11-20';  
```

**Sans index :**
```sql
-- ❌ Scan séquentiel (calcule DATE() pour chaque ligne)
EXPLAIN ANALYZE  
SELECT * FROM logs WHERE DATE(created_at) = '2025-11-20';  
```

**Avec index sur expression :**
```sql
-- Créer l'index sur DATE(created_at)
CREATE INDEX idx_logs_date ON logs((DATE(created_at)));

-- ✅ Index Scan
EXPLAIN ANALYZE  
SELECT * FROM logs WHERE DATE(created_at) = '2025-11-20';  

-- Résultat :
-- Index Scan using idx_logs_date on logs
--   Index Cond: (date(created_at) = '2025-11-20'::date)
```

**Alternative plus performante (sans fonction) :**
```sql
-- Meilleure approche : utiliser un range sans fonction
SELECT * FROM logs  
WHERE created_at >= '2025-11-20'::date  
AND created_at < '2025-11-21'::date;  

-- Index classique sur created_at suffit
CREATE INDEX idx_logs_timestamp ON logs(created_at);
```

### Exemple 3 : Recherche sur Colonnes JSON

**Contexte :** Table avec données JSON, recherche fréquente sur un champ.

```sql
-- Table de commandes avec métadonnées JSON
CREATE TABLE commandes (
    id SERIAL PRIMARY KEY,
    client_id INTEGER,
    data JSONB,
    created_at TIMESTAMP DEFAULT NOW()
);

-- Exemples de données
INSERT INTO commandes (client_id, data) VALUES
    (1, '{"user_name": "John Doe", "country": "FR", "total": 150.50}'),
    (2, '{"user_name": "Alice Smith", "country": "US", "total": 200.00}');
```

**Sans index :**
```sql
-- ❌ Scan séquentiel
SELECT * FROM commandes WHERE data->>'user_name' = 'John Doe';
```

**Avec index sur expression :**
```sql
-- Créer index sur l'extraction du champ user_name
CREATE INDEX idx_data_user_name ON commandes((data->>'user_name'));

-- ✅ Index Scan
SELECT * FROM commandes WHERE data->>'user_name' = 'John Doe';
```

**Alternative pour recherches multiples sur JSON :**
```sql
-- Index GIN pour toutes les clés/valeurs
CREATE INDEX idx_data_gin ON commandes USING gin(data);

-- Permet des recherches plus flexibles
SELECT * FROM commandes WHERE data @> '{"country": "FR"}';
```

### Exemple 4 : Normalisation de Texte

**Contexte :** Recherche sur noms avec accents et casse.

```sql
-- Table de produits
CREATE TABLE produits (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    prix NUMERIC(10,2)
);

-- Fonction de normalisation (supprimer accents + minuscules)
CREATE EXTENSION IF NOT EXISTS unaccent;

-- Index sur texte normalisé
CREATE INDEX idx_nom_normalized ON produits(
    (LOWER(unaccent(nom)))
);

-- Recherche normalisée
SELECT * FROM produits  
WHERE LOWER(unaccent(nom)) = LOWER(unaccent('Café Expresso'));  
-- Trouvera : "Café Expresso", "cafe expresso", "CAFÉ EXPRESSO"
```

### Exemple 5 : Calculs Mathématiques

**Contexte :** Recherche sur surface calculée.

```sql
-- Table de biens immobiliers
CREATE TABLE biens (
    id SERIAL PRIMARY KEY,
    adresse TEXT,
    longueur NUMERIC(6,2),  -- en mètres
    largeur NUMERIC(6,2),   -- en mètres
    prix NUMERIC(10,2)
);

-- Index sur surface calculée
CREATE INDEX idx_surface ON biens((longueur * largeur));

-- Requête : trouver biens entre 80 et 120 m²
SELECT * FROM biens  
WHERE (longueur * largeur) BETWEEN 80 AND 120  
ORDER BY longueur * largeur;  
```

**Alternative : Colonne calculée (PostgreSQL 12+) :**
```sql
-- Créer une colonne générée stockée
ALTER TABLE biens ADD COLUMN surface NUMERIC(10,2)  
GENERATED ALWAYS AS (longueur * largeur) STORED;  

-- Index classique sur colonne générée
CREATE INDEX idx_surface_generated ON biens(surface);

-- Requête simplifiée
SELECT * FROM biens  
WHERE surface BETWEEN 80 AND 120  
ORDER BY surface;  
```

### Exemple 6 : Tri Personnalisé

**Contexte :** Tri alphabétique sur nom complet.

```sql
-- Table de contacts
CREATE TABLE contacts (
    id SERIAL PRIMARY KEY,
    prenom VARCHAR(50),
    nom VARCHAR(50),
    email VARCHAR(255)
);

-- Index sur nom complet (nom + prénom)
CREATE INDEX idx_nom_complet ON contacts(
    (nom || ' ' || prenom)
);

-- Tri optimisé
SELECT * FROM contacts  
ORDER BY nom || ' ' || prenom;  
```

## Cas d'Usage Typiques

### 1. **Recherches Insensibles à la Casse**

```sql
CREATE INDEX idx_lower ON table((LOWER(colonne)));
```

**Scénario :** Emails, usernames, codes, références.

### 2. **Extraction de Composants de Dates**

```sql
CREATE INDEX idx_year ON events((EXTRACT(YEAR FROM date_col)));  
CREATE INDEX idx_month ON events((EXTRACT(MONTH FROM date_col)));  
```

**Scénario :** Rapports mensuels, statistiques annuelles.

### 3. **Recherches sur Données JSON**

```sql
CREATE INDEX idx_json_field ON table((data->>'field_name'));
```

**Scénario :** Métadonnées, configurations, données semi-structurées.

### 4. **Normalisation de Chaînes**

```sql
CREATE INDEX idx_clean_text ON table(
    (TRIM(LOWER(REPLACE(colonne, '-', ''))))
);
```

**Scénario :** Codes produits, références, numéros de série.

### 5. **Calculs Métier**

```sql
CREATE INDEX idx_prix_ttc ON produits((prix_ht * 1.20));  
CREATE INDEX idx_age ON personnes((EXTRACT(YEAR FROM age(date_naissance))));  
```

**Scénario :** Prix TTC, âge calculé, scores.

## Règles et Contraintes

### 1. **Immutabilité Requise**

PostgreSQL n'accepte que les fonctions **IMMUTABLE** dans les index sur expressions.

**Qu'est-ce qu'une fonction IMMUTABLE ?**
Une fonction qui retourne **toujours le même résultat** pour les mêmes arguments.

```sql
-- ✅ IMMUTABLE : Accepté
LOWER('ABC')  → toujours 'abc'  
ABS(-5)       → toujours 5  
DATE('2025-11-20 15:30:00')  → toujours '2025-11-20'  

-- ❌ VOLATILE : Refusé
NOW()          → change à chaque appel  
RANDOM()       → change à chaque appel  
CURRVAL()      → dépend de l'état  
```

**Exemple d'erreur :**
```sql
-- ❌ Erreur : NOW() n'est pas IMMUTABLE
CREATE INDEX idx_recent ON logs((created_at > NOW() - INTERVAL '7 days'));

-- Erreur :
-- ERROR: functions in index expression must be marked IMMUTABLE
```

### 2. **Correspondance Exacte Requise**

Pour que l'index soit utilisé, la requête doit utiliser **exactement la même expression**.

```sql
-- Index créé
CREATE INDEX idx_lower_email ON users((LOWER(email)));

-- ✅ Utilise l'index
SELECT * FROM users WHERE LOWER(email) = 'test@example.com';

-- ❌ N'utilise PAS l'index (expression différente)
SELECT * FROM users WHERE UPPER(email) = 'TEST@EXAMPLE.COM';
```

### 3. **Performance de l'Expression**

L'expression est calculée à chaque INSERT/UPDATE. Évitez les expressions coûteuses.

```sql
-- ❌ MAUVAIS : Expression très coûteuse
CREATE INDEX idx_expensive ON table(
    (
        SELECT COUNT(*) FROM autre_table
        WHERE autre_table.ref = table.id
    )
);

-- ✅ BON : Expression simple et rapide
CREATE INDEX idx_simple ON table((LOWER(colonne)));
```

### 4. **Taille de l'Index**

Les index sur expressions peuvent être plus volumineux que les index classiques.

```sql
-- Comparer les tailles
SELECT
    pg_size_pretty(pg_relation_size('idx_email')) AS "Index classique",
    pg_size_pretty(pg_relation_size('idx_email_lower')) AS "Index expression";
```

**Généralement :** La différence est négligeable (< 10%).

```sql
-- Vérifier si l'index est utilisé
EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM utilisateurs WHERE LOWER(email) = 'test@example.com';  
```

**Sortie attendue avec l'index :**
```
Index Scan using idx_email_lower on utilisateurs

```

**Sortie sans l'index (problème) :**
```
Seq Scan on utilisateurs
  Filter: (lower((email)::text) = 'test@example.com'::text)

```

```sql
-- Vérifier l'utilisation d'un index
```

## Pièges et Limitations

### ❌ Piège 1 : Expression Non-Correspondante

```sql
-- Index créé
CREATE INDEX idx_lower ON users((LOWER(email)));

-- ❌ N'utilise PAS l'index (UPPER au lieu de LOWER)
SELECT * FROM users WHERE UPPER(email) = 'TEST@EXAMPLE.COM';
```

**Solution :** Utiliser la **même expression** que dans l'index.

### ❌ Piège 2 : Fonctions Non-Immutables

```sql
-- ❌ Erreur : NOW() est VOLATILE
CREATE INDEX idx_recent ON logs((NOW() - created_at));

-- Erreur : functions in index expression must be marked IMMUTABLE
```

**Solution :** Utiliser des fonctions IMMUTABLE uniquement.

### ❌ Piège 3 : Expression Complexe et Coûteuse

```sql
-- ❌ Très coûteux à maintenir
CREATE INDEX idx_expensive ON products(
    (
        (LOWER(TRIM(REGEXP_REPLACE(name, '[^a-zA-Z0-9]', '', 'g'))))
    )
);
```

**Solution :** Simplifier l'expression ou utiliser une colonne générée.

### ❌ Piège 4 : Oublier les Doubles Parenthèses

```sql
-- ❌ Syntax Error
CREATE INDEX idx_lower ON users(LOWER(email));

-- ✅ Correct
CREATE INDEX idx_lower ON users((LOWER(email)));
```

### ❌ Piège 5 : Type de Données Différent

```sql
-- Index créé sur INTEGER
CREATE INDEX idx_str_id ON table((id::TEXT));

-- ❌ N'utilise pas l'index (VARCHAR au lieu de TEXT)
SELECT * FROM table WHERE id::VARCHAR = '123';

-- ✅ Utilise l'index (même type : TEXT)
SELECT * FROM table WHERE id::TEXT = '123';
```

## Alternatives aux Index sur Expressions

### Alternative 1 : Colonnes Générées (PostgreSQL 12+)

**Avantage :** Syntaxe de requête simplifiée.

```sql
-- Créer une colonne générée
ALTER TABLE users  
ADD COLUMN email_lower TEXT  
GENERATED ALWAYS AS (LOWER(email)) STORED;  

-- Index classique sur la colonne générée
CREATE INDEX idx_email_lower ON users(email_lower);

-- Requête simplifiée
SELECT * FROM users WHERE email_lower = 'test@example.com';
```

**Comparaison :**

| Critère | Index sur Expression | Colonne Générée + Index |
|---------|----------------------|-------------------------|
| Espace disque | Index uniquement | Index + Colonne stockée |
| Requête | `WHERE LOWER(col) = ...` | `WHERE col_lower = ...` |
| Maintenance | Automatique | Automatique |
| Lisibilité | Moins claire | Plus claire |

### Alternative 2 : Normalisation en Amont

```sql
-- Normaliser à l'insertion
INSERT INTO users (email)  
VALUES (LOWER('John@Example.COM'));  -- Stocke déjà en minuscules  

-- Index classique suffit
CREATE INDEX idx_email ON users(email);

-- Requête simple
SELECT * FROM users WHERE email = 'john@example.com';
```

**Avantage :** Simplicité maximale.

**Inconvénient :** Nécessite de normaliser dans l'application.

### Alternative 3 : Index de Texte Complet (Full-Text Search)

Pour recherches textuelles avancées :

```sql
-- Index GIN pour full-text search
CREATE INDEX idx_fts ON articles USING gin(to_tsvector('french', contenu));

-- Recherche avec opérateur @@
SELECT * FROM articles  
WHERE to_tsvector('french', contenu) @@ to_tsquery('french', 'PostgreSQL');  
```

## Index sur Expressions avec Types d'Index Spécialisés

### Avec GIN (Texte, JSON, Arrays)

```sql
-- Index GIN sur expression JSON
CREATE INDEX idx_tags ON posts USING gin((data->'tags'));

-- Recherche dans array JSON
SELECT * FROM posts WHERE data->'tags' ? 'postgresql';
```

### Avec GiST (Géométrie, Texte)

```sql
-- Index GiST sur texte normalisé
CREATE INDEX idx_text_gist ON documents  
USING gist((to_tsvector('english', contenu)));  
```

### Avec BRIN (Données Séquentielles)

```sql
-- Index BRIN sur date extraite
CREATE INDEX idx_year_brin ON logs  
USING brin((EXTRACT(YEAR FROM created_at)));  
```

## Combinaison : Index Partiel + Expression

Vous pouvez combiner index partiel et expression pour une optimisation maximale !

```sql
-- Index sur expression + condition partielle
CREATE INDEX idx_email_lower_actifs ON utilisateurs(
    (LOWER(email))
)
WHERE est_actif = TRUE;
```

**Avantages combinés :**
- Expression : Recherche insensible à la casse
- Partiel : Indexe uniquement les utilisateurs actifs

**Requête optimisée :**
```sql
-- ✅ Utilise l'index combiné
SELECT * FROM utilisateurs  
WHERE LOWER(email) = 'test@example.com'  
AND est_actif = TRUE;  
```

## Exemple Complet : Optimisation d'une Table de Produits

### Contexte

```sql
-- Table de produits e-commerce
CREATE TABLE produits (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    reference VARCHAR(50),
    prix_ht NUMERIC(10,2),
    categorie VARCHAR(100),
    date_ajout TIMESTAMP DEFAULT NOW()
);

-- 1 million de produits
-- Requêtes fréquentes :
-- 1. Recherche insensible à la casse sur nom
-- 2. Recherche sur prix TTC (TVA 20%)
-- 3. Recherche sur année d'ajout
```

### Stratégie d'Indexation

```sql
-- Index 1 : Nom en minuscules
CREATE INDEX idx_nom_lower ON produits((LOWER(nom)));

-- Index 2 : Prix TTC calculé
CREATE INDEX idx_prix_ttc ON produits((prix_ht * 1.20));

-- Index 3 : Année d'ajout
CREATE INDEX idx_annee ON produits((EXTRACT(YEAR FROM date_ajout)));

-- Index 4 : Référence nettoyée (sans tirets ni espaces)
CREATE INDEX idx_ref_clean ON produits(
    (REPLACE(REPLACE(LOWER(reference), '-', ''), ' ', ''))
);
```

### Requêtes Optimisées

```sql
-- Requête 1 : Recherche produit par nom
-- ✅ Utilise idx_nom_lower
EXPLAIN ANALYZE  
SELECT * FROM produits  
WHERE LOWER(nom) = 'iphone 15 pro';  

-- Requête 2 : Produits dans une fourchette de prix TTC
-- ✅ Utilise idx_prix_ttc
EXPLAIN ANALYZE  
SELECT * FROM produits  
WHERE (prix_ht * 1.20) BETWEEN 500 AND 1000  
ORDER BY prix_ht * 1.20;  

-- Requête 3 : Produits ajoutés en 2024
-- ✅ Utilise idx_annee
EXPLAIN ANALYZE  
SELECT * FROM produits  
WHERE EXTRACT(YEAR FROM date_ajout) = 2024;  

-- Requête 4 : Recherche par référence nettoyée
-- ✅ Utilise idx_ref_clean
EXPLAIN ANALYZE  
SELECT * FROM produits  
WHERE REPLACE(REPLACE(LOWER(reference), '-', ''), ' ', '') = 'ref123456';  
```

### Gains de Performance

**Avant optimisation :**
- Requête 1 : 2500 ms (Seq Scan)
- Requête 2 : 3200 ms (Seq Scan)
- Requête 3 : 1800 ms (Seq Scan)

**Après optimisation :**
- Requête 1 : 12 ms (Index Scan) → **208× plus rapide**
- Requête 2 : 25 ms (Index Scan) → **128× plus rapide**
- Requête 3 : 8 ms (Index Scan) → **225× plus rapide**

## Maintenance et Monitoring

### Lister les Index sur Expressions

```sql
SELECT
    schemaname,
    tablename,
    indexname,
    indexdef
FROM pg_indexes  
WHERE indexdef LIKE '%(%(%'  -- Détecte les doubles parenthèses  
ORDER BY tablename, indexname;  
```

### Analyser l'Utilisation

```sql
SELECT
    schemaname,
    tablename,
    indexname,
    idx_scan AS "Nb utilisations",
    pg_size_pretty(pg_relation_size(indexrelid)) AS "Taille",
    indexdef
FROM pg_stat_user_indexes  
JOIN pg_indexes USING (schemaname, tablename, indexname)  
WHERE indexdef LIKE '%(%(%'  
ORDER BY idx_scan DESC;  
```

### Identifier les Index Inutilisés

```sql
-- Index sur expressions jamais utilisés
SELECT
    schemaname,
    tablename,
    indexname,
    pg_size_pretty(pg_relation_size(indexrelid)) AS "Taille gaspillée"
FROM pg_stat_user_indexes  
JOIN pg_indexes USING (schemaname, tablename, indexname)  
WHERE indexdef LIKE '%(%(%'  
AND idx_scan = 0  
ORDER BY pg_relation_size(indexrelid) DESC;  
```

## Bonnes Pratiques

### ✅ DO (À Faire)

1. **Utiliser des fonctions IMMUTABLE uniquement**
   ```sql
   -- ✅ IMMUTABLE
   CREATE INDEX idx ON table((LOWER(col)));
   CREATE INDEX idx ON table((DATE(timestamp_col)));
   ```

2. **Garder les expressions simples**
   ```sql
   -- ✅ Simple et performant
   CREATE INDEX idx ON table((LOWER(email)));

   -- ❌ Complexe et coûteux
   CREATE INDEX idx ON table((LOWER(TRIM(REGEXP_REPLACE(email, '...', '')))));
   ```

3. **Documenter les index sur expressions**
   ```sql
   COMMENT ON INDEX idx_email_lower IS
   'Index sur email en minuscules pour recherches insensibles à la casse';
   ```

4. **Tester avec EXPLAIN avant de créer**
   ```sql
   -- Sans index
   EXPLAIN SELECT * FROM table WHERE LOWER(col) = 'value';

   -- Créer l'index
   CREATE INDEX ...

   -- Vérifier l'utilisation
   EXPLAIN SELECT * FROM table WHERE LOWER(col) = 'value';
   ```

5. **Préférer les colonnes générées pour les cas complexes**
   ```sql
   -- Au lieu d'un index sur expression complexe
   ALTER TABLE table ADD COLUMN computed_col
   GENERATED ALWAYS AS (expression_complexe) STORED;

   CREATE INDEX idx ON table(computed_col);
   ```

### ❌ DON'T (À Éviter)

1. **Ne pas utiliser de fonctions VOLATILE ou STABLE**
   ```sql
   -- ❌ Erreur
   CREATE INDEX idx ON table((NOW() - created_at));
   ```

2. **Ne pas oublier les doubles parenthèses**
   ```sql
   -- ❌ Syntax Error
   CREATE INDEX idx ON table(LOWER(col));

   -- ✅ Correct
   CREATE INDEX idx ON table((LOWER(col)));
   ```

3. **Ne pas créer d'index sur expressions coûteuses**
   ```sql
   -- ❌ Trop coûteux à maintenir
   CREATE INDEX idx ON table((fonction_tres_complexe(col)));
   ```

4. **Ne pas utiliser des expressions différentes dans les requêtes**
   ```sql
   CREATE INDEX idx ON table((LOWER(col)));

   -- ❌ N'utilise pas l'index
   SELECT * FROM table WHERE UPPER(col) = 'VALUE';
   ```

## Comparaison des Approches

| Approche | Avantages | Inconvénients | Quand l'utiliser |
|----------|-----------|---------------|------------------|
| **Index sur Expression** | - Transparent<br>- Pas de colonne supplémentaire | - Expression obligatoire dans requêtes<br>- Peut être moins lisible | Transformations ponctuelles |
| **Colonne Générée** | - Requêtes simples<br>- Plus lisible | - Espace disque (colonne stockée)<br>- PG 12+ uniquement | Calculs fréquents, lisibilité importante |
| **Normalisation Applicative** | - Simplicité maximale<br>- Index classique suffit | - Logique dans application<br>- Risque d'incohérence | Données contrôlées à la source |
| **Colonne Dénormalisée Manuelle** | - Contrôle total<br>- Rétrocompatible | - Maintenance manuelle (triggers)<br>- Risque de désynchronisation | Applications legacy |

---

## Résumé

### Points Clés

1. **Un index sur expression indexe le résultat d'une fonction ou expression, pas la valeur brute**

2. **Syntaxe : Doubles parenthèses obligatoires**
   ```sql
   CREATE INDEX idx ON table((expression));
   ```

3. **Condition d'utilisation : La requête doit utiliser la même expression**

4. **Restrictions :**
   - Fonctions IMMUTABLE uniquement
   - Expressions simples recommandées

5. **Cas d'usage principaux :**
   - Recherches insensibles à la casse (LOWER, UPPER)
   - Extraction de composants (DATE, EXTRACT)
   - Normalisation de texte (TRIM, REPLACE)
   - Calculs métier (prix TTC, surface)
   - Extraction JSON (data->>'field')

6. **Alternatives :**
   - Colonnes générées (PG 12+)
   - Normalisation applicative
   - Index spécialisés (GIN, GiST)
