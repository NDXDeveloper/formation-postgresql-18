# 13.5.4. Index sur Colonnes JSONB (GIN avec jsonb_path_ops)

## Introduction

PostgreSQL offre un support natif et puissant pour les données **JSON** et **JSONB** (JSON Binary). Le type JSONB permet de stocker des données semi-structurées tout en bénéficiant de capacités d'indexation avancées grâce aux **index GIN** (Generalized Inverted Index).

Ce chapitre explore comment indexer efficacement les colonnes JSONB, notamment avec l'opérateur de classe **jsonb_path_ops**, pour optimiser les requêtes sur des données JSON complexes.

---

## JSON vs JSONB : Rappel Essentiel

### Différences Fondamentales

```sql
-- JSON : Stockage textuel (comme du VARCHAR)
CREATE TABLE logs_json (
    id SERIAL PRIMARY KEY,
    data JSON
);

-- JSONB : Stockage binaire (décomposé et optimisé)
CREATE TABLE logs_jsonb (
    id SERIAL PRIMARY KEY,
    data JSONB
);
```

| Critère | JSON | JSONB |
|---------|------|-------|
| **Stockage** | Texte brut | Format binaire décomposé |
| **Performance lecture** | Lente (parsing à chaque fois) | Rapide (déjà parsé) |
| **Performance écriture** | Rapide (pas de traitement) | Plus lente (décomposition) |
| **Indexation** | Limitée | Complète (GIN, GiST) |
| **Ordre des clés** | Préservé | Non préservé |
| **Espaces blancs** | Préservés | Supprimés |
| **Doublons de clés** | Permis | Dernière valeur conservée |

**Recommandation :** Utilisez **JSONB** sauf si vous devez absolument préserver l'ordre exact du JSON d'origine.

### Exemple de Données JSONB

```sql
-- Table de produits avec métadonnées JSON
CREATE TABLE produits (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    prix NUMERIC(10,2),

);

-- Insérer des données

            "cpu": "Intel i7",
```

## Pourquoi Indexer les Colonnes JSONB ?

### Sans Index : Problème de Performance

```sql
-- Recherche sans index : Scan séquentiel
SELECT * FROM produits  
WHERE metadata->>'marque' = 'TechCorp';  

-- EXPLAIN ANALYZE :
-- Seq Scan on produits  (cost=0.00..1250.00 rows=50 width=...)
--   Filter: ((metadata ->> 'marque'::text) = 'TechCorp'::text)
--   Rows Removed by Filter: 99950
-- Planning Time: 0.123 ms
-- Execution Time: 2847.456 ms  ← TRÈS LENT sur 100K lignes !
```

**Problème :** PostgreSQL doit :
1. Parcourir **toutes les lignes** (Seq Scan)  
2. Décoder chaque colonne JSONB  
3. Extraire la clé `marque`  
4. Comparer la valeur

### Avec Index : Performance Optimale

```sql
-- Créer un index GIN
CREATE INDEX idx_metadata_gin ON produits USING gin(metadata);

-- Même requête
SELECT * FROM produits  
WHERE metadata @> '{"marque": "TechCorp"}';  

-- EXPLAIN ANALYZE :
-- Bitmap Index Scan on idx_metadata_gin  (cost=4.43..104.43 rows=50 width=...)
--   Index Cond: (metadata @> '{"marque": "TechCorp"}'::jsonb)
-- Planning Time: 0.098 ms
-- Execution Time: 12.345 ms  ← 230× PLUS RAPIDE !
```

## Opérateurs JSONB Essentiels

Avant d'aborder l'indexation, comprenons les opérateurs JSONB :

### 1. Opérateurs d'Extraction

```sql
-- -> : Obtenir un objet JSON (résultat : JSONB)
SELECT metadata->'specs' FROM produits;
-- Résultat : {"cpu": "Intel i7", "ram": "16GB", ...}

-- ->> : Obtenir une valeur texte (résultat : TEXT)
SELECT metadata->>'marque' FROM produits;
-- Résultat : TechCorp

-- #> : Chemin dans l'objet (résultat : JSONB)
SELECT metadata#>'{specs, cpu}' FROM produits;
-- Résultat : "Intel i7"

-- #>> : Chemin dans l'objet (résultat : TEXT)
SELECT metadata#>>'{specs, cpu}' FROM produits;
-- Résultat : Intel i7
```

### 2. Opérateurs de Recherche (Indexables avec GIN)

```sql
-- @> : Contient (l'objet à gauche contient celui de droite)
SELECT * FROM produits  
WHERE metadata @> '{"marque": "TechCorp"}';  

-- <@ : Est contenu dans (inverse de @>)
SELECT * FROM produits  
WHERE '{"marque": "TechCorp"}' <@ metadata;  

-- ? : Clé existe
SELECT * FROM produits  
WHERE metadata ? 'marque';  

-- ?| : Au moins une des clés existe (OR)
SELECT * FROM produits  
WHERE metadata ?| ARRAY['marque', 'fabricant'];  

-- ?& : Toutes les clés existent (AND)
SELECT * FROM produits  
WHERE metadata ?& ARRAY['marque', 'specs', 'tags'];  

-- @? : teste si un jsonpath retourne AU MOINS un élément (PostgreSQL 12+)
SELECT * FROM produits  
WHERE metadata @? '$.specs ? (@.ram == "16GB")';  -- filtre : éléments où ram = "16GB"

-- @@ : évalue un prédicat jsonpath booléen (PostgreSQL 12+)
SELECT * FROM produits  
WHERE metadata @@ '$.specs.ram == "16GB"';  -- prédicat booléen
-- ⚠️ Piège fréquent : « metadata @? '$.specs.ram == "16GB"' » renverrait TOUTES
-- les lignes ayant le chemin specs.ram, car @? vérifie seulement que le chemin
-- produit un élément (ici le booléen résultat de la comparaison), sans tester sa
-- valeur. Pour filtrer sur la valeur : utilisez @@ (prédicat) ou @? avec un
-- filtre « ? (@ ...) ».
```

**Important :** Les opérateurs `@>`, `?`, `?|`, `?&` sont **optimisés** par les index GIN.

## Index GIN : Concept

### Qu'est-ce qu'un Index GIN ?

**GIN** (Generalized Inverted Index) est un index inversé, similaire à un index de livre :

**Analogie avec un index de livre :**
- **Livre normal :** Page 1 → Contenu, Page 2 → Contenu, etc.  
- **Index de livre :** "PostgreSQL" → Pages 12, 45, 78, 134

**GIN pour JSONB :**
- Indexe **toutes les clés et valeurs** du JSON
- Crée une correspondance : `clé/valeur → lignes contenant cette clé/valeur`

### Structure Interne Simplifiée

```
Index GIN sur metadata :

"marque" -> "TechCorp"   → lignes [1, 45, 123, 456]
"marque" -> "GamerGear"  → lignes [2, 67, 234, 789]
"specs" -> "cpu"         → lignes [1, 3, 5, 7, ...]
"tags" -> "gaming"       → lignes [2, 4, 8, 12, ...]
"disponible" -> true     → lignes [1, 2, 3, 4, ...]
...
```

**Avantage :** Recherche ultra-rapide par clé/valeur.

**Inconvénient :** Index plus volumineux (indexe toutes les clés et valeurs).

---

## Types d'Index GIN pour JSONB

PostgreSQL offre **deux classes d'opérateurs** pour les index GIN sur JSONB :

### 1. GIN Standard (jsonb_ops) - Par Défaut

```sql
CREATE INDEX idx_metadata_gin ON produits USING gin(metadata);
-- Équivalent à :
CREATE INDEX idx_metadata_gin ON produits USING gin(metadata jsonb_ops);
```

**Caractéristiques :**
- Indexe **toutes les clés et valeurs** récursivement
- Supporte **tous les opérateurs JSONB**
- Index plus volumineux
- Plus flexible

**Opérateurs supportés :**
- `@>` (contient)  
- `?` (clé existe)  
- `?|` (au moins une clé existe)  
- `?&` (toutes les clés existent)  
- `@?` (JSONPath)

### 2. GIN Optimisé (jsonb_path_ops)

```sql
CREATE INDEX idx_metadata_path_ops ON produits USING gin(metadata jsonb_path_ops);
```

**Caractéristiques :**
- Indexe uniquement les **chemins complets clé→valeur**
- Supporte **uniquement l'opérateur @>**
- Index **3× plus compact**
- **Plus rapide** pour les requêtes `@>`  
- **Moins flexible** (pas de `?`, `?|`, `?&`)

**Opérateur supporté :**
- `@>` (contient) uniquement

---

## GIN Standard (jsonb_ops) : Détails

### Quand l'Utiliser ?

Utilisez GIN standard si vous avez besoin de :
- Vérifier l'**existence de clés** (`?`, `?|`, `?&`)
- Requêtes **variées** sur le JSON
- Flexibilité maximale

### Exemples d'Utilisation

```sql
-- Créer l'index
CREATE INDEX idx_metadata_gin ON produits USING gin(metadata);
```

**Requêtes supportées :**

```sql
-- 1. Contient (@>)
SELECT * FROM produits  
WHERE metadata @> '{"marque": "TechCorp"}';  

-- 2. Clé existe (?)
SELECT * FROM produits  
WHERE metadata ? 'marque';  

-- 3. Au moins une clé existe (?|)
SELECT * FROM produits  
WHERE metadata ?| ARRAY['marque', 'fabricant', 'constructeur'];  

-- 4. Toutes les clés existent (?&)
SELECT * FROM produits  
WHERE metadata ?& ARRAY['marque', 'specs', 'tags'];  

-- 5. Recherche dans sous-objet
SELECT * FROM produits  
WHERE metadata @> '{"specs": {"cpu": "Intel i7"}}';  

-- 6. Recherche dans tableau
SELECT * FROM produits  
WHERE metadata @> '{"tags": ["gaming"]}';  
```

### Taille de l'Index

```sql
-- Vérifier la taille
SELECT pg_size_pretty(pg_relation_size('idx_metadata_gin'));
-- Résultat typique : 45 MB pour 100K lignes
```

---

## GIN Optimisé (jsonb_path_ops) : Détails

### Quand l'Utiliser ?

Utilisez `jsonb_path_ops` si :
- Vous utilisez **uniquement l'opérateur @>**
- Vous cherchez des **valeurs spécifiques** dans le JSON
- L'**espace disque** est une contrainte
- Vous voulez le **maximum de performance** pour `@>`

### Différence Technique

**jsonb_ops indexe :**
```
"marque"           → lignes [1, 2, 3, ...]
"TechCorp"         → lignes [1, 45, 123, ...]
"specs"            → lignes [1, 2, 3, ...]
"cpu"              → lignes [1, 3, 7, ...]
"Intel i7"         → lignes [1, 12, 34, ...]
...
```

**jsonb_path_ops indexe :**
```
marque → TechCorp           → lignes [1, 45, 123, ...]  
specs → cpu → Intel i7      → lignes [1, 12, 34, ...]  
tags → [0] → gaming         → lignes [2, 4, 8, ...]  
...
```

**Conséquence :**
- `jsonb_path_ops` ne peut pas répondre à "Quelles lignes ont la clé 'marque' ?" (`?`)
- Mais il répond très bien à "Quelles lignes ont marque = 'TechCorp' ?" (`@>`)

### Exemples d'Utilisation

```sql
-- Créer l'index optimisé
CREATE INDEX idx_metadata_path_ops ON produits USING gin(metadata jsonb_path_ops);
```

**Requêtes supportées :**

```sql
-- ✅ Supporté : Contient (@>)
SELECT * FROM produits  
WHERE metadata @> '{"marque": "TechCorp"}';  

-- ✅ Supporté : Sous-objet
SELECT * FROM produits  
WHERE metadata @> '{"specs": {"cpu": "Intel i7"}}';  

-- ✅ Supporté : Valeur dans tableau
SELECT * FROM produits  
WHERE metadata @> '{"tags": ["gaming"]}';  

-- ✅ Supporté : Multiples conditions
SELECT * FROM produits  
WHERE metadata @> '{"marque": "TechCorp", "disponible": true}';  

-- ❌ NON supporté : Existence de clé (?)
SELECT * FROM produits  
WHERE metadata ? 'marque';  
-- Erreur ou Seq Scan (l'index n'est pas utilisé)

-- ❌ NON supporté : Existence multiple (?|, ?&)
SELECT * FROM produits  
WHERE metadata ?| ARRAY['marque', 'fabricant'];  
```

### Taille de l'Index

```sql
-- Comparer les tailles
SELECT
    pg_size_pretty(pg_relation_size('idx_metadata_gin')) AS "jsonb_ops",
    pg_size_pretty(pg_relation_size('idx_metadata_path_ops')) AS "jsonb_path_ops";

-- Résultat typique :
-- jsonb_ops     | jsonb_path_ops
-- 45 MB         | 15 MB
```

**Gain d'espace : 3× plus compact !**

---

## Comparaison : jsonb_ops vs jsonb_path_ops

| Critère | jsonb_ops (Standard) | jsonb_path_ops (Optimisé) |
|---------|----------------------|---------------------------|
| **Taille index** | Plus volumineux | **3× plus compact** |
| **Performance @>** | Rapide | **Plus rapide** (10-20% gain) |
| **Opérateur @>** | ✅ Supporté | ✅ Supporté |
| **Opérateur ?** | ✅ Supporté | ❌ Non supporté |
| **Opérateurs ?| et ?&** | ✅ Supportés | ❌ Non supportés |
| **Opérateur @? (jsonpath exists)** | ✅ Supporté | ✅ Supporté |
| **Opérateur @@ (jsonpath match)** | ✅ Supporté | ✅ Supporté |
| **Flexibilité** | Maximale | Limitée à @>, @?, @@ |
| **Cas d'usage** | Requêtes variées | Recherches de valeurs spécifiques |
| **Maintenance** | Coût normal | Coût légèrement réduit |

> **Note importante** : Contrairement à une croyance répandue, `jsonb_path_ops`  
> supporte aussi les opérateurs JSONPath `@?` et `@@` (depuis PG 12). La seule  
> restriction réelle porte sur les opérateurs d'existence de clés (`?`, `?|`, `?&`).

### Recommandation

```
┌─────────────────────────────────────┐
│ Utilisez jsonb_path_ops si :        │
│ • Vous utilisez uniquement @>       │
│ • Performance max est prioritaire   │
│ • Espace disque limité              │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│ Utilisez jsonb_ops si :             │
│ • Vous utilisez ?, ?|, ?&           │
│ • Requêtes variées                  │
│ • Flexibilité importante            │
└─────────────────────────────────────┘
```

### Exemple 2 : Application avec Logs JSON

**Contexte :** Système de logs avec événements JSON.

```sql
-- Table de logs
CREATE TABLE logs_application (
    id BIGSERIAL PRIMARY KEY,
    timestamp TIMESTAMP DEFAULT NOW(),
    niveau VARCHAR(20),
    evenement JSONB
);

-- Exemples de logs
INSERT INTO logs_application (niveau, evenement) VALUES
    ('INFO', '{
        "user_id": 12345,
        "action": "login",
        "ip": "192.168.1.100",
        "user_agent": "Chrome/120.0",
        "pays": "France",
        "duree_ms": 245
    }'),
    ('ERROR', '{
        "user_id": 67890,
        "action": "payment_failed",
        "error_code": "CARD_DECLINED",
        "montant": 99.99,
        "devise": "EUR",
        "tentatives": 3
    }');
```

**Indexation :**

```sql
-- Index optimisé pour recherches d'événements spécifiques
CREATE INDEX idx_evenement_path_ops ON logs_application  
USING gin(evenement jsonb_path_ops);  

-- Index composite : combiner timestamp + JSONB dans un seul GIN
-- ⚠️ Nécessite l'extension btree_gin (B-Tree et GIN ne peuvent
-- pas être mélangés sans elle, et GIN ne respecte pas ASC/DESC)
CREATE EXTENSION IF NOT EXISTS btree_gin;  
CREATE INDEX idx_timestamp_evenement  
    ON logs_application
    USING gin (timestamp, evenement jsonb_path_ops);

-- Alternative simple : deux index séparés (PG les combinera via BitmapAnd)
-- CREATE INDEX ON logs_application (timestamp DESC);
-- CREATE INDEX ON logs_application USING gin (evenement jsonb_path_ops);
```

**Requêtes d'analyse :**

```sql
-- Analyse 1 : Tous les événements d'un utilisateur
SELECT * FROM logs_application  
WHERE evenement @> '{"user_id": 12345}'  
ORDER BY timestamp DESC;  

-- Analyse 2 : Erreurs de paiement spécifiques
SELECT * FROM logs_application  
WHERE niveau = 'ERROR'  
AND evenement @> '{"action": "payment_failed"}';  

-- Analyse 3 : Actions depuis un pays
SELECT * FROM logs_application  
WHERE evenement @> '{"pays": "France"}';  

-- Analyse 4 : Erreurs avec code spécifique (dernières 24h)
SELECT * FROM logs_application  
WHERE timestamp > NOW() - INTERVAL '24 hours'  
AND evenement @> '{"error_code": "CARD_DECLINED"}';  
```

## Index sur Extraction JSON Spécifique

### Index sur Expression avec JSONB

Si vous interrogez toujours la **même clé JSON**, créez un index sur l'extraction :

```sql
-- Au lieu d'indexer tout le JSON
CREATE INDEX idx_metadata_gin ON produits USING gin(metadata);

-- Indexer uniquement une clé spécifique
CREATE INDEX idx_marque ON produits((metadata->>'marque'));

-- Requête optimisée
SELECT * FROM produits WHERE metadata->>'marque' = 'TechCorp';
-- Utilise idx_marque (B-Tree) au lieu de GIN
```

**Avantages :**
- Index **beaucoup plus compact** (B-Tree)
- **Plus rapide** pour requêtes sur une clé unique
- Fonctionne avec égalité, tri, plages

**Quand l'utiliser :**
- Requêtes sur **une seule clé JSON** récurrente
- Clé avec **haute cardinalité**
- Besoin de **tri** sur la valeur extraite

### Index Composite : Colonne + JSON

```sql
-- Combiner colonne classique + extraction JSON
CREATE INDEX idx_categorie_marque  
ON produits(categorie, (metadata->>'marque'));  

-- Requête ultra-optimisée
SELECT * FROM produits  
WHERE categorie = 'Informatique'  
AND metadata->>'marque' = 'Dell'  
ORDER BY prix;  
```

## Index GIN sur Tableaux JSON

### Tableaux dans JSONB

```sql
-- Table avec tags JSON
CREATE TABLE articles (
    id SERIAL PRIMARY KEY,
    titre VARCHAR(200),
    tags JSONB  -- ["postgresql", "database", "tutorial"]
);

-- Données
INSERT INTO articles (titre, tags) VALUES
    ('Guide PostgreSQL', '["postgresql", "database", "sql", "tutorial"]'),
    ('Introduction JSON', '["json", "nosql", "postgresql"]');
```

**Index GIN pour recherche dans tableaux :**

```sql
-- Index pour rechercher dans les tableaux
CREATE INDEX idx_tags_gin ON articles USING gin(tags jsonb_path_ops);

-- Recherche : Articles avec tag 'postgresql'
SELECT * FROM articles  
WHERE tags @> '["postgresql"]';  

-- Recherche : Articles avec tag 'postgresql' OU 'database'
SELECT * FROM articles  
WHERE tags @> '["postgresql"]' OR tags @> '["database"]';  

-- Alternative avec opérateur ?|
CREATE INDEX idx_tags_standard ON articles USING gin(tags jsonb_ops);  
SELECT * FROM articles  
WHERE tags ?| ARRAY['postgresql', 'database'];  
```

## Performance : Benchmarks

### Test de Performance Comparatif

**Contexte :** Table de 1 million de produits avec JSONB.

```sql
-- Configuration de test
CREATE TABLE produits_test (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(200),
    metadata JSONB
);

-- Générer 1M de lignes (simplifié)
INSERT INTO produits_test (nom, metadata)  
SELECT  
    'Produit ' || i,
    jsonb_build_object(
        'marque', 'Marque' || (i % 100),
        'prix', (random() * 1000)::numeric(10,2),
        'categorie', 'Cat' || (i % 20),
        'en_stock', random() > 0.1,
        'attributs', jsonb_build_object('couleur', 'Couleur' || (i % 10))
    )
FROM generate_series(1, 1000000) i;
```

**Test 1 : Sans Index**
```sql
EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM produits_test  
WHERE metadata @> '{"marque": "Marque42"}';  

-- Résultat :
-- Seq Scan on produits_test  (cost=0.00..52500.00 rows=10000 width=...)
-- Execution Time: 3842.567 ms
```

**Test 2 : Avec GIN Standard (jsonb_ops)**
```sql
CREATE INDEX idx_meta_standard ON produits_test USING gin(metadata);

EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM produits_test  
WHERE metadata @> '{"marque": "Marque42"}';  

-- Résultat :
-- Bitmap Heap Scan on produits_test  (cost=245.23..4582.15 rows=10000 width=...)
-- Execution Time: 67.234 ms
-- Index size: 156 MB
```

**Test 3 : Avec GIN Optimisé (jsonb_path_ops)**
```sql
CREATE INDEX idx_meta_path_ops ON produits_test USING gin(metadata jsonb_path_ops);

EXPLAIN (ANALYZE, BUFFERS)  
SELECT * FROM produits_test  
WHERE metadata @> '{"marque": "Marque42"}';  

-- Résultat :
-- Bitmap Heap Scan on produits_test  (cost=198.45..4124.78 rows=10000 width=...)
-- Execution Time: 48.123 ms
-- Index size: 52 MB
```

**Résultats :**

| Configuration | Temps (ms) | Taille Index | Gain |
|---------------|------------|--------------|------|
| Sans index | 3842 | - | Référence |
| jsonb_ops | 67 | 156 MB | **57× plus rapide** |
| jsonb_path_ops | 48 | 52 MB | **80× plus rapide** |

**Conclusion :** `jsonb_path_ops` est **28% plus rapide** et **3× plus compact** que `jsonb_ops`.

## Index GIN Partiel sur JSONB

### Combiner GIN + Partiel

```sql
-- Index uniquement sur produits en stock
CREATE INDEX idx_metadata_en_stock  
ON produits USING gin(metadata jsonb_path_ops)  
WHERE (metadata->>'en_stock')::boolean = true;  

-- Requête optimisée
SELECT * FROM produits  
WHERE metadata @> '{"marque": "TechCorp"}'  
AND (metadata->>'en_stock')::boolean = true;  
```

**Avantages :**
- Index **beaucoup plus petit**
- **Plus rapide** (moins d'entrées à parcourir)
- Utile si 80%+ des requêtes filtrent sur une condition commune

## Maintenance des Index GIN

### Taille et Croissance

```sql
-- Vérifier la taille d'un index GIN
SELECT
    indexrelname,
    pg_size_pretty(pg_relation_size(indexrelid)) AS taille,
    idx_scan AS utilisations,
    idx_tup_read AS tuples_lus
FROM pg_stat_user_indexes  
WHERE indexrelname LIKE 'idx_metadata%';  
```

### VACUUM et Index GIN

Les index GIN nécessitent une maintenance régulière :

```sql
-- VACUUM pour nettoyer les "pending list"
VACUUM ANALYZE produits;

-- Vérifier les statistiques
SELECT * FROM pg_stat_user_indexes WHERE indexrelname = 'idx_metadata_gin';
```

**Note :** GIN utilise une "pending list" pour les insertions rapides. VACUUM les intègre dans l'index principal.

### Reindexation

Si l'index GIN devient fragmenté :

```sql
-- Option 1 : REINDEX simple (verrouillage en écriture sur la table)
REINDEX INDEX idx_metadata_gin;

-- Option 2 : REINDEX CONCURRENTLY (sans blocage, PG 12+)
REINDEX INDEX CONCURRENTLY idx_metadata_gin;

-- Option 3 : Recréer manuellement (équivalent à CONCURRENTLY,
-- utile si on veut changer la définition de l'index)
CREATE INDEX CONCURRENTLY idx_metadata_gin_new  
ON produits USING gin(metadata jsonb_path_ops);  

DROP INDEX idx_metadata_gin;  
ALTER INDEX idx_metadata_gin_new RENAME TO idx_metadata_gin;  
```

## Cas d'Usage Avancés

### 1. Recherche par Similarité (Trigrammes) dans un Champ JSON

> **Précision** : la « recherche full-text » de PostgreSQL repose normalement  
> sur `tsvector` / `tsquery` (voir le chapitre 18). L'exemple ci-dessous  
> utilise plutôt l'extension `pg_trgm` pour accélérer les recherches  
> `LIKE`/`ILIKE` avec wildcards.

```sql
-- Recherche textuelle dans champs JSON
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    contenu JSONB
);

-- Index trigrammes sur une clé JSON extraite
CREATE EXTENSION IF NOT EXISTS pg_trgm;  
CREATE INDEX idx_contenu_trgm ON documents  
USING gin((contenu->>'texte') gin_trgm_ops);  

-- Accélère les recherches ILIKE avec wildcards
SELECT * FROM documents  
WHERE contenu->>'texte' ILIKE '%postgresql%';  

-- Pour du vrai full-text search (tsvector) :
-- CREATE INDEX idx_contenu_fts ON documents
-- USING gin (to_tsvector('french', contenu->>'texte'));
```

### 2. Index sur Chemin JSON Profond

```sql
-- Index sur chemin profond
CREATE INDEX idx_cpu ON produits  
USING gin((metadata#>'{specs, cpu}') jsonb_path_ops);  

-- Requête optimisée
SELECT * FROM produits  
WHERE metadata#>'{specs, cpu}' @> '"Intel i7"';  
```

### 3. Index Multi-Colonnes avec JSONB

```sql
-- ❌ INCLUDE n'accepte PAS d'expressions, seulement des colonnes simples :
-- CREATE INDEX ... ON produits(categorie) INCLUDE ((metadata->>'marque'));
-- ERROR: expressions are not supported in included columns

-- ✅ Solution 1 : matérialiser la valeur extraite dans une colonne générée
ALTER TABLE produits
    ADD COLUMN marque_extraite TEXT
    GENERATED ALWAYS AS (metadata->>'marque') STORED;

CREATE INDEX idx_categorie_metadata
    ON produits(categorie)
    INCLUDE (marque_extraite);

-- ✅ Solution 2 : index multi-colonnes B-Tree sur deux expressions
CREATE INDEX idx_cat_marque
    ON produits(categorie, (metadata->>'marque'));

-- ✅ Solution 3 : GIN multi-colonnes (nécessite pg_trgm + btree_gin si
-- on veut indexer categorie par égalité dans le même index GIN)
CREATE EXTENSION IF NOT EXISTS btree_gin;  
CREATE INDEX idx_cat_meta_gin  
    ON produits USING gin (categorie, metadata jsonb_path_ops);
```

## Pièges et Limitations

### ❌ Piège 1 : Confusion d'Opérateurs

```sql
-- Index créé
CREATE INDEX idx ON produits USING gin(metadata jsonb_path_ops);

-- ❌ N'utilise PAS l'index (opérateur ->>, pas @>)
SELECT * FROM produits WHERE metadata->>'marque' = 'TechCorp';

-- ✅ Utilise l'index
SELECT * FROM produits WHERE metadata @> '{"marque": "TechCorp"}';
```

**Solution :** Utilisez `@>` avec `jsonb_path_ops`.

### ❌ Piège 2 : jsonb_path_ops sans Support de ?

```sql
-- Index optimisé
CREATE INDEX idx ON produits USING gin(metadata jsonb_path_ops);

-- ❌ Seq Scan (opérateur ? non supporté)
SELECT * FROM produits WHERE metadata ? 'marque';

-- Solution : Utiliser jsonb_ops
CREATE INDEX idx_standard ON produits USING gin(metadata jsonb_ops);
```

### ❌ Piège 3 : Types de Données dans JSON

```sql
-- Données avec types mixtes
INSERT INTO produits (metadata) VALUES
    ('{"prix": 99.99}'),   -- nombre
    ('{"prix": "99.99"}'); -- texte

-- ❌ Ne trouve qu'une version
SELECT * FROM produits WHERE metadata @> '{"prix": 99.99}';
-- Trouve uniquement celle avec nombre, pas texte !

-- ✅ Solution : Normaliser les types
```

### ❌ Piège 4 : Index GIN Trop Large

```sql
-- ❌ MAUVAIS : JSONB gigantesque (>1MB par ligne)
CREATE TABLE logs (
    id BIGSERIAL PRIMARY KEY,
    data JSONB  -- Contient des arrays de 100K éléments
);

CREATE INDEX idx ON logs USING gin(data);
-- Index de plusieurs centaines de GB, inutilisable !

-- ✅ Solution : Index partiel ou sur clés spécifiques
CREATE INDEX idx_key ON logs((data->>'important_field'));
```

### ❌ Piège 5 : Oublier les Casts

```sql
-- Requête avec cast
SELECT * FROM produits  
WHERE (metadata->>'prix')::numeric > 100;  

-- ❌ N'utilise pas l'index GIN (cast nécessaire)

-- ✅ Solution : Index sur expression
CREATE INDEX idx_prix ON produits(((metadata->>'prix')::numeric));
```

## Bonnes Pratiques

### ✅ DO (À Faire)

1. **Choisir le bon type d'index GIN selon usage**
   ```sql
   -- Uniquement @> : jsonb_path_ops
   CREATE INDEX idx ON ma_table USING gin(col jsonb_path_ops);

   -- Opérateurs variés (?, ?|, ?&) : jsonb_ops
   CREATE INDEX idx ON ma_table USING gin(col jsonb_ops);
   ```

2. **Préférer jsonb_path_ops quand possible**
   - Plus compact (3× moins d'espace)
   - Plus rapide pour @>

3. **Index sur extraction pour clés fréquentes**
   ```sql
   -- Plus efficace qu'un GIN complet
   CREATE INDEX idx ON ma_table((jsonb_col->>'frequent_key'));
   ```

4. **Combiner avec index partiels**
   ```sql
   CREATE INDEX idx ON ma_table USING gin(col jsonb_path_ops)
   WHERE (col->>'status') = 'active';
   ```

5. **Analyser avec EXPLAIN avant de créer**
   ```sql
   EXPLAIN (ANALYZE, BUFFERS) SELECT ...;
   ```

6. **VACUUM régulier pour GIN**
   ```sql
   -- Configuration autovacuum adaptée
   ALTER TABLE ma_table SET (
       autovacuum_vacuum_scale_factor = 0.05
   );
   ```

7. **Documenter les index GIN**
   ```sql
   COMMENT ON INDEX idx_metadata_gin IS
   'GIN jsonb_path_ops pour recherches @> sur metadata.
   Réduit temps requête de 3800ms à 48ms (80× plus rapide).';
   ```

### ❌ DON'T (À Éviter)

1. **Ne pas utiliser JSON au lieu de JSONB**
   ```sql
   -- ❌ MAUVAIS
   CREATE TABLE t (data JSON);

   -- ✅ BON
   CREATE TABLE t (data JSONB);
   ```

2. **Ne pas créer d'index GIN sur JSON énormes**
   - Limite pratique : < 100KB par document JSON

3. **Ne pas confondre les opérateurs**
   ```sql
   -- ❌ ->>'key' avec GIN ne fonctionne pas bien
   -- ✅ Utiliser @> à la place
   ```

4. **Ne pas oublier VACUUM sur tables avec GIN**

5. **Ne pas sur-indexer**
   ```sql
   -- ❌ Inutile d'avoir les deux sur la même colonne
   CREATE INDEX idx1 ON ma_table USING gin(col jsonb_ops);
   CREATE INDEX idx2 ON ma_table USING gin(col jsonb_path_ops);

   -- ✅ Choisir un seul selon besoin
   ```

## Monitoring et Diagnostic

### Identifier les Requêtes Lentes sur JSONB

```sql
-- Avec pg_stat_statements
SELECT
    query,
    calls,
    mean_exec_time,
    total_exec_time
FROM pg_stat_statements  
WHERE query LIKE '%metadata%@>%'  
OR query LIKE '%metadata%?%'  
ORDER BY mean_exec_time DESC  
LIMIT 10;  
```

### Analyser l'Utilisation des Index GIN

```sql
-- ⚠️ pg_stat_user_indexes n'a PAS de colonne indexdef : on identifie la méthode
-- d'accès (gin) en joignant pg_class → pg_am.
SELECT
    s.schemaname,
    s.relname,
    s.indexrelname,
    s.idx_scan AS "Utilisations",
    s.idx_tup_read AS "Tuples lus",
    pg_size_pretty(pg_relation_size(s.indexrelid)) AS "Taille"
FROM pg_stat_user_indexes s
JOIN pg_class c ON c.oid = s.indexrelid
JOIN pg_am   am ON am.oid = c.relam
WHERE am.amname = 'gin'  
ORDER BY s.idx_scan DESC;  
```

### Vérifier les Pending Lists GIN

```sql
-- Taille réelle de la "pending list" d'un index GIN (extension pgstattuple)
CREATE EXTENSION IF NOT EXISTS pgstattuple;

SELECT pending_pages, pending_tuples
FROM pgstatginindex('idx_metadata_gin');
-- pending_pages élevé → lancer VACUUM (ou désactiver fastupdate / réduire
-- gin_pending_list_limit) pour intégrer la pending list dans l'arbre principal.
```

## Résumé

### Points Clés

1. **JSONB > JSON pour indexation**
   - Stockage binaire optimisé
   - Support complet des index GIN

2. **Deux types d'index GIN pour JSONB :**  
   - **jsonb_ops** : Flexible, tous opérateurs, plus volumineux  
   - **jsonb_path_ops** : Optimisé, uniquement @>, 3× plus compact

3. **Opérateurs principaux :**  
   - `@>` : Contient (le plus utilisé)  
   - `?` : Clé existe  
   - `?|`, `?&` : Existence multiple

4. **Performances :**
   - Index GIN : 50-200× plus rapide que Seq Scan
   - jsonb_path_ops : 20-30% plus rapide que jsonb_ops
   - Taille : jsonb_path_ops 3× plus compact

5. **Quand utiliser quoi :**
   ```
   Uniquement @> → jsonb_path_ops
   Opérateurs ?, ?|, ?& → jsonb_ops
   Une clé fréquente → Index B-Tree sur extraction
   ```

6. **Maintenance :**
   - VACUUM régulier important pour GIN
   - Surveiller la taille des index
   - Reindexer si fragmenté
