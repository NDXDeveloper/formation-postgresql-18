# 13.5.3. Index Multi-Colonnes et INCLUDE (Covering Index)

## Introduction

Un **index multi-colonnes** (également appelé **index composite** ou **index concaténé**) est un index qui porte sur plusieurs colonnes simultanément. Il permet d'optimiser les requêtes qui filtrent ou trient sur plusieurs colonnes à la fois.

La clause **INCLUDE** (disponible depuis PostgreSQL 11) permet de créer des **covering indexes** : des index qui contiennent des colonnes supplémentaires non utilisées pour le tri ou la recherche, mais disponibles pour satisfaire la requête sans accéder à la table.

Ces techniques sont essentielles pour optimiser les requêtes complexes et réduire les accès disque.

---

## Partie 1 : Index Multi-Colonnes

### Concept Fondamental

**Index simple (sur une seule colonne) :**
```sql
CREATE INDEX idx_nom ON utilisateurs(nom);
```
→ Optimise les requêtes sur `nom` uniquement.

**Index multi-colonnes (sur plusieurs colonnes) :**
```sql
CREATE INDEX idx_nom_prenom ON utilisateurs(nom, prenom);
```
→ Optimise les requêtes sur :  
- `nom` seul  
- `nom` ET `prenom`
- Mais **PAS** `prenom` seul (ordre important !)

### Pourquoi Utiliser des Index Multi-Colonnes ?

#### 1. **Requêtes avec Plusieurs Filtres**

```sql
-- Sans index multi-colonnes : Deux index séparés
CREATE INDEX idx_nom ON clients(nom);  
CREATE INDEX idx_ville ON clients(ville);  

-- Requête : PostgreSQL doit combiner deux index (Bitmap Index Scan) ou scanner un seul index
SELECT * FROM clients WHERE nom = 'Dupont' AND ville = 'Paris';
```

```sql
-- Avec index multi-colonnes : Un seul index optimisé
CREATE INDEX idx_nom_ville ON clients(nom, ville);

-- Requête : Un seul Index Scan, beaucoup plus rapide !
SELECT * FROM clients WHERE nom = 'Dupont' AND ville = 'Paris';
```

#### 2. **Tri sur Plusieurs Colonnes**

```sql
-- Index multi-colonnes
CREATE INDEX idx_date_montant ON commandes(date_commande, montant);

-- Requête avec tri : Utilise l'index pour éviter un tri coûteux
SELECT * FROM commandes  
WHERE date_commande >= '2025-01-01'  
ORDER BY date_commande, montant;  
```

#### 3. **Sélectivité Progressive**

Un index multi-colonnes agit comme un **annuaire téléphonique** :
- Trié d'abord par **nom de famille**
- Puis par **prénom** pour les personnes avec le même nom

```sql
CREATE INDEX idx_nom_prenom ON personnes(nom, prenom);

-- ✅ Très efficace : Cherche "Dupont" puis "Jean"
SELECT * FROM personnes WHERE nom = 'Dupont' AND prenom = 'Jean';

-- ✅ Efficace : Cherche tous les "Dupont"
SELECT * FROM personnes WHERE nom = 'Dupont';

-- ❌ Peu efficace : Ne peut pas utiliser l'index efficacement
SELECT * FROM personnes WHERE prenom = 'Jean';
```

---

### Syntaxe et Structure

#### Syntaxe de Base

```sql
CREATE INDEX nom_index ON nom_table(colonne1, colonne2, colonne3, ...);
```

#### Limite de Colonnes

PostgreSQL permet jusqu'à **32 colonnes** dans un index multi-colonnes.

En pratique, **2 à 4 colonnes** est le plus courant.

#### Ordre des Colonnes : CRUCIAL !

**Règle d'or :** L'ordre des colonnes dans l'index détermine son efficacité.

**Principe de base :**
```sql
CREATE INDEX idx_a_b_c ON table(a, b, c);
```

Peut être utilisé pour :
- ✅ `WHERE a = ...`  
- ✅ `WHERE a = ... AND b = ...`  
- ✅ `WHERE a = ... AND b = ... AND c = ...`  
- ⚠️  `WHERE a = ... AND c = ...` (partiellement, ignore `c`)  
- ❌ `WHERE b = ...` (ne peut pas utiliser l'index)  
- ❌ `WHERE c = ...` (ne peut pas utiliser l'index)

**Analogie : Annuaire téléphonique**

Un annuaire est trié par :
1. **Nom de famille** (A-Z)  
2. **Prénom** (A-Z pour chaque nom)

✅ Facile : Trouver "Dupont Jean"  
✅ Facile : Trouver tous les "Dupont"  
❌ Difficile : Trouver tous les "Jean" (il faut parcourir tout l'annuaire !)

### Exemples Pratiques : Index Multi-Colonnes

#### Exemple 1 : Table de Commandes

**Contexte :** E-commerce avec recherches fréquentes par client et date.

```sql
-- Table de commandes
CREATE TABLE commandes (
    id SERIAL PRIMARY KEY,
    client_id INTEGER,
    date_commande DATE,
    statut VARCHAR(50),
    montant NUMERIC(10,2)
);

-- Index multi-colonnes : client + date
CREATE INDEX idx_client_date ON commandes(client_id, date_commande DESC);
```

**Requêtes optimisées :**

```sql
-- ✅ Utilise l'index parfaitement
SELECT * FROM commandes  
WHERE client_id = 12345  
ORDER BY date_commande DESC;  

-- ✅ Utilise l'index (client + date)
SELECT * FROM commandes  
WHERE client_id = 12345  
AND date_commande >= '2025-01-01';  

-- ✅ Utilise l'index (client uniquement)
SELECT * FROM commandes  
WHERE client_id = 12345;  

-- ❌ N'utilise PAS l'index efficacement
SELECT * FROM commandes  
WHERE date_commande >= '2025-01-01';  
-- Solution : Créer un index séparé sur date_commande
```

#### Exemple 2 : Table d'Événements (Logs)

**Contexte :** Logs applicatifs avec recherches par niveau et date.

```sql
-- Table de logs
CREATE TABLE logs (
    id BIGSERIAL PRIMARY KEY,
    niveau VARCHAR(20),  -- 'DEBUG', 'INFO', 'WARNING', 'ERROR'
    message TEXT,
    created_at TIMESTAMP
);

-- Stratégie : Ordre des colonnes selon sélectivité
-- niveau : 4 valeurs distinctes (faible sélectivité)
-- created_at : beaucoup de valeurs (haute sélectivité)

-- ❌ MAUVAIS ordre (moins sélectif en premier)
CREATE INDEX idx_bad ON logs(niveau, created_at);
-- Problème : Le niveau filtre peu, l'index est moins efficace

-- ✅ BON ordre (plus sélectif en premier)
CREATE INDEX idx_good ON logs(created_at DESC, niveau);
-- Avantage : La date filtre beaucoup, puis on affine par niveau
```

**Analyse des requêtes :**

```sql
-- Requête typique : Logs récents + niveau
SELECT * FROM logs  
WHERE created_at >= NOW() - INTERVAL '1 hour'  
AND niveau = 'ERROR'  
ORDER BY created_at DESC;  

-- Avec idx_good :
-- 1. Filtre rapidement sur created_at (très sélectif)
-- 2. Puis filtre sur niveau
-- 3. Pas besoin de tri (ORDER BY déjà dans l'index)
```

#### Exemple 3 : Recherche Multi-Critères (Immobilier)

```sql
-- Table de biens immobiliers
CREATE TABLE biens (
    id SERIAL PRIMARY KEY,
    ville VARCHAR(100),
    type VARCHAR(50),      -- 'appartement', 'maison', 'studio'
    prix NUMERIC(10,2),
    surface NUMERIC(6,2),
    nb_pieces INTEGER
);

-- Index multi-colonnes pour recherche typique
CREATE INDEX idx_recherche ON biens(ville, type, prix, surface);
```

**Requêtes optimisées :**

```sql
-- ✅ Utilise l'index complètement
SELECT * FROM biens  
WHERE ville = 'Paris'  
AND type = 'appartement'  
AND prix BETWEEN 200000 AND 500000  
ORDER BY surface DESC;  

-- ✅ Utilise l'index (ville + type)
SELECT * FROM biens  
WHERE ville = 'Paris'  
AND type = 'appartement';  

-- ⚠️ Utilise l'index partiellement (ville uniquement)
SELECT * FROM biens  
WHERE ville = 'Paris'  
AND prix BETWEEN 200000 AND 500000;  
-- Note : Saute 'type', donc n'utilise pas 'prix' efficacement

-- ❌ N'utilise PAS l'index efficacement
SELECT * FROM biens  
WHERE type = 'appartement';  
-- type est en 2e position, ne peut pas utiliser l'index sans ville
```

### Choisir l'Ordre des Colonnes

#### Règle 1 : Sélectivité (Règle Générale)

**Placez les colonnes les plus sélectives en premier.**

**Sélectivité = Nombre de valeurs distinctes / Nombre total de lignes**

```sql
-- Exemple : Table de 1 000 000 de lignes

-- Colonne 'pays' : 5 valeurs distinctes (FR, US, UK, DE, ES)
-- Sélectivité = 5 / 1,000,000 = 0.000005 (faible)

-- Colonne 'email' : 980,000 valeurs distinctes (presque uniques)
-- Sélectivité = 980,000 / 1,000,000 = 0.98 (haute)

-- ✅ BON : Colonne très sélective en premier
CREATE INDEX idx_good ON table(email, pays);

-- ❌ MAUVAIS : Colonne peu sélective en premier
CREATE INDEX idx_bad ON table(pays, email);
```

#### Règle 2 : Fréquence d'Utilisation dans les Requêtes

**Si une colonne est toujours utilisée dans les requêtes, placez-la en premier.**

```sql
-- Si 90% des requêtes filtrent par client_id
-- Et seulement 20% ajoutent un filtre sur statut

-- ✅ BON : Colonne la plus utilisée en premier
CREATE INDEX idx ON commandes(client_id, statut);
```

#### Règle 3 : Égalité vs Plage

**Colonnes avec égalité (=) avant colonnes avec plage (>, <, BETWEEN).**

```sql
-- Requête typique
SELECT * FROM transactions  
WHERE type_compte = 'courant'        -- Égalité  
AND montant > 1000                   -- Plage  
AND date_transaction >= '2025-01-01'; -- Plage  

-- ✅ BON : Égalité en premier
CREATE INDEX idx ON transactions(type_compte, date_transaction, montant);

-- Explication : PostgreSQL utilise l'égalité efficacement,
-- puis peut parcourir les plages de manière ordonnée
```

#### Règle 4 : Exceptions et Cas Particuliers

Parfois, il faut **privilégier la fréquence** sur la sélectivité :

```sql
-- Cas : 99% des requêtes filtrent par statut = 'actif' (peu sélectif)
-- Et seulement 1% ajoutent un filtre sur email (très sélectif)

-- ✅ BON pour ce cas : Colonne la plus fréquente en premier
CREATE INDEX idx ON table(statut, email);

-- Alternative : Index partiel
CREATE INDEX idx ON table(email) WHERE statut = 'actif';
```

### Stratégies Avancées

#### 1. Index Multi-Colonnes avec Tri

```sql
-- Index pour tri sur plusieurs colonnes
CREATE INDEX idx_tri ON produits(categorie, prix DESC, nom);

-- Requête : Pas de tri coûteux, utilise l'index directement
SELECT * FROM produits  
WHERE categorie = 'Electronique'  
ORDER BY prix DESC, nom;  
```

#### 2. Index Multi-Colonnes avec Expression

```sql
-- Combiner multi-colonnes + expression
CREATE INDEX idx_combo ON utilisateurs(pays, (LOWER(email)));

-- Requête optimisée
SELECT * FROM utilisateurs  
WHERE pays = 'FR'  
AND LOWER(email) = 'test@example.com';  
```

#### 3. Index Multi-Colonnes Partiel

```sql
-- Combiner multi-colonnes + partiel
CREATE INDEX idx_actifs ON commandes(client_id, date_commande)  
WHERE statut IN ('en_cours', 'en_preparation');  

-- Très efficace pour requêtes ciblées
SELECT * FROM commandes  
WHERE client_id = 12345  
AND statut = 'en_cours'  
ORDER BY date_commande DESC;  
```

## Partie 2 : INCLUDE et Covering Index

### Qu'est-ce qu'un Covering Index ?

Un **covering index** (index couvrant) est un index qui contient **toutes les colonnes** nécessaires pour satisfaire une requête, sans avoir besoin d'accéder à la table principale.

**Avantage majeur :** Évite l'accès à la table (Heap Fetch), ce qui réduit drastiquement les I/O.

### Index Sans INCLUDE (Comportement Classique)

```sql
-- Index classique sur client_id
CREATE INDEX idx_client ON commandes(client_id);

-- Requête
SELECT client_id, montant FROM commandes WHERE client_id = 12345;
```

**Plan d'exécution :**
```
Index Scan using idx_client on commandes
  Index Cond: (client_id = 12345)
  → Heap Fetches: 150   ← Accès à la table pour récupérer 'montant'
```

**Problème :** PostgreSQL doit accéder à la table pour lire la colonne `montant`.

### Index Avec INCLUDE (Covering Index)

```sql
-- Covering index : inclut 'montant' sans l'utiliser pour l'indexation
CREATE INDEX idx_client_include_montant  
ON commandes(client_id)  
INCLUDE (montant);  

-- Même requête
SELECT client_id, montant FROM commandes WHERE client_id = 12345;
```

**Plan d'exécution :**
```
Index Only Scan using idx_client_include_montant on commandes
  Index Cond: (client_id = 12345)
  → Heap Fetches: 0   ← Aucun accès à la table !
```

**Avantage :** Toutes les données sont dans l'index, pas besoin d'accéder à la table.

---

### Syntaxe INCLUDE

```sql
CREATE INDEX nom_index  
ON nom_table(colonnes_indexées)  
INCLUDE (colonnes_couvertes);  
```

**Paramètres :**
- **colonnes_indexées** : Colonnes utilisées pour le tri, la recherche et le filtrage  
- **colonnes_couvertes** : Colonnes supplémentaires stockées dans l'index, mais pas utilisées pour l'indexation

### Différence Clé : Index vs INCLUDE

```sql
-- Option 1 : Index multi-colonnes classique
CREATE INDEX idx_classic ON table(col1, col2, col3);

-- Option 2 : Index avec INCLUDE
CREATE INDEX idx_include ON table(col1, col2) INCLUDE (col3);
```

**Différences :**

| Critère | Index Multi-Colonnes | Index avec INCLUDE |
|---------|----------------------|---------------------|
| **Taille de l'index** | Plus grand (col3 participe à l'arbre B-Tree) | Plus petit (col3 seulement en feuilles) |
| **Utilisation pour recherche** | col3 peut être utilisée pour filtrer | col3 ne peut PAS être utilisée pour filtrer |
| **Utilisation pour tri** | col3 peut être utilisée pour trier | col3 ne peut PAS être utilisée pour trier |
| **Covering queries** | Oui (si toutes colonnes SELECT sont indexées) | Oui (colonnes INCLUDE disponibles) |
| **Performance** | Peut être moins efficace (arbre plus profond) | Plus efficace (arbre plus compact) |

---

### Exemples Pratiques : INCLUDE

#### Exemple 1 : Table de Commandes (Optimisation SELECT)

```sql
-- Table de commandes
CREATE TABLE commandes (
    id SERIAL PRIMARY KEY,
    client_id INTEGER,
    date_commande DATE,
    statut VARCHAR(50),
    montant NUMERIC(10,2),
    note TEXT
);

-- Requête fréquente : Lister commandes d'un client avec montant
-- Sans INCLUDE
CREATE INDEX idx_client ON commandes(client_id);

EXPLAIN (ANALYZE, BUFFERS)  
SELECT client_id, date_commande, montant  
FROM commandes  
WHERE client_id = 12345;  

-- Résultat :
-- Index Scan using idx_client on commandes
--   Index Cond: (client_id = 12345)
--   Heap Fetches: 150  ← Accès table pour date_commande et montant
--   Buffers: shared hit=350
```

```sql
-- Avec INCLUDE : Covering index
CREATE INDEX idx_client_covering  
ON commandes(client_id)  
INCLUDE (date_commande, montant);  

EXPLAIN (ANALYZE, BUFFERS)  
SELECT client_id, date_commande, montant  
FROM commandes  
WHERE client_id = 12345;  

-- Résultat :
-- Index Only Scan using idx_client_covering on commandes
--   Index Cond: (client_id = 12345)
--   Heap Fetches: 0  ← Aucun accès table !
--   Buffers: shared hit=8  ← 40× moins de buffers !
```

**Gains :**
- **Heap Fetches : 150 → 0**  
- **Buffers : 350 → 8** (44× moins d'I/O)  
- **Performance : 3-10× plus rapide**

#### Exemple 2 : Recherche avec Agrégation

```sql
-- Requête : Somme des montants par client
SELECT client_id, SUM(montant)  
FROM commandes  
GROUP BY client_id;  

-- Sans INCLUDE : Doit accéder à la table pour 'montant'
CREATE INDEX idx_client ON commandes(client_id);
-- Résultat : Index Scan + Heap Fetches (lent)

-- Avec INCLUDE : Tout dans l'index
CREATE INDEX idx_client_covering  
ON commandes(client_id)  
INCLUDE (montant);  
-- Résultat : Index Only Scan (rapide !)
```

#### Exemple 3 : Tri + Sélection de Colonnes

```sql
-- Table de produits
CREATE TABLE produits (
    id SERIAL PRIMARY KEY,
    categorie VARCHAR(100),
    nom VARCHAR(200),
    prix NUMERIC(10,2),
    stock INTEGER
);

-- Requête : Top 10 produits par catégorie
SELECT nom, prix  
FROM produits  
WHERE categorie = 'Electronique'  
ORDER BY prix DESC  
LIMIT 10;  

-- Covering index parfait
CREATE INDEX idx_cat_prix_covering  
ON produits(categorie, prix DESC)  
INCLUDE (nom);  

-- Résultat :
-- Index Only Scan using idx_cat_prix_covering
--   Index Cond: (categorie = 'Electronique'::text)
--   Heap Fetches: 0
-- Tout dans l'index : filtrage, tri, colonnes SELECT !
```

#### Exemple 4 : Jointure Optimisée

```sql
-- Table de détails de commandes
CREATE TABLE details_commandes (
    id SERIAL PRIMARY KEY,
    commande_id INTEGER,
    produit_id INTEGER,
    quantite INTEGER,
    prix_unitaire NUMERIC(10,2)
);

-- Requête fréquente : Détails d'une commande
SELECT produit_id, quantite, prix_unitaire  
FROM details_commandes  
WHERE commande_id = 12345;  

-- Covering index
CREATE INDEX idx_commande_covering  
ON details_commandes(commande_id)  
INCLUDE (produit_id, quantite, prix_unitaire);  

-- Index Only Scan : Toutes les données dans l'index
```

### Quand Utiliser INCLUDE ?

#### ✅ Utilisez INCLUDE Si :

1. **Requêtes SELECT avec peu de colonnes**
   ```sql
   -- Typique : SELECT 3-5 colonnes, WHERE sur 1-2 colonnes
   SELECT col1, col2, col3, col4
   FROM table
   WHERE indexed_col = value;
   ```

2. **Colonnes de faible cardinalité dans SELECT**
   ```sql
   -- Inclure des colonnes simples : nombres, dates, textes courts
   INCLUDE (montant, date, statut)
   ```

3. **Requêtes fréquentes et critiques**
   - Réduction des I/O importante
   - Impact sur performance significatif

4. **Index déjà sélectif**
   ```sql
   -- Bon candidat : client_id très sélectif
   CREATE INDEX idx ON commandes(client_id) INCLUDE (montant, date);
   ```

#### ❌ N'Utilisez PAS INCLUDE Si :

1. **Colonnes très volumineuses**
   ```sql
   -- ❌ MAUVAIS : TEXT ou BYTEA volumineux
   CREATE INDEX idx ON table(id) INCLUDE (description_longue);
   -- Problème : Index gigantesque, contre-productif
   ```

2. **Toutes les colonnes de la table**
   ```sql
   -- ❌ MAUVAIS : Dupliquer toute la table dans l'index
   CREATE INDEX idx ON table(id) INCLUDE (col1, col2, ..., col50);
   -- Problème : Index aussi volumineux que la table !
   ```

3. **Colonnes utilisées pour filtrage ou tri**
   ```sql
   -- ❌ MAUVAIS : 'prix' utilisé dans WHERE et ORDER BY
   CREATE INDEX idx ON produits(categorie) INCLUDE (prix);
   SELECT * FROM produits WHERE categorie = 'X' AND prix > 100 ORDER BY prix;

   -- ✅ BON : 'prix' dans l'index multi-colonnes
   CREATE INDEX idx ON produits(categorie, prix);
   ```

4. **Requêtes variées sans pattern clair**
   - Si chaque requête SELECT des colonnes différentes, INCLUDE n'aide pas

### INCLUDE : Aspects Techniques

#### 1. Structure Interne

**Index B-Tree classique :**
```
Nœuds internes : contiennent (col1, col2, pointeur)  
Feuilles : contiennent (col1, col2, pointeur vers heap)  
```

**Index B-Tree avec INCLUDE :**
```
Nœuds internes : contiennent (col1, col2, pointeur)  ← INCLUDE absent  
Feuilles : contiennent (col1, col2, col_include, pointeur vers heap)  ← INCLUDE présent  
```

**Conséquence :** L'arbre est plus compact (nœuds internes plus petits), donc :
- Moins de niveaux dans l'arbre
- Parcours plus rapide
- Meilleur taux de cache

#### 2. Taille de l'Index

```sql
-- Comparer les tailles
-- Option 1 : Multi-colonnes classique
CREATE INDEX idx_multi ON table(a, b, c, d);

-- Option 2 : Avec INCLUDE
CREATE INDEX idx_include ON table(a, b) INCLUDE (c, d);

-- Vérifier la taille
SELECT
    pg_size_pretty(pg_relation_size('idx_multi')) AS "Multi-colonnes",
    pg_size_pretty(pg_relation_size('idx_include')) AS "Avec INCLUDE";
```

**Résultat typique :**
- Multi-colonnes : 150 MB
- Avec INCLUDE : 120 MB (20% plus petit)

#### 3. Maintenance (Vacuum, Insert, Update)

**INCLUDE n'augmente pas significativement le coût de maintenance**, car :
- Les colonnes INCLUDE ne sont que dans les feuilles
- Moins d'impact sur la structure de l'arbre

```sql
-- Impact INSERT/UPDATE similaire entre :
CREATE INDEX idx1 ON table(a, b) INCLUDE (c);
-- Et :
CREATE INDEX idx2 ON table(a, b, c);
```

---

### Index Only Scan : Conditions de Succès

Pour qu'un **Index Only Scan** (scan uniquement sur index) soit efficace, deux conditions :

#### 1. Toutes les Colonnes Doivent Être dans l'Index

```sql
CREATE INDEX idx ON table(col1) INCLUDE (col2, col3);

-- ✅ Index Only Scan
SELECT col1, col2, col3 FROM table WHERE col1 = value;

-- ❌ Index Scan + Heap Fetch (col4 pas dans l'index)
SELECT col1, col2, col3, col4 FROM table WHERE col1 = value;
```

#### 2. Visibility Map Doit Être À Jour (VACUUM)

PostgreSQL utilise une **Visibility Map** pour savoir quelles pages sont visibles sans vérifier le heap.

```sql
-- Si la table n'a pas été VACUUM récemment :
EXPLAIN (ANALYZE, BUFFERS)  
SELECT col1, col2 FROM table WHERE col1 = value;  

-- Résultat :
-- Index Only Scan using idx on table
--   Heap Fetches: 500  ← Doit quand même vérifier la visibilité !
```

**Solution : VACUUM régulier**
```sql
VACUUM table;
-- Ou mieux : Autovacuum configuré correctement
```

**Après VACUUM :**
```
Index Only Scan using idx on table
  Heap Fetches: 0  ← Parfait !
```

## Combinaisons Avancées

### 1. Multi-Colonnes + INCLUDE + Partiel

```sql
-- L'index ultime : Toutes les techniques combinées !
CREATE INDEX idx_ultimate  
ON commandes(client_id, date_commande DESC)  
INCLUDE (montant, statut)  
WHERE date_commande >= '2025-01-01';  

-- Requête parfaitement optimisée
SELECT client_id, date_commande, montant, statut  
FROM commandes  
WHERE client_id = 12345  
AND date_commande >= '2025-01-01'  
ORDER BY date_commande DESC;  

-- Résultat :
-- Index Only Scan using idx_ultimate
--   Index Cond: (client_id = 12345) AND (date_commande >= '2025-01-01')
--   Heap Fetches: 0
-- Tout est parfait : filtrage, tri, covering, index compact !
```

### 2. Expression + INCLUDE

```sql
-- Index sur expression avec INCLUDE
CREATE INDEX idx_email_data  
ON utilisateurs((LOWER(email)))  
INCLUDE (nom, prenom, date_inscription);  

-- Requête couverte
SELECT nom, prenom, date_inscription  
FROM utilisateurs  
WHERE LOWER(email) = 'test@example.com';  

-- Index Only Scan !
```

### 3. Index UNIQUE avec INCLUDE

```sql
-- UNIQUE index avec colonnes supplémentaires
CREATE UNIQUE INDEX idx_email_unique  
ON utilisateurs(email)  
INCLUDE (nom, date_inscription);  

-- Garantit unicité de 'email'
-- ET permet des covering queries
SELECT nom, date_inscription  
FROM utilisateurs  
WHERE email = 'test@example.com';  
```

## Stratégies de Conception

### Méthodologie : Analyser les Requêtes

**Étape 1 : Identifier les requêtes fréquentes**
```sql
-- Utiliser pg_stat_statements
SELECT
    query,
    calls,
    total_exec_time / calls AS avg_time_ms
FROM pg_stat_statements  
ORDER BY calls DESC  
LIMIT 20;  
```

**Étape 2 : Analyser avec EXPLAIN**
```sql
EXPLAIN (ANALYZE, BUFFERS)  
SELECT col1, col2, col3  
FROM table  
WHERE indexed_col = value;  
```

**Étape 3 : Identifier les opportunités**

Cherchez :
- **Seq Scan** → Besoin d'index  
- **Index Scan + Heap Fetches** → Candidat INCLUDE  
- **Sort** coûteux → Index multi-colonnes avec tri

**Étape 4 : Créer l'index optimisé**

```sql
-- Exemple de transformation
-- Avant : Index simple
CREATE INDEX idx_old ON table(col1);
-- Heap Fetches: 1000, Buffers: 2500

-- Après : Index avec INCLUDE
CREATE INDEX idx_new ON table(col1) INCLUDE (col2, col3);
-- Heap Fetches: 0, Buffers: 150
-- 16× moins d'I/O !
```

**Étape 5 : Mesurer l'impact**

```sql
-- Comparer avant/après
EXPLAIN (ANALYZE, BUFFERS)  
SELECT ...;  

-- Vérifier l'utilisation
SELECT * FROM pg_stat_user_indexes  
WHERE indexname = 'idx_new';  
```

## Maintenance et Monitoring

### Lister les Index avec INCLUDE

```sql
SELECT
    schemaname,
    tablename,
    indexname,
    indexdef
FROM pg_indexes  
WHERE indexdef LIKE '%INCLUDE%'  
ORDER BY tablename, indexname;  
```

### Analyser l'Efficacité

```sql
-- Requête pour analyser les covering index
SELECT
    schemaname,
    tablename,
    indexname,
    idx_scan AS "Utilisations",
    pg_size_pretty(pg_relation_size(indexrelid)) AS "Taille",
    idx_tup_read AS "Tuples lus",
    idx_tup_fetch AS "Tuples récupérés",
    ROUND(100.0 * idx_tup_fetch / NULLIF(idx_tup_read, 0), 2) AS "% Fetch"
FROM pg_stat_user_indexes  
WHERE schemaname NOT IN ('pg_catalog', 'information_schema')  
ORDER BY idx_scan DESC;  
```

**Interprétation :**
- **% Fetch proche de 0%** : Excellent ! Index Only Scan efficace  
- **% Fetch > 80%** : Index peu couvrant, considérer INCLUDE

### Identifier les Index Surdimensionnés

```sql
-- Index trop volumineux (> 1 GB) avec peu d'utilisation
SELECT
    schemaname,
    tablename,
    indexname,
    pg_size_pretty(pg_relation_size(indexrelid)) AS "Taille",
    idx_scan AS "Utilisations"
FROM pg_stat_user_indexes  
WHERE pg_relation_size(indexrelid) > 1073741824  -- 1 GB  
AND idx_scan < 100  
ORDER BY pg_relation_size(indexrelid) DESC;  
```

## Pièges et Limitations

### ❌ Piège 1 : INCLUDE avec Colonnes Volumineuses

```sql
-- ❌ MAUVAIS : INCLUDE sur colonne TEXT très volumineuse
CREATE INDEX idx_bad ON articles(categorie)  
INCLUDE (contenu_complet);  
-- Problème : Index gigantesque (plusieurs Go), contre-productif

-- ✅ BON : INCLUDE sur colonnes compactes uniquement
CREATE INDEX idx_good ON articles(categorie)  
INCLUDE (titre, date_publication, nb_vues);  
```

**Règle :** INCLUDE seulement des colonnes **< 100 octets** par ligne.

### ❌ Piège 2 : Ordre Incorrect dans Index Multi-Colonnes

```sql
-- Table : 1M de commandes
-- Requête fréquente :
SELECT * FROM commandes  
WHERE client_id = X AND statut = 'livree';  

-- ❌ MAUVAIS ordre
CREATE INDEX idx_bad ON commandes(statut, client_id);
-- statut: 5 valeurs → filtre mal
-- client_id: 50K valeurs → très sélectif

-- ✅ BON ordre
CREATE INDEX idx_good ON commandes(client_id, statut);
-- client_id en premier : filtrage efficace immédiat
```

### ❌ Piège 3 : INCLUDE au Lieu de Multi-Colonnes

```sql
-- Requête avec filtrage sur 'prix'
SELECT nom, prix FROM produits  
WHERE categorie = 'X' AND prix > 100;  

-- ❌ MAUVAIS : 'prix' dans INCLUDE, ne peut pas filtrer
CREATE INDEX idx_bad ON produits(categorie) INCLUDE (prix, nom);

-- ✅ BON : 'prix' dans index multi-colonnes
CREATE INDEX idx_good ON produits(categorie, prix) INCLUDE (nom);
```

**Règle :** Colonne utilisée dans WHERE, ORDER BY, ou JOIN → **index**, pas INCLUDE.

### ❌ Piège 4 : Trop d'Index Couvrants

```sql
-- ❌ Créer un index covering pour chaque requête
CREATE INDEX idx1 ON table(a) INCLUDE (b, c);  
CREATE INDEX idx2 ON table(a) INCLUDE (b, d);  
CREATE INDEX idx3 ON table(a) INCLUDE (c, d);  
-- Problème : Maintenance coûteuse (3 index sur 'a')

-- ✅ Compromis : Un index multi-colonnes + INCLUDE
CREATE INDEX idx ON table(a, b) INCLUDE (c, d);
-- Couvre plusieurs types de requêtes
```

### ❌ Piège 5 : Oublier VACUUM

```sql
CREATE INDEX idx ON table(col) INCLUDE (col2);

-- Sans VACUUM récent
EXPLAIN ANALYZE SELECT col, col2 FROM table WHERE col = X;
-- Index Only Scan
-- Heap Fetches: 500  ← Doit vérifier la visibilité !

-- Après VACUUM
VACUUM table;  
EXPLAIN ANALYZE SELECT col, col2 FROM table WHERE col = X;  
-- Index Only Scan
-- Heap Fetches: 0  ← Parfait !
```

## Bonnes Pratiques

### ✅ DO (À Faire)

1. **Analyser les requêtes avant de créer des index**
   ```sql
   -- Utiliser pg_stat_statements
   -- Identifier les Heap Fetches élevés
   EXPLAIN (ANALYZE, BUFFERS) SELECT ...;
   ```

2. **INCLUDE pour colonnes légères uniquement**
   ```sql
   -- ✅ Colonnes compactes
   INCLUDE (id, prix, date, statut)

   -- ❌ Colonnes volumineuses
   INCLUDE (description_longue, image_blob)
   ```

3. **Ordre des colonnes selon sélectivité + fréquence**
   ```sql
   -- Analyser la cardinalité
   SELECT
       COUNT(DISTINCT col1) AS card_col1,
       COUNT(DISTINCT col2) AS card_col2
   FROM table;
   ```

4. **Tester l'impact avec EXPLAIN**
   ```sql
   -- Avant
   EXPLAIN (ANALYZE, BUFFERS) SELECT ...;
   -- Créer index
   -- Après
   EXPLAIN (ANALYZE, BUFFERS) SELECT ...;
   -- Comparer les métriques
   ```

5. **Documenter les index complexes**
   ```sql
   COMMENT ON INDEX idx_covering IS
   'Covering index pour requête fréquente client + commandes récentes.
   Réduit Heap Fetches de 1500 à 0 (10× plus rapide).';
   ```

6. **VACUUM régulier pour Index Only Scan efficace**
   ```sql
   -- Configuration autovacuum
   ALTER TABLE table SET (
       autovacuum_vacuum_scale_factor = 0.05,
       autovacuum_analyze_scale_factor = 0.02
   );
   ```

### ❌ DON'T (À Éviter)

1. **Ne pas créer d'index couvrants pour toutes les requêtes**
   - Limite : 2-3 index par table en moyenne

2. **Ne pas INCLUDE des colonnes déjà dans l'index**
   ```sql
   -- ❌ Redondant
   CREATE INDEX idx ON table(a, b) INCLUDE (a, b, c);

   -- ✅ Correct
   CREATE INDEX idx ON table(a, b) INCLUDE (c);
   ```

3. **Ne pas oublier de vérifier l'utilisation réelle**
   ```sql
   -- Index créé mais jamais utilisé ?
   SELECT * FROM pg_stat_user_indexes
   WHERE indexname = 'idx_name' AND idx_scan = 0;
   ```

4. **Ne pas créer des index multi-colonnes avec mauvais ordre**
   - Toujours tester avec EXPLAIN

## Résumé

### Points Clés

#### Index Multi-Colonnes

1. **Index sur plusieurs colonnes simultanément**
   ```sql
   CREATE INDEX idx ON table(col1, col2, col3);
   ```

2. **L'ordre des colonnes est CRUCIAL**
   - Utilisable pour : `col1`, `col1+col2`, `col1+col2+col3`
   - Pas utilisable pour : `col2`, `col3`, `col2+col3`

3. **Règles d'ordre :**
   - Sélectivité (haute → basse)
   - Fréquence d'utilisation
   - Égalité avant plages

4. **Limite pratique : 2-4 colonnes**

#### Clause INCLUDE (Covering Index)

1. **Colonnes supplémentaires dans les feuilles uniquement**
   ```sql
   CREATE INDEX idx ON table(indexed_cols) INCLUDE (covered_cols);
   ```

2. **Avantages :**
   - Index Only Scan (pas de Heap Fetch)
   - Réduction drastique des I/O (3-10×)
   - Index plus compact

3. **Utilisations :**
   - Colonnes dans SELECT mais pas WHERE/ORDER BY
   - Colonnes légères (< 100 octets)
   - Requêtes fréquentes

4. **Condition : VACUUM régulier pour Index Only Scan efficace**

#### Combinaisons

- Multi-colonnes + INCLUDE + Partiel + Expression
- **L'index ultime** pour optimisation maximale
