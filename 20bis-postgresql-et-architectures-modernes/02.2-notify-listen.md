# 20bis.2.2 — NOTIFY/LISTEN pour Événements Temps Réel

## Introduction

Dans le chapitre précédent, nous avons construit un Event Store qui stocke tous les événements de notre système. Mais stocker ne suffit pas : nous devons aussi **réagir** à ces événements en temps réel.

Imaginez un tableau de bord qui affiche les commandes en cours. Avec une approche traditionnelle, l'interface doit interroger la base de données toutes les quelques secondes (polling) pour détecter les nouvelles commandes. C'est inefficace et crée une latence perceptible.

PostgreSQL offre une solution élégante : le mécanisme **NOTIFY/LISTEN**. Il permet aux applications de recevoir des notifications instantanées quand des événements se produisent, sans polling constant.

Ce chapitre vous apprendra à utiliser NOTIFY/LISTEN pour construire des systèmes réactifs et temps réel avec PostgreSQL.

---

## Le Problème du Polling

### L'Approche Traditionnelle

Sans mécanisme de notification, les applications doivent "interroger" régulièrement la base de données :

```
┌─────────────────┐                    ┌─────────────────┐
│   Application   │                    │   PostgreSQL    │
│   (Dashboard)   │                    │                 │
└────────┬────────┘                    └────────┬────────┘
         │                                      │
         │  "Y a-t-il de nouvelles commandes?"  │
         │─────────────────────────────────────►│
         │                                      │
         │  "Non, rien de nouveau"              │
         │◄─────────────────────────────────────│
         │                                      │
         │  ... attendre 5 secondes ...         │
         │                                      │
         │  "Y a-t-il de nouvelles commandes?"  │
         │─────────────────────────────────────►│
         │                                      │
         │  "Non, rien de nouveau"              │
         │◄─────────────────────────────────────│
         │                                      │
         │  ... attendre 5 secondes ...         │
         │                                      │
         │  "Y a-t-il de nouvelles commandes?"  │
         │─────────────────────────────────────►│
         │                                      │
         │  "Oui ! Une nouvelle commande"       │
         │◄─────────────────────────────────────│
         │                                      │
         ▼                                      ▼
```

### Les Inconvénients du Polling

| Problème | Impact |
|----------|--------|
| **Latence** | Délai entre l'événement et sa détection (jusqu'à l'intervalle de polling) |
| **Charge inutile** | Requêtes constantes même sans nouveaux événements |
| **Gaspillage de ressources** | CPU, mémoire, connexions utilisées pour rien |
| **Difficulté à scaler** | Plus de clients = plus de requêtes de polling |
| **Complexité** | Gérer les intervalles, les erreurs, la synchronisation |

## La Solution : NOTIFY/LISTEN

### Principe

PostgreSQL implémente un système de **publication/souscription** (pub/sub) intégré :

- **NOTIFY** : Envoie un message sur un canal nommé  
- **LISTEN** : S'abonne à un canal pour recevoir les messages  
- **UNLISTEN** : Se désabonne d'un canal

```
┌─────────────────┐                    ┌─────────────────┐
│   Application   │                    │   PostgreSQL    │
│   (Dashboard)   │                    │                 │
└────────┬────────┘                    └────────┬────────┘
         │                                      │
         │  LISTEN orders_channel               │
         │─────────────────────────────────────►│
         │                                      │
         │  "OK, vous êtes abonné"              │
         │◄─────────────────────────────────────│
         │                                      │
         │  ... connexion maintenue ouverte ... │
         │  ... aucune requête nécessaire ...   │
         │                                      │
         │                    ┌─────────────────┤
         │                    │ Nouvelle        │
         │                    │ commande !      │
         │                    │ NOTIFY envoyé   │
         │                    └─────────────────┤
         │                                      │
         │  📢 Notification: "order-123 créée"  │
         │◄─────────────────────────────────────│
         │                                      │
         │  Mise à jour immédiate du dashboard  │
         │                                      │
         ▼                                      ▼
```

### Avantages

| Avantage | Description |
|----------|-------------|
| **Temps réel** | Notification instantanée (millisecondes) |
| **Efficacité** | Aucune requête tant qu'il n'y a pas d'événement |
| **Scalabilité** | Un NOTIFY atteint tous les listeners simultanément |
| **Simplicité** | Intégré à PostgreSQL, pas de middleware externe |
| **Transactionnel** | Les notifications respectent les transactions |

## Syntaxe de Base

### LISTEN : S'abonner à un Canal

```sql
-- S'abonner au canal "orders"
LISTEN orders;

-- S'abonner à plusieurs canaux
LISTEN orders;  
LISTEN payments;  
LISTEN inventory;  

-- Les noms de canaux sont insensibles à la casse
LISTEN Orders;  -- Équivalent à LISTEN orders
```

### NOTIFY : Envoyer une Notification

```sql
-- Notification simple (sans payload)
NOTIFY orders;

-- Notification avec payload (message)
NOTIFY orders, 'order-123 created';

-- Le payload est limité à 8000 caractères
NOTIFY orders, '{"order_id": "order-123", "total": 99.99}';
```

### UNLISTEN : Se Désabonner

```sql
-- Se désabonner d'un canal spécifique
UNLISTEN orders;

-- Se désabonner de TOUS les canaux
UNLISTEN *;
```

### pg_notify() : Version Fonctionnelle

La fonction `pg_notify()` permet d'utiliser des variables et expressions :

```sql
-- Équivalent à NOTIFY orders, 'message'
SELECT pg_notify('orders', 'order-123 created');

-- Avec des variables (utile dans les fonctions PL/pgSQL)
DO $$  
DECLARE  
    v_channel TEXT := 'orders';
    v_payload TEXT := 'order-456 shipped';
BEGIN
    PERFORM pg_notify(v_channel, v_payload);
END $$;
```

## Comportement Transactionnel

### Les Notifications Respectent les Transactions

C'est un point crucial : les notifications ne sont envoyées **qu'après le COMMIT** de la transaction.

```sql
BEGIN;
    INSERT INTO orders (id, total) VALUES ('order-789', 150.00);
    NOTIFY orders, 'order-789 created';
    -- La notification est EN ATTENTE, pas encore envoyée
COMMIT;
-- Maintenant la notification est envoyée !
```

Si la transaction est annulée :

```sql
BEGIN;
    INSERT INTO orders (id, total) VALUES ('order-999', 200.00);
    NOTIFY orders, 'order-999 created';
ROLLBACK;
-- La notification n'est JAMAIS envoyée !
```

### Pourquoi C'est Important

Ce comportement garantit la **cohérence** :

- Les listeners ne reçoivent jamais de notification pour des données non commitées
- Pas de "faux positifs" dus à des transactions annulées
- L'ordre des notifications reflète l'ordre des commits

```
Transaction A                    Transaction B
    │                                │
BEGIN                            BEGIN
    │                                │
INSERT order-1                   INSERT order-2  
NOTIFY 'order-1'                 NOTIFY 'order-2'  
    │                                │
    │                            COMMIT  ──► Notification 'order-2' envoyée
    │                                │
COMMIT  ──────────────────────────────► Notification 'order-1' envoyée
    │
    ▼

Les listeners reçoivent : 'order-2' puis 'order-1'
(L'ordre des COMMIT, pas l'ordre des INSERT)
```

## Intégration avec les Triggers

La puissance de NOTIFY/LISTEN se révèle vraiment avec les **triggers**. Chaque modification de données peut automatiquement déclencher une notification.

### Trigger Basique

```sql
-- Fonction trigger pour notifier les nouvelles commandes
CREATE OR REPLACE FUNCTION notify_order_changes()  
RETURNS TRIGGER AS $$  
BEGIN  
    -- Construire le payload JSON
    PERFORM pg_notify(
        'orders',
        json_build_object(
            'action', TG_OP,
            'order_id', NEW.id,
            'total', NEW.total,
            'status', NEW.status,
            'timestamp', NOW()
        )::text
    );

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Créer le trigger sur la table orders
CREATE TRIGGER trg_order_notify  
AFTER INSERT OR UPDATE ON orders  
FOR EACH ROW  
EXECUTE FUNCTION notify_order_changes();  
```

### Trigger Avancé avec Canaux Dynamiques

```sql
-- Notification sur des canaux spécifiques selon le contexte
CREATE OR REPLACE FUNCTION notify_order_changes_advanced()  
RETURNS TRIGGER AS $$  
DECLARE  
    v_payload JSONB;
    v_channel TEXT;
BEGIN
    -- Construire le payload
    v_payload := jsonb_build_object(
        'action', TG_OP,
        'order_id', NEW.id,
        'old_status', CASE WHEN TG_OP = 'UPDATE' THEN OLD.status ELSE NULL END,
        'new_status', NEW.status,
        'total', NEW.total,
        'customer_id', NEW.customer_id,
        'timestamp', NOW()
    );

    -- Canal général pour toutes les commandes
    PERFORM pg_notify('orders', v_payload::text);

    -- Canal spécifique par statut
    v_channel := 'orders_' || NEW.status;
    PERFORM pg_notify(v_channel, v_payload::text);

    -- Canal spécifique par client (pour notifications personnalisées)
    v_channel := 'customer_' || NEW.customer_id;
    PERFORM pg_notify(v_channel, v_payload::text);

    -- Notification spéciale pour les grosses commandes
    IF NEW.total > 1000 THEN
        PERFORM pg_notify('high_value_orders', v_payload::text);
    END IF;

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_order_notify_advanced  
AFTER INSERT OR UPDATE ON orders  
FOR EACH ROW  
EXECUTE FUNCTION notify_order_changes_advanced();  
```

### Trigger pour l'Event Store

Reprenant notre Event Store du chapitre précédent :

```sql
-- Notifier chaque nouvel événement dans l'Event Store
CREATE OR REPLACE FUNCTION notify_new_event()  
RETURNS TRIGGER AS $$  
DECLARE  
    v_payload JSONB;
BEGIN
    -- Payload compact avec les infos essentielles
    v_payload := jsonb_build_object(
        'event_id', NEW.event_id,
        'stream_type', NEW.stream_type,
        'stream_id', NEW.stream_id,
        'stream_version', NEW.stream_version,
        'event_type', NEW.event_type,
        'global_position', NEW.global_position,
        'occurred_at', NEW.occurred_at
    );

    -- Canal global pour tous les événements
    PERFORM pg_notify('events', v_payload::text);

    -- Canal par type de stream (ex: 'events_Order', 'events_Account')
    PERFORM pg_notify('events_' || NEW.stream_type, v_payload::text);

    -- Canal par type d'événement (ex: 'event_OrderPlaced')
    PERFORM pg_notify('event_' || NEW.event_type, v_payload::text);

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_event_store_notify  
AFTER INSERT ON events  
FOR EACH ROW  
EXECUTE FUNCTION notify_new_event();  
```

## Intégration avec les Langages de Programmation

### Python avec psycopg

```python
import psycopg  
import select  
import json  

def listen_for_orders():
    # Connexion à PostgreSQL
    conn = psycopg.connect(
        "host=localhost dbname=myapp user=myuser password=secret",
        autocommit=True  # Important pour LISTEN
    )

    # S'abonner au canal
    conn.execute("LISTEN orders")
    print("En écoute sur le canal 'orders'...")

    # Boucle d'écoute
    while True:
        # Attendre une notification (timeout de 5 secondes)
        if select.select([conn], [], [], 5) == ([], [], []):
            # Timeout, aucune notification
            print(".", end="", flush=True)
        else:
            # Notification reçue
            conn.poll()
            while conn.notifies:
                notify = conn.notifies.pop(0)

                print(f"\nNotification reçue !")
                print(f"  Canal: {notify.channel}")
                print(f"  Payload: {notify.payload}")
                print(f"  PID: {notify.pid}")

                # Parser le JSON si applicable
                try:
                    data = json.loads(notify.payload)
                    print(f"  Données: {data}")
                except json.JSONDecodeError:
                    pass

if __name__ == "__main__":
    listen_for_orders()
```

### Go avec pgx

```go
package main

import (
    "context"
    "encoding/json"
    "fmt"
    "log"

    "github.com/jackc/pgx/v5/pgxpool"
)

type OrderNotification struct {
    Action    string  `json:"action"`
    OrderID   string  `json:"order_id"`
    Total     float64 `json:"total"`
    Status    string  `json:"status"`
    Timestamp string  `json:"timestamp"`
}

func main() {
    ctx := context.Background()

    // Pool de connexions
    pool, err := pgxpool.New(ctx,
        "postgres://myuser:secret@localhost/myapp")
    if err != nil {
        log.Fatal(err)
    }
    defer pool.Close()

    // Acquérir une connexion dédiée pour LISTEN
    conn, err := pool.Acquire(ctx)
    if err != nil {
        log.Fatal(err)
    }
    defer conn.Release()

    // S'abonner
    _, err = conn.Exec(ctx, "LISTEN orders")
    if err != nil {
        log.Fatal(err)
    }

    fmt.Println("En écoute sur le canal 'orders'...")

    // Boucle d'écoute
    for {
        notification, err := conn.Conn().WaitForNotification(ctx)
        if err != nil {
            log.Fatal(err)
        }

        fmt.Printf("Notification sur '%s'\n", notification.Channel)

        var order OrderNotification
        if err := json.Unmarshal(
            []byte(notification.Payload), &order); err == nil {
            fmt.Printf("  Commande: %s, Total: %.2f, Status: %s\n",
                order.OrderID, order.Total, order.Status)
        }
    }
}
```

## Patterns d'Utilisation

### Pattern 1 : Mise à Jour de Cache

Invalider ou mettre à jour un cache quand les données changent :

```sql
-- Trigger pour notifier les changements de produits
CREATE OR REPLACE FUNCTION notify_product_change()  
RETURNS TRIGGER AS $$  
BEGIN  
    PERFORM pg_notify(
        'cache_invalidation',
        json_build_object(
            'type', 'product',
            'action', TG_OP,
            'id', COALESCE(NEW.id, OLD.id),
            'cache_key', 'product:' || COALESCE(NEW.id, OLD.id)
        )::text
    );
    RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_product_cache  
AFTER INSERT OR UPDATE OR DELETE ON products  
FOR EACH ROW  
EXECUTE FUNCTION notify_product_change();  
```

```python
# Application : Écouter et invalider le cache Redis
```

### Pattern 2 : Dashboard Temps Réel

```sql
-- Trigger pour les métriques en temps réel
CREATE OR REPLACE FUNCTION notify_metrics()  
RETURNS TRIGGER AS $$  
BEGIN  
    -- Calculer les métriques agrégées
    PERFORM pg_notify(
        'dashboard_metrics',
        (SELECT json_build_object(
            'timestamp', NOW(),
            'orders_today', COUNT(*) FILTER (
                WHERE created_at::date = CURRENT_DATE
            ),
            'revenue_today', SUM(total) FILTER (
                WHERE created_at::date = CURRENT_DATE
            ),
            'pending_orders', COUNT(*) FILTER (
                WHERE status = 'pending'
            )
        ) FROM orders)::text
    );
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Déclencher à chaque changement de commande
CREATE TRIGGER trg_dashboard_metrics  
AFTER INSERT OR UPDATE ON orders  
FOR EACH STATEMENT  -- Une seule notification par statement
```


### Pattern 4 : File d'Attente Simple (Job Queue)

```sql
-- Table de jobs
CREATE TABLE jobs (
    id SERIAL PRIMARY KEY,
    job_type VARCHAR(100) NOT NULL,
    payload JSONB NOT NULL,
    status VARCHAR(20) DEFAULT 'pending',
    created_at TIMESTAMPTZ DEFAULT NOW(),
    started_at TIMESTAMPTZ,
    completed_at TIMESTAMPTZ
);

-- Notifier les workers quand un nouveau job arrive
CREATE OR REPLACE FUNCTION notify_new_job()  
RETURNS TRIGGER AS $$  
BEGIN  
    PERFORM pg_notify(
        'jobs_' || NEW.job_type,
        json_build_object(
            'job_id', NEW.id,
            'job_type', NEW.job_type,
            'created_at', NEW.created_at
        )::text
    );

    -- Aussi sur un canal général
    PERFORM pg_notify('jobs', NEW.id::text);

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_job_notify  
AFTER INSERT ON jobs  
FOR EACH ROW  
WHEN (NEW.status = 'pending')  
EXECUTE FUNCTION notify_new_job();  
```

```python
# Worker qui traite les jobs
async def job_worker(job_type: str):
    await conn.execute(f"LISTEN jobs_{job_type}")

    async for notify in conn.notifies():
        data = json.loads(notify.payload)
        job_id = data['job_id']

        # Réclamer le job (avec verrouillage)
        result = await conn.fetchrow("""
            UPDATE jobs
            SET status = 'processing', started_at = NOW()
            WHERE id = $1 AND status = 'pending'
            RETURNING *
        """, job_id)

        if result:
            print(f"Traitement du job {job_id}...")

            await conn.execute("""
                UPDATE jobs
                SET status = 'completed', completed_at = NOW()
                WHERE id = $1
            """, job_id)
```

## Architecture WebSocket avec NOTIFY/LISTEN

Un cas d'usage courant est de pousser les notifications PostgreSQL vers des clients web via WebSocket.

### Architecture

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐               │
│  │ Browser │    │ Browser │    │ Browser │    │ Browser │   Clients     │
│  └────┬────┘    └────┬────┘    └────┬────┘    └────┬────┘   Web         │
│       │              │              │              │                    │
│       └──────────────┴──────────────┴──────────────┘                    │
│                              │                                          │
│                              │ WebSocket                                │
│                              ▼                                          │
│                    ┌──────────────────┐                                 │
│                    │   WebSocket      │                                 │
│                    │   Server         │   Application                   │
│                    │   (Node/Python)  │   Backend                       │
│                    └────────┬─────────┘                                 │
│                             │                                           │
│                             │ LISTEN                                    │
│                             ▼                                           │
│                    ┌──────────────────┐                                 │
│                    │   PostgreSQL     │                                 │
│                    │   NOTIFY/LISTEN  │                                 │
│                    └──────────────────┘                                 │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### Exemple avec Node.js et Socket.io

```javascript
const { Server } = require('socket.io');  
const { Client } = require('pg');  
const http = require('http');  

// Serveur HTTP et WebSocket
const server = http.createServer();  
const io = new Server(server, {  
    cors: { origin: "*" }
});

// Connexion PostgreSQL pour LISTEN
const pgClient = new Client({
    connectionString: 'postgres://user:pass@localhost/myapp'
});

async function start() {
    // Connecter à PostgreSQL
    await pgClient.connect();

    // S'abonner aux canaux
    await pgClient.query('LISTEN orders');
    await pgClient.query('LISTEN events');

    // Relayer les notifications vers les WebSockets
    pgClient.on('notification', (msg) => {
        const data = JSON.parse(msg.payload);

        // Émettre à tous les clients connectés
        io.emit(msg.channel, data);

        // Ou à une room spécifique (ex: par customer_id)
        if (data.customer_id) {
            io.to(`customer:${data.customer_id}`).emit(msg.channel, data);
        }
    });

    // Gestion des connexions WebSocket
    io.on('connection', (socket) => {
        console.log('Client connecté:', socket.id);

        // Permettre aux clients de rejoindre des rooms
        socket.on('subscribe', (room) => {
            socket.join(room);
            console.log(`${socket.id} a rejoint ${room}`);
        });

        socket.on('disconnect', () => {
            console.log('Client déconnecté:', socket.id);
        });
    });

    server.listen(3000, () => {
        console.log('WebSocket server sur le port 3000');
    });
}

start().catch(console.error);
```

```javascript
// Connexion WebSocket
const socket = io('http://localhost:3000');

// S'abonner à ses propres notifications
socket.emit('subscribe', `customer:${customerId}`);

// Écouter les événements
socket.on('orders', (data) => {
    console.log('Nouvelle activité commande:', data);

    if (data.action === 'INSERT') {
        addOrderToUI(data);
    } else if (data.action === 'UPDATE') {
        updateOrderInUI(data);
    }
});

socket.on('events', (data) => {
    console.log('Nouvel événement:', data);
    appendToEventLog(data);
});

// Mise à jour de l'UI
function addOrderToUI(order) {
    const list = document.getElementById('orders-list');
    const item = document.createElement('li');
    item.textContent = `Commande #${order.order_id} - ${order.total}€`;
    list.prepend(item);
}
```

## Limites et Considérations

### Limites Techniques

| Limite | Description |
|--------|-------------|
| **Taille du payload** | Maximum 8000 caractères par notification |
| **Pas de persistance** | Les notifications sont perdues si personne n'écoute |
| **Pas de garantie de livraison** | Fire-and-forget, pas d'accusé de réception |
| **File d'attente limitée** | ~8GB par défaut, peut saturer sous forte charge |
| **Même serveur PostgreSQL** | LISTEN ne traverse pas les serveurs |

### Quand NE PAS Utiliser NOTIFY/LISTEN

| Situation | Alternative |
|-----------|-------------|
| Messages critiques ne devant pas être perdus | RabbitMQ, Kafka, table Outbox |
| Très haute fréquence (10k+ msg/sec) | Message broker dédié |
| Communication entre datacenters | Kafka, NATS, Redis Streams |
| Payloads volumineux (> 8KB) | Stocker en DB, notifier l'ID |
| Besoin de replay/historique | Event Store + LISTEN pour le temps réel |

### Bonnes Pratiques

#### 1. Garder les Payloads Petits

```sql
-- ✅ Bon : Notifier seulement l'ID, charger les détails séparément
PERFORM pg_notify('orders', json_build_object(
    'action', 'INSERT',
    'id', NEW.id
)::text);

-- ❌ Éviter : Gros payload avec toutes les données
PERFORM pg_notify('orders', row_to_json(NEW)::text);  -- Peut dépasser 8KB
```

#### 2. Nommer les Canaux de Façon Cohérente

```sql
-- Convention suggérée
'entity_action'          -- ex: order_created, user_updated
'category.subcategory'   -- ex: events.orders, cache.products
'entity:id'              -- ex: customer:123 (pour ciblage précis)
```

#### 3. Gérer les Déconnexions

```python
async def resilient_listener():
    while True:
        try:
            conn = await psycopg.AsyncConnection.connect(...)
            await conn.execute("LISTEN orders")

            async for notify in conn.notifies():
                await handle_notification(notify)

        except Exception as e:
            print(f"Erreur: {e}, reconnexion dans 5s...")
            await asyncio.sleep(5)
```

#### 4. Combiner avec l'Event Store

Pour les événements critiques, utilisez l'Event Store comme source de vérité et NOTIFY/LISTEN comme accélérateur :

```sql
-- L'événement est TOUJOURS stocké (source de vérité)
INSERT INTO events (...) VALUES (...);

-- La notification est un BONUS pour le temps réel
-- Si elle est perdue, l'événement reste dans le store
```

## Monitoring des Notifications

### Vérifier l'Utilisation de la Queue

```sql
-- Pourcentage d'utilisation de la queue de notifications
SELECT pg_notification_queue_usage();

-- Si > 0.5 (50%), risque de saturation
-- Alerter si > 0.8 (80%)
```

### Comptage des Listeners Actifs

```sql
-- Voir toutes les sessions en écoute
SELECT
    pid,
    usename,
    application_name,
    client_addr,
    state,
    query
FROM pg_stat_activity  
WHERE query LIKE '%LISTEN%'  
   OR wait_event_type = 'Client';
```

### Logs des Notifications

Activer le logging pour débugger :

```sql
-- Dans postgresql.conf
-- log_statement = 'all'  -- Attention, très verbeux

-- Ou pour une session
SET log_statement = 'all';  
NOTIFY test_channel, 'debug message';  
```

## Conclusion

Le mécanisme **NOTIFY/LISTEN** de PostgreSQL est un outil puissant pour construire des applications réactives et temps réel. Il élimine le besoin de polling constant et permet une communication instantanée entre la base de données et les applications.

Combiné avec les triggers et l'Event Store, NOTIFY/LISTEN permet de :

- Mettre à jour des dashboards en temps réel
- Invalider des caches instantanément
- Synchroniser des applications distribuées
- Alimenter des WebSockets vers les navigateurs

Bien que limité pour les cas d'usage nécessitant une garantie de livraison ou une très haute fréquence, NOTIFY/LISTEN reste la solution idéale pour de nombreux scénarios temps réel, sans ajouter de dépendance externe à votre stack PostgreSQL.

---

## Points Clés à Retenir

- **NOTIFY** : Envoie une notification sur un canal  
- **LISTEN** : S'abonne pour recevoir les notifications d'un canal  
- **Transactionnel** : Les notifications ne sont envoyées qu'après COMMIT  
- **Triggers** : Automatisent les notifications lors des modifications de données  
- **Payload** : Maximum 8000 caractères, préférer les IDs aux données complètes  
- **Pas de persistance** : Utilisez un Event Store pour les événements critiques  
- **WebSocket** : Idéal pour pousser les notifications vers les navigateurs  
- **Monitoring** : Surveillez `pg_notification_queue_usage()`
