# 20bis.1.3 — Foreign Data Wrappers pour la Fédération

## Introduction

Dans les chapitres précédents, nous avons vu que l'approche **Database per Service** isole les données de chaque microservice. Cette isolation apporte autonomie et scalabilité, mais crée un nouveau défi : **comment interroger des données réparties sur plusieurs bases de données ?**

Imaginons un responsable commercial qui souhaite un rapport croisant les données clients (base `users_db`), les commandes (base `orders_db`) et les produits (base `catalog_db`). Avec des bases séparées, impossible d'écrire une simple jointure SQL.

C'est ici qu'interviennent les **Foreign Data Wrappers (FDW)** de PostgreSQL. Cette fonctionnalité puissante permet d'accéder à des données externes — qu'elles soient dans une autre base PostgreSQL, un fichier CSV, MySQL, Oracle, ou même une API REST — **comme s'il s'agissait de tables locales**.

Ce chapitre vous apprendra à utiliser les FDW pour créer une **fédération de données** et répondre aux besoins de reporting et d'intégration dans une architecture microservices.

---

## Qu'est-ce qu'un Foreign Data Wrapper ?

### Définition

Un **Foreign Data Wrapper** (littéralement "enveloppe de données étrangères") est un mécanisme PostgreSQL qui permet de :

- Se connecter à des sources de données externes
- Créer des **tables étrangères** (foreign tables) qui représentent ces données
- Interroger ces tables avec du SQL standard
- Dans certains cas, modifier les données distantes

### Analogie Simple

Pensez aux FDW comme à des **traducteurs universels**. Vous parlez SQL à PostgreSQL, et le FDW traduit vos requêtes dans le langage compris par la source externe (protocole PostgreSQL, MySQL, fichier CSV, API HTTP, etc.).

```
┌─────────────────────────────────────────────────────────────┐
│                    PostgreSQL Local                         │
│                                                             │
│   SELECT * FROM foreign_users                               │
│         │                                                   │
│         ▼                                                   │
│   ┌───────────────┐                                         │
│   │ Foreign Data  │                                         │
│   │   Wrapper     │  ◄── Traduit SQL vers protocole cible   │
│   └───────┬───────┘                                         │
└───────────┼─────────────────────────────────────────────────┘
            │
            │  Connexion réseau
            ▼
┌─────────────────────────────────────────────────────────────┐
│              Source de Données Externe                      │
│         (PostgreSQL, MySQL, fichier, API, etc.)             │
└─────────────────────────────────────────────────────────────┘
```

### Le Standard SQL/MED

Les FDW implémentent le standard **SQL/MED** (Management of External Data), une extension du standard SQL qui définit comment accéder à des données externes. PostgreSQL a été un pionnier dans l'implémentation de ce standard.

## Architecture des Foreign Data Wrappers

### Les Composants

L'utilisation des FDW implique plusieurs objets PostgreSQL :

```
┌─────────────────────────────────────────────────────────────┐
│                    Hiérarchie FDW                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. EXTENSION          ──► Le code du wrapper               │
│        │                   (postgres_fdw, file_fdw, etc.)   │
│        ▼                                                    │
│  2. FOREIGN DATA       ──► Instance du wrapper configurée   │
│     WRAPPER                                                 │
│        │                                                    │
│        ▼                                                    │
│  3. FOREIGN SERVER     ──► Connexion vers une source        │
│        │                   (hôte, port, base de données)    │
│        ▼                                                    │
│  4. USER MAPPING       ──► Credentials pour se connecter    │
│        │                   (utilisateur/mot de passe)       │
│        ▼                                                    │
│  5. FOREIGN TABLE      ──► Représentation locale d'une      │
│                            table distante                   │
└─────────────────────────────────────────────────────────────┘
```

### Explication de Chaque Composant

| Composant | Rôle | Analogie |
|-----------|------|----------|
| **Extension** | Code qui sait communiquer avec un type de source | Le pilote d'imprimante |
| **Foreign Data Wrapper** | Instance configurée de l'extension | L'imprimante installée |
| **Foreign Server** | Adresse et paramètres de la source | L'adresse IP de l'imprimante |
| **User Mapping** | Identifiants de connexion | Login/mot de passe pour l'imprimante |
| **Foreign Table** | Vue locale sur les données distantes | Le document à imprimer |

## postgres_fdw : Fédération entre Bases PostgreSQL

Le cas d'usage le plus courant dans une architecture microservices est la fédération entre plusieurs bases PostgreSQL. L'extension **postgres_fdw** est incluse dans PostgreSQL et optimisée pour ce scénario.

### Scénario : Base de Reporting Centralisée

Imaginons trois microservices avec leurs bases dédiées :

```
┌─────────────┐     ┌─────────────┐     ┌──────────────┐
│  users_db   │     │  orders_db  │     │ catalog_db   │
│             │     │             │     │              │
│ - users     │     │ - orders    │     │ - products   │
│ - profiles  │     │ - items     │     │ - categories │
└─────────────┘     └─────────────┘     └──────────────┘
       │                   │                   │
       └───────────────────┼───────────────────┘
                           │
                           ▼
                  ┌─────────────────┐
                  │  reporting_db   │
                  │                 │
                  │ Tables locales: │
                  │ - reports       │
                  │ - dashboards    │
                  │                 │
                  │ Tables foreign: │
                  │ - fdw_users     │
                  │ - fdw_orders    │
                  │ - fdw_products  │
                  └─────────────────┘
```

### Étape 1 : Installer l'Extension

Connectez-vous à la base de reporting et installez l'extension :

```sql
-- Sur la base reporting_db
CREATE EXTENSION IF NOT EXISTS postgres_fdw;
```

Cette commande charge le code nécessaire pour communiquer avec d'autres bases PostgreSQL.

### Étape 2 : Créer le Foreign Data Wrapper

```sql
-- Généralement créé automatiquement avec l'extension,
-- mais on peut le faire explicitement :
CREATE FOREIGN DATA WRAPPER postgres_fdw_wrapper
    HANDLER postgres_fdw_handler
    VALIDATOR postgres_fdw_validator;
```

En pratique, l'extension `postgres_fdw` crée automatiquement un wrapper nommé `postgres_fdw`, donc cette étape est souvent implicite.

### Étape 3 : Définir les Serveurs Distants

Créez un serveur pour chaque base de données source :

```sql
-- Serveur pour la base users_db
CREATE SERVER users_server
    FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (
        host 'users-db.internal',      -- Hostname ou IP
        port '5432',                    -- Port PostgreSQL
        dbname 'users_db'               -- Nom de la base
    );

-- Serveur pour la base orders_db
CREATE SERVER orders_server
    FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (
        host 'orders-db.internal',
        port '5432',
        dbname 'orders_db'
    );

-- Serveur pour la base catalog_db
CREATE SERVER catalog_server
    FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (
        host 'catalog-db.internal',
        port '5432',
        dbname 'catalog_db'
    );
```

### Étape 4 : Configurer les User Mappings

Les user mappings associent un utilisateur local à des credentials sur le serveur distant :

```sql
-- Mapping pour l'utilisateur reporting_user vers users_server
CREATE USER MAPPING FOR reporting_user
    SERVER users_server
    OPTIONS (
        user 'readonly_user',           -- Utilisateur sur la base distante
        password 'secure_password_123'  -- Mot de passe
    );

-- Mapping pour orders_server
CREATE USER MAPPING FOR reporting_user
    SERVER orders_server
    OPTIONS (
        user 'readonly_user',
        password 'secure_password_456'
    );

-- Mapping pour catalog_server
CREATE USER MAPPING FOR reporting_user
    SERVER catalog_server
    OPTIONS (
        user 'readonly_user',
        password 'secure_password_789'
    );
```

**Bonne pratique de sécurité** : Utilisez des comptes en lecture seule (`readonly_user`) sur les bases sources pour éviter toute modification accidentelle.

### Étape 5 : Créer les Tables Étrangères

#### Méthode Manuelle

Définissez chaque table étrangère explicitement :

```sql
-- Table étrangère pour les utilisateurs
CREATE FOREIGN TABLE fdw_users (
    id              INTEGER,
    email           VARCHAR(255),
    name            VARCHAR(100),
    created_at      TIMESTAMPTZ,
    country         VARCHAR(2)
)
SERVER users_server  
OPTIONS (  
    schema_name 'public',      -- Schéma sur la base distante
    table_name 'users'         -- Nom de la table distante
);

-- Table étrangère pour les commandes
CREATE FOREIGN TABLE fdw_orders (
    id              INTEGER,
    user_id         INTEGER,
    total           NUMERIC(10,2),
    status          VARCHAR(20),
    created_at      TIMESTAMPTZ
)
SERVER orders_server  
OPTIONS (  
    schema_name 'public',
    table_name 'orders'
);

-- Table étrangère pour les produits
CREATE FOREIGN TABLE fdw_products (
    id              INTEGER,
    name            VARCHAR(255),
    category_id     INTEGER,
    price           NUMERIC(10,2),
    stock           INTEGER
)
SERVER catalog_server  
OPTIONS (  
    schema_name 'public',
    table_name 'products'
);
```

#### Méthode Automatique : IMPORT FOREIGN SCHEMA

Pour importer automatiquement toutes les tables d'un schéma distant :

```sql
-- Créer un schéma local pour organiser les tables importées
CREATE SCHEMA users_remote;

-- Importer toutes les tables du schéma public distant
IMPORT FOREIGN SCHEMA public
    FROM SERVER users_server
    INTO users_remote;

-- Importer seulement certaines tables
IMPORT FOREIGN SCHEMA public
    LIMIT TO (orders, order_items)
    FROM SERVER orders_server
    INTO orders_remote;

-- Exclure certaines tables
IMPORT FOREIGN SCHEMA public
    EXCEPT (internal_logs, temp_data)
    FROM SERVER catalog_server
    INTO catalog_remote;
```

### Étape 6 : Interroger les Données Fédérées

Une fois configurées, les tables étrangères s'utilisent comme des tables normales :

```sql
-- Requête simple sur une table distante
SELECT id, email, name  
FROM fdw_users  
WHERE country = 'FR';  

-- Jointure entre tables de différentes bases !
SELECT
    u.name AS customer_name,
    u.email,
    o.id AS order_id,
    o.total,
    o.created_at AS order_date
FROM fdw_users u  
JOIN fdw_orders o ON o.user_id = u.id  
WHERE o.created_at > NOW() - INTERVAL '30 days'  
ORDER BY o.total DESC  
LIMIT 100;  

-- Jointure complexe avec trois sources
SELECT
    u.name AS customer,
    p.name AS product,
    oi.quantity,
    oi.unit_price,
    o.created_at
FROM fdw_users u  
JOIN fdw_orders o ON o.user_id = u.id  
JOIN fdw_order_items oi ON oi.order_id = o.id  
JOIN fdw_products p ON p.id = oi.product_id  
WHERE o.status = 'completed'  
AND o.created_at BETWEEN '2025-01-01' AND '2025-03-31';  
```

## Optimisation des Performances

### Le Pushdown : Déléguer le Travail

L'un des avantages majeurs de `postgres_fdw` est le **pushdown** : PostgreSQL envoie certaines opérations au serveur distant au lieu de tout rapatrier localement.

#### Types de Pushdown Supportés

| Opération | Pushdown | Description |
|-----------|----------|-------------|
| **WHERE** | ✅ Oui | Les filtres sont envoyés au serveur distant |
| **ORDER BY** | ✅ Oui | Le tri est effectué à distance |
| **LIMIT** | ✅ Oui | Seules les lignes nécessaires sont transférées |
| **JOIN** | ⚠️ Partiel | Jointures entre tables du même serveur |
| **Agrégations** | ✅ Oui (PG 10+) | COUNT, SUM, AVG exécutés à distance |
| **UPDATE/DELETE** | ✅ Oui | Modifications possibles si configuré |

#### Exemple de Pushdown

```sql
-- Cette requête...
SELECT * FROM fdw_orders  
WHERE status = 'pending'  
AND created_at > '2025-01-01'  
ORDER BY created_at DESC  
LIMIT 10;  

-- ...est transformée et envoyée au serveur distant comme :
-- SELECT * FROM orders
-- WHERE status = 'pending'
-- AND created_at > '2025-01-01'
-- ORDER BY created_at DESC
-- LIMIT 10;

-- Seules 10 lignes traversent le réseau !
```

### Vérifier le Pushdown avec EXPLAIN

Utilisez `EXPLAIN VERBOSE` pour voir ce qui est envoyé au serveur distant :

```sql
EXPLAIN VERBOSE  
SELECT * FROM fdw_orders  
WHERE status = 'pending'  
LIMIT 10;  
```

Résultat (simplifié) :

```
Foreign Scan on fdw_orders
  Output: id, user_id, total, status, created_at
  Remote SQL: SELECT id, user_id, total, status, created_at
              FROM public.orders
              WHERE (status = 'pending')
              LIMIT 10
```

La ligne `Remote SQL` montre exactement la requête envoyée au serveur distant.

### Options d'Optimisation du Serveur

```sql
-- Activer le pushdown des agrégations et jointures
ALTER SERVER orders_server OPTIONS (
    ADD fetch_size '1000',           -- Lignes par batch (défaut: 100)
    ADD use_remote_estimate 'true',  -- Utiliser les stats distantes
    ADD fdw_startup_cost '100',      -- Coût de démarrage estimé
    ADD fdw_tuple_cost '0.01'        -- Coût par ligne estimé
);
```

| Option | Description |
|--------|-------------|
| `fetch_size` | Nombre de lignes récupérées par batch réseau |
| `use_remote_estimate` | Utilise `EXPLAIN` distant pour estimer les coûts |
| `fdw_startup_cost` | Coût fixe d'initialisation de la connexion |
| `fdw_tuple_cost` | Coût par ligne transférée |

### Stratégies pour Améliorer les Performances

#### 1. Créer des Index sur les Tables Sources

Les FDW bénéficient des index sur les bases distantes :

```sql
-- Sur la base orders_db (source)
CREATE INDEX idx_orders_status ON orders(status);  
CREATE INDEX idx_orders_user_id ON orders(user_id);  
CREATE INDEX idx_orders_created_at ON orders(created_at);  
```

#### 2. Utiliser des Vues Matérialisées pour le Cache

Pour les données qui ne changent pas souvent :

```sql
-- Créer une vue matérialisée des données fédérées
CREATE MATERIALIZED VIEW mv_sales_summary AS  
SELECT  
    date_trunc('day', o.created_at) AS day,
    COUNT(*) AS order_count,
    SUM(o.total) AS revenue,
    COUNT(DISTINCT o.user_id) AS unique_customers
FROM fdw_orders o  
WHERE o.status = 'completed'  
GROUP BY date_trunc('day', o.created_at);  

-- Index pour accélérer les requêtes sur la vue
CREATE INDEX idx_mv_sales_day ON mv_sales_summary(day);

-- Rafraîchir périodiquement (via pg_cron par exemple)
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_sales_summary;
```

#### 3. Limiter les Colonnes Sélectionnées

```sql
-- ❌ Éviter : transfère toutes les colonnes
SELECT * FROM fdw_orders WHERE status = 'pending';

-- ✅ Préférer : ne transfère que le nécessaire
SELECT id, user_id, total  
FROM fdw_orders  
WHERE status = 'pending';  
```

#### 4. Filtrer au Maximum

```sql
-- ❌ Filtre côté local (après transfert)
SELECT * FROM fdw_orders o  
WHERE o.id IN (SELECT order_id FROM local_flagged_orders);  

-- ✅ Filtre pushé vers le distant si possible
SELECT * FROM fdw_orders o  
WHERE o.status = 'pending'  
AND o.created_at > NOW() - INTERVAL '7 days';  
```

## Autres Foreign Data Wrappers Utiles

PostgreSQL dispose d'un écosystème riche de FDW pour se connecter à diverses sources.

### file_fdw : Fichiers CSV et Texte

Inclus dans PostgreSQL, permet de lire des fichiers comme des tables.

```sql
-- Installer l'extension
CREATE EXTENSION file_fdw;

-- Créer le serveur (local)
CREATE SERVER file_server FOREIGN DATA WRAPPER file_fdw;

-- Table étrangère pour un fichier CSV
CREATE FOREIGN TABLE import_products (
    sku         VARCHAR(50),
    name        VARCHAR(255),
    price       NUMERIC(10,2),
    category    VARCHAR(100)
)
SERVER file_server  
OPTIONS (  
    filename '/data/imports/products.csv',
    format 'csv',
    header 'true',
    delimiter ','
);

-- Interroger le fichier comme une table
SELECT * FROM import_products WHERE price > 100;
```

**Cas d'usage** : Import de données, intégration de fichiers de configuration, logs.

### mysql_fdw : Connexion à MySQL/MariaDB

Extension communautaire pour accéder à des bases MySQL.

```sql
-- Installer (nécessite compilation ou package)
CREATE EXTENSION mysql_fdw;

-- Serveur MySQL
CREATE SERVER mysql_server
    FOREIGN DATA WRAPPER mysql_fdw
    OPTIONS (
        host 'mysql.internal',
        port '3306'
    );

-- User mapping
CREATE USER MAPPING FOR reporting_user
    SERVER mysql_server
    OPTIONS (
        username 'mysql_readonly',
        password 'mysql_password'
    );

-- Table étrangère
CREATE FOREIGN TABLE legacy_customers (
    id          INTEGER,
    name        VARCHAR(255),
    email       VARCHAR(255)
)
SERVER mysql_server  
OPTIONS (  
    dbname 'legacy_db',
    table_name 'customers'
);
```

**Cas d'usage** : Migration progressive de MySQL vers PostgreSQL, fédération avec systèmes legacy.

### oracle_fdw : Connexion à Oracle

Pour les entreprises ayant des systèmes Oracle existants.

```sql
CREATE EXTENSION oracle_fdw;

CREATE SERVER oracle_server
    FOREIGN DATA WRAPPER oracle_fdw
    OPTIONS (dbserver '//oracle.internal:1521/ORCL');

CREATE USER MAPPING FOR reporting_user
    SERVER oracle_server
    OPTIONS (user 'oracle_user', password 'oracle_pwd');

CREATE FOREIGN TABLE oracle_invoices (
    invoice_id      INTEGER,
    customer_id     INTEGER,
    amount          NUMERIC(15,2),
    invoice_date    DATE
)
SERVER oracle_server  
OPTIONS (schema 'BILLING', table 'INVOICES');  
```

### mongo_fdw : Connexion à MongoDB

Pour intégrer des données NoSQL.

```sql
CREATE EXTENSION mongo_fdw;

CREATE SERVER mongo_server
    FOREIGN DATA WRAPPER mongo_fdw
    OPTIONS (address 'mongo.internal', port '27017');

CREATE USER MAPPING FOR reporting_user
    SERVER mongo_server
    OPTIONS (username 'mongo_user', password 'mongo_pwd');

CREATE FOREIGN TABLE mongo_events (
    _id             NAME,
    event_type      TEXT,
    payload         JSONB,
    timestamp       TIMESTAMPTZ
)
SERVER mongo_server  
OPTIONS (database 'events_db', collection 'events');  
```

## Gestion des Écritures Distantes

Par défaut, les tables étrangères sont en lecture seule. Mais `postgres_fdw` supporte les écritures si configuré.

### Activer les Écritures

```sql
-- Sur la table étrangère
ALTER FOREIGN TABLE fdw_orders
    OPTIONS (ADD updatable 'true');
```

### Exemples d'Écritures

```sql
-- Insertion
INSERT INTO fdw_orders (user_id, total, status)  
VALUES (42, 99.99, 'pending');  

-- Mise à jour
UPDATE fdw_orders  
SET status = 'shipped'  
WHERE id = 1001;  

-- Suppression
DELETE FROM fdw_orders  
WHERE status = 'cancelled'  
AND created_at < NOW() - INTERVAL '1 year';  
```

### Avertissements sur les Écritures

| Risque | Description |
|--------|-------------|
| **Pas de transaction distribuée** | L'écriture locale et distante ne sont pas atomiques |
| **Performance** | Les écritures distantes sont plus lentes |
| **Couplage** | Crée une dépendance directe vers le service distant |
| **Sécurité** | Nécessite des droits d'écriture sur la base source |

**Recommandation** : Dans une architecture microservices, préférez les **APIs ou événements** pour modifier les données d'un autre service. Réservez les FDW à la **lecture** pour le reporting.

## Patterns d'Architecture avec FDW

### Pattern 1 : Base de Reporting Dédiée

```
┌──────────┐ ┌──────────┐ ┌──────────┐
│Service A │ │Service B │ │Service C │
│  (OLTP)  │ │  (OLTP)  │ │  (OLTP)  │
└────┬─────┘ └────┬─────┘ └────┬─────┘
     │            │            │
     └────────────┼────────────┘
                  │ FDW (lecture seule)
                  ▼
         ┌────────────────┐
         │  Reporting DB  │
         │    (OLAP)      │
         │                │
         │ - Vues matér.  │
         │ - Agrégations  │
         │ - Dashboards   │
         └────────────────┘
```

**Avantages** :
- Isolation des requêtes analytiques lourdes
- Pas d'impact sur les bases OLTP
- Consolidation des données pour BI

### Pattern 2 : Migration Progressive

Lors d'une migration de monolithe vers microservices :

```
Phase 1: Monolithe avec toutes les données
         ┌─────────────────────────────┐
         │        monolith_db          │
         │  users | orders | products  │
         └─────────────────────────────┘

Phase 2: Extraction du service Users
         ┌─────────────┐
         │  users_db   │◄─── Nouveau service
         │   (users)   │
         └──────┬──────┘
                │ FDW
                ▼
         ┌────────────────────────────────┐
         │        monolith_db             │
         │  fdw_users | orders | products │
         └────────────────────────────────┘
         Le monolithe accède aux users via FDW

Phase 3: Tous les services extraits
         ┌─────────┐ ┌──────────┐ ┌─────────┐
         │users_db │ │orders_db │ │catalog_ │
         └─────────┘ └──────────┘ │   db    │
                                  └─────────┘
```

## Sécurité des Foreign Data Wrappers

### Bonnes Pratiques

#### 1. Utilisateurs en Lecture Seule

Créez des comptes dédiés sur les bases sources :

```sql
-- Sur la base source (orders_db)
CREATE ROLE fdw_readonly WITH LOGIN PASSWORD 'strong_password';

-- Donner uniquement les droits SELECT
GRANT CONNECT ON DATABASE orders_db TO fdw_readonly;  
GRANT USAGE ON SCHEMA public TO fdw_readonly;  
GRANT SELECT ON ALL TABLES IN SCHEMA public TO fdw_readonly;  

-- Pour les futures tables
ALTER DEFAULT PRIVILEGES IN SCHEMA public
    GRANT SELECT ON TABLES TO fdw_readonly;
```

#### 2. Chiffrement des Connexions

Activez SSL pour les connexions FDW :

```sql
CREATE SERVER secure_server
    FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (
        host 'secure-db.internal',
        port '5432',
        dbname 'secure_db',
        sslmode 'require',           -- Force SSL
        sslrootcert '/path/to/ca.crt'
    );
```

#### 3. Limiter les Tables Exposées

Ne créez des tables étrangères que pour les données nécessaires :

```sql
-- ✅ Créer uniquement les tables nécessaires au reporting
CREATE FOREIGN TABLE fdw_orders_summary (
    id          INTEGER,
    total       NUMERIC,
    status      VARCHAR,
    created_at  TIMESTAMPTZ
    -- Pas de colonnes sensibles comme payment_details
)
SERVER orders_server  
OPTIONS (schema_name 'public', table_name 'orders');  
```

#### 4. Stocker les Mots de Passe de Façon Sécurisée

Les mots de passe des user mappings sont stockés dans `pg_user_mappings`. Limitez l'accès :

```sql
-- Seul le superuser peut voir les mots de passe
REVOKE ALL ON pg_user_mappings FROM PUBLIC;
```

Pour des environnements très sensibles, utilisez l'authentification par certificat ou des solutions comme **HashiCorp Vault** pour la rotation des credentials.

## Limitations et Considérations

### Limitations Techniques

| Limitation | Description |
|------------|-------------|
| **Pas de transactions distribuées** | Chaque opération distante est une transaction séparée |
| **Latence réseau** | Les requêtes impliquent des allers-retours réseau |
| **Jointures cross-server** | Jointures entre tables de serveurs différents = tout rapatrié localement |
| **Pas de triggers distants** | Les triggers de la table source ne sont pas visibles |
| **Stats limitées** | `ANALYZE` sur tables étrangères moins précis |

### Quand NE PAS Utiliser les FDW

| Scénario | Alternative recommandée |
|----------|------------------------|
| Requêtes temps réel haute fréquence | API directe ou cache Redis |
| Écritures massives | API du service ou événements |
| Données très volumineuses | ETL vers data warehouse |
| Jointures complexes multi-serveurs | Data warehouse consolidé |
| Haute disponibilité critique | Réplication native |

### Dépannage Courant

#### Erreur de Connexion

```
ERROR: could not connect to server "orders_server"  
DETAIL: connection to server at "orders-db.internal", port 5432 failed  
```

**Solutions** :
- Vérifier que le serveur distant est accessible (ping, telnet)
- Vérifier les règles firewall
- Vérifier `pg_hba.conf` sur le serveur distant

#### Performances Dégradées

```sql
-- Analyser les tables étrangères
ANALYZE fdw_orders;

-- Vérifier le plan d'exécution
EXPLAIN (ANALYZE, VERBOSE, BUFFERS)  
SELECT * FROM fdw_orders WHERE status = 'pending';  
```

#### Timeout sur Requêtes Longues

```sql
-- Augmenter le timeout pour les connexions FDW
ALTER SERVER orders_server OPTIONS (
    ADD connect_timeout '30',
    ADD statement_timeout '300000'  -- 5 minutes en ms
);
```

## Exemple Complet : Dashboard de Ventes

Voici un exemple complet de mise en place d'un système de reporting fédéré.

### Configuration Initiale

```sql
-- ========================================
-- Base de reporting : reporting_db
-- ========================================

-- 1. Extensions
CREATE EXTENSION IF NOT EXISTS postgres_fdw;

-- 2. Schéma dédié pour les tables étrangères
CREATE SCHEMA fdw;

-- 3. Serveurs distants
CREATE SERVER users_srv FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (host 'users-db', port '5432', dbname 'users_db');

CREATE SERVER orders_srv FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (host 'orders-db', port '5432', dbname 'orders_db');

CREATE SERVER catalog_srv FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (host 'catalog-db', port '5432', dbname 'catalog_db');

-- 4. User mappings (pour l'utilisateur reporting_app)
CREATE USER MAPPING FOR reporting_app SERVER users_srv
    OPTIONS (user 'fdw_reader', password 'pwd_users');

CREATE USER MAPPING FOR reporting_app SERVER orders_srv
    OPTIONS (user 'fdw_reader', password 'pwd_orders');

CREATE USER MAPPING FOR reporting_app SERVER catalog_srv
    OPTIONS (user 'fdw_reader', password 'pwd_catalog');

-- 5. Import des schémas distants
IMPORT FOREIGN SCHEMA public
    LIMIT TO (users)
    FROM SERVER users_srv INTO fdw;

IMPORT FOREIGN SCHEMA public
    LIMIT TO (orders, order_items)
    FROM SERVER orders_srv INTO fdw;

IMPORT FOREIGN SCHEMA public
    LIMIT TO (products, categories)
    FROM SERVER catalog_srv INTO fdw;
```

### Vue Consolidée pour le Dashboard

```sql
-- Vue de synthèse des ventes
CREATE VIEW v_sales_dashboard AS  
SELECT  
    date_trunc('day', o.created_at) AS sale_date,
    cat.name AS category,
    u.country,
    COUNT(DISTINCT o.id) AS orders_count,
    COUNT(DISTINCT o.user_id) AS customers_count,
    SUM(oi.quantity) AS items_sold,
    SUM(oi.quantity * oi.unit_price) AS revenue
FROM fdw.orders o  
JOIN fdw.order_items oi ON oi.order_id = o.id  
JOIN fdw.products p ON p.id = oi.product_id  
JOIN fdw.categories cat ON cat.id = p.category_id  
JOIN fdw.users u ON u.id = o.user_id  
WHERE o.status = 'completed'  
GROUP BY  
    date_trunc('day', o.created_at),
    cat.name,
    u.country;

-- Vue matérialisée pour performances
CREATE MATERIALIZED VIEW mv_daily_sales AS  
SELECT * FROM v_sales_dashboard  
WHERE sale_date >= NOW() - INTERVAL '90 days';  

CREATE INDEX idx_mv_daily_sales_date ON mv_daily_sales(sale_date);  
CREATE INDEX idx_mv_daily_sales_category ON mv_daily_sales(category);  

-- Rafraîchissement automatique (avec pg_cron)
-- SELECT cron.schedule('refresh-sales', '0 * * * *',
--     'REFRESH MATERIALIZED VIEW CONCURRENTLY mv_daily_sales');
```

### Requêtes de Reporting

```sql
-- Top 10 des catégories ce mois
SELECT
    category,
    SUM(revenue) AS total_revenue,
    SUM(orders_count) AS total_orders
FROM mv_daily_sales  
WHERE sale_date >= date_trunc('month', CURRENT_DATE)  
GROUP BY category  
ORDER BY total_revenue DESC  
LIMIT 10;  

-- Évolution par pays
SELECT
    country,
    sale_date,
    SUM(revenue) AS daily_revenue
FROM mv_daily_sales  
WHERE sale_date >= NOW() - INTERVAL '30 days'  
GROUP BY country, sale_date  
ORDER BY country, sale_date;  
```

## Conclusion

Les **Foreign Data Wrappers** sont un outil puissant pour créer une **fédération de données** dans une architecture microservices. Ils permettent d'interroger des sources hétérogènes avec du SQL standard, sans dupliquer les données ni créer de couplage fort entre les services.

Cependant, ils ne sont pas une solution miracle. Les FDW sont idéaux pour le **reporting et l'analyse**, mais moins adaptés aux requêtes temps réel à haute fréquence ou aux écritures distribuées.

PostgreSQL, grâce à `postgres_fdw` et son écosystème de wrappers communautaires, offre une flexibilité remarquable pour intégrer des données de multiples sources dans une vue unifiée.

---

## Points Clés à Retenir

- **FDW** = mécanisme PostgreSQL pour accéder à des données externes comme des tables locales  
- **postgres_fdw** : wrapper optimisé pour la fédération entre bases PostgreSQL  
- **Pushdown** : PostgreSQL délègue filtres, tris et agrégations au serveur distant  
- **Architecture recommandée** : Base de reporting dédiée avec FDW en lecture seule  
- **Sécurité** : Utilisez des comptes readonly et SSL pour les connexions  
- **Performance** : Vues matérialisées pour cacher les résultats fédérés  
- **Limitations** : Pas de transactions distribuées, latence réseau, jointures cross-server coûteuses  
- **Cas d'usage** : Reporting, migration progressive, fédération multi-cloud
