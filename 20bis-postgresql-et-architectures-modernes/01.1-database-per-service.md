# 20bis.1.1 — Database per Service vs Shared Database

## Introduction

Lorsqu'on conçoit une application moderne basée sur une architecture microservices, une question fondamentale se pose rapidement : **comment organiser les bases de données ?** Faut-il que chaque microservice possède sa propre base de données, ou plusieurs services peuvent-ils partager une même base ?

Cette décision architecturale a des conséquences profondes sur la scalabilité, la maintenance, l'autonomie des équipes et la cohérence des données de votre système.

Dans ce chapitre, nous allons explorer les deux approches principales — **Database per Service** et **Shared Database** — en présentant leurs avantages, inconvénients et cas d'usage, avec un focus particulier sur PostgreSQL.

---

## Qu'est-ce qu'une Architecture Microservices ?

Avant d'aborder la gestion des bases de données, rappelons brièvement ce qu'est une architecture microservices.

Une **architecture microservices** consiste à découper une application en plusieurs services indépendants, chacun responsable d'une fonctionnalité métier spécifique. Par exemple, une application e-commerce pourrait être divisée en :

- Un service **Utilisateurs** (gestion des comptes)
- Un service **Catalogue** (produits et catégories)
- Un service **Commandes** (panier et achats)
- Un service **Paiements** (transactions financières)
- Un service **Notifications** (emails, SMS)

Chaque service peut être développé, déployé et mis à l'échelle indépendamment des autres.

---

## Approche 1 : Shared Database (Base de Données Partagée)

### Principe

Dans l'approche **Shared Database**, tous les microservices accèdent à une **unique base de données PostgreSQL**. Les tables de tous les services coexistent dans la même instance.

```
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  Service    │  │  Service    │  │  Service    │
│ Utilisateurs│  │  Catalogue  │  │  Commandes  │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                │                │
       └────────────────┼────────────────┘
                        │
                        ▼
              ┌─────────────────┐
              │   PostgreSQL    │
              │  (Base unique)  │
              │                 │
              │ - users         │
              │ - products      │
              │ - orders        │
              │ - payments      │
              └─────────────────┘
```

### Avantages

**1. Simplicité de mise en œuvre**

C'est l'approche la plus simple à mettre en place. Il n'y a qu'une seule base de données à installer, configurer et maintenir. Pour une petite équipe ou un projet en démarrage, cela réduit considérablement la complexité opérationnelle.

**2. Transactions ACID natives**

Avec une base partagée, vous pouvez utiliser les **transactions PostgreSQL classiques** pour garantir la cohérence des données entre plusieurs tables, même si elles "appartiennent" à différents services.

```sql
BEGIN;
  -- Créer la commande (service Commandes)
  INSERT INTO orders (user_id, total) VALUES (42, 99.99);

  -- Déduire le stock (service Catalogue)
  UPDATE products SET stock = stock - 1 WHERE id = 101;

  -- Enregistrer le paiement (service Paiements)
  INSERT INTO payments (order_id, amount, status) VALUES (1, 99.99, 'completed');
COMMIT;
```

Cette transaction garantit que soit toutes les opérations réussissent, soit aucune n'est appliquée (atomicité).

**3. Jointures directes entre services**

Vous pouvez écrire des requêtes SQL qui joignent des tables de différents domaines métier :

```sql
SELECT u.name, o.total, p.name AS product  
FROM users u  
JOIN orders o ON o.user_id = u.id  
JOIN order_items oi ON oi.order_id = o.id  
JOIN products p ON p.id = oi.product_id  
WHERE o.created_at > NOW() - INTERVAL '7 days';  
```

**4. Pas de duplication de données**

Les données existent en un seul endroit. Il n'y a pas besoin de synchroniser des informations entre plusieurs bases.

**5. Coûts d'infrastructure réduits**

Une seule instance PostgreSQL à gérer, ce qui simplifie les sauvegardes, la supervision et réduit les coûts d'hébergement.

### Inconvénients

**1. Couplage fort entre services**

C'est le problème majeur. Si le service Commandes accède directement à la table `users`, il devient dépendant de sa structure. Toute modification du schéma de `users` peut casser le service Commandes.

Ce couplage va à l'encontre du principe fondamental des microservices : **l'indépendance des services**.

**2. Point de défaillance unique (Single Point of Failure)**

Si la base de données PostgreSQL tombe en panne, **tous les services** sont impactés simultanément. Il n'y a pas d'isolation des pannes.

**3. Difficultés de mise à l'échelle**

Tous les services partagent les mêmes ressources (CPU, mémoire, connexions). Si le service Catalogue génère des requêtes lourdes, cela peut impacter les performances du service Commandes.

Il est également impossible de choisir une technologie de base de données différente pour un service ayant des besoins spécifiques (par exemple, une base orientée graphe pour des recommandations).

**4. Goulot d'étranglement organisationnel**

Si plusieurs équipes travaillent sur différents services, elles doivent coordonner leurs modifications de schéma. Les migrations deviennent complexes et risquées.

**5. Limites de connexions**

PostgreSQL a un nombre limité de connexions simultanées. Avec de nombreux services, chacun ayant son propre pool de connexions, vous pouvez rapidement atteindre cette limite.

### Cas d'usage appropriés

L'approche Shared Database est adaptée pour :

- Les **petites équipes** (moins de 5 développeurs)
- Les **projets en phase de démarrage** (MVP, prototypes)
- Les applications avec un **domaine métier simple**
- Les situations où la **cohérence transactionnelle forte** est critique
- Les environnements avec des **contraintes budgétaires** importantes

## Approche 2 : Database per Service (Base par Service)

### Principe

Dans l'approche **Database per Service**, chaque microservice possède sa **propre base de données PostgreSQL dédiée**. Aucun autre service n'a le droit d'accéder directement à cette base.

```
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  Service    │  │  Service    │  │  Service    │
│ Utilisateurs│  │  Catalogue  │  │  Commandes  │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                │                │
       ▼                ▼                ▼
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│ PostgreSQL  │  │ PostgreSQL  │  │ PostgreSQL  │
│   users_db  │  │ catalog_db  │  │  orders_db  │
└─────────────┘  └─────────────┘  └─────────────┘
```

Les services communiquent entre eux via des **APIs** (REST, gRPC) ou des **événements asynchrones** (messages), jamais par accès direct à la base d'un autre service.

### Avantages

**1. Autonomie totale des équipes**

Chaque équipe est propriétaire de son service ET de sa base de données. Elle peut :

- Modifier le schéma sans impacter les autres services
- Choisir sa stratégie d'indexation
- Effectuer des migrations indépendamment
- Déployer à son propre rythme

Cette autonomie accélère considérablement le développement dans les grandes organisations.

**2. Isolation des pannes**

Si la base de données du service Catalogue tombe en panne, les services Utilisateurs et Commandes continuent de fonctionner (en mode dégradé éventuellement).

**3. Scalabilité indépendante**

Chaque base peut être dimensionnée selon les besoins spécifiques de son service :

- Le service Catalogue peut avoir besoin de beaucoup de mémoire pour des requêtes complexes
- Le service Commandes peut nécessiter des I/O rapides pour les écritures fréquentes
- Le service Utilisateurs peut rester sur une instance modeste

**4. Liberté technologique (Polyglot Persistence)**

Bien que nous parlions de PostgreSQL, cette approche permet de choisir la technologie la plus adaptée à chaque service :

- PostgreSQL pour les données relationnelles classiques
- Redis pour le cache et les sessions
- Elasticsearch pour la recherche full-text
- MongoDB pour des documents flexibles
- Neo4j pour des données en graphe

**5. Sécurité renforcée**

L'isolation limite la surface d'attaque. Une faille dans un service ne compromet que sa propre base de données.

### Inconvénients

**1. Complexité des transactions distribuées**

Sans base partagée, impossible d'utiliser une simple transaction SQL pour modifier des données de plusieurs services. Il faut recourir à des patterns comme :

- **Saga** : orchestration de transactions compensatoires  
- **Two-Phase Commit (2PC)** : coordination distribuée (rarement recommandé)  
- **Eventual Consistency** : accepter une cohérence à terme

Ces patterns ajoutent une complexité significative au code applicatif.

**2. Jointures impossibles**

Vous ne pouvez plus écrire de jointures SQL entre les données de différents services. Pour obtenir une vue combinée, vous devez :

- Effectuer plusieurs appels API et agréger côté application
- Maintenir des vues matérialisées dénormalisées
- Utiliser un service dédié de reporting

**3. Duplication de données**

Pour éviter les appels réseau constants, les services dupliquent souvent certaines données. Par exemple, le service Commandes peut stocker une copie du nom de l'utilisateur et du nom du produit, plutôt que d'interroger les autres services à chaque affichage.

Cette duplication nécessite des mécanismes de synchronisation (événements, CDC).

**4. Coûts opérationnels accrus**

Plusieurs bases de données signifient :

- Plus d'instances à superviser
- Plus de sauvegardes à gérer
- Plus de mises à jour à orchestrer
- Plus de ressources infrastructure

**5. Complexité des requêtes de reporting**

Les analyses transversales (par exemple, "chiffre d'affaires par catégorie de produit et par région utilisateur") deviennent complexes car les données sont dispersées.

### Cas d'usage appropriés

L'approche Database per Service est adaptée pour :

- Les **grandes organisations** avec plusieurs équipes indépendantes
- Les applications avec un **trafic important** nécessitant une scalabilité fine
- Les systèmes où l'**isolation des pannes** est critique
- Les projets matures ayant besoin de **flexibilité technologique**
- Les contextes où la **cohérence à terme** est acceptable

## Variante : Schéma par Service (Compromis)

PostgreSQL offre une solution intermédiaire grâce aux **schémas** (namespaces). Chaque service utilise son propre schéma au sein d'une même instance PostgreSQL.

```
┌─────────────────────────────────────────────┐
│              PostgreSQL (instance unique)   │
│                                             │
│  ┌─────────────┐  ┌─────────────┐           │
│  │ Schema:     │  │ Schema:     │           │
│  │ users       │  │ catalog     │           │
│  │ - users     │  │ - products  │           │
│  │ - profiles  │  │ - categories│           │
│  └─────────────┘  └─────────────┘           │
│                                             │
│  ┌─────────────┐  ┌─────────────┐           │
│  │ Schema:     │  │ Schema:     │           │
│  │ orders      │  │ payments    │           │
│  │ - orders    │  │ - payments  │           │
│  │ - items     │  │ - refunds   │           │
│  └─────────────┘  └─────────────┘           │
└─────────────────────────────────────────────┘
```

### Configuration PostgreSQL

Chaque service se connecte avec un utilisateur dédié ayant accès uniquement à son schéma :

```sql
-- Créer les schémas
CREATE SCHEMA users_schema;  
CREATE SCHEMA catalog_schema;  
CREATE SCHEMA orders_schema;  

-- Créer des rôles dédiés
CREATE ROLE users_service LOGIN PASSWORD 'secret1';  
CREATE ROLE catalog_service LOGIN PASSWORD 'secret2';  
CREATE ROLE orders_service LOGIN PASSWORD 'secret3';  

-- Attribuer les permissions
GRANT USAGE ON SCHEMA users_schema TO users_service;  
GRANT ALL ON ALL TABLES IN SCHEMA users_schema TO users_service;  

-- Configurer le search_path par défaut
ALTER ROLE users_service SET search_path TO users_schema;
```

### Avantages de cette variante

- Une seule instance à maintenir
- Isolation logique entre les services
- Possibilité de transactions cross-schémas si vraiment nécessaire
- Migration progressive vers Database per Service possible

### Limites

- Pas d'isolation physique (ressources partagées)
- Un seul point de défaillance reste présent

## Tableau Comparatif

| Critère | Shared Database | Database per Service |
|---------|-----------------|---------------------|
| **Complexité initiale** | Faible | Élevée |
| **Autonomie des équipes** | Faible | Élevée |
| **Transactions ACID** | Natives | Distribuées (Saga) |
| **Jointures SQL** | Possibles | Impossibles |
| **Isolation des pannes** | Aucune | Totale |
| **Scalabilité** | Limitée | Indépendante |
| **Cohérence des données** | Forte | Éventuelle |
| **Coût infrastructure** | Faible | Élevé |
| **Duplication de données** | Aucune | Fréquente |
| **Flexibilité techno** | Aucune | Totale |

## Comment Choisir ?

La décision dépend de plusieurs facteurs. Voici un guide pour vous aider :

### Choisissez Shared Database si :

- Vous démarrez un nouveau projet avec une petite équipe
- La cohérence transactionnelle forte est une exigence métier non négociable
- Vous avez des contraintes budgétaires importantes
- Le domaine métier est relativement simple et stable
- Vous préférez la simplicité à la flexibilité

### Choisissez Database per Service si :

- Vous avez plusieurs équipes travaillant sur différents services
- Chaque service a des exigences de performance ou de stockage différentes
- L'isolation des pannes est critique pour votre activité
- Vous êtes prêt à gérer la complexité des transactions distribuées
- Votre organisation valorise l'autonomie des équipes

### Approche évolutive recommandée

Dans la pratique, beaucoup d'organisations adoptent une **approche progressive** :

1. **Phase initiale** : Shared Database ou Schema per Service pour valider le produit rapidement  
2. **Phase de croissance** : Extraire progressivement les services critiques vers leur propre base  
3. **Phase mature** : Database per Service pour les services à fort trafic, base partagée pour les services secondaires

Cette stratégie permet de ne pas sur-architecturer au départ tout en gardant la possibilité d'évoluer.

## Bonnes Pratiques avec PostgreSQL

Quelle que soit l'approche choisie, voici quelques recommandations :

### Pour Shared Database

- Utilisez des **schémas séparés** pour chaque domaine métier
- Définissez des **rôles avec permissions minimales** pour chaque service
- Évitez les jointures entre schémas dans le code applicatif courant
- Documentez clairement les "propriétaires" de chaque table
- Utilisez **PgBouncer** pour gérer le pool de connexions

### Pour Database per Service

- Mettez en place une **stratégie de sauvegarde cohérente** pour toutes les bases
- Utilisez des **outils de supervision centralisés** (Prometheus, Grafana)
- Implémentez le pattern **Outbox** pour la publication fiable d'événements
- Considérez les **Foreign Data Wrappers** pour le reporting transversal
- Standardisez les configurations PostgreSQL entre les instances

---

## Conclusion

Le choix entre **Database per Service** et **Shared Database** n'est pas binaire. Il dépend de votre contexte : taille de l'équipe, exigences de cohérence, budget, et maturité de l'organisation.

PostgreSQL, grâce à sa flexibilité (schémas, Foreign Data Wrappers, réplication logique), s'adapte aux deux modèles et permet même des approches hybrides.

L'essentiel est de comprendre les **compromis** de chaque approche et de faire un choix éclairé en fonction de vos contraintes actuelles, tout en gardant à l'esprit les évolutions futures de votre système.

---

## Points Clés à Retenir

- **Shared Database** : Simple, cohérent, mais crée du couplage et limite la scalabilité  
- **Database per Service** : Autonome, scalable, mais complexifie les transactions et les requêtes  
- **Schema per Service** : Compromis intéressant pour débuter avec PostgreSQL
- Le choix dépend de la **taille de l'équipe**, des **exigences de cohérence** et du **budget**
- Une **approche évolutive** permet de commencer simple et de migrer progressivement
