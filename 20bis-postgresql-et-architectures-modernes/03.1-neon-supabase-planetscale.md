# 20bis.3.1. Neon, Supabase, PlanetScale : Les Alternatives Serverless

## Introduction

Avec l'essor des architectures modernes, une nouvelle génération de bases de données a émergé : les bases de données **serverless**. Ces solutions permettent aux développeurs de se concentrer sur leur code plutôt que sur l'administration de l'infrastructure.

Dans ce chapitre, nous allons explorer trois acteurs majeurs de cet écosystème : **Neon**, **Supabase** et **PlanetScale**. Nous verrons leurs caractéristiques, leurs différences, et dans quels cas les utiliser.

---

## Qu'est-ce qu'une Base de Données Serverless ?

Avant de plonger dans les détails de chaque solution, clarifions ce que signifie "serverless" dans le contexte des bases de données.

### Définition Simple

Une base de données serverless est un service de base de données qui :

1. **S'adapte automatiquement** à la charge de travail (scaling automatique)  
2. **Ne nécessite pas de gestion de serveur** par l'utilisateur  
3. **Facture à l'usage** plutôt qu'à la capacité provisionnée  
4. **Peut descendre à zéro** lorsqu'il n'y a pas d'activité (scale-to-zero)

### Pourquoi C'est Important ?

Imaginez que vous lancez une application. Avec une base de données traditionnelle :

- Vous devez choisir la taille du serveur à l'avance
- Vous payez 24h/24, même si votre application dort la nuit
- Si vous avez un pic de trafic, votre base peut être submergée

Avec une base serverless :

- La capacité s'ajuste automatiquement
- Vous ne payez que ce que vous consommez
- Les pics de trafic sont gérés automatiquement

## Neon : Le PostgreSQL Cloud-Native

### Présentation

**Neon** est une plateforme serverless construite spécifiquement pour PostgreSQL. Fondée en 2021, elle a été conçue dès le départ pour tirer parti des architectures cloud modernes.

### Architecture Innovante

Neon utilise une architecture qui **sépare le calcul du stockage** :

```
┌──────────────────────────────────────────────────────────┐
│                    Architecture Neon                     │
├──────────────────────────────────────────────────────────┤
│                                                          │
│   ┌─────────────┐    ┌─────────────┐    ┌─────────────┐  │
│   │  Compute 1  │    │  Compute 2  │    │  Compute N  │  │
│   │  (endpoint) │    │  (endpoint) │    │  (endpoint) │  │
│   └──────┬──────┘    └──────┬──────┘    └──────┬──────┘  │
│          │                  │                  │         │
│          └──────────────────┼──────────────────┘         │
│                             │                            │
│                             ▼                            │
│              ┌──────────────────────────┐                │
│              │     Stockage Partagé     │                │
│              │   (Distributed Storage)  │                │
│              └──────────────────────────┘                │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

Cette séparation permet :

- **Le scale-to-zero** : Les instances de calcul peuvent s'éteindre complètement  
- **Le branching** : Créer des copies instantanées de la base de données  
- **L'efficacité des coûts** : Ne payer que pour le stockage utilisé

### Fonctionnalités Clés

#### 1. Branching de Base de Données

La fonctionnalité phare de Neon est le **branching**. Comme avec Git pour le code, vous pouvez créer des "branches" de votre base de données :

| Fonctionnalité | Description |
|----------------|-------------|
| **Branch de développement** | Créer une copie de la production pour tester |
| **Branch de preview** | Une base par pull request |
| **Restauration point-in-time** | Revenir à n'importe quel moment |

Le branching utilise la technologie **copy-on-write**, ce qui signifie qu'une branche ne consomme de l'espace que pour les données modifiées.

#### 2. Scale-to-Zero

Neon peut suspendre automatiquement les instances inactives :

- Après quelques minutes d'inactivité, l'instance s'éteint
- À la prochaine requête, elle redémarre (cold start d'environ 500ms à 2s)
- Idéal pour les environnements de développement et les applications à faible trafic

#### 3. Autoscaling

Pour les applications en production, Neon ajuste automatiquement les ressources :

- Minimum configurable (ex: 0.25 vCPU)
- Maximum configurable (ex: 4 vCPU)
- Ajustement en temps réel selon la charge

### Compatibilité PostgreSQL

Neon est **100% compatible PostgreSQL**. Cela signifie :

- Toutes les fonctionnalités SQL de PostgreSQL fonctionnent
- Les extensions populaires sont supportées (pgvector, PostGIS, etc.)
- Les outils existants (psql, pgAdmin, ORMs) fonctionnent sans modification

### Cas d'Usage Idéaux

Neon est particulièrement adapté pour :

1. **Applications en développement** : Branches gratuites pour chaque environnement  
2. **Startups** : Modèle de prix prévisible, scale-to-zero économique  
3. **Applications avec trafic variable** : Autoscaling automatique  
4. **Workflows CI/CD** : Branches éphémères pour les tests

### Limitations à Connaître

- **Cold starts** : Le redémarrage après scale-to-zero peut prendre 1-2 secondes  
- **Extensions limitées** : Toutes les extensions PostgreSQL ne sont pas disponibles  
- **Régions limitées** : Moins de régions que les clouds traditionnels

## Supabase : Le "Firebase Open Source"

### Présentation

**Supabase** se positionne comme une alternative open source à Firebase, construite sur PostgreSQL. Fondée en 2020, elle offre bien plus qu'une simple base de données.

### Une Plateforme Complète

Contrairement à Neon qui se concentre sur la base de données, Supabase fournit un **Backend-as-a-Service (BaaS)** complet :

```
┌─────────────────────────────────────────────────────────────┐
│                    Écosystème Supabase                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐  │
│  │  Database   │  │    Auth     │  │      Storage        │  │
│  │ (PostgreSQL)│  │ (Utilisat.) │  │ (Fichiers/Images)   │  │
│  └─────────────┘  └─────────────┘  └─────────────────────┘  │
│                                                             │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐  │
│  │  Realtime   │  │   Edge      │  │     API Auto-       │  │
│  │ (WebSockets)│  │  Functions  │  │     générée         │  │
│  └─────────────┘  └─────────────┘  └─────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Composants Principaux

#### 1. Base de Données PostgreSQL

Au cœur de Supabase se trouve une instance PostgreSQL complète :

- Accès direct via SQL
- Support des extensions (pgvector, PostGIS, etc.)
- Row Level Security (RLS) intégrée

#### 2. Authentification Intégrée

Supabase fournit un système d'authentification prêt à l'emploi :

| Méthode | Description |
|---------|-------------|
| Email/Mot de passe | Authentification classique |
| Magic Links | Connexion sans mot de passe par email |
| OAuth | Google, GitHub, Apple, etc. |
| Téléphone | SMS OTP |

#### 3. API Auto-générée

À partir de votre schéma de base de données, Supabase génère automatiquement :

- **API REST** : Endpoints CRUD pour chaque table  
- **API GraphQL** : Requêtes flexibles via pg_graphql  
- **Documentation Swagger** : Documentation automatique

Exemple d'utilisation côté client :

```javascript
// Récupérer tous les utilisateurs actifs
const { data, error } = await supabase
  .from('users')
  .select('*')
  .eq('active', true)
```

#### 4. Temps Réel (Realtime)

Supabase permet d'écouter les changements de la base en temps réel :

```javascript
// S'abonner aux nouvelles insertions dans la table "messages"
supabase
  .channel('messages')
  .on('postgres_changes',
      { event: 'INSERT', schema: 'public', table: 'messages' },
      (payload) => {
        console.log('Nouveau message:', payload.new)
      }
  )
  .subscribe()
```

#### 5. Stockage de Fichiers

Un système de stockage S3-compatible pour vos fichiers :

- Upload/download de fichiers
- Transformations d'images à la volée
- Contrôle d'accès basé sur les politiques PostgreSQL

#### 6. Edge Functions

Des fonctions serverless exécutées au plus près des utilisateurs :

- Basées sur Deno (TypeScript/JavaScript)
- Déployées globalement
- Idéales pour les webhooks, le traitement de paiements, etc.

### Open Source et Self-Hosted

Un avantage majeur de Supabase est son caractère **open source** :

- Le code est disponible sur GitHub
- Vous pouvez héberger Supabase vous-même (self-hosting)
- Évite le vendor lock-in

Pour le self-hosting, Supabase fournit des configurations Docker :

```bash
# Cloner le dépôt
git clone https://github.com/supabase/supabase  
cd supabase/docker  

# Lancer avec Docker Compose
docker compose up -d
```

### Cas d'Usage Idéaux

Supabase excelle pour :

1. **Applications web/mobile complètes** : Tout le backend en un seul service  
2. **Prototypage rapide** : API générée automatiquement  
3. **Applications temps réel** : Chat, notifications, dashboards live  
4. **Développeurs cherchant une alternative à Firebase** : Avec les avantages de PostgreSQL

### Limitations à Connaître

- **Moins de contrôle sur l'infrastructure** : Abstraction de haut niveau  
- **Tarification** : Peut devenir coûteux à grande échelle  
- **Complexité cachée** : L'abstraction peut masquer des problèmes de performance  
- **Dépendance à l'écosystème** : Les fonctionnalités sont interconnectées

## PlanetScale : L'Exception MySQL

### Pourquoi en Parler ?

**PlanetScale** est souvent mentionné aux côtés de Neon et Supabase, mais avec une différence majeure : il est basé sur **MySQL/Vitess**, pas PostgreSQL.

Nous l'incluons car :

1. C'est un concurrent direct dans l'espace serverless  
2. Ses concepts innovants influencent l'industrie  
3. Certains développeurs hésitent entre PostgreSQL et MySQL

### Présentation

PlanetScale est construit sur **Vitess**, la technologie développée par YouTube pour faire évoluer MySQL à l'échelle mondiale.

### Caractéristiques Distinctives

#### 1. Branching (comme Neon)

PlanetScale a popularisé le concept de database branching :

```
main (production)
  │
  ├── feature-user-profiles
  │     └── Modifications du schéma utilisateurs
  │
  └── feature-payments
        └── Nouvelles tables de paiement
```

#### 2. Non-blocking Schema Changes

Les modifications de schéma ne bloquent pas les requêtes :

- Les migrations s'exécutent en arrière-plan
- Pas de downtime
- Rollback facile

#### 3. Sharding Automatique

Vitess permet de distribuer les données automatiquement :

- Scalabilité horizontale
- Performances maintenues à grande échelle

### Tableau Comparatif avec les Solutions PostgreSQL

| Aspect | PlanetScale | Neon | Supabase |
|--------|-------------|------|----------|
| **Base** | MySQL/Vitess | PostgreSQL | PostgreSQL |
| **Branching** | ✅ | ✅ | ❌ |
| **Scale-to-zero** | ❌ | ✅ | ❌ |
| **Sharding** | ✅ Automatique | ❌ | ❌ |
| **Foreign Keys** | ⚠️ Limité | ✅ | ✅ |
| **Extensions** | ❌ | ✅ | ✅ |
| **API auto-générée** | ❌ | ❌ | ✅ |
| **Auth intégrée** | ❌ | ❌ | ✅ |

### Quand Choisir PlanetScale ?

PlanetScale peut être préférable si :

- Votre équipe est plus expérimentée en MySQL
- Vous avez besoin de sharding automatique
- Les migrations sans downtime sont critiques

### Quand Rester sur PostgreSQL ?

Préférez Neon ou Supabase si :

- Vous utilisez des fonctionnalités avancées de PostgreSQL (JSONB, arrays, extensions)
- Les clés étrangères sont importantes pour votre modèle de données
- Vous voulez utiliser des extensions comme pgvector ou PostGIS

## Autres Alternatives à Mentionner

L'écosystème des bases de données serverless évolue rapidement. Voici d'autres solutions à connaître :

### CockroachDB Serverless

- Base de données SQL distribuée
- Compatible PostgreSQL (wire protocol)
- Conçue pour la résilience globale

### Turso (libSQL)

- Fork de SQLite pour le edge
- Réplication globale
- Très faible latence

### Xata

- Base de données serverless avec recherche intégrée
- API simple et intuitive
- Fonctionnalités d'IA intégrées

### AlloyDB (Google Cloud)

- PostgreSQL compatible, optimisé par Google
- Performances améliorées pour les workloads analytiques
- Intégration native avec l'écosystème GCP

## Bonnes Pratiques pour les Bases Serverless

### 1. Gérer les Cold Starts

Les bases serverless qui scale-to-zero ont un temps de démarrage :

**Solutions :**
- Configurer un minimum de ressources pour la production
- Utiliser des health checks pour maintenir la base active
- Accepter le cold start pour les environnements de développement

### 2. Optimiser les Connexions

Les architectures serverless créent souvent de nombreuses connexions :

**Solutions :**
- Utiliser un connection pooler (PgBouncer, intégré dans Neon/Supabase)
- Configurer des timeouts appropriés
- Éviter d'ouvrir une connexion par requête HTTP

### 3. Surveiller les Coûts

Le modèle pay-per-use peut surprendre :

**Solutions :**
- Configurer des alertes de budget
- Monitorer les requêtes coûteuses
- Optimiser les requêtes pour réduire les lectures/écritures

### 4. Planifier la Portabilité

Même avec du PostgreSQL standard, certaines fonctionnalités sont propriétaires :

**Solutions :**
- Minimiser l'utilisation des APIs spécifiques au provider
- Utiliser des migrations de schéma versionnées
- Tester régulièrement sur PostgreSQL local

## Conclusion

Le paysage des bases de données serverless offre aujourd'hui des options matures et puissantes :

- **Neon** excelle pour les workflows de développement modernes avec son branching innovant et son scale-to-zero économique. C'est le choix idéal pour les équipes qui veulent du PostgreSQL pur avec une expérience développeur optimisée.

- **Supabase** est parfait pour les développeurs qui veulent un backend complet sans avoir à assembler plusieurs services. Son caractère open source et sa communauté active en font un choix sûr.

- **PlanetScale** reste une option solide pour les équipes MySQL, avec des fonctionnalités de migration de schéma particulièrement robustes.

Le choix final dépendra de vos besoins spécifiques : budget, fonctionnalités requises, expérience de l'équipe, et contraintes techniques. La bonne nouvelle est que ces trois solutions offrent des tiers gratuits suffisants pour expérimenter et faire un choix éclairé.
