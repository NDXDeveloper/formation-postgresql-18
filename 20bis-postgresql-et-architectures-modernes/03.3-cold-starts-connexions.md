# 20bis.3.3. Cold Starts et Gestion des Connexions

## Introduction

Dans le monde serverless, le terme **cold start** (démarrage à froid) revient constamment. Ce phénomène, souvent perçu comme un inconvénient, a des implications directes sur la façon dont vos applications interagissent avec PostgreSQL.

Ce chapitre explique ce qu'est un cold start, pourquoi il impacte les connexions à la base de données, et quelles stratégies adopter pour minimiser ses effets. Nous verrons également les meilleures pratiques pour gérer efficacement les connexions dans un contexte serverless.

---

## Qu'est-ce qu'un Cold Start ?

### Définition Simple

Un **cold start** se produit lorsqu'une fonction serverless doit être initialisée avant de pouvoir traiter une requête. C'est le temps nécessaire pour "réveiller" une fonction qui était inactive.

### Le Cycle de Vie d'une Fonction Serverless

Pour comprendre les cold starts, observons comment fonctionne une fonction serverless :

```
┌─────────────────────────────────────────────────────────────────────────┐
│                 CYCLE DE VIE D'UNE FONCTION SERVERLESS                  │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  COLD START (Démarrage à froid)                                         │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │ 1. Provisioning    │ 2. Téléchargement │ 3. Initialisation      │    │
│  │    du conteneur    │    du code        │    du runtime          │    │
│  │    (~100-500ms)    │    (~50-200ms)    │    (~50-500ms)         │    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                                    │                                    │
│                                    ▼                                    │
│  WARM STATE (État chaud)                                                │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │                                                                 │    │
│  │    Requête 1 ──► Exécution rapide (~10-50ms)                    │    │
│  │    Requête 2 ──► Exécution rapide (~10-50ms)                    │    │
│  │    Requête 3 ──► Exécution rapide (~10-50ms)                    │    │
│  │         ...                                                     │    │
│  │                                                                 │    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                                    │                                    │
│                            (Inactivité ~5-15 min)                       │
│                                    │                                    │
│                                    ▼                                    │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │                    INSTANCE DÉTRUITE                            │    │
│  │              (Retour à l'état froid)                            │    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### Les Phases d'un Cold Start

| Phase | Description | Durée typique |
|-------|-------------|---------------|
| **1. Provisioning** | Le cloud alloue un conteneur/micro-VM | 100-500 ms |
| **2. Téléchargement** | Le code de la fonction est chargé | 50-200 ms |
| **3. Initialisation runtime** | Node.js, Python, Java, etc. démarre | 50-500 ms |
| **4. Initialisation code** | Votre code d'init s'exécute | Variable |
| **5. Connexion DB** | Établissement de la connexion PostgreSQL | 50-300 ms |

**Total cold start** : 300 ms à 2+ secondes selon la configuration.

### Warm Start vs Cold Start

```
COLD START (Première requête ou après inactivité)
─────────────────────────────────────────────────
Requête ──► [Provisioning][Init Runtime][Init Code][Connexion DB] ──► Réponse
            └──────────────── 500ms - 2s ─────────────────────────┘


WARM START (Requêtes suivantes)
───────────────────────────────
Requête ──► [Exécution] ──► Réponse
            └── 10-50ms ──┘
```

La différence peut être **10x à 100x** en termes de latence !

## Impact des Cold Starts sur PostgreSQL

### Le Problème de la Connexion à la Base

Lors d'un cold start, la fonction doit établir une nouvelle connexion à PostgreSQL. Cette étape est souvent la plus coûteuse de l'initialisation :

```
Anatomie d'une connexion PostgreSQL
───────────────────────────────────

Client                                      PostgreSQL
  │                                              │
  │──── 1. TCP Handshake (SYN/ACK) ────────────► │  ~10-30ms
  │                                              │
  │──── 2. SSL/TLS Handshake ──────────────────► │  ~20-50ms
  │                                              │
  │──── 3. Authentification ───────────────────► │  ~10-30ms
  │         (SCRAM-SHA-256)                      │
  │                                              │
  │◄─── 4. Paramètres de session ──────────────  │  ~5-10ms
  │                                              │
  │         CONNEXION ÉTABLIE                    │
  │         Total: 50-150ms (local)              │
  │         Total: 100-300ms (distant/cloud)     │
```

### Facteurs Aggravants

Plusieurs éléments peuvent augmenter le temps de connexion :

| Facteur | Impact | Explication |
|---------|--------|-------------|
| **Distance géographique** | +50-200ms | Latence réseau entre régions |
| **SSL/TLS** | +20-50ms | Handshake cryptographique |
| **DNS Resolution** | +10-50ms | Résolution du hostname |
| **Authentification** | +10-30ms | SCRAM-SHA-256 vs md5 |
| **Charge du serveur** | Variable | Serveur occupé = connexion lente |

### Le Scénario Catastrophe : Thundering Herd

Imaginez un pic de trafic après une période d'inactivité :

```
                    THUNDERING HERD PROBLEM
                    ───────────────────────

Temps 0: Toutes les fonctions sont "froides" (inactives)

Temps 1: Pic de trafic soudain (100 requêtes simultanées)

┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐
│Cold │ │Cold │ │Cold │ │Cold │ │ ... │  100 cold starts
│Start│ │Start│ │Start│ │Start│ │     │  simultanés !
└──┬──┘ └──┬──┘ └──┬──┘ └──┬──┘ └──┬──┘
   │       │       │       │       │
   └───────┴───────┴───────┴───────┘
                   │
                   ▼
        ┌─────────────────────┐
        │     PostgreSQL      │
        │                     │
        │  100 demandes de    │  ← SURCHARGE !
        │  connexion en       │  ← Timeouts !
        │  même temps         │  ← Erreurs !
        │                     │
        └─────────────────────┘
```

Ce phénomène, appelé **"thundering herd"** (troupeau tonnant), peut :
- Saturer les connexions PostgreSQL
- Provoquer des timeouts
- Faire échouer des requêtes utilisateur
- Déclencher un effet cascade de retries

## Stratégies pour Réduire les Cold Starts

### 1. Provisioned Concurrency (AWS Lambda)

AWS permet de garder des instances "chaudes" en permanence :

```
SANS Provisioned Concurrency
────────────────────────────
Instances: 0 ──► Cold Start ──► 1 ──► Cold Start ──► 2 ...

AVEC Provisioned Concurrency (3 instances)
──────────────────────────────────────────
Instances: 3 (toujours prêtes)

Requête 1 ──► Instance chaude #1 ──► Réponse immédiate  
Requête 2 ──► Instance chaude #2 ──► Réponse immédiate  
Requête 3 ──► Instance chaude #3 ──► Réponse immédiate  
Requête 4 ──► Cold start (au-delà du provisioned)  
```

**Configuration AWS Lambda** :

```yaml
# serverless.yml (Serverless Framework)
functions:
  api:
    handler: handler.main
    provisionedConcurrency: 5  # 5 instances toujours chaudes
```

| Avantages | Inconvénients |
|-----------|---------------|
| Latence constante et prévisible | Coût fixe même sans trafic |
| Pas de cold start pour le trafic normal | Ne scale pas automatiquement |
| Connexions DB pré-établies | Dimensionnement à estimer |

### 2. Warming (Keep-Alive)

Technique qui consiste à "pinguer" régulièrement les fonctions pour les maintenir chaudes :

```
┌────────────────────────────────────────────────────────────┐
│                    STRATÉGIE DE WARMING                    │
├────────────────────────────────────────────────────────────┤
│                                                            │
│   CloudWatch Events (toutes les 5 minutes)                 │
│                         │                                  │
│                         ▼                                  │
│   ┌──────────────────────────────────────────────────────┐ │
│   │                  Lambda Warmer                       │ │
│   │                                                      │ │
│   │   ──► Ping Lambda API #1                             │ │
│   │   ──► Ping Lambda API #2                             │ │
│   │   ──► Ping Lambda API #3                             │ │
│   │                                                      │ │
│   └──────────────────────────────────────────────────────┘ │
│                                                            │
│   Les fonctions restent "chaudes" entre les pings          │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

**Exemple avec AWS EventBridge** :

```yaml
# serverless.yml
functions:
  api:
    handler: handler.main
    events:
      - http:
          path: /api
          method: any
      - schedule:
          rate: rate(5 minutes)  # Ping toutes les 5 min
          input:
            warmer: true
```

**Code de la fonction** :

```python
def handler(event, context):
    # Détecter les pings de warming
    if event.get('warmer'):
        print("Warming ping received")
        return {"statusCode": 200, "body": "warmed"}

    # Logique normale
    return process_request(event)
```

| Avantages | Inconvénients |
|-----------|---------------|
| Simple à implémenter | Coût des invocations de warming |
| Pas de changement d'architecture | Ne garantit pas N instances chaudes |
| Fonctionne sur tous les clouds | Inefficace pour le scaling |

### 3. Optimiser le Code d'Initialisation

Réduisez le temps de cold start en optimisant votre code :

#### Lazy Loading (Chargement Paresseux)

```python
# ❌ MAUVAIS : Import lourd au démarrage
import pandas as pd  
import numpy as np  
import tensorflow as tf  

def handler(event, context):
    # pandas/numpy/tf chargés même si non utilisés
    return simple_operation()


# ✅ BON : Import uniquement si nécessaire
def handler(event, context):
    if needs_ml_processing(event):
        import tensorflow as tf
        return ml_operation(tf, event)

    return simple_operation()
```

#### Minimiser les Dépendances

```
Taille du package vs Cold Start
───────────────────────────────

Package 1 MB   ──► Cold start ~200ms  
Package 10 MB  ──► Cold start ~400ms  
Package 50 MB  ──► Cold start ~800ms  
Package 250 MB ──► Cold start ~2000ms+  
```

**Conseils** :
- Utilisez des layers pour les dépendances partagées
- Excluez les fichiers inutiles (tests, docs)
- Préférez les bibliothèques légères

#### Choisir le Bon Runtime

| Runtime | Cold Start typique | Notes |
|---------|-------------------|-------|
| **Python** | 200-400ms | Bon équilibre |
| **Node.js** | 150-300ms | Parmi les plus rapides |
| **Go** | 50-150ms | Binaire compilé, très rapide |
| **Rust** | 50-150ms | Excellent, mais courbe d'apprentissage |
| **Java** | 500ms-3s | JVM lourde, utiliser GraalVM |
| **.NET** | 300-600ms | Amélioration avec .NET 6+ |

## Gestion Optimale des Connexions

### Le Défi : Connexions et Fonctions Serverless

Dans une architecture serverless, la gestion des connexions est complexe car :

1. **Chaque instance a sa propre connexion** (pas de partage entre instances)  
2. **Les instances sont éphémères** (connexions abandonnées)  
3. **Le nombre d'instances est imprévisible** (scaling automatique)

### Stratégie 1 : Connexion dans le Handler (Simple mais Inefficace)

```python
import psycopg2

def handler(event, context):
    # ❌ Nouvelle connexion à chaque invocation
    conn = psycopg2.connect(
        host="db.example.com",
        database="mydb",
        user="user",
        password="password"
    )

    try:
        cursor = conn.cursor()
        cursor.execute("SELECT * FROM users WHERE id = %s", [event['user_id']])
        result = cursor.fetchone()
        return {"user": result}
    finally:
        conn.close()  # Connexion fermée à chaque fois
```

**Problèmes** :
- Nouvelle connexion à CHAQUE requête (~100-300ms)
- Gaspillage de ressources
- Pression sur PostgreSQL

### Stratégie 2 : Réutilisation de Connexion (Recommandée)

```python
import psycopg2

# Connexion créée UNE FOIS lors du cold start
# et réutilisée pour les invocations suivantes
connection = None

def get_connection():
    global connection

    if connection is None or connection.closed:
        connection = psycopg2.connect(
            host="db.example.com",
            database="mydb",
            user="user",
            password="password"
        )

    return connection

def handler(event, context):
    conn = get_connection()  # Réutilise la connexion existante

    cursor = conn.cursor()
    cursor.execute("SELECT * FROM users WHERE id = %s", [event['user_id']])
    result = cursor.fetchone()

    # NE PAS fermer la connexion !
    return {"user": result}
```

```
Comportement avec réutilisation
───────────────────────────────

Cold Start:
  [Init Runtime] ──► [Créer Connexion] ──► [Handler] ──► Réponse
                         100-300ms

Warm Invocations:
  [Handler avec connexion existante] ──► Réponse
              ~10-50ms (pas de connexion à créer)
```

### Stratégie 3 : Gestion Robuste avec Retry et Reconnexion

```python
import psycopg2  
from psycopg2 import OperationalError  
import time  

connection = None

def get_connection(max_retries=3, retry_delay=0.5):
    global connection

    # Vérifier si la connexion existante est valide
    if connection is not None:
        try:
            # Test rapide de la connexion
            cursor = connection.cursor()
            cursor.execute("SELECT 1")
            cursor.close()
            return connection
        except OperationalError:
            # Connexion morte, on la réinitialise
            print("Connection stale, reconnecting...")
            connection = None

    # Créer une nouvelle connexion avec retry
    for attempt in range(max_retries):
        try:
            connection = psycopg2.connect(
                host="db.example.com",
                database="mydb",
                user="user",
                password="password",
                connect_timeout=5,
                options="-c statement_timeout=30000"  # 30s max par requête
            )
            print(f"Connection established (attempt {attempt + 1})")
            return connection

        except OperationalError as e:
            print(f"Connection attempt {attempt + 1} failed: {e}")
            if attempt < max_retries - 1:
                time.sleep(retry_delay * (attempt + 1))  # Backoff exponentiel
            else:
                raise

def handler(event, context):
    try:
        conn = get_connection()
        cursor = conn.cursor()
        cursor.execute("SELECT * FROM users WHERE id = %s", [event['user_id']])
        result = cursor.fetchone()
        cursor.close()
        return {"statusCode": 200, "body": {"user": result}}

    except OperationalError as e:
        # Réinitialiser la connexion pour la prochaine invocation
        global connection
        connection = None
        return {"statusCode": 503, "body": "Database temporarily unavailable"}
```

### Stratégie 5 : Context Manager pour Transactions

Pour les opérations nécessitant des transactions explicites :

```python
from contextlib import contextmanager  
import psycopg2  

connection = None

def get_connection():
    global connection
    if connection is None or connection.closed:
        connection = psycopg2.connect(...)
    return connection

@contextmanager
def transaction():
    """Context manager pour transactions avec commit/rollback automatique."""
    conn = get_connection()
    try:
        yield conn
        conn.commit()
    except Exception:
        conn.rollback()
        raise

def handler(event, context):
    with transaction() as conn:
        cursor = conn.cursor()

        # Plusieurs opérations dans une transaction
        cursor.execute("UPDATE accounts SET balance = balance - %s WHERE id = %s",
                      [event['amount'], event['from_account']])
        cursor.execute("UPDATE accounts SET balance = balance + %s WHERE id = %s",
                      [event['amount'], event['to_account']])
        cursor.execute("INSERT INTO transfers (from_id, to_id, amount) VALUES (%s, %s, %s)",
                      [event['from_account'], event['to_account'], event['amount']])

        # Commit automatique à la sortie du with

    return {"statusCode": 200, "body": "Transfer completed"}
```

## Patterns Avancés

### Pattern 1 : Connexion avec Timeout Adaptatif

Adaptez le timeout selon le contexte (cold start vs warm) :

```python
import psycopg2  
import os  

connection = None  
is_cold_start = True  

def get_connection():
    global connection, is_cold_start

    if connection is None or connection.closed:
        # Timeout plus long pour cold start (connexion initiale)
        # Timeout court pour reconnexion (le serveur est probablement OK)
        timeout = 10 if is_cold_start else 3

        connection = psycopg2.connect(
            host=os.environ['DB_HOST'],
            database=os.environ['DB_NAME'],
            user=os.environ['DB_USER'],
            password=os.environ['DB_PASSWORD'],
            connect_timeout=timeout
        )

        is_cold_start = False

    return connection
```

### Pattern 2 : Health Check de Connexion

Vérifiez périodiquement la santé de la connexion :

```python
import time

connection = None  
last_health_check = 0  
HEALTH_CHECK_INTERVAL = 60  # secondes  

def get_healthy_connection():
    global connection, last_health_check

    current_time = time.time()

    # Créer connexion si nécessaire
    if connection is None:
        connection = create_connection()
        last_health_check = current_time
        return connection

    # Health check périodique
    if current_time - last_health_check > HEALTH_CHECK_INTERVAL:
        try:
            cursor = connection.cursor()
            cursor.execute("SELECT 1")
            cursor.close()
            last_health_check = current_time
        except:
            # Connexion morte, recréer
            try:
                connection.close()
            except:
                pass
            connection = create_connection()
            last_health_check = current_time

    return connection
```

### Pattern 3 : Métriques de Connexion

Instrumentez vos connexions pour le monitoring :

```python
import time  
import logging  

connection = None  
metrics = {  
    'connection_attempts': 0,
    'connection_failures': 0,
    'connection_reuses': 0,
    'total_connection_time_ms': 0
}

def get_connection_with_metrics():
    global connection, metrics

    if connection is not None and not connection.closed:
        metrics['connection_reuses'] += 1
        return connection

    metrics['connection_attempts'] += 1
    start_time = time.time()

    try:
        connection = psycopg2.connect(...)
        elapsed_ms = (time.time() - start_time) * 1000
        metrics['total_connection_time_ms'] += elapsed_ms

        logging.info(f"New connection established in {elapsed_ms:.2f}ms")
        return connection

    except Exception as e:
        metrics['connection_failures'] += 1
        logging.error(f"Connection failed: {e}")
        raise

def handler(event, context):
    # À la fin, loguer les métriques
    conn = get_connection_with_metrics()
    # ... logique

    # Log métriques (ou envoi vers CloudWatch/Datadog)
    if context.get_remaining_time_in_millis() > 1000:
        logging.info(f"Connection metrics: {metrics}")
```

## Gestion des Erreurs de Connexion

### Types d'Erreurs Courantes

| Erreur | Cause | Solution |
|--------|-------|----------|
| `connection refused` | PostgreSQL down ou mauvais port | Vérifier l'endpoint et le port |
| `timeout expired` | Réseau lent ou serveur surchargé | Augmenter timeout, vérifier réseau |
| `too many connections` | Limite atteinte | Utiliser un pooler |
| `SSL SYSCALL error` | Connexion interrompue | Reconnexion automatique |
| `server closed connection` | Idle timeout côté serveur | Réutiliser avec vérification |

### Gestion Complète des Erreurs

```python
import psycopg2  
from psycopg2 import OperationalError, InterfaceError  
import logging  

connection = None

class DatabaseConnectionError(Exception):
    """Erreur de connexion à la base de données."""
    pass

def get_connection():
    global connection

    # Cas 1: Pas de connexion existante
    if connection is None:
        return create_new_connection()

    # Cas 2: Connexion marquée comme fermée
    if connection.closed:
        logging.info("Connection was closed, creating new one")
        return create_new_connection()

    # Cas 3: Vérifier que la connexion est vivante
    try:
        with connection.cursor() as cur:
            cur.execute("SELECT 1")
        return connection
    except (OperationalError, InterfaceError) as e:
        logging.warning(f"Connection check failed: {e}")
        return create_new_connection()

def create_new_connection(max_retries=3):
    global connection

    for attempt in range(max_retries):
        try:
            connection = psycopg2.connect(
                host="db.example.com",
                database="mydb",
                user="user",
                password="password",
                connect_timeout=5
            )
            connection.autocommit = False
            logging.info(f"Connection created (attempt {attempt + 1})")
            return connection

        except OperationalError as e:
            logging.error(f"Attempt {attempt + 1} failed: {e}")

            if "too many connections" in str(e):
                # Attendre avant de réessayer
                time.sleep(1)
            elif "connection refused" in str(e):
                # Le serveur est probablement down
                raise DatabaseConnectionError("Database server unavailable")

            if attempt == max_retries - 1:
                raise DatabaseConnectionError(f"Failed after {max_retries} attempts: {e}")

def handler(event, context):
    try:
        conn = get_connection()
        # ... logique métier
        return {"statusCode": 200, "body": "Success"}

    except DatabaseConnectionError as e:
        logging.error(f"Database error: {e}")
        return {"statusCode": 503, "body": "Service temporarily unavailable"}

    except Exception as e:
        logging.error(f"Unexpected error: {e}")
        # Invalider la connexion pour la prochaine invocation
        global connection
        if connection:
            try:
                connection.rollback()
            except:
                pass
            connection = None
        return {"statusCode": 500, "body": "Internal error"}
```

## Bonnes Pratiques Récapitulatives

### 1. Réutilisez Toujours les Connexions

```python
# ✅ Connexion globale, réutilisée entre invocations
connection = None

def handler(event, context):
    conn = get_connection()  # Réutilise l'existante
    # ...
```

### 2. Utilisez un Connection Pooler en Production

Ne connectez jamais directement à PostgreSQL depuis des fonctions serverless à fort trafic.

### 3. Gérez les Connexions Mortes

```python
# ✅ Toujours vérifier et reconnecter si nécessaire
if connection is None or connection.closed:
    connection = create_connection()
```

### 4. Configurez des Timeouts Appropriés

```python
connection = psycopg2.connect(
    # ...
    connect_timeout=5,           # Max 5s pour établir la connexion
    options="-c statement_timeout=30000"  # Max 30s par requête
)
```

### 5. Ne Fermez Pas la Connexion dans le Handler

```python
# ❌ Ne pas faire
def handler(event, context):
    conn = get_connection()
    # ...
    conn.close()  # NON ! Garde-la pour la prochaine invocation

# ✅ Faire
def handler(event, context):
    conn = get_connection()
    # ...
    # Pas de close(), la connexion reste ouverte
```

### 6. Utilisez Provisioned Concurrency pour les APIs Critiques

Pour les endpoints sensibles à la latence, le surcoût vaut le coup.

### 7. Monitorez les Connexions

Suivez ces métriques :
- Nombre de connexions actives
- Temps de création de connexion
- Taux d'échec de connexion
- Connexions réutilisées vs créées

### 8. Testez les Scénarios de Panne

Simulez :
- PostgreSQL qui redémarre
- Timeouts réseau
- Pic de charge (thundering herd)

| Aspect | Mauvaise Pratique | Bonne Pratique |
|--------|-------------------|----------------|
| **Création** | Nouvelle connexion par requête | Réutiliser la connexion globale |

## Conclusion

Les cold starts et la gestion des connexions sont des défis inhérents aux architectures serverless, mais ils sont loin d'être insurmontables.

**Points clés à retenir :**

1. **Les cold starts sont inévitables** mais peuvent être minimisés via Provisioned Concurrency, warming, et optimisation du code.

2. **La connexion à PostgreSQL est souvent le goulot d'étranglement** lors d'un cold start. Réutilisez-la entre les invocations.

3. **Un connection pooler est indispensable** en production pour protéger PostgreSQL et améliorer les performances.

4. **La robustesse est essentielle** : gérez les erreurs, reconnectez automatiquement, et monitorez vos connexions.

5. **Mesurez et optimisez** : les métriques vous montreront où investir vos efforts.

En appliquant ces stratégies, vous pouvez construire des applications serverless performantes et fiables avec PostgreSQL, tout en gardant les avantages du modèle serverless : scaling automatique, facturation à l'usage, et absence de gestion d'infrastructure.
